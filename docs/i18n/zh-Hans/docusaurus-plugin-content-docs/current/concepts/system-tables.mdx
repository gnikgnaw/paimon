# 系统表 {#system-tables}

Paimon 提供了一套非常丰富的系统表，帮助用户更好地分析和查询 Paimon 表的状态：

1. 查询数据表的状态：数据系统表（Data System Table）。
2. 查询整个 Catalog 的全局状态：全局系统表（Global System Table）。

## 数据系统表 {#data-system-table}

数据系统表包含关于每张 Paimon 数据表的元数据和信息，例如已创建的快照（snapshot）以及正在使用的配置项。用户可以通过批式查询访问系统表。

目前，Flink、Spark、Trino 和 StarRocks 支持查询系统表。

在某些情况下，需要用反引号将表名括起来以避免语法解析冲突，例如三段式访问模式：
```sql
SELECT * FROM my_catalog.my_db.`my_table$snapshots`;
```

### Snapshots 表 {#snapshots-table}

你可以通过 snapshots 表查询表的快照历史信息，包括快照中发生的记录数。

```sql
SELECT * FROM my_table$snapshots;

/*

|  snapshot_id |  schema_id |     commit_user | commit_identifier |  commit_kind |             commit_time |             base_manifest_list |            delta_manifest_list |        changelog_manifest_list |  total_record_count |  delta_record_count |  changelog_record_count |      watermark |
+--------------+------------+-----------------+-------------------+--------------+-------------------------+--------------------------------+------------------------------- +--------------------------------+---------------------+---------------------+-------------------------+----------------+
|            2 |          0 | 7ca4cd28-98e... |                 2 |       APPEND | 2022-10-26 11:44:15.600 | manifest-list-31323d5f-76e6... | manifest-list-31323d5f-76e6... | manifest-list-31323d5f-76e6... |                   2 |                   2 |                       0 |  1666755855600 |
|            1 |          0 | 870062aa-3e9... |                 1 |       APPEND | 2022-10-26 11:44:15.148 | manifest-list-31593d5f-76e6... | manifest-list-31593d5f-76e6... | manifest-list-31593d5f-76e6... |                   1 |                   1 |                       0 |  1666755855148 |
```

通过查询 snapshots 表，你可以了解该表的提交（commit）与过期（expiration）信息，并对数据进行时间旅行（time travel）。

### Schemas 表 {#schemas-table}

你可以通过 schemas 表查询表的历史 schema（表结构）。

```sql
SELECT * FROM my_table$schemas;

/*
+-----------+--------------------------------+----------------+--------------+---------+---------+-------------------------+
| schema_id |                         fields | partition_keys | primary_keys | options | comment |       update_time       |
+-----------+--------------------------------+----------------+--------------+---------+---------+-------------------------+
|         0 | [{"id":0,"name":"word","typ... |             [] |     ["word"] |      {} |         | 2022-10-28 11:44:20.600 |
|         1 | [{"id":0,"name":"word","typ... |             [] |     ["word"] |      {} |         | 2022-10-27 11:44:15.600 |
|         2 | [{"id":0,"name":"word","typ... |             [] |     ["word"] |      {} |         | 2022-10-26 11:44:10.600 |
+-----------+--------------------------------+----------------+--------------+---------+---------+-------------------------+
3 rows in set
*/
```

你可以将 snapshots 表与 schemas 表进行 JOIN，以获取给定快照的字段。

```sql
SELECT s.snapshot_id, t.schema_id, t.fields 
    FROM my_table$snapshots s JOIN my_table$schemas t 
    ON s.schema_id=t.schema_id where s.snapshot_id=100;
```

### Options 表 {#options-table}

你可以通过 options 表查询表的配置信息，这些配置是在 DDL 中指定的。未显示的配置项将采用默认值。你可以参考 [配置](../maintenance/configurations#coreoptions)。

```sql
SELECT * FROM my_table$options;

/*
+------------------------+--------------------+
|         key            |        value       |
+------------------------+--------------------+
| snapshot.time-retained |         5 h        |
+------------------------+--------------------+
1 rows in set
*/
```

### Audit log 表 {#audit-log-table}

如果你需要审计表的 changelog（变更日志），可以使用 `audit_log` 系统表。通过 `audit_log` 表，在获取表的增量数据时可以得到 `rowkind` 列。你可以使用该列进行过滤等操作以完成审计。

`rowkind` 有四种取值：

- `+I`：插入操作。
- `-U`：更新操作，表示被更新行的更新前内容。
- `+U`：更新操作，表示被更新行的更新后内容。
- `-D`：删除操作。

```sql
SELECT * FROM my_table$audit_log;

/*
+------------------+-----------------+-----------------+
|     rowkind      |     column_0    |     column_1    |
+------------------+-----------------+-----------------+
|        +I        |      ...        |      ...        |
+------------------+-----------------+-----------------+
|        -U        |      ...        |      ...        |
+------------------+-----------------+-----------------+
|        +U        |      ...        |      ...        |
+------------------+-----------------+-----------------+
3 rows in set
*/
```

对于主键表，你可以启用 `table-read.sequence-number.enabled` 配置项，从而在输出中包含 `_SEQUENCE_NUMBER` 字段。

<Tabs groupId="audit-log-sequence-number">

<TabItem value="enable-via-alter-table" label="通过 ALTER TABLE 启用">

```sql
ALTER TABLE my_table SET ('table-read.sequence-number.enabled' = 'true');
```

</TabItem>

<TabItem value="enable-via-create-table" label="通过 CREATE TABLE 启用">

```sql
CREATE TABLE my_table (
    ...
) WITH (
    'table-read.sequence-number.enabled' = 'true'
);
```

</TabItem>

</Tabs>

```sql
SELECT * FROM my_table$audit_log;

/*
+------------------+--------------------+-----------------+-----------------+
|     rowkind      | _SEQUENCE_NUMBER   |     column_0    |     column_1    |
+------------------+--------------------+-----------------+-----------------+
|        +I        |                 0  |      ...        |      ...        |
+------------------+--------------------+-----------------+-----------------+
|        -U        |                 0  |      ...        |      ...        |
+------------------+--------------------+-----------------+-----------------+
|        +U        |                 1  |      ...        |      ...        |
+------------------+--------------------+-----------------+-----------------+
3 rows in set
*/
```

:::info

`table-read.sequence-number.enabled` 配置项无法通过 SQL hints 设置。

:::

### Binlog 表 {#binlog-table}

你可以通过 binlog 表查询 binlog。在 binlog 系统表中，更新前（update before）与更新后（update after）会被打包在同一行中。

目前，binlog 表无法显示 Flink 的计算列。

```sql
SELECT * FROM T$binlog;

/*
+------------------+----------------------+-----------------------+
|     rowkind      |       column_0       |       column_1        |
+------------------+----------------------+-----------------------+
|        +I        |       [col_0]        |       [col_1]         |
+------------------+----------------------+-----------------------+
|        +U        | [col_0_ub, col_0_ua] | [col_1_ub, col_1_ua]  |
+------------------+----------------------+-----------------------+
|        -D        |       [col_0]        |       [col_1]         |
+------------------+----------------------+-----------------------+
*/
```

与 audit_log 表类似，你也可以启用 `table-read.sequence-number.enabled`，从而在 binlog 表输出中包含 `_SEQUENCE_NUMBER`：

```sql
SELECT * FROM T$binlog;

/*
+------------------+--------------------+----------------------+-----------------------+
|     rowkind      | _SEQUENCE_NUMBER   |       column_0       |       column_1        |
+------------------+--------------------+----------------------+-----------------------+
|        +I        |                 0  |       [col_0]        |       [col_1]         |
+------------------+--------------------+----------------------+-----------------------+
|        +U        |                 1  | [col_0_ub, col_0_ua] | [col_1_ub, col_1_ua]  |
+------------------+--------------------+----------------------+-----------------------+
|        -D        |                 2  |       [col_0]        |       [col_1]         |
+------------------+--------------------+----------------------+-----------------------+
*/
```

### Read-optimized 表 {#read-optimized-table}

如果你需要极致的读取性能并且能够接受读取略微陈旧的数据，可以使用 `ro`（read-optimized，读优化）系统表。
Read-optimized 系统表通过只扫描无需合并的文件来提升读取性能。

对于主键表，`ro` 系统表只扫描最顶层的文件。
也就是说，`ro` 系统表只产出最近一次全量 Compaction 的结果。

:::info

不同的桶可能在不同时间执行全量 Compaction，因此不同 key 的值可能来自不同的快照。

:::

对于 Append 表（追加表），由于所有文件都可以无需合并地读取，`ro` 系统表的行为与普通的 Append 表相同。

```sql
SELECT * FROM my_table$ro;
```

### Tags 表 {#tags-table}

你可以通过 tags 表查询表的标签（Tag）历史信息，包括这些标签基于哪些快照以及快照的一些历史信息。你还可以获取所有标签名，并按名称对特定标签的数据进行时间旅行。

## 全局系统表 {#global-system-table}

全局系统表包含 Paimon 中所有存在的表的统计信息。为方便检索，我们创建了一个名为 `sys` 的引用系统数据库。
我们可以在 Flink 或 Spark 中通过 SQL 显示所有全局系统表：

```sql
USE sys;
SHOW TABLES;
```

### All Tables 表 {#all-tables-table}

显示所有数据库中的所有表。

```sql
SELECT * FROM sys.tables;

/*
```

### All Partitions 表 {#all-partitions-table}

### ALL Options 表 {#all-options-table}

该表类似于 [Options 表](#options-table)，但它显示所有数据库中所有表的配置项。

```sql
SELECT * FROM sys.all_table_options;

/*
+---------------+--------------------------------+--------------------------------+------------------+
| database_name |                     table_name |                            key |            value |
+---------------+--------------------------------+--------------------------------+------------------+
|         my_db |                    Orders_orc  |                         bucket |               -1 |
|         my_db |                        Orders2 |                         bucket |               -1 |
|         my_db |                        Orders2 |               sink.parallelism |                7 |
|         my_db2|                      OrdersSum |                         bucket |                1 |
+---------------+--------------------------------+--------------------------------+------------------+
7 rows in set
*/
```

### Catalog Options 表 {#catalog-options-table}

你可以通过 catalog options 表查询 Catalog 的配置信息。未显示的配置项将采用默认值。你可以参考 [配置](../maintenance/configurations#coreoptions)。

```sql
SELECT * FROM sys.catalog_options;

/*
+-----------+---------------------------+
|       key |                     value |
+-----------+---------------------------+
| warehouse | hdfs:///path/to/warehouse |
+-----------+---------------------------+
1 rows in set
*/
```
