# Kafka CDC {#kafka-cdc}

## 准备 Kafka Bundled Jar {#prepare-kafka-bundled-jar}

```
flink-sql-connector-kafka-*.jar
```

## 支持的格式 {#supported-formats}
Flink 提供了多种 Kafka CDC 格式：Canal Json、Debezium Json、Debezium Avro、Ogg Json、Maxwell Json 以及 Normal Json。
如果 Kafka topic 中的某条消息是使用变更数据捕获（Change Data Capture，CDC）工具从其他数据库捕获到的变更事件，那么你可以使用 Paimon Kafka CDC。它会把解析出的 INSERT、UPDATE、DELETE 消息写入 Paimon 表。

:::info

JSON 数据源可能会缺少一些信息。例如，Ogg 和 Maxwell 格式规范并不包含字段
类型；当你把 JSON 数据源写入 Flink Kafka Sink 时，它只会保留数据和行类型，而丢弃其他信息。
同步作业会尽最大努力按如下方式处理该问题：
1. 通常，debezium-json 包含 'schema' 字段，Paimon 会从中获取数据类型。请确保你的 debezium
json 含有该字段，否则 Paimon 将使用 'STRING' 类型。
2. 如果缺少字段类型，Paimon 将默认使用 'STRING' 类型。
3. 如果缺少数据库名或表名，你将无法执行数据库同步，但仍然可以执行表同步。
4. 如果缺少主键，作业可能会创建非主键表。你可以在提交表同步作业时设置主键。

:::

## 同步表 {#synchronizing-tables}

通过在 Flink DataStream 作业中使用 [KafkaSyncTableAction](https://paimon.apache.org/docs/master/api/java/org/apache/paimon/flink/action/cdc/kafka/KafkaSyncTableAction)，或者直接通过 `flink run`，用户可以将 Kafka 一个 topic 中的一张或多张表同步到一张 Paimon 表中。

要通过 `flink run` 使用该功能，请运行以下 shell 命令。

```bash
<FLINK_HOME>/bin/flink run \
    /path/to/paimon-flink-action-@@VERSION@@.jar \
    kafka_sync_table \
    --warehouse <warehouse-path> \
    --database <database-name> \
    --table <table-name> \
    [--partition_keys <partition_keys>] \
    [--primary_keys <primary-keys>] \
    [--type_mapping to-string] \
    [--computed_column <'column-name=expr-name(args[, ...])'> [--computed_column ...]] \
    [--metadata_column <metadata-column> [--metadata_column ...]] \
    [--metadata_column_prefix <metadata-column-prefix>] \
    [--kafka_conf <kafka-source-conf> [--kafka_conf <kafka-source-conf> ...]] \
    [--catalog_conf <paimon-catalog-conf> [--catalog_conf <paimon-catalog-conf> ...]] \
    [--table_conf <paimon-table-sink-conf> [--table_conf <paimon-table-sink-conf> ...]]
```

<ConfigTable html={kafkaSyncTableHtml} />

如果你指定的 Paimon 表不存在，该 Action 会自动创建该表。其 schema 将从所有指定的 Kafka topic 的表中推导得出，它会从 topic 中获取最早的非 DDL 数据来解析 schema。如果 Paimon 表已存在，其 schema 将与所有指定的 Kafka topic 的表的 schema 进行比对。

示例 1：

```bash
<FLINK_HOME>/bin/flink run \
    /path/to/paimon-flink-action-@@VERSION@@.jar \
    kafka_sync_table \
    --warehouse hdfs:///path/to/warehouse \
    --database test_db \
    --table test_table \
    --partition_keys pt \
    --primary_keys pt,uid \
    --computed_column '_year=year(age)' \
    --kafka_conf properties.bootstrap.servers=127.0.0.1:9020 \
    --kafka_conf topic=order \
    --kafka_conf properties.group.id=123456 \
    --kafka_conf value.format=canal-json \
    --catalog_conf metastore=hive \
    --catalog_conf uri=thrift://hive-metastore:9083 \
    --table_conf bucket=4 \
    --table_conf changelog-producer=input \
    --table_conf sink.parallelism=4
```

如果在你启动同步作业时 Kafka topic 中尚不包含消息，你必须在提交作业之前手动创建该表。
你可以只定义分区键和主键，其余的列将由同步作业自动添加。

注意：在这种情况下你不应使用 --partition_keys 或 --primary_keys，因为这些键在创建表时已经定义，
之后无法修改。此外，如果你指定了计算列，还应定义计算列所使用的全部参数列。

示例 2：
如果你想同步一张主键为 'id INT' 的表，并且想计算一个分区键 'part=date_format(create_time,yyyy-MM-dd)'，
你可以先创建这样一张表（其他列可以省略）：

```sql
CREATE TABLE test_db.test_table (
    id INT,                 -- primary key
    create_time TIMESTAMP,  -- the argument of computed column part
    part STRING,            -- partition key
    PRIMARY KEY (id, part) NOT ENFORCED
) PARTITIONED BY (part);
```

然后你就可以提交同步作业：

```bash
<FLINK_HOME>/bin/flink run \
    /path/to/paimon-flink-action-@@VERSION@@.jar \
    kafka_sync_table \
    --warehouse hdfs:///path/to/warehouse \
    --database test_db \
    --table test_table \
    --computed_column 'part=date_format(create_time,yyyy-MM-dd)' \
    ... (other conf)
```

示例 3：
对于某些追加型数据（例如日志数据），可以将其视为只含 INSERT 操作类型的特殊 CDC 数据，因此你可以使用 'format=json' 把这类数据同步到 Paimon 表中。

```bash
<FLINK_HOME>/bin/flink run \
    /path/to/paimon-flink-action-@@VERSION@@.jar \
    kafka_sync_table \
    --warehouse hdfs:///path/to/warehouse \
    --database test_db \
    --table test_table \
    --partition_keys pt \
    --computed_column 'pt=date_format(event_tm, yyyyMMdd)' \
    --kafka_conf properties.bootstrap.servers=127.0.0.1:9020 \
    --kafka_conf topic=test_log \
    --kafka_conf properties.group.id=123456 \
    --kafka_conf value.format=json \
    --catalog_conf metastore=hive \
    --catalog_conf uri=thrift://hive-metastore:9083 \
    --table_conf sink.parallelism=4
```

## 同步数据库 {#synchronizing-databases}

通过在 Flink DataStream 作业中使用 [KafkaSyncDatabaseAction](https://paimon.apache.org/docs/master/api/java/org/apache/paimon/flink/action/cdc/kafka/KafkaSyncDatabaseAction)，或者直接通过 `flink run`，用户可以将多个 topic 或一个 topic 同步到一个 Paimon 数据库中。

要通过 `flink run` 使用该功能，请运行以下 shell 命令。

```bash
<FLINK_HOME>/bin/flink run \
    /path/to/paimon-flink-action-@@VERSION@@.jar \
    kafka_sync_database \
    --warehouse <warehouse-path> \
    --database <database-name> \
    [--table_mapping <table-name>=<paimon-table-name1> [--table_mapping <table-name2>=<paimon-table-name2> ...]] \
    [--table_prefix <paimon-table-prefix>] \
    [--table_suffix <paimon-table-suffix>] \
    [--table_prefix_db <db-name1>=<table-prefix1> [--table_prefix_db <db-name2>=<table-prefix2> ...]] \
    [--table_suffix_db <db-name1>=<table-suffix1> [--table_suffix_db <db-name2>=<table-suffix2> ...]] \
    [--including_tables <table-name|name-regular-expr>] \
    [--excluding_tables <table-name|name-regular-expr>] \
    [--including_dbs <database-name|name-regular-expr>] \
    [--excluding_dbs <database-name|name-regular-expr>] \
    [--type_mapping to-string] \
    [--partition_keys <partition_keys>] \
    [--primary_keys <primary-keys>] \
    [--computed_column <'column-name=expr-name(args[, ...])'> [--computed_column ...]] \
    [--metadata_column <metadata-column> [--metadata_column ...]] \
    [--metadata_column_prefix <metadata-column-prefix>] \
    [--kafka_conf <kafka-source-conf> [--kafka_conf <kafka-source-conf> ...]] \
    [--catalog_conf <paimon-catalog-conf> [--catalog_conf <paimon-catalog-conf> ...]] \
    [--table_conf <paimon-table-sink-conf> [--table_conf <paimon-table-sink-conf> ...]]
```

<ConfigTable html={kafkaSyncDatabaseHtml} />

该 Action 会为所有表构建单个合并的 Sink。对于每张要同步的 Kafka topic 的表，如果对应的 Paimon
表不存在，该 Action 会自动创建该表，其 schema 将从所有指定的 Kafka topic 的表中推导得出。如果 Paimon 表已存在，且其 schema 与从
Kafka 记录中解析出的 schema 不同，该 Action 会尝试执行 schema 演进。

示例

从一个 Kafka topic 同步到 Paimon 数据库。

```bash
<FLINK_HOME>/bin/flink run \
    /path/to/paimon-flink-action-@@VERSION@@.jar \
    kafka_sync_database \
    --warehouse hdfs:///path/to/warehouse \
    --database test_db \
    --kafka_conf properties.bootstrap.servers=127.0.0.1:9020 \
    --kafka_conf topic=order \
    --kafka_conf properties.group.id=123456 \
    --kafka_conf value.format=canal-json \
    --catalog_conf metastore=hive \
    --catalog_conf uri=thrift://hive-metastore:9083 \
    --table_conf bucket=4 \
    --table_conf changelog-producer=input \
    --table_conf sink.parallelism=4 \
    --computed_column 'pt=date_format(event_tm, yyyyMMdd)'
```

从多个 Kafka topic 同步到 Paimon 数据库。

```bash
<FLINK_HOME>/bin/flink run \
    /path/to/paimon-flink-action-@@VERSION@@.jar \
    kafka_sync_database \
    --warehouse hdfs:///path/to/warehouse \
    --database test_db \
    --kafka_conf properties.bootstrap.servers=127.0.0.1:9020 \
    --kafka_conf topic=order\;logistic_order\;user \
    --kafka_conf properties.group.id=123456 \
    --kafka_conf value.format=canal-json \
    --catalog_conf metastore=hive \
    --catalog_conf uri=thrift://hive-metastore:9083 \
    --table_conf bucket=4 \
    --table_conf changelog-producer=input \
    --table_conf sink.parallelism=4
```

## Debezium-bson {#debezium-bson}

debezium-bson 格式是 [Kafka CDC](./kafka-cdc) 所支持的格式之一。
它是通过 debezium 采集 mongodb 得到的格式，与
<a href="https://nightlies.apache.org/flink/flink-docs-stable/docs/connectors/table/formats/debezium/">debezium-json</a> 格式类似。
但是，MongoDB 没有固定的 schema，每个文档的字段类型可能各不相同，因此 JSON 中的 before/after 字段
都是 string 类型，而 debezium-json 格式要求是 JSON object 类型。

MongoDB BSON Jar 可以从 [Maven 仓库](https://mvnrepository.com/artifact/org.mongodb/bson) 下载

```
bson-*.jar
```

:::info

debezium bson 格式要求 insert/update/delete 事件消息包含完整文档，并且包含一个表示变更前文档状态的字段。
这需要将 debezium 的 capture.mode 设置为 change_streams_update_full_with_pre_image，并将 [capture.mode.full.update.type](https://debezium.io/documentation/reference/stable/connectors/mongodb.html#mongodb-property-capture-mode-full-update-type) 设置为 post_image。
在 MongoDB 6.0 版本之前，无法获取 'Update Before' 信息。因此，使用 Kafka Key 中的 id 字段作为 'Update before' 信息

:::

下面是一个以 JSON 格式从 Mongodb customers 集合中捕获的更新操作的简单示例：

```json
{
  "schema": {
    "type": "struct",
    "fields": [
      {
        "type": "string",
        "optional": true,
        "name": "io.debezium.data.Json",
        "version": 1,
        "field": "before"
      },
      {
        "type": "string",
        "optional": true,
        "name": "io.debezium.data.Json",
        "version": 1,
        "field": "after"
      },
      ...
    ]
  },
  "payload": {
    "before": "{\"_id\": {\"$oid\" : \"596e275826f08b2730779e1f\"}, \"name\" : \"Anne\", \"create_time\" : {\"$numberLong\" : \"1558965506000\"}, \"tags\":[\"success\"]}",
    "after": "{\"_id\": {\"$oid\" : \"596e275826f08b2730779e1f\"}, \"name\" : \"Anne\", \"create_time\" : {\"$numberLong\" : \"1558965506000\"}, \"tags\":[\"passion\",\"success\"]}",
    "source": {
      "db": "inventory",
      "rs": "rs0",
      "collection": "customers",
      ...
    },
    "op": "u",
    "ts_ms": 1558965515240,
    "ts_us": 1558965515240142,
    "ts_ns": 1558965515240142879
  }
}
```

这个来自 MongoDB customers 集合的文档有 4 列，_id 是 BSON ObjectID，name 是 string，
create_time 是 long，tags 是一个 string 数组。以下是 debezium-bson 格式的处理结果：

文档 Schema：

| 字段名      | 字段类型    | 键          |
|------------|------------|-------------|
| _id        | STRING     | Primary Key |
| name       | STRING     |             |
| create_time| STRING     |             |
| tags       | STRING     |             |

记录：

| RowKind | _id                      | name  | create_time                | tags                  |
|---------|--------------------------|-------|----------------------------|-----------------------|
|   -U    | 596e275826f08b2730779e1f | Anne  | 1558965506000              | ["success"]           |
|   +U    | 596e275826f08b2730779e1f | Anne  | 1558965506000              | ["passion","success"] |

由于事件消息的 schema 字段中没有文档的字段信息，因此 debezium-bson 格式不要求事件消息携带 schema 信息。具体操作如下：

- 将事件消息的 before/after 字段解析为 BSONDocument。
- 递归遍历 BSONDocument 的所有字段，并将 BsonValue 转换为 Java Object。
- before/after 的所有顶层字段都被转换为 string 类型，且 _id 固定为主键
- 如果 before/after 的顶层字段是基本类型（例如 Integer/Long 等），则直接转换为 string；如果不是，则转换为 JSON 字符串
