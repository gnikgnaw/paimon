## 0. 30 秒速通(TL;DR)

如果你只有 30 秒,记住这 8 条,够你撑过 80% 的会:

1. **结论先行**:先说结果/结论/我要你做什么,再说为什么。别讲背景讲三分钟还没到重点。
2. **用对方的语言**:对算法讲 AUC/特征/样本,对老板讲钱/风险/进度。同一件事,两套词。
3. **量化一切**:"快了很多" → "P99 从 800ms 降到 120ms";"省了不少" → "GPU 卡时省 30%,月省约 X 万"。
4. **给方案,不抛问题**:不要说"这个搞不定";要说"有 A/B 两个方案,我建议 A,因为……"。
5. **接质疑不慌**:被怼时先复述对方的点("你担心的是 X 对吧"),再回应。**先接住,再回应。**
6. **不懂别装懂**:"这块我还没拉到数据,会后 1 小时给你结论"——比胡说强一百倍。
7. **翻译技术债**:"离在线不一致" → "这次迭代的效果数据不可信,等于白干"。
8. **会前 5 分钟磨枪**:把本次会要用到的 3-5 个名词、2-3 个数字、1 个结论写在便签上。

> 🎓 **老师视角**:平台工程师最大的坑不是技术不行,是"**心里有数,嘴上没数**"。你脑子里清楚得很,一开口就被算法和老板带跑。话术不是忽悠,是把你脑子里的清晰,无损地传到对方脑子里。

## 1. 表达底层原则:先讲人话,再讲技术

### 1.1 结论先行(金字塔原理)

人脑是"栈"结构,听到一半就开始下结论。你把背景铺垫放前面,对方早就走神了。

```
❌ 新手讲法(倒金字塔,埋雷在最后)
  "我们这个特征平台呢,之前是 v1 架构,用的是 HBase 存特征,
   后来发现 HBase 在大 key 场景下有热点,然后我们调研了 Redis、
   Aerospike、还有自研 KV,做了一堆压测,最后……我们想换存储,
   需要两个人月。"
  → 老板:你前面讲的我一句没记住,你到底要啥?

✅ 高手讲法(正金字塔,结论砸脸上)
  结论:"我申请 2 人月,把特征存储从 HBase 换成 Aerospike。"
  原因:"因为 HBase 热点导致大促时特征读取 P99 飙到 2s,拖垮线上。"
  收益:"换完 P99 可控在 200ms 内,大促不再有特征超时降级。"
  代价:"2 人月,涉及一次灰度迁移,有回滚预案。"
  → 老板:批了/我有问题问 X。
```

**模板:结论 → 原因 → 收益 → 代价 → 风险**(SCQA / BLUF 的中文落地版)。

> **句式骨架(背下来,任何场合套用)**:
> - "结论是 X。" / "我的建议是 X。" / "我需要你 X。"
> - "原因有两点:一是……,二是……。"(原因压到 2-3 个,多了记不住)
> - "做完之后,Y 会从 ___ 变成 ___。"(收益量化成"从→到")
> - "代价是 N(人月/卡/钱),风险是 Z,我的兜底是 W。"

**为什么"结论先行"对平台工程师尤其重要**:你面对的是一群**比你忙、比你急、注意力比你短**的人(算法在赶实验、老板在赶决策)。他们的耐心窗口只有前 15 秒。前 15 秒没听到"你要干嘛 / 这事的结果是啥",后面讲得再好也是对牛弹琴。**把最重要的话放在你还没被打断之前说完。**

### 1.2 用对方的语言

| 同一件事 | 对算法同学说 | 对训练同学说 | 对老板说 |
|---|---|---|---|
| 特征平台升级 | 特征覆盖率从 60% 提到 95%,新增序列特征支持 | 特征拼接吞吐提升 3x,样本生产不再卡 | 算法迭代提速,事故率下降 |
| 离在线一致性治理 | 训练/推理特征对齐,AUC 不再虚高 | 样本里特征值和线上一致,不用反复对数 | 减少"白干迭代",每月省 ~N 次返工 |
| 引入数据湖(Paimon) | 样本可回填、可时间旅行、Schema 演进 | 流批一体,样本表统一,读取更快 | 存储成本降 X%,数据可追溯,合规更稳 |

> 🎓 **老师视角**:同一个技术,你要准备**三套话术**。这不是虚伪,这是专业。医生跟同行讲"心肌梗死",跟病人讲"心脏血管堵了"。你跟算法讲技术,跟老板讲价值,是同一个道理。

### 1.3 量化:把形容词换成数字

```
形容词是廉价的,数字是值钱的。

❌ "性能提升明显"      ✅ "训练吞吐 12w → 38w samples/s(↑3.2x)"
❌ "比较稳定"          ✅ "近 30 天 0 个 P1,可用性 99.95%"
❌ "省了不少资源"      ✅ "GPU 卡时 ↓30%,按当前单价月省约 X 万(经验值,需财务核)"
❌ "上线快了很多"      ✅ "特征上线周期 3 天 → 4 小时"
```

> 📌 **数字纪律**:编不出来的数字,标"**经验值/待核**"。`"约 30%(经验值)"` 永远比拍脑袋的 `"50%"` 安全。被追问时你能圆,胡编的数字一被戳穿,信任全没。

**怎么"造"出可信的数字(不是编,是量化方法)**:

| 你想表达的 | 怎么变成数字 |
|---|---|
| "训练变快了" | 对比改造前后的 `samples/s` 或单 epoch 墙钟时间 |
| "省了资源" | GPU 卡时(卡数 × 小时)前后差 × 单价(标"按当前价,待财务核") |
| "事故变少" | 近 N 天 P1/P2 数量对比,或一致性 gap 监控的告警次数 |
| "覆盖率提升" | (有值样本数 / 总样本数),按 slot 统计 |
| "上线变快" | 特征从提交到生效的端到端耗时,取中位数 |
| "稳定性" | 可用性 = 1 - 故障时长/总时长,或近 N 天 P1 数 |

> 🎓 **老师视角**:很多平台工程师"明明做了很牛的事,却讲不出价值",根子在**平时不埋点、不记账**。你要养成习惯:**每做一件事,顺手记下它的"前后对比"**——改造前吞吐多少、改造后多少。等到汇报/评审/晋升答辩时,这些数字就是你的弹药库。没数字的工作 = 没发生过的工作。

### 1.4 给方案,不抛问题

```
抛问题的人是负担,给方案的人是资产。

❌ "这个特征实时性做不到,延迟太高。"
   → 留给对方一个烂摊子,显得你无能。

✅ "实时特征延迟目前 5min。要降到秒级有两条路:
    A. 上 Flink 实时链路,1 人月,延迟到 10s,成本 +X;
    B. 维持准实时 + 关键特征单独走快链路,3 天,覆盖 80% 场景。
    我建议先做 B 兜住业务,A 排到下个季度。你看?"
   → 你成了拿主意的人。
```

> 给方案的标准结构:**A/B(/C)选项 + 各自代价 + 我的推荐 + 推荐理由 + 把决定权交回去**。注意——**一定要给"我的推荐"**,只列选项不给倾向叫"踢皮球",对方会觉得你没担当。

### 1.5 先听懂,再开口(被低估的沟通基本功)

很多冲突不是"说得不好",是"**根本没听懂对方要什么就开始回应**"。

```
┌─────────────────────────────────────────────────┐
│  听懂的三个动作                                    │
├─────────────────────────────────────────────────┤
│ ① 复述确认:"你的意思是 X,对吗?"                 │
│ ② 挖真实诉求:"你最终是想达成什么?"(问 why)    │
│ ③ 区分"要的"和"需要的":他要 A,可能真需求是 B   │
└─────────────────────────────────────────────────┘
```

**真实对话脚本(算法的"伪需求")**:

> **算法**:"给我开个 root 权限,我要自己上服务器调样本。"
> **你(❌)**:"不行,权限管控不能开。" → 直接对撞,且没解决他的问题
> **你(✅)**:"你想自己调样本,最终是想干嘛?(挖 why)——哦你是想快速试不同的特征组合对吧?那其实**不用 root,我给你一个自助的样本配置入口,你自己改配置就能试,还更安全更快**。要的是'快速试',不是'root 权限'。"

> 🎓 **老师视角**:用户(算法)提的常常是"**他以为的解决方案**",不是"他真正的问题"。他说"要 root",真需求是"想快速试特征"。**你的本事是穿透'要什么',看到'为什么要',给出更好的解法。** 这就是产品思维,也是平台工程师从"工具人"升级为"被信赖的伙伴"的关键。

## 2. 评审现场话术:如何讲方案、如何接质疑

### 2.1 讲方案的黄金结构:背景-问题-方案-权衡-收益-风险

```
┌──────────────────────────────────────────────┐
│  方案评审 6 段式(每段不超过 2 句话开场)        │
├──────────────────────────────────────────────┤
│ ① 背景:现在长什么样(1 句,给坐标)            │
│ ② 问题:痛点 + 量化影响(数字!)              │
│ ③ 方案:我要做什么(动词开头,具体)           │
│ ④ 权衡:为什么是这个,不是别的(对比表)       │
│ ⑤ 收益:做完得到什么(可度量)                │
│ ⑥ 风险:可能翻车点 + 兜底(主动说,别等问)     │
└──────────────────────────────────────────────┘
评审现场:先用 2 分钟把这 6 段讲完,再展开细节。
```

**真实对话脚本(特征平台 v2 评审)**:

> **你**:"今天评审特征平台 v2。
> ① 背景:现在特征注册靠手工配 YAML,算法上一个特征平均 3 天。
> ② 问题:近一个季度因为配置错误导致的离在线不一致事故有 4 起,每起平均废掉 1 个迭代,折算算法人力约 N 人日。
> ③ 方案:做一套特征 DSL + 自动校验 + 离在线同源生成,算法自助上线。
> ④ 权衡:可以买商业方案,但定制化差、贵;自研 1.5 个月,长期可控,选自研。
> ⑤ 收益:特征上线 3 天→4 小时,一致性事故预期降到接近 0。
> ⑥ 风险:DSL 学习成本,有平滑期;迁移期双跑校验,可回滚。
> 细节我翻到第 3 页。"

> 🎓 **老师视角**:**"风险"这一段一定要你自己主动说。** 你不说,评审的人就会去找,找到了显得你考虑不周;你主动说,显得你成熟可信。把对方"准备拿来怼你的点",提前自己说出来 + 给兜底,这叫"**抢麦克风**"。

### 2.2 应对质疑:先接住,再回应(永不正面对撞)

```
质疑来了,大脑第一反应是"防御/反驳"——这是大忌。

正确流程:
  复述确认 → 区分类型 → 回应/承接
  ┌─────────────────────────────────────┐
  │ Step1 "你担心的是 X,对吧?"          │ ← 让对方觉得被听见
  │ Step2 判断:这是 (a)技术质疑          │
  │            (b)方向质疑 (c)情绪/抢功   │
  │ Step3 分类回应                        │
  └─────────────────────────────────────┘
```

| 质疑类型 | 典型台词 | 高手接法 |
|---|---|---|
| 技术质疑(有道理) | "你这一致性怎么保证?" | "好问题。我们用同源生成 + 抽样对数,误差 <0.1%,有监控。" 认可 + 给证据 |
| 技术质疑(对方错) | "直接用 Redis 不就行了?" | "Redis 我们压过,大 key 热点扛不住大促,数据我会后发你。"不打脸,用数据 |
| 方向质疑 | "为什么现在做这个?" | "因为它卡着 Q3 的算法迭代,不做这个后面都堵着。"拉回优先级 |
| 情绪/抢功 | "这不早该做了吗?" | 见 2.3 |
| 你确实不会 | "这块底层怎么实现?" | "这块我还没完全拉通,会后给你详细方案,今天先对齐结论。"不装 |

**真实对话脚本(被怼性能)**:

> **算法**:"你这个吞吐数据是不是测出来好看的?线上肯定到不了。"
> **你(❌防御)**:"怎么会,我测得很认真!" → 升级对抗
> **你(✅承接)**:"你是担心测试环境和线上不一致对吧?(复述)合理。我这个数是单机压测峰值,线上有网络和混部干扰,我标的是'经验上限'。我可以拉一次线上影子流量实测,明天给你真实数,以那个为准。" → 你赢了信任

**质疑应对的进阶:区分"真问题"和"找茬"**

```
不是所有质疑都要硬接。先判断对方的"动机":

┌──────────────────────────────────────────────────┐
│  动机判断 → 不同打法                                │
├──────────────────────────────────────────────────┤
│ a) 真想搞懂 → 耐心给证据/数据,他是来帮你压方案的   │
│ b) 担心自己的活受影响 → 先安抚"不影响你的 X"        │
│ c) 抢话语权/刷存在 → 给个台阶"你这点很重要,         │
│    咱会后单独深聊",别在大会上纠缠               │
│ d) 纯情绪 → 不接招,"咱对齐下目标,细节会后"        │
└──────────────────────────────────────────────────┘
```

**真实对话脚本(评审跑偏,有人钻牛角尖)**:

> **某人**:"你这个序列特征的存储格式,为什么不用 Protobuf 而用 X?"(细节,且跑题)
> **你(❌)**:被带进去,花 5 分钟解释格式选型,评审时间耗光,核心方案没过。
> **你(✅)**:"格式选型是个好点,但它不影响今天要拍的方案主干。**我会后单独拉你对一下选型**,咱今天先把'要不要做、排期、资源'对齐?" → 把会拉回正轨,还显得你尊重对方。

> 🎓 **老师视角**:评审会最大的杀手不是"被怼倒",是"**被带偏到细节里,核心议题没时间拍**"。你要像主持人一样**控场**:对偏题的质疑,认可它的价值 + 约会后单聊 + 拉回主线。**会议时间是稀缺资源,你要替它把关。**

### 2.3 接"这个为什么不早做?"(最容易破防的一句)

这句话杀伤力大,因为它**像质疑你的能力/失职**。但 90% 时候它不是真的怪你,是对方想知道"现在做的合理性"。

```
❌ 防御:"之前没排期 / 之前没人提"  → 显得甩锅、被动
❌ 认怂:"是我们疏忽了"            → 没必要背锅,且不解决问题
✅ 转正:把"为什么不早做"翻译成"为什么现在正合适"
```

> **标准答法**:"之前优先级在 X(更紧急的事),这个问题在数据上是最近才暴露/放大的——上个月一致性事故从 1 起涨到 4 起,说明它到了非做不可的点。所以现在做,正是时候。"

> 🎓 **老师视角**:所有"为什么不早做"的潜台词都是"**现在做对不对**"。你别钻进"过去"里辩解,直接把话题拉到"现在/未来"的合理性。**永远朝前看,别朝后辩。**

## 3. 向上汇报老板:把技术翻译成 ROI/风险/成本/进度

老板的脑子里只有四个槽:**赚钱(ROI)、别出事(风险)、花多少(成本)、啥时候好(进度)**。你讲的任何技术,不落到这四个槽,他听不进去。

### 3.1 技术语言 → 业务语言 翻译对照表(核心资产,背下来)

| 你想说的技术事实 | 翻译成老板能听的话 |
|---|---|
| 离在线特征不一致 | "这次迭代的 AB 数据不可信,等于**白干一个迭代**,~2 周人力打水漂" |
| 没有特征平台,手工配特征 | "算法每上一个特征要 3 天且容易配错,**慢且容易出事故**" |
| 特征平台 / 样本中台 | "**提效**(算法迭代快 N 倍)+ **防事故**(一致性自动校验)" |
| 样本可回填 / 时间旅行 | "出问题能**追溯和补救**,不用从头重跑,**省抢修时间**" |
| 训练吞吐提升 3x | "同样的卡,模型迭代快 3 倍,或者**省 2/3 的 GPU 钱**" |
| GPU 利用率低 / OOM | "**卡在烧钱却没干活**,优化后月省约 X 万(经验值)" |
| 流批一体 / 数据湖 | "一套数据两用,**少维护一套链路,少一个人力**,少一个出错点" |
| 技术债 / 重构 | "现在不还,后面每个需求都**慢 30% 且越来越容易塌**" |
| Schema 演进 | "加特征不用停服、不用重刷历史,**改动零事故**" |
| P99 延迟优化 | "大促高峰**不再超时降级**,直接保住转化/GMV" |
| 监控/告警体系 | "从'出事了用户先发现'变成'我们先发现',**事故时长砍半**" |
| 灰度发布能力 | "新功能**小范围先试**,炸了也只炸 1%,不会全站翻车" |
| 限流/降级预案 | "流量洪峰来了**保住核心不雪崩**,大促能睡安稳觉" |
| 特征自助化 | "算法**不用排队等平台**,自己上特征,平台从瓶颈变高速路" |
| 资源混部/利用率优化 | "同样的机器**多干一倍的活**,等于白捡硬件预算" |

> 🎓 **老师视角**:翻译的精髓是**"损益归因"**——把技术状态映射成"赚了/亏了/省了/险了"。老板不关心"一致性",关心"会不会白干、会不会出事、要花多少钱"。你每说一个技术词,心里默问一句:**"这事的钱和风险在哪?"** 答出来,再开口。

**反向陷阱:别"翻译过度"变成吹牛**

```
翻译 ≠ 夸大。把"可能省 10%"说成"省一半",短期唬住老板,
长期一旦交付不出,你的所有数字都不再被信。

✅ 稳健翻译:"预计省 ~20%(经验值,上线后用实际账单校准)"
❌ 吹牛翻译:"直接省一半成本,立竿见影"

老板被吹过一次牛之后,会给你所有汇报打七折——这是信用的复利损耗。
```

### 3.2 向上汇报结构:钱-险-本-进(BLUF + 4 槽)

```
┌────────────────────────────────────────────┐
│  向上 1 分钟汇报骨架                          │
├────────────────────────────────────────────┤
│ 一句话结论:"项目按期,有一个风险需要你拍板"   │
│ ① ROI:做完带来 X(收入/效率/成本)           │
│ ② 风险:最大的不确定是 Y,我的预案是 Z         │
│ ③ 成本:要 N 人月 / N 张卡 / N 万             │
│ ④ 进度:M1 完成,M2 在做,预计 X 号交付        │
│ 需要你:批资源 / 拍优先级 / 协调 X 团队        │
└────────────────────────────────────────────┘
```

**真实对话脚本(申请资源)**:

> **你**:"老板,一句话:特征平台一期已上线,提效达标;现在想申请 2 人月做二期。
> ROI:二期把算法上特征从 4 小时压到分钟级,**算法团队迭代再快一档**;
> 风险:不做的话,现有手工链路在大促前是个隐患,去年大促这块出过 P1;
> 成本:2 人月,不需要额外采购;
> 进度:6 周交付,大促前一个月完成灰度。
> 需要你:确认这个优先级排在 X 项目前面。"

> 🎓 **老师视角**:向上汇报永远以"**需要你做什么**"收尾。老板的会议时间是用来"做决定"的,不是听你播报的。你不给他决策点,他会觉得这个会白开,你这个人"没产出"。

### 3.3 汇报"坏消息"的话术(故障/延期/翻车)

```
报喜不报忧 = 慢性自杀。坏消息要"快、全、带方案"地报。

公式:已发生什么 + 影响多大 + 已经做了什么 + 还需要什么 + 多久恢复
```

> **真实脚本(线上特征故障)**:
> "同步个 P1:今天 10:20 特征服务大 key 热点导致推荐侧特征超时,
> **影响**:核心推荐位降级约 20 分钟,GMV 影响在评估,初步 ~X(待核);
> **已做**:已切备用链路,服务 10:45 恢复正常;
> **根因**:某热门商品 key 流量突增,缓存击穿,正在加多级缓存 + 限流;
> **需要**:申请把缓存治理插到本周最高优;
> **复盘**:明天 11 点拉复盘会,文档我先起好。"

> 🎓 **老师视角**:坏消息里,老板最怕的不是"出事了",是"**你瞒着我**"和"**你没头绪**"。第一时间报 + 已经在动手,反而是加分项。记住顺序:**先止血(已做),再认怎么修(方案),最后才提需求。**

## 4. 与算法同学对接:接需求、建信任、push back

算法同学是你的"客户",但又不是甲方——他们的诉求往往**模糊、急、且不一定合理**。你的本事是:**接得专业、push 得礼貌。**

### 4.1 算法常见诉求的接法

| 算法诉求 | 潜台词 | 高手接法 |
|---|---|---|
| "帮我加几个 slot" | 想加特征做实验 | "可以。你要加的是离散还是连续?预估多少维?我看下当前 slot 容量和样本膨胀,**给你个不影响吞吐的方案**。" |
| "我要加序列特征" | 用户行为序列建模 | "序列长度截多少?是定长还是变长?这块影响样本大小和训练吞吐,**咱先对齐 maxlen,我评估存储和读取成本**。" |
| "要实时样本" | 想缩短反馈环 | "实时到什么程度?分钟级还是秒级?实时样本涉及流式 label 回流和一致性,**我建议先准实时兜住,真要秒级咱单开评估**。" |
| "帮我回填三个月数据" | 新特征要历史样本 | "可以回填。三个月数据量大概 X,跑批要 Y 小时、占 Z 资源。**我排个低峰跑,明天给你**。要不要先回填一周验证逻辑?" |
| "我这个特征怎么在线上对不上?" | 怀疑离在线不一致 | "咱一起对一下:你训练取的是哪个表哪个版本?线上读的是哪个 KV?**我拉个抽样对数脚本,定位是取数口径还是时效问题**。" |

> 🎓 **老师视角**:接算法需求的第一动作,**永远是"问清楚口径",不是"答能不能"**。算法说"加序列特征",你要立刻反问 maxlen、变长定长、回填范围——**问对了问题,你的专业度立刻拉满**,对方还没说完就觉得"这平台的人懂行"。

### 4.2 礼貌但坚定地 push back(拒绝不合理需求)

```
不能直接说"不行"(显得不配合),也不能照单全收(累死且出事)。
公式:认可诉求 + 摆出代价 + 给替代方案 + 把选择权交回去

结构:"这个需求我理解(认可)。但它会带来 X 代价(事实)。
       我建议 Y(替代)。当然你坚持的话,需要 Z 资源/排期(交回去)。"
```

**真实对话脚本(算法要无限加 slot)**:


> **算法**:"我想再加 50 个 slot 做实验。"
> **你**:"理解,你想多试些特征(认可)。不过现在样本已经 X TB,再加 50 个 slot 样本会膨胀 ~30%,训练吞吐会掉,大家都会变慢(代价)。我建议:**先上 10 个你最看好的,跑出来有正向再扩**;或者用特征重要性先筛一轮(替代)。你要确定 50 个都要,那我得给样本链路扩资源,排期得往后挪一周(交回去)。你看怎么弄?"

> 🎓 **老师视角**:push back 的关键是**"把代价讲成共同利益"**——"样本膨胀大家都变慢",而不是"我不想做"。让对方意识到这是**他自己和团队的代价**,他自己就会收敛需求。你不是守门员,你是"帮他做更优决策的人"。

### 4.3 用专业建立信任的小动作

- 接需求时报口径,不报情绪("我评估下"比"应该可以吧"专业一万倍)。
- 给 deadline,且留 buffer,然后**提前交付**(承诺 3 天,2 天给,信任暴涨)。
- 主动暴露数据:"这个我压测过,数据在这"——算法对**有数据的人**天然信服。
- 帮算法挡掉一次事故(比如校验拦下一个错配置),信任直接拉满。

### 4.4 三个高频对接场景的"正例 vs 反例"对照

**场景一:算法说"你这个特征上线怎么这么慢"**

> ❌ 反例:"流程就这样,得审核""不是我能决定的" → 甩锅、推流程,信任 -1
> ✅ 正例:"我看下卡在哪一步。如果是校验环节,我帮你看报错;如果是排期,我把你这个挪到今天下午跑。**端到端我盯着,2 小时内给你结论。**"

**场景二:算法怀疑"是不是平台把我的特征值算错了"**

> ❌ 反例:"不可能,我们逻辑没问题。" → 把人推到对立面,就算你对也输了关系
> ✅ 正例:"有可能,咱一起查,别冤枉也别放过。我拉同一条样本,把你算的和平台算的逐字段摆出来。**如果是平台的 bug 我马上修,如果是口径差异我给你说明。**"

**场景三:算法临时插需求"这个今天必须上"**

> ❌ 反例:"那不行,我手上一堆事。" → 显得不配合;或硬接,把别的事搞砸
> ✅ 正例:"理解你急。我手上现在有 X(在跑的重要事)。**你这个插进来,要么 X 顺延,要么我加班今晚给你——但加班赶的活儿质量我没法打包票。** 你判断下,X 和你这个哪个更不能等?"

> 🎓 **老师视角**:对接的最高境界是**"让对方感觉你和他在同一条船上"**,而不是"你是拦着他的门卫"。"咱一起查""我盯着""别冤枉也别放过"——这些话把"你 vs 我"变成"我们 vs 问题"。**关系一旦变成对立,技术上谁对都没用了。**

## 5. 与训练同学对接:吞吐/OOM/样本读取/格式

训练同学的世界:**卡、显存、吞吐、数据读得动读不动**。跟他们对话,要钻到这些细节里。

### 5.1 高频问题对话脚本

```
┌─ 场景:训练同学说"你这样本读太慢,卡 GPU" ────────┐
│ ❌ "我这边没问题啊"  → 甩锅,对方更火                │
│ ✅ 钻进去定位:                                       │
│   "咱定位下瓶颈在哪:                                 │
│    1) GPU 利用率多少?低于 80% 大概率是数据 IO 卡;   │
│    2) 你 dataloader 几个 worker?prefetch 开没?      │
│    3) 样本是行存还是列存?压缩格式?                  │
│    4) 我看下存储侧读带宽打满没。                      │
│    我先拉一下读侧 metrics,你看下 GPU 利用率曲线。"   │
└──────────────────────────────────────────────────────┘
```

| 训练同学报的问题 | 你要立刻问/查的 | 高手回应 |
|---|---|---|
| "OOM 了" | batch size?序列 maxlen?是显存还是内存 OOM? | "先看是 CUDA OOM 还是 host OOM。显存的话调 batch / 梯度累积;host 的话大概率样本一次读太多,我调读取的 buffer。" |
| "训练吞吐上不去" | GPU 利用率?数据 pipeline 还是计算瓶颈? | "GPU 打不满就是喂不饱,问题在数据侧。我看下列裁剪和 prefetch,把读侧吞吐提上来。" |
| "样本格式读不出来" | Schema 变了?版本对不上?压缩编码? | "你读的是哪个版本的 schema?最近表演进过,可能字段对不上,我给你最新 schema 和读取示例。" |
| "想要列裁剪只读几个特征" | 当前是不是整行读? | "对,可以做列裁剪,只读你要的 slot,IO 能省一大截。我给你配置项。" |
| "样本想 shuffle" | 全局还是局部 shuffle?数据量? | "全局 shuffle 成本高,大数据量我建议分片 + 块内 shuffle,效果接近,成本低很多。" |

> 🎓 **老师视角**:训练同学说"你的样本慢",**九成是"GPU 喂不饱(数据 IO 瓶颈)",而不是计算瓶颈**。你第一句话就该是"GPU 利用率多少?"——利用率低就是 IO 问题(你的活),利用率高那是模型/计算问题(他的活)。**一句话就把锅分清楚,还显得你懂训练。**

### 5.2 跨团队"分锅"的得体话术

```
出问题时别争"是谁的锅",要争"怎么最快定位"。

✅ "咱不纠结谁的问题,先把链路拆开看:
    读侧 metrics 我出,GPU 侧曲线你出,
    10 分钟后对一下,瓶颈在哪打哪。"
```

## 5.5 会议分场景作战手册(进会前对号入座)

不同的会,目标和打法完全不同。进会前先问自己:**这是哪种会?我要的结果是什么?**

```
┌─────────────┬──────────────┬─────────────────────────┐
│  会议类型    │  你的核心目标 │  开场第一句模板          │
├─────────────┼──────────────┼─────────────────────────┤
│ 方案评审会   │ 让方案被通过  │ "今天评审 X,我建议 A 方案" │
│ 需求对齐会   │ 问清口径排优先 │ "咱先对齐目标和优先级"     │
│ 故障复盘会   │ 沉淀防御措施  │ "目标是防止复发,不追责"    │
│ 进度同步会   │ 暴露风险求决策 │ "整体按期,一个点需你拍板"  │
│ 资源申请会   │ 拿到人/卡/钱  │ "申请 N,不做的代价是 X"   │
│ 技术分享会   │ 立专业人设    │ "解决了一个 X 类问题"      │
└─────────────┴──────────────┴─────────────────────────┘
```

### 5.5.1 需求对齐会:别急着接,先收敛

**真实对话脚本(算法提了一堆需求)**:

> **算法**:"我们想要实时样本、加 30 个 slot、序列特征、还要回填一年……"
> **你(❌)**:"好的好的我记一下,我尽量。" → 全盘接下,回去傻眼,样样做不好。
> **你(✅)**:"信息量挺大,咱排个序。这几个里,**哪个是卡着你这个季度目标的?** 我建议:实时样本和序列特征收益大但工作量也大,先各做一个 MVP 验证;30 个 slot 先上你最看好的 10 个;回填一年咱先回填一个月验逻辑。**这样两周内你就能开始跑,而不是等一个月啥都没有。** 你看这个优先级行不行?"

> 🎓 **老师视角**:面对"需求轰炸",新手的反应是"全记下、全答应",高手的反应是"**逼对方排序**"。一句"哪个是卡着你目标的"就能把模糊的一堆需求收敛成一个清晰的主线。**你不是来记需求的,是来帮他把需求变成可执行计划的。**

### 5.5.2 进度同步会:报喜更要报忧

**真实对话脚本(项目有延期风险)**:

> **你(❌)**:"进展挺顺利的,大部分都做完了。" → 真延期了就成了"突然爆雷"。
> **你(✅)**:"整体进度 70%,核心链路按期。**但有一个风险提前同步:** 离在线对数模块依赖的上游数据接口还没就绪,如果下周还不到,交付要顺延 3 天。我已经在和上游对接,**预案是先用 mock 数据跑通逻辑,接口到了再切真实**。需要你帮我推一下上游优先级。"

> 🎓 **老师视角**:进度会的价值不在"报喜",在"**提前暴露风险 + 求资源**"。把风险藏到最后一刻爆出来,是职业生涯最伤信任的事之一。**风险越早暴露,你越被信任;越晚爆雷,你越像在隐瞒。**

### 5.5.3 资源申请会:讲"不做的代价",不只讲"做的好处"

```
人对"损失"的敏感度,是对"收益"的 2-3 倍(损失厌恶)。
所以申请资源时:

❌ "做了这个能提效 30%"          → 老板:提效嘛,排排再说
✅ "不做的话,大促前这是个 P1 隐患, ← 损失!老板:那赶紧做
   去年这块出过事,影响 GMV X"
```

**真实对话脚本(申请 GPU 配额)**:

> **你**:"申请临时加 4 张卡 2 周。不是为了好看的利用率——是**算法 Q3 的三个重点实验现在排队排到两周后,等于这个季度的模型迭代被卡死**。加 4 张卡能让三个实验并行,Q3 目标才保得住。2 周后归还,我盯着不浪费。"

## 6. 高频问答库(20+ 条)

> 形式:**[谁问] 问题 → 高手答法**。重点不是标准答案,是**答法的姿态**:结论先行、量化、给方案、不甩锅。

### 算法常问平台

**Q1 [算法] 你们平台支持序列特征吗?最长能到多少?**
> A:"支持。当前默认 maxlen 我们配到 X,再长会影响样本大小和训练吞吐。你需要多长?告诉我,我评估下存储和读取成本给你个方案。"

**Q2 [算法] 我加个特征要多久能上线?**
> A:"标准特征自助走 DSL,校验通过后小时级生效。需要回填历史的话,看数据量,三个月数据大概 X 小时。你这个要回填吗?"

**Q3 [算法] 离在线一致性你们怎么保证?**
> A:"同源生成——训练样本和线上服务用同一套特征计算逻辑,加抽样对数监控,误差控制在 0.1% 内,超阈值告警。有 dashboard,我发你。"

**Q4 [算法] 我训练 AUC 很高但线上效果差,是不是平台问题?**
> A:"很可能是特征穿越或离在线不一致。咱定位三点:① 有没有用到未来信息(label 泄漏);② 训练取数口径和线上一致吗;③ 特征时效对齐吗。我拉对数脚本一起查。"

**Q5 [算法] 能给我实时样本吗?**
> A:"看你要的实时度。秒级要上流式 label 回流 + 一致性保障,成本高;分钟级准实时现在就能给,覆盖大部分场景。建议先准实时验证收益,真需要秒级再单独立项。"

**Q6 [算法] 帮我回填半年数据呗?**
> A:"可以。半年数据量约 X TB,跑批占 Y 资源、Z 小时。我排低峰期跑,不影响线上。要不要先回填一周让你验证逻辑,确认没问题再全量?"

**Q7 [算法] 你们的特征覆盖率/缺失率怎么样?**
> A:"当前核心特征覆盖率约 95%(经验值),有缺失率监控。哪个特征你觉得有问题,我拉具体数给你看。"

**Q8 [算法] 能不能加个 GPU/卡给我跑实验?**
> A:"卡是公共资源,我帮你看下配额和当前占用。你这个实验预计跑多久、要几张?我协调下错峰排进去,或者用低优先级队列先跑着。"

**Q9 [算法] 你这特征值和我自己算的对不上!**
> A:"咱对一下口径:你算的是哪个时间窗、哪个版本?平台是按 X 口径。大概率是时间对齐或者版本差异,我拉同一条样本逐字段对,定位差在哪。"

**Q10 [算法] 平台能支持我这个新模型结构吗?**
> A:"平台管数据和训练链路,模型结构你定。你这个结构对样本格式/输入有什么特殊要求?告诉我我适配。"

### 老板常问

**Q11 [老板] 这个平台到底有什么用?**
> A:"两个字:提效 + 防事故。算法迭代从 X 天缩到 Y 小时,过去手工配置导致的一致性事故能基本消除。"

**Q12 [老板] 投这么多人力,产出是什么?**
> A:"直接产出:算法迭代提速 N 倍、GPU 成本降约 X%、一致性事故归零。间接产出:算法能跑更多实验,模型效果天花板更高。"

**Q13 [老板] 这事不做会怎样?**
> A:"短期能跑,但每次迭代都慢 30% 且越来越脆;大促前手工链路是个 P1 隐患,去年这块出过事。不做是在欠技术债,后面利滚利。"

**Q14 [老板] 要多少资源?多久?**
> A:"N 人月,不需额外采购,复用现有集群。分两个里程碑:M1 核心链路 3 周,M2 自助化 3 周,共 6 周。"

**Q15 [老板] 风险在哪?**
> A:"最大风险是迁移期数据一致性。预案是双跑校验 + 可一键回滚,灰度放量,出问题影响可控。"

**Q16 [老板] 这个跟竞品/友商比怎么样?**
> A:"友商成熟方案在 X 上更强,但贵且定制化差。我们自研的优势是贴合自己的样本和算法迭代节奏,长期可控。"

**Q17 [老板] 为什么慢/为什么延期了?**
> A:"延期 X 天,根因是 Y(具体技术原因)被低估了。已经追上的部分是 Z,剩余 W,新预计 X 号交付。需要的话我可以砍掉非核心的 V 先保主线。"

**Q18 [老板] 这次大促能扛住吗?**
> A:"压测到 X QPS,留了 1.5 倍余量,有限流降级预案。最担心的是热点 key,已加多级缓存。整体可控,我会盯全程。"

**Q19 [老板] GPU 成本怎么这么高?**
> A:"主要是利用率有提升空间。当前平均利用率约 X%,优化数据 pipeline 和混部后预计提到 Y%,等效省约 Z(经验值,待核)。我做个专项。"

**Q20 [老板] 算法说慢是不是你们平台的锅?**
> A:"我已经定位了:其中 IO 侧我们负责的部分占 X,已在优化;计算侧那部分是模型结构问题。我和算法在一起查,不是单方面的事,这周给结论。"

### 跨职能/通用

**Q21 [PM] 这个需求技术上能做吗?**
> A:"能做。但有两个量级:小改 3 天覆盖 80% 场景,大改 2 周覆盖全。你的业务现在最需要哪个?我们按 ROI 排。"

**Q22 [测试/QA] 怎么验证一致性?**
> A:"我提供抽样对数工具:同一批 key,拉训练侧特征值和线上服务值逐字段比,出差异报告。我把脚本和用法给你。"

**Q23 [任何人] 这个我不太懂,你能保证吗?(超出你认知范围)**
> A:"这块我现在不敢拍胸脯,我需要拉数据/做个验证。给我 X 时间,我给你确定的结论。"(**这是高手答法,不是怂**)

**Q24 [算法] 平台又出问题了,你们到底行不行?(带情绪)**
> A:不接情绪,接事实:"这次问题我负责到底。根因是 X,已经 Y 止血,根治方案是 Z,DDL 是 W。这周我盯着不让它再发。"——用确定的行动平息情绪,**别陪着对方上情绪**。

**Q25 [算法] 别人家平台都有 XX 功能,咱为啥没有?**
> A:"这个功能我了解,确实有价值。它没排进来是因为优先级在 Y(更紧急的)。你要是觉得它该提前,咱评估下它能带来多少收益,我拿去排优先级。"——不被"别人家有"绑架,**回到收益和优先级**。

**Q26 [老板] 你这个项目和算法的项目,我只能投一个,投哪个?**
> A:不能只夸自己:"算法那个直接出效果,见效快;我这个是底座,不做的话算法迭代会越来越慢、越来越容易出事。**建议:算法先上拿短期收益,我这边用最小投入(N 人)先把最大的隐患堵住**,两边都不耽误。"——给老板一个**不二选一的第三方案**。

**Q27 [训练] 你们样本表又改 Schema 了,我代码跑挂了!**
> A:"抱歉影响到你了。这次加了字段 X,理论上向后兼容。你贴下报错,我看是读取姿势问题还是真不兼容。**以后 Schema 变更我提前一天在群里通知 + 给迁移说明**,避免再踩。"——认影响 + 帮定位 + 给流程改进。

**Q28 [跨团队] 这个事到底归谁?(扯皮)**
> A:"咱别先纠结归属,先看怎么最快解决。**这部分我先认领,边界处的 X 我拉你一起**,定位清楚了责任自然清楚。"——用"先解决"破"先甩锅"的僵局。

**Q29 [老板] 这个技术方案,简单说就是干啥?**
> A:用一个类比:"打个比方,**特征平台就像算法的'中央厨房'**——以前每个算法自己买菜洗菜(手工配特征),又慢又容易出问题;现在统一备料、统一质检,他们直接拿来炒(用),快还不容易出事。"——**用类比降维**,老板秒懂。

**Q30 [算法] 你保证上了这个一致性就没问题了?**
> A:不打包票:"不能保证 100%,工程上没有绝对。但我能保证:**有监控、有告警、出问题能 X 分钟内发现并定位**。从'事后才发现'变成'实时可控',这是质变。"——**承诺机制,不承诺绝对**。

> 🎓 **老师视角**:看完这 30 条,你应该发现答法有统一"语法":**先给确定的结论 → 带一个数字 → 给一个方案或下一步 → 把球踢回去(问对方一个收敛性的问题)**。把这个语法练成肌肉记忆,任何问题来了你都能套。还有一条暗线:**永远不打包票"绝对/100%",但永远给"机制/方案/下一步"**——这是工程师可信度的根。

## 7. 红线:外行话一说就露怯

### 7.1 一说就露怯的反例清单

| ❌ 外行话 | 为什么露怯 | ✅ 内行说法 |
|---|---|---|
| "这个应该没问题吧" | 不确定还嘴硬,没数据 | "我压测过,数据是 X" 或 "我去拉个数据" |
| "AUC 高就行了" | 不懂离在线 gap、不懂线上指标 | "训练 AUC 还要看线上 CTR 和一致性 gap" |
| "加个特征不就是加一列吗" | 不懂 slot/样本膨胀/吞吐影响 | "加 slot 要看样本膨胀和训练吞吐影响" |
| "实时不就是快点吗" | 不懂流式一致性/label 回流 | "实时涉及 label 回流和离在线一致性" |
| "数据存哪不都一样" | 不懂行存/列存/IO 影响 | "列存配列裁剪能省 IO,行存适合点查" |
| "重跑一遍不就好了" | 不懂回填成本/资源/时效 | "回填要 X 小时、占 Y 资源,得排低峰" |
| "GPU 不够就加卡" | 不看利用率,只会堆资源 | "先看利用率,喂不饱加卡也没用" |
| "这是算法的问题" | 急于甩锅,不定位 | "咱拆链路定位,IO 侧我查,计算侧你看" |
| "差不多""应该""大概可能" | 不量化,显得没把握 | 给具体数字或"待核"标注 |
| "这个简单,分分钟搞定" | 低估复杂度,翻车被打脸 | "看起来不复杂,我评估下有没有坑再给你时间" |
| "用户随便用,反正能扛住" | 不懂容量/限流,埋事故 | "当前压测扛 X QPS,留了余量,超了有限流降级" |
| "数据有点问题,我重刷下就行" | 不懂回刷的连锁影响 | "重刷会影响下游 X,我先评估影响面再动" |
| "AB 没差异,可能是平台没生效" | 先甩锅给平台 | "我先查特征有没有真正下发,确认了再看效果" |
| "这个需求做不了" | 一刀切,不给路 | "直接做不了,但有 A/B 两个变通,各有取舍" |

> 🎓 **老师视角**:露怯的本质不是"用错词",是**"在不确定的地方装确定"**。内行也有不懂的,但内行会说"我去拉个数据"。你把"差不多、应该、大概"换成"数字"或"我去确认",立刻从外行变内行。还有一类隐性露怯:**"这个简单"**——一旦说出口,做砸了是双倍打脸,做成了也只是"本来就简单"。**永远别低估复杂度,给自己留评估的余地。**

### 7.2 不懂时怎么得体接话(三句保命话术)

```
1. 承接:"这是个好问题 / 这块确实关键。"        ← 先稳住场子
2. 诚实:"我现在手上没有准确数据,不想拍脑袋。"   ← 诚实但不慌
3. 兜底:"给我 X 时间,我拉到数据/确认后给你结论。" ← 给确定的下一步
```

> **千万别**:① 硬编一个数字糊弄(被戳穿信任崩);② 转移话题装没听见(显得心虚);③ 把锅甩给别人("这得问 XX")——除非真不是你的领域。

## 8. 术语速查卡:开会前临阵磨枪

> 开会前 5 分钟扫一眼,确保这些词张口就来、不说错。详细解释见 [关联阅读](#-关联阅读)。

| 术语 | 一句话(够你开会用) |
|---|---|
| **slot** | 一个特征位/特征域,加 slot ≈ 加特征,影响样本大小 |
| **pass** | 训练中数据流过一遍的逻辑批次单位(搜广推常用) |
| **batch / batch size** | 一次喂给模型的样本数,影响显存和吞吐 |
| **epoch** | 全量数据训练一轮 |
| **离在线一致性** | 训练用的特征值 = 线上推理时的特征值,不一致则效果不可信 |
| **特征穿越/泄漏** | 训练时用了"未来才知道"的信息,导致 AUC 虚高 |
| **样本膨胀** | 加特征/序列变长导致样本体积变大,拖慢读取和训练 |
| **回填(backfill)** | 用历史数据重新生成样本/特征 |
| **序列特征** | 用户行为序列,有 maxlen(定长/变长) |
| **label 回流** | 用户行为(点击/转化)作为标签回到样本里 |
| **列裁剪** | 只读需要的列/slot,省 IO |
| **AUC** | 排序模型常用离线指标,但≠线上效果 |
| **P99 延迟** | 99% 请求的延迟上限,看尾部性能 |
| **GPU 利用率** | 喂得饱不饱的指标,低=IO 瓶颈 |
| **流批一体** | 一套逻辑同时处理实时流和历史批,减少不一致 |
| **Schema 演进** | 表结构变更(加字段等)不破坏历史数据 |
| **时间旅行** | 读取某个历史版本的数据(数据湖能力) |
| **OOM** | 内存/显存爆了;分 host OOM 和 CUDA OOM |
| **影子流量** | 复制线上真实流量到测试环境压测,数据更真实 |
| **特征 backfill 一致性** | 回填历史样本时,特征值要和"当时"一致,不能用现在的值 |
| **point-in-time(时点正确)** | 取特征要取"样本发生那一刻"的值,防穿越 |
| **混部** | 训练/在线/离线任务共享机器,省成本但有干扰 |
| **降级** | 服务扛不住时,牺牲部分功能保住核心(如特征超时给默认值) |
| **多级缓存** | 本地+远程多层缓存,防热点 key 击穿 |
| **灰度** | 小流量先上,验证没问题再放量 |
| **冷启动** | 新用户/新 item 没历史数据时的特征/推荐策略 |

> 🎓 **老师视角**:这张卡的用法不是"背全",是**"会前点检"**——今天这个会要聊样本回填,你就把 backfill、样本膨胀、列裁剪这三个词的说法过一遍。**临阵磨枪,不快也光。** 名词说得顺,专业感就有了一半。另外一招:**会上听到不懂的词,别装懂,当场问**"这个 X 你指的是 Y 吗?"——问对了显得你严谨,装懂被发现才是真露怯。

## 9. 模板合集:周报/评审/复盘/向上汇报

### 9.1 周报骨架(给老板看的,不是流水账)

```markdown
## 本周周报 - [你的名字] - [日期]

【一句话】本周特征平台二期进度 60%,无 P1,一个风险待拍板。

### 进展(结论 + 量化)
- 完成特征 DSL 校验模块,特征上线测试链路打通(3 天→4 小时已验证)
- 训练样本读取优化上线,吞吐 12w→30w samples/s

### 风险/需协调
- ⚠️ 大促前手工链路是隐患,建议本周拍板是否插队治理 ← 需要你

### 下周计划
- 完成离在线对数监控,接入告警
```

> 周报**别写"我做了 ABC"**,要写"**ABC 带来了什么 + 我需要你做什么决定**"。流水账周报 = 没人看。

### 9.2 方案评审文档骨架

```markdown
# [方案名] 评审

## 1. 背景与问题(1 段 + 1 个量化数字)
## 2. 目标(可度量:从 X 到 Y)
## 3. 方案设计(架构图 + 核心流程)
## 4. 方案对比(表格:自研 vs 采购 vs 现状,维度=成本/工期/可控/风险)
## 5. 收益(量化)
## 6. 风险与预案(主动列 + 兜底)
## 7. 排期与里程碑(M1/M2 + 资源)
## 8. 待决策事项(明确要评审拍什么)
```

**填好的范例(节选,可直接套)**:

```markdown
# 特征离在线一致性治理方案 评审

## 1. 背景与问题
当前训练样本和线上服务特征由两套逻辑各自生成,近一季度因不一致
导致 4 起效果回退,每起平均废掉 ~1 个迭代(约 2 周算法人力)。

## 2. 目标
离在线特征值一致性 gap 从"不可观测"→"实时监控且 <0.1%(经验阈值)"。

## 4. 方案对比
| 维度   | 现状(双套逻辑) | A 同源生成(选) | B 仅加对数监控 |
|--------|------------------|------------------|----------------|
| 一致性 | 无保障           | 根治             | 只发现不根治    |
| 工期   | -                | 4 周             | 1 周           |
| 风险   | 持续出事         | 迁移期需双跑     | 治标不治本      |

## 8. 待决策事项
- [ ] 确认选 A(同源生成),排在 X 项目之前
- [ ] 批准迁移期双跑 2 周的额外资源
```

> **评审文档的灵魂在第 4 节(对比)和第 8 节(待决策)**。对比表证明你"想全了别的路";待决策清单告诉评审"今天到底要拍什么"。没有这两节的评审,要么被质疑"考虑不周",要么开成"务虚会"。

### 9.3 故障复盘骨架(对事不对人)

```markdown
# [故障] 复盘

## 时间线(精确到分钟:发现/定位/止血/恢复)
## 影响面(范围 + 量化损失,标待核)
## 根因(技术根因,5 Why 追到底)
## 止血措施(已做)
## 根治措施(待做 + Owner + DDL)
## 经验教训(可复用的,沉淀成监控/规范/checklist)
```

**5 Why 示范(追到可防御的根因)**:

```
现象:线上特征超时,推荐降级。
 Why1 为什么超时? → 某热门商品 key 读取被打爆
 Why2 为什么被打爆? → 缓存击穿,全打到底层存储
 Why3 为什么击穿? → 该 key 缓存过期瞬间高并发回源
 Why4 为什么没防住? → 没有热点 key 探测和多级缓存
 Why5 为什么没有? → 容量设计时没考虑突发热点场景
根治:加热点探测 + 多级缓存 + 单 key 限流(Owner/DDL)
经验沉淀:新增"热点 key"监控项 + 容量评审 checklist 加一条
```

> 🎓 **老师视角**:复盘的红线是**"对事不对人"**。一旦变成追责会,以后没人敢报故障,事故只会更多更隐蔽。复盘产出必须是"**可复用的防御**"(新增一个监控、一条校验、一项规范),而不是"某人以后注意点"。5 Why 不是为了找到"谁错了",是为了挖到"**系统哪里缺了一道防线**"。

### 9.4 向上汇报骨架(口头版,1 分钟)

```
"老板,一句话:[结论]。
 ROI:[做完赚/省什么];
 风险:[最大风险 + 预案];
 成本:[人月/资源];
 进度:[里程碑 + 交付时间]。
 需要你:[拍板什么 / 批什么资源]。"
```

### 9.5 IM/群消息模板(异步同步,也要结论先行)

```
❌ 反例(挤牙膏,逼对方追问):
   "在吗" → "那个特征的事" → "就是上次说的" → ……

✅ 正例(一条说清,带结论和诉求):
   "@老板 特征平台二期已上线,提效达标(上线 3 天→4 小时)。
    一个事需要你拍板:大促前手工链路是隐患,建议插队治理,
    占 1 人 3 天。同意的话我今天排进去。详情见文档[链接]。"
```

> 🎓 **老师视角**:IM 沟通的第一句话就要有"结论 + 诉求"。**永远别发"在吗"**——它把决定"要不要理你"的成本甩给对方,且毫无信息量。把对方需要的信息和你要的东西一次性给全,这是对别人时间的尊重,也是专业的体现。

## 业界怎么做(高手 vs 新手)

| 维度 | 新手平台工程师 | 高手平台工程师 |
|---|---|---|
| 讲方案 | 从背景讲起,埋雷在最后 | 结论先行,2 分钟讲完六段式 |
| 被质疑 | 防御、对撞、急着证明自己对 | 先复述接住,再用数据回应 |
| 对老板 | 满嘴技术名词 | 全程 ROI/风险/成本/进度 |
| 对算法 | 直接答能不能 | 先问口径,再给方案 |
| 不合理需求 | 要么硬扛要么照单全收 | 认可+摆代价+给替代+交回选择权 |
| 数字 | "差不多""提升明显" | 具体数字 + "经验值/待核"标注 |
| 不懂时 | 硬编糊弄或甩锅 | "我去拉数据,X 时间给结论" |
| 报坏消息 | 拖、瞒、报喜不报忧 | 第一时间报,带止血和方案 |
| 周报 | 流水账"我做了 ABC" | "ABC 的产出 + 需要你决策" |
| 会前 | 凭感觉上 | 5 分钟点检名词/数字/结论 |

> 🎓 **老师视角**:高手和新手的差距,**80% 不在技术,在"信息如何被接收"**。同样的方案,高手讲完批了,新手讲完被毙——不是方案差,是表达差。沟通是平台工程师的**第二专业**,且常常是晋升的天花板。

## 权衡取舍

沟通本身也是权衡,别走极端:

| 取舍 | 过度一侧 | 平衡点 |
|---|---|---|
| 简洁 vs 完整 | 太简洁=信息缺失,被反复追问 | 结论先行 + 细节备在后面随时展开 |
| 自信 vs 诚实 | 太自信=装懂翻车 | 确定的事斩钉截铁,不确定的事老实说"去查" |
| 配合 vs 守门 | 全配合=累死+出事 | 合理需求快接,不合理需求礼貌 push back |
| 量化 vs 真实 | 硬凑数字=被戳穿 | 给数字但标"经验值/待核" |
| 主动 vs 越位 | 啥都管=越界惹人烦 | 自己领域主动,他人领域给建议不拍板 |
| 透明 vs 制造恐慌 | 把所有风险都报成 P0 | 分级报告,大事拍板小事知会 |

## 常见误区与反直觉

1. **误区:讲得越详细越显专业。** 反直觉:**讲得越简洁、结论越清晰才显专业。** 细节是"按需展开"的,不是一股脑倒。
2. **误区:被质疑要立刻反驳证明自己对。** 反直觉:**先承接("你担心的是 X")反而赢得主动**,对撞只会升级对抗。
3. **误区:对老板要展示技术深度。** 反直觉:**老板不关心你多牛,只关心钱/风险/进度。** 满嘴术语 = 减分。
4. **误区:坏消息晚点报、想办法先扛。** 反直觉:**坏消息越早报越加分**,瞒报是慢性自杀。
5. **误区:答应所有需求显得配合。** 反直觉:**会拒绝的人更被尊重**,什么都接 = 什么都做不好 + 出事。
6. **误区:不懂的要硬撑过去。** 反直觉:**"我去拉数据"比硬编更显专业**,内行也常说这句。
7. **误区:周报写得越满活干得越多。** 反直觉:**周报是给人看决策点的**,流水账没人看,"需要你拍板 X"才有用。
8. **误区:"为什么不早做"是在怪我。** 反直觉:**它其实在问"现在做对不对"**,朝前看别朝后辩。

## 平台侧落地 Checklist(沟通)

**会前**
- [ ] 这个会的**目标/我要的结论**是什么?(批资源?拍优先级?对齐方案?)
- [ ] 听众是谁?需要哪套语言?(算法 / 训练 / 老板)
- [ ] 准备 1 个核心结论、2-3 个关键数字、3-5 个会用到的名词(术语速查卡点检)
- [ ] 预想 3 个最可能的质疑,各准备一句承接 + 数据
- [ ] 风险点提前想好,准备主动说 + 兜底

**会中**
- [ ] 结论先行,2 分钟讲完六段式
- [ ] 被怼时:先复述对方的点,再回应
- [ ] 不懂时:不装,给"去查"的确定下一步
- [ ] 全程量化,形容词换数字
- [ ] 收尾明确"需要在场的人做什么决定"

**会后**
- [ ] 当场承诺的 follow-up 24h 内闭环(承诺 3 天 2 天给)
- [ ] 会议结论/决策邮件或群里同步一遍(留痕,防扯皮)
- [ ] 坏消息/风险已升级到该知道的人

## 对接话术

**对算法(客户思维)**
- 接需求第一句永远是"**问口径**",不是"答能不能"。
- push back 公式:认可诉求 + 摆代价 + 给替代 + 交回选择权。
- 用数据建信任:"我压测过,数据在这。"
- 帮他挡一次事故 = 信任拉满。

**对训练(钻细节)**
- 第一句问"**GPU 利用率多少**",一句话分清 IO 锅还是计算锅。
- 出问题不争锅,争"怎么最快定位":metrics 我出,曲线你出,10 分钟对。
- OOM 先分 host / CUDA;吞吐慢先看喂得饱不饱。

**对 PM/测试**
- 给量级选择题:"小改 3 天覆盖 80%,大改 2 周覆盖全,你要哪个?"
- 给可操作的工具/脚本,不给概念。

---

## 向上沟通

- 永远以"**需要你做什么**"收尾(批资源 / 拍优先级 / 协调团队)。
- 技术词必翻译成 ROI / 风险 / 成本 / 进度四个槽。
- 坏消息:先止血(已做),再说怎么修(方案),最后才提需求。
- 申请资源:用"不做的代价"比"做的好处"更有杀伤力(损失厌恶)。
- 延期:认账(根因)+ 已追回(进展)+ 新承诺(时间)+ 可砍项(止损)。
- 周报/汇报的价值在"决策点",不在"工作量"。

## 自测三问

**Q1:算法在评审会上当众质疑"你这个吞吐数据是不是测出来好看的,线上根本到不了",你怎么接?**

<details>
<summary>参考答案</summary>

不要防御性反驳("怎么会,我测得很认真")。正确三步:
1. **先承接复述**:"你担心的是测试环境和线上不一致对吧?这个担心合理。"
2. **诚实定性**:"我这个数是单机压测峰值,标的是经验上限,线上有网络和混部干扰会打折。"
3. **给确定下一步**:"我拉一次线上影子流量实测,明天给你真实数,咱以那个为准。"

核心姿态:**先接住情绪和质疑点,再用诚实+数据回应,把对抗变成协作。** 反例是急着证明自己对,只会升级对撞、双输。
</details>

**Q2:老板问"这个特征平台投这么多人力到底有什么用?"你只有一句话的机会,怎么说?**

<details>
<summary>参考答案</summary>

绝不能讲技术细节("我们做了 DSL、同源生成、Schema 演进……")。要落到老板的槽里:

"**两个字:提效 + 防事故。** 算法迭代从 3 天缩到 4 小时,过去手工配置导致的离在线不一致事故能基本消除——那种事故等于白干一个迭代。"

关键:技术翻译成**业务价值(提效)+ 风险规避(防事故)**,并把"离在线不一致"这种黑话翻译成"白干一个迭代"这种损益归因。
</details>

**Q3:训练同学甩锅说"你们平台样本读太慢,把我 GPU 卡住了",你第一句话该说什么?为什么?**

<details>
<summary>参考答案</summary>

第一句应该是:**"你 GPU 利用率现在多少?"**

原因:GPU 利用率是"喂得饱不饱"的核心判据。
- 利用率低(<80%)→ GPU 在等数据 → IO 瓶颈 → 确实是平台/读侧的活,该我优化(列裁剪、prefetch、读带宽);
- 利用率高 → GPU 在满负荷算 → 是模型结构/计算瓶颈 → 是训练侧的活。

一句话就**客观地把锅分清楚**,既不甩锅也不背冤枉锅,还显得你懂训练侧。接着就钻细节:dataloader worker 数、prefetch、样本行存列存。**不争"是谁的锅",争"怎么最快定位"。**
</details>

**Q4(加餐):老板在评审会上当众问"这个为什么不早做?",你怎么接才不掉价?**

<details>
<summary>参考答案</summary>

绝不能掉进"过去"里辩解("之前没排期""没人提"显得甩锅;"是我们疏忽"白背锅)。

正确做法:**把"为什么不早做"翻译成"为什么现在正合适",朝前看**:

"之前优先级在 X(更紧急的事),这个问题是最近才在数据上放大的——上个月相关事故从 1 起涨到 4 起,说明它到了非做不可的临界点。**所以现在做,正是时候。**"

核心:所有"为什么不早做"的潜台词都是"现在做对不对"。用数据证明"现在做的时机合理",而不是为"过去没做"找借口。**永远朝前看,别朝后辩。**
</details>
