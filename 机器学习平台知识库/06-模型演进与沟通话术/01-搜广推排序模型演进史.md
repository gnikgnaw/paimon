## 0. 30 秒速通(TL;DR)

一句话:**搜广推排序模型这十几年,本质是一条"把人从特征工程里解放出来、再把负担转嫁给平台"的演进路线。**

- **早期(LR)**:模型简单到极致,效果全靠**人工拼特征**。算法同学一半时间在写特征交叉,平台只要存好宽表、喂得动 LR。
- **中期(FM → W&D → DeepFM)**:模型开始**自动学特征交叉**,人工交叉减少,但代价是**embedding 爆炸**——参数从几百万涨到几十亿,平台第一次被"大参数"压垮。
- **行为序列时代(DIN/DIEN)**:模型开始吃**用户行为序列**,要求平台能存、能拼、能实时更新"这个人最近点过的 50 个商品",样本和特征结构彻底变复杂。
- **多目标时代(ESMM/MMoE/PLE)**:一个模型同时预估**点击 + 转化 + 时长**,要求样本带**多个 label**、且 label 之间有依赖关系(曝光→点击→转化)。
- **当下/未来(SIM 长序列、多场景、生成式)**:序列从 50 拉到**上万**,场景从 1 个变 N 个,生成式推荐(LLM for Rec)还在路上。平台压力点:**超长序列存储检索、Embedding 规模、实时性、样本供给一致性**。

**给平台研发的核心判断**:你不需要会推公式,但你必须知道——**每一代模型升级,90% 的工程压力会落到"特征供给"和"样本供给"这两根管子上**。这篇就是帮你建立这个映射。

## 1. 史前时代:为什么先讲排序模型

搜广推(搜索 / 广告 / 推荐)的系统,粗看都是一个"漏斗":

```
                 用户来了 / 发了一个 query
                          │
        ┌─────────────────┴──────────────────┐
        │           召回 (Recall)             │  从亿级物料里捞出几千个候选
        │   "宁可错捞,不可放过"  快、广        │
        └─────────────────┬──────────────────┘
                          │  候选 ~ 数千
        ┌─────────────────┴──────────────────┐
        │           粗排 (Pre-rank)           │  几千 → 几百,轻量打分
        └─────────────────┬──────────────────┘
                          │  候选 ~ 数百
        ┌─────────────────┴──────────────────┐
        │           精排 (Ranking)            │  ★ 本文主角 ★  几百 → 排序
        │   最重的模型,决定最终展示顺序        │
        └─────────────────┬──────────────────┘
                          │
        ┌─────────────────┴──────────────────┐
        │           重排 (Re-rank)            │  打散、多样性、业务规则
        └─────────────────────────────────────┘
                          │
                       最终展示
```

> 漏斗细节见 [`../01-搜广推业务场景/01-搜广推三业务本质与漏斗架构.md`](../01-搜广推业务场景/01-搜广推三业务本质与漏斗架构.md)。

**为什么单独讲"排序模型"的演进?** 因为漏斗里**精排(Ranking)是模型迭代最猛的环节**:它直接决定用户看到什么、点不点、买不买,跟钱(广告)、跟留存(推荐)、跟体验(搜索)强绑定。算法同学嘴里 90% 的"新模型",讲的都是精排。所以本文的"模型演进史",约等于"精排模型演进史"。

排序模型干的事,抽象成一句话:**给定"用户 + 物料 + 上下文"这三方信息,预测一个分数(通常是点击率 CTR 或转化率 CVR),然后按分数排序。** 整个演进史,就是"**怎么把这三方信息更聪明地揉在一起、揉出更准的分数**"。

🎓 **老师视角**:平台研发最容易犯的错,是把"模型"想象成一个黑盒,觉得"那是算法的事,我只管把数据喂进去"。错。模型的每一次结构升级,都会**反向定义你要喂什么形状的数据**。LR 时代喂的是"一行宽表",DIN 时代喂的是"一行宽表 + 一个变长行为序列",ESMM 时代喂的是"一行 + 多个 label"。**模型结构变了,你的存储 schema、特征拼接逻辑、样本生成 pipeline 全得跟着变。** 这就是为什么平台研发必须懂模型演进——不是为了训模型,是为了**预判数据形状的变化**。

## 2. LR 时代(2010 前后):人工特征工程的黄金年代

### 它解决了什么

在 LR(Logistic Regression,逻辑回归)成为主流之前,工业界排序很多还是规则 + 简单统计。LR 的价值是:**用一个可解释、可大规模并行训练、上线极快的线性模型,把"几亿维特征 → 一个概率"这件事工程化了。** 它是第一个真正能扛住工业级流量的机器学习排序模型。

### 核心思想(一句话直觉)

> **每个特征给一个权重,加起来过一个 sigmoid,得到点击概率。** 特征重要就权重大,不重要就权重小,简单粗暴但有效。

LR 是**线性模型**——它本身**不会**自动发现"年轻男性 + 篮球鞋"这种组合很猛。它只看单个特征。于是想让它聪明,只能靠人**手动把"年轻男性×篮球鞋"拼成一个新特征喂给它**。这就是"人工特征工程"的由来。

### 结构示意

```
特征(几亿维 one-hot / 交叉特征)
  x1: 用户性别=男
  x2: 用户年龄段=18-24
  x3: 物料类目=篮球鞋
  x4: 性别男 × 类目篮球鞋   ← 人工交叉,LR 自己想不到
  x5: 年龄18-24 × 类目篮球鞋 ← 人工交叉
   ...
       │  每个 xi 配一个权重 wi
       ▼
  z = w1·x1 + w2·x2 + ... + wn·xn + b
       │
       ▼
  CTR = sigmoid(z) = 1 / (1 + e^-z)
```

### 为什么早期必须靠人工交叉特征

LR 看不见特征之间的"组合效应"。但现实里,**预测信号几乎全藏在组合里**:
- 单看"男性",点击率没区分度;单看"连衣裙",也没区分度;
- 但"男性 × 连衣裙"这个组合,点击率极低——这才是有用的信号。

LR 自己学不出这个组合,**只能人提前算好、拼成一维特征喂进去**。于是算法同学大量时间花在:想哪些特征该交叉、做特征离散化(分桶)、统计各种 CTR 特征(比如"该用户历史 CTR""该物料历史 CTR")。这就是著名的"**特征工程占了机器学习 80% 的工作量**"那个年代。

> 特征工程与特征分类的全貌见 [`../03-特征数据管理/01-特征工程与特征分类体系.md`](../03-特征数据管理/01-特征工程与特征分类体系.md)。

### 对【样本 / 特征 / 平台】的新要求

| 维度 | LR 时代的要求 | 平台研发要支撑什么 |
|------|--------------|-------------------|
| 样本 | 单 label(点击 0/1),一行一个曝光 | 样本拼接:曝光日志 + 点击日志 join,打上 label |
| 特征 | 海量稀疏 one-hot + 大量**人工交叉特征** | 离线宽表存储;在线特征查询要快(KV) |
| 模型规模 | 参数 = 特征维数(几亿维,但都是稀疏的标量权重) | 稀疏参数服务(早期的 PS 雏形) |
| 实时性 | 弱,T+1 训练为主 | 离线批处理 pipeline 即可 |

🎓 **老师视角**:LR 时代平台最大的活儿其实是**"特征 KV 存储 + 特征拼接"**,而不是"算力"。因为 LR 模型本身极小(就是一堆权重),真正重的是"几亿维稀疏特征怎么存、怎么在 10ms 内查出来拼成一条样本"。**记住这个时期的特点:模型轻,数据管道重。** 后面你会看到,模型越来越重,但"数据管道重"这件事从来没轻过,反而越来越重。

🎓 **老师视角(反直觉)**:很多新人以为 LR 早就被淘汰了。其实**至今仍有大量场景在用 LR 或 LR 变体**,尤其是冷启动、小流量、强可解释要求的场景(比如风控、某些广告出价)。LR 的"可解释 + 上线快 + 几乎不吃机器"是深度模型给不了的。所以当算法同学说"这个新业务先上个 LR 跑通链路",别惊讶——那是务实,不是落后。

> **脱敏案例(LR 时代的特征拼接之痛)**
> 某电商早期 CTR 模型用 LR,算法团队维护了一张约 600 维的"人工交叉特征清单"(用户类目偏好 × 物料类目、时段 × 类目、地域 × 品牌……)。平台侧每天凌晨跑一条 Hive 链路,把曝光日志、用户画像、物料画像 join 成宽表,**再把这 600 个交叉特征逐个 SQL 算出来**。痛点有三:(1) 算法每加一个交叉特征,平台就要改一次 SQL、重跑全量,迭代周期以天计;(2) 交叉特征极度稀疏,很多组合一天只出现个位数,模型学不动;(3) 线上特征服务要存这 600 维,KV 查询压力大。**教训**:这正是 FM 出现的直接动机——把"人工拼 600 个交叉"换成"机器自动学二阶交叉",平台从此少维护一大堆易错的 SQL。这个案例也说明:LR 时代平台的瓶颈不在算力,在**"特征拼接 pipeline 的迭代效率"**。

## 3. FM / FFM(2010–2016):让机器自己做二阶交叉

### 它解决了 LR 什么痛点

LR 的痛点很直接:**人工交叉特征太累、且交叉特征极度稀疏。** "性别男 × 类目篮球鞋"这种组合,在样本里可能只出现几十次,LR 给它学的权重根本不可信(数据太少)。而且组合是爆炸的——100 个特征两两交叉就是约 5000 个,人根本拼不过来,拼出来也大多是稀疏到学不动的废特征。

FM(Factorization Machine,因子分解机)的洞见是:**别给每个"交叉组合"单独学一个权重,而是给每个特征学一个小向量(隐向量),两个特征的交叉强度 = 它们隐向量的内积。**

### 核心思想(一句话直觉)

> **每个特征背一个小向量(embedding),两个特征要不要一起放大信号,看它俩向量像不像。** 这样即使"男性×篮球鞋"这个组合样本很少,只要"男性"和"篮球鞋"各自在别的组合里学到了向量,它俩的交叉就能算出来——**信息共享,解决了稀疏问题。**

这是历史上一个关键转折:**embedding(隐向量)第一次大规模进入排序模型。** 后面所有深度模型吃 embedding 吃到吐,源头在这。

### 结构示意

```
LR(只有一阶):        FM(一阶 + 二阶自动交叉):

w1·x1 + w2·x2 + ...   w1·x1 + w2·x2 + ...        ← 一阶,和 LR 一样
                       + Σ <Vi, Vj>·xi·xj         ← 二阶,自动!不用人工拼

           特征 i ──► 隐向量 Vi = [0.2, -0.5, 0.1, ...]  (比如 8~32 维)
           特征 j ──► 隐向量 Vj = [0.1,  0.3, 0.4, ...]
           交叉强度 = Vi · Vj  (内积,一个标量)
```

FFM(Field-aware FM)是 FM 的增强:它认为"同一个特征,跟不同**字段(field)**交叉时应该用不同的向量"。比如"篮球鞋"跟"用户性别"交叉、跟"时间段"交叉,用的向量不该一样。FFM 在一些 CTR 竞赛(如 Criteo)里效果很猛,但**参数量是 FM 的 field 倍,工程开销大**,工业界全量用得没 FM 那么普遍。

| 对比项 | LR | FM | FFM |
|--------|-----|-----|------|
| 二阶交叉 | 靠人工拼 | 自动(单一隐向量) | 自动(按 field 区分隐向量) |
| 稀疏组合学得动吗 | 学不动 | 能(信息共享) | 能,且更精细 |
| 参数量 | 特征数 | 特征数 × 维度 | 特征数 × 维度 × field 数 |
| 工程开销 | 极低 | 中 | 高(参数 ×field 倍) |
| 工业界普及度 | 极高(至今仍用) | 高 | 中(竞赛多,全量少) |

### 对【样本 / 特征 / 平台】的新要求

| 维度 | 相比 LR 的变化 | 平台研发要支撑什么 |
|------|--------------|-------------------|
| 特征 | **人工交叉特征大幅减少**(机器自动二阶交叉) | 算法同学不用拼那么多交叉特征了 |
| 模型规模 | 从"标量权重"变成"每个特征一个向量" | 参数量 ×(向量维度),**第一次明显变大** |
| 训练 | 需要学 embedding,计算量上升 | 算力需求上升,但还在单机/小集群可控范围 |

🎓 **老师视角**:FM 是平台研发应该重点记住的一个"分水岭"。在 FM 之前,模型参数是"几亿个标量";FM 之后,变成"几亿个特征 × 每个一个向量"。这意味着**参数存储和传输的体量,凭空乘了一个系数(向量维度,常见经验值 8~64)**。如果你的特征参数服务(Parameter Server)是按 LR 时代的标量设计的,FM 一来就得改。**这是"embedding 吃内存"故事的第一章。**

> **脱敏案例(FM 上线后参数服务的第一次扩容)**
> 某信息流广告团队从 LR 切到 FM,隐向量维度取 16(经验值)。结果发现:原本 LR 的参数文件不到 2GB,FM 直接涨到约 30GB+——因为每个特征 ID 从"1 个权重"变成"16 个浮点数 + 偏置"。平台侧的连锁反应:(1) 参数服务器内存不够,临时扩了一倍机器;(2) 模型分发(把训好的参数推到线上)耗时从分钟级涨到十几分钟,影响了模型更新频率;(3) 在线预测时,一次打分要拉取的 embedding 量上升,网络带宽成了新瓶颈。**教训**:FM 看起来只是"算法换了个模型",但对平台是"参数体量乘了 16 倍"的硬冲击。**从这一刻起,平台研发必须把"参数规模 = 特征数 × embedding 维度"这个公式刻进脑子,每次模型升级先估这个数。**

## 4. GBDT + LR(Facebook, 2014):树来做特征变换

### 它解决了什么

FM 解决了"二阶交叉",但**高阶、非线性的特征变换**(比如对连续特征做复杂分桶、自动找出"年龄>30 且消费>1000 且最近活跃"这种组合)还是难。Facebook 2014 年那篇经典论文给了一个工程味十足的方案:**用 GBDT(梯度提升决策树)自动做特征变换,再把变换结果喂给 LR。**

### 核心思想(一句话直觉)

> **让一片决策树先把原始特征"嚼"一遍,样本落到哪些叶子节点上,就把"落在这些叶子"当成新特征喂给 LR。** 树天生擅长找非线性组合和自动分桶,LR 天生擅长大规模在线学习和快速更新——两者各取所长。

```
原始特征 ──►  [GBDT: 很多棵树]
                  树1: 落在叶子 #3
                  树2: 落在叶子 #7      ──► one-hot 编码成新特征 [0,0,1,0, 0,0,0,1, ...]
                  树3: 落在叶子 #2                              │
                  ...                                          ▼
                                                         [LR] ──► CTR
              ↑ 离线训练,变化慢          ↑ 在线快速更新,适应实时变化
```

这个架构的精髓是**"慢 + 快"的分工**:GBDT 部分重、离线训练、几天更新一次;LR 部分轻、可以高频甚至在线更新。这种"分层、不同更新频率"的思想,在今天的工业系统里仍然普遍。

### 对【样本 / 特征 / 平台】的新要求

| 维度 | 变化 | 平台研发要支撑什么 |
|------|------|-------------------|
| 特征 | 连续特征更受欢迎(树吃连续特征不用分桶) | 特征体系里数值特征的处理简化 |
| 训练 | 两阶段训练流水线(先 GBDT 再 LR) | **多阶段训练编排**:GBDT 产物要喂给 LR 训练,pipeline 变复杂 |
| 上线 | 在线要同时跑 GBDT 推理 + LR | 在线预测服务要能加载树模型 |

🎓 **老师视角**:GBDT+LR 对平台研发的真正启示,不是"树有多牛",而是**"模型可以是多阶段、多组件拼起来的"**。从这开始,你的训练 pipeline 不再是"一个模型一键训完",而是"组件 A 的输出是组件 B 的输入"。**这种 DAG 式的训练编排能力,是平台从这代开始就得有的。** 后面的 Wide&Deep、多塔模型,只会让这个 DAG 更复杂。

> **脱敏案例(两阶段训练的"产物对齐"坑)**
> 某广告团队上 GBDT+LR,踩了一个典型的工程坑:GBDT 离线每周更新一次,产出"样本落在哪些叶子"的编码规则;LR 在线高频更新。某次 GBDT 重训后叶子节点的编号变了(树结构变了),但**在线 LR 还在用旧的叶子编号**,导致特征错位、CTR 预估全乱,线上点击率掉了一截才被发现。**教训**:多阶段模型里,上游组件(GBDT)的产物是下游(LR)的"特征定义",**两者的版本必须强绑定、原子切换**。这就是为什么平台需要"模型/特征版本管理 + 一致性校验"——多阶段模型让"版本错配"成为一类高频事故。**记住:组件越多,接缝越多,接缝处就是事故高发区。**

🎓 **老师视角**:注意 GBDT **不擅长处理超高维稀疏特征**(几亿维 ID 类特征,树分裂时根本扫不过来)。所以 GBDT+LR 多用于**特征维度没那么爆炸**的场景(广告早期),而推荐里那种"几亿个 item ID"的稀疏场景,这条路走不通——这也是为什么紧接着深度模型(直接吃稀疏 embedding)会接管推荐。**模型选型从来跟"数据形状"强相关,这是平台研发要建立的第一性直觉。**

## 5. Wide & Deep(Google, 2016):记忆 + 泛化

### 它解决了什么

到这里,工业界面对一个本质矛盾:
- **记忆(Memorization)**:有些规律就是要"死记硬背"。比如"装了 A 应用的用户喜欢 B 应用"这种强共现,人工交叉特征 + 线性模型记得最准。
- **泛化(Generalization)**:有些规律要"举一反三"。比如一个从没共现过的"用户×物料"组合,得靠 embedding 的相似性去推断。深度网络擅长这个。

Wide & Deep 的答案:**两条腿一起走。** 一个 Wide 部分(线性,负责记忆,吃人工交叉特征)+ 一个 Deep 部分(DNN + embedding,负责泛化),两边的输出加起来一起预测,**联合训练**。

### 结构示意

```
            ┌──────────── Wide 部分 ────────────┐    ┌──────────── Deep 部分 ────────────┐
            │  人工交叉特征(线性)              │    │  稀疏特征 → embedding → 拼接        │
            │  "记忆":强共现、规则             │    │      │                              │
            │  app_installed × app_impression   │    │   [Embedding 层]  ← 吃内存大户      │
            │                                   │    │      │                              │
            │                                   │    │  [全连接 DNN: 几层]               │
            │                                   │    │  "泛化":没见过的组合也能推断       │
            └────────────────┬──────────────────┘    └────────────────┬──────────────────┘
                             └──────────────┬───────────────────────────┘
                                            ▼
                              CTR = sigmoid(wide输出 + deep输出)
                                   ↑ 两部分联合训练,一起更新
```

### 核心思想(一句话直觉)

> **左手记硬规律(Wide),右手猜没见过的组合(Deep),两只手一起做决定。**

Wide & Deep 是工业界深度排序模型的**真正起点**——它确立了"**Embedding + DNN**"这个延续至今的主干结构。后面 DeepFM、DCN、DIN 全是在它的"Deep 侧"做文章。

### 对【样本 / 特征 / 平台】的新要求

| 维度 | 变化 | 平台研发要支撑什么 |
|------|------|-------------------|
| 特征 | Wide 侧仍需人工交叉;Deep 侧吃大量稀疏 ID → embedding | **embedding 表规模开始爆炸**(几亿 ID × 几十维) |
| 模型规模 | 参数量从 FM 的"几亿向量"再上一台阶 | 大规模 Parameter Server 成刚需 |
| 训练 | 深度网络训练,需要 GPU / 分布式 | **分布式训练框架**(参数服务器 + worker) |
| 上线 | 在线推理要跑 DNN | 在线预测服务要支持深度模型推理(TF Serving 之类) |

🎓 **老师视角**:Wide & Deep 之后,平台研发的世界变了。在此之前你管的是"特征 KV + 简单模型";之后你要管**"embedding 大表(可能上百 GB 到 TB 级)+ 分布式训练 + 深度模型在线推理"**。这是平台架构的一次大升级。如果有人问你"为什么我们机器学习平台从这两年开始要这么多 GPU 和这么大的参数服务器",答案就在这一代——**深度模型 + 大 embedding 把算力和存储需求拉了一个数量级。**

🎓 **老师视角(给老板翻译)**:老板可能会问"Wide & Deep 不就是加了个深度网络,效果能好多少?"。你可以这样翻译:**它带来的不是"算法变聪明一点",而是"系统从此能处理没见过的长尾组合"——这对覆盖海量长尾物料/用户至关重要,直接关系到大盘的天花板。** 代价是平台投入(GPU、参数服务器)的台阶式上涨。这是一笔"用工程成本换泛化能力"的买卖。

> **脱敏案例(Wide&Deep 上线引发的平台架构换代)**
> 某推荐团队从 FM 升级到 Wide&Deep,这次升级**实质上是平台的一次架构换代**而非简单的模型迭代。变化清单:(1) embedding 表从 30GB 级涨到数百 GB(几亿 item ID × 32 维 + 用户侧 ID),单机参数服务器装不下,被迫上**分布式 Parameter Server**(参数分片到多台);(2) 训练从 CPU 单机/小集群,改为 **GPU + PS 架构的分布式训练**,平台第一次引入 GPU 集群和资源调度;(3) 在线预测从"查 KV + 算线性"变成"跑一个深度网络",上线了专门的深度模型推理服务(类 TF Serving),并要做模型热加载。**这一仗平台投入了一个季度。教训**:当算法说"上 Wide&Deep",平台研发要识别出这**不是改个模型,是上一套深度学习基础设施**——分布式训练、大参数服务、深度推理服务三件套缺一不可。把这个判断提前讲清楚,才能争取到合理的排期和资源。

## 6. DeepFM / DCN / xDeepFM(2017+):自动高阶交叉

### 它解决了 Wide & Deep 什么痛点

Wide & Deep 有个不优雅的地方:**Wide 侧还是要人工拼交叉特征。** 算法同学心想:FM 不是能自动做二阶交叉吗?能不能把 Wide 侧那个"靠人工"的部分,换成"自动学交叉"?于是这一代模型的主题就是:**彻底干掉人工交叉,让网络自动学出高阶特征交叉。**

| 模型 | 核心做法 | 一句话 |
|------|---------|--------|
| **DeepFM** | 用 FM 替换 Wide&Deep 的 Wide 侧,FM 和 DNN **共享 embedding** | "FM 管低阶交叉,DNN 管高阶,共用一套 embedding,不用人工交叉了" |
| **DCN** (Deep & Cross Network) | 设计一个 Cross 网络,**每加一层就自动多一阶交叉** | "想要几阶交叉,就叠几层 Cross,显式可控" |
| **xDeepFM** | CIN 结构,在 **vector 维度**做显式高阶交叉 | "DCN 的升级,交叉做得更细更显式" |

### 结构示意(以 DeepFM 为例)

```
                    稀疏特征 ──► [共享 Embedding 层]  ← 关键:两边共用,省参数省训练
                                      │           │
                          ┌───────────┘           └───────────┐
                          ▼                                     ▼
                   ┌──────────────┐                   ┌──────────────────┐
                   │  FM 部分      │                   │  Deep (DNN) 部分  │
                   │ 自动二阶交叉  │                   │  自动高阶交叉      │
                   └──────┬───────┘                   └─────────┬────────┘
                          └─────────────┬───────────────────────┘
                                        ▼
                                CTR = sigmoid(FM + DNN)

  对比 Wide&Deep:Wide 侧的"人工交叉"被 FM 的"自动交叉"取代 → 算法同学解放了
```

### 核心思想(一句话直觉)

> **"人工拼交叉特征"这件苦差事,正式外包给了网络结构。** 算法同学从此主要精力转向:选特征、调网络、做序列(下一代)。

### 对【样本 / 特征 / 平台】的新要求

| 维度 | 变化 | 平台研发要支撑什么 |
|------|------|-------------------|
| 特征 | **人工交叉特征基本消失**,以原始稀疏特征 + embedding 为主 | 特征体系简化为"原始特征 + embedding 配置",但 embedding 表更大 |
| 模型规模 | 网络更深更复杂,但参数大头仍是 embedding | embedding 存储继续膨胀;训练算力继续上升 |
| 训练 | 网络结构多样化,需要灵活的建模框架 | **平台要支持算法同学自定义网络结构**(从"配模型"到"写模型") |

🎓 **老师视角**:这一代对平台的核心冲击,是**"模型结构开始百花齐放"**。LR 时代你只要支持一种模型;到 DeepFM/DCN/xDeepFM,算法同学每周都想试新结构。平台不能再"为每个模型写死代码",必须提供**灵活的建模框架**(让算法同学用 TensorFlow/PyTorch 自由搭网络)+ **统一的特征接入 + 统一的训练调度**。**平台的角色从"提供特定模型"转向"提供建模与训练的基础设施"。** 这是从"工具"到"平台"的关键一跃。

🎓 **老师视角(反直觉)**:很多人以为"自动交叉"出来后,人工特征工程就死了。**没有。** 死的是"人工拼笛卡尔积交叉特征",但"选哪些特征进模型、做统计类特征(CTR/CVR 统计值)、做行为序列特征"这些更高级的特征工程,反而更重要了。**特征工程没消失,它升级了——从'拼组合'变成'设计高质量特征供给'。** 这恰恰是平台研发越来越重的原因。

> **DCN / xDeepFM 补充直觉**:DeepFM 的"自动交叉"主要到二阶(FM 部分),更高阶靠 DNN 隐式学。但 DNN 学高阶交叉是"隐式、不可控"的——你不知道它到底学没学到三阶、四阶组合。**DCN 的贡献就是把高阶交叉做成"显式可控":每叠一层 Cross 网络,就精确多一阶特征交叉**,想要几阶叠几层。xDeepFM 进一步在 vector(向量)层面而非 bit(标量)层面做显式高阶交叉,理论上更贴合 embedding 的语义。对算法是"交叉做得更细";**对平台基本无新增冲击**——它们都共用同一套 embedding 输入,平台侧的特征接入、参数服务、训练框架不变。这是个重要判断:**这一代的多个变体(DeepFM/DCN/xDeepFM)对平台是"同质"的,平台不必为每个新结构改造数据供给,只需提供灵活的建模框架让算法自由切换。**

> **脱敏案例(从"为模型写死代码"到"提供建模框架")**
> 某团队早期平台是"每上一个模型,平台同学就写一套对应的训练/推理代码"。到了 DeepFM/DCN/xDeepFM 阶段,算法每两周想试一个新结构,平台同学疲于奔命、成了瓶颈。后来平台做了一次关键重构:**把"特征接入、样本读取、分布式训练调度、模型导出"标准化成框架,模型结构交给算法用 TensorFlow 自由定义**(平台只规定输入输出接口)。重构后,算法换模型结构不再需要平台介入,迭代周期从"周"降到"天"。**教训**:这一代的核心信号是"模型结构百花齐放",平台的正确应对不是"追着每个模型写代码",而是**把自己从'提供模型'升级为'提供建模基础设施'**——这是机器学习平台真正成为"平台"的标志性一跃。

## 7. DIN / DIEN(阿里, 2017–2019):用户是会动的

### 它解决了前面所有模型什么痛点

前面所有模型(LR 到 xDeepFM),对"用户"的刻画基本是**静态画像**:性别、年龄、长期偏好类目……都是一堆"标签"。但真实的用户是**动态的、有上下文的**:
- 一个用户长期喜欢电子产品,但他**此刻**在看母婴用品(可能在给朋友挑礼物);
- 你给他推商品时,**他最近的行为序列**比他的长期标签更能说明"此刻想要什么"。

DIN(Deep Interest Network)的洞见:**别把用户行为序列压成一个固定向量,而是根据"当前要预测的这个候选物料",动态地从用户历史行为里挑出相关的那几个,重点看。** 这就是 **Attention(注意力)** 机制进入排序模型。

DIEN(Deep Interest Evolution Network)再进一步:**用户兴趣是会演化的**,用类似 GRU 的序列结构建模"兴趣随时间漂移"的过程。

### 核心思想(一句话直觉)

> **DIN:你要给我推篮球鞋,那我就重点看我历史上跟"篮球/运动"相关的行为,跟"买菜"相关的行为先放一边。** 注意力让模型"看人下菜碟"。
> **DIEN:不仅看历史,还看兴趣怎么一步步演变成现在这样。**

### 结构示意

```
   用户行为序列(变长!):  [点过A] [买过B] [看过C] [收藏D] ... 最近 N 个行为
                              │       │       │       │
   当前候选物料 ──► Attention 打分:  0.1     0.7     0.6     0.05   ← 跟候选越相关分越高
                              │       │       │       │
                              └───────┴───────┴───────┘
                                  加权求和 ──► 用户兴趣向量(随候选不同而不同!)
                                         │
                          + 其他特征 ──► [DNN] ──► CTR

  ★ 关键变化:同一个用户,对不同候选物料,算出的"兴趣向量"不一样 ★
```

| 对比项 | DIN | DIEN |
|--------|-----|------|
| 对序列的看法 | 无序集合(谁先谁后不重要) | 有序时间流(兴趣会演化) |
| 核心结构 | Attention 加权历史行为 | GRU 类序列 + 兴趣演化 + 辅助 loss |
| 对序列时间戳要求 | 可容忍乱序 | 必须严格有序、时间准确 |
| 在线推理开销 | 中 | 更重 |
| 工业落地 | 广泛(常用其简化变体) | 效果常更好但工程更重,选择性使用 |

### 对【样本 / 特征 / 平台】的新要求 ★ 这是平台压力的一次大跳变 ★

| 维度 | 变化 | 平台研发要支撑什么 |
|------|------|-------------------|
| **特征** | 新增**用户行为序列特征**(变长!"最近 50 个点击的 item_id + 类目 + 时间") | **序列特征存储**:每个用户存一条不断更新的行为序列;在线要能快速取出 |
| **样本** | 每条样本要带上"该用户在该时刻的历史行为序列" | **样本拼接复杂度暴增**:要做"时间点对齐"——不能用未来行为穿越 |
| **实时性** | 行为序列要尽量新("最近"才有价值) | **实时行为序列更新**:用户刚点了什么,几秒内要反映到序列里 |
| 模型 | Attention/序列结构,推理变重 | 在线推理延迟压力上升(序列越长越慢) |

🎓 **老师视角(本文最重要的一节之一)**:DIN/DIEN 是平台研发必须刻进 DNA 的一代。**从这代起,"用户行为序列"成了第一类特征,而它的工程难度远超普通特征:**
1. **存储**:它是变长的、每个用户一条、还在不断追加。普通特征是"用户→一个值",序列特征是"用户→一个不断增长的列表"。存储模型完全不同。
2. **实时性**:序列的价值在于"新"。用户上一秒的点击,这一秒就该进序列。这要求一条**实时行为流 pipeline**(行为日志 → 实时聚合成序列 → 写在线存储)。
3. **样本一致性(最容易踩坑)**:训练时给样本拼的序列,必须是"**该曝光发生那一刻**用户的真实历史序列",绝对不能混入曝光之后的行为——那叫**特征穿越/数据泄露**,线下指标虚高、线上一上就崩。这要求样本拼接严格按时间点对齐。

> 特征穿越、样本偏差是样本治理的核心议题,见 [`../04-样本数据管理/03-负采样与样本偏差治理.md`](../04-样本数据管理/03-负采样与样本偏差治理.md)。

🎓 **老师视角(给老板/算法翻译)**:当算法同学说"我们要上 DIN,需要用户实时行为序列",平台研发要立刻翻译成工作量清单:**(1) 建一套实时行为序列的存储与读写;(2) 改造样本拼接做时间点对齐;(3) 评估在线推理延迟。** 这三件事任意一件没做好,DIN 都上不了线。**模型一句话,平台三个月**,这句玩笑话从这代开始变真。

> **DIEN 补充直觉与对平台的额外要求**:DIN 只是"根据候选挑历史里相关的行为加权",它把序列当成一个**无序集合**看(谁先谁后无所谓)。DIEN 的进一步洞见是"**兴趣是会随时间演化的**"——你三个月前爱数码、上个月转向母婴,这种"漂移"本身是信号。DIEN 用类 GRU 的序列结构建模兴趣的时间演化,并引入"辅助 loss"监督每一步的兴趣表示。**对平台的额外要求**:序列必须**带准确的时间戳且严格有序**(DIN 可以容忍乱序,DIEN 不行);序列建模更重,在线推理延迟进一步上升。**判断**:DIEN 效果常优于 DIN,但工程更重、对序列质量(时序准确性)要求更高,很多团队权衡后线上用 DIN 或其简化变体。**这再次印证:模型选型是效果与工程成本的权衡,平台研发的延迟预算和数据质量,常常是那个"否决项"。**

> **脱敏案例(DIN 上线的"特征穿越"血案)**
> 某推荐团队上 DIN,线下 AUC 涨了 2 个点(经验值,相当可观),团队信心满满全量上线——结果线上 CTR 不升反降。排查两周,根因是**样本拼接时的特征穿越**:离线拼样本时,工程图省事,直接用"该用户当天的全部行为序列"去拼每一条曝光样本,而没有按"曝光发生的精确时刻"截断。也就是说,一条上午 9 点的曝光样本,被拼进了该用户当天 11 点、15 点的行为——**模型在训练时"偷看了未来"**,所以线下虚高;线上没有未来可看,自然打回原形。**教训**:序列特征 + 时间点对齐是 DIN 时代平台最容易翻车的地方。修复方案:样本拼接必须按曝光时间戳,只取**该时刻之前**的行为序列(point-in-time correctness)。**这条经验值得平台研发写进规范:任何序列/实时特征,线下拼样本时都要做严格的时间点对齐,线下指标异常好时第一个怀疑对象就是穿越。** 详见 [`../04-样本数据管理/03-负采样与样本偏差治理.md`](../04-样本数据管理/03-负采样与样本偏差治理.md)。

## 8. 多目标时代:ESMM / MMoE / PLE(2018–2020)

### 它解决了什么

前面的模型都在优化**单一目标**(通常是 CTR,点击率)。但真实业务要的不止点击:
- 电商要的是**成交**(CVR,转化率)、**GMV**;
- 视频要的是**完播、时长、点赞、关注**;
- 广告要的是**点击 + 转化 + ROI**。

如果给每个目标训一个模型,**成本高、且目标之间互相矛盾**(只优化 CTR 可能净推标题党)。所以诉求是:**一个模型同时预估多个目标(多任务学习),还要处理好目标之间的关系。**

### 三个代表,解决三类问题

```
ESMM(解决"样本选择偏差")        MMoE(解决"多任务怎么共享")       PLE(解决"任务互相打架")

曝光 ─► 点击 ─► 转化              ┌── Expert1 ─┐                  共享Expert + 每个任务专属Expert
 │      │       │                 │  Expert2  │── Gate ─► 任务A    ┌─共享─┐ ┌─专属A─┐ ┌─专属B─┐
 │      │       └ CVR只在点击后    │  Expert3  │── Gate ─► 任务B    └──┬──┘ └──┬───┘ └──┬───┘
 │      │         才有真实样本     └── Expert4 ─┘                      ▼任务A    ▼      ▼任务B
 │      └ pCTR                     多个"专家"共享底层,              通过门控决定共享/独占的比例
 └ 用全曝光空间建模CVR,           不同任务用不同门控权重组合         缓解"跷跷板效应"
   避开只在点击样本上训CVR的偏差
```

| 模型 | 解决的核心问题 | 一句话直觉 |
|------|--------------|-----------|
| **ESMM** (Entire Space Multi-task) | **CVR 的样本选择偏差**:CVR 传统上只在"被点击的样本"上训练,但线上要对"所有曝光"打分,训练和预测的样本空间不一致 | "通过 pCTR × pCVR = pCTCVR,在**全曝光空间**上训练,消除样本偏差" |
| **MMoE** (Multi-gate Mixture-of-Experts) | 多个任务**怎么共享底层又各有侧重** | "一组专家网络,每个任务用自己的门控,挑着用专家" |
| **PLE** (Progressive Layered Extraction) | 多任务**互相拖累(跷跷板效应)** | "共享的归共享,专属的归专属,别让任务互相抢资源" |

### ESMM 单独讲:对样本的硬要求

ESMM 是平台研发要特别理解的,因为它**直接改变了样本的结构**。传统 CVR 建模有个致命问题:

> CVR = P(转化 | 点击)。训练时只能用"被点击的样本"(因为没点击就没有转化标签)。但**线上预测时,要对所有曝光物料打 CVR 分**。训练样本(只有点击)和预测样本(所有曝光)空间不一致 → **样本选择偏差(Sample Selection Bias)**。

ESMM 的解法:不直接训 CVR,而是同时训 **CTR(曝光→点击)** 和 **CTCVR(曝光→点击→转化)**,然后 CVR = CTCVR / CTR。这两个任务都在**全部曝光样本**上训练,样本空间一致,偏差消除。

**这对样本的要求是:一条样本要同时带"是否点击"和"是否转化"两个 label,且 label 之间有依赖(没点击则必然没转化)。**

```
ESMM 的概率链(为什么能在全曝光空间训练):

   曝光(所有样本都有)
      │  pCTR = P(点击|曝光)        ← 在全部曝光样本上训练,无偏
      ▼
   点击
      │  pCVR = P(转化|点击)        ← 想直接训它,但只有点击样本 → 有偏!
      ▼
   转化

   ESMM 不直接训 pCVR,而是训两个"从曝光出发"的任务:
      pCTR    = P(点击|曝光)            ┐ 都在全曝光样本上
      pCTCVR  = P(点击且转化|曝光)       ┘ 训练,样本空间一致
   再推出:pCVR = pCTCVR / pCTR

   ★ 关键:label 的依赖链(曝光→点击→转化)被显式建模,样本要带全链路标签 ★
```

### 对【样本 / 特征 / 平台】的新要求 ★ 样本结构的又一次跳变 ★

| 维度 | 变化 | 平台研发要支撑什么 |
|------|------|-------------------|
| **样本** | 一条样本带**多个 label**(点击、转化、时长、点赞……),且 label 间有**业务依赖链**(曝光→点击→转化) | **多 label 样本生成**:从多份行为日志(曝光/点击/成交/播放)join 出一条带全标签的样本;处理 label 的时间延迟(转化可能几天后才发生 → 归因窗口) |
| **样本** | 不同 label 的正样本极度不平衡(转化远少于点击) | 样本平衡策略、加权 |
| 训练 | 多任务网络,训练资源上升 | 训练框架支持多 loss、多输出 |
| 评估 | 要同时看多个目标的指标,且看融合后的线上效果 | 评估体系支持多目标指标 |

🎓 **老师视角(平台必懂)**:ESMM/多目标时代,平台研发的战场从"特征供给"扩展到了"**样本供给**"。多 label 样本的生成,工程上比单 label 难得多:
- **多源 join**:曝光日志、点击日志、成交日志、播放日志……要按"同一次曝光"对齐拼成一条。
- **归因窗口与延迟标签**:转化可能在曝光后 1 天、7 天才发生。样本什么时候"定稿"?这涉及**延迟反馈(delayed feedback)**问题,平台要设计归因窗口、可能要做样本回填。
- **label 一致性**:多个 label 必须来自同一条曝光、同一时刻,不能错位。

🎓 **老师视角(向上沟通)**:老板问"我们能不能既提点击又提成交?"——这背后就是多目标模型 + 多 label 样本。你要让老板知道:**"一个模型多目标"在算法上不难,难的是'多 label 样本能不能稳定、准确、及时地供给'——这是平台的活儿,且转化标签有延迟,样本管道天然更复杂。** 把复杂度归位到正确的地方,是平台研发体现专业度的时刻。

> **MMoE / PLE 补充直觉**:多任务学习有个反直觉的现象叫"**跷跷板效应(seesaw)**"——你想同时优化点击和转化,结果点击涨了转化掉、或者反过来,按下葫芦浮起瓢。原因是不同任务对底层共享参数的"拉扯"方向不一致。**MMoE 的解法**:不用一个共享底座,而是一组"专家网络(Expert)",每个任务通过自己的"门控(Gate)"决定**更多采用哪几个专家**——任务相似就共享多,任务冲突就各用各的。**PLE 再进一步**:显式区分"共享专家"和"每个任务专属的专家",并分层渐进抽取,更彻底地缓解任务互相抢资源。一句话:**MMoE/PLE 是在解决"多个目标如何既共享信息又不互相打架"的结构问题。** 对平台而言,它们和单目标模型相比,**输入特征不变,但输出从 1 个变 N 个、loss 从 1 个变 N 个**——平台的训练框架要支持多输出多 loss,评估体系要支持多目标指标。

> **脱敏案例(多 label 样本的"延迟反馈"难题)**
> 某电商上 ESMM 做"点击 + 成交"双目标,样本生成时撞上了**延迟反馈(delayed feedback)**:用户今天点了商品,可能 3 天后才下单、7 天后才确认收货。问题来了——**今天生成的样本,转化 label 该填 0 还是 1?** 如果立刻填 0(还没转化),那 3 天后真转化了,这条样本就成了"假负样本",模型会低估 CVR;如果等 7 天再生成样本,模型更新就严重滞后,丢失实时性。平台最终的折中方案:设定**归因窗口**(经验值如 7 天),窗口内的转化都归因到原曝光;同时引入**样本回填/修正机制**——先按当前状态生成样本训练,转化发生后再修正 label 重新参与训练(或用专门的延迟反馈建模)。**教训**:多目标模型的工程难点 90% 不在模型,在**"带延迟的多 label 样本怎么稳定、准确、及时地供给"**。归因窗口的长短是"准确性 vs 实时性"的直接权衡,平台研发要和算法、业务一起定这个窗口,并把"延迟标签回填"做成样本管道的标准能力。

## 9. 当下与未来:多场景、长序列 SIM、生成式推荐

### 9.1 多场景 / 多域(Multi-domain)

大公司一个 App 里有 N 个推荐位:首页猜你喜欢、购物车推荐、详情页推荐、搜索结果页……每个场景数据分布不同,但又有共性。

```
单场景各训一个模型:              多场景统一建模(共享+区分):
  首页模型   (数据多,够训)         ┌──────── 共享底座 ────────┐
  购物车模型 (数据少,训不好)        │  跨场景共享的特征/参数    │
  详情页模型 (数据少,训不好)        └────┬──────┬───────┬──────┘
  ...                                   场景A塔  场景B塔  场景C塔
  问题:小场景没数据;N个模型             (各自的场景专属参数)
       维护成本爆炸                    好处:小场景借大场景的力;一套维护
```

- **痛点**:每个场景单独训一个模型,小场景数据不够、维护成本高;合成一个模型又会被大场景带偏(大场景样本多,把模型拽向自己)。
- **思路**:类似多任务,把"场景"也当成一个维度,**共享 + 区分**(代表作如 STAR 等)——底座共享、每个场景留专属参数。
- **对平台**:样本要打上**场景标识**;特征/embedding 要支持跨场景共享与隔离;一个模型服务多个场景,上线、灰度、监控都要按场景维度切分。

| 维度 | 单场景多模型 | 多场景统一模型 | 平台研发要支撑什么 |
|------|------------|--------------|-------------------|
| 小场景效果 | 差(没数据) | 好(借大场景) | 样本打场景标识,支持混合训练 |
| 维护成本 | 高(N 个模型) | 低(1 个) | 一模型多场景的部署与版本管理 |
| 监控 | 每个独立 | 要按场景拆指标 | 监控/AB 按场景维度切分 |
| 风险 | 互不影响 | 一处改动影响多场景 | 灰度要能按场景灰度,避免全炸 |

🎓 **老师视角**:多场景对平台最隐蔽的要求是**"监控和灰度要按场景切分"**。一个模型服务 5 个场景,一旦上线后某个小场景指标崩了,而大盘被大场景的好数据盖住看不出来——你就漏报了事故。**统一模型省了维护成本,但把"按场景观测"的责任推给了平台。** 别只看大盘,要有分场景的指标看板。

### 9.2 长序列建模:SIM(Search-based Interest Model)

DIN/DIEN 的行为序列通常只取**最近几十个**(经验值 50~100)。但一个重度用户可能有**几年、上万条**行为。短序列丢掉了大量长期兴趣信号。

```
DIN 时代:只看最近 50 个行为
   [................最近50个................]   ← 丢掉了更早的几千上万条

SIM 时代:从超长序列(经验值上万)里"先检索再建模"
   全部历史行为(上万条)
        │
   ┌────┴─────┐  第一步 GSU(General Search Unit):
   │ 根据当前候选,从上万条里"检索"出最相关的几十~几百条 │
   └────┬─────┘
        ▼
   ┌──────────┐  第二步 ESU(Exact Search Unit):
   │ 对检索出的子序列做精细 attention 建模(类似 DIN)  │
   └──────────┘
```

- **核心思想**:**"先检索、再精算"**——上万条行为不可能全塞进模型(太慢),所以先用一个轻量检索从超长序列里挑出跟当前候选最相关的一小撮,再精细建模。
- **对平台的硬冲击**:
  - **超长序列存储**:每个用户存上万条行为,数据量比 DIN 时代大 1~2 个量级。
  - **在线检索能力**:在线要在毫秒级从用户上万条历史里检索出相关子序列(可能用倒排/向量检索)。
  - **离线样本拼接**:给每条样本拼一条上万长度的历史序列,样本体积暴涨,存储和 IO 压力大。

> **脱敏案例(SIM 长序列的存储与 IO 爆炸)**
> 某团队从 DIN(序列长度 50)升级到 SIM(目标拉到数千~上万),平台侧遭遇三连击:(1) **存储暴涨**:用户行为序列从"每人 50 条"变成"每人数千条",在线存储容量涨了 1~2 个量级,成本陡增;(2) **样本体积爆炸**:离线给每条曝光样本拼一条上万长度的序列,单条样本从几 KB 变成几十上百 KB,训练数据总量翻了十几倍,训练 IO 成了瓶颈,一轮训练时间显著拉长;(3) **在线检索延迟**:SIM 的 GSU(检索单元)要在毫秒内从上万条历史里挑出相关的几十条,朴素遍历扛不住,平台要为行为序列建**类目倒排或向量检索索引**。**最终方案**:序列存储与短序列分层(热的短序列走高速存储,长尾长序列走成本更低的存储);样本侧用"序列 ID 引用 + 训练时按需拉取"避免每条样本物理冗余整条序列;在线引入按类目的倒排检索做 GSU。**教训**:长序列是"存储 + IO + 在线检索"三重工程难题,平台研发要在算法决定序列长度之前,就把这三笔账算清楚——**序列长度不是算法一句话的事,它直接定义了你的存储成本曲线和训练吞吐。**

### 9.3 生成式推荐 / LLM for Rec(趋势,尚在演进)

最新趋势:借鉴大语言模型(LLM)的思路做推荐。先看一个范式对比直觉:

```
传统"打分式"范式:                       生成式"生成式"范式:
  候选物料(召回来的几百个)               不预先给候选,直接"生成"要推的物料ID
       │                                       │
  逐个打分 score(user, item)             把物料编码成"语义ID序列"
       │                                  像生成文本一样自回归生成下一个物料的语义ID
  按分排序                                       │
       │                                  生成出的语义ID → 解码回真实物料
  取 TopK                                  好处:候选不受召回限制,端到端
```

几条路线:

| 路线 | 一句话 | 状态 |
|------|--------|------|
| **生成式检索**(如 TIGER 类) | 把物料编码成"语义 ID",像生成文本一样**"生成"要推荐的物料 ID**,而非传统"打分排序" | 工业界探索中 |
| **LLM 做特征/理解** | 用 LLM 理解物料内容、用户意图,产出更强的语义 embedding 喂给传统模型 | 落地较多(最务实) |
| **生成式排序 / 一体化** | 用类 Transformer 大模型统一召回+排序,scaling law 思路(模型越大效果越好) | 前沿,大厂在投入 |

- **对平台的潜在冲击(前瞻)**:
  - **算力再上台阶**:类 LLM 模型训练/推理对 GPU 的需求又是数量级跳变。
  - **语义 ID / 物料编码**:需要离线生成并维护物料的语义表示(物料动态变化,要增量编码)。
  - **推理延迟与成本**:生成式推理比打分式重得多,在线服务架构要重新设计(可能引入 KV cache、量化、蒸馏)。

| 对比项 | 传统打分式(DIN/ESMM 等) | 生成式(LLM4Rec) | 对平台的含义 |
|--------|------------------------|------------------|-------------|
| 候选来源 | 召回给定 | 可端到端生成 | 召回与排序边界可能被打破 |
| 在线开销 | 较可控 | 重(自回归生成) | 在线成本曲线全新,需严控 |
| 物料表示 | ID embedding | 语义 ID / LLM embedding | 新增离线物料编码管道 |
| 训练成本 | 大但成熟 | 极大(scaling law) | GPU 弹性与预算管理 |

🎓 **老师视角**:生成式推荐目前(2026)还在"部分场景见效、尚未全面取代"的阶段,平台研发**不必焦虑追新,但要预留架构弹性**:特别是 GPU 资源的弹性调度、超大 embedding/模型的存储、以及"在线推理成本"这条新的成本曲线。**判断:未来 3 年,排序系统大概率是"传统深度模型 + 生成式组件"混合,而非一夜替换。** 平台的本事是"两种范式都能稳定承载"。

> **脱敏案例(生成式组件的"成本曲线"冲击)**
> 某团队尝试用 LLM 做物料内容理解,产出语义 embedding 喂给传统精排模型(这是当下最务实的"LLM for Rec"落地路线之一)。效果上长尾物料的推荐质量确实有提升,但平台撞上了一条全新的**成本曲线**:(1) 物料语义 embedding 要离线批量生成,几亿物料过一遍大模型,GPU 消耗远超传统训练;(2) 物料是动态的(每天新增大量),embedding 要增量更新,需要一条"新物料 → LLM 编码 → 写入特征库"的常驻管道;(3) 一旦尝试把 LLM 放到在线链路(实时理解 query/意图),推理延迟和成本直接顶破精排预算。**最终方案**:把 LLM 严格限制在**离线 / 近线**链路(离线生成语义特征),绝不放进在线打分主路径;在线仍用轻量传统模型消费这些预生成的语义特征。**教训**:生成式时代平台研发的核心功课是管理一条**新的成本曲线**——LLM 的能力很诱人,但"什么放离线、什么放在线"的边界划分,直接决定成本是可控还是失控。**先用离线/近线方式吃到生成式的红利,把在线主路径的延迟和成本守住,是当前阶段最稳的工程策略。**

## 10. 贯穿主线:模型越进化,平台越被压榨

把整条线拉直,你会看到一条极其清晰的主线:

```
模型演进方向 ───────────────────────────────────────────────►
LR    FM   GBDT+LR  W&D   DeepFM  DIN/DIEN  ESMM/MMoE/PLE  SIM  生成式
 │     │      │      │       │        │            │         │     │
人工 ─►自动 ─►树变换─►记忆 ─►自动高阶 ─►行为序列 ─►多目标 ─►长序列─►生成
特征   二阶          +泛化   交叉      +attention   多label   上万    /LLM
       交叉

平台压力随之上涨的四根柱子:
┌─────────────────────────────────────────────────────────────────┐
│ ① Embedding 规模 :  小 → → → → → → → → → → → → → → → → → → 巨大     │
│    (FM 起步,W&D 爆发,至今几十亿~千亿参数)                       │
│ ② 序列 / 长序列   :  无 → → → → → DIN 几十 → → SIM 上万             │
│    (序列存储、实时更新、在线检索)                                 │
│ ③ 实时性          :  T+1 → → → → 准实时 → → → 秒级行为反馈          │
│    (实时特征、实时样本、实时序列)                                 │
│ ④ 样本供给复杂度  :  单label → → → → → → 多label + 延迟反馈 + 归因  │
│    (多源 join、时间点对齐、归因窗口、防穿越)                      │
└─────────────────────────────────────────────────────────────────┘
```

**一句话总结这条主线:**

> 模型的每一次进化,都是"**让算法少干一点苦活(人工特征),让平台多扛一点重活(数据/算力供给)**"。
> 算法的"自动化"红利,是用平台的"工程化"成本换来的。

### 演进主线总表(模型 → 新增数据/特征要求 → 平台压力)

> 这张表是全文的"骨架记忆"。平台研发听到任何一代模型,先在脑中定位到这一行,新增工作量就八九不离十了。

| 代际 | 代表模型 | 解决上一代的痛点 | 核心直觉(一句话) | 新增的样本/特征要求 | 平台要新支撑什么 |
|------|---------|----------------|------------------|--------------------|----------------|
| 线性 | LR | 把"亿维特征→概率"工程化 | 每个特征配权重相加过 sigmoid | 单 label;海量人工交叉特征 | 特征 KV 存储 + 人工交叉特征拼接 pipeline |
| 自动二阶 | FM / FFM | 人工交叉累且稀疏学不动 | 每个特征一个隐向量,交叉=内积 | 人工交叉减少 | embedding 首现,**参数 ×维度**,PS 雏形 |
| 树变换 | GBDT+LR | 高阶非线性变换难 | 树嚼一遍喂给 LR,慢+快分工 | 连续特征友好 | **多阶段训练编排** + 版本一致性 |
| 深度起点 | Wide&Deep | 记忆与泛化要兼得 | 左手记硬规律,右手猜新组合 | Deep 侧大量稀疏 ID→embedding | **大 embedding 爆发 + 分布式训练 + DNN 在线推理** |
| 自动高阶 | DeepFM/DCN/xDeepFM | Wide 侧还要人工交叉 | 把人工交叉外包给网络结构 | 人工交叉基本消失 | **灵活建模框架**(平台从工具→基础设施) |
| 行为序列 | DIN / DIEN | 用户被当成静态画像 | 看候选挑相关历史行为(attention) | **用户行为序列(变长)** | ★**序列存储 + 实时更新 + 防穿越对齐**★ |
| 多目标 | ESMM/MMoE/PLE | 只优化单目标;CVR 样本偏差 | 一模型多目标;ESMM 全空间建模 | **多 label + 依赖链 + 延迟标签** | ★**多 label 样本 + 归因窗口 + 延迟回填**★ |
| 长序列 | SIM | DIN 序列太短丢长期兴趣 | 上万行为先检索再精算 | **超长序列(经验值上万)** | 超长序列存储 + 在线检索 + 样本 IO 优化 |
| 生成式 | TIGER/LLM4Rec 等 | 打分式范式的天花板 | 像生成文本一样"生成"推荐 | 语义 ID / 物料语义表示 | 算力再升级 + 离线编码管道 + 在线推理成本管控 |

🎓 **老师视角(总表怎么用)**:把这张表当"查表器"用。算法说"要上 PLE",你查到"多目标"行,立刻知道:输入特征不变,但要支持多 label 样本、多 loss、多目标评估,且要警惕延迟标签。算法说"要把序列从 50 拉到 5000",你查到"长序列"行,立刻知道:存储要扩、样本 IO 要优化、在线要上检索。**这比记住任何模型的数学细节都实用——平台研发的核心能力,是把"模型语言"快速翻译成"工程工作量"。**


这就是为什么——**机器学习平台研发的价值,随着模型演进只增不减。** 模型越先进,对"高质量、低延迟、高一致性的特征与样本供给"的依赖就越深,而这正是平台研发的核心战场。

🎓 **老师视角(全文最该记住的一句)**:作为平台研发,你听算法同学讲任何一个新模型,脑子里要自动跑这个映射表——**"这个模型,对我的 ① embedding ② 序列 ③ 实时性 ④ 样本供给,分别提了什么新要求?"** 答出这四条,你就接得住任何一代模型。这比会推任何一个公式都值钱。

## 11. 一个团队的平台演进编年史(把上面串成一条故事线)

为了让"模型→平台压力"的映射真正落地,我们把前面拆散的脱敏案例,串成一个虚构但典型的团队编年史。读者可以把自己代入"平台研发负责人",感受每一步该提前预判什么。

```
时间线 ──►   T0          T1          T2          T3          T4          T5
模型     →   LR     →   FM     →   W&D    →   DIN    →   ESMM   →   SIM
平台形态 →   宽表+KV →  +PS雏形 →  分布式深 →  +序列存储→ +多label →  +长序列
                                   度学习栈    +实时流    样本管道    检索栈
```

**T0 · LR 阶段:平台只是"数据管道工"**
- 核心资产:一张离线宽表 + 一套在线特征 KV。
- 每天的活:join 曝光/点击日志打 label,算几百个人工交叉特征。
- 主要矛盾:算法每改一个交叉特征,平台就要改 SQL 重跑——**迭代效率**是瓶颈,不是算力。
- 平台研发的心智:"我是数据管道工,保证宽表准时产出、KV 查得快。"

**T1 · FM 阶段:第一次被"参数规模"教育**
- 触发点:embedding(隐向量)进入模型,参数体量乘以维度。
- 新增的活:扩参数服务、优化模型分发。
- 第一次意识到:**"参数规模 = 特征数 × 维度"这个公式,以后每次升级都要先算。**
- 平台研发的心智开始转变:"我不只是管数据,我还要管'模型参数'这个新资产。"

**T2 · Wide&Deep 阶段:平台架构换代**
- 触发点:大 embedding + 深度网络 + 泛化能力。
- 新增的活(一个季度的工程量):分布式 Parameter Server、GPU 集群与调度、深度模型在线推理服务。
- 这是质变:**平台从"特征工具"升级为"深度学习基础设施"。**
- 平台研发的心智:"算法说上 W&D,我听到的是'上一整套深度学习栈'。"

**T3 · DIN 阶段:数据从"行"变成"行 + 序列"**
- 触发点:用户行为序列 + attention。
- 新增的活:序列存储(变长追加)、实时行为流 pipeline、样本拼接的时间点对齐。
- 踩过的坑:特征穿越导致线下虚高、线上翻车(见前面的血案)。
- 平台研发的心智:"序列特征是第一类公民了,而且实时性和防穿越是新的生命线。"

**T4 · ESMM/多目标阶段:战场从特征扩到样本**
- 触发点:一个模型多目标 + CVR 样本偏差。
- 新增的活:多源日志 join 出多 label 样本、归因窗口、延迟标签回填。
- 难点认知:**90% 的复杂度在"带延迟的多 label 样本怎么稳定供给",不在模型本身。**
- 平台研发的心智:"我现在既要保特征供给,又要保样本供给,后者更难。"

**T5 · SIM/长序列阶段:存储与 IO 的极限挑战**
- 触发点:序列从几十拉到上万。
- 新增的活:超长序列分层存储、在线检索结构、样本"引用而非冗余"以省 IO。
- 成本认知:**序列长度直接定义存储成本曲线和训练吞吐,不是算法一句话的事。**
- 平台研发的心智:"我要在算法定序列长度之前,先把存储/IO/检索这三笔账摆到桌面上。"

🎓 **老师视角(编年史的总结)**:看完这条线,你会发现平台研发的角色在持续升维:
- **T0–T1**:数据管道工(保证宽表和 KV)。
- **T2**:基础设施工程师(撑起分布式训练 + 大参数 + 深度推理)。
- **T3–T4**:数据供给的架构师(序列、实时、多 label 样本,且要保证一致性与防穿越)。
- **T5 及以后**:成本与权衡的决策者(在算法做决定之前,提供"这样做要花多少存储/算力/延迟"的判断)。

**每一代模型,都在把平台研发往"更上游、更有话语权"的位置推。** 你越能提前把"模型语言→工程账单"翻译清楚,你在团队里的位置就越关键。这就是为什么这篇演进史值得平台研发反复读——**它不是算法八卦,它是你职业升维的地图。**

## 业界怎么做

- **Google**:Wide & Deep 的发源地。强调"记忆 + 泛化",其推荐/广告体系大量基于 TensorFlow 的深度模型 + 大规模 embedding。
- **Meta / Facebook**:GBDT+LR 经典论文的出处;后来 DLRM(Deep Learning Recommendation Model)开源,推动了大规模 embedding + 稀疏特征工程的工业标准,以及对内存/通信的极致优化。
- **阿里(淘天)**:DIN / DIEN / SIM / ESMM / STAR(多场景)几乎都出自阿里妈妈/搜推团队——**用户行为序列建模这条线基本是阿里定义的**。其 X-DeepLearning(XDL)等框架专为大规模稀疏 + 序列设计。
- **腾讯**:PLE(多任务,缓解跷跷板)是腾讯的代表作,广泛用于视频号/腾讯视频等多目标场景。
- **字节**:以"实时性 + 大规模"著称,推荐系统强调全链路实时(实时特征、实时样本、实时模型更新),Monolith 等系统体现其对在线学习/实时大 embedding 的投入。
- **共性**:头部公司都不是"用最新论文模型",而是"**把适合自己数据形状和业务目标的模型,工程化到极致**"。平台能力(大 embedding、实时管道、稳定样本)往往比模型本身更是护城河。

🎓 **老师视角(从业界共性提炼给平台的三条启示)**:
1. **模型会扩散,平台才是壁垒。** DIN、PLE 这些模型论文一发,全行业都能抄;但"几亿用户的实时行为序列稳定供给""千亿参数 embedding 的低延迟服务"这类平台能力,抄不走。**算法是明面的招式,平台是内功。**
2. **每家的模型选型都跟它的数据形状强绑定。** 阿里电商行为序列丰富,所以序列建模(DIN/SIM)是它的主场;腾讯多业务多目标,所以 PLE 出自腾讯;字节追极致实时,所以全链路实时是它的标签。**别盲目抄某家的模型,先看自己的数据形状像不像它。**
3. **工程化到极致 > 追最新模型。** 头部公司真正的功夫在"把一个不算最新的模型,在自己的数据和流量上打磨到极致稳定高效"。**对平台研发的启示:你的价值不在帮算法上多新的模型,而在让任何一代模型都能稳定、高效、低成本地跑起来。**

## 权衡取舍

| 取舍维度 | 一端 | 另一端 | 平台研发的判断 |
|---------|------|--------|---------------|
| 模型复杂度 vs 上线成本 | 上越新越复杂的模型 | 保持简单(LR/FM) | 看 ROI:复杂模型的效果增量,能否覆盖算力/延迟/维护成本。冷启动、小流量场景,简单模型常更划算 |
| 效果 vs 延迟 | 长序列、深网络更准 | 推理要快(精排有严格延迟预算) | 精排延迟预算通常很紧(经验值几十毫秒)。序列拉长、模型加深前,先确认在线延迟接得住 |
| 单模型多目标 vs 多模型 | 一个模型省资源、共享信息 | 多模型各自最优、解耦 | 多目标模型省成本但调参难、易跷跷板;目标差异极大时拆开更稳 |
| 实时性 vs 系统复杂度 | 秒级实时特征/样本最准 | T+1 简单稳定 | 实时管道贵且脆。先评估"实时带来的效果增量"是否值得那套实时基建 |
| Embedding 大 vs 成本 | 大 embedding 表达力强 | 小 embedding 省内存/带宽 | embedding 维度、ID 数量直接决定参数服务器规模。要有特征准入/淘汰机制控规模 |
| 追新模型 vs 打磨数据 | 上最新结构 | 把特征/样本质量做扎实 | **经验:数据质量带来的提升,常常大于换模型。** 平台研发应优先保障数据供给 |

## 常见误区与反直觉

- **误区 1:"新模型 = 更好,赶紧上"。** 反直觉:新模型的效果增量常常 < 它带来的工程/算力/延迟成本。很多场景换模型不如把特征和样本质量做扎实。**模型选型要看数据形状和业务 ROI,不是看论文新不新。**
- **误区 2:"自动特征交叉出来后,特征工程就没了"。** 反直觉:人工"拼笛卡尔积"那种交叉确实少了,但"设计高质量特征供给(统计特征、序列特征、实时特征)"反而成了胜负手。**特征工程没死,它升级到平台层了。**
- **误区 3:"模型是算法的事,跟平台关系不大"。** 反直觉:每代模型升级,90% 的落地工作量(序列存储、多 label 样本、实时管道、embedding 扩容)都在平台。**模型一句话,平台三个月。**
- **误区 4:"线下指标(AUC)涨了就能上线"。** 反直觉:线下涨、线上崩,最常见的元凶是**特征穿越 / 样本与服务不一致**——尤其在序列特征、延迟标签场景。线下指标好,先怀疑数据泄露。
- **误区 5:"序列越长越好、embedding 越大越好"。** 反直觉:都有边际递减,且直接顶着在线延迟和成本天花板。SIM 之所以要"先检索再建模",正是因为不能无脑全塞。
- **误区 6:"多目标模型把目标加进去就行"。** 反直觉:多 label 样本的生成(多源 join、归因窗口、延迟反馈、label 时间点一致性)才是难点,且任务间会跷跷板,需要 MMoE/PLE 这类结构和精细调参。
- **反直觉总结**:**模型演进的红利在算法侧显现,但成本和风险绝大部分沉淀在平台侧的"数据供给"。** 谁能稳定供给高质量特征和样本,谁就掌握了模型迭代的命脉。

## 平台侧落地 Checklist

当算法同学说"我们要上 ××模型",平台研发逐条对照:

**① Embedding / 参数规模**
- [ ] 新模型的 embedding 表预计多大(ID 数 × 维度)?现有参数服务器/存储扛得住吗?
- [ ] 是否需要特征准入(低频 ID 不入表)与淘汰(过期 ID 清理)机制控制规模?
- [ ] 训练/在线推理的内存、带宽是否需要扩容?

**② 序列 / 长序列特征**
- [ ] 是否引入用户行为序列?序列长度(经验值:DIN 几十,SIM 上万)?
- [ ] 序列特征的**存储模型**(变长追加)是否就绪?
- [ ] 在线能否在延迟预算内取出/检索序列?长序列是否需要检索结构(倒排/向量)?

**③ 实时性**
- [ ] 行为序列/实时特征要多新(秒级?分钟级?T+1?)?
- [ ] 是否需要实时行为流 pipeline(行为日志 → 实时聚合 → 在线存储)?
- [ ] 实时特征是否做到"训练与服务一致"(同一套口径)?

**④ 样本供给**
- [ ] 单 label 还是多 label?多 label 涉及哪几份日志的 join?
- [ ] 是否有延迟标签(如转化)?归因窗口多长?样本何时"定稿"、是否要回填?
- [ ] 样本拼接是否严格**时间点对齐**、**防特征穿越**?
- [ ] 多 label 正负样本是否极度不平衡?是否需要采样/加权(见负采样文档)?

**⑤ 在线推理与上线**
- [ ] 在线推理延迟在精排预算内吗(经验值几十毫秒)?
- [ ] 模型在线/离线特征口径一致(避免 training-serving skew)?
- [ ] 灰度/AB 方案、回滚方案就绪?监控覆盖核心指标?

**⑥ 训练编排**
- [ ] 是多阶段/多组件模型吗(如 GBDT+LR)?训练 DAG 编排支持吗?
- [ ] 分布式训练资源(GPU/PS)是否充足?

**对算法同学**(显得懂、接得住):
- "你这个模型主要吃**行为序列**还是吃**多 label**?序列长度大概多少?这决定我这边存储和样本拼接怎么改。"
- "CVR 这个目标,是要用 **ESMM 在全曝光空间**建模吗?那我样本得带点击和转化两个 label,转化有延迟,归因窗口你按几天算?"
- "embedding 表大概多大?要不要我们配**特征准入和淘汰**,不然参数服务器顶不住。"
- "这个序列特征,训练时拼的是**曝光那一刻**的历史吧?我做严格时间点对齐,别让我们线下穿越了。"

**对算法同学**(必须守住的底线):
- "上线前请确认线下评估**没有用未来信息**;序列和延迟标签场景我最担心穿越,我们一起核对样本口径。"
- "在线推理延迟我有预算(经验值 X ms),序列拉长/模型加深前我们先压测。"

**对算法同学**(主动给方案,不只被动接需求):
- "你想上长序列,我建议先验证'最近 200 个'够不够,真要上万再上检索栈——别一步到位把成本打爆。"
- "多目标想加'时长'这个 label?那我得确认时长日志的归因口径,以及它和点击 label 怎么时间对齐,这个方案我来出。"
- "embedding 想用 64 维?我先按'特征数 × 64'估参数服务器要多少机器,给你和老板一个成本数,再定。"

## 向上沟通

**老板问"我们模型到第几代了/跟竞对比怎么样":**
- "我们精排主力是 ××(比如 DIN+多目标),属于'**深度模型 + 行为序列 + 多目标**'这一代,跟头部公司在同一技术代际。差距通常不在模型结构,而在**数据供给的实时性和规模**——这正是平台在持续投入的。"

**老板问"为什么要这么多 GPU / 这么大的存储":**
- "从 Wide&Deep 那一代起,深度模型 + 大 embedding 把算力和存储拉高了一个数量级;上了行为序列(DIN)后又加了实时管道;上了长序列(SIM)存储再涨。**这不是浪费,是这代模型的入场券。** 不投这些,我们连同代际的模型都跑不起来。"

**老板问"能不能既提点击又提成交/时长":**
- "这是多目标模型能做的,算法上成熟。难点在**多 label 样本的稳定供给**(多源日志 join、转化有延迟要做归因)——这是平台的活儿,我们已经/需要 N 人周投入。把这块做扎实,多目标才立得住。"

**老板问"要不要赶紧上生成式推荐 / 大模型推荐":**
- "趋势是真的,但目前(2026)还在'部分场景见效'阶段,且推理成本高。我的建议是**预留算力弹性、先在小场景验证 ROI**,不盲目全量替换。同时把当前的数据供给打扎实——无论哪代模型,高质量特征和样本都是地基。"

## 自测三问

**Q1:算法同学说"我们要把精排从 DeepFM 升级到 DIN",作为平台研发,你脑子里立刻要冒出哪几项新工作?**

<details>
<summary>参考答案</summary>

DIN 相比 DeepFM 的最大变化是**引入用户行为序列 + attention**。平台要立刻评估四件事:
1. **序列存储**:用户行为序列是变长、不断追加的,需要专门的存储模型(用户→行为列表),DeepFM 时代的"用户→若干标量/ID 特征"存不下。
2. **实时更新**:序列价值在于"新",需要实时行为流 pipeline,把用户最新行为及时写进序列。
3. **样本拼接的时间点对齐**:训练样本拼的序列必须是"曝光那一刻"的真实历史,严防特征穿越(用了曝光之后的行为)。
4. **在线推理延迟**:attention + 序列让推理变重,要确认在精排延迟预算内。

(embedding 规模也会增长,但序列相关的三件事是 DIN 特有的硬骨头。)
</details>

**Q2:为什么 ESMM 要在"全曝光空间"上训练?这对样本提了什么要求?**

<details>
<summary>参考答案</summary>

传统 CVR(转化率)建模只用"被点击的样本"训练(因为没点击就没有转化标签),但**线上要对所有曝光物料打 CVR 分**——训练样本空间(只有点击)和预测样本空间(所有曝光)不一致,产生**样本选择偏差**。

ESMM 不直接训 CVR,而是同时训 **CTR(曝光→点击)** 和 **CTCVR(曝光→点击→转化)**,二者都在**全部曝光样本**上训练,再用 CVR = CTCVR / CTR 推出,样本空间一致,偏差消除。

对样本的要求:**一条样本要同时带"是否点击"和"是否转化"两个 label**,且 label 间有依赖(没点击必然没转化)。平台要从曝光/点击/成交多份日志 join 出多 label 样本,并处理**转化的延迟反馈与归因窗口**。
</details>

**Q3:贯穿整个演进史,模型对平台压力上涨的"四根柱子"是什么?给老板解释"为什么模型越先进平台越费钱",你怎么说?**

<details>
<summary>参考答案</summary>

四根柱子:**① Embedding 规模、② 序列(长度)、③ 实时性、④ 样本供给复杂度。**

给老板的解释:模型每一次进化,本质是"让算法少做人工特征这种苦活,把负担转移到平台的数据/算力供给上"。具体说——深度模型 + 大 embedding(从 Wide&Deep 起)把算力存储拉高一个数量级;行为序列(DIN)要求实时管道;长序列(SIM)要求超大存储和在线检索;多目标(ESMM)要求复杂的多 label 样本供给。**这些投入是"跑同代际模型的入场券",不是浪费;省了这些,我们连和竞对同代的模型都跑不起来。** 而且数据质量往往比模型结构更决定最终效果,所以平台在数据供给上的投入回报率很高。
</details>

## 附:算法黑话速查(平台研发随身卡片)

听到这些词,脑子里立刻闪过"它对我意味着什么":

| 黑话 | 一句话翻译 | 对平台的信号 |
|------|----------|-------------|
| CTR / CVR / CTCVR | 点击率 / 转化率 / 点击且转化率 | 多 label 样本、归因窗口 |
| embedding | 把 ID 映射成一个稠密向量 | 参数规模 = ID 数 × 维度,吃内存/带宽 |
| 特征交叉 | 把两个特征组合成信号 | 早期靠人工拼;现在网络自动,人工交叉变少 |
| 行为序列 | 用户最近点/买/看的物料列表 | 变长存储 + 实时更新 + 防穿越对齐 |
| attention | 看候选挑相关历史重点看 | 序列特征建模,推理变重 |
| 多任务 / 多目标 | 一个模型同时预估多个目标 | 多 label 样本、多 loss、跷跷板 |
| 样本选择偏差 | 训练样本空间≠预测样本空间 | ESMM 全空间建模的动因 |
| 延迟反馈 | label 在曝光后很久才到 | 归因窗口 + 样本回填 |
| 特征穿越 / 数据泄露 | 用了未来信息训练 | 线下虚高线上崩,样本要时间点对齐 |
| training-serving skew | 训练与线上特征口径不一致 | 特征口径要统一,常见线上掉点元凶 |
| Parameter Server (PS) | 存储/同步大规模稀疏参数的服务 | 大 embedding 的承载底座 |
| GSU / ESU | SIM 的检索单元 / 精算单元 | 长序列要先检索再建模 |
| 语义 ID | 把物料编码成可"生成"的 ID | 生成式推荐的离线编码管道 |

🎓 **老师视角(怎么用这张卡片)**:平台研发不需要听懂算法同学的每个公式,但**必须能把每个黑话翻译成"对我的存储/样本/算力/延迟意味着什么"**。这张表就是翻译器。开会时算法抛出一串术语,你能当场接话"那这块样本管道我得改、这块存储我得扩"——这就是"听得懂、接得住、翻译得了"的本事。
