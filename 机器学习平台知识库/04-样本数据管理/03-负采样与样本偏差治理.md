# 负采样与样本偏差治理（Negative Sampling & Bias Mitigation）

> **一句话定位**：搜广推的样本天生「正负极度不均衡 + 系统性带偏」——负采样解决「样本太多太贵」，偏差治理解决「样本本身就是歪的」。这两件事都发生在**样本落盘之前**，是平台研发能直接拍板、也最容易背锅的环节。
> **写给谁**：做样本平台 / 特征平台的研发——你不需要会推 IPW 公式，但必须能在评审现场讲清「我们为什么采样、采完为什么要校准、线上的样本到底歪在哪、平台怎么留住纠偏需要的信息」。
> **读完你能**：把「负采样 / 采样校准 / selection bias / position bias / popularity bias / ESMM / IPW / PAL」串成一套体系，并能把它翻译成「省了多少机器、会不会把 CTR 算歪、广告会不会多扣钱」讲给算法和老板听。

## 0. 30 秒速通（TL;DR）

- **负采样是什么**：搜广推一次曝光页面，点击的样本（正）只占百分之几，剩下全是没点的负样本。负样本**又多又便宜又冗余**，所以训练前**只保留一部分负样本**，叫负采样（negative sampling），核心目的是**省机器、省时间**。
- **采样必须配校准**：采样改变了正负比，模型预估的概率会**系统性偏高**。必须用一个**确定性的公式**把概率校准回真实尺度，否则 CTR 预估偏高——在广告里直接等于**计费算错、出价错、钱算错**。
- **偏差是什么**：哪怕不采样，样本本身也是**歪的**——模型只能在「曝光过的」样本上学（selection bias）、高位置天生点击多（position bias）、热门 item 天生曝光多（popularity bias）。这些偏差会让模型**学到的是系统的惯性，而不是用户的真实偏好**。
- **两大解法主线**：① selection bias → **ESMM**（在「曝光→点击→转化」的全空间上多任务建模）；② position bias → **位置特征 / PAL / 倾向得分 IPW**（训练时告诉模型「这是第几位」，预测时抹掉位置影响）。
- **你作为平台侧要交付**：**采样率可追溯**、**位置/上下文信息务必随样本落盘**、**校准所需的真实分布口径**、**全空间样本（曝光甚至未曝光）的供给能力**。

> 💡 全篇围着一句话转：**采样动的是「样本的量」，偏差治理动的是「样本的代表性」——前者要配校准把数字摆正，后者要靠平台留住信息才有得救。**

## 1. 第一性原理：搜广推的样本天生「又多又歪」

要理解负采样和偏差治理，先认清搜广推样本的两个先天缺陷。它们不是 bug，是**业务形态决定的必然**。

```
   一次信息流刷新 / 一次搜索 → 系统返回 N 条结果（曝光）
        │
        ├── 用户点了 1~2 条        → 正样本（positive）  占比 ~1%~10%
        └── 剩下 N-2 条没点        → 负样本（negative）  占比 ~90%~99%
                                       └── 又多、又便宜、又高度冗余

   先天缺陷一：正负极度不均衡 + 负样本海量      → 催生「负采样」
   先天缺陷二：能进入样本的，都是系统「挑出来给用户看的」 → 催生「偏差治理」
```

**缺陷一（量的问题）**：一天百亿级曝光，正样本可能只有几亿，负样本几十上百亿。负样本里绝大多数是「一眼就知道用户不会点」的 easy negative，信息量低、还占着算力和存储。**全量训练既贵又不划算。**

**缺陷二（质的问题）**：模型只能从「曝光日志」里学习，而曝光是**上一版模型/策略挑出来的**——

- 用户从没见过的 item，没有样本（**selection bias**）；
- 排在第 1 位的 item 天生比第 10 位点击多，哪怕它没那么相关（**position bias**）；
- 热门 item 被反复曝光，冷门 item 没机会（**popularity bias**）。

> 🎓 **老师视角（讲这段能立刻拔高）**：很多人把负采样和偏差治理混为一谈，其实它们解决的是**正交的两个问题**——
> - **负采样动的是「量」**：样本还是那些样本，只是少留几条负的。它带来的偏差是**已知的、可由公式精确还原的**（所以能校准）。
> - **偏差治理动的是「质/代表性」**：样本集合本身就不能代表「全体候选 × 全体用户」的真实反馈。这种偏差是**系统性的、无法靠简单公式还原的**，只能靠建模（ESMM）、靠特征（位置特征）、靠加权（IPW）去缓解。
>
> 在评审里能把这句话说清楚——「采样的偏差能校准，系统的偏差只能治理」——你就跳出了「调采样率」的层次，到了「理解样本代表性」的层次。

## 2. 负采样：动机、量级与第一笔账

负采样最朴素的动机就是一笔**成本账**。

| 维度 | 不采样（全量负样本） | 负采样后（如保留 1/10 负样本） |
|---|---|---|
| 样本量 | 日 ~100 亿条 | 日 ~10~20 亿条 |
| 训练时长 | 长（吞吐瓶颈） | 缩短数倍 |
| 存储成本 | 高 | 显著下降 |
| 正负比 | ~1:99（极不均衡） | ~1:9（相对友好） |
| 信息损失 | 无 | easy negative 冗余信息损失很小 |
| 副作用 | 无 | **预估概率系统性偏高，必须校准** |

> 上述为典型量级 / 经验值，不同业务差异很大，仅用于建立直觉。

负采样还有一个常被忽略的好处:**缓解类别不均衡**。1:99 的正负比会让模型「躺平预测全负」也有 99% 准确率，梯度被海量负样本淹没；采到 1:9 后，正样本的梯度信号更突出，模型更容易学。

```
   全量:  正 ▏           负 ███████████████████████████████  (1:99，正信号被淹没)
   采样后:正 ▏▏▏         负 ██████████                       (1:9，正信号凸显)
              ↑ 正样本不动，只下采样负样本
```

> 🎓 **老师视角**：注意——**正样本几乎从不下采样**。正样本是宝贵的、稀缺的、带强信息的；要动只动负样本。偶尔会对「超热门正样本」做轻微下采样以防头部主导，但那是另一回事。记住默认动作:**只下采样负样本（negative down-sampling），保留全部正样本。**

## 3. 负采样方法大全：六种采法与适用场景

负采样不是「随机扔掉一半」这么简单，采法直接决定模型学到什么。

### 3.1 六种主流采法

| 方法 | 怎么采 | 优点 | 缺点 / 风险 | 典型场景 |
|---|---|---|---|---|
| **随机负采样** | 对负样本按固定概率 w 随机保留 | 简单、无偏（分布不变） | 全是 easy negative，信息量低 | 排序基线 |
| **按曝光采样** | 以曝光为单位整页采或采条 | 保留页面上下文 | 实现稍复杂 | 精排 |
| **Batch 内负采样** | 把同 batch 内其他样本的正例当本样本的负例 | 几乎零额外成本、负例多 | 引入采样偏差需 LogQ 校正；热门 item 当负例过多 | 召回/双塔 |
| **流行度采样（popularity-based）** | 按 item 热度调整采样概率（越热越可能被采为负） | 对抗热度偏差 | 需要维护热度统计 | 召回 |
| **Hard negative（难负例）** | 主动挑「像正例但其实是负例」的样本（如曝光高位却没点、语义接近） | 信息量大、提升区分度 | 易引入噪声、可能采到假负（延迟反馈） | 召回/精排提效 |
| **全局负采样** | 从全库 item 里随机抽（不限于曝光） | 召回必须见过「没被曝光的」负例 | 与曝光分布不一致 | 召回 |

### 3.2 Easy negative vs Hard negative 的直觉

```
   候选 item 与「用户真实兴趣」的距离
   近 ←──────────────────────────────────→ 远
        正例     hard negative        easy negative
        (点了)   (差点点 / 像但没点)   (八竿子打不着)
                      ↑
            信息量最大的负例在这里:
            模型最容易搞混、最该学会区分的就是 hard negative
```

**easy negative**（随机/全局采到的，基本都是）告诉模型「这俩明显不相关」——模型一学就会，边际价值低。**hard negative** 告诉模型「这俩很像但用户就是没点」——这才是决定模型上限的信息。召回模型尤其依赖 hard negative 来提升区分度。

> 🎓 **老师视角**：hard negative 是把双刃剑。它信息量大，但**最容易和「假负样本」混淆**——一个曝光高位却没点的样本，可能是真不喜欢（好的 hard negative），也可能是**用户当时没看到/稍后才转化**（延迟反馈造成的假负，见 [[标签生成·归因窗口·延迟反馈]]）。平台侧要能区分「确定的负」和「还在归因窗口里、未盖棺定论的负」，否则 hard negative 越挖越脏。

### 3.3 平台侧的关键动作

无论哪种采法，平台研发要保证一件事:**采样率必须可追溯、可配置、随样本落盘**。原因见第 5 节——校准公式需要精确的采样率 w。

```
   样本落盘 schema 里必须有:
   ┌────────────┬───────────────────────────────────┐
   │ sample_weight / neg_sample_rate │ 这条样本对应的采样率（校准要用） │
   │ sample_strategy_id              │ 用了哪种采样策略（可回溯）       │
   │ is_sampled                      │ 是否被采样命中                   │
   └────────────┴───────────────────────────────────┘
```

## 4. 召回 vs 排序：为什么负采样策略截然不同

这是面试和评审高频考点，也是平台供给两条不同样本流的根因。

```
   召回（亿 → 千）              精排（百 → 十）
   ┌──────────────┐            ┌──────────────┐
   │ 候选 = 全库    │            │ 候选 = 上游送进来的几百个 │
   │ 负例从哪来?    │            │ 负例从哪来?               │
   │ → 全库随机/全局│            │ → 曝光了但没点的           │
   │   负采样       │            │   （曝光未点击）           │
   └──────────────┘            └──────────────┘
```

| 维度 | 召回（Recall） | 精排（Rank） |
|---|---|---|
| 负样本来源 | **全库**（绝大多数 item 用户从没见过） | **曝光未点击**（用户见过、没点） |
| 为什么 | 召回要从全库捞，必须见过「没曝光过的负例」，否则线上一遇到没见过的 item 就乱排 | 精排只需在「能被曝光的 item」里排序，用曝光未点最贴近线上分布 |
| easy/hard | 需要大量 easy negative（全局）+ 少量 hard negative | 曝光未点本身就是天然的 hard negative |
| 采样规模 | 一个正例配几十~几百负例 | 正负比相对温和 |
| 经典坑 | **样本选择偏差（SSB）**:训练用曝光样本，预测面向全库 → 分布不一致 | position bias、延迟反馈假负 |

> 🎓 **老师视角**：「召回为什么要全库负采样、精排为什么用曝光未点」——这是区分「懂搜广推」和「只懂调包」的分水岭问题。一句话记法:**召回面向全库做检索，所以负例必须来自全库;精排面向曝光集做排序，所以负例来自曝光未点。** 平台侧因此要供给**两条不同的样本流**:召回样本流需要能采到「未曝光的全库 item」，精排样本流就是曝光日志拼接（见 [[样本全生命周期与日志拼接]]）。

## 5. 采样后校准：不校准 = 把 CTR 算错 = 广告多扣钱

这是负采样**最容易被研发忽视、却最致命**的一环。

### 5.1 为什么采样会让概率偏高

负采样把负样本删掉了一部分，模型看到的「正负比」被人为抬高了。模型如实地学到了这个**被抬高的正比例**，于是预估的点击概率**系统性偏高**。

```
   真实世界:   正 1 : 负 99   →  真实 CTR ≈ 1%
   采样后(保留1/10负): 正 1 : 负 9.9 → 模型学到的「CTR」≈ 9%   ← 偏高约 9 倍!
   模型没错,它如实学了你喂的分布。错的是你没把分布还原回去。
```

### 5.2 校准公式（必须记住的核心）

设负样本的**保留率**为 `w`（0 < w ≤ 1，正样本全部保留），模型在采样数据上预估出的概率为 `p_s`，要还原成真实概率 `p`：

```
                 p_s
   p  =  ───────────────────
          p_s + (1 - p_s) / w
```

- 当 `w = 1`（不采样），公式退化为 `p = p_s`，无需校准。
- 当 `w = 0.1`（保留 1/10 负样本），把偏高的 `p_s` 压回真实尺度。

这就是经典的 **Facebook 负下采样校准公式**（He et al., 2014，公开文献）。

| 项 | 含义 | 谁负责提供 |
|---|---|---|
| `w` | 负样本保留率 / 采样率 | **平台（必须随样本落盘，否则没法校准）** |
| `p_s` | 采样空间的模型预估 | 模型 |
| `p` | 校准后的真实概率 | 校准模块 |

### 5.3 为什么广告场景里这是「红线」

```
   推荐场景: CTR 偏高 → 排序相对关系可能不变 → 影响较小（但仍影响多样性/融合）
   广告场景: CTR 偏高 → eCPM = pCTR × bid × 1000 → 排序错 + 计费错
             ↓
        pCTR 高估 → 广告主被多扣钱 / 系统超成本 → 投诉 + 资损 + 信任崩塌
```

广告的 **eCPM 排序和扣费都直接依赖 pCTR 的绝对值**（不只是相对大小）。pCTR 校准偏了，等于钱算错了。所以广告系统对校准（calibration / COPC，见 [[评估指标与训练范式名词]]）是**零容忍**的。

> 🎓 **老师视角**:校准这件事最大的坑是**「悄无声息」**。模型 AUC 一点不掉（排序能力没变），离线评估全绿，但 pCTR 的绝对值是歪的。AUC 衡量「排得对不对」，**校准衡量「数值准不准」**——这是两个维度。一个只看 AUC 的团队会完全看不见校准问题，直到广告主拿着账单来吵。平台侧要主动监控 **COPC = Σ预估点击 / Σ真实点击**，理想值≈1，偏离就报警。

### 5.4 一个带数字的校准实算例（务必能口算）

口算一遍,胜过背十遍公式。设真实 CTR ≈ 1%,负样本保留率 `w = 0.1`(保留 1/10 负样本):

```
   真实世界:   正 1, 负 99      → 真实 CTR = 1/100 = 1%
   采样后:     正 1, 负 9.9     → 采样空间「CTR」= 1/10.9 ≈ 9.17%
   模型如实学到 p_s ≈ 0.0917

   代入校准公式还原:
        p = p_s / (p_s + (1 - p_s)/w)
          = 0.0917 / (0.0917 + (1 - 0.0917)/0.1)
          = 0.0917 / (0.0917 + 9.083)
          = 0.0917 / 9.175
          ≈ 0.0100  ✓ 还原回 1%
```

| 采样率 w | 模型预估 p_s | 校准后 p | 不校准的高估倍数 |
|---|---|---|---|
| 1.0（不采样） | 1.0% | 1.0% | 1×（无需校准） |
| 0.5 | ~1.98% | 1.0% | ~2× |
| 0.1 | ~9.17% | 1.0% | ~9× |
| 0.01 | ~50.3% | 1.0% | ~50× |

> 数字为示意,用于建立「采样率越狠、不校准偏得越离谱」的直觉。

> 🎓 **老师视角**:看最后一行——采样率 0.01 时,不校准的 pCTR 高估约 50 倍。**广告系统若拿着这个 50 倍虚高的 pCTR 去算 eCPM 排序和扣费,等于灾难性资损。** 这就是为什么「采样率」必须精确落盘到每条样本:校准公式对 `w` 极度敏感,w 记错 10%,校准就错 10%。

### 5.5 Batch 内负采样的 LogQ 校正

batch 内负采样（把同 batch 其他样本的正例当负例）几乎零成本,召回双塔爱用,但有个隐藏偏差:**热门 item 更容易出现在 batch 里,于是更容易被当成别人的负例 → 被系统性打压**。

```
   朴素 batch 内负采样:  热门 item 当负例的概率 ∝ 它的出现频率
                         → 热门被过度惩罚 → 召回反而推不出热门
   LogQ 校正:  打分时减去 log(采样概率 Q_i)
               s'(i) = s(i) - log(Q_i)
               → 抵消「因为高频被多采为负」的偏差
```

| 方案 | 成本 | 偏差 | 是否需校正 |
|---|---|---|---|
| 独立全局负采样 | 高（额外采负） | 小 | 否 |
| batch 内负采样 | 极低（复用 batch） | 热门被打压 | **是（LogQ）** |
| batch 内 + LogQ 校正 | 极低 | 已修正 | —— |

**平台启示**:用 batch 内负采样时,要么平台维护 item 出现频率 `Q_i` 供 LogQ 校正,要么和算法约定好谁来估 `Q_i`。

### 5.6 公式校准 vs 后验校准

| 校准方式 | 原理 | 优点 | 缺点 | 适用 |
|---|---|---|---|---|
| **公式校准** | 用已知采样率 w 反解 | 简单、确定、零额外数据 | 只修「采样」带来的偏,且依赖 w 准确 | 所有采样场景的第一道 |
| **后验校准（保序回归 Isotonic / Platt）** | 用一份保留的真实分布,学一个单调映射把 p_s 映到真实 p | 能修「非采样原因」的系统性偏（如分布漂移） | 需额外校准集、要定期重训 | 广告强校准、长期漂移 |

> 🎓 **老师视角**:工业级广告系统常常**两者叠加**——先用公式把采样偏差修掉,再用保序回归兜住「模型本身的系统性偏 + 分布漂移」。平台侧要为后验校准准备一份**干净的、未采样的、带真实 label 的校准集**(常来自探索流量或全量回流)。这又一次说明:**纠偏/校准的能力,取决于平台留没留对的数据。**

## 6. 样本偏差体系：四种 bias 的全景地图

采样的偏差能用公式还原，**系统性偏差不能**。下面这张图是整个偏差治理的地图。

```
        全体用户 × 全库 item（真实世界的偏好）
                    │  ① 系统只挑一部分给用户看
                    ▼
            被曝光的 (user, item) 对          ← selection bias 从这里产生
                    │  ② 曝光有位置高低
                    ▼
            带位置的曝光                       ← position bias 从这里产生
                    │  ③ 热门反复曝光,冷门没机会
                    ▼
            实际点击/转化反馈                   ← popularity bias 叠加其中
                    │
                    ▼
            落进样本表 → 训练模型 → 又决定下次曝光  ← 偏差自我强化(反馈回路)
```

| Bias | 一句话 | 后果 | 主流解法 |
|---|---|---|---|
| **Selection bias（选择/曝光偏差）** | 只在「曝光过的」样本上训练，但要服务全库 | 训练分布 ≠ 服务分布（SSB） | ESMM、全空间样本、IPW |
| **Position bias（位置偏差）** | 高位天生高点击，与相关性无关 | 模型把「位置高」误学成「质量高」 | 位置特征、PAL、IPW |
| **Popularity bias（热度偏差）** | 热门 item 曝光多、点击多 | 推荐越推越热、长尾沉没 | 热度纠偏、采样调权、去偏 loss |
| **Conformity/Position 叠加等** | 从众、信任高位、品牌效应 | 反馈被社会因素污染 | 因果建模、debiasing |

> 🎓 **老师视角**:这些 bias 有一个共同的**恐怖属性——自我强化的反馈回路（feedback loop）**。模型偏向热门 → 热门曝光更多 → 样本里热门更多 → 模型更偏向热门……越滚越偏。平台侧能打破这个回路的最有力武器,是**保留「探索流量（exploration）」的样本**——用一小部分随机/打散的曝光收集无偏（或弱偏）样本，给纠偏当「锚」。能在评审里提出「我们要不要留 1% 的随机曝光做无偏样本采集」,是非常加分的平台视角。

## 7. Selection Bias 与 ESMM：只在曝光上训练的原罪

### 7.1 问题:CVR 建模的样本选择偏差（SSB）

转化率 CVR 建模是 selection bias 的重灾区。

```
   想预估的:   p(转化 | 曝光)        —— 面向所有曝光
   传统做法:   在「点击样本」上训练 p(转化 | 点击)
                        ↑
        训练空间 = 点击样本(小、有偏)
        预测空间 = 曝光样本(大、全)
        两个空间不一致 → Sample Selection Bias (SSB)
   外加:  点击样本稀少 → Data Sparsity(数据稀疏)
```

只在「点击了的样本」上学 CVR，但线上要对「所有曝光」预估 CVR——训练和预测的样本空间对不上，这就是 **SSB**。

### 7.2 解法:ESMM（Entire Space Multi-task Model，阿里，公开工作）

ESMM 的核心思想是利用一个恒等式，在**整个曝光空间**上同时建模：

```
   pCTCVR = pCTR × pCVR
   (曝光→转化)  (曝光→点击) (点击→转化)

   ┌─────────────────────────────────────────────┐
   │  在「全部曝光样本」上,同时训练两个任务:       │
   │   - CTR 塔:  学 p(点击|曝光)   ← 有曝光就有label │
   │   - CVR 塔:  学 p(转化|点击)   ← 不单独训,借乘积约束 │
   │  用 pCTCVR 和 pCTR 的 label(全空间可得)来训   │
   └─────────────────────────────────────────────┘
   效果: CVR 塔在「全曝光空间」被间接训练 → 解 SSB + 缓解稀疏
```

> 🎓 **老师视角**:你不需要会推 ESMM 的 loss,但要懂它对**平台的要求**——ESMM 要在**全曝光空间**上训练,意味着样本流不能只给「点击样本」,必须给**完整的曝光样本 + 点击 label + 转化 label**,而且三者要在同一条样本上对齐(见 [[标签生成·归因窗口·延迟反馈]] 的多 label 对齐)。当算法同学说「我要上 ESMM」,你的耳朵里应该立刻响起:「样本空间要从『点击』扩到『曝光』,样本量会涨,多 label 要对齐,延迟反馈窗口要处理」——这就是接得住。

## 8. Position Bias 与纠偏:高位天生高点击

### 8.1 问题

```
   位置:   1     2     3     ...    10
   点击率: 高 ──────────────────── 低
           ↑ 即使把一个烂 item 放第 1 位,它的点击率也会比放第 10 位高
   模型若不知道位置,会把「排第1」的高点击,误学成「这个item质量高」
```

position bias 让模型把**位置带来的点击**错误归因为**item 质量**,形成「强者恒强」的错觉。

### 8.2 三种主流纠偏

| 方法 | 思路 | 训练时 | 预测时 | 平台要求 |
|---|---|---|---|---|
| **位置特征（position as feature）** | 把「第几位」当一个特征喂进去 | 用真实位置 | 位置置默认值(如 0/第1位) | **样本必须落盘真实曝光位置** |
| **PAL（Position-bias Aware Learning，华为公开工作）** | 把 pCTR 拆成 `p(看到|位置) × p(点击|看到)` | 联合训练两项 | 只用 `p(点击|看到)`,抹掉位置项 | 同上 |
| **倾向得分 IPW（Inverse Propensity Weighting）** | 用「位置被点击的倾向」的倒数给样本加权 | 样本按 1/propensity 加权 | 无需位置 | 需估计 propensity、样本可带权 |

```
   位置特征法的关键技巧(务必理解):
   训练: position = 真实位置(第3位就填3)
   预测: position = 固定值(所有候选都当第1位)
        ↓
   这样模型打分时「位置」对所有候选一视同仁,排序只反映 item 本身质量
```

> 🎓 **老师视角**:position bias 纠偏几乎**全部依赖平台留住「真实曝光位置」**。这是个特别容易丢的字段——曝光日志如果只记了「曝光了哪些 item」却没记「它在第几位」,那对不起,position bias 永远没法治。平台侧的铁律:**曝光样本必须落盘真实位置(以及更广义的上下文:第几屏、什么场景、什么坑位)**。一旦丢了,补不回来(历史无法重建当时的位置)。这就是「平台要留住纠偏信息」最典型的例子。

## 9. Popularity Bias 与其它 bias

### 9.1 Popularity bias（热度偏差）

热门 item 因为曝光多,积累的正样本也多,模型学得更准、更敢推,于是更热——长尾 item 被系统性埋没。

```
   头部 item: 曝光多 → 样本多 → 模型推得准 → 更多曝光 (马太效应)
   长尾 item: 曝光少 → 样本少 → 模型没把握 → 更少曝光 (沉默螺旋)
```

**纠偏手段**(算法侧为主,平台侧支撑):

| 手段 | 做法 | 平台支撑 |
|---|---|---|
| 采样调权 | 负采样时按热度调整概率,给冷门 item 更多出场 | 维护 item 热度统计、采样率可配 |
| 去偏 loss | 训练时对热门样本降权(如 logQ 校正) | 样本带热度/频次字段 |
| 探索流量 | 留一部分流量随机/强制曝光长尾 | 保留 exploration 样本标记 |
| 因果去偏 | 把热度建模成混淆因子剥离 | 落盘热度作为特征 |

### 9.2 其它常见 bias（知道名字能接话即可）

- **Conformity bias（从众偏差）**:用户倾向跟随大众/高评分,反馈被社会因素污染。
- **Exposure bias**:与 selection bias 同源,强调「未曝光 ≠ 不喜欢」。
- **Trust bias**:用户信任系统,默认高位更优,与 position bias 交织。

> 🎓 **老师视角**:这些 bias 的治理**主战场在算法侧**,但**弹药库在平台侧**。算法能用什么纠偏方法,取决于平台留没留对应的信息——位置、热度、曝光全集、采样率、探索标记。**你留得越全,算法的纠偏手段越多;你留得越糙,算法只能干瞪眼。** 这句话适合写进你的平台设计文档。

### 9.3 探索与利用（E&E）:打破反馈回路的发动机

前面反复提到「探索流量」。这里展开讲——**探索与利用(Exploration & Exploitation, E&E)** 是从根上对抗反馈回路、冷启动、popularity bias 的机制。

```
   利用(Exploitation): 推模型当前认为最好的 → 短期收益最大,但越推越窄,偏差自我强化
   探索(Exploration):  故意推一些「没把握/没见过」的 → 短期有损,但收集到无偏/弱偏样本
        ↓
   平衡两者,才能既赚当下、又不让模型「近亲繁殖」
```

| 策略 | 一句话 | 平台/样本要求 |
|---|---|---|
| **ε-greedy** | 以小概率 ε 随机推 | 标记哪些是随机曝光 |
| **UCB（置信上界）** | 给「不确定」的 item 加探索奖励 | 维护 item 曝光次数/方差 |
| **Thompson Sampling** | 按后验概率采样推荐 | 维护 item 的后验分布统计 |
| **强制曝光新 item** | 新物料保量曝光破冰 | 标记新 item、保量计数 |

> 🎓 **老师视角**:E&E 收集到的**探索样本是整个偏差治理体系里最宝贵的「无偏锚点」**——它们不受当前模型偏好污染,可以拿来做无偏评估、后验校准、纠偏训练。所以平台侧务必给探索流量打上 `is_exploration` 标记并单独可取。一个成熟平台的标志,就是「能不能随时捞出一批干净的无偏样本」。能在评审里提这点,立刻显出你对「样本代表性」的体系化理解。

### 9.4 实时负采样与流式样本

随着训练从 T+1 跑批走向**流式/在线学习**(见 [[评估指标与训练范式名词]] 的训练范式),负采样也要从「离线批采」变成「**实时流采**」。

```
   离线批采:  样本落数仓后,Spark 批量按 w 下采样负样本 → 简单,但时效差(T+1)
   实时流采:  Flink 消费曝光/点击流,边拼接边按 w 决定负样本去留 → 时效好,但要解决:
              · 采样率在流里如何保证全局一致(不能各分片各采各的)
              · 正样本要等 label(点击/转化),负样本何时「定性」(延迟反馈)
              · 采样率 w 要随流写进每条样本
```

| 维度 | 离线批负采样 | 实时流负采样 |
|---|---|---|
| 时效 | T+1 / 小时级 | 秒级 / 分钟级 |
| 采样一致性 | 易（全局视角） | 难（分布式流，需协调） |
| 与延迟反馈 | 等窗口闭合后采 | 边流边定性，假负风险高 |
| 适配训练 | 批训练 | 流式 / 在线学习 |

> 🎓 **老师视角**:实时负采样最容易踩的坑是**「负样本过早定性」**——流式管线里一条「曝光未点」的样本几秒后就被当负样本采进去训练了,可结果用户 10 分钟后回来点了/转化了。这就是延迟反馈造成的**假负样本**,在实时场景被放大。平台侧要在「时效」和「label 确定性」之间设一个**等待窗口**,或者用「先当负、后回补正」的样本回写机制(见 [[标签生成·归因窗口·延迟反馈]])。

### 9.5 脱敏事故复盘:一次「采样率没落盘」引发的资损

> 以下为脱敏示意案例,用于说明本篇知识点如何在真实事故里串起来。

```
   时间线:
   T0  某广告模型为降本,把负采样率从 0.5 下调到 0.1（合理）
   T1  但采样率是写在「训练脚本的常量」里,没有随样本落盘
   T2  校准模块仍按旧的 w=0.5 校准 → pCTR 系统性高估约 1.8 倍
   T3  eCPM 排序整体抬高,广告超成本投放,广告主侧 CPC 异常上涨
   T4  广告主投诉「最近点击变贵了」,排查 3 天定位到校准 w 不一致
   根因:  采样率是「全局魔法数字」而非「每条样本的属性」,改采样率时校准没同步
   修复:  采样率 w 落盘到每条样本的 sample_weight,校准模块从样本读 w,而非读常量
```

| 这起事故踩中的知识点 | 对应本篇章节 |
|---|---|
| 采样必须配校准 | §5 |
| 采样率必须随样本落盘 | §3.3 / §10 |
| AUC 没掉但概率歪了 | §5.3 |
| 广告校准是红线 | §5.3 |

> 🎓 **老师视角**:这起事故的根因不是「算法不会校准」,而是**「采样率被当成代码常量,而不是数据属性」**——这是个纯粹的**平台/工程设计问题**,正好落在你的职责田里。把「采样率、位置、热度」这类纠偏元信息**当成样本的一等公民字段**来设计,而不是散落在脚本常量里,就是平台研发的专业度所在。

## 10. 平台侧:纠偏需要你「留住」哪些信息

把前面所有解法对平台的要求收口成一张「弹药清单」——这是本篇对平台研发最实操的产出。

| 纠偏 / 校准需求 | 平台必须留住 / 提供的东西 | 丢了的后果 |
|---|---|---|
| 负采样校准 | **每条样本的采样率 w / sample_weight** | 概率没法还原,CTR 系统性偏高 |
| Position bias | **真实曝光位置**(坑位、第几屏、第几位) | 位置偏差永远无法治理(历史不可重建) |
| Selection bias / ESMM | **完整曝光样本 + 多 label(点击/转化)对齐** | ESMM 没法在全空间训练 |
| Popularity bias | **item 热度 / 曝光频次统计** | 热度纠偏、调权无从下手 |
| 反馈回路打破 | **探索流量(exploration)样本标记** | 没有无偏锚点,偏差越滚越大 |
| 全空间召回 | **能采到未曝光全库 item 的样本流** | 召回训练分布与全库不一致 |
| 校准监控 | **真实曝光/点击分布(算 COPC 用)** | 校准漂移看不见 |

```
   平台侧一句话原则:
   ┌────────────────────────────────────────────────┐
   │  采样可以删「样本的量」,但不能删「还原/纠偏需要的元信息」 │
   │  → 位置、采样率、热度、曝光全集、探索标记,一个都不能少    │
   └────────────────────────────────────────────────┘
```

> 🎓 **老师视角**:这里有个反直觉但极重要的点——**纠偏信息的「保留」是不可逆的一次性决策**。你今天没在曝光日志里记位置,明年想治 position bias 时,历史样本是死的、补不回来的。所以平台设计样本 schema 时,要带着「未来可能要纠什么偏」的前瞻性,**宁可多记几个字段(位置/场景/采样率/热度),也不要事后追悔**。存储很便宜,丢失的信息无价。

## 11. 业界怎么做

> 以下均为公开资料/论文中的做法,具体实现各家不同,这里只取「平台研发该知道的轮廓」。

| 公司 / 工作 | 做法要点 | 对平台的启示 |
|---|---|---|
| **Facebook（He et al. 2014）** | GBDT+LR,负下采样 + 概率校准公式 | 采样必配校准,这是工业标配 |
| **Google（Wide&Deep / 召回双塔）** | batch 内负采样 + LogQ 校正(修正热门 item 被过度当负例) | batch 内负采样省成本,但要校正 |
| **阿里（ESMM / ESM²）** | 全空间多任务解 CVR 的 SSB | 样本流要给完整曝光 + 多 label |
| **华为（PAL）** | 位置偏差感知学习,pCTR=p(看到)×p(点击\|看到) | 样本必须落盘真实位置 |
| **YouTube / 召回** | 重要性采样 + 频次惩罚对抗热度偏差 | 维护热度统计、采样可配权 |
| **快手/抖音类信息流（公开分享）** | 保留探索流量收集弱偏样本、实时负采样 | exploration 样本标记 + 实时样本流 |

## 12. 权衡取舍

负采样和纠偏都没有银弹,核心是权衡。

| 取舍点 | 一端 | 另一端 | 怎么定 |
|---|---|---|---|
| 采样率 w | 采得狠:省钱多、信息损失大、校准压力大 | 采得轻:信息全、成本高 | 从 0.1~0.5 起,看 AUC/校准是否退化 |
| easy vs hard negative | easy:稳、信息少 | hard:信息多、易引噪/假负 | 召回多用 hard,但要防假负 |
| 校准方式 | 公式校准:简单、依赖 w 准确 | 后验校准(如保序回归):灵活、需额外数据 | 广告强校准场景常两者叠加 |
| 纠偏强度 | 治得猛:无偏但可能掉短期指标 | 不治:短期好看、长期长尾枯竭 | 看业务是否在意生态/长尾 |
| 探索流量 | 留得多:无偏样本多、短期收入损失 | 留得少:省钱、偏差累积 | 通常留 1%~5% |

> 🎓 **老师视角**:这张表里藏着一个经典的「短期 vs 长期」矛盾——**纠偏和探索,几乎都是「短期掉指标、长期保生态」**。老板看的是这个季度的 CTR/收入,纠偏可能让短期数字难看。所以纠偏方案能不能落地,往往不是技术问题,是**「你能不能把长期价值讲清楚」的沟通问题**(见第 16 节)。

## 13. 常见误区与反直觉

1. **「负采样了,正样本也要采」** ——错。默认**只下采样负样本**,正样本是稀缺信息,基本不动。
2. **「采样不影响 AUC,所以不用管」** ——危险。AUC(排序)确实可能不变,但**绝对概率偏了**,广告计费会错。AUC 绿 ≠ 没问题。
3. **「校准是算法的事,跟平台无关」** ——错。校准公式吃的 `w`(采样率)是**平台落盘**的;没有准确的 w,校准就是错的。
4. **「曝光未点 = 用户不喜欢」** ——不一定。可能没看到(position)、可能稍后才转化(延迟反馈假负)。把假负当 hard negative 会毒化模型。
5. **「位置偏差让算法自己处理」** ——前提是平台**留了真实位置**。没留,神仙也治不了,而且历史补不回来。
6. **「热门推荐效果好,就该多推热门」** ——短期对,长期错。热度偏差会让长尾枯竭、生态恶化、用户疲劳。
7. **「batch 内负采样免费又好用」** ——有隐藏代价:热门 item 容易被反复当负例,需要 LogQ 校正,否则热门被打压过度。
8. **「采样率拍个 0.1 就行」** ——采样率要随业务、随样本量调,且**必须记录到每条样本**,不能是个全局魔法数字拍脑袋。

## 14. 平台侧落地 Checklist

**样本 schema 层(最关键,不可逆)**
- [ ] 每条样本落盘 **采样率 / sample_weight**
- [ ] 曝光样本落盘 **真实位置 / 坑位 / 第几屏 / 场景**
- [ ] 落盘 **采样策略 ID**(可回溯用了哪种采法)
- [ ] 支持 **多 label 对齐**(曝光/点击/转化在同一条样本上)
- [ ] 落盘 **item 热度 / 频次** 字段(供热度纠偏)
- [ ] 标记 **探索流量(exploration)** 样本

**采样能力层**
- [ ] 采样率**可配置、可灰度、可按策略区分**(召回/精排不同)
- [ ] 支持随机/曝光/batch内/流行度/hard 等多种采法
- [ ] 召回样本流能采到**未曝光全库 item**
- [ ] 采样过程**可观测**(实际采样比、正负比监控)

**校准与监控层**
- [ ] 提供 **校准所需的真实分布口径**(算 COPC)
- [ ] 监控 **COPC = Σ预估 / Σ真实**,偏离 1 报警
- [ ] 监控**正负比、采样率漂移**
- [ ] 离线/在线**采样口径一致**(防训练-服务偏差,见 [[评估指标与训练范式名词]])

## 15. 对接话术:算法/训练同学怎么说、你怎么接

| 他们说 | 真实意图 | 你怎么接(显专业) |
|---|---|---|
| 「负样本太多了,采一下吧」 | 想省训练成本 | 「可以,采样率我配成可灰度的;**采样率我会落到每条样本上**,你那边校准公式直接读 sample_weight 就行,不用猜。」 |
| 「我要上 ESMM」 | 要解 CVR 的样本选择偏差 | 「那样本空间要从点击扩到**全曝光**,我得把曝光样本 + 点击/转化多 label 对齐供给,样本量会涨 X 倍,**延迟反馈窗口**也要一起定。」 |
| 「这个 item 明明很好怎么排不上去」 | 怀疑 position bias / 冷启动 | 「先确认我们样本里**有没有落真实位置**——如果有,可以加位置特征/PAL 纠偏;预测时位置统一置第一位。」 |
| 「线上 pCTR 偏高」 | 校准出问题 | 「先查采样率 w 和校准是否一致,我拉一下 **COPC** 看偏多少;大概率是采样后没校准或 w 记错了。」 |
| 「召回负样本怎么构造」 | 要全局负例 | 「召回得用**全库负采样**,不能只用曝光未点——我给你一条能采到未曝光 item 的样本流,easy/hard 比例可配。」 |
| 「曝光没点的都当负例?」 | 担心假负 | 「不能无脑当负——**还在归因窗口里的**先别盖棺,延迟转化会变成假负,我这边按窗口过滤后再给你。」 |

> 🎓 **老师视角**:接话的黄金句式是 **「可以,我会把 X 落到每条样本上,你直接读就行」**。算法同学最怕的是「平台给的样本里缺了关键信息、要自己再去捞」。你每次都能主动把「采样率/位置/热度」这些纠偏弹药备齐,他们就会越来越离不开你——这就是从「工具人」到「不可替代」的路径(见 [[00-总览与学习地图]] 的能力分层)。

## 16. 向上沟通:怎么把采样与偏差讲给老板听

老板不关心 IPW 公式,关心**钱、风险、生态**。翻译对照表:

| 技术语言 | 翻译给老板的话 |
|---|---|
| 负采样把样本从 100 亿降到 15 亿 | 「训练成本降了约 X 成,模型迭代速度快了一倍,效果不掉。」 |
| 采样后没校准,pCTR 偏高 | 「如果不修,广告 CTR 会算高,等于**给广告主多扣钱,有资损和投诉风险**。」 |
| 没留位置信息 | 「我们现在治不了位置偏差,而且**历史数据补不回来**——这是个越拖越贵的技术债,建议现在就把位置落盘。」 |
| 热度偏差 / 长尾枯竭 | 「系统会越推越热门,长尾内容沉没,**长期会让用户疲劳、生态单一、留存下滑**。我们用一点探索流量做对冲。」 |
| 探索流量损失短期收入 | 「拿出约 1%~3% 流量做探索,短期让出一点点收入,**换来无偏样本和长期模型上限**,是花小钱防大病。」 |

> 🎓 **老师视角**:向上沟通的最高境界,是把**「不可逆的技术债」前置暴露**。「位置信息现在不记,以后永远补不回来」这种话,老板一听就懂利害,也最能体现你的前瞻性。把偏差治理包装成**「现在花小钱,避免未来的大坑/资损/生态崩」**,比讲一堆 bias 术语有用一百倍。

## 17. 一图总结 + 一句话速记

```
                    搜广推样本的两宗罪
        ┌──────────────────────┬──────────────────────┐
        │   罪一:又多又贵(量)   │   罪二:又歪又偏(质)   │
        │   → 负采样             │   → 偏差治理           │
        ├──────────────────────┼──────────────────────┤
        │ 方法:随机/曝光/batch内 │ selection→ESMM(全空间) │
        │      /流行度/hard/全局 │ position →位置特征/PAL  │
        │ 召回:全库负采样        │ popularity→热度纠偏/探索│
        │ 精排:曝光未点          │ 反馈回路 →探索流量打破  │
        ├──────────────────────┼──────────────────────┤
        │ ⚠ 采样必配校准:        │ ⚠ 纠偏靠平台留信息:    │
        │   p = p_s/(p_s+(1-p_s)/w)│  位置/采样率/热度/全集 │
        │   不校准→CTR偏高→资损   │  丢了→历史补不回来      │
        └──────────────────────┴──────────────────────┘
                    平台侧:删得了样本的量,删不得纠偏的元信息
```

> **一句话速记:采样动「量」必校准,偏差治「质」靠留痕——位置、采样率、热度、全集、探索标记,一个都不能丢。**

## 18. 自测三问

1. **召回和精排的负样本来源为什么不同?**
   > 参考:召回面向**全库**做检索,线上会遇到大量没曝光过的 item,所以负例必须来自**全库随机/全局负采样**;精排只在**曝光集**里排序,用**曝光未点击**作负例最贴近线上分布。平台因此要供给两条不同样本流。

2. **负采样后为什么必须校准?给出校准公式,并说明哪一项是平台负责提供的。**
   > 参考:采样抬高了正负比,模型预估概率系统性偏高。校准公式 `p = p_s / (p_s + (1-p_s)/w)`,其中 `w` 是**负样本保留率/采样率,由平台随样本落盘提供**;不校准在广告里会导致 pCTR 高估、计费错、资损。

3. **要治理 position bias,平台必须提前做什么?为什么说这个决策不可逆?**
   > 参考:平台必须在**曝光样本里落盘真实位置**(坑位/第几位/第几屏)。一旦当时没记,历史曝光的位置**无法重建**,position bias 永远没法对历史样本纠偏——所以这是个一次性、不可逆的 schema 决策,宁可多记不可漏记。
