## 0. 30 秒速通(TL;DR)

一句话:**样本 = 特征(请求时刻的状态) + 标签(事后用户的反馈),靠一个 ID 把这两件事跨时间拼起来。**

- **样本不是"采集"出来的,是"拼接"出来的。** 没有任何一条日志天生带标签。曝光日志记录"我们给用户看了什么",反馈日志(点击/转化)记录"用户做了什么",两者发生在不同时刻、由不同的点上报,平台要靠 `request_id` / `trace_id` 把它们 join 到一起。
- **曝光是骨架,反馈是补丁。** 拼接永远以曝光为左表 `LEFT JOIN` 反馈:能 join 上的是正样本,join 不上的(在归因窗口内没反馈)就是负样本。**所以负样本是"拼不出反馈"自然产生的,不是单独采集的。**
- **难点全在时间。** 反馈天然晚于曝光(用户先看后点,先点后买),而转化可能晚几小时甚至几天。于是出现乱序、迟到(late arrival)、丢失、重复四大顽疾,平台要靠归因窗口、等待窗口、去重键、补数机制来兜。
- **两条主流水线:** 批拼接(Spark T+1,简单稳但慢一天)和流拼接(Flink 实时,新鲜但要扛大状态和等待窗口)。绝大多数成熟平台是**双轨并行**:流拼接出实时样本,批拼接当天回灌做对账兜底。
- **平台研发的核心交付:** 全链路埋点规范(ID 透传)、拼接引擎(批/流)、去重与膨胀控制、样本表 schema 与落湖(Paimon 主键表天然适配去重和补数)、样本质量监控(正负比/量级/join 命中率)。
- **量级感**(经验值/典型量级,非精确):头部信息流/电商推荐场景,**日曝光百亿级**,点击约为曝光的个位数百分比,转化又比点击低一两个数量级。样本表一天几十 TB 是常态。

## 1. 一条样本的一生:从请求到训练

### 1.1 全景时序图

先建立全局视角。一条样本从用户发起请求到最终喂进训练,要穿过在线服务、日志采集、拼接、打标、落表五个阶段:

```
时刻 t0          t0+几十ms      t0+几秒/分钟        t0+几分钟/小时/天      T+ε / T+1
 │                 │                  │                     │                    │
 用户请求 ──> 在线推理打分 ──> 渲染曝光 ──> 用户点击/转化 ──> 日志落地 ──> 拼接join ──> 打标签 ──> 样本表 ──> 训练
 │                 │                  │                     │                    │
[request_id]   [特征快照]         [impression]          [click/conv]        [join + label]

────────────────────────────────────────────────────────────────────────────────────────
 在线侧(算法服务/推荐引擎)              客户端/服务端埋点          平台侧(数据/样本流水线)
```

关键认知:**特征在 t0 这一刻被"冻结",标签在 t0 之后才发生。** 这两件事天生分离,而把它们重新缝合的过程,就是整篇文档的主题。

### 1.2 逐阶段拆解

| 阶段 | 发生在哪 | 产出什么 | 关键 ID | 平台研发关注点 |
|------|---------|---------|---------|--------------|
| 请求 | 在线推荐/搜索服务 | 一次请求上下文(user, scene, time) | 生成 `request_id` | ID 生成规范、全局唯一、可透传 |
| 推理打分 | 模型在线服务 | 候选 item 的预估分、排序结果 | request_id 透传 | **特征快照落地**(线上用的就是这份) |
| 渲染曝光 | 客户端/前端 | 实际展示给用户的 item 列表 | request_id + item_id | 曝光埋点,区分"下发"与"真曝光" |
| 用户反馈 | 客户端/服务端 | 点击、停留、转化事件 | request_id + item_id | 反馈埋点,事件时间 event_time |
| 日志落地 | 采集管道(Kafka 等) | 三类日志流 | 同上 | 不丢不重、时间戳准确 |
| 拼接 join | 平台拼接引擎 | 对齐后的"曝光+反馈"宽行 | join key | 批/流引擎、乱序迟到处理 |
| 打标签 | 拼接后处理 | label(0/1 或多目标) | — | 归因逻辑、归因窗口 |
| 落样本表 | 数据湖/存储 | 训练样本表 | 主键 | schema、格式、去重、补数 |
| 喂训练 | 训练框架 | 一行行特征+标签 | — | 读取性能、离在线一致性 |

> 🎓 **老师视角:为什么特征必须在请求时刻"冻结"?**
> 这是整条流水线最容易被忽视、却最致命的点。模型上线时用的特征是 t0 时刻的用户状态(比如"过去 7 天点击数=15");但如果你训练时拿 T+1 离线数仓重新算这个特征,算出来的可能是 t0 之后又变化过的值(变成 18 了)。这叫**特征穿越 / 离在线不一致**,会让线下 AUC 虚高、上线就崩。
> 所以成熟做法是:**在线打分时把当时用到的特征值原样落日志(feature snapshot / feature dump),拼接时直接用这份快照,而不是离线重算。** 平台研发要把"特征快照落地"做成在线服务的标配能力。详见 [样本离在线一致性](../04-样本数据管理/样本离在线一致性.md) 和 [特征平台(Feature Store)架构](../03-特征数据管理/02-特征平台(Feature Store)架构.md)。

### 1.3 平台研发要交付什么

- **统一 ID 体系**:`request_id`(一次请求)、`trace_id`(全链路追踪)、`item_id`、`user_id` 的生成与透传规范。这是后续一切 join 的地基。
- **特征快照通道**:在线服务把打分时刻的特征值打到日志里,平台提供 SDK / 旁路收集。
- **埋点 SDK 与规范**:让客户端/服务端按统一格式上报,时间戳、ID 字段不能各写各的。
- **一条端到端可观测的链路**:从 request_id 能追到这条请求最终进了哪张样本表的哪一行。

### 1.4 一个脱敏的端到端追踪案例

把抽象的"一生"落到一个具体的请求上(数据已脱敏,数值为示意):

```
[t0=10:00:00.000] 用户 U_8821 打开信息流首页
   在线服务生成 request_id = R_20260614_100000_8821ab
   召回 200 → 粗排 50 → 精排 10,精排时记录 10 个 item 的特征快照(每个 ~3KB)

[t0+12ms] 服务端下发 10 个 item(下发日志: 10 行)
[t0+800ms ~ t0+30s] 用户滑动,真正进入可视区域的有 6 个(真曝光日志: 6 行,含特征快照)
[t0+8s]   用户点击 item_C(点击日志: 1 行)
[t0+2h]   用户在商详页下单 item_C ¥99(转化日志: 1 行,可能不含 request_id)

→ 拼接产出(骨架取真曝光 6 行):
   R_..8821ab, item_A, feat..., click=0, conv=0   (负)
   R_..8821ab, item_B, feat..., click=0, conv=0   (负)
   R_..8821ab, item_C, feat..., click=1, conv=1   (正)  ← 点击精确join,转化靠 user+item+窗口归因回来
   R_..8821ab, item_D, feat..., click=0, conv=0   (负)
   R_..8821ab, item_E, feat..., click=0, conv=0   (负)
   R_..8821ab, item_F, feat..., click=0, conv=0   (负)
   (那 4 个下发了但没真曝光的 item:不进样本,否则就是假负样本)
```

这个案例里藏着本篇所有要点:真曝光 vs 下发(6 vs 10)、特征快照、点击精确 join、转化模糊归因、负样本的被动产生。后面每一节都会回到它。

> 🎓 **老师视角:request_id 的"基数(cardinality)"决定了一切系统的压力。**
> 日百亿曝光意味着每天有几十亿个不同的 request_id 在系统里流动。这个超高基数决定了:join key 的索引/分区策略、流拼接状态后端的 key 数量、去重表的主键规模、监控埋点的采样策略。平台研发做任何设计前,先问自己"这个东西要不要按 request_id 粒度存全量?"——如果要,先算清楚基数 × 单条大小,很多方案在这一步就被毙掉了。

## 2. 三类日志:曝光 / 点击 / 转化

样本拼接的原料是三类(信息流场景常见)或更多类(电商/广告会更细)行为日志。理解它们的字段、量级、产生时机、由谁上报,是设计拼接的前提。

### 2.1 三类日志对比

| 维度 | 曝光日志 impression | 点击日志 click | 转化日志 conversion |
|------|--------------------|---------------|---------------------|
| 语义 | 给用户看了什么 | 用户点了什么 | 用户产生了价值行为(下单/付费/激活) |
| 谁打的点 | 客户端为主(真曝光),部分服务端(下发) | 客户端 | 服务端为主(交易系统/业务后端) |
| 产生时机 | t0 渲染瞬间 | t0 之后秒~分钟级 | t0 之后分钟~天级 |
| 是否做骨架 | **是(左表)** | 否(右表) | 否(右表) |
| 典型量级(经验值) | 日百亿级 | 曝光的个位数百分比 | 点击的百分之几 ~ 个位数 |
| 关键字段 | request_id, item_id, user_id, scene, pos, event_time, **特征快照** | request_id, item_id, event_time, 点击位置 | request_id/订单ID, item_id, event_time, 金额, 转化类型 |
| 上报可靠性 | 较高但有丢失 | 中 | 较高(交易不能丢) |
| 延迟分布 | 几乎实时 | 短尾 | **长尾(可能几天)** |

> 量级再强调一次:这些是**典型量级/经验值**,不同业务差异极大。请勿当成精确数字对外引用。

### 2.2 字段设计的隐藏坑

**(1)"下发"不等于"真曝光"。** 服务端把 10 个 item 推给客户端,但用户只滑到了 6 个——后 4 个根本没进入屏幕。用"下发日志"当曝光会引入大量假负样本(模型以为"展示了没点",其实压根没展示)。**真曝光需要客户端在 item 进入可视区域时才上报。** 平台研发要明确:样本的骨架用真曝光还是下发,这个口径决定了样本性质。

**(2)event_time 与 log_time 是两个时间。** `event_time` 是事件真实发生的时刻(客户端打的),`log_time` / `process_time` 是日志到达管道/被处理的时刻。乱序和迟到的所有讨论,都建立在"以 event_time 为准,但只能在 log_time 看到它"这个矛盾上。**埋点必须带 event_time,且时钟要可信(客户端时间会被用户篡改,需校正)。**

**(3)曝光日志要带特征快照。** 见 1.2 老师视角。这一份快照让曝光日志通常是三类里最"胖"的,单条几 KB 到几十 KB,直接决定了样本表的体量。

```
                  一次请求的日志产生关系(一对多)
 request_id = R123
   │
   ├── impression: (R123, itemA, pos1, t0,   [feat_snapshot])   ← 骨架
   ├── impression: (R123, itemB, pos2, t0,   [feat_snapshot])
   ├── impression: (R123, itemC, pos3, t0,   [feat_snapshot])
   │
   ├── click:      (R123, itemB, t0+8s)        ← 只点了 B
   │
   └── conversion: (R123, itemB, t0+2h, ¥99)   ← 2 小时后买了 B
```

> 🎓 **老师视角:为什么转化日志"谁打的点"如此重要?**
> 点击是 UI 行为,客户端能打。但转化是**业务/资金行为**,通常发生在交易系统、支付系统,跟推荐请求隔了好几个系统。这就带来两个问题:① 转化日志里**可能没有 request_id**(下单时用户早就离开推荐页了),怎么归因回当初那次曝光?这是「归因」的核心难题。② 转化天然延迟大、长尾,引出「延迟反馈(delayed feedback)」问题。这两个都是大坑,本篇先点到,深入见 [标签生成·归因窗口·延迟反馈](../04-样本数据管理/02-标签生成·归因窗口·延迟反馈.md)。

### 2.3 一份最小可用的曝光日志字段规范(示意)

平台研发要给客户端/服务端一份强约束的埋点 schema。下面是曝光日志的最小字段集(实际会更多):

| 字段 | 类型 | 必填 | 说明 / 坑点 |
|------|------|------|-----------|
| request_id | string | 是 | 全局唯一,join 的命根子 |
| item_id | string | 是 | 配合 request_id 组成样本主键 |
| user_id | string | 是 | 转化模糊归因要用 |
| scene / channel | string | 是 | 区分场景,不同场景样本不能混训 |
| pos | int | 是 | 曝光位置,用于位置偏差纠正 |
| event_time | long(ms) | 是 | **事件真实时间**,以它为准做拼接 |
| client_ts / server_ts | long | 是 | 客户端/服务端各自时钟,用于校正 |
| is_visible | bool | 是 | 是否真曝光(进入可视区) |
| stay_time | long | 否 | 停留时长,可二次校正真曝光 |
| event_uuid | string | 是 | 客户端生成,**幂等去重键** |
| feature_snapshot | bytes/struct | 是 | 打分时刻特征快照(最胖的字段) |
| model_version | string | 否 | 哪个模型版本打的分,便于回溯 |
| sample_rate | float | 否 | 若上报侧已采样,记录采样率 |

> 🎓 **老师视角:埋点字段是"一次定义,长期负债"。**
> 客户端埋点改一次要发版、灰度、等用户升级,周期以周计,且老版本 App 永远在产生老格式日志。这意味着:① 埋点 schema 要**一步到位地预留扩展位**(比如一个 ext map 字段),别指望频繁改;② 平台拼接侧必须**兼容多版本日志格式**(老版本没 is_visible 字段怎么办?要有 fallback 规则);③ 任何字段含义变更都是灾难——同一个字段在 v1 和 v2 含义不同,会静默污染样本。**平台研发要把"日志 schema 版本"当成一等公民来管理,每条日志带 schema_version。**

### 2.4 平台研发要支撑什么

- **三类日志的统一接入**:Kafka topic / 数仓 ODS 层规范,字段标准化(schema registry)。
- **真曝光判定能力**:推动客户端上报可视曝光,或在平台侧用停留/滑动事件二次校正。
- **时间字段治理**:强制带 event_time,提供客户端时钟校正方案。
- **转化日志回流通道**:打通交易系统,把业务转化事件接入样本管道,并尽量带上可归因的 ID。
- **多版本日志兼容**:每条日志带 schema_version,拼接侧对老版本有 fallback 规则。

## 3. 日志拼接 join:把碎片对齐成样本

拼接(join / attribution join)是把分散的三类日志按 ID 对齐,缝合成"一行一个曝光、附带它的反馈与标签"的过程。

### 3.1 join 的基本范式:曝光为骨架,左连反馈

```
  impression (左表, 骨架)        click (右表)            conversion (右表)
  ┌──────────────────────┐     ┌──────────────┐       ┌────────────────┐
  │ R123, itemA, feat...  │     │ R123, itemB  │       │ R123, itemB,¥99│
  │ R123, itemB, feat...  │     └──────────────┘       └────────────────┘
  │ R123, itemC, feat...  │
  └──────────┬───────────┘
             │  LEFT JOIN ON (request_id, item_id)
             ▼
  ┌───────────────────────────────────────────────────────────┐
  │ R123, itemA, feat..., is_click=0, is_conv=0   ← 负样本     │
  │ R123, itemB, feat..., is_click=1, is_conv=1   ← 正样本     │
  │ R123, itemC, feat..., is_click=0, is_conv=0   ← 负样本     │
  └───────────────────────────────────────────────────────────┘
```

核心三条原则:

1. **以曝光为左表(骨架)。** 每一条曝光都必须产出一行样本——这是"我们做了一次展示决策"的记录,无论用户反馈与否。
2. **LEFT JOIN 反馈。** 能 join 上反馈的,label 置 1(正样本);join 不上的,label 置 0(负样本)。
3. **负样本是被动产生的。** 你不"采集"负样本,你只是把"在归因窗口内没等到反馈的曝光"标成负样本。这点想通了,负采样、样本偏差才好理解(见 [负采样与样本偏差治理](../04-样本数据管理/03-负采样与样本偏差治理.md))。

### 3.2 join key 怎么设计

join key 不是随便选 `request_id` 就完事,粒度要和样本粒度对齐:

| 样本粒度 | join key | 适用场景 |
|---------|---------|---------|
| 一次曝光一个 item | (request_id, item_id) | 信息流单 item 排序最常见 |
| 一次请求(整刷) | request_id | listwise / 整页优化 |
| 会话级 | session_id | 序列推荐、会话内行为建模 |
| 跨会话归因 | user_id + 时间窗 + item_id | 转化归因(转化时已无 request_id) |

> 🎓 **老师视角:为什么转化 join 经常退化成"模糊匹配"?**
> 点击 join 一般很干净:`(request_id, item_id)` 严丝合缝。但转化发生时往往拿不到 request_id,只能靠 `(user_id, item_id, 转化时间在曝光后的归因窗口内)` 去"找回"当初那次曝光。如果一个用户在窗口内对同一 item 有多次曝光,到底归因给哪一次?这就是归因模型(末次归因 / 首次归因 / 线性归因)的活儿。**平台研发要意识到:点击 join 是精确匹配,转化 join 是带归因策略的近似匹配,二者的引擎逻辑和容错完全不同。**

### 3.3 多目标:一条样本多个 label

现代推荐是多目标的:同一条曝光要同时拼 `is_click`、`is_like`、`is_finish`、`is_conv`、`stay_time`……所以拼接产出的不是单个 0/1,而是一组标签向量。

```
 样本宽行 = [ request_id, item_id, feature_snapshot(几百维) | label_click, label_like, label_finish, label_conv, stay_time, ... ]
            └──────────── 特征(t0 冻结)──────────┘ └────────── 多个标签(事后拼来)────────┘
```

平台研发要把"标签"做成**可扩展、可配置**的:算法新增一个目标(比如"是否分享"),不应该要改拼接引擎代码,而应是配置一类新反馈日志接进来。

### 3.4 拼接的伪 SQL:让概念落到可执行

把"曝光左连反馈 + 归因窗口"写成伪 SQL,算法和数据同学一看就懂(以点击为例):

```sql
-- 骨架:真曝光
SELECT
    imp.request_id,
    imp.item_id,
    imp.feature_snapshot,
    imp.pos,
    imp.event_time            AS imp_time,
    -- 点击精确 join:同 request_id + item_id,且点击在曝光之后、窗口内
    IF(clk.request_id IS NOT NULL, 1, 0)  AS label_click,
    -- 转化模糊归因:无 request_id,靠 user+item+时间窗
    IF(cvr.user_id IS NOT NULL, 1, 0)     AS label_conv
FROM impression imp
LEFT JOIN click clk
    ON  imp.request_id = clk.request_id
    AND imp.item_id    = clk.item_id
    AND clk.event_time BETWEEN imp.event_time AND imp.event_time + INTERVAL '30' MINUTE
LEFT JOIN conversion cvr
    ON  imp.user_id = cvr.user_id
    AND imp.item_id = cvr.item_id
    AND cvr.event_time BETWEEN imp.event_time AND imp.event_time + INTERVAL '7' DAY
WHERE imp.is_visible = TRUE          -- 只用真曝光做骨架
```

注意三处区别:① 点击 join 用 `request_id+item_id` 精确匹配,转化 join 退化成 `user_id+item_id` 近似;② 两者归因窗口不同(点击 30 分钟,转化 7 天);③ 骨架 `WHERE is_visible = TRUE` 过滤掉非真曝光。**这段伪 SQL 是批拼接的骨干,流拼接是它的"增量+带状态"版本。**

### 3.5 平台研发要交付什么

- **可配置的拼接 DSL / 规则引擎**:声明式描述"用哪些日志、按什么 key、什么窗口、产出哪些 label",避免每个业务写一套 hardcode。
- **点击 join 与转化 join 的不同实现**:前者精确,后者带归因策略。
- **label 可插拔**:新增反馈类型 = 加配置,不改引擎。

## 4. 拼接的工程难题:乱序 / 迟到 / 丢失 / 重复

这一节是平台研发真正要扛的地方。理想里 join 很简单,现实里时间和分布式把它撕碎。

### 4.1 四大顽疾总览

```
理想:     impression ──> click ──> conversion   (有序、不丢、不重、即时到达)

现实:
  乱序 ── click 的 log_time 早于 impression 的 log_time(网络/缓冲导致)
  迟到 ── conversion 在曝光后 3 天才到(用户拖延下单)
  丢失 ── 客户端崩溃/弱网,曝光或点击根本没上报
  重复 ── 客户端重试 / 管道 at-least-once,同一事件上报多次
```

| 顽疾 | 成因 | 后果 | 平台对策 |
|------|------|------|---------|
| 乱序 out-of-order | 网络抖动、客户端缓冲批量上报、多管道 | 点击先于曝光到达,按 process_time 处理会漏拼 | **以 event_time 为准 + 等待窗口 / watermark** |
| 迟到 late arrival | 转化长尾、客户端离线缓存 | 样本生成时标签还没到,正样本被误标成负 | **归因窗口 + 延迟反馈处理 + 补数回灌** |
| 丢失 loss | 弱网、崩溃、采样丢弃 | 假负样本(点了但没上报)/ 骨架缺失 | **多通道冗余、对账、可接受的丢失率监控** |
| 重复 duplicate | at-least-once 投递、客户端重试 | 同一曝光多行、点击被重复计数 | **幂等去重键 + 主键表 upsert** |

### 4.2 乱序:为什么不能按"到达时间"拼

```
事件真实顺序(event_time):  impression(t0) ──> click(t0+5s)
到达拼接引擎顺序(log_time): click 先到 ────────> impression 后到(网络慢)

如果按 log_time 处理:click 到的时候找不到 impression → 漏拼 → 这条点击丢失 → 假负样本
正确做法:按 event_time 对齐,引擎要"等一会儿"让晚到的曝光也进来
```

这就是**等待窗口(wait window)** 和 **watermark** 的来由:流拼接不能一来就处理,要给乱序留缓冲。等多久?等得久样本新鲜度差、状态大;等得短漏拼多。这是个典型权衡(见第 5 节)。

### 4.3 迟到:延迟反馈的经典困境

转化可能 t0+3 天才发生。但训练等不了 3 天。于是:

- **方案 A(等满窗口再出样本)**:等归因窗口关闭(如 7 天)再生成样本,标签准但样本延迟 7 天,模型学的是"一周前的世界"。
- **方案 B(先出负样本,后补正)**:曝光后先按"负样本"出,等转化真来了再"翻正"(回灌 / fake negative correction)。样本新鲜,但需要修正机制,且短期内正样本被低估。

> 🎓 **老师视角:延迟反馈是"业务越成功越疼"的问题。**
> 转化窗口越长(比如游戏付费、电商大件),延迟反馈越严重。方案 B(先负后补)是工业界主流,因为它兼顾新鲜度,但它要求**样本表能被"事后修改"**——同一个 (request_id, item_id) 先写了 label=0,3 天后要 update 成 label=1。**这正是数据湖主键表(如 Apache Paimon 主键表 + Changelog)的杀手级场景**:用主键 upsert 天然支持"先写后改",还能产出 changelog 让下游增量感知标签翻转。用纯 append 的 Hive 表做这事会非常痛苦。展开见第 7 节和 [02-标签生成·归因窗口·延迟反馈](../04-样本数据管理/02-标签生成·归因窗口·延迟反馈.md)。

### 4.4 丢失:用对账逼近真相

丢失无法消灭,只能监控和逼近。平台研发要建**三类日志的量级对账**:

```
 健康度对账(每小时/每天):
   曝光数      ≈ 在线服务下发数 × 真曝光率         (偏离 → 曝光埋点丢了)
   点击数/曝光数 ≈ 历史 CTR 均值 ± 容忍带          (突降 → 点击埋点丢了)
   join 命中率  = 能拼上骨架的反馈数 / 总反馈数      (突降 → 骨架缺失或 key 错)
```

任何一个指标跳水,样本质量就出问题。这套对账是样本流水线的"心电图"。

### 4.5 重复:幂等去重键

重复来自 at-least-once 语义和客户端重试。对策:

- 为每条日志设计**幂等键**:曝光用 `(request_id, item_id, pos)`,点击用 `(request_id, item_id, event_time 取整)` 或客户端生成的 `event_uuid`。
- 落表时用**主键 upsert**(同键覆盖)而非 append,天然去重。
- 流拼接里用状态去重(记住已处理的 event_uuid,但要配 TTL,否则状态无限膨胀)。

### 4.6 一个迟到翻正的脱敏时序案例

```
[Day1 10:00] 曝光 (R_x, item_C, feat...)            → 流拼接出样本: label_conv = 0  (先按负写)
[Day1 10:00 + 30min] 等待窗口关闭,样本落表           → 样本表: (R_x, item_C) conv=0
[Day3 15:00] 用户下单 item_C ¥99                     → 转化迟到 2 天,流窗口早关
                                                       走旁路 → 触发翻正
[Day3 15:05] upsert (R_x, item_C) conv=1, amount=99  → 主键覆盖,changelog 记录 0→1
[Day4 凌晨] T+1 批拼接全量重拼 Day1 数据              → 复算确认 conv=1,与流翻正对账一致
```

时间线把"先负后翻正"讲透了:样本在 Day1 就新鲜可用(label=0),Day3 转化到了再 upsert 翻正,Day4 批拼接做最终对账。**如果没有主键可更新存储,Day3 这一步要么重刷整个 Day1 分区,要么靠下游"取最新"的脏逻辑——都很痛苦。**

> 🎓 **老师视角:翻正窗口和"训练消费时机"要算清楚。**
> 假设转化归因窗口 7 天,那么 Day1 的样本要到 Day8 才算"标签完全稳定"。但训练等不了 7 天。所以现实是:训练会消费"还在翻正中"的样本,模型短期内看到的正样本是偏少的(延迟反馈偏差)。平台研发要给算法**两个口径**:① 实时口径(新鲜但标签未稳);② T+7 稳定口径(标签准但滞后)。算法按模型策略选,或两者结合做延迟反馈建模。这个"口径选择权"必须交给算法,平台只提供能力。

### 4.7 平台研发要交付什么

- **以 event_time 为基准的拼接语义**(不是 process_time)。
- **可配置的等待窗口与归因窗口**(不同业务不同值)。
- **延迟反馈的回灌/翻正机制**(配合主键表)。
- **三类日志对账与 join 命中率监控**(心电图)。
- **幂等去重**(去重键 + 主键 upsert + 状态 TTL)。

### 4.8 事故复盘:一次乱序+丢失导致的样本错位(脱敏)

真实世界里,样本问题往往不是单点失效,而是几个顽疾叠加。下面是一个脱敏复盘,平台研发应该把这类复盘当成"压力测试用例"。

**背景**:某信息流场景,流拼接等待窗口设为 10 分钟,以 process_time 近似当 event_time(为省事,没严格用 event_time)。某日上游 Kafka 一个分区积压。

**现象**:当天模型线下 AUC 正常,但上线后 CTR 预估系统性偏低,线上点击率掉了约 8%(经验量级,示意)。

**排查时间线**:
```
[发现] 算法报"线上 CTR 比预估低很多" → 怀疑特征穿越 → 查特征快照,正常
[定位1] 看样本质量大盘:当天 click_join_rate 从基线 ~95% 跌到 ~78%  ← 心电图报警(但当时阈值设太松,没自动告警)
[定位2] 看 Kafka:点击流某分区积压 ~25 分钟,远超 10 分钟等待窗口
[根因] 积压的点击到达时,对应曝光的状态已因窗口到期被清理 → 点击 join 不上
        → 这些"其实点了"的曝光被错误标成负样本(假负样本)
        → 假负样本拉低了正样本占比 → 模型学得"大家都不点" → 线上预估偏低
[叠加] 因为用 process_time 近似 event_time,积压让乱序雪上加霜,正常该拼上的也漏了
```

**根因链**:① 等待窗口(10min)< 实际乱序延迟(25min);② 用 process_time 而非 event_time,放大了积压的影响;③ join 命中率监控阈值过松,没能第一时间告警。三个问题叠加,产出了一批"假负样本",污染了当天全量样本。

**修复**:
- 短期:用 T+1 批拼接重拼当天数据(批天然不受积压影响,数据都到齐了),覆盖污染的实时样本,重训。
- 中期:拼接严格改用 event_time + watermark,等待窗口按实测乱序分布的 P99 设定(而非拍脑袋 10min)。
- 长期:join 命中率监控改为"同比基线 ± 容忍带"动态阈值,跌破即自动熔断当天实时样本、回退到批口径。

| 复盘维度 | 当时的错 | 改正后 |
|---------|---------|--------|
| 时间语义 | process_time 近似 | 严格 event_time + watermark |
| 等待窗口 | 拍脑袋 10min | 按乱序 P99 实测设定 |
| 监控阈值 | 静态、过松 | 动态基线 + 自动熔断 |
| 兜底 | 无 | 批拼接随时可覆盖实时样本 |

> 🎓 **老师视角:样本问题的"延迟暴露"特性,是平台研发最该警惕的。**
> 这个事故最坑的地方是:**样本错了,但训练不报错、线下指标也正常**——因为线下用的是同一批被污染的样本,自评自然好看。问题要等到上线、对比线上真实反馈才暴露,而那时已经训了一版坏模型、损失了流量。这就是为什么**样本质量监控必须前置到"出样本的那一刻",而不是等模型上线后用效果倒推**。平台研发的护城河,正是这套"在样本层就拦住问题"的能力——它让你比算法更早发现数据出了事。这也直接关联 [样本离在线一致性](../04-样本数据管理/样本离在线一致性.md)。

## 5. 批拼接 vs 流拼接:T+1 Spark 与实时 Flink

拼接引擎的两条技术路线,平台研发的核心架构选型。

### 5.1 两条路线时序对比

```
批拼接(Spark / T+1):
  Day D 全天日志落地 ──────────────────> Day D+1 凌晨跑批 ──> 全量 join ──> 出 D 的样本
  ├ 优点:实现简单、口径稳定、天然处理乱序(全天数据都到齐了)、易回溯
  └ 缺点:样本延迟一天,模型最快也只能 T+1 更新

流拼接(Flink / 实时):
  曝光流 ┐
  点击流 ┼──> Flink 状态拼接(KeyBy request_id,缓存曝光等反馈)──> 等待窗口到期 ──> 实时出样本(秒~分钟级)
  转化流 ┘
  ├ 优点:样本新鲜,支撑在线学习 / 准实时模型更新
  └ 缺点:大状态(曝光要缓存等反馈)、等待窗口权衡、迟到要旁路补、运维重
```

### 5.2 详细对比表

| 维度 | 批拼接(Spark T+1) | 流拼接(Flink 实时) |
|------|--------------------|---------------------|
| 样本新鲜度 | T+1(慢一天) | 秒~分钟级 |
| 乱序处理 | 天然解决(全量到齐) | 靠 watermark + 等待窗口 |
| 迟到处理 | 天然解决(窗口内) | 需要旁路补数 / 翻正 |
| 状态成本 | 无(批量算) | **高**(曝光缓存在状态后端) |
| 实现复杂度 | 低 | 高 |
| 运维成本 | 低(失败重跑) | 高(7×24,状态/checkpoint) |
| 口径稳定性 | 高(易复算复核) | 中(实时口径需对账) |
| 适合模型 | 天级全量/增量训练 | 在线学习、准实时更新 |
| 资源弹性 | 好(跑完释放) | 差(常驻) |

补充更细的几个工程维度,这些往往是真正决定选型的"暗线":

| 维度 | 批拼接(Spark T+1) | 流拼接(Flink 实时) |
|------|--------------------|---------------------|
| 数据回溯/重算 | 极易(重跑某天即可) | 难(要回拨 offset + 重建状态) |
| 调试/排错 | 易(数据静止可反复查) | 难(状态在飞,难复现) |
| 故障爆炸半径 | 小(单天作业,隔离) | 大(常驻作业挂了影响实时全链路) |
| 特征快照处理 | 直接 join,无状态压力 | 必须旁路,否则状态爆 |
| 升级/发版 | 随时改逻辑下次生效 | 需状态兼容迁移,谨慎 |
| 成本模型 | 按需占用,峰谷分明 | 常驻占用,按峰值预留 |

**一个简单的选型判断清单**(满足越多越该上流):
- [ ] 模型需要分钟/小时级更新(在线学习、准实时召回更新)?
- [ ] 实时样本能带来可量化的线上收益(冷启动、热点快速捕捉)?
- [ ] 团队有 Flink 状态作业的运维能力(checkpoint/状态后端/积压治理)?
- [ ] 有 T+1 批拼接作兜底对账(不裸奔纯流)?

如果以上多数为否,**老老实实先把 T+1 批拼接做扎实**——它能覆盖绝大多数模型的需求,且省心、稳定、易回溯。流拼接是"业务确实需要新鲜度"时的升级项,不是默认项。

### 5.3 流拼接的"大状态"为什么是头号难题

流拼接里,**曝光先到、反馈后到**,所以引擎必须把曝光**缓存在状态**里,等着后续点击/转化来 join。状态大小约等于:

```
状态量 ≈ 等待窗口时长 × 单位时间曝光量 × 单条曝光大小(含特征快照)
```

代入经验量级:窗口 30 分钟、日曝光百亿(约每秒百万级)、单条几 KB 含特征快照——状态轻松到 TB 级。这就要求:

- **特征快照不要全进状态**:可以只缓存 join key + 轻量字段,特征快照旁路存储,出样本时再回查。
- **窗口尽量短 + 迟到旁路**:窗口内拼不上的反馈走旁路补数,而不是无限拉长窗口。
- **状态后端要扛得住**:RocksDB 状态后端 + 合理 checkpoint + TTL 清理。

> 🎓 **老师视角:几乎没有平台是"纯流"或"纯批",都是 Lambda 式双轨。**
> 现实里成熟平台普遍是:**流拼接出实时样本喂在线学习/准实时模型,同时批拼接在 T+1 全量重拼一遍做"对账兜底和回灌"。** 因为流拼接的迟到、丢失、口径漂移,最终都要靠 T+1 的全量批拼接来校正——批是流的"标准答案"。平台研发要接受这个复杂度:你大概率要同时维护两套拼接,并保证它们口径一致(这又回到离在线一致性)。业界也有用 Paimon 这类「流批一体」的湖格式,让同一张样本表既被流写入又被批回灌,减少两套存储的口径分裂。

### 5.4 流拼接的内部机制:一张图看懂状态流转

```
   曝光流 ─KeyBy(request_id)─┐
                            ▼
                  ┌──────────────────────┐
                  │  Flink Keyed State    │   每个 request_id 一个状态桶
                  │  存:轻量曝光行 + key  │   (特征快照旁路,不进状态)
                  │  注册:定时器 timer    │   到 event_time+窗口 触发
                  └──────────┬───────────┘
   点击流 ─KeyBy(request_id)─┘   到达时查状态,匹配上则更新 label
                             │
                  ┌──────────▼───────────┐
                  │ 定时器触发(窗口到期) │ → 输出样本(此刻 label 已收敛)
                  │ + 清理状态(TTL)     │   旁路回查特征快照拼成宽行
                  └──────────────────────┘
                             │
              迟到的反馈(窗口已关)→ 进旁路流 → 批 / 准实时翻正回灌
```

关键设计点:
- **KeyBy request_id**,让同一请求的曝光和反馈落到同一并发实例,才能 join。
- **曝光进状态等反馈**,用定时器在 `event_time + 等待窗口` 触发输出。
- **特征快照不进状态**(否则状态爆炸),只存 join key,出样本时旁路回查(KV 存储 / 外部表)。
- **窗口到期即输出 + 清状态**;窗口后才到的迟到反馈走旁路,交给批/准实时翻正。

### 5.5 数字感:流拼接到底要多大资源

用经验量级粗算(示意,非精确):

```
  日曝光 100 亿 → 平均 ~12 万条/秒,高峰可能 50~100 万条/秒
  等待窗口 30 分钟 → 状态里同时驻留约 (12万/s × 1800s) ≈ 2 亿个曝光 key
  单 key 轻量行(去掉特征快照)约 200~500 字节
  → 状态量级 ≈ 几十 GB ~ 上百 GB(若不旁路特征快照,直接 TB 级)
```

这解释了两件事:① 为什么"特征快照旁路"是流拼接的生死线;② 为什么窗口长短是资源的直接乘数,不能随意拉长。**对老板/SRE 沟通流拼接资源时,就用这个乘法公式:状态 ≈ 窗口 × QPS × 单条大小。**

### 5.6 等待窗口与状态后端的工程细节

窗口和状态不是配一个数就完事,有一串工程开关要拧:

| 工程项 | 怎么做 | 坑点 |
|--------|--------|------|
| 窗口时长 | 按反馈乱序/到达延迟的 P99 实测设定,不拍脑袋 | 太短漏拼(假负),太长状态爆 + 样本不新鲜 |
| 状态后端 | 大状态用 RocksDB state backend(落盘),不用纯堆内存 | 堆内存状态 OOM;RocksDB 要给够本地磁盘 + 调 block cache |
| 状态 TTL | 给状态设 TTL = 窗口 + 余量,过期自动清 | 不设 TTL 状态无限涨;TTL 太短会误清还要用的曝光 |
| Checkpoint | 大状态用增量 checkpoint,间隔按可容忍重算量定 | 全量 checkpoint 在 TB 状态下会拖垮作业 |
| 反压/积压 | 监控算子反压 + Kafka lag,积压超窗口要熔断/告警 | 积压一旦超过窗口,就是 4.8 那类事故 |
| 倾斜 | 热点 user/item 会让某并发实例状态畸大 | 对 key 加盐 / 二次散列打散热点 |
| 迟到旁路 | 窗口后到的反馈输出到 side output,交批翻正 | 不处理就是永久漏拼;不能简单丢弃 |

> 🎓 **老师视角:流拼接的运维成本,本质是"为新鲜度付的常驻税"。**
> 批拼接跑完就释放资源、挂了重跑就行,几乎没有常驻运维负担。流拼接是 7×24 常驻的:状态后端要喂磁盘、checkpoint 要存对象存储、积压要盯、热点要治、升级要状态兼容迁移……这些都是为"样本新鲜几小时"付出的持续成本。**平台研发给老板算账时要诚实:流拼接不是"更先进所以要上",而是"业务需要分钟级样本新鲜度,才值得付这份常驻税"。** 如果模型本来就 T+1 全量训练,硬上流拼接是给自己找罪受。这条权衡见后文 [权衡取舍](#权衡取舍)。

### 5.7 平台研发要交付什么

- **批/流两套拼接引擎**,或一套流批一体引擎,并保证口径对齐。
- **流拼接的状态治理**:特征快照旁路、窗口策略、状态后端调优、TTL、热点打散。
- **T+1 对账与回灌作业**:把流样本和批样本对账,差异回灌修正。
- **统一的拼接配置**:同一套 DSL 同时下发给批和流,避免两套逻辑漂移。
- **积压熔断**:Kafka lag 超过窗口时,自动熔断实时样本、回退批口径。

## 6. 样本去重、膨胀与一次曝光多 item

样本数量直接等于存储成本和训练时长,膨胀控制是平台研发的硬指标。

### 6.1 去重 dedup

去重要分清"在去重什么":

| 去重层次 | 重复来源 | 去重键 | 手段 |
|---------|---------|--------|------|
| 日志级去重 | at-least-once、客户端重试 | event_uuid / (request_id,item_id,pos) | 主键 upsert / 状态去重 |
| 样本级去重 | 同一曝光被拼了多次 | (request_id, item_id) | 拼接后按主键收敛 |
| 跨天去重 | 流样本 + 批样本都写了同一条 | (date, request_id, item_id) | 主键表覆盖,批为准 |

### 6.2 样本膨胀:一次曝光多 item

一次请求曝光 N 个 item,就产生 N 行样本。整页/列表场景下,膨胀是天然的:

```
 一次刷新 = 曝光 10 个 item  →  10 行样本
 用户日均刷 50 次            →  500 行/人/天
 日活 1 亿                   →  500 亿行/天  (经验量级,非精确)
```

膨胀的应对不是"少拼",而是**采样和裁剪**:

- **负样本下采样**:负样本远多于正样本,训练前对负样本按比例采样(见第 8 节和 [03-负采样与样本偏差治理](../04-样本数据管理/03-负采样与样本偏差治理.md))。
- **特征瘦身**:特征快照里冗余、低区分度的字段裁掉,单条样本变小。
- **冷数据降存**:历史样本转低成本存储,只保留近 N 天热样本供频繁训练。

> 🎓 **老师视角:膨胀问题本质是"曝光决定下限"。**
> 你不能为了省存储而少记曝光——曝光是骨架,漏记曝光等于丢样本、引入偏差。所以膨胀的下限被曝光量牢牢锁死,平台研发能动的只有:单条变小(特征瘦身)、总条数受控(采样)、存储分层(冷热)。**对老板要讲清楚:样本存储成本随业务曝光量线性增长,这是结构性的,不是优化能消灭的,只能管理。**

### 6.3 一次曝光多 item 的归因陷阱

整页曝光 10 个 item,用户点了第 3 个。**那其余 9 个一定是负样本吗?** 不一定——用户可能只看了前 5 个就点了第 3 个,后 5 个根本没真曝光(回到 2.2 的"真曝光"问题)。把没看到的也当负样本,会污染样本。平台研发要支持**基于真曝光/停留位置的负样本裁剪**。

### 6.4 一次容量测算:把膨胀算成钱

平台研发要能当场给老板/SRE 算出样本规模和成本(经验量级,示意):

```
  日活 DAU            = 1 亿
  人均日刷新次数      = 50 次/人
  每次曝光 item 数    = 8 个(真曝光)
  → 日曝光样本行数    = 1亿 × 50 × 8 = 400 亿行/天

  单条样本大小:
    特征快照 ~3KB + 标签/元数据 ~0.2KB ≈ 3.2KB(列存压缩后约 1/3 ~ 1/5)
  → 压缩后单条 ~ 0.7~1KB
  → 日样本表大小 ≈ 400亿 × 0.8KB ≈ 32 TB/天(未采样)

  若负样本下采样到 1/5:
  → 日样本表 ≈ 7~8 TB/天
  → 保留 30 天热样本 ≈ 200+ TB
```

这套测算让"膨胀"从抽象变具体:**采样率、特征大小、保留天数是平台研发手里能调的三个旋钮,每个都直接换算成存储账单。** 跟老板谈成本优化时,就摆这三个旋钮,而不是空谈"优化存储"。

> 🎓 **老师视角:特征快照大小是被低估的成本杠杆。**
> 大家容易盯着"样本条数"砍,但单条样本里特征快照常占 90%+ 的体积。把特征从 500 维砍到 300 维,或把高基数 ID 特征哈希压缩,带来的存储收益往往比砍样本条数更大,且不影响样本分布(不引入采样偏差)。**所以做样本降本,先做特征瘦身(无损或低损),再做负采样(有损、要校正)——顺序别反了。** 这条要和特征平台联动,见 [特征平台架构](../03-特征数据管理/02-特征平台(Feature Store)架构.md)。

### 6.5 平台研发要交付什么

- **三层去重能力**(日志级/样本级/跨天)。
- **可配置采样**(按 label 分层采样,且记录采样率用于训练侧权重校正)。
- **特征瘦身工具**(无用字段下线、特征重要性反馈)。
- **冷热分层存储策略**。

## 7. 样本格式与落湖:TFRecord / Parquet / Paimon

样本最终要落地成可被训练框架高效读取的格式。这是平台研发和存储打交道最深的一节。

### 7.1 三种主流落地形态对比

| 维度 | TFRecord | Parquet | 数据湖主键表(Paimon) |
|------|----------|---------|----------------------|
| 本质 | TF 序列化二进制流 | 列存文件格式 | 湖表(底层也是 Parquet/ORC + 元数据 + LSM) |
| 读取方 | TensorFlow 友好 | 通用(Spark/Flink/各框架) | 通用 + 流批一体读 |
| 是否支持 update | 否(只能重写) | 否(append/重写) | **是(主键 upsert)** |
| 是否支持 changelog | 否 | 否 | **是** |
| schema 演进 | 弱(proto 改动麻烦) | 中(列增删) | **强(schema evolution)** |
| 列裁剪/谓词下推 | 弱 | **强** | **强** |
| 适合场景 | 纯 TF 训练、固定 schema | 通用离线样本、多框架 | 延迟反馈翻正、流批一体、增量样本 |

补几个落地维度,帮你选型时不漏要点:

| 维度 | TFRecord | Parquet | Paimon 主键表 |
|------|----------|---------|--------------|
| 写入幂等/去重 | 无,需上游保证 | 无,需上游保证 | **主键天然去重** |
| 小文件治理 | 自己合并 | 自己合并 | **内置 compaction** |
| 训练随机读/shuffle | 顺序流式好,随机弱 | 列存,随机一般 | 同 Parquet,但可增量 |
| 跨引擎一致性 | 仅 TF 生态 | 好 | **好(流批同表)** |
| 元数据/快照能力 | 无 | 无(靠目录约定) | **快照、时间旅行、回滚** |
| 典型角色 | 训练投喂层 | 通用交换层 | **样本权威存储层** |

一条常见的"三层落地"工作流(很多平台的实际形态):

```
   流拼接(Flink) ─upsert─┐
                         ▼
   批回灌(Spark) ─upsert─> Paimon 主键样本表 (权威存储: 去重/翻正/schema演进/changelog)
                         │
                         ├─ 在线分析/抽样/质量监控 直接查 Paimon
                         │
                         └─ 定时导出 ─> TFRecord/Parquet ─> 喂 TF/PyTorch 训练
                            (导出时做:负采样、列裁剪、特征瘦身、shuffle)
```

> 🎓 **老师视角:"权威存储"和"投喂格式"分层,是这一节最该记住的架构判断。**
> 新手容易纠结"到底用 TFRecord 还是 Paimon",其实它们在不同层、不冲突。**Paimon 主键表当权威存储**——负责把样本正确地攒起来、改起来(去重、翻正、schema 演进),它是单一事实来源(single source of truth)。**TFRecord/Parquet 当投喂格式**——从权威存储按训练需要导出一份(采过样、裁过列、shuffle 过),用完即弃、可随时重导。这样分层的好处:权威数据只有一份不会漂,投喂格式可以随训练框架/采样策略千变万化。**平台研发的设计原则:可变的、需要被修正的东西放权威层;只读的、为性能优化的东西放投喂层。**

### 7.2 为什么延迟反馈场景偏爱 Paimon 主键表

回到第 4.3 的延迟反馈:曝光后先出 label=0,转化来了要改成 label=1。

```
 纯 append(Hive/TFRecord/裸 Parquet):
   先写  (R123, itemB, label=0)
   3天后 想改成 label=1  →  只能"再写一行 label=1 + 下游自己取最新"或"全量重刷分区"  →  痛苦

 Paimon 主键表(主键 = (request_id, item_id)):
   先写  upsert (R123, itemB, label=0)
   3天后 upsert (R123, itemB, label=1)   →  主键覆盖,自动收敛为最新值
                                         →  Changelog 还能告诉下游"这条从0翻成1了"
```

这就是为什么样本表用主键表落湖,在有延迟反馈/迟到/去重需求时,工程上干净得多:

- **去重**:主键天然去重(第 6 节)。
- **翻正/回灌**:upsert 直接覆盖,不用重刷分区。
- **流批一体**:同一张表流写、批回灌、流读、批读,减少 Lambda 双存储的口径分裂(第 5.3 老师视角)。
- **增量读**:训练侧可只读 changelog 增量样本,不必全量扫。

> 🎓 **老师视角:别把"格式"和"存储抽象"混为一谈。**
> TFRecord / Parquet 是**文件格式**,Paimon 是**表格式 / 湖存储抽象**(底层数据还是 Parquet/ORC)。它们不是非此即彼:很多平台是 **Paimon 主键表做样本的"权威存储"(支持 upsert/翻正/去重),再导出成 TFRecord 喂 TF 训练**。也就是说,湖表负责"正确地攒和改样本",TFRecord 负责"高效地喂模型",各司其职。平台研发要想清楚这条链路,不要为了上湖而丢掉训练读取性能,也不要为了 TF 方便而把"可修改"这个能力丢掉。

### 7.3 schema 演进:样本表最容易爆的雷

特征会不断新增/下线,标签会新增目标。样本表 schema 几乎天天在变。雷点:

- 新增特征列:老分区没这列,训练读到 null,要约定默认值。
- 删除/重命名:训练侧硬编码列名会直接挂。
- 类型变更:int → long、float 精度变化,序列化不兼容。

对策:**用支持 schema evolution 的存储(Paimon/Parquet 列增删)** + **schema 版本管理 + 样本表 schema 与特征平台对齐**(特征定义变更要联动样本 schema)。这又和 [特征平台架构](../03-特征数据管理/02-特征平台(Feature Store)架构.md) 强相关。

更系统地,把 schema 变更按"兼容性"分级处理:

| 变更类型 | 兼容性 | 风险 | 平台对策 |
|---------|--------|------|---------|
| 新增列(末尾追加) | 后向兼容 | 低 | 老分区/老数据读为 null,约定默认值;Paimon 支持 add column |
| 删除列 | 前向破坏 | 中 | 不物理删,先标 deprecated,确认无消费者再下线;训练侧用列名而非下标读 |
| 重命名列 | 破坏 | 高 | 等价于删+增,禁止直接 rename;新增新列 + 双写过渡期 |
| 类型收窄(long→int) | 破坏 | 高 | 禁止;只允许放宽(int→long、float→double) |
| 类型放宽(int→long) | 兼容 | 低 | Paimon/Parquet 多支持,仍需训练侧验证反序列化 |
| 调整列顺序 | 取决于读法 | 中 | **强制按列名读,不按位置读**,则顺序无关 |

落到几条硬规则,平台研发要写进样本平台的"宪法":
1. **训练读样本一律按列名(name-based),禁止按下标(position-based)。** 这一条让"新增列/调顺序"从破坏性变成无害。
2. **只增不删、只放宽不收窄。** 删除走 deprecated 软下线流程。
3. **每张样本表带 schema_version,每次变更登记 + 通知下游消费者。**
4. **特征平台变更联动:** 特征上下线在 Feature Store 注册时,自动生成对应的样本表 schema 变更工单,而不是靠人记。

> 🎓 **老师视角:样本 schema 的麻烦在于"双向兼容窗口"。**
> 样本表不是一次性的——你今天改了 schema,但昨天、上周的历史分区还是老 schema,而训练经常要"读最近 30 天"跨多个 schema 版本的数据。这意味着:新 schema 要能读老数据(后向兼容,老数据缺的列补默认值),写新数据的作业也不能让老的读作业挂(前向兼容)。**Paimon 这类湖表的 schema evolution 帮你管住了存储层的兼容,但训练侧的读取代码(尤其 TFRecord 那种 schema 弱的)才是真正容易崩的地方。** 所以"按列名读 + 默认值约定"这两条,价值比换什么存储都大。展开见 [特征平台架构](../03-特征数据管理/02-特征平台(Feature Store)架构.md)。

### 7.4 Paimon 样本表的一个落地骨架(示意 DDL)

把上面的讨论落成一张 Paimon 主键样本表的建表骨架(字段精简示意):

```sql
CREATE TABLE sample_ctr (
    dt              STRING,                 -- 分区:天
    request_id      STRING,
    item_id         STRING,
    feature_snapshot STRING,                -- 或拆成多列 / 用 bytes
    pos             INT,
    label_click     INT,
    label_conv      INT,
    conv_amount     DOUBLE,
    sample_rate     DOUBLE,                 -- 采样率,落表供训练校正
    label_update_time TIMESTAMP(3),         -- 翻正发生的时间
    PRIMARY KEY (dt, request_id, item_id) NOT ENFORCED
) PARTITIONED BY (dt) WITH (
    'bucket' = '-1',                        -- 或固定 bucket 数,按基数定
    'merge-engine' = 'deduplicate',         -- 同主键保留最新(天然去重 + 翻正)
    'changelog-producer' = 'lookup',        -- 产出 changelog 供下游增量消费
    'snapshot.time-retained' = '7 d'        -- 配合翻正窗口的保留期
);
```

要点对应前文:`PRIMARY KEY` 实现去重和翻正;`merge-engine=deduplicate` 让后写覆盖先写;`changelog-producer` 让下游能增量感知"label 0→1"的翻转;分区 + 保留期配合归因窗口管理存储。**流拼接作业 upsert 写入,T+1 批作业也 upsert 回灌对账,同一张表流批共写——这正是流批一体的样本中台形态。**

> 🎓 **老师视角:`merge-engine` 选 deduplicate 还是 partial-update,取决于"标签是整行覆盖还是逐列补齐"。**
> 如果翻正时你能拿到完整一行(特征+所有 label),用 `deduplicate` 整行覆盖最简单。但很多时候反馈是**分多次、逐目标到达**的:点击先到、转化后到、分享更晚到,每次只更新一个 label 列。这时整行覆盖会把还没到的列覆盖成空。Paimon 的 `partial-update` merge-engine 正好解决这个——**同主键多次写入,逐列合并,非空列才覆盖**。平台研发要根据"反馈是整行到还是逐列到"来选 merge-engine,这是 Paimon 落样本表最该想清楚的一个开关。

### 7.5 平台研发要交付什么

- **样本表的"权威存储"选型**(推荐主键湖表)+ **训练导出格式**(TFRecord/Parquet)。
- **schema 演进治理**(版本管理、默认值约定、与特征平台联动)。
- **增量读取能力**(changelog 喂增量训练)。
- **存储成本管控**(列存压缩、冷热分层、TTL)。

## 8. 样本质量:正负比、量级、采样

样本质量是模型效果的天花板。"Garbage in, garbage out",平台研发是这道关口的守门人。

### 8.1 核心质量指标

| 指标 | 含义 | 健康参考(经验值) | 异常的含义 |
|------|------|------------------|-----------|
| 样本量级 | 日产出样本行数 | 日百亿级曝光样本(头部) | 跳水 → 上游日志丢失 |
| 正负比 | 正样本 / 负样本 | CTR 场景常 1:几十~1:几百 | 突变 → 拼接或归因出错 |
| join 命中率 | 反馈拼上骨架的比例 | 越高越好,持续监控基线 | 下跌 → key 错/骨架缺失 |
| 标签延迟分布 | 转化在曝光后多久到 | 业务相关,看长尾 | 长尾变长 → 延迟反馈加重 |
| 特征缺失率 | 特征快照空值比例 | 越低越好 | 升高 → 在线快照通道异常 |
| 重复率 | 同主键重复行比例 | 接近 0(去重后) | 升高 → 去重失效 |

### 8.2 正负比与采样

负样本天然远多于正样本(大部分曝光不被点击)。直接全量训练:① 样本量爆炸;② 正样本被淹没,模型偏向"全预测负"。所以:

- **负样本下采样**:按比例丢弃部分负样本,平衡正负比、缩减样本量。
- **采样后要做权重校正 / 预估值校准**:采样改变了数据分布,模型预估的概率会偏离真实 CTR,需要在训练或预测时校正(如 prior correction)。
- **采样率要记录进样本元数据**:训练侧才能正确还原。

> 🎓 **老师视角:采样不是平台单方面能定的,是和算法的"合同"。**
> 平台可以提供采样能力,但**采多少、怎么采、怎么校正,是算法的建模决策**。平台研发千万别擅自改采样率而不通知算法——你改了分布,人家模型的校准全废了,线上 CTR 预估会系统性偏高/偏低。**正确姿势:平台把采样做成可配置、可记录、可回溯的能力,采样率随样本落表,算法按需配置,出问题能追溯到是哪次采样配置导致的。** 偏差治理整体见 [03-负采样与样本偏差治理](../04-样本数据管理/03-负采样与样本偏差治理.md)。

### 8.3 样本偏差:平台要意识到的几类

- **曝光偏差**:只有被推荐系统选中曝光的 item 才有样本,没曝光的 item 永远没机会(幸存者偏差)。
- **位置偏差**:位置靠前的天然 CTR 高,样本里混入了位置效应。
- **延迟反馈偏差**:延迟转化未到时被标负,短期样本正样本偏少。

平台研发不一定要解决这些(那是算法的活),但**要保留足够信息让算法能纠偏**:位置字段、曝光来源、采样率、标签生成时间等都要落进样本。

### 8.4 质量监控落到可执行:几条监控 SQL(示意)

监控大盘背后就是一组定时跑的聚合查询(示意,针对 Paimon/Hive 样本表):

```sql
-- 1) 每天样本量级与正负比(对比历史基线告警)
SELECT dt,
       COUNT(*)                              AS total,
       SUM(label_click)                      AS pos_click,
       SUM(label_click) / COUNT(*)           AS ctr,         -- 突变即告警
       SUM(label_conv)  / NULLIF(SUM(label_click),0) AS cvr
FROM sample_ctr WHERE dt = '${today}' GROUP BY dt;

-- 2) join 命中率:反馈拼上骨架的比例(以点击为例)
SELECT
    COUNT_IF(clk.request_id IS NOT NULL) / COUNT(*) AS click_join_rate
FROM click clk
LEFT JOIN impression imp
    ON clk.request_id = imp.request_id AND clk.item_id = imp.item_id
WHERE clk.dt = '${today}';      -- 命中率跳水 = 骨架缺失或 key 错

-- 3) 重复率:同主键出现多次的比例(去重后应接近 0)
SELECT 1 - COUNT(DISTINCT request_id, item_id) / COUNT(*) AS dup_rate
FROM sample_ctr WHERE dt = '${today}';

-- 4) 特征缺失率:特征快照为空的比例
SELECT COUNT_IF(feature_snapshot IS NULL OR feature_snapshot = '') / COUNT(*) AS feat_null_rate
FROM sample_ctr WHERE dt = '${today}';
```

每条 SQL 对应 8.1 表里的一个指标,跑出来对比历史基线,偏离容忍带就告警。**这就是样本流水线的"心电图"的具体实现——平台研发要把它做成自动调度 + 阈值告警,而不是出事了再手动查。**

> 🎓 **老师视角:监控要监"比率"而非"绝对值",且要有"基线"。**
> 绝对量级天然随大盘流量波动(周末/大促),盯绝对值会被流量波动淹没。盯**比率**(CTR、join 命中率、重复率、缺失率)才能剥离流量、暴露真问题。而比率也要有动态基线(同比上周同一天 ± 容忍带),否则正常的业务节律会触发误报。**好的样本监控不是"告诉你数字是多少",而是"告诉你今天和过去比哪里不对劲"。** 这套监控建好了,上游埋点一丢、拼接一出 bug,你比算法更早发现——这是平台研发的价值体现。

### 8.5 平台研发要交付什么

- **样本质量监控大盘**(8.1 全部指标 + 告警基线)。
- **分层采样能力 + 采样率落表 + 校正信息**。
- **纠偏所需字段的完整留存**(位置、来源、时间、采样率)。
- **样本血缘**:从样本表能追到来源日志、拼接配置版本、特征版本。

## 业界怎么做

> 以下为业界公开实践的**模式归纳**,非某家精确架构,细节请以一手资料为准。

- **双轨 Lambda(批+流)是主流。** 实时流拼接(Flink)出新鲜样本喂准实时/在线学习,T+1 批拼接(Spark)做全量重拼、对账、延迟反馈回灌。批是流的"标准答案"。
- **特征快照随曝光落日志**,拼接时直接用快照而非离线重算,这是解决离在线一致性的工业共识(对应各家"样本服务 / sample dump"能力)。
- **延迟反馈普遍用"先负后翻正"**:广告/电商场景尤其重视,配合主键可更新存储。学术上对应 delayed feedback model 系列工作,工程上对应可 upsert 的样本表。
- **湖格式做样本权威存储正在普及。** Iceberg/Hudi/Paimon 这类支持 upsert、schema evolution、流批一体的表格式,被越来越多平台用作样本中台底座;Paimon 的主键表 + Changelog 在"延迟反馈翻正 + 增量喂训"上契合度高。
- **真曝光优先**:头部信息流普遍区分"下发"和"可视曝光",用可视曝光当样本骨架以降低假负样本。
- **样本与特征平台打通**:样本 schema 与 Feature Store 的特征定义联动,特征上下线自动影响样本生成。

## 权衡取舍

| 取舍点 | 选 A | 选 B | 判断依据 |
|--------|------|------|---------|
| 拼接引擎 | 批 T+1(简单稳) | 流实时(新鲜) | 模型更新频率;多数选双轨 |
| 等待窗口 | 长(漏拼少) | 短(新鲜、状态小) | 反馈延迟分布 + 状态成本 |
| 延迟反馈 | 等满窗口(标签准) | 先负后翻正(新鲜) | 转化窗口长短;长则倾向翻正 |
| 负样本 | 全量(无偏) | 下采样(省资源) | 算法建模需求 + 资源;务必校正 |
| 样本存储 | 裸 Parquet/TFRecord(读快) | 主键湖表(可改/去重/流批) | 是否需要翻正/去重/增量 |
| 真曝光判定 | 用下发(简单) | 用可视曝光(准) | 假负样本对效果的影响 |
| 状态里放什么 | 全字段(出样本快) | 仅 key + 旁路特征(省状态) | 状态预算;大流量必旁路 |

核心心法:**新鲜度、准确性、成本三者不可能同时拉满。** 批拼接牺牲新鲜度换准确和省心,流拼接牺牲成本和复杂度换新鲜,延迟反馈翻正牺牲短期准确换新鲜。平台研发的工作就是把这些旋钮做成可配置,让业务按场景调。

## 常见误区与反直觉

1. **"负样本要去采集。"** 错。负样本是曝光 LEFT JOIN 反馈后拼不上自然产生的,你只需要完整记录曝光。
2. **"特征训练时离线重算就行,何必落快照。"** 错且危险。离线重算会引入特征穿越,线下虚高线上崩。必须用请求时刻的快照。
3. **"下发日志就是曝光。"** 错。下发 ≠ 用户真看到,用下发当曝光会造大量假负样本。
4. **"流拼接比批拼接先进,应该全切流。"** 反直觉:成熟平台几乎都是双轨,流要批来校正。纯流口径漂移没人兜底。
5. **"join 不上就是数据错了。"** 不一定。归因窗口内本就没反馈的曝光,join 不上是正常的(=负样本)。要区分"应该拼上却没拼上"(命中率异常)和"本就没反馈"(正常负样本)。
6. **"按到达时间(process_time)拼接最简单。"** 错。乱序会让你漏拼大量反馈,必须按 event_time。
7. **"采样率平台自己定就行。"** 错。采样改变分布,必须和算法约定并落表校正,否则线上预估值系统性偏差。
8. **"样本表用 Hive append 就够了。"** 在有延迟反馈翻正/去重需求时很痛苦,主键可更新湖表干净得多。
9. **"转化 join 和点击 join 一样写。"** 错。点击是精确 join,转化常常拿不到 request_id,要带归因策略做近似匹配。
10. **"窗口开大点总没错,反正不漏拼。"** 反直觉:窗口越大,流状态越大(TB 级)、样本越不新鲜,运维成本陡增。窗口要和延迟分布精算。

## 向上沟通

**老板可能问 → 你怎么答:**

- **"样本为什么要存这么多、这么贵?"**
  → "样本量随业务曝光量**线性增长**,日曝光百亿级、单条带几百维特征快照,一天几十 TB 是结构性的。我们能做的是特征瘦身、负样本采样、冷热分层把成本管住,但消灭不了——曝光越多,样本越多,这是业务成功的代价。"

- **"为什么模型不能实时更新,要等一天?"**
  → "如果走 T+1 批拼接,样本天然延迟一天;要实时就得上流拼接,代价是大状态、高运维、口径要对账。我们在推双轨:实时流出新鲜样本给准实时模型,T+1 批做兜底校正。这是新鲜度和稳定性的平衡。"

- **"为什么转化数据对不上 / 模型对转化预估不准?"**
  → "转化有**延迟反馈**:用户可能几天后才下单,样本生成时还没等到,先标成了负样本,后续要翻正回灌。窗口越长(大件/付费)越明显。我们用主键表 upsert 做翻正,并和算法约定校正方式。这不是 bug,是这类问题的固有特性。"

- **"这套样本流水线的风险点在哪?"**
  → "三个:① 埋点丢失导致样本失真(靠对账心电图监控);② 离在线特征不一致导致线上崩(靠特征快照);③ 拼接口径漂移(靠批校正流)。我们都有监控和兜底机制。"

## 自测三问

**Q1:为什么训练用的特征必须是"请求时刻的快照",离线重算不行?**
> 参考答案:因为特征会随时间变化。模型上线时用的是 t0 时刻的特征值,若训练时用 T+1 离线数仓重算,会算到 t0 之后变化过的值,造成**特征穿越 / 离在线不一致**,线下 AUC 虚高、上线效果崩。所以要在线打分时把特征值原样落日志(特征快照),拼接时直接用快照。

**Q2:负样本是怎么来的?为什么说它是"被动产生"的?**
> 参考答案:拼接以曝光为骨架,LEFT JOIN 点击/转化等反馈。在归因窗口内能 join 上反馈的是正样本(label=1),join 不上的就被标成负样本(label=0)。所以你不单独"采集"负样本,只要完整记录曝光,负样本就是"拼不出反馈的曝光"自然产生的。这也是为什么完整记录曝光骨架如此重要。

**Q3:延迟反馈为什么是难题?为什么主键可更新的湖表(如 Paimon 主键表)适合它?**
> 参考答案:转化可能在曝光后几小时到几天才发生,但训练等不了那么久。主流做法"先负后翻正":曝光后先按 label=0 出样本,等转化真到了再把它改成 label=1。这要求样本表能"事后修改同一条记录"。Paimon 主键表用主键 (request_id, item_id) 做 upsert,翻正只需覆盖写,无需重刷分区,还能产出 changelog 让下游增量感知标签翻转;纯 append 表(Hive/裸 Parquet)做这事要全量重刷,非常痛苦。
