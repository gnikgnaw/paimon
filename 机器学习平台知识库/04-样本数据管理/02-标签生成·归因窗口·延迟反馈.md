## 0. 30 秒速通(TL;DR)

如果你只有 30 秒,记住这 8 条:

1. **label 是监督信号**:点击 label(click)、转化 label(conversion)、时长/完播 label(dwell/finish)、多目标 label(CTR + CVR + 时长一起学)。它们都从行为日志生成,而不是天生就在样本里。
2. **生成 label = 把"曝光样本"和"后续行为"做关联(join)**。曝光是底座(impression),点击/转化是叠加上去的正向信号。没点的就是负样本(neg)。
3. **归因 attribution 回答的是"这次转化算给哪一次曝光/点击"**。主流口径:last-click(末次点击)、first-click(首次)、multi-touch(多触点分配)。口径不同,正样本数和 label 分布都会变。
4. **归因窗口 attribution window 决定"多久之内的行为才算这次曝光的功劳"**。点击后 7 天内下单才算转化 → 那 7 天就是窗口。窗口越长,转化越多但反馈越慢。
5. **延迟反馈 delayed feedback 是核心难题**:转化往往发生在曝光很久之后(电商下单 1~7 天、广告激活 1~30 天)。训练时 label 还没到,导致**假负样本**(fake negative)——当下标 0,未来其实是 1。
6. **延迟反馈解法两大流派**:① 等窗口(label 准但时效差);② 即时训练 + 回补/纠正(fake negative weighting、DFM、ESDFM、importance weighting、负样本延迟校正)。绝大多数大厂走的是 ②。
7. **平台侧必须交付的能力**:① **可更新 label 的样本存储**(label 能从 0 改成 1 并重写);② **归因计算服务**;③ **延迟反馈样本回补链路**(样本回流/重写);④ **label 延迟看板**。
8. **标签穿越 label leakage 是隐形炸弹**:用"未来才发生的信息"当特征或 label 衍生,离线 AUC 爆高、上线扑街。平台要在样本拼接时卡住时间边界。

> 🎓 **老师视角**:平台研发最容易犯的错,是把 label 当成"算法的事"。错。label 的**正确性、时效性、可回补性**全是平台链路的工程指标。算法只能在你给的 label 上做拟合;你给的 label 系统性偏了 5%,算法神仙难救。所以这一章你要当**主场**来打。

> 💡 **怎么用这篇文档**:第 1~3 节讲"label 怎么来"(生成、归因、窗口),是基础概念;第 4~5 节是核心难题延迟反馈及其解法;第 6~7 节讲两条容易踩雷的横向问题(多任务对齐、穿越);第 8~11 节是平台落地的硬交付(监控、去噪、数学直觉、存储实现);第 12 节给演进路线。**急用的话,先看 TL;DR + 第 5 节 + 平台侧 Checklist 三块,就能接住 80% 的对接需求。**

## 1. 标签 label 是什么:从日志到监督信号

### 1.1 概念对齐:label 不是数据,是"被定义出来的信号"

在监督学习里,样本 = `特征 X` + `标签 y`。模型学的是 `f(X) → y`。
在搜广推场景里:

- `X`(特征)= 用户特征 + item 特征 + 上下文特征,反映"曝光那一刻我看到了什么"。
- `y`(label)= 用户对这次曝光的"反应结果"。

关键认知:**label 不是日志里现成的一个字段,而是你用一套口径,从多张行为日志里"加工"出来的。** 同一批曝光,口径不同,label 就不同。

### 1.2 搜广推常见 label 类型

| label 类型 | 业务含义 | 典型取值 | 建模目标 | 难点 |
|-----------|---------|---------|---------|------|
| 点击 label(click) | 用户点没点 | 0/1 | CTR(点击率) | 反馈快(秒级),但有误点 |
| 转化 label(conversion) | 点完之后有没有"成交动作" | 0/1 | CVR(转化率) | **延迟严重**,正样本稀疏 |
| 时长 label(dwell/duration) | 停留/观看时长 | 连续值或分桶 | 时长回归/分类 | 噪声大,需去机刷/误触 |
| 完播 label(finish) | 视频是否看完 | 0/1 或完播率 | 完播率 | 不同时长 item 不可比 |
| 互动 label(like/comment/share) | 点赞评论转发收藏 | 0/1 | 多互动目标 | 极度稀疏 |
| 多目标 label | CTR/CVR/时长一起 | 多列并存 | 多任务学习 MTL | 各 label 反馈速度不一致 |

> 🎓 **老师视角**:注意到"反馈速度不一致"这一列了吗?这是平台的核心痛点伏笔。click 是秒级到的,conversion 可能 7 天后才到,时长是会话结束才到。**同一条曝光样本的多个 label,是在不同时间点陆续"长出来"的**。这意味着你的样本表不能是"一次写死",而必须支持"label 字段后续更新"。记死这一点,后面所有设计都围绕它。

### 1.3 label 是怎么从日志生成的:join 的本质

核心动作:**以曝光(impression)为骨架,把后续行为日志关联上去。**

```
原始日志(三条流,时间先后到达):

  曝光流 impression          点击流 click              转化流 conversion
  ┌──────────────────┐      ┌──────────────────┐     ┌──────────────────┐
  │ req_id=R1        │      │ req_id=R1        │     │ user=U1          │
  │ user=U1 item=A   │      │ user=U1 item=A   │     │ item=A           │
  │ ts=10:00:00      │      │ ts=10:00:05      │     │ ts=次日 14:00    │
  │ trace_id=T1      │      │ trace_id=T1      │     │ order_id=O9      │
  └──────────────────┘      └──────────────────┘     └──────────────────┘
          │                         │                         │
          │   join on (req_id/trace_id 或 user+item+时间窗)    │
          ▼                         ▼                         ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │ 训练样本(以曝光为主键)                                          │
  │ user=U1 item=A 上下文=... | click=1 | conversion=?(待回补)      │
  └─────────────────────────────────────────────────────────────────┘
```

生成逻辑(伪代码):

```sql
-- click label:曝光 LEFT JOIN 点击,匹配上则 1,否则 0
SELECT
  imp.*,
  CASE WHEN clk.trace_id IS NOT NULL THEN 1 ELSE 0 END AS click_label
FROM impression imp
LEFT JOIN click clk
  ON imp.trace_id = clk.trace_id          -- 精确 trace 关联
 AND clk.ts BETWEEN imp.ts AND imp.ts + INTERVAL '30' MINUTE;  -- 时间约束防误关联
```

> 🎓 **老师视角**:这里有两个工程陷阱,你必须替算法同学守住:
> ① **关联键要"够稳"**:最好的关联键是 `trace_id`(一次曝光的全链路唯一 ID),其次才是 `user+item+时间窗`。用后者会把"同一用户同一商品的另一次曝光的点击"错关联进来,污染 label。
> ② **时间边界要明确**:click 必须发生在 impression **之后**且在合理窗口内。否则会把上一次会话的点击算进来。这正是后面"标签穿越"的前置防线。
> 详细的日志拼接工程见 [01-样本全生命周期与日志拼接](../04-样本数据管理/01-样本全生命周期与日志拼接.md)。

### 1.4 负样本从哪来:曝光未点击 = 负

正样本好理解(点了/转化了)。负样本的定义往往被忽视但极重要:

- **CTR 的负样本** = 曝光了但没点击的样本。
- **CVR 的负样本** = 点击了但没转化的样本(注意 CVR 的样本空间通常是"点击集合",不是"曝光集合")。

这里埋着一个建模一致性问题:**CTR 在曝光空间训练,CVR 在点击空间训练,但线上要预估 `pCTCVR = pCTR × pCVR`,两个空间不一致会产生 sample selection bias(样本选择偏差)。** 这就是 ESMM(Entire Space Multi-task Model)要解决的问题。负采样与样本偏差的系统治理见 [03-负采样与样本偏差治理](../04-样本数据管理/03-负采样与样本偏差治理.md)。

> 🎓 **老师视角**:当算法同学跟你说"我要全曝光空间的 CVR 样本"时,别愣。他要做的就是 ESMM 这类全空间建模:CVR 不在点击子空间训练,而是在整个曝光空间上,用 `pCTR×pCVR` 联合监督。对平台的要求是:**你要能同时提供曝光骨架 + click label + conversion label 三者对齐的样本**,而不是切好的点击子集。

## 2. 归因 attribution:这次转化算给谁

### 2.1 为什么需要归因

一个用户从"看到"到"转化",中间可能经历多次曝光/点击:

```
时间轴 ───────────────────────────────────────────────────────────►
  Day1          Day2           Day4            Day5
  曝光A          点击A           曝光B+点击B       下单(转化!)
  (搜索结果)     (进详情页)      (信息流广告)      (¥299)

问题:这一次 ¥299 的转化,功劳算给谁?
  - last-click:算给 Day4 的点击B(末次)
  - first-click:算给 Day2 的点击A(首次)
  - multi-touch:A 和 B 按规则分(如各 50%,或时间衰减)
```

**归因(attribution)= 把一次结果(转化)分配给促成它的某次/某些触点(曝光或点击)的规则。** 它直接决定:哪些样本被标成正样本(label=1)。

### 2.2 三种主流归因口径对比

| 归因模型 | 规则 | 正样本归属 | 优点 | 缺点 | 常见场景 |
|---------|------|-----------|------|------|---------|
| last-click(末次点击) | 转化算给最后一次点击 | 末次触点 | 简单、主流、与"临门一脚"直觉一致 | 低估前期种草触点 | 效果广告 CVR 默认口径 |
| first-click(首次) | 算给第一次触点 | 首次触点 | 重视拉新/种草 | 低估临门一脚 | 品牌/拉新评估 |
| multi-touch(多触点) | 按规则分配权重 | 多个触点 | 全链路公允 | 实现复杂、权重难定 | 全域归因、MTA 分析 |
| linear / time-decay | 平均分 / 越近权重越高 | 多个触点 | 介于两者之间 | 仍需定参数 | 进阶归因分析 |
| position-based(U型) | 首末各 40%,中间分 20% | 多个触点 | 兼顾首末 | 经验参数 | 营销归因 |

> 🎓 **老师视角**:对**训练样本**而言,98% 的团队默认用 **last-click**,因为它能干净地把"一次转化 ↔ 一次点击"一一对应,样本好造。**multi-touch 更多用在归因分析/预算分配(MTA)**,而不是直接造训练 label,因为"一次转化分给 0.5 个正样本"在训练里很别扭。你跟算法对口径时,先问一句:"训练 label 用 last-click 行吗?" 大概率行。

### 2.3 归因的工程实现:一个有状态的关联

归因不是简单的两表 join,而是**有状态的回溯**:拿到一条转化,要往前回溯这个用户在窗口内的所有触点,按口径选出"被归因的那一条",再把对应曝光样本的 label 从 0 改成 1。

```
延迟反馈样本回补的归因子流程(last-click):

   转化事件到达
   (user=U1, item=A, ts=T_conv)
        │
        ▼
   ┌──────────────────────────────────────────┐
   │ 在 [T_conv - window, T_conv] 内            │
   │ 回溯 U1 对 item=A 的所有点击触点           │
   └──────────────────────────────────────────┘
        │
        ▼
   选出末次点击 click(ts 最大且 ≤ T_conv)
        │
        ▼
   定位该点击对应的曝光样本(trace_id 关联)
        │
        ▼
   把该样本 conversion_label: 0 → 1,重写样本   ←★ 这就是延迟反馈回补
```

> 🎓 **老师视角**:看到 ★ 那一步没有?**归因的输出,本质就是一个"把历史样本的 label 改写"的请求。** 这就要求你的样本存储**必须可更新(mutable label)**,不能是一次性追加(append-only)就锁死的。这是本文最重要的工程结论之一,后面第 5、8 节会反复回到它。Apache Paimon 这类支持主键更新(primary-key upsert)的湖格式,天然适配这个诉求——你可以用 `trace_id` 或 `sample_id` 当主键,后到的转化直接 upsert 覆盖 label 列。

## 3. 归因窗口 attribution window:多久之内才算数

### 3.1 定义

**归因窗口 attribution window = 从触点(曝光/点击)发生起,往后多长时间内的转化才算这次触点的功劳。**

- 例:点击后 **7 天**内下单算转化(7-day click-through window)。
- 例:曝光后 **1 天**内下单算转化(1-day view-through window)。

超过窗口的转化,**不算这次触点的 label**(可能算给后来的别次触点,或干脆不算)。

### 3.2 窗口长短的影响:一张权衡表

| 维度 | 短窗口(如 1 天) | 长窗口(如 30 天) |
|------|----------------|------------------|
| 捕获的转化数 | 少(漏掉慢转化) | 多(几乎全捕获) |
| 正样本比例 | 偏低 | 偏高、更真实 |
| label 时效(多久能定稿) | 快(1 天后样本就稳) | 慢(要等 30 天) |
| 假负样本严重程度 | 较轻 | **严重**(大量样本在窗口内才变正) |
| 训练新鲜度 | 高(能快速用上) | 低(要么等、要么回补) |
| 适用业务 | 即时性强:游戏激活、信息流点击 | 决策周期长:大件电商、教育、金融 |

### 3.3 窗口与延迟反馈是一枚硬币的两面

```
窗口选择 → 直接决定延迟反馈的严重程度

   曝光/点击     转化分布(典型电商,简化示意)
   ts=0          │
   │             │ ████████                      ← 当天转化(快)
   │             │ ████                          ← 第2天
   │             │ ██                            ← 第3~4天
   │             │ █                             ← 第5~7天(慢长尾)
   ▼             └──────────────────────────────►  转化发生时间
                 0  1   2   3   4   5   6   7 天

  若窗口=1天:只认当天那 ~一半转化 → 漏标大量正样本(但样本 1 天就定稿)
  若窗口=7天:认全部转化 → 标得全 → 但样本要等 7 天才"稳",期间全是假负
```

> 🎓 **老师视角**:窗口不是越长越好,也不是越短越好。它是**"label 完整性 vs label 时效性"的旋钮**。平台研发要做的不是替算法定这个值,而是:① 把窗口做成**可配置参数**(不同业务线、不同 label 类型可配不同窗口);② 让样本存储能区分"已定稿样本"和"窗口内待观察样本";③ 在看板里展示"当前窗口下,转化在第 N 天的累计回流比例"(见第 8 节)。**你提供旋钮和仪表盘,算法来拧。**

### 3.4 量级参考(典型/经验值,非精确)

> 以下为行业**经验量级**,不同业务差异极大,仅作直觉锚点:

- 信息流点击转化窗口:常见 **1~7 天**。
- 电商加购/下单窗口:常见 **7~14 天**;大件/耐用品可到 **30 天**。
- 广告联盟激活/付费:常见 **7~30 天**。
- 转化"当天到达比例":电商类经验上**约一半到七成**当天到,长尾拖到一周;具体以你自己的看板为准,**不要照搬**。

## 4. 延迟反馈 delayed feedback:搜广推 CVR 建模的头号难题

### 4.1 问题本质:训练时 label 还没"长出来"

这是整章的核心。把场景拉直:

```
时间轴 ──────────────────────────────────────────────────────────►
  T0           T0+1h          T0+1d              T0+5d
  曝光+点击     训练任务跑了     转化真的发生了!     (但模型早训完了)
  click=1       此刻 conv=0      conv 应该=1
  conv=??       (还没转化)
               │
               ▼
        训练用的是 conv=0 这个【假负样本 fake negative】
        ——当下是负,未来会变正
```

**延迟反馈(delayed feedback)= 正向信号(尤其是转化)在触点之后延迟很久才到达,导致训练时刻的 label 偏负。** 后果:

- 模型**系统性低估** pCVR(因为大量未来的正样本,现在被当成负样本喂进去了)。
- 越是"慢转化"的人群/品类,被低估得越狠 → 出价/排序系统性偏差。

### 4.2 为什么这是搜广推特有的"硬骨头"

| 因素 | 说明 |
|------|------|
| 在线学习需求强 | 推荐/广告要快速捕捉新内容、新活动,等不起长窗口 |
| 转化天然慢且长尾 | 下单/激活/付费决策周期长,转化分布有重尾 |
| 正样本本就稀疏 | CVR 量级常在 **个位数百分比甚至更低**(经验值),漏标正样本影响被放大 |
| 出价/计费直接挂钩 | pCVR 偏差直接影响 eCPM、ROI,是真金白银 |

> 🎓 **老师视角**:你跟算法同学聊延迟反馈时,核心矛盾就一句话:**"我想现在就用最新数据训练(时效),但现在的 label 是错的(假负)。"** 等窗口能让 label 变对,但牺牲时效;不等,label 就是错的。所有解法都在这个矛盾里腾挪。平台的价值在于:**提供"先用、后改"的样本基础设施**,让算法既能即时训练,又能在转化到达后回补纠正。

### 4.3 假负样本的代价:一个直觉算例(示意)

> 纯示意,数字为说明用途,非真实统计:

假设真实 CVR = 5%,即 100 次点击有 5 次最终转化。但训练时刻只有 2 次转化已到达(其余 3 次还在路上):

- 训练看到的"观测 CVR" = 2/100 = 2%。
- 模型若直接拟合,会学到偏低的 pCVR ≈ 2%,系统性低估 60%。
- 越是"慢转化品类",这个低估越严重 → 模型会**歧视慢转化人群/商品**,形成恶性循环。

这就是为什么延迟反馈**不能简单忽略**,必须在样本层或模型层校正。

## 5. 延迟反馈的解法全景

解法分两大流派,平台研发要支撑的能力不同。

### 5.1 流派一:等待窗口(wait & label)

最朴素:**等到归因窗口结束,label 定稿了再训练。**

```
等待窗口方案:

  曝光点击发生 ───────[ 等满 N 天窗口 ]───────► label 定稿 ──► 进训练
  T0                                          T0+N

  优点:label 准,无假负
  缺点:训练数据永远"滞后 N 天",对新内容/活动反应慢
```

- **优点**:label 完全正确,实现最简单。
- **缺点**:时效差。对在线学习、实时推荐基本不可接受。
- **平台侧支撑**:只需"延迟 N 天再生成训练集"的调度,样本可以是 append-only,**不需要可变 label**。这是唯一不强依赖样本回补的方案。

> 🎓 **老师视角**:别看不起等待窗口。对**转化快、对时效不敏感**的业务(如离线天级重训、决策周期短的场景),它最稳、最省事。**不要为了用上 DFM 而强行上 DFM。** 工程的第一原则是够用就好。

### 5.2 流派二:即时训练 + 回补/纠正(主流)

立刻用当前(含假负的)样本训练,之后转化到达时再纠正。这一流派又分几种技术路线:

#### (a) Fake Negative + 回补(最直观的工程派)

- 样本先以 negative 进入训练(承认它现在是假负)。
- 转化到达后,**追加一条 positive 样本** 或 **把原样本 label 改写为 positive**。
- 关键工程:**样本可更新 / 样本回流重写**。

```
Fake Negative 回补流:

  T0  点击 → 生成样本 (conv=0) → 即时进训练流          ①先当负样本用
       │
       │  T0+3d  转化到达
       ▼
  归因服务回溯 → 定位原样本 → upsert conv: 0→1         ②回补改写
       │
       ▼
  改写后的样本重新进入训练流(或作为修正信号)          ③纠正模型
```

#### (b) Importance Weighting(重要性加权,经典 DFM 思路)

Chapelle 2014 提出的延迟反馈模型(Delayed Feedback Model, DFM)思路:**联合建模"是否转化"和"转化延迟时间"**。对当前观测为负的样本,根据"它未来还可能转化"的概率做加权校正,而不是简单当 0。

- 平台无需改 label,但要**提供延迟时间字段**(转化距点击多久),供模型建模延迟分布。

#### (c) ESDFM / 后续流派(即时 + 假负校正的工业升级)

ESDFM(Elapsed-Time Sampling Delayed Feedback Model)等:**即时把每条样本先当负样本喂一次,转化到达后再喂一条正样本**,并用重要性采样权重纠正"同一样本被当负又当正"带来的分布偏差。这类方法是工业界流式 CVR 的常见选型。

#### (d) 负样本延迟校正 / 多窗口标定

不立刻定 label,而是**在多个时间窗口分别快照 label**,让模型感知"在 1 天窗口下是负,在 7 天窗口下是正",显式建模窗口与转化的关系。

### 5.3 五种解法对平台的要求对比

| 方案 | label 是否需可变 | 是否需延迟时间字段 | 时效 | 实现复杂度 | 平台交付重点 |
|------|----------------|------------------|------|-----------|------------|
| 等待窗口 | 否(append-only 即可) | 否 | 差 | 低 | 延迟调度 |
| Fake Negative 回补 | **是(核心)** | 建议有 | 好 | 中 | 可变 label 存储 + 回补链路 |
| Importance Weighting / DFM | 否(可加正样本) | **是(必需)** | 好 | 高(偏算法) | 延迟时间字段 + 流式拼接 |
| ESDFM 类 | 是(先负后正双写) | 是 | 好 | 高 | 双写 + 权重计算支持 |
| 多窗口标定 | **是(多快照)** | 是 | 中 | 中 | 多版本 label 列 + 窗口配置 |

> 🎓 **老师视角**:看这张表,你会发现**除了"等待窗口",其余四种都强依赖"label 可改写或多版本"**。所以平台研发只要打算支持主流延迟反馈方案,**"可变 label 的样本存储 + 样本回补链路"就是你必须建的地基**,绕不过去。这也是为什么很多团队从 Hive append-only 样本表,迁移到支持 upsert 的湖格式(如 Paimon 主键表)——本质就是为了让 label 能被后到的转化改写。

### 5.4 平台侧样本回补链路:参考架构

```
            ┌──────────────────────────────────────────────────────┐
            │                    实时样本流                          │
  曝光/点击 →│  日志拼接 → 即时生成样本(conv=0)→ 写样本存储 → 训练   │
            └──────────────────────────────────────────────────────┘
                                   │ 主键: sample_id / trace_id
                                   ▼
            ┌──────────────────────────────────────────────────────┐
            │              可变 label 样本存储(支持 upsert)         │
            │   sample_id | features | click | conv | conv_ts | ver  │
            └──────────────────────────────────────────────────────┘
                                   ▲
                                   │ upsert conv:0→1, 写 conv_ts
            ┌──────────────────────────────────────────────────────┐
  转化流 →  │  归因服务(窗口内回溯 + 口径选触点)→ 回补请求           │
            └──────────────────────────────────────────────────────┘
                                   │
                                   ▼
            ┌──────────────────────────────────────────────────────┐
            │   样本回流:被改写样本重新进入训练流 / 触发增量重训     │
            └──────────────────────────────────────────────────────┘
```

> 🎓 **老师视角**:这套链路有三个你必须盯死的工程点:
> ① **幂等**:同一转化可能因重试被消费多次,回补必须幂等(用 `order_id` 去重),否则 label 被加重或正样本翻倍。
> ② **回补延迟可控**:转化到达后多久能完成回补?这本身是个 SLA,要进看板(第 8 节)。
> ③ **训练消费方要能感知"样本被改写了"**:要么订阅 changelog(Paimon 主键表能产出 changelog),要么定期全量重算。**千万别让训练方还在用改写前的旧 label。**

## 6. 多目标 / 多任务 label 对齐与一致性

### 6.1 多目标 label 为什么难对齐

现代推荐/广告普遍是**多任务学习(MTL)**:一个模型同时预估 CTR、CVR、时长、完播等。这要求一条样本上**同时挂多个 label**:

```
一条样本的多 label,在不同时间陆续到达:

  T0          T0(秒级)    会话结束        T0+5d
  曝光样本 →   click=1  →   dwell=35s  →   conv=1
  生成        立即到       几分钟内到      延迟很久到
  │           │            │              │
  └───────────┴────────────┴──────────────┘
        同一 sample_id,label 字段分时段"长齐"
```

问题:**各 label 反馈速度不同步**。click 秒级到,conv 几天后到。如果你为了等 conv 而推迟整条样本的生成,click 这种快 label 的时效就被拖垮了。

### 6.2 对齐策略与一致性要求

| 策略 | 做法 | 优点 | 代价 | 平台支撑 |
|------|------|------|------|---------|
| 整条等齐再写 | 等所有 label 到齐才生成样本 | 简单一致 | 时效被最慢 label 拖累 | 延迟调度 |
| 分列回补 | 先写快 label,慢 label 后续 upsert | 各 label 各取所需 | 需可变 label + 版本管理 | upsert 存储 + changelog |
| 多样本视图 | 同一行为生成 CTR 样本和 CVR 样本两套 | 各任务样本空间清晰 | 存储翻倍、需保证对齐 | 一致性校验 |

> 🎓 **老师视角**:多任务 label 一致性的"红线"是——**多个 label 必须来自同一条曝光、同一套特征快照**。最怕的是:CTR label 用了 A 时刻的特征,CVR label 回补时特征已经变了(用户画像更新了),结果同一 `sample_id` 的特征和 label 对不上。**特征快照(feature snapshot)必须在曝光时刻冻结并随样本一起存,回补 label 时绝不重新取特征。** 这点和离在线一致性强相关,见 [样本离在线一致性](../04-样本数据管理/样本离在线一致性.md)。

### 6.3 label 一致性的三条平台铁律

1. **特征在曝光时刻冻结**:label 后补,特征绝不后补(否则就是穿越,见第 7 节)。
2. **同一 sample_id 全局唯一且可追溯**:多 label 回补都打到同一行。
3. **label 版本/时间戳可见**:每个 label 带 `事件时间` 和 `回补时间`,便于排查和监控延迟。

## 7. 标签穿越 label leakage:用未来当现在的隐形炸弹

### 7.1 什么是标签穿越

**标签穿越(label leakage / data leakage)= 训练时用到了"预测时刻不可能知道"的未来信息**,导致离线指标虚高、上线崩盘。

延迟反馈场景里,穿越尤其隐蔽,因为 label 本来就是"未来"才到的,稍不注意就把未来信息也带进了**特征**:

```
危险示例(用未来信息当特征):

  曝光时刻 T0 要预估 pCVR
  特征里却混进了:
    - "该用户当天总下单数"  ← 包含了 T0 之后的下单,泄露 label!
    - "该 item 当天转化率"  ← 用了未来转化,泄露!
    - 用归因回补后的 conv 字段去衍生别的特征  ← 直接穿越!

  离线 AUC:0.95 (假高)
  上线效果:暴跌    ← 因为线上预测时根本拿不到这些未来信息
```

### 7.2 延迟反馈与穿越的纠缠

这是本章最微妙的地方:**延迟反馈要求"label 后补",而后补的 label 极易被误用为"特征"。**

```
正确的时间边界:

  ┌─────────────── T0 之前的信息 ───────────────┐ │ ┌── T0 之后 ──┐
  │ 可以当特征(用户历史、item 历史统计)        │T0│ │ 只能当 label │
  └─────────────────────────────────────────────┘ │ └─────────────┘
                                                   预测时刻
  铁律:任何 ts > T0 的信息,只能进 label,绝不能进 feature。
```

### 7.3 平台如何在工程上卡死穿越

| 防线 | 做法 |
|------|------|
| 时间边界校验 | 拼接时强制 `feature.ts ≤ label_event_time`,越界报警 |
| 特征快照冻结 | 曝光时刻冻结特征,后续回补 label 不碰特征(见 6.2) |
| 统计特征 point-in-time 计算 | "近 7 天转化率"等统计量必须按曝光时刻截断,用 point-in-time join |
| label 字段隔离 | 回补的 conv 列打标记,禁止被特征生产流读取 |
| 离线/在线一致性比对 | 同一请求线上线下特征比对,差异即可能穿越或时间错位 |

> 🎓 **老师视角**:当算法同学拿着"离线 AUC 0.92"来邀功,而你知道这个特征是天级统计的,你就该追一句:**"这个统计特征是按曝光时刻截断算的,还是用了全天数据?"** 十有八九能逮到穿越。**平台研发是穿越的最后一道防线**,因为只有你掌握着拼接链路里"谁能在什么时间点看到什么数据"的真相。point-in-time correctness 是你的看家本领。

## 8. label 延迟与回流监控:平台的看板义务

延迟反馈一旦上了"即时训练 + 回补",你就**必须**有看板,否则线上 label 偏了都不知道。

### 8.1 必须监控的核心指标

| 指标 | 含义 | 报警意义 |
|------|------|---------|
| label 回流曲线 | 转化在点击后第 1/2/.../N 天的累计到达比例 | 比例突变 = 上游链路/归因异常 |
| 当前观测正样本率 vs 定稿正样本率 | 即时 label 与最终 label 的差距 | 差距过大 = 假负严重,需调方案 |
| 回补延迟 P50/P99 | 转化到达后多久完成 label 改写 | P99 飙升 = 回补链路堵塞 |
| 回补成功率 / 幂等冲突率 | 回补是否丢失或重复 | 重复 = 正样本被夸大 |
| 窗口外转化比例 | 多少转化掉在窗口外被丢弃 | 过高 = 窗口设太短 |
| 各 label 字段就绪率 | 多 label 中每个的到齐进度 | 监控多任务对齐 |

### 8.2 label 回流曲线(平台看板核心)

```
累计回流比例
100% ┤                              ┌──────────●  ← 第7天接近定稿
     │                      ┌───────┘
 75% ┤              ┌───────┘
     │       ┌──────┘
 50% ┤   ●───┘                  ← 当天约一半(经验值,示意)
     │  ╱
 25% ┤ ╱
     │╱
  0% ●────┬────┬────┬────┬────┬────┬────►
     0    1    2    3    4    5    6    7  天

  用途:① 定窗口的依据;② 估算"即时 label 偏差"幅度;
        ③ 曲线异常立刻报警(如某天回流骤降 = 上游转化日志断了)
```

> 🎓 **老师视角**:这条曲线是你跟算法、跟老板沟通的**通用语言**。算法看它定窗口、估偏差;老板看它判断"我们的 label 体系健不健康"。**建好这条曲线的看板,是延迟反馈项目里平台研发最高 ROI 的交付物之一。** 它把"看不见的 label 偏差"变成了"看得见的一张图"。

### 8.3 监控落到组件

平台研发要交付的监控组件清单:

1. **回流统计作业**:按 `conv_ts - click_ts` 的延迟分桶,产出日级回流曲线。
2. **label diff 作业**:对比"即时样本 label"与"窗口定稿后 label",算偏差率。
3. **回补链路 SLA 监控**:埋点回补请求的端到端延迟、成功率、幂等冲突。
4. **异常报警**:回流曲线突变、回补 P99 超阈、窗口外转化比例异常,接告警系统。

## 9. label 去噪与质量:垃圾进垃圾出

延迟反馈解决的是"label 还没到"的问题;但即使到了,label 本身也可能是脏的。**label 质量是模型上限的真实天花板**,而清洗 label 又是平台研发的活。

### 9.1 label 噪声的常见来源

| 噪声来源 | 表现 | 影响 | 平台处理 |
|---------|------|------|---------|
| 误触 / 误点 | 滑动时误点,秒退 | 假正(click=1 但无意图) | 用停留时长门槛过滤(如 dwell < 阈值不算有效点击) |
| 机刷 / 爬虫 / 作弊流量 | 异常高频点击转化 | 污染正样本分布 | 反作弊名单、设备指纹、频控过滤 |
| 重复曝光 / 重复上报 | 同一行为多条日志 | label 翻倍 | 按业务唯一键去重(幂等) |
| 退款 / 取消订单 | 转化又被撤销 | 假正(conv=1 应改回 0) | 监听退款流,反向回补 conv:1→0 |
| 归因错配 | 关联键不稳,串了别次触点 | label 张冠李戴 | 用 trace_id 精确关联,时间约束兜底 |
| 时长 label 噪声 | 后台播放、挂机 | 时长虚高 | 前台可见性校验、异常值截断 |

### 9.2 退款/取消:被忽视的"反向延迟反馈"

```
正向延迟反馈:  conv  0 ──(下单到达)──► 1
反向延迟反馈:  conv  1 ──(退款到达)──► 0     ←★ 同样需要回补能力!

时间轴:
  T0 点击    T0+2d 下单(conv:0→1)    T0+5d 退款(conv:1→0)
     │            │                        │
     └── 样本 conv 字段在 5 天内被改了两次,平台都要 upsert 跟上
```

> 🎓 **老师视角**:很多团队只做了"正向回补"(下单补成正),却忘了"反向回补"(退款撤回正样本)。结果模型把"下单后退款"的人也当成优质转化学,出价偏高、亏钱。**只要你的存储支持 upsert label,正向反向其实是同一套链路**——区别只是监听的事件流不同(下单流 vs 退款流)。给老板讲的时候,这是个能体现"我们考虑得比别家细"的点。

### 9.3 有效行为定义:label 不是日志里的原始事件

平台要和算法一起把"什么算有效正样本"定义清楚,并固化成口径配置:

- **有效点击**:停留 ≥ N 秒(如 3s)才算,过滤误触。
- **有效转化**:剔除退款/取消/异常订单。
- **有效时长**:前台可见 + 截断极端值(如 > 6 小时按异常处理)。

> 🎓 **老师视角**:这些"有效性门槛"绝不能硬编码在某个 SQL 里。要做成**口径配置中心**:门槛、窗口、归因口径全部可配、有版本。因为算法会反复调试这些阈值,你不能每次都改代码发版。把口径做成配置,是平台研发成熟度的标志。

## 10. 延迟反馈解法的数学直觉(给平台研发的够用版)

你不需要会推公式,但要能听懂算法在说什么,才能"接得住、翻译得了"。这里给够用版直觉。

### 10.1 为什么"直接当负样本"是错的:偏差从哪来

设真实转化概率为 `p`。理想训练应在"最终 label"上做最大似然。但即时训练用的是"观测 label"(转化还没到 → 标 0)。观测分布 ≠ 真实分布,这就是**有偏(biased)**。

```
                 真实正样本(最终会转化)
                 ┌──────────────────────────┐
   即时观测:      ████░░░░░░░░░░░░░░░░░░░░░░░░  ← 只看到已到达的一小部分正样本
                 已到达        还在路上(被当成负)
   模型学到的 p ≈ (已到达正样本) / 全部  <  真实 p     → 系统性低估
```

### 10.2 三类纠偏思路的直觉

| 方法 | 一句话直觉 | 对平台的诉求 |
|------|-----------|------------|
| Importance Weighting(DFM) | "这条现在是负,但它有 q 的概率未来会转化,所以给它的负样本损失打个折扣"——用权重把有偏分布纠回无偏 | 提供延迟时间字段,让模型估计延迟分布 |
| Fake Negative Weighting(FNW) | "所有样本先当负样本进一次(承认有假负),转化到达后再当正样本进一次,用重要性权重抵消重复" | 先负后正**双写**样本 |
| Fake Negative Calibration(FNC) | 类似 FNW,但侧重对即时模型输出做**校准(calibration)**而非改训练流 | 即时样本 + 后验校准数据 |

### 10.3 DFM 的核心:把"延迟"也当成要学的东西

Chapelle 的延迟反馈模型同时学两件事:

1. `P(转化 | 特征)` —— 最终会不会转化。
2. `P(延迟时间 | 特征, 转化)` —— 如果会转化,多久之后转(通常假设指数分布)。

有了第 2 个,模型就能推断:"这条样本现在标 0,但根据它的延迟分布,它大概率只是还没转化,而不是真的不会转化",从而修正损失。

```
                    ┌─────────────────────┐
   特征 X ─────────►│  转化头 P(C|X)        │──► 是否会转化
                    ├─────────────────────┤
                    │  延迟头 P(D|X,C=1)    │──► 转化延迟分布(如指数)
                    └─────────────────────┘
   平台只要交付:每条转化样本带 D = conv_ts - click_ts(延迟时长)
```

> 🎓 **老师视角**:你只要记住一句话翻译给老板/自己:**"DFM 不仅学'会不会转化',还学'多久才转化',所以它能在转化没到时就猜出'这只是没到、不是不转'。"** 而你作为平台,唯一硬交付就是**那个延迟时长字段 `conv_ts - click_ts`**。没有这个字段,DFM 这条路根本走不了。这就是为什么前面 Checklist 把"延迟时间字段"列为必交付项。

## 11. 样本回补的存储实现:以 Paimon 主键表为例

延迟反馈所有方案的地基都是"可变 label 存储 + 通知下游"。本节给一个具体落地形态,帮你把抽象需求映射到真实组件。本知识库依托 Apache Paimon,正好它的主键表(primary-key table)天然适配。

### 11.1 为什么 append-only 表扛不住延迟反馈

```
传统 Hive append-only 样本表:
  写入即不可改 → 转化后到 → 只能"再追加一行" → 同一曝光出现两行
  → 训练方看到两条冲突 label,要自己去重/取最新 → 复杂且易错

Paimon 主键表(以 sample_id 为主键):
  写入即 upsert → 转化后到 → upsert 同一主键 → label 列原地更新
  → 训练方永远看到"每个 sample_id 一行、最新 label" → 干净
```

### 11.2 样本表 schema 设计(示意)

```
CREATE TABLE sample_table (
  sample_id     STRING,        -- 主键:全局唯一(=trace_id)
  event_time    TIMESTAMP,     -- 曝光时刻 T0(分区/水位用)
  features      BYTES,         -- ★曝光时刻冻结的特征快照(绝不后改)
  click_label   INT,           -- 秒级到达
  conv_label    INT,           -- 延迟到达,可被 upsert 改写(0→1→0退款)
  conv_ts       TIMESTAMP,     -- 转化时刻(算延迟用)
  label_version INT,           -- 回补版本号,每次改写 +1
  PRIMARY KEY (sample_id) NOT ENFORCED
) PARTITIONED BY (dt);
```

要点对照前面铁律:
- `features` 一次写死,**回补只动 `conv_label` / `conv_ts` / `label_version`,绝不重写 features**(防穿越,见第 7 节)。
- `PRIMARY KEY (sample_id)` 让后到的转化用 upsert 命中同一行。
- `label_version` 让下游和看板能识别"这条被回补过几次"。

### 11.3 回补写入与下游通知

```
实时链路:
  曝光点击 → 写 (sample_id, features, click=1, conv=0, ver=0)
                                │
  转化流到达 → 归因服务定位 sample_id
                                ▼
            upsert (sample_id, conv=1, conv_ts=..., ver=1)   ← 只带要改的列
                                │
                                ▼ Paimon 产出 changelog(变更流)
  训练消费方订阅 changelog → 感知"这条 label 变了" → 增量更新/重训信号
```

> 🎓 **老师视角**:这里有个常被忽视的关键——**部分列更新(partial-update)**。回补时你只想改 `conv_label`,不想碰 `features`。Paimon 的 partial-update 合并引擎(merge-engine)正好支持"只更新指定列,其余列保留"。如果用错合并引擎(比如 deduplicate 整行覆盖),你回补时如果没带上 features,就会把特征冲掉成 null——这就是事故。**选对 merge-engine 是这条链路的隐形地雷,务必和数据同学确认。** Paimon 主键表的 changelog 能力(changelog-producer)则解决了"下游怎么知道 label 改了"的通知问题,这两个能力配合起来,正是延迟反馈回补链路的理想底座。

### 11.4 回补幂等的工程实现

```
幂等三件套:
  ① 业务唯一键去重:用 order_id 维护"已处理转化"集合,重复转化丢弃
  ② upsert 天然幂等:同一 sample_id 重复 upsert 同值,结果一致
  ③ 版本/时间戳护栏:回补带 conv_ts,若来的事件比已记录的旧则忽略(防乱序)
```

> 🎓 **老师视角**:消息队列(Kafka 等)的语义通常是 at-least-once,转化事件**一定会重复**。如果你的回补不是幂等的,正样本就会被多记 → CVR label 虚高 → 模型高估 → 出价过高烧钱。**幂等不是"锦上添花",是延迟反馈链路的生死线。** 把"幂等冲突率"放进看板(第 8 节)就是为了兜住这条线。

## 12. label 体系演进路线图(平台建设三阶段)

不要一上来就追求 ESDFM 全套。label 体系是演进出来的。给你一条务实的三阶段路线,也方便向上沟通要资源。

### 12.1 三阶段对比

| 阶段 | 形态 | label 存储 | 延迟反馈处理 | 适合谁 | 主要工作量 |
|------|------|-----------|------------|--------|----------|
| 阶段一:能用 | 等待窗口 + append-only | 不可变 | 等窗口定稿再训练 | 起步期、转化快、天级重训 | 调度 + 基础 join |
| 阶段二:够快 | 即时训练 + 回补 | 可变 upsert(主键表) | Fake Negative 回补 + changelog | 有实时推荐、CVR 偏低明显 | 归因服务 + 回补链路 + 看板 |
| 阶段三:够准 | DFM/ESDFM + 多窗口标定 | 多版本 label + 延迟字段 | 重要性加权 / 先负后正双写 | 大规模广告、CVR 是核心收入 | 延迟字段 + 双写 + 纠偏支持 |

### 12.2 演进时序图

```
能用 ───────────► 够快 ───────────► 够准
阶段一            阶段二              阶段三
│                 │                   │
├ 等待窗口         ├ 上主键表(upsert) ├ 加 conv_ts 延迟字段
├ append 样本表    ├ 建归因服务         ├ 先负后正双写
└ 天级重训         ├ 回补链路 + 幂等     ├ 支持 DFM/ESDFM 纠偏
                  ├ label 回流看板      └ 多窗口 label 标定
                  └ 防穿越校验

判断升级时机:回流曲线显示"假负偏差"显著 + 业务要时效 → 一升二
            CVR 是核心收入、偏差影响出价 ROI → 二升三
```

> 🎓 **老师视角**:这张路线图是你向上要资源的"作战地图"。**关键判断点不是"技术先进",而是"假负偏差有多痛 + 业务多需要时效"。** 你拿回流曲线(第 8 节)证明"假负偏差有 X%、正在让 CVR 系统性低估",老板自然批阶段二的投入。**用数据驱动演进,别用"别家都上了"驱动。** 而且阶段二的"可变 label 存储 + 回补 + 看板"是后续一切的地基,一次建好长期复用——这是最高 ROI 的投资,要重点争取。

## 业界怎么做

> 以下为公开技术方案的**方向性**总结,作直觉锚点,细节以各家论文/分享为准。

- **阿里(ESMM / ESMM2)**:在**全曝光空间**联合建模 CTR 与 CVR,用 `pCTCVR = pCTR × pCVR` 监督,解决 CVR 样本选择偏差。对平台的要求是提供曝光骨架 + 多 label 对齐样本。
- **延迟反馈建模(DFM,Criteo / Chapelle 2014)**:联合建模"是否转化"与"转化延迟分布",对当前为负的样本做重要性加权校正。需要平台提供**延迟时间字段**。
- **ESDFM / FNW / FNC(腾讯、阿里等流式 CVR 方案)**:即时把样本当负样本喂入,转化到达后补正样本,并用重要性采样纠偏。需要平台支持**先负后正双写 + 样本回补**。
- **多触点归因(MTA)**:多用于**预算分配/效果评估**,而非直接造训练 label;训练 label 仍以 last-click 为主。
- **湖格式做可变样本**:不少团队用支持主键 upsert 的湖格式(如 Apache Paimon、Hudi)承载样本表,以 `sample_id` 为主键支持 label 回补改写,并用 changelog 通知下游训练消费方。
- **Feature Store + point-in-time**:统一特征快照与 point-in-time join,防止统计特征穿越,保障离在线一致。

## 权衡取舍

| 决策点 | 选项 A | 选项 B | 怎么选 |
|-------|--------|--------|--------|
| label 时效 vs 准确 | 即时训练(有假负) | 等待窗口(label 准) | 时效敏感→A+回补;转化快→B |
| 归因口径 | last-click(简单) | multi-touch(全面) | 训练 label 选 A;归因分析选 B |
| 窗口长度 | 短(时效高) | 长(完整) | 看转化延迟分布,用回流曲线定 |
| 样本存储 | append-only(简单) | 可变 upsert(支持回补) | 要做延迟反馈→必须 B |
| 多 label 对齐 | 整条等齐(一致) | 分列回补(时效) | 多任务+在线学习→分列回补 |
| 回补通知下游 | 订阅 changelog(实时) | 定期全量重算(简单) | 在线学习→changelog;天级→全量 |

> 🎓 **老师视角**:所有权衡的轴心只有一根:**时效 vs 正确**。延迟反馈这件事,本质就是"我能不能先用着错的、再慢慢改对"。平台的全部工程价值,就是把这个"先用后改"做得幂等、可监控、对下游透明。把这根轴想透,具体方案怎么选你自己就有判断了。

## 常见误区与反直觉

1. **误区:label 是算法的事,平台只管搬数据。**
   反:label 的正确性/时效性/可回补性是平台的工程指标。label 系统性偏 5%,模型救不回来。

2. **误区:转化到了就直接覆盖 label 就行。**
   反:不幂等的回补会让正样本翻倍/偏多;必须用 `order_id` 去重、保证幂等。

3. **误区:窗口越长越好,转化抓得全。**
   反:窗口越长,假负越严重、样本定稿越慢、时效越差。是旋钮,不是越大越好。

4. **误区:延迟反馈一定要上 DFM / ESDFM。**
   反:转化快、对时效不敏感的业务,等待窗口最稳最省。别为炫技上复杂模型。

5. **反直觉:回补 label 时去重新取最新特征,数据更"新"应该更好。**
   反:这是典型穿越!特征必须在曝光时刻冻结,回补只动 label,绝不动特征。

6. **反直觉:离线 AUC 越高越好。**
   反:AUC 异常高常常是穿越的信号(用了未来统计特征),不是模型变强了。

7. **误区:CVR 就在点击样本上训练。**
   反:那会有样本选择偏差;ESMM 在全曝光空间建模,平台要提供曝光骨架对齐样本。

8. **误区:多个 label 各写各的就行。**
   反:多 label 必须挂在同一 `sample_id`、同一特征快照上,否则特征与 label 对不齐。

## 平台侧落地 Checklist

交付延迟反馈/label 体系时,逐条对照:

- [ ] **样本主键**:每条样本有全局唯一、可追溯的 `sample_id` / `trace_id`。
- [ ] **可变 label 存储**:支持按主键 upsert label(从 0 改 1),如 Paimon 主键表。
- [ ] **特征快照冻结**:曝光时刻冻结特征并随样本存储,回补时绝不重取。
- [ ] **归因服务**:窗口内回溯触点,口径(last-click 默认)可配。
- [ ] **归因窗口可配**:按业务线/label 类型配置不同窗口。
- [ ] **回补链路幂等**:用业务唯一键去重,防止 label 重复回补。
- [ ] **延迟时间字段**:存 `conv_ts`、`conv_ts - click_ts`,供 DFM 类模型用。
- [ ] **多 label 对齐**:同一 sample_id 挂多 label,带各自事件时间。
- [ ] **下游通知**:回补改写通过 changelog / 重算通知训练消费方,防止用旧 label。
- [ ] **穿越防线**:拼接强制 `feature.ts ≤ T0`;统计特征 point-in-time 计算。
- [ ] **label 回流看板**:回流曲线、即时 vs 定稿偏差、回补 SLA、窗口外比例。
- [ ] **异常报警**:回流突变、回补 P99 超阈、幂等冲突接告警。
- [ ] **离在线一致性比对**:同请求线上线下特征 diff,见离在线一致性文档。
- [ ] **退款/取消反向回补**:监听退款流,把已转化样本 conv 反向改回 0(见第 9.2 节)。
- [ ] **合并引擎选型确认**:回补用 partial-update,确保只改 label 列、不冲掉特征(见第 11.3 节)。
- [ ] **有效行为口径配置化**:点击/转化/时长的有效性门槛做成可配版本,不硬编码(见第 9.3 节)。

## 对接话术

**当算法同学说"线上 CVR 一直预估偏低":**
> "先别改模型。我拉一下 label 回流曲线——很可能是延迟反馈,当前样本里有一批假负样本(转化还没回流)。我们要么拉长定稿窗口,要么上回补链路把后到的转化补成正样本。你想要哪种?我这边样本表已经支持 upsert label。"

**当算法同学要"全曝光空间的 CVR 样本"(ESMM):**
> "可以。我给你曝光骨架 + click label + conversion label 三者对齐到同一 sample_id 的样本,CVR 不切点击子集,在全空间监督。延迟反馈的 conv 我用回补链路补,带 conv_ts 字段给你建模延迟。"

**当算法同学问"能不能给我'用户当天总转化数'当特征":**
> "这个要小心穿越。如果是预估时刻 T0 之后的转化也算进去,就泄露 label 了。我只能给你按 T0 截断的 point-in-time 统计,离线 AUC 可能没那么好看,但上线才扛得住。"

**当数仓/上游说"转化日志延迟到一两天很正常":**
> "理解。那我们的归因窗口和 label 定稿就得据此设。我把回流曲线监控上,你们链路一旦断流或延迟突变,我这边告警能立刻发现,别等模型效果掉了才查。"

## 向上沟通

**老板问:"我们 CVR 模型为什么没别人准?"**
> "很可能不是模型差,是 label 体系。搜广推 CVR 有个公认难题叫延迟反馈:转化发生在曝光后好几天,训练时还没回来,模型当成'没转化'学,系统性低估。我们已经建了三块地基:① 样本 label 可回补(转化后到自动补成正样本);② label 回流看板(偏差看得见);③ 防穿越校验(离线指标不虚高)。这让我们的 label 比纯等窗口的方案既快又准。"

**老板问:"这块要不要继续投入?"**
> "建议投。label 是模型的天花板——特征和模型再好,label 偏了全白搭。延迟反馈的回补链路和监控看板是一次性建好、长期受益的基础设施,所有 CVR/转化类业务共用。ROI 很高,而且是别家容易忽视、我们能拉开差距的地方。"

**一句话汇报:**
> "我们把'看不见的 label 偏差'变成了'看得见的回流曲线 + 自动回补',让 CVR 这类延迟转化场景的样本既快又准。"

```
│                                    │                                  │
│                                    ▼                                  │
│   ┌──────────────── 平台必须交付的四件套 ────────────────┐           │
│   │ 1 可变 label 样本存储(upsert)                        │           │
│   │ 2 归因计算服务(窗口+口径可配)                        │           │
│   │ 3 延迟反馈回补链路(幂等+通知下游)                    │           │
│   │ 4 label 延迟回流看板(偏差可见+报警)                  │           │
│   └────────────────────────────────────────────────────┘           │
│                                                                       │
│   穿越红线:feature.ts ≤ T0,label 后补、特征绝不后补                 │
└─────────────────────────────────────────────────────────────────────┘
```

**一句话速记:**
> **label 是模型的天花板;延迟反馈让 label "先错后对",平台的活就是把"先用后改"做得幂等、可监控、不穿越。**

## 自测三问

**Q1:为什么延迟反馈会让 CVR 模型系统性低估?平台用什么方案纠正?**

<details>
<summary>参考答案</summary>

转化(conversion)往往在曝光/点击后很久才发生(电商 1~7 天、广告激活更长)。训练发生在转化到达之前,所以那些"未来会转化"的样本在当下被标成 0(假负样本 fake negative)。模型拟合这批偏负的 label,就会把 pCVR 学低,且越慢转化的人群被低估越狠。

平台纠正方案:① **等待窗口**——等 label 定稿再训练(准但慢);② **即时训练 + 回补**——先以负样本进训练,转化到达后由归因服务把对应样本 label upsert 改成 1(或补一条正样本),配合 DFM/ESDFM 等重要性加权纠偏。主流是 ②,前提是平台提供**可变 label 存储 + 幂等回补链路 + 下游通知**。
</details>

**Q2:归因口径(last-click / multi-touch)和归因窗口分别影响什么?平台要交付什么?**

<details>
<summary>参考答案</summary>

- **归因口径**决定"一次转化算给哪次触点",即哪条样本被标成正样本。训练 label 主流用 last-click(一一对应、样本好造);multi-touch 多用于归因分析/预算分配。口径变了,正样本归属和 label 分布都会变。
- **归因窗口**决定"触点之后多久内的转化才算数"。窗口越长,捕获转化越全、正样本越真实,但 label 定稿越慢、假负越严重、时效越差。是"完整性 vs 时效性"的旋钮。

平台交付:**归因计算服务**(回溯触点、口径可配)、**窗口可配置**(按业务线/label 类型)、以及**label 回流曲线看板**作为定窗口和估偏差的依据。
</details>

**Q3:延迟反馈要求 label 后补,这和标签穿越有什么关系?平台怎么守住边界?**

<details>
<summary>参考答案</summary>

延迟反馈要求"label 后补",而后补的 label(及其衍生的统计量)极易被误当成特征喂进模型,造成**标签穿越**:用了预测时刻 T0 之后才知道的信息,离线 AUC 虚高、上线崩盘。

铁律:**任何 ts > T0 的信息只能进 label,绝不能进 feature。** 平台守边界的手段:① 曝光时刻**冻结特征快照**,回补只改 label、绝不重取特征;② 拼接强制 `feature.ts ≤ T0`,越界报警;③ 统计类特征(如"近 7 天转化率")必须 **point-in-time** 按曝光时刻截断计算;④ 回补的 conv 列隔离打标,禁止被特征生产流读取;⑤ 离在线特征一致性比对,差异即可能穿越。
</details>
