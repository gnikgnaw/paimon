## 0. 30 秒速通（TL;DR）

1. **特征 = 模型的输入信号**。搜广推里特征不是一两个 Excel 列，而是动辄成百上千维、混合多种类型、来自多个数据源、有不同时效要求的「信号集合」。
2. **八大类要刻进脑子**：数值、类别 ID、统计滑窗、行为序列、交叉、embedding、上下文、画像。它们的**存储格式、更新频率、量级、计算位置**完全不同——这正是平台研发要分别支撑的东西。
3. **平台研发的核心抓手不是「算法准不准」，而是四件事**：① 用什么**存储格式**（定长数值 vs 变长序列 vs 高基数 ID）；② **更新频率**（秒级/分钟级/天级）；③ **计算位置**（在线实时算 vs 离线批算）；④ **防穿越**（特征的时间点必须早于 label 的时间点）。
4. **深度模型也常需要分桶**。不是「上了深度学习就不用特征工程」——连续值分桶后转 embedding，往往比直接喂裸数值更稳更好。这是个反直觉但极其常见的事实。
5. **最容易出事的两类特征是统计特征和序列特征**：它们都依赖「截至某时刻的历史」，一旦离线计算时不小心用了未来数据，就是**特征穿越（label leakage）**，线下 AUC 虚高、线上暴跌。平台必须在生产链路上把「时间点对齐」做成基础设施，不能指望每个算法同学手动小心。
6. **记一句话**：你不需要会调模型，但你要让每一类特征都「存得下、更得动、算得对、不穿越」。

> 本文是「特征数据管理」系列的第一篇，先把**特征本身**讲透——有哪些类、怎么变换、各自对平台意味着什么。
> 后续 [02-特征平台(Feature Store)架构](../03-特征数据管理/02-特征平台(Feature Store)架构.md) 讲怎么把这些落成系统，[03-特征生产链路与特征穿越治理](../03-特征数据管理/03-特征生产链路与特征穿越治理.md) 专门讲防穿越。

## 1. 为什么平台研发必须懂特征分类

很多平台同学的误区是：「特征是算法同学的事，我只管把数据存进去、查出来就行。」

这在小公司、单一模型场景也许成立。但在搜广推这种**特征驱动**的系统里，不懂分类你会在三个地方翻车：

- **存储选型翻车**：你把一个高基数稀疏 ID 特征（比如 user_id，基数上亿）当成普通字符串列存进宽表，结果存储爆炸、查询拖垮。它本该走 embedding table 或哈希分桶。
- **时效需求翻车**：算法说「我要实时点击序列」，你按天级离线表给他，模型上线后发现「用户刚点的东西下一刷推荐里完全没反应」，效果差一大截。
- **穿越事故翻车**：你帮算法同学拼了一张训练样本表，把「用户当天总点击数」直接 join 进去，结果这个数包含了 label 发生之后的点击——线下 AUC 0.85，线上掉到 0.68，全组排查一周才发现是你拼表穿越了。

```
        算法/训练同学                平台研发                  老板
        ─────────────                ──────────                ────
   「我要近7天CTR特征」  ──►  「这是统计滑窗特征,        「为什么上这个
   「点击序列定长50」          需要离线批计算+防穿越,      特征要花两周?」
   「user/item embedding」     在线KV查询;序列要变长存储」      ▲
            ▲                          │                        │
            └──────────────────────────┴────────────────────────┘
                你是中间的「翻译层」+「能力供给层」
                听得懂需求 → 翻译成存储/计算/时效 → 向上解释成本
```

🎓 **老师视角**：平台研发对特征的理解深度，直接决定了你是「被动接需求的工具人」还是「能主动给算法同学设计方案的合伙人」。前者每次都是算法说啥你做啥、做错了背锅；后者能在需求评审时就指出「这个特征会穿越」「这个量级你们扛不住在线计算」。本文目标是把你变成后者。

### 1.1 搜广推特征到底有多大规模（建立量级直觉）

平台同学最需要先有的是「数量级直觉」，否则做容量规划全靠拍脑袋。给一组经验值（不同公司差异极大，仅建立直觉）：

| 维度 | 小厂/单业务（典型量级） | 大厂头部业务（典型量级） |
|---|---|---|
| 特征条数（feature 个数） | 几十 ~ 几百 | 几百 ~ 几千 |
| 单次请求拼的特征值数（含展开后） | 几百 | 几千 ~ 上万 |
| sparse ID 总基数（所有 ID 特征加起来） | 千万级 | 百亿 ~ 千亿级 |
| embedding 参数量 | 几百 MB ~ 几 GB | 几十 GB ~ TB 级 |
| 在线特征查询 QPS | 千 ~ 万 | 十万 ~ 百万 |
| 在线单次特征拉取延迟预算 | 个位数 ~ 几十 ms | 严格控制在 10ms 级 |

这组数字告诉平台研发三件事：① **sparse 特征（高基数 ID）是存储和参数量的大头**，不是数值特征；② **在线延迟预算极紧**，特征拉取慢一点点就影响整个推荐链路超时；③ **请求时要并发拉成百上千个特征**，这决定了在线特征存储必须是低延迟 KV + 批量查询能力。

### 1.2 特征的「四象限」：平台研发的心智模型

把特征按「在线/离线」和「实时变/慢变」分成四象限，这是平台研发最实用的心智模型：

```
                    实时变(秒/分钟级)
                          ▲
        在线序列拼接       │      实时统计(近1h CTR)
        当前session行为    │      热点item曝光数
                          │
  在线 ◄───────────────────┼───────────────────► 离线
                          │
        item静态属性       │      用户长期画像
        类目/价格档        │      近30天消费(T+1)
                          ▼
                    慢变(天级)
```

- **右下（离线慢变）**：最好做，T+1 批跑 + KV 缓存，成本低。画像、长期统计在这。
- **左上（在线实时变）**：最难做，要常驻流计算 + 实时存储，成本高。实时统计、序列在这。
- 平台研发的本能反应应该是：**能往右下挪就往右下挪**。一个特征如果天级更新够用，绝不要做成实时——实时的工程成本可能是天级的 10 倍。要逼着算法同学回答「你这个特征真的需要实时吗，慢一天会掉多少点？」

## 2. 特征分类体系：八大类全景

先给一张总表，这是全文的骨架，建议截图贴工位。后面每个小节展开。

| 特征类别 | 典型例子 | 数据形状 | 基数/量级（典型） | 更新频率（典型） | 计算位置 | 存储建议 |
|---|---|---|---|---|---|---|
| 数值（连续） | 价格、停留时长、年龄 | 标量 float | 低（值域连续） | 天级~实时 | 离线为主 | 宽表数值列 |
| 类别（离散 ID） | city、品类、user_id | 整数 ID | 几十 ~ 上亿 | 天级（属性）/实时（行为） | 离线+在线 | ID 映射表 + embedding table |
| 统计（滑窗） | 近7天点击数、近1h CTR | 标量/小向量 | 中 | 分钟级~小时级 | 离线批+在线增量 | 时序聚合表/KV |
| 序列（行为） | 最近50个点击item | 变长数组 | 长度数十~数百 | 实时 | 在线拼接为主 | 变长 list / 行为日志 |
| 交叉（组合） | 用户性别×品类 | 组合 ID | 笛卡尔积可爆炸 | 跟随源特征 | 离线预计算 or 模型内 | 哈希后 ID |
| Embedding | user/item 向量 | 稠密向量 | 维度 8~256 | 天级（重训）~分钟级（增量） | 离线训练+在线查 | embedding table / 向量库 |
| 上下文 | 请求时间、场景、GPS | 标量/类别 | 低~中 | 实时（请求时产生） | 在线生成 | 请求上下文，不落库 |
| 画像（profile） | 用户长期偏好、item 标签 | 混合（数值+ID+向量） | 大宽表 | 天级（T+1） | 离线批 | profile 宽表/KV |

> 「典型量级」均为经验值，不同公司差异巨大，仅用于建立直觉。

### 2.1 数值特征（连续）

**是什么**：可以比大小、做算术的连续值。比如商品价格、用户年龄、视频时长、停留秒数、历史成交金额。

**对平台意味着什么**：
- **存储**：最省事的一类，普通宽表里一个 `float`/`double`/`int` 列即可。
- **更新频率**：属性类（如年龄）天级足够；行为派生类（如「当前停留时长」）可能实时。
- **坑**：值域差异大（价格 0.1~99999，年龄 0~120），不归一化会让模型训练不稳；缺失值（NULL）必须有约定的填充策略，不能裸 NULL 进模型。

🎓 **老师视角**：连续特征看着最简单，但它有个反直觉点——**深度模型里连续特征常常不是直接喂的，而是先分桶再转 embedding**（见 [3](#3-特征变换让原始数据变成模型能吃的形状) 和 [常见误区](#常见误区与反直觉)）。所以平台要支持「同一个数值列，既能原值导出，也能按分桶规则离散化导出」。别以为存个 float 就完事了。

**数值特征的隐藏坑——量纲与单位漂移**：同一个「金额」特征，有的上游给的是「元」、有的给的是「分」，差 100 倍。模型对这种漂移极其敏感（归一化统计量是基于旧量纲算的，新数据换了量纲直接失真）。平台研发要在特征入口做**单位/量纲的契约校验**，并监控数值特征的分布（均值、分位点）是否突变（PSI 漂移告警）。一个上游悄悄把单位从元改成分，下游所有用这个特征的模型一起出问题，是真实发生过的事故。

### 2.2 类别特征（离散 ID）

**是什么**：取值是离散的、互相之间没有大小关系的标识。又分两种：
- **低基数类别**：city（几千）、性别（2~3）、品类（几百~几千）、星期几（7）。
- **高基数 ID**：user_id（上亿）、item_id（千万~亿）、广告 creative_id。

**对平台意味着什么**：
- **存储**：类别要做 **ID 映射**（feature → 整数编号，即 vocabulary/词表），高基数 ID 通常配 embedding table，不能当字符串硬存进模型。
- **量级**：低基数类别可以 one-hot 或小 embedding；高基数 ID 必须 embedding（one-hot 上亿维不现实）或 **哈希分桶**（hash trick，把上亿 ID 映射到固定桶数，牺牲一点碰撞换存储）。
- **更新频率**：item 属性类天级；新 user/item 不断产生，**词表会膨胀**，平台要处理「冷启动 ID」「OOV（out-of-vocabulary）」。

```
高基数 ID 的两条路:
  ┌─────────────┐
  │ user_id 1亿 │
  └──────┬──────┘
         ├──► 路线A: 全量词表 + embedding table
         │     优点:无碰撞,效果好  缺点:词表巨大,新ID需更新
         │
         └──► 路线B: hash(user_id) % N (如 N=1000万)
               优点:固定大小,新ID自动落桶  缺点:碰撞,两个用户共享向量
```

🎓 **老师视角**：高基数 ID 是搜广推区别于普通机器学习的标志性特征。平台研发要重点关注**词表（vocabulary）的生命周期**：谁来生成、多久更新、新 ID 怎么进、过期 ID 怎么淘汰、在线和离线词表如何一致。词表不一致是线上线下不一致的常见根因（详见 [04-样本数据管理/样本离在线一致性](../04-样本数据管理/样本离在线一致性.md)）。

**词表生命周期五件事（平台研发的活清单）**：

| 阶段 | 问题 | 平台要提供的能力 |
|---|---|---|
| 生成 | 从哪些数据、按什么频率统计出 ID→编号映射 | 离线批扫历史，按频次过滤低频 ID |
| 下发 | 词表怎么同步到训练侧和在线服务侧 | 版本化下发，训练/在线用同一版本号 |
| 新增（OOV） | 训练时没见过的新 ID 怎么办 | 预留 OOV 桶或哈希兜底，不能崩 |
| 淘汰 | 长期不活跃的 ID 占着 embedding 怎么清 | 按活跃度/频次定期淘汰，回收 embedding 槽位 |
| 一致性 | 离线编号和在线编号是否一致 | 同一份词表 + 同一版本号，校验对齐 |

**哈希 trick 的碰撞到底有多严重？** 经验上，把基数 N 的 ID 哈希到 M 个桶，当 M 远大于实际活跃 ID 数（如 10 倍）时碰撞影响小；当 M 和活跃数接近时碰撞严重，效果掉点明显。平台研发要能给算法同学算这笔账：「你 5000 万活跃 user，哈希到 1000 万桶，平均每桶 5 个用户共享一个向量，掉点风险高；建议至少 5000 万桶或上全量词表。」

### 2.3 统计特征（滑窗 count/sum/ratio）

**是什么**：在一个**时间窗口**内对行为做聚合。比如：
- 近 7 天某用户点击数（count）
- 近 1 小时某 item 曝光数（count）
- 近 30 天某用户对某品类的购买金额（sum）
- 近 1 天某 item 的 CTR = 点击 / 曝光（ratio）

**对平台意味着什么**：这是平台研发**工作量最大、最容易出穿越事故**的一类，单列 [4](#4-统计特征滑窗特征深挖时效与穿越风险) 详讲。核心：
- **计算**：离线批（Spark/Flink 批）算长窗口，在线增量（Flink 流）算短窗口。
- **时效**：近 1 小时 CTR 这种要分钟级甚至秒级更新；近 30 天金额天级即可。
- **穿越**：统计窗口的右边界**必须严格小于** label 时间，否则把「未来」算进了「现在」。

🎓 **老师视角**：统计特征常有「多个维度组合」的爆发式增长——「用户×品类近7天点击」「用户×店铺近7天购买」「item×城市近1天曝光」……每加一个维度组合就是一组新特征，生产成本线性甚至超线性增长。平台研发要把统计特征生产做成**配置化的模板**（指定主体、窗口、聚合函数、过滤条件就能产出一组特征），而不是每来一个需求手写一段 SQL/Flink job——后者会让你淹没在无穷无尽的「再加一个统计特征」需求里。模板化 + 防穿越内建，是统计特征生产的正确工程姿势。

### 2.4 序列特征（用户行为序列）

**是什么**：一个用户**按时间排序**的行为列表，是变长数组。比如「最近 50 个点击的 item_id」「最近 20 个购买的品类」。

**对平台意味着什么**：
- **存储**：变长 list，不能用定长数值列。常见做法是存成数组类型，或单独的行为日志按 user 聚合。
- **形状处理**：模型需要**定长**输入，所以平台/特征处理层要做**截断（truncate，取最近 N 个）**和 **padding（补零到 N）**。
- **量级**：序列长度典型 20~200；超长序列（如阿里 SIM 做到上千甚至上万）需要专门的检索式建模和存储。
- **更新**：实时——用户刚点的东西要尽快进序列，否则 DIN/DIEN 这类模型失去意义。

单列 [5](#5-行为序列特征深挖dindien-为什么需要它) 详讲。

### 2.5 交叉特征（组合）

**是什么**：把两个或多个特征组合成一个新特征。比如「用户性别 × 商品品类」「城市 × 时段」。组合能捕捉单特征捕捉不到的交互信号（女性 + 化妆品的点击率远高于各自单独看）。

**对平台意味着什么**：
- **人工交叉**：离线预计算成新的组合 ID 列，存进宽表。问题是**笛卡尔积爆炸**（性别 3 × 品类 5000 = 1.5 万还好；user_id × item_id 直接上天文数字，必须哈希）。
- **模型自动交叉**：FM/DCN/DeepFM 这类模型在网络内部自动学交叉，平台**不用预计算**，只要把原始特征喂进去。
- 平台要回答的关键问题：**这个交叉到底要不要我预计算？** 见 [6](#6-交叉特征深挖人工交叉-vs-模型自动交叉)。

### 2.6 Embedding 特征

**是什么**：把高基数 ID 或一段文本/图像，映射成一个低维稠密向量（如 32 维 float）。embedding 是「学出来的特征」，本身就编码了语义相似性。

**对平台意味着什么**：
- **来源**：① 模型训练副产物（embedding table，随模型一起训）；② 预训练向量（如 graph embedding、双塔召回向量、文本向量）。
- **存储**：embedding table 可能很大（1 亿 ID × 32 维 × 4 字节 ≈ 12.8 GB，仅一个特征）；在线服务用 KV 或向量库存取。
- **更新**：随模型重训（天级）或增量训练（分钟级）更新；召回向量库可能近实时更新。
- 这里和向量检索强相关，详见关联阅读里的特征平台架构。

🎓 **老师视角**：embedding 是平台研发最容易低估存储成本的特征。一个看似普通的「item embedding」乘上亿级基数就是几十 GB，还要乘以「多个模型各自一份」「在线副本数」。做容量规划时一定要把 embedding table 单独拎出来算。

**embedding table 容量速算公式**（平台研发必备）：

```
单表大小 ≈ ID基数 × embedding维度 × 每参数字节数
例: 1亿 user × 32维 × 4字节(fp32) ≈ 12.8 GB
    再 × 5个模型各一份 ≈ 64 GB
    再 × 在线3副本 ≈ 192 GB 在线内存
```

由此可推出平台研发省 embedding 成本的几个杠杆：
- **降维**：32 维降到 16 维直接砍一半（但可能掉点，要 A/B）。
- **降精度**：在线用 fp16 甚至 int8 量化，砍一半到四分之三内存。
- **淘汰低频 ID**：长尾 ID 占了大量槽位但贡献极小，按频次淘汰能显著瘦身。
- **多模型共享 embedding**：相近任务共用一份 embedding table，而不是各训各的。

🎓 **老师视角**：embedding 的**版本一致性**是另一个隐形坑。embedding table 是随模型训练出来的，模型 v2 的第 1000 号 embedding 和模型 v1 的第 1000 号**含义完全不同**。在线服务换模型时，embedding table 必须**和模型同版本一起切换**，不能模型换了 v2、embedding 还是 v1——那等于给新模型喂了一堆随机向量。平台要把「模型 + embedding table + 特征处理统计量」当成一个**原子的版本包**一起发布。

### 2.7 上下文特征（时间/场景/位置）

**是什么**：描述「这次请求发生时的环境」的特征。请求时间（小时、星期、是否节假日）、场景（首页 feed / 搜索结果页 / 详情页猜你喜欢）、位置（GPS、城市）、设备（iOS/Android）、网络（WiFi/4G）。

**对平台意味着什么**：
- **生成时机**：大部分在**请求发生的那一刻**由在线服务生成，**不预先落库**——因为它就是「此刻」的状态。
- **计算位置**：在线，几乎零成本（从请求里直接拿）。
- **坑**：上下文特征的离线训练样本，必须**回放当时的上下文**而不是用「现在」的上下文，否则又是穿越。比如离线重算样本时，「是否节假日」要用样本发生那天的日历，不是跑批那天的。

### 2.8 画像特征（user/item profile）

**是什么**：对一个 user 或 item 的**长期、稳定**的刻画集合。user profile：长期偏好品类 top5、活跃时段、消费等级、人生阶段。item profile：标签、类目、价格档、历史质量分。

**对平台意味着什么**：
- **存储**：典型的**大宽表**（一个 user 一行，几百上千列），或拆成 KV（user_id → profile blob）。
- **更新**：天级 T+1 批跑为主（长期画像不需要实时）。
- **量级**：宽，但更新慢、读多写少，适合预计算 + KV 缓存。
- 它是其他特征的「底座」，很多统计/交叉特征其实是从 profile 派生的。

🎓 **老师视角**：画像特征的更新延迟容易被忽视。「T+1」意味着今天用的画像反映的是昨天及之前的状态。对长期偏好（消费等级、人生阶段）这没问题；但如果有人把「实时性敏感的东西」（如本次 session 兴趣）硬塞进 T+1 画像，就会严重滞后。平台研发要在画像建模时帮算法**区分「慢变量」和「快变量」**：慢变量进 T+1 画像，快变量走实时统计或在线序列，别混为一谈。画像膨胀（一个 user 上千列、大部分用不上）也是常见问题，要配合特征重要性定期瘦身。

| 特征类别 | 读写比 | 是否落库 | 离线/在线主战场 | 平台最该关注的风险 |
|---|---|---|---|---|
| 数值 | 读多 | 是 | 离线 | 归一化/缺失值约定 |
| 类别 ID | 读多 | 是 | 两者 | 词表一致性、OOV |
| 统计滑窗 | 读多写多 | 是 | 两者 | **穿越**、时效 |
| 序列 | 读多写多 | 是 | 在线拼接 | 变长存储、截断padding |
| 交叉 | 读多 | 视情况 | 离线 | 笛卡尔积爆炸 |
| Embedding | 读多 | 是 | 离线训+在线查 | 存储容量、版本一致 |
| 上下文 | 不落库 | 否 | 在线 | 离线回放穿越 |
| 画像 | 读多写少 | 是 | 离线 | 宽表膨胀、更新延迟 |

## 3. 特征变换：让原始数据变成模型能吃的形状

原始特征几乎不能直接喂模型，要经过变换。这一节列清单，并重点回答平台同学最常问的——**为什么深度模型也常需要分桶**。

### 3.1 变换清单（对照表）

| 变换 | 适用特征 | 做什么 | 平台支撑要点 |
|---|---|---|---|
| 分桶（bucketize） | 连续数值 | 把连续值切成若干区间，每区间一个 ID | 桶边界要存下来、线上线下用同一份 |
| 归一化/标准化 | 连续数值 | min-max 归一化 / z-score 标准化 | 要存 min/max 或 mean/std（统计量） |
| log 变换 | 长尾数值 | 取 log 压缩长尾（金额、播放量） | 注意 0 值要 log(x+1) |
| 哈希编码 | 高基数 ID | hash 到固定桶数 | 哈希函数线上线下必须完全一致 |
| 离散化 | 连续/有序 | 比分桶更泛，含等频/等距/自适应 | 同分桶 |
| 缺失值处理 | 所有 | 填默认值/特殊值/均值 | 约定统一的「缺失」表示 |
| 异常值截断（clip） | 数值 | 把超过 P99 的值截到 P99 | 截断阈值是统计量，要存 |

### 3.2 分桶三种方式

```
原始价格分布(长尾): 大量便宜货 + 少量奢侈品
  0│███████████
  价格 ──────────────────────────────────►

等距分桶: 按值域均分  → 便宜货全挤在第一个桶,区分度差
  [0-1000][1000-2000][2000-3000]...
   ███████████  ▏        ▏

等频分桶: 按样本数均分 → 每桶样本数相同,长尾被拉开,常用
  [0-50][50-120][120-300][300-99999]
   ██████ ██████ ██████   ██████

自适应分桶: 按对label的区分度切(如用决策树切点) → 效果导向
  按 CTR 突变点切边界
```

| 分桶方式 | 切法 | 优点 | 缺点 | 何时用 |
|---|---|---|---|---|
| 等距（equal-width） | 值域均分 | 简单直观 | 长尾下大量样本挤一桶 | 分布均匀时 |
| 等频（equal-frequency） | 样本数均分 | 抗长尾，最常用 | 桶边界依赖数据分布 | 默认首选 |
| 自适应（如树分桶） | 按对 label 区分度切 | 效果好 | 依赖 label、计算重、易穿越 | 精细调优 |

### 3.3 为什么深度模型也常需要分桶（重点）

这是平台同学最容易误解的点：「不是上了深度学习就端到端、不用特征工程了吗？」

**不是。** 即便是 DNN，连续特征常常先分桶再转 embedding，原因：

1. **非线性表达更容易**：直接喂裸数值（如年龄 25），DNN 要靠多层网络去拟合「年龄对结果的非线性影响」。分桶后每个年龄段一个 embedding，模型直接给每段学一个独立向量，**非线性关系一步到位**，训练更稳、更快收敛。
2. **抗异常值和尺度问题**：分桶天然把极端值收进边界桶，不怕一个 999999 的脏数据把梯度带飞。
3. **统一处理离散和连续**：分桶后连续特征也变成了 ID，可以和类别特征一样走 embedding，工程上统一。
4. **记忆能力**：分桶 embedding 给了模型「记住特定区间」的能力（记忆），配合 DNN 的「泛化」，这正是 Wide&Deep 思路的体现。

🎓 **老师视角**：当算法同学说「这个连续特征我要分 64 个桶，等频」，平台研发要听懂的是：① 你要提供**算分桶边界的能力**（离线对历史数据统计分位点）；② 边界要**存成可下发的配置**；③ **线上线下必须用同一份边界**，否则同一个用户离线训练分到第 3 桶、在线推理分到第 5 桶，就是线上线下不一致。这是个典型的「平台要做成基础设施而不是让算法手搓」的点。

🎓 **老师视角**：所有「需要统计量」的变换（归一化的 mean/std、分桶的边界、截断的 P99 阈值）都有一个共性——**这些统计量是从训练集算出来的，必须随模型版本固化下来，线上推理时复用同一份**。平台应该把这些统计量当成「模型的一部分」来管理（versioned），而不是每次在线服务自己临时算。临时算 = 线上线下不一致的温床。

### 3.4 缺失值与异常值：被低估的「脏数据」专题

平台研发往往低估缺失值/异常值的破坏力。它们不是「算法的事」，因为脏数据从**数据生产链路**进来，治理责任在平台。

**缺失值（missing）的处理选项与陷阱**：

| 策略 | 做法 | 适用 | 陷阱 |
|---|---|---|---|
| 填默认值 | 缺失填 0 / -1 | 数值特征 | 0 可能是有意义的真实值，混淆 |
| 填特殊标记 | 缺失填专门的「缺失桶」 | 类别/分桶后 | 最推荐，模型能学「缺失」本身的含义 |
| 填统计值 | 缺失填均值/中位数 | 数值特征 | 统计值要固化，且会抹掉「缺失」信号 |
| 不处理交给模型 | GBDT 等可原生处理缺失 | 树模型 | DNN 一般不行，要先填 |

**最重要的一条**：缺失本身常常是**有信息的信号**（「这个用户没填年龄」本身就说明了点什么）。所以业界深度模型里普遍把缺失映射到一个**专门的缺失 ID/桶**，让模型自己去学缺失的含义，而不是粗暴填 0 抹掉它。

**异常值（outlier）截断**：长尾数值（金额、时长）经常有脏值（如时长 9999999 秒）。做法是按分位点截断（clip 到 P99 / P1）。注意：**截断阈值是从训练集算的统计量，要固化并线上线下复用**——又是统计量一致性问题。

🎓 **老师视角**：缺失值和异常值的处理规则，本质都是「随特征定义固化、线上线下一致执行」的统计量/规则。平台研发应该在特征定义里把「缺失怎么填、异常怎么截」作为特征的**元数据**一起管理和下发，而不是散落在算法的训练脚本和在线代码两处各写一遍（那必然会写得不一样）。

### 3.5 变换发生在哪一层：平台的关键架构决策

变换逻辑放在哪执行，是平台架构的重要决策：

```
原始特征 ──► [变换层] ──► 模型输入
              │
   选项A: 在"特征生产时"变换好,存变换后的值
          优点:在线读取即用,延迟低  缺点:换变换规则要重算全量
   选项B: 存原始值,在"模型输入前"实时变换
          优点:规则灵活可调  缺点:每次推理都算,占延迟
   选项C: 变换内嵌进模型(如TF的特征列/预处理层)
          优点:训练推理用同一份代码,天然一致  缺点:耦合模型
```

经验取舍：**简单、稳定的变换（如固定分桶、归一化）倾向内嵌模型或生产时算**（选 A/C，保证一致性）；**频繁调整的实验性变换**倾向存原始值实时变换（选 B，灵活）。越来越多团队走选 C（把预处理做进模型计算图），因为它从根上消除了训练-服务偏差。

🎓 **老师视角**：这个决策直接关系到「线上线下一致性」这个搜广推平台的命门。如果变换在离线 Python 脚本里写一遍、在线 C++/Java 服务里又写一遍，两份代码迟早写得不一样，造成偏差。最稳的方向是**变换逻辑单一来源**——要么内嵌模型、要么用同一套特征处理库被两边调用。这是 [02-特征平台(Feature Store)架构](../03-特征数据管理/02-特征平台(Feature Store)架构.md) 要解决的核心问题之一。

## 4. 统计特征/滑窗特征深挖：时效与穿越风险

统计特征是平台研发的主战场，单独深挖。

### 4.1 怎么算

以「用户近 7 天点击数」为例：

```
时间轴 ─────────────────────────────────────────►
        D-7   D-6   D-5  ...  D-1    D(今天) 
         ●     ●     ●         ●      ▲样本发生时刻 T
        └──────── 统计窗口 [T-7d, T) ────┘
                                         ↑
                            窗口右边界必须 < T(严格小于)
```

两种实现：

| 维度 | 离线批计算 | 在线增量计算 |
|---|---|---|
| 引擎 | Spark / Flink 批 | Flink 流 / 实时 KV 累加 |
| 适用窗口 | 长窗口（近 7/30 天） | 短窗口（近 1h / 实时） |
| 时效 | T+1 / 小时级 | 秒级~分钟级 |
| 成本 | 周期性重算，CPU 重 | 常驻流，状态存储贵 |
| 穿越风险 | **高**（容易误用未来数据） | 中（天然按时间到达，但乱序要处理） |

### 4.2 时效：长窗口慢、短窗口快

经验法则：**窗口越短、越需要捕捉即时变化的特征，时效要求越高**。
- 近 30 天消费金额 → 天级 T+1 足够（用户消费习惯变化慢）。
- 近 1 小时 item CTR → 分钟级（热点变化快，慢了就推过气内容）。
- 实时「本次 session 点击数」→ 秒级，在线算。

### 4.2.1 长短窗口拼接：lambda 式的双路计算

一个常见且棘手的场景：算法要「近 30 天 + 近 1 小时」组合的统计特征。30 天的量级只能离线批算（实时流存 30 天状态太贵），1 小时的又必须实时。于是出现**双路计算 + 在线拼接**：

```
  近30天统计 ──(离线批,T+1)──► 离线结果KV ─┐
                                            ├─► 在线拼成完整特征 ──► 模型
  近1小时统计 ──(在线流,分钟级)──► 实时KV ──┘
              ↑ 注意:两路的口径/单位/平滑方式必须完全一致
```

这就是经典的 lambda 架构在特征侧的体现。平台研发的难点是**保证两路口径一致**：同样的去重逻辑、同样的平滑参数、同样的过滤规则。两路由不同引擎（Spark 批 vs Flink 流）实现，最容易在细节上跑偏（比如批侧去重了、流侧没去重），导致特征值在「近 30 天」和「近 1 小时」之间口径不一致，模型学到的是噪声。

🎓 **老师视角**：能用单一引擎（如 Flink 同时跑批和流、流批一体）统一两路口径，就尽量统一，能从根上避免「两套代码两个口径」。这也是为什么流批一体的计算框架在特征平台里越来越受欢迎——它让「同一份逻辑跑离线和在线」成为可能，正对应了特征平台「define once, serve both」的核心诉求。

### 4.3 穿越风险（重中之重）

**特征穿越 / label leakage**：特征里混进了 label 时刻**之后**才知道的信息。

典型穿越事故：
1. **窗口右边界越界**：算「近 7 天点击数」时，把样本发生当天剩余时间的点击也算进去了。正确做法：窗口右边界 = 样本时刻 T，且**开区间**。
2. **用「最新快照」拼历史样本**：拿今天的 user profile 去拼 30 天前的样本。profile 是「现在」的，30 天前不长这样。
3. **统计量用了全量数据**：算分桶边界、归一化 mean 时用了包含未来的全量，应只用样本时刻之前的数据（严格场景）。

```
穿越长这样:
  样本时刻 T ─────┐
  特征窗口  ──────┼──────►  ✗ 窗口右边界超过了 T,吃到了未来
                  │
                  T 之后发生的点击,模型在预测时根本不知道
                  → 线下AUC虚高,线上暴跌
```

🎓 **老师视角**：穿越是搜广推平台的「头号杀手级 bug」，因为它**线下指标全是好的**，离线评估根本发现不了，上线才暴雷，排查极难。平台研发的正确姿势：**把「时间点对齐」做成生产链路的强制能力**——所有特征生产都带 event_time，拼样本时按 label 的 event_time 做 point-in-time join，框架层禁止用未来数据。不要指望算法同学每次手动小心。这一整套治理见 [03-特征生产链路与特征穿越治理](../03-特征数据管理/03-特征生产链路与特征穿越治理.md)。

### 4.4 ratio 类特征的两个坑

CTR、转化率这类 ratio = 分子/分母 的统计特征，有两个平台研发要替算法把关的坑：

1. **小样本除法不稳（要做平滑）**：某 item 只曝光了 2 次、点击 1 次，CTR = 50%，这显然不可信。业界普遍做**贝叶斯平滑/拉普拉斯平滑**：`CTR = (点击 + α) / (曝光 + α + β)`，用先验把小样本拉回均值。平台要支持把平滑参数配置化，而不是硬编码。
2. **分母为 0**：曝光为 0 时 CTR 没意义，要约定一个缺省值（通常是全局均值或特殊值），不能产出 NaN/Inf 进模型。

### 4.5 脱敏案例：一次真实的滑窗穿越

> **场景（脱敏）**：某信息流推荐，算法上线一个「item 近 1 天 CTR」特征，离线评估 AUC 从 0.71 涨到 0.79，团队大喜。上线后线上 CTR 不升反降。
>
> **排查**：离线算「item 近 1 天 CTR」时，用的是「自然日整天」的聚合（00:00~23:59），而一个样本发生在当天 10:00。这意味着这个 CTR 统计窗口**包含了 10:00 之后到 23:59 的曝光点击**——也就是未来。模型在离线学到了「未来这个 item 很火」这一信息，AUC 自然虚高；但线上预测时 10:00 那一刻根本拿不到未来，特征失效，效果回落。
>
> **修复**：把窗口改成滚动的「截至样本时刻前 24 小时」开区间，重算样本。AUC 回落到 0.73（真实水位），线上小幅正收益。
>
> **平台教训**：「自然日聚合」是穿越重灾区。平台应默认提供**按 event_time 右开区间**的滚动窗口聚合原语，而不是让算法自己写「按 dt 分区 join」——后者极易越界。

## 5. 行为序列特征深挖：DIN/DIEN 为什么需要它

### 5.1 序列长什么样

```
用户 u123 的点击序列(按时间正序):
  item_88 → item_42 → item_17 → ... → item_9 → item_30
   t=09:01   t=09:05   t=10:22         t=昨天    t=刚刚

模型要定长输入(假设 N=50):
  - 序列长度 > 50 → 截断(truncate),取最近50个
  - 序列长度 < 50 → padding 补 0(或补特殊 PAD_ID)到 50
  - 同时要带 mask 标记哪些是真实、哪些是 padding

  [item_9, item_30, ..., PAD, PAD, PAD]  长度=50
  mask: [1, 1, ..., 0, 0, 0]
```

### 5.2 DIN/DIEN 为什么需要它

普通模型把序列「池化」成一个平均向量，丢失了顺序和「哪个历史行为和当前候选相关」的信息。

- **DIN（Deep Interest Network）**：用 **attention** 让「当前候选 item」去和「历史序列里每个 item」算相关性，相关的历史行为权重高。所以它必须吃到**原始序列**，而不是预先池化的平均。
- **DIEN（Deep Interest Evolution Network）**：在 DIN 基础上用 GRU 建模兴趣的**演化过程**，更依赖序列的**时间顺序**。

结论：**只要算法上 DIN/DIEN/SIM/BST 这类序列建模，平台就必须能提供「保序的、变长的、带时间戳的」行为序列**，不能只给一个聚合统计值。

**序列建模的演进与平台的存储压力**：行为序列建模这几年从「短序列」走向「超长序列」，对平台存储的要求逐级升级：

| 模型 | 序列长度（典型量级） | 建模方式 | 对平台的存储/检索要求 |
|---|---|---|---|
| 平均池化（pooling） | 任意 | 取平均，丢顺序 | 最简单，存聚合即可 |
| DIN | 几十（如 50） | target attention | 存原始短序列 |
| DIEN | 几十 | GRU 建模兴趣演化 | 存原始有序短序列 |
| BST | 几十~百 | Transformer | 同上，稍长 |
| SIM / ETA | 上千~上万（终身行为） | 先检索再建模 | **需专门的行为检索结构**（按品类/相关性快速从超长历史里捞 top-k） |

🎓 **老师视角**：当算法说「我们要上 SIM，用户终身行为序列」，平台研发要拉响警报——这不再是「存最近 50 个」那么简单，而是要存用户**全部历史行为（可能上万条）**，并提供**快速检索**能力（给定当前候选 item，从上万条历史里快速找出相关的 top-k）。这等于多了一套「行为检索系统」，存储和工程量级完全不同。务必在需求评审阶段就把这个成本摆到台面上，不要等算法说「不就是序列长一点吗」轻易答应。

### 5.3 对样本和存储的要求

| 要求 | 说明 | 平台支撑 |
|---|---|---|
| 保序 | 必须按时间排序 | 存时间戳，读时排序 |
| 变长存储 | 不同用户序列长度不同 | array/list 类型，或行为日志聚合 |
| 实时更新 | 刚点的要尽快进序列 | 在线拼接，从实时行为流取最近 N 个 |
| 防穿越 | 序列里只能有 T 之前的行为 | 按 event_time 截断到样本时刻 |
| 截断/padding | 转定长 | 特征处理层统一做，配置化 N |
| 存储量级 | N×行为，乘用户数 | 长序列（>200）考虑专门检索结构 |

```
在线拼序列的位置:
  请求到来 ──► 在线服务 ──► 从行为流/KV 取该user最近N个行为
                            ──► 截断 + padding ──► 拼进特征 ──► 喂模型
  注意:取的必须是"截至此刻"的行为,这天然不穿越(在线就是实时)
  但离线训练样本回放时,必须截到"样本那一刻",这才是穿越高发区
```

🎓 **老师视角**：序列特征的穿越比统计特征更隐蔽。在线服务取「当前最近 N 个行为」天然不穿越（因为在线就是此刻）。但**离线生成训练样本**时，你要回放「这个样本发生的那一刻，用户的最近 N 个行为是什么」——如果偷懒用了「用户现在的最近 N 个行为」，就把未来行为混进了历史序列。平台必须支持**按 event_time 重建历史序列**，这是离在线一致性的关键，详见 [04-样本数据管理/样本离在线一致性](../04-样本数据管理/样本离在线一致性.md)。

### 5.4 序列不止是 item_id：多通道对齐

实战里行为序列往往不是一条「item_id 列表」，而是**多个并列的子序列必须严格对齐**：

```
位置:        0       1       2      ...    N-1
item_id:   [88,     42,     17,    ...,   PAD]
category:  [电子,   服饰,   电子,  ...,   PAD]   ← 每个item的品类
action:    [点击,   收藏,   点击,  ...,   PAD]   ← 行为类型
time_gap:  [3600,   1200,   86400, ...,   PAD]   ← 距今时间差(秒)
           └──────── 这几条必须一一对应、同样截断同样padding ────┘
```

- DIN/DIEN 不光看你点了哪个 item，还看品类、行为类型、时间间隔。这些是**并列的多通道序列**，截断和 padding 必须**完全同步**（同一个位置被截/被补）。
- 平台支撑要点：把「行为序列」建模成一个**结构化的行为对象列表**（每个行为是一个含多字段的 record），而不是孤立的几个数组各存各的——后者极易在截断时错位，导致 item_88 配上了 item_42 的品类。这是序列特征工程里非常隐蔽的 bug。

🎓 **老师视角**：当算法说「我序列要带上品类和停留时长」，平台研发要立刻意识到这是**多通道对齐**问题，存储和处理层都要保证「一个行为的所有属性绑在一起截断/padding」。建议在存储层就存成「行为对象的有序列表」（一行为一 record，含 item/category/action/timestamp），读取时整体截断，从根上杜绝错位。

## 6. 交叉特征深挖：人工交叉 vs 模型自动交叉

### 6.1 两条路线

```
                    要不要平台预计算交叉特征?
                              │
        ┌─────────────────────┴─────────────────────┐
   人工交叉(显式)                              模型自动交叉(隐式)
        │                                            │
  离线把 A×B 拼成新ID列                     FM/DCN/DeepFM 网络内部学
  存进宽表/哈希                              平台只喂原始特征
        │                                            │
  优点:可控、可解释、对简单模型(LR)必需        优点:省存储、自动发现高阶交叉
  缺点:笛卡尔积爆炸、要人工选组合              缺点:黑盒、算力换存储、调参难
```

### 6.2 对比表

| 维度 | 人工交叉 | 模型自动交叉 |
|---|---|---|
| 谁来做 | 平台/算法离线预计算 | 模型网络内部 |
| 典型模型 | LR、GBDT+LR | FM、FFM、DCN、DeepFM、xDeepFM |
| 存储 | 要存组合特征（可能很大） | 不额外存 |
| 算力 | 离线一次性算 | 在线推理时算 |
| 可解释 | 强（看得见组合） | 弱（黑盒） |
| 扩展性 | 差（组合数爆炸） | 好（自动） |
| 平台是否预计算 | **要** | **不要** |

### 6.3 平台是否要预计算交叉特征？

这是平台同学常被问的决策题。判断逻辑：

1. **模型用的是 LR/GBDT 这类「不会自己学交叉」的** → 交叉信息全靠人工，**平台要预计算**关键组合。
2. **模型用的是 FM/DCN/DeepFM** → 模型自己学交叉，**平台一般不预计算**，只保证原始特征喂全。
3. **少数高价值、强先验的交叉**（如「用户常买品类 × 当前 item 品类是否一致」）即使在深度模型下也可能预计算，因为这是强信号、显式给比让模型自己学更稳。

🎓 **老师视角**：当算法说「帮我把 user_id × item_id 交叉一下」，平台研发要立刻警觉——这是**笛卡尔积爆炸**的典型（上亿 × 千万）。正确反应是：① 这种超高基数交叉**必须哈希**到固定桶；② 或者反问「你是不是该用 FM 让模型自己学，而不是预计算」。直接照做去拼 user_id×item_id 的全量交叉表，存储会爆炸。

### 6.4 高阶交叉与「交叉爆炸」的量级直觉

二阶交叉（两两组合）已经不少，三阶、四阶交叉组合数呈指数增长：

| 阶数 | 例子 | 组合数量级（假设各特征基数千级） | 平台态度 |
|---|---|---|---|
| 一阶（原始） | 性别、品类、城市 | 各几千 | 直接存 |
| 二阶 | 性别×品类 | 千 × 千 = 百万 | 可预计算，注意控量 |
| 三阶 | 性别×品类×城市 | 百万 × 千 = 十亿 | 一般交给模型自学 |
| 四阶+ | 更多组合 | 天文数字 | **必须**交给 DCN/xDeepFM 自学 |

结论给平台研发：**人工预计算只适合低阶、强先验的少量交叉**；高阶交叉是深度交叉网络（DCN/xDeepFM）的活，平台不要去预计算，那是无底洞。

## 7. 特征存储格式与计算位置：平台研发的真正落点

前面每类特征都提了「存储」和「计算位置」，这一节系统收口——这是平台研发**最该上心**的部分，因为它直接决定系统的成本、延迟和复杂度。

### 7.1 存储格式：离线一种、在线一种，但要等价

特征通常**双存储**：离线供训练、在线供推理。两边格式不同但**语义必须等价**。

```
        离线存储(训练用)                在线存储(推理用)
        ────────────────                ────────────────
  数据湖/数仓: Hive/Iceberg/Paimon       低延迟KV: Redis/HBase/自研
  列式、批量扫描、按dt分区                点查/批量查、毫秒级、按key
  存全量历史(带event_time)               存"最新"或"最近窗口"
        │                                        │
        └──────── 同一份特征定义 ────────────────┘
                  (define once, serve both)
```

| 特征形状 | 离线存储建议 | 在线存储建议 | 关键约束 |
|---|---|---|---|
| 标量数值 | 列式宽表一列 | KV value 一字段 | 量纲一致 |
| 低基数类别 | 宽表 ID 列 | KV value | 词表一致 |
| 高基数 ID | 宽表 + embedding table | embedding 服务/参数服务 | 编号一致 |
| 变长序列 | array/list 列 或 行为日志 | 行为流 + 在线拼接 | 按 event_time 截断 |
| embedding 向量 | 向量列 / parquet | KV blob 或向量库 | 版本一致 |
| 画像宽表 | 大宽表（列存） | KV（user_id→profile） | T+1 同步 |

🎓 **老师视角**：离线用列存（Hive/Iceberg/Paimon 这类湖格式，擅长批量扫描全量历史训练），在线用 KV（擅长按 key 毫秒级点查）——这是两种完全不同的存储引擎，因为**训练和推理的访问模式截然相反**：训练要「扫一亿行的某几列」，推理要「按一个 user_id 拿它的全部特征」。平台研发的核心难题不是选哪个存储，而是**保证两边产出的特征值在语义上完全一致**（define once, serve both）。

### 7.2 计算位置：在线算 vs 离线算的决策树

```
              这个特征怎么算?
                    │
        ┌───────────┴────────────┐
   依赖"当前请求上下文"吗?      纯历史聚合?
   (此刻时间/位置/session)        │
        │是                      │
   必须在线算(请求时生成)    更新频率要求?
                            ┌────┴─────┐
                         实时/分钟级   天级/小时级
                            │            │
                        在线流计算    离线批计算
                        +实时存储     +KV缓存(首选,便宜)
```

| 计算位置 | 适合的特征 | 优点 | 代价 |
|---|---|---|---|
| 离线批 | 长窗口统计、画像、低阶交叉 | 便宜、好维护、可重算 | 时效差（T+1/小时） |
| 在线流 | 短窗口统计、实时热点 | 时效高 | 常驻流，状态存储贵，运维重 |
| 在线请求时生成 | 上下文、序列拼接、模型内交叉 | 天然实时、不落库 | 占用在线延迟预算 |

🎓 **老师视角**：平台研发要建立一个铁律——**默认离线，除非证明必须在线**。在线计算（流计算 + 实时存储）的工程和运维成本是离线的数倍，且占用宝贵的在线延迟预算。每当算法要求一个实时特征，先问「天级够不够？慢一天掉几个点？」，让需求方为「实时」这个昂贵选项付出举证责任。很多所谓的实时需求，其实天级就够。

### 7.3 脱敏案例：把实时特征「降级」成天级省下大半成本

> **场景（脱敏）**：某电商推荐，算法要求 30 多个「用户近 N 天行为统计」特征全部做成实时（分钟级），平台搭了一套 Flink 流 + 实时 KV，状态存储和机器成本很高，且经常因流计算反压告警。
>
> **复盘**：和算法一起做消融，发现这 30 多个里只有 3 个短窗口特征（近 1 小时点击/曝光）对实时性敏感，其余近 7 天、近 30 天的长窗口特征做成 **T+1 天级**对线上指标几乎无影响（A/B 验证掉点在噪声范围内）。
>
> **结果**：3 个保留实时，其余降级到离线批 + KV 缓存。流计算集群规模缩小一半多，反压告警消失，线上指标无损。
>
> **平台教训**：「全都要实时」往往是需求方的默认偷懒选项。平台研发用**消融数据**逼出「哪些真的需要实时」，是降本的有效手段。本质上是把 [1.2 四象限](#12-特征的四象限平台研发的心智模型) 里能往右下挪的特征都挪过去。

## 8. 特征重要性与特征筛选：平台怎么支撑特征实验

### 8.1 为什么要做

- 特征不是越多越好：冗余/无用特征增加存储、计算、在线延迟，还可能引入噪声和穿越风险。
- 上线一个特征是有**长期成本**的（持续生产、存储、监控），所以要评估它「值不值」。
- 老板会问「这 200 个特征哪些真有用？能不能砍一半省成本？」——这需要特征重要性数据支撑。

### 8.2 常见方法（了解即可，平台不一定亲自做）

| 方法 | 思路 | 平台支撑点 |
|---|---|---|
| 模型自带重要性 | GBDT 的 feature importance、attention 权重 | 提供训练产物的导出 |
| 置换重要性（permutation） | 打乱某特征看指标掉多少 | 提供「关掉某特征重跑评估」的能力 |
| 消融实验（ablation） | 去掉一组特征对比 AUC | 支持特征分组开关、A/B |
| 相关性/IV/PSI | 统计指标筛选 | 提供离线特征统计报告 |

### 8.3 平台怎么支撑特征实验（重点，这才是你的活）

平台研发的价值不在「算重要性」，而在**让特征实验变快变便宜**：

```
特征实验的平台支撑:
  ┌──────────────────────────────────────────────┐
  │ 1. 特征注册/下线开关:加一个特征不用改代码,配置化   │
  │ 2. 特征版本管理:每个实验用哪些特征可追溯           │
  │ 3. 特征统计报告:覆盖率、分布、缺失率、PSI 漂移      │
  │ 4. 离线快速回放:换一组特征能快速重算样本/重跑评估   │
  │ 5. 在线灰度:新特征先小流量,看在线指标               │
  │ 6. 血缘:这个特征被哪些模型用,下线影响谁            │
  └──────────────────────────────────────────────┘
```

🎓 **老师视角**：算法同学评判特征看 AUC，平台研发评判特征看「**这个特征上线后我的系统多扛了多少存储/延迟/生产成本**」。一个好平台能让算法「半天加一个特征做实验、不行就下掉」，而不是「加个特征要排期两周改链路」。你把特征实验的边际成本压到足够低，算法同学的迭代速度就上去了，这是平台的核心价值。具体系统设计见 [02-特征平台(Feature Store)架构](../03-特征数据管理/02-特征平台(Feature Store)架构.md)。

### 8.4 特征监控：上线只是开始

特征上线后会「悄悄变质」，平台必须做**持续监控**，否则模型在不知不觉中退化。三类要监控的东西：

| 监控项 | 看什么 | 报警含义 |
|---|---|---|
| 覆盖率/缺失率 | 有多少样本能拿到该特征值 | 突然大量缺失 = 上游链路断了 |
| 分布漂移（PSI） | 特征值分布相比训练时是否偏移 | PSI 偏大 = 数据分布变了，模型可能失效 |
| 在离线一致性 | 同一请求离线和在线算出的值是否一致 | 不一致 = 训练-服务偏差，效果打折 |

**PSI（Population Stability Index）经验阈值**（典型量级，仅供参考）：
- PSI < 0.1：分布基本稳定。
- 0.1 ~ 0.25：有一定漂移，关注。
- PSI > 0.25：显著漂移，需排查或考虑重训。

🎓 **老师视角**：在离线一致性监控是搜广推平台的「金标准」检查项。做法是**抽样**线上请求，把它在线算出的特征值记下来，再用离线链路对同一请求、同一时刻重算一遍，对比两者差异。差异率超阈值就报警。很多团队没做这一步，结果训练-服务偏差长期存在却毫不知情，白白损失效果。这一项强烈建议作为平台基础设施落地，细节见 [04-样本数据管理/样本离在线一致性](../04-样本数据管理/样本离在线一致性.md)。

### 8.5 特征生命周期：从生到死

把一个特征从诞生到退役串起来，平台研发要支撑的全流程：

```
  ① 提需求 ──► ② 定义注册 ──► ③ 离线生产 ──► ④ 在线供给
   (算法)       (元数据/契约)   (批/流计算)    (KV/embedding服务)
                                                    │
  ⑧ 下线 ◄── ⑦ 评估价值 ◄── ⑥ 持续监控 ◄── ⑤ 进模型训练/推理
  (清存储,通知血缘下游)  (重要性/成本)  (覆盖率/PSI/一致性)
```

每一环平台都要有对应能力，缺一环就会出问题：缺②就没有契约、量纲随便变；缺⑥就在不知情中退化；缺⑧就是「只增不减」、存储成本越滚越大、没人敢下线（因为不知道谁在用——这又回到血缘）。

## 业界怎么做

- **统一在离线特征定义（避免训练-服务偏差）**：业界（如 Feast、Tecton、阿里 FeatureStore、字节等内部平台）的核心思想是**特征定义一次、离线在线共用**，从根上消除「离线一套逻辑、在线一套逻辑」导致的不一致。
- **Point-in-time correct join 作为标配**：成熟特征平台都把「按 event_time 做时间点对齐 join」做成框架原语，防穿越不靠人。
- **连续特征分桶 + embedding 是 CTR 模型标配**：Google Wide&Deep、各家精排模型普遍对连续特征分桶。
- **行为序列从短到长演进**：DIN（短序列）→ DIEN（兴趣演化）→ SIM/ETA（超长序列检索），存储和检索结构随之升级。
- **哈希处理高基数 ID**：海量 sparse ID 普遍用 hashing trick 控制 embedding table 规模，碰撞换存储。
- **特征平台分「在线存储（低延迟 KV，如 Redis/HBase/自研）」+「离线存储（数仓/数据湖，如 Hive/Iceberg/Paimon）」双层**，这正是 Feature Store 的经典双存储架构。

## 权衡取舍

| 取舍点 | A 方案 | B 方案 | 怎么选 |
|---|---|---|---|
| 高基数 ID 编码 | 全量词表+embedding（无碰撞、效果好、词表大） | 哈希分桶（固定大小、有碰撞） | 看基数和存储预算，超大基数倾向哈希 |
| 交叉特征 | 人工预计算（可控、爆炸风险） | 模型自动学（省存储、黑盒） | 看模型能力，深度模型倾向自动 |
| 统计特征计算 | 离线批（便宜、慢） | 在线流（贵、快） | 看时效要求，短窗口必须在线 |
| 序列长度 | 短（50，省资源、信息少） | 长（数百+，信息多、贵） | 看模型和收益，长序列要专门结构 |
| 特征数量 | 多（信息全、成本高、易穿越） | 精（省成本、可能漏信息） | 用特征重要性数据做减法 |
| 特征更新频率 | 实时（新鲜、贵） | 天级（便宜、可能过时） | 看特征变化速度，慢变量天级足够 |

## 常见误区与反直觉

1. **「上了深度学习就不用特征工程」**——错。连续特征分桶、行为序列构造、交叉信号设计，深度模型一样需要，有时更需要（见 [3.3](#33-为什么深度模型也常需要分桶重点)）。
2. **「特征越多越好」**——错。冗余特征增加成本、延迟、噪声、穿越风险。要做减法。
3. **「线下 AUC 高就是好特征」**——警惕，**AUC 异常高极可能是穿越**。线下越漂亮越要怀疑穿越。
4. **「在线服务自己实时算归一化/分桶就行」**——错。统计量必须用训练时固化的那份，临时算会造成线上线下不一致。
5. **「序列特征在线取最近 N 个就不会穿越」**——在线确实不穿越，但**离线回放历史样本**时极易穿越，这才是高发区。
6. **「user_id × item_id 交叉一下」**——天文数字笛卡尔积，必须哈希或交给模型自学，别真去拼全量表。
7. **「上下文特征离线用现在的环境算就行」**——错。「是否节假日」「时段」要用样本**当时**的环境，不是跑批当天的。
8. **「embedding 就是几个 float，能多大」**——一个特征乘上亿基数就是几十 GB，乘模型数和副本数更夸张，是容量规划的隐形大头。
9. **「特征上线就不用管了」**——错。特征会因上游变化而漂移、失效（PSI 报警），必须持续监控覆盖率、分布和在离线一致性。
10. **「序列长一点不就是 N 改大点」**——错。短序列（DIN，几十）和超长序列（SIM，上万）是两种存储和检索架构，成本天差地别。
11. **「缺失值填 0 就行」**——危险。0 可能是有意义的真实值；缺失本身往往是有信息的信号，应映射到专门的缺失桶让模型学。

## 平台侧落地 Checklist

**存储与格式**
- [ ] 每类特征都明确了**存储格式**（数值列 / ID 映射+embedding table / 变长 list / KV）。
- [ ] 离线（湖/数仓）与在线（KV）**双存储语义等价**，遵循 define once、serve both。
- [ ] embedding table 单独做**容量规划**（基数 × 维度 × 字节 × 模型数 × 副本数），并评估降维/量化/淘汰空间。

**更新与计算**
- [ ] 每个特征都有**明确的更新频率**（实时/分钟/小时/天）并据此选了计算引擎。
- [ ] 每个特征都标注了**计算位置**（在线/离线/两者），默认离线、实时需举证。
- [ ] 长短窗口拼接（lambda）场景下，**两路口径完全一致**（去重/平滑/过滤）。

**防穿越与一致性（命门）**
- [ ] 所有特征生产都带 **event_time**，拼样本走 **point-in-time join**，框架层禁止用未来数据（防穿越）。
- [ ] 需要统计量的变换（归一化/分桶/截断）的**统计量随模型版本固化**，线上线下用同一份。
- [ ] 高基数 ID 有**词表生命周期**管理（生成/更新/OOV/淘汰），且**线上线下词表一致**。
- [ ] 序列特征支持**变长存储 + 截断 padding + 按 event_time 重建历史序列**，多通道严格对齐。
- [ ] 模型 + embedding table + 特征统计量作为**原子版本包**一起发布。
- [ ] 有**在离线一致性抽样校验**，差异超阈值报警。

**治理与运维**
- [ ] 高基数交叉强制走**哈希**，并防笛卡尔积爆炸；高阶交叉交给模型自学。
- [ ] 缺失值/异常值规则作为特征**元数据**统一管理下发，缺失优先映射到专门桶。
- [ ] 提供**特征统计报告**（覆盖率、缺失率、分布、PSI 漂移）和**特征实验快速开关**。
- [ ] 有**特征血缘**：每个特征被哪些模型用，下线影响范围可查。
- [ ] 统计特征生产**模板化**（主体+窗口+聚合+过滤可配），防穿越内建。

## 对接话术

**算法同学说**：「我要个用户近 7 天点击序列特征，定长 50，不够补零。」
**你接**：「这是变长行为序列特征，我用 list 存、带时间戳；定长 50 我在特征处理层做截断+padding，并出 mask。离线训练样本我会按你样本的 event_time 重建当时的序列，防止把未来行为穿越进去。在线推理直接取实时行为流最近 50 个。要不要顺带给你 DIN 用的 attention mask？」

**算法同学说**：「这个连续特征帮我等频分 64 桶。」
**你接**：「行，我离线对历史数据算 64 个等频分位点作为桶边界，固化成配置随模型版本走，线上线下用同一份边界，保证一致。缺失值我们约定落到哪个桶？」

**算法同学说**：「帮我把 user_id 和 item_id 交叉一下当特征。」
**你接**：「这俩都是超高基数，直接交叉是笛卡尔积爆炸，得哈希到固定桶数（比如 1000 万），会有碰撞但可控。不过如果你模型是 DeepFM/DCN，其实可以不预计算，让模型自己学交叉，省一大块存储——你这边模型是哪种？」

**算法同学说**：「这些统计特征我全要实时的。」
**你接**：「实时这套（流计算 + 实时存储）成本是天级的好几倍，还占在线延迟。咱们先区分一下：哪几个是短窗口、对实时性真敏感的？我建议先把它们做实时，长窗口（近 7/30 天）的先上 T+1，我们 A/B 看看实时到底带来多少增量，值不值这个成本再说。」

**算法同学说**：「我序列里要带上品类、停留时长。」
**你接**：「那就是多通道序列了，item/品类/时长要绑在一起截断 padding，不能各存各的否则会错位。我在存储层把它存成行为对象的有序列表，一个行为一条 record，读取时整体截断，给你保证对齐。」

## 向上沟通

**老板问**：「为什么上一个新特征要排期两周，不能快点？」
**你答**：「上一个特征不是加一列那么简单，它要：① 离线天天生产、② 在线低延迟可查、③ 防止穿越（用错时间点会让效果造假）、④ 监控它的分布漂移。我们正在把这套做成自助平台，目标是把加特征从两周压到半天，这样算法迭代速度能上一个台阶。」

**老板问**：「我们有 200 个特征，能砍掉一半省成本吗？」
**你答**：「能评估。我会拉出每个特征的重要性和使用血缘，把低贡献、高成本（尤其是大 embedding 和实时统计）的特征列出来，和算法一起做消融实验确认砍掉不掉点。预计能省下 [X]% 的特征存储和在线计算成本，但要留 2 周做 A/B 验证不影响线上指标。」

**老板问**：「上次那个上线就掉点的事故，到底是谁的锅？」
**你答**：「是特征穿越——离线训练时不小心用到了预测时刻拿不到的未来信息，导致线下指标虚高、线上回落。这不是某个人粗心，是我们缺少防穿越的基础设施。我正在把『按时间点对齐（point-in-time join）』做成生产链路的强制能力，从框架层禁止用未来数据，让这类事故不再依赖个人小心，而是系统兜底。」

**老板问**：「为什么我们的在线推理偶尔效果比离线评估差？」
**你答**：「最常见的根因是『训练-服务偏差』：同一个特征，离线训练算出来的值和在线推理算出来的值不完全一样（比如分桶边界、词表、归一化统计量两边不一致）。我们要做在离线一致性抽样校验来量化这个差异并报警。把偏差压下去，线上线下的表现就会对齐。」

## 自测三问

**Q1：算法同学要一个「用户当天总点击数」特征拼进训练样本，你直接 join 进去了，线下 AUC 飙到 0.92，要不要上线？**

<details>
<summary>参考答案</summary>

**不能直接上**，要高度怀疑穿越。「当天总点击数」包含了样本发生**之后**的点击，这是经典穿越。线下 AUC 异常高（0.9+ 在 CTR 场景很可疑）往往就是穿越信号。正确做法：把它改成「截至样本 event_time 之前的累计点击数」，用 point-in-time join 重新生成，再看 AUC。
</details>

**Q2：一个连续特征要等频分 64 桶。这个分桶的桶边界应该在哪算、存在哪、谁用？**

<details>
<summary>参考答案</summary>

桶边界应在**离线对训练集历史数据**算分位点得到（注意严格场景下只能用样本时刻之前的数据），**固化成配置随模型版本管理**，**线上推理和离线训练用同一份边界**。绝不能让在线服务每次自己临时算，否则同一个值离线分到第 3 桶、在线分到第 5 桶，造成线上线下不一致。
</details>

**Q3：算法要上 DIN 模型，需要用户行为序列。在线服务取「用户最近 50 个点击」不会穿越；那离线生成训练样本时为什么容易穿越，平台要怎么支撑？**

<details>
<summary>参考答案</summary>

在线取「最近 50 个」天然是此刻状态，不穿越。但离线回放历史样本时，如果偷懒用「用户**现在**的最近 50 个行为」，就把样本时刻**之后**发生的行为混进了历史序列——穿越。平台必须支持**按样本的 event_time 重建当时的最近 50 个行为**（point-in-time 重建序列），并做截断+padding。这是离在线一致性的关键能力。
</details>
