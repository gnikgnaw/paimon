## 0. 30 秒速通（TL;DR）

- **特征生产有两条链路**：**离线（Spark/批，T+1）**——扫历史明细，跑大窗口统计、画像类特征，产出落离线特征表（数仓/数据湖）；**实时（Flink/流，秒级）**——消费行为流增量更新，产出写在线 KV（Redis/HBase 等）供服务读。两条链路**算的是同一批特征**，这就埋下了一致性与穿越的雷。
- **特征回填 backfill**：新特征上线，老样本里没有它的值，必须"补历史"。回填**最容易穿越**，因为你站在"今天"去算"过去那一刻"的特征，一不小心就把未来的数据算进去了。
- **特征穿越 / feature leakage**：样本里用了"预测那一刻还不知道的信息"。三大典型：① 用了 t0 之后才产生的统计值；② 用了 label 衍生出来的特征；③ 滑窗右边界错位（窗口边界越过了 t0）。
- **后果**：穿越会让**离线 AUC 虚高**（模型偷看了答案），上线却原形毕露、效果暴跌，且极难自查。
- **防穿越核心机制**：**point-in-time 正确性**——取"预测那一刻 t0 及之前"的特征值，而不是"现在最新"的值。离线靠**事件时间 as-of join** 实现。
- **你作为平台侧要交付**：事件时间对齐能力、安全的回填工具、穿越检测 / point-in-time join、特征漂移与覆盖率看板、新鲜度监控。

> 💡 本文与同目录《[样本离在线一致性](../04-样本数据管理/样本离在线一致性.md)》互补：那篇站在**样本侧**讲"训练值=服务值"；本篇站在**特征侧**讲"特征怎么生产、回填、防穿越"。穿越 / point-in-time 这两个词两篇都会碰，本文讲特征生产视角的细节，不重复一致性那篇的快照体系。

## 1. 全景：特征从原始日志到喂进模型，走的是哪条路

先建立一张地图。一个特征的"一生"，从用户产生行为，到最终变成模型输入的一个数字，要经过**采集 → 生产 → 存储 → 服务/拼样本**四段。而"生产"这一段，分裂成**离线**和**实时**两条物理上独立的链路。

```
                         ┌──────────────────────────────────────────┐
   用户行为/曝光埋点 ───▶│         原始日志（Kafka 流 + 数仓落表）        │
                         └───────────────┬──────────────┬───────────┘
                                         │              │
                       ┌─────────────────┘              └─────────────────┐
                       ▼  （批读历史明细）                  （流式消费增量）  ▼
            ┌────────────────────────┐              ┌────────────────────────┐
            │  离线特征生产 (Spark/批)  │              │  实时特征生产 (Flink/流)  │
            │  T+1，跑大窗口/画像        │              │  秒级，增量更新统计        │
            └───────────┬────────────┘              └───────────┬────────────┘
                        ▼                                        ▼
            ┌────────────────────────┐              ┌────────────────────────┐
            │  离线特征表（数仓/数据湖） │              │  在线特征 KV（Redis/HBase）│
            │  供：拼训练样本、回填      │              │  供：在线服务实时读取      │
            └───────────┬────────────┘              └───────────┬────────────┘
                        │                                        │
                        ▼                                        ▼
              【训练侧】拼样本时取特征               【服务侧】打分时取特征
                        └──────────── 必须是同一个值 ────────────┘
```

**关键认知（这是全篇的地基）**：

- **同一个特征，往往要在两条链路上各生产一遍。** 比如"用户近 7 天点击数"：离线用 Spark 扫 7 天点击明细 group by 算出来，落离线表；实时用 Flink 维护一个 7 天滑窗计数器，写进 KV。两套代码、两个引擎、两个产出位置。
- **离线表服务训练，KV 服务在线打分。** 这是两条链路的"出海口"不同。
- **穿越问题主要发生在"离线链路 + 拼样本/回填"这一侧**，因为离线是"事后站在今天回看历史"，时间维度最容易错位；实时链路天然只能看到"此刻及之前"，反而不容易穿越（但会有延迟/乱序问题，见 §3、§10）。

> 🎓 **老师视角**：很多平台研发把"特征"理解成一张静态的表，这是新手视角。**特征的本质是"某个实体在某个时间点的状态"**——它带着强烈的时间属性。一旦你脑子里给每个特征值都挂上"这是 t 时刻的值"这个隐含维度，穿越、point-in-time、新鲜度这些概念会瞬间通透。评审时你能说出"特征是带时间戳的状态量，不是一张维表"，就已经比一半人站得高了。

### 1.1 为什么非要两条链路，不能合一条

| 诉求 | 离线链路满足 | 实时链路满足 |
|---|---|---|
| 跑**大窗口**特征（近 30/90 天画像） | ✅ 批处理擅长全量扫描 | ❌ 流式维护超大窗口状态成本爆炸 |
| **秒级新鲜**（用户刚点的立刻可见） | ❌ T+1 天级延迟 | ✅ 秒级 |
| **历史回溯/重算**（补历史、改口径重跑） | ✅ 可重跑任意历史区间 | ❌ 流过去了就过去了 |
| **吞吐成本**（一天百亿样本算特征） | ✅ 批量摊薄成本低 | ⚠️ 实时算每条都占资源 |
| **低延迟读取**（服务 P99 几十 ms） | ❌ 扫表太慢 | ✅ KV 点查 |

一句话：**离线管"全、准、可回溯"，实时管"快、新鲜"**。两者目标互斥，所以物理上必须分开，这也正是后续一切麻烦的根源。

## 2. 离线特征生产链路（Spark / 批 / T+1）

### 2.1 它在干什么

离线链路的典型形态：**每天凌晨一个调度任务（T+1），用 Spark/Hive 批量扫昨天（及更早）的行为明细，按实体（user/item/...）聚合出特征，写进离线特征表。**

```
   调度时刻：次日 02:00（T+1）
        │
        ▼
   ┌─────────────────────────────────────────────┐
   │ Spark 批任务                                   │
   │  ① 读 dt=昨天 的行为明细（点击/曝光/转化/停留）  │
   │  ② 读历史画像表（user/item 维度的累积状态）      │
   │  ③ JOIN + GROUP BY + 窗口聚合                  │
   │     - 近 7/30/90 天点击数、CTR、消费额          │
   │     - 用户长期偏好画像（类目分布、活跃度）       │
   │     - item 侧统计（曝光量、历史 CTR、新鲜度）    │
   │  ④ 写离线特征表（按 dt 分区）                   │
   └─────────────────────────────────────────────┘
        │
        ▼
   离线特征表 feature_offline (partition by dt)
        │
        ├──▶ 拼训练样本（与 label join）
        └──▶ 回填（backfill）历史样本
```

### 2.2 离线链路适合产哪些特征

| 特征类型 | 例子 | 为什么放离线 |
|---|---|---|
| **大窗口统计** | 近 30/90 天消费额、活跃天数 | 窗口太大，流式维护状态成本高 |
| **画像类** | 用户长期类目偏好、价格敏感度 | 变化慢，T+1 足够新鲜 |
| **复杂聚合** | 多表 join 后的交叉统计、序列特征 | 批处理表达力强、易回溯 |
| **离线模型产物** | 用户/物品 embedding（双塔离线导出） | 本来就是离线批量算的 |

详见特征分类体系：《[特征工程与特征分类体系](../03-特征数据管理/01-特征工程与特征分类体系.md)》。

### 2.3 离线链路的产出与时间语义（重点）

离线特征表**必须明确两个时间**，否则后面一定穿越：

- **数据时间（event time / 统计截止时间）**：这条特征值统计到了"哪一刻"。比如"截至 6/13 24:00 的近 7 天点击数"。
- **生产时间（processing time / 跑批时刻）**：这个任务"什么时候算出来的"。比如 6/14 02:00 跑出来的。

```
   时间轴 ──────────────────────────────────────────────────▶
        6/13 10:00        6/13 24:00         6/14 02:00
            │                 │                  │
        某次预测 t0       统计截止时间        跑批生产时间
            │                 │                  │
            └──── 如果训练用了"6/13 24:00 截止"的特征去对齐"10:00 的预测"
                  → 多算进了 10:00~24:00 的行为 → 穿越！（见 §6）
```

> 🎓 **老师视角**：离线特征表里"那一行特征值"代表的是"统计截止时间那一刻的状态"，**不是"全天任意时刻都成立的值"**。新手最常见的错误，就是拿一张"按天聚合"的特征表，去和"当天某个具体时刻的预测"做 join，默认这一天里特征值不变。在搜广推里特征是秒级变化的，"一天一个值"这种粗粒度本身就是穿越的温床。平台研发要做的是**让离线表带上足够细的时间版本**，或者用 point-in-time 机制（§8）保证对齐到 t0。

**看代码：一个典型的离线特征生产 SQL（近 7 天点击数）**

```sql
-- 离线 T+1：为每个 user 算"截至 dt 当天 24:00 的近 7 天点击数"
-- 注意：这张表的语义是"以 dt 24:00 为截止时间"的快照
INSERT OVERWRITE TABLE feature_offline PARTITION (dt = '2026-06-13')
SELECT
    user_id,
    COUNT(*)                         AS clicks_7d,   -- 近 7 天点击数
    COUNT(DISTINCT item_id)          AS items_7d,    -- 近 7 天去重商品数
    MAX(click_time)                  AS last_click   -- 末次点击时间（也是个版本线索）
FROM   click_log
WHERE  click_time >= '2026-06-07 00:00:00'
  AND  click_time <  '2026-06-14 00:00:00'           -- 截止 dt 当天 24:00
GROUP  BY user_id;
```

> ⚠️ 注意这张表的"截止时间"是 `dt 当天 24:00`。如果直接拿它去对齐"6/13 白天某次预测"，就把那次预测之后的行为算进去了——这正是 §2.3 强调的"全天一个值"陷阱。要么把粒度做细（如按小时甚至事件级版本），要么走 point-in-time（§8）。

### 2.4 平台研发在离线链路要交付什么

- **统一的特征生产框架/模板**：让算法同学声明式地配特征（窗口、聚合函数、实体键），平台编译成标准 Spark 任务，而不是每人手写一坨 SQL。
- **时间分区与版本规范**：强制每张特征表带 `event_time`/`dt` 且语义明确，禁止"全天一个值"的粗口径直接进样本。
- **血缘与口径登记**：特征从哪些源表算来、用了什么窗口，登记可查（防 §4 口径飘移、便于排查穿越）。
- **幂等与可重跑**：任务必须支持"重跑某一天分区结果不变"，否则改口径回溯、补数据时会产生不可复现的特征值。
- **回填入口**：见 §5。

## 3. 实时特征生产链路（Flink / 流 / 秒级）

### 3.1 它在干什么

实时链路：**Flink 作业消费行为流（Kafka），用状态（state）维护实时统计，秒级更新后写进在线 KV，供服务直接读。**

```
   行为流（Kafka）：点击/曝光/加购/下单 …… 持续流入
        │
        ▼
   ┌─────────────────────────────────────────────┐
   │ Flink 流作业                                   │
   │  ① keyBy(user_id / item_id)                   │
   │  ② 维护 keyed state：滑动窗口计数、累加器        │
   │     - session 内点击数（事件触发即更新）         │
   │     - 近 1 小时/近 1 天滑窗 CTR                 │
   │     - 实时序列（最近 N 次点击 item 列表）         │
   │  ③ 处理 event time + watermark（应对乱序）      │
   │  ④ sink 写在线 KV（Redis/HBase/Tair）          │
   └─────────────────────────────────────────────┘
        │
        ▼
   在线特征 KV：key=user_id, value={session_clicks:2, rt_ctr_1h:0.03, ...}
        │
        ▼
   在线服务打分时点查 KV，拿到"此刻"的实时特征
```

### 3.2 实时链路适合产哪些特征

| 特征类型 | 例子 | 为什么放实时 |
|---|---|---|
| **session 级行为** | 本次会话点了几个、看了哪些类目 | 必须秒级，离线 T+1 完全失效 |
| **短窗口实时统计** | 近 5 分钟/1 小时点击、实时 CTR | 时效性直接决定效果 |
| **实时序列** | 最近 N 次交互的 item 序列 | DIN/DIEN 这类模型强依赖 |
| **实时热度/趋势** | item 近 10 分钟曝光暴涨 | 抓热点、打击爆款滞后 |

### 3.3 实时链路的核心难点：乱序、延迟、状态

实时链路一般**不容易"穿越"**（它物理上只能消费到"已发生"的事件），但它有自己的三座大山：

```
   理想：事件按时间顺序到达
   现实：网络抖动/分区/重试 → 事件乱序、迟到

   事件真实发生:  e1(10:00:01)  e2(10:00:02)  e3(10:00:03)
   实际到达顺序:  e1            e3            e2(迟到!)
                                              ↑
                          如果 watermark 已经推过 10:00:02，
                          e2 可能被丢弃或进侧输出 → 特征算少了
```

- **乱序（out-of-order）**：靠 **event time + watermark** 容忍一定迟到，但 watermark 越宽容、特征越"等得久"，新鲜度越差——这是一对矛盾（见 §10）。
- **延迟/迟到数据**：超过 watermark 容忍度的迟到事件被丢，导致实时特征"少算"——这正是离在线 diff 的常见来源（§9）。
- **状态规模与一致性**：keyed state 随 key 基数（亿级用户）膨胀，要 checkpoint、要考虑 failover 后状态恢复，否则重启丢状态、特征跳变。

> 🎓 **老师视角**：实时链路的"穿越"风险低，但"算不准"风险高。两条链路的病灶其实是镜像的——**离线病在"时间错位（穿越）"，实时病在"数据不全（迟到丢失）"**。平台研发评审实时特征时，头号要追问的不是"会不会穿越"，而是"watermark 多大、迟到数据怎么处理、failover 状态会不会丢"。

### 3.4 实时特征也会"反向穿越"（容易被忽略）

实时链路虽难穿越，但有一种隐蔽的"反向穿越"：**实时作业的处理时点早于事件本应纳入特征的时点**。比如做训练样本时，如果用"实时 KV 在某后续时刻的值"去回灌历史样本，而不是"打分那一刻 KV 的值"，就会把后到的实时更新混进来。

```
   预测 t0=10:00:00，打分时 KV 里 session_clicks=2
   10:00:03  迟到事件到达，KV 更新为 session_clicks=3
   ❌ 若拼样本时直接读"现在的 KV"=3 → 比 t0 那一刻多了 1 → 反向穿越/不一致
   ✅ 正确：用打分那一刻的快照值=2（见一致性篇的特征快照）
```

> ⚠️ 这说明：**即使特征生产在实时链路，拼样本时也不能"事后再读 KV"**——KV 是会被后续事件持续覆盖的，读到的永远是"现在"而非"t0 那一刻"。这正是特征快照（feature logging）存在的根本原因之一，详见《[样本离在线一致性](../04-样本数据管理/样本离在线一致性.md)》§5。

### 3.5 平台研发在实时链路要交付什么

- **统一的实时特征作业框架**：封装 keyBy / 窗口 / 状态 / sink，算法同学配置即可，避免人人手写 Flink。
- **watermark / 迟到策略可配且可观测**：把"丢了多少迟到数据"做成指标，否则实时特征"悄悄算少了"没人知道。
- **KV 写入的新鲜度监控**：从"事件发生"到"KV 可读"的端到端延迟（§10）。
- **状态规模与 checkpoint 治理**：状态后端（如 RocksDB）选型、TTL 清理过期 key、checkpoint 周期与对齐，防止状态爆炸拖垮作业。
- **KV 写入的版本/时间戳**：sink 时带上 event_time，便于服务侧判断新鲜度、便于离在线 diff 定位。

## 4. 两条链路对照与"双跑同一特征"的陷阱

### 4.1 离线 vs 实时：一张对照表

| 维度 | 离线特征生产 | 实时特征生产 |
|---|---|---|
| 引擎 | Spark / Hive（批） | Flink（流） |
| 触发 | 调度，T+1（天级/小时级） | 事件驱动，秒级 |
| 数据源 | 数仓历史明细（全量） | 行为流（增量） |
| 产出落地 | 离线特征表（数仓/数据湖） | 在线 KV（Redis/HBase） |
| 主要服务对象 | 拼训练样本、回填 | 在线打分 |
| 擅长 | 大窗口、画像、可回溯重算 | 短窗口、session、实时序列 |
| 主要风险 | **穿越**（时间错位） | **迟到丢数**（算不全）、状态爆炸 |
| 新鲜度 | 天级 | 秒级 |
| 重算/回溯 | ✅ 易 | ❌ 难（流过去就没了） |

### 4.2 致命陷阱：同一特征双跑，口径必然飘

很多"中窗口"特征（如近 1 天点击数）会**同时**在离线和实时各产一份——离线 T+1 算一份进样本，实时算一份进 KV 供服务。**两套代码、两拨人、两个迭代节奏，口径一定会飘**：

```
   特征：用户近 1 天点击数
   ┌─────────────────────────┐      ┌─────────────────────────┐
   │ 离线 Spark               │      │ 实时 Flink               │
   │  "近 1 个自然日"          │  vs  │  "滚动 24 小时"           │
   │  缺失填 0                 │      │  缺失填 -1                │
   │  按 user+item+hour 去重   │      │  未去重                   │
   └─────────────────────────┘      └─────────────────────────┘
          同一条样本两边能差出一倍 → 离在线不一致
```

> ⚠️ 这个"两套实现口径飘移"是**离在线一致性问题**的根因之一，本文不展开治理（快照、统一定义等），**详见《[样本离在线一致性](../04-样本数据管理/样本离在线一致性.md)》§4.2 / §7**。本文这里只点明：**特征生产链路天然分裂成两条，是一致性问题的物理来源。** 解一致性，要么消灭两套计算（特征快照），要么统一口径（统一特征定义 / Feature Store，见 §11 与《[特征平台架构](../03-特征数据管理/02-特征平台(Feature Store)架构.md)》）。

> 🎓 **老师视角**：当算法同学说"这个中窗口特征到底放离线还是实时"，你要能反问一句关键的："**它要不要进在线服务？要的话就得双跑，那一致性怎么保？走快照还是统一定义？**" 这一问就把对话从"功能实现"拉到了"架构权衡"，瞬间体现你的段位。

## 5. 特征回填 backfill：新特征上线要补历史

### 5.1 为什么需要回填

新特征不会凭空有历史。算法同学想加一个特征"用户近 30 天加购次数"，但：

- **训练要历史样本**：模型要用过去 N 天的样本训练，可这些老样本里**根本没有这个新特征的值**。
- **解法只有一个：回填（backfill）**——站在今天，用历史明细，把每条老样本在它**各自预测时刻 t0** 的这个特征值"补算"出来。

```
   今天 = 6/14，新特征"近30天加购数"上线
        │
        ▼
   要训练用 5/15 ~ 6/13 的历史样本
        │
        ▼
   每条历史样本都缺这个特征 → 回填：
     对 5/20 10:00 的那条样本 → 算"截至 5/20 10:00 的近30天加购数"
     对 6/01 15:00 的那条样本 → 算"截至 6/01 15:00 的近30天加购数"
        ▲
   关键：要算"那条样本预测时刻 t0 的值"，不是"今天的值"
```

### 5.2 回填为什么是穿越重灾区

回填的本质是**"站在未来（今天）去重算过去某一刻的特征"**——你手上**握着全部未来数据**，太容易顺手就把未来算进去了。

```
   ❌ 错误回填（穿越）：
      对 5/20 的样本，直接算"截至今天 6/14 的近30天加购数"
      → 把 5/20 之后到 6/14 的加购行为全算进去了 → 严重穿越

   ❌ 更隐蔽的错误：
      对 5/20 的样本算"以 5/20 为右边界的近30天"，但右边界用了"5/20 当天 24:00"
      而样本预测发生在 5/20 10:00 → 多算了 10:00~24:00 → 仍然穿越

   ✅ 正确回填：
      对每条样本，严格以它各自的 t0 为右边界（exclusive），只算 t0 之前的行为
      → 这就是 point-in-time 回填（§8）
```

> 🎓 **老师视角**：回填是平台研发**最容易翻车、也最能体现专业度**的活儿。新手写个 SQL `WHERE event_time >= t0 - 30d` 就跑了，漏了上界 `AND event_time < t0`，离线 AUC 蹭一下涨上去，皆大欢喜——直到上线暴跌。**回填工具必须把"以每条样本各自的 t0 为右边界"做成框架级强约束，不让算法同学手写时间条件。** 这是平台的护城河。

**看代码：错误回填 vs 正确 point-in-time 回填**

```sql
-- ❌ 错误：所有样本共用一个固定右边界（甚至"今天"），漏了"各样本各自 t0"
SELECT s.req_id, s.t0,
       COUNT(*) AS addcart_30d
FROM   samples s
JOIN   addcart_log a ON a.user_id = s.user_id
WHERE  a.event_time >= s.t0 - INTERVAL 30 DAY        -- 只限下界！
GROUP  BY s.req_id, s.t0;

-- ✅ 正确：以每条样本各自的 t0 为右边界（exclusive），逐样本卡死
SELECT s.req_id, s.t0,
       COUNT(*) AS addcart_30d
FROM   samples s
JOIN   addcart_log a ON a.user_id = s.user_id
WHERE  a.event_time >= s.t0 - INTERVAL 30 DAY
  AND  a.event_time <  s.t0                           -- 关键：上界 = 该样本 t0
GROUP  BY s.req_id, s.t0;
```

> ⚠️ 两段 SQL 只差一行 `AND a.event_time < s.t0`。这一行就是"防穿越命门"。但更深的问题是：**这种逐样本时间范围 join 在亿级样本上极其昂贵**（每条样本都要按自己的 t0 扫一段历史）——这正是 §5.3 说回填贵的根本原因，也是为什么生产级回填要用专门的 as-of join 算子而非朴素 SQL（见 §8.3）。

### 5.3 回填的算力成本（评审会被问钱）

回填是**一次性的大规模重算**，成本经常被低估：

> 📊 **量级感**：回填要对"历史样本数 × 特征复杂度"做全量计算。假设回填 30 天、每天百亿曝光、要算几十个新特征，相当于把"30 天的离线特征生产"集中在一次任务里跑完，且每条样本要做 point-in-time 对齐（as-of join 比普通 join 重得多）。这类任务**消耗的算力常是日常 T+1 任务的几十倍**，跑几个小时到一天很常见，还会和日常调度抢资源。

回填成本控制手段：

| 手段 | 做法 | 收益 |
|---|---|---|
| **限定回填窗口** | 只回填训练真正需要的 N 天，不贪多 | 线性降本 |
| **负样本采样回填** | 只对采样后的负样本回填，正样本全回 | 省一大半 |
| **错峰调度** | 避开日常 T+1 高峰，用离线集群空闲时段 | 不挤占线上 |
| **增量物化** | 把中间聚合结果物化复用，避免每个特征重扫明细 | 省扫描 IO |
| **批量上线** | 攒一批新特征一起回填，共享一次明细扫描 | 摊薄扫描成本 |
| **按天分区并行** | 把回填区间按 dt 切片并行跑，缩短墙钟时间 | 提速（不降总量） |

> ⚠️ **采样回填的坑**：对负样本采样后样本正负比变了，预估值会偏高，下游要做采样率校准（calibration）。这点与负采样一致，能主动提说明你懂"省成本不破坏建模正确性"。

### 5.4 平台研发要交付的回填工具

- **声明式回填入口**：算法同学只声明"特征定义 + 回填区间 + 样本表"，平台负责按 t0 安全回填。
- **强制 point-in-time**：右边界恒为每条样本各自 t0，框架层面杜绝手写时间条件穿越。
- **成本预估与配额**：提交回填前预估扫描量/算力，给配额防止打爆集群。
- **进度与可重入**：回填可能跑很久，要支持断点续跑、按分区幂等，失败重跑不产生脏数据。
- **回填后的穿越自检**：回填完跑一遍穿越检测（§6.4 / §9），通过才允许进训练样本。

## 6. 特征穿越 / 未来信息泄漏：定义与典型场景

### 6.1 定义

> **特征穿越（feature leakage / data leakage / 未来信息泄漏）**：训练样本里某个特征，用到了"预测时刻 t0 之后才产生、或本不该在 t0 知道"的信息。等价于"模型在训练时偷看了一部分答案"。

时间轴一张图说清：

```
   预测时刻 t0
        │  ←─── 合法：只能用 t0 之前已知的信息 ───┤
   ─────●━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━▶ 时间
        │                                    ↑
        │                              "未来" = 穿越禁区
        └── 任何越过 t0 取到的信息，都是穿越
```

### 6.2 三大典型穿越场景（务必背熟）

**① 用了 t0 之后才知道的统计值（窗口右边界越界）**

```
   特征：item 当天总曝光数
   样本 t0 = 6/13 10:00
   ❌ 用了"6/13 全天（24:00 截止）的总曝光" → 把 10:00~24:00 的未来曝光算进来了
   ✅ 只能用"截至 6/13 10:00 的累计曝光"
```
这是回填最常见的穿越（§5.2）。

**② 用了 label 衍生 / 与 label 同源的特征（最隐蔽、最致命）**

```
   场景：预测"用户是否会下单"
   ❌ 特征里混进了"本次会话最终是否产生支付" → 这就是 label 本身的变体！
   ❌ 特征"订单金额" → 只有下单了才有金额，等于直接泄漏了 label
   ❌ 特征"用户当天总转化数" → 包含了这条样本自己的那次转化
```
label 泄漏往往藏在派生特征里，AUC 会高得离谱（接近 1.0 是强烈信号）。

**③ 滑窗右边界错位 / 统计口径含当前事件**

```
   特征：用户近 1 小时点击数
   ❌ 滑窗 [t0-1h, t0]（闭区间，含 t0） → 把"当前这次点击/这次曝光"算进去
      尤其当 label 就是"是否点击"时，等于把答案算进特征
   ✅ 滑窗 [t0-1h, t0)（右开区间，严格小于 t0）
```

### 6.3 穿越场景速查表

| 场景 | 病因 | 典型表现 |
|---|---|---|
| 窗口右边界越过 t0 | 回填/拼样本只限下界没限上界 | 离线 AUC 虚高 0.01~0.05 |
| 用了 label 衍生特征 | 特征与标签同源 | AUC 异常高（接近 1） |
| 滑窗含当前事件 | 区间闭合错（用了闭区间） | 自相关，强信号特征虚高 |
| 用了"最新值"维表 | 维表只存当前值，无版本 | join 到的是今天的值而非 t0 |
| 跨样本聚合含未来 | "用户当天总数"含本条之后的样本 | 全天聚合类特征普遍虚高 |
| 数据延迟反向穿越 | 离线补齐了在线当时还没到的数据 | 离在线 diff（详见一致性篇 §4.3） |

> 🎓 **老师视角**：穿越和"离在线不一致"是**两个相关但不同**的问题，评审时一定要分清，否则会被算法同学问住。
> - **穿越** = 时间错位，离线**单边**就错了（哪怕没有在线对照，它也是错的，因为它用了未来）。
> - **不一致** = 离线值 ≠ 在线值（两边对不上）。
> 一个特征可以"不穿越但不一致"（口径飘了），也可以"一致但穿越"（离在线都用了未来、错得一样）。本文治穿越，《[样本离在线一致性](../04-样本数据管理/样本离在线一致性.md)》治不一致。**两道防线都要有。**

### 6.4 平台怎么检测穿越（不能只靠人 review）

- **时间边界静态检查**：扫描特征定义/回填 SQL，凡是窗口缺上界、用闭区间含 t0、join 未带版本时间的，直接告警/拦截。
- **AUC 异常检测**：单特征 AUC（用单个特征预测 label 的区分度）突然爆高、或加入某特征后整体 AUC 跳涨"好得不真实"，触发穿越嫌疑（§7）。
- **特征-label 时间一致性校验**：自动核对每个特征的"统计截止时间 ≤ 样本 t0"。
- **shuffle 检验**：把特征值在时间上打乱后 AUC 应大幅下降；若打乱后 AUC 仍高，说明特征里有非时序泄漏（label 同源）。

## 7. 为什么穿越会让"离线 AUC 虚高、上线就崩"

这是评审里最值钱的一段，能讲透你就立住了。

### 7.1 机理

```
   训练时（离线）：              上线时（在线）：
   特征 X 偷看了 t0 之后的       特征 X 只能拿到 t0 之前的真实信息
   "答案相关信息"                （未来还没发生，根本拿不到）
        │                              │
        ▼                              ▼
   模型学到："X 高 → label=1"      X 退化成普通信息，没有预测力
   （其实是因为 X 本身含了 label）       │
        │                              ▼
        ▼                        线上模型抓不到那个"虚假强信号"
   离线 AUC 虚高 🎉              线上效果暴跌 📉
```

**核心**：穿越特征在离线**和 label 高度相关（因为它本身就含未来/含 label）**，模型把它当成强信号学进去了；但上线时这个特征拿不到未来，**信号消失**，模型像被抽掉了拐杖。

### 7.2 事故时间线（典型画像）

```
  Day 0  算法：新特征上线，离线 AUC 0.80 → 0.84，巨大提升 🎉
  Day 1  灰度：线上 CTR 不涨反跌 📉
  Day 2  全量：CTR 跌 4%，老板问话
  Day 3  排查：怀疑模型、怀疑分布、怀疑特征服务……
  Day 5  有人查到新特征是"用户当天总转化数"——含了本条样本之后的转化
         → 典型 label 穿越
  Day 6  去掉该特征重训，AUC 回落到 0.805（真实水平），线上 CTR 回升 ✅
```

> 📊 **量级感**：穿越导致的离线 AUC 虚高，常见 0.01~0.05。在搜广推里 AUC 真实涨 0.001 都值得开香槟，所以**"某个新特征让 AUC 跳涨 0.02 以上"几乎是穿越的红色警报**——好得不真实，先查穿越，别急着上线。

> 🎓 **老师视角**：把这句话刻进 DNA——**"离线指标涨得越夸张，越要先怀疑穿越，而不是先庆祝。"** 你在评审里主动说"这个提升幅度不太正常，先跑个穿越检测再上线"，比任何技术细节都更能让算法和老板信任你。这是从"被动接需求"到"主动把关质量"的分水岭。

### 7.3 两类穿越的判别：时间穿越 vs label 穿越

虽然都叫"穿越"，两类的成因和体检手段不同，分清才能对症下药：

| | 时间穿越（窗口越界/滑窗错位） | label 穿越（特征与标签同源） |
|---|---|---|
| 成因 | 取了 t0 之后的统计值 | 特征里直接/间接含了 label 信息 |
| AUC 表现 | 虚高 0.01~0.05 | 极端虚高，常接近 0.9+ 甚至 1.0 |
| 体检手段 | 检查窗口右边界 / point-in-time | 检查特征定义是否依赖结果字段 |
| 修复 | 卡死 t0 上界、用右开区间 | 删除该特征或重新定义口径 |
| 隐蔽度 | 中（SQL 一行就能查） | 高（藏在派生特征里，要追血缘） |

> 🎓 **老师视角**：时间穿越靠"时间边界检查"能机器化拦截；label 穿越更阴险，因为它可能藏在一个看似无害的派生特征里（比如"用户当天会话价值"，而会话价值是由后续转化算出来的）。**抓 label 穿越要靠特征血缘——追这个特征到底是从哪些源字段算来的，有没有沾到结果侧字段。** 这就是为什么 §2.4 强调"血缘登记"是平台必交付项。

## 8. Point-in-Time 正确性：防穿越的核心机制

### 8.1 核心定义

> **Point-in-Time（PIT）正确性 / 时点正确**：拼样本或回填时，每条样本只能 join 到"它各自预测时刻 t0 及之前"的特征值——取的是"**那一刻的快照**"，而不是"现在的最新值"。

```
   ❌ 普通 join（穿越）：             ✅ point-in-time join（as-of join）：
   样本 ⋈ 特征表."当前最新值"        样本.t0 ⋈ 特征表中 version_time ≤ t0 的"最近一条"
```

### 8.2 前提：特征必须是"带版本时间戳的变更流"，而非"当前值维表"

普通维表只存"现在的值"，无法回到过去。PIT 要求特征以**变更记录**存储——每次值变化留一条，带 `version_time`：

```
  user_id | feature      | value | version_time
  A       | 近7天点击数   | 4     | 6/13 09:30
  A       | 近7天点击数   | 5     | 6/13 09:58   ◀── 样本 t0=10:00，应 join 这条（≤t0 的最近一条）
  A       | 近7天点击数   | 9     | 6/13 23:00   ◀── 这是"未来"，PIT 必须排除
```

样本 t0=10:00 → as-of join 向前找到最近一条 `version_time ≤ 10:00` 的，即 `value=5`。绝不会取到 23:00 的 9。

### 8.3 离线如何用事件时间做 PIT join

```
   ┌──────────────┐         ┌─────────────────────────┐
   │  样本表        │         │  特征变更流表             │
   │  req_id, t0   │         │  key, value, version_time │
   └──────┬───────┘         └────────────┬────────────┘
          │                              │
          └──────── as-of join ──────────┘
             ON key 相等
             AND version_time <= t0          ← 卡死上界，防穿越
             取满足条件中 version_time 最大的那条（最接近 t0 的过去值）
```

实现要点：

- **本质是 as-of join（按时间向前最近匹配）**：Spark 可用 range join + 窗口函数排序取 top-1，或专用 as-of join 算子；Flink 有 temporal table join。
- **右边界严格 `<= t0`（或 `< t0`，取决于是否含 t0 当刻）**：这一行就是防穿越命门。
- **比普通 join 重**：要排序、要找最近邻，数据放大明显——这也是回填贵的原因（§5.3）。

**看代码：用窗口函数实现 as-of join（Spark SQL 思路）**

```sql
-- 对每条样本，取特征变更流里 version_time <= t0 的、最接近 t0 的那一条
WITH joined AS (
  SELECT
      s.req_id, s.t0, f.value, f.version_time,
      ROW_NUMBER() OVER (
        PARTITION BY s.req_id
        ORDER BY f.version_time DESC        -- 离 t0 最近的过去值排第一
      ) AS rn
  FROM   samples s
  JOIN   feature_changelog f
    ON   f.key = s.user_id
   AND   f.version_time <= s.t0             -- 防穿越命门：只取 t0 之前
)
SELECT req_id, t0, value AS feat_value
FROM   joined
WHERE  rn = 1;                              -- 每条样本只保留最近的一条
```

> ⚠️ 这种"每条样本 join 一段历史再排序取 top-1"的模式，在亿级样本上会产生巨大的中间数据放大。生产里常用：① 把样本和变更流都按时间排序后做单遍归并（merge as-of），避免全量 join + 排序；② 限制变更流回看窗口（如只看 t0 前 N 天的变更）。这也是为什么平台要把 PIT join 封装成专用算子，而不是让每个人手写上面这段 SQL。

### 8.4 PIT 的局限（评审要主动说）

```
   PIT 解决了"时点"（不穿越），但没解决"口径"
        │
        ▼
   特征值仍是"离线这套逻辑"算的；
   若离线算子和在线算子口径不同 → 仍然离在线不一致（§4.2）
```

> 🎓 **老师视角**：PIT vs 特征快照的本质区别——**快照是"把线上当时的答案抄下来"，PIT 是"按规则把过去那一刻的答案重算出来"**。能抄就别算（快照天然不穿越且一致），所以快照优先；PIT 是快照覆盖不到时的兜底——**正是 §5 的历史回填、冷启动、新特征追溯场景**。这两个机制的分工，详见《[样本离在线一致性](../04-样本数据管理/样本离在线一致性.md)》§5/§6，本文不重复快照体系。

## 9. 特征监控：漂移 / 缺失 / 覆盖 / 离在线 diff

特征生产出来不等于万事大吉。**特征会无声地坏掉**——上游格式变了、Flink 作业挂了、口径改了没同步，模型却照常吃着"坏特征"在跑。监控是让坏特征"出声"的唯一办法。

### 9.1 四类必须上看板的监控

```
   ┌─────────────────────────────────────────────────────────┐
   │  ① 分布漂移 (drift)   特征分布今天 vs 历史 偏了多少          │
   │  ② 缺失率 (missing)   该特征为 null/默认值的占比             │
   │  ③ 覆盖率 (coverage)  有多少样本/实体能取到这个特征          │
   │  ④ 离在线 diff        同一特征 离线值 vs 在线值 不一致占比     │
   └─────────────────────────────────────────────────────────┘
```

| 监控项 | 含义 | 常见病因 | 经验阈值（举例） |
|---|---|---|---|
| **分布漂移 PSI** | 特征分布相对基线偏移度 | 上游变更、口径改、流量突变 | PSI<0.1 健康，>0.25 严重 |
| **缺失率** | null/默认值占比 | 上游字段缺、join 不上、作业延迟 | 突变即告警 |
| **覆盖率** | 能取到该特征的样本占比 | 新实体冷启动、KV 没写到 | 低于历史基线告警 |
| **离在线 diff** | 离线值≠在线值的占比 | 两套口径飘、迟到丢数 | 关键特征<0.1% |

### 9.2 分布漂移（drift）：最常用的健康信号

```
   基线分布（上周）        今天分布
      ▁▃▅▇▅▃▁      vs      ▁▃▇▇▃▁▁
          │                    │
          └──── PSI 计算偏移度 ──┘
   PSI < 0.1   分布稳定，健康
   0.1~0.25    有偏移，关注
   > 0.25      显著漂移，模型可能失效，告警
```

> 📊 **名词点**：**PSI（Population Stability Index，群体稳定性指数）** 是搜广推/风控衡量分布漂移的标配。能脱口而出"PSI > 0.25 要告警"，是典型的内行信号。注意区分两种漂移：**特征漂移（feature/data drift，输入分布变了）** 和 **概念漂移（concept drift，输入→label 的关系变了）**——前者特征监控能抓，后者要靠模型效果监控。

**PSI 怎么算（评审被追问能答出来）**：把特征值分成 N 个桶，比较"基线分布"和"当前分布"在每个桶的占比：

```
   PSI = Σ over buckets ( (cur% - base%) × ln(cur% / base%) )

   例：某特征分 5 桶
   桶   base%   cur%    (cur-base)·ln(cur/base)
   B1   0.20    0.10    (-0.10)·ln(0.5)  = 0.069
   B2   0.30    0.25    ...
   ...                  累加 → PSI
   类别型特征直接按取值分桶；连续型按分位数/等宽分桶。
```

> ⚠️ PSI 的工程坑：① 桶的边界要固定（用基线期的分位数定桶），否则桶一变 PSI 没意义；② 某桶占比为 0 会让 ln 爆炸，要做平滑（加极小值）；③ 节假日/大促分布天然突变，要支持"豁免窗口"避免误报。

> 🎓 **老师视角**：漂移看板最容易犯的错是"只看全局均值"。一个特征均值没动，但**分布形状变了（双峰变单峰）、或某个高价值人群的缺失率飙了**，全局均值完全看不出来。好的看板要**分位数 + 空值率 + 分人群/分场景切片**，而不是一个干巴巴的均值曲线。这是平台研发能体现"懂数据"的地方。

### 9.3 缺失率与覆盖率：一对最容易混淆的指标

这两个常被混为一谈，评审时分清能加分：

```
   覆盖率 (coverage)：分母是"全体应该有值的实体/样本"
       = 能取到该特征的样本数 / 总样本数
       低 → 这个特征"很多对象根本没有"（如新用户没有历史行为画像）

   缺失率 (missing)：通常指"本来有、现在突然取不到/变默认值"
       = null 或默认值的样本数 / 总样本数
       突增 → 生产链路坏了（上游断流、join 失败、KV 没写到）
```

- **覆盖率天然偏低不一定是 bug**：冷启动特征对新用户覆盖率本就低，关键看"相对历史基线有没有掉"。
- **缺失率突增几乎一定是事故**：昨天 2%、今天 40%，说明上游表没产出或 join 键变了——这是最该秒级告警的信号。

> 🎓 **老师视角**：缺失率监控最大的价值是**当哨兵**。模型对缺失不敏感（默认值填上照样打分），所以特征"消失"不会报错、不会崩，只会让模型悄悄变蠢。**缺失率突变告警，往往是你比算法同学更早发现"特征链路挂了"的唯一手段。** 把它做成强告警，是平台研发刷存在感的高性价比动作。

### 9.4 离在线 diff 监控

这一项与一致性篇高度相关，本文只点接口：定期抽样请求，对同一特征比对"离线重算/快照值"与"在线 KV 值"，统计逐特征不一致率。**详细的校验手段（采样比对/影子重算/分布监控）见《[样本离在线一致性](../04-样本数据管理/样本离在线一致性.md)》§8，本文不重复。**

### 9.5 平台研发要交付的监控能力

- **特征级监控自动化**：新特征上线即自动接入漂移/缺失/覆盖监控，无需手配。
- **基线管理**：自动维护历史基线，支持节假日/大促等已知分布突变的"豁免标注"，减少误报。
- **分层告警**：关键特征（高重要度）严阈值，长尾特征宽阈值，避免告警疲劳。
- **下钻能力**：从告警一键下钻到"哪个场景/人群/上游表"出的问题。

## 10. 特征时效与新鲜度（freshness）

### 10.1 什么是新鲜度

> **特征新鲜度（freshness）**：从"用户行为真实发生"到"该行为反映进特征、被服务读到"的**端到端延迟**。

```
   行为发生 ───▶ 进 Kafka ───▶ Flink 处理 ───▶ 写入 KV ───▶ 服务读到
   t_event        +Δ1            +Δ2            +Δ3          t_read

   新鲜度延迟 = t_read - t_event = Δ1 + Δ2 + Δ3
   （实时特征经验值：亚秒级 ~ 数秒级；越实时的场景要求越紧）
```

### 10.2 新鲜度为什么影响效果

```
   用户刚点了 3 个"运动鞋" → 强烈实时兴趣信号
   ┌─────────────────────────────────────────────┐
   │ 新鲜度 1 秒：下一刷立刻推运动鞋 → 转化↑           │
   │ 新鲜度 5 分钟：用户早划走了 → 信号失效            │
   │ 新鲜度 T+1：明天才生效 → 完全错过窗口            │
   └─────────────────────────────────────────────┘
```

对**实时兴趣、session、热点**类特征，新鲜度几乎直接决定效果；对**长期画像**类特征，新鲜度天级足够，过度追求实时反而浪费成本。

### 10.3 新鲜度 vs 准确性 vs 成本：三角权衡

```
            准确性（等齐迟到数据再算）
                  ╱╲
                 ╱  ╲
                ╱    ╲
       新鲜度 ╱        ╲ 成本
      （越快越好）     （实时算越多越贵）
```

- **watermark 容忍度大** → 等迟到数据 → 准确性↑ 但新鲜度↓。
- **watermark 容忍度小** → 算得快 → 新鲜度↑ 但可能丢迟到数据、准确性↓。
- **更新频率高/窗口细** → 实时性↑ 但 Flink 资源、KV 写压力、成本↑。

| 特征类型 | 新鲜度要求 | 更新频率 | 取舍倾向 |
|---|---|---|---|
| session 实时行为 | 亚秒~秒级 | 事件触发 | 新鲜度优先，可容忍少量丢迟到 |
| 短窗口实时统计 | 秒~分钟级 | 滑窗/微批 | 新鲜度与准确平衡 |
| 中窗口（近1天） | 分钟~小时级 | 定时刷 | 准确优先 |
| 长期画像 | 天级 | T+1 | 成本优先，无需实时 |

> 🎓 **老师视角**：新手以为"特征越新鲜越好"，于是把什么都往实时塞——这是成本灾难。**新鲜度是有边际效益的**：session 特征晚 1 秒可能就废了，长期画像晚 1 天毫无影响。平台研发的价值在于**按特征类型给出"恰到好处的新鲜度"**，而不是无脑全实时。评审时能说"这个特征不需要实时，T+1 就够，省下的算力可观"，是懂成本的表现。

### 10.4 更新频率对效果的影响（与新鲜度相关但不同）

新鲜度是"延迟"，更新频率是"多久刷一次"，两者都影响效果但侧重不同：

```
   事件触发更新：每来一个事件就更新特征   → 最新鲜，但写放大、成本最高
   微批更新：    每 N 秒/N 条批量更新一次   → 平衡新鲜度与写压力
   定时刷新：    每小时/每天重算一次        → 成本低，新鲜度差
```

- **更新太疏**：特征"跟不上"用户状态变化，实时兴趣类特征效果掉得最明显。
- **更新太频**：KV 写 QPS 暴涨、状态频繁刷盘，成本与稳定性双输；且对慢变特征是浪费。
- **典型经验**：高频行为特征（session/实时序列）走事件触发或秒级微批；统计类走分钟级微批；画像类走小时/天级。

> 📊 **量级感**：把一个亿级用户的特征从"小时级刷新"提到"秒级事件触发"，KV 写 QPS 可能从万级跳到百万级，Flink/存储成本可能翻数倍。**所以"提高更新频率"在评审上等价于"加钱"，必须用效果收益来论证。**

> 🎓 **老师视角**：算法同学常说"这个特征能不能更实时点"，背后真实诉求往往是"效果差，想试试更新鲜会不会好"。你要能反问："是新鲜度不够（延迟大），还是更新频率不够（刷太疏），还是根本是口径/覆盖问题？" 把"我要更实时"拆成可定位的子问题，是平台研发的专业价值所在。

### 10.5 平台研发要交付的新鲜度能力

- **端到端新鲜度埋点**：测量 `t_read - t_event`，做成 P50/P99 延迟看板。
- **新鲜度 SLA 分级**：按特征类型定义新鲜度目标，超标告警。
- **更新频率可配**：让特征按类型选择事件触发/微批/定时，平台兜住成本与稳定性。
- **迟到数据可观测**：暴露"丢弃的迟到事件量"，让"实时特征悄悄算少了"显形（呼应 §3.3）。

> 🎓 这一节是"显视野"的弹药库。汇报时点到这些，老板会觉得你调研过行业。

| 厂商/方案 | 特征生产与防穿越的核心做法 | 关键词 |
|---|---|---|
| **Uber（Michelangelo / Palette）** | 离在线统一特征定义，DSL 描述，双 runtime；特征库 Palette | 统一定义, 双 runtime |

| **Google（TFX / TFT）** | Transform 把预处理逻辑做成训练/服务共享的计算图，从源头防 skew | TFT, transform graph |
| **Meta** | 大规模 feature logging（快照）为主，弱化离线重算，规避穿越 | feature logging |
| **阿里** | 特征中心 + 实时特征（Flink），离在线一致与实时化并重 | 特征中心, 实时特征 |

- **快照派**：把线上当时的特征值抄下来（feature logging），物理上不穿越、不需要 PIT 回算。
- **重算派 / Feature Store**：用统一定义 + point-in-time join，离线按规则准确重算，支持历史回填。
- **成熟平台两者结合**：快照保线上链路，PIT + 统一定义保历史回填与回溯。

## 12. 权衡取舍

> 🎓 评审里只会喊"防穿越"不够，能讲清取舍才是内行。

**离线 vs 实时生产（放哪条链路）**

| 维度 | 放离线 | 放实时 |
|---|---|---|
| 新鲜度 | ✗ 天级 | ✓ 秒级 |
| 大窗口能力 | ✓ 强 | ✗ 状态成本爆炸 |
| 可回溯/回填 | ✓ 易 | ✗ 难 |
| 穿越风险 | ⚠️ 高（要 PIT 治理） | ✓ 低 |
| 成本 | ✓ 摊薄低 | ⚠️ 实时算每条都占资源 |

**防穿越机制：快照 vs PIT 重算**

| 维度 | 特征快照 | Point-in-Time 重算 |
|---|---|---|
| 防穿越 | ★★★★★ 物理不可能含未来 | ★★★★ 实现对了才不穿越 |
| 历史回填 | ✗ 没打点的历史拿不到 | ✓ 可回填任意历史 |
| 存储/算力 | 存储高（快照大） | 算力高（as-of join 重） |
| 口径一致 | ★★★★★ 就是线上值 | ★★★ 仍受离线口径影响 |

**典型组合策略**：主力靠快照防线上链路穿越与不一致；**新特征上线/历史回填靠 PIT 兜底**；中窗口双跑特征推统一定义降一致性风险；新鲜度按特征类型分级，不盲目全实时。

## 13. 常见误区与反直觉

- ❌ **"离线 AUC 涨了就是好特征"** → 涨得越夸张越可能是穿越；先跑穿越检测再庆祝。
- ❌ **"实时特征不会穿越，所以不用管时间"** → 实时确实少穿越，但有迟到丢数、状态丢失，会"算不全"，照样坏。
- ❌ **"回填就是把新特征 SQL 跑一遍历史"** → 漏了"以每条样本各自 t0 为右边界"就是大规模穿越。
- ❌ **"特征越新鲜越好，全上实时"** → 长期画像无需实时，盲目全实时是成本灾难。
- ❌ **"做了 point-in-time 就万无一失"** → PIT 只解决时点，不解决离在线口径（§8.4）。
- ❌ **"穿越和离在线不一致是一回事"** → 是两个问题、两道防线，本文治穿越，一致性篇治不一致（§6.3）。
- 🤯 **反直觉**：一个特征"在离线和在线都偷看了未来、错得一模一样"（一致但穿越），比"算对但离在线不一致"更隐蔽——因为它通过了离在线 diff 校验，却整体把模型带歪。**所以穿越检测不能只靠离在线比对，必须有独立的时间边界校验（§6.4）。**

---

## 14. 平台侧落地 Checklist

把全篇翻译成"你的平台要交付什么能力"：

- [ ] **离线特征生产框架**：声明式配特征，强制带 `event_time`/版本，禁止"全天一个值"进样本。
- [ ] **实时特征生产框架**：封装 keyBy/窗口/状态/sink；watermark 与迟到策略可配、可观测。
- [ ] **事件时间对齐能力**：离线与实时统一事件时间语义，支持按 t0 对齐。
- [ ] **安全回填工具**：声明式回填，框架级强制 point-in-time（右边界恒为各样本 t0），带成本预估与配额。
- [ ] **Point-in-Time / as-of join 能力**：特征带 `version_time`，支持向前最近匹配。
- [ ] **穿越检测**：时间边界静态检查 + 单特征 AUC 异常 + shuffle 检验，回填/上线前必过。
- [ ] **特征监控看板**：分布漂移 PSI / 缺失率 / 覆盖率 / 离在线 diff，分层告警 + 下钻。
- [ ] **新鲜度监控**：端到端 `t_read - t_event` 看板，按特征类型分级 SLA。
- [ ] **新特征上线门禁**：穿越检测 + 一致性比对双通过，才允许进训练样本。

> 📌 性价比最高、算法同学最买账的三件：**安全回填工具** + **穿越检测** + **漂移看板**。

## 16. 向上沟通

> 🎓 老板不关心 as-of join 怎么写，关心**风险、钱、收益**。三句话模板：

1. **用事故讲风险**：*"特征穿越会让模型离线看着好、上线掉点，且平均要好几天才定位，每次相当于一次线上事故 + 一次算法信任损耗。"*
2. **用 ROI 讲价值**：*"建回填工具 + 穿越检测，能在上线前拦住穿越特征，把'上线才发现暴跌'变成'上线前就拦下'，每避免一次 CTR 掉点 ≈ XX 收入。"*
3. **用取舍讲成本**：*"回填是一次性大算力，我们用采样 + 错峰 + 增量物化控制成本；新鲜度按特征分级，不盲目全实时，省下的算力可观。"*

> 💡 一句话向上总结：**"特征生产链路决定模型迭代的速度，穿越治理决定迭代的可信度——链路慢，算法等；穿越漏，算法白跑还摔跤。"**

## 17. 一图总结 + 一句话速记

```
                        特征生产链路与穿越治理
                                │
     ┌──────────────┬───────────┼────────────┬──────────────┐
   两条链路        回填          穿越           PIT           监控
     │              │            │             │              │
  离线(批/T+1)   补历史样本    ①窗口越界      取t0那一刻      漂移PSI
  实时(流/秒级)   易穿越!       ②label衍生     而非最新值     缺失/覆盖率
  双跑→口径飘     大算力        ③滑窗右错位    带version_time  离在线diff
     │           PIT回填        →AUC虚高崩     as-of join     新鲜度
     └──────── 离线管全准可溯，实时管快新鲜，两边对齐靠时间 ────────┘
```

> 📌 **一句话速记**：*特征是带时间戳的状态量——离线批量算、实时流式算，回填和拼样本时只准取"预测那一刻 t0"的值（point-in-time），取到了未来就是穿越，穿越会让离线 AUC 虚高、上线暴跌。*

## 18. 自测三问

> 检验你是否真懂（答得上来，评审现场就稳了）：

1. **为什么"特征回填"是穿越的重灾区？平台怎么从框架层面杜绝？**
   <details><summary>参考答案</summary>回填是"站在今天去重算过去某一刻的特征"，手上握着全部未来数据，极易把 t0 之后的行为算进去（漏上界、用全天口径、闭区间含 t0）。平台要把"以每条样本各自 t0 为右边界的 point-in-time 回填"做成框架级强约束，不让算法同学手写时间条件，并在回填后强制跑穿越检测才允许进样本。</details>
2. **point-in-time join 解决了穿越，为什么还可能离在线不一致？**
   <details><summary>参考答案</summary>PIT 只保证"时点正确"（取 t0 之前的值，不穿越），但特征值仍是"离线这套算子"算出来的；如果离线和在线的计算口径（窗口、去重、默认值）不同，值照样对不上。穿越和不一致是两个问题、两道防线——PIT 治时点，统一定义/快照治口径一致。</details>
3. **算法同学说新特征让离线 AUC 涨了 0.03，你第一反应该做什么、为什么？**
   <details><summary>参考答案</summary>先怀疑穿越，跑穿越检测（时间边界检查 + 单特征 AUC + shuffle 检验），确认不是 label 泄漏或窗口越界再上线。因为搜广推真实涨 0.001 都难得，跳涨 0.03 "好得不真实"，是穿越的红色警报；先拦住比上线暴跌再回滚划算得多。</details>
