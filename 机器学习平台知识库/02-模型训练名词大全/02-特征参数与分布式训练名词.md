## 0. 30 秒速通(TL;DR)

如果你只有 30 秒,记住这张「认知锚点」:

| 你听到的词 | 它到底是什么 | 对平台研发意味着什么 |
|---|---|---|
| **sparse 特征** | 用户 ID、商品 ID、广告 ID 这类离散 ID,取值空间上亿 | 你要支撑「超大稀疏参数存储」(GB~TB 级),这是平台最重的活 |
| **dense 特征** | 年龄、价格、点击率这类连续数值,几十~几千维 | 量小,直接进 MLP,几乎不构成存储压力 |
| **embedding** | 把一个稀疏 ID 映射成低维稠密向量(如 16/32 维) | 模型 95%+ 的参数都在这,是显存/内存/网络的主要矛盾 |
| **embedding table** | 所有 ID 的 embedding 向量的大表 | 这就是那个动辄上百 GB 甚至 TB 的「大家伙」,决定你要不要上 PS |
| **feasign / 特征签名** | ID 被 hash 成 64bit 整数当 key | 你要管 hash 空间、冲突率、特征准入与淘汰(防止表无限膨胀) |
| **PS(参数服务器)** | Server 存参数、Worker 算梯度的架构 | 稀疏参数分片放 Server,你要支撑 Server 的扩缩容、容错、网络带宽 |
| **AllReduce / Ring-AllReduce** | 多卡之间同步稠密梯度的集合通信 | 稠密为主的训练用它,你要保障 NCCL/RDMA 网络、GPU 拓扑 |
| **Horovod / NCCL** | AllReduce 的工程实现(框架层 / 通信库) | 平台要预装、调优、对齐版本,排查「卡了/不收敛」常落到这层 |
| **同步 vs 异步(BSP/ASP/SSP)** | 梯度是不是要等所有人对齐再更新 | 影响吞吐和效果,异步快但有 staleness,你要能解释「为什么慢/为什么飘」 |
| **数据并行 vs 模型并行** | 切数据还是切模型 | embedding 太大塞不下单机 → 模型并行 + sharding,是你要扛的架构题 |

**一句话总纲**:搜广推模型 = 一堆 sparse ID(占参数 99%,走 PS / embedding sharding)+ 少量 dense 数值(走 MLP,用 AllReduce 同步)。平台研发的核心战场,就是**怎么存得下那张超大 embedding table,怎么在几百台机器间高效同步参数**。

## 1. sparse vs dense 特征:模型的两种「燃料」

### 1.1 直觉先行

想象你给模型喂两种「燃料」:

- **dense(稠密)特征**:像一杯水,密密实实,每一维都有意义。例如用户年龄=28、商品价格=99.9、最近 7 天点击率=0.03。它本身就是数值,几乎可以直接进网络。
- **sparse(稀疏)特征**:像一个巨大的衣帽间,里面有一亿个挂钩(每个挂钩=一个 ID),但你这次只用其中几十个。例如「用户 ID=83746521」「点击过的商品 ID=[101, 9982, 30021]」。它的取值空间极大,但每条样本只点亮极少数几个。

```
                      一条样本(一次曝光)
   ┌──────────────────────────────────────────────────────────┐
   │  DENSE(稠密)                  SPARSE(稀疏)              │
   │  ──────────────                ──────────────             │
   │  age      = 28                 user_id  = 83746521        │
   │  price    = 99.9               item_id  = 9982            │
   │  ctr_7d   = 0.03               city_id  = 310             │
   │  ...(几十~几千维)            click_seq= [101,9982,...]  │
   │                                ...(几十~几百个 ID)      │
   └──────────────────────────────────────────────────────────┘
         │                                  │
         │ 直接(归一化后)进网络            │ 先查 embedding table 变成向量
         ▼                                  ▼
   ┌───────────┐                    ┌──────────────────────┐
   │  dense 向量│                    │ 每个 ID → 16/32 维向量 │
   │ [28,99.9,..]│                   │ 拼接 / pooling 成大向量 │
   └─────┬─────┘                    └──────────┬───────────┘
         │                                      │
         └──────────────┬───────────────────────┘
                        ▼
                ┌───────────────┐
                │   拼接 concat   │
                └───────┬───────┘
                        ▼
                ┌───────────────┐
                │   MLP / DNN    │ ← 这里的权重才是「dense 参数」
                └───────────────┘
```

### 1.2 关键差异(量级是重点)

| 维度 | dense 特征 | sparse 特征 |
|---|---|---|
| 取值形态 | 连续数值(浮点) | 离散 ID(整数/字符串) |
| 维度规模 | 几十 ~ 几千维(典型量级) | **取值空间可上亿甚至百亿**(典型量级,如全站用户 ID) |
| 每条样本激活量 | 几乎全维都有值 | 极稀疏,只点亮几十~几百个 ID |
| 进模型方式 | 归一化后直接进 MLP | **先查 embedding table** 转成稠密向量再进 MLP |
| 参数贡献 | MLP 权重,占比小(MB~GB) | embedding table,**占模型参数 95%+**(GB~TB) |
| 对平台的压力 | 显存/算力(GPU 算 MLP) | **存储 + 网络**(海量 KV 的 push/pull) |

### 1.3 脱敏案例

某中型推荐场景(脱敏,数量级示意):
- dense 特征约 **300 维**,全部权重 + MLP 加起来约 **几十 MB**。
- sparse 特征约 **80 个 slot**(user/item/各种交叉),总 ID 空间约 **几十亿**,embedding 维度 16,embedding table 约 **几百 GB**。
- 结论:99.9% 的参数在 sparse 侧。平台谈「模型大小」「训练扛不扛得住」,基本只在谈 sparse。

> 🎓 **老师视角**:很多平台新人一上来盯着 GPU 显存,觉得「模型大=显存不够」。错。搜广推的痛点不在「算」,在「存稀疏 ID 的 embedding」。dense 的 MLP 小到可以忽略,真正的大象是那张 embedding table。**你判断一个搜广推训练任务重不重,第一句话该问的是「稀疏维度多大、embedding 维度几维」,而不是「几张卡」。**

> 🎓 **老师视角**:为什么 sparse 不能像 dense 一样直接进网络?因为 ID 本身是「名字」不是「数值」——user_id=83746521 不代表它比 83746520「大一点点」。直接当数值喂进去,网络会学到完全错误的大小关系。embedding 的本质就是「给每个名字学一个有意义的坐标」,这是后面第 2 节的核心。

### 1.4 还有一类容易被忽略的:变长 sparse(序列特征)

sparse 不只是「单个 ID」,还有一类是**变长 ID 列表**,这是搜广推的精华所在:

- **行为序列**:用户最近点击过的 100 个商品 ID `[101, 9982, 30021, ...]`,长度不固定(有人点 3 个,有人点几百个)。
- **多值离散**:一个商品挂的多个标签 `["手机", "数码", "5G"]`。

```
   定长 sparse(单值)          变长 sparse(多值 / 序列)
   ──────────────────          ────────────────────────────
   user_id = 83746521          click_seq = [101, 9982, 30021, ...](长度可变)
       │                            │  │  │
       ▼                            ▼  ▼  ▼  每个 ID 各查一个 embedding
   查 1 个 embedding            查 N 个 embedding,再 pooling(sum/mean/attention)
                                    │
                                    ▼
                            压成 1 个定长向量进 MLP
```

| 形态 | 例子 | 处理方式 | 对平台意味着 |
|---|---|---|---|
| 单值 sparse | user_id, city_id | 查 1 个 embedding | 简单,batch 内 ID 数可预估 |
| 多值/序列 sparse | click_seq, tag_list | 查 N 个再 pooling | **batch 的 ID 数波动大**,pull 流量不稳定,要按峰值预留带宽 |

> 🎓 **老师视角**:序列特征是平台容量规划的隐藏炸弹。一个普通用户行为序列 50 个 ID,但有「超级活跃用户」序列上千个 ID。如果你按平均值规划 PS 的 pull 带宽,遇到一批长序列样本就会瞬间打满。所以**容量要按 p99 而不是均值规划**,这是序列特征教给平台研发的一课。

## 2. embedding / embedding table / embedding 维度:参数大头从哪来

### 2.1 embedding 是什么:把「名字」翻译成「坐标」

embedding(嵌入)= 把一个高维稀疏的离散 ID,映射成一个低维稠密的实数向量。

```
   稀疏 ID(one-hot 视角,逻辑上是亿维)         稠密 embedding(实际存储,16 维)
   ┌────────────────────────────────┐          ┌──────────────────────────┐
   item_id = 9982                              查表(lookup)
   [0,0,0,...,1,...,0,0,0]  ← 第 9982 位是 1  ────────────►  [0.12, -0.88, 0.34, ... , 0.05]
    └──────  1 亿维  ──────┘                                  └──── 16 维(典型)────┘
```

- 逻辑上,一个 ID 是「亿维 one-hot 向量」;物理上,我们不会真的存亿维,而是**用 ID 去一张表里查出对应的那一行 16 维向量**。
- 这一行向量是**可学习参数**:训练过程中,反向传播会更新它,让「行为相似的 ID」在向量空间里靠得近(比如两个都被同一批人点击的商品)。

### 2.2 embedding table:那张「大家伙」

embedding table = 所有 ID 的 embedding 向量堆在一起的大矩阵。

```
   embedding table(逻辑形态)
   ┌──────────────┬───────────────────────────────────┐
   │   key(ID)    │   value(16 维 float 向量)          │
   ├──────────────┼───────────────────────────────────┤
   │ 0x3F2A...01  │ [0.12, -0.88, ... , 0.05]           │
   │ 0x91BC...77  │ [-0.34, 0.20, ... , 0.91]           │
   │   ...        │   ...                                │
   │ (几十亿行)    │                                     │
   └──────────────┴───────────────────────────────────┘

   单行大小 = embedding 维度 × 每个数的字节
            = 16 × 4 bytes(fp32) = 64 bytes / 行
   表大小   ≈ 行数 × 单行大小(+ optimizer 状态)
```

**容量心算公式(平台研发必背)**:

```
embedding table 显存/内存 ≈ 有效 ID 数 × embedding 维度 × 字节数 × (1 + optimizer 倍数)
```

- 字节数:fp32=4,fp16/bf16=2。
- optimizer 倍数:用 Adam/AdaGrad 时,除了参数本身还要存动量/二阶矩,**实际占用通常是参数本身的 2~3 倍**(典型量级)。这是平台最容易踩的坑——你按参数大小估了 100GB,结果实际跑起来 250GB。

**脱敏量级示例**:

| 有效 ID 数(典型量级) | 维度 | fp32 参数 | 含 Adam(×3)实际占用 |
|---|---|---|---|
| 1 亿 | 16 | ~6.4 GB | ~19 GB |
| 10 亿 | 16 | ~64 GB | ~190 GB |
| 50 亿 | 32 | ~640 GB | ~1.9 TB |

### 2.3 embedding 维度:不是越大越好

embedding 维度(dim)= 每个 ID 用几维向量表示。常见 **8 / 16 / 32 / 64**(典型量级)。

| 维度选择 | 效果影响 | 资源影响 |
|---|---|---|
| 维度小(如 8) | 表达能力弱,可能欠拟合 | 表小,省内存/带宽 |
| 维度大(如 64) | 表达能力强,但易过拟合且边际收益递减 | 表线性变大,push/pull 流量翻倍 |
| 混合维度(不同 slot 不同 dim) | 高频重要特征给大维,长尾给小维 | 复杂但省资源,业界常用优化 |

> 🎓 **老师视角**:算法同学说「我把 embedding 维度从 16 调到 32」,你脑子里要立刻闪过一句话:**「存储和网络流量直接翻倍」**。这不是一个免费操作。维度 ×2,embedding table ×2,每步 push/pull 的数据量 ×2,PS 网卡压力 ×2。所以这种调参不是算法一个人的事,是要跟你确认资源够不够的事。

> 🎓 **老师视角**:为什么 embedding 是「参数大头」而不是 MLP?因为 MLP 的参数量 = 层与层之间的连接数,几百×几百,撑死几百万个参数。而 embedding table 的参数量 = ID 数 × 维度,几十亿 × 16 = 几百亿个参数。**两者差三四个数量级。** 记住:谈搜广推的「大模型」,大在 embedding,不在网络深度。(这跟 NLP 的大语言模型恰好相反——那边大在 dense 的 Transformer 权重。)

### 2.4 embedding 怎么被「用」:lookup → pooling → 进网络

平台研发要理解 embedding 在一个 step 里的完整生命周期,这样才知道每个环节卡在哪、压力在哪:

```
   ① lookup(查表)        ② pooling(聚合)        ③ concat(拼接)       ④ forward(前向)
   ─────────────          ──────────────          ────────────         ───────────
   batch 内所有 ID         多值/序列特征的            把所有 slot 的         进 MLP 算预测
   去 table 查向量         N 个向量 sum/mean         向量首尾拼成
   (PS 的 pull 在此)      压成 1 个                 一个大向量
        │                      │                       │                    │
   网络/IO 压力           CPU/GPU 算力              内存带宽            GPU 算力
```

- **lookup 是稀疏侧最重的 IO 操作**:一个 batch 可能要查几十万个不同 ID,对应几十万次 KV 查询(本地或远程 PS)。
- **pooling** 把变长压成定长——这是序列特征能进固定结构 MLP 的关键。
- 反向时,梯度只回传给「这个 batch 实际查过的那些 ID」,其余几十亿行纹丝不动。**这就是稀疏更新「稀疏」二字的来源**,也是为什么稀疏天然适合异步(第 6 节会用到)。

### 2.5 省 embedding 内存的常见手段(平台要知道有哪些旋钮)

| 手段 | 思路 | 省多少(典型量级) | 代价 |
|---|---|---|---|
| **混合精度(fp16/bf16)** | 用 2 字节存代替 4 字节 | embedding 内存减半 | 数值稳定性需关注 |
| **混合维度** | 高频 slot 大维、长尾小维 | 视分布,可省 20%~50% | 配置复杂 |
| **量化(int8 等)** | 训练后/训练中量化向量 | 再减半甚至更多 | 精度损失需评估 |
| **准入淘汰**(见第 3 节) | 不给低频/过期 ID 分配空间 | 视长尾占比,可省一大截 | 可能丢长尾召回 |
| **分层存储** | 热的放显存/内存,冷的放 SSD | 突破单机内存上限 | 冷数据访问慢 |

> 🎓 **老师视角**:这张表是你跟算法/老板谈「内存不够怎么办」时的弹药库。顺序通常是:先准入淘汰(免费且该做)→ 混合精度(简单收益大)→ 混合维度/量化(要评估掉点)→ 分层存储(终极手段,改架构)。**别一上来就喊「加机器」——很多时候是这些旋钮没拧。**

## 3. feature signature / 哈希 / 碰撞 / 准入淘汰:ID 是怎么变成 sign 的

### 3.1 为什么要做特征签名(feasign)

原始 ID 五花八门:user_id 是数字,item_category 是字符串「电子产品/手机」,交叉特征是「user_id × item_id」拼出来的超长串。模型/PS 不想处理这些乱七八糟的格式,它只想要**一个统一的 64bit 整数当 key**。

于是有了 **feature signature(特征签名,简称 feasign)**:把任意原始特征经过 hash,变成一个 64bit 无符号整数。

```
   原始特征(各种格式)                    hash 编码                    feasign(64bit key)
   ┌────────────────────────┐                                      ┌──────────────────┐
   │ "user_id=83746521"      │                                     │ 0x9F3A...2C81     │
   │ "item_cate=手机"        │  ──── slot << 高位 | hash(value) ──► │ 0x21BC...0077     │
   │ "u×i=83746521_9982"     │                                     │ 0x77DE...A130     │
   └────────────────────────┘                                      └──────────────────┘
                                                                            │
                                                                            ▼
                                                              拿这个 64bit 当 key 去 PS / table 查 embedding
```

- 常见做法是把 **slot(特征域编号)放高位 + value 的 hash 放低位**,这样不同 slot 即使 value 相同也不会撞在一起(比如 city_id=101 和 item_id=101 不会共用一个 embedding)。
- 这个 64bit key 就是后面 PS 里的 key,也是 embedding table 的 key。**整个稀疏侧都围着 feasign 转。**

### 3.2 哈希碰撞(hash collision)与碰撞率

把任意 ID 压进 64bit,理论上不同 ID 可能 hash 到同一个值——这叫**哈希碰撞**。碰撞了就意味着两个本来不同的 ID 共用一个 embedding,会引入噪声、损害效果。

| 方案 | 思路 | 碰撞率 | 代价 |
|---|---|---|---|
| 全量保留 ID(无 hash 压缩) | 每个 ID 单独一行 | 0 | 表无限膨胀,存储爆炸 |
| 64bit hash(主流) | 压到 64bit 空间 | 极低(64bit 空间巨大,几十亿 ID 碰撞概率近乎可忽略,典型量级) | 几乎无损,工程主流 |
| 固定大小 hash bucket(如压到 2^25 桶) | 强行限制表大小 | **明显升高**,需评估 | 省内存,但效果可能掉点 |

> 🎓 **老师视角**:这里有个反直觉点。如果 hash 到 64bit 全空间,碰撞率低到可以忽略;但有些团队为了控表大小,会做「hash 到固定桶数」(feature hashing trick),把几十亿 ID 硬压到几千万桶。这时候碰撞率不是 0 了,是要权衡的——**省了内存,可能掉了 AUC**。当算法同学跟你说「我们用 hash bucket 控制了表大小」,你要敏感:这是用效果换资源的决策,值不值得要数据说话。

**两种「hash」别混为一谈**(平台研发最容易绕进去的坑):

| 名称 | 目的 | 是否有损 | 典型场景 |
|---|---|---|---|
| **编码 hash(到 64bit)** | 把各种格式的 ID 统一成 64bit 整数 key | 几乎无损(空间巨大) | 所有稀疏特征,标配 |
| **压缩 hash / hashing trick(到固定桶)** | 强行限制 embedding 行数 | **有损**(碰撞率高) | 内存吃紧、长尾极多时的取舍 |

> 🎓 **老师视角**:在线/离线一致性在这里也藏着一个雷——**hash 函数和 slot 编号必须线上线下完全一致**。如果离线训练用了 hashA、在线 serving 用了 hashB,同一个 ID 会算出不同的 feasign,查到错误的 embedding,模型预测直接崩。平台研发要把「hash 实现 + slot 配置」当成**线上线下共享的契约**来管理,改一处必须两边同步。这类「特征穿越/不一致」事故,排查起来极其痛苦,提前规范远胜事后救火。

### 3.3 特征准入(admit)与淘汰(eviction / expire):防止表无限膨胀

每天都有新 ID 进来(新用户、新商品),表会无限长大。必须有机制控制:

- **特征准入(feature admit)**:新 ID 第一次出现时,不一定立刻给它分配 embedding。常见策略:**出现次数达到阈值 N 次才准入**(如出现 ≥3 次才建 embedding)。理由:只出现一两次的 ID(可能是噪声/爬虫)给了 embedding 也学不好,纯浪费空间。
- **特征淘汰(feature eviction / expire)**:长期没被访问的 ID,踢出表。常见策略:**按时间过期**(N 天没出现就删)或 **按 LRU / 访问频次** 淘汰。

```
   新 ID 出现 ──► 计数器 +1 ──► 是否 ≥ 准入阈值?
                                 │否            │是
                                 ▼              ▼
                            暂不建 embedding   建 embedding,进表
                                                  │
                                  ┌───────────────┘
                                  ▼
                         周期性扫描:多久没访问了?
                                  │
                       ≥ 过期阈值 ─┴─► 淘汰(释放空间)
```

| 机制 | 触发条件(典型量级) | 平台要支撑的能力 |
|---|---|---|
| 准入 admit | 累计出现 ≥ 3~10 次 | 维护每个 feasign 的访问计数 |
| 时间淘汰 expire | 90 天未出现(可配) | 记录 last-access 时间戳,周期扫描清理 |
| 频次/LRU 淘汰 | 内存达水位时淘汰最冷的 | 维护访问统计,支持按水位触发 |

> 🎓 **老师视角**:准入和淘汰是平台研发**直接负责**的能力,不是算法的玩具。没有它们,embedding table 会随时间无限膨胀,直到把 PS 内存撑爆。所以当你设计稀疏存储时,**「准入阈值、过期天数」必须是可配置的运营旋钮**,而且要能监控「表的行数随时间的曲线」。一条平稳的曲线说明准入淘汰在工作;一条只涨不降的曲线说明你的表迟早炸。

> 🎓 **老师视角**:准入淘汰还有个隐藏价值——**它让在线/离线一致性变可控**。在线 serving 的模型只能装得下有限的 ID;如果离线训练把所有长尾 ID 都留着,导出模型时还得再裁一遍。在训练侧就做好准入淘汰,导出的模型天然就是「精简版」,在线加载更顺。

## 4. 参数服务器 PS 架构:Worker / Server 怎么分工

### 4.1 为什么搜广推天然适配 PS

回到第 2 节的结论:embedding table 动辄几百 GB ~ TB,**单机内存/显存根本装不下**。怎么办?把这张大表「切片」分散到多台机器上存——这就是 **参数服务器(Parameter Server,PS)** 的核心动机。

```
                         ┌─────────────────────────────────────────┐
                         │         Server 节点(存参数)             │
                         │  ┌────────┐ ┌────────┐ ┌────────┐        │
                         │  │Server 0│ │Server 1│ │Server N│        │
                         │  │分片 A   │ │分片 B   │ │分片 ... │        │ ← 超大稀疏 embedding
                         │  └───▲────┘ └───▲────┘ └───▲────┘        │   按 key 哈希分片存这里
                         └──────┼──────────┼──────────┼─────────────┘
                pull(拉参数)   │          │          │   push(推梯度)
                         ┌──────┴──────────┴──────────┴─────────────┐
                         │         Worker 节点(算梯度)             │
                         │  ┌────────┐ ┌────────┐ ┌────────┐        │
                         │  │Worker 0│ │Worker 1│ │Worker M│        │ ← 读样本、前向反向、
                         │  │持 dense │ │持 dense │ │持 dense │        │   稠密参数各自持有
                         │  └────────┘ └────────┘ └────────┘        │
                         └───────────────────────────────────────────┘
                              ▲              ▲             ▲
                              │              │             │
                           读训练样本(分片喂给不同 Worker)
```

### 4.2 PS 的工作流(一个 step 的生命周期)

```
   1. Worker 读一批样本,知道这批需要哪些 feasign(ID)
                      │
   2. pull:Worker 向各 Server 拉取这批 ID 的 embedding(只拉用到的,不拉全表!)
                      │
   3. 前向 forward:embedding + dense 一起算出预测值
                      │
   4. 算 loss、反向 backward:得到每个参数的梯度
                      │
   5. push:Worker 把稀疏梯度推回对应 Server,Server 用 optimizer 更新参数
                      │
   6. 稠密参数(MLP)在 Worker 间另行同步(见第 5 节 AllReduce)
```

**核心设计精髓**:稀疏侧只 pull / push「这一批用到的几十万个 ID」,而不是整张几十亿行的表。这就是 PS 能扛海量稀疏的关键——**通信量正比于 batch 用到的 ID 数,而不是表的总大小**。

### 4.3 Server 存什么、Worker 存什么

| 角色 | 存什么 | 主要瓶颈 | 平台要支撑 |
|---|---|---|---|
| **Server** | 超大**稀疏 embedding**(按 key 分片) | 内存容量 + 网络带宽(被 N 个 Worker 同时 pull/push) | Server 扩缩容、分片均衡、容错(挂了能恢复)、高带宽网络 |
| **Worker** | **稠密参数**(MLP,各自一份)+ 计算 | CPU/GPU 算力 + 读样本 IO | Worker 弹性伸缩、样本数据就近读取、failover |

| 对比项 | PS 架构 | AllReduce 架构(见第 5 节) |
|---|---|---|
| 擅长 | 超大稀疏参数 | 中等规模稠密参数 |
| 参数存放 | 集中在 Server,分片 | 每个 Worker 各持一份完整副本 |
| 通信模式 | Worker↔Server 的 pull/push(星型) | Worker↔Worker 的集合通信(环/树型) |
| 典型场景 | 搜广推(亿级稀疏 ID) | CV/NLP(稠密大模型) |
| 扩展瓶颈 | Server 网卡带宽、热点 key | 网络拓扑、最慢节点 |

> 🎓 **老师视角**:面试和对接里最容易混的就是「PS vs AllReduce 二选一」。错。**搜广推的现代训练通常是「PS + AllReduce 混合」**:稀疏 embedding 走 PS(因为太大,只能分片集中存),稠密 MLP 走 AllReduce(因为小,每个 Worker 存一份各自同步更快)。所以你听到一个系统「既有 Server 又用 NCCL」,不矛盾,这是标配。

> 🎓 **老师视角**:PS 最大的运维痛点是**热点 key**。某些超高频 ID(比如爆款商品、超级大 V 用户)的 embedding 会被几乎每个 batch 都 pull,导致存它的那个 Server 网卡被打爆,而其他 Server 闲着。这叫「数据倾斜/热点」。平台侧的解法有:热点 key 多副本、按访问频次重新分片。当算法说「训练突然变慢了」,你排查清单里一定要有「是不是某个 Server 热点了」。

> 🎓 **老师视角**:容错对 PS 是生死问题。一次大模型训练可能跑几天,几百台机器里挂一台是常态。如果 Server 挂了导致一部分参数丢失且无法恢复,几天的训练白费。所以**「Server 定期 checkpoint + 故障自动拉起 + 从 checkpoint 恢复」是平台必须提供的能力**,不是可选项。

### 4.4 PS 的弹性与流式训练(进阶,但平台必懂)

搜广推的 PS 和传统离线训练有两个工业级特性,平台研发尤其要懂:

- **流式 / 增量训练(streaming / online learning)**:模型不是「训完就停」,而是**持续不断地喂新样本、实时更新参数、定期导出上线**。这意味着 PS 是一个长期在线的服务,而不是一次性作业。表会随时间增长(新 ID 不断进),所以准入淘汰不是可选优化,是**让流式训练能长期跑下去的前提**。
- **弹性扩缩容(elastic scaling)**:在线请求有高峰低谷,训练资源也想动态调整。先进的 PS 支持在不停训练的情况下增减 Server/Worker,参数自动重新分片。

```
   传统离线训练               流式 / 在线训练
   ───────────────           ──────────────────────────────
   样本集固定 → 训 N 轮 → 停   样本源源不断流入(Kafka 等)
                                 │
                                 ▼
                  Worker 持续消费 → push 梯度 → Server 实时更新
                                 │
                       定期(如每小时)导出模型快照上线
                                 │
                       后台持续做准入(新 ID 进)+ 淘汰(老 ID 出)
```

| 对比项 | 离线批训练 | 流式/在线训练 |
|---|---|---|
| 样本 | 固定数据集 | 持续流入 |
| 模型更新 | 训完导出 | 边训边导出(分钟~小时级) |
| 表大小 | 相对稳定 | 持续变化,强依赖淘汰 |
| 平台压力 | 一次性算力 | **长期在线服务化**(SLA、监控、容错全要) |

> 🎓 **老师视角**:很多平台新人以为「训练 = 跑个批作业」。在头部搜广推场景,训练越来越像**一个 7×24 的在线服务**——模型几小时甚至几分钟就更新一版上线,追实时性。这彻底改变了平台研发的视角:你维护的不是「作业」,是「服务」,要有 SLA、要有监控告警、要能热升级、挂了要秒级恢复。这是搜广推平台最硬核也最有价值的部分。

### 4.5 push / pull 到底传什么、传多少

平台研发对 PS 网络的把控,落到 push/pull 这两个动作的数据量上:

```
   pull(Worker ← Server):拉这一批用到的 ID 的 embedding 向量
     流量 ≈ batch 内去重后的 ID 数 × embedding 维度 × 字节
   push(Worker → Server):把这一批 ID 的稀疏梯度推回去
     流量 ≈ 同上量级(梯度和向量同形状)
```

- 关键词是「**去重后的 ID 数**」:一个 batch 里很多样本会共用热门 ID(如爆款商品),去重后实际 pull 的量比想象中小——但热门 ID 也正是热点 Server 的来源。
- 维度、序列长度、batch 大小**任何一个翻倍,push/pull 流量都成比例上涨**。这就是为什么前面反复强调「提维度/加序列不是免费的」。

| 影响因子 | 对 pull/push 流量的影响 | 平台动作 |
|---|---|---|
| embedding 维度 | 线性 | 提维前评估带宽 |
| 序列长度 | 线性(且 p99 远超均值) | 按 p99 规划,必要时截断 |
| batch size | 线性 | 与 Worker 数联动看全局 batch |
| 去重率 | 反向(热门多则去重高) | 监控热点 key |

> 🎓 **老师视角**:当你做 PS 网络容量规划时,**别用「ID 总数」估流量,要用「单 batch 去重后 ID 数 × 维度 × 字节 × steps/秒」**。前者会把你吓死(几十亿),后者才是真实的每秒网络压力(可能几十万 ID 级)。算清这笔账,你才知道 25G 网卡够不够、要不要上 100G/RDMA。

## 5. AllReduce / Ring-AllReduce / Horovod / NCCL:稠密参数怎么同步

### 5.1 AllReduce 解决什么问题

数据并行下,每个 Worker 拿不同的样本算出**不同的梯度**。要让所有 Worker 的模型保持一致,就得把这些梯度「汇总求和(或求平均)再分发回去」。这个「所有节点的数据规约后,结果广播给所有节点」的操作,就叫 **AllReduce**。

```
   规约前:每个 Worker 有自己的梯度
   Worker0: g0    Worker1: g1    Worker2: g2    Worker3: g3
       └──────────────┴───── AllReduce(求和/平均)─────┴──────────┘
                              │
   规约后:每个 Worker 都拿到 (g0+g1+g2+g3)
   Worker0: G     Worker1: G     Worker2: G     Worker3: G   ← 完全一致,各自更新
```

### 5.2 Ring-AllReduce:为什么是「环」

最朴素的做法是「所有人把梯度发给一个中心节点求和再发回」——中心节点网卡瞬间爆炸(带宽不随节点数扩展)。**Ring-AllReduce(环形)** 把节点连成一个环,每个节点只跟左右邻居通信,分块流水线传递。

```
       ┌──────► Worker0 ──────┐
       │                       ▼
   Worker3                  Worker1     每个节点只跟下一个邻居发送
       ▲                       │        梯度切成块,边传边累加(流水线)
       └────── Worker2 ◄───────┘
```

**关键性质**:Ring-AllReduce 的通信量**几乎与节点数无关**(带宽最优),这就是它能扩展到几百上千卡的原因。代价是延迟随节点数线性增长(环越大,转一圈越久),所以超大规模会用树形/分层 AllReduce 优化。

### 5.3 名词分层:别把它们摆在同一层

这是最容易混的地方。它们不是并列关系,而是**分属不同抽象层**:

```
   ┌──────────────────────────────────────────────────────────┐
   │ 算法/概念层:  AllReduce / Ring-AllReduce  ← 「要做什么」    │
   ├──────────────────────────────────────────────────────────┤
   │ 框架/编排层:  Horovod(也包括各框架内置的分布式)          │
   │               ← 「谁来组织 Worker 们做 AllReduce」          │
   ├──────────────────────────────────────────────────────────┤
   │ 通信库/底层:  NCCL(GPU 间)/ MPI / Gloo(CPU)           │
   │               ← 「具体用什么把字节从这张卡搬到那张卡」      │
   └──────────────────────────────────────────────────────────┘
```

| 名词 | 它在哪一层 | 一句话 | 对平台研发意味着 |
|---|---|---|---|
| **AllReduce** | 概念/算法 | 多节点梯度规约+广播的语义 | 理解它你才懂稠密同步在干嘛 |
| **Ring-AllReduce** | 算法实现 | AllReduce 的带宽最优拓扑 | 知道为什么能扩展到上千卡 |
| **Horovod** | 框架/编排层 | Uber 开源,统一封装 TF/PyTorch 的 AllReduce 训练 | 平台要预装、对齐版本、提供启动器 |
| **NCCL** | 通信库底层(NVIDIA) | GPU 间集合通信的高性能实现 | 你要装 NCCL、配 RDMA/NVLink、调 GPU 拓扑;「卡死/不收敛」常落到这层 |
| **Gloo / MPI** | 通信库底层 | CPU 场景或无 NCCL 时的后端 | CPU 训练或调试时的备选后端 |

### 5.4 PS vs AllReduce 再对比一次(稠密同步视角)

| 对比项 | PS 同步稠密 | AllReduce 同步稠密 |
|---|---|---|
| 谁存参数 | Server 集中(可能有副本) | 每个 Worker 各持完整一份 |
| 通信形态 | Worker→Server→Worker(星型,易热点) | Worker↔Worker(环/树,均衡) |
| 扩展性 | Server 带宽是瓶颈 | 带宽最优,扩展性更好 |
| 适合 | 参数极大且稀疏 | 参数规模适中且稠密 |

> 🎓 **老师视角**:面试常考「Ring-AllReduce 为什么比 PS 同步稠密快」。答案核心一句话:**PS 是星型(所有 Worker 砸向 Server,Server 网卡成瓶颈,带宽不随规模扩展),Ring 是环型(每个节点只与邻居通信,总带宽需求几乎不随节点数增长)**。但注意这只对稠密成立——稀疏参数太大,每个 Worker 存一份完整副本根本不现实,所以稀疏还得回 PS。这就是为什么混合架构是搜广推的最优解。

> 🎓 **老师视角**:平台研发排查「多卡训练卡住不动」,80% 的概率在 NCCL 这一层:GPU 拓扑没识别对、RDMA 网卡没配好、NCCL 版本和驱动不匹配、被防火墙挡了端口。所以你必须把 `NCCL_DEBUG=INFO` 这类调试手段、网络拓扑检查、版本矩阵当成平台标配知识。这是「框架同学找你」而不是「算法同学找你」的高频问题。

### 5.5 通信优化:gradient bucketing 与 compute/communication overlap

平台研发会听到几个「让 AllReduce 更快」的术语,提前知道意思:

- **gradient bucketing(梯度分桶)**:不是每个参数算完就单独发,而是攒一桶再发,减少通信次数(小包多 → 网络效率低)。
- **compute/communication overlap(计算通信重叠)**:反向传播是从后往前算的,后面层的梯度先算完——那就**一边继续算前面层、一边把后面层的梯度发出去**,把通信时间藏在计算时间里。

```
   不重叠(串行)                        重叠(流水线)
   |==反向计算全部==|==AllReduce==|       |==反向计算==|
                                              |==AllReduce 后面层==|
   总时间 = 计算 + 通信                  |==继续算前面层==|
                                         总时间 ≈ max(计算, 通信)
```

> 🎓 **老师视角**:这两个优化通常框架(Horovod / DDP)默认就做了,但**调参不当会失效**——比如 bucket 太小通信碎、太大重叠不起来。当框架同学说「多卡扩展性不好,卡数翻倍但速度没翻倍」,除了排查网络硬件,也要看通信重叠是否生效。这是「为什么加了卡却不快」的常见软件层答案。

## 6. 同步 vs 异步训练(BSP / ASP / SSP):梯度什么时候对齐

### 6.1 三种范式一张图讲清

```
   BSP(同步,Bulk Synchronous Parallel)
   Worker0 |==算==|--等--|==算==|--等--|     所有人算完 → 同步屏障 → 一起更新
   Worker1 |===算==|-等-|===算==|-等-|        谁慢等谁(木桶效应)
   Worker2 |=算=|----等----|=算=|---等--|     ← 最慢的 Worker 拖累全场(straggler)
                ▲                 ▲
              同步屏障           同步屏障

   ASP(异步,Asynchronous Parallel)
   Worker0 |==算==|更新|==算==|更新|==算==|   各算各的,谁好了就推 Server、立刻更新
   Worker1 |===算==|更新|===算==|更新|        不等任何人 → 最快,但用的可能是旧参数
   Worker2 |=算=|更新|=算=|更新|=算=|更新|     ← 存在 staleness(梯度基于过期参数)

   SSP(有界异步,Stale Synchronous Parallel)
   允许快的 Worker 领先慢的,但不能超过 S 步(staleness 上界 = S)
   领先太多 → 强制等一等 → 在「快」和「准」之间折中
```

### 6.2 staleness(梯度陈旧度)是异步的核心代价

异步训练里,Worker 拉参数、算梯度、推回去之间有时间差。等它的梯度推回 Server 时,Server 上的参数可能已经被别的 Worker 更新过好几轮了——也就是说,**这个梯度是基于「过期的参数」算的**,这叫 **staleness(陈旧度)**。staleness 越高,梯度越「不准」,可能导致收敛慢、震荡甚至不收敛。

### 6.3 三者对比

| 范式 | 全称 | 吞吐 | 效果/收敛 | staleness | 适用 |
|---|---|---|---|---|---|
| **BSP** | Bulk Synchronous | 受最慢节点拖累(低) | 等价单机,最稳 | 0 | 同构集群、追求精度复现 |
| **ASP** | Asynchronous | 最高(无等待) | 可能震荡/掉点 | 不受控 | 异构集群、稀疏更新场景、容忍精度波动 |
| **SSP** | Stale Synchronous | 介于两者 | 折中 | 有界(≤ S) | 想兼顾速度与稳定 |

> 🎓 **老师视角**:有个广为流传的经验——**大规模稀疏推荐训练对异步的容忍度比 CV/NLP 高**。为什么?因为稀疏梯度很「局部」:这个 batch 只更新它用到的几十万个 ID 的 embedding,不同 Worker 大概率在更新**不同的 ID**,冲突天然就少,staleness 的伤害也就小。所以搜广推历史上大量用 ASP 异步 PS 训练,跑得飞快。而稠密的大模型梯度是「全局」的,异步冲突严重,通常坚持同步。这个差异你要能讲出来。

> 🎓 **老师视角**:同步训练最大的运维敌人叫 **straggler(落后者)**——一台机器因为磁盘慢、网络抖、被别的任务挤占而变慢,BSP 下它会拖垮整个集群(所有人等它)。所以平台研发对同步训练要重点监控「每个 Worker 的 step 耗时分布」,发现某台持续偏慢就要能踢掉重调度。**「为什么训练吞吐上不去」很多时候不是算法问题,是你集群里有一台病机器。**

> 🎓 **老师视角**:当算法同学抱怨「换了集群之后模型 AUC 飘了、复现不了」,你要敏感地问一句「是不是从同步切到异步了,或者 Worker 数变了」。异步训练的结果跟 Worker 数、调度时序有关,**天然不可严格复现**。这不是 bug,是异步的固有特性。能讲清这点,你在算法同学眼里就从「运维」升级成「懂行的平台」。

### 6.4 一个常被忽略的点:batch size 会随 Worker 数变

数据并行下,如果每个 Worker 各自的「本地 batch」是 256,那么 8 个 Worker 的**全局有效 batch size 就是 256 × 8 = 2048**。这件事很多人没意识到,但它会实打实影响效果:

```
   本地 batch=256                全局有效 batch
   Worker0 [256]  ┐
   Worker1 [256]  ├─ 梯度平均 → 等价于一个 batch=2048 的更新
   ...            │
   Worker7 [256]  ┘
```

- 全局 batch 变大 → 梯度更平滑、但**学习率通常要跟着调大**(经验法则:线性缩放,batch ×k 则 lr ×k,典型量级,需 warmup)。
- 所以「加机器」不是纯粹的加速——它隐含改变了有效 batch,可能需要算法重新调学习率,否则效果会变。

| Worker 数 | 本地 batch | 全局有效 batch | 学习率(需配合调整) |
|---|---|---|---|
| 1 | 256 | 256 | base lr |
| 8 | 256 | 2048 | ~8× base(经验,需 warmup) |
| 32 | 256 | 8192 | 大 batch 区,可能需更复杂 scheduler |

> 🎓 **老师视角**:这是平台研发和算法之间最容易扯皮的灰色地带。平台说「我帮你把卡从 8 张加到 32 张」,以为是纯加速;结果算法发现效果掉了,回头怪平台。真相是全局 batch 翻了 4 倍、学习率没跟着调。**所以「扩卡」这件事要主动告知算法「全局 batch 会变,记得调 lr」**——这一句话能避免一场无意义的甩锅。

## 7. 数据并行 vs 模型并行:embedding 太大怎么切

### 7.1 两种并行的本质区别

```
   数据并行(Data Parallelism)—— 切「数据」,模型每台一份完整副本
   ┌──────────┐  ┌──────────┐  ┌──────────┐
   │ 模型副本   │  │ 模型副本   │  │ 模型副本   │   每台机器装得下完整模型
   │  (完整)   │  │  (完整)   │  │  (完整)   │   只是喂不同的数据分片
   └────▲─────┘  └────▲─────┘  └────▲─────┘   梯度靠 AllReduce 同步
     数据片1        数据片2        数据片3

   模型并行(Model Parallelism)—— 切「模型」,一份模型拆到多台
   ┌──────────┐  ┌──────────┐  ┌──────────┐
   │ 模型分片 A │→│ 模型分片 B │→│ 模型分片 C │   单台装不下完整模型
   │ (前几层)  │  │ (中间层)  │  │ (后几层)  │   把模型本身切开分散存/算
   └──────────┘  └──────────┘  └──────────┘
        同一批数据在分片间流动(前向/反向跨机器传激活值)
```

### 7.2 何时用哪个

| 判断 | 用数据并行 | 用模型并行 |
|---|---|---|
| 模型单机装得下? | 是 → 优先数据并行(简单、扩展好) | 否 → 必须模型并行 |
| 瓶颈是什么? | 数据多、想加速 | 参数大、单机塞不下 |
| 通信代价 | 同步梯度(AllReduce) | 传激活值(机器间串行依赖,更复杂) |
| 工程复杂度 | 低 | 高 |

**第一选择永远是数据并行**——它简单、扩展性好。只有当模型大到单机装不下,才被迫上模型并行。

### 7.3 embedding 太大:这正是模型并行的经典场景

搜广推里「单机装不下」的那个东西,99% 是 embedding table。所以处理方式是 **embedding sharding(embedding 分片)**——本质就是对 embedding 这部分做模型并行:

```
   一张 1TB 的 embedding table 装不下任何单机
                      │
                      ▼  按 key 哈希分片(本质是模型并行)
   ┌────────────┐ ┌────────────┐ ┌────────────┐
   │ Shard 0     │ │ Shard 1     │ │ Shard 2     │   每片只存一部分 ID 的 embedding
   │ key%3==0    │ │ key%3==1    │ │ key%3==2    │   合起来才是完整的表
   │ (存 Server0)│ │ (存 Server1)│ │ (存 Server2)│
   └────────────┘ └────────────┘ └────────────┘

   而稠密 MLP(小,单机装得下)→ 走数据并行,每个 Worker 一份,AllReduce 同步
```

**所以搜广推的标准架构是「混合并行」**:
- **稀疏 embedding** → 模型并行(分片到多个 Server / PS)
- **稠密 MLP** → 数据并行(每 Worker 一份,AllReduce 同步)

| 切分对象 | 用的并行方式 | 落地形态 | 平台要支撑 |
|---|---|---|---|
| embedding table(大) | 模型并行 | sharding 到多 Server / PS | 分片路由、热点均衡、容错 |
| MLP(小) | 数据并行 | 每 Worker 一副本 | AllReduce / NCCL 网络 |

### 7.4 一次完整容量推演(脱敏,平台研发实战范式)

把前面的公式串起来,演示「拿到一个训练需求,怎么估资源」。这是平台研发最该练熟的硬功夫:

```
   需求(算法给的数字,脱敏):
     · 有效稀疏 ID 数:约 20 亿
     · embedding 维度:32
     · optimizer:Adam(状态约 2×)
     · 期望全局 batch:8192,本地 batch 256
   ───────────────────────────────────────────────
   ① 参数本身 = 20亿 × 32 × 4B(fp32) ≈ 256 GB
   ② 含 optimizer = 256 × (1+2) ≈ 768 GB
   ③ 切混合精度(fp16 存参数)≈ 384 GB(省一半)
   ④ Server 数:若单 Server 可用 ~100GB → 至少 4~8 台(留余量+热点冗余)
   ⑤ Worker 数:全局 8192 / 本地 256 = 32 个 Worker
   ⑥ 网络:序列特征按 p99 估每步 pull/push 流量 → 决定网卡是 25G 还是 100G
   ───────────────────────────────────────────────
   结论给老板/算法:约 4~8 台 Server + 32 个 Worker,
                    需高带宽网络;提维度/加序列长度会改变这个估算。
```

| 推演环节 | 用到的概念 | 来自本文 |
|---|---|---|
| 参数大小 | 容量公式 | 第 2.2 节 |
| optimizer 倍数 | Adam 状态 | 第 2.2 节 |
| 省内存 | 混合精度/淘汰 | 第 2.5、3.3 节 |
| Server 数 | embedding sharding | 第 4、7 节 |
| Worker 数 | 全局/本地 batch | 第 6.4 节 |
| 网络带宽 | pull/push、序列 p99 | 第 1.4、4 节 |

> 🎓 **老师视角**:能现场做出这套推演,你在需求评审会上就是「能拍板要多少资源」的那个人。注意三个最容易漏的乘数:**optimizer 倍数(×2~3)、序列特征的 p99(不是均值)、混合精度的减半**。漏掉前两个会低估炸内存/炸带宽,漏掉第三个会高估浪费预算。把这套当肌肉记忆。



> 🎓 **老师视角**:模型并行有个隐藏代价——**跨机器串行依赖**。数据并行各台独立算,只在最后同步一次;模型并行里分片 B 必须等分片 A 算完才能开始(前向),反向又得倒着来,中间还要传激活值。这意味着模型并行的通信和调度复杂得多,能不用就不用。embedding 之所以适合「分片式模型并行」,恰恰因为它是「查表」操作——每个 ID 独立查,分片之间没有串行依赖,这是个特例。纯 MLP 的层间模型并行才是真正难啃的(那是大语言模型的领域)。

## 8. 全名词串讲:一条样本的完整旅程

把前面所有名词串成一个故事——这是检验你是否真懂的终极测试。跟着一条曝光样本走一遍:

```
   ① 一条曝光样本进来:含 dense(age=28...)和 sparse(user_id, click_seq=[...])
                              │
   ② 每个 sparse 特征 → feasign:slot 编号 + value 的 64bit hash(第 3 节)
      └─ 新 ID?查准入计数,够阈值才有 embedding;否则暂不分配
                              │
   ③ Worker 收集这一批所有 feasign,向 PS 的各 Server pull 对应 embedding(第 4 节)
      └─ 只 pull 这批用到的,不是全表;序列特征会 pull 很多 → 流量按 p99 规划
                              │
   ④ lookup 出的向量做 pooling + concat,和 dense 拼一起,进 MLP forward(第 2 节)
                              │
   ⑤ 算 loss → backward,得到梯度:
      ├─ 稀疏梯度:只回传这批用到的 ID → push 回对应 Server,Server 用 optimizer 更新
      └─ 稠密梯度:Worker 间用 Ring-AllReduce / NCCL 同步(第 5 节)
                              │
   ⑥ 同步策略决定节奏:BSP 等所有人 / ASP 各走各的(有 staleness)/ SSP 折中(第 6 节)
                              │
   ⑦ 后台持续:准入新 ID、淘汰冷 ID,控制 embedding table 大小(第 3 节)
                              │
   ⑧ (流式场景)每隔一段时间导出模型快照上线(第 4.4 节)
```

**把名词按「它属于哪一侧」归位**,你就再也不会混:

| 归属 | 名词 | 一句话角色 |
|---|---|---|
| 特征侧 | sparse / dense / 序列特征 | 模型的两种燃料 |
| 编码侧 | feasign / hash / 碰撞 / 准入 / 淘汰 | 把 ID 变成可管理的 key,并控表 |
| 参数侧 | embedding / table / 维度 | 参数大头,稀疏侧的本体 |
| 架构侧 | PS / Server / Worker / push / pull | 怎么存下大表、怎么传参数 |
| 同步侧 | AllReduce / Ring / Horovod / NCCL | 稠密参数怎么对齐 |
| 节奏侧 | BSP / ASP / SSP / staleness | 梯度什么时候对齐,快还是准 |
| 并行侧 | 数据并行 / 模型并行 / sharding | 切数据还是切模型 |

> 🎓 **老师视角**:如果你能不看文档把上面这条旅程从头讲到尾,你就已经超过了大多数只会背单词的人。**关键不是记住每个名词的定义,而是知道它们在这条数据流上的位置和因果关系**——为什么 feasign 在 pull 之前、为什么稀疏走 push 而稠密走 AllReduce、为什么稀疏能异步而稠密要同步。这种「串起来」的能力,就是你在算法和老板之间「接得住、翻译得了」的底气。

## 权衡取舍

| 决策点 | 选项 A | 选项 B | 怎么权衡 |
|---|---|---|---|
| embedding 维度 | 小维(省资源) | 大维(更准) | 边际收益递减,先小后调;每加倍=存储/带宽加倍 |
| hash 空间 | 64bit 全空间(无损) | 固定 bucket(省内存) | bucket 省内存但碰撞掉点,要用 AUC 数据决策 |
| 准入阈值 | 高(表小、稳) | 低(覆盖长尾、表大) | 高阈值丢长尾召回,低阈值表膨胀,看场景长尾价值 |
| 训练形态 | 离线批训(稳、可复现) | 流式在线训(实时、追时效) | 时效要求高上流式,但要为「训练即服务」付出 SLA/运维成本 |
| 同步策略 | BSP(准、稳、慢) | ASP(快、可能飘) | 稀疏推荐可偏异步;追求复现/稠密大模型偏同步 |
| 并行方式 | 数据并行(简单) | 模型并行(被迫) | 装得下就数据并行,装不下才模型并行 |
| Server 副本 | 单副本(省内存) | 多副本(防热点/容错) | 热点 key 和容错需求决定,通常关键 Server 多副本 |
| 参数精度 | fp32(稳) | fp16/bf16(省一半内存) | 混合精度是省 embedding 内存的常用手段,注意数值稳定性 |

## 常见误区与反直觉

1. **「模型大 = 显存不够」** ❌ —— 搜广推的「大」在 embedding table(内存/存储),不在 GPU 显存的算力。第一反应该问稀疏维度,不是问几张卡。

2. **「embedding 维度调大是免费的」** ❌ —— 维度翻倍 = 存储翻倍 + 网络流量翻倍 + optimizer 状态翻倍。这是要算资源账的。

3. **「hash 一定会丢精度」** ❌ —— hash 到 64bit 全空间碰撞率近乎可忽略;只有强行压到小 bucket 才会明显掉点。要区分「无损 hash」和「有损压缩 hash」。

4. **「PS 和 AllReduce 二选一」** ❌ —— 搜广推标配是混合:稀疏 PS + 稠密 AllReduce。

5. **「异步训练更先进所以更好」** ❌ —— 异步快但有 staleness、不可复现。稀疏推荐能容忍,稠密大模型通常坚持同步。先进与否取决于场景。

6. **「训练慢肯定是算法问题」** ❌ —— 很多时候是平台问题:Server 热点、straggler 病机、NCCL 网络没配好。平台研发要先自查这三样。

7. **「embedding table 会一直长大没办法」** ❌ —— 准入 + 淘汰就是用来控表的,这是平台研发的核心职责。表只涨不降说明你的淘汰没生效。

8. **「optimizer 不占空间」** ❌ —— Adam/AdaGrad 的动量/二阶矩通常让实际占用是参数本身的 2~3 倍。估容量必须把它算进去。

9. **「加卡 = 纯加速,效果不变」** ❌ —— 数据并行加 Worker 会放大全局有效 batch,通常要同步调大学习率,否则效果会变。扩卡要主动告知算法。

10. **「序列特征按平均长度规划带宽就行」** ❌ —— 超活跃用户的序列可长达上千 ID,带宽必须按 p99 而非均值规划,否则遇到长序列 batch 瞬间打满。

11. **「embedding 维度越大效果越好」** ❌ —— 边际收益递减且易过拟合,高频特征大维、长尾小维的混合维度往往比一刀切大维更优。

12. **「线上线下 hash 不一样无所谓」** ❌ —— hash 函数和 slot 编号是线上线下的契约,不一致会导致同一 ID 查到错误 embedding,预测直接崩,且极难排查。

## 平台侧落地 Checklist

**稀疏存储(embedding / PS)**
- [ ] embedding table 容量监控:行数随时间曲线(应平稳,不应只涨)
- [ ] 容量估算公式落地:有效 ID 数 × 维度 × 字节 × optimizer 倍数(2~3)
- [ ] 特征准入阈值、过期天数可配置,且有默认安全值
- [ ] hash 方案明确:64bit 全空间 or bucket?碰撞率是否评估过
- [ ] Server 分片均衡 + 热点 key 检测(网卡/QPS 偏斜告警)

## 对接话术

**算法同学说**:「我把这个 slot 的 embedding 维度从 16 提到 32。」
**你接**:「行,那这个 slot 的 embedding 存储和每步 push/pull 流量会翻倍,我先看下对应 Server 的内存和网卡余量,如果接近水位我们可能要先扩 Server 或者只对高频 slot 提维度。」

**算法同学说**:「训练吞吐怎么越来越慢?」
**你接**:「我先排三样:有没有 Server 热点(某个高频 ID 把单台网卡打爆)、有没有 straggler(某台 Worker 持续偏慢拖累同步)、NCCL 网络有没有降级。先看监控再定位是平台还是模型问题。」

**算法同学说**:「换集群后 AUC 飘了、复现不了。」
**你接**:「确认下是不是同步策略变了或者 Worker 数变了——异步训练的结果跟 Worker 数和调度时序有关,本来就不严格可复现。要复现得固定同步训练 + 固定 Worker 数 + 固定种子。」

**算法同学说**:「我们用 hash bucket 把表压小了。」
**你接**:「省内存我支持,但 bucket 会带来 hash 碰撞,可能掉点。我们对比一下压缩前后的离线 AUC,确认掉点在可接受范围再上线。」

**算法同学说**:「帮我把训练从 8 卡扩到 32 卡。」
**你接**:「可以,但提醒一句:数据并行下全局有效 batch 会从 2048 变 8192,学习率大概率要跟着调(还可能要 warmup),不然效果会变。资源我来加,lr 你这边对一下,免得到时候以为是平台的问题。」

**算法同学说**:「这个特征我们想加进序列里,长度放开到 500。」
**你接**:「序列加长直接抬高每步的 pull 流量,而且超活跃用户会更夸张。我按 p99 评估下网卡和 PS 带宽,可能要给序列做截断或采样,不然长序列 batch 会把网络打满。」

## 向上沟通

**老板问**:「这个模型为什么这么吃资源?」
> 「搜广推模型的资源大头不在算力,在稀疏特征的 embedding 表——我们有几十亿个 ID,每个 ID 一个向量,表本身就几百 GB,加上优化器状态实际占用是它的 2~3 倍。这是行业普遍情况,不是我们做得差。我们已经用特征准入淘汰把表控制在合理规模。」

**老板问**:「为什么不能像别人一样几张卡就训完?」
> 「那种是 CV/NLP 的稠密模型,几张卡确实够。我们是推荐,瓶颈是『海量稀疏参数存不下单机』,所以必须用参数服务器把表分片到多台机器——这是架构差异决定的,不是堆卡能解决的。」

**老板问**:「训练能不能再快一点?」
> 「有两条路:一是切到异步训练,吞吐能上去,代价是效果可能有小波动、结果不严格可复现;二是优化稀疏侧热点和网络。建议先优化工程瓶颈,异步作为备选,因为它会影响效果稳定性,要算法一起评估。」

**老板问**:「这套架构稳不稳,会不会训一半崩了?」
> 「我们有 checkpoint 和故障自动恢复——几百台机器跑几天,挂机器是常态,但能从最近的检查点恢复,不会从头再来。这是平台已经兜底的能力。」

## 一图总结 + 一句话速记

```
                         搜广推一次分布式训练全景
   ┌────────────────────────────────────────────────────────────────────┐
   │  特征:  SPARSE(亿级 ID,99% 参数)      DENSE(数值,几十~千维)    │
   │            │                                  │                       │
   │     feasign(64bit hash)+ 准入/淘汰       归一化                       │
   │            │                                  │                       │
   │            ▼                                  ▼                       │
   │  ┌─────────────────────┐            ┌─────────────────┐              │
   │  │ embedding table(TB) │            │   dense 向量     │              │
   │  │ 模型并行 → 分片到 PS │            │                  │              │
   │  └──────────┬──────────┘            └────────┬────────┘              │
   │             │ pull/push(只传 batch 用到的)   │                       │
   │             ▼                                  ▼                       │
   │  ┌─────────────────────┐            ┌─────────────────────────────┐  │
   │  │  Server(存稀疏分片) │◄──────────►│ Worker(算 + 存稠密 MLP)     │  │
   │  │  扩缩容/热点/容错     │            │ 数据并行 + AllReduce(NCCL)  │  │
   │  └─────────────────────┘            └─────────────────────────────┘  │
   │             同步策略:BSP(准)/ ASP(快有 staleness)/ SSP(折中)    │
   └────────────────────────────────────────────────────────────────────┘
```

**一句话速记**:
> **稀疏 ID 经 feasign 哈希、靠准入淘汰控表,变成 TB 级 embedding 分片到 PS(模型并行);稠密 MLP 每机一份靠 AllReduce/NCCL 同步(数据并行);同步求稳、异步求快——平台研发的活,就是把这张大表存得下、把这些机器同步得快、还得挂了能恢复。**

## 自测三问

**Q1:为什么说搜广推模型的「大」主要在 embedding table 而不在 MLP?容量怎么估?**

<details>
<summary>参考答案</summary>

因为 MLP 的参数量是「层间连接数」,几百×几百,撑死几百万个;而 embedding table 的参数量是「ID 数 × 维度」,几十亿 × 16 = 几百亿个,差三四个数量级。容量估算公式:**有效 ID 数 × embedding 维度 × 字节数(fp32=4)×(1 + optimizer 倍数,Adam 约 2~3)**。比如 10 亿 ID × 16 维 × 4B ≈ 64GB 参数,含 Adam 实际约 190GB。
</details>

**Q2:PS 和 AllReduce 是不是二选一?为什么搜广推常用混合架构?**

<details>
<summary>参考答案</summary>

不是二选一,搜广推标配是混合。原因:**稀疏 embedding 太大(TB 级),每个 Worker 不可能各存一份完整副本,只能分片集中到 Server,走 PS(本质是模型并行)**;而**稠密 MLP 很小,每个 Worker 存一份完整副本各自更新,用 Ring-AllReduce 同步更快(数据并行,带宽最优、无 Server 热点)**。所以「Server + NCCL 同时出现」不矛盾,是最优解。
</details>

**Q3:异步训练(ASP)为什么快,代价是什么?为什么搜广推比 CV/NLP 更能容忍异步?**

<details>
<summary>参考答案</summary>

快是因为 Worker 不用等彼此(无同步屏障,没有 straggler 拖累)。代价是 **staleness(梯度陈旧度)**——梯度可能基于已被别人更新过的旧参数,导致收敛慢/震荡/不可复现。搜广推更能容忍,是因为**稀疏梯度很局部**:每个 batch 只更新它用到的少数 ID 的 embedding,不同 Worker 大概率在更新不同 ID,冲突天然少,staleness 伤害小。而稠密大模型梯度是全局的,异步冲突严重,通常坚持同步。
</details>
