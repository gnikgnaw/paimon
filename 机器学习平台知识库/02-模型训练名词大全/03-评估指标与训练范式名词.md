## 0. 30 秒速通（TL;DR）

- **AUC**：衡量模型**排序能力**的指标——"模型给正样本打的分，是否普遍高于负样本"。搜广推用它不用准确率（accuracy），是因为正样本极稀（CTR 常年 1%~5%），准确率会被"全预测不点击"骗到 95%+ 却毫无用处。AUC 取值 0.5（瞎猜）~1.0（完美），线上模型常在 **0.65~0.85（经验值，随业务差异极大）**。
- **GAUC（Group AUC）**：把 AUC **按用户（或按 query / 按场景）分组算完再加权平均**。因为线上排序是"在同一个人面前排不同 item"，跨用户比大小没意义。GAUC 比 AUC 更贴近线上效果，是搜广推最受信的离线指标。**对平台：要保留 `user_id` 维度，离线评估不能把它丢了。**
- **logloss / 交叉熵**：模型训练时**真正在最小化的损失函数**，衡量"预测概率离真实 label 有多远"。它同时关心排序**和**数值准不准，AUC 只关心排序。
- **COPC（Click Over Predicted Click）= 实际点击数 / 预估点击数之和**，等价物是 **calibration（校准）**。广告 CTR 预估必须校准，因为 **eCPM = bid × pCTR** 直接进**计费和排序**——pCTR 偏高 10%，相当于多收/少收 10% 的钱。**对平台：校准要拿到真实曝光分布、按时段/人群分桶统计 COPC。**
- **离线指标 vs 在线指标**：离线看 AUC/GAUC/logloss，**线上看 CTR/CVR/RPM/时长**，并且**只认 A/B 实验**。离线涨线上不涨是常态（样本偏差、穿越、校准漂移、线上线下不一致）。
- **训练范式四件套**：
  - **全量训练（full/batch）**：从头扫全部历史样本，周期性（天级）重训，是地基和"重置点"。
  - **增量训练（incremental）**：在昨天的模型（base）上只喂新样本继续训（delta），**热启动 warm-start**，省算力、追时效。
  - **在线学习（Online Learning / ODL）**：样本一边产生一边训练，模型按**分钟级**滚动更新。
  - **流式训练（streaming）**：基于流式样本（Flink/Kafka）持续训练，是在线学习的工程形态。
- **平台研发的总命题**：指标决定**你要保留/统计哪些维度**（user_id、曝光分布、时段），范式决定**你要提供什么形态的样本流和模型产出链路**（全量大表 vs 按时间切片的增量流 vs 实时样本流）。

> 💡 全篇一句话：**指标是"怎么判模型好不好"，范式是"怎么把模型攒出来"——平台研发不调模型，但要把这两件事的工程底座铺平。**

## 1. 先建坐标系：指标和范式，到底在回答什么问题

新人最容易把这一堆词混成一锅。先用一张图把它们摆到正确的位置上：

```
                      一个推荐/广告模型的生命周期
                                  │
        ┌─────────────────────────┴──────────────────────────┐
        │                                                     │
   【怎么训出来 / 更新】                                 【怎么判好不好】
     训练范式 Paradigm                                    评估指标 Metrics
        │                                                     │
   ┌────┼────────┬──────────┐                      ┌──────────┼───────────┐
   │    │        │          │                      │          │           │
 全量  增量    在线学习    流式              离线指标            在线指标
 full  incr.   ODL       streaming      （样本上算）         （真实流量上看）
   │    │        │          │                │                    │
 天级  小时级  分钟级    秒/分钟级       AUC/GAUC/logloss     CTR/CVR/RPM/时长
                                          /COPC/校准           （只认 A/B）
```

**两条主线，互相牵制：**
- **范式决定时效**：全量天级、增量小时级、在线分钟级——越实时，对样本流和训练链路要求越高。
- **指标决定信任**：离线指标用来快速筛模型（便宜、快），在线指标用来定生死（贵、慢、但真实）。

```
   离线指标  ──"预测"──▶  在线指标
   （便宜/快/可重复）        （昂贵/慢/真实）
        │                      ▲
        └── 二者背离 = 平台和算法一起头疼的事 ──┘
```

> 🎓 **老师视角（开篇先立这个框架）**：很多平台研发把"指标"当成算法同学的事，跟自己无关。这是最大的认知误区。**指标决定了你的数据要怎么留**：要算 GAUC 你就不能丢 user_id；要算校准你就得留住真实曝光的分布；要做在线学习你的样本就得按时间严格有序地流。**指标是需求，平台是供给**。把这层关系理顺，你在评审里就从"工具人"变成了"能预判需求的人"。

## 2. AUC：搜广推的"默认指标"，本质是排序能力

### 2.1 一句话定义

**AUC（Area Under the ROC Curve，ROC 曲线下面积）**：随机抽一个正样本和一个负样本，模型给正样本打的分**高于**负样本的概率。

```
   AUC 的几何直觉（最好记的一种）：

   把所有样本按模型打分从高到低排成一队：

   分数高 ───────────────────────────────▶ 分数低
   [+] [+] [-] [+] [-] [-] [+] [-] [-] [-]
    ↑正样本应该尽量靠前

   AUC = P( 随机一个正样本排在 随机一个负样本 前面 )

   AUC = 1.0  →  所有正样本都排在所有负样本前面（完美排序）
   AUC = 0.5  →  正负完全混着（等于瞎猜）
   AUC < 0.5  →  排反了（模型在帮倒忙，通常是 bug）
```

### 2.2 为什么搜广推用 AUC，不用准确率（accuracy）

这是面试和评审都爱问的点，必须能脱口而出。核心原因：**搜广推的正样本极度稀疏。**

```
   一天 1 亿次广告曝光，CTR = 2%：
      正样本（点击） =   200 万
      负样本（未点击） = 9800 万

   一个偷懒的模型："全部预测为不点击"
      准确率 accuracy = 9800万 / 1亿 = 98%   ← 看着很高！
      但它一个点击都没预测对，业务价值 = 0
```

准确率被极度不平衡的样本"骗"了。而 AUC 只看**排序**——只要模型能把那 2% 的点击者排到前面，AUC 就高，跟正负比例无关。

| 指标 | 衡量什么 | 受样本不平衡影响 | 受阈值影响 | 搜广推适用性 |
|---|---|---|---|---|
| Accuracy 准确率 | 预测对的比例 | **严重**（被多数类主导） | 是（要选阈值） | ❌ 几乎不用 |
| Precision/Recall | 命中精度/召回 | 受影响 | 是 | 部分场景（如风控） |
| **AUC** | **排序能力** | **不敏感** | **否（无需阈值）** | ✅ 默认指标 |
| logloss | 概率准不准 | 受影响 | 否 | ✅ 配合 AUC 看 |

> 🎓 **老师视角**：记住一句话——**搜广推的产物是"一个排好序的列表"，不是"一个是/否的判断"**。用户面对的是"哪条广告排第一"，不是"这条广告会不会被点"。所以衡量模型，本质是衡量**排序质量**，AUC 正中靶心。你跟算法同学说"我知道你们看 AUC 是因为它衡量排序、对正负不平衡不敏感"，他立刻知道你是懂行的。

### 2.3 AUC 的"无阈值"特性，对平台意味着什么

AUC 不需要设阈值，意味着**离线评估时不用调判定门限**，平台只要：
- 保留每条样本的 **模型打分（score / pCTR）** 和 **真实 label**；
- 提供一个能在亿级样本上算 AUC 的离线评估作业（通常基于 Spark/Flink，按分桶近似算，精确算要全局排序，成本高）。

```
   平台要交付的最小评估数据：
   ┌───────────┬─────────┬─────────┐
   │ sample_id │  score  │  label  │     ← 算 AUC 只需要这两列 + 用于分组的列
   └───────────┴─────────┴─────────┘
```

### 2.4 量级与脱敏案例

> 📊 **脱敏案例（量级为经验值）**：某信息流推荐 CTR 模型，离线 AUC 从 `0.7150 → 0.7180`，算法同学说"涨了 3 个千分点（3 个 thousandth）"。在搜广推里，**AUC 涨 0.001（1 个 thousandth）就可能对应线上 CTR 涨百分之零点几**，是值得上线 A/B 的量级。所以你会看到大家盯着小数点后第三、第四位较劲——这不是矫情，是因为基数巨大，千分点的 AUC 背后是百万级的收益。

## 3. GAUC（Group AUC）：为什么它比 AUC 更像线上

### 3.1 全局 AUC 的"假"在哪

普通 AUC 是**全局**算的：把所有用户、所有 item 的打分混在一起排序。但**线上排序从来不是全局排序**——它是"在用户 A 面前排 A 的候选，在用户 B 面前排 B 的候选"。跨用户比大小，业务上根本不发生。

```
   全局 AUC 的陷阱：

   用户A的样本：  打分都偏高（A是活跃用户，特征值大）
       item1 score=0.9 (label=1)
       item2 score=0.8 (label=0)
   用户B的样本：  打分都偏低（B是新用户）
       item3 score=0.3 (label=1)
       item4 score=0.2 (label=0)

   全局排序：  0.9(+) > 0.8(-) > 0.3(+) > 0.2(-)
   全局 AUC 看起来很高，因为 A 的负样本(0.8) 排在 B 的正样本(0.3) 前面 → 白送的"对"

   但这个"对"在线上毫无意义：A 的 item 永远不会和 B 的 item 同框竞争！
```

**问题本质**：全局 AUC 把"区分不同用户的整体活跃度"也算进了排序能力，但线上只需要"在同一个人内部排对"。模型可能擅长"区分用户"却不擅长"在用户内部排序"，全局 AUC 会高估它。

### 3.2 GAUC 怎么算

**GAUC = 按 group（通常是 user）分别算 AUC，再按 group 的权重（通常是该用户的曝光/样本数）加权平均。**

```
   GAUC 计算流程：

   Step 1  样本按 user_id 分组
           ┌─ user A: [样本...] → 组内算 AUC_A = 0.72,  权重 w_A = 曝光数 120
           ├─ user B: [样本...] → 组内算 AUC_B = 0.68,  权重 w_B = 曝光数  30
           └─ user C: [样本...] → 组内算 AUC_C = ...

   Step 2  过滤无效组
           - 组内全是正样本 or 全是负样本 → AUC 无定义 → 丢弃该组
           - 组内样本数 < 阈值（如 < 2）→ 丢弃

   Step 3  加权平均
                Σ ( w_i × AUC_i )
        GAUC = ───────────────────      （w_i 常取该 user 的曝光次数或点击次数）
                     Σ w_i
```

公式直白版：

```
GAUC = Σ_u  (impression_u × AUC_u)  /  Σ_u impression_u
```

### 3.3 AUC vs GAUC 对比

| 维度 | 全局 AUC | GAUC（Group AUC） |
|---|---|---|
| 排序范围 | 所有样本混在一起 | 每个 user（group）内部 |
| 贴近线上 | 较弱（含跨用户区分能力） | **强**（就是线上排序的形态） |
| 需要的数据维度 | score + label | score + label + **group key（user_id）** |
| 计算复杂度 | 一次全局排序 | 每组排序后加权（可并行，但要 shuffle by user） |
| 对长尾用户 | 不区分 | 受分组阈值/加权方式影响大 |
| 业界共识 | 入门指标 | **更受信，常作主指标** |

> 🎓 **老师视角**：GAUC 的精髓是**"消除用户间的偏置，只考核组内排序"**。你可以这样跟老板打比方："全局 AUC 像是把全校学生的考试名次混在一起排；GAUC 是先在每个班里排名次再综合——因为录取（线上展现）是按班（按用户场景）来的，跨班比分没意义。"另外，group 不一定是 user：**搜索场景常按 query 分组，多场景模型常按 scene 分组**。听到算法说"按 query 算 GAUC"，你就知道分组键变了，样本里要留住 `query_id`。

### 3.4 GAUC 对平台研发意味着什么（重点）

GAUC 的全部工程负担都压在一个词上：**你必须保留分组维度（user_id / query_id / scene_id）。**

```
   平台要交付的评估数据（比 AUC 多一列分组键）：
   ┌───────────┬─────────┬─────────┬───────────┐
   │ sample_id │  score  │  label  │  user_id  │  ← GAUC 多出来的关键列
   └───────────┴─────────┴─────────┴───────────┘

   ⚠️ 常见翻车：样本表为了"脱敏/省空间/做特征哈希"把 user_id 丢了或哈希碰撞了
       → 算法同学想算 GAUC 时发现分不了组 → 找平台返工
```

平台落地要点：
- 样本 schema 里**显式保留** group key，不能在特征工程阶段被吃掉（哈希进 embedding 不影响，但评估列要单独留原始 id）。
- 评估作业要支持 **shuffle by user_id 后分组算 AUC**，注意**数据倾斜**（超级活跃用户/爬虫用户样本量巨大，单 group 可能拖死一个 reducer）。
- 提供**最小组样本数阈值**和**加权方式**配置项（按曝光加权 vs 等权，结果差异显著）。

> 📊 **脱敏案例**：某团队离线 AUC 0.78 看着很好，上线却平。后来补算 GAUC 只有 0.61——模型其实"会区分活跃用户，但在单个用户内部几乎排不对序"。这个 gap 就是全局 AUC 的虚高。**平台当时因为没留 user_id，补算 GAUC 花了两天重跑样本**，这就是没提前规划分组维度的代价。

## 4. logloss / 交叉熵：训练真正在优化的东西

### 4.1 它俩是一回事

**logloss（对数损失）= 二分类交叉熵（Binary Cross-Entropy）**，是 CTR/CVR 模型训练时**实际最小化的损失函数**。AUC 是"验收指标"，logloss 是"训练目标"——模型梯度下降优化的是 logloss，不是 AUC。

```
   单条样本的 logloss：

      label=1（点击了）：  loss = -log(p)         p 越接近 1，loss 越小
      label=0（没点击）：  loss = -log(1 - p)     p 越接近 0，loss 越小

   合并写：  loss = -[ y·log(p) + (1-y)·log(1-p) ]

   整体 logloss = 所有样本 loss 的平均
```

```
   直觉图：预测越"自信地错"，惩罚越狠（指数级）

   loss
    ▲
    │ \                      label=1 时的 loss = -log(p)
    │  \
    │   \___
    │       \____
    │            \________
    └──────────────────────▶ p（预测点击概率）
    p=0(自信说不点，结果点了)        p=1(预测对了, loss→0)
    loss 巨大
```

### 4.2 logloss 和 AUC 的分工

| 维度 | AUC | logloss / 交叉熵 |
|---|---|---|
| 角色 | 验收指标（排序好不好） | 训练目标（梯度优化它） |
| 关心排序 | ✅ 只关心排序 | ✅ 间接关心 |
| 关心数值准不准 | ❌ 不关心（只看相对大小） | ✅ **关心**（预测概率离 label 多远） |
| 能反映校准 | 不能 | **能**（系统性偏高/偏低会推高 logloss） |
| 单调缩放打分后 | 不变（如全 ×2） | **会变** |

关键反差：**把所有打分都乘 2，AUC 一点不变（排序没动），但 logloss 会变（数值偏了）**。所以广告这种"既要排序对、又要数值准"的场景，必须**AUC 和 logloss 一起看**。

> 🎓 **老师视角**：教新人记一句——**"AUC 管位次，logloss 管分数"**。考试只看名次（AUC），那把所有人分数同比例放大不影响名次；但如果要按分数发奖学金（按 pCTR 计费），那分数本身准不准就要命了，这就是 logloss / 校准的地盘。听到算法说"AUC 没动但 logloss 涨了"，多半是**校准漂了**或**分布变了**，是个该警觉的信号。

### 4.3 对平台意味着什么

logloss 本身平台不用算太多（训练框架自带），但要理解它的两个工程含义：
- **logloss 是训练 loss 曲线的主角**：平台的训练监控看板上那条"loss 下降曲线"通常就是它。loss 不降 / 突然 NaN，是平台要能报警和定位的（学习率、脏样本、梯度爆炸）。
- **logloss 异常 = 样本或分布有问题的早期信号**：训练 loss 正常但评估 logloss 飙高，往往是**训练集和评估集分布不一致**（时间切片、采样比例问题），平台要能按时间/数据源切分对比。

## 5. COPC 与 Calibration 校准：广告为什么必须"算得准"

### 5.1 COPC 是什么

**COPC（Click Over Predicted Click）= 实际点击数总和 / 预估点击数（pCTR）总和。**

```
   一批曝光样本：
      实际点击数        = Σ label        （比如 10000 次曝光里真点了 200 次）
      预估点击数        = Σ pCTR         （模型对这 10000 次预测的概率之和，比如 220）

      COPC = 200 / 220 = 0.909

   COPC = 1.0    →  完美校准（预估总量 = 实际总量）
   COPC < 1.0    →  模型高估（预测点击数 > 实际，pCTR 偏高）
   COPC > 1.0    →  模型低估（预测点击数 < 实际，pCTR 偏低）
```

COPC 是**校准（calibration）**最常用的单值度量。校准的完整含义是：**模型预测概率 = 真实发生频率**——即"模型说有 30% 概率点击的那批曝光，真实点击率确实接近 30%"。

```
   校准曲线（Calibration / Reliability Curve）：

   实际点击率
    ▲
   1│                              ╱ ← 理想：y=x（预测=实际）
    │                          ╱
    │                      ╱  · 实测点（按 pCTR 分桶统计真实CTR）
    │                  ╱   ·
    │              ╱ ·
    │          ╱·
    │      ╱·
    0└────────────────────────────▶ 预测点击率 pCTR
     0                              1

   实测点贴着 y=x → 校准好；系统性偏离 → 需要校准（calibration）
```

### 5.2 为什么广告 CTR 必须校准（排序模型可以不那么在意）

这是整篇最该讲透的"价值点"。核心在广告的计价机制：

```
   广告排序与计费（GSP/eCPM 体系）：

      eCPM = bid（广告主出价） × pCTR（模型预估点击率） × 1000

      排序：按 eCPM 从高到低排（决定谁展现）
      计费：按 eCPM 折算扣费（决定收多少钱）

   ┌──────────────────────────────────────────────┐
   │  pCTR 偏高 10%  →  eCPM 虚高 10%               │
   │     → 排序错乱（不该上的广告上了）            │
   │     → 计费偏差（多扣广告主的钱 / 平台少收）   │
   │     → 直接是钱和信任问题，不是"掉点"问题      │
   └──────────────────────────────────────────────┘
```

对比推荐排序：纯推荐只要**相对顺序**对就行（用 AUC/GAUC 衡量），pCTR 整体偏高偏低不影响"谁排前面"。**但广告里 pCTR 是个进了乘法公式、要换算成钱的绝对数值**，所以必须校准。

| 场景 | 要排序对（AUC/GAUC） | 要数值准（COPC/校准） | 为什么 |
|---|---|---|---|
| 纯推荐（信息流排序） | ✅ 必须 | 弱（可选） | 只决定展现顺序 |
| 广告 CTR 计费 | ✅ 必须 | ✅ **必须** | pCTR 进 eCPM，影响计费/排序 |
| 广告预算控制/ROI | ✅ | ✅ **必须** | 预估量决定花钱节奏 |
| CVR 预估（转化） | ✅ | ✅ 常需要 | 影响 oCPC/oCPX 出价 |

### 5.3 模型为什么会"不准"，怎么校准

不准的常见来源：
- **负采样**：训练时为了平衡正负比把负样本下采样了，pCTR 会系统性偏高，**必须用采样率反推校准**（这是平台和样本工程的硬交接点，详见 `04-样本数据管理/03-负采样与样本偏差治理.md`）。
- **分布漂移**：训练样本的时段/人群分布 ≠ 线上真实曝光分布。
- **冷启动 / 长尾**：新广告、新用户预估系统性偏差。

常见校准手段（了解即可，知道平台要喂什么数据给它）：

```
   负采样后的解析校准（最常见）：
      训练时负样本采样率 = r（如保留 1/10 负样本）
      校准后 pCTR_cal = pCTR / ( pCTR + (1 - pCTR)/r )

   分桶校准 / Isotonic Regression / Platt Scaling：
      用一批"真实曝光 + 真实label"拟合一条 校准映射 f(pCTR) → 校准后概率
      ⚠️ 这批数据必须是【线上真实曝光分布】，不能是采样过的训练集！
```

> 🎓 **老师视角**：把校准讲成"翻译损失修正"。模型在"采样/失真的训练世界"里学会了一套概率，线上是"真实世界"，两边比例不同，所以要做一次"汇率换算"——COPC 就是这个汇率偏离 1 的程度。能把"负采样 → pCTR 偏高 → 必须校准 → 校准要真实曝光分布"这条链讲顺，你就抓住了广告平台最容易出事故、也最值钱的一环。

### 5.4 COPC / 校准对平台研发意味着什么（重点）

这是平台研发**直接要扛**的需求：

```
   要支撑校准，平台必须提供：

   ① 真实曝光分布的样本
      - 不能只给采样后的训练集
      - 要有一份"未经负采样 / 或记录了采样率"的曝光流，用来统计真实CTR

   ② 按维度分桶统计 COPC 的能力
      ┌─────────┬───────────┬───────────┬───────┐
      │  时段   │  pCTR分桶  │ 实际点击  │ COPC  │   ← 不能只算一个全局COPC
      ├─────────┼───────────┼───────────┼───────┤
      │ 0-1点   │ [0.0,0.1) │   ...     │ 0.95  │
      │ 高峰    │ [0.1,0.2) │   ...     │ 1.12  │   ← 高峰期偏低 → 该时段要单独校准
      └─────────┴───────────┴───────────┴───────┘

   ③ 采样率的可追溯
      - 每条样本要能拿到它的负采样率 r，否则没法做解析校准

   ④ 校准监控看板 + 报警
      - 实时/准实时盯 COPC 偏离 1 的程度，飘了要能报警
      - 因为 COPC 飘 = 直接的计费风险
```

> 📊 **脱敏案例（量级经验值）**：某广告系统一次模型上线后全局 COPC 看着是 1.0，没人报警。但**按时段拆开**发现凌晨 COPC=0.7、高峰 COPC=1.3，正负相抵被全局指标掩盖了——高峰期严重高估导致超投、凌晨低估导致少花预算。教训：**COPC 必须分桶看，全局 COPC=1 可能是两个反向偏差抵消的假象**。这直接定义了平台监控看板的设计：要支持多维下钻。

## 6. 离线指标 vs 在线指标：离线涨 ≠ 线上涨

### 6.1 两个世界的指标

```
   ┌────────────── 离线（Offline）──────────────┐   ┌──────── 在线（Online）────────┐
   │  在历史样本上算，便宜、快、可重复            │   │  在真实流量上看，贵、慢、真实   │
   │                                             │   │                               │
   │   AUC / GAUC      排序能力                   │   │   CTR    点击率                │
   │   logloss         概率准不准                 │   │   CVR    转化率                │
   │   COPC / 校准     量级准不准                 │   │   RPM    千次展现收入（广告）  │
   │   loss 曲线       训练收敛                   │   │   时长/停留/留存  用户体验      │
   │                                             │   │   GMV / 成交   电商            │
   │   ✅ 用来"筛"模型，决定要不要上线A/B          │   │   ✅ 用来"定"模型，A/B显著才推全 │
   └─────────────────────────────────────────────┘   └───────────────────────────────┘
                        │                                            ▲
                        └──── 离线只是"预测"，A/B 才是"判决" ─────────┘
```

### 6.2 为什么离线涨了线上不一定涨（核心认知）

这是平台研发**必须能向老板解释**的现象。原因有一串：

| # | 离线涨线上没涨的原因 | 本质 | 平台能做什么 |
|---|---|---|---|
| 1 | **离在线不一致**（特征值对不上） | 训练用的特征 ≠ 线上真实特征 | 特征快照/一致性校验（见关联阅读） |
| 2 | **样本偏差 / 幸存者偏差** | 训练样本只有"被展现过"的，没见过被模型自己滤掉的 | 负采样治理、随机流量样本 |
| 3 | **穿越（时间泄漏）** | 离线用了预测时刻还不存在的未来信息，离线虚高 | point-in-time 样本拼接 |
| 4 | **校准漂移** | 排序对了但 pCTR 不准，影响广告计费/出价 | COPC 监控、校准链路 |
| 5 | **指标 ≠ 业务目标** | AUC 涨但优化的不是用户真正在意的（如多样性、时长） | 多目标评估、约束指标 |
| 6 | **A/B 没显著 / 流量不够** | 离线 0.001 的涨幅在线上被噪声淹没 | 实验平台、样本量/显著性 |
| 7 | **线上线下数据口径不同** | 评估集和线上人群/时段分布不一致 | 评估集贴近线上分布 |

```
   典型剧情（平台研发要能复述）：

   离线 AUC +0.003 🎉  →  上线 A/B  →  线上 CTR 持平甚至 -0.5% 😱
                                          │
              ┌───────────────────────────┼───────────────────────────┐
              ▼                           ▼                           ▼
        特征对不齐                    样本有穿越                  校准飘了
      (离在线不一致)              (离线虚高)                (排序对但计费偏)
```

> 🎓 **老师视角**：这是平台研发**最能体现价值**的话题。算法同学常被老板追问"你离线明明涨了为什么线上没涨"，这时候**懂工程的平台研发能帮算法把锅分清楚**：是不是特征没对齐？样本是不是穿越了？是不是 A/B 流量不够显著？你能把这 7 条说清楚，老板会觉得"平台是懂业务的，不是只管机器的"。记住主张：**离线指标是"实验室读数"，A/B 才是"临床试验"，永远以 A/B 为准。**

### 6.3 对平台意味着什么

- **离线评估要尽量贴近线上**：评估集的时间、人群、采样要尽量还原线上分布；最好用**未来一天的真实曝光**做评估（time-based split），而不是随机切。
- **要打通 A/B 实验平台**：模型版本 → 实验分组 → 在线指标回收，形成"离线筛 → A/B 判 → 推全"的闭环。
- **要能做离在线指标对比看板**：同一模型版本，离线 AUC/GAUC 和线上 CTR/RPM 摆在一起，长期积累"离线涨 X 个点 → 线上大概涨 Y"的经验映射（这是非常值钱的平台资产）。

## 7. 训练范式总览：全量 / 增量 / 在线 / 流式

从这里开始换主线，从"怎么判好坏"转到"怎么把模型攒出来"。

```
   时效轴（更新一次模型要多久）：

   慢 ◀───────────────────────────────────────────────────────▶ 快
   天级               小时级            分钟级           秒~分钟级
   ┌─────────┐      ┌─────────┐     ┌──────────────┐  ┌──────────┐
   │ 全量训练 │      │ 增量训练 │     │ 在线学习 ODL  │  │ 流式训练  │
   │  Full   │      │  Incr.  │     │ Online Learn │  │ Streaming│
   └─────────┘      └─────────┘     └──────────────┘  └──────────┘
   扫全部历史        base+新样本      样本到了就学      流式样本持续训
   重训(重置)        warm-start      分钟级出模型      工程形态

   样本要求：
   离线大表        按时间切片的       严格时序、         流式样本
   (天级分区)       增量样本流         不能穿越          (Kafka/Flink)
```

四种范式不是互斥的，**生产里通常是组合拳**：

```
   典型组合：全量打底 + 增量追新 + （部分场景）在线学习兜实时

   每天凌晨：全量训练出一个干净的 base 模型（重置累积误差）
        │
        └─▶ 白天：在 base 上做增量训练，每小时/每几分钟出一个 delta
                │
                └─▶ 高时效场景：在线学习，分钟级滚动更新
```

| 范式 | 更新周期 | 样本来源 | 是否热启动 | 主要诉求 | 主要风险 |
|---|---|---|---|---|---|
| 全量 Full/Batch | 天级 | 全部历史（数仓大表） | 否（从头或固定init） | 稳定、可复现、重置误差 | 慢、贵、追不上时效 |
| 增量 Incremental | 小时/分钟级 | 新增时间切片 | **是（warm-start）** | 时效、省算力 | 误差累积、分布漂移 |
| 在线学习 ODL | 分钟级 | 实时样本流 | 是（持续） | 极致时效（热点/突发） | 稳定性、脏样本、回滚难 |
| 流式训练 Streaming | 秒~分钟 | Flink/Kafka 流 | 是（持续） | 实时性 + 工程化 | 一致性、乱序、状态管理 |

## 8. 全量训练（Full / Batch）：地基与重置点

### 8.1 定义

**全量训练**：每个训练周期（通常天级）**从头扫一遍全部（或一个长窗口的）历史样本**，训练出一个完整模型。它是模型的"地基"，也是误差的"重置点"。

```
   全量训练时间线：

   Day1 全部样本 ──train──▶ Model_v1
   Day2 全部样本 ──train──▶ Model_v2   （重新扫 Day1+Day2 或固定窗口）
   Day3 全部样本 ──train──▶ Model_v3
        │
        每次都是"扫海量历史，跑很久，但干净"
```

### 8.2 特点

| 优点 | 缺点 |
|---|---|
| 结果稳定、可复现（同样本同结果） | 慢——亿/百亿样本跑数小时 |
| 没有误差累积（每次重置） | 贵——占大量算力 |
| 适合做基线 / 大版本迭代 | 时效差——抓不住当天热点 |
| 分布覆盖全（长窗口历史） | 用旧分布可能不贴近今天 |

### 8.3 对平台意味着什么

- 提供**按时间分区的全量样本大表**（如 Hive/数据湖天级分区），支持大窗口扫描。
- 调度上是**重资源批作业**：要排队、要资源隔离、要 checkpoint/重试（跑 6 小时挂了不能从头来）。
- **可复现性**：固定随机种子、固定样本范围、固定特征版本——全量模型常作"基线"，必须能复现。
- 全量产出的模型是后续增量的 **base**（见下一节）。

> 🎓 **老师视角**：全量训练在生产里的真正角色是**"定期重置"**。增量训练跑久了会累积误差、被脏样本带偏、和真实分布越飘越远，所以工业界普遍**"全量打底 + 增量追新"**——每天凌晨用全量重训一个干净 base，把累积误差清零，白天再增量追时效。你理解了这个"重置"逻辑，就明白为什么平台要同时支撑这两条链路，而不是二选一。

---

## 9. 增量训练（Incremental）：base + delta 与 warm-start

### 9.1 定义与核心概念

**增量训练**：不从头训，而是**加载上一个模型作为起点（base / 热启动 warm-start），只喂新增的样本继续训练**，产出更新后的模型。

```
   增量训练 = warm-start（热启动）

   ┌──────────┐    加载参数    ┌──────────┐  喂新样本  ┌──────────┐
   │ base模型 │ ──warm-start─▶ │ 继续训练  │ ─────────▶│ 新模型   │
   │ (昨天/上 │   (不从随机    │ 只用新增  │           │ (base+   │
   │  个pass) │    初始化开始) │ 时间切片  │           │  delta)  │
   └──────────┘                └──────────┘            └──────────┘

   对比冷启动（cold-start / 从随机初始化训）：
      冷启动每次都从零学，慢且抖；warm-start 继承已学知识，快且稳。
```

**几个必须分清的词：**

| 术语 | 含义 | 类比 |
|---|---|---|
| **warm-start 热启动** | 用已有模型参数初始化，不从随机开始 | 接着上次进度继续看书 |
| **cold-start 冷启动** | 从随机初始化训练（也指新用户/新item无历史） | 从第一页重新看 |
| **base 模型** | 增量的起点（通常是全量或上个 pass 产出） | 主干 |
| **delta 更新** | 在 base 上由新样本带来的参数增量 | 补丁/增补 |

### 9.2 base + delta：模型是怎么"叠"出来的

```
   base + delta 思路（参数视角）：

   Model_new = Model_base  +  Δ(由新样本训练得到)

   工程上有两种落地：
   ① 全参数滚动：每次增量后保存完整新模型（简单，存储大）
   ② base + delta 分发：base 不变，只下发 delta（embedding 增量等）
                        → 模型分发省带宽，适合大 embedding 表
```

```
   一天的增量链路（典型）：

   00:00 全量 base ──▶ 09:00 增量 ──▶ 10:00 增量 ──▶ 11:00 增量 ──▶ ...
                       (喂09点样本)   (喂10点样本)   (喂11点样本)
                          ▲ 每次都 warm-start 自上一个模型
```

### 9.3 增量训练的风险（平台要兜住）

| 风险 | 说明 | 平台对策 |
|---|---|---|
| **误差累积** | 一直 warm-start，脏样本/偏差会越叠越歪 | 定期全量重置（天级）做 base |
| **分布漂移** | 新样本分布突变（大促、突发事件）带偏模型 | 监控特征/label 分布、异常熔断 |
| **样本时序错乱** | 增量必须按时间顺序喂，乱序=变相穿越 | 样本按时间严格切片、严格有序 |
| **回滚难** | 增量链路坏了，要能回到某个干净版本 | 每个 pass 存 checkpoint，可回滚 |

### 9.4 对平台意味着什么（重点）

增量训练把样本和链路的要求从"大表"升级为"按时间切片的有序流"：

```
   平台要交付：

   ① 按时间切片的样本流
      Day 大表  →  细到 小时/分钟 的样本分片，且严格按事件时间有序
      ┌────────┬────────┬────────┬────────┐
      │ 09:00  │ 09:15  │ 09:30  │ 09:45  │  ← 增量按片喂，不能乱序
      └────────┴────────┴────────┴────────┘

   ② 模型版本管理 / checkpoint
      - 每个增量产出要可保存、可加载（warm-start 的前提）
      - 要能"指定从某个 base 重新增量"（回滚/补训）

   ③ base 与 delta 的衔接调度
      - 全量何时产出新 base、增量何时切到新 base，要有调度协议
      - 切 base 时刻的样本边界不能丢/不能重（exactly-once 语义最好）

   ④ 样本时效保障
      - 增量的价值就在"快"，样本从产生到可训练的延迟要监控（label 回流延迟尤其关键，见下一节样本时效）
```

> 🎓 **老师视角**：跟算法对接增量训练，最容易出锅的不是算法，是**样本时序和边界**。warm-start 的本质是"接着上次继续学"，所以**喂样本的顺序和时间窗口必须和模型预期完全对齐**——多喂一片、漏喂一片、顺序错了，都等于给模型"喂了穿越数据"，离线看不出来，线上掉点。平台研发要把"按时间严格切片、不重不漏不乱序"当成增量链路的第一原则。

## 10. 在线学习（Online Learning / ODL）与流式训练

### 10.1 定义

**在线学习（Online Learning，搜广推常叫 ODL = Online Deep Learning）**：样本**一边在线上产生、一边几乎实时地用来更新模型**，模型按**分钟级**滚动出新版本。**流式训练（streaming）**是它的工程形态——基于流计算（Flink/Kafka）持续消费样本流训练。

```
   从 batch 到 online 的连续谱：

   全量(天)  ──▶  增量(小时)  ──▶  在线学习/流式(分钟~秒)
   攒一大批再训        攒一小批再训        来一点学一点

   在线学习数据流：
   线上曝光/点击 ──▶ 实时样本拼接 ──▶ 样本流(Kafka) ──▶ 流式训练 ──▶ 分钟级新模型 ──▶ 推送上线
        (label回流)        (join特征/label)                  (持续更新)                  │
        ▲──────────────────────────────────────────────────────────────────────────────┘
                          闭环：模型越新 → 越贴当下 → 抓住热点/突发
```

### 10.2 为什么要在线学习

核心价值：**抓住时效性极强的信号**——突发热点、大促、直播、新闻、库存/价格剧变。这些场景里，"昨天的模型"甚至"一小时前的模型"都已经过时。

| 场景 | 为什么需要在线学习 |
|---|---|
| 突发热点/新闻 | 几分钟内流量结构剧变，老模型完全抓不住 |
| 大促/秒杀 | 用户行为、商品热度分钟级变化 |
| 直播/短视频 | 内容生命周期极短，冷启动要极快 |
| 实时竞价广告 | 出价/转化环境实时波动 |

### 10.3 在线学习的代价（平台要扛的复杂度）

```
   在线学习把所有"实时性难题"一次性全摊给平台：

   ┌─ 样本实时拼接：特征和 label 要在流里实时 join（label 还可能延迟到达）
   ├─ 时序与乱序：流数据天然乱序，处理不好=穿越
   ├─ 脏样本即时影响：一条脏样本可能立刻带歪线上模型（batch 有缓冲，online 没有）
   ├─ 稳定性/熔断：模型分钟级在变，崩了影响面是实时的
   ├─ 回滚：要能秒级回退到上一个好版本
   └─ 一致性：流式状态管理、exactly-once、checkpoint
```

| 维度 | 增量训练 | 在线学习 / 流式 |
|---|---|---|
| 触发方式 | 定时（每小时） | 事件驱动（样本到达即学） |
| 样本形态 | 时间切片文件 | 持续流（Kafka/Flink） |
| 出模型周期 | 小时级 | 分钟级 |
| 脏样本影响 | 下个批次才生效，可拦 | **几乎实时**，更危险 |
| 工程复杂度 | 中 | **高**（流式状态/乱序/一致性） |
| 回滚 | 切回上个版本 | 要秒级回退机制 |

### 10.4 label 延迟——在线学习最棘手的工程点

```
   曝光发生在 t0，但 label（点击/转化）可能很晚才到：

   t0 曝光 ────────────────────────────────────▶
              点击可能 t0+3秒          转化可能 t0+几小时甚至几天
                 │                            │
        ┌────────┴───────┐          ┌─────────┴─────────┐
        │ CTR：label较快  │          │ CVR：label极慢     │
        │ 等几分钟窗口     │          │ 归因窗口长，要等   │
        └────────────────┘          └───────────────────┘

   平台抉择：等多久算"没点击=负样本"？等太短漏正样本，等太长丢时效。
   → 需要"负样本回补 / fake-negative 修正"等机制（属样本工程深水区）
```

> 🎓 **老师视角**：在线学习听起来很酷（"模型分钟级自我进化"），但**它是把工程复杂度和风险拉满的范式**。你要在评审里清醒地提醒：在线学习的收益主要在**强时效场景**，对很多稳态业务，增量训练就够了，没必要为了"实时"上一套又贵又难维护的流式训练。**label 延迟和脏样本即时传播是两座大山**——能把这两点讲清楚，说明你真懂这套东西的代价，而不是被名词唬住。

### 10.5 对平台意味着什么（重点）

```
   要支撑在线学习/流式，平台至少要建：

   ① 实时样本流水线：特征+label 流式 join，处理乱序/迟到（watermark）
   ② 流式训练引擎：消费样本流持续训练，状态/参数服务器实时更新
   ③ 模型秒级/分钟级分发：新模型快速推到线上推理服务
   ④ 实时监控 + 熔断：COPC/AUC/分布的实时看板，异常自动停更/回滚
   ⑤ 版本与回滚：每个分钟级模型可追溯、可秒级回退
   ⑥ exactly-once / checkpoint：样本不重不漏，训练状态可恢复
```

## 11. 按 pass / 分钟级产出模型：模型版本是怎么流出来的

### 11.1 pass 在这里的含义

注意：**pass 在不同语境含义不同**（训练里"一个 pass"常指扫一遍数据，详见 `01-训练流程名词(pass·slot·batch·epoch).md`）。但在**增量/在线学习的产出语境**里，"按 pass 产出"通常指：**每处理完一个时间片/一批样本，就产出一个模型版本（一个 pass = 一次模型快照）**。

```
   "按 pass / 分钟级产出模型" 的版本流：

   样本流 ───┬──── pass_1 (09:00-09:01) ──train──▶ Model@09:01 ──▶ 上线
            ├──── pass_2 (09:01-09:02) ──train──▶ Model@09:02 ──▶ 上线
            ├──── pass_3 (09:02-09:03) ──train──▶ Model@09:03 ──▶ 上线
            └──── ...                              每个pass一个版本

   每个 pass：warm-start 自上一个 pass → 训这一片样本 → 存checkpoint → 推线上
```

### 11.2 模型产出的版本视图

| 范式 | "一个版本"对应什么 | 产出频率 | 版本号常见形态 |
|---|---|---|---|
| 全量 | 一次全量训练 | 天级 | `model_20260614` |
| 增量 | 一个时间片增量 | 小时级 | `model_20260614_10` |
| 在线/流式 | 一个 pass（分钟片） | 分钟级 | `model_20260614_1032` |

```
   模型版本流（平台要管的核心对象）：

   [base全量] → [delta] → [delta] → ... 一天几十~几百个版本
        │          │         │
        每个版本都要：可存储、可加载、可上线、可回滚、可追溯训练样本范围
```

### 11.3 对平台意味着什么

- **模型版本管理是核心能力**：分钟级产出意味着一天几百个版本，要有版本仓库、元数据（训练样本范围、base 指针、指标）、生命周期（保留/清理）。
- **产出与上线解耦**：训出来的 pass 不一定立刻全量上线，要支持灰度/A-B/影子，坏版本能拦在门口。
- **每个 pass 可追溯**：出问题要能回答"这个 model@10:32 是基于哪个 base、喂了哪段样本训出来的"——这是排障和合规的基础。

> 🎓 **老师视角**：跟老板/算法解释"按 pass 产出"时，把它类比成**游戏的自动存档**——每过一小段就存一个档（pass），随时能读档（warm-start 继续）或回到某个档（回滚）。平台研发的活，就是把这套"存档系统"做得**不丢档、读档快、坏档能扔**。这个比喻能瞬间让非技术的人理解版本管理为什么重要。

## 12. 过拟合 / 泛化 / 正负样本比 / 样本时效：搅动指标的四只手

这一节把会"扰动指标读数"的四个因素讲清楚——它们解释了"为什么指标会异常"，也直接定义了平台的样本责任。

### 12.1 过拟合 vs 泛化

**过拟合（overfitting）**：模型把训练样本"背下来了"，包括噪声，在训练集上指标极好，在没见过的数据上崩。**泛化（generalization）**：模型在没见过的新数据上的表现——这才是真正要的。

```
   过拟合的经典曲线：

   指标(loss)
    ▲
    │＼                          验证集 loss（泛化）
    │ ＼                    ╱── 开始上升 = 过拟合了
    │  ＼________________╱
    │            ＼______________ 训练集 loss（一直降）
    │                    ＼______
    └───────────┼──────────────────▶ 训练轮数/时间
            最佳停止点
           (early stopping)
```

```
   过拟合 vs 欠拟合：

   欠拟合(underfit)        刚好(good fit)         过拟合(overfit)
   训练差/验证差          训练好/验证好           训练极好/验证差
   模型太弱              泛化好                  背了噪声
   AUC: 0.6/0.6         0.78/0.77              0.95/0.65
```

| 现象 | 训练集表现 | 验证/线上表现 | 平台/算法对策 |
|---|---|---|---|
| 欠拟合 | 差 | 差 | 加特征、加模型容量 |
| 良好泛化 | 好 | 好（接近训练） | 维持 |
| 过拟合 | 极好 | 明显变差 | 正则、early-stopping、更多样本、dropout |

**对平台**：要支持**训练集/验证集按时间切分**（time-based split，绝不能随机切——随机切会穿越），提供**验证集指标曲线监控**支撑 early-stopping。

### 12.2 正负样本比

搜广推天然正样本稀少（CTR 1%~5%），**正负样本比**严重失衡。处理方式（负采样）会反过来影响指标和校准。

```
   原始：正:负 = 1:50（CTR 2%）
        │
   负采样保留 1/10 负样本 → 正:负 = 1:5（训练更高效、loss梯度更稳）
        │
   ⚠️ 代价：pCTR 系统性偏高  →  必须校准（回到第5节 COPC）
   ⚠️ AUC 基本不受采样影响（排序），logloss/COPC 受影响（数值）
```

| 处理方式 | 对训练 | 对 AUC | 对校准/COPC | 平台责任 |
|---|---|---|---|---|
| 不采样 | 慢、梯度被负样本主导 | — | 不偏 | 存全量负样本（贵） |
| 负采样 | 快、稳 | 几乎不变 | **系统偏高，需校准** | **记录采样率**供校准 |
| 正样本加权 | 平衡梯度 | 影响小 | 看实现 | 透传权重 |

> 🎓 **老师视角**：正负样本比这件事，平台研发要死死记住一条——**只要动了正负比（负采样/加权），就欠下了一笔"校准债"**，而还这笔债（解析校准）需要**采样率**。所以平台的样本里**必须可追溯每条样本的采样率**，否则算法想校准时拿不到数据。这是 §5（COPC）和 §12.2（正负比）的硬连接点，详见 `04-样本数据管理/03-负采样与样本偏差治理.md`。

### 12.3 样本时效

**样本时效**：从"线上发生行为"到"样本可被训练用"的延迟。它直接决定增量/在线学习的上限。

```
   样本时效链路（每一段都在加延迟）：

   曝光发生 → 特征落盘 → label回流(点击/转化) → 样本拼接join → 落到训练样本流
     t0          +ms          +秒~小时~天            +分钟           可训练
                                  ▲
                       CVR 的 label 可能延迟数小时~数天（归因窗口）
                       → 这是在线学习/增量"快不起来"的根本制约
```

| 时效水平 | 典型延迟 | 能支撑的范式 |
|---|---|---|
| 天级 | 数小时~1天 | 全量 |
| 小时级 | 分钟~小时 | 增量 |
| 分钟级 | 秒~分钟 | 在线学习/流式（CTR） |
| 实时 | 秒级 | 流式（label快的场景） |

**对平台**：样本时效是**增量/在线学习链路的核心 SLA**。要监控每段延迟，尤其 **label 回流延迟**（CVR 归因窗口长，往往是瓶颈），并提供"延迟到达样本怎么补训"的机制。

> 🎓 **老师视角**：新人常以为"在线学习快不快取决于训练引擎快不快"。错——**真正的天花板是样本时效，尤其 label 什么时候到**。CTR 的 label 几秒到，能做分钟级；CVR 的 label 可能几天后才确认，再快的引擎也没用。平台研发要把"label 回流延迟监控"当成在线学习链路的一类指标来建，这是体现专业度的细节。

### 12.4 四只手如何搅动指标——汇总

```
   ┌──────────────┬────────────────────────────────────────┐
   │ 过拟合        │ 训练AUC虚高、线上崩 → 要time-split验证   │
   │ 正负样本比    │ 负采样推高pCTR → 欠校准债，要记采样率    │
   │ 样本时效      │ label延迟限制时效 → 增量/在线的真天花板  │
   │ 离在线不一致  │ 离线涨线上不涨 → 见关联阅读             │
   └──────────────┴────────────────────────────────────────┘
   平台研发：这四只手对应的，全是【样本和评估数据】的工程责任。
```

## 13. 业界怎么做

| 公司/体系 | 范式与指标实践（公开资料概括，量级为经验值） | 平台启示 |
|---|---|---|
| 阿里（盖坤等 GAUC 提出方） | 业界 GAUC 的重要推广者，深度 CTR（DIN/DIEN）评估以 GAUC 为主；广告强调校准 | 主指标用 GAUC，留 user 维度 |
| Google（Smart Bidding/Ads） | CTR 预估极重校准（calibration），有专门校准层；离线 logloss + 在线 A/B 双轨 | 校准独立成模块，监控 COPC |
| Meta（Facebook Ads，GBDT+LR 经典论文） | 经典论文明确讨论负采样后用解析公式校准 pCTR；强调 normalized entropy（归一化交叉熵） | 负采样必配校准，记采样率 |
| 字节 / 快手（信息流/短视频） | 在线学习/流式训练做到分钟级，强时效场景（热点、直播）依赖 ODL | 流式样本流 + 分钟级模型分发 |
| 业界通用（推荐系统） | "全量打底 + 增量追新" 是主流组合；A/B 实验平台为最终裁判 | 双链路 + 实验闭环 |

```
   一个被广泛采用的"组合范式"（多数大厂的形态）：

   离线全量(天) ─打底─▶ 增量(小时) ─追新─▶ [强时效场景再叠] 在线学习(分钟)
        │                                              │
        └──── 离线 GAUC/logloss 筛 ──▶ A/B 实验判 ──▶ 推全 ◀──── COPC 实时校准监控
```

> 量级与具体做法随公司/时期差异极大，以上为公开资料与经验值概括，不作为精确事实。

## 14. 权衡取舍

```
   核心三组权衡（平台研发要会主持这些讨论）：

   ① 时效 vs 稳定
      在线学习(快,抖)  ◀──────────────▶  全量(慢,稳)
      增量是折中；越实时越难维护、越易被脏样本带偏

   ② 离线指标(便宜快) vs 在线A/B(贵慢真)
      离线只能"筛"，A/B 才能"判"；省 A/B = 赌博

   ③ 排序(AUC/GAUC) vs 校准(COPC)
      纯推荐重排序；广告两者都要，校准漂=钱的问题
```

| 取舍 | 选 A 的代价 | 选 B 的代价 | 平台立场 |
|---|---|---|---|
| 实时 vs 稳定 | 工程复杂、风险高 | 抓不住热点 | 按业务时效需求定，不为实时而实时 |
| 离线 vs A/B | 离线可能骗人 | A/B 贵且慢 | 离线筛 + A/B 判，建闭环 |
| 全量 vs 增量 | 慢、贵 | 误差累积 | 两者都要（全量打底+增量追新） |
| 不采样 vs 负采样 | 慢、存储贵 | 欠校准债 | 采样+记采样率+校准 |
| 等 label vs 抢时效 | 漏正样本 | 时效差 | 按 CTR/CVR 分别定窗口 |

> 🎓 **老师视角**：评审里最值钱的能力是**"帮团队把权衡显性化"**。算法说"我要上在线学习"，你不要直接答"行/不行"，而是把权衡摆出来："在线学习能拿到分钟级时效，代价是要建实时样本流、实时校准监控、秒级回滚，维护成本翻几倍——我们的业务真的需要分钟级吗，还是增量小时级就够？" 这一问，你就从执行者变成了决策参与者。

## 15. 常见误区与反直觉

| # | 误区 | 真相 |
|---|---|---|
| 1 | "准确率高=模型好" | 搜广推正样本稀，准确率被多数类骗。看 AUC/GAUC |
| 2 | "AUC 高线上就好" | AUC 是离线排序指标，线上看 CTR/RPM 且只认 A/B |
| 3 | "全局 AUC 就够了" | 线上是组内排序，GAUC 更真；要留 user_id |
| 4 | "排序对了就行，pCTR 不用准" | 广告 pCTR 进 eCPM 计费，必须校准（COPC） |
| 5 | "COPC=1 就校准好了" | 全局 COPC=1 可能是反向偏差抵消，要分桶看 |
| 6 | "AUC 没变模型就没问题" | AUC 不反映校准；logloss/COPC 可能已飘 |
| 7 | "在线学习越实时越好" | 时效拉满=复杂度/风险拉满；多数场景增量够用 |
| 8 | "训练引擎快=模型更新快" | 真天花板是样本时效，尤其 label 回流延迟 |
| 9 | "增量训练可以一直增量" | 会累积误差，必须定期全量重置 |
| 10 | "训练集随机切验证集" | 会穿越！必须按时间切（time-based split） |
| 11 | "负采样不影响指标" | AUC 几乎不变，但 pCTR/logloss/COPC 受影响，欠校准债 |
| 12 | "离线涨了上线就稳赢" | 离线↑线上↓是常态，原因一长串（见 §6） |

> 🎓 **老师视角**：误区 5（COPC=1 假象）和误区 10（随机切穿越）是**最能区分"听过名词"和"真懂"的两道分水岭**。能主动提这两条，评审里立刻显出深度。

## 16. 平台侧落地 Checklist

**评估指标侧：**
- [ ] 样本/评估数据**保留 `user_id`（及 `query_id`/`scene_id`）**，支撑 GAUC 分组
- [ ] 评估作业支持**全局 AUC + GAUC**，GAUC 可配最小组样本数、加权方式
- [ ] GAUC 计算处理**数据倾斜**（超活跃用户/爬虫单组过大）
- [ ] 同时输出 **AUC、GAUC、logloss**，不只看一个
- [ ] **校准/COPC 监控**：可按时段/人群/pCTR 分桶下钻，全局+分桶都看
- [ ] 提供**真实曝光分布**样本（未采样或记采样率），供校准统计
- [ ] 每条样本**可追溯负采样率**，支撑解析校准
- [ ] 评估集采用 **time-based split**（贴近线上未来分布），不随机切

**训练范式侧：**
- [ ] **全量链路**：大表分区扫描、资源隔离、checkpoint/重试、可复现（固定种子/样本范围/特征版本）
- [ ] **增量链路**：按时间**严格有序、不重不漏**的样本切片；warm-start 加载；base/delta 调度协议
- [ ] **模型版本管理**：版本仓库 + 元数据（base 指针、样本范围、指标），分钟级产出可承载
- [ ] **回滚能力**：每个 pass 可追溯、可（秒级/分钟级）回退到干净版本
- [ ] **在线学习/流式**（如需）：实时样本 join、乱序/迟到处理（watermark）、exactly-once、实时熔断
- [ ] **样本时效监控**：分段延迟，尤其 **label 回流延迟**（CVR 归因窗口）
- [ ] **产出与上线解耦**：灰度/影子/A-B，坏版本拦在门口

**闭环侧：**
- [ ] 打通 **A/B 实验平台**：模型版本 → 实验分组 → 在线指标回收
- [ ] **离在线指标对比看板**：长期积累"离线涨 X → 线上涨 Y"映射
- [ ] 训练 **loss / 验证集曲线监控**，支撑 early-stopping、异常报警

## 17. 对接话术：算法/训练同学怎么说、你怎么接

| 算法同学说 | 真实意图 | 你（平台）该接的话 / 要做的事 |
|---|---|---|
| "这版离线 AUC 涨了 3 个千分点" | 想上 A/B | "评估集是 time-split 的吗？我顺手补个 GAUC，再走 A/B 看 CTR" |
| "AUC 不准，要看 GAUC" | 要组内排序指标 | "样本里 user_id 留着呢，我加 GAUC 评估，按曝光加权对吧？" |
| "线上 COPC 飘了" | 校准出问题，影响计费 | "我拉分时段/分桶的 COPC 看是不是局部偏；采样率我能给你做解析校准" |
| "这个模型走在线学习" | 要分钟级更新 | "实时样本流、乱序处理、秒级回滚、实时 COPC 监控我得先建，label 延迟你们能接受多久？" |
| "增量训练接着昨天的 base" | warm-start | "我保证样本按时间严格切片、不重不漏；base 指针和切换边界我们对一下" |
| "全量重训一版做 base" | 要干净重置点 | "我固定样本范围和种子保证可复现，跑完作为今天增量的 base" |
| "logloss 涨了但 AUC 没动" | 怀疑校准/分布 | "可能是校准漂或训练/评估分布不一致，我按时间和数据源切开对比给你" |
| "样本能不能更快到" | 要更高时效 | "瓶颈大概率在 label 回流，我把分段延迟监控拉出来定位" |

> 🎓 **老师视角**：对接的精髓是**把模糊的指标诉求翻译成具体的数据/链路需求**。算法说"看 GAUC"，你脑子里立刻弹出"要 user_id、要分组评估、要防倾斜"；算法说"上在线学习"，你立刻弹出"实时流、乱序、回滚、label 延迟"。这套"词 → 工程需求"的条件反射，就是平台研发的核心竞争力。

## 18. 向上沟通：怎么把指标与范式讲给老板听

老板要的不是公式，是**风险、成本、收益、决策**。给几套话术：

**讲"离线涨了为什么线上没涨"（最常被问）：**
> "离线指标像实验室读数，A/B 才是真实临床。离线涨可能因为特征对不齐、样本穿越、或校准漂了——这几类问题都在平台侧，我们有监控能定位。所以我们的流程是离线先筛掉差的，A/B 实验做最终判决，不会拿离线数字直接下结论。"

**讲"为什么要做 GAUC / 留 user_id"：**
> "全局 AUC 像把全校学生名次混排，但我们的产品是在每个用户面前排序，所以要按用户分组算（GAUC），这更接近真实效果。代价是样本要保留用户维度，这点存储成本换的是评估可信度。"

**讲"为什么广告必须校准（COPC）"：**
> "广告的预估点击率直接进计费公式，预估偏高 10% 就相当于多收/少收 10% 的钱。所以我们不光要排序对，还要数值准，专门有 COPC 监控盯着，飘了会报警——这是钱和广告主信任的问题。"

**讲"要不要上在线学习"（控制预期/成本）：**
> "在线学习能做到分钟级更新，适合热点、大促这种强时效场景。但它要建实时样本流、实时监控、秒级回滚，维护成本和风险比增量训练高一个量级。建议先确认业务是否真需要分钟级，否则增量（小时级）性价比更高。"

```
   向上沟通万能结构：

   现象/诉求 → 用一个生活类比解释 → 平台能/已经做什么 → 成本与风险 → 给决策建议
```

> 🎓 **老师视角**：跟老板沟通永远把"钱、风险、决策"放前面，把"公式、术语"放后面或不放。COPC 不要讲"实际点击除以预估点击",讲"预估偏 10% 等于钱偏 10%"。这种翻译能力，比你懂多少公式更能决定你在组织里的位置。

## 19. 一图总结 + 一句话速记

```
 ┌───────────────────────────── 评估指标与训练范式全景 ─────────────────────────────┐
 │                                                                                  │
 │   【怎么判好坏 · 指标】                        【怎么攒模型 · 范式】              │
 │                                                                                  │
 │   离线 ┌─ AUC ........ 排序能力(无阈值)        全量 Full ... 天级/重置/可复现     │
 │        ├─ GAUC ....... 组内排序(留user_id)      │                                 │
 │        ├─ logloss .... 训练目标/数值准          增量 Incr. .. 小时/warm-start     │
 │        └─ COPC/校准 .. 量级准(广告计费命脉)      │            base+delta           │
 │                                                  │                                 │
 │   在线 ┌─ CTR/CVR/RPM/时长                       在线学习 ODL  分钟级/强时效        │
 │        └─ 只认 A/B ... 最终判决                   │            流式 streaming        │
 │                                                  └─ 按 pass 产出/版本流            │
 │                                                                                  │
 │   搅动指标四只手：过拟合 · 正负样本比 · 样本时效 · 离在线不一致                   │
 │                                                                                  │
 │   平台研发的总命题：                                                             │
 │      指标 → 决定保留哪些维度(user_id/曝光分布/采样率)                            │
 │      范式 → 决定样本流形态(大表/有序切片/实时流)与模型产出链路                    │

 └──────────────────────────────────────────────────────────────────────────────────┘
```

**一句话速记：**
> **AUC 看排序、GAUC 看组内、logloss 看数值、COPC 看校准；全量打底、增量追新、在线抢时效——指标决定我留什么数据，范式决定我建什么链路。**

## 20. 自测三问

**Q1：算法同学说"离线 AUC 涨了但要补 GAUC 才敢上线"，作为平台研发，你需要确认什么、可能踩什么坑？**

<details>
<summary>参考答案</summary>

- **要确认**：样本/评估数据里是否保留了 `user_id`（GAUC 分组键）。若被特征工程哈希掉或脱敏丢了，就分不了组。
- **可能踩的坑**：
  1. 没留 user_id → 要重跑样本补字段，可能耽误几天（真实案例）。
  2. **数据倾斜**——超活跃用户/爬虫单个 group 样本量巨大，分组算 AUC 时拖死个别 reducer。
  3. 加权方式（按曝光加权 vs 等权）和最小组样本阈值会显著改变 GAUC 数值，要和算法对齐口径。
  4. 全局 AUC 高但 GAUC 低很常见（模型会区分用户但组内排不对），别被全局 AUC 误导。
- **本质**：GAUC = 按 user 分组算 AUC 再加权平均，更贴近"线上在同一个人面前排序"的真实形态。
</details>

**Q2：广告团队反馈"线上 COPC 看起来是 1.0，但收入和预期对不上"。可能是什么问题？平台该提供什么能力定位？**

<details>
<summary>参考答案</summary>

- **最可能**：**全局 COPC=1 是反向偏差抵消的假象**——比如凌晨低估（COPC<1）、高峰高估（COPC>1），正负相抵后全局看着是 1，但局部严重偏，导致高峰超投、凌晨少花预算。
- **平台要提供**：**分桶/分维度 COPC 下钻**能力——按时段、人群、pCTR 区间分别统计 COPC，而不只看全局一个值。
- **其他可能**：负采样率没正确用于校准（pCTR 系统偏移）、训练分布≠线上曝光分布、某人群/广告冷启动偏差。
- **延伸**：校准统计必须用**真实曝光分布**的数据（未采样或记录了采样率），不能用采样过的训练集；每条样本要可追溯采样率以支撑解析校准。
- **为什么要命**：pCTR 进 eCPM = bid×pCTR，直接影响排序和计费，COPC 偏=钱偏。
</details>

**Q3：算法要把某模型从"增量训练（小时级）"升级成"在线学习（分钟级）"，平台需要新增哪些能力？最大的工程瓶颈通常是什么？**

<details>
<summary>参考答案</summary>

- **需要新增**：
  1. **实时样本流水线**：特征+label 流式 join，处理乱序/迟到（watermark）。
  2. **流式训练引擎**：消费样本流持续训练，参数/状态实时更新。
  3. **模型分钟级/秒级分发**：新模型快速推到线上推理。
  4. **实时监控 + 熔断**：实时 COPC/AUC/分布看板，异常自动停更。
  5. **秒级回滚** + **版本可追溯**（每个 pass 存档）。
  6. **exactly-once / checkpoint**：样本不重不漏，状态可恢复。
- **最大瓶颈**：**样本时效，尤其 label 回流延迟**。CTR label 几秒到、可做分钟级；CVR label 可能几小时~几天后才确认（归因窗口长），再快的引擎也突破不了这个天花板。所以"等多久判负样本"是核心抉择，可能需要负样本回补机制。
- **额外提醒**：在线学习把脏样本的影响变成**近实时传播**（batch 还能拦，online 拦不住），稳定性和熔断比增量重要得多；要向团队点明"实时=复杂度/风险拉满"，确认业务真需要分钟级。
</details>
