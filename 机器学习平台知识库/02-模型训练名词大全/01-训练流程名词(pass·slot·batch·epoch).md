## 0. 30 秒速通（TL;DR）

- **slot**：搜广推里"一类特征"的槽位/编号。LR/参数服务器时代把"用户ID""广告ID""近7天点击数"……每一**类（field）**特征叫一个 slot。一个模型"几百上千个 slot"是常态。slot 决定了**样本怎么拼、特征怎么编号、参数服务器怎么分片**。
- **pass**：**一词两义**。① 通用 ML 里 `pass ≈ epoch`，过一遍数据；② 搜广推流式训练里 **`一个 pass = 一个时间片（如一小时）的增量数据`**，训练完产出一个模型版本（base + delta）。听到"pass"先问一句"你说的是哪种 pass"。
- **batch / mini-batch / batch_size**：一次前向+反向用多少条样本。影响**显存/内存占用、收敛稳定性、吞吐**。搜广推 batch_size 经验量级几千到几万，远大于 CV/NLP。
- **epoch / step / iteration**：`step（=iteration）= 处理一个 batch`；`一个 epoch = 过完整份数据 = 样本数 / batch_size 个 step`。搜广推**常只过一遍数据（one-pass）**，不跑多 epoch。
- **为什么 one-pass**：数据量大到跑多遍不划算 + 时效性要求高（模型要追上最新分布）+ 样本只看一次天然抗过拟合。
- **平台侧一句话**：这些词背后全是**对样本流、特征编号体系、参数服务器分片、checkpoint/版本管理、增量回流时效**的能力要求——你要支撑的就是这些。

> 💡 全篇围着一句话转：**"算法说的每个训练流程名词，对你都是一句'平台要提供 XX 数据或 XX 链路能力'的需求翻译。"**

## 1. 先建坐标系：一条样本从生到死的训练流水线

在逐个拆词之前，先把这些名词挂到同一张流水线图上，否则单看每个词都是孤立的。**所有名词都是这条流水线上的一个刻度或一个动作。**

```
  原始日志/特征快照              样本拼接                       训练
 ┌──────────────┐   slot 配置   ┌──────────────┐   batch 切分  ┌────────────────────┐
 │ 曝光/点击/特征 │ ───────────▶ │ 一条 instance │ ───────────▶ │  for each batch:     │
 │   (回流)      │  按 slot 抽取  │ = label + 多个 │  按 batch_size │   forward → loss     │
 └──────────────┘  并编号成 sign │   slot 的特征  │   切成 mini-   │   backward → 更新参数 │ ← 一个 step
                                └──────────────┘   batch        │   global_step += 1   │
                                                                └─────────┬──────────┘
                                                                          │ 攒满"一个时间片"
                                                                          ▼
                            ┌───────────────────────────────────────────────────┐
                            │  过完一个 pass（一份数据 / 一个小时分片）              │
                            │   → 存 checkpoint → 产出模型版本（base 或 +delta）    │
                            └───────────────────────────────────────────────────┘
                                                                          │
                            通用 ML：再过同一份数据 = 下一个 epoch ──────────┘（搜广推常省略）
```

读图要点（先有这张图，后面每节都在放大其中一块）：

- **slot** 作用在「样本拼接」阶段：决定一条样本由哪些"类"的特征拼成、每个特征怎么编号。
- **batch / batch_size** 作用在「训练」阶段：把样本切成一小撮一小撮喂给模型。
- **step / iteration** = 处理完一个 batch、参数更新一次。
- **pass / epoch** = 把"一份数据"整个过一遍这个宏观刻度。
- **checkpoint / 模型版本** = 过完一个 pass 后落盘、对外发布。

> 🎓 **老师视角（讲这段能立刻显出层次）**：很多平台研发把这些词当"算法黑话"被动记忆，记不住也接不住。**正确的姿势是把它们全挂到这张流水线上**——slot 在拼接环节、batch 在喂数环节、pass/epoch 在宏观调度环节、checkpoint 在产出环节。评审现场你能指着流水线说"你说的 pass 是这一格，对应我这边的增量回流时效要求"，瞬间就从"记名词"升级到"懂链路"。

### 1.1 名词速查总表（先扫一眼，建立全局印象）

下表把全篇要讲的词压成一页。**每行最右列就是这个词对你——平台研发——的真实含义**，这是全篇的灵魂：

| 名词 | 一句话含义 | 所在阶段 | 对平台研发意味着 |
|---|---|---|---|
| **slot** | 一类特征（field/特征域） | 拼接 | 配置中心、参数分片单元、加 slot=全链路工程 |

| **sign** | feature 哈希后的唯一整数 key | 拼接 | 离在线哈希一致（红线）、冲突率监控 |
| **embedding** | sign 对应的可学习向量 | 训练/存储 | 参数存储、分片、checkpoint 体积 |
| **instance/sample** | 一条样本=label+多 slot 特征 | 拼接 | 拼接引擎吞吐、格式标准、schema 演进 |

| **base / delta** | 全量底座 / 增量更新 | 流式训练 | 跑批+流式协同、版本拼装、防漂移 |
| **warmup** | 训练初期 lr 从小爬升 | 训练 | 监控初期稳定性、配合大 batch |

## 2. slot：搜广推特征的"槽位/特征域"

### 2.1 词源：slot = 特征槽位 / 特征域（field）

`slot`，字面"槽位"。在搜广推语境里，它指**一类特征（feature field / feature group）**。直觉模型：你有一排带编号的槽，每个槽专门放"某一类信息"。

```
  slot 1   slot 2   slot 3   slot 4   slot 5  ...  slot 873
 ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐    ┌────────┐
 │用户ID │ │广告ID │ │用户   │ │广告  │ │近7天 │ ...│用户城市 │
 │      │ │      │ │年龄段 │ │行业  │ │点击数│    │        │
 └──────┘ └──────┘ └──────┘ └──────┘ └──────┘    └────────┘
   ↑        ↑                                          ↑
 一个槽 = 一"类"特征，全模型成百上千个这样的槽
```

**关键认知：slot 是"类"，不是"值"。** "用户ID"是 slot 5、"广告ID"是 slot 7；具体到某条样本里"用户ID=10086"才是这个 slot 取到的**值**。

### 2.2 为什么 LR / 参数服务器时代要把每类特征叫一个 slot

这要回到搜广推早期的主力模型 **LR（逻辑回归）+ 大规模稀疏特征**的工程现实：

```
  线性模型：score = w·x = Σ_i  w_i × x_i
                              ↑      ↑
                          每个特征   特征取值(多为0/1，稀疏)
                          都有自己
                          的权重 w_i
```

- 搜广推特征**极度稀疏、维度极高**：把"用户ID""广告ID"这种 ID 类特征 one-hot 展开，维度轻松上**亿甚至几十亿**。
- 这么多维特征，**没法手写、没法一个个起名字**。于是工程上按"类"组织：把同一类（同一 field）的所有取值归到一个 **slot** 下，给 slot 一个整数编号。
- 训练框架（参数服务器 PS）就按 slot 来**组织参数、分片存储、配置特征处理逻辑**。slot 成了特征工程和参数管理的**基本组织单元**。

> 🎓 **老师视角（反直觉点）**：新人常以为"slot 是深度学习的东西"。**恰恰相反——slot 是 LR/PS 时代为了管住"上亿维稀疏特征"逼出来的组织手段**，深度模型（embedding 时代）只是继承并复用了它。你理解了"slot 是稀疏特征的工程组织单元"，就理解了它为什么至今仍是搜广推平台的核心配置项——它跟模型结构无关，跟"特征怎么编号、参数怎么分片"强相关，而这恰恰是平台的活。

### 2.3 "几百上千个 slot" 是什么概念

这是评审里很容易被问到的"量级感"。给几个**典型量级/经验值**（不是绝对数字，各家差异大）：

| 维度 | 典型量级（经验值） | 说明 |
|---|---|---|
| 一个成熟广告/推荐模型的 slot 数 | 几百 ~ 一两千个 slot | 早期 LR 模型可能几十个，大型深度模型常上千 |
| 单个 slot 下的取值空间（基数） | 几个 ~ 几十亿 | "性别"slot 只有 2~3 个值；"用户ID"slot 可达数亿 |
| 全模型展开后的总特征维度 | 十亿 ~ 千亿级 | 各 slot 基数之和，稀疏存储 |
| 模型参数量（含 embedding） | 几十 GB ~ 数 TB | 主要被高基数 slot 的 embedding 吃掉 |

> 📊 **量级直觉**：你可以把"一个有 1000 个 slot 的模型"理解成"这个模型同时看 1000 类信息来做一次判断"。其中**少数几个高基数 ID 类 slot（用户ID、广告ID、商品ID）吃掉了 90%+ 的参数和存储**——这对平台意味着参数分片、embedding 存储、checkpoint 体积全被这几个大 slot 主导。

### 2.4 slot 配置如何驱动样本与特征拼接

这是**平台研发最该关心的一节**：slot 不是个名词，它是一份**配置（slot conf / feature conf）**，这份配置直接驱动你的样本拼接链路。

```
            ┌─────────────── slot 配置（一份配置文件/元数据）───────────────┐
            │  slot 5  : 来源=user 表.user_id     类型=ID      处理=hash      │
            │  slot 7  : 来源=ad 表.ad_id         类型=ID      处理=hash      │
            │  slot 41 : 来源=快照.clicks_7d      类型=数值     处理=分桶       │
            │  slot 99 : 来源=user×ad 交叉        类型=组合     处理=hash      │
            └───────────────────────────────┬──────────────────────────────┘
                                            │ 驱动
       原始数据(日志/快照/维表)              ▼
       ┌─────────────────┐    按 slot 配置抽取/拼接/编号    ┌───────────────────────┐
       │ user 表 / ad 表  │ ──────────────────────────────▶│ 一条 instance：         │
       │ 特征快照 / 交叉   │                                │ label=1                 │
       └─────────────────┘                                │ slot5:sign_a slot7:sign_b│
                                                           │ slot41:sign_c ...       │
                                                           └───────────────────────┘
```

slot 配置一改，你的拼接链路就跟着变。具体到平台研发，**slot 配置驱动了这几件你要支撑的事**：

1. **取数**：每个 slot 声明"从哪张表/哪个快照字段取"，你要保证这些数据源可达、口径一致（→ 离在线一致性）。
2. **编号/哈希**：ID 类 slot 的原始值要经过 hash 变成 sign（见下节），你要保证**离在线用同一套哈希**，否则同一个"用户ID=10086"离线在线编出两个号，模型直接学废。
3. **拼接**：把多个 slot 的特征按格式拼成一条 instance，你要支撑这套拼接引擎的吞吐（一天百亿条样本）。
4. **增删 slot**：算法加一个 slot = 你要支持"新增特征域"全链路（取数→哈希→拼接→训练 schema 变更→参数空间扩容），且不能影响存量。

> 🎓 **老师视角**：当算法同学说"我加 5 个 slot 试试"，你脑子里要立刻翻译成一串平台动作：**新增 5 个数据源接入、5 套哈希规则离在线对齐、样本 schema 变更、PS 参数空间扩容、存储/带宽涨多少**。能当场说出"加 slot 不是改个配置那么简单，涉及离在线哈希一致和参数扩容"，你在算法同学眼里立刻是"懂行的平台"。

### 2.5 脱敏案例画像：一次"加 slot"引发的事故

> 🛒 **背景（脱敏）**：某信息流推荐模型约 600 个 slot。算法同学想试一个新特征"用户最近浏览的 3 个类目"，编号为 slot 612，让平台加上。
>
> **平台只改了离线**：在样本拼接侧接好了 slot 612 的取数和哈希，离线样本里出现了 slot 612。重训后离线 AUC 涨了 0.003，算法很兴奋，推上线。
>
> **线上翻车**：上线后 CTR 没涨反而微跌。排查三天，最后发现——**在线特征服务还没接 slot 612**，线上打分时这个 slot 恒为空（走默认值），而离线样本里它是有真实值的。模型重度依赖了一个"线上根本拿不到"的特征。
>
> **根因**：加 slot 是"离线 + 在线 + 哈希一致 + 上线门禁"四件套，这次只做了离线一件。
>
> **平台该有的拦截**：新特征上线前的**一致性门禁**——离线样本里出现的每个 slot，必须在线上也能取到且哈希一致，否则不允许进训练。这一条门禁能把这次三天事故消灭在 Day 0。

> 📊 **量级直觉**：这类"离在线 slot 不对齐"事故在没有门禁的平台上**反复发生**，且每次定位都要 1~3 天，因为离线指标完全正常、所有人第一反应是查模型。有门禁 + 一致性看板的平台能把它降到分钟级发现。

## 3. slot 与 feature / sign / embedding 的关系

这四个词天天混着说，必须一次理清。它们是**从"类"到"值"再到"参数"的一条链**：

```
   slot（特征域/类）         feature（具体特征值）        sign（特征签名/编号）        embedding（向量参数）
 ┌────────────────┐       ┌─────────────────┐        ┌──────────────────┐       ┌──────────────────┐
 │ slot 5 = 用户ID  │ ──取值─▶│ user_id = 10086  │ ──hash─▶│ sign = 0x9af3...  │ ──查表─▶│ [0.12,-0.3,...,0.8]│
 │（一类信息）       │       │（这条样本的实际值）  │        │（64位整数指纹/key） │       │（这个值对应的向量） │
 └────────────────┘       └─────────────────┘        └──────────────────┘       └──────────────────┘
        类                       值                        值的唯一编号                值的可学习参数
```

逐层解释：

| 概念 | 是什么 | 类比 | 平台关心点 |
|---|---|---|---|
| **slot** | 一类特征（field） | "身份证号"这个**字段** | 配置、分片、拼接的组织单元 |
| **feature** | slot 在某条样本里取到的具体值 | 某人身份证号"110..." | 取数口径、缺失值 |
| **sign（特征签名）** | feature 经哈希得到的唯一整数 key（常见 64bit） | 身份证号的指纹哈希 | **离在线哈希必须一致**；哈希冲突率 |
| **embedding** | sign 对应的可学习向量参数 | 给这个指纹分配的"画像向量" | 参数存储、分片、checkpoint 体积 |

### 3.1 为什么要 sign（特征签名）这一步

原始特征值五花八门（字符串、长 ID、交叉特征），没法直接当数组下标。工程上统一做法：

```
   原始特征 "用户ID=10086" + slot编号
            │  拼接 + 哈希（如 MurmurHash / CityHash → 64bit）
            ▼
        sign = 14820374019283...    ← 全局唯一（近似）整数 key
            │  作为参数服务器里的 key
            ▼
        查到 embedding 向量 / LR 权重
```

- **带上 slot 编号一起哈希**：这样"slot 5 的值 10086"和"slot 6 的值 10086"会编成不同的 sign——同一个数字在不同特征域里含义不同，必须区分开。
- **哈希冲突**：64bit 空间下冲突概率极低但非零，两个不同特征撞到同一个 sign 会共享参数。平台要监控冲突率（经验上控制在可忽略级别）。

> 🎓 **老师视角（最容易翻车、最能显专业的一点）**：**离线生成 sign 的哈希函数，必须和在线一字不差。** 我踩过的坑：离线用了带 seed 的哈希、在线没带，或者一边大端一边小端，结果同一个"用户ID=10086"离线编成 `A`、在线编成 `B`，模型在 `A` 上学了半天，线上喂的是 `B` 的参数（基本是随机初始化值）→ 离线 AUC 正常、上线效果稀烂，还极难查。**评审里你能主动说"sign 哈希离在线一致性是我必盯的红线"，立刻和"只会喊上 Feature Store"的人拉开差距。**（详见 ../04-样本数据管理/样本离在线一致性.md）

### 3.2 LR 时代 vs 深度时代：sign 后面挂什么

```
   LR 时代：     sign ──▶ 一个标量权重 w（一维）
   深度时代：     sign ──▶ 一个 embedding 向量（如 8/16/32 维）──▶ 喂进 DNN
```

- LR 时代每个 sign 对应**一个权重标量**，参数总量 ≈ 总特征数。
- 深度时代每个 sign 对应**一个 embedding 向量**（维度 d，典型 8~64），参数量 ≈ 总特征数 × d，直接膨胀几十倍——这就是为什么深度搜广推模型动辄 TB 级、为什么参数服务器（PS）成了刚需。

> 📊 **量级感**：假设某 ID 类 slot 有 2 亿个不同取值，embedding 维度 16，每个 float 4 字节 → 单这一个 slot 的 embedding 就要 `2e8 × 16 × 4 ≈ 12.8 GB`。一个模型里好几个这种大 slot，参数到几百 GB~TB 级就很自然了。**这直接决定你的参数存储方案、checkpoint 体积、分布式分片策略。**

### 3.3 稀疏参数怎么存：定长数组 vs 动态哈希表

sign → embedding 这一步在工程上有两种存储路线，**直接决定参数能不能无限增长、平台怎么扩容**：

```
  路线 A：定长数组（固定哈希空间）           路线 B：动态哈希表（按需开辟）
 ┌────────────────────────┐             ┌────────────────────────┐
 │ sign mod N → 数组下标     │             │ HashMap<sign, embedding>│
 │ 预分配 N 个 embedding 槽   │             │ 来一个新 sign 就 new 一个 │
 │ N 固定（如 2^28）         │             │ 容量随真实特征数增长       │
 └────────────────────────┘             └────────────────────────┘
   优点：实现简单、内存可预估             优点：不浪费、无哈希冲突
   缺点：N 设小→冲突多，N 设大→浪费      缺点：实现复杂、要做准入/淘汰防爆
```

- **路线 A（定长/哈希取模）**：早期/资源受限场景常见。`sign mod N` 落到固定大小的参数数组里。N 设太小，多个不同特征撞同一个槽（**哈希冲突**）共享参数，掉精度；N 设太大，大量槽空着浪费内存。
- **路线 B（动态哈希表）**：现代大规模稀疏训练框架（PS）主流。每个真实出现过的 sign 才分配一份参数，**理论上无冲突**，但参数量会随长尾特征无限膨胀，必须配"准入 + 淘汰"。

### 3.4 特征准入（admission）与淘汰（eviction）：防参数爆炸

动态哈希表路线下，长尾 sign（只出现一两次的特征）会把参数表撑爆，所以要做**准入和淘汰**——这是搜广推 PS 的标配机制，也是平台研发要支撑的硬能力：

| 机制 | 做什么 | 常见策略（经验） | 解决什么 |
|---|---|---|---|
| **特征准入（admission）** | 决定一个新 sign 要不要进参数表 | 出现频次 ≥ 阈值才准入；或概率准入（counting bloom filter 计数） | 挡掉只出现一两次的长尾噪声特征，防表膨胀 |
| **特征淘汰（eviction）** | 把"过期/不再活跃"的 sign 踢出参数表 | 一段时间未被更新（如 N 天没出现）就淘汰；或按重要性/L2 范数淘汰 | 回收过期特征（下线的广告/商品 ID）占用的内存 |

```
  新 sign 来了
      │  ┌─ 出现次数 < 阈值 ─▶ 不准入，本次当默认值处理（不建参数）
      ▼  │
  准入判定 ┤
         └─ 出现次数 ≥ 阈值 ─▶ 准入，分配 embedding，开始学习
                                      │
   定期扫描 ──▶ 某 sign 已 N 天没更新 ──▶ 淘汰，回收内存
```

> 🎓 **老师视角（这点能直接区分"懂搜广推平台"和"只懂通用训练"）**：通用深度学习（CV/NLP）里没有"特征准入/淘汰"这回事——它们的参数是固定结构。**唯独大规模稀疏搜广推，参数表是动态生长的，不做准入会被长尾特征撑爆内存，不做淘汰会被下线的 ID 永久占用资源。** 你在评审里能讲出"准入挡噪声、淘汰回内存，这俩是稀疏 PS 区别于普通训练的关键机制"，立刻显出你懂的是搜广推这套特有工程，而不是泛泛的炼丹。补一刀：**准入/淘汰的阈值是效果与成本的旋钮**——准入卡太严会丢掉有用的长尾特征（掉点），卡太松会浪费内存，这又是一个典型权衡。

> 📊 **量级感**：做了准入后，参数表里实际存活的 sign 数常比"理论出现过的不同特征数"小一个数量级（大量长尾被挡）。这对 checkpoint 体积、PS 内存、恢复速度都是数量级的节省。

## 4. pass：通用 ML 与搜广推工业语境的"一词两义"

`pass` 是这篇里**最容易把平台研发绕晕**的词，因为它在两个语境下含义完全不同。

### 4.1 含义 A：通用 ML 里 pass ≈ epoch

在教科书和通用深度学习语境里：

> **a pass over the data = 把训练集完整过一遍 = 一个 epoch。**

"single-pass algorithm"就是"只过一遍数据的算法"。这个含义下 `pass` 和 `epoch` 基本等价，描述的是"数据被完整遍历的次数"。

### 4.2 含义 B：搜广推流式训练里，一个 pass = 一个时间片的增量数据

在搜广推、尤其是**流式/在线训练**（百度 abacus/凤巢这类语境）里，`pass` 被赋予了一个**工业化的、和时间强绑定的**含义：

> **一个 pass = 一个时间片（如一小时 / 一个数据分片）的增量样本。**

```
  时间轴 ───────────────────────────────────────────────────────▶
        │  pass 1  │  pass 2  │  pass 3  │  pass 4  │ ...
        │ 10:00~   │ 11:00~   │ 12:00~   │ 13:00~   │
        │ 11:00 的 │ 12:00 的 │ 13:00 的 │ 14:00 的 │
        │ 增量样本  │ 增量样本  │ 增量样本  │ 增量样本  │
        └────┬─────┘────┬─────┘────┬─────┘────┬─────┘
             │过这一片    │过这一片    │过这一片    │过这一片
             ▼          ▼          ▼          ▼
          模型 v1     模型 v2     模型 v3     模型 v4   ← 每个 pass 产出一个模型版本
        （base+delta1）(+delta2)  (+delta3)  (+delta4)
```

**核心区别**：
- 含义 A 的 pass 是"把**同一份固定数据**过第几遍"。
- 含义 B 的 pass 是"把**时间轴上新到的一片增量数据**过一遍"，每片数据**只过一次**，过完就产一个模型版本，然后等下一片。

### 4.3 两义对照表（务必背下来）

| 维度 | 含义 A：pass ≈ epoch（通用 ML） | 含义 B：pass = 时间片增量（搜广推流式） |
|---|---|---|
| 数据是否固定 | 固定训练集，反复过 | 时间轴上不断新增，过完即弃 |
| 一个 pass 处理什么 | 整个训练集一遍 | 一个时间窗口（如 1 小时）的新样本 |
| 同一份数据过几遍 | 多遍（多 epoch） | 通常一遍（one-pass） |
| 产出 | 训练若干 epoch 后产一个模型 | 每个 pass 产一个模型版本 |
| 时间属性 | 弱（离线批） | 强（和 wall-clock 时间对齐） |
| 平台关注 | 总训练时长、收敛 | 增量回流时效、版本节奏、base+delta |

> 🎓 **老师视角（接话神技）**：听到算法说"pass"，**第一反应不要急着答，先在心里判断语境**：如果在聊"流式/在线训练/每小时出模型"，他说的几乎一定是含义 B；如果在聊"离线重训/跑几轮"，那是含义 A。拿不准就直接问一句："你说的 pass 是'一小时增量那个 pass'，还是'过几遍数据那个 epoch'？"——这一问，算法同学立刻知道你是懂的，而不是在装懂。

## 5. pass 的工业含义：base + delta 与按 pass 产出模型版本

把含义 B 讲透，因为它是搜广推流式训练的核心机制，也是**对平台链路要求最高**的一块。

### 5.1 base + delta：全量底座 + 增量更新

流式训练不会每个 pass 都从头训，而是**在上一版模型基础上增量更新**：

```
  ┌──────────┐   pass1   ┌──────────┐   pass2   ┌──────────┐
  │  base    │ ────────▶ │ base     │ ────────▶ │ base     │
  │ (全量底座) │  +delta1  │ +delta1  │  +delta2  │ +delta1  │  ...
  │          │           │          │           │ +delta2  │
  └──────────┘           └──────────┘           └──────────┘
   每天/每周               每小时一个 delta（一个 pass 产出）
   重训一次 base
```

- **base（基线/全量模型）**：周期性（如每天凌晨）用较长窗口数据全量重训，作为"底座"。纠正增量累积的偏差、清理过期特征。
- **delta（增量）**：每个 pass（如每小时）在当前模型上用新增样本增量训练，产出一个增量更新。
- **线上模型** = base + 截至当前的所有 delta。

> 📊 **量级感（典型经验值，各家不同）**：
> - delta（pass）节奏：**分钟级 ~ 小时级**（凤巢类广告系统常做到分钟级/小时级增量）。
> - base 重训节奏：**天级 ~ 周级**。
> - 模型上线延迟：从"用户产生行为"到"该行为影响线上模型"，成熟系统可压到**分钟~小时级**。

### 5.2 按 pass 产出模型版本：版本号即时间片

每个 pass 训完，落一个**带版本号的模型快照**，版本号往往直接编码时间片：

```
  model/
   ├── base_20260614/                ← 当天 base
   ├── delta_20260614_10/            ← pass: 10点片
   ├── delta_20260614_11/            ← pass: 11点片
   └── delta_20260614_12/  ← 当前线上 = base_20260614 + delta_10 + delta_11 + delta_12
```

这套机制对平台研发意味着一组**硬能力要求**：

1. **增量样本回流时效**：pass 是小时级，样本就必须小时级（甚至分钟级）回流并拼好 label。回流慢一拍，整个 pass 节奏就崩（→ ../04-样本数据管理/样本离在线一致性.md 的延迟反馈问题）。
2. **模型版本管理**：每个 pass 一个版本，一天几十上百个版本，要支持版本登记、回滚、清理、灰度。
3. **热更新/热加载**：线上要能不停服地加载新 delta，平台得提供模型热更新通道。
4. **断点续训**：pass 是连续的，必须从上一个 pass 的 checkpoint 续训，不能从头来（见第 9 节 checkpoint）。
5. **坏 pass 隔离**：某个时间片样本异常（如埋点故障），要能识别并跳过/回滚该 pass，否则坏数据被增量学进模型且难撤销。

> 🎓 **老师视角（反直觉，评审高分点）**：流式增量训练有个隐患——**delta 一直累加，偏差会漂移**（catastrophic forgetting、近期样本主导）。所以才需要**周期性 base 全量重训来"校正"**。很多人只知道"增量出模型快"，不知道"base 重训是给增量擦屁股的"。你能讲出"base + delta 是速度和稳定性的折中：delta 保时效，base 防漂移"，就讲到了机制设计的本质，而不是停在名词复述。

### 5.3 一个流式 pass 的时序（平台视角）

```
  T=10:00 ─────────────────────────────────────────────────────▶ T=11:00
   │ 用户在 10:00~11:00 产生曝光/点击/转化
   │     │ 行为埋点 + 特征快照 旁路落盘（秒级）
   │     ▼
   │  ┌─────────────┐   等 label（归因窗口）   ┌──────────────┐
   │  │ 在线特征日志   │ ─────────────────────▶ │ pass 样本就绪   │
   │  └─────────────┘                         └──────┬───────┘
   │                                                 │ T≈11:0x 触发训练
   │                                                 ▼
   │                                  从上一 pass checkpoint 续训
   │                                  过这一片样本（one-pass）→ 产出 delta
   │                                                 │
   │                                                 ▼
   │                                  写 checkpoint + 登记模型版本 + 推送热更新
   └─────────────────────────────────────────────── ▼ 线上加载 delta_10
```

> 📌 注意 label 的延迟：点击可能几秒到几分钟，转化可能几小时到几天。所以"10 点片的样本"往往不能 10:01 就训，要等一个**归因窗口**，这是流式训练时效的天然下限。

### 5.4 pass 节奏的两难：等 label 还是先训？（延迟反馈）

这是流式 pass 设计里**最现实的工程矛盾**，平台研发必然撞上。一个 pass 要不要等所有 label 都回来再训？两难：

```
   等满归因窗口再训                     不等，先用"目前已知"的 label 训
 ┌──────────────────────┐           ┌──────────────────────────┐
 │ 时效差：转化要等几小时~几天 │           │ 时效好：行为一来就能训        │
 │ → pass 严重滞后于线上     │           │ → 模型几乎实时跟上            │
 │ 标签准：label 完整        │           │ 标签偏：晚到的正样本还没回来    │
 └──────────────────────┘           │   → 把它当负样本，正样本被低估   │
                                     └──────────────────────────┘
```

工业界常见折中（这些方案名你能说出来就很专业）：

| 方案 | 做法 | 代价 |
|---|---|---|
| **短窗口截断** | 只等一个较短归因窗口（如点击等几分钟），过期未到就定 label | 长尾转化丢失，CVR 类偏差 |
| **延迟反馈建模（delayed feedback model）** | 先按当前 label 训，晚到的正样本到了再**纠正/回填**（做 importance weighting 或重训该样本） | 实现复杂，要支持"label 回填重算" |
| **正样本补回 / fake negative 校正** | 先把未转化样本当负样本，转化真到了再补一条正样本/翻转 label | 样本流要支持"追加修正" |

> 🎓 **老师视角**：这一节是"pass 时效"和"样本正确性"打架的典型现场。**平台研发要提供的不是某个固定答案，而是"可配的归因窗口 + 支持 label 回填/修正的样本流"能力**，让算法去权衡。能讲出"延迟反馈是流式 pass 绕不开的两难，我提供可配窗口和回填能力让算法做取舍"，比只说"我们做流式训练"深一个层次。详见 ../04-样本数据管理/样本离在线一致性.md 的延迟反馈部分。

## 6. batch / mini-batch / batch_size

### 6.1 含义

- **batch**：一次喂给模型做"前向计算 + 反向更新"的一组样本。
- **mini-batch**：相对"全量梯度下降（一次用全部样本）"和"随机梯度下降（一次一条）"而言，**一次用一小撮（几十~几万条）** 的折中做法。现代训练几乎全是 mini-batch，所以日常说的 batch 就是 mini-batch。
- **batch_size**：一个 batch 里的样本条数。

```
  全量数据 = 一份很大的样本集
  ┌────────────────────────────────────────────────────┐
  │ ▢▢▢▢ ▢▢▢▢ ▢▢▢▢ ▢▢▢▢ ▢▢▢▢ ▢▢▢▢ ▢▢▢▢ ▢▢▢▢ ... │
  └─┬────┘└─┬──┘└─┬──┘└─┬──┘
    batch1  batch2  batch3 ...   ← 每个 batch 含 batch_size 条样本
    │
    └─▶ forward → 算 loss → backward → 更新一次参数 = 一个 step
```

### 6.2 batch_size 对显存/内存、收敛、吞吐的影响

这是算法和平台都关心的"调参旋钮"，三个维度的影响要分清：

| 增大 batch_size | 影响 | 方向 |
|---|---|---|
| 显存/内存占用 | 一个 batch 的中间激活、梯度都要驻留 | **线性上升** ⚠️ 容易 OOM |
| 梯度估计 | 平均更多样本，梯度更稳、噪声更小 | 收敛更平滑，但可能陷入"太稳"而泛化变差 |
| 单 step 信息量 | 一步看更多样本，需要的 step 数变少 | step 数↓，但每 step 更慢 |
| 吞吐（样本/秒） | 大 batch 更能打满 GPU/CPU 并行 | **吞吐↑**（到瓶颈前） |
| 学习率配合 | 大 batch 通常要配合调大 lr + warmup | 否则收敛变慢/变差 |

> 🎓 **老师视角（反直觉）**：新人以为"batch_size 越大越好（吞吐高嘛）"。**真相是有上限**：① 受显存/内存物理约束（超了直接 OOM）；② batch 太大梯度噪声太小，反而可能**泛化变差**（著名的"large-batch generalization gap"）；③ 大 batch 必须配套调大学习率并加 warmup，否则前期不稳。平台研发要知道这点，因为算法常会找你"把 batch_size 翻倍提吞吐"，你得能反问一句"显存够吗？lr 和 warmup 跟着调了吗？"——这就显出你不是只会改配置。

### 6.3 搜广推的"大 batch"训练

搜广推的 batch_size **量级远大于 CV/NLP**：

| 领域 | batch_size 典型量级（经验值） | 原因 |
|---|---|---|
| CV（图像） | 几十 ~ 几百 | 单样本大（图像张量重），显存吃紧 |
| NLP（大模型预训练） | 几十万 ~ 几百万 token（按 token 算） | 序列长 |
| **搜广推** | **几千 ~ 几万条 instance** | 单样本是稀疏特征（轻），靠大 batch 打满吞吐、稳定稀疏梯度 |

- 搜广推单条样本是一堆稀疏 sign，**单样本计算很轻**，所以能塞很大的 batch。
- 大 batch 对**稀疏特征的梯度更新更稳定**（很多 sign 在一个 batch 里才出现几次，batch 大才有足够统计量）。
- 大 batch + 大规模分布式（数据并行）是搜广推训练吞吐的标配。

> 📊 **平台含义**：batch_size 直接决定**单 worker 的内存占用**和**参数服务器的通信粒度（每个 batch 要 pull/push 一批 sign 的参数）**。算法调大 batch，你要关注 worker 内存、PS 通信带宽、以及 batch 内去重（同一 batch 里重复 sign 只拉一次参数）是否生效。

### 6.4 一个 batch 在分布式下到底发生了什么（PS pull/push 时序）

平台研发必须理解一个 batch 在参数服务器架构下的完整往返，因为**这条往返就是搜广推训练的性能瓶颈所在**：

```
  Worker（算梯度）                              PS（存参数，按 sign 分片到多机）
   │ 1. 拿到一个 batch（如 1 万条 instance）
   │ 2. 抽出 batch 里所有 sign，去重         ← batch 内去重：1万条样本可能引用
   │    （比如去重后 30 万个唯一 sign）            几十万 sign，重复的只拉一次
   │ 3. 按 sign 路由，向各 PS 分片 ──pull──▶  查出这 30 万 sign 的 embedding
   │ 4. ◀──── 收到 embedding ────────────
   │ 5. forward → loss → backward 算梯度
   │ 6. 把 30 万 sign 的梯度 ──push──▶        PS 用梯度更新对应 embedding（应用 lr/优化器）
   │ 7. global_step += 1，进入下一个 batch
```

要点（评审会追问的）：

- **batch 内去重**：一个 batch 里同一个 sign（如热门广告ID）出现成千上万次，**去重后只 pull/push 一次**，否则通信量爆炸。去重比常很高（热门特征反复出现），这是关键优化。
- **pull/push 是网络往返**：每个 batch 都要和 PS 来回一次。batch 越大、唯一 sign 越多，通信量越大。**搜广推训练常是"通信瓶颈"而非"计算瓶颈"**，和 CV/NLP（计算瓶颈）相反。
- **稠密参数 vs 稀疏参数**：DNN 部分的稠密参数（MLP 权重）量小但全量同步；embedding 部分的稀疏参数量大但每 batch 只动一小撮。两者通信模式不同，平台常分开处理。

> 🎓 **老师视角（区分内行的一刀）**：CV/NLP 训练优化重点在"算得快"（GPU 利用率、算子融合）；**搜广推训练优化重点在"传得少传得快"（batch 内去重、通信压缩、参数分片均衡）**。你能讲出"搜广推是通信密集型，瓶颈在 PS pull/push 而非 GPU"，就抓住了搜广推训练性能的命门——很多只做过通用训练的人会答错这一点。

## 7. epoch / step / iteration：换算关系与 one-pass

### 7.1 三者定义与换算

```
  ┌──────────────────────────────────────────────────────────────┐
  │  step（= iteration）：处理完"一个 batch"、参数更新一次          │
  │  epoch：把"整份训练数据"完整过一遍                              │
  │                                                                │
  │  一个 epoch 的 step 数 =  样本总数 / batch_size                 │
  │  总 step 数            =  epoch 数 × (样本总数 / batch_size)     │
  │  global_step          =  从训练开始累计到现在的 step 总数        │
  └──────────────────────────────────────────────────────────────┘
```

举个具体例子（便于建立直觉）：

> 假设训练集 **1 亿条样本**，`batch_size = 10000`：
> - 一个 epoch 的 step 数 = `1e8 / 1e4 = 10000` 个 step。
> - 跑 3 个 epoch，总 step = `3 × 10000 = 30000`，`global_step` 最终到 30000。
> - 若是 **one-pass（1 个 epoch）**，就只有 10000 个 step。

| 术语 | 中文 | 粒度 | 关系 |
|---|---|---|---|
| iteration / step | 迭代/步 | 一个 batch | 最小单位 |
| epoch | 轮 | 一遍数据 | = (样本数/batch_size) 个 step |
| pass（含义A） | 遍 | 一遍数据 | ≈ epoch |
| global_step | 全局步 | 累计 | 跨 epoch 累加，不清零 |

> ⚠️ **iteration 和 step 是同义词**，都指"处理一个 batch"。但偶尔有人把"iteration"当"epoch"用，听到要确认。**step 几乎总是指 batch 级**，比较安全。

### 7.2 为什么搜广推常说"流式 one-pass、不跑多 epoch"

"one-pass"= 每条样本**只被训练看到一次**（1 个 epoch）。这是搜广推和 CV/NLP 的一个**显著差异**，放到第 8 节专门讲透为什么。这里先记结论：

```
  CV/NLP 离线训练：       样本 ▢ 被看 epoch1, epoch2, epoch3 ...（多遍，刷分）
  搜广推流式训练：         样本 ▢ 只被看 1 次 → 用完即弃（one-pass）
```

> 🎓 **老师视角**：能把"step / epoch / pass"的换算脱口而出（step=样本数/batch_size），是面试和评审里典型的"基本功信号"。但更高一层的信号是：你知道**搜广推根本不在 epoch 这个维度上反复刷**，而是在"pass=时间片"维度上滚动——这说明你理解的是工业训练范式，不是教科书范式。

## 8. 为什么搜广推"流式 one-pass、不跑多 epoch"

这是用户点名最想搞懂的反直觉点，单列一节讲透。三条原因，缺一不可：

### 8.1 原因一：数据量大到跑多遍不划算

```
  CV/NLP：    几十万~几百万样本 → 跑 10~100 epoch 才够，反复刷有性价比
  搜广推：     一天就 百亿级 曝光样本 → 跑一遍已经吃掉巨量算力
              再跑第二遍 epoch = 算力翻倍，边际收益却很小
```

- 搜广推**永远不缺数据**（每天新增百亿级），瓶颈是算力和时效，不是样本量。
- 既然样本管够，与其把旧数据刷第二遍，不如把算力花在**消化新到的数据**上。

### 8.2 原因二：时效性要求高——模型必须追上最新分布

```
  用户兴趣、热点、广告库存 时刻在变（分布漂移 / concept drift）
        │
  搜广推要求模型尽快学到"刚刚发生的事"
        │
  如果花时间反复跑旧数据的 epoch → 模型更新被拖慢 → 落后于线上真实分布 → 掉点
```

- 搜广推数据有**强时效性**：今天的爆款明天就凉，模型必须**快速跟上**。
- 反复跑 epoch 会**拖慢模型上线节奏**，与"尽快追上最新分布"直接冲突。one-pass + 流式增量才是对的范式。

### 8.3 原因三：样本只看一次，天然抗过拟合

```
  多 epoch（看多遍）：      模型反复见同一批样本 → 容易"背答案"→ 过拟合
  one-pass（看一遍）：      每条样本只见一次 → 模型学的是"普遍规律"而非"记住个例"
                          相当于一种天然的正则化
```

- 在海量数据下，**每条样本只看一次本身就是一种强正则**——模型没机会去"记住"个别样本，被迫去学可泛化的模式。
- 这也是为什么搜广推 one-pass 不仅不掉效果，反而往往**泛化更好**。

### 8.4 三因素汇总

| 原因 | 一句话 | 与 epoch 的冲突 |
|---|---|---|
| 数据量大 | 数据管够，刷旧的不划算 | 多 epoch = 算力浪费 |
| 时效性高 | 模型要追最新分布 | 多 epoch = 拖慢更新 |
| 抗过拟合 | 只看一次=天然正则 | 多 epoch = 反而易过拟合 |

> 🎓 **老师视角（这段讲好直接封神）**：很多平台研发以为"搜广推 one-pass 是因为穷/省算力"。**其实算力只是表层，深层是三因素共振：数据无限 + 时效苛刻 + 看一次更抗过拟合，三者刚好都指向 one-pass。** 你能把这三条串起来讲，老板和算法都会觉得"这人是真懂搜广推训练范式，不是背名词"。补一刀更显认知：**正因为 one-pass，样本质量和离在线一致性的容错极低——模型只看一次，喂错了就错了，没有"下一个 epoch 纠正"的机会**，这就把"样本平台必须做对"的重要性拔到了战略高度。

## 9. 其他高频名词：warmup / lr schedule / global step / checkpoint / shuffle / sample

### 9.1 learning rate（学习率）与 lr schedule（学习率调度）

- **learning rate（lr，学习率）**：每次参数更新迈多大步。太大震荡甚至发散，太小收敛慢。
- **lr schedule（学习率调度）**：lr 不固定，随训练进程按规则变化。常见：

```
  lr
   │      ┌── 训练中后期逐步衰减（decay）：稳住、收敛
   │     /│╲
   │    / │ ╲____
   │   /  │      ╲────____
   │  /   │              ╲────  ← 余弦/阶梯/指数衰减
   │ /    │
   └─/────┴────────────────────▶ step
     warmup
   （前期从小到大爬升，防止一开始就被大梯度冲飞）
```

### 9.2 warmup（预热）

训练**最开始**用很小的 lr，再逐步升到目标 lr。

- **为什么**：训练初期参数是随机初始化的，梯度方向很不靠谱。一上来就用大 lr 容易把模型"冲坏"（loss 飞了 / NaN）。warmup 让模型先"热身"稳定下来。
- **和大 batch 强相关**：batch 越大越需要 warmup（前面第 6 节提过），因为大 batch + 大 lr 初期最容易炸。

### 9.3 global step（全局步）

- 从训练**最开始**累计到现在的 step 总数，**不随 epoch 清零**。
- 用途：lr schedule 按 global step 算、日志/监控按 global step 打点、checkpoint 按 global step 命名。
- 流式训练里 global step 可以非常大（持续累加），是判断"训练进行到哪了"的全局时钟。

### 9.4 checkpoint（检查点）

训练过程中**周期性把模型参数（+优化器状态）落盘**，是搜广推训练的命脉之一。

```
  训练中 ── 每 N step / 每个 pass ──▶ 写 checkpoint（参数 + optimizer 状态 + global_step）
                                          │
        机器挂了 / 想发布模型 ◀───────────────┘
            └─▶ 从最近 checkpoint 恢复，断点续训（不用从头）
```

- **作用**：① 容错（机器/任务挂了从这恢复）；② 续训（流式训练每个 pass 从上个 checkpoint 续）；③ 发布（checkpoint 转成线上可加载的模型版本）。
- **搜广推的特殊性**：参数量 TB 级，checkpoint **又大又慢**。要做增量 checkpoint、分片并行写、异步落盘，否则写 checkpoint 把训练卡住。

> 📊 **量级感**：TB 级模型一次全量 checkpoint 落盘可能要**几分钟到几十分钟**，期间若阻塞训练就是大问题。所以平台要支持**异步/增量 checkpoint**（只存变化的参数分片）。

**全量 checkpoint vs 增量 checkpoint（搜广推必须搞清的区别）**：

```
  全量 checkpoint：每次把整张 TB 级参数表全写一遍 → 又慢又占存储
  增量 checkpoint：只写"上次 checkpoint 以来被更新过的 sign 的参数" → 快、省
                   （搜广推每个 pass 只动一小撮 sign，增量优势巨大）
```

- 搜广推一个 pass 通常只更新参数表里**很小一部分** sign，所以增量 checkpoint 收益极大——只落变化的那部分。
- 但增量链要能**正确恢复**：恢复时需要"某个全量基点 + 之后所有增量"，平台要管好这条链不断、不丢、可回放。
- **checkpoint 一致性**：分布式下 worker 和多台 PS 要在"同一个 global_step"打一致的 checkpoint（类似分布式快照/barrier），否则恢复出来参数版本对不齐。这是平台要保证的强一致点。

> 🎓 **老师视角**：很多人以为 checkpoint 就是"存个文件"。在 TB 级稀疏模型上，**checkpoint 本身就是一个分布式一致性问题 + 存储成本问题**：要跨多台 PS 在同一 step 打齐快照（一致性）、要增量只存变化（成本）、要支持快速恢复续训（容错）。能讲到"checkpoint 是分布式快照而非存文件"，是很硬的平台认知信号。

### 9.5 shuffle（打散）

把样本顺序打乱再喂给模型。

- **为什么**：如果样本按某种顺序排列（如按时间、按用户聚堆），一个 batch 里全是相似样本，梯度有偏，收敛差。shuffle 让每个 batch 尽量是"独立同分布"的随机抽样。
- **流式训练的矛盾**：流式天然是按时间到达的，**没法全局 shuffle**（未来样本还没到）。常用**局部 shuffle**（在一个滑动 buffer 内打散）来缓解。

> 🎓 **老师视角**：shuffle 在流式场景是个被低估的坑。流式 one-pass + 不能全局 shuffle，意味着**样本的时间局部相关性会带偏梯度**。平台能提供"shuffle buffer / 局部打散"能力，是流式训练质量的隐形保障。能在评审里提这点，说明你想到了"流式范式下连 shuffle 都不一样"。

### 9.6 sample / instance（样本/实例）

- **sample / instance**：一条训练样本，= **一个 label + 该次预测用到的所有 slot 的特征**。前面流水线图里那条 instance 就是它。
- 一条 instance 就是模型的一个学习单位；batch 由若干 instance 组成；一份数据由海量 instance 组成。
- 注意区分 sample（样本/实例）和 "sampling（采样）"——后者是负采样那套，别混（→ 负采样相关文档）。

### 9.7 这一组名词的"平台翻译"速查

| 名词 | 平台要支撑什么 |
|---|---|
| lr schedule / warmup | 训练框架透出调度配置；监控 loss 是否在 warmup 期稳定 |
| global step | 训练监控/日志/checkpoint 命名的统一时钟 |
| checkpoint | 异步/增量 checkpoint、断点续训、checkpoint→线上模型发布通道 |
| shuffle | 流式 shuffle buffer / 离线打散能力 |
| sample / instance | 样本拼接引擎吞吐、instance 格式标准、schema 管理 |

## 10. 两种训练形态：流式（streaming）vs 跑批（batch）

把前面所有名词收口到两种训练形态——这是平台架构选型的根。

```
   ┌────────────────────── 跑批 / 离线训练（batch）──────────────────────┐
   │ 样本：数仓里固定的一份历史数据（如最近 30 天）                          │
   │ 节奏：天级/周级触发一次，跑完几个 epoch 产一个模型                       │
   │ 名词：epoch 有意义、可多遍、可全局 shuffle                             │
   │ 适合：base 全量重训、模型结构实验、冷启动                              │
   └────────────────────────────────────────────────────────────────┘

   ┌────────────────────── 流式 / 在线训练（streaming）──────────────────┐
   │ 样本：时间轴上不断到达的增量样本流                                     │
   │ 节奏：分钟/小时级 pass，每个 pass 增量产一个 delta                      │
   │ 名词：one-pass、pass=时间片、base+delta、局部 shuffle                 │
   │ 适合：线上模型持续更新、追实时分布                                     │
   └────────────────────────────────────────────────────────────────┘
```

### 10.1 对比表

| 维度 | 跑批 / batch | 流式 / streaming |
|---|---|---|
| 数据 | 固定历史集 | 持续增量流 |
| epoch | 有意义，常多遍 | 基本只 one-pass |
| pass 含义 | ≈ epoch（含义A） | 时间片增量（含义B） |
| shuffle | 可全局 | 只能局部（buffer） |
| 产出节奏 | 天/周 | 分钟/小时 |
| 时效 | 低 | 高 |
| 典型用途 | base 重训、实验 | delta 增量、线上更新 |
| 平台核心压力 | 大吞吐批处理、算力调度 | 低延迟回流、版本管理、热更新 |

### 10.2 现实里两者并存

成熟搜广推平台**两种形态同时跑**：

```
   每天/每周  ┌─── 跑批：全量重训 base ────────────────┐
             │                                       ▼
   时间轴 ────┼──pass1──pass2──pass3── ... ──▶  线上模型 = base + Σ delta
             │   ↑流式：每小时增量产 delta，热更新到线上
             └───────────────────────────────────────
```

- **跑批**负责 base（稳、纠偏、可做实验）。
- **流式**负责 delta（快、追分布）。
- 两者拼在一起就是 base + delta 机制——这就是为什么前面 pass 那节和这里能对上。

> 🎓 **老师视角**：平台研发常被问"我们到底该上流式还是跑批"。**正确答案几乎总是'都要，且要让两者协同'**：流式没有 base 兜底会漂移，跑批没有流式追不上时效。你能讲出"base 用跑批、delta 用流式、二者协同"这套组合，而不是二选一，就是架构师视角。

| **字节（推荐系统）** | 公开实践强调实时化推荐、流式样本、模型分钟级更新 | 流式样本, 实时更新 |
| **Google** | TFX/TensorFlow 体系；`feature_column`/特征域、tf.Example 里的 feature 概念；强调 training-serving 一致 | feature column, tf.Example, TFX |
| **Meta** | 大规模 embedding（DLRM）、feature logging 回流做样本、大 batch 数据并行 | DLRM, feature logging, embedding |
| **微软/开源（DeepRec、HugeCTR 等）** | 大规模稀疏训练引擎，embedding 分片、PS、大 batch | sparse training, embedding sharding |

**两条可观察的共性（公开口径归纳）**：

1. **slot/sign + 参数服务器 + 大 batch**，几乎是国内大规模稀疏搜广推训练的事实标准范式。
2. **流式增量（pass/delta）+ 周期 base 重训**，是追时效与保稳定的通用折中。

## 12. 权衡取舍

> 🎓 评审里能讲清"调这个旋钮的代价"才是内行。

| 旋钮 | 调大/启用的收益 | 代价 | 平台要盯 |
|---|---|---|---|
| **batch_size ↑** | 吞吐↑、稀疏梯度更稳 | 显存/内存↑（OOM 风险）、泛化可能变差、需配 lr+warmup | worker 内存、PS 通信、OOM |
| **多 epoch（跑批反复刷）** | 小数据下效果↑ | 算力翻倍、时效↓、过拟合风险↑ | 搜广推一般不划算 |
| **pass 节奏 ↑（更频繁出 delta）** | 模型更追实时分布 | 回流/训练/发布全链路压力↑、坏 pass 影响更快 | 回流时效、版本管理、坏 pass 隔离 |
| **base 重训频率 ↑** | 漂移纠正更及时 | 大额算力、调度复杂 | 算力调度、与流式协同 |
| **更多 slot** | 信息更丰富、可能涨点 | 取数/哈希/拼接/参数/存储全线↑、离在线一致风险↑ | 全链路扩容、哈希一致 |
| **checkpoint 更频繁** | 容错粒度更细、续训更稳 | 落盘 IO/存储↑、可能拖慢训练 | 异步/增量 checkpoint |

**典型取舍结论**：
- batch_size 不是越大越好，受显存和泛化双重约束，**配套 lr/warmup 才能用大 batch**。
- pass 节奏越快越追实时，但**回流和坏 pass 容错跟不上就是灾难**，节奏要和链路能力匹配。
- slot 加得越多越可能涨点，但**每个 slot 都是一条要离在线对齐的链路**，加 slot ≠ 改配置。

## 13. 常见误区与反直觉

- ❌ **"slot 是深度学习的概念"** → 反了。slot 是 LR/参数服务器时代为管住上亿维稀疏特征逼出来的组织单元，深度模型只是继承复用。
- ❌ **"pass 就是 epoch"** → 在搜广推流式语境里，pass = 一个时间片的增量数据，过完产一个模型版本，和 epoch 不是一回事。听到 pass 先确认语境。
- ❌ **"epoch 越多效果越好"** → 搜广推数据无限 + 时效苛刻 + 看一次更抗过拟合，反而 one-pass 是主流，多 epoch 常是浪费甚至有害。
- ❌ **"batch_size 越大越好"** → 受显存约束、可能泛化变差、必须配 lr+warmup，存在最优区间。
- ❌ **"加几个 slot 就是改个配置"** → 加 slot = 新增数据源接入 + 离在线哈希对齐 + 样本 schema 变更 + 参数空间扩容 + 存储带宽上涨，是全链路工程。
- ❌ **"sign 的哈希离在线随便对一下就行"** → 哈希函数/seed/字节序但凡不一致，同一特征离在线编号不同，模型直接学废且极难查（红线问题）。
- 🤯 **反直觉一**：one-pass 不是"为了省钱将就"，而是"数据无限 + 时效 + 抗过拟合"三因素共振下的**更优范式**。
- 🤯 **反直觉二**：流式增量训练里，**base 全量重训不是浪费，而是给 delta 累积偏差"擦屁股"** 的必要环节；只 delta 不 base，模型会漂移。
- 🤯 **反直觉三**：因为 one-pass 样本只看一次，**样本质量/一致性的容错极低**——没有"下个 epoch 纠正"的机会，这让样本平台的重要性高于一般人想象。

## 14. 平台侧落地 Checklist

把全篇翻译成"你的平台要交付/支撑什么"：

- [ ] **slot 配置/元数据中心**：统一管理 slot 编号、来源、类型、处理方式；支持新增/下线 slot 的全链路。
- [ ] **离在线一致的 sign 哈希**：离线与在线使用同一套哈希（函数/seed/字节序对齐），并监控哈希冲突率。（红线，见 ../04-样本数据管理/样本离在线一致性.md）
- [ ] **样本拼接引擎**：按 slot 配置把多源数据拼成标准 instance，支撑百亿级/天吞吐，schema 可演进。
- [ ] **增量样本回流通道**：满足 pass（分钟/小时级）节奏，处理 label 归因窗口与延迟反馈。
- [ ] **参数服务器/embedding 存储**：按 slot/sign 分片、支持 TB 级参数、动态扩容。
- [ ] **base + delta 训练编排**：跑批出 base、流式出 delta、二者协同与版本拼装。
- [ ] **模型版本管理**：每 pass 一个版本，支持登记/灰度/回滚/清理；版本号编码时间片。
- [ ] **checkpoint 体系**：异步/增量 checkpoint、断点续训、checkpoint→线上模型发布。
- [ ] **坏 pass 隔离**：识别异常时间片样本，支持跳过/回滚，防坏数据被增量学进模型。
- [ ] **训练监控**：loss/AUC、global step、warmup 期稳定性、OOM、回流时效、版本节奏看板。
- [ ] **shuffle 能力**：离线全局 shuffle + 流式局部 shuffle buffer。
- [ ] **batch/资源配置透出**：batch_size、worker 内存、PS 通信粒度可配可观测。

> 📌 性价比最高、算法最买账的三件：**slot 配置中心 + 离在线一致的 sign 哈希 + 增量回流通道（撑起 pass 节奏）**。

## 16. 向上沟通：翻译成 ROI / 风险讲给老板

> 🎓 老板不关心 step 怎么换算，关心**模型多久能更新、出事多久能恢复、要花多少钱**。三句话模板：

1. **用"模型更新速度"讲价值**：*"我们把 slot/样本/增量回流这套链路做扎实，模型就能从'天级更新'提到'小时级更新'，更快追上用户最新兴趣，直接关系 CTR/转化。"*
2. **用"事故与容错"讲风险**：*"流式训练样本只看一次，喂错了没法回炉。我们建坏 pass 隔离 + 一致性校验，能在坏数据污染模型前拦住，把一次潜在线上事故消灭在源头。"*
3. **用"成本与扩容"讲取舍**：*"算法每加一批 slot、每提一档 batch，背后是存储、算力、带宽的真金白银。我们把 slot 配置中心、参数分片、异步 checkpoint 做好，能在不无脑堆机器的前提下支撑模型长大。"*

> 💡 一句话向上总结：**"这些训练名词背后，是'模型更新有多快、出事恢复有多稳、规模扩张有多省'三件事——这正是训练平台的价值。"**

## 17. 一图总结 + 一句话速记

```
                         训练流程名词全景（平台视角）
   ┌──────────────┬──────────────┬───────────────┬──────────────┬──────────────┐
   │   slot       │   sample      │   batch        │   pass        │   产出        │
   │（特征域/类）   │（一条 instance）│（一撮样本）      │（含义A:遍数据  │（模型版本）    │
   │              │ =label+多slot │ batch_size 条   │  含义B:时间片）│              │
   └──────┬───────┴──────┬───────┴───────┬───────┴──────┬───────┴──────┬───────┘
          │feature→sign   │拼接           │ step=处理1batch│ one-pass     │ checkpoint
          │→embedding     │（按slot配置）   │ epoch=过完整份  │ base+delta   │ →热更新
          ▼              ▼               ▼              ▼              ▼
     离在线哈希一致     样本拼接引擎      显存/lr/warmup   增量回流时效    版本管理/坏pass隔离
     （红线）          百亿/天吞吐       OOM/PS通信       归因窗口        异步增量checkpoint
   ───────────────────────────────────────────────────────────────────────────
   每个词 → 一句"平台要交付 XX 数据/链路能力"
```

> 📌 **一句话速记**：**slot 是"看哪几类信息"，sign 是"每个值的编号"（离在线必须一致），batch 是"一口喂多少"，pass/epoch 是"过几遍数据"——搜广推只过一遍（one-pass）、按时间片滚动出 base+delta；这一切背后都是平台要撑住的样本流、哈希一致、参数分片和版本管理。**

## 18. 自测三问

> 检验你是否真懂（答得上来，评审现场就稳了）：

1. **算法说"帮我加 3 个 slot"，你要在脑子里展开成哪些平台动作？为什么"加 slot 不只是改配置"？**
   <details><summary>参考答案要点</summary>展开为：新增数据源接入（每个 slot 声明来源表/快照字段）→ 离在线一致的 sign 哈希规则 → 样本拼接 schema 变更 → 参数服务器参数空间扩容 → 存储/带宽评估 → 一致性校验。因为 slot 驱动的是从取数到拼接到参数的整条链路，任一环（尤其哈希一致）出错都会让模型学废。</details>

2. **搜广推训练为什么常 one-pass、不跑多 epoch？这对样本平台的容错意味着什么？**
   <details><summary>参考答案要点</summary>三因素共振：① 数据量大（百亿/天），刷旧数据不划算；② 时效性高，多 epoch 拖慢更新、追不上分布；③ 只看一次=天然正则，抗过拟合。对样本平台的含义：每条样本只被训练看到一次，喂错没有"下个 epoch 纠正"的机会，所以样本质量与离在线一致性容错极低，必须前置校验、坏 pass 可隔离回滚。</details>

3. **搜广推流式训练里的 pass，和教科书里的 epoch 有什么区别？base+delta 为什么需要 base？**
   <details><summary>参考答案要点</summary>教科书 epoch/pass(含义A) = 把同一份固定数据过一遍，可多遍。搜广推 pass(含义B) = 时间轴上一个时间片（如一小时）的增量样本，过一遍即弃，每个 pass 产一个模型版本（delta）。需要 base 是因为：delta 持续累加会让模型漂移（近期样本主导、遗忘），周期性 base 全量重训用于纠偏，base+delta 是"时效（delta）与稳定（base）"的折中。</details>
