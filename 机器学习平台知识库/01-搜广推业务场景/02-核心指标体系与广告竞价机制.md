## 0. 30 秒速通(TL;DR)

1. **指标分三层**:模型层(CTR/CVR/AUC,算法同学盯)→ 产品层(点击率/转化率/时长/留存,产品同学盯)→ 业务层(GMV/收入/DAU/LTV,老板盯)。**下层是上层的因,上层是下层的果**,但传导有损耗、有时滞。
2. **广告核心公式只有一个**:`eCPM = pCTR × bid × 1000`(CPC 计费下)。它既是**排序依据**,又是**收入预期**。看懂它,广告商业化你就懂了一半。
3. **二价拍卖(GSP)**:赢家按"第二名的价格"扣费,目的是**让广告主敢报真实价值**(报真话是最优策略)。
4. **校准(calibration)是平台研发的命门**:pCTR 预估偏高 → eCPM 虚高 → 广告主被多扣钱 / 实际收入对不上预期。**AUC 高不代表校准好**,这是两回事。
5. **指标会骗人**:幸存者偏差(只看曝光过的)、辛普森悖论(分组反转)、平均值掩盖分布。AB 实验选错指标,结论全反。
6. **对平台研发的总要求**:把**指标计算口径、样本 label 定义、校准所需的分布**这三件事做对、做稳、可追溯。其他都是上层建筑。

> 🎓 **老师视角**:很多平台研发把自己定位成"管道工"——数据从 A 搬到 B 就行。错。你搬的每一个 label、每一个特征分布,都在影响别人花的钱和信的数。**你是会计,不是搬运工。** 这篇文档的所有内容,最后都要落到"我这条链路出错,谁会受伤、伤多少"。

## 1. 指标金字塔:三层传导

### 1.1 为什么要分层

搜广推系统里,指标多到能列一整页。如果不分层,你会陷入"算法说 A 指标好,产品说 B 指标坏,老板问 C 指标"的鸡同鸭讲。分层的本质是**按"谁能直接影响它 / 谁为它负责"来归类**。

```
                    ┌─────────────────────────────────────┐
   顶层:业务指标     │  GMV  收入  DAU/MAU  LTV  ARPU  留存   │  ← 老板/财务盯
   (结果,慢)        │  「公司活得好不好」                     │
                    └───────────────▲─────────────────────┘
                                    │ 传导(有损耗、有时滞)
                    ┌───────────────┴─────────────────────┐
   中层:产品指标     │  点击率  转化率  停留时长  人均时长     │  ← 产品经理盯
   (行为,中)        │  下单率  复购率  次日留存             │  「用户用得爽不爽」
                    └───────────────▲─────────────────────┘
                                    │ 传导(模型只影响"排序质量")
                    ┌───────────────┴─────────────────────┐
   底层:模型指标     │  CTR  CVR  AUC  GAUC  LogLoss  校准   │  ← 算法工程师盯
   (能力,快)        │  「模型预测准不准、排序好不好」          │
                    └─────────────────────────────────────┘
```

### 1.2 逐层传导关系(关键!)

- **模型层 → 产品层**:模型把"用户更可能点/买的内容"排到前面 → 用户实际点击率/转化率上升。
- **产品层 → 业务层**:更多点击/转化 → 更多成交 → GMV 上升;用户用得爽 → 留存高 → DAU 涨。

**但传导不是 1:1 的**,中间有三个"漏":

| 传导环节 | 漏在哪 | 典型表现 |
|---|---|---|
| 模型 → 产品 | 离线 AUC 涨 ≠ 线上点击率涨 | 离线涨 1%,线上 AB 平 or 跌(分布偏移、特征穿越) |
| 产品 → 业务 | 点击率涨 ≠ GMV 涨 | 标题党涨点击,但转化差、退货高,GMV 不动 |
| 短期 → 长期 | 短期收入涨 ≠ LTV 涨 | 加广告涨收入,用户体验差,留存掉,长期亏 |

> 🎓 **老师视角**:当算法同学兴奋地说"离线 AUC 涨了"时,你要会问一句:**"线上 AB 跑了吗?哪个产品指标动了?"** 离线指标只是"实验室体检报告",上线效果要看产品指标,而能不能拿去跟老板邀功,要看业务指标。三层之间**不能直接划等号**,这是平台研发最该有的直觉。

### 1.3 平台研发要支撑什么

- **每一层指标都要有唯一的"计算口径"和"产出链路"**。最怕的是同一个"转化率",报表系统算一个数、实验平台算另一个数、算法离线评估又是第三个数——**三套口径,三场撕逼**。
- 平台要提供**指标血缘(lineage)**:这个 GMV 是哪些订单算的?去掉了退款吗?含不含跨境?口径一变,所有依赖它的实验结论都要重算。

### 1.4 指标的"时滞":为什么传导不能立等可取

三层指标不仅有损耗,还有**时间差**。模型指标当天就能看,产品指标要等用户行为(小时~天级),业务指标尤其是 LTV/留存要等数天到数月。

```
   时滞:越往上层,指标"成熟"得越慢
   模型层 AUC    ── 离线训练完即得(分钟~小时)
   点击率/转化率 ── 当天/次日(转化还要等归因窗口)
   7日/30日留存  ── 必须等够 7 天 / 30 天
   LTV           ── 数月甚至更久,常用模型预估(pLTV)
```

| 指标 | 成熟时滞(经验) | 平台研发影响 |
|---|---|---|
| AUC/校准 | 分钟~小时 | 可快速迭代 |
| 点击率 | 当天 | 实验可较快看初步结论 |
| 转化率 | 等归因窗口(1~7 天) | 实验不能提前下结论 |
| 留存 | 7~30 天 | 长周期实验回收 |
| LTV | 数月 | 多用 pLTV 模型预估,误差大 |

> 🎓 **老师视角**:时滞决定了"实验跑多久才能看结论"。老板想三天看 GMV 涨没涨,但若主指标是 7 日留存,三天的数据根本不成熟,提前下结论就是赌博。**平台研发做实验平台时,要把每个指标的"最短成熟期"硬编码进去**,不到期不让出"显著"结论,这是对抗"老板催着上线"的技术护栏。

## 2. 模型层指标:CTR / CVR / AUC

这一层是算法同学的"母语",平台研发至少要能听懂、能复现、能对账。

### 2.1 CTR(Click-Through Rate,点击率)

- **公式**:`CTR = 点击次数 / 曝光次数`
- **业务含义**:内容/广告"被点的能力"。注意区分两个东西:
  - **后验 CTR(real CTR)**:线上真实统计出来的,是个**事实**。
  - **pCTR(predicted CTR)**:模型预估的点击概率,是个**预测**。**广告计费用的是 pCTR**。

| 概念 | 来自 | 用途 | 谁负责 |
|---|---|---|---|
| 后验 CTR | 日志统计 | 报表、监控、做 label | 平台数据 |
| pCTR | 模型打分 | 排序、计费(eCPM) | 算法模型 |

> 典型量级(经验值,不同业务差异极大):信息流广告 CTR 1%~5%;搜索广告因意图强,可到 5%~15%;开屏/Banner 可能不到 1%。**别记死数字,记数量级**。

**CTR 还有一个最容易被忽略的细节:分母怎么定义。** 同样叫"曝光",可以是"广告被请求返回"、"广告被渲染到屏幕"、"广告进入可视区域(可见曝光 viewable impression)"。三者数量级能差 20%~30%。平台研发如果把"请求"当曝光、把"渲染"当曝光混用,CTR 会无声地飘。

```
   一次广告的生命周期里有好几个"曝光"口径:

   广告返回 ──► 渲染到页面 ──► 进入可视区(>50%像素>1秒)──► 用户点击
     │             │                 │                       │
   请求曝光      渲染曝光          可见曝光(viewable)         点击
   (最大)        (中)              (最严,IAB标准)
   ────────────────────────────────────────────────────
   CTR 的分母用哪个,直接决定这个数大还是小
```

> 🎓 **老师视角**:面试里问"CTR 怎么算"人人会答,但工程里真正出事的是"曝光到底从哪个埋点取"。**平台研发的价值恰恰在这种没人愿意抠的细节上**。建议:计费、报表、训练 label 三处的曝光口径必须显式声明且一致,文档里写死,别留"默认"。

### 2.2 CVR(Conversion Rate,转化率)

- **公式**:`CVR = 转化次数 / 点击次数`(注意分母是**点击**,不是曝光)
- **业务含义**:点了之后"真的下单/注册/付费"的能力。
- **坑点**:转化的"定义"和"归因窗口"是平台研发要钉死的——转化算"加购"还是"支付成功"?点击后 1 小时算还是 7 天算?换个口径,CVR 能差一倍。详见 [样本标签生成·归因窗口](../04-样本数据管理/02-标签生成·归因窗口·延迟反馈.md)。

**脱敏案例(典型量级,非真实数据)**:某电商业务,把转化口径从"支付成功"偷偷改成"提交订单",CVR 从约 2.0% 跳到约 3.2%。算法侧 pCVR 模型用新 label 重训,oCPC 出价随之被抬高,广告主成本上涨却没等量转化,一周内投诉量翻倍。**根因不是模型,是口径变更没有版本化、没有通知下游。** 这就是平台研发为什么要把口径当"接口"来管。

| 转化口径(由浅到深) | CVR 量级(示意) | 谁爱用 |
|---|---|---|
| 加购 add-to-cart | 高 | 想让数据好看的 |
| 提交订单 | 中 | 中间口径 |
| 支付成功 | 低 | 财务认可的 |
| 支付且不退款(净转化) | 最低 | 算真账的 |

### 2.3 CTCVR 与全链路

电商里常说 `pCTCVR = pCTR × pCVR`,表示"从曝光到转化"的整体概率。这是 ESMM 等多任务模型的基础。对平台研发的意义:**CVR 样本有"样本选择偏差"**——只有点击过的才有 CVR label,而线上要对所有曝光打分,训练和预测的样本空间不一致。

### 2.4 AUC / GAUC(排序能力)

- **AUC(Area Under ROC Curve)**:衡量"模型把正样本排在负样本前面"的能力。取值 0.5(瞎猜)~1.0(完美)。**只关心排序,不关心绝对值**。
- **GAUC(Group AUC)**:按 user(或 session)分组各算 AUC 再加权平均。**搜广推里 GAUC 比 AUC 更有意义**,因为线上是"给同一个用户排他能看到的候选",而不是把所有人的所有 item 混在一起排。

```
AUC 的直觉:随机抽一个正样本、一个负样本,
模型给正样本打分更高的概率。

  正样本打分分布   ░░░░░░▓▓▓▓████        AUC 高 = 两堆分得开
  负样本打分分布 ████▓▓▓░░
                ────────────────► 打分
```

| 指标 | 关心什么 | 高 ≠ 好的陷阱 |
|---|---|---|
| AUC | 排序对不对 | **AUC 高但校准差,计费照样错** |
| GAUC | 单用户内排序对不对 | 更贴近线上,但样本少的用户噪声大 |
| LogLoss | 概率值准不准(含校准) | 对极端值敏感 |
| 校准(Cal) | pCTR 是否等于真实 CTR | AUC 完全不反映它 |

**AUC 还有几个工程上必须知道的性质:**

- **AUC 对类别不平衡不敏感**(这是优点也是缺点)。CTR 任务正样本极少(1%~5%),AUC 仍然稳定,但这也意味着 AUC 看不出"绝对概率准不准"。
- **AUC 不可加**:不能把两天的 AUC 平均成两天合并的 AUC。平台做指标聚合时,要存原始打分对或用 PR-AUC/分桶统计,别简单平均。
- **GAUC 的权重要想清楚**:按曝光数加权还是按用户数加权?重度用户曝光多,按曝光加权会被他们主导。这是平台研发实现 GAUC 计算时必须和算法对齐的参数。

```
   GAUC 计算:每个用户单独算 AUC,再加权

   user_1: AUC=0.70  权重 w1 (该用户曝光数/点击数)
   user_2: AUC=0.65  权重 w2
   user_3: 全是正样本或全是负样本 → AUC 无法定义 → 丢弃
   ...
   GAUC = Σ(AUC_u × w_u) / Σ w_u
          ↑ "单组无正负样本"的处理方式不同,GAUC 能差几个点
```

> 🎓 **老师视角**:GAUC 里"某个用户在这批样本里全是负样本(没点任何东西)"的情况非常常见,这种用户的 AUC 算不出来。是丢弃、还是计 0.5、还是其他处理?**不同处理方式下 GAUC 能差好几个千分点**。平台研发实现评估库时,这个细节必须和算法白纸黑字对齐,否则"你算 0.68 我算 0.66"的撕逼永远停不下来。

> 🎓 **老师视角**:面试和工作中最常见的误解是**"AUC 高 = 模型好 = 计费准"**。错!AUC 只看排序。一个模型把所有 pCTR 都乘以 2,AUC 一点不变(排序没变),但每一笔计费都会高一倍。**广告系统真正要命的是校准,不是 AUC。** 这件事第 8 节会再砸一遍,因为它就是平台研发的核心战场。

### 2.5 平台研发要支撑什么

- 提供**统一的离线评估口径**:同样的 AUC/GAUC 计算逻辑,算法离线、平台监控用同一套库,避免"你算 0.72 我算 0.70"。
- 提供**校准监控**:线上分桶统计 `平均 pCTR vs 实际 CTR`,任一桶偏差超阈值就告警。

## 3. 商业化指标:CPM / eCPM / CPC / CPA / ROI

广告的钱怎么算、平台怎么赚,全在这一节。**这一节是广告平台研发的发动机舱**。

### 3.1 核心定义速查表

| 指标 | 全称 | 公式 | 站在谁的视角 |
|---|---|---|---|
| CPM | Cost Per Mille | 每千次**曝光**的成本 | 广告主付的钱 |
| eCPM | effective CPM | `eCPM = 收入 / 曝光数 × 1000` | **平台**衡量一个广告位赚钱能力 |
| CPC | Cost Per Click | 每次**点击**的成本 | 广告主按点击付费 |
| CPA | Cost Per Action | 每次**转化**的成本 | 广告主按转化付费 |
| ROI | Return On Investment | `ROI = 广告带来的收入 / 广告花费` | 广告主算账 |
| ROAS | Return On Ad Spend | 同 ROI,广告口径更常用 | 广告主算账 |

### 3.2 eCPM:整个广告系统的"通用货币"

**为什么需要 eCPM?** 因为广告位要在 CPM、CPC、CPA 不同计费的广告之间排序——它们计费单位都不一样,没法直接比。eCPM 把它们**统一换算成"每千次曝光能给平台带来多少钱"**,这样就能放一起比了。

```
不同计费方式 → 统一折算成 eCPM(每千次曝光预期收入)→ 排序

  CPM 广告:  eCPM = 出价(本来就是千次曝光价)
  CPC 广告:  eCPM = pCTR × CPC出价 × 1000
  CPA 广告:  eCPM = pCTR × pCVR × CPA出价 × 1000
                    └──────┬──────┘
                      = 转化概率链

  排序:谁的 eCPM 高,谁排前面 → 平台收入最大化
```

**`eCPM = pCTR × bid × 1000` 这个公式要刻进 DNA**(CPC 场景)。直觉:出价是"点一次愿意付多少",pCTR 是"曝光一次能点多少次",两者相乘就是"曝光一次的期望收入",乘 1000 就是千次。

**三种计费混排同一个广告位(脱敏示意,看 eCPM 怎么拉平它们)**:

| 广告 | 计费 | 出价 | pCTR | pCVR | eCPM(元/千次) |
|---|---|---|---|---|---|
| A | CPM | 40(千次) | — | — | 40 |
| B | CPC | 1.5(点击) | 3% | — | 0.03×1.5×1000 = 45 |
| C | CPA | 60(转化) | 4% | 2% | 0.04×0.02×60×1000 = 48 |

排序结果:**C(48) > B(45) > A(40)**。注意 C 是 CPA,eCPM 里多乘了一层 pCVR——**转化预估链越长,eCPM 越依赖校准,误差也越容易累积**。C 的 pCVR 若高估,它会不公平地抢到 A、B 的位置,且平台预期收入虚高。这就是为什么"转化型广告"对校准的要求比"点击型"还苛刻。

> 典型量级(经验值):成熟信息流广告 eCPM 大致几十到一两百元 / 千次曝光;搜索广告因意图强,eCPM 通常更高。**这个数严重依赖行业、流量质量,别背死。**

**eCPM 拆解练习(脱敏,示意量级)**:一条 CPC 广告,广告主出价 1.5 元/点击,模型预估 pCTR=3%。
```
   eCPM = pCTR × bid × 1000
        = 0.03 × 1.5 × 1000
        = 45 元 / 千次曝光
```
另一条 CPM 广告直接出价 40 元/千次。则第一条(45)排在前面。**注意:这里用的是 pCTR(预估),不是后验 CTR**。如果这条广告真实 CTR 只有 2%,平台预期收入(按 45 算)就会高于实收,差的就是校准误差。这一笔一笔累加,就是月底"收入对不上预测"的来源。

### 3.3 CPC / CPA / ROI 算账

- **CPC**:广告主关心"我花的每一块钱买了几个点击"。`CPC = 总花费 / 总点击`。
- **CPA**:广告主更关心"买了几个真实转化"。`CPA = 总花费 / 总转化`。
- **ROI/ROAS**:广告主最终的账。`ROAS = 转化带来的 GMV / 广告花费`。比如花 1 万带来 5 万 GMV,ROAS=5。广告主一般有个 ROAS 红线,跌破就停投。

**一条龙算账(脱敏示意,把商业化指标串起来)**:
```
   投放 10 万元广告
        │  eCPM ≈ 50 元/千次 → 曝光 ≈ 200 万次
        │  CTR ≈ 3%          → 点击 ≈ 6 万次
        │  CPC = 花费/点击    → 10万 / 6万 ≈ 1.67 元/点击
        │  CVR ≈ 2%          → 转化 ≈ 1200 单
        │  CPA = 花费/转化    → 10万 / 1200 ≈ 83 元/单
        │  客单价 500 元      → GMV ≈ 60 万元
        ▼
   ROAS = 60万 / 10万 = 6   (达标,继续投)
```
**这条链上任何一个指标算错,广告主的账就错。** 比如平台把 CTR 口径从"可见曝光"偷换成"请求曝光",曝光分母变大、CTR 显示变小,广告主会误判"点击太贵"而砍预算——其实是口径问题,不是效果问题。**平台研发守住口径,就是守住广告主的信任。**

> 🎓 **老师视角**:平台和广告主的目标**不完全一致**,这是理解广告系统的关键张力。平台想最大化 eCPM(短期收入),广告主想最大化 ROI(花得值)。**如果平台只顾 eCPM、不管广告主 ROI,广告主就会跑路**,长期收入反而崩。所以现代广告系统(oCPC/智能出价)本质是在"平台收入"和"广告主 ROI"之间找平衡。第 7 节讲。

### 3.4 平台研发要支撑什么

- **计费口径必须和 eCPM 公式严格对齐**:扣的钱 = 真实计费逻辑,绝不能"排序用一套 pCTR,计费用另一套"。
- **eCPM 的每个因子都要可追溯**:某条广告 eCPM 突然跌,要能拆出来是 pCTR 掉了、出价降了、还是 pCVR 链路出问题。**因子级监控**是广告平台的标配。

## 4. 业务层指标:GMV / ARPU / LTV / DAU / 留存 / 时长

这一层是老板和财务的语言。平台研发要会**把模型/商业化指标翻译成这层指标**。

### 4.1 速查表

| 指标 | 公式 / 含义 | 业务意义 |
|---|---|---|
| GMV | Gross Merchandise Volume,成交总额 | 平台规模(注意:不等于收入,含未付款/退款口径差异) |
| 收入 Revenue | 平台实际赚到的钱(佣金/广告费) | 真金白银 |
| DAU/MAU | 日活/月活跃用户数 | 大盘活跃度;DAU/MAU 比值看"粘性" |
| 留存率 | 第 N 日还回来的用户比例 | 次留、7 留、30 留;产品健康度 |
| 人均时长 | 总时长 / DAU | 用户投入度(信息流/视频核心指标) |
| ARPU | Average Revenue Per User = 收入 / 用户数 | 单用户变现能力 |
| ARPPU | 收入 / 付费用户数 | 付费用户变现能力 |
| LTV | Life Time Value,用户全生命周期价值 | 一个用户一辈子能贡献多少钱 |

### 4.2 LTV 与 CAC:商业模式是否成立的底线

```
   一个用户的经济账:
   ┌─────────────┐         ┌──────────────────────────┐
   │ 获客成本 CAC │  vs     │ 生命周期价值 LTV          │
   │ (买量花的钱) │         │ (这人一辈子贡献的收入)     │
   └─────────────┘         └──────────────────────────┘
            必须  LTV > CAC,且 LTV/CAC 越大越健康
            (经验上常说 LTV/CAC > 3 才算健康,行业差异大)
```

**LTV 估算的直觉(脱敏示意)**:某 App 月 ARPU ≈ 8 元,月留存(月流失率约 20%)对应平均存活约 5 个月,则粗略 `LTV ≈ 8 × 5 = 40 元`。若买一个用户的 CAC = 15 元,则 LTV/CAC ≈ 2.7,**接近健康线但不算宽裕**。这时若为了短期收入加广告导致月留存恶化(存活掉到 4 个月),LTV 直接掉到 32,LTV/CAC 跌破 2.2——**短期多赚的广告费,远不够补回留存恶化造成的 LTV 损失**。这就是"借高利贷"的量化版本。

> 🎓 **老师视角**:为什么老板天天问留存而不只问收入?因为 **LTV ≈ ARPU × 用户存活时长**,而存活时长由留存决定。留存掉一点,LTV 指数级缩水,买来的用户全亏。所以**任何"涨短期收入但掉留存"的策略,本质是借高利贷**。当商业化同学说"加个广告位 eCPM 能涨 5%"时,平台研发要有意识去问:**"留存的 AB 看了吗?"**

### 4.2.1 留存率的几个口径(平台研发必须钉死)

"次日留存 30%" 这句话本身是没定义的,因为:

| 维度 | 口径选项 | 影响 |
|---|---|---|
| 分母(谁) | 新增用户 / 全体活跃 / 某次活动进来的 | 新增留存 vs 活跃留存差很大 |
| 留存判定(回来算什么) | 打开 App / 有核心行为 / 有付费 | 越严越低 |
| 时间窗(第 N 日) | 自然日 / 滚动 24 小时 | 跨零点行为算哪天 |

```
   Day0 新增 1000 人
        │
   Day1 回来 300 人  → 次日留存 = 300/1000 = 30%
   Day7 回来 150 人  → 7 日留存 = 150/1000 = 15%
   Day30 回来 80 人  → 30 日留存 = 80/1000 = 8%
        ↓
   留存曲线一般"先陡降后拖尾",拖尾的高度 ≈ 这个产品的"忠实用户基本盘"
```

> 🎓 **老师视角**:留存曲线的**拖尾高度**比次留更能反映产品价值——次留可以靠 push 和补贴硬拉,但 30 日、90 日还回来的人才是真爱。平台研发做留存指标时,要支持**长窗口回收**和**按"激活来源"分群**,否则买量团队会用"次留好看"的渠道骗预算,而这些用户两周后全跑光。

### 4.2.2 DAU/MAU 与粘性

- **DAU/MAU 比值**(也叫 stickiness)反映用户多频繁回来。比值 0.5 意味着平均一个月活用户里有一半天天来。社交/工具类高,低频电商类低。
- **平台研发的坑**:DAU 去重逻辑。同一人多设备?游客转登录?这些 join/去重错误会让 DAU 虚高,直接污染所有人均指标(因为它们的分母是 DAU)。



### 4.3 平台研发要支撑什么

- **业务指标的口径是政治问题**:GMV 含不含退款、含不含跨境、含不含刷单,每个口径背后都是不同部门的 KPI。平台要把口径**显式化、版本化**,别让人偷偷改。
- **长周期指标(LTV/留存)的归因极难**:用户今天的留存,可能是 7 天前某次推荐造成的。平台要支撑**长窗口、跨天的指标计算和实验回收**。

## 5. 广告计费模式:谁承担风险

计费模式的核心区别是**"风险落在谁身上"**——平台还是广告主。

```
   计费节点越靠后,平台承担的风险越大
   曝光 ───────► 点击 ───────► 转化
    │             │             │
   CPM           CPC           CPA / oCPC
  广告主担风险   各担一半      平台担更多风险
 (曝光了就付)  (点了才付,    (要预测转化,
              平台担"点不点")   担"转不转化")
```

| 模式 | 广告主何时付费 | 风险归属 | 平台为啥愿意/不愿意 |
|---|---|---|---|
| CPM | 每次曝光 | 广告主担全部(曝光不点白花钱) | 平台旱涝保收,但广告主不爱 |
| CPC | 每次点击 | 平台担"点击率"风险 | 平台要预估准 pCTR,否则亏 |
| CPA | 每次转化 | 平台担"转化"风险 | 平台基本不直接做,太亏 |
| oCPC | 按点击扣,以转化为目标优化 | 平台担优化责任,广告主担最终扣费 | **现代主流**,双赢折中 |
| oCPX | oCPC 的泛化(目标可为下载/付费/ROI) | 同上 | 同上 |

> 🎓 **老师视角**:为什么纯 CPA 几乎没平台敢做?因为转化离平台太远——用户点了广告,落地页烂、客服差、商品贵,都会导致不转化,**这些锅不该平台背,但 CPA 让平台背了**。所以行业演化出 oCPC:**广告主设转化目标,平台用模型帮你优化,但实际扣费仍按点击**。风险分摊更合理。这就是为什么你今天看到的广告系统几乎都在跑 oCPC/oCPX。

**风险量化(脱敏示意)**:假设平台对一条广告纯 CPA 计费、约定 CPA=80 元,广告主预期转化率 2%。若广告主落地页突然变烂,真实 CVR 掉到 1%:
```
   平台为拿到 1 个转化要送出的点击数翻倍(2% → 1%)
   平台成本(流量)≈ 翻倍,但收入仍按 80 元/转化
   → 平台每个转化的"流量成本"吃掉利润甚至倒贴
   ── 转化端的锅(落地页),却由平台的流量成本买单
```
oCPC 把它改成"按点击扣费、以转化为优化目标":落地页变烂导致转化少,广告主自己的 CPA 升高、自己承担,平台只承担"点击预估"这部分风险。**风险落到了能控制它的人头上**,这才是健康的机制设计。

### 5.1 平台研发要支撑什么

- 不同计费模式**对 label 和样本的要求不同**:CPC 要点击 label,CPA/oCPC 要转化 label(且涉及归因窗口、延迟反馈)。平台要为每种模式准备对应的样本流。
- **计费防作弊**:CPC 要防刷点击,CPA 要防刷转化。异常流量过滤是计费链路的一部分。

## 6. 广告竞价机制:GSP / eCPM 排序 / 保留价

### 6.1 一次广告请求的竞价全流程

```
用户刷到一个广告位,后台发生了什么:

 ① 召回      从千万广告里粗选出几百/几千个候选(定向、预算过滤)
    │
 ▼
 ② 预估      对每个候选打分 pCTR(、pCVR)
    │
 ▼
 ③ 排序      算 eCPM = pCTR × bid × 1000,降序
    │        (有的还乘"广告体验分"等因子做生态调控)
 ▼
 ④ 过保留价   eCPM < 保留价(reserve price)的直接淘汰
    │
 ▼
 ⑤ 拍卖计费   GSP:赢家按"足够赢过第二名的最低价"扣费
    │
 ▼
 ⑥ 展示+回传  曝光/点击/转化日志回流 → 成为明天的训练样本
```

### 6.2 GSP(广义第二价格拍卖)为什么用"二价"

- **一价(GFP)的问题**:你出价多少就扣多少 → 广告主会拼命**压价试探**,出价不稳定、系统震荡,大家都不敢报真实价值。
- **二价(GSP/Vickrey 思想)**:赢家扣的是"刚好赢过下一名所需的价格"。**最优策略是报真实价值**(报高了多花冤枉钱,报低了可能丢量),博弈论上更稳定。

```
   三个广告竞同一个位,按 eCPM 排序:
     广告A  eCPM = 50   ← 赢家
     广告B  eCPM = 30   ← 第二名
     广告C  eCPM = 10

   一价:A 按自己的 50 扣
   二价:A 按"刚好压过 B 的价"扣
        → A 实际 CPC = (B的eCPM / A的pCTR) / 1000
        即 A 只需付"赢过 B 所必需的钱",不必付满自己出价
```

**把上面的二价 CPC 算到底(脱敏示意)**:设 A 的 pCTR=4%、出价 1.25 元/点击,故 `eCPM_A = 0.04×1.25×1000 = 50`;B 的 eCPM=30。二价下 A 实际扣费(每次点击)≈ 让 A 的 eCPM 刚好等于 B 的 eCPM 所需的点击价:
```
   actual_CPC_A = eCPM_B / (pCTR_A × 1000)
                = 30 / (0.04 × 1000)
                = 0.75 元 / 点击
```
A 原本愿意出 1.25,实际只付 0.75。**省下的 0.5 元正是"二价让利",换来的是 A 敢报真实的 1.25 而不是反复试探压价。** 平台研发实现计费时,这个折算公式必须和排序口径中的 pCTR 完全一致,否则扣的钱就不是这个数。

| 维度 | 一价 GFP | 二价 GSP |
|---|---|---|
| 赢家扣费 | 自己的出价 | 赢过第二名所需的价 |
| 广告主策略 | 拼命压价、博弈 | 报真实价值最优 |
| 系统稳定性 | 差(震荡) | 好 |
| 现实采用 | 程序化展示广告有回归一价的趋势 | 经典搜索/信息流广告主流 |

> 🎓 **老师视角**:"二价"听起来像平台吃亏(少收钱了),但它换来的是**广告主敢报真实出价、生态稳定**,长期总收入更高。这是一个经典的"短期让利换长期繁荣"设计。注意业界近年(尤其展示广告、Google Ad Manager)有**回归一价(first-price)**的趋势,因为多层中介下二价变得不透明。**别把"二价"当成永恒真理,要知道它解决的是什么问题。**

### 6.3 保留价(Reserve Price)

平台设的"地板价":eCPM 低于它的广告不让展示。作用:**保护用户体验 + 抬高成交价**。保留价是平台调节"收入 vs 体验"的旋钮之一。

保留价还有一个二价拍卖里的微妙作用:**它本身也参与"第二价格"的计算**。当只有一个广告参竞,或第二名很弱时,赢家不是按"第二名"扣,而是按"保留价"扣——否则平台就几乎收不到钱。所以保留价是二价机制的"兜底地板"。

```
   只有一个广告 A 参竞,eCPM=50,保留价=20:
     没有保留价 → 二价无"第二名"可比 → 趋近 0,平台白送
     有保留价   → A 按 20 扣费(保留价兜底)
   保留价 = 二价拍卖的"隐形第二名"
```

### 6.4 生态调控因子(rank score 不只看钱)

成熟平台的排序公式往往不是纯 eCPM,而是:
```
   rank_score = eCPM × quality_factor
              = pCTR × bid × 1000 × (体验分/相关性/落地页质量…)
```
目的是不让"出价高但体验差"的广告霸屏。这一步对平台研发的意义:**计费要和排序解耦清楚**——排序用了 quality_factor,但计费的二价折算要明确是否含这个因子,含错了钱就算错。Google 的 Ad Rank、各家的"广告质量分"都是这个思路。

### 6.5 平台研发要支撑什么

- 竞价链路**毫秒级**,pCTR 预估服务的延迟、降级、兜底都是平台研发的活。预估服务挂了,要有兜底分(用后验 CTR 或默认值),不能让整个广告位空着。
- **计费日志的准确与幂等**:同一次点击不能重复扣费,网络重试要幂等。这是钱的事,错不起。

## 7. 智能出价 oCPC / oCPX:平台帮广告主调价

### 7.1 它解决什么

广告主不懂怎么调 CPC 出价才能拿到最多转化又不超成本。oCPC 让广告主只设"我要转化、每个转化最多花 X 元",**平台用模型(pCVR)帮它动态折算成每次点击该出多少**。

```
   广告主视角(简单)         平台内部(复杂)
   ┌──────────────┐        ┌─────────────────────────────┐
   │ 目标:加微好友 │        │ 实时算 pCVR                  │
   │ 成本上限:30元 │  ───►  │ 动态出价 bid_click =          │
   └──────────────┘        │   目标CPA × pCVR             │
                           │ → 折算 eCPM 参与竞价          │
                           │ → 实际仍按点击 GSP 扣费       │
                           └─────────────────────────────┘
```

### 7.2 oCPC 的两阶段(常见工程实现)

| 阶段 | 做什么 | 平台研发要点 |
|---|---|---|
| 第一阶段(冷启) | 用广告主出价跑,**积累转化数据** | 转化样本不够 pCVR 不准,要凑够量级(经验:几十个转化起步) |
| 第二阶段(智能出价) | pCVR 够稳了,平台接管自动出价 | 模型校准、成本控制(PID/反馈控制)是核心 |

### 7.3 oCPX

oCPC 的泛化:优化目标不止"点击后转化",可以是 APP 下载、激活、付费、甚至 ROI。X = 任意目标。

> 🎓 **老师视角**:oCPC 把"出价"这件事从广告主手里**接管到了平台的算法里**。这意味着:**平台的 pCVR 校准直接决定广告主的实际成本**。pCVR 高估 → 平台以为容易转化 → 出高价抢量 → 实际转化少 → 广告主成本爆了 → 投诉/跑路。所以 oCPC 体系下,**校准从"影响收入对账"升级为"影响广告主信任"**,平台研发的样本/校准责任更重了。

### 7.4 平台研发要支撑什么

- oCPC 严重依赖**转化样本的及时回收**。转化有延迟(用户今天点、三天后才下单),这就是**延迟反馈(delayed feedback)**问题,平台要设计转化样本的等待窗口与补回机制。详见 [标签生成·归因窗口·延迟反馈](../04-样本数据管理/02-标签生成·归因窗口·延迟反馈.md)。
- 成本控制需要**实时的花费/转化回流**,出价策略才能闭环调节。

### 7.5 成本控制(PID)的直觉:出价是个"恒温器"

oCPC 第二阶段的核心是**把实际 CPA 控制在广告主目标附近**。这通常用控制论里的反馈调节(类似 PID):实际成本高了就压出价,低了就抬出价。

```
   目标 CPA = 30 元
        │
   ┌────┴───────────────────────────────────┐
   │  实测 CPA 35 元(超了)→ 调低出价系数      │
   │  实测 CPA 25 元(还有空间)→ 调高出价系数  │  ← 像空调恒温
   └────────────────────────────────────────┘
   出价系数 bid_ratio 随实测成本不断微调,把 CPA 拉回目标
```

> 🎓 **老师视角**:这里平台研发的隐形责任是**回流数据的实时性和完整性**。如果转化回流延迟了 6 小时,控制器看到的是"几小时前的旧成本",调节就会滞后甚至震荡(超调)。**控制系统的稳定性,首先取决于反馈信号的质量**——而反馈信号正是你这条数据链路产出的。延迟反馈在 oCPC 里不是学术问题,是真金白银的成本失控风险。

## 8. 指标与样本/特征/校准的关系

**这一节是整篇文档对平台研发最关键的部分。** 前面讲的所有指标,最终都建立在你产出的样本和特征之上。

### 8.1 label 怎么定义,指标就怎么算

- CTR 的 label = 这次曝光有没有被点 → 你怎么 join 曝光日志和点击日志,决定了 CTR 算得对不对。
- CVR 的 label = 点击后有没有转化 → 转化定义(加购?支付?)+ 归因窗口(几天内算)直接决定 label。

```
   曝光日志 ──┐
              ├─ join(by request_id / 曝光ID) ─► 训练样本(带 click label)
   点击日志 ──┘
                          │
   转化日志 ──── 归因窗口内 join ──► 加 conversion label
                          │
              ↑ 这里 join 错一个 key、窗口设错一天,
                label 就错,所有下游指标和模型一起错
```

> 🎓 **老师视角**:平台研发最容易低估的就是"join 的正确性"。曝光和点击用什么 key 关联?去重逻辑对不对?跨天的点击怎么算?**label 错了,模型学的是错的目标,指标算的是错的数,而且没人能从模型效果上一眼看出来**——它会安静地烂掉。这是平台研发最该死磕的细节。

### 8.2 为什么 pCTR 预估偏高会多扣广告主钱

回到 `eCPM = pCTR × bid × 1000`,以及二价计费里 CPC 的折算。**如果 pCTR 系统性偏高**:

```
   pCTR 高估 1.2 倍
        │
        ▼
   eCPM 虚高 → 这条广告"看起来"比实际值钱
        │
        ▼
   ① 排序错位:不值钱的广告排到了前面(劣币驱逐良币)
   ② 计费偏差:二价折算下广告主实际单价被推高
        │
        ▼
   广告主 ROI 变差 → 投诉 / 降预算 / 跑路 → 平台长期收入受损
```

**校准(calibration)的定义**:在每个 pCTR 分桶里,`平均 pCTR ≈ 实际后验 CTR`。校准好,这条线就是一条 45 度对角线。

```
  实际CTR
   ▲
   │            ╱ 理想:完美校准(y=x)
   │          ╱
   │        ╱ ··· 实际曲线偏在下方
   │      ╱  ···      = pCTR 高估(高估了点击)
   │    ╱ ···              → 计费偏高、排序失真
   │  ╱···
   └──────────────► 预测pCTR
```

### 8.3 校准为什么会坏(平台研发要盯的)

| 原因 | 说明 | 平台侧征兆 |
|---|---|---|
| 样本采样未还原 | 负样本下采样后没做概率还原 | pCTR 系统性偏高 |
| 训练/线上特征不一致(穿越) | 离线用了线上拿不到的未来特征 | 离线 AUC 虚高,线上崩 |
| 数据分布漂移 | 大促/新流量分布变了,模型没跟上 | 校准曲线整体偏移 |
| label 延迟未回收 | 转化还没回来就当负样本 | CVR 系统性低估 |

> 🎓 **老师视角**:负采样后忘记做**概率还原(re-calibration)**,是新人最常踩的雷。为了训练效率,负样本通常下采样(比如保留 1/10),这会让模型输出的概率整体偏高,**必须用公式还原回真实分布**(如 `p = p' / (p' + (1-p')/w)`)。平台研发如果不知道这一层,就会眼睁睁看着计费整体偏高还找不到原因。**采样比例必须随样本一路透传到模型侧。**

### 8.3.1 负采样概率还原(把公式讲透)

设真实负样本被以比例 `w`(如 `w=0.1`,即保留 1/10)下采样。下采样后模型学到的概率 `p'` 偏高,要还原成真实概率 `p`:

```
   下采样:正样本全留,负样本只留 w 比例
        │
   模型在"被抬高正例占比"的数据上学到 p'(偏高)
        │
   还原公式:  p = p' / ( p' + (1 - p') / w )

   数值感受(w=0.1):
     p' = 0.30  →  p ≈ 0.041   (还原后大幅降低)
     p' = 0.50  →  p ≈ 0.091
   不还原直接拿 p' 去算 eCPM,计费会高好几倍
```

| 是否还原 | pCTR 表现 | eCPM | 后果 |
|---|---|---|---|
| 不还原(直接用 p') | 系统性偏高 | 虚高 | 计费整体偏高、排序失真 |
| 还原(用 p) | 贴近真实 CTR | 准确 | 计费正确、校准曲线贴对角线 |

**平台研发的动作**:把采样比例 `w` 作为**样本元数据**和样本一起落盘、一起进特征流,模型侧才有东西可还原。`w` 丢了,这条链路就无法事后修正。

> 🎓 **老师视角**:这是"样本即契约"最锋利的体现。`w` 这个看似不起眼的小数,丢了之后整个计费偏高,而且**从模型 AUC 上完全看不出来**(还原是单调变换,不改排序、不改 AUC)。只有看校准曲线才暴露。所以校准监控不是锦上添花,是平台研发的"对账底线"。



### 8.4 平台研发要支撑什么(汇总)

- 提供**校准所需的分布数据**:负采样比例、各特征分布、线上 pCTR vs 后验 CTR 分桶统计。
- 保证**训练-服务一致性(train-serving skew 监控)**:同一条样本,离线算的特征和线上算的特征必须一致。
- 保证**样本时序正确**:绝不能让"未来"信息泄漏到训练样本里(特征穿越),这是校准与 AUC 一起崩的头号杀手。

### 8.5 常见校准方法(听得懂即可,平台研发需知道有哪些旋钮)

| 方法 | 思路 | 适用 | 平台侧成本 |
|---|---|---|---|
| 分桶校准(binning) | 按 pCTR 分桶,用桶内后验 CTR 替换 | 简单、可解释、好监控 | 需维护桶内统计,样本少的桶抖 |
| Platt scaling | 用 sigmoid 拟合一层"再校准" | 概率整体偏移时 | 轻,需小批校准集 |
| 保序回归 isotonic | 单调映射,不假设形状 | 偏移形状复杂时 | 需较多校准样本 |
| 在线动态校准 | 用近实时后验持续微调 | 分布漂移快(大促) | 重,需实时回流链路 |

```
   校准是排序之外的"第二道工序":
   原始模型打分 ──► [ 校准层 ] ──► 用于计费的 pCTR
                      ↑
        分布漂移/采样还原/在线反馈都在这层修正
   把校准做成独立模块 → 出问题能单独定位、单独回滚
```

> 🎓 **老师视角**:大厂的工程共识是**把校准从模型主网络里解耦成独立一层**。原因很实在:模型迭代快、责任在算法;校准关乎计费、责任在平台。解耦后,"上周计费偏高"可以独立排查校准层,不用每次都去翻模型,也方便灰度和回滚。**能独立问责的模块,才是可运维的模块。**

## 9. 指标陷阱:统计学不会骗你,但会被你用错

### 9.1 幸存者偏差(Survivorship Bias)

- **现象**:CVR 只能在"被点击过"的样本上算,而点击本身是有偏的(排在前面的才被点)。用这种数据训练,模型只见过"幸存者",对没被曝光的内容一无所知。
- **后果**:**样本选择偏差(SSB)**,模型在全曝光空间上预测会偏。ESMM 这类模型就是为了缓解它。

### 9.2 辛普森悖论(Simpson's Paradox)

- **现象**:分组看每组 B 都比 A 好,合起来却 A 比 B 好。
- **典型场景**:新策略在"高活用户""低活用户"两组都涨转化,但因为新策略把流量更多导给了低转化人群,**整体转化率反而跌**。

```
        转化率    A策略    B策略
  高活组          8%       9%     ← B 赢
  低活组          2%       3%     ← B 赢
  ─────────────────────────────
  合计           5.5%     4.2%    ← A 赢?!(因人群权重不同)
```

> 🎓 **老师视角**:辛普森悖论是 AB 实验里最阴的坑。**看总指标涨跌之前,先看人群构成有没有变。** 平台研发做实验平台时,一定要支持**分层(stratified)分析**——按关键维度拆开看,否则结论可能整个反过来。

### 9.3 平均值骗局

- 人均时长 50 分钟,可能是 10% 重度用户拉高的,90% 用户根本没那么久。
- **对策**:看分位数(P50/P90)、看分布直方图,别只看 mean。平台报表要默认带分位数。

### 9.4 AB 实验的指标选择

| 陷阱 | 说明 |
|---|---|
| 只看代理指标 | 点击率涨了就上线,不看 GMV/留存 → 标题党 |
| 短期看代理,长期看核心 | 短期收入涨,长期留存跌,需长周期回收 |
| 指标太多 = 总能找到一个涨的 | 多重比较问题,要事先定主指标(OEC) |
| 样本量/显著性不够 | 涨跌可能只是噪声,要做显著性检验 |

> 🎓 **老师视角**:做实验平台的平台研发,要逼着业务方**事前声明主指标(OEC, Overall Evaluation Criterion)**。不让人"实验跑完再挑一个涨的指标来汇报"——那是数据造假的温床。**事前定主指标,是实验平台的道德底线。**

### 9.5 比率指标的辛普森陷阱再加一刀:比值的可加性

平台报表里最常见的隐性 bug:**把多天/多桶的比率直接平均**。CTR 是比率,`A天CTR=2%`、`B天CTR=4%`,两天合并的 CTR **不是 3%**,而是 `(A点击+B点击)/(A曝光+B曝光)`。如果两天曝光量差 10 倍,简单平均会严重失真。

```
   Day A: 点击 20  / 曝光 1000  = 2%
   Day B: 点击 40  / 曝光  500  = 8%
   简单平均 = (2%+8%)/2 = 5%        ✗ 错
   正确合并 = (20+40)/(1000+500) = 4%  ✓ 对
   ── 比率必须"先合分子分母,再相除",不能平均比率
```

> 🎓 **老师视角**:这条规则简单到像废话,但我见过太多报表和监控因为"平均了比率"而长期报错数,且因为偏差不大没人发现。**平台研发实现任何聚合时,比率类指标一律存"分子、分母"两列,展示时现算**,绝不存"已经算好的比率"再去平均。这是数据工程的铁律。

## 10. 指标体系在不同业务/漏斗阶段的差异

同一套指标名词,放到搜索、推荐、广告三类业务,以及漏斗的召回/粗排/精排/重排不同阶段,**权重和口径完全不同**。平台研发若用"一套指标打天下",一定会和某条业务线对不上。

### 10.1 三业务的核心指标侧重

```
            搜索 Search        推荐 Recommend       广告 Ads
   主指标   相关性/转化         时长/留存/互动        eCPM/收入/ROI
   模型     pCTR(意图强)       pCTR+pCVR+时长         pCTR×pCVR×bid
   特点     query 强信号        无明确意图,靠兴趣      多了"出价"这一维
   校准     重要               次要(纯排序)          极重要(关乎计费)
```

| 业务 | 第一关注指标 | 为何 | 平台研发重点 |
|---|---|---|---|
| 搜索 | 相关性 + 转化率 | 用户带着明确意图 | query 理解、相关性 label |
| 推荐(信息流/视频) | 人均时长 + 留存 | 无意图,靠粘住用户 | 时长 label、负反馈(划走)采集 |
| 推荐(电商) | GMV + 转化 | 直接变现 | 转化归因、跨品类穿越防护 |
| 广告 | eCPM + 广告主 ROI | 双边市场 | 校准、计费、防作弊 |

> 🎓 **老师视角**:最典型的撕逼是"推荐侧不在乎校准,广告侧把校准当命"。如果平台用同一个评估模板,推荐同学会嫌你天天报校准告警,广告同学会嫌你不报。**平台研发要按业务线配置不同的指标看板和告警阈值**,别做"一刀切"的指标平台。

### 10.2 漏斗各阶段的指标差异

```
   召回(千万→千) ─► 粗排(千→百) ─► 精排(百→十) ─► 重排(十→几)
      │                │              │               │
   关心"覆盖率/        关心"粗排与     关心 AUC/        关心多样性/
    召回率/有没有      精排一致性"      GAUC/校准        生态/打散
    漏掉好的"
   指标:Recall@K      一致性(对齐率)   排序精度          业务约束指标
```

- **召回阶段**看 Recall@K(有没有把该召的召回来),不看精排那套 AUC。
- **粗排**最关键的是"和精排的一致性"——粗排淘汰的东西精排再也看不到,粗排错杀就是天花板。
- **精排**才是 AUC/GAUC/校准的主战场。
- **重排**引入多样性、生态、频控等**业务约束指标**,常常要牺牲一点单点 eCPM 换长期体验。

> 🎓 **老师视角**:平台研发常犯的错是"拿精排的 AUC 去考核召回"。召回的目标是"别漏",不是"排得准"。用错指标去考核某一层,会逼着算法往错误方向优化。**先搞清楚这一层在漏斗里的职责,再选指标。** 漏斗结构详见 [搜广推三业务本质与漏斗架构](../01-搜广推业务场景/01-搜广推三业务本质与漏斗架构.md)。

## 11. AB 实验:显著性与置信区间的工程坑

平台研发做实验平台,最危险的不是算不出指标,而是**算出来的涨跌到底是不是真的**。这一节是实验平台的"质检关"。

### 11.1 为什么需要显著性检验

两组指标有差异,可能是策略真有效,也可能只是**随机波动**。显著性检验回答的就是:"这个差异,纯靠运气能蒙出来的概率有多大?"(p 值)。p < 0.05 是常见门槛,意思是"靠运气蒙出这么大差异的概率小于 5%"。

```
   对照组 CTR ≈ 2.00%   实验组 CTR ≈ 2.05%
        差异 +0.05pp,这是真的涨,还是噪声?
        │
   看置信区间:  [-0.02pp, +0.12pp]  → 区间跨 0 → 不显著!不能上线
   或:          [+0.01pp, +0.09pp]  → 区间不含 0 → 显著,可信
   ── 只报一个点估计(+0.05pp)而不报区间,是耍流氓
```

### 11.2 工程上最容易踩的几个坑

| 坑 | 现象 | 平台侧对策 |
|---|---|---|
| 偷看(peeking) | 实验跑一半看到显著就停 | 固定样本量/用序贯检验,禁止中途叫停 |
| 多重比较 | 看 20 个指标总有 1 个"显著" | 事前定主指标(OEC),次指标做校正 |
| 样本量不足 | 区间太宽,涨跌都是噪声 | 事前算功效(power)/最小可检测效应 MDE |
| 分流不均/污染 | AA 实验都能"显著" | 上线前先跑 AA 实验自检 |
| 比率指标方差算错 | 用户内多次曝光不独立 | 用 delta method / bootstrap 修正方差 |

> 🎓 **老师视角**:最坑的是**比率指标的方差**。CTR 是"点击/曝光",但同一个用户的多次曝光高度相关(不是独立样本)。如果按"每次曝光独立"算方差,会**严重低估方差 → p 值虚小 → 把噪声当成显著结论上线**。这就是为什么严肃的实验平台要用 delta method 或 bootstrap、并以"用户"而非"曝光"为随机单元。平台研发不懂这层,实验平台就是个"自信地给出错误结论"的机器,比没有还危险。

### 11.3 AA 实验:实验平台的体检

上线任何策略前,先把用户随机分两组、**两组用完全相同的策略**(AA),理论上所有指标都应不显著。如果 AA 都能跑出"显著差异",说明分流有偏、指标方差算错或埋点有问题。**AA 实验是实验平台的自检底线**,平台研发应把它做成常态化保障。

## 12. 广告主视角的出价博弈(脱敏案例画像)

理解广告主怎么想,才能理解为什么机制要这么设计。下面是一个脱敏的画像式案例(数字为示意量级)。

```
   广告主"老王"卖瑜伽垫,目标 ROAS=4(花1元要赚4元GMV)
   客单价 100 元,毛利率 40% → 一单毛利 40 元
   实测点击→转化率约 2%  →  一次点击期望毛利 = 40 × 2% = 0.8 元
        │
   理性最高出价 = 0.8 元/点击(再高就亏毛利)
        │
   ┌────────────────────────────────────────────────┐
   │ 一价机制:老王怕出 0.8 真被扣 0.8,试探着出 0.5、0.6… │
   │           → 出价不稳、量忽高忽低、平台收入震荡        │
   │ 二价机制:老王放心出真实的 0.8,实际只按"压过第二名"扣 │
   │           → 出价稳定、量稳定、平台长期收入更高        │
   └────────────────────────────────────────────────┘
```

| 老王的痛点 | 平台的机制回应 |
|---|---|
| "我不知道出多少合适" | oCPC:你设目标 ROAS/CPA,我帮你折算点击出价 |
| "我怕出高价被宰" | 二价 GSP:报真实价值最优,不必试探 |
| "转化要好几天才回来,成本算不准" | 平台需做延迟反馈回收,否则成本失真 |
| "我的转化少,模型学不准" | oCPC 冷启阶段先用出价跑,攒够转化再智能出价 |

> 🎓 **老师视角**:平台研发要建立"双边市场"的意识——广告系统不只是平台的赚钱机器,**广告主是另一边的客户,他们的 ROI 算得过来,才会持续投钱**。你保证的延迟反馈回收、校准准确,直接决定老王的成本算不算得准、还愿不愿意留下。**技术链路的质量,最终体现在客户留存上。** 延迟反馈的机制设计详见 [标签生成·归因窗口·延迟反馈](../04-样本数据管理/02-标签生成·归因窗口·延迟反馈.md)。

## 业界怎么做

- **Google / Meta**:广告排序的工业级标准就是 eCPM 排序 + 拍卖。Google 历史上用 GSP,近年广告交易(Ad Manager)转向**一价拍卖**,因为多层中介下二价透明度差。Meta 的 oCPM/价值优化是 oCPX 思路的代表。
- **阿里 / 字节 / 腾讯**:oCPC/oCPX 是国内主流。阿里妈妈、巨量引擎、腾讯广告都有完整的智能出价体系;阿里的 ESMM 就是为解决 CVR 样本选择偏差和数据稀疏提出的。
- **校准工程**:大厂普遍有独立的**校准模块**(Platt scaling / isotonic regression / 分桶校准)和**校准监控大盘**,把校准当一等公民,而不是模型的副产品。
- **实验平台**:Microsoft、Google 的 A/B 实验体系(可查 Kohavi 的著作)是行业范本,强调 OEC、分层分析、防多重比较。

> 🎓 **老师视角**:业界趋势是**把"校准"和"出价/计费"从模型里解耦出来**,做成独立可控、可监控的模块。原因正是本文反复强调的:模型 AUC 高低是算法的事,但**钱算得对不对是平台的事**,两者必须能分开问责。

## 权衡取舍

| 取舍 | A 方 | B 方 | 怎么权衡 |
|---|---|---|---|
| 收入 vs 体验 | 多塞广告涨 eCPM | 少塞广告保留存 | 用长周期 LTV 而非短期收入做决策 |
| AUC vs 校准 | 提排序能力 | 保计费准确 | 广告场景校准优先,纯推荐排序 AUC 优先 |
| 二价 vs 一价 | 二价激励真实出价 | 一价透明简单 | 单层拍卖用二价,多层中介倾向一价 |
| 平台收入 vs 广告主 ROI | 短期榨流量 | 让广告主赚到 | oCPC 折中,长期看广告主留存 |
| 实时性 vs 准确性 | 实时样本快速迭代 | 等齐转化样本再训 | 延迟反馈场景要等窗口,牺牲实时换准确 |

## 常见误区与反直觉

1. **"AUC 涨了模型就更好"** —— 错。AUC 只看排序,不看校准,广告计费照样可能错。
2. **"点击率涨了就该上线"** —— 错。可能是标题党,要看转化和 GMV。
3. **"eCPM 高的广告对平台一定好"** —— 短期是,但若 pCTR 高估,长期坑广告主、伤生态。
4. **"二价拍卖平台吃亏"** —— 短期少收,长期因出价真实、生态稳而总收入更高。
5. **"转化率高的渠道就是好渠道"** —— 小心辛普森悖论和样本量,可能是人群结构差异。
6. **"负采样只是为了训练快,不影响结果"** —— 错。不做概率还原,pCTR 整体偏高,计费全错。
7. **"GMV 涨了就是赚了"** —— GMV 不是收入,要看退款、补贴、获客成本后的净值。
8. **"人均时长涨了用户更爱了"** —— 可能是少数重度用户拉的,看分位数。

## 平台侧落地 Checklist

- [ ] 每个指标有**唯一计算口径**,并有口径版本与血缘(谁定义、含不含退款/异常流量)。
- [ ] 模型层指标(AUC/GAUC/校准)有**统一评估库**,算法与平台共用同一套。
- [ ] 上线了**校准监控大盘**:分桶 pCTR vs 后验 CTR,偏差超阈值告警。
- [ ] **负采样比例随样本透传**到模型侧,概率还原逻辑落地并可验证。
- [ ] **train-serving skew 监控**:离线特征与线上特征一致性比对。
- [ ] **特征穿越检测**:样本时序校验,禁止未来信息泄漏。
- [ ] label 生成链路明确:曝光/点击/转化 join key、去重、**归因窗口**、延迟反馈补回。
- [ ] 计费链路**幂等 + 防作弊**(去重点击、异常流量过滤)。
- [ ] pCTR 预估服务有**延迟监控、降级与兜底分**。
- [ ] 实验平台支持**主指标(OEC)事前声明 + 分层分析 + 显著性检验**。
- [ ] 关键指标报表默认带**分位数(P50/P90)**,不只给均值。

## 对接话术

**对算法同学**:
> "你这个 AUC 涨了 3 个千分点,**校准曲线**动了吗?上线后我这边监控的**线上 pCTR vs 后验 CTR** 别偏。还有负采样比例记得透传给我,不然计费会整体偏高。"

**对商业化/策略同学**:
> "你要加这个广告位,eCPM 预期涨多少我能配合算。但**留存的 AB** 麻烦也挂上,我们用 LTV 看,不只看短期收入。oCPC 的话,转化样本回收窗口我按几天配?"

**对数据/报表同学**:
> "这个 GMV 口径含不含退款和跨境?定下来后我**版本化**,别中途改,不然所有依赖它的实验结论得重算。"

## 向上沟通

**当老板问"算法说 AUC 涨了,GMV 能涨多少?"**
> "AUC 是模型实验室指标,涨不等于线上涨,更不直接等于 GMV。要看线上 AB 里**产品指标(点击率/转化率)**有没有动,再看是否传导到 GMV。中间有损耗。建议等 AB 的 GMV 显著性结果,我这边保证口径一致、校准没问题。"

**当老板问"为什么这个月广告收入和预测对不上?"**
> "大概率是**校准偏移**。eCPM = pCTR × 出价,如果 pCTR 系统性高估(可能因采样未还原或分布漂移),预期收入就会虚高于实收。我已上线分桶校准监控,正在定位是哪一层。"

**当老板说"加两个广告位,收入立刻涨"**
> "短期 eCPM 会涨,但要警惕**借高利贷**:广告多了伤体验,留存掉则 LTV 缩水,长期可能净亏。建议先做留存 AB,用 LTV/CAC 算总账再决定。"

## 一图总结 + 一句话速记

```
 ┌──────────────────────────────────────────────────────────────┐
 │  业务层  GMV / 收入 / DAU / LTV / ARPU / 留存   ← 老板语言      │
 │     ▲  传导有损耗、有时滞,别划等号                            │
 │  产品层  点击率 / 转化率 / 时长 / 留存          ← 产品语言      │
 │     ▲  线上 AB 才算数,离线指标只是体检报告                    │
 │  模型层  CTR / CVR / AUC / GAUC / 校准          ← 算法语言      │
 └──────────────────────────────────────────────────────────────┘
       核心公式:eCPM = pCTR × bid × 1000(广告通用货币)
       竞价:eCPM 排序 + GSP 二价(报真实价值最优)
       计费命门:校准(pCTR 准不准)= 平台研发的钱与信任
       平台三件事:口径对 + label 对 + 校准对
```

**一句话速记**:**模型给概率、拍卖定价格、校准管对账;平台研发不调模型,但样本、label、校准这三样错了,别人的钱和信的数就全错。**

## 自测三问

**Q1:算法同学说"离线 AUC 从 0.71 涨到 0.73",你作为平台研发应该追问什么?为什么?**

<details>
<summary>参考答案</summary>

至少追问三点:① **线上 AB 跑了吗,哪个产品指标动了?**(AUC 是离线排序指标,涨不等于线上点击率/转化率涨,可能有分布偏移或特征穿越)② **校准动了吗?**(AUC 只看排序,把所有 pCTR 乘个常数 AUC 不变,但计费会全错;广告场景校准比 AUC 更要命)③ **特征有没有穿越?**(离线 AUC 虚高的头号原因是用了线上拿不到的未来特征)。本质:离线指标 ≠ 线上效果 ≠ 计费正确,三者要分开验证。
</details>

**Q2:`eCPM = pCTR × bid × 1000` 中,如果 pCTR 被系统性高估 1.2 倍,会发生什么连锁反应?**

<details>
<summary>参考答案</summary>

① eCPM 虚高,这条广告"看起来"更值钱;② 排序错位,实际不值钱的广告被排到前面(劣币驱逐良币);③ 二价计费折算下,广告主实际单价被推高,**多扣钱**;④ 广告主 ROI 变差 → 投诉、降预算、跑路 → 平台长期收入受损。根因常见于**负采样后未做概率还原**或**数据分布漂移**。平台研发要靠分桶校准监控(平均 pCTR vs 后验 CTR)发现并定位。
</details>

**Q3:某新策略在"高活用户"和"低活用户"两个分组里转化率都比老策略高,但整体转化率却更低。这是什么现象?平台研发应该怎么支撑分析?**

<details>
<summary>参考答案</summary>

这是**辛普森悖论**:新策略把更多流量导向了本身转化率就低的人群,导致加权后的整体指标反而下降,尽管每个子组都更好。平台研发应在实验平台支持**分层(stratified)分析**——按关键维度(活跃度、人群、渠道)拆开看,并监控各组的**流量/人群构成是否变化**。只看总指标会得出完全相反的错误结论。
</details>
