# 搜广推三业务本质与漏斗架构

> **一句话定位**：搜索、广告、推荐是同一套"排序机器"的三种业务皮肤——本质都是"在算力和延迟预算内,从海量候选里挑出最该给用户看的那几条",而平台研发的工作,就是给这台机器供血(样本)、供料(特征)、铺路(算力链路)。
> **写给谁**：不亲自调模型,但要在算法、训练同学和老板之间做"翻译官"和"接得住需求"的平台研发工程师。
> **读完你能**:看懂搜广推三者的本质差异,讲清"为什么都归到排序"、漏斗每一阶段(召回/粗排/精排/重排/混排)的候选量级、算力预算、样本与特征要求,并能把这些翻译成平台侧要支撑的具体能力。

## 0. 30 秒速通(TL;DR)

如果只给你 30 秒,记住这几条:

1. **三业务的根本差异在"用户意图强弱"**:搜索有明确 query(强意图)、推荐没有明确意图(平台猜)、广告夹在中间且多了个出钱的金主(广告主)。意图越强,相关性权重越高;意图越弱,平台越有"塑造"空间。
2. **三业务的目标函数不同**:搜索优化相关性+满意度、推荐优化时长/留存/生态、广告优化收入(同时受三方博弈约束)。但它们底层都退化成"预估某个概率(CTR/CVR/时长)然后排序"。
3. **"排序"是统一抽象**:不管哪个业务,系统都在对一个候选列表打分排序。所以底层引擎、样本流、特征体系可以大量复用——这是平台能统一支撑的根本原因。
4. **漏斗是工程对算力的妥协**:候选从"亿"级一路收敛到用户看到的"十"级,分成**召回 → 粗排 → 精排 → 重排 → 混排**。越往上候选越少、模型越重、延迟预算越紧。
5. **每一阶段对平台的要求都不同**:召回要的是海量候选索引+多路负采样样本;精排要的是曝光样本+实时特征;重排要的是 list-wise 上下文与多样性策略。**平台研发的核心活儿,就是为不同阶段供给不同的样本流、特征和候选规模。**
6. **冷启动**(新用户/新物料/新广告)是另一条主线:靠 side info(画像/内容特征)+ 探索(E&E)补行为数据的空缺,平台要支撑"探索流量"的分配与回收。

> 🎓 **老师视角**:你不需要会推导 DIN 或 DeepFM,但你必须能在白板上画出这张漏斗图,并指着每一层说出"这层候选多少、延迟多少毫秒、用什么样本、谁负责"。这是平台研发的"作战地图",画不出来,需求评审时你就会被算法同学牵着走。

## 1. 三业务本质:意图、目标、博弈

### 1.1 一句话区分搜、广、推

把这三个业务摆在"用户意图强弱"这根轴上,一切就清晰了:

```
 用户意图强 ◄──────────────────────────────────────────► 用户意图弱
 │                          │                              │
 搜索 (Search)            广告 (Ads)                   推荐 (Rec)
 "我要找运动鞋"          "你可能想买/平台想卖"          "随便逛逛,你猜我想看啥"
 query 表达明确          有时有 query(搜索广告)        无 query
                         有时无 query(信息流广告)
 ───────────────────────────────────────────────────────────────
 相关性是天条            收入是直接目标               长期生态是目标
                         (但不能伤用户体验)           (时长/留存/多样性)
```

- **搜索**:用户主动输入 query,意图被显式表达。系统的天职是"理解 query → 找到最相关的结果"。用户对"不相关"零容忍——搜"运动鞋"出来"洗发水",用户立刻流失。
- **推荐**:用户没有明确表达想要什么(信息流、首页猜你喜欢、短视频流)。系统要从历史行为、上下文、相似人群里**猜**意图。容错度比搜索高(你刷到一个不感兴趣的视频,划走就行),但要对长期留存负责。
- **广告**:本质是"付费的推荐/搜索结果"。它**夹在中间**——既可能挂在搜索结果里(搜索广告,如电商关键词广告),也可能混在信息流里(原生信息流广告)。它多了一个角色:**出钱的广告主**。

### 1.2 目标函数的差异

| 业务 | 核心优化目标 | 典型代理指标 | 约束条件 |
|------|------------|------------|---------|
| 搜索 | 相关性 + 用户满意度 | CTR、停留、query 改写率、首条点击率 | 时延、相关性下限 |
| 推荐 | 长期价值(留存/时长/生态) | CTR、时长、完播、互动、次留 | 多样性、健康度、信息茧房 |
| 广告 | 平台收入(eCPM 最大化) | eCPM = CTR × CVR × bid(出价) | 用户体验、广告主 ROI、预算约束 |

> 关键洞察:搜索和推荐优化的是"用户侧的好",广告优化的是"平台收入",但**收入又不能透支用户体验和广告主 ROI**——否则用户跑了、广告主不投了,生态崩盘。所以广告是个典型的三方博弈,见 1.3。

### 1.3 广告的三方博弈

```
              ┌───────────────────────────────┐
              │          平 台 (Platform)       │
              │  目标:长期收入最大化 + 生态健康   │
              └───────────┬───────────────────┘
                          │ 撮合 / 拍卖 (Auction)
            ┌─────────────┴──────────────┐
            ▼                             ▼
   ┌─────────────────┐          ┌──────────────────┐
   │   用户 (User)     │          │  广告主 (Advertiser)│
   │ 想看相关/有用内容  │◄────────►│  想要转化/低成本获客 │
   │ 体验差就流失       │   广告    │  ROI 差就停投       │
   └─────────────────┘          └──────────────────┘
```

- **用户**:希望少看广告、看到的广告也要相关。体验崩了就用脚投票。
- **广告主**:希望花最少的钱换最多的转化(ROI)。平台坑了广告主,广告主停投,收入归零。
- **平台**:要在两者之间撮合,通过**竞价机制(Auction)**让广告位分配给"出价 × 质量"最高的广告,同时长期不能竭泽而渔。

> 🎓 **老师视角**:很多平台研发只把广告当"另一个推荐",这是大坑。广告系统有一整套搜推没有的东西:**计费(billing)、预算控制(budget pacing)、出价(bid)、竞价/拍卖(GSP/VCG)、广告主结算**。这些直接关系到钱,链路对一致性和准确性的要求远高于搜推——预估算错可以慢慢迭代,**计费算错是要赔钱/打官司的**。当算法同学说"复用推荐的样本流就行",你要追问:"广告的计费日志、归因窗口、反作弊过滤,这套流你打算怎么接?"

### 1.4 意图强弱如何反推到技术选型

| 维度 | 搜索 | 推荐 | 广告 |
|------|------|------|------|
| 意图来源 | query(显式) | 历史行为+上下文(隐式) | 兼有,且叠加广告主定向 |
| 相关性权重 | 极高(硬约束) | 中(软约束) | 中高(不相关广告 ROI 也差) |
| 探索空间 | 小(不能乱给) | 大(可大胆探索兴趣) | 中(探索受预算约束) |
| 第三方角色 | 无 | 无 | 广告主(出钱方) |
| 对平台研发的额外要求 | query 理解链路、倒排索引 | 实时行为流、兴趣探索 | 计费、竞价、预算、归因 |

> 🎓 **老师视角**:意图越弱,平台对"给什么"的话语权越大,探索(exploration)空间也越大。这就是为什么推荐系统敢做大规模兴趣探索,而搜索不敢——搜"身份证补办"你给人家推个游戏,那是事故。平台研发要理解这个差异,因为它决定了**探索流量的预算**该给哪个业务多、哪个业务少。

### 1.5 一个常被忽略的差异:反馈信号的"密度与延迟"

三业务不光意图不同,**用户反馈信号的密度和延迟**也大相径庭,而这直接决定平台样本链路的设计:

```
                 反馈信号                  延迟           对样本链路的影响
 搜索:  点击 / 停留 / query 改写        秒级(即时)      样本可近实时回流
 推荐:  点击 / 完播 / 时长 / 互动       秒~分钟           短窗口归因即可
 广告:  点击(秒) → 转化(小时~天)     极长(跨天)       归因窗口长,样本要"等"
```

- 搜索的反馈最即时:用户点没点、改没改 query,几秒内就知道。
- 推荐的反馈也较快:完播、停留时长在几秒到几分钟内确定。
- **广告最麻烦**:点击是秒级的,但"转化"(下单、激活、注册)可能在点击后**几小时到几天**才发生。这意味着广告的 CVR 样本要等一个很长的**归因窗口(attribution window)**才能确定 label,样本链路要支持"延迟 label 回填"。

| 业务 | label 延迟(经验值) | 样本链路设计要点 |
|------|---------------------|-----------------|
| 搜索 | 秒级 | 近实时样本流即可满足 |
| 推荐 | 秒~分钟 | 短归因窗口,实时/近线回流 |
| 广告(CTR) | 秒级 | 同推荐 |
| 广告(CVR) | 小时~天 | 长归因窗口、延迟 label 回填、样本去重 |

> 🎓 **老师视角**:广告 CVR 的"长延迟归因"是平台研发最容易低估的工程量。你不能像推荐那样"曝光后几分钟拼好样本就发车",而要维护一个**延迟 label 的回填机制**——曝光样本先落盘等着,转化信号在几小时/几天后到达再回填 label。期间还要处理:归因口径(首次点击归因还是末次?)、跨设备归因、重复转化去重。这套机制如果照搬推荐的短窗口设计,广告 CVR 模型的样本会大面积"漏 label"(把真转化当成了未转化),模型直接学歪。**所以遇到广告需求,第一句就该问:归因窗口多长?label 延迟回填怎么做?**

## 2. 为什么都归到"排序"问题

### 2.1 三业务的统一抽象

剥掉业务皮肤,三者在系统层面做的是同一件事:

```
输入: 一个 (用户 user, 上下文 context, 候选集 candidates[]) 
输出: 候选集按"某个分数"排序后的有序列表 ranked_list[]

搜索: 分数 ≈ 相关性(query, item) × 满意度预估
推荐: 分数 ≈ f(CTR, 时长, 互动, ...) 的多目标融合
广告: 分数 ≈ eCPM = pCTR × pCVR × bid   (排序即定价的基础)
```

无论哪个业务,系统都在**对一个候选列表打分(score)并排序(sort)**。而打分的核心,几乎都落在**预估某个概率**上:

- **pCTR**(predicted Click-Through Rate,点击率预估):用户点这条的概率。
- **pCVR**(predicted Conversion Rate,转化率预估):点了之后转化(购买/下载/注册)的概率。
- **时长/完播/互动预估**:推荐里更看重的目标。

> 排序(ranking)= 预估(prediction)+ 比较(comparison)。预估是核心,排序是结果。

### 2.2 这个统一抽象对平台意味着什么

因为三业务底层都是"特征 → 模型 → 打分 → 排序",所以:

| 可复用的层 | 复用程度 | 说明 |
|-----------|---------|------|
| 特征体系(feature store) | 高 | 用户/物料/上下文特征大量共用 |
| 样本生产框架(sample join) | 高 | 曝光/点击/转化日志拼接逻辑相似 |
| 训练框架(training infra) | 高 | 同一套分布式训练/参数服务器 |
| 在线推理(serving) | 高 | 同一套打分服务、模型加载、AB 框架 |
| 业务专属逻辑 | 低 | 广告计费/竞价、搜索 query 理解需独立 |

> 🎓 **老师视角**:"都是排序问题"这句话,是平台研发能用一套基础设施同时服务搜广推的根本理由,也是你向老板论证"为什么要建统一平台、而不是三个团队各搞各的"的核心论据。但要诚实:**统一的是引擎和数据流,不统一的是业务上层逻辑**。别把"统一平台"吹成"一套代码全搞定",评审时会被算法同学当场拆穿。

### 2.3 排序 ≠ 只看预估分

排序虽然以预估为核心,但最终展示顺序还会被一堆"业务规则"修正:

```
预估分 (pCTR/pCVR/...)
   │
   ├─ × 出价 bid           ← 广告:排序即定价
   ├─ × 多目标融合权重       ← 推荐:CTR×时长×互动 加权
   ├─ + 多样性 / 打散惩罚    ← 重排:别连着 5 个同类
   ├─ + 业务加权 / 扶持      ← 新内容扶持、商业化策略
   └─ + 强插 / 置顶规则      ← 运营干预
   ▼
最终展示顺序
```

> 这一步往往是平台研发"踩坑最多"的地方:算法同学交付的是预估分,但线上最终顺序被无数策略规则改写。**当业务方说"线上效果和离线评估对不上",八成是这堆规则在中间动了手脚。** 平台要提供"全链路打分可追溯(score trace)"的能力,把每一步的加权/惩罚记录下来,否则排障时你只能干瞪眼。

### 2.4 排序的三种范式:point/pair/list-wise

排序在学术和工程上有三种"学习目标"的范式,理解它直接关系到你给不同阶段供什么样本:

```
 point-wise(逐点):  对每个候选独立预估一个分,再排序
                     样本单位 = 单条 (user, item, label)
                     用在:召回、粗排、精排 主流做法
                     特点:简单、可并行、海量样本友好

 pair-wise(逐对):   学习"A 应该排在 B 前面"
                     样本单位 = 一对 (item_A > item_B)
                     用在:部分搜索排序(相对序更重要)
                     特点:更贴近"排序"本质,样本构造成本高

 list-wise(逐列表):  直接优化整个列表的好坏
                     样本单位 = 一个列表 + 用户对列表的整体反馈
                     用在:重排(re-rank)、生成式排序
                     特点:能建模 item 间相互影响,样本最贵
```

| 范式 | 样本单位 | 平台供给难度 | 典型阶段 |
|------|---------|------------|---------|
| point-wise | 单条 (user,item,label) | 低(主流样本流) | 召回/粗排/精排 |
| pair-wise | 一对 item | 中(要构造对) | 部分搜索排序 |
| list-wise | 整个列表 | 高(要拼 list 级样本) | 重排/生成式 |

> 🎓 **老师视角**:"排序范式"听起来很学术,但它对平台是实打实的样本形态差异。point-wise 你只要一行行的曝光样本;list-wise 你要把"一次请求里展示的整个列表 + 用户对每个位置的反馈"打包成一条样本——这是完全不同的拼接逻辑和存储结构。当重排团队说"我们要做 list-wise 重排",你心里要立刻响起警报:**这不是改个模型那么简单,是要我新建一条 list 级样本流。** 别等到开发到一半才发现样本平台喂不出 list 样本。

## 3. 多阶段漏斗架构总览

### 3.1 为什么要分阶段(漏斗)

理想情况下,我们想用最好的模型给**每一个**候选打分。但现实是:

- 候选库可能有**上亿**(全站商品/视频/内容)。
- 用户一次请求的延迟预算只有**几十到几百毫秒**。
- 最好的精排模型给一个候选打分要几毫秒——给一亿个候选打分?算力直接爆炸。

于是工程上做了一个经典妥协:**分级筛选**。用便宜的模型快速从海量里粗筛,留下少量候选,再用昂贵的模型精细打分。这就是漏斗(funnel)。

### 3.2 漏斗全景图

```
        候选量级(典型/经验值)        阶段              模型复杂度    延迟预算(经验值)
 ┌──────────────────────────────────────────────────────────────────────────┐
 │  亿级 (10^8)         全量候选库 (item pool)                                  │
 │       │                                                                    │
 │       ▼  ┌──────────────────┐                                              │
 │  千级  │   召回 Recall       │  多路召回,轻量/向量检索   ~10-30 ms          │
 │ (10^3)│  亿 → 千            │  双塔/i2i/向量/规则        (多路并发)          │
 │       ▼  └──────────────────┘                                              │
 │  百级  ┌──────────────────┐                                                │
 │ (10^2)│   粗排 Pre-Rank     │  轻量模型(双塔/浅层)      ~10-20 ms          │
 │       │  千 → 百            │  打分快、精度够用           │                 │
 │       ▼  └──────────────────┘                                              │
 │  十级  ┌──────────────────┐                                                │
 │ (10^1)│   精排 Rank         │  重模型(DIN/DeepFM/多目标) ~20-50 ms         │
 │       │  百 → 十            │  全量特征、交叉、序列建模    │                 │
 │       ▼  └──────────────────┘                                              │
 │  ~10  ┌──────────────────┐                                                 │
 │       │   重排 Re-Rank      │  list-wise,多样性/打散      ~5-10 ms          │
 │       │  调整最终顺序        │  上下文感知、生成式重排      │                 │
 │       ▼  └──────────────────┘                                              │
 │  最终  ┌──────────────────┐                                                │
 │ 展示  │   混排 Mixed-Rank   │  广告与自然结果混合          ~3-5 ms          │
 │       │  插入广告/多业务流    │  广告位分配、流量分发         │                │
 │       ▼  └──────────────────┘                                              │
 │  用户看到的最终列表 (~10-20 条)                                              │
 └──────────────────────────────────────────────────────────────────────────┘

 注:量级和延迟均为"经验值/典型量级",不同公司、不同场景差异很大。
     纯推荐流不一定有混排;纯搜索可能没有独立粗排;广告会插在自然流的混排环节。
```

### 3.3 漏斗各阶段速查表

| 阶段 | 输入→输出量级(经验值) | 核心问题 | 模型典型 | 延迟预算(经验值) |
|------|---------------------|---------|---------|----------------|
| 召回 Recall | 亿 → 千 | 别漏掉好的(高召回) | 双塔、i2i、向量 ANN、规则 | 10-30 ms(多路并发) |
| 粗排 Pre-Rank | 千 → 百 | 快速粗筛,精度够用 | 双塔、浅层 DNN | 10-20 ms |
| 精排 Rank | 百 → 十 | 排准(高精度) | DIN/DeepFM/多目标重模型 | 20-50 ms |
| 重排 Re-Rank | 十 → 十(调序) | 整体最优、多样性 | list-wise、PRM、生成式 | 5-10 ms |
| 混排 Mixed-Rank | 多流 → 一流 | 广告/自然结果混合 | 规则+轻量模型 | 3-5 ms |

> 🎓 **老师视角**:漏斗每往上一层,**候选量降一个数量级,单条打分的算力预算就能升一个数量级**。这是个守恒游戏:总算力 ≈ 候选量 × 单条算力。所以你会看到召回模型简单到只是个向量内积,精排模型复杂到几百个特征+注意力机制——不是召回团队懒,是算力不允许它复杂。理解这个守恒,你才能在容量评估时算清楚账:"精排 QPS 翻倍,机器要加多少?"

## 4. 召回(Recall):亿到千

### 4.1 召回在干什么

召回的目标是从**全量候选库(亿级)**里快速捞出一个**几百到几千**的候选子集,送给后面的排序。核心 KPI 是**召回率(recall)**——好东西别漏掉。它不追求把顺序排准(那是精排的活),只追求"该捞的都捞到了"。

### 4.2 多路召回(Multi-Channel Recall)

实践中没有哪一路召回能覆盖所有用户需求,所以业界普遍用**多路召回并行 + 合并去重**:

```
         ┌─────────────────────────────────────────────┐
         │              用户请求 (user, context)          │
         └───────────────────────┬─────────────────────┘
            ┌──────────┬─────────┼─────────┬──────────┐
            ▼          ▼         ▼         ▼          ▼
        ┌───────┐ ┌────────┐ ┌──────┐ ┌────────┐ ┌────────┐
        │ 协同   │ │ 双塔向量 │ │ i2i  │ │ 热门/  │ │ 规则/  │
        │ 过滤   │ │ ANN 检索 │ │ 相似 │ │ 兜底   │ │ 运营   │
        │ (CF)  │ │(Two-Tower)│ │ 物品 │ │       │ │ 强插   │
        └───┬───┘ └────┬────┘ └──┬───┘ └───┬────┘ └───┬────┘
            └──────────┴────┬────┴─────────┴──────────┘
                            ▼
                  ┌──────────────────┐
                  │  合并 + 去重 + 配额  │  (每路给定配额, merge)
                  └────────┬─────────┘
                            ▼
                  候选子集 (几百~几千)
```

- **协同过滤(CF)**:看过这个的人还看了什么。
- **双塔向量召回(Two-Tower + ANN)**:用户塔和物品塔各出一个向量,在线用近似最近邻(ANN,如 HNSW/IVF)检索。这是当今工业界主力。
- **i2i(item-to-item)**:基于物品相似度,从用户最近行为的物品扩展。
- **热门/兜底**:保证任何情况下都有结果返回。
- **规则/运营**:强插、扶持新内容。

### 4.3 召回的样本与平台要求(重点)

召回最反直觉、也最坑平台研发的一点:**召回的负样本不能只用曝光未点击,必须做负采样**。

```
精排负样本:  曝光了但用户没点  ← 这些是"展示过的"
召回负样本:  全库随机采 + 曝光未点 ← 因为召回要学"从全库里区分好坏",
                                    只用曝光样本会让模型"没见过世面"
```

为什么?因为召回面对的是**全量候选库**,而曝光样本只是经过层层漏斗筛选后的"幸存者"——它们都是相对"好"的。如果只用曝光样本训练召回,模型从没见过"全库里那些根本不相关的东西",上线后就会把一堆离谱候选捞上来。所以召回必须**从全库随机负采样(random negative sampling)**,让模型见过"真正的烂候选"。

| 召回对平台的供给要求 | 具体内容 |
|---------------------|---------|
| 全库候选索引 | 物品向量库、ANN 索引的构建与实时更新 |
| 负采样能力 | 全库随机采样、batch 内负采样、流式负采样 |
| 多路并发调度 | 多路召回并行、超时降级、配额管理 |
| 向量服务 | 在线向量检索引擎(ANN)、向量版本管理 |
| 实时物料入库 | 新物品分钟级进入召回库(冷启动相关) |

> 🎓 **老师视角**:负采样是平台研发和算法之间最容易扯皮的地方。算法同学要"全库随机负采样",意味着平台要在样本生产链路里支持"从亿级物料池高效采样并拼特征"——这跟精排"用现成曝光日志拼样本"完全是两套数据流。如果你一开始只按精排那套设计样本平台,等召回团队来提需求时你会推倒重来。**所以样本平台的设计,第一天就要把"召回负采样流"和"精排曝光流"当成两条独立管道来规划。** 负采样和样本偏差的细节,见 [`../04-样本数据管理/03-负采样与样本偏差治理.md`](../04-样本数据管理/03-负采样与样本偏差治理.md)。

### 4.4 双塔召回的工程要点(平台最该懂的一路)

双塔(Two-Tower)是当今工业界召回的主力,平台研发要重点理解它的工程形态,因为它对在线服务的要求最特殊:

```
 离线/近线侧                         在线侧
 ┌─────────────┐                 ┌──────────────────────────┐
 │ 物品塔(item) │ ── 算出所有 ──► │ 物品向量库 (vector index) │
 │ 把物品编码成  │    物品向量      │  HNSW / IVF 等 ANN 索引    │
 │ 一个向量      │    (离线批算)    └────────────┬─────────────┘
 └─────────────┘                              │ ANN 检索
 ┌─────────────┐                              │ (取 top-K 最近邻)
 │ 用户塔(user) │ ── 在线实时 ──► 用户向量 ─────┘
 │ 把用户编码成  │    (打分时算)
 │ 一个向量      │
 └─────────────┘
```

关键工程点:
- **物品向量离线/近线预计算**:物品塔的输出对每个请求是不变的,所以可以离线批量算好,在线只需查索引。这是双塔"快"的根本原因。
- **用户向量在线实时计算**:用户塔的输入随上下文变化,必须在线算。
- **ANN 索引(HNSW/IVF)**:用近似最近邻把"亿级向量里找 top-K"压到毫秒级。索引的构建、更新、版本切换都是平台的活。
- **双塔的天生短板**:用户塔和物品塔分离,**两者之间没有特征交叉**(只在最后做内积),所以精度天花板低于精排——这是它换取速度付出的代价,也是为什么双塔只用在召回/粗排而非精排。

| 双塔召回对平台的特殊要求 | 说明 |
|------------------------|------|
| 物品向量批量产出 | 离线/近线算全库物品向量,产出量 = 全库物品数 |
| 向量索引构建与热更新 | 新物品向量增量入索引,索引版本与模型版本对齐 |
| 用户向量在线推理 | 用户塔在线低延迟出向量 |
| 向量与模型版本一致性 | 物品向量、用户塔、索引必须同版本,错版本=乱召回 |

> 🎓 **老师视角**:双塔召回最隐蔽的线上事故是**"版本错配"**:模型更新了用户塔,但物品向量还是旧版本算的,两套向量不在同一空间,内积出来的"相似度"全是噪声,召回质量断崖下跌且不报错(系统照常返回结果)。所以平台必须把"用户塔模型版本 + 物品向量版本 + ANN 索引版本"绑成一个原子单元一起切换。这种"静默故障"最难查——监控指标得专门盯召回质量(如 top-K 命中曝光的比例),否则你只会看到大盘缓慢下滑却找不到原因。

### 4.5 多路召回的合并与配额

多路召回回来后要合并,这里有几个平台要支撑的策略点:

```
 协同过滤路 ─┐  配额 200
 双塔向量路 ─┤  配额 500    ┌──────────────┐
 i2i 路    ─┼─────────────►│ 去重 + 配额裁剪 │──► 合并候选(~1000)
 热门兜底路 ─┤  配额 100    │ + 截断 + 降级   │
 运营强插路 ─┘  配额 50     └──────────────┘
```

- **配额(quota)**:每路给多少名额,直接影响最终候选的"成分构成",是个需要 AB 调优的策略参数。
- **去重(dedup)**:多路可能召回同一物品,要去重。
- **超时降级(timeout fallback)**:某路慢了/挂了,不能拖垮整体,要能快速降级(返回已有的、或走兜底路)。

> 🎓 **老师视角**:多路召回的"配额"看似小事,实则是业务话语权的分配:商业化想多给广告路配额、生态想多给新内容路配额、算法想多给主模型路配额。平台要把配额做成**可配置、可 AB、可监控每路贡献**,否则各方在你的代码里硬塞 if-else,半年后没人敢动这段召回合并逻辑。**让配额"数据化、可观测"是平台的护城河。**

## 5. 粗排(Pre-Rank):千到百

### 5.1 粗排的定位:夹心层

粗排夹在召回(千级)和精排(百级)之间,是个"承上启下"的过渡层:

```
召回(千) ──► 粗排(千→百) ──► 精排(百→十)
              │
              └─ 既要比召回准,又要比精排快
                 算力预算介于两者之间
```

为什么不直接召回→精排?因为召回给出的几千个候选,直接全塞给精排重模型,**精排扛不住**(几千 × 几毫秒 = 几十毫秒甚至更多,延迟爆)。粗排就是为了把"几千"先砍到"几百",让精排的输入规模可控。

### 5.2 粗排的核心矛盾:一致性 vs 算力

粗排最难的地方在于:它和精排目标一致(都想排准),但算力差好几倍。如果粗排和精排的排序结果差异太大,会出现**"粗排截断了精排本想要的好候选"**——这叫**粗精排一致性问题**。

| 方案 | 思路 | 优点 | 缺点 |
|------|------|------|------|
| 双塔粗排 | 用户塔/物品塔分离,向量内积打分 | 极快,物品向量可预计算 | 缺乏特征交叉,精度有限 |
| 浅层 DNN | 轻量全连接网络 | 比双塔有交叉,精度更高 | 比双塔慢 |
| 蒸馏(distillation) | 用精排模型蒸馏粗排 | 提升粗精排一致性 | 训练链路更复杂 |

### 5.3 粗排对平台的要求

| 供给要求 | 说明 |
|---------|------|
| 物品向量预计算 | 双塔粗排的物品向量离线/近线批量算好,在线只查 |
| 轻量打分服务 | 比精排更高 QPS、更低延迟的打分链路 |
| 粗精排一致性监控 | 监控粗排 top-N 与精排 top-N 的重合度 |
| 蒸馏样本流 | 若用蒸馏,需要精排打分作为软标签喂给粗排 |

> 🎓 **老师视角**:很多中小场景**没有独立粗排**,召回直接接精排——候选量不大时,这完全合理,别为了"架构完整"硬造一层粗排。粗排是当候选量大到精排扛不住时才引入的。所以你做容量规划时的判断准则是:**"召回输出量 × 精排单条耗时 > 延迟预算"** 才需要粗排。能用一句话讲清"我们为什么需要/不需要粗排",你在架构评审里就是清醒的那一个。

## 6. 精排(Rank):百到十

### 6.1 精排是"主力部队"

精排面对几百个候选,用**最重的模型、最全的特征**精细打分,选出 top-10 左右。这是整个系统投入算力和模型复杂度最高的地方,也是算法同学迭代最频繁的战场。

```
 输入: 几百个候选 + 全量特征(用户画像、物品属性、上下文、
                          用户行为序列、交叉特征 ...)
   │
   ▼  重模型 (DIN / DeepFM / DCN / 多目标 MMoE / ...)
   │  - 行为序列建模 (attention)
   │  - 特征交叉
   │  - 多目标预估 (pCTR, pCVR, 时长 ...)
   ▼
 输出: 每个候选的多个预估分 → 融合 → 排序 → top-10
```

### 6.2 精排的样本:曝光样本(重点)

精排的训练样本是**曝光样本(impression samples)**:

```
正样本: 曝光 + 点击(+ 转化)
负样本: 曝光 + 未点击
特征:   打分时刻的实时特征快照(feature snapshot)
```

这与召回完全不同:精排只关心"在已经展示给用户的候选里,排得准不准",所以用曝光日志做样本天经地义。但这里藏着一个平台研发必须懂的坑:**特征穿越(feature leakage)**。

```
错误做法: 训练时用"事后特征"(如:这个物品当天的总点击数)
          → 线上打分时根本拿不到"当天总点击数"(还没发生)
          → 离线指标虚高,上线暴跌

正确做法: 记录"打分那一刻"的特征快照,训练用快照
          → 保证训练/推理特征一致 (train-serving consistency)
```

| 精排对平台的供给要求 | 说明 |
|---------------------|------|
| 曝光样本拼接 | 曝光/点击/转化多日志 join,处理归因延迟 |
| 实时特征快照 | 打分时落特征快照,杜绝特征穿越 |
| 在线实时特征 | 用户实时行为序列、上下文特征的低延迟读取 |
| 训练-推理一致性 | 同一套特征计算逻辑离线/在线对齐 |
| 高性能打分服务 | 几百候选 × 重模型,在几十毫秒内完成 |

> 🎓 **老师视角**:"训练-推理一致性(train-serving consistency)"是平台研发的**头号生命线**。算法同学离线 AUC 涨了 2 个点,上线却没动甚至跌——十有八九是特征在离线和在线算得不一样(特征穿越、口径不一致、时间窗对不齐)。这个锅平台研发很难甩,因为特征链路是你建的。**所以特征快照(feature snapshot / feature logging)能力,是精排平台的第一优先级,没有它,所有离线评估都是耍流氓。** 模型本身的演进史(从 LR 到 DIN 到生成式),见 [`../06-模型演进与沟通话术/01-搜广推排序模型演进史.md`](../06-模型演进与沟通话术/01-搜广推排序模型演进史.md)。

### 6.3 一个脱敏案例:特征穿越是怎么发生的

讲个真实味道的脱敏故事,帮你建立"嗅觉":

```
算法同学加了个新特征:"item_ctr_today"(物品当天的实时点击率)
离线训练:用 T 日的全天日志算这个特征 → AUC +1.5%,大喜
上线:T 日中午用户请求时,"当天点击率"只统计到中午,
      和训练用的"全天点击率"口径完全不同
      → 训练时模型见到的是"上帝视角的全天数据"
      → 上线时拿到的是"只到此刻的半天数据"
      → 离线虚高,上线归零
```

根因:训练特征用了**包含未来信息**的统计口径(全天),而线上打分时此刻还没到"全天"。这就是典型的**特征穿越(leakage)**——它不是 bug,是数据口径的时间对齐问题,极其隐蔽。

正确做法:

```
打分时刻 (T 日 12:00),把当时实际能拿到的特征值
            ("item_ctr_截止12:00") 落成快照,记到样本里
            ↓
训练时直接用这个快照,而不是回头去算"全天 ctr"
            ↓
保证:训练见到的 = 线上能拿到的  (train-serving 一致)
```

| 穿越类型 | 表现 | 平台的防御手段 |
|---------|------|---------------|
| 时间穿越 | 用了打分时刻之后才有的统计值 | 特征快照(只记当时值) |
| label 穿越 | 特征里混进了 label 信息 | 特征/label 来源审计 |
| 口径不一致 | 离线/在线同名特征算法不同 | 离在线特征同源(同一套计算逻辑) |
| 时间窗错位 | 离线窗口和在线窗口不对齐 | 统一时间窗定义、对账 |

> 🎓 **老师视角**:防特征穿越的终极武器只有一个——**特征日志(feature logging):在线打分那一刻,把喂给模型的特征原样记下来,训练就用这份记录,绝不离线回算。** 这样训练和推理用的是同一份"事实",物理上杜绝穿越。代价是存储成本(每次打分都要落特征),但相比"离线评估全是假的"这个代价,完全值得。当算法同学嫌特征日志占存储想关掉时,你要顶住:**关掉它,你们所有的离线实验结论都不可信。**

## 7. 重排(Re-Rank)与混排(Mixed-Rank)

### 7.1 重排:从"逐条最优"到"整体最优"

精排是 **point-wise** 的——它独立给每个候选打分,没考虑"放在一起好不好看"。但用户看到的是一个**列表**,列表的整体体验不等于单条分数之和:

```
精排 top-5 (按分数): [篮球鞋A, 篮球鞋B, 篮球鞋C, 篮球鞋D, 篮球鞋E]
                      ↑ 单看每条都"最优",但连着 5 双篮球鞋,体验差!

重排后:            [篮球鞋A, 跑鞋X, 篮球鞋B, 运动袜Y, 篮球鞋C]
                      ↑ 牺牲一点单条分数,换整体多样性 + 上下文协同
```

重排要解决的问题:
- **多样性(diversity)**:别让列表全是一类。
- **打散(scatter)**:同类目/同作者/同品牌之间拉开距离。
- **上下文感知(context-aware)**:item 之间互相影响(list-wise 建模,如 PRM、生成式重排)。
- **业务策略**:新内容扶持、生态调控、强插置顶。

### 7.2 混排:广告与自然结果的融合

混排(也叫混合排序/广告插入)是**多业务流的汇合点**,典型是把广告插进自然推荐/搜索结果里:

```
 自然结果流 (推荐/搜索) ──┐
                         ├──► 混排引擎 ──► 最终列表
 广告流 (Ads)        ────┘    │
                              ├─ 广告位分配 (哪几个位置给广告)
                              ├─ 广告/自然结果的价值对齐 (eCPM vs 自然分)
                              └─ 体验约束 (广告密度上限、间隔)
```

混排的核心是**不同业务流"价值口径"的对齐**:自然结果的"分"和广告的"eCPM"是两套量纲,混排要把它们换算到可比较的尺度上,再决定每个位置给谁。这是商业化和体验拉扯最激烈的战场。

| 重排/混排对平台的供给要求 | 说明 |
|--------------------------|------|
| list-wise 样本 | 以"整个列表+用户反馈"为单位的样本(非单条) |
| 上下文特征 | 列表内 item 间的相对位置、共现特征 |
| 多流融合链路 | 自然流 + 广告流的低延迟合并 |
| 价值对齐配置 | eCPM 与自然分的换算、广告密度策略 |
| 策略可配置/可灰度 | 打散规则、广告位、扶持权重需热更新+AB |

> 🎓 **老师视角**:重排和混排是平台研发最容易"背锅但讲不清"的环节,因为这里塞满了**运营策略和商业化规则**。线上一个"为什么这条排第一"的问题,可能要追到混排的广告插入逻辑、重排的打散惩罚、运营的强插配置三层。**平台必须提供"端到端打分链路可追溯"——把每条 item 从召回到最终位置的每一步加权/惩罚/插入原因都记录下来。** 没有这个,你在跨团队排障会议上只能背锅。广告竞价与计费机制的细节,见 [`../01-搜广推业务场景/02-核心指标体系与广告竞价机制.md`](../01-搜广推业务场景/02-核心指标体系与广告竞价机制.md)。

## 8. 每阶段的预算与供给对照

把前面散落的信息汇成一张"作战总表",这是你最该背下来的一页:

### 8.1 算力/延迟/候选量级总对照(经验值)

| 阶段 | 候选量级 | 单条算力 | 延迟预算 | 模型复杂度 | 核心 KPI |
|------|---------|---------|---------|-----------|---------|
| 召回 | 亿→千 | 极低(向量内积) | 10-30 ms | 双塔/ANN/规则 | 召回率 |
| 粗排 | 千→百 | 低 | 10-20 ms | 双塔/浅层 DNN | 粗精一致性 |
| 精排 | 百→十 | 高(重模型) | 20-50 ms | DIN/MMoE/多目标 | AUC/线上效果 |
| 重排 | 十→十 | 中(list-wise) | 5-10 ms | PRM/生成式 | 多样性/整体收益 |
| 混排 | 多流→一流 | 低-中 | 3-5 ms | 规则+轻量 | 收入/体验平衡 |

### 8.2 样本与特征供给对照(平台研发最关心)

| 阶段 | 样本类型 | 负样本来源 | 特征要求 | 平台核心活儿 |
|------|---------|-----------|---------|-------------|
| 召回 | 点击/正反馈 | **全库随机负采样** + 曝光未点 | 用户/物品向量特征 | 全库采样链路、向量库、ANN 索引 |
| 粗排 | 曝光样本(或蒸馏) | 曝光未点 / 精排软标签 | 轻量特征子集 | 物品向量预计算、蒸馏样本流 |
| 精排 | **曝光样本** | 曝光未点击 | 全量+实时+序列特征 | **特征快照、train-serving 一致性** |
| 重排 | **list-wise 列表样本** | 列表内未交互 | 上下文/位置/共现特征 | list 级样本拼接、上下文特征 |
| 混排 | 列表+广告反馈 | — | 多流价值口径 | 多流融合、价值对齐、可追溯 |

> 🎓 **老师视角**:这张表是你设计样本平台和特征平台时的"需求清单"。注意三条独立的数据流:**(1) 召回的全库负采样流;(2) 精排的曝光快照流;(3) 重排的 list-wise 流。** 它们的数据形态、采集方式、拼接逻辑都不同。新平台最常见的失败,就是只为精排建了一条流,后来召回、重排团队各自来提需求时被迫不断打补丁,最终样本平台变成一堆 if-else 的缝合怪。**第一天就按三条流规划,远比后面重构便宜。**

### 8.3 一个脱敏案例:延迟预算怎么算

某中等规模信息流推荐场景(脱敏,数字为经验量级):

```
端到端延迟预算: ~150 ms (用户可感知阈值)
   ├─ 召回(多路并发):  ~25 ms   (候选库 ~5000万,输出 ~2000)
   ├─ 粗排:           ~15 ms   (2000 → 300)
   ├─ 精排:           ~40 ms   (300 → 30, 重模型)
   ├─ 重排:            ~8 ms   (30 → 30, 打散+多样性)
   ├─ 混排(插广告):    ~5 ms   (30 + 广告 → 最终 ~20 条)
   └─ 网络/序列化/其他: ~50 ms
```

> 这个账本的意义:当算法同学说"精排模型想加个序列特征,单条耗时从 1ms 涨到 1.5ms",你立刻能算:300 候选 × 0.5ms = 多 150ms?不,精排是 batch 打分,实际增量取决于并行度。**关键是你得有这本账,且能把模型复杂度的变化翻译成延迟和机器成本。** 这是平台研发在容量评审里的核心价值。

## 9. 冷启动:没有行为数据怎么办

### 9.1 三类冷启动

```
┌─────────────┬──────────────────────────┬─────────────────────────┐
│  冷启动类型   │        缺什么              │        怎么补             │
├─────────────┼──────────────────────────┼─────────────────────────┤
│ 新用户冷启动  │ 用户没有历史行为           │ 注册信息/设备/地域/         │
│ (user)      │                          │ 首次填的兴趣/相似人群        │
├─────────────┼──────────────────────────┼─────────────────────────┤
│ 新物料冷启动  │ 物品没有被曝光/点击过       │ 内容特征(标题/图片/类目/    │
│ (item)      │                          │ 作者)、相似物品迁移、扶持流量 │
├─────────────┼──────────────────────────┼─────────────────────────┤
│ 新广告冷启动  │ 广告没有 CTR/CVR 数据      │ 广告主历史、创意内容特征、    │
│ (ad)        │ (但要花广告主的钱!)       │ 探索预算、相似广告先验        │
└─────────────┴──────────────────────────┴─────────────────────────┘
```

冷启动的本质:**模型靠"行为数据"学习,但新用户/新物料/新广告恰恰没有行为数据。** 这是个鸡生蛋问题——没数据就排不准,排不准就没曝光,没曝光就更没数据。

### 9.2 两大武器:Side Info + 探索(E&E)

**武器一:Side Info(辅助信息 / 内容特征)**
没有行为,就用"内容本身"和"身份属性"。新视频没人看过,但它有标题、封面、类目、作者——用这些内容特征,模型能给出一个"先验估计"。这就是为什么内容理解(NLP/CV 提取物料特征)对冷启动至关重要。

**武器二:探索与利用(Exploration & Exploitation, E&E)**
利用(exploitation)= 给用户已知他喜欢的(赚稳的);探索(exploration)= 故意给点没把握的,**为了收集数据**。

```
纯利用: 永远给最优 → 新物料永远没机会曝光 → 永远冷
纯探索: 全随机 → 用户体验崩 → 收入崩
E&E:    用一点流量做探索,平衡"当下收益"和"长期学习"
        常用:ε-greedy、Thompson Sampling、UCB、Bandit
```

> 🎓 **老师视角**:E&E 对平台研发的最大挑战是**"探索流量"是一种需要管理的稀缺资源**。给新物料多少探索流量?探索的反馈怎么快速回收进模型(近线/实时更新)?探索带来的短期收益损失谁买单?这些都不是模型问题,是**流量分配与数据回收的工程问题**,正是平台研发的主场。尤其广告冷启动——探索花的是**广告主的真金白银**,探索策略错了是要赔钱的,平台必须有"探索预算护栏"。

### 9.3 冷启动对平台的供给要求

| 供给要求 | 说明 |
|---------|------|
| 实时物料入库 | 新物品分钟级进入召回/排序候选池 |
| 内容特征管线 | NLP/CV 提取物料 side info,实时灌入特征库 |
| 探索流量分配 | 给新物料/新广告分配可控的探索流量 |
| 近线/实时反馈回收 | 探索产生的行为快速回流模型(分钟级) |
| 冷启动效果监控 | 新物料破冰率、新用户次留等专项看板 |

### 9.4 一个脱敏案例:新物料破冰的探索预算

某内容平台(脱敏,数字为经验量级)给新物料的探索机制:

```
每天新增物料: ~50 万条
策略:为"零曝光新物料"开辟一个 "探索流量池",约占大盘 3%~5% 流量
   ├─ 新物料进池 → 强制获得一定曝光(破冰)
   ├─ 快速观察前 N 次曝光的 CTR/完播 (近线分钟级回收)
   ├─ 表现好的 → 毕业,进入正常漏斗与老物料公平竞争
   └─ 表现差的 → 限流/淘汰,不再浪费探索预算
关键指标:破冰率(N 小时内获得足够曝光以判定优劣的新物料占比)
```

这套机制对平台的考验集中在两点:**(1) 探索流量的精准分配**(既不能太多伤大盘收益,也不能太少让新物料饿死);**(2) 反馈的快速回收**(探索的价值在于"快速判优劣",回收慢了探索就白做)。

> 🎓 **老师视角**:探索流量是"花钱买信息"。3% 的流量看着不多,但乘上大盘体量就是巨大的收益让渡——这笔账老板一定会算。你要能讲清楚:"我们让出 3% 流量做探索,换来的是新物料更快破冰、内容生态更健康、长期供给更丰富。"并且要有数据证明这 3% 花得值(破冰率、新物料长期贡献)。**探索预算是平台和业务反复拉扯的数字,把它做成可观测、可调节、能算 ROI 的旋钮,是你的价值所在。**

## 10. 多目标 / 多场景 / 多任务在漏斗里的位置

### 10.1 三个容易混淆的"多"

| 概念 | 含义 | 例子 | 主要落在漏斗哪层 |
|------|------|------|----------------|
| 多目标(multi-objective) | 同时优化多个**业务目标** | CTR + 时长 + 互动 + 转化 | 精排(多目标预估+融合) |
| 多任务(multi-task) | 一个模型同时学多个**预估任务** | MMoE 同时学 pCTR/pCVR | 精排(模型结构) |
| 多场景(multi-scene) | 一套体系服务多个**业务场景** | 首页/详情页/搜索结果页 | 全链路(召回到精排) |

> 注意区分:**多目标是"想要什么"(业务诉求),多任务是"怎么学"(模型手段),多场景是"用在哪"(业务覆盖)。** 算法同学常把它们混着说,你要能拆清楚——尤其在需求评审时,"多场景"往往意味着平台要支持模型/特征的场景化隔离与共享,这是实打实的工程量。

### 10.2 多目标融合在精排的位置

```
精排重模型
   ├─ pCTR  ─┐
   ├─ pCVR  ─┤
   ├─ 时长   ─┼──► 融合公式 (加权 / 学习式融合) ──► 最终排序分
   ├─ 互动   ─┤      score = pCTR^a × 时长^b × ...
   └─ 完播   ─┘      (权重 a,b 由业务/在线学习决定)
```

多目标的难点不在"预估多个值",而在"**怎么融合**"——融合公式的权重直接决定了业务导向(更激进拉时长?还是更稳拉留存?)。这个权重往往要做大量 AB 实验调优。

### 10.3 对平台的要求

| 供给要求 | 说明 |
|---------|------|
| 多目标样本 | 同一曝光样本上挂多个 label(点击/转化/时长...) |
| 多任务训练框架 | 支持 MMoE/PLE 等共享底层、多塔输出的结构 |
| 融合权重可配置 | 融合公式权重热更新 + AB 实验 |
| 多场景特征/模型管理 | 场景间特征共享与隔离、模型版本按场景路由 |

> 🎓 **老师视角**:多场景对平台研发是个"指数级复杂度放大器"。3 个场景 × 5 个目标 × N 个模型版本,如果没有好的**特征/模型的"共享-隔离"治理机制**,平台会被组合爆炸淹没。这也是为什么大厂都在推"一套模型多场景(STAR 等)"——不只是为了效果,更是为了**治理成本**。当老板问"为什么要做多场景统一建模",效果是一方面,**运维成本和迭代效率是更能打动他的论据**。

## 11. 把漏斗串起来:在线/离线一体的平台视角

前面逐层拆解了漏斗,最后从平台研发的"上帝视角"把整条链路串起来——这才是你的主战场全貌:

```
         离 线 / 近 线 (Offline / Nearline)              在 线 (Online)
 ┌──────────────────────────────────────┐   ┌──────────────────────────────┐
 │ 日志采集(曝光/点击/转化/上下文)        │   │     用户请求 (user, context)   │
 │        │                              │   │            │                  │
 │        ▼                              │   │            ▼                  │
 │ 样本生产:三条流                        │   │  ┌──────────────────────┐    │
 │  ├ 召回全库负采样流                     │   │  │ 召回→粗排→精排→重排→混排 │    │
 │  ├ 精排曝光快照流  ◄── 特征日志 ────────┼───┼──┤ (各层查特征库/向量库)    │   │
 │  └ 重排 list-wise 流                    │   │  └──────────┬───────────┘    │
 │        │                              │   │            │ 落特征快照        │
 │        ▼                              │   │            ▼                  │
 │ 训练(召回/粗排/精排/重排各自训练)       │   │      返回结果给用户            │
 │        │                              │   │            │                  │
 │        ▼ 产出模型 + 物品向量            │   │            ▼ 用户反馈           │
 │ 模型/向量推送 ──────────────────────────┼──►│  模型/索引加载、AB 路由        │
 │        ▲                              │   │            │                  │
 │        └───── 反馈回流(行为日志)──────┼───┼────────────┘                  │
 └──────────────────────────────────────┘   └──────────────────────────────┘
            ↑ 平台研发的主战场:数据流转的每一环 ↑
```

这张图的每一条线都是平台研发要负责的"管道",对照看你的职责清单:

| 环节 | 平台研发职责 | 出问题的典型症状 |
|------|------------|----------------|
| 日志采集 | 曝光/点击/转化日志完整、不丢、可对齐 | 样本缺失、label 不准 |
| 特征日志 | 在线打分特征落快照 | 特征穿越、离线虚高 |
| 样本生产 | 三条流的拼接、归因、去重 | 样本偏差、label 漏拼 |
| 训练调度 | 资源调度、增量/全量训练、产出物管理 | 训练慢、产出物版本乱 |
| 模型/向量推送 | 模型、物品向量、索引的原子发布 | 版本错配、静默故障 |
| 在线服务 | 各层打分服务、AB 路由、降级 | 延迟超标、抖动、雪崩 |
| 反馈回流 | 行为日志回流闭环、实时/近线更新 | 模型更新滞后、探索失效 |

> 🎓 **老师视角**:算法同学的世界是"模型",你的世界是"模型周围的一切数据流转"。一个搜广推系统能不能稳定迭代,90% 取决于这张图里的管道是否健壮、可观测、可追溯——而不是模型本身有多 fancy。**记住这句话:模型决定效果的上限,平台决定效果能不能稳定、低成本地达到那个上限,并且出了问题查得清。** 评审、汇报、排障,你都可以回到这张图,指着具体的环节说话,这是你区别于"只懂某一层"的同学的核心竞争力。

## 业界怎么做

- **阿里**:电商搜索/推荐/广告体系成熟,贡献了 DIN/DIEN(行为序列建模)、MMoE/PLE(多任务)、ESMM(CVR 建模解样本偏差)、STAR(多场景统一建模)。广告侧 OCPC/OCPM 智能出价是标杆。
- **字节(抖音/今日头条)**:信息流推荐的代表,极致的实时性(实时特征、实时模型更新)、强探索(冷启动破冰快),漏斗各层高度工程化。
- **快手**:短视频推荐,强调多目标(时长/完播/互动/关注)的复杂融合与生态平衡。
- **腾讯**:广告(PLE 多任务出自腾讯)、微信生态多场景推荐。
- **Google/Meta**:Wide&Deep、DCN(Google)、DLRM(Meta)等经典排序模型;YouTube 的召回-排序两段式是推荐系统教科书级案例。
- **通用趋势**:召回向量化(双塔+ANN)、排序大模型化与生成式(生成式推荐/重排)、多场景统一建模、实时化(streaming 样本与模型更新)。

> 共识:架构上的"召回→粗排→精排→重排→混排"漏斗,以及"双塔召回 + 重模型精排"的组合,已是行业事实标准。差异主要在实时性、探索强度、多场景治理的工程成熟度。

## 权衡取舍

| 取舍点 | A 方 | B 方 | 平台研发的判断 |
|--------|------|------|---------------|
| 漏斗层数 | 层多(召回/粗排/精排/重排/混排齐全) | 层少(召回直接精排) | 候选量小就别堆层,按"精排能否扛住召回输出"决定是否要粗排 |
| 召回路数 | 多路(覆盖全、效果好) | 少路(链路简单、好维护) | 多路有边际递减且抬高延迟与维护成本,定期下线低贡献路 |
| 模型复杂度 | 重模型(效果好) | 轻模型(快、省) | 越靠漏斗上层越能上重模型;下层(召回)堆复杂度性价比极低 |
| 实时性 | 实时特征/实时更新(效果好) | 批处理(稳定、便宜) | 实时链路贵且脆,只对"实时收益高"的特征/场景做实时 |
| 探索强度 | 强探索(长期收益) | 弱探索(短期稳) | 探索是花钱买数据,要算 ROI,广告侧尤其要有预算护栏 |
| 多场景 | 统一建模(治理省) | 各场景独立(灵活) | 场景多到运维爆炸时才上统一建模,小场景独立反而敏捷 |

> 🎓 **老师视角**:平台研发在这些取舍里的角色不是"选边站",而是**"把每个选择的成本算清楚,摆到桌面上让业务方做决策"**。算法同学天然倾向"更复杂=更好",但复杂度的账(机器、延迟、维护、稳定性)是你来算的。你的杀手锏是一句话:"这个方案效果 +X%,但机器成本 +Y%、P99 延迟 +Z ms、多一条要长期维护的链路,值不值?"

## 常见误区与反直觉

1. **误区:召回就是"粗一点的精排"。**
   反直觉真相:召回的目标(高召回率、别漏)和精排(高精度、排准)**根本不同**,连样本(全库负采样 vs 曝光样本)都不一样。把召回当精排做,负样本就错了。

2. **误区:离线 AUC 涨了,上线就一定涨。**
   反直觉真相:离线和线上对不上是常态,头号嫌疑是**特征穿越**和 **train-serving 不一致**。这口锅常落在平台头上。

3. **误区:候选越多越好,召回多捞点没坏处。**
   反直觉真相:候选量直接吃掉后链路算力和延迟预算,且超过一定量后效果边际递减。**召回量是要精打细算的预算,不是越大越好。**

4. **误区:广告就是带价格的推荐,复用推荐链路即可。**
   反直觉真相:广告有计费、竞价、预算、归因、反作弊一整套,**对一致性和准确性的要求远高于搜推(算错要赔钱)**。

5. **误区:重排只是"再排一次"。**
   反直觉真相:重排是从 point-wise 到 list-wise 的范式切换,要的是 list 级样本和上下文特征,**和精排是两套数据形态**。

6. **误区:冷启动是算法的事,平台不用管。**
   反直觉真相:冷启动的核心是**探索流量分配 + 反馈快速回收**,这是纯粹的工程/平台问题。

7. **误区:多目标 = 多任务。**
   反直觉真相:多目标是"业务想要啥",多任务是"模型怎么学",多场景是"用在哪"——三件不同的事,常被混着说。

## 平台侧落地 Checklist

设计/评审搜广推平台时,逐项核对:

- [ ] **样本流分清三条管道**:召回全库负采样流 / 精排曝光快照流 / 重排 list-wise 流,各自的采集、拼接、存储独立规划。
- [ ] **特征快照(feature logging)**:精排打分时落特征快照,杜绝特征穿越,保证 train-serving 一致。
- [ ] **召回基建**:向量库、ANN 在线检索、物品向量构建与更新、多路并发调度+超时降级。
- [ ] **粗精排一致性监控**:监控粗排/精排 top-N 重合度,有蒸馏时跑通软标签样本流。
- [ ] **全链路打分可追溯(score trace)**:记录每条 item 从召回到最终位置的每一步加权/惩罚/插入原因。
- [ ] **延迟预算账本**:端到端及各层延迟预算清晰,有容量模型把模型复杂度变化翻译成延迟和机器成本。
- [ ] **广告专属链路**:计费、预算 pacing、竞价、归因、反作弊,与搜推数据流隔离,一致性要求拉满。
- [ ] **冷启动支撑**:实时物料入库、内容特征管线、探索流量分配与护栏、近线反馈回收。
- [ ] **多场景/多目标治理**:特征/模型的共享-隔离机制、融合权重热更新、按场景的模型路由。
- [ ] **AB 实验与灰度**:策略(打散/广告位/扶持权重)可配置、可灰度、可回滚。

## 向上沟通

向**老板/非技术决策者**汇报时,把技术翻译成业务和成本语言:

- **讲架构**:"我们的推荐/搜索/广告系统都是一台'排序机器',从上亿候选里层层筛到用户看到的十几条。底层引擎和数据可以复用,这就是我们建统一平台的价值——一套基建,三个业务都能跑。"
- **讲投入**:"想让模型更准,就要更复杂的模型和更多特征,这直接换算成更多机器和更高延迟。每一次效果提升,我都能给您算清楚成本账,帮您判断值不值。"
- **讲冷启动**:"新内容/新广告没有数据,要靠'探索流量'去试错收集数据,这本质是花一点短期收益买长期增长。广告侧探索花的是广告主的钱,我们做了预算护栏防止失控。"
- **讲多场景**:"做多场景统一建模,不只是效果好一点,更重要的是省运维成本、提迭代效率——3 个场景各搞各的,长期是组合爆炸,人力扛不住。"
- **一句话锚定价值**:"算法同学负责让模型更聪明,我们平台负责让这份聪明能稳定、低成本、可追溯地落到线上——没有我们,他们的离线效果上不了线,上了线也说不清。"

## 自测三问

**Q1:为什么召回不能只用"曝光未点击"作为负样本,而精排却可以?**

参考答案:召回面对的是**全量候选库(亿级)**,要学会"从全库里区分好坏",而曝光样本是经过漏斗层层筛选的"幸存者",都相对优质——只用它们训练,召回模型从没见过"全库里真正离谱的烂候选",上线会把一堆不相关候选捞上来。所以召回必须做**全库随机负采样**。精排只对"已经决定展示的候选"排序,任务是"在曝光集里排准",所以用曝光样本(曝光未点击作负样本)天经地义。本质:**两者的任务边界不同,所以样本空间不同。**

**Q2:算法同学说"离线 AUC 涨了 3 个点,但上线没效果",作为平台研发你先怀疑什么?怎么排查?**

参考答案:首先怀疑 **train-serving 不一致**,尤其是**特征穿越(feature leakage)**——训练用了线上打分时刻拿不到的"未来/事后特征"(如当天总点击数),导致离线虚高。排查路径:(1) 核对训练特征与线上打分特征的口径和时间窗是否一致;(2) 检查是否使用了打分时刻的**特征快照**而非事后回填;(3) 对比同一请求的离线/在线特征值是否逐字段一致。如果特征对齐没问题,再看是否被混排/重排的线上策略规则改写了顺序(用全链路 score trace 排查)。

**Q3:一个新场景候选量只有几千,算法同学想照搬"召回→粗排→精排→重排"四层架构,你怎么判断?**

参考答案:**不一定需要粗排,甚至重排也可酌情简化。** 判断准则是"召回输出量 × 精排单条耗时 vs 延迟预算"——候选量小(几千召回、输出几百到精排),精排重模型完全扛得住时,粗排只会增加链路复杂度和粗精排一致性的维护负担,没有收益。架构层数是为了应对算力约束,不是为了"完整"。我会建议:先召回直接接精排,跑起来看延迟和效果,等候选量/QPS 增长到精排扛不住时再引入粗排。**别为架构完整性付不必要的工程成本。**
