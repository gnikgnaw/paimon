## 0. 30 秒速通(TL;DR)

一句话:**ML 平台是一条「数据 → 样本 → 特征 → 训练 → 模型 → 推理 → 实验」的流水线,每一层都是一个独立组件,组件之间靠「数据契约」对接;而数据湖底座(以 Paimon 为例)是把这条流水线「离线」和「在线」两条腿拼成同一根脊柱的关键。**

- **分层**:数据层(日志/数仓/数据湖/KV)→ 样本平台 → 特征平台(Feature Store)→ 训练平台 → 模型管理 → 推理服务(serving)→ 实验平台(AB),外加贯穿全局的元数据/血缘/调度。
- **核心矛盾**:离线(训练,要全量、可回溯、可复现)和在线(serving,要低延迟、点查)对同一份数据有完全不同的诉求。这就是「离在线一致性」问题的根源。
- **湖仓一体的价值**:用一份存储、一套 schema、一套血缘,同时服务批(Spark 训练读)和流(Flink 实时写/读),避免离线一套链路、在线一套链路导致的「特征穿越」和「训练/服务偏差(train-serving skew)」。
- **Paimon 扮演什么**:它是一个带 **LSM 主键表** 的湖格式。
  - 主键表 + upsert → 样本 label 回填、特征回填、延迟反馈,原地更新而不是重写整张表。
  - 流批一体 → 同一张表 Flink 流式写、Spark 批量读,离在线共用一份真相。
  - Changelog → 行为变更产生 +I/-U/+U/-D 变更流,驱动下游特征/样本增量更新。
  - Time Travel / Snapshot → 取「训练发起那一刻」的数据版本,做 point-in-time 关联和可复现训练。
  - Compaction / 分区 → 把流式写产生的小文件合并成大文件,保证批量训练读吞吐。
- **对平台研发意味着什么**:你不需要懂模型怎么收敛,但你要保证「算法同学拿到的样本是对的、可复现的、读得快的」,并且「线上线下用的是同一份特征定义」。这就是你的简历主菜。

## 1. ML 平台整体分层架构:一张大图看懂全局

先看全局。下面这张图是搜广推场景下一个相对完整的 ML 平台分层,从下往上读(数据在最底,模型/实验在最上)。

```
┌──────────────────────────────────────────────────────────────────────────────┐
│                        实验平台 (A/B Experiment Platform)                       │
│   分流(flow split) · 指标(CTR/CVR/GMV) · 显著性检验 · 互斥/正交层(layer)        │
└───────────────▲──────────────────────────────────────────────▲───────────────┘
                │ 模型版本 → 实验桶绑定                          │ 在线指标回流
┌───────────────┴──────────────────────────────────────────────┴───────────────┐
│                        推理服务 (Online Serving)                                │
│   模型加载 · 特征拉取(在线 Feature Store)· 打分(scoring)· 灰度/回滚              │
└───────────────▲────────────────────────────▲──────────────────▲──────────────┘
                │ 模型文件+签名                │ 在线特征读        │ 打分日志(曝光/点击)
┌───────────────┴──────────┐  ┌──────────────┴───────┐          │
│   模型管理 (Model Registry)│  │  特征平台 Feature Store│          │
│  模型仓 · 版本 · 评估指标   │  │  在线库(KV/Redis)     │          │
│  审批 · 上线/回滚           │  │  离线库(湖/数仓)      │          │
└───────────────▲──────────┘  │  特征定义/注册中心     │          │
                │ 训练产出模型  │  特征计算(批/流)       │          │
┌───────────────┴──────────┐  └──────▲────────▲───────┘          │
│   训练平台 (Training)      │         │        │                  │
│  资源调度(K8s/Yarn)       │◄────────┘        │ 特征回填          │
│  分布式训练(PS/AllReduce) │  读样本+特征      │                  │
│  超参/任务编排            │                   │                  │
└───────────────▲──────────┘                   │                  │
                │ 读训练样本                     │                  │
┌───────────────┴───────────────────────────────┴──────────────────┴──────────┐
│                        样本平台 (Sample Platform)                              │
│   样本拼接(join label+feature)· 负采样 · 延迟反馈 · 样本版本/快照              │
└───────────────▲────────────────────────────────────────────────────────────┘
                │ 行为日志 + 特征快照
┌───────────────┴────────────────────────────────────────────────────────────┐
│                        数据层 (Data Layer)                                     │
│  行为日志(Kafka/日志流) · 数仓(Hive/数仓表) · 数据湖(Paimon/Iceberg) · KV(在线)│
└──────────────────────────────────────────────────────────────────────────────┘

         ┌──────────────────────────────────────────────────────────┐
横切关注: │ 元数据/血缘(Metadata & Lineage) · 调度(Scheduler) ·        │
         │ 多租户/配额(Quota) · 监控告警 · 权限治理(RBAC)            │
         └──────────────────────────────────────────────────────────┘
```

读图要点:

- **纵向是数据流**(从下往上:原始数据 → 样本 → 特征 → 训练 → 模型 → 服务),**也有回流**(右侧:打分日志、在线指标重新落回数据层,形成闭环)。
- **横切关注**(元数据、调度、多租户、监控)不是某一层,而是贯穿所有层。平台研发大量的活就在这条横线上。
- **Feature Store 是个「双库」结构**:离线库给训练读,在线库给 serving 读。这两个库必须保证「同一个特征定义、同一份计算逻辑」,否则就是 train-serving skew。这是后面第 6 节的重头戏。

🎓 **老师视角(全局认知)**:很多新人把 ML 平台理解成「训练一个模型的工具」。错。**ML 平台 90% 的复杂度不在训练,而在数据**——样本怎么拼对、特征怎么不穿越、离线在线怎么一致、历史怎么复现。模型训练本身是算法同学的事,但「喂给模型的东西对不对」是平台研发的事。你简历上最值钱的不是「我搭了个训练集群」,而是「我保证了百亿样本的离在线一致性和可复现」。

## 2. 各组件职责与数据流:逐层拆解

逐层说清楚每个组件「干什么、吃什么、吐什么」。

### 2.1 数据层(Data Layer)

- **职责**:存原始数据。包括实时行为日志(曝光/点击/转化,通常走 Kafka 等消息流)、离线数仓(Hive 风格的 ODS/DWD/DWS 分层表)、数据湖(Paimon/Iceberg/Hudi,带 ACID 和更新能力)、在线 KV(Redis/HBase,给 serving 点查)。
- **吃**:业务系统埋点、服务端日志、第三方数据。
- **吐**:给样本平台和特征平台当原料。

### 2.2 样本平台(Sample Platform)

- **职责**:把「label」和「特征快照」拼成训练样本(sample joining)。处理负采样、延迟反馈(delayed feedback)、样本去重、样本版本管理。
- **关键难点**:**point-in-time correctness**——拼样本时,特征值必须取「请求发生那一刻」的版本,不能用未来的特征(那叫特征穿越/数据泄漏)。
- **吐**:训练样本表(通常落数据湖,带分区,如 `dt=2026-06-13`)。

### 2.3 特征平台(Feature Store)

- **职责**:定义、计算、存储、服务特征。是离在线一致性的核心承载点。
- **三个子模块**:
  - **特征注册中心(Registry)**:特征的「字典」——名字、类型、来源、计算逻辑、owner。
  - **特征计算**:批计算(Spark,跑离线特征)+ 流计算(Flink,跑实时特征如「用户近 1 小时点击数」)。
  - **双存储**:离线库(湖/数仓,给训练全量扫描)+ 在线库(KV,给 serving 毫秒点查)。
- **吐**:特征值,供样本平台拼接(离线)和 serving 拉取(在线)。

### 2.4 训练平台(Training Platform)

- **职责**:资源调度(K8s/Yarn 分配 GPU/CPU)、分布式训练(Parameter Server 或 AllReduce 架构)、任务编排、超参管理。
- **吃**:训练样本(来自样本平台)+ 特征(来自 Feature Store 离线库)。
- **吐**:模型文件 + 训练指标(loss/AUC),交给模型管理。

补充一点你需要懂的「训练侧」知识(够你接话即可,不必深到调参):

- **PS(Parameter Server)架构**:适合搜广推的**大稀疏模型**(特征上亿到百亿维 embedding)。worker 算梯度,server 存参数、聚合更新。瓶颈常在参数拉取/推送的通信和 server 内存。
- **AllReduce 架构**:适合稠密模型(CV/NLP)。各 worker 持完整模型副本,梯度环形规约同步。瓶颈在带宽。
- **平台研发的接触点**:不是去优化算法,而是保证「**worker 能高吞吐地把样本喂进去**」——这就直接回到底座:Spark/Flink 读 Paimon 的吞吐、数据本地性、文件数(小文件会拖垮数据加载)。**很多训练慢其实慢在 data loading,不在 GPU**。

| 架构 | 适合模型 | 瓶颈 | 与底座的关系 |
|---|---|---|---|
| Parameter Server | 大稀疏(搜广推) | 通信/server 内存 | 样本读吞吐要稳 |
| AllReduce | 稠密(CV/NLP) | 带宽 | 数据加载不能成瓶颈 |

🎓 **老师视角(训练慢往往不是 GPU 的锅)**:算法同学抱怨「训练慢、GPU 利用率低」,九成情况是**数据喂不饱**——样本读吞吐跟不上、小文件太多、序列化反序列化开销大。这时候该动的是底座(compaction、列裁剪、读并行度),不是去加 GPU。你能一眼看出「这是 data loading 瓶颈不是算力瓶颈」,就比一般运维强一截。

### 2.5 模型管理(Model Registry)

- **职责**:模型仓库(存模型文件)、版本管理、离线评估指标、上线审批、回滚。
- **吐**:经审批的模型版本,推给推理服务。

### 2.6 推理服务(Online Serving)

- **职责**:加载模型,实时拉取在线特征,对请求打分,支持灰度发布和秒级回滚。
- **吐**:打分结果(给业务)+ 打分日志(曝光特征快照,回流到数据层,这是下一轮样本的原料)。

### 2.7 实验平台(A/B Platform)

- **职责**:流量分桶、模型/策略与实验桶绑定、在线指标统计、显著性检验、互斥/正交分层。
- **吐**:实验结论(模型 A 比 B 的 CTR 提升 X%),决定模型是否全量上线。

数据流闭环用一张「环形」图表达更清楚:

```
   行为日志/曝光点击 ──┐
        ▲             ▼
        │        [样本平台] 拼 label+特征快照
   [Serving]           │
   打分+回流日志        ▼
        ▲        [Feature Store] 特征计算
        │             │
   [模型管理]          ▼
        ▲        [训练平台] 读样本+特征 → 训练
        │             │
        └─────────────┘  模型产出 → 评估 → 上线 → Serving
```

下面这张表把每个组件的「输入/输出/典型存储/平台研发关注点」对齐:

| 组件 | 输入 | 输出 | 典型存储 | 平台研发关注点 |
|---|---|---|---|---|
| 数据层 | 埋点/日志 | 原始表/流 | Kafka / 数仓 / 湖 / KV | 数据接入、schema 管理、时效 |
| 样本平台 | label + 特征快照 | 训练样本表 | 数据湖(分区表) | point-in-time、样本版本、回填 |
| 特征平台 | 原始数据 | 特征值(离线+在线) | 湖(离线)/ KV(在线) | 离在线一致、特征注册、穿越防护 |
| 训练平台 | 样本 + 特征 | 模型 + 指标 | 对象存储 / 模型仓 | 调度、资源隔离、数据读吞吐 |
| 模型管理 | 模型 + 指标 | 模型版本 | 模型仓 | 版本、审批、回滚 |
| 推理服务 | 模型 + 在线特征 | 打分 + 日志 | 内存 / KV | 延迟、灰度、特征一致 |
| 实验平台 | 模型版本 + 流量 | 实验结论 | 指标系统 | 分流正确性、指标可信 |

🎓 **老师视角(数据流的方向感)**:记住一个判断口诀——**「向上是加工,向右是回流」**。数据从底层向上每经过一层都被加工得更「贴近模型」(原始日志 → 样本 → 特征 → 模型);而 serving 产生的日志向右回流,变成下一轮训练的原料。**这个闭环的延迟,直接决定模型多久能学到最新的用户行为**。平台研发优化这个闭环延迟(比如从 T+1 缩到分钟级),就是实打实的业务价值。

## 3. 组件间接口:数据契约怎么定

组件能不能解耦、能不能独立演进,全看「接口契约」定得好不好。这里讲最关键的几条链路契约。

### 3.1 样本平台 ↔ 特征平台

样本平台拼样本时要向特征平台要「某用户/某 item 在某时刻的特征快照」。契约要素:

- **特征 key**:用什么主键关联(user_id / item_id / request_id)。
- **时间语义**:用 event time 还是 processing time?point-in-time 关联必须用 event time。
- **版本**:取哪个特征版本(对应到湖的哪个 snapshot)。
- **schema**:特征字段的名字、类型、缺失值约定。

### 3.2 特征平台 ↔ 训练 / 特征平台 ↔ serving

这是 train-serving skew 的发生地。契约要素:

- **同一份特征定义**:离线计算和在线计算必须用同一套逻辑(理想是同一份代码/同一个引擎)。
- **同一份数据来源**:离线库和在线库的数据应可追溯到同一个上游(下一节讲 Paimon 怎么用一张表喂两边)。
- **schema 演进规则**:加字段、改类型必须双边同步,不能离线加了在线没加。

### 3.3 训练 ↔ 模型管理 ↔ serving

- **模型签名(signature)**:模型期望的输入特征列表、顺序、类型。serving 拉特征时必须按签名对齐。
- **模型版本号**:训练产出唯一版本,实验平台和 serving 都用这个号绑定。

下面这张表对比「契约定得好 vs 不好」的后果:

| 契约项 | 定得好 | 定得不好的后果 |
|---|---|---|
| 时间语义 | 统一 event time | 特征穿越,离线 AUC 虚高,上线暴跌 |
| 特征定义 | 离在线同源同逻辑 | train-serving skew,模型线上不 work |
| schema 演进 | 双边同步 + 兼容规则 | 上线后特征对不齐,打分错乱 |
| 模型签名 | 显式声明输入列 | serving 喂错特征,静默劣化 |
| 版本绑定 | 模型/样本/特征都有版本号 | 出问题无法定位、无法复现 |

🎓 **老师视角(契约就是边界)**:平台研发最容易踩的坑是「为了快,组件之间直接读对方的内部表」。短期省事,长期变成一坨没人敢动的耦合。**好的接口契约 = schema + 时间语义 + 版本 + owner**,四个齐全才算数。你定契约时多问一句「这个字段出问题找谁、怎么复现」,就能挡掉一半的事故。

### 3.4 一条样本的接口视角:从曝光到训练就绪

把上面三条契约串起来,看一条样本是怎么「跨组件」走完一生的。这是理解契约价值的最佳切片。

```
  时刻         发生的事                          涉及契约                   底座动作
  ─────────────────────────────────────────────────────────────────────────────────
  T0 曝光      serving 打分, 落曝光日志           模型签名(特征列表)         记录请求时刻的特征快照
                (req_123, user, item, feats)
                      │
  T0+Δ        样本平台收到曝光                    样本↔特征(key+时间语义)    upsert(req_123, label=0)
                      │
  T1 点击      用户点击, 回流点击日志              event time 对齐            upsert(req_123, label=1)
                      │
  T2 转化      几小时后转化(延迟反馈)            主键 upsert                upsert(req_123, cvr=1)
                      │
  T3 训练就绪  样本拼接完成, 绑定 snapshot         版本绑定                  snapshot=N 固定下来
                      │
  T4 训练      Spark 读 snapshot=N 全量           特征定义(同源)            time travel 取一致视图
```

读这张图你要抓住一个反直觉点:**一条样本的 label 不是一次写定的,而是被多次 upsert 逐步「长」出来的**。曝光时 label=0,点击改 1,转化补 cvr——这正是为什么 ML 样本必须用主键表而不能用追加表。如果用 Hive 追加,你会得到 req_123 的三条冲突记录,还得自己去重取最新,既慢又容易错。

下面这张表把「一条样本的生命周期」和「对应的契约/底座能力」对齐,方便你查:

| 阶段 | 数据动作 | 依赖契约 | 依赖底座能力 |
|---|---|---|---|
| 曝光记录 | 写特征快照 | 模型签名 | 主键表写入 |
| label 拼接 | 多次 upsert | 样本↔特征 key + 时间 | 主键 upsert |
| 延迟反馈 | 异步补 label | event time 语义 | 流式 upsert + changelog |
| 训练就绪 | 固定数据版本 | 版本绑定 | snapshot |
| 训练读取 | 全量扫描一致视图 | 特征定义同源 | time travel + 分区裁剪 |

🎓 **老师视角(样本是「活的」)**:新人最容易误以为「样本写进去就不变了」。在搜广推里恰恰相反——**样本是会被持续修改的活数据**(延迟反馈、回填、纠错)。一旦你接受这个事实,你就会明白为什么 ML 数据底座必须支持 upsert + changelog + 版本快照,而传统「不可变追加」的大数据范式在这里水土不服。

## 4. 数据湖底座:为什么 ML 平台需要湖仓一体

前面三节讲了「逻辑分层」。现在落到「物理底座」:这些样本表、特征表、到底存在哪、用什么存。

### 4.1 三代存储方案对比

| 维度 | 纯数仓(Hive) | 数仓 + KV 双链路 | 湖仓一体(Paimon 等) |
|---|---|---|---|
| 更新能力 | 只能整分区重写 | KV 可更新,数仓不行 | 主键表原地 upsert |
| 流批 | 批为主,流很弱 | 离线批 + 在线流两套 | 同一张表流批共用 |
| 离在线一致 | 难,两套链路 | **最难**,两套代码两份数据 | 一份存储一套 schema |
| 历史回溯 | 靠分区,粗 | KV 基本没历史 | snapshot 级 time travel |
| 小文件 | 还好 | KV 无此问题 | 需 compaction 治理 |
| 适合场景 | T+1 离线训练 | 早期妥协方案 | 实时+可复现的现代 ML |

### 4.2 为什么必须湖仓一体

搜广推 ML 对数据有四个「既要又要」:

1. **既要全量回溯(训练),又要低延迟点查(serving)**。
2. **既要批写入(离线特征),又要流写入(实时特征)**。
3. **既要更新(label 回填),又要不可变审计(可复现)**。
4. **既要离线,又要在线,还要二者一致**。

传统「数仓 + KV 双链路」方案会把同一个特征写两遍代码、存两份数据——这是 train-serving skew 的根源。湖仓一体的核心主张是:**用一份存储、一套 schema、一套血缘,尽可能让离线和在线共享同一份真相**。

```
       传统双链路(易 skew)                  湖仓一体(一份真相)
   ┌──────────┐   ┌──────────┐          ┌──────────────────────┐
   │ 离线特征  │   │ 在线特征  │          │   数据湖 (Paimon)      │
   │ 计算(批) │   │ 计算(流) │          │   主键表 / 流批一体     │
   └────┬─────┘   └────┬─────┘          └───────┬───────┬──────┘
        │ 两套代码      │ 两套代码                │ 批读   │ 流读
        ▼              ▼                         ▼        ▼
     数仓表          KV库                      训练     serving
   (训练读)        (serving读)               (同一份)  (派生在线库)
        ↑ 逻辑可能不一致 ↑                          ↑ 同源,易对齐 ↑
```

🎓 **老师视角(湖仓不是银弹)**:湖仓一体解决的是「一致性和可复现」,**不是** serving 延迟。serving 还是要把数据派生到 KV 里做毫秒点查——湖不可能直接扛在线点查 QPS。所以现实架构是「**湖做真相源 + KV 做在线缓存,二者同源派生**」。你跟老板讲架构时不要说「上了湖就不要 KV 了」,那是外行话;要说「湖统一了真相源,KV 从湖派生,从而保证离在线一致」。

## 5. Apache Paimon 在其中扮演什么角色

> 说明:本节是「结合 Apache Paimon 的架构示例」,概念为主。目的是讲清 Paimon 的几项核心能力如何具体支撑「样本管理」和「特征管理」。

Apache Paimon 是一个 **lake format**,核心是把「湖格式」和 **LSM(Log-Structured Merge-tree)** 结构结合,从而在湖上获得高效的实时更新能力。对 ML 平台来说,它提供四张王牌:**主键表 upsert、流批一体、changelog、time travel**,加上一套 **snapshot / 分区 / compaction** 的存储治理机制。

### 5.1 主键表 + upsert:样本/特征的更新

Paimon 的 **primary key table** 用 LSM merge-tree 组织数据:写入按主键排序成 SortedRun,分层(Level)存放,读时做 merge-on-read,后台 compaction 合并。这带来一个对 ML 极其关键的能力——**按主键原地 upsert,而不是整表/整分区重写**。

为什么 ML 需要这个:

- **延迟反馈改 label**:广告转化(CVR)可能在曝光后几小时甚至几天才回来。曝光时样本 label=0,转化回来要改成 label=1。主键表 upsert 一行就改完,不用重写整张样本表。
- **特征回填(backfill)**:新增一个特征,要给历史样本补上;或者修了某个特征的计算 bug 要重刷。upsert 让你按 key 精准更新受影响的行。
- **特征更新**:用户画像类特征(如「用户标签」)会随时间变化,upsert 保证每个 key 只有一行最新值。

```
  样本表(主键 = request_id)的生命周期

  T0 曝光  : upsert (req_123, label=0, features=...)  ──┐
  T1 点击  : upsert (req_123, label=1, ...)             │  同一主键
  T2 转化  : upsert (req_123, cvr_label=1, ...)         │  逐步 upsert
                                                        ▼
  读取时   : merge-on-read 返回 req_123 的最新合并结果

  对比纯数仓: 每次改 label 都要重写整个 dt 分区(可能上亿行)
```

| 更新场景 | 纯数仓做法 | Paimon 主键表做法 |
|---|---|---|
| 改单条 label | 重写整分区 | upsert 一行 |
| 特征回填 | 重跑整批 | upsert 受影响 key |
| 延迟反馈 | 攒批后整体重算 | 流式 upsert 实时改 |
| 去重 | distinct 整表 | 主键天然去重 |

🎓 **老师视角(upsert 是 ML 数据的刚需)**:传统大数据「数据不可变、只追加」的信条,在 ML 样本场景下行不通——**label 本质上是会变的**(延迟反馈)。Paimon 主键表把「可更新」带到了湖上,这正是它比 Hive 强、比纯 Iceberg(早期更偏 append)更贴 ML 的地方。你要能跟算法同学讲清:「延迟反馈不用你攒一周再重算,主键表 upsert 实时改,样本永远是最新的」。

#### 5.1.1 LSM 内部:upsert 在湖上为什么能高效

要在汇报里站得住,你得能讲清「为什么湖上能高效 upsert」。一句话:**靠 LSM 把随机更新转成顺序写 + 后台合并**。

```
   写入路径(LSM merge-tree, 概念示意):

   upsert 流  ──►  内存缓冲(按主键排序)
                        │ flush
                        ▼
                   Level 0  [run][run][run]   ← 新写的小文件, 可能键重叠
                        │ compaction(合并+去重, 保留最新)
                        ▼
                   Level 1  [ sorted run ]     ← 合并后, 键有序不重叠
                        │
                        ▼
                   Level n  [ 大 sorted run ]  ← 越底层文件越大、越稳定

   读取路径: merge-on-read —— 跨 Level 按主键归并, 同 key 取最新版本
```

关键点拆给你:

- **写不就地改**:upsert 不去原文件里改某一行(对象存储也不支持随机改),而是追加一个新版本到 L0,后台 compaction 时再把同一主键的多版本合并、保留最新。所以「upsert 一行」对写入侧其实是「顺序追加 + 延迟合并」,这才便宜。
- **merge engine 决定「怎么合」**:Paimon 支持多种 merge engine。`deduplicate`(默认,同 key 留最新整行)适合大多数样本场景;`partial-update`(同 key 不同列分别更新)适合「曝光写一部分列、转化补另一部分列」这种特征/样本宽表拼接;`aggregation`(按列做 sum/max 等聚合)适合实时累计特征(如「累计点击数」)。
- **读有代价**:merge-on-read 要跨层归并,L0 文件越多读越慢——这就是 5.5 节 compaction 的由来。

| merge engine | 行为 | ML 典型用途 |
|---|---|---|
| deduplicate | 同 key 留最新整行 | 一般样本/画像特征 |
| partial-update | 同 key 按列分别更新 | 曝光+转化宽表拼接 |
| aggregation | 同 key 按列聚合 | 实时累计类特征 |
| first-row | 同 key 留第一条 | 去重保首次 |

🎓 **老师视角(partial-update 是宽表拼接的神器)**:搜广推样本常常是「一行很多列,不同列来自不同时刻/不同上游」。比如曝光阶段写用户/item 特征,转化阶段补 cvr 标签。用 `partial-update` merge engine,曝光流写它那几列、转化流写它那几列,Paimon 自动按主键合并成一整行——你不用写复杂的 join。这是把「多流 join」下沉到存储层的典型手法,能省掉一大坨样本拼接代码。

### 5.2 流批一体:统一离在线

同一张 Paimon 表,**Flink 可以流式写入/读取,Spark 可以批量读取**。这是「离在线统一」的物理基础。

典型用法:

- **Flink 流式写**:实时行为日志进来,Flink 实时计算特征/拼样本,流式 upsert 到 Paimon 表。
- **Spark 批量读**:训练任务用 Spark 全量扫描这张表的某个 snapshot 做训练。
- **Flink 流式读**:下游实时任务(如实时特征派生到在线 KV)流式消费这张表的变更。

```
                    ┌─────────────────────────────┐
   实时日志 ──Flink──►│      Paimon 样本/特征表       │
   (流式写 upsert)    │  primary key + LSM + snapshot │
                    └──────┬───────────────┬────────┘
                           │ Spark 批读     │ Flink 流读
                           ▼               ▼
                      训练(全量扫描)   派生到在线 KV(serving)
                      读某个 snapshot   消费 changelog
```

| 引擎 | 对 Paimon 的典型操作 | 服务对象 |
|---|---|---|
| Flink | 流式写 upsert / 流式读 changelog | 实时样本特征、在线派生 |
| Spark | 批量全表/分区扫描 | 离线训练读样本 |
| 二者共享 | 同一张表、同一套 schema、同一份血缘 | 离在线一致 |

🎓 **老师视角(流批一体 = 一份真相两种读法)**:流批一体的本质不是「同一个引擎跑流和批」,而是「**同一份存储,既能被流式增量消费,又能被批量全量扫描**」。对平台研发的意义:你不用再维护「离线一张 Hive 表、实时一个 Kafka topic、二者对账」的痛苦链路,Paimon 一张表两种读法,从源头消除了对账问题。

### 5.3 Changelog:行为变更流驱动更新

Paimon 主键表可以产生 **changelog**——即每次数据变化的变更流(类似 CDC 的 +I 插入 / -U 更新前 / +U 更新后 / -D 删除)。下游可以流式消费这个 changelog,实现增量驱动。

在 ML 平台里的价值:

- **特征/样本增量更新**:上游样本表一旦有 upsert(比如 label 从 0 改成 1),changelog 就吐出这条变更,下游派生表(如训练宽表、在线特征)只更新这一条,不用全量重算。
- **在线特征同步**:Flink 消费 changelog,把变更实时同步到在线 KV,保证在线特征跟着离线真相走。
- **触发式计算**:行为变更可作为下游流计算的触发信号(如「用户新增一次点击」触发「近 1 小时点击数」重算)。

```
   Paimon 主键表 upsert
        │
        ▼  产生 changelog (+I / -U / +U / -D)
   ┌────┴─────────────────────────────┐
   ▼                  ▼                ▼
 训练宽表          在线 KV          实时统计特征
 (增量更新)       (同步 serving)   (触发重算)
```

| 没有 changelog | 有 changelog |
|---|---|
| 下游只能定时全量重算 | 下游按变更增量更新 |
| 在线特征靠定时刷新,延迟大 | 变更实时推送,秒级同步 |
| 改一条要扫全表找差异 | 直接拿到变更行 |

🎓 **老师视角(changelog 是「增量」的引擎)**:很多人以为流计算就是「数据来一条算一条」,但真正难的是「上游一条数据被**改了**之后,下游怎么知道、怎么只更新受影响的部分」。changelog 就是答案——它把「状态变更」变成「事件流」。对平台研发:有了 changelog,你才能搭出「离线改一条、在线秒级跟上」的离在线同步链路,而不是靠定时全量刷。

### 5.4 Time Travel / 快照:可复现训练与 point-in-time

Paimon 每次 commit 产生一个 **snapshot**(快照),指向当时的全量数据状态。你可以按 snapshot id 或时间戳读取「历史某一时刻」的表——这就是 **time travel**。

对 ML 的两个杀手级用途:

1. **可复现训练(reproducibility)**:训练任务记录「我读的是 snapshot=12345」。半年后想复现这次训练、排查线上问题,直接 time travel 回 snapshot 12345,拿到一模一样的样本。**没有这个,模型出了问题你根本无法复盘**。
2. **point-in-time 关联**:拼样本时,特征要取「请求发生那一刻」的版本。time travel 让你能精确取到那个时间点的特征快照,从源头防止特征穿越(用了未来数据)。

```
  snapshot 时间线:

  s100        s101        s102 (训练发起)      s103        s104
   │           │           │                   │           │
   ▼           ▼           ▼                   ▼           ▼
  ──────────────────────────────────────────────────────────►  时间

  训练任务 A 记录: read snapshot=102
       │
       └──► 半年后复现: time travel 回 s102 → 拿到完全一致的样本
            排查问题:   对比 s102 与 s103 → 定位是哪次变更引入的偏差
```

| 能力 | 没有 time travel | 有 time travel(Paimon snapshot) |
|---|---|---|
| 复现训练 | 几乎不可能,数据已被覆盖 | 按 snapshot 精确还原 |
| point-in-time | 容易取错版本,导致穿越 | 取指定时刻快照,不穿越 |
| 问题排查 | 无法对比历史 | 快照间 diff 定位变更 |
| 审计 | 无 | 每次 commit 有据可查 |

🎓 **老师视角(可复现是科学的底线)**:训练模型本质是做实验,**实验不可复现就不是科学,是玄学**。算法同学说「我这版模型 AUC 0.78」,如果你的平台不能复现出当时的样本,这个数字就无法验证、无法 debug。Paimon 的 snapshot 让「数据版本」变成一等公民——这是平台研发能交付的最硬核的「工程可信度」。面试/汇报时这一点最能体现你的专业。

### 5.5 Snapshot / 分区 / Compaction:对样本读吞吐的意义

流式 upsert 有个副作用:**频繁小批量写入会产生大量小文件**。小文件多了,批量训练读时要打开海量文件、寻址开销巨大,读吞吐崩掉。Paimon 用三个机制治理:

- **分区(partition)**:按 `dt` 等维度分区,训练读某天样本只扫该分区,避免全表。
- **Snapshot + manifest**:每个 snapshot 通过 manifest 索引到具体数据文件,读时按需裁剪(只读相关 manifest/文件)。
- **Compaction**:后台把小文件合并成大文件、把 LSM 各层合并,减少文件数和 merge 开销,保证读吞吐稳定。

```
  流式写产生小文件         compaction 合并后
  ┌──┐┌──┐┌──┐┌──┐         ┌────────────┐
  │f1││f2││f3││f4│   ───►   │  big file  │   训练读:打开 1 个文件
  └──┘└──┘└──┘└──┘         └────────────┘   而不是 N 个
  读要开 N 个文件                            吞吐稳定
```

| 机制 | 解决什么 | 对训练读的意义 |
|---|---|---|
| 分区 | 数据范围裁剪 | 只扫相关分区,扫描量降一个量级 |
| snapshot/manifest | 文件级裁剪 + 一致性视图 | 读到一致快照,按需读文件 |
| compaction | 小文件 / LSM 层数 | 文件数可控,读吞吐稳定 |

🎓 **老师视角(小文件是流式湖的头号运维敌人)**:一旦你让 Flink 高频流式写 Paimon,小文件问题几乎必然出现。表现是「训练读越来越慢、任务超时」。平台研发要把 **compaction 策略**(同步还是异步、触发频率、资源配额)纳入平台默认治理,而不是等算法同学投诉「读不动了」才去救火。**「写得爽 ≠ 读得动」,二者的平衡就是 compaction 的艺术**。

#### 5.5.1 写放大 vs 读放大:compaction 的权衡本质

compaction 不是「越勤越好」,它在三个量之间做权衡——**写放大、读放大、空间放大**。这是 LSM 系统的经典三角,平台研发必须有概念。

```
              compaction 频率
   稀疏(懒)  ◄──────────────────────────►  频繁(勤)
   ─────────────────────────────────────────────────
   写放大     低(少合并, 写得快)          高(反复重写同份数据)
   读放大     高(L0 文件多, 读要归并很多)   低(层数少, 读快)
   空间放大   高(同 key 多版本堆着)        低(及时合并去冗余)
   ─────────────────────────────────────────────────
   适合       高吞吐流式写、读不频繁         训练读频繁、要求稳定吞吐
```

落到 ML 平台:

- **写侧**(Flink 实时拼样本):希望写延迟低、不阻塞,倾向**异步 compaction**(写的归写,后台慢慢合)。
- **读侧**(Spark 批量训练读):希望文件数少、吞吐稳,倾向更积极的 compaction。
- **平台研发的活**:给样本表配一个折中策略——异步 compaction 为主,再用「专门的 compaction 作业 / `dedicated compaction`」在训练读之前把目标分区合并到位。**让算法同学读之前数据已经是「整理好的」状态**。

| 调控手段 | 作用 | 经验取向 |
|---|---|---|
| 同步 vs 异步 compaction | 写时合 / 后台合 | 流式高频写选异步 |
| compaction 触发阈值 | 多少文件触发合并 | 读密集场景调低阈值 |
| 独立 compaction 作业 | 解耦写与合并资源 | 大表强烈推荐 |
| 分区级 TTL/过期 snapshot | 控制冗余与空间 | 必配, 否则元数据膨胀 |

#### 5.5.2 脱敏案例:一次「训练读越来越慢」的排障

> 以下为脱敏示意案例,数字为典型量级/经验值,非真实生产数据。

某搜广推团队的实时样本表用 Flink 秒级 upsert,初期一切正常。两周后算法同学反馈「训练任务从 40 分钟拖到 3 小时,经常超时」。排障路径:

1. **看 `$files` 系统表**:发现单个 `dt` 分区下小文件数从几百涨到数十万——典型小文件爆炸。
2. **看 `$snapshots`**:snapshot 数量巨大,旧 snapshot 从未过期,manifest 也跟着膨胀,scan 规划阶段就很慢。
3. **根因**:只开了写入端的轻量 compaction,跟不上秒级写入产生小文件的速度;且没配 snapshot 过期。
4. **处置**:① 上独立 compaction 作业按分区合并;② 配 snapshot 过期(保留近 N 个);③ 训练读前对目标分区做一次 full compaction。
5. **结果**:训练读恢复到 ~40 分钟级别,且稳定。

这个案例的教训写进 Checklist 了:**compaction 和 snapshot 过期必须默认开启并托管,不能等出事**。

### 5.6 湖仓一体 vs 数仓+KV 双写:一次深度对照

这是面试和架构评审最常被追问的题:「为什么不用成熟的『数仓离线算 + KV 在线写』,非要上湖?」你要能从「数据一致性、运维成本、可复现」三个维度讲透,而不是只说「湖更先进」。

先把两套架构的数据流画出来对照:

```
   ① 数仓 + KV 双写(传统)
   ┌─────────┐                    ┌─────────┐
   │原始数据  │                    │实时数据  │
   └────┬────┘                    └────┬────┘
        │ 离线 Spark 计算 A            │ 在线 Flink 计算 B
        ▼                             ▼
   ┌─────────┐                    ┌─────────┐
   │ 数仓表   │  ←需对账→           │  KV 库   │
   │(训练读) │  (逻辑/口径漂移)    │(serving)│
   └─────────┘                    └─────────┘
   痛点: 计算逻辑 A≠B, 数据两份, 对账永无止境, skew 高发

   ② 湖仓一体(Paimon 真相源 + KV 派生)
   ┌─────────┐  ┌─────────┐
   │原始数据  │  │实时数据  │
   └────┬────┘  └────┬────┘
        ▼            ▼ 统一写入(同一套计算)
        ┌──────────────────────┐
        │  Paimon 真相源(一份)  │
        └──────┬────────┬───────┘
        Spark批读│        │Flink 消费 changelog 派生
               ▼        ▼
            训练    ──► KV 库(serving)
            一份真相, 同源派生, 无需对账
```

| 维度 | 数仓+KV 双写 | 湖仓一体(Paimon) | 谁赢 |
|---|---|---|---|
| 一致性 | 两套计算,易漂移 | 一份真相,派生 | 湖 |
| 对账成本 | 持续投入,永不结束 | 几乎消除 | 湖 |
| 更新能力 | 数仓不能改,KV 能改 | 主键 upsert 统一 | 湖 |
| 可复现 | KV 无历史,难复现 | snapshot/tag 精确复现 | 湖 |
| serving 延迟 | KV 毫秒级 | KV 派生,仍毫秒级 | 平 |
| 迁移成本 | 存量已建好 | 需改造接入 | 双写(短期) |
| 长期 TCO | 两套链路两份存储 | 一份存储 + 派生 | 湖 |

注意最后两行:双写不是一无是处——**存量系统、迁移成本、团队熟悉度**是它的现实优势。所以正确结论不是「无脑上湖」,而是:

- **新建系统**:直接湖仓一体,避免一开始就背双写的债。
- **存量系统**:渐进迁移,先把「最痛的、skew 最严重的」特征/样本表迁到湖,KV 改成从湖派生;不要一次性大爆破。

🎓 **老师视角(架构辩论的落点是「成本和风险」)**:被追问「为什么上湖」时,新人会答「湖支持 ACID、time travel、流批一体」——这是在背特性。资深的答法是落到**业务成本**:「双写架构里离线在线两套计算逻辑会漂移,对账是个无底洞,而且 KV 没历史导致模型出问题无法复现。湖把真相源统一成一份,KV 从它派生,从根上消掉对账成本和 skew,代价是一次性的接入改造。」——**讲特性是初级,讲成本和风险是高级**。

### 5.7 Paimon 能力到 ML 场景的全景映射

把第 5 章所有能力收口成一张「能力 → ML 价值」对照表,方便你贴到设计文档里:

| Paimon 能力 | 对应 ML 场景 | 解决的痛点 |
|---|---|---|
| 主键表 + upsert | 延迟反馈改 label、画像更新 | 不用重写整表 |
| partial-update merge engine | 多流拼接样本宽表 | 省掉复杂 join |
| aggregation merge engine | 实时累计特征 | 边写边聚合 |
| 流批一体 | Flink 写 + Spark 读 | 离在线共用一份表 |
| changelog | 在线特征同步、增量计算 | 离线改一条在线秒级跟上 |
| snapshot + time travel | 可复现训练、point-in-time | 不穿越、可复盘 |
| tag | 命名训练基线 | 长期保留关键版本 |
| 分区 + manifest 裁剪 | 训练读裁剪 | 扫描量降一个量级 |
| compaction | 训练读吞吐 | 治小文件、稳吞吐 |
| 系统表 | 治理与排障 | 可观测 |

## 6. 离在线一致性、血缘、元数据治理、多租户、成本

这一节是平台研发的「治理基本功」,也是简历上的加分项。

### 6.1 离在线一致性(train-serving consistency)

核心矛盾前面反复说了:离线训练用的特征,和在线 serving 用的特征,如果不一致,模型就劣化。一致性破坏的三种典型来源:

1. **逻辑不一致**:离线 Spark 一套计算代码,在线一套,慢慢漂移。
2. **时间不一致**:离线用了未来数据(穿越),在线只有当下数据。
3. **数据源不一致**:离线读数仓,在线读 KV,二者没对齐。

Paimon 视角的解法:**湖做真相源,在线 KV 从湖的 changelog 派生**——同源、同 schema,逻辑漂移和数据源不一致从根上被压住;point-in-time(time travel)压住时间不一致。

```
            ┌────────── Paimon 真相源 ──────────┐
            │  样本/特征主键表 + snapshot         │
            └───────┬────────────────┬──────────┘
        Spark 批读  │                │ Flink 消费 changelog
        (训练特征)   ▼                ▼  派生
                  训练 ◄── 同源同逻辑 ──► 在线 KV(serving)
```

详见 [../04-样本数据管理/样本离在线一致性.md](../04-样本数据管理/样本离在线一致性.md)。

### 6.2 数据血缘(lineage)

血缘 = 「这份数据从哪来、被谁加工、流向哪去」。ML 平台的血缘要打通:**特征 → 样本 → 模型 → 实验**。价值:

- 出问题能反查(模型劣化 → 哪个特征变了 → 哪个上游表改了)。
- 改 schema 能评估影响面(改这个特征会影响哪些模型)。
- 合规审计(这个模型用了哪些数据)。

Paimon 的 snapshot/manifest 元数据天然提供「表级 + 文件级」的版本血缘基础,平台层再叠加「特征-样本-模型」的逻辑血缘。

血缘的「反查」价值最能体现在事故现场:

```
   线上现象: 模型 v37 CTR 下跌 3%
        │ 沿血缘反查
        ▼
   模型 v37 ──用了──► 样本快照 snapshot=8821
                          │
                          ▼ 该快照来自
                     特征表 user_profile(改过 schema?)
                          │
                          ▼ 对比 snapshot diff
                     发现: feat_xxx 在 8820→8821 之间分布突变
                          │
                          ▼
                     定位上游: 某埋点字段口径变更 → 根因
```

没有血缘 + snapshot,这条反查链每一步都得靠人肉猜。有了它,排障从「几天」压到「几十分钟」。这是你能给老板讲的最硬的「降事故时长」成果。

### 6.3 元数据与治理

- **Catalog**:统一管理库表元数据(Paimon 支持 FileSystem / Hive / REST 等 catalog)。平台研发要选一个统一 catalog,避免元数据散落。生产里多用 REST catalog 或 Hive catalog 做统一入口,配合权限和审计。
- **系统表(system tables)**:Paimon 内置 `$snapshots`、`$schemas`、`$manifests`、`$files`、`$options`、`$tags` 等元数据表。这些是平台研发做治理/排障的「仪表盘」——比如用 `$files` 看小文件分布、用 `$snapshots` 看快照堆积、用 `$schemas` 看 schema 演进历史。**会查这些系统表,是平台研发排障的基本功**。
- **schema 演进**:Paimon 支持 schema evolution(加列、改类型),平台要定演进规则和兼容策略:只允许向后兼容(加可空列、安全类型放宽),禁止破坏性变更直接上线。
- **tag(标签快照)**:可以给某个 snapshot 打 tag(如 `train_v37_base`),长期保留不被过期清理,作为「可复现训练」的命名锚点。比记 snapshot 数字 ID 更可读。
- **数据质量**:行数、空值率、分布漂移监控,异常告警。

| 治理诉求 | Paimon 提供的抓手 | 平台层补什么 |
|---|---|---|
| 版本/审计 | snapshot + tag | 训练-版本绑定记录 |
| 排障可观测 | 系统表($files/$snapshots…) | 仪表盘 + 告警 |
| schema 安全 | schema evolution | 兼容规则 + 审批流 |
| 元数据统一 | catalog(REST/Hive) | 权限 + 入口收敛 |

### 6.4 多租户与配额

- **隔离**:不同业务线/团队的数据库表、计算资源、存储配额要隔离。
- **权限(RBAC)**:谁能读哪张特征表、谁能改样本表 owner。
- **配额(quota)**:防止某个任务把集群资源/存储打爆。

### 6.5 成本:存算分离

湖仓一体天然走 **存算分离**(数据在对象存储,计算按需起)。成本抓手:

- **存储**:对象存储便宜,但要靠分区 + 生命周期(TTL,如样本只留 90 天)+ compaction 控制冗余。
- **计算**:训练 GPU 贵,要靠调度提升利用率;读吞吐靠 compaction/分区裁剪降扫描量。

| 治理维度 | 不做的后果 | 平台研发抓手 |
|---|---|---|
| 离在线一致 | 模型线上劣化 | 同源派生 + point-in-time |
| 血缘 | 出事无法定位 | 表级+逻辑级血缘打通 |
| 元数据 | 表散落、schema 乱 | 统一 catalog + 演进规则 |
| 多租户 | 资源互相打架 | 隔离 + RBAC + 配额 |
| 成本 | 账单失控 | 存算分离 + TTL + compaction |

🎓 **老师视角(治理是平台研发的护城河)**:训练框架谁都能接一个,真正区分「初级」和「资深」平台研发的,是治理能力——**你能不能让一个百人算法团队在同一个平台上互不打架地跑实验,出了问题十分钟定位,半年后还能复现**。这些「看不见的工程」才是简历上最值钱的。

## 7. 平台研发在整体架构里的位置与交付物

你(平台研发)不在任何单一层,而是**横跨数据层到 Feature Store、并贯穿元数据/血缘/调度横切层**。你的核心交付物:

```
   平台研发的「责任田」(粗框)

   ┌──────────────────────────────────────────────┐
   │ 实验平台   推理服务   模型管理                   │  ← 算法/工程协作
   ├──────────────────────────────────────────────┤
   │ 训练平台(调度/资源)                            │  ← 平台研发主战场之一
   ├──────────────────────────────────────────────┤
   │ 特征平台 Feature Store ★★★                     │  ← 平台研发核心
   ├──────────────────────────────────────────────┤
   │ 样本平台 ★★★                                   │  ← 平台研发核心
   ├──────────────────────────────────────────────┤
   │ 数据湖底座(Paimon)★★★  +  数据层接入           │  ← 平台研发地基
   ├──────────────────────────────────────────────┤
   │ 元数据 / 血缘 / 调度 / 多租户 / 监控(横切)★★★  │  ← 贯穿全程
   └──────────────────────────────────────────────┘
```

交付物清单(也是你简历可以写的「成果」):

- 一套样本/特征表的存储规范(主键设计、分区策略、TTL、compaction 策略)。
- 离在线一致性方案(同源派生链路 + point-in-time 保证)。
- 可复现训练机制(snapshot/版本绑定 + 血缘)。
- 统一元数据 catalog 与 schema 演进规则。
- 多租户隔离与配额、监控告警体系。

### 7.1 平台演进三阶段路线图

平台不是一天建成的。给你一张「从 0 到成熟」的演进路线图,既能指导落地节奏,也能在汇报里展示你的全局规划能力。

```
  阶段一: 离线可用            阶段二: 离在线一致          阶段三: 实时+可复现
  (T+1, 跑通链路)            (消 skew, 上湖底座)         (分钟级, 工程可信)
  ────────────────────────────────────────────────────────────────────────
  数仓批样本                  Paimon 主键表做真相源        Flink 秒级 upsert 样本
  Hive 离线特征               在线 KV 从 changelog 派生     实时特征 + 在线学习
  人工拼样本                  point-in-time 防穿越          snapshot/tag 全量可复现
  无版本/难复现               统一 catalog + 血缘起步        compaction 托管 + 自动治理
  ────────────────────────────────────────────────────────────────────────
  目标: 模型能训出来          目标: 线上不再无故劣化        目标: 快、稳、可复盘
  风险: skew、不可复现        风险: 小文件、迁移成本        风险: 实时链路稳定性
```

| 维度 | 阶段一 | 阶段二 | 阶段三 |
|---|---|---|---|
| 时效 | T+1 | 小时级 | 分钟/秒级 |
| 一致性 | 弱(双链路) | 同源派生 | 同源 + 实时同步 |
| 可复现 | 几乎没有 | snapshot 起步 | tag + 血缘完整 |
| 底座 | 数仓+KV | 引入 Paimon | Paimon 流批一体 |
| 主要风险 | skew/穿越 | 小文件/迁移 | 实时稳定性 |
| 平台研发重心 | 跑通 | 消 skew + 治理 | 实时化 + 自动化 |

落地建议:**不要跳级**。很多团队想一步到位上阶段三的实时链路,结果一致性和治理(阶段二)没打牢,实时只是把错误更快地放大。正确顺序是先消 skew、立可复现(阶段二),再追实时(阶段三)。

🎓 **老师视角(演进比一步到位更难也更值钱)**:能画出这张路线图,说明你不只会干活,还懂「**什么时候该干什么、什么不该现在干**」。老板最怕的是研发只盯着炫技(上来就搞实时在线学习),却把地基(一致性、可复现)落下。你在评审里说出「我们先打阶段二的一致性,实时放阶段三」,这就是 staff 级别的判断力,直接写进简历的「架构演进规划」经历里。

## 业界怎么做

> 以下为业界公开常见做法的概括,数字为典型量级/经验值,非特定公司承诺。

- **Feature Store 化**:业界普遍把特征抽成独立的 Feature Store(开源代表如 Feast,大厂多为自研),核心都是「离线在线双库 + 统一定义」来压 train-serving skew。
- **湖格式选型**:Paimon / Iceberg / Hudi 是主流三家。Paimon 的差异化在 **LSM 主键表 + 流式 upsert + 原生 changelog**,对「实时更新 + 流批一体」的 ML 样本/特征场景贴合度高;Iceberg 生态最广但早期偏 append;Hudi 也主打 upsert,与 Paimon 各有侧重。
- **离在线统一趋势**:从「数仓+KV 双链路」向「湖做真相源 + KV 派生」演进,是近几年明显趋势。
- **实时化**:样本/特征链路从 T+1 向分钟级/秒级演进,实时样本拼接 + 在线学习(online learning)在头部搜广推团队已是标配诉求。
- **典型量级(经验值)**:大型搜广推样本日增可达百亿行级,特征数千到上万维;训练读样本要求吞吐稳定在 GB/s 量级——这些都直接逼着底座必须解决小文件和分区裁剪。

## 权衡取舍

| 取舍点 | 选 A | 选 B | 经验建议 |
|---|---|---|---|
| 存储底座 | 数仓+KV 双链路 | 湖仓一体(Paimon) | 新建优先湖仓一体;存量渐进迁移 |
| compaction | 同步(写时合并) | 异步(后台合并) | 高频写优先异步,降写延迟 |
| 实时度 | T+1 批 | 分钟/秒级流 | 按业务收益定,实时不是越快越好 |
| 一致性来源 | 离线在线各算 | 同源派生 | 一律同源派生,别图省事各算 |
| 样本存留 | 长留全量 | TTL 滚动 | 按可复现需求 + 成本平衡定 TTL |
| 特征更新 | 整批重算 | 主键 upsert | 更新型特征用 upsert |

🎓 **老师视角(取舍要绑业务收益)**:不要为了「技术先进」上实时、上湖。每个取舍都要能回答「**这给业务带来多少收益、花多少成本**」。比如把样本从 T+1 做到分钟级,如果业务模型对时效不敏感(用户兴趣本来就慢变),那这笔投入就是浪费。资深平台研发的判断力,体现在「敢于不做某些事」。

## 常见误区与反直觉

- **误区:上了数据湖就能扛在线点查,可以干掉 KV。**
  反直觉:湖的强项是批量扫描和一致性,**不是毫秒点查**。serving 仍需 KV,湖做真相源、KV 做派生缓存。
- **误区:流批一体就是「一个引擎同时跑流和批」。**
  反直觉:它的本质是「**一份存储两种读法**」,而非引擎统一。Paimon 一张表既被 Flink 流读也被 Spark 批读,这才是关键。
- **误区:离线 AUC 高就能上线。**
  反直觉:离线高常常是**特征穿越**造成的假象。没有 point-in-time 保证,离线指标越漂亮越要警惕。
- **误区:可复现是「锦上添花」。**
  反直觉:可复现是**底线**,不是加分项。不能复现的训练,等于不能 debug 的系统。
- **误区:小文件是小问题,反正能读。**
  反直觉:小文件会让训练读吞吐**断崖式下跌**,是流式湖的头号运维敌人,必须靠 compaction 主动治理。
- **误区:特征改了,在线定时刷一下就行。**
  反直觉:定时刷会引入分钟级到小时级的不一致窗口,变更期间线上用的是旧特征——用 changelog 做事件驱动同步才是正解。
- **误区:upsert 是「就地改那一行」,所以很快。**
  反直觉:湖上(对象存储)不支持随机改,upsert 实际是「追加新版本 + 后台 compaction 合并」。理解这点你才懂为什么会有小文件和读放大问题。
- **误区:实时化是平台先进性的最高目标,越实时越好。**
  反直觉:实时只在「业务对时效敏感」时才有收益。慢变特征(如长期画像)做实时是浪费,且放大稳定性风险。先把一致性和可复现做扎实,再谈实时。
- **误区:Spark 批读 Paimon 和读 Hive 没区别,迁过去就行。**
  反直觉:Paimon 主键表读是 merge-on-read,文件组织、compaction、snapshot 都会影响读性能,迁移后要重新调读并行度和 compaction 策略,不能照搬 Hive 经验。

## 平台侧落地 Checklist

- [ ] 样本/特征表统一用湖主键表,主键设计明确(req_id / user_id+item_id 等)。
- [ ] 分区策略定好(通常按 `dt`),配合训练读裁剪。
- [ ] compaction 策略默认开启(优先异步),纳入平台托管,不靠人工。
- [ ] 离在线同源:在线 KV 从湖 changelog 派生,禁止离线在线各算一套。
- [ ] point-in-time:拼样本走 event time + time travel,防穿越。
- [ ] 可复现:训练任务绑定 snapshot id / 数据版本,记录到元数据。
- [ ] 统一 catalog 管理元数据,定 schema 演进与兼容规则。
- [ ] 血缘打通:特征 → 样本 → 模型 → 实验。
- [ ] TTL / 生命周期:样本按需滚动删除,控存储成本。
- [ ] 多租户:库表/资源/存储隔离 + RBAC + 配额。
- [ ] 监控:行数、空值率、分布漂移、读吞吐、小文件数告警。
- [ ] snapshot 过期 + 关键版本打 tag(训练基线长留)。
- [ ] 系统表纳入排障仪表盘($files 看小文件、$snapshots 看堆积、$schemas 看演进)。
- [ ] 演进按阶段推进:先一致性 + 可复现(阶段二),再实时化(阶段三),不跳级。

## 对接话术

**对算法/训练同学:**

- 「你拼样本走 event time,我用 time travel 给你取请求那一刻的特征快照,保证不穿越。」
- 「延迟反馈不用你攒批重算,主键表 upsert 实时改 label,样本永远最新。」
- 「你训练时记一下读的是哪个 snapshot,以后复现/排查我能精确还原这份样本。」
- 「读样本慢了八成是小文件,我把 compaction 调一下,你别动代码。」

**对数仓/数据同学:**

- 「我们把样本特征收敛到 Paimon 主键表,一张表 Flink 流写、Spark 批读,你不用再为我维护一套对账链路。」
- 「在线 KV 我从这张表的 changelog 派生,数据源跟你这边同源,出问题好对齐。」

**对 serving/工程同学:**

- 「在线特征我用 Flink 消费 changelog 实时同步,变更秒级到 KV,不是定时刷。」
- 「模型签名里的特征列表我们对齐一下,serving 按签名拉特征,别错位。」

**常被问到的几句,准备好答案:**

- 问「为啥不直接读湖做 serving?」答:「湖扛不了在线点查 QPS 和延迟,它是真相源;在线还是 KV,但 KV 从湖派生,所以一致。」
- 问「迁到 Paimon 风险大不大?」答:「核心风险是小文件治理和读性能调优,我们已经把 compaction 做成平台托管,迁移按表灰度,先迁最痛的样本表。」
- 问「历史数据怎么办?」答:「存量数据导入后打 tag 作为基线,新链路接上流式写,老链路并行跑一段对账,确认一致再切。」

## 自测三问

**Q1:为什么说「离线 AUC 高不一定能上线」?这跟数据湖底座有什么关系?**

参考答案:离线 AUC 高常常是「特征穿越」(用了请求时刻之后才有的未来数据)造成的假象,上线后拿不到未来数据,效果暴跌。这本质是 point-in-time correctness 问题。数据湖底座(如 Paimon)的 time travel / snapshot 能力,让拼样本时按 event time 精确取「请求那一刻」的特征版本,从源头防穿越。所以底座的版本能力直接决定离线指标可信度。

**Q2:Paimon 的主键表 upsert 和 changelog 分别解决 ML 平台的什么问题?举例说明。**

参考答案:主键表 upsert 解决「更新」——延迟反馈改 label(曝光时 label=0,转化回来 upsert 成 1)、特征回填、画像特征更新,都不用重写整张表,按主键原地改一行。changelog 解决「增量驱动」——上游一旦 upsert,changelog 吐出变更流(+I/-U/+U/-D),下游(训练宽表、在线 KV、实时统计)按变更增量更新,实现「离线改一条、在线秒级跟上」,而不是定时全量刷。

**Q3:作为平台研发,你会如何保证「离在线一致性」?讲清原理。**

参考答案:一致性破坏有三个来源——逻辑不一致(离线在线各写一套计算)、时间不一致(穿越)、数据源不一致(离线读数仓/在线读 KV 没对齐)。我的方案是「**湖做真相源,在线 KV 从湖的 changelog 派生**」:同源同 schema 压住逻辑和数据源不一致;point-in-time(time travel)压住时间不一致。再叠加血缘(特征→样本→模型)做出问题反查,和监控(分布漂移)做主动发现。核心一句:别让离线在线各算一套,一律同源派生。

**Q4(加餐):为什么流式写 Paimon 一段时间后训练读会变慢?你怎么治?这背后是什么权衡?**

参考答案:流式高频 upsert 会在 LSM 的 L0 不断追加小文件,文件数膨胀;同时 snapshot 若不过期,manifest 也膨胀,scan 规划变慢。两者叠加导致训练读变慢甚至超时。治理手段:开独立 compaction 作业按分区合并、配 snapshot 过期(保留近 N 个 + 给关键版本打 tag 长留)、训练读前对目标分区做 full compaction。背后的权衡是 LSM 的「写放大/读放大/空间放大」三角——compaction 越勤读越快但写放大越高,流式写场景要在「写延迟低」和「读吞吐稳」之间折中,典型做法是异步 compaction + 读前整理。
