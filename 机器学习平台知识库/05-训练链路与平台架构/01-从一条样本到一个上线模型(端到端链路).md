## 0. 30 秒速通(TL;DR)

- **一条样本的一生**:用户来请求 → 在线服务给候选打分,同时**把当时用到的特征拍快照存下来**(关键!)→ 用户看没看、点没点产生**曝光/反馈日志** → 把"快照特征"和"行为标签"按某个 key **拼接**成一条带 label 的样本 → 经过**采样/清洗** → 喂给**训练**算梯度更新参数 → **离线评估** AUC/GAUC → **小流量灰度 / AB** → **全量上线热更新到 serving** → **线上监控** → 效果**回流**指导下一轮。
- **整条链路是个环,不是一条线**:今天上线的模型,会影响明天的曝光,曝光又变成明天的训练样本。所以一旦某环出问题(比如特征穿越、样本偏置),会**自我强化、越滚越歪**。
- **平台研发的核心职责**:不是调模型,而是**保证这条流水线每个接口的数据是「对的、齐的、准时的、可追溯的」**。你交付的是 SLA、是契约、是可回滚,不是 AUC。
- **两条主链路**:**离线批训练**(T+1 跑批,产出全量/增量版本,稳但慢)和 **流式训练 / 在线学习**(实时样本流,分钟级 / pass 级出 delta 模型,快但脆)。大厂推荐系统两条往往**并存**。
- **最常爆的三个雷**:① 特征穿越(训练用了线上拿不到的未来信息);② 在线/离线特征不一致(train-serving skew);③ 样本回流断流但没人发现(模型「饿死」在旧数据上)。
- **数据新鲜度直接换算成钱**:在强时效场景(信息流、电商大促),样本从产生到进训练每多延迟 1 小时,效果可能掉 0.X%~百分之几(经验值,场景差异大),平台要把「端到端时延」当一级指标盯。

## 1. 先建立大局观:为什么是「闭环」而不是「流水线」

很多新同学第一次听这条链路,脑子里画的是一条**直线**:数据进来 → 训练 → 上线 → 完事。这是错的,而且这个错误会让你在排查问题时完全找不到北。

正确的心智模型是一个**自反馈的环**:

```
              ┌──────────────────────────────────────────────┐
              │                                                │
              ▼                                                │
        ┌───────────┐    曝光/点击     ┌───────────┐           │
   用户 │ 在线推荐  │ ──────────────▶ │ 用户行为  │           │
  ──────▶│ (serving) │                 │  日志     │           │
        └───────────┘                 └─────┬─────┘           │
              ▲                             │                 │
              │ 加载新模型                   ▼                 │
        ┌───────────┐                 ┌───────────┐           │
        │  上线/灰度 │◀──── 评估通过 ──│ 样本+训练 │───────────┘
        └───────────┘                 └───────────┘
                                       (本轮模型影响下一轮曝光)
```

**为什么这件事对平台研发是「天大的事」?**

因为环意味着**误差会被反馈放大**。举两个真实会发生的场景:

1. **曝光偏置(exposure bias)**:模型只给用户看它「以为」用户喜欢的东西,用户也只能在这些里点。于是训练样本里全是模型已经认可的内容,模型越来越「自信」,探索越来越少。这不是模型 bug,是**链路结构性问题**,要靠平台层面的随机曝光 / EE(Explore-Exploit)流量来打破。
2. **脏数据自我繁殖**:某天上游埋点改了字段含义没通知,样本里某个特征全变成默认值,模型学歪;歪掉的模型上线后曝光更差,产生更歪的样本……如果回流和监控没卡住,**三天就能把一个好模型喂废**。

🎓 **老师视角**:你要记住一句话——**「在监督学习里,模型只能学到你喂给它的世界」**。线性流水线思维会让你只盯「这一趟数据对不对」;闭环思维会让你问「这趟数据是被上一版模型怎么塑造出来的、我这版又会怎么塑造下一趟」。平台研发的高级感,就在于你能在设计阶段就把「打破闭环偏置」的口子(随机流量、纠偏样本、负采样策略)留出来,而不是等算法同学来追着你要。

## 2. 端到端全链路总图

把闭环展开成「按时间推进」的全链路,这是全篇的**主图**,后面每一节都是在放大其中一块。

```
 时间轴 ───────────────────────────────────────────────────────────────────▶

 ① 在线打分                ② 日志                ③ 样本工程            ④ 特征供给
 ┌─────────────┐          ┌──────────┐          ┌──────────────┐     ┌────────────┐
 │ 用户请求     │          │ 曝光日志  │          │ join by key   │     │ 训练特征    │
 │ → 召回/排序  │ 特征快照 │ 点击日志  │ 实时/T+1 │ + label 打标  │     │ 读取(slot) │
 │ → 打分返回   │─────────▶│ 转化日志  │─────────▶│ + 采样/去噪   │────▶│ 在线一致性  │
 │ → 拍特征快照 │          │ 时长/停留 │          │ + 特征对齐    │     │ 校验        │
 └─────────────┘          └──────────┘          └──────────────┘     └─────┬──────┘
   feature snapshot          raw logs            labeled samples            │
                                                                            ▼
 ⑦ 监控/回流          ⑥ 上线/灰度          ⑤ 评估              ④' 训练
 ┌────────────┐      ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
 │ 线上指标    │      │ 模型推到serving│    │ 离线AUC/GAUC  │     │ 读样本→前向   │
 │ CTR/CVR     │◀─────│ 灰度比例控制   │◀───│ 分组对比基线   │◀────│ →反向→更新参数 │
 │ 特征漂移    │ 回流 │ 一致性校验     │ 通过│ 业务指标预估    │ 产出│ → 产出模型版本 │
 │ 模型衰减    │      │ 可回滚         │     │ 卡阈值放行     │     │ base+delta    │
 └─────┬──────┘      └──────────────┘     └──────────────┘     └──────────────┘
       │
       └──────────────────────▶ 反哺特征/样本/下一轮训练(回到 ①)
```

**逐环节的「平台职责 / 输入 / 输出 / 卡点」速查表:**

| 环节 | 平台核心职责 | 输入(交付物来源) | 输出(交付物) | 最常见卡点 |
|---|---|---|---|---|
| ① 在线打分 + 特征快照 | 在打分时**原子地**记录用到的特征 | 用户请求、特征服务 | feature snapshot(带 request_id) | 快照丢失/不全、采样过低 |
| ② 曝光/反馈日志 | 埋点规范、日志可靠落地、去重 | 客户端/服务端埋点 | 标准化行为日志流 | 埋点漏报、重复、口径不一 |
| ③ 样本拼接+标签+采样 | join 引擎、归因窗口、采样策略 | 快照 + 行为日志 | labeled samples | 拼接丢失、标签穿越、回补 |
| ④ 特征供给 | 训练特征读取、在线/离线一致性 | 样本 + 特征仓 | 训练可读的特征张量 | train-serving skew |
| ⑤ 训练 | 调度、资源、checkpoint、产出版本 | 特征 + 模型代码 | 模型版本(base+delta) | OOM、慢节点、PS 热点 |
| ⑥ 评估 | 离线指标计算、基线对比、卡口 | 模型 + 测试集 | 通过/拒绝 + 指标报告 | 评估集穿越、指标虚高 |
| ⑦ 灰度/AB | 流量切分、分桶、统计显著性 | 候选模型 + 实验配置 | 实验结论 | 分桶污染、观测期不足 |
| ⑧ 上线/热更新 | 模型分发、版本切换、回滚 | 通过实验的模型 | serving 加载新模型 | 加载抖动、版本不一致 |
| ⑨ 监控/回流 | 全链路指标、漂移检测、血缘 | 线上日志 + 指标 | 告警 + 回流数据 | 断流无感、漂移漏报 |

🎓 **老师视角**:这张表你应该背下来「平台职责」那一列。面试和向上汇报时,**别人问你「机器学习平台是干嘛的」,你不要答「跑训练任务的」**——你要答「保障这 9 个环节的数据契约和 SLA」。前者是运维视角,后者是平台架构师视角,差一个段位。

## 3. 环节逐个拆:在线打分与特征快照

### 3.1 这一环到底发生了什么

用户发起一次请求(刷一下信息流、打开一个商品页),在线推荐系统在**几十毫秒**内完成:召回(从亿级物料里捞出几千个候选)→ 粗排 → 精排(模型打分)→ 重排 → 返回 Top-N。

**对样本链路而言,最关键的不是「打了多少分」,而是「打分时用了哪些特征值」。** 这就是**特征快照(feature snapshot)**。

### 3.2 为什么必须当场拍快照,不能事后补

这是整条链路最反直觉、也最重要的设计点之一。

```
   错误做法(事后补特征)                正确做法(在线拍快照)
   ─────────────────────                ─────────────────────
   T0: 打分(用 user_ctr_7d=0.12)        T0: 打分,同时记录:
        ↓                                     request_id=abc
   T0+6h: 训练时去特征仓查 user_ctr_7d      feature_snapshot={
        ↓ 但此时已经变成 0.15!                 user_ctr_7d: 0.12,
   ❌ 训练用的特征 ≠ 线上打分用的特征          item_price: 99, ...
   ❌ 模型学到的关系是错的(穿越)            }
                                          ↓
                                          T0+6h: 训练直接读快照里的 0.12
                                          ✅ 训练特征 == 线上特征
```

特征值是**随时间变化**的(用户的 7 天点击率、物料的实时热度、库存……)。如果训练时去「现查」,查到的是**未来时刻**的值,这就是经典的**特征穿越 / 数据穿越(label leakage / feature leakage)**。模型在离线评估时 AUC 高得离谱,一上线就垮——因为线上根本拿不到未来的值。

🎓 **老师视角**:「特征穿越」是平台研发面试的高频题,也是线上事故的高频因。你只要记住一句话:**「训练所见,即线上所见」(train as you serve)**。实现这句话的工程手段,99% 的工业系统选择的是**在线拍快照**,而不是离线对齐时间点(后者理论可行但极易出错,且很多实时特征根本无法回溯)。

### 3.3 平台职责与交付物

| 项 | 内容 |
|---|---|
| **平台职责** | 提供快照 SDK / 切面,在打分链路无侵入地落快照;控制采样率;保证 request_id 全链路透传 |
| **输入** | 用户请求、特征服务返回值、模型打分上下文 |
| **输出交付物** | feature snapshot 流(通常落 Kafka → 数据湖,带 request_id、timestamp、模型版本号) |
| **典型量级** | 大厂精排每天快照量可达**百亿~千亿条**;单条快照特征数百到数千个 slot;为控成本常按 1%~100% 采样(曝光样本一般全采,未曝光按需) |
| **常见卡点** | ① 快照写入拖慢打分 RT(必须异步、降级);② 采样率过低导致样本量不够;③ 快照只存了精排没存粗排,粗排模型没法训 |

🎓 **老师视角**:快照的「采样率」是个经典的成本/效果权衡。全量快照存储和计算成本极高,但采太狠又会让长尾用户/物料样本不足。平台研发要做的是**提供「可配置、可分层」的采样能力**(比如曝光位全采、未曝光按 item 热度分层采),而不是一刀切。这个口子留得好不好,直接决定算法同学能不能做精细化建模。

## 4. 环节逐个拆:曝光/反馈日志

### 4.1 行为日志的层次

用户的反馈不是只有「点没点」,而是一个**漏斗**:

```
   曝光(impression) ──▶ 点击(click) ──▶ 停留/播放(dwell/play) ──▶ 转化(conversion)
        100%               ~5-10%            ~80% of click          ~1-3% of click
   (经验值,场景差异极大,仅示意漏斗形状)
```

每一层都是一个**标签来源**:CTR 模型用点击,CVR 模型用转化,时长模型用 dwell,多目标模型全都要。

### 4.2 平台要解决的脏活

| 问题 | 表现 | 平台对策 |
|---|---|---|
| **去重** | 客户端重试、双端上报导致一次曝光报多条 | 基于 (request_id, item_id, 事件类型) 幂等去重 |
| **乱序** | 点击日志比曝光日志先到 | 用事件时间(event time)而非处理时间,窗口等待 |
| **丢失** | 客户端弱网、进程被杀 | 服务端兜底埋点 + 丢失率监控 |
| **口径不一** | 「曝光」定义:进入可视区?停留 >1s? | 平台统一埋点规范 + 字典,强制对齐 |

🎓 **老师视角**:日志这一环,平台研发最容易低估的是**「口径治理」**的工作量。一个「曝光」的定义,产品、客户端、算法、数仓四方理解可能都不一样。你做的不只是管道,而是**定义和守护「事件的语义契约」**。一旦口径漂了,下游所有样本和指标全错,而且**错得很隐蔽**(数据照常产出,只是含义变了)。所以埋点 schema 变更必须走平台审批 + 影响面分析,这是制度问题不是技术问题。

### 4.3 交付物与量级

- **输出**:标准化、去重、带事件时间的行为日志流(Kafka topic / 数据湖表)。
- **典型量级**:大厂日活级产品,曝光日志日增**百亿~万亿行**,是整条链路里数据量最大的一环。
- **SLA**:实时链路要求曝光日志**秒级~分钟级**可消费;T+1 链路要求次日凌晨**前置时间**(比如 6:00 前)产出完整分区。

## 5. 环节逐个拆:样本拼接 + 打标签 + 采样

这是整条链路里**逻辑最复杂、bug 最多、最考验平台功力**的一环。也是 [04-样本数据管理/01-样本全生命周期与日志拼接](../04-样本数据管理/01-样本全生命周期与日志拼接.md) 的核心主题,这里讲它在端到端链路里的位置。

### 5.1 拼接(join):把特征快照和行为日志缝起来

```
   feature snapshot              behavior logs
   ┌──────────────────┐         ┌──────────────────┐
   │ request_id: abc  │         │ request_id: abc  │
   │ item_id: 1001    │   JOIN  │ item_id: 1001    │
   │ features: {...}  │ ◀─────▶ │ click: 1         │
   │ ts: T0           │  by key │ dwell: 12s       │
   └──────────────────┘         │ convert: 0       │
                                └──────────────────┘
            │
            ▼
   ┌─────────────────────────────────────────────┐
   │ 一条完整训练样本                              │
   │ features: {...}  label_ctr: 1  label_cvr: 0  │
   └─────────────────────────────────────────────┘
```

join 的 key 通常是 `(request_id, item_id)` 或更细的 `(request_id, item_id, position)`。

### 5.2 打标签的两个魔鬼细节:归因窗口 & 标签回补

**归因窗口(attribution window)**:用户**什么时候**的转化算这次曝光的功劳?

```
   T0 曝光 ──────────────────────────────────────────▶
            │←─ 点击 ─┤                    
                       │←──── 转化归因窗口(如 7 天)────┤
                                                        ↑
                                            这天的转化还算 T0 的 label 吗?
```

- 窗口太短:漏掉延迟转化(用户加购物车 3 天后才买),正样本偏少。
- 窗口太长:出模型时还不知道最终 label,**时效性差**。
- **CVR 场景特有的「延迟反馈(delayed feedback)」难题**:转化可能几小时甚至几天后才发生,流式训练时这条样本到底先当负样本还是等?这是个专门的研究方向(平台要支持「假负样本回补」机制)。

**标签回补(label backfill)**:先以负样本进入样本流,等真实转化到了再**回补**修正。平台要支持样本的「**先出后改**」或「**延迟出样**」两种模式。

### 5.3 采样:为什么不能全量喂

| 采样手段 | 目的 | 注意点 |
|---|---|---|
| **负采样(negative sampling)** | 正负样本极不平衡(CTR 正样本可能 <5%),负样本太多浪费算力 | 采样后要**校准(calibration)**,否则预估值整体偏高 |
| **重要性采样 / 加权** | 让模型更关注高价值样本 | 权重要回传给 loss |
| **去噪 / 过滤** | 剔除爬虫、异常点击、误触 | 规则 + 模型双重过滤 |

🎓 **老师视角**:负采样后**一定要做概率校准**,这是平台研发最容易踩的「无声坑」。你采样改变了正负比例,模型输出的 pCTR 就不再是真实概率,而广告计费、ROI 计算都依赖真实概率。算法同学知道要校准,但**校准逻辑放在训练侧还是 serving 侧、用什么公式**,往往要平台来统一约定和实现。你要主动问一句「这个采样校准在哪做」,别等出了计费事故才发现两边都以为对方做了。

### 5.4 平台职责与卡点

| 项 | 内容 |
|---|---|
| **平台职责** | 提供 join 引擎(批 + 流)、归因窗口配置、采样框架、标签回补机制、样本质量校验 |
| **输出交付物** | labeled & sampled samples(训练直接可读的格式,如 TFRecord / Parquet / 自定义列存) |
| **典型量级** | 拼接成功率必须监控,健康值 **>99%**;丢失率每升高 1% 都要告警 |
| **常见卡点** | ① join 丢失(快照或日志一方缺失);② 标签穿越(用了归因窗口之外的转化);③ 回补导致样本重复或不一致;④ 采样后忘了校准 |

## 6. 环节逐个拆:特征供给(训练侧)

样本里已经带了特征快照,但训练时还要做**特征工程的最后一公里**:把原始特征值转成模型能吃的张量。详见 [03-特征数据管理/02-特征平台(Feature Store)架构](../03-特征数据管理/02-特征平台(Feature Store)架构.md)。

### 6.1 训练-服务一致性(train-serving consistency)

这是 Feature Store 存在的**头号理由**。同一套特征计算逻辑,必须在**在线打分**和**离线训练**两条路径上产出**完全一致**的结果。

```
                    ┌──── 在线路径(serving)────┐
   原始特征逻辑      │ Java/C++ 实现,低延迟       │ ── pCTR
   (如 hash + 分桶) │                            │
        │           └────────────────────────────┘
        │           ┌──── 离线路径(training)────┐
        └──────────▶│ Spark/Python 实现           │ ── 训练
                    └────────────────────────────┘
                          ⚠️ 两套实现极易不一致!
```

**两套实现 = 两套 bug 的可能**。比如在线用了浮点、离线用了双精度;在线对缺失值填 0、离线填均值。这种细微差异叫 **train-serving skew**,会让模型「学的」和「用的」对不上,效果莫名其妙变差,极难排查。

🎓 **老师视角**:解决 skew 的现代答案是「**特征逻辑统一**」——要么用快照(根本不在离线重算,直接用线上当时的值,这是最彻底的);要么用同一份特征定义代码在两侧执行(Feature Store 的 transformation 复用)。作为平台研发,你要警惕任何「在线一套、离线一套」的特征实现,**这是 skew 的温床**。在做架构评审时,看到「离线重新计算实时特征」的方案,要立刻举手。

### 6.2 slot 概念与 ID 化

工业推荐模型的特征通常以 **slot** 组织(一个 slot = 一类特征,如 slot 1=user_id,slot 2=item_id,slot 3=user_ctr_7d……),离散特征经过 hash 变成 **feature sign / fea-sign**,落到 embedding 表里。这部分名词详见 [02-模型训练名词大全/01-训练流程名词(pass·slot·batch·epoch)](../02-模型训练名词大全/01-训练流程名词(pass·slot·batch·epoch).md)。

平台研发要理解:**特征数量爆炸是常态**,推荐大模型的 embedding 参数可达**百亿~万亿**,绝大部分参数都在稀疏 embedding 上。这直接决定了训练架构必须用 **PS(Parameter Server)** 来分布式存储和更新这些参数。

## 7. 环节逐个拆:训练(读样本/算梯度/更新参数)

### 7.1 训练内部的循环

```
   ┌─────────────────────────────────────────────────────────┐
   │  for each batch:                                          │
   │    ① 读样本 (sample reader, 流水线预取避免 IO 等待)        │
   │    ② 前向 forward:特征 → embedding lookup → 网络 → 预估值 │
   │    ③ 算 loss(预估值 vs label)                            │
   │    ④ 反向 backward:算梯度                                │
   │    ⑤ 更新参数:dense 在 worker,sparse embedding 推到 PS  │
   └─────────────────────────────────────────────────────────┘
                              │
                              ▼ 每隔 N 个 batch / 一个 pass
                        产出 checkpoint / 模型版本
```

### 7.2 推荐训练的典型架构:PS-Worker

```
        ┌──────────┐   ┌──────────┐   ┌──────────┐
        │ Worker 1 │   │ Worker 2 │   │ Worker N │   读样本+算梯度
        └────┬─────┘   └────┬─────┘   └────┬─────┘
             │ pull/push embedding & 梯度    │
        ┌────┴────────────┴────────────────┴─────┐
        │   Parameter Server (PS) 集群             │   存稀疏参数
        │   shard0   shard1   ...   shardK          │   (百亿~万亿)
        └──────────────────────────────────────────┘
```

- **Worker**:读样本、前向反向算梯度。
- **PS**:分片存储海量稀疏 embedding,接收梯度更新参数。
- 与通用深度学习的 **AllReduce(数据并行)** 不同,推荐场景因稀疏参数巨大,主流是 **PS 架构**(也有 PS + AllReduce 混合:dense 用 AllReduce,sparse 用 PS)。

### 7.3 平台职责与卡点

| 项 | 内容 |
|---|---|
| **平台职责** | 资源调度(GPU/CPU/PS)、任务编排、checkpoint 管理、容错恢复、慢节点检测、产出模型版本 |
| **输出交付物** | 模型版本(完整 base 模型 + 可选的 delta 增量) |
| **典型量级** | 离线全量训练一轮数小时~数十小时;增量训练分钟级~小时级;流式 pass 级出模型可达分钟级 |
| **常见卡点** | ① PS 热点(某些高频 ID 把单个 shard 打满);② 慢节点拖慢同步;③ OOM(embedding 太大);④ checkpoint 太大导致保存/恢复慢;⑤ 数据读取成为瓶颈(GPU 等数据) |

🎓 **老师视角**:平台研发盯训练,**第一要盯的不是「准不准」(那是算法的事),而是「快不快、稳不稳、省不省」**:GPU 利用率(经验上能把整体利用率做到 50%+ 已经不错)、单 step 耗时、是否有数据 IO 等待、PS 是否倾斜、failover 后能否秒级恢复。你优化掉一个数据读取瓶颈,可能比算法调一周参省的钱还多。**「训练成本」是平台研发能直接背的核心指标。**

### 7.4 容错与慢节点:大规模训练的两个长期顽疾

推荐训练动辄几十上百个节点连续跑数小时,**「有节点会挂」「有节点会慢」是常态而非异常**,平台必须把这两件事当成设计前提。

```
   容错(fault tolerance)              慢节点(straggler)
   ─────────────────────              ─────────────────────
   节点宕机 / 网络抖动                  某 worker 因硬件/数据倾斜变慢
        ↓                                   ↓
   从最近 checkpoint 恢复               同步训练时,所有节点等最慢的那个
   (而非从头重训)                      → 整体被拖慢(木桶效应)
        ↓                                   ↓
   要点:checkpoint 频率 vs 成本         对策:慢节点检测 + 任务重分配
        checkpoint 太密 → 拖慢训练            / 备份任务(backup worker)
        太疏 → 故障损失大                     / 异步训练降低同步等待
```

| 问题 | 平台对策 | 权衡 |
|---|---|---|
| **节点宕机** | 周期性 checkpoint + 自动 failover 恢复 | checkpoint 越密恢复损失越小,但占 IO、拖慢训练 |
| **慢节点(straggler)** | 检测 + 重调度 + backup worker | 同步训练受木桶效应制约,异步训练缓解但带来梯度延迟 |
| **PS 热点 shard** | 高频 ID 打散、shard 再均衡 | 打散增加通信,要权衡 |
| **大 checkpoint** | 增量 checkpoint、异步保存 | 实现复杂度上升 |

🎓 **老师视角**:同步 vs 异步训练是个经典权衡,平台研发要能讲清楚。**同步**:每步所有节点对齐梯度,收敛稳定、可复现,但被**最慢节点拖累**(木桶效应),节点越多越明显。**异步**:节点各跑各的、不互相等,吞吐高、抗慢节点,但存在**梯度延迟(stale gradient)**,收敛性差一些。推荐稀疏场景因 embedding 更新天然稀疏(不同样本碰的 ID 不同),对异步的容忍度比稠密 CV/NLP 模型高,所以异步 PS 在推荐界用得多。你不需要自己推导收敛性,但要知道**「为什么推荐能用异步而很多场景不行」**——答案是稀疏。

## 8. 环节逐个拆:评估(离线 AUC/GAUC)

模型训出来不能直接上线,先**离线评估**——用没参与训练的数据集打分,看指标。

### 8.1 核心指标

| 指标 | 含义 | 推荐场景为什么用它 |
|---|---|---|
| **AUC** | 排序能力:随机一对正负样本,模型给正样本打分更高的概率 | 推荐本质是排序问题,AUC 直接衡量排序质量 |
| **GAUC** | 分组 AUC(按 user 分组算 AUC 再加权平均) | 更贴合推荐「给每个用户内部排序」的真实场景,比全局 AUC 更可信 |
| **LogLoss** | 预估概率的准确度 | 广告计费依赖预估值绝对准确,不只看排序 |
| **Calibration** | 预估 CTR 与真实 CTR 的比值 | 校验有没有系统性高估/低估 |

🎓 **老师视角**:为什么推荐界偏爱 **GAUC** 而不只看 AUC?因为全局 AUC 可能被「区分活跃用户和不活跃用户」这种**简单任务**刷高,但用户真正在意的是「在我个人的候选列表里排得对不对」。GAUC 把它拆到每个用户内部算,更能反映线上体感。平台研发要知道这个区别,因为**评估指标的口径定义、分组逻辑、加权方式都要平台来实现和固化**,口径一旦不一致,A 模型和 B 模型就没法公平比。

### 8.2 评估这一环的隐形雷:评估集穿越

- 评估集如果和训练集**时间上重叠或泄露**,AUC 会虚高。正确做法是**时间切分(temporal split)**:用 T 日及之前训练,用 T+1 日评估。
- 平台要**强制**评估集与训练集的时间隔离,不能让算法同学手滑用了同一份数据。

### 8.3 平台职责

| 项 | 内容 |
|---|---|
| **平台职责** | 提供标准评估流程、指标计算库、基线模型对比、卡口阈值(不达标不许进灰度) | 
| **输出交付物** | 评估报告 + 放行/拦截决策 |
| **常见卡点** | 评估集穿越、指标实现不一致、基线选错(和一个弱基线比当然赢) |

---

## 9. 环节逐个拆:灰度 / AB 实验

离线指标好 ≠ 线上效果好(离线在线一致性是另一个大坑)。所以必须**小流量线上验证**。

### 9.1 灰度 vs AB 实验

```
   全量流量 100%
   ┌─────────────────────────────────────────────┐
   │ 对照组(老模型) 95%  │ 实验组(新模型) 5%   │  ← AB 实验:对比效果
   └─────────────────────────────────────────────┘
                                  │
                       逐步放量 5% → 20% → 50% → 100%   ← 灰度:控制风险
```

- **灰度(canary/gray release)**:重点是**控风险**,先放一点点,出问题立刻回滚,逐步放量。
- **AB 实验**:重点是**比效果**,实验组 vs 对照组,看统计显著的业务指标差异。
- 两者常结合:先小流量灰度(看会不会崩),稳了再做正式 AB(看效果好不好)。

### 9.2 分桶与流量隔离

平台要保证:
- **分桶正交 / 互斥**:不同实验的流量不能互相污染(用分层实验框架,如 Google 的 overlapping experiment 体系)。
- **分桶一致性**:同一用户在实验期内稳定落在同一桶,否则体验跳变、数据不可信。
- **观测期足够**:实验要跑够时间(通常**至少一周**,覆盖工作日/周末波动),才能下结论。

🎓 **老师视角**:平台研发在 AB 这一环最该懂的是「**别让人偷偷看中间结果就下结论**」。实验跑了两天看着涨了就全量,极可能是**新奇效应(novelty effect)**或随机波动,过几天就回落甚至变负。你要在平台上把「最小观测期」「显著性检验」做成**硬卡口**,而不是靠人自觉。这是平台帮业务「守住科学性」的价值。

### 9.3 交付物与卡点

| 项 | 内容 |
|---|---|
| **平台职责** | 流量切分框架、分桶管理、指标实时统计、显著性检验、一键回滚 |
| **输出交付物** | 实验结论(涨跌幅 + 置信度)+ 放量决策 |
| **典型量级** | 实验组初始流量常 1%~5%;观测期 3~14 天 |
| **常见卡点** | 分桶污染、流量不足导致不显著、新奇效应误判、回滚不及时 |

---

## 10. 环节逐个拆:全量上线与模型热更新

实验通过,模型全量上线。**关键词:热更新、不停服、可回滚。**

### 10.1 模型分发与加载

```
   训练产出模型 ──▶ 模型仓库(版本化) ──▶ 分发(推/拉) ──▶ serving 节点加载
        │                                                      │
        │                                              ┌────────┴────────┐
        │                                              │ 双 buffer 切换:  │
        │                                              │ 加载新模型到备用  │
        │                                              │ buffer,校验通过  │
        │                                              │ 后原子切换流量    │
        │                                              └─────────────────┘
        └──▶ 全程带版本号,任何节点可追溯加载的是哪个版本
```

- **热更新**:serving 不重启,在线把新模型加载进内存,**原子切换**(双 buffer:加载到备用,校验 OK 后切换),避免请求打到加载到一半的模型。
- **灰度加载**:不是所有 serving 节点同时切,先切一部分,观察无异常再全切。
- **回滚**:必须能**秒级回退**到上一个版本(版本号 + 模型文件都留着)。

🎓 **老师视角**:模型热更新最隐蔽的坑是「**加载抖动**」——切换瞬间,部分请求用新模型、部分用旧模型,或者加载占用资源导致 RT 飙升。平台研发要保证切换是**原子的、资源隔离的**。还有一个高级坑:**embedding 增量更新**——大模型几十 GB,不可能每次全量推,要支持只推变化的部分(delta)。这就引出下一节的版本管理。

### 10.2 交付物与卡点

| 项 | 内容 |
|---|---|
| **平台职责** | 模型版本仓、分发系统、热加载、双 buffer、灰度加载、一键回滚 |
| **输出交付物** | serving 集群加载新版本,流量平滑切换 |
| **常见卡点** | 加载抖动、新旧版本不一致(分发未完成就切流量)、大模型分发慢、回滚演练没做过(真出事才发现回滚也是坏的) |

---

## 11. 环节逐个拆:线上监控与效果回流

模型上线**不是终点,是新一轮的起点**。

### 11.1 要监控什么

```
   ┌─ 业务指标:CTR / CVR / 时长 / GMV(掉了说明模型或链路出问题)
   │
   │─ 模型指标:线上 pCTR 分布、校准比值(预估准不准)
   │
   │─ 特征监控:特征覆盖率、缺失率、分布漂移(feature drift)
   │
   │─ 链路指标:样本拼接率、回流延迟、训练成功率、serving RT
   │
   └─ 数据漂移:线上数据分布 vs 训练时分布(distribution shift)
```

### 11.2 效果回流:闭环合拢

线上产生的曝光和反馈,**回流**成为下一轮训练样本。这就是第 1 节说的「环」合拢的地方。

平台要保证:
- **回流不断流**:这是最容易「无声死亡」的环节。回流断了,模型还在用旧样本训,效果缓慢衰减,**没有报错、没有告警,只是慢慢变差**——除非你专门监控「样本新鲜度」。
- **回流不延迟**:延迟越大,模型越「过时」,详见第 14 节时延预算。
- **回流不偏置**:别只回流模型曝光过的,要保留随机/探索流量样本打破闭环偏置。

🎓 **老师视角**:我带新人时一定强调——**「会上线的工程师很多,会监控回流断流的工程师很少」**。模型衰减是温水煮青蛙:今天 AUC 掉 0.001,没人觉得是事;一个月后业务指标掉了几个点,回头查才发现回流三周前就断了。平台研发要做的是**把「样本新鲜度」「回流延迟」「模型最后更新时间」做成大盘上最显眼的指标**,断流立刻红灯。这种「主动暴露慢性病」的能力,是平台成熟度的标志。

### 11.3 交付物与卡点

| 项 | 内容 |
|---|---|
| **平台职责** | 全链路监控大盘、漂移检测、告警、回流管道、血缘追踪 |
| **输出交付物** | 健康度大盘 + 告警 + 回流数据(回到样本环节) |
| **常见卡点** | 回流断流无感知、漂移漏报、监控只看业务指标不看链路指标 |

## 12. 两条链路对比:离线批训练 vs 流式训练

整条链路有两种「跑法」,大厂往往**并存**。这是本篇最重要的架构对比。

### 12.1 离线批训练(batch training)

```
   T 日数据(全天攒齐)
        │
        ▼ 凌晨跑批
   ┌──────────────────────────────┐
   │ 全量训练(从头) 或 增量训练   │
   │ (在昨天 base 上继续训)        │
   └──────────────────────────────┘
        │
        ▼ T+1 早上产出
   新模型版本 ──▶ 评估 ──▶ 上线
```

- **时效**:T+1(次日),或日内增量(几小时一轮)。
- **特点**:数据攒齐再训,稳定、可复现、易调试、易回滚。
- **适用**:对时效不极致敏感的场景、需要全量重训纠偏、复杂大模型离线训练。

### 12.2 流式训练 / 在线学习(streaming / online learning)

```
   实时样本流(Kafka)
        │  (拼接也是流式的)
        ▼ 持续消费
   ┌──────────────────────────────┐
   │ 在线学习:样本来一批,更新一批 │
   │ 每个 pass(若干分钟)出一个    │
   │ delta 模型                     │
   └──────────────────────────────┘
        │
        ▼ 分钟级
   delta 模型 ──▶ (轻量校验) ──▶ 热更新到 serving
```

- **时效**:分钟级 / pass 级,模型几乎「实时」跟上线上变化。
- **特点**:快,但复杂、脆弱、难复现、出问题难回滚(参数一直在变)。
- **适用**:强时效场景(信息流、直播、大促、热点事件)、需要快速捕捉用户兴趣漂移。

### 12.3 全面对比表

| 维度 | 离线批训练 | 流式训练 / 在线学习 |
|---|---|---|
| **出模型频率** | 小时级~天级 | 分钟级~pass 级 |
| **数据新鲜度** | T+1 ~ 数小时 | 分钟级 |
| **架构复杂度** | 中(成熟、可控) | 高(流式拼接 + 在线更新 + 实时分发) |
| **可复现性** | 好(数据快照固定) | 差(参数连续演化) |
| **回滚难度** | 易(切版本号) | 难(状态连续,要保 base + delta 链) |
| **资源模式** | 周期性峰值(凌晨集中) | 持续常驻 |
| **调试难度** | 低 | 高(线上问题难离线复现) |
| **延迟反馈处理** | 简单(数据攒齐了) | 难(要假负样本 + 回补) |
| **典型适用** | 召回模型、复杂大模型、纠偏全量训 | 精排实时性强的场景、热点捕捉 |
| **对平台的要求** | 调度 + 资源管理 + 版本管理 | 流式 join + 在线 PS + 模型热更新 + 强监控 |

🎓 **老师视角**:别陷入「流式一定比离线高级」的迷思。流式训练的**复杂度和运维成本是指数级上升**的,而很多场景的边际收益并不值这个钱。成熟团队的做法常是 **「离线全量打底 + 流式增量提鲜」**:离线训练保证模型基础质量和纠偏(防止在线学习把模型带歪),流式只学增量、定期被离线版本「校正」回来。作为平台研发,你要能判断**「这个业务到底需不需要上流式」**,而不是技术上能做就做——这是替公司省钱的判断力。

## 13. 模型版本管理:base+delta、回滚、热更新

### 13.1 base + delta 模型

大模型几十 GB,频繁全量分发不现实。工业方案是 **base + delta**:

```
   base 模型(完整,几十 GB)         ← 离线定期全量产出(如每天 / 每周)
        │
        ├── delta_1(只含变化的参数,几百 MB)  ← 流式/增量产出
        ├── delta_2
        ├── delta_3
        │
   serving 加载 = base + 最新若干 delta 合并
```

- **base**:完整模型,体积大,产出慢,作为基准。
- **delta**:相对上一版的**增量**(主要是被更新过的 embedding 行),体积小,产出快,频繁分发。
- serving 端:加载 base,然后不断应用 delta 做热更新。

### 13.2 版本号与回滚

```
   model_v20260613_full        ← base
   model_v20260614_0900_delta  ← 时间戳 + 类型,全局唯一
   model_v20260614_0915_delta
        ...
   回滚 = 切回某个已知好的 (base + delta 序列) 组合
```

| 能力 | 平台要保证 |
|---|---|
| **唯一版本号** | 每个模型版本全局唯一、可追溯(谁训的、用了哪天数据、什么代码) |
| **版本血缘** | 模型 → 训练任务 → 样本版本 → 特征版本 → 代码 commit,全链路可查 |
| **秒级回滚** | 保留近 N 个版本,一键切回 |
| **灰度比例** | 新版本按流量比例逐步替换旧版本 |
| **base/delta 一致性** | serving 加载的 base 和 delta 必须配套,版本错配要拦截 |

🎓 **老师视角**:版本管理最被低估的价值是「**回滚演练**」。很多团队的回滚按钮**从来没在真实事故里按过**,等真出事才发现:旧版本文件被清理了 / 回滚后特征对不上 / 回滚本身有 bug。平台研发要把「回滚」当成一个**需要定期演练的能力**,而不是一个假设永远好用的按钮。这跟数据库备份一样——没恢复过的备份等于没有备份。

## 14. 端到端时延预算与「数据新鲜度→效果」

### 14.1 把链路当成一个「时延账本」

```
   事件发生(T0)
   │ ① 日志落地延迟         ~秒级 (流) / 攒到次日 (批)
   │ ② 拼接/打标延迟        ~分钟级 (流) / 小时级 (批)
   │   └─ 含归因窗口等待(CVR 可能等数小时~数天)
   │ ③ 训练消费延迟         ~分钟级 (流) / 数小时 (批)
   │ ④ 出模型 + 校验        ~分钟级 (流) / 数十分钟 (批)
   │ ⑤ 分发 + 热更新        ~秒级~分钟级
   ▼
   模型反映该事件(T0 + 端到端时延)
   端到端:流式 ~分钟级到小时级 | 批 ~T+1
```

### 14.2 数据新鲜度怎么换算成钱

- **核心规律**:新鲜度越高,模型越能捕捉当下兴趣/热点,效果越好——但**边际收益递减**,且各场景斜率差异极大。
- **强时效场景**(信息流、直播、电商大促、突发热点):新鲜度极其关键。从 T+1 提到小时级、再到分钟级,业务指标常有可观提升(经验值,具体场景需 AB 实测)。
- **弱时效场景**(用户长期兴趣、稳定品类):T+1 甚至 T+N 足够,上流式纯属浪费。

🎓 **老师视角**:平台研发要把「**端到端数据新鲜度**」做成一个**可度量、可分解、可优化**的一级指标——不是「我们有流式」就完事,而是能说出「从用户点击到模型反映,P50 是 X 分钟、P99 是 Y 分钟,瓶颈在归因窗口等待这一段」。能把时延账本拆到每一段,你才知道**钱该花在优化哪一环**(很多时候瓶颈不在训练,而在 join 或归因窗口)。这是从「会用」到「会优化」的分水岭。

### 14.3 新鲜度 vs 成本对照

| 新鲜度档位 | 端到端时延 | 架构 | 成本 | 适用 |
|---|---|---|---|---|
| T+1 | 约 1 天 | 离线批 | 低(峰值算力) | 大多数稳态场景 |
| 小时级 | 数小时 | 批增量 | 中 | 中等时效 |
| 分钟级 | 几分钟~几十分钟 | 流式 | 高(常驻 + 复杂) | 强时效、热点 |
| 准实时(在线学习) | 分钟内 | 在线学习 | 很高 | 极致时效、大促 |

## 15. 整条链路的可观测性:血缘、指标、断点排查

### 15.1 数据血缘(data lineage)

```
   线上指标异常
        ↓ 顺着血缘往上查
   模型版本 v20260614 ──▶ 训练任务 #8821 ──▶ 样本版本 sample_20260613
        ↓                                          ↓
   代码 commit abc123                       特征版本 feat_20260613
        ↓                                          ↓
   配置快照                                  上游日志分区 / 埋点 schema
```

血缘的价值:**任何一个环节出问题,能在几分钟内定位「波及范围」和「根因方向」**,而不是从头到尾人肉排查。

### 15.2 各环节核心监控指标速查

| 环节 | 必盯指标 | 异常含义 |
|---|---|---|
| 特征快照 | 快照量、采样率、写入延迟 | 量骤降 = 打分异常或快照写入挂了 |
| 日志 | 各类事件量、去重率、丢失率 | 量异常 = 埋点/客户端问题 |
| 拼接 | 拼接成功率(>99%)、丢失率 | 成功率掉 = join key 或一方缺失 |
| 样本 | 样本量、正负比、新鲜度 | 正负比突变 = 标签或采样出问题 |
| 训练 | 成功率、单 step 耗时、GPU 利用率、loss 曲线 | loss 不收敛/突变 = 数据或代码问题 |
| 评估 | AUC/GAUC、与基线 diff | 突高 = 可能穿越;突低 = 数据问题 |
| 上线 | 加载成功率、版本一致性、RT | RT 飙升 = 加载抖动 |
| 线上 | CTR/CVR、pCTR 校准、特征漂移 | 校准跑偏 = 采样/分布问题 |
| 回流 | 回流延迟、样本新鲜度、最后更新时间 | 新鲜度老化 = 回流断流 |

### 15.3 断点排查的标准动作

🎓 **老师视角**:教你一个排查口诀——**「从结果倒推,沿血缘上溯,在接口验数」**。线上效果掉了(结果),先看是 serving 问题还是模型问题(看 pCTR 分布有没有变);若是模型,沿血缘找到训练任务和样本版本;在每个**环节接口处验数据**(快照量对不对、拼接率掉没掉、正负比变没变),哪一环的「输入正常但输出异常」,根因就在那一环。**永远在接口处对数,不要凭感觉猜。** 一条链路有了清晰的环节接口和每个接口的监控,排查时间能从「几天」压到「几十分钟」。

## 16. 一个脱敏的端到端排查案例(把前面串起来)

理论讲完,来一个**贴近真实**的脱敏案例,把第 1~15 节的知识串成一次实战。

### 16.1 现场:某信息流场景,CTR 缓慢下滑

```
   时间线(脱敏)
   D 日:某团队上线了新精排模型 v_new,AB 显著正向(+1.x%),全量。
   D+3:大盘 CTR 开始缓慢下滑,每天掉一点点,没人警觉。
   D+10:CTR 累计掉了约 2-3 个点,业务方报警,平台介入排查。
```

第一反应很容易跑偏:「是不是新模型其实不行?回滚到老模型!」——但回滚后**CTR 没恢复**。这说明根因**不在模型本身**,而在链路。这正是「从结果倒推、沿血缘上溯、在接口验数」的用武之地。

### 16.2 沿血缘上溯的排查过程

```
   现象:CTR 下滑(结果)
     ↓ 看 serving 还是模型?→ 线上 pCTR 分布整体偏低且越来越偏
     ↓ 看模型新鲜度 → 模型最后更新时间停在 D+2!(本该分钟级更新)
     ↓ 看回流/样本环节 → 样本新鲜度老化,流式样本流在 D+2 凌晨断了
     ↓ 看拼接接口 → 拼接成功率从 99.x% 掉到 70%(输入正常,输出异常)
     ↓ 看上游 → 某次埋点发布把 item_id 字段类型从 string 改成了 long
     ↓ 根因:join key 类型不匹配 → 大量样本拼接失败 → 样本量锐减
              → 流式训练「饿肚子」→ 模型停更 → 越来越过时 → CTR 下滑
```

### 16.3 复盘:每个环节本该拦住它

| 本该拦住的环节 | 为什么没拦住 | 应补的能力 |
|---|---|---|
| 埋点变更 | schema 改动没走审批 + 影响面分析 | 强制 schema 变更审批 |
| 拼接监控 | 拼接率掉了但告警阈值太松 | 拼接率掉破 95% 立刻告警 |
| 样本新鲜度 | 没把样本新鲜度做成显眼指标 | 新鲜度老化红灯上大盘 |
| 模型更新时间 | 没监控「模型最后更新时间」 | 停更超 N 分钟告警 |

🎓 **老师视角**:这个案例的精髓是——**「回滚模型救不了链路病」**。新人遇到效果掉第一反应是动模型,但**链路断了,换哪个模型都白搭**。它也说明:一个**字段类型的小改动**,能通过「拼接失败 → 样本饿死 → 模型停更」这条隐蔽链路,在十天后变成一次业务事故。平台研发的价值,就是在每个接口处埋下「输入正常但输出异常」的探针,让这种慢性病在第一天就亮红灯,而不是第十天靠人肉考古。这一个案例,胜过背十遍监控清单。

## 17. 平台研发的「全环节交付物」总清单

把第 3~11 节散落的交付物收口成一张「我到底要产出什么」的总表,这是你向上汇报和自查时的底稿。

| 环节 | 平台研发的硬交付物(代码/系统/契约) |
|---|---|
| 在线打分 | 快照 SDK/切面、request_id 透传机制、采样配置中心 |
| 日志 | 埋点 schema 注册中心、去重/乱序处理管道、日志质量大盘 |
| 样本 | join 引擎(批+流)、归因窗口配置、采样+校准框架、回补机制、样本质量校验 |
| 特征供给 | 特征读取层、train-serving 一致性校验工具、slot/ID 化管理 |
| 训练 | 调度系统、PS/资源管理、checkpoint+failover、训练监控、版本产出 |
| 评估 | 评估流程、指标库、时间隔离校验、卡口阈值系统 |
| 灰度/AB | 流量切分框架、分桶管理、显著性检验、最小观测期卡口 |
| 上线 | 模型版本仓、base/delta 分发、双 buffer 热更新、回滚系统 |
| 监控/回流 | 全链路血缘系统、新鲜度/回流大盘、漂移检测、告警 |

🎓 **老师视角**:注意这张表里**没有一项是「调模型参数」**。这就是平台研发和算法工程师的根本分工——你交付的是**「让模型能被安全、高效、可追溯地生产和上线的基础设施」**。当老板问「这季度你做了啥」,你的答案应该落在这张表的某几格里(比如「把拼接引擎从批改造成流批一体,端到端新鲜度从 T+1 提到分钟级」),而不是「我帮算法调了个模型」。把交付物说清楚,你的价值才被看见。

## 业界怎么做

| 体系 | 代表性做法 | 给平台研发的启示 |
|---|---|---|
| **特征/训练一致性** | 用特征快照 + Feature Store 复用特征定义,消除 train-serving skew | 一致性靠架构保证,不靠人盯 |
| **大规模稀疏训练** | PS 架构存百亿~万亿 embedding,worker 算梯度;dense+sparse 混合并行 | 推荐训练不是「堆 GPU」,是稀疏参数工程 |
| **流批一体** | 离线全量打底 + 流式增量提鲜,流式被离线定期校正 | 别单押流式,组合拳更稳 |
| **实验平台** | 分层正交实验框架(overlapping experiments)、最小观测期硬卡口 | 实验科学性要平台兜底 |
| **模型分发** | base+delta 增量分发、双 buffer 热更新、秒级回滚 | 上线能力 = 分发 + 热更 + 回滚三件套 |
| **可观测性** | 全链路血缘 + 样本新鲜度大盘 + 漂移检测 | 把慢性病做成显眼的红灯 |

(以上为业界通行模式的归纳,具体实现各家有差异。)

## 权衡取舍

| 取舍点 | A 选项 | B 选项 | 怎么判断 |
|---|---|---|---|
| 新鲜度 vs 成本 | 流式分钟级 | 离线 T+1 | 时效敏感且 AB 验证有收益才上流式 |
| 快照采样率 | 全量(贵但全) | 采样(省但缺长尾) | 按曝光/热度分层采,平衡成本与覆盖 |
| 归因窗口 | 长窗口(准但慢) | 短窗口(快但漏) | 看转化延迟分布 + 时效要求 |
| 在线学习 vs 离线纠偏 | 纯在线(快但易跑偏) | 离线打底 | 用离线定期校正在线,组合最稳 |
| 全量 vs 增量训练 | 全量(纠偏彻底) | 增量(快省) | 增量提鲜 + 周期性全量纠偏 |
| 监控覆盖 | 全链路重监控(贵) | 只盯业务指标(漏) | 关键接口必盯,新鲜度/回流必盯 |

## 常见误区与反直觉

1. **「离线 AUC 高就能上线」** —— 错。离线在线一致性是独立的大坑,AUC 高可能是穿越或评估集泄露。必须 AB 验证。
2. **「流式训练比离线高级,应该全上」** —— 错。流式复杂度和成本指数上升,很多场景不值。组合拳才是王道。
3. **「训练慢就加 GPU」** —— 常错。推荐训练瓶颈经常在数据 IO、PS 倾斜、归因等待,不在算力。先做时延账本再花钱。
4. **「模型上线就完事了」** —— 大错。上线是新一轮起点,回流断流、模型衰减是温水煮青蛙式的慢性死亡。
5. **「特征训练时现查就行」** —— 致命错。这就是特征穿越,必须用在线快照。
6. **「采样后预估值能直接用」** —— 错。负采样改变了概率分布,必须校准,否则计费/ROI 全错。
7. **「回滚按钮肯定能用」** —— 危险假设。没演练过的回滚等于没有回滚。
8. **「数据照常产出就说明链路健康」** —— 错。口径漂移、含义变化时数据照样产出,只是全错了,极其隐蔽。

## 平台侧落地 Checklist

**特征快照环节**
- [ ] 快照 SDK 无侵入接入,不拖慢打分 RT(异步 + 降级)
- [ ] request_id 全链路透传,快照带模型版本号
- [ ] 采样率可配置、可分层(曝光全采、未曝光分层采)
- [ ] 快照量监控,骤降告警

**日志环节**
- [ ] 埋点 schema 统一管理,变更走审批 + 影响面分析
- [ ] 去重(幂等 key)、乱序(event time)、丢失率监控
- [ ] 各类事件量大盘

**样本环节**
- [ ] join 引擎支持批 + 流,拼接率 >99% 且监控
- [ ] 归因窗口可配置,支持延迟反馈的假负 + 回补
- [ ] 采样框架 + 校准逻辑统一约定(明确在训练侧还是 serving 侧)
- [ ] 正负比、样本新鲜度监控

**训练环节**
- [ ] 资源调度 + 容错 + 慢节点检测
- [ ] checkpoint + failover 秒级恢复(演练过)
- [ ] GPU 利用率、单 step 耗时、PS 倾斜监控
- [ ] 产出模型带完整血缘(数据/代码/配置版本)

**评估环节**
- [ ] 评估集与训练集强制时间隔离(防穿越)
- [ ] 指标库统一(AUC/GAUC/校准口径固定)
- [ ] 基线对比 + 卡口阈值(不达标不进灰度)

**上线环节**
- [ ] 模型版本仓 + base/delta 配套校验
- [ ] 双 buffer 热更新,原子切换,资源隔离
- [ ] 灰度加载 + AB 框架(分桶正交、最小观测期硬卡口)
- [ ] 秒级回滚 + 定期回滚演练

**监控/回流环节**
- [ ] 全链路血缘可查(模型→任务→样本→特征→代码)
- [ ] 样本新鲜度 / 回流延迟 / 模型最后更新时间放大盘最显眼处
- [ ] 特征漂移、pCTR 校准、分布 shift 检测
- [ ] 断流、漂移自动告警

## 对接话术

**对算法/训练同学:**
- 「你这个特征是在线能实时拿到的吗?如果不能,训练就只能用快照里的历史值,别指望我离线给你现算未来的值。」
- 「你做了负采样,校准放在哪做?训练侧还是 serving 侧?咱俩得对齐,不然计费会出事。」
- 「归因窗口你定多长?窗口长了出模型就慢,你得在时效和正样本召回之间取舍,定了我来落配置。」
- 「这个模型离线 AUC 涨了不代表能全量,先走灰度 + AB,观测期我建议至少一周。」

**对数仓/数据同学:**
- 「埋点 schema 要改的话先报备,我评估下游样本和指标的影响面,不能直接上。」
- 「曝光日志 T+1 分区几点能齐?我下游拼接的 SLA 卡在这。」

**对 SRE/运维:**
- 「模型热更新走双 buffer,切换是原子的,但加载会占内存,资源要留余量。」
- 「回滚要能秒级切版本,我们得定期演练,不能等出事才发现回滚也坏了。」

## 向上沟通

**给老板讲「机器学习平台是干嘛的」(一句话版):**
> 「我们保障从『用户看到一次推荐』到『模型被更新上线』这条流水线上,数据是对的、齐的、准时的、可追溯的、可回滚的。算法同学负责模型聪不聪明,我们负责这条流水线不出事、出事能秒回滚、还要越来越省钱。」

**讲价值(三个可量化抓手):**
- **稳**:链路 SLA 达成率、事故 MTTR(平均恢复时间)、回滚成功率。
- **快**:端到端数据新鲜度(从行为到模型反映的 P50/P99 时延),直接影响时效场景业务指标。
- **省**:训练成本 / GPU 利用率、存储成本,平台优化能直接省真金白银。

**讲风险(让老板有数):**
- 「目前回流监控做到位了,但回滚演练还没纳入常态,建议排期——这是出大事故时的最后一道保险。」
- 「流式链路复杂度高,人力投入和收益要按场景算账,不是所有业务都该上。」

```
   ┌──────────────────────────────────────────────────────────────────┐
   │   一条样本到一个上线模型 = 一个自反馈的环                          │
   │                                                                    │
   │   打分(拍快照) → 日志 → 拼接打标采样 → 特征供给 → 训练           │
   │        ▲                                              │             │
   │        │                                              ▼             │
   │   监控回流 ◀── 上线热更新 ◀── 灰度/AB ◀── 评估(GAUC)              │
   │                                                                    │
   │   平台职责贯穿全程:对的·齐的·准时的·可追溯·可回滚                 │
   │   两条跑法:离线批(稳)  +  流式(鲜),组合拳最优                  │
   │   三大雷:特征穿越 · train-serving skew · 回流断流无感             │
   └──────────────────────────────────────────────────────────────────┘
```

**一句话速记**:**「训练所见即线上所见,模型上线即新轮开始;平台不调模型,平台守住数据的『对齐、准时、可回滚』。」**

## 自测三问

**Q1:为什么训练特征必须用「在线拍的快照」,而不能训练时去特征仓现查?**

参考答案:因为特征值随时间变化(如用户 7 天点击率)。训练时现查查到的是**未来时刻**的值,而线上打分时只能拿到**当时**的值,二者不一致就是**特征穿越**——离线 AUC 虚高、上线垮掉。在线拍快照保证「训练所见即线上所见」(train as you serve),这是消除穿越最彻底的工程手段。很多实时特征事后根本无法精确回溯,也只能靠当场快照。

**Q2:离线批训练和流式训练各自的核心权衡是什么?成熟团队通常怎么组合?**

参考答案:离线批——稳定、可复现、易回滚,但时效差(T+1);流式——分钟级新鲜度,但架构复杂、难复现、难回滚、要处理延迟反馈。成熟团队常用「**离线全量打底 + 流式增量提鲜**」:离线训练保证基础质量并定期纠偏(防止在线学习把模型带歪),流式只学增量并被离线版本周期性校正。是否上流式要按场景算账(时效敏感且 AB 有收益才值),不是技术能做就做。

**Q3:模型上线后,哪个环节最容易「无声死亡」?平台研发该怎么防?**

参考答案:**效果回流**最容易无声断流。回流断了,模型继续用旧样本训,效果**缓慢衰减但不报错**(温水煮青蛙)。防法:把「**样本新鲜度 / 回流延迟 / 模型最后更新时间**」做成监控大盘上最显眼的一级指标,断流/老化立刻红灯告警;而不是只盯业务指标——等业务指标掉了再回头查,往往已经晚了几周。
