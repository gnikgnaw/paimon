# Apache Paimon 查询优化与多维排序策略源码深度分析

> **版本**：1.5-SNAPSHOT　**源码模块**：`paimon-core`（Scan/Split/Lookup/Clustering）、`paimon-common`（文件索引、Z-Order/Hilbert 算法）、`paimon-spark`/`paimon-flink`（多维排序作业）　**核对日期**：2026-06

**一句话定位**：查询优化做的是同一件事——**用尽量便宜的元数据判断，提前把"肯定不含命中数据"的文件/行剔掉**，把 PB 级表的扫描收敛到几个文件甚至几行；多维排序则是它的"前置施肥"——通过 Z-Order/Hilbert 空间填充曲线重排数据物理布局，让多列范围查询的 min/max 同时变窄，使后续每层裁剪都更有效。

读完本文你应能回答：① 一条带多列谓词的查询，从 partition 裁剪到最终读行，要穿过哪几层漏斗、每层的成本/精度/源码入口分别是什么；② 为什么单列 ORDER 排序只对头部列友好，而 Z-Order"字节交错"能让参与的每一列都受益——它在数学上保持了什么；③ ORDER / ZORDER / HILBERT 三种排序何时各自更优，Hilbert 凭什么比 Z-Order 局部性更好、代价在哪；④ 主键表与追加表的谓词下推为什么走两套过滤逻辑，主键表的 value 过滤为什么必须是"条件性"的（否则会漏数据）；⑤ `rawConvertible` 跳过 merge 需要同时满足哪几个条件，DV 启用为什么让它恒成立；⑥ Lookup 点查从上层往下查、Level 0 遍历 / Level 1+ 二分的依据是什么；⑦ 哪些配置/数据形态会让裁剪退化（Level 0 堆积、高基数列建 Bitmap、Limit 下推被悄悄禁用）。

> 阅读约定：本文每个机制按"① 要解决什么问题 → ② 设计原理与取舍 → ③ 关键源码（精选片段 + `路径:行号`）→ ④ 风险/陷阱/边界 → ⑤ 收益与代价"组织。源码行号以本次核对为准（master，commit `55f4fd175` 附近）；与旧稿不符处用"（已修正）"标注。文件索引的内部结构（Bloom/Bitmap/BSI/RangeBitmap 怎么编码）由 **13 号文档**主讲，本篇只讲"查询如何命中索引"；Lookup 点查的 LSM 底层、DeletionVector 的产生分别由 **01 §6** 与 **04** 主讲，本篇点到即引。

## 1. 快速理解（核心问题 / 概念速查 / 高频陷阱）

### 1.1 核心问题：裁剪漏斗 + 多维排序，本质都在缩小扫描面

**① 要解决什么问题**

一张生产表轻松到几十万个数据文件。一次 `SELECT ... WHERE` 如果老老实实读所有 Manifest、反序列化每条 `ManifestEntry`、打开每个数据文件，延迟从秒级跌到分钟级，交互式查询直接没法用。查询优化要回答的就是一句话：**怎样用最便宜的判断，最早地剔掉"肯定不命中"的文件和行。**

**② 设计原理与取舍——漏斗的成本/精度梯度**

Paimon 的答案是一条"成本递增、数据量递减"的漏斗：越靠前的过滤越便宜、越粗，越靠后的越贵、越精。关键在于**让每一层都尽量便宜地砍掉一大批数据，把昂贵的判断留给少数幸存者**。

| 漏斗层 | 过滤依据 | I/O / 计算成本 | 精度 | 源码入口 |
|--------|---------|---------------|------|---------|
| ① 分区裁剪 | partition 谓词 vs `partitionStats` | 最低（只读 ManifestList 聚合） | 分区级 | `ManifestsReader.filterManifestFileMeta:166` |
| ② bucket/level 裁剪 | `minBucket/maxBucket`、`minLevel/maxLevel` | 最低（同上，纯整数比较） | Manifest 级 | 同上 `:167-189` |
| ③ Entry 行过滤 | partition/bucket/level（反序列化前） | 低（操作二进制 InternalRow） | 单文件元数据 | `AbstractFileStoreScan.createEntryRowFilter:526` |
| ④ 文件 min/max 跳过 | `keyStats`/`valueStats` | 中（完整反序列化 DataFileMeta） | 文件级范围 | `KeyValueFileStoreScan.filterByStats:141`、`AppendOnly…:125` |
| ⑤ 文件索引命中 | Bloom/Bitmap/BSI（嵌入或独立） | 中-高（位图运算 / 额外 I/O） | 行级（部分） | `AppendOnlyFileStoreScan:161`、`FileIndexEvaluator.evaluate:45` |
| ⑥ Split 合并 + 点查 | `rawConvertible`、LookupLevels | 决定读期是否 merge | 读取方式 | `MergeTreeSplitGenerator:69`、`LookupUtils.lookup:35` |

**多维排序是这条漏斗的"前置施肥"**：第④层 min/max 跳过的有效性，完全取决于"同一文件内的数据在各列上是否聚拢"。按写入顺序落盘时，每个文件的各列 min/max 都很宽，第④层几乎裁不掉东西；用 Z-Order/Hilbert 重排后，每个文件在多个列上的 min/max 都收窄，第④层才真正发挥作用。所以本篇把"裁剪链路"和"多维排序"放在一起讲——**它们是同一目标的两端：排序决定数据布局有多可裁，漏斗决定能裁掉多少。**

一句话设计哲学：**便宜的判断尽量早做、尽量多砍；昂贵的判断只对幸存者做；而排序决定了"幸存者"能有多少。**

### 1.3 完整查询加速链路（六级漏斗一张图）

以一条 `SELECT * FROM t WHERE dt='2024-01-01' AND city='BJ' AND amount>1000` 为例（t 主键含 dt，bucket=hash），数据从"全表数十万文件"收敛到"几个 Split、若干行"的全过程：

```
0. 规划入口   FileStoreScan.plan()                                    AbstractFileStoreScan.java:297
─────────────────────────────────────────────────────────────────────
① partition  ManifestsReader 用 partitionStats 比对 dt='2024-01-01'   ManifestsReader.java:166,191
   裁剪       → 跳过整个 Manifest（连内容都不读）                       —— 成本最低、收益最大的一刀
② bucket/    同一步比对 minBucket/maxBucket、minLevel/maxLevel         ManifestsReader.java:167-189
   level 裁剪 → compaction 选层 / 指定 bucket 时再砍一批 Manifest
③ Entry 行   读入幸存 Manifest，反序列化前用 InternalRow getter        AbstractFileStoreScan.java:526
   过滤       逐条比 partition/bucket/level → 避免无谓的完整反序列化
④ 文件 min/  完整反序列化后，主键表用 keyStats、追加表用 valueStats     KeyValueFileStoreScan.java:141
   max 跳过   比对 dt/city/amount 范围 → 文件级跳过（已修正行号）        AppendOnlyFileStoreScan.java:125
⑤ 文件索引   amount>1000 命中 BSI、city='BJ' 命中 Bloom/Bitmap         AppendOnlyFileStoreScan.java:161
```

每往后一级，处理的数据量都数量级地下降；而**多维排序（§4）通过把数据在 city/amount 上聚拢，让第④⑤级能砍掉的文件成倍增加**——没有好的物理布局，再多层漏斗也只是空转。

### 1.4 高频生产陷阱

**陷阱 1：Level 0 文件堆积让第④层和 Split 优化双双退化。** Level 0 文件之间 key 范围 overlap，既不能逐文件做 `wholeBucketFilter`（只能整桶判断，见 §2.6），也无法 `rawConvertible`（§6.1 要求 `level != 0`）。后果：本该被 min/max 砍掉的文件全部幸存、本该直读的 Split 全部退化成 merge。规避：及时触发 compaction；Lookup 场景配 `lookup` changelog-producer 主动清 L0。

**陷阱 2：高基数列误建 Bitmap 索引导致索引爆炸。** Bitmap 为每个不同值维护一张位图，`user_id` 这类百万基数列会生成百万张位图，存储和构建都爆炸。规避：高基数等值用 Bloom Filter，低基数（status/type）用 Bitmap，数值范围用 BSI（空间固定 O(bits×n)，不随基数增长）。索引选型细节见 **13 号文档**。

**陷阱 3：主键表强开 value 过滤会漏数据。** 主键表同一 key 可能散在多个文件，`ScanMode.ALL` 下若按单文件 valueStats 直接跳文件，可能漏掉参与 merge 的旧版本。所以 value 过滤是**条件性**的：只有 `valueFilterForceEnabled`（上层显式开）或 CHANGELOG 模式下 LOOKUP/FULL_COMPACTION producer 才启用（`KeyValueFileStoreScan.java:195-211`）。不要盲目打开。

**陷阱 4：Limit 下推被悄悄禁用却以为生效。** `limitPushdownEnabled()`（`KeyValueFileStoreScan.java:223`）的禁用条件比想象中多：PARTIAL_UPDATE / AGGREGATE 引擎（rowCount≠最终行数）、DV 启用（删除行不计有效行）、**以及存在 valueFilter 或 keyFilter 时**（已修正：旧稿遗漏了后两个条件）。即只要带了过滤谓词，Limit 下推就不生效——这是正确性优先的设计，不是 bug。

**陷阱 5：在低基数列上做 Z-Order/Hilbert 等于白做。** 空间填充曲线靠"各列都有足够多的取值去交错"才能拉开文件间区分度。`gender`、`status` 这种 2-3 个值的列，排序后文件间仍大量 overlap。规避：选中高基数、且查询频率相当的 2-4 列；排序列超过 4 列后每列分到的 bit 太少、区分度都不够。

**陷阱 6：全量 Clustering 当增量用，写放大爆炸。** `compaction.clustering.full-compaction=true` 每次读写整张表（写放大 2x）。增量模式只排 Level 0、只合并 overlap 的 Level 1（§5.1），写放大可压到 0.4x 量级。流式写入务必用增量，全量只作为偶发兜底。

**陷阱 7：把 Lookup 当批扫描的银弹。** LookupLevels 只加速"按主键点查单行"，对范围扫描无益；且它要在本地磁盘建 SST 缓存，Flink failover 后逐文件重建可能数分钟。规避：大文件场景开远程 Lookup 文件预构建上传（§7.1 / 01 §6 陷阱）；点查机制底层详见 **01 §6**。

## 2. 多层文件过滤：Scan 漏斗

### 2.1 FileStoreScan 继承体系与 plan() 三段流程

**① 要解决什么问题**

`FileStoreScan.plan()` 要在面对数十万文件时秒级产出 `Split[]`。如果它先把所有 Manifest 反序列化成对象再过滤，单这一步就能拖垮查询。所以 plan() 本身被切成三段，越早的过滤越粗越便宜。

**② 设计原理与取舍**

继承体系把"通用过滤逻辑"和"表类型差异"分离：

```
FileStoreScan (接口)
  └─ AbstractFileStoreScan      通用三段流程 + Entry 行过滤
       ├─ KeyValueFileStoreScan  主键表：keyStats 主过滤、条件性 value 过滤、wholeBucketFilter
       └─ AppendOnlyFileStoreScan 追加表：valueStats 全列过滤、直接用文件索引
```

`plan()` 三段流程（`AbstractFileStoreScan.java:297`）：

```
① readManifests()            ManifestsReader 用聚合元数据做 Manifest 级粗过滤（连内容都不读）   :299
② readManifestEntries()      读入幸存 Manifest，逐条 Entry 做行过滤 → 统计过滤              :303
③ postFilterManifestEntries() 主键表专属后过滤：整桶过滤 / Limit 下推（可选）                 :306
```

第②步内部又分"反序列化前的行过滤"（§2.3）和"反序列化后的 `filterByStats`"（§2.4），分别对应漏斗的第③④层。

**③ 关键源码**：`AbstractFileStoreScan.java:466` 把 `filterByStats(ManifestEntry)` 声明为抽象方法，由两个子类各自实现——这是主键表/追加表过滤策略分叉的核心钩子。

### 2.2 第一层：Manifest 级粗过滤（ManifestsReader）

**① 要解决什么问题**

一个 Manifest 文件可能聚合了几千条 ManifestEntry。如果它的分区范围、bucket 范围跟查询完全不沾边，应当**连读都不读**，直接整文件跳过。

**② 设计原理与取舍**

`ManifestFileMeta` 在每个 Manifest 上预存了一份"对内部全部条目的聚合统计"（`ManifestFileMeta.java:64-99`，已修正：旧稿标 `:44-61`）：

| 字段 | 过滤用途 | 谁来用 |
|------|---------|--------|
| `partitionStats: SimpleStats` | 分区 min/max/nullCounts → 分区谓词裁剪 | 所有查询 |
| `minBucket/maxBucket` | 指定 bucket 不在范围则跳过；`onlyReadRealBuckets` 时 `maxBucket<0` 也跳 | bucket 定向读 |
| `minLevel/maxLevel` | compaction 按层选文件、`levelFilter` 时裁剪 | compaction / 流读 |
| `minRowId/maxRowId` | 按行号范围裁剪（增量/分裂读） | row-range 读 |
| `numAddedFiles/numDeletedFiles` | 辅助：纯删除 Manifest 在 ALL 模式可直接过滤 | 内部 |

**③ 关键源码**（`ManifestsReader.filterManifestFileMeta:166`）——先做最便宜的整数比较（bucket/level），再做 partition 谓词：

```java
private boolean filterManifestFileMeta(ManifestFileMeta manifest) {
    Integer minBucket = manifest.minBucket();   // :167
    Integer maxBucket = manifest.maxBucket();
    if (minBucket != null && maxBucket != null) {
        if (onlyReadRealBuckets && maxBucket < 0) return false;
        if (specifiedBucket != null && (specifiedBucket < minBucket || specifiedBucket > maxBucket))
            return false;
    }
    // ... minLevel/maxLevel、levelFilter 同理
    if (partitionFilter != null) {            // :191 —— 最关键的一刀
        SimpleStats stats = manifest.partitionStats();
        if (!partitionFilter.test(numAddedFiles + numDeletedFiles,
                stats.minValues(), stats.maxValues(), stats.nullCounts()))
            return false;
    }
    return true;
}
```

**④ 风险/陷阱**：这一层是"成本最低、收益最大"的一刀，但它依赖 Manifest 数量受控。Manifest 文件过多时，光遍历 `ManifestFileMeta` 也会变慢——需要定期 Manifest compaction（小文件治理见 **11 号文档**）。

**⑤ 收益**：分区设计得当时，这一层就能把数十万文件砍到几百个，且**不读取任何 Manifest 内容**。这是 Paimon 相对 Iceberg 的一个差异点：Iceberg 的 manifest list 也有 partition summary，但 Paimon 还额外带了 bucket/level/rowId 范围，能利用 LSM 层级结构多砍一刀。

### 2.3 第二层：ManifestEntry 行过滤（反序列化前）

**① 要解决什么问题**

`ManifestEntry` 完整反序列化要解开 `DataFileMeta` 里的 keyStats、valueStats（每个都是 BinaryRow + BinaryArray）。对一个有几千条 Entry 的 Manifest，逐条完整反序列化代价不小。能不能在"还是二进制 InternalRow"时就把明显不匹配的条目甩掉？

**② 设计原理与取舍**：`createEntryRowFilter()`（`AbstractFileStoreScan.java:526`，已修正：旧稿标 `:526-567` 区间）直接在 `InternalRow` 二进制层面工作，用 `ManifestEntrySerializer.partitionGetter()` 等静态 getter 零拷贝抽取 partition / bucket / level / 文件名，逐条判断：

1. `partitionFilter.test(partition)`
2. `bucketFilter.test(partition, bucket, totalBucket)`（partition-aware bucket 过滤）
3. level 精确/范围匹配
4. 文件名过滤（增量读取场景）

**⑤ 收益与代价**：避免了对不需要的条目做完整对象图反序列化，省的是大量短命对象的分配与 GC 压力。代价是这层只能用"简单标量字段"，碰不到 keyStats/valueStats——精细的 min/max 判断留给下一层。

### 2.4 第三层：文件级统计过滤 + Schema Evolution

**① 要解决什么问题**

行过滤后幸存的条目被完整反序列化，现在能访问 `DataFileMeta` 的列级统计了。这是真正用 min/max 把"分区命中但数据范围不命中"的文件砍掉的一层——也是多维排序的收益兑现处。

**② 设计原理与取舍——主键表/追加表分叉**

| | 主键表 `KeyValueFileStoreScan` | 追加表 `AppendOnlyFileStoreScan` |
|---|---|---|
| 主过滤字段 | `keyStats`（主键按序存储，范围精确） | `valueStats`（无主键，全列统计） |
| value 过滤 | **条件性**（§2.5，merge 语义所限） | 始终用 `inputFilter` |
| 文件索引 | 仅在 value 过滤分支里用 embeddedIndex | `filterByStats` 末尾直接用 |

主键表 `filterByStats`（`KeyValueFileStoreScan.java:141`，已修正行号）——核心是用 `keyStats` 做 min/max 判断，并对每个 schemaId 缓存逆映射后的过滤器：

```java
protected boolean filterByStats(ManifestEntry entry) {
    if (isValueFilterEnabled() && !filterByValueFilter(entry)) return false;  // 条件性 value 过滤
    Predicate notEvolvedFilter = notEvolvedKeyFilterMapping.computeIfAbsent(
            entry.file().schemaId(),
            id -> fieldKeyStatsConverters.filterUnsafeFilter(id, keyFilter, /*keepNewFieldFilter*/ true));
    if (notEvolvedFilter == null) return true;
    DataFileMeta file = entry.file();
    SimpleStatsEvolution.Result stats = fieldKeyStatsConverters.getOrCreate(file.schemaId())
            .evolution(file.keyStats(), file.rowCount(), null);

```

追加表 `filterByStats`（`AppendOnlyFileStoreScan.java:125`，已修正：旧稿标 `:121-158`）多一步：valueStats 过滤通过后，若 `fileIndexReadEnabled` 再走 `testFileIndex(embeddedIndex)`（`:161`，文件索引命中见 §3）。

**SimpleStats 为何用 BinaryRow**：所有列的 min/max 紧凑塞进一个连续 `byte[]`，相比 `Object[]` 省掉每个值的对象头与引用——数万文件下内存差距显著；且谓词比较直接在二进制上做，零反序列化。（结构见 `stats/SimpleStats.java`。）

**Schema Evolution——为什么不能直接拿当前谓词比对旧文件**：旧文件是在旧 schema 下写的，列序、类型可能都变了。`SimpleStatsEvolution` 把"当前 schema 的谓词"逆映射（devolve）到旧文件 schema：
- `indexMapping`：当前字段 → 旧字段索引；当前新增字段映射为 -1，其 nullCount 视为 rowCount（旧数据该列全 null）。
- `castFieldGetters`：类型变更时转换（如 INT→BIGINT）。
- `keepNewFieldFilter=true`：保留新增字段的谓词——例如新增列 `c` 上的 `c>3`，旧文件该列全 null，可被安全过滤掉。

`DataFileMeta`（现为接口，`io/DataFileMeta.java`）中与查询相关的字段：`minKey/maxKey`（IntervalPartition 与二分用）、`keyStats`/`valueStats`、`valueStatsCols`（稠密统计列标识）、`embeddedIndex`、`rowCount`、`deleteRowCount`（判 rawConvertible 用，§6.1）。

### 2.5 主键表的 value 过滤为什么是条件性的

**① 要解决什么问题 / ④ 风险**：主键表同一个 key 可能散落在多个文件（不同版本待 merge）。如果在 `ScanMode.ALL`（批读全量快照）下，凭某个文件的 valueStats 不匹配 `amount>1000` 就把它跳掉，可能漏掉这个文件里某条"旧版本不满足、但 merge 后满足"的记录——**直接导致结果错误**。这就是为什么 value 过滤不能无条件开。

**② 设计原理**（`KeyValueFileStoreScan.isValueFilterEnabled:195`）：

```java
private boolean isValueFilterEnabled() {
    if (valueFilter == null) return false;
    switch (scanMode) {
        case ALL:       return valueFilterForceEnabled;   // 须上层显式开（确认无漏数据风险）
        case DELTA:     return false;
        case CHANGELOG: return changelogProducer == LOOKUP
                            || changelogProducer == FULL_COMPACTION;  // 这两种 producer 已 full-merge
        default: throw new UnsupportedOperationException();
    }
}
```

LOOKUP / FULL_COMPACTION producer 下，读到的文件已是全量合并产物，单文件 valueStats 能代表最终值，故可安全过滤。这是"正确性优先于性能"的典型取舍。

### 2.6 后过滤：wholeBucketFilter 与 Limit 下推

**① 要解决什么问题**：value 过滤之外，主键表还有两个只能"等所有条目收齐后"才能做的判断：① 对一个 bucket 整体能否跳过（解决 overlap 下的过滤难题）；② Limit 下推。`postFilterManifestEntries`（`KeyValueFileStoreScan.java:236`）按 (partition, bucket) 分组后逐桶处理。

**② 设计原理**：

`noOverlapping(entries)`（`:336`）判断同桶文件是否全在同一非零 level（即 key 范围互不重叠）：
- **无 overlap** → `filterWholeBucketPerFile`（`:279`）逐文件过滤，可精确砍。
- **有 overlap**（含 Level 0）→ `filterWholeBucketAllFiles`（`:289`）**整桶判断**：只有桶内所有文件都不匹配才能跳整桶。原因同 §2.5——overlap 的文件 merge 后结果会变，单文件 valueStats 不可信。

Limit 下推（`limitPushdownEnabled:223`）——禁用条件比旧稿描述的多：

```java
public boolean limitPushdownEnabled() {
    if (limit == null || limit <= 0) return false;
    return mergeEngine != PARTIAL_UPDATE      // 部分更新：rowCount ≠ 最终行数
            && mergeEngine != AGGREGATE       // 聚合：merge 后行数减少
            && !deletionVectorsEnabled        // DV：删除行不计有效行
            && valueFilter == null            // （已修正：旧稿遗漏）带谓词时 rowCount 不可信

}
```

且 Limit 仅对 `noOverlapping` 的桶累积生效（`:249`）。**陷阱**：只要查询带了 WHERE 谓词，Limit 下推就不生效——这是正确性使然，别误以为"加了 LIMIT 1 就只扫一个文件"。

## 3. 文件索引如何命中查询

> 文件索引（Bloom/Bitmap/BSI/RangeBitmap）的**内部编码结构**由 **13 号文档**主讲。本节只讲一件事：**查询谓词如何被翻译成对索引的调用、索引返回什么、Scan 与 Read 两阶段各用哪种存储形态。**

**① 要解决什么问题**

min/max 只能做"文件级要么全读要么不读"。对高选择性查询（`user_id='x'` 在百万行里命中 10 行），min/max 完全无能为力——文件的 min/max 范围照样覆盖 'x'。文件索引补上"行级/概率级"这一层精度。

**② 设计原理与取舍**：四种索引各擅其一，没有万能索引。Bloom 省空间但只判"可能存在"；Bitmap 给精确行号但怕高基数；BSI 给数值范围且空间不随基数涨；RangeBitmap 唯一支持 TopN。详细取舍见 **13 号文档**，本节给"查询侧"视角。

### 3.1 FileIndexPredicate 三值逻辑与 Visitor 评估

`FileIndexPredicate`（`paimon-common/.../fileindex/FileIndexPredicate.java`）是文件索引的统一评估入口，可从嵌入字节或独立文件路径构造。它把"谓词树"翻译成"对每列索引 Reader 的调用"：

```
evaluate(Predicate)
  ├─ getRequiredNames(predicate)   提取谓词涉及的列名
  ├─ reader.readColumnIndex(name)  为每列加载相关索引 Reader（可多个：同列可建多种索引）
  └─ Visitor 遍历谓词树
       ├─ LeafPredicate:  predicate.function().visit(reader, fieldRef, literals)
       └─ CompoundPredicate: AND 任一 SKIP 即 SKIP；OR 任一 REMAIN 即 REMAIN
```

**三值逻辑 `FileIndexResult`**（`fileindex/FileIndexResult.java`）是设计的关键：
- `REMAIN`——文件可能命中，保留（Bloom 的"可能存在"、不支持的谓词都归此）。
- `SKIP`——确定不命中，整文件跳过。
- `BitmapIndexResult`——精确行号位图（RoaringBitmap32），可只读命中行（行级跳过）。

**为什么用 Visitor + 基类默认 REMAIN**：每种索引只支持部分谓词。`FileIndexReader` 基类对所有 `visitXxx` 默认返回 `REMAIN`（保守，绝不漏数据），各索引只覆盖自己支持的方法。这样新增索引类型、新增谓词类型互不干扰——这是"绝不能因为索引不支持某谓词就误判 SKIP"这一正确性底线的工程体现。

### 3.2 四种索引各自命中什么谓词

> 编码结构细节见 **13 号文档**；这里只给"查询能否命中"的速查。

| 索引 | 命中的谓词 | 返回 | 适用列 | 关键源码 |
|------|-----------|------|--------|---------|
| **Bloom Filter** | `=` | REMAIN/SKIP（仅文件级） | 高基数等值（user_id/order_id） | `bloomfilter/BloomFilterFileIndex.java` |
| **Bitmap** | `=` `!=` `IN` `NOT IN` `IS (NOT) NULL` | BitmapIndexResult（行号） | 低基数（status/type） | `bitmap/BitmapFileIndex.java` |
| **BSI** | `=` `!=` `<` `<=` `>` `>=` `BETWEEN` `IN` `IS NULL` | BitmapIndexResult | 数值范围（amount/age） | `bsi/BitSliceIndexBitmapFileIndex.java` |
| **Range Bitmap** | 同 BSI + **TopN** | BitmapIndexResult | 有序/半有序数值、TopN | `rangebitmap/RangeBitmapFileIndex.java` |

要点（从查询命中视角）：
- **Bloom 只回 REMAIN/SKIP**（`visitEqual` 里 `filter.testHash(...) ? REMAIN : SKIP`），默认 `fpp=0.1`——10% 误报换更小的索引体积，便于嵌入。
- **Bitmap 回精确行号**，配合 `RawFileSplitRead` 的 `FormatReaderContext.selection`，让格式层只读命中行对应的数据块——这是相对 Parquet/ORC RowGroup（几千~几万行粒度）跳过的本质提升。
- **BSI 把数值按二进制位分片**，空间固定 O(bits×n) 不随基数涨，范围查询 O(bits) 次位运算搞定——解决 Bitmap 在高基数数值列上的爆炸。
- **Range Bitmap 是唯一支持 `visitTopN` 的索引**，用有序字典直接定位 Top-K 行号，免去"全读再排序"。

### 3.3 嵌入式 vs 独立索引：Scan 用前者，Read 用后者

| 模式 | 存储位置 | 适用 | 阶段 |
|------|---------|------|------|
| 嵌入式 | `DataFileMeta.embeddedIndex` 字节，随 Manifest 读入 | 小索引（Bloom，几百 B~几 KB） | **Scan** 用，无额外 I/O |
| 独立文件 | `extraFiles` 中 `.idx` 文件，需额外 I/O | 大索引（Bitmap 可数 MB） | **Read** 用 |

**为什么分两种**：Scan 是热路径，要快速过滤海量文件，大索引嵌入会让 Manifest 膨胀、拖慢 Scan；所以小索引嵌入（Scan 即可用），大索引独立（已确定要读的少数文件再付额外 I/O）。Read 阶段两者都查：先看 `embeddedIndex`，没有再找 `.idx`（`FileIndexEvaluator.java:119-125`）。

### 3.4 Read 阶段评估与 DV 协同（FileIndexEvaluator）

`RawFileSplitRead` 创建文件读取器前调 `FileIndexEvaluator.evaluate()`（`io/FileIndexEvaluator.java:45`）：

```
evaluate(file, dv, filter, topN, limit)
  ├─ 无 filter 无 topN → createBaseSelection(file,dv).limit(limit)   仅处理 Limit   :60
  ├─ createBaseSelection(file, dv)  全行位图 [0,rowCount) AND NOT 删除向量          :100
  ├─ 有 filter → predicate.evaluate(filter) 再 AND baseSelection
  ├─ 有 topN   → predicate.evaluateTopN(topN, selection)                          :81
  └─ 结果 == baseSelection → 返回 REMAIN（无谓词收益，避免无谓传位图）
```

**DV 与索引的位图级协同**（`createBaseSelection:100`）：先生成覆盖 `[0, rowCount)` 的全行位图，`andNot` 掉 DV 标记的删除行，得到"有效行位图"；文件索引结果再 `and` 它。因为 DV 和 Bitmap 索引同用 RoaringBitmap32，这一步是纯位运算、无需逐行判断。DV 机制本身见 §8 与 **04 号文档**。

## 4. 多维排序：空间填充曲线为什么有效

> 本章是本篇主菜——把"空间填充曲线为什么能让多列范围查询都受益"讲透。**06 号文档会引用本章**。Z-Order/Hilbert 的核心算法在 `paimon-common`（`ZIndexer`/`HilbertIndexer`），与计算引擎解耦；Spark/Flink 侧只是把它接到 `repartitionByRange + sortWithinPartitions`。

### 4.1 核心原理：单列排序的局限与 Z-Order 的破解

**① 要解决什么问题**

第 2 章讲过，第④层 min/max 跳过只在"文件内数据各列聚拢"时才有效。问题是：怎么排序，才能让多个列的 min/max **同时**变窄？

**字典序排序（ORDER BY a, b, c）的根本局限**：它只对**前缀列**友好。按 (a, b) 排序后，每个文件的 a 范围很窄（强可裁），但 b 只在 a 相等的小段内有序——跨文件看 b 的 min/max 几乎覆盖全域；c 更糟。所以 `WHERE b = ?`（不带 a 条件）时，几乎所有文件的 b 范围都命中，裁不掉。

```
ORDER BY (col1, col2)，每文件存 4 行：
col1: 1 2 3 4 | 5 6 7 8        ← col1 每文件范围窄：[1,4]、[5,8]，强可裁
col2: a b c d | a b c d        ← col2 每文件范围都是 [a,d]，全覆盖，裁不掉
```

**② 设计原理与取舍——Z-Order 让每列"轮流贡献高位比特"**

空间填充曲线把多维点 (a, b, c) 映射成一个一维值，且**保持局部性**：多维空间里靠近的点，映射后的一维值也大概率靠近。按这个一维值排序落盘，每个文件就对应多维空间里的一小块"邻域"——于是 a、b、c 的 min/max **同时**都窄。

Z-Order 实现局部性的手段是**比特交错（bit interleaving）**：把各列的二进制位轮流取出来拼成 Z-value。其妙处在于——

```
col1=a7..a0, col2=b7..b0  →  Z = a7 b7 a6 b6 a5 b5 ... a0 b0
                                 ↑↑   每列的最高位都排在 Z 的最前面
```

Z-value 的**最高位由所有列的最高位轮流占据**。按 Z 排序时，先按"所有列的最高位"分大块，再按次高位细分……**每一列都参与了高位的划分**，因此每一列都获得了"文件级区分度"。这正是它优于字典序的本质：字典序把全部高位权重给了第一列，Z-Order 把高位权重在各列间均分。

```
Z-Order 排序后（同样每文件 4 行）：
col1: 1 1 2 2 | 3 3 4 4        ← col1 每文件范围窄
col2: a b a b | a b a b        ← col2 每文件范围也窄（不再全覆盖）
两列同时可裁 —— 这就是多维查询都受益的来源
```

**代价**：① 单列查询时，Z-Order 不如对该列做字典序（因为该列只拿到一半高位）；② 排序需要全表 shuffle，是批作业，不适合实时写入路径；③ 字符串等变长类型必须截断/填充到定长才能等权交错（见 §4.2）。

一句话哲学：**字典序把区分度全押在头部列；空间填充曲线把区分度摊给每一列——以"单列略弱"换"多列皆可裁"。**

### 4.2 Z-Order 字节交错算法（ZOrderByteUtils / ZIndexer）

**源码路径**：算法核心 `paimon-common/.../sort/zorder/ZOrderByteUtils.java`、`ZIndexer.java`（引擎无关）；Spark 侧 UDF 接入 `spark/sort/SparkZOrderUDF.java`、`ZorderSorter.java`。

**Step 1：每列转"字典序可比较的定长字节"**。直接拿原始二进制交错会出错——负整数补码的最高位是 1，会排在正数前面；IEEE 754 浮点的符号位同理。所以每种类型要先转成"按无符号字节比较即等价于按数值比较"的形式（`ZOrderByteUtils.java` 的 `*ToOrderedBytes` 系列，`:65-164`）：

| 类型 | 转换 | 为什么 |
|------|------|--------|
| 有符号整数（int/long/...） | 翻转符号位（XOR `0x80...`） | 让负数排到正数前，无符号比较即数值序 |
| 浮点（float/double） | 重解释为 long 后 `XOR (lval>>63) \| Long.MIN_VALUE` | 处理 IEEE 754 符号-量级表示，负数反序 |
| 字符串/二进制 | UTF-8 后截断或补零到定长（`byteTruncateOrFill:156`） | 变长无法等权交错，必须定长 |

**为什么统一用 `PRIMITIVE_BUFFER_SIZE = 8`**（`ZOrderByteUtils.java:45`）：所有原始类型都补到 8 字节（Long 宽度），保证**每列向 Z-value 贡献相同数量的 bit**——这是"各维度在 Z-curve 上权重相等"的前提。若某列只占 2 字节而另一列占 8 字节，前者的低位会与后者的高位交错，权重失衡，局部性被破坏。

**Step 2：逐位交错 `interleaveBits`**（`ZOrderByteUtils.java:184`，已核对）。算法是一个朴素双指针循环：从源列按"列→位"轮询取 bit，写入目标字节的对应位；某列字节用完后跳过该列（`do { ++sourceColumn; ... } while (columnsBinary[sourceColumn].length <= sourceByte)`，`:217-232`）——这就是"变长输入也能交错、用完即跳过"的处理。

```
输入 col_A=[a7..a0], col_B=[b7..b0]
输出      [a7 b7 a6 b6 a5 b5 a4 b4 a3 b3 a2 b2 a1 b1 a0 b0]
```

`ZIndexer`（`ZIndexer.java:65`）是把上面串起来的引擎无关入口：构造时按 `orderColumns` 解析列下标与每列的 `RowProcessor`（`:72-90`），运行时对每行各列转有序字节、调 `interleaveBits` 产出 Z-value 字节数组。Spark/Flink 各自的 sorter 复用它（Spark 的 `SparkZOrderUDF.sortedLexicographically/interleaveBytes` 是同一套逻辑的 UDF 包装，`SparkZOrderUDF.java:323,328`）。

### 4.3 Hilbert 曲线：更好的局部性、更贵的计算

**源码路径**：`paimon-common/.../sort/hilbert/HilbertIndexer.java`（核心，依赖第三方库 `org.davidmoten.hilbert.HilbertCurve`）；Spark 侧 `spark/sort/HilbertSorter.java`、`SparkHilbertUDF.java`。

**① 为什么还要 Hilbert——Z 曲线的"跳跃点"问题**：Z 曲线在每个"Z"形拐角处会有长跳——一维上相邻的两个 Z-value，多维坐标可能隔得很远（比如从一个象限右下角直接跳到下一象限左上角）。这意味着同一个文件里偶尔会混入"多维上很远"的点，拉宽了 min/max。Hilbert 曲线是连续的分形曲线，**相邻的 Hilbert 值对应的多维点一定相邻**，没有跳跃点，所以每文件的多维邻域更紧凑、min/max 更窄。

**② 实现差异**：与 Z-Order 在"字节级"交错不同，`HilbertIndexer` 走"long 级"映射——每列先转成一个 `long`（`hilbertValue(row)`，`HilbertIndexer.java:101,311`），再把这组 long 喂给 `HilbertCurve` 算出曲线上的一维位置，最后 `hilbertCurvePosBytes` 转字节（`:96-104`）。曲线映射本身是递归/位运算密集的，比 Z-Order 单纯交错字节贵得多。

**③ 收益与代价**：局部性更好（实践中文件跳过率通常优于 Z-Order，地理空间经纬度场景尤其明显），代价是排序作业的 CPU 开销更高、且依赖外部库。维度越多（3 列以上）Hilbert 相对 Z-Order 的优势越值得这份代价。

### 4.4 ORDER / ZORDER / HILBERT 何时各自更优

这是用户最该带走的决策表：

| 查询模式 | 最优排序 | 原因 |
|---------|---------|------|
| 单列范围/等值（含"主列+次要列"） | **ORDER** | 字典序把全部高位给头部列，单列区分度最强 |
| 两列等权联合过滤 | **ZORDER** | 比特交错让两列同时可裁，计算便宜 |
| 三列及以上等权 / 地理经纬度 | **HILBERT** | 无跳跃点，多维局部性最佳，值回计算代价 |
| 写入已天然有序的列（如自增时间） | 不排或 ORDER | 已有序，Z/Hilbert 反而打乱、白付 shuffle |

配套陷阱（呼应 §1.4 陷阱 5）：排序列必须中高基数、查询频率相当、控制在 2-4 列；低基数列（gender/status）做 Z/Hilbert 等于白做；列数超 4 后每列分到的高位 bit 太少，各列区分度都不够。

### 4.5 TableSorter：排序为何放在计算引擎层

**为什么不在存储层做**：多维排序要读全表、跨节点 shuffle、重分布——这是个标准批作业，天然需要 Spark/Flink 的分布式 shuffle 能力。存储层只管读写与元数据，把排序留给计算引擎，保持了存储层简洁、也复用了引擎成熟的排序设施。

`TableSorter`（Spark `spark/sort/TableSorter.java:65`，Flink `flink/sorter/TableSorter.java`）是工厂，按 `CoreOptions.OrderType` 选策略：

```java
public static TableSorter getSorter(FileStoreTable table, OrderType orderType, List<String> orderColumns) {
    switch (orderType) {
        case ORDER:   return new OrderSorter(table, orderColumns);   // repartitionByRange + sortWithinPartitions
        case ZORDER:  return new ZorderSorter(table, orderColumns);  // 生成 ZVALUE 列再按其排序
        case HILBERT: return new HilbertSorter(table, orderColumns); // 生成 H_COLUMN 列再按其排序
        case NONE:    return /* no-op */;
    }
}
```

三个 Sorter 共用同一收尾套路（以 `ZorderSorter.java:46` 为例）：先把多列折成单列排序键（ORDER 直接用原列、ZORDER 用 `ZVALUE`、HILBERT 用 `H_COLUMN`），再 `repartitionByRange(key)`（按键范围分区，相近键落同分区/同文件）+ `sortWithinPartitions(key)`（分区内排序），最后丢弃临时键列。`repartitionByRange` 是关键——它保证了"相近 Z/Hilbert 值进同一文件"，min/max 收窄才真正落到文件级。

## 5. Clustering Compaction：在存储层增量排序

### 5.1 两阶段架构如何压低写放大

**① 要解决什么问题**

§4 的多维排序是"批作业、读全表"——流式写入下不可能每来一批就全表重排。能不能像 LSM compaction 一样，**只对新写入的增量排序，已排序数据只在必要时合并**，把写放大压住？这就是 Clustering Compaction。

**② 设计原理与取舍——借 LSM level 区分"已排/未排"**

`ClusteringCompactManager`（`…clustering/ClusteringCompactManager.java`）专为"按聚簇列排序的主键表"设计，两阶段：

```
ClusteringFiles 用 level 天然区分两态（无需额外 sorted 标记）：
  level 0 = unsortedFiles（新写入，待排序）
  level 1 = sortedFiles  （已按聚簇列有序）

Phase 1 排序重写：把所有 level 0 文件按聚簇列外部排序 → 写成 level 1
Phase 2 归并合并：对已存在的 level 1 文件，检测聚簇列范围 overlap / 碎片 → 合并
```

**为什么两阶段能压低写放大**：Phase 1 只碰增量（level 0），Phase 2 只合并真正 overlap/碎片的 level 1 段。相比"每次全表重排 2x 写放大"，增量模式下大部分已排序数据原样不动——写放大可压到 0.4x 量级（读增量+碎片、写同等量，相对全表）。

**③ 关键源码**（`compact(boolean fullCompaction)`，`ClusteringCompactManager.java:195`，已修正：旧稿标 `:196-251`）：

```java
private CompactResult compact(boolean fullCompaction) {
    List<DataFileMeta> unsortedFiles = fileLevels.unsortedFiles();        // :202 level 0
    List<DataFileMeta> existingSortedFiles = fileLevels.sortedFiles();    // :204 快照，避免混入 Phase1 新文件
    for (DataFileMeta file : unsortedFiles) {                             // Phase 1
        result.after().addAll(fileRewriter.sortAndRewriteFile(file, kvSerializer, kvSchemaType, keyIndex));
    }
    List<List<DataFileMeta>> mergeGroups = fullCompaction                 // Phase 2
            ? singletonList(existingSortedFiles)                          // :214 全量：整批一组
            : fileRewriter.pickMergeCandidates(existingSortedFiles);      // :217 增量：智能选段
    // ... 执行各 group 的 mergeAndRewriteFiles
}
```

注意 `existingSortedFiles` 在 Phase 1 之前就快照，避免把 Phase 1 刚产出的 level 1 文件又卷入 Phase 2。

### 5.2 ClusteringFileRewriter：排序、去重、合并

**源码路径**：`…clustering/ClusteringFileRewriter.java`。

**Phase 1 `sortAndRewriteFile`**（`:120`）：建 `BinaryExternalSortBuffer`（外部排序，内存不足自动 spill，§7.3）→ 把文件所有 KV 写入缓冲 → 迭代排序结果，经 `keyIndex.checkKey` 去重后写新文件。去重两模式：
- **FIRST_ROW**：`if (!keyIndex.checkKey(keyBytes)) continue;`（`:165`）——重复 key 直接跳过，保留首个。
- **DEDUPLICATE**：标记 DV（删除旧位置），读取时过滤；DV 机制详见 **04 号文档**。

**Phase 2 `pickMergeCandidates`**（`:192`）：按聚簇列 minKey 排序文件 → 检测 overlap 切 section → **含 overlap（≥2 文件）或过小（`< targetFileSize/2`，阈值在 `:199` `smallSectionThreshold`）的 section** 标记待合并 → 相邻待合并 section 并成一个 merge group。为什么用 `targetFileSize/2`：小于目标一半算"碎片"该合，但不去碰两个接近目标大小的文件（合它们只是浪费 I/O）——在"控碎片"和"省 I/O"间取平衡。

**Phase 2 `mergeAndRewriteFiles`**（`:228`）：min-heap 多路归并按聚簇列重写。**Spill 优化**（`:235`）：输入文件数 > `spillThreshold` 时，把较小的文件先 spill 到行式压缩临时文件，只保留较大文件直读——减少同时打开的文件句柄，避开 "too many open files"。为什么 spill 小文件而非大文件：小文件 spill 读写字节少、开销低，大文件保留列式格式直读还能享受向量化。

### 5.3 增量 vs 全量聚簇

| 模式 | 触发 | Phase 1 | Phase 2 |
|------|------|---------|---------|
| 增量（默认） | `fullCompaction=false` | 仅排 level 0 | `pickMergeCandidates` 智能选 overlap/碎片段 |
| 全量 | `fullCompaction=true` | 同增量 | 所有 sorted 文件作为单一 merge group（≈全表重排） |

**为什么增量是默认**：全量读写整张表、I/O 极高，只适合偶发兜底（如聚簇列变更后一次性重排）。流式写入应配增量 + 合理触发间隔，把聚簇列选在中高基数、查询常用的过滤列上（呼应 §1.4 陷阱 6）。`ClusteringFiles`（`…/ClusteringFiles.java`）只维护 unsorted/sorted 两态而非传统多级 Levels——Clustering 关心的是"在聚簇列上有没有序"，两态足够、管理更简。

## 6. Split 生成：跳过 merge 的 rawConvertible

### 6.1 rawConvertible 判定的三个条件

**① 要解决什么问题**

Scan 漏斗砍剩的文件要分给并行 task 读。怎么分 Split，既最大化并行度，又**尽量避免不必要的 merge**？主键表读取若每个 Split 都走 SortMergeReader 多路归并（§7），CPU/内存开销远大于"直读原始文件"。能直读的关键标记就是 `rawConvertible`。

**② 设计原理与取舍**：`SplitGenerator` 产出 `SplitGroup`（`table/source/SplitGenerator.java`），`rawConvertible=true` 表示这组文件可跳过 merge 直读（再叠 DV 过滤）。`alwaysRawConvertible()`（`MergeTreeSplitGenerator.java:64`）：

```java
public boolean alwaysRawConvertible() {
    return deletionVectorsEnabled || mergeEngine == FIRST_ROW;
}
```

DV 启用时恒可直读——因为更新/删除都通过标记 DV 实现，读取只需按位图过滤已删行，根本不需要跨文件 merge（DV 机制见 §8 / **04 号文档**）。

`splitForBatch` 的逐批判定（`MergeTreeSplitGenerator.java:69`，已核对）：

```java
boolean rawConvertible = files.stream().allMatch(f -> f.level() != 0 && withoutDeleteRow(f));
boolean oneLevel = (files 的 level 集合大小 == 1);
if (rawConvertible && (deletionVectorsEnabled || mergeEngine == FIRST_ROW || oneLevel)) {
    return BinPacking.packForOrdered(files, weightFunc, targetSplitSize).stream()
            .map(SplitGroup::rawConvertibleGroup).collect(toList());   // :78
}
```

**三个条件**（缺一不可）：
1. 所有文件 `level != 0`——Level 0 文件间 key 可能 overlap，直读会读出未合并的多版本（这就是 §1.4 陷阱 1 的根因）。
2. 所有文件无删除行（`withoutDeleteRow`，`:151`）——否则需要 merge 处理 delete。
3. 满足之一：DV 启用 / FIRST_ROW 引擎 / **所有文件同一 level**（同 level 文件无 overlap，天然可直读）。

### 6.2 IntervalPartition 两层划分提高并行度

**源码路径**：`mergetree/compact/IntervalPartition.java`。把"可能 overlap 的一组文件"切成不重叠的 **Section**，每个 Section 内再切 **SortedRun**——`partition()` 返回 `List<List<SortedRun>>`（`:67`）。

**外层 Section 划分**（`partition():67`）：文件按 minKey 排序，维护当前 section 右边界 `bound`：下一个文件 `minKey > bound` → 新建 section；否则并入并 `bound = max(bound, maxKey)`。结果：**Section 间 key 范围不重叠**。

**内层 SortedRun 划分**（`partition(List):93`）：用优先队列（按各 run 末文件 maxKey 排序）贪心分组——文件 minKey 大于队首 run 的末 maxKey 则追加该 run，否则开新 run。注释明确这一层是"把单个 Section 内的 SortedRun 数最小化"（`:51-57`）。

**为什么 Section 不重叠重要**：一个 Split 里的文件要一起 merge。把文件切成互不重叠的 Section 后，**不同 Section 可独立并行处理**——并行度从"bucket 数"提升到"bucket 数 × 平均 Section 数"。

### 6.3 BinPacking 装箱与 openFileCost

**源码路径**：`paimon-common/.../utils/BinPacking.java`。两种装箱：
- `packForOrdered`（`:33`）：保序顺序装，累积权重超 target 就开新箱——**Split 生成用它**，保持文件的时间/key 范围连续，利于 merge 与缓存局部性。
- `packForFixedBinNumber`（`:61`）：先按权重排序再优先队列贪心分到固定箱数——并行度固定的场景用。

**权重函数 `max(file.fileSize(), openFileCost)`**：`openFileCost` 默认 4MB（`CoreOptions.SOURCE_SPLIT_OPEN_FILE_COST:649`，已核对）。它给"打开文件的固定成本"兜底——一个 1MB 小文件按 4MB 计权，避免单个 Split 塞进几百个小文件后打开文件本身成了瓶颈。`source.split.target-size` 默认 128MB（`:643`，已核对）。

### 6.4 主键表 vs 追加表 Split 策略

| 维度 | `MergeTreeSplitGenerator`（主键表） | `AppendOnlySplitGenerator`（追加表） |
|------|-----------------------------------|------------------------------------|
| 输入排序 | 无（交给 IntervalPartition） | 按 `minSequenceNumber` 排序 |
| Section 划分 | 必要（处理 key overlap） | 不需要（追加表文件本就无 overlap） |
| rawConvertible | 条件性（§6.1 三条件） | **始终 true**（无 merge 语义） |
| 装箱单位 | Section（一组文件） | 单个文件 |
| 流式 Split | 不拆（所有文件一个 Split） | BUCKET_UNAWARE 模式拆分，其余不拆 |
| 权重函数 | `max(fileSize, openFileCost)` | 同左 |

追加表始终 `rawConvertible` 是它读取天然快的来源——没有主键 merge，文件就是最终数据。

## 7. Lookup 点查与 SortMergeReader

> 本节两个主题：**Lookup 点查**的 LSM 底层（LookupLevels/SST 缓存/远程文件）由 **01 §6** 主讲，这里只讲"查询如何走这条路、上→下查与 Level0/1+ 策略的依据"；**SortMergeReader**（rawConvertible 不成立时的归并读）是本篇主讲。

### 7.1 LookupLevels 多层查找（引用 01 §6）

**① 何时走 Lookup**：流式 Join / 维表 `FOR SYSTEM_TIME AS OF` / CDC 后按主键点查——QPS 高、要毫秒级单行查找。范围扫描不走这里（§1.4 陷阱 7）。

**② 查找策略**（`LookupUtils.lookup:35`，已核对）：

```java
public static <T> T lookup(Levels levels, InternalRow key, int startLevel, ...) {
    for (int i = startLevel; i < levels.numberOfLevels(); i++) {
        T result = (i == 0)
                ? level0Lookup.apply(key, levels.level0())   // Level 0：遍历
                : lookup(key, levels.runOfLevel(i));         // Level 1+：二分
        if (result != null) return result;                   // 命中即止（上层更新）
    }
}
```

- **从上往下查、命中即止**：LSM 中上层数据更新。同 key 第一个命中的就是最新值，无需继续往下。
- **Level 0 遍历**（`lookupLevel0:59`）：L0 文件 key 范围 overlap，TreeSet 按 sequence 降序，逐个查"key 是否在 [minKey,maxKey]"，最新者先命中。这也是"L0 堆积拖慢点查"（§1.4 陷阱 1）的来源。
- **Level 1+ 二分**（`lookup:79`）：每层是一个无 overlap 的 SortedRun，二分定位"maxKey ≥ target 的最小文件"，只查一个文件。

**③ 本地 SST 缓存与远程文件**（`LookupLevels.createLookupFile:182`、`remoteSst:274`）：远程对象存储随机读延迟高，故把数据文件转成本地 SST（带 Bloom Filter，`bfGenerator` 按行数定大小，`:242`），用 Caffeine LRU 缓存（按磁盘大小淘汰）。Flink 扩容/failover 时本地无缓存需重建——大文件场景开远程 Lookup 文件（compaction 时预构建上传、启动直接下载）。**这套底层详见 01 §6**，不再展开。

### 7.2 LoserTree vs MinHeap：同 key 快速路径

**① 何时走 SortMergeReader**：当 §6.1 的 `rawConvertible` 不成立（有 L0、有删除行、跨多 level 且非 DV/FIRST_ROW），就得多路归并合并同 key 的多版本。引擎可配（`sort-engine`，默认 `LOSER_TREE`，`CoreOptions.java:600`，已核对）：

```java
switch (sortEngine) {              // SortMergeReader.createSortMergeReader:41
    case MIN_HEAP:   return new SortMergeReaderWithMinHeap<>(...);
    case LOSER_TREE: return new SortMergeReaderWithLoserTree<>(...);
}
```

| 维度 | MinHeap | LoserTree |
|------|---------|-----------|
| 复杂度 | O(log N)/元素 | O(log N)/元素，**常数更小** |
| Batch 模式 | 多 batch（每次 readBatch 返回新迭代器） | 单 batch（后续 next() 迭代） |
| 同 key 处理 | PriorityQueue 逐个 poll | **状态机快速路径批量处理** |
| 缓存友好 | 堆节点对象 | 数组存储，更友好 |
| 适用 | 少量 reader、大 batch | 大量 reader（compaction 5-10 路）、追最优 |

**② LoserTree 的核心优化——同 key 快速路径**（`LoserTree.java:45`）：LSM 里多个 reader 常含同 key 不同版本。LoserTree 用 6 状态机（`State` 枚举，`:338`，已核对——LOSER/WINNER 各 NEW_KEY/SAME_KEY/POPPED 共 6 个）管理。winner 被 pop（`WINNER_POPPED`）后，若 `firstSameKeyIndex >= 0`（树中有同 key 节点），**直接换位、不从底部逐级比较**（`:143-150`）——同 key 的 N 个版本合并从 O(N log N) 降到 O(N)。compaction 多路且同 key 频繁，故默认 LoserTree。

### 7.3 Spill 与外部排序

`BinaryExternalSortBuffer`（`paimon-common/.../sort/`）：内存写 `BinaryInMemorySortBuffer`，不足时 spill 到磁盘通道，读时 `BinaryExternalMerger` 多路归并——§5.2 的 Clustering Phase 1 排序即用它。

Phase 2 `mergeAndRewriteFiles` 的独立 spill 策略（`ClusteringFileRewriter.java:235`，已核对）：输入文件数 > `spillThreshold` 时，**较小的文件先 spill 到行式压缩临时文件**，较大文件保留列式直读。为什么 spill 小不 spill 大：小文件 spill 字节少、开销低；大文件保列式格式可享向量化读取。这同时压低了同时打开的文件句柄数，避开 "too many open files"。

## 8. DV 对查询路径的影响（引用 04）

> DeletionVector 的**产生与维护**由 **04 号文档**主讲；本节只讲它**如何改变查询路径**。

DV 用 RoaringBitmap32 标记文件中已删除的行位置，使主键表读取从"全量 merge"变为"直读 + 按位图过滤"。对查询的三处影响：

**① Scan 阶段——rawConvertible 恒成立**：DV 启用时 `alwaysRawConvertible()` 返回 true（§6.1），所有 Split 跳过 merge。代价是 **Limit 下推被禁**（`limitPushdownEnabled` 含 `!deletionVectorsEnabled`，§2.6）——文件 rowCount 不等于有效行数（要减去 DV 删除行），不能据它精确截断。

**② Read 阶段——ApplyDeletionVectorReader**（`deletionvectors/ApplyDeletionVectorReader.java`）：迭代每条记录时 `deletionVector.isDeleted(position)` 检查行位置是否被标删，是则跳过。**基于行位置而非 key** —— 无需知道 key 结构、对所有表类型通用、单次检查 O(1)。

**③ 与文件索引的位图级协同**（`FileIndexEvaluator.createBaseSelection:100`，§3.4）：全行位图 `[0,rowCount)` 先 `andNot` DV 得有效行，文件索引结果再 `and` 它。DV 与 Bitmap 索引同用 RoaringBitmap32，纯位运算完成，无需逐行判断——这是"删除"和"索引命中"两个过滤在读期高效合一的关键。

## 9. 全局统计信息（ANALYZE）

**① 与文件级 SimpleStats 的分工**：文件级 `SimpleStats`（§2.4）只有 min/max/nullCounts，写入时顺带收集、用于查询过滤。但查询优化器（Flink/Spark）选 join 顺序、聚合并行度还需要**全局 NDV、数据分布**——这是文件级统计给不了的，需要 ANALYZE 全表收集。

**② 模型**（`stats/Statistics.java`、`ColStats.java`）：`Statistics` 关联 `snapshotId`/`schemaId`，含 `mergedRecordCount`/`mergedRecordSize` 与 `Map<String, ColStats>`；`ColStats` 含 `colId`、`distinctCount`(NDV)、`min`/`max`、`nullCount`、`avgLen`、`maxLen`。

**③ 读写与一致性**（`stats/StatsFileHandler.java`）：`writeStats` 先把字段按对应 schema 序列化成字符串再写 JSON 文件；统计文件名写入 `Snapshot.statistics()`，使统计与数据版本一致、支持时间旅行查询的成本估算。手动 ANALYZE（Flink/Spark Procedure）而非自动收集——避免自动收集对写入路径的不确定性影响，用户自选低峰期、自选大表。

> NDV 的近似算法（HyperLogLog 等）属收集端实现细节，本篇不展开（旧稿此处标注为"推测"，已删除以免误导）。

## 10. 调优速查与设计决策总结

### 10.1 与 Iceberg 过滤机制对比

| 维度 | Paimon | Iceberg |
|------|--------|---------|
| Manifest 过滤 | partitionStats + **bucket/level/rowId 范围** | partition summary（per-field min/max） |
| 文件统计 | SimpleStats（BinaryRow，零反序列化） | column metrics（含 NaN 计数，Avro） |
| Schema Evolution | SimpleStatsEvolution 动态逆映射 | 基于 field-id 静态映射 |
| 文件索引 | Bloom/Bitmap/BSI/RangeBitmap（嵌入+独立，**行级**） | Bloom（独立 Puffin，仅文件级） |
| 多维排序 | ORDER/ZORDER/HILBERT，且 **Clustering 增量** | ZORDER，仅全表重写 |
| Split | IntervalPartition + BinPacking + **rawConvertible** | FileScanTask 按大小拆，总是 merge |
| Lookup 点查 | **原生**（LookupLevels + 本地 SST + 远程文件） | 无（需外部 KV） |

Paimon 强在：行级文件索引、原生点查、存储层增量聚簇、DV+索引位图级协同、Manifest 级 bucket/level 范围裁剪。Iceberg 强在：NaN 计数、更成熟的 Z-Order 生态与 Manifest caching/rewrite、更灵活的格式支持。

### 10.2 关键配置参数

| 参数 | 默认值 | 作用 | 调优 |
|------|-------|------|------|
| `file-index.bloom-filter.columns` | - | Bloom 索引列 | 高基数等值列 |
| `file-index.bitmap.columns` | - | Bitmap 索引列 | 低基数等值/IN 列 |
| `file-index.bsi.columns` | - | BSI 索引列 | 数值范围列 |
| `file-index.read-enabled` | true（`CoreOptions.java:391`） | Scan/Read 是否用文件索引 | 保持开启 |
| `sort-engine` | LOSER_TREE（`:600`，已核对） | 多路归并引擎 | 多路 compaction 用 LOSER_TREE |
| `source.split.target-size` | 128MB（`:643`，已核对） | 目标 Split 大小 | 按并行度调 |
| `source.split.open-file-cost` | 4MB（`:649`，已核对） | 打开文件固定成本 | 小文件多时调大 |
| `lookup.cache-file-retention` | 1h（`:1277`） | Lookup SST 缓存保留 | 频繁查询调大 |
| `lookup.cache-max-disk-size` | 无限制 | Lookup 缓存磁盘上限 | 按本地盘调 |
| `clustering.columns` | - | 聚簇列 | 多维查询常用过滤列，2-4 列中高基数 |

### 10.3 最佳实践清单

1. **分区先行**：partition 裁剪是第①层、最便宜的一刀；用查询最频繁的低基数维度（如日期）分区。
2. **控住 Level 0**：及时 compaction——L0 堆积会让 min/max 跳过、`rawConvertible`、Lookup 点查同时退化（§1.4 陷阱 1）。
3. **按查询模式选排序**：单列 ORDER；两列等权 ZORDER；三列以上/经纬度 HILBERT（§4.4）。流式无序写入用 Clustering 增量模式落地。
4. **按列类型选索引**：高基数等值 Bloom、低基数 Bitmap、数值范围 BSI、TopN RangeBitmap；同列可叠多索引由 `FileIndexPredicate` 自动组合（索引选型细节见 13）。
5. **主键表启 DV 换 rawConvertible**：所有 Split 跳 merge 直读，读延迟显著降低；代价是额外 DV 存储 + Limit 下推失效（§8）。
6. **Lookup 场景**：本地盘留够、开远程 Lookup 文件加速 failover（§7.1 / 01 §6）。
7. **Split 大小**：约等于 `总量 / (并行度 × 2~4)`；小文件多时调大 `open-file-cost`。

### 10.4 设计决策总结

| 决策点 | 选择 | 取舍/代价 | 收益 |
|--------|------|----------|------|
| 多层裁剪 | Manifest→Entry→文件→索引 漏斗 | 实现复杂 | 便宜的判断早做、多砍，昂贵的只对幸存者做 |
| 统计存储 | SimpleStats 用 BinaryRow | 不易调试 | 零反序列化比较、内存省 |
| Schema 演进 | 谓词逆映射到旧 schema | 每 schemaId 缓存映射 | 旧文件免重写仍可被裁 |
| 主键 value 过滤 | 条件性（ALL 须显式开） | 部分场景过滤受限 | **正确性优先**，不漏 merge 数据 |
| Limit 下推 | 带谓词/DV/PU/AGG 即禁 | 这些场景不加速 | rowCount 不可信时不误截断 |
| 多维排序 | 提供 ORDER/ZORDER/HILBERT | 用户需选、批作业开销 | 各列区分度按需分配，多列皆可裁 |
| Z-Order 交错 | 各列定长（8B）等权交错 | 字符串须截断/填充 | 各维度权重相等、局部性好 |
| Hilbert | long 级曲线映射 | 计算更贵、依赖外部库 | 无跳跃点、局部性最优 |
| Clustering | 两阶段增量 | 实现复杂 | 写放大从 2x 压到 0.4x 量级 |
| Split | rawConvertible + IntervalPartition | 严格条件判定 | 能直读则跳 merge、不重叠段可并行 |
| 归并引擎 | 默认 LoserTree（可配 MinHeap） | 维护两套 | 同 key 快速路径，compaction 更快 |
| DV | 行位置位图（RoaringBitmap32） | 额外存储、Limit 失效 | rawConvertible 恒成立、与索引位运算合一 |
| 全局统计 | 手动 ANALYZE、关联 Snapshot | 需主动触发 | 与数据版本一致、供优化器选计划 |
