## 1. 查询规划与多层文件过滤

### 1.1 FileStoreScan 整体架构

Paimon 的查询规划核心入口是 `FileStoreScan.plan()`，它采用**多层漏斗式过滤**架构，从粗粒度到细粒度逐层裁剪无关文件，最大限度减少 I/O 操作。

**为什么这么做**: 数据湖中一个表可能有数万个数据文件，如果全量扫描 Manifest 并读取所有文件，查询延迟会非常高。通过多层过滤，可以在元数据层面就剔除大量不相关文件，只读取真正需要的数据。

**好处**: 每一层过滤的代价不同 -- Manifest 级过滤只需要比较聚合统计信息(几十字节)，Entry 行过滤利用反序列化前的原始二进制数据，文件级过滤才需要完整的统计信息。这种分层设计使得每一层的计算/IO 成本逐级递增，但数据量逐级递减。

```
源码路径:
- paimon-core/.../operation/FileStoreScan.java (接口定义)
- paimon-core/.../operation/AbstractFileStoreScan.java (核心实现)
- paimon-core/.../operation/KeyValueFileStoreScan.java (主键表)
- paimon-core/.../operation/AppendOnlyFileStoreScan.java (追加表)
```

继承体系:

```
FileStoreScan (接口)
    +-- AbstractFileStoreScan (抽象类, 通用过滤逻辑)
            +-- KeyValueFileStoreScan (主键表专用, 分离 key/value 过滤)
            +-- AppendOnlyFileStoreScan (追加表专用, 单一过滤器)
```

`plan()` 方法的核心执行流程 (见 `AbstractFileStoreScan.java:297-349`):

```
1. readManifests()         -> ManifestsReader 做 Manifest 级粗过滤
2. readManifestEntries()   -> 读 Manifest 文件内的条目, 逐条做行过滤+统计过滤
3. postFilterManifestEntries() -> 可选的后过滤(整桶过滤、Limit 下推)
```

### 1.2 第一层: Manifest 级过滤 (ManifestsReader)

**源码路径**: `paimon-core/.../operation/ManifestsReader.java:166-207`

`ManifestsReader.filterManifestFileMeta()` 是第一层过滤，在读取 Manifest 文件内容之前，仅凭 `ManifestFileMeta` 的聚合元数据判断整个 Manifest 文件是否可以跳过。

**ManifestFileMeta 包含的过滤维度** (见 `manifest/ManifestFileMeta.java:44-61`):

| 字段 | 类型 | 过滤用途 |
|------|------|---------|
| `partitionStats` | `SimpleStats` | 分区范围裁剪 -- 每个 Manifest 记录其内所有条目的分区统计信息(min/max/nullCounts) |
| `minBucket` / `maxBucket` | `Integer` | Bucket 范围裁剪 -- 如果指定查询 bucket=5，但 Manifest 的 bucket 范围是 [10, 20]，则跳过 |
| `minLevel` / `maxLevel` | `Integer` | Level 范围裁剪 -- Compaction 选取特定 Level 文件时可跳过不相关 Manifest |
| `minRowId` / `maxRowId` | `Long` | Row Range 裁剪 -- 支持基于行号范围的过滤 |
| `numAddedFiles` / `numDeletedFiles` | `long` | 辅助信息 -- 仅含删除条目的 Manifest 在 ALL 模式下可直接过滤 |

**为什么在 Manifest 层面做过滤**: Manifest 文件可能包含数千个数据文件条目，每个条目需要反序列化。如果一个 Manifest 的分区范围完全不匹配查询谓词，跳过整个 Manifest 可避免读取和反序列化数千条 Entry。

过滤逻辑核心代码 (简化自 `ManifestsReader.java:166-207`):

```java
private boolean filterManifestFileMeta(ManifestFileMeta manifest) {
    // 1. Bucket 范围过滤
    if (specifiedBucket != null && bucket 不在 [minBucket, maxBucket] 范围) return false;
    
    // 2. Level 范围过滤
    if (specifiedLevel != null && level 不在 [minLevel, maxLevel] 范围) return false;
    if (levelMinMaxFilter != null && !levelMinMaxFilter.test(minLevel, maxLevel)) return false;
    
    // 3. 分区谓词过滤 -- 利用 partitionStats 的 min/max/nullCounts
    if (partitionFilter != null) {
        SimpleStats stats = manifest.partitionStats();
        long totalEntries = manifest.numAddedFiles() + manifest.numDeletedFiles();
        if (!partitionFilter.test(totalEntries, stats.minValues(), stats.maxValues(), stats.nullCounts()))
            return false;
    }
    
    // 4. Row Range 过滤
    if (rowRangeIndex != null && !rowRangeIndex.intersects(minRowId, maxRowId)) return false;
    
    return true;
}
```

### 1.3 第二层: ManifestEntry 级行过滤

**源码路径**: `AbstractFileStoreScan.java:526-567`

通过第一层过滤后的 Manifest 文件需要被读取，此时在反序列化每一行 ManifestEntry 时，Paimon 使用 `createEntryRowFilter()` 进行**原始行级过滤**。

**为什么这么做**: ManifestEntry 的完整反序列化包括 DataFileMeta 中的 keyStats、valueStats 等复杂结构，代价较高。通过在 `InternalRow` 层面(反序列化前)用轻量级 getter 函数提取分区、bucket、level 等简单字段进行过滤，可以避免对不需要的条目做完整反序列化。

**好处**: 减少了大量不必要的对象创建和内存分配。该过滤器直接操作 `InternalRow` 的二进制格式，使用 `ManifestEntrySerializer.partitionGetter()` 等静态方法提取字段，是零拷贝的高效操作。

核心过滤条件:
1. **分区过滤** -- `partitionFilter.test(partition)`
2. **Bucket 过滤** -- `bucketFilter.test(partition, bucket, totalBucket)` (支持 partition-aware 的 bucket 过滤)
3. **Level 过滤** -- 精确 level 匹配或范围过滤
4. **文件名过滤** -- 用于增量读取等场景

### 1.4 第三层: 文件级统计信息过滤 (filterByStats)

通过行过滤后的条目会被完整反序列化为 `ManifestEntry` 对象，此时可以访问 `DataFileMeta` 中的完整统计信息进行精细过滤。

**KeyValueFileStoreScan.filterByStats()** (见 `KeyValueFileStoreScan.java:141-165`):

```java
protected boolean filterByStats(ManifestEntry entry) {
    // 1. Value 过滤(条件性启用)
    if (isValueFilterEnabled() && !filterByValueFilter(entry)) return false;
    
    // 2. Key 统计信息过滤 -- 使用 keyStats (min/max/nullCounts)
    // 通过缓存避免重复计算 Schema 映射
    Predicate notEvolvedFilter = notEvolvedKeyFilterMapping.computeIfAbsent(
        entry.file().schemaId(),
        id -> fieldKeyStatsConverters.filterUnsafeFilter(
            entry.file().schemaId(), keyFilter, true));
    if (notEvolvedFilter == null) return true;
    
    DataFileMeta file = entry.file();
    SimpleStatsEvolution.Result stats = fieldKeyStatsConverters
        .getOrCreate(file.schemaId())
        .evolution(file.keyStats(), file.rowCount(), null);
    return notEvolvedFilter.test(file.rowCount(), stats.minValues(), stats.maxValues(), stats.nullCounts());
}
```

**为什么主键表使用 keyStats 做主过滤**: 主键表的数据按主键排序存储，keyStats 能精确反映每个文件的主键范围。当查询条件包含主键谓词时，可以高效地通过 min/max 范围判断跳过文件。

**AppendOnlyFileStoreScan 的过滤策略** (见 `AppendOnlyFileStoreScan.java:121-158`):

追加表没有主键，所有列都使用 valueStats 进行过滤。过滤流程分为两步:
1. **统计信息过滤** -- 使用 valueStats 的 min/max/nullCounts 进行初步过滤
2. **文件索引过滤** -- 如果启用了 fileIndexReadEnabled，进一步使用嵌入的文件索引(如 Bloom Filter)进行精细过滤

这与主键表的差异在于:
- 主键表: 先用 keyStats 过滤，再条件性地用 valueStats 和文件索引
- 追加表: 直接用 valueStats 过滤，然后可选地用文件索引

**AppendOnlyFileStoreScan.filterByStats()** (见 `AppendOnlyFileStoreScan.java:121-158`):

```java
protected boolean filterByStats(ManifestEntry entry) {
    // 1. 获取 Schema 映射后的过滤器
    Predicate notEvolvedFilter = notEvolvedFilterMapping.computeIfAbsent(
        entry.file().schemaId(),
        id -> simpleStatsEvolutions.filterUnsafeFilter(
            entry.file().schemaId(), inputFilter, true));
    if (notEvolvedFilter == null) return true;
    
    // 2. valueStats min/max 过滤
    SimpleStatsEvolution evolution = simpleStatsEvolutions.getOrCreate(entry.file().schemaId());
    SimpleStatsEvolution.Result stats = evolution.evolution(
        entry.file().valueStats(), entry.file().rowCount(), entry.file().valueStatsCols());
    boolean result = notEvolvedFilter.test(rowCount, stats.minValues(), stats.maxValues(), stats.nullCounts());
    if (!result) return false;
    
    // 3. 文件索引过滤(如 Bloom Filter) -- 仅在启用时使用
    if (fileIndexReadEnabled) {
        return testFileIndex(entry.file().embeddedIndex(), entry);
    }
    return true;
}
```

**DataFileMeta 中的统计信息字段** (见 `io/DataFileMeta.java:62-91`):

| 字段 | 用途 |
|------|------|
| `minKey` / `maxKey` | 主键范围边界(BinaryRow)，用于 IntervalPartition 和二分查找 |
| `keyStats` | 主键列的 SimpleStats(min/max/nullCounts)，用于主键谓词过滤 |
| `valueStats` | 值列的 SimpleStats，用于非主键列谓词过滤 |
| `valueStatsCols` | 稠密存储模式下标识哪些列有统计信息 |
| `embeddedIndex` | 嵌入的文件索引(Bloom Filter 等)字节 |
| `rowCount` | 行数，用于统计信息计算和 Limit 下推 |
| `deleteRowCount` | 删除行数，用于判断是否可以做 rawConvertible |

### 1.5 SimpleStats 结构详解

**源码路径**: `paimon-core/.../stats/SimpleStats.java`

`SimpleStats` 是 Paimon 中最核心的统计信息结构，用于文件级和 Manifest 级的谓词过滤。

```java
public class SimpleStats {
    private final BinaryRow minValues;    // 每列最小值, 以 BinaryRow 形式存储
    private final BinaryRow maxValues;    // 每列最大值
    private final BinaryArray nullCounts; // 每列 null 计数(Long 数组)
}
```

**为什么采用 BinaryRow 存储**: 与使用 Java 对象数组(如 `Object[]`)相比，`BinaryRow` 将所有列值紧凑地存储在连续的字节数组中。这带来两个好处:
1. **极低的内存开销** -- 避免了 Java 对象头和引用的额外开销，对于数万个文件的场景，内存节省显著。
2. **零反序列化比较** -- 谓词评估时可以直接在二进制数据上比较，无需将每列值反序列化为 Java 对象。

### 1.6 Schema Evolution 对统计信息的影响

**源码路径**: `paimon-core/.../stats/SimpleStatsEvolution.java`, `SimpleStatsEvolutions.java`

数据文件可能是在旧 Schema 下写入的，而查询谓词基于当前 Schema。`SimpleStatsEvolution` 负责统计信息的 Schema 映射:

1. **indexMapping** -- 旧 Schema 字段到新 Schema 字段的索引映射。新增字段映射值为 -1，对应的 nullCount 设为 rowCount(意味着旧数据中该字段全为 null)。
2. **castFieldGetters** -- 类型变更时的类型转换(如 INT -> BIGINT)。
3. **denseFields 优化** -- 当 valueStatsCols 非 null 时，说明统计信息是稠密存储的(仅包含部分列)，需要先做稠密到稀疏的映射。

**filterUnsafeFilter** 方法 (见 `SimpleStatsEvolutions.java`): 将当前 Schema 下的 Predicate 逆向映射("devolve")到旧 Schema 下，使之可以直接应用于旧文件的统计信息。`keepNewFieldFilter=true` 使得新增字段的过滤条件被保留 -- 例如新增字段 `c` 的谓词 `c > 3`，对于旧文件(该字段不存在，等同全 null)可以安全地过滤。

### 1.7 主键表 vs 追加表谓词下推差异

| 维度 | KeyValueFileStoreScan | AppendOnlyFileStoreScan |
|------|----------------------|------------------------|
| **主过滤字段** | keyStats (主键统计) | valueStats (全列统计) |
| **Value 过滤** | 条件性启用，受 scanMode/mergeEngine 影响 | 始终使用 inputFilter |
| **文件索引** | 仅在 value 过滤中使用 (embeddedIndex) | 在 filterByStats 中直接使用 |
| **Whole Bucket 过滤** | 支持(整桶判断，解决 overlapping 问题) | 不需要(无 merge 语义) |
| **Limit 下推** | 受限(PARTIAL_UPDATE/AGGREGATE/DV 禁用) | 受限(DV/数据演化 禁用) |

**为什么主键表的 Value 过滤是条件性的**: 主键表存在 merge 语义 -- 同一主键可能在多个文件中出现不同值。在 `ScanMode.ALL` 模式下(批读取全量快照)，如果启用 value 过滤，可能会漏掉某些文件中需要参与 merge 的记录。因此只有在 `valueFilterForceEnabled`(由上层显式启用)或 `CHANGELOG` 模式下的 LOOKUP/FULL_COMPACTION producer 才启用 (见 `KeyValueFileStoreScan.java:196-211`)。

### 1.8 后过滤: wholeBucketFilter 与 Limit 下推

**源码路径**: `KeyValueFileStoreScan.java:232-254`

`postFilterManifestEntries()` 是主键表独有的后处理步骤，在所有 Manifest 条目都收集完毕后执行:

1. **按 (partition, bucket) 分组** -- 使用 `LinkedHashMap` 保序
2. **判断 noOverlapping** -- 如果同一桶内所有文件都在同一非零 Level，则无 overlap
3. **wholeBucketFilter** -- 对有 overlap 的桶做整桶判断，对无 overlap 的桶做逐文件判断
4. **Limit 下推** -- 仅对无 overlap 的桶生效，累积 rowCount 直到达到 limit

**wholeBucketFilter 的设计意图** (见 `KeyValueFileStoreScan.java:238-242, 285-299`):

**执行逻辑**:
1. 如果 `noOverlapping == true`:
   - 调用 `filterWholeBucketPerFile(entries)` -- 逐文件过滤
   
2. 如果 `noOverlapping == false`:
   - 调用 `filterWholeBucketAllFiles(entries)` -- 整桶判断
   - **特殊情况**: 当 `!deletionVectorsEnabled && (mergeEngine == PARTIAL_UPDATE || mergeEngine == AGGREGATE)` 时，直接返回所有文件，不做过滤
   - **原因**: 部分更新和聚合引擎下，单个文件的 valueStats 不能代表 merge 后的最终结果

**设计原理**:
- 当同一 bucket 内的文件存在 key overlap(多 level 或 level 0)时，即使一个文件的 valueStats 不匹配谓词，其中的记录可能在 merge 后改变结果，所以只能整桶判断:只有桶内所有文件都不匹配才能跳过整个桶
- 当文件间无 overlap(同一非零 level)时，可以逐文件过滤

## 2. 文件索引加速查询

### 2.1 FileIndexPredicate 评估流程

**源码路径**: `paimon-common/.../fileindex/FileIndexPredicate.java`

`FileIndexPredicate` 是文件索引的统一评估入口，支持从嵌入字节或文件路径构造。评估流程:

```
evaluate(Predicate)
  +-- getRequiredNames(predicate)     // 提取谓词涉及的列名
  +-- reader.readColumnIndex(name)    // 为每个列加载所有相关索引 Reader
  +-- FileIndexPredicateTest.test()   // 用 Visitor 模式遍历谓词树
        +-- visit(LeafPredicate)      // 叶子节点: 调用每个 FileIndexReader 的 visit 方法
        |     predicate.function().visit(fileIndexReader, fieldRef, literals)
        +-- visit(CompoundPredicate)  // 复合节点: AND/OR 组合
              +-- AND: 短路求值, 任一 SKIP 则 SKIP
              +-- OR: 任一 REMAIN 则 REMAIN
```

**FileIndexResult** (见 `fileindex/FileIndexResult.java`) 是一个三值逻辑接口:
- `REMAIN` -- 文件可能包含匹配数据，需要保留
- `SKIP` -- 文件确定不包含匹配数据，可以跳过
- `BitmapIndexResult` -- 返回精确的行号位图(RoaringBitmap32)，仅读取命中的行

**为什么使用 Visitor 模式**: 不同类型的索引(Bloom、Bitmap、BSI、Range Bitmap)支持不同的谓词操作。`FileIndexReader` 基类为所有操作提供默认返回 `REMAIN`(保守策略)，各索引实现只需覆盖自己支持的操作。这种设计使得新增索引类型和新增谓词类型互不干扰。

### 2.2 四种索引的查询应用

#### 2.2.1 Bloom Filter

**源码路径**: `paimon-common/.../fileindex/bloomfilter/BloomFilterFileIndex.java`

| 特性 | 说明 |
|------|------|
| 支持的谓词 | `=` (等值查询) |
| 返回类型 | `REMAIN` / `SKIP` (文件级粗过滤) |
| 适用场景 | 高基数列的等值查询(如 user_id, order_id) |
| 配置参数 | `items` (预期元素数, 默认 1000000), `fpp` (误报率, 默认 0.1) |

核心实现 (见 `BloomFilterFileIndex.Reader.visitEqual`):
```java
public FileIndexResult visitEqual(FieldRef fieldRef, Object key) {
    return key == null || filter.testHash(hashFunction.hash(key)) ? REMAIN : SKIP;
}
```

**为什么默认误报率是 0.1**: 较高的 fpp 使得 Bloom Filter 占用更少空间(嵌入到 DataFileMeta 中时更紧凑)。对于大表的大量文件，10% 的误报率在整体过滤效果和存储开销间取得了平衡。用户可根据需要调低 fpp。

### 2.4 读取阶段的索引评估 (FileIndexEvaluator)

**源码路径**: `paimon-core/.../io/FileIndexEvaluator.java`

`RawFileSplitRead.createFileReader()` 在创建文件读取器前会调用 `FileIndexEvaluator.evaluate()`:

```
FileIndexEvaluator.evaluate()
  +-- 无过滤条件且无 TopN? -> 仅处理 Limit (创建 baseSelection 并截断)
  +-- createBaseSelection(file, dv) -> 全行位图 AND NOT 删除向量
  +-- 有过滤条件? -> predicate.evaluate(filter) AND baseSelection
  +-- 有 TopN? -> predicate.evaluateTopN(topN, selection)
```

**DV 与索引的协同**: `createBaseSelection` 先生成覆盖 `[0, rowCount)` 的全行位图，然后减去 DV 中标记删除的行。这确保了文件索引评估的结果不包含已删除的行。

## 3. 多维排序策略

### 3.1 TableSorter 体系架构

**源码路径**: 
- Spark: `paimon-spark/paimon-spark-common/.../spark/sort/TableSorter.java`
- Flink: `paimon-flink/paimon-flink-common/.../flink/sorter/TableSorter.java`

Paimon 的多维排序通过计算引擎实现(需要分布式排序能力)。`TableSorter` 是抽象基类，通过工厂方法根据 `OrderType` 选择排序策略:

```java
public static TableSorter getSorter(FileStoreTable table, OrderType orderType, List<String> orderColumns) {
    switch (orderType) {
        case ORDER:   return new OrderSorter(table, orderColumns);
        case ZORDER:  return new ZorderSorter(table, orderColumns);
        case HILBERT: return new HilbertSorter(table, orderColumns);
        case NONE:    return /* 无操作 */;
    }
}
```

**为什么通过计算引擎实现**: 多维排序需要读取全表数据并重新分布排序，这是一个批处理作业，需要分布式计算框架的 shuffle 能力。Paimon 在 Spark 和 Flink 的引擎集成层分别实现了排序，保持了存储层的简洁性。

### 3.2 OrderSorter: 字典序排序

**源码路径**: `spark/sort/OrderSorter.java`

```java
public Dataset<Row> sort(Dataset<Row> input) {
    Column[] sortColumns = orderColNames.stream().map(input::col).toArray(Column[]::new);
    return input.repartitionByRange(sortColumns).sortWithinPartitions(sortColumns);
}
```

**为什么先 repartitionByRange 再 sortWithinPartitions**:
1. `repartitionByRange` -- 按排序列做范围分区，保证相近值的数据落在同一分区
2. `sortWithinPartitions` -- 每个分区内局部排序

**好处**: 单列或有明确优先级的多列查询，字典序排序使得数据在存储层高度有序，min/max 统计信息的区分度最好，文件跳过率最高。

### 3.3 ZorderSorter: Z-Order 字节交错算法

**源码路径**: `spark/sort/ZorderSorter.java`

Z-Order 排序的核心步骤:
1. **将每列值转为字典序字节表示** -- 通过 `SparkZOrderUDF.sortedLexicographically()` 为每种类型调用对应的转换 UDF
2. **字节交错(interleave)** -- 将所有列的字节数组按位交错成一个 Z-value
3. **按 Z-value 范围分区并排序** -- `repartitionByRange(ZVALUE).sortWithinPartitions(ZVALUE)`
4. **丢弃 Z-value 列**

**为什么需要字节交错**: Z-Order 排序保留了多维数据的空间局部性。交错后，多维空间中相近的点在一维 Z-value 中也倾向于相近，使得同一文件中包含的数据在多个维度上都具有较小的 min/max 范围，从而提高多列联合过滤的跳过率。

### 3.4 HilbertSorter: Hilbert 曲线

**源码路径**: `spark/sort/HilbertSorter.java`

Hilbert 排序与 Z-Order 类似，但使用 Hilbert 曲线代替 Z 曲线:
```java
public Dataset<Row> sort(Dataset<Row> df) {
    Column hilbertColumn = hilbertValue(df);
    Dataset<Row> hilbertValueDF = df.withColumn(H_COLUMN, hilbertColumn);
    return hilbertValueDF.repartitionByRange(H_COLUMN).sortWithinPartitions(H_COLUMN);
}
```

**为什么提供 Hilbert 曲线选项**: Hilbert 曲线比 Z 曲线有更好的空间局部性保持性(不存在 Z 曲线中的"跳跃点")。理论和实践表明，Hilbert 排序后数据的 min/max 范围更紧凑，能获得比 Z-Order 高约 10-20% 的文件跳过率。代价是 Hilbert 曲线的计算比 Z-Order 更复杂。

### 3.5 Z-Order 字节交错算法原理 (ZOrderByteUtils)

**源码路径**: `paimon-common/.../sort/zorder/ZOrderByteUtils.java`

核心思想是将多维数据映射到一维空间，同时尽量保持空间局部性。

**Step 1: 类型到有序字节的转换**

每种类型都需要转换为字典序可比较的字节表示:

| 类型 | 转换方法 | 原理 |
|------|---------|------|
| 有符号整数 | XOR `0x8000000000000000L` | 翻转符号位，使负数在正数之前 |
| 浮点数 | XOR `((lval >> 63) \| Long.MIN_VALUE)` | IEEE 754 符号量级整数重解释，将浮点数按位重解释为 long 后翻转符号位 |
| 字符串 | UTF-8 编码 + 截断/填充到固定长度 | 保证等长比较 |

**Step 2: 位交错 (interleaveBits)**

(见 `ZOrderByteUtils.java:184-220`)

```
输入: col_A = [a7 a6 a5 a4 a3 a2 a1 a0], col_B = [b7 b6 b5 b4 b3 b2 b1 b0]
输出: [a7 b7 a6 b6 a5 b5 a4 b4 a3 b3 a2 b2 a1 b1 a0 b0]
```

算法逐列、逐位、轮询式地从源列中取出 bit，写入目标字节数组。当一列的 bit 用完后，仅从剩余列取 bit。

**为什么使用 PRIMITIVE_BUFFER_SIZE = 8**: 所有原始类型统一转为 8 字节(Long 宽度)表示，使得每列贡献相同数量的 bit 到交错结果中，保证各维度在 Z-curve 上的权重相同。

## 4. Compaction 排序优化 (Clustering)

### 4.1 ClusteringCompactManager 两阶段架构

**源码路径**: `paimon-core/.../mergetree/compact/clustering/ClusteringCompactManager.java`

ClusteringCompactManager 是为支持"按聚簇列排序"的主键表设计的 Compaction 管理器。它采用**两阶段**架构:

**Phase 1: 排序重写 (Sort & Rewrite)**
- 读取所有未排序文件(level 0)
- 按聚簇列外部排序后写入新的 level 1 文件
- 排序过程中通过 `ClusteringKeyIndex` 做去重检查(FIRST_ROW 跳过重复, DEDUPLICATE 标记 DV)

**Phase 2: 归并合并 (Merge)**
- 对 Phase 1 之前已存在的 sorted 文件，按聚簇列范围检测 overlap
- 将 overlap 的文件归为一组，相邻的小文件也合并
- 使用多路归并(min-heap)重写

**为什么需要两阶段**: Phase 1 处理增量写入的新数据(将其排序化)，Phase 2 处理已排序数据间的碎片和 overlap 问题。分开处理避免了每次 compaction 都读取全量数据。

**好处**: 增量聚簇模式下，只有新写入的数据(level 0 文件)需要完整排序，已排序的文件只在 overlap 或碎片化时才需要合并，大幅降低了写放大。

核心执行流程 (见 `ClusteringCompactManager.java:196-251`):
```java
private CompactResult compact(boolean fullCompaction) {
    // Phase 1: Sort and rewrite all unsorted (level 0) files
    List<DataFileMeta> unsortedFiles = fileLevels.unsortedFiles();
    List<DataFileMeta> existingSortedFiles = fileLevels.sortedFiles(); // 快照, 避免混入 Phase 1 新文件
    for (DataFileMeta file : unsortedFiles) {
        List<DataFileMeta> sortedFiles = fileRewriter.sortAndRewriteFile(file, ...);
        result.before().add(file);
        result.after().addAll(sortedFiles);
    }

    // Phase 2: Merge sorted files based on overlap
    List<List<DataFileMeta>> mergeGroups;
    if (fullCompaction) {
        mergeGroups = singletonList(existingSortedFiles);
    } else {
        mergeGroups = fileRewriter.pickMergeCandidates(existingSortedFiles);
    }
    // ... 执行合并
}
```

### 5.2 IntervalPartition Section 划分

**源码路径**: `paimon-core/.../mergetree/compact/IntervalPartition.java`

IntervalPartition 将一组有 key 范围重叠可能性的文件划分为 **Section**(不重叠的段)和 **SortedRun**(段内的有序运行)。

**两层分区算法**:

**外层: Section 划分** (见 `IntervalPartition.java:67-91`)

文件按 minKey 排序后，维护当前 section 的右边界 bound:
- 如果下一个文件的 minKey > bound -> 新建 section
- 否则加入当前 section, 更新 bound = max(bound, file.maxKey)

这确保了同一 section 内的文件 key 范围可能重叠，不同 section 间的 key 范围不重叠。

**内层: SortedRun 划分** (见 `IntervalPartition.java:93-125`)

使用贪心+优先队列算法，对每个 section 内的文件进一步分组:
- 维护多个 SortedRun，用优先队列按各 run 的最后文件 maxKey 排序
- 对每个文件: 如果 minKey > 最小 maxKey 的 run 的最后文件 maxKey -> 追加到该 run
- 否则 -> 创建新 run

**为什么 Section 间不重叠很重要**: Split 中包含的所有文件需要一起做 merge 排序。如果能将文件划分为不重叠的 Section，不同 Section 可以独立并行处理，提高查询并行度。

### 5.3 BinPacking 装箱算法

**源码路径**: `paimon-common/.../utils/BinPacking.java`

Paimon 使用两种装箱算法:

**packForOrdered** (有序装箱, 见 `BinPacking.java:33-58`):
- 保持输入顺序，顺序装入
- 当累积权重超过 targetWeight 时开新箱
- 用于 Split 生成(保持文件的时间/范围顺序)

**packForFixedBinNumber** (固定箱数装箱, 见 `BinPacking.java:61-81`):
- 先按权重排序，再用优先队列贪心分配到指定数量的箱
- 用于并行度固定的场景

**为什么 Split 生成使用有序装箱**: 文件的顺序反映了数据的时间或 key 范围顺序。保持顺序使得同一 Split 内的文件在 key 范围上连续，有利于 merge 效率和缓存局部性。

## 6. Lookup 点查优化

### 6.1 LookupLevels 多层查找

**源码路径**: `paimon-core/.../mergetree/LookupLevels.java`, `LookupUtils.java`

LookupLevels 实现了基于 LSM-Tree 层级结构的高效点查。

**查找策略** (见 `LookupUtils.java:35-57`):
```java
public static <T> T lookup(Levels levels, InternalRow key, int startLevel, ...) {
    for (int i = startLevel; i < levels.numberOfLevels(); i++) {
        if (i == 0) {
            result = level0Lookup(key, levels.level0());  // Level 0: 遍历所有文件
        } else {
            result = lookup(key, levels.runOfLevel(i));   // Level 1+: 二分查找
        }
        if (result != null) break;  // 找到即返回(新数据在上层)
    }
}
```

**Level 0 查找** (见 `LookupUtils.java:59-77`):
- Level 0 的文件无序(TreeSet 按 sequence 排序)
- 需要**遍历所有文件**，检查 key 是否在 [minKey, maxKey] 范围内
- 找到第一个匹配即返回(TreeSet 按 sequence 降序，最新的先匹配)

**Level 1+ 查找** (见 `LookupUtils.java:79-119`):
- 每个 Level 是一个 SortedRun(文件间无 overlap, 按 key 有序)
- 使用**二分查找**定位目标文件: 找到 maxKey >= target 的最小文件
- 只需查询一个文件

**为什么从上层往下查**: LSM-Tree 中，上层数据更新，下层数据更旧。对同一 key，上层的值是最新的。从上往下查找，第一个命中的就是最新值，无需继续。

### 6.2 LookupFile 缓存机制

**源码路径**: `paimon-core/.../mergetree/LookupFile.java`, `LookupLevels.java:182-207`

每个数据文件会被转换为本地 Lookup 文件(SST 格式)，存储在本地磁盘上。

**LookupFile 缓存** (见 `LookupFile.java:123-132`):
```java
public static Cache<String, LookupFile> createCache(Duration fileRetention, MemorySize maxDiskSize) {
    return Caffeine.newBuilder()
        .expireAfterAccess(fileRetention)  // 访问超时淘汰
        .maximumWeight(maxDiskSize.getKibiBytes())  // 磁盘空间限制(KiB)
        .weigher(LookupFile::fileWeigh)    // 权重 = 文件大小(KiB)
        .removalListener(LookupFile::removalCallback)  // 淘汰时删除本地文件
        .build();
}
```

**创建 Lookup 文件的过程** (见 `LookupLevels.java:239-272`):
1. 创建本地临时文件
2. 尝试下载远程预构建的 Lookup 文件(如果存在)
3. 若无远程文件，从数据文件逐条读取 KV，写入 LookupStoreWriter
4. 写入时同时构建 Bloom Filter

**为什么缓存 Lookup 文件到本地**: 远程文件系统(S3/HDFS)的随机读取延迟高，不适合点查场景。将文件转换为本地 SST 格式后:
- 利用本地 SSD 的随机读取能力
- 附带 Bloom Filter 快速判断 key 是否存在
- Caffeine 缓存管理 LRU 淘汰和磁盘空间

### 6.3 Bloom Filter 加速

Lookup 场景下的 Bloom Filter 与文件索引的 Bloom Filter 不同:
- **文件索引 Bloom Filter**: 在 Scan 阶段判断文件是否可能包含匹配数据
- **Lookup Bloom Filter**: 在本地 LookupFile 中，通过 `LookupStoreWriter` 构建，用于在读取 SST 文件前快速排除不存在的 key

```java
// LookupLevels.createSstFileFromDataFile
LookupStoreWriter kvWriter = lookupStoreFactory.createWriter(localFile, bfGenerator.apply(file.rowCount()));
```

`bfGenerator` 根据文件行数动态生成 Bloom Filter Builder，行数越多，Bloom Filter 越大，误报率越低。

### 6.4 远程 Lookup 文件下载

**源码路径**: `LookupLevels.java:209-233`, `274-307`

Paimon 支持将 Lookup 文件预先上传到远程存储，新启动的节点可以直接下载而不需要从数据文件重建:

```java
Optional<RemoteSstFile> remoteSstFile = remoteSst(file);
// 远程文件命名: {dataFileName}.{length}.{processorId}.{serVersion}.lookup
```

**为什么需要远程 Lookup 文件**: 在 Flink 任务恢复或扩容时，新 TM 节点需要重建 Lookup 文件。如果数据文件很大，逐条读取并重建 SST 文件非常耗时。预构建并上传的远程 Lookup 文件可以直接下载使用，大幅缩短恢复时间。

## 7. SortMergeReader 优化

### 7.1 LoserTree vs MinHeap 对比

**源码路径**: `paimon-core/.../mergetree/compact/SortMergeReader.java`

Paimon 提供两种 SortMerge 引擎，通过 `sort-engine` 配置选择:

```java
static <T> SortMergeReader<T> createSortMergeReader(..., SortEngine sortEngine) {
    switch (sortEngine) {
        case MIN_HEAP:   return new SortMergeReaderWithMinHeap<>(...);
        case LOSER_TREE: return new SortMergeReaderWithLoserTree<>(...);
    }
}
```

| 维度 | MinHeap | LoserTree |
|------|---------|-----------|
| 时间复杂度 | O(log N) per element | O(log N) per element, 但常数更小 |
| Batch 模式 | 多 batch(每次 readBatch() 返回一个新的 RecordIterator) | 单 batch(readBatch() 仅返回一个 RecordIterator，后续通过 next() 迭代) |
| 内存效率 | 每个 reader 的 batch 独立消费，可能产生多个中间 batch | 全局管理，所有 reader 的数据在一个 batch 中处理 |
| 相同 key 处理 | 通过 PriorityQueue poll+collect 逐个处理 | 通过状态机快速路径批量处理 |
| 适用场景 | 少量 reader、大 batch | 大量 reader、需要最优合并性能 |

**MinHeap 实现特点** (见 `SortMergeReaderWithMinHeap.java`):
- 使用 `PriorityQueue<Element>` 维护所有 reader 的当前记录
- 每次 readBatch() 调用时，从所有 reader 读取第一条记录放入堆
- 通过 SortMergeIterator 逐条 poll 最小元素，当某个 reader 的 batch 耗尽时，重新读取该 reader 的下一个 batch
- 可能产生多个 RecordIterator(多 batch)

**LoserTree 实现特点** (见 `SortMergeReaderWithLoserTree.java`):
- readBatch() 仅返回一个 RecordIterator，后续通过 next() 逐条获取记录
- 通过 `loserTree.adjustForNextLoop()` + `popWinner()` 获取下一条记录
- 同 key 合并通过快速路径完成，无需重复比较

### 7.2 LoserTree 核心状态机

**源码路径**: `paimon-core/.../mergetree/compact/LoserTree.java`

LoserTree 相比标准败者树的关键优化是**同 key 快速路径**。在 LSM-Tree 中，多个 reader 可能包含相同 key 的不同版本。LoserTree 通过 6 种状态高效管理同 key 的合并:

```
State 枚举 (见 LoserTree.java:338-356):
  WINNER_WITH_NEW_KEY   -- 赢家, 新 key
  WINNER_WITH_SAME_KEY  -- 赢家, 与全局赢家相同 key
  WINNER_POPPED         -- 赢家已被弹出, 等待推进
  LOSER_WITH_NEW_KEY    -- 败者, 新 key
  LOSER_WITH_SAME_KEY   -- 败者, 与全局赢家相同 key
  LOSER_POPPED          -- 败者已被弹出(用于 adjustForNextLoop)
```

**popWinner() + adjust() 的同 key 快速路径** (见 `LoserTree.java:105-115, 143-160`):

当 winner 被 pop 后(状态变为 WINNER_POPPED)，如果 `firstSameKeyIndex >= 0`(树中有相同 key 的节点)，直接交换位置，无需从底部逐级比较。这使得同 key 的 N 个版本合并只需 O(N) 次操作，而非 O(N log N)。

**为什么 LoserTree 更适合 Compaction**: Compaction 时通常有多个 SortedRun 参与合并(可能 5-10 个)，且同 key 出现在多个 run 中的概率较高。LoserTree 的同 key 快速路径和更优的缓存友好性(数组而非堆节点对象)在此场景下性能更佳。

### 7.3 Spill 机制与外部排序

**源码路径**: `paimon-common/.../sort/BinaryExternalSortBuffer.java`, `ClusteringFileRewriter.java`

ClusteringFileRewriter 在 Phase 1 排序时使用 `BinaryExternalSortBuffer`，这是一个支持磁盘 Spill 的外部排序缓冲区:

1. **内存阶段**: 数据写入内存中的 `BinaryInMemorySortBuffer`
2. **Spill 阶段**: 当内存不足时，将排序后的数据 spill 到磁盘通道
3. **Merge 阶段**: 读取时使用 `BinaryExternalMerger` 做多路归并

Phase 2 的 `mergeAndRewriteFiles` 有独立的 Spill 策略 (见 `ClusteringFileRewriter.java:234-244`):
- 当输入文件数 > `spillThreshold` 时，按文件大小排序
- 较小的文件先 spill 到行式压缩临时文件(`ChannelWriterOutputView`)
- 保留较大的文件直接读取，减少 Spill I/O

**为什么 Spill 较小的文件而非较大的**: 小文件的 Spill 开销低(读写字节少)，而大文件保持原始列式格式读取效率更高(列式格式的向量化读取优势保留)。

## 8. DV 对查询的影响

### 8.1 DV 在 Scan 阶段的作用

DV(Deletion Vectors) 通过位图标记数据文件中已删除的行，避免了传统 merge-on-read 的全量合并。

在 Scan 阶段，DV 的主要影响:

1. **rawConvertible 判定** -- DV 启用时，`alwaysRawConvertible()` 返回 true，因为不需要 merge
2. **Limit 下推受限** -- `KeyValueFileStoreScan.limitPushdownEnabled()` 在 DV 启用时返回 false:
   ```java
   public boolean limitPushdownEnabled() {
       // 前置条件: limit 必须有效
       if (limit == null || limit <= 0) {
           return false;
       }
       
       // 禁用条件:
       // 1. PARTIAL_UPDATE 引擎 - 部分字段更新，rowCount 不等于最终记录数
       // 2. AGGREGATE 引擎 - 聚合后记录数可能减少
       // 3. deletionVectorsEnabled - DV 标记的删除行不计入有效行数
       return mergeEngine != PARTIAL_UPDATE 
               && mergeEngine != AGGREGATE 
               && !deletionVectorsEnabled;
   }
   ```
   **为什么 DV 启用时禁止 Limit 下推**: 文件的 rowCount 不等于有效行数(需要减去 DV 标记的删除行)，因此不能基于 rowCount 做精确的 Limit 截断。

### 8.2 DV 在 Read 阶段的作用

**源码路径**: `paimon-core/.../deletionvectors/ApplyDeletionVectorReader.java`

```java
public class ApplyDeletionVectorReader implements FileRecordReader<InternalRow> {
    public FileRecordIterator<InternalRow> readBatch() {
        FileRecordIterator<InternalRow> batch = reader.readBatch();
        return new ApplyDeletionFileRecordIterator(batch, deletionVector);
    }
}
```

`ApplyDeletionFileRecordIterator` 在迭代过程中，通过 `deletionVector.isDeleted(returnedPosition)` 检查每条记录的位置是否在 DV 位图中被标记，是则跳过。

**为什么 DV 基于行位置而非 key**: 基于位置的方式无需知道数据的 key 结构，适用于所有表类型(主键表和追加表)。读取时文件格式层提供行位置，DV 查询只需要 O(1) 的位图检查。

### 8.3 DV 与文件索引的协同

**源码路径**: `io/FileIndexEvaluator.java:100-108`

DV 和文件索引在 Read 阶段协同工作:

```java
private static BitmapIndexResult createBaseSelection(DataFileMeta file, DeletionVector dv) {
    BitmapIndexResult selection = new BitmapIndexResult(
        () -> RoaringBitmap32.bitmapOfRange(0, file.rowCount()));  // 全行位图
    if (dv instanceof BitmapDeletionVector) {
        selection = selection.andNot(((BitmapDeletionVector) dv).get());  // 减去删除行
    }
    return selection;
}
```

流程:
1. 创建全行位图 [0, rowCount)
2. AND NOT 删除向量 -> 有效行位图
3. 文件索引评估结果 AND 有效行位图 -> 最终需要读取的行

**好处**: DV 和 Bitmap 索引共享 RoaringBitmap32 格式，可以通过高效的位运算组合，无需在行级别逐条判断。

## 9. 统计信息收集

### 9.1 Statistics 与 ColStats 模型

**源码路径**: `paimon-core/.../stats/Statistics.java`, `ColStats.java`

Paimon 的全局统计信息(ANALYZE 结果)通过 `Statistics` 类表示:

```java
public class Statistics {
    long snapshotId;          // 对应的快照 ID
    long schemaId;            // 对应的 Schema ID
    Long mergedRecordCount;   // merge 后的总记录数
    Long mergedRecordSize;    // merge 后的总数据大小(字节)
    Map<String, ColStats<?>> colStats;  // 列级统计信息
}
```

`ColStats` (见 `ColStats.java`) 包含:

| 字段 | 类型 | 说明 |
|------|------|------|
| `colId` | int | 列 ID |
| `distinctCount` | Long | 不同值数量 (NDV) |
| `min` / `max` | Comparable<T> | 最小/最大值 |
| `nullCount` | Long | null 值数量 |
| `avgLen` | Long | 平均列长度 |
| `maxLen` | Long | 最大列长度 |

**为什么需要全局统计信息**: 文件级 SimpleStats 已经支持查询过滤，但查询优化器(Flink/Spark)需要全局的 NDV、数据分布等信息来选择最优的 join 策略、聚合策略和并行度。ANALYZE 操作收集这些全局统计并持久化。

### 9.2 StatsFileHandler 读写机制

**源码路径**: `paimon-core/.../stats/StatsFileHandler.java`

```java
public class StatsFileHandler {
    // 写入: 先序列化字段到字符串, 再写入 JSON 文件
    public String writeStats(Statistics stats) {
        stats.serializeFieldsToString(schemaManager.schema(stats.schemaId()));
        return statsFile.write(stats);
    }
    
    // 读取: 读 JSON 文件, 再反序列化字段

        String file = snapshot.statistics();
        return file == null ? Optional.empty() : Optional.of(readStats(file));
    }
}
```

统计信息文件与 Snapshot 关联: `Snapshot.statistics()` 指向对应的统计文件名。

### 9.3 ANALYZE 操作流程

ANALYZE 操作通过 Flink/Spark 的 Procedure 或 SQL 触发:
1. 扫描指定快照的所有 Manifest 条目
2. 对每个数据文件读取样本数据或全量数据
3. 计算 NDV(使用 HyperLogLog 等近似算法)、min/max、nullCount 等
4. 聚合为 `Statistics` 对象
5. 通过 `StatsFileHandler.writeStats()` 写入文件
6. 在下一次 Commit 时将统计文件名写入 Snapshot

## 10. 查询性能调优建议

### 10.1 与 Iceberg 过滤机制对比

| 维度 | Paimon | Iceberg |
|------|--------|---------|
| **Manifest 过滤** | partitionStats + bucket/level 范围 | partition summary (min/max per partition field) |
| **文件统计** | SimpleStats (min/max/nullCounts, BinaryRow 格式) | Column-level metrics (min/max/null_count/nan_count, Avro 格式) |
| **Schema Evolution** | SimpleStatsEvolution 动态映射 | 基于 field-id 的静态映射 |
| **文件索引** | Bloom/Bitmap/BSI/Range Bitmap (嵌入式+独立) | Bloom Filter (独立 Puffin 文件) |
| **多维排序** | ORDER/ZORDER/HILBERT (Spark/Flink 实现) | ZORDER (Spark 实现, 源自 Iceberg) |
| **Split 优化** | IntervalPartition + BinPacking + rawConvertible | FileScanTask 基于大小拆分 |
| **Lookup 点查** | LookupLevels + 本地 SST 缓存 + 远程下载 | 无原生支持(需外部 KV 层) |

**Paimon 的优势**:
- 更丰富的文件索引类型(Bitmap/BSI/Range Bitmap 提供行级过滤)
- 原生的 Lookup 点查优化(适合流式更新+实时查询场景)
- Clustering Compaction 在存储层实现增量排序优化
- DV + 文件索引的位图级协同
- Manifest 级别的 minBucket/maxBucket/minLevel/maxLevel 可以在不读取 Manifest 内容的情况下淘汰 Manifest 文件

**Iceberg 的优势**:
- NaN 计数(对浮点数据更精确)
- 更成熟的 Z-Order 实现和社区生态
- Manifest caching 和 Manifest rewrite 优化
- 更灵活的文件格式支持

### 10.2 关键配置参数

| 参数 | 默认值 | 说明 | 调优建议 |
|------|-------|------|---------|
| `file-index.bloom-filter.columns` | - | 启用 Bloom Filter 的列 | 高基数列的等值查询 |
| `file-index.bitmap.columns` | - | 启用 Bitmap 索引的列 | 低基数列的等值/IN 查询 |
| `file-index.bsi.columns` | - | 启用 BSI 索引的列 | 数值列的范围查询 |
| `file-index.read-enabled` | true | 是否在 Scan/Read 时使用文件索引 | 建议保持开启 |
| `sort-engine` | LOSER_TREE | 排序合并引擎 | 大量文件 compaction 使用 LOSER_TREE |
| `source.split.target-size` | 128MB | 目标 Split 大小 | 根据集群并行度调整 |
| `lookup-cache-file-retention` | 1h | Lookup 文件缓存保留时间 | 频繁查询时增大 |
| `lookup-cache-max-disk-size` | 无限制 | Lookup 缓存磁盘限制 | 根据本地磁盘大小设置 |
| `clustering-columns` | - | 聚簇列 | 多维查询的常用过滤列 |

### 10.3 最佳实践

**1. 分区设计先行**
- 分区裁剪是第一层、成本最低的过滤
- 以查询最频繁的低基数维度(如日期)做分区

**2. 为高频查询列配置文件索引**
- 等值查询: Bloom Filter (高基数) 或 Bitmap (低基数)
- 范围查询: BSI 或 Range Bitmap
- 组合使用: 同一列可以同时配置多种索引，FileIndexPredicate 会自动组合结果

**3. 多维排序提高文件跳过率**
- 单列范围查询: 使用 ORDER
- 两列等权查询: 使用 ZORDER
- 三列及以上: 使用 HILBERT (更好的空间局部性)

**4. 主键表启用 DV 获得 rawConvertible**
- 启用 DV 后，所有 Split 可以跳过 merge 直接读取
- 显著降低查询延迟，代价是额外的 DV 存储

**5. Clustering Compaction 优化多维查询**
- 配置 `clustering-columns` 后，Compaction 会按指定列排序数据
- 适合写入时数据无序但查询有多维过滤需求的场景
- 增量模式下写放大较低

**6. Lookup 点查场景优化**
- 确保本地磁盘足够存放 Lookup 缓存
- 启用远程 Lookup 文件加速任务恢复
- Bloom Filter 的行数参数影响 Lookup 缓存命中率

**7. Split 大小与并行度平衡**
- `scan.split-target-size` 过大 -> 并行度不够
- 过小 -> 太多 Split 导致调度开销
- 建议: 总数据量 / (并行度 * 2~4)
