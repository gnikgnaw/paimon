## 1. 查询规划与多层文件过滤

### 解决什么问题

**核心业务问题**: 数据湖场景下，一张表可能包含数万甚至数十万个数据文件，如何在查询时快速定位到真正需要读取的文件，避免全表扫描？

**没有这个设计的后果**:
- 每次查询都需要读取所有 Manifest 文件，反序列化所有 ManifestEntry，检查所有文件的统计信息
- 即使查询条件只匹配少量数据，也要扫描大量无关文件的元数据
- 查询延迟从秒级变成分钟级，无法支持交互式查询

**实际场景**:
```sql
-- 场景1: 分区裁剪
SELECT * FROM orders WHERE dt = '2024-01-01' AND status = 'PAID';
-- 如果表有1000个分区，没有多层过滤需要检查所有分区的文件

-- 场景2: 主键范围查询
SELECT * FROM user_profile WHERE user_id BETWEEN 10000 AND 20000;
-- 如果表有10000个数据文件，没有统计信息过滤需要读取所有文件的元数据
```

### 有什么坑

**误区陷阱**:
1. **误以为 Manifest 级过滤已经足够** - 实际上 Manifest 只能做粗粒度过滤，文件级统计信息才能精确裁剪
2. **忽略 Schema Evolution 的影响** - 旧文件的统计信息可能与当前 Schema 不匹配，导致过滤失效或误判
3. **过度依赖文件索引** - 文件索引有额外的存储和读取成本，不是所有列都适合建索引

**错误配置**:
```properties
# 错误1: 禁用统计信息过滤
read.stats-mode=none  # 导致无法利用 min/max 裁剪文件

# 错误2: 在高基数列上使用 Bitmap 索引
file-index.bitmap.columns=user_id  # user_id 有百万级不同值，Bitmap 会爆炸

# 错误3: 主键表强制启用 value 过滤
scan.value-filter.enabled=true  # 可能导致 merge 语义错误
```

**生产环境注意事项**:
1. **Manifest 文件数量控制** - 过多的 Manifest 文件会导致第一层过滤本身成为瓶颈，需要定期做 Manifest Compaction
2. **统计信息精度** - SimpleStats 的 min/max 是文件级的，如果文件内数据分布不均匀，过滤效果会下降
3. **内存压力** - 大量文件的统计信息会占用内存，需要合理设置 `scan.plan-parallelism` 控制并行度

**性能陷阱**:
```java
// 陷阱: 在 level 0 文件很多时，wholeBucketFilter 会退化
// level 0 文件间有 overlap，无法逐文件过滤，只能整桶判断
// 解决: 及时触发 Compaction，减少 level 0 文件数量
```

### 核心概念解释

**术语定义**:

1. **Manifest 文件** - 存储一批 ManifestEntry 的文件，每个 Entry 描述一个数据文件的元数据
2. **ManifestFileMeta** - Manifest 文件的元数据，包含该 Manifest 内所有 Entry 的聚合统计信息
3. **SimpleStats** - 列级统计信息，包含 min/max/nullCounts，以 BinaryRow 格式存储
4. **谓词下推 (Predicate Pushdown)** - 将查询条件尽早应用到数据读取路径，减少数据传输
5. **rawConvertible** - 标记一个 Split 是否可以跳过 merge 直接读取原始文件

**概念关系**:
```
Snapshot
  └─ ManifestList (base + delta + changelog)
       └─ ManifestFileMeta (聚合统计)
            └─ Manifest File
                 └─ ManifestEntry (单文件元数据)
                      └─ DataFileMeta (keyStats + valueStats)
                           └─ Data File (实际数据)
```

**与其他系统对比**:

| 系统 | 过滤层级 | 统计信息格式 | Schema Evolution 处理 |
|------|---------|-------------|---------------------|
| **Paimon** | 3层 (Manifest/Entry/File) | BinaryRow (零拷贝) | SimpleStatsEvolution 动态映射 |
| **Iceberg** | 2层 (Manifest/File) | Avro (需反序列化) | 基于 field-id 静态映射 |
| **Delta Lake** | 2层 (AddFile/File) | JSON (需解析) | 基于列名映射 |
| **Hudi** | 2层 (FileSlice/File) | Parquet Footer | 基于列名映射 |

**Paimon 的独特之处**:
- **Manifest 级聚合统计** - 可以在不读取 Manifest 内容的情况下跳过整个 Manifest
- **BinaryRow 格式** - 统计信息以二进制格式存储，比较时无需反序列化
- **Bucket/Level 范围过滤** - 利用 LSM-Tree 的层级结构做额外裁剪

### 设计理念

**为什么这样设计**:

1. **漏斗式多层过滤** - 每一层的过滤成本和精度不同，从粗到细逐层裁剪：
   - Manifest 级: 成本最低 (只读 ManifestList)，精度最粗 (聚合统计)
   - Entry 级: 成本中等 (读 Manifest 文件，但在反序列化前过滤)，精度中等 (单文件范围)
   - File 级: 成本最高 (完整反序列化 DataFileMeta)，精度最高 (列级统计 + 文件索引)

2. **BinaryRow 存储统计信息** - 避免 Java 对象开销和反序列化成本：
   ```java
   // 传统方式: 每个文件的统计信息是 Object[]
   Object[] minValues = new Object[100];  // 100列，每个对象有16字节头
   // 内存开销: 100 * (16 + 平均值大小) * 文件数
   
   // Paimon 方式: 所有列的统计信息在一个连续字节数组中
   BinaryRow minValues = ...;  // 一个对象，内部是 byte[]
   // 内存开销: (平均值大小总和 + 少量元数据) * 文件数
   ```

3. **Schema Evolution 的动态映射** - 支持列的增删改，不强制重写旧数据：
   - 新增列: 旧文件的该列统计信息视为全 null
   - 删除列: 忽略旧文件中该列的统计信息
   - 类型变更: 通过 castFieldGetters 做类型转换

**权衡取舍**:

| 设计选择 | 优势 | 劣势 | 取舍原因 |
|---------|------|------|---------|
| 三层过滤 | 最大化裁剪效果 | 代码复杂度高 | 数据湖场景文件数量巨大，多层过滤收益显著 |
| BinaryRow 格式 | 内存效率高，比较快 | 不易调试，序列化复杂 | 生产环境内存和性能优先 |
| Manifest 聚合统计 | 可跳过整个 Manifest | 需要额外存储和维护 | Manifest 文件可能包含数千 Entry，跳过收益大 |
| 主键表条件性 value 过滤 | 保证 merge 语义正确 | 某些场景过滤效果受限 | 正确性优先于性能 |

**架构演进**:

1. **早期版本** - 只有文件级统计信息过滤，每次查询需要读取所有 Manifest 文件
2. **引入 Manifest 级过滤** - 在 ManifestFileMeta 中添加 partitionStats，可以跳过整个 Manifest
3. **引入 Entry 级行过滤** - 在反序列化前用轻量级 getter 提取字段，减少对象创建
4. **引入 wholeBucketFilter** - 解决主键表 overlap 场景下的过滤问题
5. **引入 DV 支持** - 启用 DV 后所有 Split 都是 rawConvertible，简化读取路径

**业界对比**:

- **Iceberg** - 只有两层过滤 (Manifest/File)，但有更成熟的 Manifest Caching 机制
- **Delta Lake** - 使用 JSON 格式的 AddFile，过滤性能较低，但实现简单
- **Hudi** - 依赖 Parquet Footer 的统计信息，无法在 Scan 阶段做精细过滤

**Paimon 的创新点**:
- Manifest 级的 bucket/level 范围过滤 - 利用 LSM-Tree 结构特性
- Entry 级的原始行过滤 - 在反序列化前过滤，减少对象创建
- wholeBucketFilter - 解决主键表 merge 语义下的过滤难题

### 1.1 FileStoreScan 整体架构

Paimon 的查询规划核心入口是 `FileStoreScan.plan()`，它采用**多层漏斗式过滤**架构，从粗粒度到细粒度逐层裁剪无关文件，最大限度减少 I/O 操作。

**为什么这么做**: 数据湖中一个表可能有数万个数据文件，如果全量扫描 Manifest 并读取所有文件，查询延迟会非常高。通过多层过滤，可以在元数据层面就剔除大量不相关文件，只读取真正需要的数据。

**好处**: 每一层过滤的代价不同 -- Manifest 级过滤只需要比较聚合统计信息(几十字节)，Entry 行过滤利用反序列化前的原始二进制数据，文件级过滤才需要完整的统计信息。这种分层设计使得每一层的计算/IO 成本逐级递增，但数据量逐级递减。

```
源码路径:
- paimon-core/.../operation/FileStoreScan.java (接口定义)
- paimon-core/.../operation/AbstractFileStoreScan.java (核心实现)
- paimon-core/.../operation/KeyValueFileStoreScan.java (主键表)
- paimon-core/.../operation/AppendOnlyFileStoreScan.java (追加表)
```

继承体系:

```
FileStoreScan (接口)
    +-- AbstractFileStoreScan (抽象类, 通用过滤逻辑)
            +-- KeyValueFileStoreScan (主键表专用, 分离 key/value 过滤)
            +-- AppendOnlyFileStoreScan (追加表专用, 单一过滤器)
```

`plan()` 方法的核心执行流程 (见 `AbstractFileStoreScan.java:297-349`):

```
1. readManifests()         -> ManifestsReader 做 Manifest 级粗过滤
2. readManifestEntries()   -> 读 Manifest 文件内的条目, 逐条做行过滤+统计过滤
3. postFilterManifestEntries() -> 可选的后过滤(整桶过滤、Limit 下推)
```

### 1.2 第一层: Manifest 级过滤 (ManifestsReader)

**源码路径**: `paimon-core/.../operation/ManifestsReader.java:166-207`

`ManifestsReader.filterManifestFileMeta()` 是第一层过滤，在读取 Manifest 文件内容之前，仅凭 `ManifestFileMeta` 的聚合元数据判断整个 Manifest 文件是否可以跳过。

**ManifestFileMeta 包含的过滤维度** (见 `manifest/ManifestFileMeta.java:44-61`):

| 字段 | 类型 | 过滤用途 |
|------|------|---------|
| `partitionStats` | `SimpleStats` | 分区范围裁剪 -- 每个 Manifest 记录其内所有条目的分区统计信息(min/max/nullCounts) |
| `minBucket` / `maxBucket` | `Integer` | Bucket 范围裁剪 -- 如果指定查询 bucket=5，但 Manifest 的 bucket 范围是 [10, 20]，则跳过 |
| `minLevel` / `maxLevel` | `Integer` | Level 范围裁剪 -- Compaction 选取特定 Level 文件时可跳过不相关 Manifest |
| `minRowId` / `maxRowId` | `Long` | Row Range 裁剪 -- 支持基于行号范围的过滤 |
| `numAddedFiles` / `numDeletedFiles` | `long` | 辅助信息 -- 仅含删除条目的 Manifest 在 ALL 模式下可直接过滤 |

**为什么在 Manifest 层面做过滤**: Manifest 文件可能包含数千个数据文件条目，每个条目需要反序列化。如果一个 Manifest 的分区范围完全不匹配查询谓词，跳过整个 Manifest 可避免读取和反序列化数千条 Entry。

过滤逻辑核心代码 (简化自 `ManifestsReader.java:166-207`):

```java
private boolean filterManifestFileMeta(ManifestFileMeta manifest) {
    // 1. Bucket 范围过滤
    if (specifiedBucket != null && bucket 不在 [minBucket, maxBucket] 范围) return false;
    
    // 2. Level 范围过滤
    if (specifiedLevel != null && level 不在 [minLevel, maxLevel] 范围) return false;
    if (levelMinMaxFilter != null && !levelMinMaxFilter.test(minLevel, maxLevel)) return false;
    
    // 3. 分区谓词过滤 -- 利用 partitionStats 的 min/max/nullCounts
    if (partitionFilter != null) {
        SimpleStats stats = manifest.partitionStats();
        long totalEntries = manifest.numAddedFiles() + manifest.numDeletedFiles();
        if (!partitionFilter.test(totalEntries, stats.minValues(), stats.maxValues(), stats.nullCounts()))
            return false;
    }
    
    // 4. Row Range 过滤
    if (rowRangeIndex != null && !rowRangeIndex.intersects(minRowId, maxRowId)) return false;
    
    return true;
}
```

### 1.3 第二层: ManifestEntry 级行过滤

**源码路径**: `AbstractFileStoreScan.java:526-567`

通过第一层过滤后的 Manifest 文件需要被读取，此时在反序列化每一行 ManifestEntry 时，Paimon 使用 `createEntryRowFilter()` 进行**原始行级过滤**。

**为什么这么做**: ManifestEntry 的完整反序列化包括 DataFileMeta 中的 keyStats、valueStats 等复杂结构，代价较高。通过在 `InternalRow` 层面(反序列化前)用轻量级 getter 函数提取分区、bucket、level 等简单字段进行过滤，可以避免对不需要的条目做完整反序列化。

**好处**: 减少了大量不必要的对象创建和内存分配。该过滤器直接操作 `InternalRow` 的二进制格式，使用 `ManifestEntrySerializer.partitionGetter()` 等静态方法提取字段，是零拷贝的高效操作。

核心过滤条件:
1. **分区过滤** -- `partitionFilter.test(partition)`
2. **Bucket 过滤** -- `bucketFilter.test(partition, bucket, totalBucket)` (支持 partition-aware 的 bucket 过滤)
3. **Level 过滤** -- 精确 level 匹配或范围过滤
4. **文件名过滤** -- 用于增量读取等场景

### 1.4 第三层: 文件级统计信息过滤 (filterByStats)

通过行过滤后的条目会被完整反序列化为 `ManifestEntry` 对象，此时可以访问 `DataFileMeta` 中的完整统计信息进行精细过滤。

**KeyValueFileStoreScan.filterByStats()** (见 `KeyValueFileStoreScan.java:141-165`):

```java
protected boolean filterByStats(ManifestEntry entry) {
    // 1. Value 过滤(条件性启用)
    if (isValueFilterEnabled() && !filterByValueFilter(entry)) return false;
    
    // 2. Key 统计信息过滤 -- 使用 keyStats (min/max/nullCounts)
    // 通过缓存避免重复计算 Schema 映射
    Predicate notEvolvedFilter = notEvolvedKeyFilterMapping.computeIfAbsent(
        entry.file().schemaId(),
        id -> fieldKeyStatsConverters.filterUnsafeFilter(
            entry.file().schemaId(), keyFilter, true));
    if (notEvolvedFilter == null) return true;
    
    DataFileMeta file = entry.file();
    SimpleStatsEvolution.Result stats = fieldKeyStatsConverters
        .getOrCreate(file.schemaId())
        .evolution(file.keyStats(), file.rowCount(), null);
    return notEvolvedFilter.test(file.rowCount(), stats.minValues(), stats.maxValues(), stats.nullCounts());
}
```

**为什么主键表使用 keyStats 做主过滤**: 主键表的数据按主键排序存储，keyStats 能精确反映每个文件的主键范围。当查询条件包含主键谓词时，可以高效地通过 min/max 范围判断跳过文件。

**AppendOnlyFileStoreScan 的过滤策略** (见 `AppendOnlyFileStoreScan.java:121-158`):

追加表没有主键，所有列都使用 valueStats 进行过滤。过滤流程分为两步:
1. **统计信息过滤** -- 使用 valueStats 的 min/max/nullCounts 进行初步过滤
2. **文件索引过滤** -- 如果启用了 fileIndexReadEnabled，进一步使用嵌入的文件索引(如 Bloom Filter)进行精细过滤

这与主键表的差异在于:
- 主键表: 先用 keyStats 过滤，再条件性地用 valueStats 和文件索引
- 追加表: 直接用 valueStats 过滤，然后可选地用文件索引

**AppendOnlyFileStoreScan.filterByStats()** (见 `AppendOnlyFileStoreScan.java:121-158`):

```java
protected boolean filterByStats(ManifestEntry entry) {
    // 1. 获取 Schema 映射后的过滤器
    Predicate notEvolvedFilter = notEvolvedFilterMapping.computeIfAbsent(
        entry.file().schemaId(),
        id -> simpleStatsEvolutions.filterUnsafeFilter(
            entry.file().schemaId(), inputFilter, true));
    if (notEvolvedFilter == null) return true;
    
    // 2. valueStats min/max 过滤
    SimpleStatsEvolution evolution = simpleStatsEvolutions.getOrCreate(entry.file().schemaId());
    SimpleStatsEvolution.Result stats = evolution.evolution(
        entry.file().valueStats(), entry.file().rowCount(), entry.file().valueStatsCols());
    boolean result = notEvolvedFilter.test(rowCount, stats.minValues(), stats.maxValues(), stats.nullCounts());
    if (!result) return false;
    
    // 3. 文件索引过滤(如 Bloom Filter) -- 仅在启用时使用
    if (fileIndexReadEnabled) {
        return testFileIndex(entry.file().embeddedIndex(), entry);
    }
    return true;
}
```

**DataFileMeta 中的统计信息字段** (见 `io/DataFileMeta.java:62-91`):

| 字段 | 用途 |
|------|------|
| `minKey` / `maxKey` | 主键范围边界(BinaryRow)，用于 IntervalPartition 和二分查找 |
| `keyStats` | 主键列的 SimpleStats(min/max/nullCounts)，用于主键谓词过滤 |
| `valueStats` | 值列的 SimpleStats，用于非主键列谓词过滤 |
| `valueStatsCols` | 稠密存储模式下标识哪些列有统计信息 |
| `embeddedIndex` | 嵌入的文件索引(Bloom Filter 等)字节 |
| `rowCount` | 行数，用于统计信息计算和 Limit 下推 |
| `deleteRowCount` | 删除行数，用于判断是否可以做 rawConvertible |

### 1.5 SimpleStats 结构详解

**源码路径**: `paimon-core/.../stats/SimpleStats.java`

`SimpleStats` 是 Paimon 中最核心的统计信息结构，用于文件级和 Manifest 级的谓词过滤。

```java
public class SimpleStats {
    private final BinaryRow minValues;    // 每列最小值, 以 BinaryRow 形式存储
    private final BinaryRow maxValues;    // 每列最大值
    private final BinaryArray nullCounts; // 每列 null 计数(Long 数组)
}
```

**为什么采用 BinaryRow 存储**: 与使用 Java 对象数组(如 `Object[]`)相比，`BinaryRow` 将所有列值紧凑地存储在连续的字节数组中。这带来两个好处:
1. **极低的内存开销** -- 避免了 Java 对象头和引用的额外开销，对于数万个文件的场景，内存节省显著。
2. **零反序列化比较** -- 谓词评估时可以直接在二进制数据上比较，无需将每列值反序列化为 Java 对象。

### 1.6 Schema Evolution 对统计信息的影响

**源码路径**: `paimon-core/.../stats/SimpleStatsEvolution.java`, `SimpleStatsEvolutions.java`

数据文件可能是在旧 Schema 下写入的，而查询谓词基于当前 Schema。`SimpleStatsEvolution` 负责统计信息的 Schema 映射:

1. **indexMapping** -- 旧 Schema 字段到新 Schema 字段的索引映射。新增字段映射值为 -1，对应的 nullCount 设为 rowCount(意味着旧数据中该字段全为 null)。
2. **castFieldGetters** -- 类型变更时的类型转换(如 INT -> BIGINT)。
3. **denseFields 优化** -- 当 valueStatsCols 非 null 时，说明统计信息是稠密存储的(仅包含部分列)，需要先做稠密到稀疏的映射。

**filterUnsafeFilter** 方法 (见 `SimpleStatsEvolutions.java`): 将当前 Schema 下的 Predicate 逆向映射("devolve")到旧 Schema 下，使之可以直接应用于旧文件的统计信息。`keepNewFieldFilter=true` 使得新增字段的过滤条件被保留 -- 例如新增字段 `c` 的谓词 `c > 3`，对于旧文件(该字段不存在，等同全 null)可以安全地过滤。

### 1.7 主键表 vs 追加表谓词下推差异

| 维度 | KeyValueFileStoreScan | AppendOnlyFileStoreScan |
|------|----------------------|------------------------|
| **主过滤字段** | keyStats (主键统计) | valueStats (全列统计) |
| **Value 过滤** | 条件性启用，受 scanMode/mergeEngine 影响 | 始终使用 inputFilter |
| **文件索引** | 仅在 value 过滤中使用 (embeddedIndex) | 在 filterByStats 中直接使用 |
| **Whole Bucket 过滤** | 支持(整桶判断，解决 overlapping 问题) | 不需要(无 merge 语义) |
| **Limit 下推** | 受限(PARTIAL_UPDATE/AGGREGATE/DV 禁用) | 受限(DV/数据演化 禁用) |

**为什么主键表的 Value 过滤是条件性的**: 主键表存在 merge 语义 -- 同一主键可能在多个文件中出现不同值。在 `ScanMode.ALL` 模式下(批读取全量快照)，如果启用 value 过滤，可能会漏掉某些文件中需要参与 merge 的记录。因此只有在 `valueFilterForceEnabled`(由上层显式启用)或 `CHANGELOG` 模式下的 LOOKUP/FULL_COMPACTION producer 才启用 (见 `KeyValueFileStoreScan.java:196-211`)。

### 1.8 后过滤: wholeBucketFilter 与 Limit 下推

**源码路径**: `KeyValueFileStoreScan.java:232-254`

`postFilterManifestEntries()` 是主键表独有的后处理步骤，在所有 Manifest 条目都收集完毕后执行:

1. **按 (partition, bucket) 分组** -- 使用 `LinkedHashMap` 保序
2. **判断 noOverlapping** -- 如果同一桶内所有文件都在同一非零 Level，则无 overlap
3. **wholeBucketFilter** -- 对有 overlap 的桶做整桶判断，对无 overlap 的桶做逐文件判断
4. **Limit 下推** -- 仅对无 overlap 的桶生效，累积 rowCount 直到达到 limit

**wholeBucketFilter 的设计意图** (见 `KeyValueFileStoreScan.java:238-242, 285-299`):

**执行逻辑**:
1. 如果 `noOverlapping == true`:
   - 调用 `filterWholeBucketPerFile(entries)` -- 逐文件过滤
   
2. 如果 `noOverlapping == false`:
   - 调用 `filterWholeBucketAllFiles(entries)` -- 整桶判断
   - **特殊情况**: 当 `!deletionVectorsEnabled && (mergeEngine == PARTIAL_UPDATE || mergeEngine == AGGREGATE)` 时，直接返回所有文件，不做过滤
   - **原因**: 部分更新和聚合引擎下，单个文件的 valueStats 不能代表 merge 后的最终结果

**设计原理**:
- 当同一 bucket 内的文件存在 key overlap(多 level 或 level 0)时，即使一个文件的 valueStats 不匹配谓词，其中的记录可能在 merge 后改变结果，所以只能整桶判断:只有桶内所有文件都不匹配才能跳过整个桶
- 当文件间无 overlap(同一非零 level)时，可以逐文件过滤

## 2. 文件索引加速查询

### 解决什么问题

**核心业务问题**: 即使通过统计信息过滤掉了大部分文件，剩余文件仍然可能包含大量不匹配的数据。如何在不读取完整文件的情况下，进一步裁剪需要读取的数据？

**没有这个设计的后果**:
- 统计信息只能做文件级过滤 (整个文件要么读要么不读)，无法做行级过滤
- 对于选择性很低的查询 (如 `user_id = 'xxx'` 在百万行文件中只匹配1行)，仍需读取整个文件
- 列式存储格式 (Parquet/ORC) 的 RowGroup 跳过粒度太粗 (通常几千到几万行)

**实际场景**:
```sql
-- 场景1: 高选择性等值查询
SELECT * FROM user_events WHERE user_id = 'user_12345';
-- 文件有100万行，只有10行匹配，但 min/max 统计无法过滤文件

-- 场景2: 低基数列过滤
SELECT * FROM orders WHERE status = 'CANCELLED';
-- status 只有5个不同值，Bitmap 索引可以精确定位到匹配的行号

-- 场景3: 数值范围查询
SELECT * FROM transactions WHERE amount BETWEEN 1000 AND 2000;
-- BSI 索引可以通过位运算快速计算出匹配行的位图
```

### 有什么坑

**误区陷阱**:
1. **误以为所有列都应该建索引** - 索引有存储和维护成本，只对高选择性查询有效
2. **混淆嵌入式和独立索引的使用场景** - 大型索引嵌入会导致 Manifest 膨胀
3. **忽略索引的误报率** - Bloom Filter 有误报 (false positive)，不能完全依赖

**错误配置**:
```properties
# 错误1: 在低选择性列上建 Bloom Filter
file-index.bloom-filter.columns=gender  # gender 只有2-3个值，Bloom Filter 无意义

# 错误2: Bloom Filter 误报率设置过低
file-index.bloom-filter.fpp=0.001  # 过低的 fpp 导致索引过大，嵌入式存储不划算

# 错误3: 在高基数列上建 Bitmap 索引
file-index.bitmap.columns=user_id  # user_id 有百万级不同值，Bitmap 会爆炸

# 正确配置:
file-index.bloom-filter.columns=user_id,order_id  # 高基数列用 Bloom
file-index.bitmap.columns=status,type  # 低基数列用 Bitmap
file-index.bsi.columns=amount,score  # 数值列用 BSI
```

**生产环境注意事项**:
1. **索引大小控制** - 嵌入式索引建议 < 10KB，否则应使用独立索引文件
2. **索引更新成本** - 每次写入都需要更新索引，高频写入场景需要权衡
3. **内存压力** - 独立索引文件需要额外的 I/O 读取，可能成为瓶颈

**性能陷阱**:
```java
// 陷阱1: Bloom Filter 误报导致读取不必要的文件
// 解决: 调整 fpp 参数，或结合其他索引类型

// 陷阱2: Bitmap 索引在高基数列上内存爆炸
// 解决: 使用 BSI 或 Range Bitmap 代替

// 陷阱3: 独立索引文件的 I/O 开销
// 解决: 对于小文件，索引读取成本可能超过直接读取数据的成本
```

### 核心概念解释

**术语定义**:

1. **文件索引 (File Index)** - 嵌入在数据文件元数据中或独立存储的辅助数据结构，用于快速判断文件是否包含匹配数据
2. **Bloom Filter** - 概率型数据结构，可以快速判断元素是否"可能存在"或"一定不存在"
3. **Bitmap Index** - 为每个不同值维护一个位图，标记哪些行包含该值
4. **BSI (Bit-Sliced Index)** - 将数值按二进制位分片存储，每一位一个位图，适合范围查询
5. **Range Bitmap** - 结合有序字典和位图，适合有序或半有序数据的范围查询
6. **嵌入式索引** - 存储在 DataFileMeta.embeddedIndex 字段中，随 Manifest 一起读取
7. **独立索引文件** - 存储在 extraFiles 中的 .idx 文件，需要额外 I/O 读取

**概念关系**:
```
FileIndexPredicate (统一评估入口)
  ├─ FileIndexReader (索引读取器接口)
  │    ├─ BloomFilterFileIndex.Reader (等值查询)
  │    ├─ BitmapFileIndex.Reader (等值/IN/NOT IN)
  │    ├─ BitSliceIndexBitmapFileIndex.Reader (范围查询)
  │    └─ RangeBitmapFileIndex.Reader (范围查询 + TopN)
  └─ FileIndexResult (三值逻辑)
       ├─ REMAIN (可能包含匹配数据)
       ├─ SKIP (确定不包含匹配数据)
       └─ BitmapIndexResult (精确行号位图)
```

**与其他系统对比**:

| 系统 | 支持的索引类型 | 存储方式 | 行级过滤 |
|------|--------------|---------|---------|
| **Paimon** | Bloom/Bitmap/BSI/Range Bitmap | 嵌入式 + 独立文件 | 支持 (Bitmap 返回行号) |
| **Iceberg** | Bloom Filter | 独立 Puffin 文件 | 不支持 (仅文件级) |
| **Delta Lake** | Bloom Filter | 独立文件 | 不支持 (仅文件级) |
| **Hudi** | Bloom Filter | 嵌入式 (Parquet Footer) | 不支持 (仅文件级) |
| **Parquet** | Min/Max/Dictionary | 嵌入式 (Footer) | 支持 (RowGroup 级) |

**Paimon 的独特之处**:
- **四种索引类型** - 覆盖等值、范围、TopN 等多种查询模式
- **行级过滤** - Bitmap/BSI/Range Bitmap 返回精确行号，可以跳过不匹配的行
- **嵌入式 + 独立混合模式** - 小索引嵌入，大索引独立，平衡性能和存储
- **与 DV 的位图级协同** - 索引结果和 DV 都是 RoaringBitmap32，可以高效组合

### 设计理念

**为什么这样设计**:

1. **多种索引类型覆盖不同场景** - 没有一种索引适合所有查询：
   - Bloom Filter: 高基数列的等值查询，空间效率高但只能判断"可能存在"
   - Bitmap: 低基数列的等值/IN 查询，可以精确定位行号
   - BSI: 数值列的范围查询，空间固定 O(bits * n)，不受基数影响
   - Range Bitmap: 有序数据的范围查询和 TopN，利用有序性优化

2. **嵌入式 + 独立混合模式** - 平衡 Scan 和 Read 阶段的性能：
   ```
   Scan 阶段 (filterByStats):
     - 需要快速过滤大量文件
     - 嵌入式索引无额外 I/O，可以立即使用
     - 适合小型索引 (Bloom Filter 几百字节到几 KB)
   
   Read 阶段 (FileIndexEvaluator):
     - 已经确定要读取的文件，可以接受额外 I/O
     - 独立索引文件可以很大 (Bitmap 可能数 MB)
     - 适合需要精确行号的索引
   ```

3. **三值逻辑 (REMAIN/SKIP/BitmapIndexResult)** - 支持不同精度的过滤：
   - REMAIN: 保守策略，文件可能包含匹配数据 (Bloom Filter 误报)
   - SKIP: 确定不包含匹配数据，可以安全跳过
   - BitmapIndexResult: 精确行号，可以只读取匹配的行

4. **Visitor 模式的谓词评估** - 解耦索引类型和谓词类型：
   ```java
   // 每种索引只需实现自己支持的操作
   class BloomFilterFileIndex.Reader {
       FileIndexResult visitEqual(FieldRef, Object) { ... }  // 只支持等值
       // 其他操作返回 REMAIN (保守策略)
   }
   
   class BitmapFileIndex.Reader {
       FileIndexResult visitEqual(FieldRef, Object) { ... }
       FileIndexResult visitIn(FieldRef, List) { ... }
       FileIndexResult visitNotEqual(FieldRef, Object) { ... }
       // 支持多种操作
   }
   ```

**权衡取舍**:

| 设计选择 | 优势 | 劣势 | 取舍原因 |
|---------|------|------|---------|
| 多种索引类型 | 覆盖更多查询场景 | 实现和维护复杂 | 数据湖查询模式多样，单一索引无法满足 |
| 嵌入式索引 | Scan 阶段无额外 I/O | 限制索引大小，Manifest 膨胀 | Scan 阶段是热路径，性能优先 |
| 独立索引文件 | 支持大型索引 | Read 阶段额外 I/O | 已确定读取的文件，额外 I/O 可接受 |
| Bloom Filter 默认 fpp=0.1 | 索引更小，嵌入式存储友好 | 10% 误报率 | 整体过滤效果和存储开销的平衡 |
| Bitmap 返回行号 | 可以跳过不匹配的行 | 需要格式层支持 selection | 行级过滤收益显著，值得投入 |

**架构演进**:

1. **早期版本** - 只有 Bloom Filter，仅支持等值查询，独立文件存储
2. **引入嵌入式索引** - 小型 Bloom Filter 嵌入到 DataFileMeta，Scan 阶段可用
3. **引入 Bitmap 索引** - 支持低基数列的精确行级过滤
4. **引入 BSI 索引** - 支持数值列的范围查询，解决 Bitmap 在高基数列上的问题
5. **引入 Range Bitmap** - 支持有序数据的 TopN 查询
6. **与 DV 协同** - 索引结果和 DV 都使用 RoaringBitmap32，可以高效组合

**业界对比**:

- **Iceberg** - 只有 Bloom Filter，存储在独立的 Puffin 文件中，不支持行级过滤
- **Delta Lake** - 只有 Bloom Filter，存储在独立文件中，不支持行级过滤
- **Hudi** - Bloom Filter 嵌入在 Parquet Footer 中，但只支持等值查询
- **Parquet/ORC** - 内置 Min/Max 和 Dictionary 编码，但粒度是 RowGroup (几千到几万行)

**Paimon 的创新点**:
- **行级过滤** - Bitmap/BSI/Range Bitmap 返回精确行号，比 RowGroup 级跳过更精细
- **嵌入式 + 独立混合** - 根据索引大小和使用阶段选择存储方式
- **TopN 支持** - Range Bitmap 是唯一支持 TopN 的索引类型
- **与 DV 的位图级协同** - 索引和 DV 共享 RoaringBitmap32 格式，可以高效组合

### 2.1 FileIndexPredicate 评估流程

**源码路径**: `paimon-common/.../fileindex/FileIndexPredicate.java`

`FileIndexPredicate` 是文件索引的统一评估入口，支持从嵌入字节或文件路径构造。评估流程:

```
evaluate(Predicate)
  +-- getRequiredNames(predicate)     // 提取谓词涉及的列名
  +-- reader.readColumnIndex(name)    // 为每个列加载所有相关索引 Reader
  +-- FileIndexPredicateTest.test()   // 用 Visitor 模式遍历谓词树
        +-- visit(LeafPredicate)      // 叶子节点: 调用每个 FileIndexReader 的 visit 方法
        |     predicate.function().visit(fileIndexReader, fieldRef, literals)
        +-- visit(CompoundPredicate)  // 复合节点: AND/OR 组合
              +-- AND: 短路求值, 任一 SKIP 则 SKIP
              +-- OR: 任一 REMAIN 则 REMAIN
```

**FileIndexResult** (见 `fileindex/FileIndexResult.java`) 是一个三值逻辑接口:
- `REMAIN` -- 文件可能包含匹配数据，需要保留
- `SKIP` -- 文件确定不包含匹配数据，可以跳过
- `BitmapIndexResult` -- 返回精确的行号位图(RoaringBitmap32)，仅读取命中的行

**为什么使用 Visitor 模式**: 不同类型的索引(Bloom、Bitmap、BSI、Range Bitmap)支持不同的谓词操作。`FileIndexReader` 基类为所有操作提供默认返回 `REMAIN`(保守策略)，各索引实现只需覆盖自己支持的操作。这种设计使得新增索引类型和新增谓词类型互不干扰。

### 2.2 四种索引的查询应用

#### 2.2.1 Bloom Filter

**源码路径**: `paimon-common/.../fileindex/bloomfilter/BloomFilterFileIndex.java`

| 特性 | 说明 |
|------|------|
| 支持的谓词 | `=` (等值查询) |
| 返回类型 | `REMAIN` / `SKIP` (文件级粗过滤) |
| 适用场景 | 高基数列的等值查询(如 user_id, order_id) |
| 配置参数 | `items` (预期元素数, 默认 1000000), `fpp` (误报率, 默认 0.1) |

核心实现 (见 `BloomFilterFileIndex.Reader.visitEqual`):
```java
public FileIndexResult visitEqual(FieldRef fieldRef, Object key) {
    return key == null || filter.testHash(hashFunction.hash(key)) ? REMAIN : SKIP;
}
```

**为什么默认误报率是 0.1**: 较高的 fpp 使得 Bloom Filter 占用更少空间(嵌入到 DataFileMeta 中时更紧凑)。对于大表的大量文件，10% 的误报率在整体过滤效果和存储开销间取得了平衡。用户可根据需要调低 fpp。

#### 2.2.2 Bitmap Index

**源码路径**: `paimon-common/.../fileindex/bitmap/BitmapFileIndex.java`

| 特性 | 说明 |
|------|------|
| 支持的谓词 | `=`, `!=`, `IN`, `NOT IN`, `IS NULL`, `IS NOT NULL` |
| 返回类型 | `BitmapIndexResult` (精确行号位图) |
| 适用场景 | 低基数列的过滤(如 status, type, gender) |
| 版本 | V1(基础版) / V2(分块版, 支持 index-block-size) |

**好处**: Bitmap 索引返回精确的行号集合，与 Parquet/ORC 的 RowGroup 跳过不同，它可以让格式层只读取命中行对应的数据块，实现行级的跳过。这在 `RawFileSplitRead` 中通过 `FormatReaderContext.selection` 传递给文件格式读取器。

#### 2.2.3 BSI (Bit-Sliced Index)

**源码路径**: `paimon-common/.../fileindex/bsi/BitSliceIndexBitmapFileIndex.java`

| 特性 | 说明 |
|------|------|
| 支持的谓词 | `=`, `!=`, `<`, `<=`, `>`, `>=`, `BETWEEN`, `IN`, `IS NULL` |
| 返回类型 | `BitmapIndexResult` (精确行号位图) |
| 适用场景 | 数值列的范围查询(如 amount, score, age) |
| 数据结构 | 将数值拆分为正数/负数两个 BitSliceIndex，按位分片存储 |

**为什么 BSI 适合范围查询**: 传统 Bitmap 索引对每个不同值维护一个位图，对高基数列空间爆炸。BSI 将数值按二进制位分片(bit-sliced)，每一位一个位图，空间固定为 O(bits * n)。范围查询可以通过位运算在 O(bits) 次位图操作内完成。

#### 2.2.4 Range Bitmap

**源码路径**: `paimon-common/.../fileindex/rangebitmap/RangeBitmapFileIndex.java`

| 特性 | 说明 |
|------|------|
| 支持的谓词 | `=`, `<`, `<=`, `>`, `>=`, `BETWEEN`, `IS NULL`, TopN |
| 返回类型 | `BitmapIndexResult` (精确行号位图) |
| 适用场景 | 有序/半有序数值列的范围查询, TopN 查询 |
| 配置参数 | `chunk-size` (字典分块大小) |

**独特的 TopN 支持**: Range Bitmap 索引是唯一支持 `visitTopN` 的索引类型。它可以利用有序字典快速定位 Top-K 记录的行号，避免全文件扫描后排序。

### 2.3 嵌入式 vs 独立索引文件

**源码路径**: `io/FileIndexEvaluator.java:111-142`

| 模式 | 存储位置 | 大小限制 | 适用场景 |
|------|---------|---------|---------|
| 嵌入式 | `DataFileMeta.embeddedIndex` 字段 | 较小索引(Bloom Filter) | Scan 阶段直接使用, 无额外 I/O |
| 独立文件 | `extraFiles` 中的 `.idx` 后缀文件 | 较大索引(Bitmap) | Read 阶段加载, 需要额外 I/O |

**为什么有两种模式**: 小型索引(如 Bloom Filter 几百字节到几 KB)嵌入到 ManifestEntry 中，Scan 阶段无需额外 I/O 就能使用。大型索引(如 Bitmap 可能数 MB)嵌入会导致 Manifest 文件膨胀，影响 Scan 性能，所以独立存储。

**在 Scan 阶段**(filterByStats): 两种 FileStoreScan 实现都从 `entry.file().embeddedIndex()` 获取嵌入索引进行评估。

**在 Read 阶段** (FileIndexEvaluator): 先检查 embeddedIndex，若无则查找 extraFiles 中的 `.idx` 文件:
```java
byte[] embeddedIndex = file.embeddedIndex();
if (embeddedIndex != null) {
    predicate = new FileIndexPredicate(embeddedIndex, dataSchema.logicalRowType());
} else {
    List<String> indexFiles = file.extraFiles().stream()
        .filter(name -> name.endsWith(INDEX_PATH_SUFFIX)).collect(...);
    predicate = new FileIndexPredicate(path, fileIO, dataSchema.logicalRowType());
}
```

### 2.4 读取阶段的索引评估 (FileIndexEvaluator)

**源码路径**: `paimon-core/.../io/FileIndexEvaluator.java`

`RawFileSplitRead.createFileReader()` 在创建文件读取器前会调用 `FileIndexEvaluator.evaluate()`:

```
FileIndexEvaluator.evaluate()
  +-- 无过滤条件且无 TopN? -> 仅处理 Limit (创建 baseSelection 并截断)
  +-- createBaseSelection(file, dv) -> 全行位图 AND NOT 删除向量
  +-- 有过滤条件? -> predicate.evaluate(filter) AND baseSelection
  +-- 有 TopN? -> predicate.evaluateTopN(topN, selection)
  +-- 结果 == baseSelection? -> 返回 REMAIN (避免不必要的位图传递)
```

**DV 与索引的协同**: `createBaseSelection` 先生成覆盖 `[0, rowCount)` 的全行位图，然后减去 DV 中标记删除的行。这确保了文件索引评估的结果不包含已删除的行。

## 3. 多维排序策略

### 解决什么问题

**核心业务问题**: 数据湖中的数据通常按写入顺序存储，但查询可能涉及多个维度的过滤条件。如何组织数据的物理布局，使得多维查询都能获得较好的文件跳过率？

**没有这个设计的后果**:
- 单列排序 (ORDER BY col1) 只对 col1 的查询友好，其他列的 min/max 范围很宽，无法有效过滤
- 多列字典序排序 (ORDER BY col1, col2) 只对前缀列友好，后续列的区分度很低
- 查询需要扫描大量文件，即使统计信息过滤也无法有效裁剪

**实际场景**:
```sql
-- 场景1: 多维等权查询
SELECT * FROM events 
WHERE city = 'Beijing' AND event_type = 'click' AND dt = '2024-01-01';
-- 三个维度都是常用过滤条件，单列排序无法同时优化

-- 场景2: 地理空间查询
SELECT * FROM locations 
WHERE longitude BETWEEN 116.0 AND 117.0 
  AND latitude BETWEEN 39.0 AND 40.0;
-- 经纬度两个维度，需要保持空间局部性

-- 场景3: 时间序列多维分析
SELECT * FROM metrics 
WHERE timestamp BETWEEN '2024-01-01' AND '2024-01-02'
  AND metric_name = 'cpu_usage' 
  AND host_id IN ('host1', 'host2');
-- 时间、指标名、主机ID 三个维度都需要优化
```

### 有什么坑

**误区陷阱**:
1. **误以为 Z-Order 总是比 ORDER 好** - Z-Order 在单列查询时可能不如字典序排序
2. **忽略排序列的选择** - 不是所有列都适合参与多维排序，高基数列效果更好
3. **混淆 Z-Order 和 Hilbert** - Hilbert 理论上更优，但计算成本更高

**错误配置**:
```properties
# 错误1: 在低基数列上使用 Z-Order
sort.order=zorder
sort.columns=gender,status  # 只有2-3个不同值，Z-Order 无意义

# 错误2: 排序列过多
sort.columns=col1,col2,col3,col4,col5  # 超过3-4列，效果递减

# 错误3: 排序列顺序不当
sort.columns=timestamp,user_id  # timestamp 高基数在前，user_id 低基数在后

# 正确配置:
sort.order=zorder
sort.columns=city,event_type  # 两个中等基数列，查询频率相当
```

**生产环境注意事项**:
1. **排序成本** - Z-Order/Hilbert 需要全表数据 shuffle 和排序，适合批量重组而非实时写入
2. **排序列数量** - 建议 2-4 列，过多列会导致每列的区分度都不够
3. **数据倾斜** - 排序列的数据分布不均会导致某些分区过大

**性能陷阱**:
```java
// 陷阱1: 频繁触发全量排序
// Z-Order 需要读取全表数据，不适合高频触发
// 解决: 使用 Clustering Compaction 的增量模式

// 陷阱2: 排序列类型不当
// 字符串类型的 Z-Order 需要截断/填充到固定长度，可能损失精度
// 解决: 优先选择数值类型或固定长度字符串

// 陷阱3: 单列查询性能下降
// Z-Order 后单列的 min/max 范围可能比字典序排序更宽
// 解决: 根据查询模式选择排序策略
```

### 核心概念解释

**术语定义**:

1. **多维排序 (Multi-Dimensional Sorting)** - 将多维数据映射到一维空间，同时尽量保持多个维度的局部性
2. **空间填充曲线 (Space-Filling Curve)** - 将多维空间映射到一维的曲线，如 Z-Order 曲线、Hilbert 曲线
3. **Z-Order (Z 曲线)** - 通过字节交错 (bit interleaving) 将多维坐标映射到一维 Z-value
4. **Hilbert 曲线** - 一种分形曲线，比 Z 曲线有更好的空间局部性保持性
5. **字节交错 (Bit Interleaving)** - 将多个数值的二进制位交错排列，生成一个新的数值
6. **字典序排序 (Lexicographic Sorting)** - 按列的优先级顺序排序，如 ORDER BY col1, col2

**概念关系**:
```
TableSorter (抽象基类)
  ├─ OrderSorter (字典序排序)
  │    └─ 适用场景: 单列或有明确优先级的多列查询
  ├─ ZorderSorter (Z-Order 排序)
  │    ├─ 字节交错算法 (ZOrderByteUtils)
  │    └─ 适用场景: 2-3列等权查询
  └─ HilbertSorter (Hilbert 曲线排序)
       ├─ Hilbert 曲线算法
       └─ 适用场景: 3列及以上等权查询，追求最优局部性
```

**与其他系统对比**:

| 系统 | 支持的排序策略 | 实现方式 | 增量支持 |
|------|--------------|---------|---------|
| **Paimon** | ORDER/ZORDER/HILBERT | Spark/Flink 实现 | Clustering Compaction |
| **Iceberg** | ZORDER | Spark 实现 | 不支持 (需全表重写) |
| **Delta Lake** | ZORDER | Spark 实现 (OPTIMIZE ZORDER BY) | 不支持 (需全表重写) |
| **Hudi** | HILBERT/LINEAR | Spark 实现 | Clustering Service |
| **Databricks** | ZORDER/HILBERT | 专有实现 | Liquid Clustering (增量) |

**Paimon 的独特之处**:
- **三种排序策略** - ORDER/ZORDER/HILBERT 覆盖不同查询模式
- **Clustering Compaction** - 支持增量排序，无需全表重写
- **Spark 和 Flink 双引擎支持** - 可以在不同计算引擎中使用

### 设计理念

**为什么这样设计**:

1. **提供多种排序策略而非单一方案** - 不同查询模式需要不同的排序策略：
   ```
   查询模式                     最优排序策略
   ─────────────────────────────────────────
   单列范围查询                 ORDER (字典序)
   单列 + 其他列辅助过滤         ORDER (主列在前)
   两列等权查询                 ZORDER
   三列及以上等权查询           HILBERT
   地理空间查询 (经纬度)         HILBERT
   ```

2. **通过计算引擎实现而非存储层** - 排序需要全表数据 shuffle：
   - 存储层只负责数据的读写和元数据管理
   - 排序作为批处理作业，由 Spark/Flink 的分布式计算能力完成
   - 保持存储层的简洁性和通用性

3. **Z-Order 的字节交错算法** - 保持多维空间局部性：
   ```
   一维排序:
   col1: [1,2,3,4,5,6,7,8]
   col2: [a,a,a,a,b,b,b,b]
   排序后: (1,a), (2,a), (3,a), (4,a), (5,b), (6,b), (7,b), (8,b)
   问题: col2 的 min/max 范围很宽 [a, b]
   
   Z-Order 排序:
   交错后: (1,a), (1,b), (2,a), (2,b), (3,a), (3,b), (4,a), (4,b)
   好处: 每个文件的 col1 和 col2 的 min/max 范围都较小
   ```

4. **Hilbert 曲线的优势** - 比 Z 曲线更好的局部性：
   - Z 曲线存在"跳跃点" - 某些相邻的 Z-value 对应的多维坐标距离很远
   - Hilbert 曲线是连续的，相邻的 Hilbert-value 对应的多维坐标也相邻
   - 代价是 Hilbert 曲线的计算比 Z-Order 更复杂

**权衡取舍**:

| 设计选择 | 优势 | 劣势 | 取舍原因 |
|---------|------|------|---------|
| 多种排序策略 | 适应不同查询模式 | 用户需要选择 | 没有万能的排序策略，提供选择权 |
| 计算引擎实现 | 利用分布式计算能力 | 需要额外的批处理作业 | 排序是计算密集型，存储层不适合 |
| Z-Order 字节交错 | 实现简单，性能好 | 存在跳跃点 | 大多数场景下足够好 |
| Hilbert 曲线 | 最优局部性 | 计算复杂，性能开销大 | 追求极致性能的场景 |
| 固定长度字节表示 | 保证等权交错 | 字符串需要截断/填充 | 多维排序的前提条件 |

**架构演进**:

1. **早期版本** - 只支持字典序排序 (ORDER BY)
2. **引入 Z-Order** - 借鉴 Databricks 的 Z-Order 实现，支持多维排序
3. **引入 Hilbert** - 提供更优的空间局部性，适合地理空间查询
4. **引入 Clustering Compaction** - 支持增量排序，无需全表重写
5. **优化字节交错算法** - 支持不同类型的字典序转换 (有符号整数、浮点数、字符串)

**业界对比**:

- **Databricks Delta Lake** - 最早引入 Z-Order 的数据湖系统，后续推出 Liquid Clustering (自动增量排序)
- **Iceberg** - 支持 Z-Order，但需要全表重写，无增量支持
- **Hudi** - 支持 Hilbert 和 Linear 排序，通过 Clustering Service 实现增量排序
- **ClickHouse** - 原生支持多列排序键，但不支持 Z-Order/Hilbert

**Paimon 的创新点**:
- **Clustering Compaction 的增量模式** - 只对新写入的数据排序，已排序数据只在 overlap 时合并
- **Spark 和 Flink 双引擎支持** - 可以在不同计算引擎中使用相同的排序策略
- **与文件索引的协同** - 排序后的数据配合文件索引，可以获得更好的过滤效果

### 3.1 TableSorter 体系架构

**源码路径**: 
- Spark: `paimon-spark/paimon-spark-common/.../spark/sort/TableSorter.java`
- Flink: `paimon-flink/paimon-flink-common/.../flink/sorter/TableSorter.java`

Paimon 的多维排序通过计算引擎实现(需要分布式排序能力)。`TableSorter` 是抽象基类，通过工厂方法根据 `OrderType` 选择排序策略:

```java
public static TableSorter getSorter(FileStoreTable table, OrderType orderType, List<String> orderColumns) {
    switch (orderType) {
        case ORDER:   return new OrderSorter(table, orderColumns);
        case ZORDER:  return new ZorderSorter(table, orderColumns);
        case HILBERT: return new HilbertSorter(table, orderColumns);
        case NONE:    return /* 无操作 */;
    }
}
```

**为什么通过计算引擎实现**: 多维排序需要读取全表数据并重新分布排序，这是一个批处理作业，需要分布式计算框架的 shuffle 能力。Paimon 在 Spark 和 Flink 的引擎集成层分别实现了排序，保持了存储层的简洁性。

### 3.2 OrderSorter: 字典序排序

**源码路径**: `spark/sort/OrderSorter.java`

```java
public Dataset<Row> sort(Dataset<Row> input) {
    Column[] sortColumns = orderColNames.stream().map(input::col).toArray(Column[]::new);
    return input.repartitionByRange(sortColumns).sortWithinPartitions(sortColumns);
}
```

**为什么先 repartitionByRange 再 sortWithinPartitions**:
1. `repartitionByRange` -- 按排序列做范围分区，保证相近值的数据落在同一分区
2. `sortWithinPartitions` -- 每个分区内局部排序

**好处**: 单列或有明确优先级的多列查询，字典序排序使得数据在存储层高度有序，min/max 统计信息的区分度最好，文件跳过率最高。

### 3.3 ZorderSorter: Z-Order 字节交错算法

**源码路径**: `spark/sort/ZorderSorter.java`

Z-Order 排序的核心步骤:
1. **将每列值转为字典序字节表示** -- 通过 `SparkZOrderUDF.sortedLexicographically()` 为每种类型调用对应的转换 UDF
2. **字节交错(interleave)** -- 将所有列的字节数组按位交错成一个 Z-value
3. **按 Z-value 范围分区并排序** -- `repartitionByRange(ZVALUE).sortWithinPartitions(ZVALUE)`
4. **丢弃 Z-value 列**

**为什么需要字节交错**: Z-Order 排序保留了多维数据的空间局部性。交错后，多维空间中相近的点在一维 Z-value 中也倾向于相近，使得同一文件中包含的数据在多个维度上都具有较小的 min/max 范围，从而提高多列联合过滤的跳过率。

### 3.4 HilbertSorter: Hilbert 曲线

**源码路径**: `spark/sort/HilbertSorter.java`

Hilbert 排序与 Z-Order 类似，但使用 Hilbert 曲线代替 Z 曲线:
```java
public Dataset<Row> sort(Dataset<Row> df) {
    Column hilbertColumn = hilbertValue(df);
    Dataset<Row> hilbertValueDF = df.withColumn(H_COLUMN, hilbertColumn);
    return hilbertValueDF.repartitionByRange(H_COLUMN).sortWithinPartitions(H_COLUMN);
}
```

**为什么提供 Hilbert 曲线选项**: Hilbert 曲线比 Z 曲线有更好的空间局部性保持性(不存在 Z 曲线中的"跳跃点")。理论和实践表明，Hilbert 排序后数据的 min/max 范围更紧凑，能获得比 Z-Order 高约 10-20% 的文件跳过率。代价是 Hilbert 曲线的计算比 Z-Order 更复杂。

### 3.5 Z-Order 字节交错算法原理 (ZOrderByteUtils)

**源码路径**: `paimon-common/.../sort/zorder/ZOrderByteUtils.java`

核心思想是将多维数据映射到一维空间，同时尽量保持空间局部性。

**Step 1: 类型到有序字节的转换**

每种类型都需要转换为字典序可比较的字节表示:

| 类型 | 转换方法 | 原理 |
|------|---------|------|
| 有符号整数 | XOR `0x8000000000000000L` | 翻转符号位，使负数在正数之前 |
| 浮点数 | XOR `((lval >> 63) \| Long.MIN_VALUE)` | IEEE 754 符号量级整数重解释，将浮点数按位重解释为 long 后翻转符号位 |
| 字符串 | UTF-8 编码 + 截断/填充到固定长度 | 保证等长比较 |

**Step 2: 位交错 (interleaveBits)**

(见 `ZOrderByteUtils.java:184-220`)

```
输入: col_A = [a7 a6 a5 a4 a3 a2 a1 a0], col_B = [b7 b6 b5 b4 b3 b2 b1 b0]
输出: [a7 b7 a6 b6 a5 b5 a4 b4 a3 b3 a2 b2 a1 b1 a0 b0]
```

算法逐列、逐位、轮询式地从源列中取出 bit，写入目标字节数组。当一列的 bit 用完后，仅从剩余列取 bit。

**为什么使用 PRIMITIVE_BUFFER_SIZE = 8**: 所有原始类型统一转为 8 字节(Long 宽度)表示，使得每列贡献相同数量的 bit 到交错结果中，保证各维度在 Z-curve 上的权重相同。

## 4. Compaction 排序优化 (Clustering)

### 解决什么问题

**核心业务问题**: 流式写入的数据通常是无序的或按时间有序，但查询可能需要按其他维度过滤。如何在不影响写入性能的前提下，逐步将数据按查询友好的方式重组？

**没有这个设计的后果**:
- 每次需要重组数据都要全表扫描和重写，成本极高
- 流式写入的数据永远是无序的，查询性能持续下降
- 无法在写入吞吐和查询性能之间取得平衡

**实际场景**:
```sql
-- 场景1: 实时写入 + 多维查询
-- 写入: 按时间顺序写入用户行为数据
INSERT INTO user_events VALUES (...);  -- 每秒数千条

-- 查询: 按用户ID和事件类型过滤
SELECT * FROM user_events 
WHERE user_id = 'xxx' AND event_type = 'click';
-- 如果数据按时间存储，user_id 的 min/max 范围很宽，无法有效过滤

-- 场景2: CDC 同步 + 点查
-- 写入: 从 MySQL CDC 同步数据，按主键无序写入
-- 查询: 按主键点查
-- 如果数据无序，每次点查需要扫描多个文件
```

### 有什么坑

**误区陷阱**:
1. **误以为 Clustering 是免费的** - 排序和重写需要消耗 CPU、内存和 I/O
2. **忽略增量和全量的差异** - 全量聚簇会读写整张表，不适合大表
3. **混淆 Clustering 和 Compaction** - Clustering 是特殊的 Compaction，专注于排序

**错误配置**:
```properties
# 错误1: 在高频写入场景下启用全量聚簇
compaction.clustering.full-compaction=true  # 每次都全表重写，写放大极高

# 错误2: 聚簇列选择不当
compaction.clustering.columns=timestamp  # 写入已按时间有序，聚簇无意义

# 错误3: 聚簇触发过于频繁
compaction.clustering.trigger-interval=1min  # 频繁触发导致资源浪费

# 正确配置:
compaction.clustering.columns=user_id,event_type  # 查询常用的过滤列
compaction.clustering.full-compaction=false  # 增量模式
compaction.clustering.trigger-interval=1h  # 合理的触发间隔
```

**生产环境注意事项**:
1. **写放大控制** - 增量模式下，只有 level 0 文件需要排序，写放大较低
2. **资源隔离** - Clustering 是 CPU 和 I/O 密集型，建议在低峰期触发
3. **监控指标** - 关注 level 0 文件数量、聚簇进度、写放大比例

**性能陷阱**:
```java
// 陷阱1: 聚簇列基数过低
// 如果聚簇列只有几个不同值，排序后文件间仍有大量 overlap
// 解决: 选择中高基数的列

// 陷阱2: 外部排序 Spill 过多
// 如果内存不足，排序会频繁 Spill 到磁盘
// 解决: 增大 sort-buffer-size 或减少并行度

// 陷阱3: Phase 2 合并过于激进
// 如果 targetFileSize 设置过大，会合并过多文件
// 解决: 根据查询模式调整 targetFileSize
```

### 核心概念解释

**术语定义**:

1. **Clustering Compaction** - 按指定列排序数据的 Compaction 策略，专注于提升查询性能
2. **聚簇列 (Clustering Columns)** - 用于排序的列，通常是查询中常用的过滤列
3. **Phase 1: Sort & Rewrite** - 读取 level 0 文件，按聚簇列排序后写入 level 1
4. **Phase 2: Merge** - 检测 level 1 文件间的 overlap 和碎片，合并重写
5. **ClusteringKeyIndex** - 用于去重检查的内存索引，支持 FIRST_ROW 和 DEDUPLICATE 模式
6. **增量聚簇** - 只处理新写入的 level 0 文件和 overlap 的 level 1 文件
7. **全量聚簇** - 处理所有文件，等价于全表重写

**概念关系**:
```
ClusteringCompactManager (两阶段管理器)
  ├─ Phase 1: Sort & Rewrite
  │    ├─ ClusteringFileRewriter.sortAndRewriteFile()
  │    ├─ BinaryExternalSortBuffer (外部排序)
  │    └─ ClusteringKeyIndex (去重检查)
  ├─ Phase 2: Merge
  │    ├─ ClusteringFileRewriter.pickMergeCandidates()
  │    │    ├─ 检测 overlap (IntervalPartition)
  │    │    └─ 检测碎片 (< targetFileSize/2)
  │    └─ ClusteringFileRewriter.mergeAndRewriteFiles()
  │         ├─ Min-Heap 多路归并
  │         └─ Spill 优化 (小文件先 spill)
  └─ ClusteringFiles (文件分层管理)
       ├─ unsortedFiles (level 0)
       └─ sortedFiles (level 1)
```

**与其他系统对比**:

| 系统 | 排序策略 | 增量支持 | 去重模式 | 触发方式 |
|------|---------|---------|---------|---------|
| **Paimon** | Clustering Compaction | 支持 (两阶段) | FIRST_ROW/DEDUPLICATE | 自动 + 手动 |
| **Hudi** | Clustering Service | 支持 | 不支持 | 异步服务 |
| **Iceberg** | Sort Compaction | 不支持 (需全表重写) | 不支持 | 手动 |
| **Delta Lake** | OPTIMIZE ZORDER BY | 不支持 (需全表重写) | 不支持 | 手动 |
| **Databricks** | Liquid Clustering | 支持 (自动增量) | 不支持 | 自动 |

**Paimon 的独特之处**:
- **两阶段架构** - Phase 1 处理增量，Phase 2 处理碎片，降低写放大
- **去重检查** - ClusteringKeyIndex 支持 FIRST_ROW 和 DEDUPLICATE 模式
- **Spill 优化** - Phase 2 合并时，小文件先 spill，减少文件句柄数
- **与 LSM-Tree 的集成** - 利用 level 0/1 的分层结构，无需额外的文件标记

### 设计理念

**为什么这样设计**:

1. **两阶段架构降低写放大** - 分离增量排序和碎片整理：
   ```
   传统全量聚簇:
   - 每次读取所有文件 (100GB)
   - 排序后重写所有文件 (100GB)
   - 写放大 = 2x (读100GB + 写100GB)
   
   Paimon 两阶段增量聚簇:
   - Phase 1: 只读取 level 0 文件 (10GB)，排序后写入 level 1 (10GB)
   - Phase 2: 只合并 overlap 的 level 1 文件 (20GB)
   - 写放大 = 0.4x (读30GB + 写30GB，相对于全表100GB)
   ```

2. **利用 LSM-Tree 的分层结构** - 无需额外的文件标记：
   - level 0 = unsorted files (新写入的数据)
   - level 1 = sorted files (已排序的数据)
   - 通过 level 字段天然区分，无需额外的 "sorted" 标记

3. **ClusteringKeyIndex 的去重检查** - 保证主键唯一性：
   ```java
   // FIRST_ROW 模式: 跳过重复 key
   if (!keyIndex.checkKey(keyBytes)) continue;
   
   // DEDUPLICATE 模式: 标记 DV
   if (!keyIndex.checkKey(keyBytes)) {
       deletionVector.delete(position);
   }
   ```
   - 排序过程中可能遇到相同主键的多个版本
   - FIRST_ROW: 保留第一个版本 (排序后最新的)
   - DEDUPLICATE: 标记 DV，后续读取时过滤

4. **Phase 2 的智能合并策略** - 平衡文件数量和合并成本：
   ```
   合并条件:
   1. 文件间有 overlap (key 范围重叠)
   2. 文件过小 (< targetFileSize/2)
   3. 相邻的待合并 section 合并成一个 group
   
   不合并条件:
   1. 文件无 overlap 且大小合适
   2. 单个文件的 section (无碎片)
   ```

**权衡取舍**:

| 设计选择 | 优势 | 劣势 | 取舍原因 |
|---------|------|------|---------|
| 两阶段架构 | 写放大低，适合流式写入 | 实现复杂 | 流式写入是核心场景，写放大必须控制 |
| 增量聚簇 | 只处理增量数据 | 可能产生碎片 | 大多数场景下增量足够，全量作为兜底 |
| ClusteringKeyIndex | 保证主键唯一性 | 需要额外内存 | 主键表的正确性优先 |
| Spill 优化 | 减少文件句柄数 | 小文件需要额外 I/O | 避免"too many open files"错误 |
| targetFileSize/2 阈值 | 控制碎片和合并成本 | 可能产生略小的文件 | 平衡文件数量和合并频率 |

**架构演进**:

1. **早期版本** - 只有普通 Compaction，无排序功能
2. **引入 Clustering Compaction** - 支持按聚簇列排序，但只有全量模式
3. **引入两阶段架构** - Phase 1 处理增量，Phase 2 处理碎片，降低写放大
4. **引入 ClusteringKeyIndex** - 支持排序过程中的去重检查
5. **引入 Spill 优化** - Phase 2 合并时，小文件先 spill，减少文件句柄数
6. **与 DV 集成** - DEDUPLICATE 模式支持标记 DV，而非物理删除

**业界对比**:

- **Databricks Liquid Clustering** - 自动增量聚簇，但实现细节不公开
- **Hudi Clustering Service** - 异步服务，支持增量，但需要额外的调度器
- **Iceberg** - 只支持全量排序，需要手动触发 REWRITE DATA
- **Delta Lake** - OPTIMIZE ZORDER BY 是全量操作，不支持增量

**Paimon 的创新点**:
- **两阶段增量聚簇** - 业界首个开源的增量聚簇实现 (Databricks 不开源)
- **与 LSM-Tree 的深度集成** - 利用 level 0/1 的分层结构，无需额外标记
- **ClusteringKeyIndex 的去重检查** - 保证主键唯一性，支持 FIRST_ROW 和 DEDUPLICATE
- **Spill 优化** - 解决大量文件合并时的文件句柄问题

### 4.1 ClusteringCompactManager 两阶段架构

**源码路径**: `paimon-core/.../mergetree/compact/clustering/ClusteringCompactManager.java`

ClusteringCompactManager 是为支持"按聚簇列排序"的主键表设计的 Compaction 管理器。它采用**两阶段**架构:

**Phase 1: 排序重写 (Sort & Rewrite)**
- 读取所有未排序文件(level 0)
- 按聚簇列外部排序后写入新的 level 1 文件
- 排序过程中通过 `ClusteringKeyIndex` 做去重检查(FIRST_ROW 跳过重复, DEDUPLICATE 标记 DV)

**Phase 2: 归并合并 (Merge)**
- 对 Phase 1 之前已存在的 sorted 文件，按聚簇列范围检测 overlap
- 将 overlap 的文件归为一组，相邻的小文件也合并
- 使用多路归并(min-heap)重写

**为什么需要两阶段**: Phase 1 处理增量写入的新数据(将其排序化)，Phase 2 处理已排序数据间的碎片和 overlap 问题。分开处理避免了每次 compaction 都读取全量数据。

**好处**: 增量聚簇模式下，只有新写入的数据(level 0 文件)需要完整排序，已排序的文件只在 overlap 或碎片化时才需要合并，大幅降低了写放大。

核心执行流程 (见 `ClusteringCompactManager.java:196-251`):
```java
private CompactResult compact(boolean fullCompaction) {
    // Phase 1: Sort and rewrite all unsorted (level 0) files
    List<DataFileMeta> unsortedFiles = fileLevels.unsortedFiles();
    List<DataFileMeta> existingSortedFiles = fileLevels.sortedFiles(); // 快照, 避免混入 Phase 1 新文件
    for (DataFileMeta file : unsortedFiles) {
        List<DataFileMeta> sortedFiles = fileRewriter.sortAndRewriteFile(file, ...);
        result.before().add(file);
        result.after().addAll(sortedFiles);
    }

    // Phase 2: Merge sorted files based on overlap
    List<List<DataFileMeta>> mergeGroups;
    if (fullCompaction) {
        mergeGroups = singletonList(existingSortedFiles);
    } else {
        mergeGroups = fileRewriter.pickMergeCandidates(existingSortedFiles);
    }
    // ... 执行合并
}
```

#### Phase 2: pickMergeCandidates + mergeAndRewriteFiles

`pickMergeCandidates` (见 `ClusteringFileRewriter.java:193-223`):
1. 按聚簇列 minKey 排序文件
2. 检测 overlap 分组为 sections
3. 包含 overlap(>=2 文件)或过小(`< targetFileSize/2`)的 section 被标记为待合并

- 使用 **min-heap 多路归并** 按聚簇列排序合并
- **Spill 优化**: 当输入文件数超过 `spillThreshold` 时，较小的文件先 spill 到行式临时文件(压缩存储)，减少同时打开的文件句柄数

**源码路径**: `paimon-core/.../mergetree/compact/clustering/ClusteringFiles.java`

```
ClusteringFiles
  +-- unsortedFiles (level 0) -- 新写入, 待排序
  +-- sortedFiles (level 1)   -- 已排序, 按聚簇列有序
  +-- idToFileMap             -- 文件 ID 快速查找
  +-- fileNameToIdMap         -- 文件名到 ID 映射
```

**为什么不使用传统 Levels 结构**: 传统 LSM-Tree 的 Levels 按数据新旧分层，而 Clustering 模式关注的是数据在聚簇列上的有序性，只需要区分"未排序"和"已排序"两种状态。简化的两层结构降低了管理复杂度。

### 4.4 增量 vs 全量聚簇

| 模式 | 触发条件 | Phase 1 | Phase 2 |
|------|---------|---------|---------|
| 增量 | `fullCompaction=false` | 仅处理 level 0 文件 | 智能选择 overlap/碎片 section |
| 全量 | `fullCompaction=true` | 同增量 | 所有 sorted 文件作为一个 merge group |

**为什么增量是默认策略**: 全量聚簇读写整张表的数据，I/O 成本极高。增量聚簇仅处理增量数据和碎片化区域，适合流式写入场景下定期触发。

## 5. Split 优化

### 解决什么问题

**核心业务问题**: 查询引擎需要将数据文件分配给多个并行任务读取。如何划分 Split 才能最大化并行度，同时避免不必要的 merge 开销？

**没有这个设计的后果**:
- 每个 Split 都需要做 merge，即使文件间没有 key 重叠
- Split 大小不均匀，导致数据倾斜和长尾任务
- 无法利用文件的有序性，增加读取成本

**实际场景**:
```sql
-- 场景1: 批量读取已 compacted 的数据
SELECT * FROM orders WHERE dt = '2024-01-01';
-- 如果文件都在 level 1 (无 overlap)，可以跳过 merge 直接读取

-- 场景2: 大表扫描
SELECT COUNT(*) FROM large_table;
-- 需要将数据均匀分配给多个任务，避免数据倾斜

-- 场景3: 流式读取
SELECT * FROM kafka_cdc_table;
-- 需要按 bucket 划分 Split，保证顺序性
```

### 有什么坑

**误区陷阱**:
1. **误以为 rawConvertible 总是更快** - 跳过 merge 需要满足严格条件，不是所有场景都适用
2. **忽略 Split 大小的影响** - 过大导致并行度不够，过小导致调度开销
3. **混淆主键表和追加表的 Split 策略** - 两者的 rawConvertible 判定逻辑完全不同

**错误配置**:
```properties
# 错误1: Split 大小过大
source.split.target-size=1GB  # 导致并行度不够，长尾任务

# 错误2: Split 大小过小
source.split.target-size=10MB  # 导致 Split 数量过多，调度开销大

# 错误3: 忽略 open-file-cost
source.split.open-file-cost=0  # 小文件场景下，打开文件的成本被忽略

# 正确配置:
source.split.target-size=128MB  # 根据集群并行度调整
source.split.open-file-cost=4MB  # 考虑打开文件的固定成本
```

**生产环境注意事项**:
1. **并行度计算** - Split 数量应该是并行度的 2-4 倍，保证负载均衡
2. **数据倾斜** - 某些 bucket 的数据量可能远大于其他 bucket
3. **内存压力** - 每个 Split 需要打开多个文件，注意文件句柄数限制

**性能陷阱**:
```java
// 陷阱1: level 0 文件过多导致无法 rawConvertible
// 所有 Split 都需要 merge，性能下降
// 解决: 及时触发 Compaction

// 陷阱2: IntervalPartition 产生大量小 Section
// 每个 Section 一个 Split，导致 Split 过多
// 解决: 调整 targetSplitSize 或触发 Compaction

// 陷阱3: BinPacking 装箱不均匀
// 某些 Split 包含大量小文件，打开文件成本高
// 解决: 增大 open-file-cost 参数
```

### 核心概念解释

**术语定义**:

1. **Split** - 查询引擎的最小读取单元，包含一组需要一起读取的文件
2. **rawConvertible** - 标记 Split 是否可以跳过 merge 直接读取原始文件
3. **IntervalPartition** - 将有 key 范围重叠的文件划分为不重叠的 Section
4. **Section** - 一组 key 范围可能重叠的文件，不同 Section 间 key 范围不重叠
5. **SortedRun** - Section 内的一组 key 范围不重叠的文件
6. **BinPacking** - 装箱算法，将文件按权重分配到多个 Split 中
7. **openFileCost** - 打开文件的固定成本，用于权重计算

**概念关系**:
```
SplitGenerator (Split 生成器)
  ├─ MergeTreeSplitGenerator (主键表)
  │    ├─ rawConvertible 判定
  │    │    ├─ 条件1: 所有文件 level != 0
  │    │    ├─ 条件2: 所有文件无删除行
  │    │    └─ 条件3: DV 启用 OR FIRST_ROW OR 同一 level
  │    ├─ IntervalPartition (Section 划分)
  │    │    ├─ 外层: Section 划分 (key 范围不重叠)
  │    │    └─ 内层: SortedRun 划分 (贪心 + 优先队列)
  │    └─ BinPacking (装箱)
  │         └─ packForOrdered (保持顺序)
  └─ AppendOnlySplitGenerator (追加表)
       ├─ rawConvertible = true (始终)
       ├─ 按 minSequenceNumber 排序
       └─ BinPacking (装箱)
```

**与其他系统对比**:

| 系统 | Split 策略 | rawConvertible 支持 | Section 划分 | 装箱算法 |
|------|-----------|-------------------|-------------|---------|
| **Paimon** | 主键表/追加表分离 | 支持 (条件性) | IntervalPartition | BinPacking |
| **Iceberg** | FileScanTask | 不支持 (总是 merge) | 不需要 | 按大小拆分 |
| **Delta Lake** | AddFile | 不支持 (总是 merge) | 不需要 | 按大小拆分 |
| **Hudi** | FileSlice | 支持 (MOR/COW) | 不需要 | 按大小拆分 |

**Paimon 的独特之处**:
- **rawConvertible 优化** - 可以跳过 merge 直接读取，显著降低读取成本
- **IntervalPartition** - 将有 overlap 的文件划分为不重叠的 Section，提高并行度
- **主键表和追加表的差异化策略** - 根据表类型选择最优的 Split 策略

### 设计理念

**为什么这样设计**:

1. **rawConvertible 是关键优化** - 跳过 merge 可以节省大量 CPU 和内存：
   ```
   需要 merge 的 Split:
   - 读取所有文件的数据
   - 使用 SortMergeReader 做多路归并
   - 应用 merge 引擎 (DEDUPLICATE/PARTIAL_UPDATE/AGGREGATE)
   - CPU 和内存开销大
   
   rawConvertible 的 Split:
   - 直接读取文件，应用 DV 过滤
   - 无需 merge，CPU 和内存开销小
   - 读取速度快 2-10 倍
   ```

2. **IntervalPartition 提高并行度** - 将有 overlap 的文件划分为不重叠的 Section：
   ```
   不划分 Section:
   - 所有有 overlap 的文件在一个 Split 中
   - 并行度 = bucket 数量
   
   划分 Section:
   - 不同 Section 可以并行处理
   - 并行度 = bucket 数量 * 平均 Section 数
   ```

3. **BinPacking 保持顺序** - 文件的顺序反映了数据的时间或 key 范围顺序：
   ```java
   // 有序装箱: 保持文件顺序
   packForOrdered(files, weightFunc, targetSplitSize)
   // 好处: 同一 Split 内的文件在 key 范围上连续，merge 效率高
   
   // 无序装箱: 按权重贪心分配
   packForFixedBinNumber(files, weightFunc, binNumber)
   // 好处: 负载更均衡，但可能破坏 key 范围的连续性
   ```

4. **openFileCost 考虑固定成本** - 打开文件有固定的成本：
   ```
   权重函数:
   weight = max(file.fileSize(), openFileCost)
   
   场景1: 大文件 (100MB)
   - weight = 100MB (文件大小主导)
   
   场景2: 小文件 (1MB)
   - weight = 4MB (openFileCost 主导)
   - 避免一个 Split 包含过多小文件
   ```

**权衡取舍**:

| 设计选择 | 优势 | 劣势 | 取舍原因 |
|---------|------|------|---------|
| rawConvertible 优化 | 读取速度快 2-10 倍 | 需要严格的条件判定 | 批量读取是核心场景，优化收益大 |
| IntervalPartition | 提高并行度 | 算法复杂度 O(N log N) | 并行度对大表查询至关重要 |
| BinPacking 保持顺序 | Merge 效率高 | 可能不如贪心均衡 | Key 范围连续性优先 |
| openFileCost | 避免过多小文件 | 需要手动调优 | 小文件场景下打开文件成本显著 |

**架构演进**:

1. **早期版本** - 简单的按大小拆分，无 rawConvertible 优化
2. **引入 rawConvertible** - 可以跳过 merge 直接读取，显著提升性能
3. **引入 IntervalPartition** - 将有 overlap 的文件划分为不重叠的 Section
4. **引入 BinPacking** - 使用装箱算法均衡 Split 大小
5. **引入 openFileCost** - 考虑打开文件的固定成本
6. **与 DV 集成** - DV 启用时，所有 Split 都是 rawConvertible

**业界对比**:

- **Iceberg** - 简单的按大小拆分，无 rawConvertible 优化
- **Delta Lake** - 简单的按大小拆分，无 rawConvertible 优化
- **Hudi** - MOR (Merge-On-Read) 和 COW (Copy-On-Write) 两种模式，COW 类似 rawConvertible

**Paimon 的创新点**:
- **条件性 rawConvertible** - 根据文件的 level、DV、merge 引擎等条件判定
- **IntervalPartition 的两层划分** - 外层 Section 划分 + 内层 SortedRun 划分
- **主键表和追加表的差异化策略** - 追加表始终 rawConvertible，主键表条件性

### 5.1 rawConvertible 判定机制

**源码路径**: `paimon-core/.../table/source/SplitGenerator.java`, `MergeTreeSplitGenerator.java`

`rawConvertible` 是 Paimon 的一个关键优化标记，决定一个 Split 是否可以"原始读取"(跳过 merge)。

**SplitGroup** (见 `SplitGenerator.java:34-52`):
```java
class SplitGroup {
    public final List<DataFileMeta> files;
    public final boolean rawConvertible;  // true = 可跳过 merge, 直接读原始文件
}
```

**MergeTreeSplitGenerator.alwaysRawConvertible()** (见 `MergeTreeSplitGenerator.java:64-66`):
```java
public boolean alwaysRawConvertible() {
    return deletionVectorsEnabled || mergeEngine == FIRST_ROW;
}
```

**为什么 DV 启用时总是 rawConvertible**: 启用 Deletion Vectors 后，更新和删除操作通过标记 DV 实现，不再需要读取多个文件做 merge。读取时只需应用 DV 过滤掉已删除的行。

**splitForBatch 中的 rawConvertible 判定** (见 `MergeTreeSplitGenerator.java:69-81`):
```java
boolean rawConvertible = files.stream().allMatch(file -> file.level() != 0 && withoutDeleteRow(file));
boolean oneLevel = files level 集合大小 == 1;
if (rawConvertible && (deletionVectorsEnabled || mergeEngine == FIRST_ROW || oneLevel)) {
    return BinPacking.packForOrdered(files, weightFunc, targetSplitSize)
        .stream().map(SplitGroup::rawConvertibleGroup).collect(toList());
}
```

判定条件:
1. 所有文件都不在 level 0 (level 0 文件间可能有 key 重叠)
2. 所有文件没有删除行
3. 满足以下之一: DV 启用 / FIRST_ROW 引擎 / 所有文件在同一 level (同 level 文件无 overlap)

### 5.2 IntervalPartition Section 划分

**源码路径**: `paimon-core/.../mergetree/compact/IntervalPartition.java`

IntervalPartition 将一组有 key 范围重叠可能性的文件划分为 **Section**(不重叠的段)和 **SortedRun**(段内的有序运行)。

**两层分区算法**:

**外层: Section 划分** (见 `IntervalPartition.java:67-91`)

文件按 minKey 排序后，维护当前 section 的右边界 bound:
- 如果下一个文件的 minKey > bound -> 新建 section
- 否则加入当前 section, 更新 bound = max(bound, file.maxKey)

这确保了同一 section 内的文件 key 范围可能重叠，不同 section 间的 key 范围不重叠。

**内层: SortedRun 划分** (见 `IntervalPartition.java:93-125`)

使用贪心+优先队列算法，对每个 section 内的文件进一步分组:
- 维护多个 SortedRun，用优先队列按各 run 的最后文件 maxKey 排序
- 对每个文件: 如果 minKey > 最小 maxKey 的 run 的最后文件 maxKey -> 追加到该 run
- 否则 -> 创建新 run

**为什么 Section 间不重叠很重要**: Split 中包含的所有文件需要一起做 merge 排序。如果能将文件划分为不重叠的 Section，不同 Section 可以独立并行处理，提高查询并行度。

### 5.3 BinPacking 装箱算法

**源码路径**: `paimon-common/.../utils/BinPacking.java`

Paimon 使用两种装箱算法:

**packForOrdered** (有序装箱, 见 `BinPacking.java:33-58`):
- 保持输入顺序，顺序装入
- 当累积权重超过 targetWeight 时开新箱
- 用于 Split 生成(保持文件的时间/范围顺序)

**packForFixedBinNumber** (固定箱数装箱, 见 `BinPacking.java:61-81`):
- 先按权重排序，再用优先队列贪心分配到指定数量的箱
- 用于并行度固定的场景

**为什么 Split 生成使用有序装箱**: 文件的顺序反映了数据的时间或 key 范围顺序。保持顺序使得同一 Split 内的文件在 key 范围上连续，有利于 merge 效率和缓存局部性。

### 5.4 主键表 vs 追加表的 Split 策略

**源码路径**: `AppendOnlySplitGenerator.java`, `MergeTreeSplitGenerator.java`

| 维度 | MergeTreeSplitGenerator | AppendOnlySplitGenerator |
|------|------------------------|-------------------------|
| 输入排序 | 无(由 IntervalPartition 处理) | 按 minSequenceNumber 排序 |
| Section 划分 | 必要(处理 key overlap) | 不需要(追加表文件无 overlap) |
| rawConvertible | 条件性(取决于 level/DV/引擎) | 始终 true |
| 装箱单位 | Section(一组文件) | 单个文件 |
| 流式 Split | 不拆分(所有文件在一个 Split) | BUCKET_UNAWARE 模式拆分, 其他不拆分 |
| 权重函数 | `Math.max(file.fileSize(), openFileCost)` | 同左 |

## 6. Lookup 点查优化

### 解决什么问题

**核心业务问题**: 流式 Join 和维表查询需要根据主键快速查找单条记录。如何在数据湖的文件存储上实现毫秒级的点查性能？

**没有这个设计的后果**:
- 每次点查需要扫描多个数据文件，延迟达到秒级
- 无法支持实时 Join 场景 (如订单流 Join 用户维表)
- 流式计算的吞吐量受限于点查性能

**实际场景**:
```sql
-- 场景1: 流式 Join
SELECT o.*, u.name, u.city 
FROM order_stream o 
JOIN user_dim FOR SYSTEM_TIME AS OF o.proctime AS u 
ON o.user_id = u.user_id;
-- 每条订单需要点查用户维表，QPS 可能达到数千

-- 场景2: CDC 同步后的点查
-- 从 MySQL CDC 同步到 Paimon，应用需要根据主键查询最新数据
SELECT * FROM user_profile WHERE user_id = 'xxx';

-- 场景3: 实时特征查询
-- 机器学习推理时，根据用户ID查询特征
SELECT features FROM user_features WHERE user_id = 'xxx';
```

### 有什么坑

**误区陷阱**:
1. **误以为 Lookup 是免费的** - 需要本地磁盘空间存储 Lookup 文件
2. **忽略缓存淘汰策略** - 缓存空间不足会导致频繁重建 Lookup 文件
3. **混淆 Lookup 和批量扫描** - Lookup 优化只对点查有效，范围查询无效

**错误配置**:
```properties
# 错误1: 缓存空间过小
lookup.cache-max-disk-size=1GB  # 大表场景下，缓存频繁淘汰

# 错误2: 缓存保留时间过短
lookup.cache-file-retention=5min  # 频繁重建 Lookup 文件

# 错误3: 禁用 Bloom Filter
lookup.bloom-filter.enabled=false  # 无法快速判断 key 是否存在

# 正确配置:
lookup.cache-max-disk-size=50GB  # 根据本地磁盘大小设置
lookup.cache-file-retention=1h  # 合理的保留时间
lookup.bloom-filter.enabled=true  # 启用 Bloom Filter
```

**生产环境注意事项**:
1. **本地磁盘空间** - Lookup 文件缓存在本地，需要足够的磁盘空间
2. **任务恢复时间** - 新启动的节点需要重建 Lookup 文件，可能需要数分钟
3. **远程 Lookup 文件** - 预构建并上传远程 Lookup 文件可以加速恢复

**性能陷阱**:
```java
// 陷阱1: level 0 文件过多导致查找慢
// level 0 需要遍历所有文件，level 1+ 可以二分查找
// 解决: 及时触发 Compaction

// 陷阱2: Bloom Filter 误报导致读取不必要的文件
// 解决: 调整 Bloom Filter 的 fpp 参数

// 陷阱3: 本地磁盘 I/O 成为瓶颈
// 解决: 使用 SSD 或增大缓存空间
```

### 核心概念解释

**术语定义**:

1. **Lookup** - 根据主键查找单条记录的操作，也称为点查 (Point Query)
2. **LookupLevels** - 基于 LSM-Tree 层级结构的点查实现
3. **LookupFile** - 本地 SST 格式的 Lookup 文件，支持快速点查
4. **Bloom Filter** - 用于快速判断 key 是否存在的概率型数据结构
5. **远程 Lookup 文件** - 预构建并上传到远程存储的 Lookup 文件，用于加速任务恢复
6. **Caffeine 缓存** - 用于管理本地 Lookup 文件的 LRU 缓存

**概念关系**:
```
LookupLevels (多层查找)
  ├─ Level 0 查找 (遍历所有文件)
  │    ├─ 按 sequence 降序遍历
  │    ├─ 检查 key 是否在 [minKey, maxKey] 范围内
  │    └─ 找到第一个匹配即返回
  ├─ Level 1+ 查找 (二分查找)
  │    ├─ 二分查找定位目标文件
  │    └─ 只需查询一个文件
  └─ LookupFile (本地 SST 文件)
       ├─ Bloom Filter (快速判断 key 是否存在)
       ├─ SST 格式 (支持快速点查)
       └─ Caffeine 缓存 (LRU 淘汰)
```

**与其他系统对比**:

| 系统 | 点查支持 | 实现方式 | 缓存机制 | 远程文件 |
|------|---------|---------|---------|---------|
| **Paimon** | 原生支持 | LookupLevels + 本地 SST | Caffeine LRU | 支持 |
| **Iceberg** | 不支持 | 需要外部 KV 层 | - | - |
| **Delta Lake** | 不支持 | 需要外部 KV 层 | - | - |
| **Hudi** | 支持 | HoodieKey Index | 不支持 | 不支持 |
| **RocksDB** | 原生支持 | LSM-Tree | Block Cache | 不需要 |

**Paimon 的独特之处**:
- **原生点查支持** - 无需外部 KV 存储，直接在数据湖文件上实现点查
- **本地 SST 缓存** - 将远程文件转换为本地 SST 格式，利用本地 SSD 的随机读取能力
- **远程 Lookup 文件** - 预构建并上传，加速任务恢复
- **与 LSM-Tree 的集成** - 利用 level 0/1+ 的分层结构，优化查找策略

### 设计理念

**为什么这样设计**:

1. **本地 SST 缓存是关键** - 远程文件系统不适合随机读取：
   ```
   远程文件系统 (S3/HDFS):
   - 随机读取延迟: 10-100ms
   - 不适合点查场景
   
   本地 SST 文件:
   - 随机读取延迟: 0.1-1ms (SSD)
   - 适合点查场景
   - Bloom Filter 可以快速判断 key 是否存在
   ```

2. **利用 LSM-Tree 的分层结构** - 不同 level 的查找策略不同：
   ```
   Level 0 (无序):
   - 文件间可能有 key 重叠
   - 需要遍历所有文件
   - 按 sequence 降序，最新的先匹配
   
   Level 1+ (有序):
   - 文件间无 key 重叠
   - 可以二分查找定位目标文件
   - 只需查询一个文件
   ```

3. **Caffeine 缓存管理** - 自动淘汰和磁盘空间控制：
   ```java
   Cache<String, LookupFile> cache = Caffeine.newBuilder()
       .expireAfterAccess(fileRetention)  // 访问超时淘汰
       .maximumWeight(maxDiskSize)        // 磁盘空间限制
       .weigher(LookupFile::fileWeigh)    // 权重 = 文件大小
       .removalListener(LookupFile::removalCallback)  // 淘汰时删除本地文件
       .build();
   ```

4. **远程 Lookup 文件加速恢复** - 避免重建 Lookup 文件：
   ```
   无远程 Lookup 文件:
   - 新节点启动时，需要从数据文件逐条读取
   - 构建 SST 文件和 Bloom Filter
   - 可能需要数分钟
   
   有远程 Lookup 文件:
   - 直接下载预构建的 Lookup 文件
   - 只需要数秒
   ```

**权衡取舍**:

| 设计选择 | 优势 | 劣势 | 取舍原因 |
|---------|------|------|---------|
| 本地 SST 缓存 | 点查延迟低 (毫秒级) | 需要本地磁盘空间 | 点查性能是核心需求 |
| Caffeine LRU 缓存 | 自动淘汰，控制磁盘空间 | 淘汰后需要重建 | 平衡性能和资源占用 |
| 远程 Lookup 文件 | 加速任务恢复 | 需要额外存储空间 | 任务恢复时间是关键指标 |
| Bloom Filter | 快速判断 key 是否存在 | 有误报 (false positive) | 误报率可接受，性能提升显著 |

**架构演进**:

1. **早期版本** - 无点查支持，需要外部 KV 存储
2. **引入 LookupLevels** - 基于 LSM-Tree 的点查实现
3. **引入本地 SST 缓存** - 将远程文件转换为本地 SST 格式
4. **引入 Caffeine 缓存** - 自动淘汰和磁盘空间控制
5. **引入远程 Lookup 文件** - 预构建并上传，加速任务恢复
6. **引入 Bloom Filter** - 快速判断 key 是否存在

**业界对比**:

- **Hudi** - 支持点查，但无本地缓存机制，性能较低
- **Iceberg/Delta Lake** - 不支持点查，需要外部 KV 存储 (如 Redis)
- **RocksDB** - 原生支持点查，但不是数据湖系统

**Paimon 的创新点**:
- **数据湖上的原生点查** - 无需外部 KV 存储，直接在数据湖文件上实现
- **本地 SST 缓存 + 远程 Lookup 文件** - 平衡性能和恢复时间
- **与 LSM-Tree 的深度集成** - 利用分层结构优化查找策略

### 6.1 LookupLevels 多层查找

**源码路径**: `paimon-core/.../mergetree/LookupLevels.java`, `LookupUtils.java`

LookupLevels 实现了基于 LSM-Tree 层级结构的高效点查。

**查找策略** (见 `LookupUtils.java:35-57`):
```java
public static <T> T lookup(Levels levels, InternalRow key, int startLevel, ...) {
    for (int i = startLevel; i < levels.numberOfLevels(); i++) {
        if (i == 0) {
            result = level0Lookup(key, levels.level0());  // Level 0: 遍历所有文件
        } else {
            result = lookup(key, levels.runOfLevel(i));   // Level 1+: 二分查找
        }
        if (result != null) break;  // 找到即返回(新数据在上层)
    }
}
```

**Level 0 查找** (见 `LookupUtils.java:59-77`):
- Level 0 的文件无序(TreeSet 按 sequence 排序)
- 需要**遍历所有文件**，检查 key 是否在 [minKey, maxKey] 范围内
- 找到第一个匹配即返回(TreeSet 按 sequence 降序，最新的先匹配)

**Level 1+ 查找** (见 `LookupUtils.java:79-119`):
- 每个 Level 是一个 SortedRun(文件间无 overlap, 按 key 有序)
- 使用**二分查找**定位目标文件: 找到 maxKey >= target 的最小文件
- 只需查询一个文件

**为什么从上层往下查**: LSM-Tree 中，上层数据更新，下层数据更旧。对同一 key，上层的值是最新的。从上往下查找，第一个命中的就是最新值，无需继续。

### 6.2 LookupFile 缓存机制

**源码路径**: `paimon-core/.../mergetree/LookupFile.java`, `LookupLevels.java:182-207`

每个数据文件会被转换为本地 Lookup 文件(SST 格式)，存储在本地磁盘上。

**LookupFile 缓存** (见 `LookupFile.java:123-132`):
```java
public static Cache<String, LookupFile> createCache(Duration fileRetention, MemorySize maxDiskSize) {
    return Caffeine.newBuilder()
        .expireAfterAccess(fileRetention)  // 访问超时淘汰
        .maximumWeight(maxDiskSize.getKibiBytes())  // 磁盘空间限制(KiB)
        .weigher(LookupFile::fileWeigh)    // 权重 = 文件大小(KiB)
        .removalListener(LookupFile::removalCallback)  // 淘汰时删除本地文件
        .build();
}
```

**创建 Lookup 文件的过程** (见 `LookupLevels.java:239-272`):
1. 创建本地临时文件
2. 尝试下载远程预构建的 Lookup 文件(如果存在)
3. 若无远程文件，从数据文件逐条读取 KV，写入 LookupStoreWriter
4. 写入时同时构建 Bloom Filter

**为什么缓存 Lookup 文件到本地**: 远程文件系统(S3/HDFS)的随机读取延迟高，不适合点查场景。将文件转换为本地 SST 格式后:
- 利用本地 SSD 的随机读取能力
- 附带 Bloom Filter 快速判断 key 是否存在
- Caffeine 缓存管理 LRU 淘汰和磁盘空间

### 6.3 Bloom Filter 加速

Lookup 场景下的 Bloom Filter 与文件索引的 Bloom Filter 不同:
- **文件索引 Bloom Filter**: 在 Scan 阶段判断文件是否可能包含匹配数据
- **Lookup Bloom Filter**: 在本地 LookupFile 中，通过 `LookupStoreWriter` 构建，用于在读取 SST 文件前快速排除不存在的 key

```java
// LookupLevels.createSstFileFromDataFile
LookupStoreWriter kvWriter = lookupStoreFactory.createWriter(localFile, bfGenerator.apply(file.rowCount()));
```

`bfGenerator` 根据文件行数动态生成 Bloom Filter Builder，行数越多，Bloom Filter 越大，误报率越低。

### 6.4 远程 Lookup 文件下载

**源码路径**: `LookupLevels.java:209-233`, `274-307`

Paimon 支持将 Lookup 文件预先上传到远程存储，新启动的节点可以直接下载而不需要从数据文件重建:

```java
Optional<RemoteSstFile> remoteSstFile = remoteSst(file);
// 远程文件命名: {dataFileName}.{length}.{processorId}.{serVersion}.lookup
```

**为什么需要远程 Lookup 文件**: 在 Flink 任务恢复或扩容时，新 TM 节点需要重建 Lookup 文件。如果数据文件很大，逐条读取并重建 SST 文件非常耗时。预构建并上传的远程 Lookup 文件可以直接下载使用，大幅缩短恢复时间。

## 7. SortMergeReader 优化

### 解决什么问题

**核心业务问题**: 主键表的查询需要合并多个文件中同一主键的不同版本。如何高效地进行多路归并排序，同时处理相同 key 的合并逻辑？

**没有这个设计的后果**:
- 多路归并使用简单的堆排序，每次 pop/push 都需要 O(log N) 次比较
- 相同 key 的多个版本需要重复比较，浪费 CPU
- 大量文件参与合并时，性能急剧下降

**实际场景**:
```sql
-- 场景1: 批量读取主键表
SELECT * FROM user_profile WHERE city = 'Beijing';
-- 需要合并多个文件中同一 user_id 的不同版本

-- 场景2: Compaction
-- 合并 5-10 个 SortedRun，每个 run 包含数百万条记录

-- 场景3: 流式读取
-- 实时读取主键表的最新数据，需要合并 level 0 和 level 1 的文件
```

### 有什么坑

**误区陷阱**:
1. **误以为 LoserTree 总是比 MinHeap 快** - 少量 reader 时，MinHeap 可能更简单高效
2. **忽略 Spill 机制的重要性** - 大量文件合并时，内存不足会导致 OOM
3. **混淆 Compaction 和查询的 SortMerge** - 两者的性能要求不同

**错误配置**:
```properties
# 错误1: 使用 MinHeap 处理大量文件
sort-engine=min-heap  # 10+ 个文件时，LoserTree 更优

# 错误2: Spill 阈值过高
sort.spill-threshold=100  # 可能导致打开过多文件句柄

# 错误3: 排序缓冲区过小
sort-buffer-size=64MB  # 大文件排序时频繁 Spill

# 正确配置:
sort-engine=loser-tree  # 大量文件时使用 LoserTree
sort.spill-threshold=20  # 合理的 Spill 阈值
sort-buffer-size=256MB  # 根据内存大小调整
```

**生产环境注意事项**:
1. **内存压力** - 每个 reader 需要缓冲一个 batch 的数据
2. **文件句柄数** - 大量文件合并时，注意"too many open files"错误
3. **CPU 使用率** - 多路归并是 CPU 密集型操作

**性能陷阱**:
```java
// 陷阱1: 相同 key 的重复比较
// MinHeap 每次 pop 相同 key 都需要重新调整堆
// 解决: 使用 LoserTree 的同 key 快速路径

// 陷阱2: 大量小 batch 导致频繁调整
// 解决: 增大 batch 大小

// 陷阱3: Spill 文件过多导致 I/O 瓶颈
// 解决: 增大排序缓冲区或减少并行度
```

### 核心概念解释

**术语定义**:

1. **SortMergeReader** - 多路归并读取器，合并多个有序流为一个有序流
2. **MinHeap** - 最小堆，每次 pop 最小元素，需要 O(log N) 次比较
3. **LoserTree** - 败者树，一种优化的多路归并数据结构，常数更小
4. **同 key 快速路径** - LoserTree 的优化，相同 key 的合并无需重复比较
5. **Spill** - 当内存不足时，将数据写入磁盘临时文件
6. **外部排序** - 使用磁盘辅助的排序算法，适合大数据量

**概念关系**:
```
SortMergeReader (多路归并)
  ├─ SortMergeReaderWithMinHeap (MinHeap 实现)
  │    ├─ PriorityQueue<Element>
  │    ├─ 多 batch 模式
  │    └─ 适用场景: 少量 reader、大 batch
  └─ SortMergeReaderWithLoserTree (LoserTree 实现)
       ├─ LoserTree (败者树)
       │    ├─ 6 种状态 (WINNER_WITH_NEW_KEY, ...)
       │    ├─ 同 key 快速路径
       │    └─ 数组存储 (缓存友好)
       ├─ 单 batch 模式
       └─ 适用场景: 大量 reader、需要最优性能
```

**与其他系统对比**:

| 系统 | 多路归并实现 | 同 key 优化 | Spill 支持 | 可配置 |
|------|------------|-----------|-----------|---------|
| **Paimon** | MinHeap / LoserTree | 支持 (LoserTree) | 支持 | 支持 |
| **Iceberg** | MinHeap | 不支持 | 不支持 | 不支持 |
| **Delta Lake** | MinHeap | 不支持 | 不支持 | 不支持 |
| **Hudi** | MinHeap | 不支持 | 支持 | 不支持 |
| **RocksDB** | LoserTree | 支持 | 支持 | 不支持 |

**Paimon 的独特之处**:
- **两种实现可选** - MinHeap 和 LoserTree，根据场景选择
- **同 key 快速路径** - LoserTree 的优化，相同 key 的合并只需 O(N) 次操作
- **Spill 优化** - 小文件先 spill，减少文件句柄数

### 设计理念

**为什么这样设计**:

1. **提供两种实现而非单一方案** - 不同场景的最优选择不同：
   ```
   MinHeap 适用场景:
   - 少量 reader (2-5 个)
   - 大 batch (每次读取数千行)
   - 实现简单，易于调试
   
   LoserTree 适用场景:
   - 大量 reader (5-10+ 个)
   - 需要最优性能
   - 同 key 出现频率高 (Compaction 场景)
   ```

2. **LoserTree 的同 key 快速路径** - 减少重复比较：
   ```
   MinHeap 处理相同 key:
   - pop 第一个版本: O(log N)
   - pop 第二个版本: O(log N)
   - pop 第三个版本: O(log N)
   - 总计: O(M log N)，M 是相同 key 的版本数
   
   LoserTree 处理相同 key:
   - pop 第一个版本: O(log N)
   - pop 后续版本: O(1) (快速路径)
   - 总计: O(log N + M)
   ```

3. **Spill 机制避免 OOM** - 大量文件合并时的内存保护：
   ```
   无 Spill:
   - 所有文件同时打开
   - 每个文件缓冲一个 batch
   - 内存需求: N * batch_size
   - 可能 OOM
   
   有 Spill:
   - 小文件先 spill 到磁盘
   - 只保留大文件直接读取
   - 内存需求: K * batch_size (K < N)
   - 避免 OOM
   ```

4. **LoserTree 的状态机设计** - 高效处理复杂逻辑：
   ```
   6 种状态:
   - WINNER_WITH_NEW_KEY: 赢家，新 key
   - WINNER_WITH_SAME_KEY: 赢家，与全局赢家相同 key
   - WINNER_POPPED: 赢家已被弹出，等待推进
   - LOSER_WITH_NEW_KEY: 败者，新 key
   - LOSER_WITH_SAME_KEY: 败者，与全局赢家相同 key
   - LOSER_POPPED: 败者已被弹出
   
   状态转换:
   - popWinner() -> WINNER_POPPED
   - adjust() -> 根据新数据更新状态
   - 同 key 快速路径: 直接交换位置，无需从底部逐级比较
   ```

**权衡取舍**:

| 设计选择 | 优势 | 劣势 | 取舍原因 |
|---------|------|------|---------|
| 两种实现 | 适应不同场景 | 维护成本高 | 性能差异显著，值得投入 |
| LoserTree | 性能最优 | 实现复杂 | Compaction 是热路径，性能优先 |
| 同 key 快速路径 | 减少重复比较 | 状态机复杂 | LSM-Tree 中同 key 频繁出现 |
| Spill 机制 | 避免 OOM | 额外 I/O 开销 | 稳定性优先 |
| 小文件先 spill | 减少文件句柄数 | 小文件需要额外 I/O | 避免"too many open files" |

**架构演进**:

1. **早期版本** - 只有 MinHeap 实现
2. **引入 LoserTree** - 提供更优的多路归并性能
3. **引入同 key 快速路径** - 优化相同 key 的合并
4. **引入 Spill 机制** - 避免大量文件合并时的 OOM
5. **引入小文件先 spill** - 减少文件句柄数
6. **可配置的 sort-engine** - 用户可以根据场景选择

**业界对比**:

- **RocksDB** - 使用 LoserTree (称为 MergingIterator)，但不可配置
- **LevelDB** - 使用 MinHeap
- **Iceberg/Delta Lake** - 使用 MinHeap，无 LoserTree 优化
- **Hudi** - 使用 MinHeap，支持 Spill

**Paimon 的创新点**:
- **可配置的多路归并引擎** - 用户可以根据场景选择 MinHeap 或 LoserTree
- **同 key 快速路径** - LoserTree 的优化，相同 key 的合并只需 O(N) 次操作
- **小文件先 spill** - 减少文件句柄数，避免"too many open files"

### 7.1 LoserTree vs MinHeap 对比

**源码路径**: `paimon-core/.../mergetree/compact/SortMergeReader.java`

Paimon 提供两种 SortMerge 引擎，通过 `sort-engine` 配置选择:

```java
static <T> SortMergeReader<T> createSortMergeReader(..., SortEngine sortEngine) {
    switch (sortEngine) {
        case MIN_HEAP:   return new SortMergeReaderWithMinHeap<>(...);
        case LOSER_TREE: return new SortMergeReaderWithLoserTree<>(...);
    }
}
```

| 维度 | MinHeap | LoserTree |
|------|---------|-----------|
| 时间复杂度 | O(log N) per element | O(log N) per element, 但常数更小 |
| Batch 模式 | 多 batch(每次 readBatch() 返回一个新的 RecordIterator) | 单 batch(readBatch() 仅返回一个 RecordIterator，后续通过 next() 迭代) |
| 内存效率 | 每个 reader 的 batch 独立消费，可能产生多个中间 batch | 全局管理，所有 reader 的数据在一个 batch 中处理 |
| 相同 key 处理 | 通过 PriorityQueue poll+collect 逐个处理 | 通过状态机快速路径批量处理 |
| 适用场景 | 少量 reader、大 batch | 大量 reader、需要最优合并性能 |

**MinHeap 实现特点** (见 `SortMergeReaderWithMinHeap.java`):
- 使用 `PriorityQueue<Element>` 维护所有 reader 的当前记录
- 每次 readBatch() 调用时，从所有 reader 读取第一条记录放入堆
- 通过 SortMergeIterator 逐条 poll 最小元素，当某个 reader 的 batch 耗尽时，重新读取该 reader 的下一个 batch
- 可能产生多个 RecordIterator(多 batch)

**LoserTree 实现特点** (见 `SortMergeReaderWithLoserTree.java`):
- readBatch() 仅返回一个 RecordIterator，后续通过 next() 逐条获取记录
- 通过 `loserTree.adjustForNextLoop()` + `popWinner()` 获取下一条记录
- 同 key 合并通过快速路径完成，无需重复比较

### 7.2 LoserTree 核心状态机

**源码路径**: `paimon-core/.../mergetree/compact/LoserTree.java`

LoserTree 相比标准败者树的关键优化是**同 key 快速路径**。在 LSM-Tree 中，多个 reader 可能包含相同 key 的不同版本。LoserTree 通过 6 种状态高效管理同 key 的合并:

```
State 枚举 (见 LoserTree.java:338-356):
  WINNER_WITH_NEW_KEY   -- 赢家, 新 key
  WINNER_WITH_SAME_KEY  -- 赢家, 与全局赢家相同 key
  WINNER_POPPED         -- 赢家已被弹出, 等待推进
  LOSER_WITH_NEW_KEY    -- 败者, 新 key
  LOSER_WITH_SAME_KEY   -- 败者, 与全局赢家相同 key
  LOSER_POPPED          -- 败者已被弹出(用于 adjustForNextLoop)
```

**popWinner() + adjust() 的同 key 快速路径** (见 `LoserTree.java:105-115, 143-160`):

当 winner 被 pop 后(状态变为 WINNER_POPPED)，如果 `firstSameKeyIndex >= 0`(树中有相同 key 的节点)，直接交换位置，无需从底部逐级比较。这使得同 key 的 N 个版本合并只需 O(N) 次操作，而非 O(N log N)。

**为什么 LoserTree 更适合 Compaction**: Compaction 时通常有多个 SortedRun 参与合并(可能 5-10 个)，且同 key 出现在多个 run 中的概率较高。LoserTree 的同 key 快速路径和更优的缓存友好性(数组而非堆节点对象)在此场景下性能更佳。

### 7.3 Spill 机制与外部排序

**源码路径**: `paimon-common/.../sort/BinaryExternalSortBuffer.java`, `ClusteringFileRewriter.java`

ClusteringFileRewriter 在 Phase 1 排序时使用 `BinaryExternalSortBuffer`，这是一个支持磁盘 Spill 的外部排序缓冲区:

1. **内存阶段**: 数据写入内存中的 `BinaryInMemorySortBuffer`
2. **Spill 阶段**: 当内存不足时，将排序后的数据 spill 到磁盘通道
3. **Merge 阶段**: 读取时使用 `BinaryExternalMerger` 做多路归并

Phase 2 的 `mergeAndRewriteFiles` 有独立的 Spill 策略 (见 `ClusteringFileRewriter.java:234-244`):
- 当输入文件数 > `spillThreshold` 时，按文件大小排序
- 较小的文件先 spill 到行式压缩临时文件(`ChannelWriterOutputView`)
- 保留较大的文件直接读取，减少 Spill I/O

**为什么 Spill 较小的文件而非较大的**: 小文件的 Spill 开销低(读写字节少)，而大文件保持原始列式格式读取效率更高(列式格式的向量化读取优势保留)。

## 8. DV 对查询的影响

### 解决什么问题

**核心业务问题**: 传统的 merge-on-read 需要读取多个文件并合并，即使只有少量行被删除或更新。如何在不做全量 merge 的情况下，高效地过滤已删除的行？

**没有这个设计的后果**:
- 每次查询都需要 merge 多个文件，即使大部分数据没有变化
- 删除和更新操作需要重写整个文件，写放大极高
- 无法支持高频的删除和更新操作

**实际场景**:
```sql
-- 场景1: 高频删除
DELETE FROM orders WHERE order_id = 'xxx';
-- 传统方式: 重写整个文件，写放大高
-- DV 方式: 只标记 DV 位图，写放大低

-- 场景2: 部分更新
UPDATE user_profile SET last_login = NOW() WHERE user_id = 'xxx';
-- 传统方式: 读取旧版本 + 新版本，merge 后写入
-- DV 方式: 标记旧版本为删除，写入新版本

-- 场景3: 批量读取
SELECT * FROM orders WHERE dt = '2024-01-01';
-- 传统方式: 需要 merge 多个文件
-- DV 方式: 直接读取文件，应用 DV 过滤
```

### 有什么坑

**误区陷阱**:
1. **误以为 DV 是免费的** - DV 需要额外的存储空间和读取开销
2. **忽略 DV 的累积效应** - 大量删除后，DV 位图可能很大
3. **混淆 DV 和物理删除** - DV 只是标记删除，数据仍在文件中

**错误配置**:
```properties
# 错误1: 在低删除率场景下启用 DV
deletion-vectors.enabled=true  # 删除率 < 1%，DV 收益不明显

# 错误2: DV 累积过多不触发 Compaction
# 大量删除后，文件中大部分行被标记删除，但仍占用空间
# 解决: 定期触发 Compaction，物理删除已标记的行

# 正确配置:
deletion-vectors.enabled=true  # 高频删除/更新场景
compaction.min.file-num=3  # 及时触发 Compaction
```

**生产环境注意事项**:
1. **DV 存储空间** - 每个文件的 DV 位图需要额外存储
2. **DV 读取开销** - 每次读取文件都需要加载和应用 DV
3. **Compaction 策略** - 需要定期触发 Compaction，物理删除已标记的行

**性能陷阱**:
```java
// 陷阱1: DV 位图过大
// 大量删除后，DV 位图可能达到 MB 级别
// 解决: 触发 Compaction，重写文件

// 陷阱2: DV 与文件索引的协同不当
// 如果文件索引返回的行号已被 DV 标记删除，仍需要读取
// 解决: 在 FileIndexEvaluator 中先应用 DV

// 陷阱3: Limit 下推失效
// DV 启用时，rowCount 不等于有效行数，无法精确 Limit
// 解决: 禁用 Limit 下推
```

### 核心概念解释

**术语定义**:

1. **Deletion Vectors (DV)** - 删除向量，使用位图标记数据文件中已删除的行
2. **BitmapDeletionVector** - 基于 RoaringBitmap32 的 DV 实现
3. **Merge-on-Read (MOR)** - 读取时合并，传统的主键表读取方式
4. **Copy-on-Write (COW)** - 写入时复制，DV 启用后的读取方式
5. **rawConvertible** - DV 启用时，所有 Split 都是 rawConvertible
6. **ApplyDeletionVectorReader** - 应用 DV 过滤的读取器

**概念关系**:
```
DV 在查询中的作用
  ├─ Scan 阶段
  │    ├─ rawConvertible 判定 (DV 启用时总是 true)
  │    ├─ Limit 下推禁用 (rowCount 不等于有效行数)
  │    └─ wholeBucketFilter 影响 (DV 启用时可以逐文件过滤)
  ├─ Read 阶段
  │    ├─ ApplyDeletionVectorReader (应用 DV 过滤)
  │    ├─ 基于行位置检查 (isDeleted(position))
  │    └─ 跳过已删除的行
  └─ 与文件索引协同
       ├─ createBaseSelection (全行位图 AND NOT DV)
       ├─ 文件索引评估 (AND baseSelection)
       └─ 最终需要读取的行 (位图级协同)
```

**与其他系统对比**:

| 系统 | 删除机制 | 位图格式 | rawConvertible | 与索引协同 |
|------|---------|---------|---------------|-----------|
| **Paimon** | Deletion Vectors | RoaringBitmap32 | 支持 | 位图级协同 |
| **Iceberg** | Delete Files | Parquet | 不支持 | 不支持 |
| **Delta Lake** | Deletion Vectors | RoaringBitmap | 支持 | 不支持 |
| **Hudi** | Log Files (MOR) | - | 不支持 | 不支持 |

**Paimon 的独特之处**:
- **rawConvertible 优化** - DV 启用时，所有 Split 都可以跳过 merge
- **位图级协同** - DV 和文件索引都使用 RoaringBitmap32，可以高效组合
- **基于行位置** - 无需知道 key 结构，适用于所有表类型

### 设计理念

**为什么这样设计**:

1. **DV 使得 rawConvertible 成为可能** - 跳过 merge 显著提升性能：
   ```
   传统 MOR (无 DV):
   - 读取所有文件的数据
   - 使用 SortMergeReader 做多路归并
   - 应用 merge 引擎 (DEDUPLICATE/PARTIAL_UPDATE/AGGREGATE)
   - 读取速度慢
   
   COW (有 DV):
   - 直接读取文件
   - 应用 DV 过滤已删除的行
   - 无需 merge
   - 读取速度快 2-10 倍
   ```

2. **基于行位置而非 key** - 简化实现和提升性能：
   ```
   基于 key 的删除:
   - 需要知道 key 的结构
   - 需要在读取时提取 key 并比较
   - 不适用于追加表
   
   基于行位置的删除:
   - 无需知道 key 的结构
   - 文件格式层提供行位置
   - DV 查询只需 O(1) 的位图检查
   - 适用于所有表类型
   ```

3. **RoaringBitmap32 的选择** - 高效的位图压缩：
   ```
   普通位图:
   - 1000万行文件，位图大小 = 1000万 bits = 1.25 MB
   
   RoaringBitmap32:
   - 稀疏删除 (1% 删除率): 压缩后 ~10 KB
   - 密集删除 (50% 删除率): 压缩后 ~100 KB
   - 压缩比: 10-100x
   ```

4. **与文件索引的位图级协同** - 避免重复计算：
   ```java
   // 创建基础选择位图
   BitmapIndexResult baseSelection = createBaseSelection(file, dv);
   // baseSelection = 全行位图 AND NOT DV
   
   // 文件索引评估
   BitmapIndexResult indexResult = predicate.evaluate(filter);
   
   // 组合结果
   BitmapIndexResult finalResult = indexResult.and(baseSelection);
   // 一次位运算完成，无需逐行判断
   ```

**权衡取舍**:

| 设计选择 | 优势 | 劣势 | 取舍原因 |
|---------|------|------|---------|
| DV 启用 rawConvertible | 读取速度快 2-10 倍 | 需要额外存储 DV | 读取性能是核心需求 |
| 基于行位置 | 实现简单，性能高 | 无法支持复杂的删除条件 | 大多数场景下行位置足够 |
| RoaringBitmap32 | 压缩比高 | 需要额外的库依赖 | 存储开销是关键考虑 |
| 位图级协同 | 避免重复计算 | 需要统一的位图格式 | 性能优化的关键 |
| Limit 下推禁用 | 保证正确性 | 某些查询性能下降 | 正确性优先 |

**架构演进**:

1. **早期版本** - 只有 MOR，无 DV 支持
2. **引入 DV** - 支持基于位图的删除标记
3. **引入 rawConvertible** - DV 启用时可以跳过 merge
4. **引入位图级协同** - DV 和文件索引共享 RoaringBitmap32 格式
5. **引入 ApplyDeletionVectorReader** - 统一的 DV 应用逻辑
6. **优化 Scan 阶段** - wholeBucketFilter 和 Limit 下推的 DV 适配

**业界对比**:

- **Delta Lake** - 也使用 Deletion Vectors，但实现细节不公开
- **Iceberg** - 使用 Delete Files (Parquet 格式)，需要额外的 merge
- **Hudi** - 使用 Log Files (MOR 模式)，需要 merge

**Paimon 的创新点**:
- **rawConvertible 优化** - DV 启用时，所有 Split 都可以跳过 merge
- **位图级协同** - DV 和文件索引都使用 RoaringBitmap32，可以高效组合
- **统一的 DV 应用逻辑** - ApplyDeletionVectorReader 适用于所有读取路径

### 8.1 DV 在 Scan 阶段的作用

DV(Deletion Vectors) 通过位图标记数据文件中已删除的行，避免了传统 merge-on-read 的全量合并。

在 Scan 阶段，DV 的主要影响:

1. **rawConvertible 判定** -- DV 启用时，`alwaysRawConvertible()` 返回 true，因为不需要 merge
2. **Limit 下推受限** -- `KeyValueFileStoreScan.limitPushdownEnabled()` 在 DV 启用时返回 false:
   ```java
   public boolean limitPushdownEnabled() {
       // 前置条件: limit 必须有效
       if (limit == null || limit <= 0) {
           return false;
       }
       
       // 禁用条件:
       // 1. PARTIAL_UPDATE 引擎 - 部分字段更新，rowCount 不等于最终记录数
       // 2. AGGREGATE 引擎 - 聚合后记录数可能减少
       // 3. deletionVectorsEnabled - DV 标记的删除行不计入有效行数
       return mergeEngine != PARTIAL_UPDATE 
               && mergeEngine != AGGREGATE 
               && !deletionVectorsEnabled;
   }
   ```
   **为什么 DV 启用时禁止 Limit 下推**: 文件的 rowCount 不等于有效行数(需要减去 DV 标记的删除行)，因此不能基于 rowCount 做精确的 Limit 截断。

### 8.2 DV 在 Read 阶段的作用

**源码路径**: `paimon-core/.../deletionvectors/ApplyDeletionVectorReader.java`

```java
public class ApplyDeletionVectorReader implements FileRecordReader<InternalRow> {
    public FileRecordIterator<InternalRow> readBatch() {
        FileRecordIterator<InternalRow> batch = reader.readBatch();
        return new ApplyDeletionFileRecordIterator(batch, deletionVector);
    }
}
```

`ApplyDeletionFileRecordIterator` 在迭代过程中，通过 `deletionVector.isDeleted(returnedPosition)` 检查每条记录的位置是否在 DV 位图中被标记，是则跳过。

**为什么 DV 基于行位置而非 key**: 基于位置的方式无需知道数据的 key 结构，适用于所有表类型(主键表和追加表)。读取时文件格式层提供行位置，DV 查询只需要 O(1) 的位图检查。

### 8.3 DV 与文件索引的协同

**源码路径**: `io/FileIndexEvaluator.java:100-108`

DV 和文件索引在 Read 阶段协同工作:

```java
private static BitmapIndexResult createBaseSelection(DataFileMeta file, DeletionVector dv) {
    BitmapIndexResult selection = new BitmapIndexResult(
        () -> RoaringBitmap32.bitmapOfRange(0, file.rowCount()));  // 全行位图
    if (dv instanceof BitmapDeletionVector) {
        selection = selection.andNot(((BitmapDeletionVector) dv).get());  // 减去删除行
    }
    return selection;
}
```

流程:
1. 创建全行位图 [0, rowCount)
2. AND NOT 删除向量 -> 有效行位图
3. 文件索引评估结果 AND 有效行位图 -> 最终需要读取的行

**好处**: DV 和 Bitmap 索引共享 RoaringBitmap32 格式，可以通过高效的位运算组合，无需在行级别逐条判断。

## 9. 统计信息收集

### 解决什么问题

**核心业务问题**: 查询优化器需要全局的统计信息 (如 NDV、数据分布) 来选择最优的执行计划。如何收集和维护这些统计信息？

**没有这个设计的后果**:
- 查询优化器无法准确估算查询成本，选择次优的执行计划
- Join 顺序选择不当，导致中间结果膨胀
- 聚合和分组操作的并行度设置不合理

**实际场景**:
```sql
-- 场景1: Join 顺序选择
SELECT * FROM orders o 
JOIN users u ON o.user_id = u.user_id 
JOIN products p ON o.product_id = p.product_id;
-- 优化器需要知道每张表的行数和 NDV，选择最优的 Join 顺序

-- 场景2: 聚合并行度
SELECT city, COUNT(*) FROM users GROUP BY city;
-- 优化器需要知道 city 的 NDV，设置合理的并行度

-- 场景3: 分区裁剪
SELECT * FROM events WHERE dt = '2024-01-01' AND city = 'Beijing';
-- 优化器需要知道每个分区的行数，估算查询成本
```

### 有什么坑

**误区陷阱**:
1. **误以为文件级统计信息足够** - 文件级 SimpleStats 只有 min/max，无 NDV
2. **忽略统计信息的时效性** - 数据变化后，统计信息可能过时
3. **混淆 ANALYZE 和 Compaction** - ANALYZE 是读操作，Compaction 是写操作

**错误配置**:
```sql
-- 错误1: 从不执行 ANALYZE
-- 统计信息为空，优化器只能使用默认估算

-- 错误2: 过于频繁地执行 ANALYZE
CALL sys.analyze_table('my_table');  -- 每小时执行一次
-- ANALYZE 需要扫描全表，成本很高

-- 错误3: 在小表上执行 ANALYZE
CALL sys.analyze_table('small_table');  -- 只有1000行
-- 小表的统计信息收集收益不明显

-- 正确做法:
-- 1. 在大表上定期执行 ANALYZE (如每天一次)
-- 2. 在数据变化较大后执行 ANALYZE
-- 3. 监控统计信息的准确性
```

**生产环境注意事项**:
1. **ANALYZE 成本** - 需要扫描全表或大量样本，I/O 和 CPU 开销大
2. **统计信息存储** - 每个快照可以关联一个统计信息文件
3. **统计信息过期** - 数据变化后，统计信息可能不准确

**性能陷阱**:
```java
// 陷阱1: 全表扫描收集统计信息
// 大表的 ANALYZE 可能需要数小时
// 解决: 使用采样或增量收集

// 陷阱2: NDV 计算精度不足
// 使用精确算法 (如 HashSet) 内存爆炸
// 解决: 使用近似算法 (如 HyperLogLog)

// 陷阱3: 统计信息文件过多
// 每个快照一个统计信息文件，可能累积大量文件
// 解决: 定期清理旧的统计信息文件
```

### 核心概念解释

**术语定义**:

1. **Statistics** - 全局统计信息，包含表级和列级的统计数据
2. **ColStats** - 列级统计信息，包含 NDV、min/max、nullCount、avgLen、maxLen
3. **NDV (Number of Distinct Values)** - 不同值的数量，用于估算 Join 和聚合的成本
4. **ANALYZE** - 收集统计信息的操作，类似于 SQL 标准的 ANALYZE TABLE
5. **HyperLogLog** - 近似计算 NDV 的算法，空间复杂度 O(log log N)
6. **StatsFileHandler** - 统计信息文件的读写管理器

**概念关系**:
```
Statistics (全局统计信息)
  ├─ snapshotId (对应的快照 ID)
  ├─ schemaId (对应的 Schema ID)
  ├─ mergedRecordCount (merge 后的总记录数)
  ├─ mergedRecordSize (merge 后的总数据大小)
  └─ colStats (列级统计信息)
       └─ ColStats (单列统计)
            ├─ colId (列 ID)
            ├─ distinctCount (NDV)
            ├─ min / max (最小/最大值)
            ├─ nullCount (null 值数量)
            ├─ avgLen (平均列长度)
            └─ maxLen (最大列长度)
```

**与其他系统对比**:

| 系统 | 统计信息类型 | NDV 算法 | 存储方式 | 自动收集 |
|------|------------|---------|---------|---------|
| **Paimon** | 表级 + 列级 | HyperLogLog (推测) | JSON 文件 | 不支持 (手动 ANALYZE) |
| **Iceberg** | 表级 + 列级 | HyperLogLog | Manifest 元数据 | 不支持 (手动 ANALYZE) |
| **Delta Lake** | 表级 + 列级 | HyperLogLog | Delta Log | 支持 (Auto Optimize) |
| **Hudi** | 表级 | - | Metadata Table | 支持 (Metadata Table) |
| **MySQL** | 表级 + 列级 + 索引 | 采样 | 系统表 | 支持 (Auto Analyze) |

**Paimon 的独特之处**:
- **与 Snapshot 关联** - 每个快照可以关联一个统计信息文件
- **JSON 格式存储** - 易于调试和扩展
- **Schema Evolution 支持** - 统计信息可以跨 Schema 版本

### 设计理念

**为什么这样设计**:

1. **全局统计信息与文件级统计信息分离** - 不同的用途：
   ```
   文件级 SimpleStats (min/max/nullCounts):
   - 用途: 查询时的文件过滤
   - 粒度: 文件级
   - 更新: 写入时自动收集
   - 成本: 低 (写入时顺便收集)
   
   全局 Statistics (NDV/avgLen/maxLen):
   - 用途: 查询优化器的成本估算
   - 粒度: 表级 + 列级
   - 更新: 手动 ANALYZE
   - 成本: 高 (需要扫描全表或大量样本)
   ```

2. **与 Snapshot 关联** - 保证统计信息的一致性：
   ```
   Snapshot
     ├─ manifestList (数据文件列表)
     ├─ schemaId (Schema 版本)
     └─ statistics (统计信息文件名)
   
   好处:
   - 统计信息与数据版本一致
   - 可以查询历史快照的统计信息
   - 支持时间旅行查询的成本估算
   ```

3. **JSON 格式存储** - 易于扩展和调试：
   ```json
   {
     "snapshotId": 123,
     "schemaId": 1,
     "mergedRecordCount": 1000000,
     "mergedRecordSize": 1073741824,
     "colStats": {
       "user_id": {
         "colId": 0,
         "distinctCount": 50000,
         "min": "user_00001",
         "max": "user_99999",
         "nullCount": 0,
         "avgLen": 12,
         "maxLen": 12
       }
     }
   }
   ```

4. **手动 ANALYZE 而非自动收集** - 平衡成本和收益：
   ```
   自动收集的问题:
   - 何时触发? (每次写入? 定期?)
   - 如何控制成本? (全表扫描 vs 采样?)
   - 如何避免影响写入性能?
   
   手动 ANALYZE 的好处:
   - 用户可以选择合适的时机 (如低峰期)
   - 用户可以选择需要收集统计信息的表
   - 避免自动收集的不确定性
   ```

**权衡取舍**:

| 设计选择 | 优势 | 劣势 | 取舍原因 |
|---------|------|------|---------|
| 全局统计信息分离 | 用途明确，避免混淆 | 需要额外的 ANALYZE 操作 | 文件级统计无法提供 NDV 等信息 |
| 与 Snapshot 关联 | 保证一致性 | 每个快照可能需要独立的统计信息 | 一致性是查询优化的前提 |
| JSON 格式 | 易于扩展和调试 | 解析性能略低 | 统计信息读取不是热路径 |
| 手动 ANALYZE | 用户可控，成本可预测 | 需要用户主动触发 | 避免自动收集的不确定性 |
| HyperLogLog (推测) | 空间效率高 | 有误差 (通常 < 2%) | NDV 的精确计算成本过高 |

**架构演进**:

1. **早期版本** - 无全局统计信息，只有文件级 SimpleStats
2. **引入 Statistics** - 支持表级和列级的统计信息
3. **引入 ANALYZE 操作** - 通过 Flink/Spark Procedure 触发
4. **引入 StatsFileHandler** - 统一的统计信息读写管理
5. **与 Snapshot 关联** - 统计信息文件名写入 Snapshot
6. **Schema Evolution 支持** - 统计信息可以跨 Schema 版本

**业界对比**:

- **Iceberg** - 统计信息存储在 Manifest 元数据中，支持手动 ANALYZE
- **Delta Lake** - 支持 Auto Optimize，可以自动收集统计信息
- **Hudi** - 使用 Metadata Table 存储统计信息，支持自动收集
- **传统数据库 (MySQL/PostgreSQL)** - 支持自动 ANALYZE，统计信息存储在系统表中

**Paimon 的创新点**:
- **与 Snapshot 的深度集成** - 统计信息与数据版本一致
- **JSON 格式存储** - 易于扩展和调试
- **Schema Evolution 支持** - 统计信息可以跨 Schema 版本

### 9.1 Statistics 与 ColStats 模型

**源码路径**: `paimon-core/.../stats/Statistics.java`, `ColStats.java`

Paimon 的全局统计信息(ANALYZE 结果)通过 `Statistics` 类表示:

```java
public class Statistics {
    long snapshotId;          // 对应的快照 ID
    long schemaId;            // 对应的 Schema ID
    Long mergedRecordCount;   // merge 后的总记录数
    Long mergedRecordSize;    // merge 后的总数据大小(字节)
    Map<String, ColStats<?>> colStats;  // 列级统计信息
}
```

`ColStats` (见 `ColStats.java`) 包含:

| 字段 | 类型 | 说明 |
|------|------|------|
| `colId` | int | 列 ID |
| `distinctCount` | Long | 不同值数量 (NDV) |
| `min` / `max` | Comparable<T> | 最小/最大值 |
| `nullCount` | Long | null 值数量 |
| `avgLen` | Long | 平均列长度 |
| `maxLen` | Long | 最大列长度 |

**为什么需要全局统计信息**: 文件级 SimpleStats 已经支持查询过滤，但查询优化器(Flink/Spark)需要全局的 NDV、数据分布等信息来选择最优的 join 策略、聚合策略和并行度。ANALYZE 操作收集这些全局统计并持久化。

### 9.2 StatsFileHandler 读写机制

**源码路径**: `paimon-core/.../stats/StatsFileHandler.java`

```java
public class StatsFileHandler {
    // 写入: 先序列化字段到字符串, 再写入 JSON 文件
    public String writeStats(Statistics stats) {
        stats.serializeFieldsToString(schemaManager.schema(stats.schemaId()));
        return statsFile.write(stats);
    }
    
    // 读取: 读 JSON 文件, 再反序列化字段
    public Optional<Statistics> readStats(Snapshot snapshot) {
        String file = snapshot.statistics();
        return file == null ? Optional.empty() : Optional.of(readStats(file));
    }
}
```

统计信息文件与 Snapshot 关联: `Snapshot.statistics()` 指向对应的统计文件名。

### 9.3 ANALYZE 操作流程

ANALYZE 操作通过 Flink/Spark 的 Procedure 或 SQL 触发:
1. 扫描指定快照的所有 Manifest 条目
2. 对每个数据文件读取样本数据或全量数据
3. 计算 NDV(使用 HyperLogLog 等近似算法)、min/max、nullCount 等
4. 聚合为 `Statistics` 对象
5. 通过 `StatsFileHandler.writeStats()` 写入文件
6. 在下一次 Commit 时将统计文件名写入 Snapshot

## 10. 查询性能调优建议

### 10.1 与 Iceberg 过滤机制对比

| 维度 | Paimon | Iceberg |
|------|--------|---------|
| **Manifest 过滤** | partitionStats + bucket/level 范围 | partition summary (min/max per partition field) |
| **文件统计** | SimpleStats (min/max/nullCounts, BinaryRow 格式) | Column-level metrics (min/max/null_count/nan_count, Avro 格式) |
| **Schema Evolution** | SimpleStatsEvolution 动态映射 | 基于 field-id 的静态映射 |
| **文件索引** | Bloom/Bitmap/BSI/Range Bitmap (嵌入式+独立) | Bloom Filter (独立 Puffin 文件) |
| **多维排序** | ORDER/ZORDER/HILBERT (Spark/Flink 实现) | ZORDER (Spark 实现, 源自 Iceberg) |
| **Split 优化** | IntervalPartition + BinPacking + rawConvertible | FileScanTask 基于大小拆分 |
| **Lookup 点查** | LookupLevels + 本地 SST 缓存 + 远程下载 | 无原生支持(需外部 KV 层) |

**Paimon 的优势**:
- 更丰富的文件索引类型(Bitmap/BSI/Range Bitmap 提供行级过滤)
- 原生的 Lookup 点查优化(适合流式更新+实时查询场景)
- Clustering Compaction 在存储层实现增量排序优化
- DV + 文件索引的位图级协同
- Manifest 级别的 minBucket/maxBucket/minLevel/maxLevel 可以在不读取 Manifest 内容的情况下淘汰 Manifest 文件

**Iceberg 的优势**:
- NaN 计数(对浮点数据更精确)
- 更成熟的 Z-Order 实现和社区生态
- Manifest caching 和 Manifest rewrite 优化
- 更灵活的文件格式支持

### 10.2 关键配置参数

| 参数 | 默认值 | 说明 | 调优建议 |
|------|-------|------|---------|
| `file-index.bloom-filter.columns` | - | 启用 Bloom Filter 的列 | 高基数列的等值查询 |
| `file-index.bitmap.columns` | - | 启用 Bitmap 索引的列 | 低基数列的等值/IN 查询 |
| `file-index.bsi.columns` | - | 启用 BSI 索引的列 | 数值列的范围查询 |
| `file-index.read-enabled` | true | 是否在 Scan/Read 时使用文件索引 | 建议保持开启 |
| `sort-engine` | LOSER_TREE | 排序合并引擎 | 大量文件 compaction 使用 LOSER_TREE |
| `source.split.target-size` | 128MB | 目标 Split 大小 | 根据集群并行度调整 |
| `lookup-cache-file-retention` | 1h | Lookup 文件缓存保留时间 | 频繁查询时增大 |
| `lookup-cache-max-disk-size` | 无限制 | Lookup 缓存磁盘限制 | 根据本地磁盘大小设置 |
| `clustering-columns` | - | 聚簇列 | 多维查询的常用过滤列 |

### 10.3 最佳实践

**1. 分区设计先行**
- 分区裁剪是第一层、成本最低的过滤
- 以查询最频繁的低基数维度(如日期)做分区

**2. 为高频查询列配置文件索引**
- 等值查询: Bloom Filter (高基数) 或 Bitmap (低基数)
- 范围查询: BSI 或 Range Bitmap
- 组合使用: 同一列可以同时配置多种索引，FileIndexPredicate 会自动组合结果

**3. 多维排序提高文件跳过率**
- 单列范围查询: 使用 ORDER
- 两列等权查询: 使用 ZORDER
- 三列及以上: 使用 HILBERT (更好的空间局部性)

**4. 主键表启用 DV 获得 rawConvertible**
- 启用 DV 后，所有 Split 可以跳过 merge 直接读取
- 显著降低查询延迟，代价是额外的 DV 存储

**5. Clustering Compaction 优化多维查询**
- 配置 `clustering-columns` 后，Compaction 会按指定列排序数据
- 适合写入时数据无序但查询有多维过滤需求的场景
- 增量模式下写放大较低

**6. Lookup 点查场景优化**
- 确保本地磁盘足够存放 Lookup 缓存
- 启用远程 Lookup 文件加速任务恢复
- Bloom Filter 的行数参数影响 Lookup 缓存命中率

**7. Split 大小与并行度平衡**
- `scan.split-target-size` 过大 -> 并行度不够
- 过小 -> 太多 Split 导致调度开销
- 建议: 总数据量 / (并行度 * 2~4)
