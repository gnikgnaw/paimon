# Apache Paimon 索引机制深度分析

> **版本**：1.5-SNAPSHOT　**源码模块**：`paimon-common`（`org.apache.paimon.fileindex`，四类文件索引主体）、`paimon-core`（`index`/`deletionvectors`/`mergetree`，索引落地与读取协同）、`paimon-api`（`FileIndexOptions`/`CoreOptions` 配置）　**核对日期**：2026-06

**一句话定位**：Paimon 用一套"按谓词类型分工"的文件内嵌索引（Bloom Filter / Bitmap / BSI / Range Bitmap）把对象存储上"读一行也得读整个文件"的代价，压成"先看几 KB 索引决定跳文件还是只读匹配行"。本文是「文件索引」主讲文档，04（Deletion Vector）、12（查询优化）会引用它。

读完本文你应能回答：① 四类索引各自的数据结构是什么、分别擅长什么谓词，为什么不是一种索引打天下；② Bloom Filter 为什么只能做"跳文件"、Bitmap/BSI/Range Bitmap 为什么能精确到行；③ 高基数列为什么不能建 Bitmap、低基数列建 Bloom Filter 为什么浪费；④ `FileIndexResult` 的三态（REMAIN/SKIP/BitmapIndexResult）如何让"跳文件"和"跳行"两种索引在 AND/OR 里自然共存；⑤ 索引内嵌 Manifest 还是写独立 `.idx`，由哪个配置、默认多大决定（旧稿此处错得离谱，本文已修正）；⑥ 为什么 BSI 要把正负数分开建、Range Bitmap 凭什么独家支持 TopN 和字符串范围；⑦ 哪些写法会让索引悄悄失效、哪些配置会让写入性能腰斩。

## 1. 快速理解（核心问题 / 概念速查 / 高频陷阱）

### 1.1 核心问题：索引到底在跟什么成本作斗争

**① 要解决什么问题**

对象存储上数据以列式格式（Parquet/ORC）成百上千个文件躺着，列式存储的最小可读单元是 Row Group / Stripe。没有索引时，一个 `WHERE id = 'x'` 的点查只能把所有候选文件的整个列块拉下来、解压、扫描——网络 I/O、解压 CPU、对象存储 API 调用三重浪费，延迟与文件数成正比。索引要回答两个递进的问题：**这个文件值得读吗（文件级跳过）？文件里哪几行值得读（行级过滤）？**

**② 设计原理与取舍**

Paimon 没有做"一种万能索引"，而是按**谓词类型**拆成四类，每类只在自己最擅长的查询模式上做到极致。一句话设计哲学：**没有银弹，只有按查询模式分工的权衡——空间、精度、可支持的谓词三者不可兼得。**

- 等值且只需判断"在不在"：Bloom Filter，空间最省（每元素几 bit），但有误判、只能跳文件。
- 等值/IN/NULL 且要精确到行：Bitmap 倒排，空间随基数膨胀，只宜低基数列。
- 数值范围（`>` `<` `between`）要精确到行：BSI，把每一位拆成独立 bitmap 做位运算。
- 范围 + 字符串 + TopN：Range Bitmap，BSI 之上加字典编码，独家支持 `ORDER BY col LIMIT N` 下推。

横向对比同类湖格式（理解 Paimon 索引定位的关键）：

| 系统 | 主要索引 | 存储位置 | 能否行级过滤 |
|------|---------|---------|---------|
| **Paimon** | Bloom / Bitmap / BSI / Range Bitmap（SPI 可扩展） | 内嵌 Manifest 或独立 `.idx` | **能**（Bitmap/BSI/Range Bitmap 返回精确行号） |
| Iceberg | Parquet/ORC 列级 Bloom Filter + min/max | 文件 Footer / Puffin | 仅 Page 级 |
| Hudi | Bloom Filter（主键索引为主） | `.hoodie` 元数据 | 不支持 |
| Delta Lake | Data Skipping（min/max 统计） | `_delta_log` | 不支持 |

Paimon 的差异点：**行级过滤能力**——其它格式最多跳文件或跳 Page，Paimon 的 Bitmap 系列能直接交出"该读哪几行"的精确 RoaringBitmap，配合 `ApplyBitmapIndexRecordReader` 在读取层只吐出命中行（详见 §7.3）。

### 1.3 四类文件索引选型速查

| 索引 | SPI 标识 | 数据结构 | 擅长谓词 | 行级? | 适用列基数 | 空间/写入代价 | 致命短板 |

### 1.4 高频生产陷阱

**陷阱 1：把 Bloom Filter 当成能行级过滤。** Bloom Filter 的 `Reader` 只实现了 `visitEqual`，返回 `REMAIN`/`SKIP`（`BloomFilterFileIndex.java:132-134`），它只能回答"这个文件可能有/一定没有"，无法回答"哪几行"。文件里 1 亿行只有 1 行命中，Bloom Filter 仍要你读整个文件。要跳行得用 Bitmap/BSI/Range Bitmap。

**陷阱 2：给高基数列建 Bitmap。** Bitmap 的 Writer 在内存里维护 `Map<Object, RoaringBitmap32> id2bitmap`（`BitmapFileIndex.java:85`），每个不同值一个 bitmap。用户 ID 这种亿级基数列会直接把写入内存撑爆、索引体积超过数据本身。高基数等值用 Bloom Filter，低基数（状态、类型、地区）才用 Bitmap。

**陷阱 3：给字符串列建 BSI。** BSI 的 `getValueMapper` 只为数值类型（TinyInt/SmallInt/Int/BigInt/Date/Time/Timestamp/Decimal）注册了映射，其它类型 `defaultMethod` 直接抛 `UnsupportedOperationException`（`BitSliceIndexBitmapFileIndex.java:392-397`）。字符串范围查询要用 Range Bitmap（其 `KeyFactory` 支持 Char/VarChar，`KeyFactory.java:56-63`）。

**陷阱 4：误以为内嵌阈值是 32KB。** 真实配置是 `file-index.in-manifest-threshold`，默认 **500 字节**（`CoreOptions.java:385-389`，**已修正**：旧稿写成 `file-index-in-manifest-threshold` 默认 32KB，键名和默认值都错）。默认值这么小意味着绝大多数索引都会落成独立 `.idx` 文件；只有极小的索引才内嵌进 Manifest。想减少小文件数量可调大它，但 Manifest 会膨胀、scan 时反序列化变慢。

**陷阱 5：索引失效而不自知。** ① `UPPER(col) = 'X'` 这类函数包裹或隐式类型转换会让谓词无法下推到索引；② 改了索引配置，存量数据文件不会自动重建索引，要手动 `CALL sys.rewrite_file_index(...)`（走 `FileIndexProcessor`，`paimon-core/.../index/FileIndexProcessor.java`）；③ Bloom Filter 对 `col IS NULL` 无能为力（它把 null 直接当 `REMAIN`，`BloomFilterFileIndex.java:133`）。

**陷阱 6：BSI/Range Bitmap 写大文件时 OOM。** BSI 的 `StatsCollectList` 把所有值暂存进 `List<Long>`，源码注释明写 `// todo: Find a way to reduce the risk of out-of-memory.`（`BitSliceIndexBitmapFileIndex.java:168-169`）。单文件越大、行数越多，暂存内存越高。控制 `target-file-size` / `compaction.max-file-size`，别让单个数据文件过大。

**陷阱 7：多个 Bloom Filter 做 AND 误判率累积。** 单个 FPP=0.1，但若一列叠多个 Bloom Filter 或多列 AND，误判是概率相乘/相关的，整体"假命中"率会显著高于单个值。需要高精度时调小 `fpp`（代价是空间翻倍），或改用精确的 Bitmap。

## 2. 文件内嵌索引体系总览

### 2.1 接口体系与 SPI 扩展

**① 要解决什么问题**

四类索引数据结构差异巨大，但读写流程要统一：写期都要"收集列值 → 序列化成字节"，读期都要"按谓词访问 → 返回过滤结果"。需要一组抽象既能统一调用、又能让第三方（全文检索、向量）插进来。

**② 设计原理与取舍**

一套四接口 + SPI：

- `FileIndexerFactory.identifier()` 返回类型名（如 `bloom-filter`），`create(DataType, Options)` 产出 `FileIndexer`；
- `FileIndexer.createWriter()` / `createReader(in, start, length)` 分别给写、读两端；
- `FileIndexWriter.write(Object)` 逐值收集，`serializedBytes()` 一次性吐出索引字节；
- `FileIndexReader` 对每种谓词有一个 `visitXxx`（`visitEqual`/`visitIn`/`visitLessThan`/`visitGreaterThan`/`visitIsNull`/`visitBetween`/`visitTopN`…），返回 `FileIndexResult`。

设计哲学：**写入 = "把列值喂给 Writer"，读取 = "把谓词派发给 Reader 的 visit 方法"，索引类型差异完全藏在实现里。** 谓词与索引的对接靠 `FileIndexReader` 的 visitor，谁支持哪种谓词就只重写哪个 visit（不重写的走父类默认值——见 §7.2）。

**③ 关键源码**

SPI 通过 `ServiceLoader` 静态加载，重复 identifier 会告警（`FileIndexerFactoryUtils.java:35-47`）：

```java
static {
    ServiceLoader<FileIndexerFactory> serviceLoader = ServiceLoader.load(FileIndexerFactory.class);
    for (FileIndexerFactory indexerFactory : serviceLoader) {
        if (factories.put(indexerFactory.identifier(), indexerFactory) != null) {
            LOG.warn("Found multiple FileIndexer for type: " + indexerFactory.identifier() + " ...");
        }
    }
}
```

`META-INF/services/org.apache.paimon.fileindex.FileIndexerFactory` 恰好注册四个 Factory（已核对）：

| SPI 标识 | FileIndexer 实现 | Factory（identifier 来源） |
|---------|-----------------|---------|
| `bloom-filter` | `BloomFilterFileIndex` | `BloomFilterFileIndexFactory.java:29` |
| `bitmap` | `BitmapFileIndex` | `BitmapFileIndexFactory.java:29` |
| `bsi` | `BitSliceIndexBitmapFileIndex` | `BitSliceIndexBitmapFileIndexFactory.java:29` |
| `range-bitmap` | `RangeBitmapFileIndex` | `RangeBitmapFileIndexFactory.java:29` |

**④ 风险 / 边界**：第三方扩展（paimon-tantivy 全文检索、paimon-lumina 向量）通过同一 SPI 注册自己的 `FileIndexerFactory`，无需改核心代码；但同一 identifier 被注册两次只会保留一个并告警，扩展时务必用唯一标识。

**⑤ 收益与代价**：收益是索引类型与文件格式、与核心读写完全解耦，新增一种索引零侵入；代价是所有索引都被框死在"逐值 write + 一次性 serializedBytes"的写期模型里——这正是 BSI/Range Bitmap 必须把全部值暂存内存的根因（§5、§6）。

### 2.2 FileIndexFormat 存储格式与内嵌/独立决策

**① 要解决什么问题**

一个数据文件可能给多列、每列多种索引建索引。读取时只想 seek 到"这一列的这一种索引"那几个字节，不想把整块索引反序列化。

**② 设计原理与取舍**

自包含二进制格式：Header 记录每列、每索引在 Body 里的 `(start, length)`，Body 是各索引序列化字节的拼接。读取时先读 Header，按需 `seek(bodyStart + start)` 取目标块即可。比 Protobuf/Avro 省掉了 schema 元数据，且天然支持随机访问。`magic = 1493475289347502L`、当前 `version = V_1(1)`、空列用 `EMPTY_INDEX_FLAG = -1` 标记（`FileIndexFormat.java:99-116`，类注释里有完整 ASCII 布局图）。

**内嵌 vs 独立 `.idx` 决策（重要，旧稿此处错误）**：由 `file-index.in-manifest-threshold` 决定，**默认 500 字节**（`CoreOptions.java:385-389`）。

- 索引字节 ≤ 阈值 → 嵌入 `DataFileMeta`，随 Manifest 持久化（小而快，scan 阶段就能用）；
- 索引字节 > 阈值 → 写独立 `.idx` 文件，挂在 `DataFileMeta.extraFiles`。

> **（已修正）**：旧稿写键名 `file-index-in-manifest-threshold`、默认 32KB，两处都错。真实键名是 `file-index.in-manifest-threshold`，默认 500B。默认值这么小，意味着实践中绝大多数索引落成独立 `.idx`，只有极小索引才内嵌。

**③ 关键源码**：写期产物的内嵌/独立选择由 `DataFileIndexWriter` 按 `fileIndexInManifestThreshold()` 判定；存量重建走 `FileIndexProcessor`（`paimon-core/.../index/FileIndexProcessor.java:79` 读阈值，`process()` 读旧 `.idx`、重算、按阈值重新落地）。

**④ 风险 / 边界**：阈值调大可减少 `.idx` 小文件数，但 Manifest 体积上升、scan 时反序列化变慢——这是"小文件数"与"Manifest 大小"之间的权衡。改索引配置后存量文件不会自动重建，需 `CALL sys.rewrite_file_index(...)`。

**⑤ 收益与代价**：收益是随机访问 + 与数据文件同生命周期（原子提交、随删自动失效，不存在索引/数据不一致）；代价是索引膨胀直接转成对象存储的小文件数量与 API 调用成本。

## 3. Bloom Filter 索引

**① 要解决什么问题**

最常见的等值点查（`WHERE col = 'x'`）只想知道一件事：**这个文件值得读吗？** 用最少的字节、最快的判断把"一定没有"的文件直接跳掉，哪怕偶尔放过几个"其实没有"的文件也认。

**② 设计原理与取舍**

位数组 + k 个哈希函数。写期对每个值算哈希、置位；读期对查询值算哈希、查位：有 0 → 一定没有（`SKIP`），全 1 → 可能有（`REMAIN`，可能误判）。一句话哲学：**用可控的误判率换极致空间，且只回答"跳不跳文件"。**

为什么是默认首选：空间最省（一个 bit 数组，写入开销几乎可忽略）、判断 O(1)、通过 `FastHash` 适配所有类型。代价是三个硬限制——只支持等值、有误判、无法处理 NULL。

底层用 64 位的 `BloomFilter64` 而非 32 位：哈希空间 2^64 让大基数下的冲突率显著更低，对长文件更稳。`FastHash` 按类型分派哈希：字节类型（CHAR/VARCHAR/BINARY 等）走 xxHash（`net.openhft.hashing.LongHashFunction`，`FastHash.java:49/67`），数值类型走整数位扰动哈希，Timestamp 按精度转 millis/micros 后再哈希。

**③ 关键源码**

默认参数与 Writer/Reader 的极简对照（`BloomFilterFileIndex.java`）：

```java
private static final int DEFAULT_ITEMS = 1_000_000;   // L50 预期元素数
private static final double DEFAULT_FPP = 0.1;        // L51 误判率

// Writer：L88-98，注意是 new，不是旧稿写的 BloomFilter64.create(...)
this.filter = new BloomFilter64(items, fpp);
public void write(Object key) { if (key != null) filter.addHash(hashFunction.hash(key)); }

// Reader：只实现 visitEqual，null 直接放行
public FileIndexResult visitEqual(FieldRef f, Object key) {            // L132-134
    return key == null || filter.testHash(hashFunction.hash(key)) ? REMAIN : SKIP;
}
```

> **（已修正）**：旧稿写 `BloomFilter64.create(items, fpp)`，实际构造是 `new BloomFilter64(items, fpp)`（`BloomFilterFileIndex.java:89`）。Reader 仅重写 `visitEqual`——这从源码层面证明了"Bloom Filter 只能等值、只能跳文件"。

**④ 风险 / 陷阱 / 边界**

- **NULL 失效**：`visitEqual` 里 `key == null` 直接返回 `REMAIN`（L133），所以 `IS NULL` / `IS NOT NULL` 用 Bloom Filter 等于没建。
- **范围失效**：没有 `visitLessThan` 等实现，给时间戳、金额等范围列建 Bloom Filter 完全无效（走父类默认 `REMAIN`，永不跳文件）。
- **`items` 配小了误判飙升**：`items` 应略大于文件实际行数；配得过小，FPP 实际值远超设定。
- **误判率在 AND 下累积**：多个 Bloom Filter 联合时整体假命中率高于单个 FPP。

**⑤ 收益与代价**

收益：以每元素几 bit 的极低成本，把海量不相关文件挡在 I/O 之外，是高基数等值点查的性价比之王。代价：概率性（默认约 10% 文件会被"可能有"放过、需实读确认）、谓词覆盖窄（仅等值、不含 NULL）。**适用**：高基数、等值、跳文件即可的场景；**不适用**：需要精确到行、范围、NULL 过滤——那是 Bitmap/BSI/Range Bitmap 的活。


## 4. Bitmap 倒排索引

**① 要解决什么问题**

Bloom Filter 只能跳文件。但 1GB 文件里 1000 万行只有 100 行命中时，跳不了文件等于白读 999.99 万行。低基数列（状态、类型、地区）的等值/IN 查询要的是**精确到行**：直接交出"该读哪几行"，外加 Bloom Filter 干不了的 `NOT IN`、`IS NULL`、`IS NOT NULL`。

**② 设计原理与取舍**

倒排："值 → 命中行号集合"，行号集合用 `RoaringBitmap32` 压缩。RoaringBitmap 把 32 位空间分块，每块按密度选 Array/Bitmap/Run 容器，稀疏时空间 ∝ 基数而非最大行号。一句话哲学：**用 RoaringBitmap 把行号集合压到很小，换来精确行级过滤 + 原生位运算的多条件组合。**

谓词全靠位运算落地：`=`/`IN` 是命中值 bitmap 的 OR；`NOT IN` 是 OR 后 `flip`；`IS NULL` 查独立的 nullBitmap；`IS NOT NULL` 是 nullBitmap 取反。这就是为什么 Bitmap 天然支持 Bloom Filter 做不到的那些谓词。

代价直接写在数据结构里：Writer 在内存维护 `Map<Object, RoaringBitmap32> id2bitmap`，**内存与基数成正比**——这是"高基数列禁用 Bitmap"的物理根因，不是经验法则。

**③ 关键源码**（`BitmapFileIndex.java`）

```java
// Writer：每个不同值一个 bitmap，内存 ∝ 基数（L85, L97-106）

private final RoaringBitmap32 nullBitmap = new RoaringBitmap32();
```

**单值优化为何重要**：基数偏高、大量值只出现一次时，逐个存空 bitmap 极浪费；负数 offset 把"单行命中"内联进元信息，省掉 bitmap 序列化。读时 `readBitmap` 见到负 offset 直接 `RoaringBitmap32.bitmapOf(-1 - offset)` 还原（`BitmapFileIndex.java:266-268`）。

**④ 风险 / 陷阱 / 边界**

- **高基数列内存爆炸**：`id2bitmap` 与基数同步膨胀，建索引前先 `SELECT COUNT(DISTINCT col)`，基数 > ~1000 别用 Bitmap，改 Bloom Filter。
- **小文件得不偿失**：RoaringBitmap 序列化有固定开销，文件太小时索引体积占比反而高。
- **NULL 是一等公民**：单独 nullBitmap 让 `IS NULL` O(1) 命中——这是相对 Bloom Filter 的独特优势。

**⑤ 收益与代价**

收益：精确行级过滤 + 多条件 AND/OR 走 RoaringBitmap 原生位运算（§7.2）+ 可与 DV 做 `andNot`（§7.3）。代价：空间/写入内存 ∝ 基数，注定只服务低基数列。**适用**：低基数等值/IN/NULL 且选择率低；**不适用**：高基数列、范围查询（范围请看 BSI/Range Bitmap）。


## 5. BSI 位切片索引

**① 要解决什么问题**

数值范围查询（`amount > 1000`、`age BETWEEN 18 AND 65`）在 OLAP 里极常见。Bloom Filter 完全不支持范围；Bitmap 理论上能做（把 `> 1000` 拆成上千个值的 OR），但要为每个不同值存一个 bitmap、查询要 OR 一大堆，空间和算力都不划算。需要一种**直接对数值大小做位运算**的行级范围索引。

**② 设计原理与取舍**

Bit-Sliced Index：把数值的**每一位**单独存成一个 bitmap（"slice"）。N 行的 BIGINT 列最多 64 个 slice，每个 slice 是"第 k 位为 1 的行号集合"。范围查询不再遍历值，而是从高位到低位做几十次位运算就能算出 `< v` / `> v` 的行集。一句话哲学：**把"比大小"翻译成"逐位的 bitmap 比较"，slice 数 = 位宽，与基数无关。**

为什么必须正负数分开建：负数补码直接按位比较会得出 `-1 > 1` 的错误结论。Paimon 用绝对值分别建 positive/negative 两个 BSI，查询时按边界值正负分流（如 `col < 正数v` = `positive.lt(v)` ∪ `所有负数`）。这是 BSI 正确性的关键，不是优化。

代价同样写在数据结构里：Writer 的 `StatsCollectList` 必须**暂存全部值**到 `List<Long>` 才能一次性切片，源码注释直言这是 OOM 风险点。

**③ 关键源码**（`BitSliceIndexBitmapFileIndex.java`）

**④ 风险 / 陷阱 / 边界**

- **仅数值类型**：给字符串建 BSI 直接抛异常，字符串范围用 Range Bitmap。
- **写期 OOM**：值全暂存，大文件高基数时内存峰值高；控制单文件大小（`target-file-size`）。
- **位宽即成本**：BIGINT 64 个 slice，范围查询要做 ~位宽 次位运算；位越宽查询越重，但与基数无关——这正是它相对 Bitmap 的优势所在。

**⑤ 收益与代价**

收益：数值范围、`between`、等值/IN、NULL 全部行级精确，空间随位宽（常数）而非基数增长，高基数数值列也扛得住。代价：写期暂存全部值、仅限数值、不支持字符串与 TopN。**适用**：高/中基数数值列的范围与等值；**不适用**：字符串范围、TopN（交给 Range Bitmap）。


## 6. Range Bitmap 索引

**① 要解决什么问题**

BSI 解决了数值范围，但留下两个缺口：① 它只认数值，字符串范围（`name BETWEEN 'a' AND 'm'`）一律抛异常；② OLAP 里极常见的 `ORDER BY col LIMIT N`（TopN）无索引可用，只能全量排序。Range Bitmap 要把"范围 + 字符串 + TopN"一次性补齐——这是四类索引里实现最重、能力最全的一类。

**② 设计原理与取舍**

在 BSI 之上加一层 **ChunkedDictionary（分块字典）**：先把任意可比较的原始值（含字符串）排序、映射成紧凑的整数 code，再对 code 建 BSI。一句话哲学：**先用字典把"任意类型的序"压成"整数的序"，范围/TopN 就退化成 BSI 已经会做的整数位运算。**

- **类型扩展靠 `KeyFactory`**：它为 Char/VarChar（`StringKeyFactory`）、数值、Decimal、Timestamp 等分别提供 converter 与比较器（`KeyFactory.java:51-88`，Char/VarChar 在 L56-63）。这是 Range Bitmap 能做字符串范围、而 BSI 不能的根因。
- **TopN 独家能力**：`visitTopN` 按排序方向走 `bitmap.bottomK`（升序）/ `bitmap.topK`（降序），还能接收上一步过滤出的 `foundSet` 先缩集再取 TopN（`RangeBitmapFileIndex.java:166-183`）。这是四类索引中唯一实现 `visitTopN` 的，`ORDER BY col LIMIT N` 的下推全靠它。
- **字典分块**：字典按 `chunk-size`（`CHUNK_SIZE="chunk-size"`，`RangeBitmapFileIndex.java:50`）切块，定长/变长 chunk 支持按需加载，避免高基数列一次性把整本字典读进内存。

为什么不直接给 BSI 加字符串支持就好？因为字符串没有天然的"位"可切片，必须先字典编码；而一旦引入字典，就顺带获得了"有序整数 code"，TopN 也就水到渠成——所以这两个能力是同一个设计决策的产物，不是分别加的。

**③ 关键源码**（`RangeBitmapFileIndex.java`）

Reader 把每种谓词都翻译成一次 `RangeBitmap` 调用，全部返回行级 `BitmapIndexResult`：

**④ 风险 / 陷阱 / 边界**

- **实现最重、写入最贵**：要先收集值、建字典、再建 BSI，写期开销与内存压力是四类里最高的。不要对"只做等值"的列建 Range Bitmap，那是用大炮打蚊子——等值用 Bloom Filter 或低基数 Bitmap 即可。
- **TopN 命中有条件**：`evaluateTopN` 里若结果基数已 ≤ k 会直接返回、不再走索引（`FileIndexPredicate.java:91-102`）；且只取 `orders.get(0)`，多列排序只用第一列做索引下推，其余列仍需引擎排序。

**⑤ 收益与代价**

收益：四类里能力最全——数值/字符串范围、等值/IN、NULL、`between`、以及独家的 TopN 全部行级精确。代价：实现与写入成本最高、字典带来额外空间与一层间接。**适用**：字符串范围查询、`ORDER BY col LIMIT N` 下推、需要"一列顶多种谓词"的高价值列；**不适用**：纯等值（用 Bloom）、纯数值范围且不需要 TopN（用更轻的 BSI）。

## 7. 索引与查询的协同：谓词如何命中索引

四类索引各自能跳文件或跳行，但一条 `WHERE` 里往往是多列、多谓词、AND/OR 嵌套。本章讲清楚：一个 `Predicate` 树是怎么被评估成"读/跳文件/读哪几行"的单一结论的，以及"跳文件"和"跳行"两种结果如何在同一棵树里自然共存。这是 04（DeletionVector）、12（查询优化）会引用的核心机制。

### 7.1 FileIndexPredicate 评估流程

**① 要解决什么问题**

读取一个数据文件前，要先用它内嵌的索引判断"这个文件 / 这文件的哪几行值得读"。输入是查询下推下来的 `Predicate` 树，输出要是一个能直接驱动"跳文件 / 跳行"的结论。

**② 设计原理与取舍**

`FileIndexPredicate.evaluate(predicate)` 是入口（`FileIndexPredicate.java:75`）。流程三步：① `getRequiredNames` 收集谓词涉及的列名（`L117`）；② 对每个列名 `reader.readColumnIndex(name)` 拿到该列上**所有**索引的 `FileIndexReader`（一列可叠多种索引）；③ 用内部 `FileIndexPredicateTest` 这个 `PredicateVisitor` 把整棵树折叠成一个 `FileIndexResult`。一句话哲学：**谓词树的结构（AND/OR/叶子）由 visitor 递归，索引类型差异由叶子处 `function().visit(reader, …)` 派发——两层正交。**

**③ 关键源码**（`FileIndexPredicate.java`）

叶子谓词：对同一列的多个索引 Reader 依次 `.and()`，任一 `SKIP` 立即短路返回（`L156-177`，已核对）：

```java
public FileIndexResult visit(LeafPredicate predicate) {
    FileIndexResult compoundResult = REMAIN;
    FieldRef fieldRef = ...;
    for (FileIndexReader r : columnIndexReaders.get(fieldRef.name())) {
        compoundResult = compoundResult.and(
                predicate.function().visit(r, fieldRef, predicate.literals()));
        if (!compoundResult.remain()) return compoundResult;   // 短路：已能跳文件
    }
    return compoundResult;
}
```

复合谓词：`Or` 把子结果依次 `.or()`，否则（AND）依次 `.and()` 并短路（`L179-204`）。`predicate.function().visit(reader, …)` 就是把"等值/范围/IN"这个动作派发到 Reader 的对应 `visitXxx`——谁实现了哪个 visit，谁就能处理那种谓词；没实现的走父类默认 `REMAIN`（永不跳，安全兜底）。

**④ 风险 / 陷阱 / 边界**

- **函数包裹 = 索引失效**：`UPPER(col)='X'`、隐式类型转换会让谓词在下推前就无法对齐到某个裸列，`getRequiredNames` 拿不到、或 `fieldRefOptional` 为空（`L158-161` 返回 `REMAIN`），索引等于没建。
- **未实现的谓词静默放行**：给 Bloom Filter 列写范围条件，Reader 没有 `visitLessThan`，走默认 `REMAIN`——不会报错，但永远跳不了文件，容易"以为有索引在生效"。
- **TopN 走单独入口**：TopN 不在 `evaluate` 里，而在 `evaluateTopN`（`L91`）；若结果基数已 ≤ k 直接返回不走索引（`L97-102`），且只用 `orders.get(0)` 第一排序列。

**⑤ 收益与代价**

收益：一套 visitor 把"任意谓词树 + 任意索引组合"统一折叠成单一结论，新增索引/谓词零侵入；短路让"能跳就早跳"。代价：能力边界对用户不可见——谓词稍一变形（函数、类型转换）就静默失效，排障只能靠看是否真读了文件。

### 7.2 三态逻辑：REMAIN / SKIP / BitmapIndexResult

**① 要解决什么问题**

Bloom Filter 只能给出"读/跳文件"（布尔），Bitmap 系列能给出"读哪几行"（行号集合）。两类结果要能在同一棵 AND/OR 树里混合运算，不能因为类型不同就互不兼容。

**② 设计原理与取舍**

`FileIndexResult` 设计成三态而非布尔（`FileIndexResult.java`）：

- `REMAIN`（L24）：保留，无法过滤。`and(X)=X`，`or(_)=REMAIN`。
- `SKIP`（L42）：跳过整个文件。`and(_)=SKIP`，`or(X)=X`。
- `BitmapIndexResult`（`bitmap/BitmapIndexResult.java:29`）：携带 `RoaringBitmap32` 行号集，`remain()` 即"bitmap 非空"。

关键在 `BitmapIndexResult.and/or`（`L41/L50`）：两个 BitmapResult 相遇走 RoaringBitmap 原生 `and`/`or`（精确行级交并）；与 REMAIN/SKIP 相遇则回落到父类默认实现，自动退化成"跳文件/全保留"。一句话哲学：**用代数闭合的三态，让"跳文件"和"跳行"在 AND/OR 下自动取正确语义——调用方无需关心对端是哪类索引。**

举例：`bloom(col_a='x') AND bitmap(col_b IN (...))`——Bloom 返回 `REMAIN`（可能有），与 `BitmapResult` 做 `and`，`REMAIN.and(X)=X`，结果就是 Bitmap 的精确行集；若 Bloom 返回 `SKIP`，`SKIP.and(_)=SKIP`，整个文件直接跳过，Bitmap 都不用算（且 `BitmapIndexResult` 是 `LazyField`，懒求值，跳文件时根本不触发 bitmap 计算）。

**③ 关键源码**

| 结果 | `and(X)` | `or(X)` | 来源 |
|------|---------|---------|------|
| `REMAIN` | `X` | `REMAIN` | `FileIndexResult.java:24` |
| `SKIP` | `SKIP` | `X` | `FileIndexResult.java:42` |
| `BitmapIndexResult` | 同为 Bitmap→`RoaringBitmap32.and`，否则回落父类 | 同理 `or` | `BitmapIndexResult.java:41/50` |

**④ 风险 / 陷阱 / 边界**

- **行级结果是懒求值**：`BitmapIndexResult` 继承 `LazyField`，bitmap 只在 `get()`/`remain()` 时才算。AND 短路 + 懒求值共同保证"被跳掉的分支不付计算代价"，但也意味着同一结果多次 `get()` 要注意只算一次（已由 LazyField 缓存）。
- **三态不可与布尔混用**：调用方拿到 `FileIndexResult` 只应问 `remain()` 和组合 `and/or`，不要 instanceof 判死，否则新增结果类型会漏处理。

**⑤ 收益与代价**

收益：异构索引零成本组合，行级与文件级在一套代数里统一；懒求值让无效分支不耗 CPU。代价：三态语义对初学者不直观（`SKIP.or` 会"复活"对端），读源码需先吃透这张真值表。

### 7.3 行级过滤与 DV 的联合

**① 要解决什么问题**

索引算出"该读第 3、7、100 行"后，读取层要真的只吐这几行；而 MOW 表里这些行可能已被逻辑删除，得再扣掉。

**② 设计原理与取舍**

当评估结果是 `BitmapIndexResult` 时，读取层用 `ApplyBitmapIndexRecordReader` + `ApplyBitmapIndexFileRecordIterator`（`paimon-common/.../fileindex/bitmap/`，两类均已核对存在）把列式文件的行迭代器套一层，只放行 bitmap 命中的行。与删除向量的联合靠 `BitmapIndexResult.andNot(deletion)`（`BitmapIndexResult.java:58`）：从命中行集中按位扣除 DV 标记的已删除行。一句话哲学：**索引交"候选行"，DV 交"作废行"，一次 `andNot` 得到"真正要返回的行"——全程 RoaringBitmap 位运算，不落地中间行。**

> DeletionVector 的数据结构（32/64 位 RoaringBitmap、Magic Number、与 Iceberg Position Delete 的兼容、`BucketedDvMaintainer` 在 compaction 阶段生成 DV）是 **04 号文档的主讲内容**，本文不展开，详见 `04-DeletionVector...` 与 `01-核心存储引擎分析.md §7.2`。这里只需知道：DV 以"按行号的 RoaringBitmap"形式与文件索引的行级结果同构，所以能直接 `andNot`。

**③ 关键源码**

```java
// BitmapIndexResult.java:58 —— 行级结果扣除已删除行
public BitmapIndexResult andNot(RoaringBitmap32 deletion) {
    return new BitmapIndexResult(() -> RoaringBitmap32.andNot(get(), deletion));
}
```

**④ 风险 / 陷阱 / 边界**

- **行级过滤的收益取决于选择率**：命中率高（如 60% 的行都命中）时，跳行省不下多少 I/O，反而多了 bitmap 构建与位运算开销；选择率越低（命中越少）行级过滤越值。
- **行号语义是文件内行号**：文件索引返回的是 `[0, 文件行数)` 内的局部行号，不是全表 Row ID；跨文件/全表定位是"全局索引"的范畴（见 §9）。

**⑤ 收益与代价**

收益：把"跳行"和"扣删除行"统一成位运算，读取层只物化真正要返回的行，对低选择率查询是数量级的 I/O 节省。代价：bitmap 构建与 `andNot` 有固定开销，高选择率/小文件场景可能得不偿失。

## 8. 索引配置与最佳实践

### 8.1 配置方式与关键开关

**① 要解决什么问题**

用户需要声明"给哪列建哪种索引、参数多少、索引存哪"，且改了配置后存量文件要能补建。

**② 设计原理与取舍**

配置走 `file-index.<index-type>.columns` 前缀，由 `FileIndexOptions` 解析（前缀 `file-index`、后缀 `columns`，`FileIndexOptions.java:35-37/56-90`）。同一列可声明多种索引，查询时 §7.1 自动对同列多索引做 AND。

```sql
WITH (
  'file-index.bloom-filter.columns' = 'user_id,device',   -- 高基数等值，跳文件
  'file-index.bloom-filter.user_id.fpp'  = '0.05',         -- 单列覆盖 FPP（默认 0.1）
  'file-index.bloom-filter.user_id.items'= '2000000',      -- 预期元素数（默认 1,000,000）
  'file-index.bitmap.columns'       = 'status',            -- 低基数等值/IN/NULL，跳行
  'file-index.bsi.columns'          = 'amount',            -- 数值范围，跳行
  'file-index.range-bitmap.columns' = 'city_name'          -- 字符串范围/TopN，跳行
)
-- Map 嵌套列：'file-index.bloom-filter.columns' = 'props[country]'
```

**关键开关与默认值（核对自源码）**：

| 配置项 | 默认值 | 作用 | 源码 |
|--------|--------|------|------|
| `file-index.in-manifest-threshold` | **500 B**（已修正，非 32KB） | ≤ 阈值内嵌 Manifest，> 阈值写独立 `.idx` | `CoreOptions.java:385-389` |
| `file-index.read.enabled` | `true` | 读期是否启用文件索引过滤 | `CoreOptions.java:391` |
| `file-index.<type>.columns` | 无 | 声明索引列 | `FileIndexOptions.java:35-37` |
| `file-index.bloom-filter.<col>.fpp` | 0.1 | Bloom 误判率 | `BloomFilterFileIndex.java:51` |
| `file-index.bloom-filter.<col>.items` | 1,000,000 | Bloom 预期元素数 | `BloomFilterFileIndex.java:50` |
| `file-index.range-bitmap.<col>.chunk-size` | 类型相关 | Range Bitmap 字典分块大小 | `RangeBitmapFileIndex.java:50` |

> **（已修正）**：旧稿写键名 `file-index-in-manifest-threshold`（连字符）、默认 32KB——键名和默认值都错。真实键名是 `file-index.in-manifest-threshold`（点分），默认 **500 B**（`CoreOptions.java:386-388`，测试用例 `AppendOnlySimpleTableTest.java:789` 亦用 `"500 B"` 为基准）。默认值这么小，意味着实践中绝大多数索引落成独立 `.idx`，只有极小索引才内嵌进 Manifest。

**③ 关键源码**：存量重建走 `CALL sys.rewrite_file_index(...)`（Flink/Spark 均有 `RewriteFileIndexProcedure`），底层 `FileIndexProcessor.process()` 读旧 `.idx`、按当前配置重算、再按 `fileIndexInManifestThreshold()`（`FileIndexProcessor.java:79`）重新落地。

**④ 风险 / 陷阱 / 边界**

- **改配置不重建存量**：新配置只对新写文件生效，老文件需手动 `rewrite_file_index`，否则老数据查询不享受新索引。
- **阈值调大是双刃**：减少 `.idx` 小文件数，但 Manifest 膨胀、scan 反序列化变慢——在"对象存储小文件数"与"Manifest 大小"间权衡。
- **`file-index.read.enabled=false` 会全局关闭过滤**：排障时可临时关闭对比，但生产别误关。

**⑤ 收益与代价**：收益是声明式、列粒度、可叠加、可补建；代价是配置项分散（列级 override 嵌在 key 里）、改配置的存量重建需人工触发且要全读数据文件（大文件有 I/O 成本）。

### 8.2 选型与组合策略

把四类索引的选型压成一张"按查询模式 + 列基数"的决策表（依据见 §3–§6 各章数据结构分析）：

| 查询模式 | 列基数 | 首选 | 为什么 / 不选谁 |
|---------|-------|------|----------------|
| `col = x`（只跳文件够用） | 高基数 | **Bloom Filter** | 每元素几 bit，性价比之王；不选 Bitmap（高基数内存爆） |
| `col = x` / `IN` 且要跳行 | 低基数（< ~1000） | **Bitmap** | 返回精确行号；高基数禁用 |
| `IS NULL` / `IS NOT NULL` | 任意 | **Bitmap / BSI / Range Bitmap** | 三者支持 null；Bloom 不支持 |
| 数值范围 `>` `<` `between` | 任意（数值） | **BSI** | slice 数 = 位宽与基数无关，比 Range Bitmap 轻 |
| 字符串范围 | 任意 | **Range Bitmap** | BSI 仅数值，字符串必须字典编码 |
| `ORDER BY col LIMIT N` | 任意 | **Range Bitmap** | 唯一实现 `visitTopN` |
| 全文检索 / 向量检索 | — | **全局索引**（§9） | 文件级索引无法跨文件返回全局 Row ID |

**组合策略**：同列叠 `bloom-filter`（粗筛跳文件）+ `bitmap`（精确跳行）是常见搭配——Bloom 先把大量无关文件挡掉，剩下文件用 Bitmap 跳行，§7.1 自动 AND。但别无脑全配：每种索引都要写期开销和空间（§8.3），只给"真正高频被过滤的列"建。

**何时不该建索引**：① 选择率高（过滤后还剩大半行）的列，索引省不下 I/O；② 几乎不作过滤条件的列；③ 小文件为主的表（索引固定开销占比高）；④ 写入吞吐已是瓶颈时谨慎加重索引（尤其 BSI/Range Bitmap）。

### 8.3 索引对写入的代价

**① 要解决什么问题**：索引不是免费的，写期要额外收集值、占内存、序列化。建索引前要知道每类的写入代价量级。

**② 设计原理与取舍**：代价直接由数据结构决定（呼应 §3–§6）：

| 索引 | 写入开销 | 写期内存 | 根因 |
|------|---------|---------|------|
| Bloom Filter | 低 | O(bit 数组) | 仅一个 bit 数组，逐值 `addHash` |
| Bitmap | 中-高 | **O(基数)** | `Map<值, RoaringBitmap>` 每个不同值一个 bitmap |
| BSI | 中 | **O(行数)** | `StatsCollectList` 暂存全部值到 `List<Long>` |
| Range Bitmap | 中-高 | O(字典 + BSI) | 先建字典再建 BSI |

**④ 风险 / 陷阱 / 边界**

- **BSI/Range Bitmap 写大文件 OOM**：BSI 的 `StatsCollectList` 暂存全部值，源码注释直言 `// todo: Find a way to reduce the risk of out-of-memory.`（`BitSliceIndexBitmapFileIndex.java:168`）。单文件越大、行数越多，写期内存峰值越高——用 `target-file-size` / `compaction.max-file-size` 控制单文件大小。
- **Bitmap 高基数内存爆**：`id2bitmap` 与基数同步膨胀（§4），建前先 `COUNT(DISTINCT col)`。
- **多 Bloom Filter AND 误判累积**：单 FPP=0.1，多列 AND 时整体假命中率高于单值，高精度需调小 `fpp`（空间翻倍）或改用精确 Bitmap。

**⑤ 收益与代价**：收益是查询期跳文件/跳行的 I/O 节省；代价是写期 CPU/内存与对象存储的索引文件（小文件数 / Manifest 体积）。**核心权衡：索引把成本从"读期、按查询次数重复付"前移到"写期、一次性付"——查询越频繁、选择率越低，越划算。**

## 9. 其余索引概览（各有主讲，交叉引用）

本文聚焦"文件内嵌的四类列索引"。Paimon 里还有几类与"索引"沾边、但职责与主讲文档不同的机制，这里只给定位与边界，不展开实现——避免与主讲文档重复。

| 机制 | 一句话定位 | 与四类文件索引的本质区别 | 主讲文档 | 关键源码入口 |
|------|-----------|------------------------|---------|-------------|
| **DeletionVector（DV）** | MOR/MOW 下记录"某文件哪些行已逻辑删除"的行号 bitmap | 它标的是"作废行"，不是"候选行"；与本文行级结果同构，靠 `andNot` 联合（§7.3） | **04**（机制主讲）、`01 §7.2`（概览） | `paimon-core/.../deletionvectors/` |
| **Hash Index（动态 Bucket 索引）** | 动态桶表记录 `key hash → bucket`，保证同一 key 始终落同一桶 | 写期用于 bucket 路由，不参与查询过滤；只存 hashCode 不存值 | `01 §10.3`（动态桶索引维护） | `paimon-core/.../index/`（`HashBucketAssigner`、`PartitionIndex`、`HashIndexFile`） |
| **Lookup（点查加速）** | 把 SST/列式文件转成本地 KV + 内嵌 Bloom，支持微秒级点查 | 是"读取加速结构"而非"过滤索引"；服务 Lookup Join / Partial-Update 的按主键点查 | `01 §6`（LookupLevels） | `paimon-core/.../mergetree/LookupLevels.java`、`LookupFile.java` |
| **全局索引（Global Index）** | 跨全表返回**全局 Row ID**（`RoaringNavigableMap64`），分 BTree / Bitmap 两类 | 作用域是整表、返回全局行号；构建需全表扫描+外排，成本远高于文件索引 | 全文检索见 paimon-tantivy、向量见 paimon-lumina | `paimon-common/.../globalindex/`（`GlobalIndexer`、`BTreeGlobalIndexer`、`BitmapGlobalIndex`） |

**几个易混点（只点关键，详见各主讲文档）**：

- **文件索引 vs 全局索引**：文件索引行号是"文件内 `[0, 文件行数)`"，全局索引是"全表 `[0, 表总行数)`"；后者通过按行范围分片（rowRange）+ `offset()` 把分片内行号偏移到全局空间。普通等值/范围用文件索引，只有全文/向量检索这种必须跨文件的才上全局索引。
- **Hash Index 不是查询索引**：它的唯一职责是写入期把 key 路由到确定的 bucket（动态桶），查询加速靠 bucket 裁剪而非它本身。`SimpleHashBucketAssigner` 用于 overwrite（不加载历史索引）。
- **Lookup 里的 Bloom Filter ≠ 本文的 Bloom Filter 文件索引**：Lookup 文件内嵌的 Bloom Filter 用于"点查时快速判负"（key 不存在直接跳过磁盘 I/O），与列级 `bloom-filter` 文件索引是两套东西，只是同名思想。

## 10. 设计决策总结

| 决策点 | 选择 | 取舍 / 代价 | 收益 |
|--------|------|------------|------|
| 一种万能索引 vs 按谓词分工 | **四类分工**（Bloom/Bitmap/BSI/Range Bitmap） | 用户需懂选型；配错=没建 | 每类在自己谓词上做到极致，空间/精度/谓词三者按需取舍 |
| Bloom Filter 的能力边界 | 只 `visitEqual`、只跳文件、允许误判 | 不支持范围/NULL/IN；约 10% 文件假命中需实读 | 每元素几 bit，高基数等值性价比之王 |
| Bitmap 行号集结构 | `RoaringBitmap32` 倒排 + 单值负数内联 + V2 块加载 | 内存/空间 ∝ 基数，只宜低基数 | 精确行级 + 原生位运算多条件组合 + null 一等公民 |
| 数值范围如何做 | **BSI 位切片**（slice=位宽，与基数无关）+ 正负数分建 | 仅数值；写期暂存全部值有 OOM 风险 | 高基数数值列范围也扛得住，空间随位宽常数增长 |
| 字符串范围 + TopN | **Range Bitmap = 字典编码 + BSI** | 实现最重、写入最贵 | 独家支持字符串范围与 `ORDER BY col LIMIT N` 下推 |
| 结果类型 | **三态** `REMAIN`/`SKIP`/`BitmapIndexResult` | 语义不直观、需吃透真值表 | 跳文件与跳行在 AND/OR 下代数闭合、异构索引零成本组合 |
| 行级结果求值 | `BitmapIndexResult` 懒求值（`LazyField`） | 多次 get 需缓存（已处理） | 被短路掉的分支不付 bitmap 计算代价 |
| 索引存哪 | 阈值 `file-index.in-manifest-threshold`（默认 **500B**） | 默认值极小→多数落独立 `.idx` 小文件；调大则 Manifest 膨胀 | 小索引内嵌随 scan 即用，大索引独立避免撑大 Manifest |
| 索引与数据生命周期 | 内嵌 DataFileMeta / 挂 extraFiles | 改配置不自动重建，需 `rewrite_file_index` | 随数据原子提交、随删自动失效，无索引/数据不一致 |
| 扩展性 | SPI `FileIndexerFactory`（`ServiceLoader`） | 同 identifier 注册两次只留一个并告警 | 全文/向量等第三方索引零侵入接入 |
| 与 DV 的协同 | 行级 `BitmapIndexResult.andNot(dv)` | 高选择率/小文件时位运算开销可能不划算 | "候选行 − 作废行"一次位运算得最终行集，不物化中间行 |

**与 Iceberg 横向定位**：Iceberg 主要靠 Parquet/ORC 列级 Bloom Filter + min/max（Puffin/Footer），最多做到 Page 级跳过；Paimon 的差异化在于 **Bitmap/BSI/Range Bitmap 能返回精确行号集**（行级过滤）、**同列可叠加多索引自动 AND/OR**、**Range Bitmap 独家 TopN 下推**、以及 **SPI 可扩展**。代价是 Paimon 这套索引更"重"、配置与选型门槛更高，且生态成熟度不及 Parquet/ORC 原生 Bloom Filter。
