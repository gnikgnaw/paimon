## 1. Compaction 调优

### 解决什么问题

**核心业务问题**: LSM-Tree 架构的主键表在持续写入时会产生大量小文件,导致读取性能急剧下降。没有 Compaction,每次查询都需要扫描所有文件,读放大严重。

**实际场景**:
- CDC 实时入湖场景: 每秒产生数百个小文件,查询延迟从毫秒级上升到秒级
- 高频更新场景: 同一个 key 的多个版本分散在不同文件中,点查需要读取所有版本
- 存储空间膨胀: 历史版本未清理,存储成本是实际数据量的 3-5 倍

**没有这个设计的后果**:
1. 读取性能线性下降: 文件数量从 10 个增长到 1000 个,查询延迟增长 100 倍
2. 元数据爆炸: Manifest 文件记录数百万个小文件,扫描 manifest 本身就需要数十秒
3. 文件系统压力: HDFS NameNode 或 S3 listing 操作超时
4. 内存溢出: 打开过多文件句柄导致 OOM

### 有什么坑

**误区 1: 盲目减小 compaction-trigger 追求读性能**
```properties
# 错误配置
num-sorted-run.compaction-trigger = 2  # 过于激进
```
**后果**: Compaction 过于频繁,写入吞吐下降 50% 以上,checkpoint 超时频繁。

**误区 2: 写入和 Compaction 混在一起**
**现象**: 写入任务的 checkpoint 时间从 30 秒突增到 5 分钟,甚至超时失败。
**原因**: Compaction 占用大量 CPU 和 I/O,与写入争抢资源。
**解决**: 使用 Dedicated Compaction 模式 (`write-only = true`)。

**误区 3: 忽略 stop-trigger 的重要性**
```properties
# 危险配置
num-sorted-run.compaction-trigger = 10
num-sorted-run.stop-trigger = 100  # 过大
```
**后果**: 在 Compaction 跟不上时,文件数量疯狂增长到 100 个才阻塞写入,此时读取已经完全不可用。

**生产环境注意事项**:
- 监控 `maxLevel0FileCount` 指标,超过 `stop-trigger` 的 80% 就应该告警
- Compaction 线程数不足时,增大 `compaction.max-file-num` 无效,需要增加并行度
- 对象存储 (S3/OSS) 场景下,Compaction 的网络 I/O 成本远高于本地磁盘

**性能陷阱**:
- 空间放大触发全量 Compaction 时,会合并所有层的数据,耗时可能达到小时级
- Off-peak 配置的时区问题: 配置的是 JVM 时区,不是数据中心时区
- 在 Kubernetes 环境中,Compaction 的 CPU 限流会导致任务假死

### 核心概念解释

**Sorted Run**: LSM-Tree 中的基本单位,表示一组有序且 key 范围不重叠的文件。
- Level0: 每个文件是一个 sorted run (key 范围可能重叠)
- Level1+: 每个 level 是一个 sorted run (level 内文件 key 范围不重叠)

**读放大 vs 写放大**:
- 读放大: 读取一条记录需要扫描的文件数量。sorted run 越多,读放大越大。
- 写放大: 一条记录从写入到最终落盘被写入的次数。Compaction 越频繁,写放大越大。
- 平衡点: `num-sorted-run.compaction-trigger` 就是这个平衡点的控制旋钮。

**与 RocksDB 的对比**:
- Paimon 的 Universal Compaction 直接借鉴 RocksDB 的实现
- 区别: Paimon 是分布式存储,Compaction 可以在独立的 Flink 任务中执行
- RocksDB 的 Compaction 是单机多线程,Paimon 可以跨机器并行

**与 Hudi/Iceberg 的对比**:
- Hudi: 使用 Copy-on-Write 或 Merge-on-Read,Compaction 是可选的
- Iceberg: 主要依赖 snapshot 隔离,没有 LSM-Tree 的 Compaction 概念
- Paimon: 强制 Compaction,保证读取性能的确定性

### 设计理念

**为什么选择 LSM-Tree**: 
1. 写入友好: 顺序写入 level0,避免随机 I/O
2. 更新高效: 同一 key 的多个版本可以在内存中合并,减少磁盘写入
3. 压缩友好: 大文件压缩比更高,节省存储成本

**为什么借鉴 RocksDB Universal Compaction**:
- 经过 Facebook 生产环境验证的成熟算法
- 相比 Leveled Compaction,写放大更低 (适合写多读少场景)
- 空间放大可控 (通过 `max-size-amplification-percent` 限制)

**权衡取舍**:
- 选择更小的 `compaction-trigger`: 牺牲写入吞吐,换取稳定的读取性能
- 选择更大的 `compaction-trigger`: 牺牲读取性能,换取更高的写入吞吐
- Dedicated Compaction: 牺牲运维复杂度 (需要额外任务),换取写入和读取的解耦

**架构演进**:
- 早期版本: 只支持写入内 Compaction,运维简单但性能受限
- 当前版本: 支持 Dedicated Compaction,适应大规模生产环境
- 未来方向: 自适应 Compaction (根据负载自动调整参数)

**业界对比**:
- Paimon 的 Compaction 是强制的,保证读取性能的下界
- Hudi 的 Compaction 是可选的,灵活但需要人工介入
- Iceberg 依赖 snapshot 隔离,避免了 Compaction 的复杂性,但更新性能较差

### 1.1 Compaction 核心参数解析

Paimon 的主键表采用 LSM-Tree 架构，Compaction 策略基于 RocksDB 的 Universal Compaction 实现。核心实现在 `UniversalCompaction.java`。

#### 1.1.1 `num-sorted-run.compaction-trigger`

- **源码位置**: `CoreOptions.java:745-751`
- **默认值**: `5`
- **含义**: 触发 Compaction 的 Sorted Run 数量阈值。包含 level0 文件 (一个文件一个 sorted run) 和高层级的 run (一个 level 一个 sorted run)。

**源码影响路径**:

在 `UniversalCompaction.pick()` 方法 (第 67-107 行) 中，Compaction 选择按以下优先级依次执行:

```
步骤 0: earlyFullCompact (如果配置了 optimization-interval 等触发条件)
步骤 1: pickForSizeAmp -> 检查空间放大是否超阈值
步骤 2: pickForSizeRatio -> 基于 size ratio 选择合并候选
步骤 3: 如果 runs.size() > numRunCompactionTrigger -> 强制选择合并
```

关键判断在 `pickForSizeAmp()` (第 125-147 行) 和 `pickForSizeRatio()` (第 149-182 行) 中:
- 如果 `runs.size() < numRunCompactionTrigger`，则步骤 1 和步骤 2 都直接返回 null
- 只有当 sorted run 数量达到触发阈值时才会尝试合并

**为什么重要**: 该值直接决定了读放大与写放大的平衡点。值越小，合并越频繁 (写放大高但读性能好)；值越大，合并越少 (写吞吐高但读需要扫描更多文件)。

#### 1.1.2 `num-sorted-run.stop-trigger`

- **源码位置**: `CoreOptions.java:753-759`
- **默认值**: `num-sorted-run.compaction-trigger + 3` (在 `numSortedRunStopTrigger()` 方法第 2987-2993 行计算)
- **含义**: 当 sorted run 数量超过该值时，写入将被阻塞等待 Compaction 完成。

**源码影响路径**:

在 `MergeTreeCompactManager.shouldWaitForLatestCompaction()` (第 109-111 行):
```java
public boolean shouldWaitForLatestCompaction() {
    return levels.numberOfSortedRuns() > numSortedRunStopTrigger;
}
```

还有 `shouldWaitForPreparingCheckpoint()` (第 114-117 行):
```java
public boolean shouldWaitForPreparingCheckpoint() {
    return levels.numberOfSortedRuns() > (long) numSortedRunStopTrigger + 1;
}
```

**为什么重要**: 这是写入背压的最后防线。当 Compaction 跟不上写入速度时，该参数控制最大允许的文件堆积量，超过则暂停写入。

**好处**: 防止 sorted run 无限增长导致读取性能急剧下降和内存不足。

#### 1.1.3 `compaction.max-size-amplification-percent`

- **源码位置**: `CoreOptions.java:798-804`
- **默认值**: `200`
- **含义**: 空间放大百分比阈值，即存储一个字节数据所需的额外存储空间百分比 (针对 changelog 模式表)。

**源码影响路径**:

在 `UniversalCompaction.pickForSizeAmp()` (第 125-147 行):
```java
long candidateSize = runs.subList(0, runs.size() - 1).stream()
        .map(LevelSortedRun::run).mapToLong(SortedRun::totalSize).sum();
long earliestRunSize = runs.get(runs.size() - 1).run().totalSize();
// 当除最底层外的所有层总大小超过最底层的 maxSizeAmp% 时触发全量合并
if (candidateSize * 100 > maxSizeAmp * earliestRunSize) {
    return CompactUnit.fromLevelRuns(maxLevel, runs);
}
```

**为什么重要**: 当空间放大超过阈值时，会触发全量 Compaction (将所有 sorted run 合并到最大层)，这是一个代价昂贵的操作，但能有效控制存储空间。

#### 1.1.4 `compaction.size-ratio`

- **源码位置**: `CoreOptions.java:813-820`
- **默认值**: `1`
- **含义**: 在比较 sorted run 大小时的百分比灵活度。

**源码影响路径**:

在 `UniversalCompaction.pickForSizeRatio()` (第 163-182 行):
```java
if (candidateSize * (100.0 + sizeRatio + ratioForOffPeak()) / 100.0 < next.run().totalSize()) {
    break; // 候选集大小 < 下一个 run 大小的 (100+ratio)%, 停止扩展
}
candidateSize += next.run().totalSize();
candidateCount++;
```

**为什么重要**: 较大的 `size-ratio` 意味着更多的 sorted run 会被包含进合并候选集，产生更积极的合并行为。

#### 1.1.5 Off-Peak Compaction

- **源码位置**: `CoreOptions.java:822-854`
- **配置项**:
  - `compaction.offpeak.start.hour` (默认 -1，禁用)
  - `compaction.offpeak.end.hour` (默认 -1，禁用)
  - `compaction.offpeak-ratio` (默认 0)

**源码影响路径**:

在 `UniversalCompaction.ratioForOffPeak()` (第 184-186 行):
```java
private int ratioForOffPeak() {
    return offPeakHours == null ? 0 : offPeakHours.currentRatio(LocalDateTime.now().getHour());
}
```

在 `pickForSizeRatio` 计算中，off-peak ratio 会被加到 sizeRatio 上，使得低峰时段合并更积极。

**好处**: 可以在业务低峰期 (如凌晨 2 点到早上 6 点) 执行更积极的 Compaction，减少对在线业务的影响。

#### 1.1.6 Early Full Compaction

- **源码位置**: `EarlyFullCompaction.java`
- **相关配置**:
  - `compaction.optimization-interval`: 周期性触发全量合并的间隔
  - `compaction.total-size-threshold`: 总大小低于此阈值时强制全量合并
  - `compaction.incremental-size-threshold`: 增量大小超过此阈值时强制全量合并

**源码影响路径**:

在 `EarlyFullCompaction.tryFullCompact()` (第 70-107 行)，按优先级检查三个条件:
1. 距上次全量合并是否超过 `fullCompactionInterval`
2. 所有 run 总大小是否小于 `totalSizeThreshold`
3. 非最大层的增量大小是否超过 `incrementalSizeThreshold`

**为什么有用**: `compaction.optimization-interval` 确保读优化系统表 (read-optimized) 的查询时效性，适合需要定期保证查询性能的场景。

### 1.2 Dedicated Compaction vs 写入内 Compaction

#### 写入内 Compaction (默认模式)

- 写入和 Compaction 在同一个 Flink Job 中执行
- 优点: 运维简单，不需要额外任务
- 缺点: 写入和 Compaction 争抢资源，可能导致写入延迟

#### Dedicated Compaction

- **启用方式**: 写入任务设置 `write-only = true` (源码位置: `CoreOptions.java:634-641`)
- **源码说明**: `"If set to true, compactions and snapshot expiration will be skipped. This option is used along with dedicated compact jobs."`

**为什么这么做**: 将 Compaction 负载从写入链路分离出来，避免 Compaction 造成的写入背压和 checkpoint 超时。

**好处**:
1. 写入不会因 Compaction 被阻塞 (不会触发 `shouldWaitForLatestCompaction()` 返回 true)
2. 可以独立调整 Compaction 资源
3. Compaction 可以在低峰期更积极地执行

### 1.3 场景化推荐配置

#### 高吞吐写入场景

```properties
# 放宽 Compaction 触发条件，减少写入阻塞
num-sorted-run.compaction-trigger = 10
num-sorted-run.stop-trigger = 15
# 使用 Dedicated Compaction
write-only = true
# 提高空间放大容忍度
compaction.max-size-amplification-percent = 400
```

**为什么**: 增大触发阈值减少 Compaction 频率，配合 dedicated compaction 完全消除写入路径上的合并开销。

#### 读多写少场景

```properties
# 更积极的 Compaction，保证读取性能
num-sorted-run.compaction-trigger = 3
num-sorted-run.stop-trigger = 6
compaction.size-ratio = 5
# 开启定期优化合并
compaction.optimization-interval = 30min
# 控制空间放大
compaction.max-size-amplification-percent = 100
```

**为什么**: 低触发阈值确保文件数量少，读取扫描更少文件。定期优化合并确保读取性能稳定。

#### 延迟敏感场景

```properties
# 适中的触发条件
num-sorted-run.compaction-trigger = 5
num-sorted-run.stop-trigger = 8
# 使用 Dedicated Compaction 避免写入抖动
write-only = true
# 低峰期更积极合并
compaction.offpeak.start.hour = 2
compaction.offpeak.end.hour = 8
compaction.offpeak-ratio = 20
```

**为什么**: Dedicated Compaction 消除写入毛刺，off-peak 配置在低峰期补回 Compaction 进度。

## 2. 内存管理调优

### 解决什么问题

**核心业务问题**: 多 bucket 并发写入时,每个 bucket 需要独立的内存缓冲区,总内存需求 = bucket 数量 × write-buffer-size。内存不足会导致 OOM 或频繁溢写,严重影响性能。

**实际场景**:
- 高基数分区表: 1000 个分区 × 256 bucket = 256,000 个潜在 writer,内存需求达到 TB 级
- 实时宽表写入: 单表 512 个 bucket,每个 256MB buffer,总需求 128GB
- 批量回填历史数据: 同时写入数百个分区,内存瞬间耗尽

**没有这个设计的后果**:
1. OOM 频繁: TaskManager 反复重启,任务永远无法完成
2. 写入阻塞: 内存不足时 writer 相互等待,吞吐下降到个位数 MB/s
3. 数据丢失风险: OOM 导致 checkpoint 失败,已写入数据无法提交

### 有什么坑

**误区 1: 盲目增大 write-buffer-size**
```properties
# 危险配置
write-buffer-size = 1gb  # 看起来很爽
bucket = 256
# 实际内存需求: 256GB!
```
**后果**: 单个 TaskManager 内存只有 16GB,瞬间 OOM。

**误区 2: 关闭溢写功能追求性能**
```properties
write-buffer-spillable = false  # 想避免磁盘 I/O
```
**后果**: 内存不足时无路可退,直接 OOM。生产环境必须开启溢写作为最后防线。

**误区 3: 忽略 page-size 的影响**
```properties
page-size = 4kb  # 想节省内存
```
**后果**: 内存分配次数暴增,GC 压力巨大,吞吐下降 50%。

**生产环境注意事项**:
- 监控 `bufferPreemptCount` 指标,持续增长说明内存严重不足
- 对象存储场景下,溢写到本地磁盘可能比直接写对象存储更快
- Kubernetes 环境中,容器内存限制必须大于 JVM 堆内存 + 堆外内存 + 系统开销

**性能陷阱**:
- 内存抢占机制是同步的,抢占时会阻塞当前 writer
- 溢写文件的清理依赖 checkpoint 完成,checkpoint 失败会导致磁盘堆积
- 多个 bucket 共享内存池时,热点 bucket 会"饿死"冷门 bucket

### 核心概念解释

**MemoryOwner**: 内存使用者的抽象接口,每个 bucket 的 writer 都是一个 MemoryOwner。
- `setMemoryPool()`: 绑定到共享内存池
- `memoryOccupancy()`: 报告当前占用量
- `flushMemory()`: 被抢占时释放内存 (溢写到磁盘)

**内存抢占 (Preemption)**: 当内存池满时,找到占用最多的 MemoryOwner,强制其溢写释放内存。
- 类似操作系统的页面置换算法
- 优先牺牲"富有"的 writer,保证"贫穷"的 writer 能继续工作

**磁盘溢写 (Spill)**: 将内存中的排序数据临时写入磁盘,后续合并时再读回。
- 溢写文件是临时的,checkpoint 后会被删除
- 溢写数据会被压缩 (默认 zstd),减少磁盘占用

**与 Spark/Flink 的对比**:
- Spark: 使用 Tungsten 内存管理,基于 page 的分配
- Flink: 使用 MemorySegment,固定大小的内存块
- Paimon: 借鉴 Flink 的设计,但增加了跨 bucket 的抢占机制

### 设计理念

**为什么需要内存抢占**:
- 数据倾斜是常态: 某些 bucket 的数据量远超其他 bucket
- 静态分配低效: 给每个 bucket 预留固定内存会导致大量浪费
- 动态平衡: 让热点 bucket 多用内存,冷门 bucket 少用内存

**为什么需要磁盘溢写**:
- 内存是有限的: 无论如何抢占,总内存是固定的
- 防止 OOM: 溢写是最后的安全阀,牺牲性能保证任务不挂
- 渐进式降级: 内存不足时性能下降,但不会完全失败

**权衡取舍**:
- 大 write-buffer: 减少 flush 频率,但增加内存压力和 OOM 风险
- 小 write-buffer: 内存安全,但 flush 频繁,Compaction 压力大
- 开启溢写: 防止 OOM,但增加磁盘 I/O 和延迟

**架构演进**:
- 早期版本: 每个 bucket 独立内存,容易 OOM
- 当前版本: 共享内存池 + 抢占机制,内存利用率提升 3-5 倍
- 未来方向: 基于机器学习的自适应内存分配

### 2.1 核心内存参数

#### 2.1.1 `write-buffer-size`

- **源码位置**: `CoreOptions.java:657-662`
- **默认值**: `256 MB`
- **含义**: 在将数据转换为排序的磁盘文件之前，在内存中积累的数据量。

**源码影响路径**:

`write-buffer-size` 决定了 `SortBufferWriteBuffer` (第 65-74 行) 中 sort buffer 的大小。当 buffer 满后会触发 flush 操作，将内存中的排序数据写入 level0 文件。

**为什么重要**: 更大的 write buffer 意味着:
1. 每次 flush 产生更大的 level0 文件，减少 Compaction 频率
2. 内存中合并更多记录，减少重复 key 的磁盘写入
3. 但会增加内存消耗

#### 2.1.2 `page-size`

- **源码位置**: `CoreOptions.java:701-705`
- **默认值**: `64 KB`
- **含义**: 内存页大小，是内存分配的最小单元。

**源码影响路径**:

在 `MemoryPoolFactory` (第 46 行):
```java
this.totalBufferSize = (long) totalPages * innerPool.pageSize();
```

页大小影响内存分配粒度。较大的页减少分配次数但可能浪费空间，较小的页更灵活但增加管理开销。

#### 2.1.3 `sort-spill-threshold`

- **源码位置**: `CoreOptions.java:606-612`
- **默认值**: `numSortedRunStopTrigger + 1` (见 `sortSpillThreshold()` 方法第 2803-2810 行，当未显式配置时自动计算)
- **含义**: 当排序读取器的最大数量超过此值时，将尝试磁盘溢写。

**为什么重要**: 防止合并排序时因为打开过多的 sorted run 读取器而导致 OOM。当 sorted run 数量超过此阈值时，会将部分数据溢写到磁盘。

### 2.2 MemoryOwner 和内存分配机制

#### 源码分析: `MemoryOwner.java` 和 `MemoryPoolFactory.java`

`MemoryOwner` 接口定义了内存使用者的三个核心方法:

```java
public interface MemoryOwner {
    void setMemoryPool(MemorySegmentPool memoryPool);  // 设置内存池
    long memoryOccupancy();                             // 当前内存占用量
    void flushMemory() throws Exception;                // 释放内存 (溢写到磁盘)
}
```

**内存抢占机制** (`MemoryPoolFactory.java` 第 73-93 行):

当某个 `MemoryOwner` 需要分配内存页但内存池已满时:
1. 遍历所有 `MemoryOwner`，找到占用内存最大的 (排除请求者自身)
2. 调用最大内存占用者的 `flushMemory()` 方法，将其数据溢写到磁盘
3. 释放的内存可供当前请求者使用

```java
private void preemptMemory(MemoryOwner owner) {
    long maxMemory = 0;
    MemoryOwner max = null;
    for (MemoryOwner other : owners) {
        if (other != owner && other.memoryOccupancy() > maxMemory) {
            maxMemory = other.memoryOccupancy();
            max = other;
        }
    }
    if (max != null) {
        max.flushMemory();  // 强制最大占用者溢写
        ++bufferPreemptCount;
    }
}
```

**为什么这么设计**: 多个 bucket 的 writer 共享同一个内存池，通过抢占机制实现动态平衡，避免热点 bucket 占满内存导致其他 bucket 无法写入。

### 2.3 磁盘溢写机制

#### `write-buffer-spillable`

- **源码位置**: `CoreOptions.java:672-676`
- **默认值**: `true`
- **含义**: write buffer 是否可以溢写到磁盘。

#### `write-buffer-spill.max-disk-size`

- **源码位置**: `CoreOptions.java:665-670`
- **默认值**: 无限制 (`MemorySize.MAX_VALUE`)
- **含义**: 溢写到磁盘的最大容量限制。

#### `sort-spill-buffer-size`

- **源码位置**: `CoreOptions.java:614-618`
- **默认值**: `64 MB`
- **含义**: 溢写排序时每次写入磁盘的数据量。

#### `spill-compression`

- **源码位置**: `CoreOptions.java:620-625`
- **默认值**: `zstd`
- **含义**: 溢写数据的压缩算法。支持 zstd、lz4、lzo 等。

**为什么需要溢写**: 当 write-buffer 内存不足且无法通过抢占获取更多内存时，溢写机制可以将内存中的排序数据临时写入磁盘，防止 OOM。后续合并时再从磁盘读回。

#### `write-max-writers-to-spill`

- **源码位置**: `CoreOptions.java:685-690`
- **默认值**: `10`
- **含义**: 批量追加写入时，如果 writer 数量超过此值，自动开启 buffer cache 和溢写功能。

**为什么**: 大量分区的批写入场景下，每个分区一个 writer，writer 过多会导致内存不足。

### 2.4 内存调优建议

#### 高内存场景 (充足的 TaskManager 内存)

```properties
write-buffer-size = 512mb
page-size = 128kb
write-buffer-spillable = false
```

**为什么**: 大 write buffer 减少 flush 频率，关闭溢写避免磁盘 I/O，适合内存充足且 bucket 数量有限的场景。

**好处**: 减少 I/O，降低 Compaction 压力。

#### 内存紧张场景 (多分区/多 bucket)

```properties
write-buffer-size = 64mb
page-size = 32kb
write-buffer-spillable = true
write-buffer-spill.max-disk-size = 10gb
spill-compression = zstd
spill-compression.zstd-level = 1
write-max-writers-to-spill = 5
```

**为什么**: 缩小 write buffer 减少总内存需求，开启溢写防止 OOM，使用 zstd 压缩减少磁盘占用。

**好处**: 在有限内存下支持更多并发 writer，避免 OOM 导致任务失败。

#### 监控关键指标

通过 `WriterBufferMetric` (源码: `WriterBufferMetric.java`) 提供的指标进行监控:
- `numWriters`: 当前 writer 数量
- `bufferPreemptCount`: 内存抢占次数 (高值表示内存不足)
- `usedWriteBufferSizeByte`: 已使用的 write buffer 大小
- `totalWriteBufferSizeByte`: 总 write buffer 大小

**告警阈值建议**: 如果 `bufferPreemptCount` 持续快速增长，说明内存严重不足，应增加 `write-buffer-size` 或增加 TaskManager 内存。

## 3. Snapshot 与 Changelog 管理

### 解决什么问题

**核心业务问题**: 每次 commit 都会产生新的 snapshot,不清理会导致存储空间无限增长,同时历史 snapshot 对应的数据文件无法删除,造成存储成本失控。

**实际场景**:
- 高频写入场景: 每秒产生一个 snapshot,一天 86,400 个,一个月 260 万个
- 流式消费场景: 下游任务从 snapshot 100 开始消费,但当前已经到 snapshot 10000,中间 9900 个 snapshot 无法删除
- 存储成本爆炸: 实际数据 1TB,但因为历史版本未清理,存储占用 10TB

**没有这个设计的后果**:
1. 存储成本失控: 每月存储费用从 1000 元增长到 10000 元
2. Manifest 扫描超时: 需要扫描数百万个 snapshot 的 manifest,查询延迟从秒级上升到分钟级
3. 文件系统压力: HDFS NameNode 内存不足,S3 listing 操作超时
4. 任务启动慢: Flink 任务启动时需要扫描所有 snapshot,启动时间从秒级上升到小时级

### 有什么坑

**误区 1: 设置过短的保留时间**
```properties
# 危险配置
snapshot.time-retained = 10min
snapshot.num-retained.min = 3
```
**后果**: 下游消费任务因为网络抖动暂停 15 分钟,恢复后发现需要的 snapshot 已被删除,任务失败。

**误区 2: 忽略 Consumer 的保护机制**
**现象**: 配置了 `snapshot.num-retained.min = 10`,但实际保留了 1000 个 snapshot。
**原因**: 有下游 Consumer 正在消费,`consumerManager.minNextSnapshot()` 保护了这些 snapshot。
**解决**: 检查 Consumer 列表,删除不再使用的 Consumer。

**误区 3: 同步过期导致 checkpoint 超时**
```properties
snapshot.expire.execution-mode = sync  # 默认值
snapshot.expire.limit = 1000  # 过大
```
**后果**: 一次过期 1000 个 snapshot,删除数百万个文件,checkpoint 超时。

**生产环境注意事项**:
- 监控 snapshot 数量,超过 1000 个就应该排查为什么没有过期
- 异步过期模式下,过期速度必须跟上 snapshot 生成速度,否则会积压
- Changelog 解耦后,snapshot 过期不会影响 changelog 的保留

**性能陷阱**:
- 过期操作是串行的,删除大量小文件时非常慢
- 对象存储的删除操作有 QPS 限制,可能被限流
- 过期时需要读取所有待删除 snapshot 的 manifest,内存消耗大

### 核心概念解释

**Snapshot**: 表的一个不可变快照,包含该时刻的所有数据文件信息。
- 每次 commit 产生一个新 snapshot
- Snapshot 通过 manifest list 引用数据文件
- 删除 snapshot 不会立即删除数据文件,需要判断文件是否被其他 snapshot 引用

**Changelog**: 记录数据变更的日志,用于流式消费。
- `changelog-producer = none`: 不生成 changelog
- `changelog-producer = input`: 直接使用输入的 changelog
- `changelog-producer = lookup`: 通过 lookup 生成 changelog
- `changelog-producer = full-compaction`: 通过全量 Compaction 生成 changelog

**Changelog 解耦**: Changelog 的生命周期独立于 Snapshot。
- 未解耦: 删除 snapshot 时同时删除对应的 changelog
- 解耦后: snapshot 过期不影响 changelog,changelog 可以保留更长时间

**与 Hudi/Iceberg 的对比**:
- Hudi: 使用 timeline 管理 commit,支持 savepoint 和 archive
- Iceberg: 使用 metadata.json 管理 snapshot,支持 snapshot expiration
- Paimon: 使用 snapshot 文件 + earliest hint,支持 changelog 解耦

### 设计理念

**为什么需要 snapshot 过期**:
- 存储成本: 历史 snapshot 占用大量存储空间
- 查询性能: 过多 snapshot 导致 manifest 扫描慢
- 运维复杂度: snapshot 数量过多难以管理

**为什么需要多重保护机制**:
1. `num-retained.min`: 保证最少快照数,防止误删
2. `time-retained`: 保证时间窗口,支持时间旅行查询
3. `Consumer 保护`: 防止删除正在被消费的 snapshot
4. `expire.limit`: 防止一次删除过多,影响性能

**为什么需要 Changelog 解耦**:
- 不同的生命周期需求: Snapshot 用于批查询 (保留短),Changelog 用于流消费 (保留长)
- 存储成本优化: Snapshot 对应的数据文件可以更快释放,Changelog 文件通常更小
- 灵活性: 可以独立配置 snapshot 和 changelog 的保留策略

**权衡取舍**:
- 保留更多 snapshot: 支持更长的时间旅行,但存储成本高
- 保留更少 snapshot: 节省存储,但回溯能力受限
- 同步过期: 保证及时清理,但可能影响写入延迟
- 异步过期: 不影响写入,但可能积压

**架构演进**:
- 早期版本: 只支持基于数量的过期,不支持时间保留
- 当前版本: 支持多重保护机制,支持 changelog 解耦
- 未来方向: 支持基于存储成本的自适应过期策略

### 3.1 快照保留策略参数

#### `snapshot.num-retained.min`

- **源码位置**: `CoreOptions.java:495-500`
- **默认值**: `10`
- **含义**: 最少保留的完成快照数量，必须 >= 1。

#### `snapshot.num-retained.max`

- **源码位置**: `CoreOptions.java:503-508`
- **默认值**: `Integer.MAX_VALUE` (实际显示为 "infinite")
- **含义**: 最多保留的完成快照数量。

#### `snapshot.time-retained`

- **源码位置**: `CoreOptions.java:510-514`
- **默认值**: `1h`
- **含义**: 已完成快照的最长保留时间。

### 3.2 ExpireSnapshots 实现原理

**核心源码**: `ExpireSnapshotsImpl.java` 第 92-152 行

过期算法按以下步骤计算需要过期的快照范围:

```
1. 获取最新快照 ID (latestSnapshotId) 和最早快照 ID (earliest)
2. 计算 min = max(latestSnapshotId - retainMax + 1, earliest)
   -- 由 snapshot.num-retained.max 保护
3. 计算 maxExclusive = latestSnapshotId - retainMin + 1
   -- 由 snapshot.num-retained.min 保护
4. maxExclusive = min(maxExclusive, consumerManager.minNextSnapshot())
   -- 由 Consumer 保护 (除非 consumer.changelog-only=true)
5. maxExclusive = min(maxExclusive, earliest + maxDeletes)
   -- 由 snapshot.expire.limit 保护，防止一次过期太多
6. 从 min 遍历到 maxExclusive，检查 snapshot.time-retained
   -- 如果快照时间在保留时间内，提前退出
```

**过期执行流程** (`innerExpireUntil()` 第 163-278 行):

```
1. 收集所有待过期的快照 (并行读取)
2. 获取 tag 对应的快照 (用于创建数据文件跳过器)
3. 删除合并树文件 (merge tree files)
4. 删除 changelog 文件 (如果未解耦)
5. 清理空目录
6. 删除 manifest 文件
7. 删除 snapshot 文件本身
8. 更新 earliest hint 文件
```

### 3.3 快照过期触发时机

快照过期在每次 commit 后自动触发 (除非 `write-only = true`)。

#### `snapshot.expire.execution-mode`

- **源码位置**: `CoreOptions.java:536-540`
- **可选值**:
  - `sync` (默认): 同步执行。如果文件太多可能耗时较长，阻塞流处理。
  - `async`: 异步执行。如果快照生成速度超过删除速度，会造成文件积压。

#### `snapshot.expire.limit`

- **源码位置**: `CoreOptions.java:542-547`
- **默认值**: `50`
- **含义**: 单次允许过期的最大快照数量。

**为什么需要限制**: 一次过期太多快照会产生大量文件删除操作，可能导致 checkpoint 超时或影响写入延迟。

### 3.4 Changelog 保留策略

#### `changelog.num-retained.min/max`

- **源码位置**: `CoreOptions.java:516-528`
- **默认值**: 无 (不单独控制 changelog)

#### `changelog.time-retained`

- **源码位置**: `CoreOptions.java:530-534`
- **默认值**: 无

**设计说明**: 当配置了 changelog 保留策略时，Changelog 可以与 Snapshot 生命周期解耦 (`isChangelogDecoupled()`)。这意味着即使 Snapshot 被过期，对应的 Changelog 仍然可以独立保留，用于流式消费。

### 3.5 快照管理最佳实践

#### 流式写入场景

```properties
# 保留足够的快照用于故障恢复和回溯
snapshot.num-retained.min = 20
snapshot.num-retained.max = 200
snapshot.time-retained = 2h
# 限制单次过期量，避免 checkpoint 超时
snapshot.expire.limit = 30
# 高吞吐场景使用异步过期
snapshot.expire.execution-mode = async
```

**为什么**: 保留 2 小时快照用于故障恢复，每次最多过期 30 个快照避免阻塞。

#### 需要长时间流式消费的场景

```properties
snapshot.num-retained.min = 10
snapshot.time-retained = 1h
# 解耦 Changelog 生命周期
changelog-producer = lookup
changelog.time-retained = 24h
changelog.num-retained.min = 100
```

**为什么**: Snapshot 短期保留节省存储，Changelog 独立保留 24 小时满足下游流式消费需求。

**好处**: 存储成本可控 (Snapshot 对应的数据文件可以更快释放)，同时下游消费者有足够的回溯窗口。

## 4. 分区管理

### 解决什么问题

**核心业务问题**: 时间分区表会持续产生新分区,历史分区不清理会导致存储空间无限增长,同时查询时需要扫描所有分区,性能下降。

**实际场景**:
- 日志表: 每天一个分区,一年 365 个分区,保留 3 年就是 1095 个分区
- 实时数仓: 小时级分区,一天 24 个,一个月 720 个,查询需要扫描所有分区
- 存储成本: 实际只需要最近 30 天数据,但保留了 3 年,存储成本是实际需求的 36 倍

**没有这个设计的后果**:
1. 存储成本失控: 每月存储费用从 1000 元增长到 36000 元
2. 查询性能下降: 扫描 1000 个分区需要数分钟,即使只查询最近一天
3. 元数据爆炸: Manifest 中记录数百万个文件,扫描 manifest 本身就超时
4. 运维困难: 手动删除分区容易误删,且操作繁琐

### 有什么坑

**误区 1: 分区时间格式配置错误**
```properties
# 错误配置
partition.timestamp-formatter = yyyy-MM-dd
# 但实际分区值是: dt=20240101 (没有分隔符)
```
**后果**: 无法解析分区时间,分区永远不会过期,存储空间持续增长。

**误区 2: 过期检查间隔过长**
```properties
partition.expiration-check-interval = 24h
partition.expiration-time = 1h
```
**后果**: 分区在 1 小时后就应该过期,但要等 24 小时才检查一次,实际保留时间是 1-25 小时。

**误区 3: 一次删除过多分区**
```properties
partition.expiration-max-num = 10000  # 过大
partition.expiration-batch-size = 10000
```
**后果**: 一次删除 10000 个分区,需要删除数百万个文件,任务 OOM 或超时。

**生产环境注意事项**:
- 分区过期是在写入任务中执行的,过期时间过长会阻塞写入
- 分区时间的解析依赖 JVM 时区,确保时区配置正确
- 分区过期会删除整个分区目录,确保没有其他任务依赖这些分区

**性能陷阱**:
- 分区过期需要遍历所有分区,分区数量过多时遍历本身就很慢
- 删除分区时需要递归删除目录,对象存储的删除操作有 QPS 限制
- 分区过期和 snapshot 过期是独立的,可能导致文件被删除两次 (浪费 I/O)

### 核心概念解释

**分区过期策略**:
- `values-time`: 从分区值中提取时间 (如 `dt=2024-01-01`),与当前时间比较
- `update-time`: 使用分区最后更新时间,与当前时间比较

**分区时间提取**:
- `timestamp-formatter`: 时间格式化器,如 `yyyy-MM-dd HH:mm:ss`
- `timestamp-pattern`: 从多个分区字段组合时间,如 `$year-$month-$day $hour:00:00`

**批量删除机制**:
- `expiration-max-num`: 单次最多删除的分区数量 (防止一次删除过多)
- `expiration-batch-size`: 分批删除的批次大小 (防止内存不足)

**与 Hive 的对比**:
- Hive: 需要手动执行 `ALTER TABLE DROP PARTITION`,或使用外部调度工具
- Paimon: 自动过期,无需人工介入
- Hive: 分区过期是元数据操作,不涉及数据删除 (如果是外部表)
- Paimon: 分区过期会物理删除数据文件

### 设计理念

**为什么需要自动分区过期**:
- 降低运维成本: 无需人工编写脚本定期删除分区
- 避免误操作: 自动过期基于配置,不会误删重要分区
- 及时清理: 与写入任务集成,实时检查和清理

**为什么需要随机化初始检查时间**:
```java
long rndSeconds = ThreadLocalRandom.current().nextLong(checkIntervalSeconds);
this.lastCheck = LocalDateTime.now().minusSeconds(rndSeconds);
```
- 避免雷击: 所有任务在同一时刻检查分区,导致文件系统压力突增
- 分散负载: 随机化后,检查操作均匀分布在时间轴上

**为什么需要批量删除**:
- 内存保护: 一次删除 10000 个分区需要加载所有分区的元数据,可能 OOM
- 性能保护: 分批删除可以在批次之间释放资源,避免长时间阻塞

**权衡取舍**:
- 更短的检查间隔: 及时清理,但增加 I/O 开销
- 更长的检查间隔: 减少开销,但分区保留时间不精确
- 更大的批次大小: 删除更快,但内存压力大
- 更小的批次大小: 内存安全,但删除慢

**架构演进**:
- 早期版本: 只支持基于分区值的过期,不支持 update-time
- 当前版本: 支持两种策略,支持批量删除
- 未来方向: 支持基于存储成本的智能过期 (如保留最近 1TB 数据)

### 4.1 PartitionExpire 实现原理

**核心源码**: `PartitionExpire.java`

#### 过期检查逻辑 (第 140-149 行):

```java
List<Map<String, String>> expire(LocalDateTime now, long commitIdentifier) {
    if (checkInterval.isZero()
            || now.isAfter(lastCheck.plus(checkInterval))
            || (endInputCheckPartitionExpire && Long.MAX_VALUE == commitIdentifier)) {
        List<Map<String, String>> expired =
                doExpire(now.minus(expirationTime), commitIdentifier);
        lastCheck = now;
        return expired;
    }
    return null;
}
```

三种触发条件:
1. `checkInterval` 为零: 每次 commit 都检查
2. 距上次检查超过 `checkInterval`: 定时触发
3. 流任务结束时 (`endInputCheckPartitionExpire && commitIdentifier == Long.MAX_VALUE`)

#### 随机化初始检查时间 (第 82-87 行):

```java
long rndSeconds = 0;
long checkIntervalSeconds = checkInterval.toMillis() / 1000;
if (checkIntervalSeconds > 0) {
    rndSeconds = ThreadLocalRandom.current().nextLong(checkIntervalSeconds);
}
this.lastCheck = LocalDateTime.now().minusSeconds(rndSeconds);
```

**为什么**: 避免所有任务在同一时刻执行分区过期检查，分散 I/O 压力。

#### 过期策略

- **`values-time`** (默认): 从分区值中提取时间，与当前时间比较。通过 `PartitionValuesTimeExpireStrategy` 实现。
- **`update-time`**: 比较分区最后更新时间与当前时间。

### 4.2 分区过期配置

#### `partition.expiration-time`

- **源码位置**: `CoreOptions.java:1141-1148`
- **默认值**: 无 (不配置则不启用分区过期)
- **含义**: 分区的过期时间间隔。超过此时间的分区将被删除。

#### `partition.expiration-check-interval`

### 4.4 分区表管理建议

#### 日级分区表

```properties
partition.expiration-time = 30d
partition.expiration-check-interval = 1h
partition.expiration-max-num = 50
partition.timestamp-formatter = yyyy-MM-dd
partition.timestamp-pattern = $dt
```

**为什么**: 保留 30 天数据，每小时检查一次，每次最多删除 50 个分区，避免一次性删除过多分区导致文件系统压力过大。

#### 小时级分区表 (数据量大)

```properties
partition.expiration-time = 7d
partition.expiration-check-interval = 30min
partition.expiration-max-num = 100
partition.expiration-batch-size = 20
partition.timestamp-formatter = yyyy-MM-dd HH:mm:ss
partition.timestamp-pattern = $dt $hour:00:00
```

**为什么**: 小时级分区数量多 (每天 24 个)，缩短检查间隔和增大最大删除数，同时设置 batch-size 为 20 避免一次性删除 100 个分区导致 OOM。

## 5. Bucket 选择策略

### 解决什么问题

**核心业务问题**: Bucket 是 Paimon 的并行单元,bucket 数量直接决定写入和读取的并行度。选择不当会导致严重的性能问题。

**实际场景**:
- Bucket 过少: 1TB 数据只有 4 个 bucket,写入并行度只有 4,吞吐只有 40MB/s
- Bucket 过多: 10GB 数据配置 1024 个 bucket,产生 10 万个小文件,查询超时
- 数据倾斜: 某个 bucket 的数据量是其他 bucket 的 100 倍,成为瓶颈

**没有这个设计的后果**:
1. 写入瓶颈: 并行度不足,写入吞吐远低于硬件能力
2. 小文件问题: 每个 bucket 独立产生文件,bucket 过多导致文件数量爆炸
3. 内存不足: 每个 bucket 需要独立的 write buffer,bucket 过多导致 OOM
4. Compaction 压力: 更多 bucket 意味着更多独立的 Compaction 任务

### 有什么坑

**误区 1: 盲目使用动态 bucket**
```properties
bucket = -1  # 动态 bucket
```
**后果**: 数据倾斜严重时,动态 bucket 会创建大量 bucket,反而加剧小文件问题。

**误区 2: 固定 bucket 数量不考虑数据增长**
```properties
bucket = 16  # 初期数据量 10GB
# 一年后数据量增长到 1TB,但 bucket 数量不变
```
**后果**: 单个 bucket 的数据量从 625MB 增长到 64GB,Compaction 耗时从秒级上升到小时级。

**误区 3: 跨分区更新场景使用 hash dynamic**
```properties
bucket = -1
# 但有跨分区更新 (cross-partition-update = true)
```
**后果**: 应该使用 `KEY_DYNAMIC` 模式,但实际使用了 `HASH_DYNAMIC`,导致数据错乱。

**生产环境注意事项**:
- Bucket 数量一旦确定就无法修改 (除非重建表),必须提前规划
- 动态 bucket 的 bucket 数量会持续增长,需要定期监控
- 对象存储场景下,bucket 过多会导致 listing 操作超时

**性能陷阱**:
- Bucket 数量必须是 2 的幂次方 (如 16, 32, 64),否则 hash 分布不均
- 动态 bucket 的 `target-row-num` 设置过小会导致 bucket 数量爆炸
- 固定 bucket 的 bucket-key 选择不当会导致严重的数据倾斜

### 核心概念解释

**Bucket 模式**:
- `HASH_FIXED` (bucket > 0): 固定桶数,通过 hash(bucket-key) % bucket_num 分配
- `HASH_DYNAMIC` (bucket = -1, 无跨分区更新): 动态桶,根据数据量自动增减
- `KEY_DYNAMIC` (bucket = -1, 有跨分区更新): 基于 key 的动态桶,保证同一 key 在同一 bucket
- `POSTPONE_MODE` (bucket = -2): 延迟桶分配,用于特殊场景

**Bucket 与并行度的关系**:
- 写入并行度 = min(bucket 数量, Flink 并行度)
- 读取并行度 = bucket 数量 (每个 bucket 可以独立读取)
- Compaction 并行度 = bucket 数量 (每个 bucket 独立 Compaction)

**与 Hive/Spark 的对比**:
- Hive: 使用 bucket 进行数据分桶,但 bucket 数量可以动态调整
- Spark: 使用 partition 进行数据分区,partition 数量可以动态调整
- Paimon: Bucket 数量固定 (除非使用动态 bucket),更接近 HBase 的 region 概念

### 设计理念

**为什么需要 Bucket**:
- 并行化: 每个 bucket 可以独立写入和读取,实现并行处理
- 数据局部性: 相同 bucket-key 的数据在同一 bucket,支持高效的点查和更新
- Compaction 隔离: 每个 bucket 独立 Compaction,避免全局锁

**为什么需要固定 Bucket**:
- 确定性: 相同的 key 总是路由到相同的 bucket,支持幂等写入
- 简单性: 不需要维护 bucket 到 key 的映射关系
- 性能: Hash 计算开销极低,适合高吞吐场景

**为什么需要动态 Bucket**:
- 灵活性: 数据量未知时,动态调整 bucket 数量
- 避免小文件: 数据量小时使用少量 bucket,避免产生大量小文件
- 避免大文件: 数据量大时自动增加 bucket,避免单个 bucket 过大

**权衡取舍**:
- 固定 bucket: 性能好,但需要提前规划,不灵活
- 动态 bucket: 灵活,但可能产生过多 bucket,增加运维复杂度
- 更多 bucket: 并行度高,但小文件多,内存消耗大
- 更少 bucket: 文件大,内存省,但并行度低

**架构演进**:
- 早期版本: 只支持固定 bucket
- 当前版本: 支持动态 bucket,支持跨分区更新
- 未来方向: 支持 bucket 数量的在线调整 (类似 HBase 的 region split)

### 5.1 Bucket 模式源码分析

**核心源码**: `KeyValueFileStore.bucketMode()` (第 91-101 行):

```java
public BucketMode bucketMode() {
    int bucket = options.bucket();
    switch (bucket) {
        case -2:
            return BucketMode.POSTPONE_MODE;
        case -1:
            return crossPartitionUpdate ? BucketMode.KEY_DYNAMIC : BucketMode.HASH_DYNAMIC;
        default:
            return BucketMode.HASH_FIXED;
    }
}
```

三种模式:
1. **Fixed Bucket** (`bucket > 0`): 固定桶数，通过 hash 分配。适合已知数据规模的场景。
2. **Dynamic Bucket** (`bucket = -1`): 动态桶模式。
   - `HASH_DYNAMIC`: 无跨分区更新时使用
   - `KEY_DYNAMIC`: 有跨分区更新时使用
3. **Postpone Bucket** (`bucket = -2`): 延迟桶分配模式。

### 5.2 固定 Bucket 数量的影响

#### Bucket 过少的影响

1. **写入瓶颈**: 每个 bucket 对应一个 writer，bucket 少意味着并行度低
2. **文件过大**: 单个 bucket 积累过多数据，Compaction 耗时长
3. **读取延迟**: 查询集中在少数 bucket，无法充分利用并行度

#### Bucket 过多的影响

1. **小文件问题**: 每个 bucket 独立产生 level0 文件，bucket 过多导致大量小文件
2. **内存消耗**: 每个 bucket 需要独立的 write buffer，bucket 多则总内存需求大
3. **Compaction 压力**: 更多的 bucket 意味着更多的独立 Compaction 任务
4. **元数据开销**: manifest 中需要记录更多文件信息

### 5.3 Dynamic Bucket 配置

#### `dynamic-bucket.target-row-num`

- **源码位置**: `CoreOptions.java:1411-1417`
- **默认值**: `2,000,000`
- **含义**: 动态桶模式下每个桶的目标行数。

**为什么有用**: 动态桶根据数据量自动增减桶数，target-row-num 控制桶的粒度。较大的值意味着更少的桶和更大的文件，较小的值意味着更多的桶和更好的查询并行度。

### 5.4 Bucket 数量选择公式和建议

#### 固定 Bucket 估算公式

```
bucket_num = ceil(daily_data_size_per_partition / target_file_size / num_sorted_run_compaction_trigger)
```

其中:
- `daily_data_size_per_partition`: 每个分区每天写入的数据量
- `target_file_size`: 目标文件大小 (默认 128 MB)，源码 `CoreOptions.java:713-723`
- `num_sorted_run_compaction_trigger`: Compaction 触发阈值 (默认 5)

#### 举例

- 每天每分区写入 100 GB 数据
- target-file-size = 128 MB
- num-sorted-run.compaction-trigger = 5
- bucket_num = ceil(100 * 1024 / 128 / 5) = ceil(160) = 160

#### 实践建议

| 分区每日数据量 | 建议 Bucket 数 | 说明 |
|-------------|-------------|------|
| < 1 GB | 1-2 | 数据量小，避免小文件 |
| 1-10 GB | 4-8 | 保证基本并行度 |
| 10-100 GB | 16-64 | 平衡并行度和文件大小 |
| 100 GB-1 TB | 64-256 | 高并行度写入 |
| > 1 TB | 使用 Dynamic Bucket | 数据量大且不确定，使用动态桶 |

**好处**: 合理的 bucket 数量可以在写入吞吐、Compaction 效率、查询并行度之间取得平衡。

## 6. 文件格式与压缩

### 解决什么问题

**核心业务问题**: 数据文件的格式和压缩算法直接影响存储成本、写入性能和查询性能。选择不当会导致成本翻倍或性能下降 10 倍。

**实际场景**:
- 存储成本: 1TB 原始数据,使用 snappy 压缩后 600GB,使用 zstd 压缩后 300GB,成本差一倍
- 查询性能: 列式存储 (Parquet) 查询 10 列耗时 1 秒,行式存储 (Avro) 需要 10 秒
- 写入性能: zstd level=9 压缩耗时 10 秒,lz4 只需要 1 秒

**没有这个设计的后果**:
1. 存储成本失控: 使用无压缩或低压缩比算法,存储费用是最优方案的 3-5 倍
2. 查询性能差: 使用行式存储,列式查询需要读取全部数据,I/O 放大 10 倍
3. 写入延迟高: 使用高压缩级别,写入吞吐下降 50% 以上
4. 兼容性问题: 使用不兼容的格式,无法与 Hive/Spark 生态集成

### 有什么坑

**误区 1: 盲目追求高压缩比**
```properties
file.compression = zstd
file.compression.zstd-level = 22  # 最高级别
```
**后果**: 压缩耗时从 1 秒增加到 30 秒,写入吞吐下降 90%,但压缩比只提升 10%。

**误区 2: 所有层级使用相同格式**
```properties
file.format = parquet  # 所有层级都用 parquet
```
**后果**: Level0 文件频繁写入和合并,parquet 的列式写入开销大,写入性能差。

**误区 3: target-file-size 设置过小**
```properties
target-file-size = 16mb  # 想要更细粒度的裁剪
```
**后果**: 文件数量暴增,元数据开销大,查询时打开文件的开销超过读取数据的开销。

**生产环境注意事项**:
- 对象存储场景下,文件大小应该是 API 调用粒度的整数倍 (如 S3 的 5MB)
- Parquet 的 row group 大小应该小于 target-file-size,否则无法充分利用列裁剪
- 压缩算法的选择要考虑 CPU 资源,CPU 不足时高压缩比反而降低吞吐

**性能陷阱**:
- Parquet 的字典编码在高基数列上会失效,反而增加存储空间
- ORC 的 stripe 大小过大会导致内存不足
- 压缩算法的解压速度差异巨大,zlib 解压比 lz4 慢 5 倍

### 核心概念解释

**文件格式**:
- `Parquet`: 列式存储,支持列裁剪和谓词下推,适合 OLAP 查询
- `ORC`: 列式存储,Hive 生态兼容性好,支持 ACID
- `Avro`: 行式存储,schema 复杂或全行读取时性能好
- `CSV/JSON`: 文本格式,调试方便但性能差

**压缩算法**:
- `zstd`: 高压缩比 + 快速解压,综合最优
- `lz4`: 极快的压缩和解压,适合写入密集场景
- `snappy`: 平衡压缩比和速度,Hadoop 生态常用
- `zlib/gzip`: 高压缩比但慢,适合冷数据归档

**按层级配置**:
- Level0: 频繁写入和合并,使用快速格式和压缩 (Avro + lz4)
- Level1+: 稳定存储,使用高压缩比 (Parquet + zstd)

**与 Hive/Iceberg 的对比**:
- Hive: 默认使用 ORC,支持 Parquet
- Iceberg: 默认使用 Parquet,支持 ORC 和 Avro
- Paimon: 默认使用 Parquet,支持按层级配置不同格式

### 设计理念

**为什么默认使用 Parquet**:
- 列式存储: 查询只需要读取需要的列,I/O 大幅减少
- 压缩友好: 相同列的数据类型相同,压缩比更高
- 生态兼容: Spark/Presto/Trino 等引擎都原生支持 Parquet

**为什么默认使用 zstd**:
- 压缩比高: 比 snappy 高 30-50%,比 lz4 高 50-100%
- 速度快: 压缩和解压速度都很快,比 zlib 快 5-10 倍
- 可调节: 支持 level 1-22,可以根据场景调整

**为什么支持按层级配置**:
- 写入优化: Level0 使用快速格式,减少写入延迟
- 存储优化: 高层级使用高压缩比,节省存储成本
- 灵活性: 不同层级的访问模式不同,应该使用不同的配置

**权衡取舍**:
- Parquet vs Avro: 列式查询快 vs 全行读取快
- zstd vs lz4: 高压缩比 vs 极速压缩
- 大文件 vs 小文件: 压缩比高 vs 裁剪粒度细
- 高压缩级别 vs 低压缩级别: 存储省 vs 写入快

**架构演进**:
- 早期版本: 只支持全局配置,所有层级使用相同格式
- 当前版本: 支持按层级配置,灵活性大幅提升
- 未来方向: 支持基于数据特征的自适应格式选择 (如高基数列使用 ORC,低基数列使用 Parquet)

### 6.1 文件格式选择

**源码位置**: `CoreOptions.java:285-297`

Paimon 支持的文件格式:

| 格式 | 常量 | 适用场景 |
|------|------|---------|
| Parquet | `FILE_FORMAT_PARQUET` (默认) | 列式查询优化，适合 OLAP 场景 |
| ORC | `FILE_FORMAT_ORC` | 列式存储，Hive 生态兼容性好 |
| Avro | `FILE_FORMAT_AVRO` | 行式存储，适合 schema 复杂或全行读取 |
| CSV/Text/JSON | - | 简单数据交换，不推荐生产使用 |

#### 按层级指定格式

- **源码位置**: `CoreOptions.java:307-316`
- **配置示例**: `'file.format.per.level' = '0:avro,3:parquet'`

**为什么**: Level0 文件通常较小且需要快速写入，使用 Avro (行式) 可以减少写入延迟；高层级文件较大且偏读取优化，使用 Parquet (列式) 提升查询性能。

### 6.2 压缩算法配置

#### `file.compression`

- **源码位置**: `CoreOptions.java:318-323`
- **默认值**: `zstd`
- **说明**: `"For faster read and write, it is recommended to use zstd."`

#### `file.compression.zstd-level`

- **源码位置**: `CoreOptions.java:325-330`
- **默认值**: `1`
- **说明**: `"For higher compression rates, it can be configured to 9, but the read and write speed will significantly decrease."`

#### 按层级指定压缩

- **源码位置**: `CoreOptions.java:299-305`
- **配置示例**: `'file.compression.per.level' = '0:lz4,1:zstd'`

**为什么**: Level0 文件频繁写入和合并，使用 lz4 (高速低压缩) 减少写放大；高层级文件稳定存储时间长，使用 zstd (高压缩比) 节省存储空间。

#### 压缩算法对比

| 算法 | 压缩比 | 压缩速度 | 解压速度 | 推荐场景 |
|------|-------|---------|---------|---------|
| zstd (level=1) | 高 | 快 | 快 | 默认推荐，综合最优 |
| zstd (level=9) | 很高 | 慢 | 快 | 存储成本敏感，可牺牲写入速度 |
| lz4 | 中 | 很快 | 很快 | 写入密集，延迟敏感 |
| snappy | 中 | 快 | 快 | 与 Hadoop 生态兼容 |
| zlib | 高 | 慢 | 中 | 冷数据归档 |

### 6.3 target-file-size 的影响

**源码位置**: `CoreOptions.java:713-723` 和 `targetFileSize()` 方法第 2895-2898 行:

```java
public long targetFileSize(boolean hasPrimaryKey) {
    return options.getOptional(TARGET_FILE_SIZE)
            .orElse(hasPrimaryKey ? VALUE_128_MB : VALUE_256_MB)
            .getBytes();
}
```

- 主键表默认 128 MB
- 追加表默认 256 MB

**Compaction 文件大小计算** (`compactionFileSize()` 第 2961-2966 行):
```java
public long compactionFileSize(boolean hasPrimaryKey) {
    return targetFileSize(hasPrimaryKey) / 10 * 7;  // target 的 70%
}
```

只有大于 `compactionFileSize` (即 target 的 70%) 的文件才会参与 Compaction。

**为什么这么设计**: 避免刚写入的文件立即被合并 (因为压缩后的文件可能略小于 target)，减少不必要的重复合并。

#### target-file-size 调优建议

| 场景 | 建议值 | 原因 |
|------|-------|------|
| HDFS 存储 | 128-256 MB | 匹配 HDFS block 大小，减少远程读取 |
| 对象存储 (S3/OSS) | 256-512 MB | 减少 API 调用次数和 listing 开销 |
| 快速查询 | 64-128 MB | 更小的文件实现更细粒度的裁剪 |
| 高吞吐写入 | 256 MB | 减少 Compaction 频率 |

### 6.4 文件块大小

- **源码位置**: `CoreOptions.java:378-383`
- **默认值**: ORC stripe 64 MB，Parquet row group 128 MB

**好处**: 较大的块大小减少元数据开销和 seek 次数，但增加内存占用。根据存储系统特性调整。

## 7. 监控指标

### 解决什么问题

**核心业务问题**: 生产环境的 Paimon 表可能出现各种性能问题,没有监控指标就无法及时发现和定位问题,导致故障扩大。

**实际场景**:
- 写入延迟突增: 从 100ms 上升到 10 秒,但不知道是 Compaction 慢还是内存不足
- 查询超时: 查询从 1 秒变成 1 分钟,但不知道是文件太多还是分区裁剪失效
- 存储成本异常: 1TB 数据占用 10TB 存储,但不知道是空间放大还是 snapshot 未清理

**没有这个设计的后果**:
1. 故障发现滞后: 问题发生 1 小时后才被用户投诉,影响范围扩大
2. 定位困难: 只能靠猜测和试错,排查时间从分钟级上升到小时级
3. 无法优化: 不知道瓶颈在哪里,盲目调参反而恶化问题
4. 容量规划失败: 不知道资源使用情况,扩容不及时或过度扩容

### 有什么坑

**误区 1: 只监控 Flink 指标,不监控 Paimon 指标**
**现象**: Flink 的 checkpoint 时间正常,但写入延迟很高。
**原因**: Compaction 在 checkpoint 之外执行,Flink 指标看不到。
**解决**: 必须监控 Paimon 的 `CompactionMetrics`。

**误区 2: 告警阈值设置不合理**
```
# 错误的告警规则
WARN if maxLevel0FileCount > 100
```
**后果**: `num-sorted-run.stop-trigger = 8`,但告警阈值是 100,等告警时写入已经被阻塞很久了。

**误区 3: 忽略趋势监控,只看瞬时值**
**现象**: `bufferPreemptCount` 当前值是 10,看起来不高。
**问题**: 但增长速率是 100/分钟,说明内存严重不足,很快就会 OOM。

**生产环境注意事项**:
- 监控指标的采集频率要足够高 (建议 10 秒),否则可能错过瞬时峰值
- 不同表的告警阈值应该不同,根据表的配置动态调整
- 监控系统本身要高可用,否则监控失效时无法发现问题

**性能陷阱**:
- 过多的监控指标会增加 JVM 的 GC 压力
- 监控指标的计算本身有开销,如 `compactionThreadBusy` 需要遍历所有 Compaction 线程
- 监控数据的存储和查询也有成本,需要定期清理历史数据

### 核心概念解释

**CommitMetrics**: 提交相关的指标,反映写入和 Compaction 的活跃度。
- `lastCommitDuration`: 最近一次提交耗时,包括文件写入、manifest 更新、snapshot 创建
- `lastTableFilesAdded`: 新增文件数,反映写入活跃度
- `lastTableFilesCommitCompacted`: Compaction 产生的文件数,反映合并活跃度

**CompactionMetrics**: Compaction 相关的指标,反映合并的健康状况。
- `maxLevel0FileCount`: 最大 Level0 文件数,接近 `stop-trigger` 说明即将阻塞写入
- `compactionThreadBusy`: Compaction 线程繁忙度,超过 80% 说明 Compaction 跟不上
- `compactionQueuedCount`: 队列中等待的 Compaction 数,持续增长说明积压

**ScanMetrics**: 扫描相关的指标,反映查询的性能。
- `lastScanDuration`: 最近一次扫描耗时,包括 manifest 扫描和文件裁剪
- `lastScanSkippedTableFiles`: 跳过的文件数,跳过率低说明裁剪效果差

**WriterBufferMetric**: 写入缓冲相关的指标,反映内存使用情况。
- `bufferPreemptCount`: 缓冲区抢占次数,持续增长说明内存不足
- `usedWriteBufferSizeByte`: 已使用的 Write Buffer 大小,接近总大小说明内存紧张

### 设计理念

**为什么需要多维度的指标**:
- 写入指标: 反映数据摄入的健康状况
- Compaction 指标: 反映后台合并的健康状况
- 查询指标: 反映数据读取的健康状况
- 内存指标: 反映资源使用的健康状况

**为什么需要瞬时值和累计值**:
- 瞬时值 (如 `lastCommitDuration`): 反映当前状态,用于实时告警
- 累计值 (如 `commitDuration` 直方图): 反映历史趋势,用于性能分析

**为什么需要百分位数指标**:
- 平均值会被异常值拉高,无法反映真实情况
- P99/P999 可以反映长尾延迟,更适合 SLA 监控

**权衡取舍**:
- 更多指标: 信息更全面,但增加开销和复杂度
- 更少指标: 开销小,但可能遗漏关键信息
- 更高采集频率: 及时发现问题,但增加存储成本
- 更低采集频率: 节省成本,但可能错过瞬时峰值

**架构演进**:
- 早期版本: 只有基本的 Flink 指标,无法定位 Paimon 特有的问题
- 当前版本: 提供丰富的 Paimon 指标,覆盖写入、Compaction、查询、内存等维度
- 未来方向: 支持基于指标的自动调优 (如根据 `compactionThreadBusy` 自动调整 `compaction-trigger`)

## 8. 常见问题排查

### 解决什么问题

**核心业务问题**: 生产环境的 Paimon 表会遇到各种性能问题,需要系统化的排查方法快速定位根因,避免盲目试错。

**实际场景**:
- 写入延迟突增: 从 100ms 上升到 10 秒,业务数据积压
- 查询超时: 原本 1 秒的查询变成 1 分钟,用户投诉
- 任务 OOM: TaskManager 反复重启,任务永远无法完成
- Compaction 积压: Level0 文件数量持续增长,即将阻塞写入

**没有这个设计的后果**:
1. 排查时间长: 从小时级上升到天级,影响业务连续性
2. 误判问题: 调整了错误的参数,问题没解决反而恶化
3. 重复发生: 没有找到根因,问题修复后又复发
4. 经验无法复用: 每次都要重新排查,无法积累经验

### 有什么坑

**误区 1: 只看表象,不看根因**
**现象**: 写入延迟高,直接增大 TaskManager 内存。
**问题**: 实际是 Compaction 跟不上,增加内存无效,应该使用 Dedicated Compaction。

**误区 2: 盲目调参,不看监控**
**现象**: 查询慢,直接把 `num-sorted-run.compaction-trigger` 从 5 改成 2。
**问题**: 实际是分区裁剪失效,调整 Compaction 参数无效,反而增加写放大。

**误区 3: 单点优化,忽略全局**
**现象**: 优化了写入性能,但 Compaction 积压导致查询变慢。
**问题**: 写入和 Compaction 是相互影响的,需要全局平衡。

**生产环境注意事项**:
- 排查问题前先保存现场 (监控截图、日志、配置),避免调整后无法复现
- 每次只调整一个参数,观察效果后再调整下一个
- 调整参数后要等待足够长的时间 (至少 1 个 checkpoint 周期) 才能判断效果

**性能陷阱**:
- 某些问题是周期性的 (如 off-peak Compaction),需要观察完整周期
- 某些问题是累积的 (如 snapshot 未清理),短期看不出影响
- 某些问题是级联的 (如内存不足导致溢写,溢写导致 I/O 高,I/O 高导致 checkpoint 超时)

### 核心概念解释

**排查路径**: 从现象到根因的系统化方法。
1. 确认现象: 通过监控指标确认问题的具体表现
2. 定位瓶颈: 通过指标对比找到瓶颈环节
3. 分析根因: 结合源码和配置分析为什么出现瓶颈
4. 制定方案: 根据根因制定解决方案
5. 验证效果: 调整后观察指标是否改善

**关键指标的含义**:
- `compactionThreadBusy > 80%`: Compaction 线程满负荷,跟不上写入速度
- `bufferPreemptCount 快速增长`: 内存不足,writer 之间频繁抢占
- `maxLevel0FileCount 接近 stop-trigger`: 即将阻塞写入,紧急状态
- `lastCommitAttempts > 3`: 提交冲突频繁,可能有多个写入任务

**常见问题的根因分类**:
- 资源不足: CPU、内存、磁盘 I/O、网络带宽
- 配置不当: Compaction 参数、内存参数、bucket 数量
- 数据特征: 数据倾斜、高基数、大量更新
- 外部依赖: 文件系统慢、网络抖动、下游消费慢

### 设计理念

**为什么需要系统化的排查方法**:
- 避免盲目试错: 有明确的排查路径,不会遗漏关键环节
- 快速定位: 通过指标快速缩小问题范围
- 经验复用: 排查路径可以文档化,新人也能快速上手

**为什么需要多层次的排查**:
- 第一层: 看监控指标,确认问题的具体表现
- 第二层: 看日志和异常,找到错误信息
- 第三层: 看源码,理解问题的根本原因
- 第四层: 看配置和数据特征,找到触发条件

**为什么需要分类的解决方案**:
- 不同问题的解决方案不同,需要对症下药
- 同一现象可能有多个根因,需要逐一排查
- 某些问题需要组合方案,单一调整无效

**权衡取舍**:
- 快速修复 vs 根治: 临时调整参数可以快速缓解,但可能复发
- 局部优化 vs 全局平衡: 优化某个环节可能影响其他环节
- 保守调整 vs 激进调整: 小步调整更安全,但见效慢

**架构演进**:
- 早期版本: 缺乏监控指标,排查主要靠日志和猜测
- 当前版本: 提供丰富的监控指标,支持系统化排查
- 未来方向: 支持自动诊断和修复 (如检测到 Compaction 积压自动调整参数)

### 8.1 写入延迟高的排查路径

```
1. 检查 compactionThreadBusy 指标
   ↓ 如果 > 80%
   说明 Compaction 跟不上写入速度
   → 增大 num-sorted-run.compaction-trigger
   → 切换为 Dedicated Compaction (write-only=true)
   → 增加 Compaction 并行度

2. 检查 bufferPreemptCount
   ↓ 如果快速增长
   说明内存不足，writer 之间频繁抢占
   → 增大 write-buffer-size
   → 减少 bucket 数量 (减少 writer 数量)
   → 增大 TaskManager 内存

3. 检查 maxLevel0FileCount
   ↓ 如果接近 numSortedRunStopTrigger
   说明 Compaction 积压严重，写入被阻塞
   → 增大 num-sorted-run.stop-trigger (临时缓解)
   → 分析 Compaction 为何慢 (文件过大? I/O 瓶颈?)

4. 检查 lastCommitAttempts
   ↓ 如果 > 3
   说明提交冲突频繁
   → 检查是否有多个写入任务写同一表
   → 考虑使用不同的 commit-user-prefix
```

### 8.2 读取慢的排查路径

```
1. 检查 lastScanDuration
   ↓ 如果 > 10s
   说明 manifest 扫描耗时
   → 检查 lastScannedManifests 数量
   → 增大 manifest.full-compaction-threshold-size 触发 manifest 合并
   → 检查 manifest 缓存命中率 (manifestHitCache / total)

2. 检查 lastScanResultedTableFiles
   ↓ 如果过多 (比如 > 1000)
   → 检查分区裁剪是否生效 (WHERE 条件是否包含分区列)
   → 检查 file index 是否配置
   → 检查 Compaction 是否正常，文件数量是否过多

3. 检查 maxLevel0FileCount
   ↓ 如果偏高
   → Level0 文件需要全部扫描且 key range 重叠
   → 增强 Compaction (减小 compaction-trigger)
   → 考虑使用 compaction.optimization-interval 定期全量合并

4. 检查数据倾斜
   → 对比 maxTotalFileSize 和 avgTotalFileSize
   → 如果差异大说明某些 bucket 数据量远超其他
   → 重新评估 bucket-key 和 bucket 数量
```

### 8.3 Compaction 积压的排查

```
1. 检查 compactionQueuedCount
   ↓ 如果持续 > 0
   说明 Compaction 线程不够

2. 检查 avgCompactionTime
   ↓ 如果持续增长
   可能原因:
   a. target-file-size 过大导致单次合并数据量过多
   b. 空间放大触发全量合并 (maxSizeAmp 过低)
   c. 磁盘/网络 I/O 瓶颈

3. 检查 compactionInputSize / compactionOutputSize 比率
   → 如果输入远大于输出，说明合并效率好 (大量重复 key 被合并)
   → 如果接近 1:1，说明数据重复率低，合并属于"白做功"
   → 考虑增大 compaction-trigger 减少不必要的合并

4. 解决方案优先级:
   a. 使用 Dedicated Compaction 分离写入和合并资源
   b. 配置 off-peak hours 在低峰期更积极合并
   c. 增大 Compaction 线程池
   d. 调整触发参数减少合并频率
```

### 8.4 OOM 场景和解决方案

#### 场景 1: 写入 OOM

**原因**: 大量 bucket 的 writer 同时持有 sort buffer

**源码依据**: 每个 bucket 的 writer 需要 `SortBufferWriteBuffer`，其内存由 `MemoryPoolFactory` 统一管理。但如果 bucket 数量 * 实际每个 buffer 使用内存 > 总内存，即使有抢占机制也无法完全避免 OOM。

**解决方案**:
```properties
write-buffer-size = 64mb          # 减小单个 buffer
write-buffer-spillable = true      # 开启溢写
write-buffer-spill.max-disk-size = 5gb
write-max-writers-to-spill = 5     # 超过 5 个 writer 就开启溢写
```

#### 场景 2: Compaction OOM

**原因**: 合并大量文件时需要同时打开所有输入文件的 reader

**源码依据**: `sort-spill-threshold` 控制合并读取器的上限 (默认 `numSortedRunStopTrigger + 1`)。

**解决方案**:
```properties
sort-spill-threshold = 5            # 限制同时打开的 reader 数量
sort-spill-buffer-size = 32mb       # 减小溢写 buffer
local-sort.max-num-file-handles = 128  # 限制外部排序的 fan-in (默认值)
```

#### 场景 3: 扫描 Compaction 文件 OOM (Append 表)

**原因**: 无感知 bucket 的 Append 表 Compaction 时扫描文件过多

**源码依据**: `compaction.file-num-limit` (默认 100,000)。

**解决方案**:
```properties
compaction.file-num-limit = 10000   # 限制扫描文件数
```

#### 场景 4: 分区过期 OOM

**原因**: 一次删除过多分区

**源码依据**: `partition.expiration-batch-size` 控制分批删除。

**解决方案**:
```properties
partition.expiration-max-num = 50
partition.expiration-batch-size = 10
```

## 9. 生产环境最佳实践

### 解决什么问题

**核心业务问题**: 生产环境的配置需要在性能、成本、稳定性之间取得平衡,没有最佳实践会导致频繁调整配置,影响业务稳定性。

**实际场景**:
- 新表上线: 不知道如何配置参数,只能用默认值,上线后发现性能不达标
- 配置调优: 每次调整都要重启任务,影响业务连续性
- 成本优化: 存储成本过高,但不知道如何在不影响性能的前提下降低成本
- 故障恢复: 任务失败后不知道如何快速恢复,数据丢失风险

**没有这个设计的后果**:
1. 性能不达标: 写入吞吐只有预期的 50%,查询延迟是预期的 10 倍
2. 成本失控: 存储成本是预期的 3 倍,每月多花费数万元
3. 稳定性差: 任务频繁 OOM 或 checkpoint 超时,可用性低于 99%
4. 运维复杂: 每个表都需要单独调优,运维成本高

### 有什么坑

**误区 1: 所有表使用相同配置**
**现象**: 用一套"通用配置"应用到所有表。
**问题**: 不同表的数据特征和访问模式不同,通用配置无法满足所有场景。

**误区 2: 只关注性能,忽略成本**
```properties
# 追求极致性能的配置
write-buffer-size = 1gb
num-sorted-run.compaction-trigger = 2
snapshot.time-retained = 7d
```
**后果**: 性能很好,但内存需求和存储成本都很高,不可持续。

**误区 3: 只关注成本,忽略性能**
```properties
# 追求极致成本的配置
write-buffer-size = 32mb
num-sorted-run.compaction-trigger = 20
snapshot.time-retained = 10min
```
**后果**: 成本很低,但写入延迟高,查询超时,业务不可用。

**生产环境注意事项**:
- 配置模板只是起点,需要根据实际数据特征和负载调整
- 关键配置 (如 bucket 数量) 一旦确定就无法修改,必须提前规划
- 配置调整要有灰度过程,先在测试环境验证,再在生产环境小流量试验

**性能陷阱**:
- 某些配置的效果有延迟 (如 Compaction 参数调整后要等数小时才能看到效果)
- 某些配置是相互依赖的 (如 `write-buffer-size` 和 `bucket` 数量)
- 某些配置在不同环境下效果不同 (如对象存储 vs 本地磁盘)

### 核心概念解释

**配置模板**: 针对典型场景的推荐配置,作为起点快速上线。
- 主键表 (流式写入): 高频更新,需要平衡写入和查询性能
- 主键表 (批量更新): 低频大批量更新,需要优化 Compaction
- Append 表: 只追加不更新,需要优化文件大小和过期策略

**工作负载分类**:
- CDC 实时入湖: 高频小批量更新,需要大 write buffer 和 Dedicated Compaction
- 宽表聚合: 需要频繁合并,需要激进的 Compaction 策略
- 维表: 主要用于点查,需要文件索引和 lookup cache

**Tag 和 Branch 的运维场景**:
- Tag: 保留关键时间点的数据快照,用于时间旅行查询和数据审计
- Branch: 隔离测试和修复操作,不影响主分支

### 设计理念

**为什么需要配置模板**:
- 降低上手门槛: 新用户不需要理解所有参数,直接使用模板快速上线
- 避免常见错误: 模板经过验证,避免了常见的配置错误
- 提供最佳实践: 模板体现了社区的最佳实践和经验

**为什么需要场景化配置**:
- 不同场景的需求不同: 流式写入和批量写入的最优配置完全不同
- 不同阶段的需求不同: 开发阶段和生产阶段的配置重点不同
- 不同规模的需求不同: 小表和大表的配置策略不同

**为什么需要 Tag 和 Branch**:
- 时间旅行: 查询历史某一时刻的数据,用于数据审计和回溯分析
- 隔离测试: 在 Branch 上测试 Schema 变更或数据修复,不影响主分支
- 灾难恢复: 通过 Tag 快速回滚到已知的良好状态

**权衡取舍**:
- 性能 vs 成本: 高性能需要更多资源,需要根据业务价值权衡
- 稳定性 vs 灵活性: 保守配置更稳定,激进配置更灵活但风险高
- 通用性 vs 定制化: 通用配置运维简单,定制化配置性能更优

**架构演进**:
- 早期版本: 只有默认配置,用户需要自己摸索
- 当前版本: 提供配置模板和最佳实践,降低上手门槛
- 未来方向: 支持自动配置推荐 (根据数据特征和负载自动生成配置)

### 9.1 推荐配置模板

#### 主键表 (流式写入)

```properties
# -- 文件格式 --
file.format = parquet
file.compression = zstd
file.compression.zstd-level = 1
target-file-size = 128mb

# -- 写入 --
write-buffer-size = 256mb
write-buffer-spillable = true

# -- Compaction (Dedicated 模式) --
write-only = true
num-sorted-run.compaction-trigger = 5
num-sorted-run.stop-trigger = 8
compaction.max-size-amplification-percent = 200
compaction.size-ratio = 1

# -- 快照 --
snapshot.num-retained.min = 20
snapshot.num-retained.max = 500
snapshot.time-retained = 2h
snapshot.expire.limit = 30
snapshot.expire.execution-mode = sync

# -- Changelog --
changelog-producer = lookup
changelog.time-retained = 12h

# -- 分区 (按日分区) --
partition.expiration-time = 90d
partition.expiration-check-interval = 1h
partition.expiration-max-num = 50
```

#### 主键表 (批量更新)

```properties
# -- 文件格式 --
file.format = parquet
file.compression = zstd
target-file-size = 256mb

# -- 写入 --
write-buffer-size = 256mb
commit.force-compact = true

# -- Compaction --
num-sorted-run.compaction-trigger = 3
compaction.max-size-amplification-percent = 100

# -- 快照 --
snapshot.num-retained.min = 5
snapshot.num-retained.max = 30
snapshot.time-retained = 1h
```

#### Append 表 (日志/事件类)

```properties
# -- 文件格式 --

file.compression = zstd
target-file-size = 256mb

# -- 写入 --
write-buffer-for-append = true
write-buffer-size = 128mb

# -- Compaction (Append 表参数) --
compaction.min.file-num = 5
compaction.delete-ratio-threshold = 0.2

# -- 快照 --

snapshot.time-retained = 30min

# -- 分区 --
partition.expiration-time = 30d
```

### 9.2 不同工作负载的配置策略

#### CDC 实时入湖

```properties
changelog-producer = lookup
# 或者 changelog-producer = full-compaction 配合 full-compaction.delta-commits = 3
write-only = true
num-sorted-run.compaction-trigger = 5
# 对于有大量更新的表，增加 write buffer 提高去重效率
write-buffer-size = 512mb
```

**为什么**: CDC 场景同一 key 可能频繁更新，大 write buffer 可以在内存中合并更多更新，减少磁盘 I/O 和 Compaction 负担。

#### 宽表聚合

```properties
merge-engine = aggregation
# 或 merge-engine = partial-update
num-sorted-run.compaction-trigger = 3
compaction.max-size-amplification-percent = 100
```

**为什么**: 聚合表需要频繁合并来保证查询结果准确性，更激进的 Compaction 策略能让查询看到最新的聚合结果。

#### 维表 (Lookup Join)

```properties
merge-engine = deduplicate
# 或 first-row
bucket = 固定值，根据数据量计算
# 查询加速
file-index.read.enabled = true
lookup.cache-max-memory-size = 256mb
```

**为什么**: 维表主要用于点查 (lookup join)，启用文件索引和 lookup cache 能显著减少不必要的文件读取。

### 9.3 Tag 和 Branch 的运维使用场景

#### Tag 的自动创建

**源码位置**: `CoreOptions.java:1682-1707`

```properties
# 自动按天创建 Tag
tag.automatic-creation = watermark
tag.creation-period = daily
tag.num-retained-max = 30
tag.default-time-retained = 30d
tag.creation-delay = 0
```

**使用场景**:
1. **时间旅行查询**: 通过 Tag 查询历史某一时刻的数据快照
2. **批处理回溯**: 批任务从指定 Tag 读取数据
3. **数据审计**: 保留关键时间点的数据状态

**为什么 `tag.creation-delay = 0`**: 默认不延迟创建 Tag。如果需要允许延迟数据进入当前 Tag 对应的快照，可以配置为 10min 或其他值，避免因为少量延迟数据导致跨 Tag 查询。

#### Tag 保留策略

- `tag.num-retained-max`: 最多保留的自动创建 Tag 数量 (源码: `CoreOptions.java:1722-1727`)
- `tag.default-time-retained`: Tag 的默认保留时间 (源码: `CoreOptions.java:1729-1735`)
- `tag.time-expire-enabled`: 是否开启按时间过期 (默认 true) (源码: `CoreOptions.java:1737-1741`)

#### Branch 的运维场景

1. **A/B 测试**: 在 Branch 上修改 Schema 或写入测试数据，不影响主分支
2. **数据修复**: 在 Branch 上执行数据修复操作，验证后再合并
3. **Schema 演进测试**: 在 Branch 上测试新的 Schema 变更

**实践建议**:
- 使用 `scan.fallback-snapshot-branch` 配置链式读取，主分支没有的分区自动从指定 Branch 读取
- Branch 上的独立 Compaction 不影响主分支
- 及时清理不再使用的 Branch，避免存储浪费
