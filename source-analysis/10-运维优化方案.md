## 1. 快速理解（运维心智模型 / 配置速查 / 高频事故）

### 1.1 运维心智模型：四个旋钮组与一条因果链

运维 Paimon 表，本质上是在调四组旋钮，平衡**写吞吐、读延迟、内存安全、存储成本**四个互相牵制的目标：

| 旋钮组 | 代表参数 | 调大 | 调小 | 监控指标 |
|--------|---------|------|------|---------|
| **Compaction 频率** | `num-sorted-run.compaction-trigger` | 写更快、读更慢、写放大低 | 读更快、写更慢、写放大高 | `compactionThreadBusy`、`maxLevel0FileCount` |
| **写入内存** | `write-buffer-size` | flush 少、文件大、内存压力大 | 内存安全、flush 频繁、小文件多 | `bufferPreemptCount`、`numWriters` |
| **数据并行度** | `bucket` | 并行度高、小文件多 | 文件大、并行度低 | `maxTotalFileSize`（倾斜）|
| **保留窗口** | `snapshot.*` / `partition.expiration-*` | 回溯能力强、存储贵 | 存储省、回溯受限 | snapshot 数量、存储占用 |

**一条因果链（运维必须背下来）**：写入太快 → WriteBuffer 频繁 flush 出小文件 → Level 0 sorted run 堆积 → 达到 `compaction-trigger` 触发合并 → compaction 跟不上则 sorted run 继续涨 → 达到 `stop-trigger` **阻塞写入**（背压）。这条链上每一环都有对应的旋钮和指标：发现写入抖动，就沿这条链从 `compactionThreadBusy` → `maxLevel0FileCount` → `bufferPreemptCount` 逐段定位（详见 §9）。

> 机制层面"为什么是 LSM、sorted run 如何归并、universal compaction 三级策略"见 `01 §3/§8` 与 `23`，本文不再展开。

### 1.3 八个高频生产事故

**事故 1：`compaction-trigger` 配太小（如 2）追求读性能。** Compaction 过于频繁，写放大飙升，写吞吐可降 50% 以上、checkpoint 频繁超时。运维信号：`compactionThreadBusy` 长期 >80% 但 `maxLevel0FileCount` 很低。修：回调到 5，或改用 Dedicated Compaction（§2.2）。

**事故 2：`stop-trigger` 配太大（如 100）。** 这是写入背压的**最后防线**，配大了意味着 sorted run 涨到 100 才阻塞写入，而此时读取早已不可用（Level 0 要全量归并）。默认 `trigger+3`（见 `numSortedRunStopTrigger()` 在 `CoreOptions.java:3068`）。运维：给 `maxLevel0FileCount` 配 `stop-trigger × 0.8` 的告警，别等阻塞才发现。

**事故 3：`write-buffer-size` × `bucket` 乘出 OOM。** 内存预算 ≈ `并发 writer 数 × write-buffer-size`。配 `1gb` × `256 bucket` = 256GB 内存需求，单 TaskManager 16GB 必然 OOM。修：按 §3.1 公式反推 buffer 大小，并保留 `write-buffer-spillable=true` 兜底。

**事故 4：关掉 `write-buffer-spillable` 追求性能。** 溢写是 OOM 的最后安全阀，关掉后内存满了无路可退，直接 OOM 任务挂掉、checkpoint 失败、已写数据无法提交。生产环境**永远不要关**。

**事故 5：`snapshot` 保留窗口太短，下游消费断流。** 配 `time-retained=10min` + `num-retained.min=3`，下游因网络抖动暂停 15min，恢复后所需快照已被删，任务失败。注意：正在被 Consumer 消费的快照受 `consumerManager.minNextSnapshot()` 保护（`ExpireSnapshotsImpl.java:135`），所以"配了 min=10 却保留了 1000 个"通常是有残留 Consumer 没清理。

**事故 6：分区时间格式不匹配，分区永不过期。** 配 `partition.timestamp-formatter=yyyy-MM-dd` 但实际分区值是 `dt=20240101`（无分隔符），解析失败 → 分区永远不过期 → 存储无限增长。修：让 formatter/pattern 与真实分区值严格对齐（§5）。

**事故 7：分区过期检查间隔 ≫ 过期时间，保留时间漂移。** 配 `check-interval=24h` + `expiration-time=1h`，分区本该 1h 过期，却要等 24h 才检查一次，实际保留 1–25h 不可控。原则：`check-interval` 应远小于 `expiration-time`。

**事故 8：误用异步过期/异步 compaction 后积压。** `snapshot.expire.execution-mode=async` 时若删除速度跟不上 snapshot 生成速度，文件持续积压；MOW（deletion-vectors）表强行异步 compaction 还会导致新写数据可见延迟（详见 `01 §1.3 陷阱 8`）。运维：监控 snapshot 数量趋势，持续上升即说明过期没跟上。

## 2. Compaction 运维调优

> 机制原理（universal compaction 三级 pick、CompactManager 异步调度）详见 `01 §8` 与 `23`。本节只讲运维旋钮、配错后果与排障动作。

### 2.1 五个核心旋钮与取舍

**① 要解决什么运维问题**：LSM 主键表持续写入会不断生成 Level 0 小文件，sorted run 越堆越多，读取归并路数线性上涨。Compaction 把多个 run 合并成大文件，控制读放大；但合并本身吃 CPU/IO，配错了会反噬写入。运维要做的是把 compaction 调到"既不积压、又不抢写入资源"的平衡点。

**② 五个旋钮的取舍**：

| 旋钮 | 默认值 | 行号（已修正）| 调大的影响 | 调小的影响 | 适用判断 |
|------|-------|------|----------|----------|---------|
| `num-sorted-run.compaction-trigger` | 5 | `CoreOptions.java:756` | 合并少、写快、读慢、写放大低 | 合并勤、读快、写慢、写放大高 | 写多读少调大，读多写少调小 |
| `num-sorted-run.stop-trigger` | trigger+3 | `:764` | 容忍更多堆积、背压晚 | 背压早、写入更易卡 | 不要随意调大（背压最后防线）|
| `compaction.max-size-amplification-percent` | 200 | `:809` | 容忍更多空间放大、全量合并少 | 频繁全量合并、I/O 高 | 存储贵调小，写多调大 |
| `compaction.size-ratio` | 1 | `:824` | 合并候选集更大、更激进 | 合并更保守 | 一般保持默认 |
| `compaction.force-up-level-0` | false | `:817` | 候选总是包含全部 L0 文件 | — | Lookup/读优化场景可开 |

> 注意：`compaction-trigger` 这一个参数同时影响 **compaction 频率、默认层数（`numLevels()=trigger+1`，`CoreOptions.java:3076`）、归并时的 run 数（峰值内存）** 三件事，调大写更快但单次归并内存更高、查询更慢。机制详解见 `01 §1.3 陷阱 7`。

**③ 关键源码（运维只需理解触发条件）**：`UniversalCompaction.pick()`（`UniversalCompaction.java:67`，已修正——旧稿行号一致但路径误标 mergetree 顶层）按优先级依次尝试：

```
步骤 1 pickForSizeAmp   —— 空间放大超阈值 → 全量合并（代价最贵）
步骤 2 pickForSizeRatio —— 按 size-ratio 凑候选集 → 增量合并
步骤 3 runs > trigger   —— 强制取 (runs - trigger + 1) 个最老 run 合并
```

`runs.size() < trigger` 时步骤 1/2 都返回 null（`pick()` 第 75 行）。空间放大判定在 `pickForSizeAmp()`（`UniversalCompaction.java:125`）：

```java
// UniversalCompaction.java:139（已修正，旧稿 125-147 范围偏移）
if (candidateSize * 100 > maxSizeAmp * earliestRunSize) {
    return CompactUnit.fromLevelRuns(maxLevel, runs); // 触发全量合并
}
```

`stop-trigger` 背压判定在 `MergeTreeCompactManager.shouldWaitForLatestCompaction()`（`MergeTreeCompactManager.java:109`，行号核对一致）：sorted run 数超过 `numSortedRunStopTrigger` 即阻塞写入；`shouldWaitForPreparingCheckpoint()`（`:114`）则用 `stop-trigger+1` 作为 checkpoint 前的等待阈值。

**④ 配错会怎样 / 怎么发现**：
- `compaction-trigger` 配 2：写吞吐降 50%+、checkpoint 频超时。信号：`compactionThreadBusy` 长期 >80% 而 `maxLevel0FileCount` 很低。
- `stop-trigger` 配 100：sorted run 涨到 100 才背压，此时 Level 0 要全量归并、读取已不可用。信号：`maxLevel0FileCount` 远超默认 8 但写入未被阻塞。
- `max-size-amplification-percent` 配太低（如 50）：频繁触发全量合并，`maxCompactionInputSize` 反复出现峰值、`avgCompactionTime` 拉长。

**⑤ 怎么修**：写入抖动优先用 Dedicated Compaction（§2.2）把合并搬出写入链路，而不是一味调 trigger；读慢且 L0 高才调小 trigger 或配 `compaction.optimization-interval` 做定期全量合并（§2.3）。

### 2.2 Dedicated Compaction：写入与合并解耦

**① 要解决什么运维问题**：默认（写入内 compaction）下写入和合并在同一个 Flink Job 抢 CPU/IO，最典型的事故是 checkpoint 时间从 30s 突增到 5min 甚至超时。Dedicated Compaction 把合并拆到独立的 compact job，写入侧只追加不合并。

**② 取舍**：

| 模式 | 运维复杂度 | 写入稳定性 | 资源利用 | 适用 |
|------|----------|----------|---------|------|
| 写入内 compaction（默认）| 低（单 job）| 易抖动 | 写入与合并争抢 | 中小规模、对延迟不敏感 |
| Dedicated（`write-only=true`）| 高（多一个 compact job）| 平稳 | 可独立扩缩 | 大规模、checkpoint 敏感、CDC 入湖 |

**③ 关键源码**：`write-only`（`CoreOptions.java:634`，行号核对一致）官方描述：`"If set to true, compactions and snapshot expiration will be skipped. This option is used along with dedicated compact jobs."` 注意它同时跳过 **snapshot 过期**——所以开了 `write-only` 后必须由 compact job（或独立的 expire 动作）负责过期，否则快照永不清理。

**④ 配错会怎样**：只在写入侧开 `write-only=true` 却没起 compact job → 文件永不合并、`maxLevel0FileCount` 一路涨到 `stop-trigger` 把写入卡死，同时 snapshot 永不过期、存储疯涨。

**⑤ 怎么修**：开 `write-only` 必须配套一个 dedicated compaction job（Flink `compact` action / procedure），并确认它在跑、`compactionThreadBusy` 有读数。

### 2.3 Off-Peak 与 Early Full Compaction

**Off-Peak（低峰更激进合并）**：`compaction.offpeak.start.hour` / `end.hour`（默认 -1 禁用，`CoreOptions.java:833`/`:841`，已修正）划定低峰时段，`compaction.offpeak-ratio` 在该时段把额外比例叠加到 `size-ratio` 上，使合并更激进。源码：`ratioForOffPeak()`（`UniversalCompaction.java:184`）取 `LocalDateTime.now().getHour()` 判定——**坑：用的是 JVM 时区，不是数据中心时区**，跨时区部署务必核对 `user.timezone`。

**Early Full Compaction（定期/按量强制全量合并）**：由 `EarlyFullCompaction` 实现，三类触发条件——`compaction.optimization-interval`（距上次全量合并的间隔）、总大小阈值、非顶层增量大小阈值。最常用的是 `optimization-interval`，它保证 read-optimized 系统表的查询时效性。**坑**：`changelog-producer=full-compaction` 若不配触发间隔，可能数小时才出一次 changelog（详见 `01 §1.3 陷阱 5` 与 `24`）。

### 2.4 场景化推荐配置

```properties
# 高吞吐写入：放宽触发 + Dedicated，消除写入路径上的合并开销

num-sorted-run.stop-trigger = 15
write-only = true                              # 配套独立 compact job
compaction.max-size-amplification-percent = 400
```

```properties
# 读多写少：更勤合并 + 定期全量，保证读延迟稳定
num-sorted-run.compaction-trigger = 3
num-sorted-run.stop-trigger = 6
compaction.size-ratio = 5
compaction.optimization-interval = 30min
compaction.max-size-amplification-percent = 100
```

```properties

num-sorted-run.stop-trigger = 8
write-only = true
compaction.offpeak.start.hour = 2             # 注意 JVM 时区
compaction.offpeak.end.hour = 8
compaction.offpeak-ratio = 20
```

## 3. 内存与溢写调优

> 内存共享池、SortBuffer、溢写排序的机制原理详见 `18`。本节只讲内存预算公式、配错后果与 OOM 排障。

### 3.1 内存预算公式与核心参数

**① 要解决什么运维问题**：每个并发 writer（≈ 一个 partition×bucket 组合）需要一块独立的排序缓冲区。多 bucket / 多分区批量回填时，writer 数瞬间膨胀，总内存需求成倍上涨，配不好就是反复 OOM。运维核心是**先估总内存、再反推单 buffer 大小、最后留溢写兜底**。

**② 内存预算公式（运维必背）**：

```
峰值写入内存 ≈ 并发 writer 数 × write-buffer-size
并发 writer 数 ≈ min(同时写入的 partition×bucket 组合数, Flink 写入并行度上的实际承载)
```

例：单 TaskManager 给写入预留 8GB 堆外内存，预计同时写 16 个分区 × 4 bucket = 64 个 writer，则 `write-buffer-size ≤ 8GB / 64 = 128MB`。注意这是**峰值**估算；Paimon 用共享内存池 + 抢占（§3.2）做动态均衡，但总量是硬上限，超了只能靠溢写。

**核心参数**：

| 参数 | 默认值 | 行号（已修正）| 运维含义 |
|------|-------|------|---------|
| `write-buffer-size` | 256 MB | `CoreOptions.java:657` | 单 writer 排序缓冲；满则 flush 出一个 L0 文件 |
| `page-size` | 64 KB | `:701` | 内存分配最小单元；配太小（4KB）GC 压力大 |
| `write-buffer-spillable` | true | `:672` | OOM 最后安全阀，**生产勿关** |
| `sort-spill-threshold` | `stopTrigger+1` | `:606`（计算见 `sortSpillThreshold()` `:2868`，已修正）| 归并读取器超此数则溢写，防 compaction OOM |
| `write-max-writers-to-spill` | 10 | `:685` | 批量 append writer 超此数自动开启 buffer cache + 溢写 |

**③ 关键源码**：`write-buffer-size` 决定 `SortBufferWriteBuffer` 的排序缓冲容量，满则 flush 成 L0 文件；`page-size` 决定 `MemoryPoolFactory` 的总页字节数（`MemoryPoolFactory.java:46`：`this.totalBufferSize = (long) totalPages * innerPool.pageSize();`）。

**④ 配错会怎样 / 怎么发现**：
- `write-buffer-size=1gb` × `bucket=256`：256GB 内存需求，16GB TM 瞬间 OOM。
- `page-size=4kb`：分配次数暴增、GC 抖动、吞吐降约一半。
- 关 `write-buffer-spillable`：内存满后直接 OOM。
- 发现手段：监控 `bufferPreemptCount`（抢占次数）持续快速增长 = 内存严重不足；`numWriters` 远超 bucket 数 = 分区扇出过大。

**⑤ 怎么修**：按上面公式反推 buffer 大小；内存仍紧张则缩 buffer + 开溢写 + 限 `write-max-writers-to-spill`；不要靠盲目加 TM 内存（治标不治本，且分区扇出大时无底洞）。

### 3.2 共享内存池与抢占（运维视角）

> 抢占的完整机制详见 `18`。运维只需知道：它如何均衡热点 bucket、以及哪个指标暴露内存不足。

多个 writer 共享同一个内存池。当某 writer 申请页而池满时，`MemoryPoolFactory.preemptMemory()`（`MemoryPoolFactory.java:73`，行号核对一致）会找到**占用最大的其他 writer**强制其 `flushMemory()` 溢写，腾出内存给申请者，并自增 `bufferPreemptCount`：

```java
// MemoryPoolFactory.java:73 起（精选）
private void preemptMemory(MemoryOwner owner) {
    // 找占用最大的 other（排除自己），强制其 flushMemory() 溢写
    if (max != null) {
        max.flushMemory();
        ++bufferPreemptCount;   // 运维核心指标：抢占次数
    }
}
```

**运维要点**：
- 抢占是**同步**的——抢占时会短暂阻塞申请者，频繁抢占 = 写入毛刺。
- `bufferPreemptCount` 是内存压力的**第一信号**：偶发可接受，**持续快速增长**必须扩内存或缩 buffer。
- 抢占只在 writer 之间挪内存，**总量不变**；总量不够时最终落到溢写（§3.3）。

### 3.3 溢写参数与调优建议

| 参数 | 默认值 | 行号（已修正）| 含义 |
|------|-------|------|------|
| `write-buffer-spillable` | true | `CoreOptions.java:672` | 是否允许 write buffer 溢写磁盘 |
| `write-buffer-spill.max-disk-size` | 无限制 | `:665` | 溢写磁盘容量上限 |
| `sort-spill-buffer-size` | 64 MB | `:614` | 溢写排序时每次写盘的数据量 |
| `spill-compression` | zstd | `:620` | 溢写压缩算法（zstd/lz4/lzo）|

**坑**：溢写文件依赖 checkpoint 完成后才清理，**checkpoint 反复失败会导致溢写文件在本地磁盘堆积**，最终撑爆本地盘。排障时若发现 TM 本地磁盘满，先查 checkpoint 是否健康。

```properties
# 内存充足、bucket 数有限：大 buffer 少 flush
write-buffer-size = 512mb
page-size = 128kb
write-buffer-spillable = true          # 即使内存充足也建议留兜底（勿设 false）
```

```properties
# 内存紧张、多分区多 bucket：缩 buffer + 溢写
write-buffer-size = 64mb
page-size = 32kb
write-buffer-spillable = true
write-buffer-spill.max-disk-size = 10gb
spill-compression = zstd
write-max-writers-to-spill = 5
```

监控用 `WriterBufferMetric`（`WriterBufferMetric.java`）：`numWriters`、`bufferPreemptCount`、`usedWriteBufferSizeByte`、`totalWriteBufferSizeByte`。告警：`bufferPreemptCount` 增速 > 10/min 即报警（详见 §8.4）。

## 4. 快照与 Changelog 过期运维

> Snapshot 双清单结构、时间旅行机制详见 `01 §12` 与 `17`；Changelog 四种 producer 的产生机制详见 `24`。本节只讲过期运维：护栏、限额、解耦实践。

### 4.1 过期算法的五道护栏

**① 要解决什么运维问题**：每次 commit 产生一个 snapshot，不清理则存储无限增长、manifest 扫描变慢、任务启动变慢。但清理过头会删掉下游正在用的快照。过期算法用**五道护栏**在"清得干净"和"别误删"之间取平衡。

**② 五道护栏（运维必须知道每道由谁控制）**：

| 护栏 | 控制参数/来源 | 作用 |
|------|-------------|------|
| 最多保留 | `snapshot.num-retained.max`（`CoreOptions.java:503`）| 上限，超出的最老快照可删 |
| 最少保留 | `snapshot.num-retained.min`（`:495`，默认 10）| 下限，至少留这么多 |
| Consumer 保护 | `consumerManager.minNextSnapshot()` | 不删下游正在消费的快照 |
| 单次限额 | `snapshot.expire.limit`（`:542`，默认 50）| 一次最多删多少，防止过删过多 |
| 时间窗口 | `snapshot.time-retained`（`:510`，默认 1h）| 保留时间内的快照不删 |

**③ 关键源码**：过期范围计算在 `ExpireSnapshotsImpl`（`ExpireSnapshotsImpl.java:101` 起，行号核对一致）：

```java
// ExpireSnapshotsImpl.java:126 起（精选护栏叠加逻辑）
long min = Math.max(latestSnapshotId - retainMax + 1, earliest);        // num-retained.max
long maxExclusive = latestSnapshotId - retainMin + 1;                    // num-retained.min
maxExclusive = Math.min(maxExclusive,
        consumerManager.minNextSnapshot().orElse(Long.MAX_VALUE));       // Consumer 保护
maxExclusive = Math.min(maxExclusive, earliest + maxDeletes);           // expire.limit
// 再从 min 遍历到 maxExclusive，命中 time-retained 内的快照即提前停止
```

**④ 配错会怎样**：`num-retained.min=3` + `time-retained=10min`，下游抖动 15min 后所需快照被删、消费失败。**怎么发现**：配了 `min=10` 却保留了上千个 snapshot，几乎一定是**残留 Consumer** 把 `minNextSnapshot()` 顶住了——去 `$consumers` 系统表查并清理无用 Consumer。

**⑤ 怎么修**：流式场景把 `num-retained.min` 提到 20+、`time-retained` 给足故障恢复窗口（如 2h）；不要靠缩短保留来省存储（应改用 Changelog 解耦，§4.3）。

### 4.2 同步/异步过期与单次限额

`snapshot.expire.execution-mode`（`CoreOptions.java:536`）：
- `sync`（默认）：commit 后同步过期。文件多时阻塞流处理。
- `async`：异步过期。若删除速度跟不上 snapshot 生成速度则积压。

`snapshot.expire.limit`（`:542`，默认 50）：单次最多过期的快照数，防止一次删数百万文件导致 checkpoint 超时。

**坑**：`write-only=true` 会**跳过过期**（见 §2.2），此时必须由 compact job 或独立 expire 动作兜底，否则快照永不清理。

```properties
# 流式写入：足够回溯 + 限单次量 + 高吞吐用异步
snapshot.num-retained.min = 20
snapshot.num-retained.max = 200
snapshot.time-retained = 2h
snapshot.expire.limit = 30
snapshot.expire.execution-mode = async   # 但需监控 snapshot 数量趋势防积压
```

### 4.3 Changelog 解耦与最佳实践

当配置了 `changelog.num-retained.min/max`（`CoreOptions.java:516`）或 `changelog.time-retained`（`:530`）时，Changelog 生命周期与 Snapshot **解耦**（`isChangelogDecoupled()`）：即使 snapshot 过期，对应 changelog 仍可独立保留供下游流式消费。

**运维价值**：snapshot 短期保留（省数据文件存储）+ changelog 长期保留（满足下游回溯），两者分开调。

```properties
# 需要长时间流式消费：snapshot 短留、changelog 长留
snapshot.num-retained.min = 10
snapshot.time-retained = 1h
changelog-producer = lookup
changelog.time-retained = 24h
changelog.num-retained.min = 100
```

> changelog producer 的选择（none/input/lookup/full-compaction）及其延迟/成本权衡详见 `24`。

## 5. 分区过期运维

> 时间旅行、分区生命周期的机制详见 `17`。本节聚焦分区过期的配置坑与运维动作。

**① 要解决什么运维问题**：时间分区表持续产新分区，历史分区不清理则存储无限增长、查询扫描分区数膨胀。Paimon 支持自动过期，免去手动 `DROP PARTITION`，但**配置稍有不对就静默失效**（分区永不过期），是高频事故。

**② 两种策略 + 四个旋钮**：

| 项 | 取值 | 说明 |
|----|------|------|
| 过期策略 | `values-time`（默认）/ `update-time` | 前者从分区值提取时间，后者用分区最后更新时间 |
| `partition.expiration-time` | 无（不配则不启用）`:1152` | 超过此时长的分区被删 |
| `partition.expiration-check-interval` | 1h `:1161` | 检查频率，须 ≪ expiration-time |
| `partition.expiration-max-num` | 100 `:1167` | 单次最多删的分区数 |
| `partition.expiration-batch-size` | =max-num `:1173` | 分批删除批大小，防 OOM |
| `partition.timestamp-formatter` | `yyyy-MM-dd[ HH:mm:ss]` `:1182` | 从分区值解析时间的格式 |
| `partition.timestamp-pattern` | — `:1201` | 多字段组合时间，如 `$year-$month-$day $hour:00:00` |

> 行号均已对当前 `CoreOptions.java` 重新核对（旧稿偏移已修正）。

**③ 关键源码**：过期触发在 `PartitionExpire.expire()`（`PartitionExpire.java:140`，行号核对一致），三种触发条件——`checkInterval` 为零（每次 commit 都查）、距上次检查超过 `checkInterval`、流任务结束（`commitIdentifier == Long.MAX_VALUE`）。初始检查时间做了**随机化**（`PartitionExpire.java:82`：`ThreadLocalRandom...nextLong(checkIntervalSeconds)`），避免所有任务同时检查造成文件系统压力突增。分批删除在 `:167`：`Lists.partition(expired, expireBatchSize)`。

**④ 配错会怎样（两大经典静默故障）**：
- **格式不匹配 → 永不过期**：formatter 配 `yyyy-MM-dd` 但分区值是 `20240101`（无分隔符），解析失败，分区永远不删，存储疯涨。**怎么发现**：存储持续上涨但分区数从不下降。
- **检查间隔 ≫ 过期时间 → 保留漂移**：`check-interval=24h` + `expiration-time=1h`，实际保留 1–25h 不可控。

**⑤ 怎么修**：上线前用真实分区值验证 formatter/pattern 能解析出时间；`check-interval` 设为远小于 `expiration-time`；分区多时设 `batch-size` 限制单批删除量防 OOM。

```properties
# 日级分区表
partition.expiration-time = 30d
partition.expiration-check-interval = 1h
partition.expiration-max-num = 50
partition.timestamp-formatter = yyyy-MM-dd
partition.timestamp-pattern = $dt
```

```properties
# 小时级分区表（量大、分区多）
partition.expiration-time = 7d
partition.expiration-check-interval = 30min
partition.expiration-max-num = 100
partition.expiration-batch-size = 20        # 分批删，防一次删 100 个 OOM
partition.timestamp-formatter = yyyy-MM-dd HH:mm:ss
partition.timestamp-pattern = $dt $hour:00:00
```

## 6. Bucket 容量规划

> 五种 BucketMode、KeyAndBucketExtractor、动态桶索引维护的机制详见 `16` 与 `01 §10`。本节只讲容量规划公式、改不动的约束与倾斜排查。

**① 要解决什么运维问题**：bucket 是 Paimon 的并行单元，数量直接决定写入/读取/compaction 并行度和单桶数据量。**bucket 数定了基本改不动**（改需重建表或专门的 rescale 操作），所以必须上线前算好。过少 → 写瓶颈、单桶巨大；过多 → 小文件爆炸、元数据膨胀、内存吃紧。

**② 四种模式与判断（运维只需会选）**：

| 模式 | `bucket` 取值 | 选用条件 | 运维注意 |
|------|--------------|---------|---------|
| HASH_FIXED | `>0` | 数据规模已知 | 数量定后不可改，需预留增长 |
| HASH_DYNAMIC | `-1`，无跨分区更新 | 规模不定 | 桶数随数据增长，需监控 |
| KEY_DYNAMIC | `-1`，有跨分区更新 | 主键不含全部分区字段 | 启动需读全表主键建索引 |
| POSTPONE_MODE | `-2` | 让系统自适应延迟分配 | 特殊场景 |

判定逻辑（`KeyValueFileStore.bucketMode()`）：`-2`→POSTPONE，`-1`→按 `crossPartitionUpdate` 选 KEY_DYNAMIC / HASH_DYNAMIC，`>0`→HASH_FIXED。**坑**：跨分区更新场景误用 HASH_DYNAMIC（而非 KEY_DYNAMIC）会导致同一 key 在新旧分区各写一份、数据重复（详见 `01 §1.3 陷阱 4`）。

**③ 容量规划公式**：

```
bucket_num = ceil(单分区数据量 / target-file-size / compaction-trigger)
```

例：单分区 100GB、`target-file-size=128MB`、`compaction-trigger=5` → `ceil(100*1024/128/5) = 160`。经验上让**单桶稳定数据量落在 1–5GB**。动态桶用 `dynamic-bucket.target-row-num`（`CoreOptions.java:1422`，默认 2,000,000）控制每桶目标行数——配太小桶数会爆炸。

| 单分区每日数据量 | 建议 bucket | 说明 |
|---------------|-----------|------|
| < 1 GB | 1–2 | 避免小文件 |
| 1–10 GB | 4–8 | 保证基本并行度 |
| 10–100 GB | 16–64 | 平衡并行度与文件大小 |
| 100GB–1TB | 64–256 | 高并行度写入 |
| > 1TB / 不确定 | 动态桶 | 自适应 |

**④ 配错会怎样 / 怎么发现**：
- 过少：写吞吐远低于硬件能力；单桶巨大、compaction 从秒级变小时级。
- 过多：小文件爆炸、对象存储 listing 超时、`numWriters` 高导致内存吃紧。
- 倾斜：对比 `maxTotalFileSize` 与平均值，差距大说明 bucket-key 选得不好、热点集中。

**⑤ 怎么修**：上线前按公式估算并预留 1–2 年增长；倾斜则重选 bucket-key；规模实在不定就用动态桶。**注意：固定桶数改不了，规划务必保守。**

## 7. 文件格式与压缩调优

> 文件格式与 IO 层机制详见 `20`；文件索引（Bloom/Bitmap/BSI）详见 `13`。本节只讲格式/压缩的运维选型与配错后果。

**① 要解决什么运维问题**：格式和压缩直接决定存储成本、写入吞吐、查询 I/O。选错可能成本翻倍或查询慢 10 倍。运维要按**层级冷热**和**存储介质**选型。

**② 选型决策表**：

| 维度 | 选项 | 何时用 |
|------|------|--------|
| 格式 | Parquet（默认）| 列式 OLAP 查询，生态兼容最好 |
| 格式 | ORC | Hive 生态、列式 |
| 格式 | Avro | 行式，L0 频繁写、或全行读取 |
| 压缩 | zstd（默认 level=1）| 综合最优 |
| 压缩 | lz4 | 写入密集、延迟敏感（L0）|
| 压缩 | zlib/gzip | 冷数据归档 |

**按层级配置**（运维最实用的优化）：L0 频繁写合并用快速格式/压缩，高层稳定存储用高压缩比。
- `file.format.per.level`（`CoreOptions.java:307` 区域）：如 `'0:avro,3:parquet'`
- `file.compression.per.level`（`:299` 区域）：如 `'0:lz4,1:zstd'`

**③ 关键参数与源码**：
- `file.compression`（`CoreOptions.java:318`，行号核对一致）默认 zstd；`file.compression.zstd-level` 默认 1，官方注释明确"调到 9 压缩率更高但读写显著变慢"。
- `target-file-size`（`:713`）：PK 表默认 128MB、append 表默认 256MB（`targetFileSize()` `:2959`，已修正）。`compactionFileSize() = target × 70%`（`:3042`，已修正）——只有大于 target 70% 的文件才参与 compaction，避免刚写入就被反复合并。

**④ 配错会怎样**：
- `zstd-level=22`：压缩耗时从 1s 涨到 30s，写吞吐降约 90%，压缩率只多约 10%——得不偿失。
- `target-file-size=16mb`：小文件爆炸、元数据开销大、查询打开文件的开销超过读数据本身。

**⑤ 怎么修 / 调优建议**：

| 场景 | target-file-size | 原因 |
|------|-----------------|------|
| HDFS | 128–256 MB | 匹配 block 大小，减少远程读 |
| 对象存储 S3/OSS | 256–512 MB | 减少 API 调用与 listing 开销 |
| 快速查询 | 64–128 MB | 更细裁剪粒度 |
| 高吞吐写入 | 256 MB | 减少 compaction 频率 |

压缩默认 zstd level=1 即可；只有存储成本特别敏感且 CPU 富余时才调高 level，且建议按层级只对高层调高。

## 8. 监控指标与告警阈值

> Flink 自带指标看不到 Paimon 内部的 compaction/内存状态——必须额外接 Paimon 的四类指标。下面表格中的字段名、繁忙度算法均已对源码核对。

**为什么必须监控 Paimon 自有指标**：Flink checkpoint 时间正常 ≠ 写入健康，因为 compaction 在 checkpoint 之外执行、内存抢占也不反映在 Flink 指标里。运维要盯的四类指标分别对应 §1.1 因果链上的四个环节：CommitMetrics（提交/合并活跃度）、CompactionMetrics（合并健康）、ScanMetrics（查询裁剪效果）、WriterBufferMetric（内存压力）。

**两个监控原则**：① 阈值要跟着表配置走——`maxLevel0FileCount` 告警阈值应是该表 `stop-trigger × 0.8`，而不是写死 100；② 看趋势不只看瞬时——`bufferPreemptCount` 当前值 10 不可怕，但 100/min 的增速说明马上要 OOM。

## 9. 故障排查 SOP

> 排查原则：先看指标定位环节、再结合配置/源码找根因，**每次只调一个参数、等至少一个 checkpoint 周期再判断效果**。下面三条链路（写入慢 / 读慢 / compaction 积压）+ 四类 OOM 是生产最高频的问题。

### 9.1 写入延迟高的排查路径

```
1. 检查 compactionThreadBusy 指标
   ↓ 如果 > 80%
   说明 Compaction 跟不上写入速度
   → 增大 num-sorted-run.compaction-trigger
   → 切换为 Dedicated Compaction (write-only=true)
   → 增加 Compaction 并行度

2. 检查 bufferPreemptCount
   ↓ 如果快速增长
   说明内存不足，writer 之间频繁抢占
   → 增大 write-buffer-size
   → 减少 bucket 数量 (减少 writer 数量)
   → 增大 TaskManager 内存

3. 检查 maxLevel0FileCount
   ↓ 如果接近 numSortedRunStopTrigger
   说明 Compaction 积压严重，写入被阻塞
   → 增大 num-sorted-run.stop-trigger (临时缓解)
   → 分析 Compaction 为何慢 (文件过大? I/O 瓶颈?)

4. 检查 lastCommitAttempts
   ↓ 如果 > 3
   说明提交冲突频繁
   → 检查是否有多个写入任务写同一表
   → 考虑使用不同的 commit-user-prefix
```

### 9.2 读取慢的排查路径

```
1. 检查 lastScanDuration
   ↓ 如果 > 10s
   说明 manifest 扫描耗时
   → 检查 lastScannedManifests 数量
   → 增大 manifest.full-compaction-threshold-size 触发 manifest 合并
   → 检查 manifest 缓存命中率 (manifestHitCache / total)

2. 检查 lastScanResultedTableFiles
   ↓ 如果过多 (比如 > 1000)
   → 检查分区裁剪是否生效 (WHERE 条件是否包含分区列)
   → 检查 file index 是否配置
   → 检查 Compaction 是否正常，文件数量是否过多

3. 检查 maxLevel0FileCount
   ↓ 如果偏高
   → Level0 文件需要全部扫描且 key range 重叠
   → 增强 Compaction (减小 compaction-trigger)
   → 考虑使用 compaction.optimization-interval 定期全量合并

4. 检查数据倾斜
   → 对比 maxTotalFileSize 和 avgTotalFileSize
   → 如果差异大说明某些 bucket 数据量远超其他
   → 重新评估 bucket-key 和 bucket 数量
```

### 9.3 Compaction 积压的排查

```
1. 检查 compactionQueuedCount
   ↓ 如果持续 > 0
   说明 Compaction 线程不够

2. 检查 avgCompactionTime
   ↓ 如果持续增长
   可能原因:
   a. target-file-size 过大导致单次合并数据量过多
   b. 空间放大触发全量合并 (maxSizeAmp 过低)
   c. 磁盘/网络 I/O 瓶颈

3. 检查 compactionInputSize / compactionOutputSize 比率
   → 如果输入远大于输出，说明合并效率好 (大量重复 key 被合并)
   → 如果接近 1:1，说明数据重复率低，合并属于"白做功"
   → 考虑增大 compaction-trigger 减少不必要的合并

4. 解决方案优先级:
   a. 使用 Dedicated Compaction 分离写入和合并资源
   b. 配置 off-peak hours 在低峰期更积极合并
   c. 增大 Compaction 线程池
   d. 调整触发参数减少合并频率
```

### 9.4 OOM 场景和解决方案

> 四类 OOM 各有不同根因和限流参数，对症下药：

#### 场景 1: 写入 OOM

**原因**: 大量 bucket 的 writer 同时持有 sort buffer

**源码依据**: 每个 bucket 的 writer 需要 `SortBufferWriteBuffer`，其内存由 `MemoryPoolFactory` 统一管理。但如果 bucket 数量 * 实际每个 buffer 使用内存 > 总内存，即使有抢占机制也无法完全避免 OOM。

**解决方案**:
```properties
write-buffer-size = 64mb          # 减小单个 buffer
write-buffer-spillable = true      # 开启溢写
write-buffer-spill.max-disk-size = 5gb
write-max-writers-to-spill = 5     # 超过 5 个 writer 就开启溢写
```

#### 场景 2: Compaction OOM

**原因**: 合并大量文件时需要同时打开所有输入文件的 reader

**源码依据**: `sort-spill-threshold`（`CoreOptions.java:606`）控制合并读取器的上限，未显式配置时默认 `numSortedRunStopTrigger + 1`（计算见 `sortSpillThreshold()` `CoreOptions.java:2868`，已修正）。

**解决方案**:
```properties
sort-spill-threshold = 5            # 限制同时打开的 reader 数量
sort-spill-buffer-size = 32mb       # 减小溢写 buffer
local-sort.max-num-file-handles = 128  # 限制外部排序的 fan-in (默认值)
```

#### 场景 3: 扫描 Compaction 文件 OOM (Append 表)

**原因**: 无感知 bucket 的 Append 表 Compaction 时扫描文件过多

**源码依据**: `compaction.file-num-limit` (默认 100,000)。

**解决方案**:
```properties
compaction.file-num-limit = 10000   # 限制扫描文件数
```

#### 场景 4: 分区过期 OOM

**原因**: 一次删除过多分区

**源码依据**: `partition.expiration-batch-size` 控制分批删除。

**解决方案**:
```properties
partition.expiration-max-num = 50
partition.expiration-batch-size = 10
```

## 11. 设计决策总结

| 决策点 | 选择 | 取舍 / 代价 | 收益（运维视角）|
|--------|------|------------|---------|
| compaction 频率旋钮 | 单参数 `compaction-trigger` 控制频率/层数/归并 run 数 | 一参数三重作用，调优需理解联动 | 旋钮少、心智简单 |
| 写入背压 | `stop-trigger` 硬阻塞写入 | 写入被卡 | 防 sorted run 无限增长拖垮读取 |
| 写入与合并解耦 | `write-only` + 独立 compact job | 多一个作业、需自管过期 | 写入平稳、checkpoint 不超时、资源可独立扩缩 |
| 内存管理 | 共享池 + 抢占 + 溢写兜底 | 抢占同步阻塞、溢写有 IO 代价 | 数据倾斜下不静态浪费内存、OOM 有最后安全阀 |
| 快照过期 | 五道护栏叠加（max/min/Consumer/limit/time）| 残留 Consumer 会顶住快照 | 既清得干净又不误删下游在用的快照 |
| Changelog 解耦 | 独立保留策略 | 多一组配置 | 数据文件早释放省存储、changelog 长留供回溯 |
| 分区过期 | 自动按时间/更新时间过期 | 格式配错会静默永不过期 | 免手动 DROP PARTITION、随写入实时清理 |
| Bucket | 固定桶数定后不可改 | 必须提前规划 | hash 路由开销极低、幂等写入、并行隔离 |
| 文件格式/压缩 | 默认 Parquet+zstd，支持按层级配置 | 按层级配置需额外理解 | L0 快写、高层省存储，各得其所 |
| 监控 | 暴露 Commit/Compaction/Scan/WriterBuffer 四类指标 | 需额外接入告警 | Flink 指标看不到的内部状态可观测、可定位 |
