## 1. Compaction 调优

### 1.1 Compaction 核心参数解析

Paimon 的主键表采用 LSM-Tree 架构，Compaction 策略基于 RocksDB 的 Universal Compaction 实现。核心实现在 `UniversalCompaction.java`。

#### 1.1.1 `num-sorted-run.compaction-trigger`

- **源码位置**: `CoreOptions.java:745-751`
- **默认值**: `5`
- **含义**: 触发 Compaction 的 Sorted Run 数量阈值。包含 level0 文件 (一个文件一个 sorted run) 和高层级的 run (一个 level 一个 sorted run)。

**源码影响路径**:

在 `UniversalCompaction.pick()` 方法 (第 67-107 行) 中，Compaction 选择按以下优先级依次执行:

```
步骤 0: earlyFullCompact (如果配置了 optimization-interval 等触发条件)
步骤 1: pickForSizeAmp -> 检查空间放大是否超阈值
步骤 2: pickForSizeRatio -> 基于 size ratio 选择合并候选
步骤 3: 如果 runs.size() > numRunCompactionTrigger -> 强制选择合并
```

关键判断在 `pickForSizeAmp()` (第 125-147 行) 和 `pickForSizeRatio()` (第 149-182 行) 中:
- 如果 `runs.size() < numRunCompactionTrigger`，则步骤 1 和步骤 2 都直接返回 null
- 只有当 sorted run 数量达到触发阈值时才会尝试合并

**为什么重要**: 该值直接决定了读放大与写放大的平衡点。值越小，合并越频繁 (写放大高但读性能好)；值越大，合并越少 (写吞吐高但读需要扫描更多文件)。

#### 1.1.2 `num-sorted-run.stop-trigger`

- **源码位置**: `CoreOptions.java:753-759`
- **默认值**: `num-sorted-run.compaction-trigger + 3` (在 `numSortedRunStopTrigger()` 方法第 2987-2993 行计算)
- **含义**: 当 sorted run 数量超过该值时，写入将被阻塞等待 Compaction 完成。

**源码影响路径**:

在 `MergeTreeCompactManager.shouldWaitForLatestCompaction()` (第 109-111 行):
```java
public boolean shouldWaitForLatestCompaction() {
    return levels.numberOfSortedRuns() > numSortedRunStopTrigger;
}
```

还有 `shouldWaitForPreparingCheckpoint()` (第 114-117 行):
```java
public boolean shouldWaitForPreparingCheckpoint() {
    return levels.numberOfSortedRuns() > (long) numSortedRunStopTrigger + 1;
}
```

**为什么重要**: 这是写入背压的最后防线。当 Compaction 跟不上写入速度时，该参数控制最大允许的文件堆积量，超过则暂停写入。

**好处**: 防止 sorted run 无限增长导致读取性能急剧下降和内存不足。

#### 1.1.3 `compaction.max-size-amplification-percent`

- **源码位置**: `CoreOptions.java:798-804`
- **默认值**: `200`
- **含义**: 空间放大百分比阈值，即存储一个字节数据所需的额外存储空间百分比 (针对 changelog 模式表)。

**源码影响路径**:

在 `UniversalCompaction.pickForSizeAmp()` (第 125-147 行):
```java
long candidateSize = runs.subList(0, runs.size() - 1).stream()
        .map(LevelSortedRun::run).mapToLong(SortedRun::totalSize).sum();
long earliestRunSize = runs.get(runs.size() - 1).run().totalSize();
// 当除最底层外的所有层总大小超过最底层的 maxSizeAmp% 时触发全量合并
if (candidateSize * 100 > maxSizeAmp * earliestRunSize) {
    return CompactUnit.fromLevelRuns(maxLevel, runs);
}
```

**为什么重要**: 当空间放大超过阈值时，会触发全量 Compaction (将所有 sorted run 合并到最大层)，这是一个代价昂贵的操作，但能有效控制存储空间。

#### 1.1.4 `compaction.size-ratio`

- **源码位置**: `CoreOptions.java:813-820`
- **默认值**: `1`
- **含义**: 在比较 sorted run 大小时的百分比灵活度。

**源码影响路径**:

在 `UniversalCompaction.pickForSizeRatio()` (第 163-182 行):
```java
if (candidateSize * (100.0 + sizeRatio + ratioForOffPeak()) / 100.0 < next.run().totalSize()) {
    break; // 候选集大小 < 下一个 run 大小的 (100+ratio)%, 停止扩展
}
candidateSize += next.run().totalSize();
candidateCount++;
```

**为什么重要**: 较大的 `size-ratio` 意味着更多的 sorted run 会被包含进合并候选集，产生更积极的合并行为。

#### 1.1.5 Off-Peak Compaction

- **源码位置**: `CoreOptions.java:822-854`
- **配置项**:
  - `compaction.offpeak.start.hour` (默认 -1，禁用)
  - `compaction.offpeak.end.hour` (默认 -1，禁用)
  - `compaction.offpeak-ratio` (默认 0)

**源码影响路径**:

在 `UniversalCompaction.ratioForOffPeak()` (第 184-186 行):
```java
private int ratioForOffPeak() {
    return offPeakHours == null ? 0 : offPeakHours.currentRatio(LocalDateTime.now().getHour());
}
```

在 `pickForSizeRatio` 计算中，off-peak ratio 会被加到 sizeRatio 上，使得低峰时段合并更积极。

**好处**: 可以在业务低峰期 (如凌晨 2 点到早上 6 点) 执行更积极的 Compaction，减少对在线业务的影响。

#### 1.1.6 Early Full Compaction

- **源码位置**: `EarlyFullCompaction.java`
- **相关配置**:
  - `compaction.optimization-interval`: 周期性触发全量合并的间隔
  - `compaction.total-size-threshold`: 总大小低于此阈值时强制全量合并
  - `compaction.incremental-size-threshold`: 增量大小超过此阈值时强制全量合并

**源码影响路径**:

在 `EarlyFullCompaction.tryFullCompact()` (第 70-107 行)，按优先级检查三个条件:
1. 距上次全量合并是否超过 `fullCompactionInterval`
2. 所有 run 总大小是否小于 `totalSizeThreshold`
3. 非最大层的增量大小是否超过 `incrementalSizeThreshold`

**为什么有用**: `compaction.optimization-interval` 确保读优化系统表 (read-optimized) 的查询时效性，适合需要定期保证查询性能的场景。

### 1.2 Dedicated Compaction vs 写入内 Compaction

#### 写入内 Compaction (默认模式)

- 写入和 Compaction 在同一个 Flink Job 中执行
- 优点: 运维简单，不需要额外任务
- 缺点: 写入和 Compaction 争抢资源，可能导致写入延迟

#### Dedicated Compaction

- **启用方式**: 写入任务设置 `write-only = true` (源码位置: `CoreOptions.java:634-641`)
- **源码说明**: `"If set to true, compactions and snapshot expiration will be skipped. This option is used along with dedicated compact jobs."`

**为什么这么做**: 将 Compaction 负载从写入链路分离出来，避免 Compaction 造成的写入背压和 checkpoint 超时。

**好处**:
1. 写入不会因 Compaction 被阻塞 (不会触发 `shouldWaitForLatestCompaction()` 返回 true)
2. 可以独立调整 Compaction 资源
3. Compaction 可以在低峰期更积极地执行

### 1.3 场景化推荐配置

#### 高吞吐写入场景

```properties
# 放宽 Compaction 触发条件，减少写入阻塞
num-sorted-run.compaction-trigger = 10
num-sorted-run.stop-trigger = 15
# 使用 Dedicated Compaction
write-only = true
# 提高空间放大容忍度
compaction.max-size-amplification-percent = 400
```

**为什么**: 增大触发阈值减少 Compaction 频率，配合 dedicated compaction 完全消除写入路径上的合并开销。

#### 读多写少场景

```properties
# 更积极的 Compaction，保证读取性能
num-sorted-run.compaction-trigger = 3
num-sorted-run.stop-trigger = 6
compaction.size-ratio = 5
# 开启定期优化合并
compaction.optimization-interval = 30min
# 控制空间放大
compaction.max-size-amplification-percent = 100
```

**为什么**: 低触发阈值确保文件数量少，读取扫描更少文件。定期优化合并确保读取性能稳定。

#### 延迟敏感场景

```properties
# 适中的触发条件
num-sorted-run.compaction-trigger = 5
num-sorted-run.stop-trigger = 8
# 使用 Dedicated Compaction 避免写入抖动
write-only = true
# 低峰期更积极合并
compaction.offpeak.start.hour = 2
compaction.offpeak.end.hour = 8
compaction.offpeak-ratio = 20
```

**为什么**: Dedicated Compaction 消除写入毛刺，off-peak 配置在低峰期补回 Compaction 进度。

## 2. 内存管理调优

### 2.1 核心内存参数

#### 2.1.1 `write-buffer-size`

- **源码位置**: `CoreOptions.java:657-662`
- **默认值**: `256 MB`
- **含义**: 在将数据转换为排序的磁盘文件之前，在内存中积累的数据量。

**源码影响路径**:

`write-buffer-size` 决定了 `SortBufferWriteBuffer` (第 65-74 行) 中 sort buffer 的大小。当 buffer 满后会触发 flush 操作，将内存中的排序数据写入 level0 文件。

**为什么重要**: 更大的 write buffer 意味着:
1. 每次 flush 产生更大的 level0 文件，减少 Compaction 频率
2. 内存中合并更多记录，减少重复 key 的磁盘写入
3. 但会增加内存消耗

#### 2.1.2 `page-size`

- **源码位置**: `CoreOptions.java:701-705`
- **默认值**: `64 KB`
- **含义**: 内存页大小，是内存分配的最小单元。

**源码影响路径**:

在 `MemoryPoolFactory` (第 46 行):
```java
this.totalBufferSize = (long) totalPages * innerPool.pageSize();
```

页大小影响内存分配粒度。较大的页减少分配次数但可能浪费空间，较小的页更灵活但增加管理开销。

#### 2.1.3 `sort-spill-threshold`

- **源码位置**: `CoreOptions.java:606-612`
- **默认值**: 无 (默认计算为 `numSortedRunStopTrigger + 1`，见 `sortSpillThreshold()` 方法第 2803-2810 行)
- **含义**: 当排序读取器的最大数量超过此值时，将尝试磁盘溢写。

**为什么重要**: 防止合并排序时因为打开过多的 sorted run 读取器而导致 OOM。当 sorted run 数量超过此阈值时，会将部分数据溢写到磁盘。

### 2.2 MemoryOwner 和内存分配机制

#### 源码分析: `MemoryOwner.java` 和 `MemoryPoolFactory.java`

`MemoryOwner` 接口定义了内存使用者的三个核心方法:

```java
public interface MemoryOwner {
    void setMemoryPool(MemorySegmentPool memoryPool);  // 设置内存池
    long memoryOccupancy();                             // 当前内存占用量
    void flushMemory() throws Exception;                // 释放内存 (溢写到磁盘)
}
```

**内存抢占机制** (`MemoryPoolFactory.java` 第 73-93 行):

当某个 `MemoryOwner` 需要分配内存页但内存池已满时:
1. 遍历所有 `MemoryOwner`，找到占用内存最大的 (排除请求者自身)
2. 调用最大内存占用者的 `flushMemory()` 方法，将其数据溢写到磁盘
3. 释放的内存可供当前请求者使用

```java
private void preemptMemory(MemoryOwner owner) {
    long maxMemory = 0;
    MemoryOwner max = null;
    for (MemoryOwner other : owners) {
        if (other != owner && other.memoryOccupancy() > maxMemory) {
            maxMemory = other.memoryOccupancy();
            max = other;
        }
    }
    if (max != null) {
        max.flushMemory();  // 强制最大占用者溢写
        ++bufferPreemptCount;
    }
}
```

**为什么这么设计**: 多个 bucket 的 writer 共享同一个内存池，通过抢占机制实现动态平衡，避免热点 bucket 占满内存导致其他 bucket 无法写入。

### 2.3 磁盘溢写机制

#### `write-buffer-spillable`

- **源码位置**: `CoreOptions.java:672-676`
- **默认值**: `true`
- **含义**: write buffer 是否可以溢写到磁盘。

#### `write-buffer-spill.max-disk-size`

- **源码位置**: `CoreOptions.java:665-670`
- **默认值**: 无限制 (`MemorySize.MAX_VALUE`)
- **含义**: 溢写到磁盘的最大容量限制。

#### `sort-spill-buffer-size`

- **源码位置**: `CoreOptions.java:614-618`
- **默认值**: `64 MB`
- **含义**: 溢写排序时每次写入磁盘的数据量。

#### `spill-compression`

- **源码位置**: `CoreOptions.java:620-625`
- **默认值**: `zstd`
- **含义**: 溢写数据的压缩算法。

**为什么需要溢写**: 当 write-buffer 内存不足且无法通过抢占获取更多内存时，溢写机制可以将内存中的排序数据临时写入磁盘，防止 OOM。后续合并时再从磁盘读回。

#### `write-max-writers-to-spill`

- **源码位置**: `CoreOptions.java:685-690`
- **默认值**: `10`
- **含义**: 批量追加写入时，如果 writer 数量超过此值，自动开启 buffer cache 和溢写功能。

**为什么**: 大量分区的批写入场景下，每个分区一个 writer，writer 过多会导致内存不足。

### 2.4 内存调优建议

#### 高内存场景 (充足的 TaskManager 内存)

```properties
write-buffer-size = 512mb
page-size = 128kb
write-buffer-spillable = false
```

**为什么**: 大 write buffer 减少 flush 频率，关闭溢写避免磁盘 I/O，适合内存充足且 bucket 数量有限的场景。

**好处**: 减少 I/O，降低 Compaction 压力。

#### 内存紧张场景 (多分区/多 bucket)

```properties
write-buffer-size = 64mb
page-size = 32kb
write-buffer-spillable = true
write-buffer-spill.max-disk-size = 10gb
spill-compression = zstd
spill-compression.zstd-level = 1
write-max-writers-to-spill = 5
```

**为什么**: 缩小 write buffer 减少总内存需求，开启溢写防止 OOM，使用 zstd 压缩减少磁盘占用。

**好处**: 在有限内存下支持更多并发 writer，避免 OOM 导致任务失败。

#### 监控关键指标

通过 `WriterBufferMetric` (源码: `WriterBufferMetric.java`) 提供的指标进行监控:
- `numWriters`: 当前 writer 数量
- `bufferPreemptCount`: 内存抢占次数 (高值表示内存不足)
- `usedWriteBufferSizeByte`: 已使用的 write buffer 大小
- `totalWriteBufferSizeByte`: 总 write buffer 大小

**告警阈值建议**: 如果 `bufferPreemptCount` 持续快速增长，说明内存严重不足，应增加 `write-buffer-size` 或增加 TaskManager 内存。

### 3.2 ExpireSnapshots 实现原理

**核心源码**: `ExpireSnapshotsImpl.java` 第 92-152 行

过期算法按以下步骤计算需要过期的快照范围:

```
1. 获取最新快照 ID (latestSnapshotId) 和最早快照 ID (earliest)
2. 计算 min = max(latestSnapshotId - retainMax + 1, earliest)
   -- 由 snapshot.num-retained.max 保护
3. 计算 maxExclusive = latestSnapshotId - retainMin + 1
   -- 由 snapshot.num-retained.min 保护
4. maxExclusive = min(maxExclusive, consumerManager.minNextSnapshot())
   -- 由 Consumer 保护 (除非 consumer.changelog-only=true)
5. maxExclusive = min(maxExclusive, earliest + maxDeletes)
   -- 由 snapshot.expire.limit 保护，防止一次过期太多
6. 从 min 遍历到 maxExclusive，检查 snapshot.time-retained
   -- 如果快照时间在保留时间内，提前退出
```

**过期执行流程** (`innerExpireUntil()` 第 163-278 行):

```
1. 收集所有待过期的快照 (并行读取)
2. 获取 tag 对应的快照 (用于创建数据文件跳过器)
3. 删除合并树文件 (merge tree files)
4. 删除 changelog 文件 (如果未解耦)
5. 清理空目录
6. 删除 manifest 文件
7. 删除 snapshot 文件本身
8. 更新 earliest hint 文件
```

### 3.3 快照过期触发时机

快照过期在每次 commit 后自动触发 (除非 `write-only = true`)。

#### `snapshot.expire.execution-mode`

- **源码位置**: `CoreOptions.java:536-540`
- **可选值**:
  - `sync` (默认): 同步执行。如果文件太多可能耗时较长，阻塞流处理。
  - `async`: 异步执行。如果快照生成速度超过删除速度，会造成文件积压。

#### `snapshot.expire.limit`

- **源码位置**: `CoreOptions.java:542-547`
- **默认值**: `50`
- **含义**: 单次允许过期的最大快照数量。

**为什么需要限制**: 一次过期太多快照会产生大量文件删除操作，可能导致 checkpoint 超时或影响写入延迟。

### 3.4 Changelog 保留策略

#### `changelog.num-retained.min/max`

- **源码位置**: `CoreOptions.java:516-528`
- **默认值**: 无 (不单独控制 changelog)

#### `changelog.time-retained`

- **源码位置**: `CoreOptions.java:530-534`
- **默认值**: 无

**设计说明**: 当配置了 changelog 保留策略时，Changelog 可以与 Snapshot 生命周期解耦 (`isChangelogDecoupled()`)。这意味着即使 Snapshot 被过期，对应的 Changelog 仍然可以独立保留，用于流式消费。

### 3.5 快照管理最佳实践

#### 流式写入场景

```properties
# 保留足够的快照用于故障恢复和回溯
snapshot.num-retained.min = 20
snapshot.num-retained.max = 200
snapshot.time-retained = 2h
# 限制单次过期量，避免 checkpoint 超时
snapshot.expire.limit = 30
# 高吞吐场景使用异步过期
snapshot.expire.execution-mode = async
```

**为什么**: 保留 2 小时快照用于故障恢复，每次最多过期 30 个快照避免阻塞。

#### 需要长时间流式消费的场景

```properties
snapshot.num-retained.min = 10
snapshot.time-retained = 1h
# 解耦 Changelog 生命周期
changelog-producer = lookup
changelog.time-retained = 24h
changelog.num-retained.min = 100
```

**为什么**: Snapshot 短期保留节省存储，Changelog 独立保留 24 小时满足下游流式消费需求。

**好处**: 存储成本可控 (Snapshot 对应的数据文件可以更快释放)，同时下游消费者有足够的回溯窗口。

## 4. 分区管理

### 4.1 PartitionExpire 实现原理

**核心源码**: `PartitionExpire.java`

#### 过期检查逻辑 (第 140-149 行):

```java
List<Map<String, String>> expire(LocalDateTime now, long commitIdentifier) {
    if (checkInterval.isZero()
            || now.isAfter(lastCheck.plus(checkInterval))
            || (endInputCheckPartitionExpire && Long.MAX_VALUE == commitIdentifier)) {
        List<Map<String, String>> expired =
                doExpire(now.minus(expirationTime), commitIdentifier);
        lastCheck = now;
        return expired;
    }
    return null;
}
```

三种触发条件:
1. `checkInterval` 为零: 每次 commit 都检查
2. 距上次检查超过 `checkInterval`: 定时触发
3. 流任务结束时 (`endInputCheckPartitionExpire && commitIdentifier == Long.MAX_VALUE`)

#### 随机化初始检查时间 (第 82-87 行):

```java
long rndSeconds = 0;
long checkIntervalSeconds = checkInterval.toMillis() / 1000;
if (checkIntervalSeconds > 0) {
    rndSeconds = ThreadLocalRandom.current().nextLong(checkIntervalSeconds);
}
this.lastCheck = LocalDateTime.now().minusSeconds(rndSeconds);
```

**为什么**: 避免所有任务在同一时刻执行分区过期检查，分散 I/O 压力。

#### 过期策略

- **`values-time`** (默认): 从分区值中提取时间，与当前时间比较。通过 `PartitionValuesTimeExpireStrategy` 实现。
- **`update-time`**: 比较分区最后更新时间与当前时间。

### 4.2 分区过期配置

#### `partition.expiration-time`

- **源码位置**: `CoreOptions.java:1141-1148`
- **默认值**: 无 (不配置则不启用分区过期)

## 5. Bucket 选择策略

### 5.1 Bucket 模式源码分析

**核心源码**: `KeyValueFileStore.bucketMode()` (第 91-101 行):

```java
public BucketMode bucketMode() {
    int bucket = options.bucket();
    switch (bucket) {
        case -2:
            return BucketMode.POSTPONE_MODE;
        case -1:
            return crossPartitionUpdate ? BucketMode.KEY_DYNAMIC : BucketMode.HASH_DYNAMIC;
        default:
            return BucketMode.HASH_FIXED;
    }
}
```

三种模式:
1. **Fixed Bucket** (`bucket > 0`): 固定桶数，通过 hash 分配。适合已知数据规模的场景。
2. **Dynamic Bucket** (`bucket = -1`): 动态桶模式。
   - `HASH_DYNAMIC`: 无跨分区更新时使用
   - `KEY_DYNAMIC`: 有跨分区更新时使用
3. **Postpone Bucket** (`bucket = -2`): 延迟桶分配模式。

### 5.2 固定 Bucket 数量的影响

#### Bucket 过少的影响

1. **写入瓶颈**: 每个 bucket 对应一个 writer，bucket 少意味着并行度低
2. **文件过大**: 单个 bucket 积累过多数据，Compaction 耗时长
3. **读取延迟**: 查询集中在少数 bucket，无法充分利用并行度

#### Bucket 过多的影响

1. **小文件问题**: 每个 bucket 独立产生 level0 文件，bucket 过多导致大量小文件
2. **内存消耗**: 每个 bucket 需要独立的 write buffer，bucket 多则总内存需求大
3. **Compaction 压力**: 更多的 bucket 意味着更多的独立 Compaction 任务
4. **元数据开销**: manifest 中需要记录更多文件信息

### 5.3 Dynamic Bucket 配置

#### `dynamic-bucket.target-row-num`

- **源码位置**: `CoreOptions.java:1411-1417`
- **默认值**: `2,000,000`
- **含义**: 动态桶模式下每个桶的目标行数。

**为什么有用**: 动态桶根据数据量自动增减桶数，target-row-num 控制桶的粒度。较大的值意味着更少的桶和更大的文件，较小的值意味着更多的桶和更好的查询并行度。

### 5.4 Bucket 数量选择公式和建议

#### 固定 Bucket 估算公式

```
bucket_num = ceil(daily_data_size_per_partition / target_file_size / num_sorted_run_compaction_trigger)
```

其中:
- `daily_data_size_per_partition`: 每个分区每天写入的数据量
- `target_file_size`: 目标文件大小 (默认 128 MB)，源码 `CoreOptions.java:713-723`
- `num_sorted_run_compaction_trigger`: Compaction 触发阈值 (默认 5)

#### 举例

- 每天每分区写入 100 GB 数据
- target-file-size = 128 MB
- num-sorted-run.compaction-trigger = 5
- bucket_num = ceil(100 * 1024 / 128 / 5) = ceil(160) = 160

#### 实践建议

| 分区每日数据量 | 建议 Bucket 数 | 说明 |
|-------------|-------------|------|
| < 1 GB | 1-2 | 数据量小，避免小文件 |
| 1-10 GB | 4-8 | 保证基本并行度 |
| 10-100 GB | 16-64 | 平衡并行度和文件大小 |
| 100 GB-1 TB | 64-256 | 高并行度写入 |
| > 1 TB | 使用 Dynamic Bucket | 数据量大且不确定，使用动态桶 |

**好处**: 合理的 bucket 数量可以在写入吞吐、Compaction 效率、查询并行度之间取得平衡。

## 6. 文件格式与压缩

### 6.1 文件格式选择

**源码位置**: `CoreOptions.java:285-297`

Paimon 支持的文件格式:

| 格式 | 常量 | 适用场景 |
|------|------|---------|
| Parquet | `FILE_FORMAT_PARQUET` (默认) | 列式查询优化，适合 OLAP 场景 |
| ORC | `FILE_FORMAT_ORC` | 列式存储，Hive 生态兼容性好 |
| Avro | `FILE_FORMAT_AVRO` | 行式存储，适合 schema 复杂或全行读取 |
| CSV/Text/JSON | - | 简单数据交换，不推荐生产使用 |

#### 按层级指定格式

- **源码位置**: `CoreOptions.java:307-316`
- **配置示例**: `'file.format.per.level' = '0:avro,3:parquet'`

**为什么**: Level0 文件通常较小且需要快速写入，使用 Avro (行式) 可以减少写入延迟；高层级文件较大且偏读取优化，使用 Parquet (列式) 提升查询性能。

### 6.2 压缩算法配置

#### `file.compression`

- **源码位置**: `CoreOptions.java:318-323`
- **默认值**: `zstd`
- **说明**: `"For faster read and write, it is recommended to use zstd."`

#### `file.compression.zstd-level`

- **源码位置**: `CoreOptions.java:325-330`
- **默认值**: `1`
- **说明**: `"For higher compression rates, it can be configured to 9, but the read and write speed will significantly decrease."`

#### 按层级指定压缩

- **源码位置**: `CoreOptions.java:299-305`
- **配置示例**: `'file.compression.per.level' = '0:lz4,1:zstd'`

**为什么**: Level0 文件频繁写入和合并，使用 lz4 (高速低压缩) 减少写放大；高层级文件稳定存储时间长，使用 zstd (高压缩比) 节省存储空间。

#### 压缩算法对比

| 算法 | 压缩比 | 压缩速度 | 解压速度 | 推荐场景 |
|------|-------|---------|---------|---------|
| zstd (level=1) | 高 | 快 | 快 | 默认推荐，综合最优 |
| zstd (level=9) | 很高 | 慢 | 快 | 存储成本敏感，可牺牲写入速度 |
| lz4 | 中 | 很快 | 很快 | 写入密集，延迟敏感 |
| snappy | 中 | 快 | 快 | 与 Hadoop 生态兼容 |
| zlib | 高 | 慢 | 中 | 冷数据归档 |

### 6.3 target-file-size 的影响

**源码位置**: `CoreOptions.java:713-723` 和 `targetFileSize()` 方法第 2895-2898 行:

```java
public long targetFileSize(boolean hasPrimaryKey) {
    return options.getOptional(TARGET_FILE_SIZE)
            .orElse(hasPrimaryKey ? VALUE_128_MB : VALUE_256_MB)
            .getBytes();
}
```

- 主键表默认 128 MB
- 追加表默认 256 MB

**Compaction 文件大小计算** (`compactionFileSize()` 第 2961-2966 行):
```java
public long compactionFileSize(boolean hasPrimaryKey) {
    return targetFileSize(hasPrimaryKey) / 10 * 7;  // target 的 70%
}
```

只有大于 `compactionFileSize` (即 target 的 70%) 的文件才会参与 Compaction。

**为什么这么设计**: 避免刚写入的文件立即被合并 (因为压缩后的文件可能略小于 target)，减少不必要的重复合并。

#### target-file-size 调优建议

| 场景 | 建议值 | 原因 |
|------|-------|------|
| HDFS 存储 | 128-256 MB | 匹配 HDFS block 大小，减少远程读取 |
| 对象存储 (S3/OSS) | 256-512 MB | 减少 API 调用次数和 listing 开销 |
| 快速查询 | 64-128 MB | 更小的文件实现更细粒度的裁剪 |
| 高吞吐写入 | 256 MB | 减少 Compaction 频率 |

### 6.4 文件块大小

- **源码位置**: `CoreOptions.java:378-383`
- **默认值**: ORC stripe 64 MB，Parquet row group 128 MB

**好处**: 较大的块大小减少元数据开销和 seek 次数，但增加内存占用。根据存储系统特性调整。

## 8. 常见问题排查

### 8.1 写入延迟高的排查路径

```
1. 检查 compactionThreadBusy 指标
   ↓ 如果 > 80%
   说明 Compaction 跟不上写入速度
   → 增大 num-sorted-run.compaction-trigger
   → 切换为 Dedicated Compaction (write-only=true)
   → 增加 Compaction 并行度

2. 检查 bufferPreemptCount
   ↓ 如果快速增长
   说明内存不足，writer 之间频繁抢占
   → 增大 write-buffer-size
   → 减少 bucket 数量 (减少 writer 数量)
   → 增大 TaskManager 内存

3. 检查 maxLevel0FileCount
   ↓ 如果接近 numSortedRunStopTrigger
   说明 Compaction 积压严重，写入被阻塞
   → 增大 num-sorted-run.stop-trigger (临时缓解)
   → 分析 Compaction 为何慢 (文件过大? I/O 瓶颈?)

4. 检查 lastCommitAttempts
   ↓ 如果 > 3
   说明提交冲突频繁
   → 检查是否有多个写入任务写同一表
   → 考虑使用不同的 commit-user-prefix
```

### 8.2 读取慢的排查路径

```
1. 检查 lastScanDuration
   ↓ 如果 > 10s
   说明 manifest 扫描耗时
   → 检查 lastScannedManifests 数量
   → 增大 manifest.full-compaction-threshold-size 触发 manifest 合并
   → 检查 manifest 缓存命中率 (manifestHitCache / total)

2. 检查 lastScanResultedTableFiles
   ↓ 如果过多 (比如 > 1000)
   → 检查分区裁剪是否生效 (WHERE 条件是否包含分区列)
   → 检查 file index 是否配置
   → 检查 Compaction 是否正常，文件数量是否过多

3. 检查 maxLevel0FileCount
   ↓ 如果偏高
   → Level0 文件需要全部扫描且 key range 重叠
   → 增强 Compaction (减小 compaction-trigger)
   → 考虑使用 compaction.optimization-interval 定期全量合并

4. 检查数据倾斜
   → 对比 maxTotalFileSize 和 avgTotalFileSize
   → 如果差异大说明某些 bucket 数据量远超其他
   → 重新评估 bucket-key 和 bucket 数量
```

### 8.3 Compaction 积压的排查

```
1. 检查 compactionQueuedCount
   ↓ 如果持续 > 0
   说明 Compaction 线程不够

2. 检查 avgCompactionTime
   ↓ 如果持续增长
   可能原因:
   a. target-file-size 过大导致单次合并数据量过多
   b. 空间放大触发全量合并 (maxSizeAmp 过低)
   c. 磁盘/网络 I/O 瓶颈

3. 检查 compactionInputSize / compactionOutputSize 比率
   → 如果输入远大于输出，说明合并效率好 (大量重复 key 被合并)
   → 如果接近 1:1，说明数据重复率低，合并属于"白做功"
   → 考虑增大 compaction-trigger 减少不必要的合并

4. 解决方案优先级:
   a. 使用 Dedicated Compaction 分离写入和合并资源
   b. 配置 off-peak hours 在低峰期更积极合并
   c. 增大 Compaction 线程池
   d. 调整触发参数减少合并频率
```

### 8.4 OOM 场景和解决方案

#### 场景 1: 写入 OOM

**原因**: 大量 bucket 的 writer 同时持有 sort buffer

**源码依据**: 每个 bucket 的 writer 需要 `SortBufferWriteBuffer`，其内存由 `MemoryPoolFactory` 统一管理。但如果 bucket 数量 * 实际每个 buffer 使用内存 > 总内存，即使有抢占机制也无法完全避免 OOM。

**解决方案**:
```properties
write-buffer-size = 64mb          # 减小单个 buffer
write-buffer-spillable = true      # 开启溢写
write-buffer-spill.max-disk-size = 5gb
write-max-writers-to-spill = 5     # 超过 5 个 writer 就开启溢写
```

#### 场景 2: Compaction OOM

**原因**: 合并大量文件时需要同时打开所有输入文件的 reader

**源码依据**: `sort-spill-threshold` 控制合并读取器的上限 (默认 `numSortedRunStopTrigger + 1`)。

**解决方案**:
```properties
sort-spill-threshold = 5            # 限制同时打开的 reader 数量
sort-spill-buffer-size = 32mb       # 减小溢写 buffer
local-sort.max-num-file-handles = 64  # 限制外部排序的 fan-in
```

#### 场景 3: 扫描 Compaction 文件 OOM (Append 表)

**原因**: 无感知 bucket 的 Append 表 Compaction 时扫描文件过多

**源码依据**: `compaction.file-num-limit` (默认 100,000)。

**解决方案**:
```properties
compaction.file-num-limit = 10000   # 限制扫描文件数
```

#### 场景 4: 分区过期 OOM

**原因**: 一次删除过多分区

**源码依据**: `partition.expiration-batch-size` 控制分批删除。

**解决方案**:
```properties
partition.expiration-max-num = 50
partition.expiration-batch-size = 10
```

```properties
# -- Compaction (Append 表参数) --
compaction.min.file-num = 5
compaction.delete-ratio-threshold = 0.2

# -- 快照 --
snapshot.num-retained.min = 5
snapshot.time-retained = 30min

# -- 分区 --
partition.expiration-time = 30d
```

### 9.2 不同工作负载的配置策略

#### CDC 实时入湖

```properties
changelog-producer = lookup
# 或者 changelog-producer = full-compaction 配合 full-compaction.delta-commits = 3
write-only = true
num-sorted-run.compaction-trigger = 5
# 对于有大量更新的表，增加 write buffer 提高去重效率
write-buffer-size = 512mb
```

**为什么**: CDC 场景同一 key 可能频繁更新，大 write buffer 可以在内存中合并更多更新，减少磁盘 I/O 和 Compaction 负担。

#### 宽表聚合

```properties
merge-engine = aggregation
# 或 merge-engine = partial-update
num-sorted-run.compaction-trigger = 3
compaction.max-size-amplification-percent = 100
```

**为什么**: 聚合表需要频繁合并来保证查询结果准确性，更激进的 Compaction 策略能让查询看到最新的聚合结果。

#### 维表 (Lookup Join)

```properties
merge-engine = deduplicate
# 或 first-row
bucket = 固定值，根据数据量计算
# 查询加速
file-index.bloom-filter.columns = primary_key_column
lookup-cache-max-memory-size = 256mb
```

**为什么**: 维表主要用于点查 (lookup join)，bloom filter 和 lookup cache 能显著减少不必要的文件读取。

### 9.3 Tag 和 Branch 的运维使用场景

#### Tag 的自动创建

**源码位置**: `CoreOptions.java:1682-1687`

```properties
# 自动按天创建 Tag
tag.automatic-creation = watermark
tag.creation-period = daily
tag.num-retained-max = 30
tag.default-time-retained = 30d
tag.creation-delay = 10min
```

**使用场景**:
1. **时间旅行查询**: 通过 Tag 查询历史某一时刻的数据快照
2. **批处理回溯**: 批任务从指定 Tag 读取数据
3. **数据审计**: 保留关键时间点的数据状态

**为什么 `tag.creation-delay = 10min`**: 允许 10 分钟的延迟数据进入当前 Tag 对应的快照，避免因为少量延迟数据导致跨 Tag 查询。

#### Tag 保留策略

- `tag.num-retained-max`: 最多保留的自动创建 Tag 数量 (源码: `CoreOptions.java:1722-1727`)
- `tag.default-time-retained`: Tag 的默认保留时间 (源码: `CoreOptions.java:1729-1735`)
- `tag.time-expire-enabled`: 是否开启按时间过期 (默认 true) (源码: `CoreOptions.java:1737-1741`)

#### Branch 的运维场景

1. **A/B 测试**: 在 Branch 上修改 Schema 或写入测试数据，不影响主分支
2. **数据修复**: 在 Branch 上执行数据修复操作，验证后再合并
3. **Schema 演进测试**: 在 Branch 上测试新的 Schema 变更

**实践建议**:
- 使用 `scan.fallback-snapshot-branch` 配置链式读取，主分支没有的分区自动从指定 Branch 读取
- Branch 上的独立 Compaction 不影响主分支
- 及时清理不再使用的 Branch，避免存储浪费
