# Apache Paimon 时间旅行与版本管理深度分析

> **版本**：1.5-SNAPSHOT　**源码模块**：`paimon-core`（`Snapshot` / `CommitKind` / `ExpireConfig` 定义在 `paimon-api`）　**核对日期**：2026-06

**一句话定位**：Paimon 把"表在任意时刻的完整状态"抽象成一条**单调递增的 Snapshot 链**，并在其上叠加 **Tag（命名长期保留）、Branch（独立演进空间）、Consumer（流式消费位点）、Rollback（物理回退）、Expire（带多重保护的过期）** 五套机制，让一份对象存储上的湖表同时具备时间旅行、版本固化、实验隔离、断点续读和自动清理能力。

## 1. 快速理解（核心问题 / 概念 / 陷阱）

### 1.1 核心问题：一条 Snapshot 链如何撑起五套版本能力

**① 要解决什么问题**

实时湖表跑在 S3/OSS/HDFS 上，对象存储只能顺序追加、不支持原地改写，却要同时满足一堆互相冲突的需求：误操作要能秒级回退、监管要能查"昨晚 8 点的状态"、流任务要能断点续读、长期基准点要能跨年保留、并发读写要互不脏读、存储又不能无限膨胀。如果没有统一的版本抽象，每个需求都得单独造轮子（备份副本、外部 offset 存储、手工删文件），既贵又危险。

**② 设计原理与取舍**

Paimon 的答案是一条 **单调递增、内容自包含的 Snapshot 链**：每次提交原子写出一个 `snapshot-{id}` JSON 文件，文件本身不含数据，只引用 ManifestList（详见 `01-核心存储引擎分析.md §12.2`）。其余五套能力都是这条链的"叠加层"：

| 能力 | 本质 | 与 Snapshot 链的关系 |
|------|------|---------------------|
| 时间旅行 | 按 id / 时间 / tag / watermark 定位某个 Snapshot | 在链上"查找"，不改链 |
| Tag | 对某个 Snapshot 的命名长期引用 | 把链上一个点"钉住"不被过期 |
| Branch | 独立的 snapshot/schema/tag/consumer 目录 | 复制一个起点，另起一条链 |
| Consumer | 持久化"下一个待读 Snapshot id" | 在链上记一个游标 |
| Rollback | 删除目标之后的所有 Snapshot 元数据 | **物理截断**这条链 |
| Expire | 删除过早的 Snapshot + 不再引用的数据文件 | **物理修剪**链头 |

一句话设计哲学：**所有版本能力都退化为"在一条不可变追加的链上做查找 / 钉住 / 分叉 / 截断 / 修剪"，把分布式版本管理的复杂度压进单调 id + 自包含 JSON 这两个简单约定里。**

**Snapshot ID 为什么单调递增、文件为什么是 JSON**：单调 id 让时间旅行的"按版本号定位"O(1)、"按时间二分"O(log N)，且由提交协调者单点生成、天然全局有序，规避了分布式时钟偏移；JSON 自包含让运维能 `cat snapshot-42` 直接读、新增字段时旧读取器靠 `@JsonIgnoreProperties(ignoreUnknown=true)` 忽略——这也是 Tag 能"继承 Snapshot 格式"的前提（§3.1）。代价是每次提交多写几 KB 元数据，相对数据量可忽略。

### 1.2 核心概念速查表

### 1.3 高频生产陷阱

**陷阱 1：假设 Snapshot ID 从 1 连续。** id 单调递增但**不连续**——过期会从链头删除。误以为"从 1 遍历到 latest"会撞 `FileNotFoundException`。正确做法：以 `earliestSnapshotId()`/`latestSnapshotId()` 为边界（`SnapshotManager.java:259/191`）；`earliestSnapshot()` 本身带"最早快照刚好被别的 job 过期"的重试（`:229`）。

**陷阱 2：以为 `num-retained.max` 和 `time-retained` 是"或"关系。** 实际是**取更严格者**：过期上界被 retainMax、retainMin、Consumer、maxDeletes 逐级 `Math.min` 收紧，再被 timeRetain 提前终止（§8.2）。配 `max=100` + `time-retained=1h`，1 小时内涌入 200 个快照，最终也只保留约 100 个。

**陷阱 3：把 ExpireConfig 内部默认值当成用户侧默认值。** 用户侧（`CoreOptions`）默认：`snapshot.num-retained.min=10`、`max=Integer.MAX_VALUE`（无限）、`time-retained=1h`、`snapshot.expire.limit=50`（`CoreOptions.java:495/503/510/542`）。`ExpireConfig` 内部 Builder 的裸默认是 `min=1`、`max=MAX`、`timeRetain=∞`、`maxDeletes=MAX`（`ExpireConfig.java:107-110`），仅在未走 CoreOptions 构造时生效。**（已修正：旧稿把 `retainMin` 默认写成 10、`maxDeletes` 写成 50 挂在 ExpireConfig 名下，混淆了两层默认值。）**

**陷阱 4：以为创建 Tag 后就能随便删 Snapshot 文件。** Tag 只在**走 ExpireSnapshots 正规过期**时保护数据文件（§8.3），手工 `rm snapshot-*` 绕过过期，Tag 引用的数据文件仍会被后续过期当成无引用清掉。

**陷阱 5：回滚后不重置 Consumer。** 回滚物理删除目标之后的 Snapshot，Consumer 的 `nextSnapshot` 可能指向已不存在的 id，流任务重启即失败（§5.3）。回滚前先建 Tag 当安全网，回滚后用 `resetConsumer` 或从 Flink savepoint 恢复。

**陷阱 6：`consumer-changelog-only=true` 误用。** 开启后 Consumer **不再保护 Snapshot**、只保护 Changelog（`ExpireSnapshotsImpl.java:134`）。仅适合"纯消费 Changelog"的下游；若任务还要读完整快照（如冷启动全量），其依赖的 Snapshot 会被照常过期导致失败。

**陷阱 7：FastForward 的破坏性。** FastForward 会删除 main 上 `>= 分支最早 SnapshotId` 的所有 snapshot/schema/tag（`FileSystemBranchManager.java:184-189`），不可逆。分支创建后若 main 又有重要提交，这些提交会被覆盖丢失。

**陷阱 8：自动 Tag 的 `watermark` vs `process-time` 选错。** 流表用 `process-time` 会让 Tag 名（按处理时间）与数据事件时间错位几小时；流场景应选 `watermark`（`TagTimeExtractor`），并注意流空闲时靠 `idlenessTimeout` 兜底创建（`TagAutoCreation.java:113-133`）。

## 2. Snapshot 链与时间旅行

### 2.1 Snapshot 数据模型与双清单

**① 要解决什么问题**：要在不可变追加的对象存储上提供"某时刻完整状态"的入口，同时让创建快照足够便宜（不能拷数据）、让全量读和增量读都高效。

**② 设计原理与取舍**：Snapshot 本身不含数据，只通过 ManifestList 间接引用数据文件，于是"提交一个快照 = 写一个几 KB 的 JSON"。关键是**双清单（base + delta）**：`baseManifestList` 是到本快照为止的全量文件引用（供全表扫描），`deltaManifestList` 只是本次提交的增量（供流式增量读和快速过期）。早期只有单清单的湖格式，增量读要 diff 前后两个快照的 manifest（O(N)）；Paimon 提交时就把增量算好，增量读退化为"直接读 delta 清单"（O(1)）。代价是每次提交多写 2~3 个清单文件。ManifestList 三层索引的细节由 `01-核心存储引擎分析.md §12.2/§12.3` 主讲，本文只用到"Snapshot → ManifestList"这一层。

**③ 关键源码**（`Snapshot.java`，paimon-api）：核心字段是 `id`（单调递增）、`schemaId`（引用的 schema 版本，§6 回滚要用）、`baseManifestList` / `deltaManifestList` / `changelogManifestList`（三类清单，后者可空，供 CDC）、`commitKind`、`timeMillis`（提交时间，时间旅行二分要用）、`watermark`（可空，watermark 二分与自动 Tag 要用）。`CommitKind` 枚举（`Snapshot.java:454`，已修正：旧稿写"454-470"，枚举体在 454 起）：

| CommitKind | 含义 | 对流式增量的影响 |
|------------|------|-----------------|
| `APPEND` | 追加新数据文件 | `DeltaFollowUpScanner` 唯一会扫的类型（§7.3） |
| `COMPACT` | Compaction 重组文件 | 不产生新数据，DELTA 模式跳过 |
| `OVERWRITE` | 覆盖/删除现有文件 | 流式需特殊处理受影响分区 |
| `ANALYZE` | 统计信息收集 | 不影响数据 |

**④ 风险 / 边界**：`changelogManifestList`、`watermark`、`nextRowId`、`statistics` 等都是可空字段——读取早期版本写出的快照时这些可能为 null，watermark 二分（§2.3）正是为此做了"跳过无 watermark 快照"的处理。

**⑤ 收益与代价**：自包含 JSON + 双清单 = 原子提交、零拷贝快照、O(1) 增量读；代价是元数据文件随提交线性累积，必须靠 §8 的过期机制修剪。

### 2.2 SnapshotManager：读取、缓存与重试

**① 要解决什么问题**：在对象存储上稳定地"按 id 拿到一个 Snapshot"——既要快（高频访问要缓存），又要稳（对象存储有最终一致性窗口），还要能支持分支（每个分支一套独立 snapshot 目录）和 REST Catalog（快照可能存在服务端）。

**② 设计原理与取舍**：`SnapshotManager`（`SnapshotManager.java:60`）构造时接收 `branch`，规范化后拼进路径 `{branchPath}/snapshot/snapshot-{id}`（`snapshotPath` 在 `:109`，已修正：旧稿标"第 84 行"指的是 `normalizeBranch` 调用处）。它持有一个可空的 Caffeine `Cache<Path, Snapshot>`（热点快照命中内存）和一个可空的 `SnapshotLoader`（REST Catalog 用，REST 抛 `UnsupportedOperationException` 时回退文件系统，见 `latestSnapshot()` 在 `:167`）。

**③ 关键源码**：对象存储最终一致性的处理在 `tryFromPath`（`SnapshotManager.java:760`，已修正：旧稿标 761-787）——读 JSON 解析失败（非 `FileNotFoundException`）时最多重试 10 次、每次 sleep 200ms：

```java
// SnapshotManager.tryFromPath（精简）
while (retryNumber++ < 10) {
    content = fileIO.readFileUtf8(path);   // FileNotFoundException 直接抛出，不重试
    try { return Snapshot.fromJson(content); }
    catch (Exception e) { exception = e; Thread.sleep(200); }
}
throw new RuntimeException("Retry fail after 10 times", exception);
```

常用方法行号（本次核对）：`snapshot(long)` 带缓存 `:124`、`tryGetSnapshot(long)` 抛 FNF `:136`、`latestSnapshot()` `:167`、`latestSnapshotId()` `:191`、`earliestSnapshot()` 带过期重试 `:213/229`、`earliestSnapshotId()` `:259`、`deleteSnapshot(long)` `:159`。

**④ 风险 / 边界**：重试只挡"写后读不一致"，挡不住"写后立刻从另一节点读"——自动化脚本紧跟写操作做读取时建议加 1~2s 延迟。`FileNotFoundException` 故意不重试（文件确实不存在就该快速失败）。

**⑤ 收益与代价**：缓存 + hint（§2.5）让稳态下定位快照只需 1~2 次 read；重试换来对象存储上的稳定性，代价是异常路径下最坏阻塞 2s。

### 2.3 时间旅行的两种定位：版本号直取与时间戳/watermark 二分

**① 要解决什么问题**：用户想读"历史某状态"，但表达方式各异——有人知道精确 Snapshot id，有人只知道时间点，有人按事件时间（watermark）定位。底层要把这些都映射到链上的某个 Snapshot。

**② 设计原理与取舍**：分两类。**版本号直取**（snapshot-id / tag）是 O(1)：id 直接 `snapshot(id)`，tag 经 `TagManager` 取出再 `trimToSnapshot()`。**时间定位**（timestamp / watermark）靠二分：因为 `timeMillis` 随 id 严格递增，天然有序，O(N) 降到 O(log N)，每次只读一个 JSON。这是"单调 id + 时间随 id 递增"这条不变式的直接红利。

**③ 关键源码**：`earlierOrEqualTimeMills` 在 `SnapshotManager.java:291`（标准二分，`mid = earliest + (latest-earliest)/2` 防溢出），对称的 `laterOrEqualTimeMills` 在 `:325`。**watermark 二分更复杂**（`earlierOrEqualWatermark` 在 `:354`），原因是 watermark 字段可空：

- 先判 `latest` 的 watermark 是否为 `Long.MIN_VALUE`（无 watermark），是则直接返回 null，省一次 IO；
- 从 earliest 起线性扫描，找到**第一个有 watermark 的快照**作为二分起点（早期快照可能 watermark 为 null）；
- 二分过程中遇到 watermark 为 null 的 mid，还要向左回退找最近一个有 watermark 的快照。

**④ 风险 / 边界**：① `scan.timestamp` 是 UTC 毫秒，传本地时区会定位到错误快照；② 二分依赖"时间随 id 递增"，若系统时钟回拨导致某快照 `timeMillis` 倒挂，二分结果可能不符直觉；③ watermark 模式下若全表无 watermark（如纯批写入），返回 null。

**⑤ 收益与代价**：O(log N) 定位让百万级快照表的时间旅行仍是毫秒级；代价是 watermark 路径实现复杂、对空值敏感。

### 2.4 统一入口 TimeTravelUtil 与四种 StartingScanner

**① 要解决什么问题**：让用户用一个 `scan.version` 就能时间旅行，不必记 4 个不同的 scan key。

**② 设计原理与取舍**：`TimeTravelUtil`（`table/source/snapshot/TimeTravelUtil.java:54`）维护一组 `SCAN_KEYS`（snapshot-id / tag-name / watermark / timestamp / timestamp-millis），`adaptScanVersion`（`:149`）把通用 `scan.version` 自适应路由：是已存在的 Tag 名 → tag-name；以 `watermark-` 前缀开头 → watermark；纯数字 → snapshot-id；否则当 Tag 名（不存在则抛异常）。

|---------|----------|----------|

### 2.5 Hint 文件加速与失效兜底

**① 要解决什么问题**：定位首尾快照（earliest/latest）是最高频操作，但在对象存储上 `list` 一个目录下的全部 `snapshot-*` 文件很贵。

**② 设计原理与取舍**：在 `snapshot/` 目录放两个明文小文件 `EARLIEST` / `LATEST`，内容就是首尾快照 id。读 hint 即得 id，免 list。但 hint 可能因并发提交而过时，所以读 hint **必须带自校验**。

**③ 关键源码**（`HintFileUtils.java:43`，已修正：旧稿标 44-55）：`findLatest` 读到 LATEST hint 后，检查 `snapshot-(hint+1)` 是否存在——存在说明有更新的快照、hint 已过时，回退 `findByListFiles` 全量扫描：

```java
Long snapshotId = readHint(fileIO, LATEST, dir);
if (snapshotId != null && snapshotId > 0
        && !fileIO.exists(file.apply(snapshotId + 1))) {
    return snapshotId;            // 确认没有更新的，hint 有效
}
return findByListFiles(fileIO, Math::max, dir, prefix);  // 回退
```

`findEarliest`（`:57`）对称：hint 指向的文件存在才信，否则回退 list。`readHint` 自身还带 3 次重试（`:69`）。

**④ 风险 / 边界**：自定义代码若直接 `readHint` 而不做"next 是否存在"的校验，会读到陈旧 id。FastForward / 回滚会主动删/改 hint（`deleteLatestHint`、`commitLatestHint`、`commitEarliestHint`，均在 `SnapshotManager.java:732/736/740`），所以这些操作后 hint 状态是受控的。

**⑤ 收益与代价**：稳态下首尾定位是 O(1) read；代价是 hint 失效时退化为一次目录 list，且必须严格走自校验逻辑才安全。

## 3. Tag 机制：命名引用与长期保留

### 3.1 Tag 数据模型：为什么继承 Snapshot

**① 要解决什么问题**：Snapshot 默认 1 小时就被过期（§8），但月末快照、审计基准点这类"长期固定点"必须能跨年保留，且要有人类可读的名字（"回滚到月末 Tag"远比"回滚到 Snapshot 12345"好用）。同时希望 Tag 文件能被老版本读取器、回滚逻辑、Iceberg 兼容层直接当 Snapshot 用。典型场景：金融表每月留一个 Tag、保留 7 年（共 84 个），比配 `snapshot.time-retained=7y` 累积数十万快照省下约 1000 倍元数据。

**② 设计原理与取舍**：Tag 选择 **继承 Snapshot 而非组合**（`Tag.java:44`）：

```java
@JsonIgnoreProperties(ignoreUnknown = true)
public class Tag extends Snapshot {
    @Nullable private final LocalDateTime tagCreateTime;   // 供 TTL 计算
    @Nullable private final Duration tagTimeRetained;      // 单 Tag TTL
}
```

继承换来三个直接收益：① 老版本（<=0.7）读取器靠 `@JsonIgnoreProperties(ignoreUnknown=true)` 忽略两个新字段，仍能把 Tag 当 Snapshot 读；② `trimToSnapshot()` 零成本把 Tag 退回纯 Snapshot，供回滚（§5.1）/分支创建（§4.2）复用；③ Tag 文件能直接反序列化成 Snapshot。代价是 Tag 文件含 Snapshot 全字段（约 1~2KB），比"只存一个 id 的纯引用"重几百字节——但这正是 Tag 能在 Snapshot 过期后独立兜底的根本原因（§5.1 的 Tag 回滚、§8.3 的数据文件保护都依赖 Tag 自带完整信息）。

**对比：Tag 为什么不学 Iceberg 只存 id**

| 维度 | Paimon（继承 Snapshot） | Iceberg（SnapshotRef 只存 id） |
|------|------------------------|-------------------------------|
| Tag 文件内容 | 完整 Snapshot 字段 + 2 个 TTL 字段 | 仅 snapshot-id 引用 |
| Snapshot 过期后 | Tag 仍可独立恢复完整状态 | 引用悬空，需 Snapshot 仍在 |
| TTL / 自动创建 | 内建支持 | 不支持，需外部调度 |
| 体积 | 重（KB 级） | 轻（几十字节） |

**③ 关键源码**：写入时的版本兼容处理在 `createOrReplaceTag`（`TagManager.java:166`，已修正：旧稿标 163-166，那是注释行）——`timeRetained` 为 null 时**不写 `tagCreateTime` 字段**，生成的 JSON 与旧版 Snapshot 完全一致，保证老读取器兼容：

```java
String content = timeRetained != null
    ? Tag.fromSnapshotAndTagTtl(snapshot, timeRetained, LocalDateTime.now()).toJson()
    : snapshot.toJson();   // 无 TTL 时退化为纯 Snapshot JSON
```

**④ 风险 / 边界**：`tagCreateTime`/`tagTimeRetained` 可空——无 TTL 的 Tag 在过期时只能靠"文件修改时间"近似 createTime（§3.3）。

**⑤ 收益与代价**：一个文件同时是"命名引用 + 完整状态备份 + 老版本兼容 + 回滚兜底源"；代价是体积偏大，但 Tag 数量通常很少（几十到几百），总开销可忽略。

### 3.2 TagManager CRUD 与删除的跳过集合

**① 要解决什么问题**：Tag 是命名引用，CRUD 看似简单，但**删除**很危险——Tag 引用的数据文件可能同时被相邻 Tag 或 Snapshot 引用，乱删会损坏其它版本。

**② 设计原理与取舍**：Tag 文件存于 `{branchPath}/tag/tag-{tagName}`，CRUD 都是单文件原子操作（`overwriteFileUtf8` / `rename` / `deleteQuietly`）。删除的安全性靠两道判断 + 一个"跳过集合"：

- **删除三态**（`deleteTag`，`TagManager.java:233`）：① 对应 Snapshot 仍在链中 → 只删 Tag 元文件，数据文件归 Snapshot 管；② Snapshot 已过期但还有别的 Tag 指向同一 Snapshot → 也只删元文件；③ Snapshot 已过期且这是**最后一个**引用它的 Tag → 才真正清理数据文件。
- **跳过集合**（`doClean`，`TagManager.java:277`，已修正：旧稿标 284-297，那是 skip-set 构建区间）：清数据文件前，先把**左邻 Tag** 和**右邻最近快照**（earliest snapshot 与右邻 Tag 取 id 更小者）引用的文件收进 skip 集，只删"本 Tag 独占"的文件。这与 §8.3 过期时的 Tag 保护是同一思路：**相邻引用即保护边界**。

**③ 关键源码**：CRUD 入口与行号——`createTag`（`:128`，名称非空校验 + `ignoreIfExists` 静默返回 + 委托 `createOrReplaceTag`）；`replaceTag`（`:143`，替换时只保留 `IcebergCommitCallback` 类回调，见 `:151`）；`deleteTag`（`:233`）；`renameTag`（`:195`，`fileIO.rename` 原子重命名）；读取 `get`/`getOrThrow`（`:349/359`）、`tags()` 返回 `SortedMap<Snapshot, List<String>>`（`:379`）。`tags()` 用 SortedMap 是因为**一个 Snapshot 可被多个 Tag 指向**，按 Snapshot id 排序便于删除/过期时定位左右邻。

**④ 风险 / 陷阱 / 边界**：
- Tag 名已存在且未设 `ignoreIfExists=true` 会抛异常——自动化脚本必须捕获，否则任务失败。
- "删最后一个 Tag 才清数据"这一级联行为常被误解：多 Tag 指向同一 Snapshot 时删前 N-1 个都不动数据。
- Tag **只在走 ExpireSnapshots 正规过期时保护数据文件**（§8.3）；手工 `rm snapshot-*` 绕过过期，Tag 引用的数据文件会在后续过期被当无引用删掉。
- `createOrReplaceTag` 内会 `validateNoAutoTag`，避免手动 Tag 与自动 Tag 命名撞车。

**⑤ 收益与代价**：跳过集合让"删 Tag"成为只影响本 Tag 独占文件的安全操作；代价是删除前要读相邻 Tag/Snapshot 的 manifest 构建 skip 集，删除有一定 IO 开销。

### 3.3 Tag TTL：双重过期取先到者

**源码路径**: `paimon-core/src/main/java/org/apache/paimon/tag/TagTimeExpire.java`

**为什么需要 Tag TTL**: 自动创建的 Tag 如果不清理会持续累积，占用存储和元数据空间。TTL 机制允许 Tag 在创建后经过指定时间自动过期。

```java
public class TagTimeExpire {
    private LocalDateTime olderThanTime;  // 全局 "older than" 阈值

    public List<String> expire() {
        for (Pair<Tag, String> pair : tagManager.tagObjects()) {
            Tag tag = pair.getLeft();
            LocalDateTime createTime = tag.getTagCreateTime();
            Duration timeRetained = tag.getTagTimeRetained();

            // 如果 Tag 没有 createTime/timeRetained（旧版本 Tag）
            // 但配置了 olderThanTime，则用文件修改时间作为 createTime
            if (createTime == null || timeRetained == null) {
                if (olderThanTime != null) {
                    createTime = DateTimeUtils.toLocalDateTime(
                        fileIO.getFileStatus(tagPath).getModificationTime());
                }
            }

            // 两个过期条件 OR 关系:
            // 1. createTime + timeRetained < now (TTL 到期)
            // 2. olderThanTime > createTime (全局 "比某时间更老")
            boolean isReachTimeRetained = timeRetained != null
                && LocalDateTime.now().isAfter(createTime.plus(timeRetained));
            boolean isOlderThan = olderThanTime != null
                && olderThanTime.isAfter(createTime);
        }
    }
}
```

`expire()` 在 `TagTimeExpire.java:61`（已修正：旧稿把方法标在 552，那是早期行号）。

**取先到者**：`isReachTimeRetained`（单 Tag 自带 TTL 到期）或 `isOlderThan`（全局 `olderThanTime` 阈值）**任一成立**即删——是 OR 关系，所以谁先满足谁生效。无 `createTime`/`timeRetained` 的旧 Tag，在配了全局 `olderThanTime` 时退化为用**文件修改时间**当 createTime。

**④ 风险 / 边界**：只配了全局 TTL 而忽略某个 Tag 自带的短 `timeRetained`，会让该 Tag 比预期更早消失（取先到者）。无 TTL 的老 Tag 仅在配了全局阈值时才会被清理，否则永久保留。

**⑤ 收益与代价**：单 Tag TTL + 全局阈值两层独立控制（月末 Tag 留 7 年、日常 Tag 留 7 天可共存）；代价是双重条件下用户需自己想清"谁先到期"。

### 3.4 自动 Tag 创建（TagAutoCreation）

**源码路径**: `paimon-core/src/main/java/org/apache/paimon/tag/TagAutoCreation.java`

**为什么需要自动 Tag**: 在生产环境中，手动管理 Tag 太繁琐。自动 Tag 按时间周期（小时/天/自定义）创建，实现定期数据保护。

**核心组件协作**:

```
TagAutoManager
  ├── TagAutoCreation  (自动创建)
  │     ├── TagTimeExtractor    (时间提取: 进程时间 / Watermark)
  │     ├── TagPeriodHandler    (周期管理: DAILY / HOURLY / TWO_HOURS / 自定义)
  │     └── TagManager          (实际创建)
  └── TagTimeExpire    (TTL 过期)
```

**TagTimeExtractor 两种模式**（`tag/TagTimeExtractor.java`）:

| 模式 | 配置值 | 时间来源 | 适用场景 |
|------|--------|----------|----------|
| `ProcessTimeExtractor` | `tag.automatic-creation=process-time` | `Snapshot.timeMillis()` | 批处理或不关心事件时间 |
| `WatermarkExtractor` | `tag.automatic-creation=watermark` | `Snapshot.watermark()` | 流处理，基于数据事件时间 |

**TagPeriodHandler 周期策略**（`tag/TagPeriodHandler.java`）:

| 策略 | 周期 | Tag 名称格式 | 示例 |
|------|------|-------------|------|
| `DailyTagPeriodHandler` | 1 天 | `yyyy-MM-dd` | `2024-12-25` |
| `HourlyTagPeriodHandler` | 1 小时 | `yyyy-MM-dd HH` | `2024-12-25 14` |
| `TwoHoursTagPeriodHandler` | 2 小时 | `yyyy-MM-dd HH` | `2024-12-25 14` |
| `PeriodDurationTagPeriodHandler` | 自定义 | `yyyyMMddHHmm` | `202412251430` |

**自动创建流程**（`TagAutoCreation.run()` 第 141-156 行）:

```
1. 从 nextSnapshot 开始遍历已存在的 Snapshot
2. 对每个 Snapshot 调用 timeExtractor.extract() 获取时间
3. 如果时间 >= nextTag (减去 delay)，创建 Tag
4. 如果配置了 numRetainedMax，清理超出数量的旧 Tag
5. 更新 nextTag 和 nextSnapshot，继续循环
```

**关于 automaticCompletion**（`TagAutoCreation.java:173`）: 配 `tag.automatic-completion=true` 后，若应创建 Tag 但时间已跨越多个周期，用 nextTag 作为实际 Tag 时间而非当前数据时间，确保 Tag 序列连续不跳期。`numRetainedMax` 超量清理在 `:183`，且只清自动 Tag（手动 Tag 不计入，故总 Tag 数可能超过 `tag.num-retained-max`）。

**关于 idlenessTimeout**（`TagAutoCreation.java:113`）: watermark 模式下若流空闲（长时间无新数据），watermark 停滞超过 `idlenessTimeout` 会强制推进以生成 Tag，避免因流空闲导致 Tag 长期不出。

**④ 风险 / 边界**：流表错配 `process-time` 会让 Tag 名（处理时间）与数据事件时间错位数小时；`numRetainedMax` 只约束自动 Tag。**⑤ 收益与代价**：自动 Tag 把"定期保护"从人工运维变成配置项；代价是 watermark 模式对流空闲敏感、需要 `idlenessTimeout` 兜底。

## 4. Branch 机制：独立空间与 FastForward

> 本节讲分支机制的源码原理（路径隔离、零拷贝创建、FastForward 覆盖）。分支用于"特征实验隔离 / A-B 测试"的应用场景与实践，详见 `41-分支机制与特征实验隔离.md`。

### 4.1 路径规则：隔离元数据、共享数据文件

**① 要解决什么问题**：要在生产表上做实验（新 ETL、Schema 变更、A/B），又不能影响线上、不想拷贝整张表的数据。

**② 设计原理与取舍**：Branch 是"独立演进路径"——**隔离元数据、共享数据文件**。隔离的是 snapshot / schema / tag / consumer（可变、各分支独立提交回滚建 Tag）；共享的是 data / manifest（不可变，多分支引用同一文件无冲突，零拷贝）。main 分支特殊地映射到表根目录（兼容引入分支前的旧布局），其余分支放在 `branch/branch-{name}/` 下：

```java
static String branchPath(Path tablePath, String branch) {
    return isMainBranch(branch)
        ? tablePath.toString()                                        // main → 表根目录
        : tablePath.toString() + "/branch/" + BRANCH_PREFIX + branch; // 其它分支
}
```

```
table/
├── snapshot/ schema/ tag/ consumer/   # main 的独立元数据
├── manifest/ data/                    # 所有分支共享（不可变）
└── branch/branch-dev/
    └── snapshot/ schema/ tag/ consumer/   # dev 分支独立元数据
```

`BranchManager` 设计成**接口**（`BranchManager.java:32`），有 `FileSystemBranchManager`（直接操作目录）和 `CatalogBranchManager`（委托 REST，§4.4）两个实现——同一上层 API 适配两种后端。分支名校验 `validateBranch`（`BranchManager.java:87`，已修正：旧稿标 86-99）：不能是 main、不能空白、不能纯数字（避免与 Snapshot id 混淆）。

**④ 风险 / 边界**：① main 与子分支共享 data，若两边同时 compaction 理论上可能产生同名文件——Paimon 靠文件名带随机 UUID 规避；② main 的 ExpireSnapshots **不感知子分支引用**：main 过期掉某 Snapshot 后，子分支若还有 Tag 指向它，该 Tag 会悬空（§8.3 的保护只在同一分支内生效）。建议创建分支时同步在 main 建保护 Tag。③ `dropBranch` 只删分支元数据目录、不清数据文件（可能被 main 引用），分支独有数据需靠 main 过期自然回收。

**⑤ 收益与代价**：零拷贝起分支、存储高效、操作简单；代价是跨分支引用关系不被过期机制感知，需用户自己用 Tag 兜底。

### 4.2 分支创建的两种模式

**① 要解决什么问题**：起一条分支需要确定"从哪个点开始演进"——是从某个历史固定点（Tag），还是从空表重新积累。

**② 设计原理与取舍**：两种模式都**只复制元数据、不碰数据文件**：

- **基于 Tag 创建**（`FileSystemBranchManager.java:113`，已修正：旧稿标 97-128）：Tag → `trimToSnapshot` 拿到起点快照，把该 Tag 文件、对应 Snapshot 文件、以及 `0..schemaId` 的全部 Schema 文件复制进分支目录。开销 O(schema 数量)，与数据量无关。
- **空分支创建**（`:89`，已修正：旧稿标 74-95）：只复制最新 Schema，不复制任何 Snapshot，分支从空表开始积累。

为什么必须复制 Schema：Schema id 是全局的，若分支不复制就改 Schema 会污染 main；复制后分支才能独立演进 Schema。

**③ 关键源码**（基于 Tag 创建，精简）：

```java
public void createBranch(String branchName, String tagName, boolean ignoreIfExists) {
    Snapshot snapshot = tagManager.getOrThrow(tagName).trimToSnapshot();
    fileIO.copyFile(tagManager.tagPath(tagName), …copyWithBranch(branchName).tagPath(tagName), true);
    fileIO.copyFile(snapshotManager.snapshotPath(snapshot.id()), …snapshotPath(snapshot.id()), true);
    copySchemasToBranch(branchName, snapshot.schemaId());  // 复制 0..schemaId 的 Schema
}
```

**④ 风险 / 边界**：`createBranch(name, tagName)` 要求 Tag 存在（`getOrThrow` 否则抛异常）；`ignoreIfExists=false` 且分支已存在会抛异常。
**⑤ 收益与代价**：创建分支是轻量元数据操作；代价是分支与 main 共享 data，分支删除时数据文件不会自动清（§4.3）。

### 4.3 FastForward 覆盖式合并

**① 要解决什么问题**：分支实验验证通过后，要把分支结果合回 main。完整三方合并要做 Schema 冲突检测、数据文件冲突解决，成本极高；而 Paimon 分支的典型用法是"实验—验证—合并"，不是长期并行开发。

**② 设计原理与取舍**：选择 **FastForward（覆盖式）而非 merge**——把分支的 snapshot/schema/tag 整体覆盖到 main，**丢弃 main 上分叉点之后的变更**。行为确定、无需用户处理冲突。代价是激进：main 在分支创建后产生的新提交会被无声覆盖。

**③ 关键源码**（`FileSystemBranchManager.java:166`，已修正：旧稿标 156-210）：

```
1. fastForwardValidate：分支不能是 main、不能是当前分支（BranchManager.java:118）
2. earliestSnapshotId = 分支最早快照 id；earliestSchemaId = 其 schemaId
3. 删除 main 上 id >= earliestSnapshotId 的 snapshot/tag、id >= earliestSchemaId 的 schema
4. deleteLatestHint() 删 LATEST hint（避免读到旧的最新值）
5. copyFiles 把分支 snapshot/schema/tag 目录复制进 main（tag 目录存在才复制）
6. invalidateCache() 失效快照缓存
```

**④ 风险 / 陷阱 / 边界**：
- **破坏性不可逆**：步骤 3 删除的 main 提交无法恢复，合并前务必确认 main 自分叉后无重要变更（合并前先给 main 建 Tag 当安全网）。
- `fastForwardValidate`（`BranchManager.java:118`，已修正：旧稿标 101-114）禁止对 main 或当前分支 fast-forward。
- 删 LATEST hint 后下次读 latest 会回退到 list 扫描（§2.5），短暂略慢但正确。

**⑤ 收益与代价**：合并语义简单确定、零冲突解决；代价是只能覆盖、无法保留 main 分叉后变更，仅适合单向"实验回灌主干"。

## 5. Snapshot 回滚：物理删除与 Tag 兜底

**为什么需要回滚**：

1. **数据误操作恢复**：ETL 任务写入错误数据、误删除数据、或执行了错误的 UPDATE/DELETE 操作，需要快速恢复到历史正确状态。没有回滚机制，只能通过备份恢复（耗时数小时）或重跑历史任务（可能无法重现）。

2. **实验失败回退**：在生产环境测试新功能时，如果发现问题需要立即回退。回滚机制可以秒级恢复到测试前的状态，避免长时间停服。

3. **合规性要求**：某些行业（如金融）要求数据变更可追溯、可回退。回滚机制提供了审计友好的版本管理能力。

4. **Compaction 异常恢复**：Compaction 任务可能因 bug 产生错误的合并结果，回滚可以快速恢复到 Compaction 前的状态，避免数据损坏扩散。

**实际场景**：某电商公司的订单表，凌晨 2 点运行的 ETL 任务因代码 bug 将所有订单状态改为"已取消"，早上 8 点用户投诉激增。运维团队执行：
```sql
-- 查询凌晨 1:50 的 Snapshot ID
SELECT snapshot_id FROM my_table$snapshots WHERE commit_time < '2024-12-25 01:50:00' ORDER BY snapshot_id DESC LIMIT 1;
-- 回滚到该 Snapshot
CALL sys.rollback_to('db.my_table', 12345);
```
30 秒内恢复正常，避免了数百万损失。

### 5.1 rollbackTo 两种目标与三层兜底

**① 要解决什么问题**：脏数据写入后要秒级回退到历史正确状态，且即便目标 Snapshot 已被过期、也得能恢复。

**② 设计原理与取舍**：核心抉择是 **物理删除目标之后的元数据，而非像 Iceberg 那样移动 current-snapshot-id 指针**。原因是 Paimon 的 Snapshot id 单调递增且被后续提交直接引用（如 Consumer 的 nextSnapshot、自动 Tag 的 nextSnapshot）——若只回退指针，新提交会生成与"被跳过"的 id 冲突的 id。物理删除让 Snapshot 链始终连续、无"有效/无效"状态机。代价是**不可逆**：删掉的元数据无法找回（回滚前应先建 Tag 当安全网）。

| 维度 | Paimon（物理删除） | Iceberg（指针回退） |
|------|-------------------|---------------------|
| 实现 | 删除目标之后的 snapshot/changelog/tag | 改 current-snapshot-id |
| 可逆 | 否 | 是 |
| Snapshot 链 | 始终连续，无空洞状态 | 历史 Snapshot 保留 |
| 与 id 复用 | 不冲突（已删） | 需保证不复用旧 id |

**两种目标 + 三层兜底**（`AbstractFileStoreTable.rollbackTo`，按 id 在 `:528`、按 tag 在 `:553`，已修正：旧稿标 527-562）：

**③ 关键源码**（按 Snapshot id 回滚，精简）：

```java
public void rollbackTo(long snapshotId) {
    try {
        snapshotManager.rollback(Instant.snapshot(snapshotId));   // 1. REST Catalog 优先
    } catch (UnsupportedOperationException e) {
        try {
            Snapshot snapshot = snapshotManager.tryGetSnapshot(snapshotId);
            rollbackHelper().cleanLargerThan(snapshot);            // 2. 文件系统：读快照并清理
        } catch (FileNotFoundException ex) {                       // 3. 快照已过期：找同 id 的 Tag
            for (Map.Entry<Snapshot, List<String>> e2 : tagManager().tags().entrySet()) {
                if (e2.getKey().id() == snapshotId) { rollbackTo(e2.getValue().get(0)); return; }
            }
            throw new IllegalArgumentException("Snapshot doesn't exist");
        }
    }

```

三层兜底：① REST Catalog 走 `SnapshotLoader.rollback`；② 文件系统直接读快照 + `cleanLargerThan`；③ 快照已过期时遍历 Tag 找同 id 者，委托给"按 Tag 回滚"。按 Tag 回滚（`:553`）额外调 `createSnapshotFileIfNeeded`——若 earliest 已比 Tag 对应快照更新（因过期），`cleanLargerThan` 会删光所有快照，必须把 Tag 内容写回 snapshot 目录，否则表变成"无任何 Snapshot"不可读。这正是 §3.1 "Tag 自带完整 Snapshot 信息"在回滚上的兑现。

**④ 风险 / 陷阱 / 边界**：① 不可逆，回滚前先建 Tag；② 回滚期间有并发写入会产生 id 冲突（提交协议会检测重试，但建议先停写）；③ 按 Tag 回滚重建的 Snapshot 与原 id 相同，需确保 id 生成器正确恢复；④ 子分支回滚不影响 main，但回滚后 fast-forward 会覆盖 main 历史（§4.3）。

**⑤ 收益与代价**：秒级元数据级回退、Snapshot 链干净、有 Tag 即可在快照过期后恢复；代价是不可逆 + 需配套安全 Tag。

### 5.3 回滚对数据文件与 Consumer 的影响

**回滚只动元数据，不动数据文件。** 三层清理只删 snapshot/changelog/tag **元文件**，被它们引用的数据文件原地不动——原因：① 数据文件可能被其它分支/Tag/并发查询引用，立删可能让读取失败；② 立即扫描全表全分支的引用关系成本极高；③ 交给 §8 的 ExpireSnapshots 异步自然回收。收益：回滚是 O(待删快照数) 的轻量元数据操作，与数据量无关，秒级完成；代价是回滚后存储不会立即释放，需等过期或手动触发过期。

**回滚对 Consumer 是隐患。** 回滚不直接改 Consumer 文件，但若某 Consumer 的 `nextSnapshot > retained.id`，它指向的快照已被删，流式读取会发现快照不存在而失败（与 §1.3 陷阱 5 呼应）。处置：① 重启 Flink 任务从 savepoint 恢复；② 用 `ConsumerManager.resetConsumer(id, new Consumer(...))` 手动把位点拉回 `<= retained.id`。**最佳实践：回滚前先停写、停消费、建保护 Tag，回滚后统一重置 Consumer。**

## 6. Schema 回滚：引用检查保护

**① 要解决什么问题**：`ALTER TABLE` 改错了列类型/分区键后要单独回退 Schema，而不连带回退数据。Paimon 的 Schema 是独立文件、独立 id（不像 Iceberg 跟随 Snapshot），所以需要一套独立的 Schema 回滚，且必须保证不把"仍被某 Snapshot 引用的 Schema"删掉（否则那个 Snapshot 永远读不出来）。

**② 设计原理与取舍**：与 Snapshot 回滚一样是**物理删除 + 从大到小删**，但多了一道**引用检查闸门**——删之前先扫所有 Snapshot/Tag/Changelog 引用的 schemaId 集合，若存在 `id > targetSchemaId` 的引用就直接抛异常拒绝回滚。这意味着 **Schema 回滚必须在 Snapshot 回滚之后**（先回滚 Snapshot 释放对新 Schema 的引用，再回滚 Schema），顺序不能颠倒。不重写数据文件：Parquet/ORC 支持 Schema 演进，旧 Schema 读新 Schema 写的文件时忽略多余列即可，重写 O(数据量) 代价太高。

**③ 关键源码**（`SchemaManager.rollbackTo`，`SchemaManager.java:1147`）：

**④ 风险 / 陷阱 / 边界**：① 引用检查严格——有 Snapshot/Tag/Changelog 引用更新 Schema 时回滚失败，必须先回滚 Snapshot；② 物理删除不可逆；③ Schema id 不连续时，删除循环会跳过不存在的 id；④ Catalog 提供 `catalogSchemaRollback` 回调（REST/Hive）时优先走 Catalog 回滚，行为可能与文件系统回滚不同（如可逆性）；⑤ 子分支回滚 Schema 不影响 main，但 fast-forward 会覆盖 main 的 Schema。

**⑤ 收益与代价**：引用检查把"删坏 Snapshot"挡在门外，回滚安全；代价是操作有顺序约束（先 Snapshot 后 Schema），且回滚 Schema 不重写数据、查询时靠 Schema 演进忽略多余列。

**源码路径**: `paimon-core/src/main/java/org/apache/paimon/schema/SchemaManager.java` 第 1147 行起（已修正：旧稿标 1147-1185）

```java
public void rollbackTo(long targetSchemaId, SnapshotManager snapshotManager,
                        TagManager tagManager, ChangelogManager changelogManager) {
    // 1. 验证目标 Schema 存在
    checkArgument(schemaExists(targetSchemaId));

    // 2. 收集所有被 Snapshot/Tag/Changelog 引用的 Schema ID
    Set<Long> usedSchemaIds = new HashSet<>();
    snapshotManager.pickOrLatest(snapshot -> {
        usedSchemaIds.add(snapshot.schemaId());
        return false;
    });
    tagManager.taggedSnapshots().forEach(s -> usedSchemaIds.add(s.schemaId()));
    changelogManager.changelogs().forEachRemaining(c -> usedSchemaIds.add(c.schemaId()));

    // 3. 检查是否有引用了更新 Schema 的 Snapshot 存在
    Optional<Long> conflict = usedSchemaIds.stream()
        .filter(id -> id > targetSchemaId)
        .min(Long::compareTo);
    if (conflict.isPresent()) {
        throw new RuntimeException("Cannot rollback, schema still referenced");
    }

    // 4. 删除所有比目标更新的 Schema 文件
    List<Long> toBeDeleted = listAllIds().stream()
        .filter(id -> id > targetSchemaId)
        .collect(Collectors.toList());
    toBeDeleted.sort((o1, o2) -> Long.compare(o2, o1)); // 从大到小删
    for (Long id : toBeDeleted) {
        fileIO.delete(toSchemaPath(id), false);
    }
}
```

**为什么要检查引用关系**: Schema 回滚比 Snapshot 回滚更危险。如果有 Snapshot 引用了更新的 Schema，删除该 Schema 会导致 Snapshot 不可读。因此必须先确保没有现存引用。

**好处**: 通过严格的引用检查，确保 Schema 回滚是安全的。

## 7. Consumer 机制：消费位点与过期保护

### 7.1 ConsumerManager 与 minNextSnapshot

**① 要解决什么问题**：流任务消费 Paimon 表要能断点续读（重启从上次位点继续，不重复不丢失），多任务消费同一表要各记各的进度，且 Paimon 的过期机制必须能感知"谁还在消费哪个快照"以免误删。仅靠 Flink checkpoint 不够——checkpoint 可能丢失/损坏，且存储层无法读到引擎的 offset。

**② 设计原理与取舍**：Consumer 极简——一个文件 `{branchPath}/consumer/consumer-{consumerId}` 只存一个 `nextSnapshot`（下一个待读快照 id）。结合 Snapshot 链，这一个 Long 就足以恢复完整消费状态，文件约 50 字节。最关键的设计是 `minNextSnapshot()`——取所有 Consumer 的最小 nextSnapshot，交给 ExpireSnapshots 当过期保护下界（§8.2）：只要有 Consumer 还要读快照 N，N 之前的就不能过期。它与 Flink checkpoint **互为备份**（checkpoint 丢了从 Consumer 恢、Consumer 丢了从 checkpoint 恢），且引擎无关（Flink/Spark/自定义消费者都能用）。

**③ 关键源码**（`ConsumerManager.java:44`）：

```java
public OptionalLong minNextSnapshot() {                       // :82
    return listOriginalVersionedFiles(fileIO, consumerDirectory(), CONSUMER_PREFIX)
        .map(this::consumer).filter(Optional::isPresent).map(Optional::get)
        .mapToLong(Consumer::nextSnapshot).reduce(Math::min);
}
```

其余：`consumer(id)` 读（`:66`）、`resetConsumer(id, c)` 用 `overwriteFileUtf8` 原子重置（`:70`）、`deleteConsumer(id)`（`:78`）、`consumers()` 列举（`:141`）。`Consumer` 类只有 `nextSnapshot` 一个字段。

**④ 风险 / 陷阱 / 边界**：① 多任务共用同一 Consumer ID 会互相覆盖位点 → 每任务唯一 ID（含任务名 + 环境）；② 停掉的任务不会自动删 Consumer，其旧 `nextSnapshot` 会一直把 `minNextSnapshot` 钉住、阻止过期 → 手动 `deleteConsumer` 或配 `consumer.expiration-time`；③ 回滚后 `nextSnapshot` 可能指向已删快照（§5.3）；④ 并行度 >1 时由 Source 第 0 并行度统一写，避免并发覆盖。

**⑤ 收益与代价**：极轻量、引擎无关、与过期机制天然联动；代价是生命周期需用户管理（建/删），停掉的僵尸 Consumer 会拖住过期。

**⑤ 收益与代价**：极轻量、引擎无关、与过期机制天然联动；代价是生命周期需用户管理（建/删），停掉的僵尸 Consumer 会拖住过期。

### 7.2 Consumer 过期清理

**① 要解决什么问题**：陷阱 ② 的根源——停掉的任务留下僵尸 Consumer，其 `nextSnapshot` 持续把 `minNextSnapshot` 钉死，使 Snapshot 永远无法过期、存储膨胀。需要一套自动清理。

**② 设计原理与取舍**：两种清理都基于**文件修改时间**而非业务时间——`consumer.expiration-time` 走 `expire(LocalDateTime)`，删除修改时间早于阈值的 Consumer 文件；另有 `clearConsumers(including, excluding)` 按正则批量清理。选用文件修改时间是因为 Consumer 没有"创建时间"字段，且每次 checkpoint 重置位点都会刷新修改时间，活跃任务自然不会被误删。

**③ 关键源码**（`ConsumerManager.java`）：

```java
public void expire(LocalDateTime expireDateTime) {                 // :95
    listVersionedFileStatus(fileIO, consumerDirectory(), CONSUMER_PREFIX)
        .forEach(status -> {
            LocalDateTime modTime = DateTimeUtils.toLocalDateTime(status.getModificationTime());
            if (expireDateTime.isAfter(modTime)) fileIO.deleteQuietly(status.getPath());
        });
}
```

`clearConsumers(Pattern including, Pattern excluding)` 在 `:112`。

**④ 风险 / 边界**：`consumer.expiration-time` 按**修改时间**计——若任务长时间运行但卡在某快照不提交、Consumer 文件不刷新，可能被误删。配置阈值应明显大于任务正常的 checkpoint/提交间隔。

**⑤ 收益与代价**：自动回收僵尸 Consumer、解除对过期的阻塞；代价是阈值配小了会误删卡顿但仍存活的任务的位点。

### 7.3 增量扫描与 FollowUpScanner（概览）

> Consumer 记录的位点最终是被流式增量扫描消费的。本节只讲与版本管理强相关的扫描骨架；Changelog 如何产生（input/lookup/full-compaction 四种 producer）由 `24-Changelog 产生机制.md` 主讲，Flink 写入/Source 链路详见 `15-Flink 写入链路.md`。

**① 要解决什么问题**：流任务要持续读"上次位点之后的新提交"，既要批流一体（先全量后增量），又要避免重复/丢数据。

**② 设计原理与取舍**：扫描拆成 **StartingScanner（定起点）+ FollowUpScanner（判断后续每个快照是否要扫）** 两段——起点逻辑复杂（latest/earliest/from-snapshot/from-timestamp/full…），后续逻辑简单（只判"这个快照要不要扫"），拆开后两类策略可自由组合。`DataTableStreamScan`（`table/source/DataTableStreamScan.java:74`）维护 `nextSnapshotId`（而非 lastConsumed，与 Consumer 的 `nextSnapshot` 语义一致）：`tryFirstPlan`（`:154`）定起点并置 `nextSnapshotId = start.id + 1`，之后循环取 `nextSnapshotId` 对应快照、判 OVERWRITE、`shouldScanSnapshot` + `scan`、自增。

**③ 关键源码**——FollowUpScanner 三实现（均在 `table/source/snapshot/`，已核对）：

| Scanner | 触发条件 | 扫描模式 | 对应 changelog.producer |
|---------|----------|----------|-------------------------|
| `DeltaFollowUpScanner` | 仅 `APPEND`（`DeltaFollowUpScanner.java:35`） | DELTA | none |
| `ChangelogFollowUpScanner` | 有 `changelogManifestList` | CHANGELOG | input/lookup/full-compaction |
| `AllDeltaFollowUpScanner` | 所有快照 | DELTA | FILE_MONITOR |

初始扫描的 level 过滤（`DataTableStreamScan.java:158`，已修正：旧稿标 158-167）：LOOKUP 模式起始只读 `level > 0`（level 0 数据会在后续 compaction 产生 changelog，避免重复消费）；FULL_COMPACTION 模式只读最高 level（`numLevels-1`）。

**④ 风险 / 陷阱 / 边界**：① `changelog.producer=none` 下 COMPACT 快照被跳过，若 compaction 合并了未消费数据可能丢数 → 主键表用 input/lookup；② OVERWRITE 快照需重扫受影响分区，处理不当会重复/丢失；③ LOOKUP 起始过滤 level 0 后，若 compaction 不运行，level 0 数据永不被消费；④ 起始位点优先 `from-snapshot` + Consumer，`latest` 会丢历史、`from-timestamp` 不够精确。

**⑤ 收益与代价**：策略解耦让批流一体与多种 changelog 语义共用一套扫描框架；代价是 producer/scanner/level 过滤三者必须搭配正确，错配即数据重复或丢失。

**Consumer 位点如何参与过期保护**（衔接 §8.2）：

```java
// ExpireSnapshotsImpl.expire()（:134）
if (!expireConfig.isConsumerChangelogOnly()) {
    maxExclusive = Math.min(maxExclusive,
        consumerManager.minNextSnapshot().orElse(Long.MAX_VALUE));
}
```

正在消费的最小位点之前的 Snapshot 不会被过期——这就是 §7.1 `minNextSnapshot` 的终点。

## 8. ExpireSnapshots 多重保护

**为什么需要过期**：

1. **存储空间管理**：Snapshot 和数据文件会持续累积，如果不清理，存储成本会线性增长。ExpireSnapshots 自动删除过期的 Snapshot 和不再被引用的数据文件，控制存储成本。

2. **元数据性能优化**：过多的 Snapshot 文件会降低元数据操作的性能（如列出所有 Snapshot、查找最早 Snapshot）。定期过期可以保持元数据目录的精简。

3. **数据安全保护**：过期操作必须确保不会删除"正在使用"的 Snapshot（如被 Consumer 消费、被 Tag 引用、被时间旅行查询依赖）。多重保护机制防止误删。

4. **灵活的保留策略**：不同场景对数据保留的需求不同（如开发环境保留 1 小时，生产环境保留 7 天）。ExpireConfig 提供了丰富的配置参数，支持多维度的保留策略。

**实际场景**：某公司的订单表，每小时产生 100 个 Snapshot（每分钟一次提交）。如果不过期，一个月会累积 72000 个 Snapshot 文件，元数据目录达到 GB 级别，列出 Snapshot 需要数秒。配置 `snapshot.time-retain=24h` 和 `snapshot.num-retained.min=100` 后，保持 Snapshot 数量在 100-2400 之间，元数据操作延迟降低 100 倍。

**高频陷阱（过期专属）**：

1. **retainMin 和 retainMax 的关系误区**：
   - `retainMin`：最少保留数量（即使超过时间限制也保留）
   - `retainMax`：最多保留数量（即使未超过时间限制也删除）
   - 实际保留数量在 `[retainMin, retainMax]` 之间，取决于时间限制
   
   很多用户误以为 `retainMax` 是"目标保留数量"，导致配置错误。

2. **timeRetain 的计算基准**：
   - `snapshot.time-retain=1h` 表示"保留最近 1 小时内提交的 Snapshot"
   - 基准是 Snapshot 的 `timeMillis`（提交时间），而非当前时间
   - 如果系统时钟回拨，可能导致过期逻辑异常
   
   生产环境必须确保时钟同步（如使用 NTP）。

3. **maxDeletes 限流的副作用**：
   - `snapshot.max-deletes=50` 限制单次过期最多删除 50 个 Snapshot
   - 如果积压了 1000 个过期 Snapshot，需要 20 次过期任务才能清理完
   - 在积压期间，存储空间仍然被占用
   
   建议根据 Snapshot 产生速率调整 `maxDeletes`（如每分钟产生 10 个 Snapshot，设置 `maxDeletes=100`）。

4. **Consumer 保护失效的场景**：
   - 如果配置 `snapshot.expire.consumer-changelog-only=true`，Consumer 只保护 Changelog，不保护 Snapshot
   - 此时如果 Consumer 消费速度慢，其依赖的 Snapshot 可能被过期，导致流式任务失败
   - 只有在"Consumer 只消费 Changelog"的场景下才能启用此配置

5. **Tag 保护的数据文件清理延迟**：
   - Tag 保护的是数据文件，不保护 Snapshot 元数据文件
   - 如果 Snapshot 被过期但 Tag 仍然存在，数据文件会被保留，但 Snapshot 文件已删除
   - 通过 Tag 查询时，会从 Tag 文件中恢复 Snapshot 信息，但可能与原始 Snapshot 略有差异（如缺少某些统计信息）

6. **Changelog 独立生命周期的配置陷阱**：
   - 如果 `changelog.num-retained.max > snapshot.num-retained.max`，会自动启用 `changelogDecoupled`
   - 此时 Changelog 和 Snapshot 独立过期，可能导致"Snapshot 已过期但 Changelog 仍存在"
   - 某些查询（如时间旅行 + Changelog）可能失败
   
   建议明确配置 `changelog-decoupled=true/false`，避免自动推断。

### 8.0 设计原理与取舍（多重保护总览）

**① 要解决什么问题**：Snapshot 与数据文件随提交无限累积，必须修剪以控存储和元数据规模；但修剪绝不能误删"仍在用"的快照（被 Consumer 消费、被 Tag/Branch 引用、被 timeRetain 保护）。

**② 设计原理与取舍**：过期不是"删早于某时间的快照"这么简单，而是一个**过期上界被多道闸门逐级 `Math.min` 收紧、再被 timeRetain 提前终止**的算法（§8.2）。四道闸门各管一件事：

| 保护 | 角色 | 为什么需要 |
|------|------|-----------|
| **retainMax 上限** | 算下界 min | 即使时间限制很长，也不留过多快照 |
| **retainMin 下限** | 算上界初值 | 即使时间很短/时钟异常，至少留 N 个，防全删 |
| **Consumer 保护** | 收紧上界 | 不删任何 Consumer 还要消费的快照（§7.1） |
| **maxDeletes 限流** | 收紧上界 | 单次最多删 N 个，避免长阻塞、分摊成本 |
| **timeRetain** | 循环提前终止 | 保留提交时间未超期的快照 |

外加 **Tag/Branch 对数据文件的引用保护**（§8.3）：快照元文件可以删，但它引用的数据文件若被 Tag 需要则跳过——靠"跳过集合"实现，与 §3.2 删 Tag 的思路同源。

**Changelog 解耦**：默认 changelog 随 snapshot 一起过期；当任一 changelog 保留参数 > 对应 snapshot 参数时自动启用 `changelogDecoupled`，changelog 走独立保留（CDC 下游可能延迟消费，需 changelog 留得比 snapshot 久）。

**③④⑤** 见下面 §8.1（配置与默认值）、§8.2（算法逐闸门）、§8.3（Tag/Branch 数据保护）。

### 8.2 过期上界的四道闸门 + timeRetain 提前终止

**源码路径**: `paimon-core/src/main/java/org/apache/paimon/table/ExpireSnapshotsImpl.java`（`expire()` 在 `:98`，已修正：旧稿标 92-152）

过期范围是 `[earliestId, maxExclusive)`，`maxExclusive` 被四道闸门逐级 `Math.min` 收紧，再被 timeRetain 在循环里提前终止：

```
Step 1  下界  min = max(latestId - retainMax + 1, earliest)          // :126 不超 retainMax
Step 2  上界  maxExclusive = latestId - retainMin + 1                // :131 至少留 retainMin
Step 3  Consumer 保护  if (!consumerChangelogOnly)                   // :134
            maxExclusive = min(maxExclusive, minNextSnapshot)        //   不删 Consumer 还要读的
Step 4  限流  maxExclusive = min(maxExclusive, earliest + maxDeletes) // :142 单次最多删 maxDeletes
Step 5  时间保护（从 min 起遍历）                                       // :144
            for (id = min; id < maxExclusive; id++)
                nextSnapshot = tryGetSnapshot(id + 1)                //   注意：看“下一个”快照
                if (olderThanMills <= nextSnapshot.timeMillis())     //   :150
                    return expireUntil(earliest, id)                 //   提前终止
Step 6  return expireUntil(earliest, maxExclusive)                   // :159
```

**关键修正（timeRetain 看的是"下一个快照"）**：旧稿写"`snapshot(id).timeMillis() >= now - timeRetain` 就终止"，与源码不符。源码实际取 `tryGetSnapshot(id + 1)` 判断——注释明言"a snapshot can only be expired if its next snapshot has been alive longer than snapshotTimeRetain"，即**只有当某快照的下一个快照都已存活超过 timeRetain，该快照才允许过期**，保护更强（避免删掉"虽老但其后继仍很新"的快照）。且循环从 `min` 起、而非从 `earliest` 起。

**多重保护流程**：

```
retainMax → 算下界min ┐
retainMin → 算上界    ├─→ Consumer 收紧 → maxDeletes 收紧 → [min, maxExclusive)
                      ┘                                          │
                                                  从 min 遍历，遇“下一个快照”仍在 timeRetain 内 → 提前终止
                                                                 └→ 否则 expireUntil(earliest, maxExclusive)
```

### 8.3 Tag / Branch 保护与数据文件安全

**元文件可删、数据文件按引用保护。** 过期会删 Snapshot 元文件，但其引用的数据文件若被 Tag/后续快照需要则**跳过**。`expireUntil` → `innerExpireUntil`（`ExpireSnapshotsImpl.java:171`）里：

```java
List<Snapshot> taggedSnapshots = tagManager.taggedSnapshots();          // :199
// 对每个待过期快照，构建“被 Tag 引用的数据文件跳过器”，跳过它们
Predicate<ExpireFileEntry> skipper =
    snapshotDeletion.createDataFileSkipperForTags(taggedSnapshots, id);  // :215
snapshotDeletion.cleanUnusedDataFiles(snapshot, skipper);               // :224
```

manifest 清理同理构建跳过集（`findSkippingTags` + `manifestSkippingSet`，`:244` 起），保护被 Tag 和保留快照引用的 manifest。

**为什么 Tag 只保护数据文件、不保护 Snapshot 元文件**：① Snapshot 元文件仅 1~2KB，留着不留着对存储无所谓；② Tag 自带完整 Snapshot 信息（§3.1），元文件删了也能从 Tag 恢复；③ 若还要保护元文件就得维护元文件引用计数，徒增复杂度。所以过期后可能出现"Snapshot 元文件已删、但数据文件因 Tag 保留"的状态——这正是 §3.2 删最后一个 Tag 才真正清数据的前提。

**Branch 的保护边界（重要）**：上述跳过集只覆盖**当前分支的 Tag 和快照**，**不感知其它分支的引用**（呼应 §4.1 风险②）。main 过期某快照时不会检查子分支是否还引用它的数据文件——若子分支共享了该数据文件且 main 把它删了，子分支会损坏。**实践：跨分支共享起点时，在 main 上对该起点建保护 Tag**，借 Tag 的数据文件保护兜住跨分支引用。

**为什么需要 Tag 保护**: Snapshot 过期后其数据文件可能仍被 Tag 需要——Tag 的设计初衷就是"Snapshot 过期后仍能访问该时间点数据"，这道保护让该承诺在过期路径上兑现。

## 9. 实战场景

### 场景一: 数据回滚恢复

**问题**: 错误的 ETL 任务写入了脏数据。

**解决方案**:

```sql
-- 方案 A: 按 Snapshot ID 回滚
CALL sys.rollback_to('database_name.table_name', 42);

-- 方案 B: 按 Tag 回滚（Snapshot 已过期时）
CALL sys.rollback_to_tag('database_name.table_name', 'backup-20241225');
```

**原理**: RollbackHelper 删除目标 Snapshot 之后的所有 Snapshot/Changelog/Tag 元数据文件，并更新 LATEST hint。数据文件保持不变，后续过期会自然清理不再引用的文件。

**关键注意事项**:
- 回滚是不可逆操作（被删除的 Snapshot 元数据不可恢复）
- 回滚后需要手动重置或重启所有 Consumer/Flink 流式任务
- 建议在回滚前先创建一个 Tag 作为安全网

### 场景二: A/B 测试分支

**问题**: 需要在不影响生产数据的情况下测试新的 ETL 逻辑。

**解决方案**:

```sql
-- 1. 基于当前最新 Tag 创建实验分支
CALL sys.create_branch('db.table', 'experiment', 'daily-20241225');

-- 2. 在实验分支上运行新 ETL
-- (通过 'branch' = 'experiment' 表选项路由到分支)

-- 3. 验证通过后，fast-forward 合并回 main
CALL sys.fast_forward('db.table', 'experiment');

-- 4. 或者验证失败，直接删除分支
CALL sys.drop_branch('db.table', 'experiment');
```

**原理**: 分支创建只复制 Schema/Snapshot/Tag 元文件（不复制数据），FastForward 将分支的元数据覆盖到 main。整个过程对数据文件零拷贝。分支用于"特征实验隔离 / A-B 实验"的完整工程实践详见 `41-分支机制与特征实验隔离.md`。

### 场景三: 定期 Tag 备份策略

**配置示例**:

```sql
ALTER TABLE my_table SET (
    'tag.automatic-creation' = 'watermark',        -- 基于 watermark 创建
    'tag.creation-period' = 'daily',               -- 每天一个 Tag
    'tag.creation-delay' = '10 min',               -- 允许 10 分钟延迟
    'tag.num-retained-max' = '30',                 -- 最多保留 30 个自动 Tag
    'tag.default-time-retained' = '7 d'            -- 每个 Tag TTL 7 天
);
```

**运行效果**: 每天自动创建一个以日期命名的 Tag（如 `2024-12-25`），超过 30 个时删除最老的，且每个 Tag 创建 7 天后自动过期。两种清理策略取先到者。

## 10. 与 Iceberg 版本管理对比

| 维度 | Apache Paimon | Apache Iceberg |
|------|---------------|----------------|
| **快照模型** | Snapshot（JSON 文件，引用 ManifestList） | Snapshot（Avro/JSON，引用 ManifestList） |
| **快照 ID** | 单调递增 Long（从 1 开始） | 单调递增 Long（时间戳-based） |
| **时间旅行** | `scan.snapshot-id` / `scan.timestamp` / `scan.tag-name` / `scan.watermark` | `snapshot-id` / `as-of-timestamp` / `branch` / `tag` |
| **Tag** | 独立 Tag 文件（继承 Snapshot 格式），支持 TTL | Ref 机制，Tag 是一种 SnapshotRef（type=TAG） |
| **Branch** | 独立目录结构，共享数据文件 | Ref 机制，Branch 是一种 SnapshotRef（type=BRANCH），每个 Branch 有独立的 Snapshot 链 |
| **Branch 存储** | `{table}/branch/branch-{name}/` 独立 snapshot/schema/tag | 在同一 metadata 目录中，通过 `refs` 表区分 |
| **合并** | FastForward（覆盖式） | cherry-pick / fast-forward |
| **回滚** | 删除目标之后的元数据文件 | `rollback-to-snapshot` / `rollback-to-timestamp`，通过设置 current-snapshot-id |
| **回滚安全性** | 删除后续 Snapshot 文件（不可逆） | 只更改 current-snapshot-id 指针（可逆）|
| **Consumer** | 基于文件系统的 Consumer ID + nextSnapshot | 无内建 Consumer，依赖引擎的 checkpoint 机制 |
| **过期保护** | retainMin + retainMax + timeRetain + Consumer 保护 + expireLimit | `min-snapshots-to-keep` + `max-snapshot-age-ms` |
| **Schema 回滚** | 独立的 Schema 回滚机制（检查引用后删除） | 通过 Snapshot 回滚间接恢复（Schema 跟随 Snapshot） |
| **自动 Tag** | 内建 TagAutoCreation（process-time / watermark 模式） | 无内建自动 Tag |
| **Changelog** | 内建 Changelog 机制（与 Snapshot 解耦或绑定） | 无内建 Changelog，依赖 incremental scan |

**关键设计差异分析**:

1. **Branch 实现**: Paimon 使用物理目录隔离，每个分支有独立的 snapshot/schema/tag 子目录。Iceberg 使用逻辑引用（SnapshotRef），所有分支的 Snapshot 共享同一元数据目录。Paimon 的方式更适合文件系统场景，Iceberg 的方式更适合有事务支持的 Catalog。

2. **回滚机制**: Paimon 的回滚是物理删除（不可逆），Iceberg 的回滚是指针移动（可逆）。Paimon 选择物理删除是因为其 Snapshot ID 单调递增且被后续提交引用，不能简单跳过；Iceberg 可以指针回退因为其 metadata log 独立于 Snapshot 链。

3. **Consumer 内建**: Paimon 的 Consumer 机制直接集成在存储层，与过期机制联动。Iceberg 依赖计算引擎（如 Flink/Spark）自身的 checkpoint 机制跟踪消费位点。Paimon 的方式更内聚但引入了存储层的复杂度。

4. **自动 Tag**: Paimon 内建自动 Tag 创建和过期，Iceberg 需要外部调度来创建 Tag。这反映了 Paimon 对流式处理场景的更深度优化。

## 11. 设计决策总结

| 决策点 | 选择 | 取舍 / 代价 | 收益 |
|--------|------|-------------|------|
| 版本抽象 | 单调递增、自包含 JSON 的 Snapshot 链 | 元数据随提交线性累积，须靠过期修剪 | 版本号 O(1) 定位、时间 O(log N) 二分、可 `cat` 调试、新增字段向后兼容 |
| 时间定位 | 时间戳/watermark 二分 | watermark 可空致实现复杂、对空值敏感 | 百万级快照仍毫秒级定位，每次仅一次 IO |
| 首尾定位 | EARLIEST/LATEST hint + next 自校验 | hint 失效时退化为目录 list | 稳态免 list，O(1) read |
| Tag 表示 | 继承 Snapshot（自带完整信息 + TTL 字段） | 比纯 id 引用重几百字节 | 过期后独立兜底、老版本兼容、回滚源、Iceberg 兼容 |
| Tag 删除 | 左右邻"跳过集合" + 删最后引用才清数据 | 删除前需读相邻 manifest 构建 skip 集 | 安全删除，不误删共享数据文件 |
| Tag TTL | 单 Tag TTL 与全局 olderThan 取先到者 | 双条件下需用户想清谁先到期 | 长短期 Tag 可共存的灵活保留 |
| Branch 布局 | 隔离元数据、共享数据文件 | 跨分支引用不被过期感知，需 Tag 兜底 | 零拷贝起分支、存储高效 |
| Branch 合并 | FastForward 覆盖式 | 不可逆、丢弃 main 分叉后变更 | 语义确定、零冲突解决 |
| 回滚 | 物理删除目标之后元数据 | 不可逆，需回滚前建 Tag 安全网 | Snapshot 链干净、id 不冲突、秒级 |
| 回滚兜底 | 快照过期时遍历 Tag 恢复 + 重建快照文件 | 需 Tag 在过期前已建 | "只要有 Tag 就能回滚" |
| Schema 回滚 | 物理删除 + 引用检查闸门 | 必须先回滚 Snapshot 再回滚 Schema | 杜绝删坏被引用的 Schema |
| Consumer | 只存一个 nextSnapshot 的文件 | 生命周期需用户管理，僵尸 Consumer 拖住过期 | 极轻量、引擎无关、与过期联动 |
| 过期上界 | 四道闸门逐级 Math.min + timeRetain 提前终止 | 配置维度多，易误解 max/min 关系 | 多重保护防误删，行为可预测 |
| timeRetain 判据 | 看"下一个快照"是否仍在保留期 | 略反直觉 | 保护"虽老但后继仍新"的快照，更安全 |
| 数据文件保护 | Tag/保留快照引用即跳过删除 | 只覆盖当前分支，不跨分支 | 元文件可删、数据按引用安全回收 |
