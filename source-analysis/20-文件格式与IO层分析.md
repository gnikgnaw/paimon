# Apache Paimon 文件格式与 IO 层源码深度分析

> **版本**：1.5-SNAPSHOT　**源码模块**：`paimon-format`（ORC/Parquet/Avro/CSV/JSON/Text 实现）、`paimon-common`（`org.apache.paimon.fs` FileIO 抽象、`org.apache.paimon.format` 格式 SPI、compression 内部压缩）、`paimon-core`（KeyValue 读写工厂、路径工厂）　**核对日期**：2026-06

**一句话定位**：文件格式与 IO 层是 Paimon 落到磁盘/对象存储的"最后一公里"——它一边用 `FileFormat` SPI 把"流式快写"（行式 Avro）和"分析快读"（列式 ORC/Parquet）按 LSM 层级分开取舍，一边用 `FileIO` 抽象把对象存储"不能原地改、rename 非原子、有可见延迟"的物理特性兜住，让上层引擎对底层存储无感。

读完本文你应能回答：① 为什么 LSM 不同层级要用不同格式，Level 0 用行式、高层用列式背后的取舍是什么；② `FileFormat`/`FormatReaderFactory`/`FormatWriterFactory` 这套 SPI 抽象怎么把 ORC/Parquet SDK 的差异关在 `paimon-format` 里；③ 列裁剪和谓词下推是怎么一层层落到 ORC 的 `SearchArgument` 和 Parquet 的 `FilterCompat` 上的，开 Deletion Vector 后为什么 Row Group 过滤会被禁掉；④ ORC 零拷贝向量映射、异步预取、Parquet 的 `RequestedSchema` 缓存各自在省什么；⑤ zstd/lz4/snappy 三种压缩算法、文件格式层压缩 vs 内部块压缩两套体系各自的代价权衡；⑥ `FileIO` 怎么在 HDFS 上靠原子 rename、在 S3 上靠 Multi-Part Upload 做原子提交，对象存储的可见延迟怎么用 `writerMetadata` 绕开；⑦ 哪些配置（`file.format`、`file.format.per.level`、`file.compression.zstd-level`、`file-reader-async-threshold`）配错会拖慢写入或退化查询。

## 1. 快速理解（核心问题 / 概念 / 陷阱）

### 1.1 核心问题：格式层与 IO 层各自在解决什么

这一层其实是两件几乎正交的事，放在一篇里是因为它们都"贴着磁盘"：

**格式层（`paimon-format`）要解决的问题**：一份 LSM 数据，在不同层级有完全不同的访问模式。Level 0 是 memtable 刚溢写出来的、生命周期极短、马上会被 compaction 吃掉，写入速度是唯一关心的事；高层级是 compaction 沉淀下来、要被反复分析查询的冷数据，读取性能（列裁剪、谓词下推、压缩比）才是关键。用一种格式通吃必然两头不讨好——行式格式写得快但读分析慢，列式格式读分析快但写入要先攒批、建索引、编码。Paimon 的答案是**让格式可按层级配置**：`file.format.per.level='0:avro'` 让 L0 走行式快写，其余层走列式（默认 `parquet`）。这与 RocksDB 的分层 compaction 哲学一脉相承。

**IO 层（`paimon-common` 的 `org.apache.paimon.fs`）要解决的问题**：对象存储（S3/OSS）的物理特性和文件系统（HDFS/Local）根本不同——不能原地改写、rename 实质是 copy+delete（非原子）、写完有短暂可见延迟。而 Paimon 的快照提交、schema 变更都依赖"原子地让一个新文件出现"。`FileIO` 抽象把这套差异关进实现里：HDFS 用原子 rename，S3 用 Multi-Part Upload 的 complete 操作，本地用 `ATOMIC_MOVE`，上层引擎完全无感。同时 `FileIO` 是 `Serializable` 的，因为它要被 Flink/Spark 序列化分发到每个 Task。

一句话设计哲学：**格式层用"按层级特化"换写读两端的最优；IO 层用"统一抽象 + 实现内兜底"把对象存储的不原子、有延迟藏起来。**

### 1.3 高频生产陷阱

**陷阱 1：以为默认格式是 ORC。（已修正）** 旧稿全文反复把"Paimon 默认 ORC"当前提，实际 `file.format` 默认值是 **`parquet`**（`CoreOptions.java:295`，`defaultValue(FILE_FORMAT_PARQUET)`）。这会影响选型判断：默认表写出来的是 Parquet 文件，Row Group 默认 128MB（见 `file.block-size` 描述 `CoreOptions.java:383`），不是 ORC 的 64MB Stripe。

**陷阱 2：Level 0 用列式格式拖慢写入。** L0 是 memtable 高频溢写，列式格式要先攒批、建字典/索引、列编码，写延迟高且文件生命周期极短（很快被 compaction 重写），等于白做功。规避：`file.format.per.level='0:avro'` 让 L0 走行式快写，高层再列式。

**陷阱 3：盲目调高 `file.compression.zstd-level`。** zstd level 从 1（默认 `CoreOptions.java:328`）调到 9，压缩比提升有限但压缩速度可能掉 5–8 倍，直接拖慢写入吞吐。只在冷数据归档、存储成本敏感且写少读多时调高；热数据/L0 反而该用 lz4。

**陷阱 4：开了 Deletion Vectors 却奇怪谓词没下推。** 开 DV（或 bitmap 索引）后 ORC 的 Row Group 级过滤会被**主动禁用**（`OrcReaderFactory.java:293` 的 `!deletionVectorsEnabled` 条件），因为 Row Group 过滤会跳行、改变行号，而 DV 依赖绝对行号。这是正确行为不是 bug——Stripe 级裁剪仍在，只是行组级精筛让位给了行级删除。

**陷阱 5：`file-reader-async-threshold` 配错或以为所有格式都异步。（已修正）** 异步预取**只对 ORC 生效**（`KeyValueFileReaderFactory.java:120` 显式判断 `.orc` 后缀），且阈值配置项是 `file-reader-async-threshold`（默认 10MB，`CoreOptions.java:1850`），不是旧稿写的 `read.async-threshold`。Parquet 大文件不会走异步。

**陷阱 6：误以为 Avro 能列裁剪/谓词下推。** Avro 是行式，`createReaderFactory` 直接忽略 `filters` 参数（`AvroFileFormat.java:80` 只 `new AvroBulkFormat(projectedRowType)`），投影只在反序列化后裁，没有文件级跳过。所以 Avro 只适合 L0/Manifest/Changelog 这类"写多、整读、不分析"的小文件，绝不要把分析型大表设成 `file.format=avro`。

**陷阱 7：把 `manifest.format` 改成列式。** Manifest 存 `DataFileMeta`，行少、字段多、每次提交都写，是典型的小文件高频写场景，默认 `avro`（`CoreOptions.java:442`）最合适。改成 orc/parquet 只会增加写开销，且可能影响旧快照可读性。

**陷阱 8：对象存储上写完立刻按文件路径重读统计。** S3 写完文件有短暂"不可见"窗口，提取统计时若重新打开文件可能读不到。Paimon 用 `SimpleStatsExtractor.extract(..., writerMetadata)` 从写入器内存元数据直接取统计绕开（`SimpleStatsExtractor.java`），自定义路径别走"先 close 再 reopen 算统计"。

## 2. 三种格式的定位与选型取舍

**① 要解决什么问题**

一份数据在 LSM 树里经历从"刚写入的热数据"到"沉淀的冷数据"的完整生命周期，每个阶段的访问模式天差地别；外部还要和 Spark（偏好 Parquet）、Hive（偏好 ORC）生态互通。如果只用一种格式，要么牺牲写入吞吐（列式格式攒批+编码慢），要么牺牲查询性能（行式格式没法列裁剪/谓词下推）。所以 Paimon 同时支持行式（Avro）与列式（ORC/Parquet）三种主格式，加上文本格式（CSV/JSON/Text）用于导入导出。

**② 设计原理与取舍：行式 vs 列式的本质分歧**

| 维度 | Avro（行式） | ORC（列式） | Parquet（列式） |
|------|------|------|------|
| 存储方式 | 整行连续 | Stripe（默认 64MB）→ 列 Stream | Row Group（默认 128MB）→ Column Chunk → Page |
| 写入开销 | 低（直接序列化，无索引） | 中（攒批、列编码、轻量索引） | 中高（需缓完整列数据再编码） |
| 列裁剪 | 反序列化后才裁，无文件级跳过 | 文件级（只读需要的列 Stream） | 文件级 + 嵌套递归裁剪（Dremel） |
| 谓词下推 | 不支持 | Stripe + Row Group 统计 | Row Group 统计 + 可选 Bloom Filter |
| 压缩比 | 中 | 高（RLE+字典+列存聚簇） | 高（字典+RLE+Delta） |
| 适用阶段 | L0、Manifest、Changelog | 高层数据、Hive 互通 | 高层数据（**Paimon 默认**）、Spark 互通 |

设计哲学一句话：**行式格式赌"写完很快被重写、整读不分析"，省掉一切列存的预处理；列式格式赌"写一次读多次、要分析"，用写入端的攒批编码成本换读取端的列裁剪/谓词下推/高压缩比。** LSM 的分层天然把这两类需求在时间轴上分开了——L0 是前者、高层是后者——所以"按层级选格式"不是花哨功能，而是 LSM 与列存结合的必然产物。

**为什么 Avro 没有谓词下推**：行式存储里一行的各列字节是交错连续的，没有"某一列的 min/max/null_count"这种列级元数据可供文件层过滤，只能顺序扫描整块再反序列化。源码上 `AvroFileFormat.createReaderFactory` 直接丢弃 `filters` 参数（`AvroFileFormat.java:76`），列投影 `projectedRowType` 也只是反序列化后取需要的字段。

**为什么列式格式适合分析**：列存把同一列的值聚簇存放，① 只读查询涉及的列（列裁剪省 I/O）；② 同列同类型值压缩比天然更高；③ 每个数据块带 min/max 统计，谓词能在 Stripe/Row Group 粒度跳过整块（谓词下推省 I/O）。代价是写入端必须先把一批行"转置"成列、构建字典与索引，所以写延迟更高、需要攒够批才划算。

**③ 三者在 Paimon 里的实际角色**

- **Parquet**：`file.format` 默认值（`CoreOptions.java:295`，`FILE_FORMAT_PARQUET`）。嵌套类型（ROW/ARRAY/MAP）原生支持且能递归裁剪，与 Spark 集成最成熟，还支持 Variant Shredding。
- **ORC**：与 Hive 生态契合最好。Paimon 对 ORC 的集成最深——支持零拷贝向量映射、大文件异步预取（见 §4）。
- **Avro**：`manifest.format` 默认值（`CoreOptions.java:442`），也是 L0/Changelog 的推荐格式。写得快、文件头开销小，适合"小文件、行少、高频写"。
- **CSV/JSON/Text**：文本格式，仅用于外部导入导出，详见 §6。

**④ 风险/陷阱/边界**

- 分析型大表误设 `file.format=avro` → 查询退化为全扫，列裁剪/谓词下推全失效。
- L0 误用列式 → 高频溢写时反复做列编码白功，写吞吐下降。
- 改 `manifest.format` 为列式 → 增加提交开销、可能影响旧快照可读性。
- 格式变更（`ALTER TABLE ... SET ('file.format'='parquet')`）只影响**新写文件**，旧文件保持原格式；读取时按文件名后缀自动识别格式（见 §10），故新旧格式可共存。

**⑤ 收益与代价**

收益：写读两端都拿到接近最优的格式，且能跨引擎互通。代价：配置维度变多（全局格式 + per-level + 压缩 + zstd-level），配错就两头不讨好；多格式实现也增大了 `paimon-format` 的维护面。下文 §4–§6 分别讲透三类格式的工厂实现。

## 3. FileFormat SPI 抽象与格式发现

**① 要解决什么问题**

格式实现要依赖 ORC/Parquet/Avro 各自的 SDK，这些 SDK 又笨重又互相冲突。如果让 `paimon-core` 直接依赖它们，核心包会被 SDK 污染、且加新格式必须改核心代码。需要一层抽象，把"用什么格式"和"格式怎么实现"解耦，让 core 只面向接口编程、运行时再动态加载实现。

**② 设计原理与取舍：抽象在 common、实现在 format、SPI 串联**

分层是关键：
- `paimon-common` 定义抽象 `FileFormat`、`FormatReaderFactory`、`FormatWriterFactory`、`SimpleStatsExtractor`，**不依赖任何格式 SDK**。
- `paimon-format` 实现 `OrcFileFormat`/`ParquetFileFormat`/`AvroFileFormat`，依赖各 SDK，并通过 SPI（`FileFormatFactory`）注册。
- `paimon-core` 只依赖 `paimon-common` 的抽象，运行时按 `identifier`（"orc"/"parquet"/"avro"）经 `ServiceLoader` 发现并实例化对应实现。

设计哲学：**用 SPI 把"格式选择"变成运行时的字符串到工厂的查表，加新格式只需新增一个 `FileFormatFactory` 实现 + SPI 注册文件，零侵入核心。**

### 3.1 FileFormat / FormatReaderFactory / FormatWriterFactory

`FileFormat` 是抽象基类（`FileFormat.java:43`，paimon-common），四个核心方法：

```java
public abstract FormatReaderFactory createReaderFactory(           // :62
        RowType dataSchemaRowType, RowType projectedRowType, @Nullable List<Predicate> filters);
public abstract FormatWriterFactory createWriterFactory(RowType type);   // :66
public abstract void validateDataFields(RowType rowType);                // :69
public Optional<SimpleStatsExtractor> createStatsExtractor(...);         // :71（默认返回 empty）
```

三个抽象的职责分工，恰好对应"读三件事"：
- **`createReaderFactory` 的三个参数就是列裁剪+谓词下推的入口**：`dataSchemaRowType` 是文件全 schema，`projectedRowType` 是投影后只要的列（列裁剪），`filters` 是谓词（下推）。列式格式会把这两者落到 SDK（ORC `SearchArgument`、Parquet `FilterCompat`），行式 Avro 则只用 `projectedRowType` 在反序列化后裁、丢弃 `filters`。
- **`createWriterFactory(type)`** 建出的工厂，其 `create(out, compression)` 接收输出流和压缩算法——压缩算法是写时才传入的，使得"按层级配压缩"成为可能（见 §8）。
- **`createStatsExtractor`** 决定该格式能否"文件级提取统计"：ORC/Parquet 覆盖它返回提取器（从 Footer 读 min/max），Avro 不覆盖（返回 empty），于是 core 改走逐行收集（见 §11）。

**FormatContext** 统一承载格式参数（`FileFormatFactory.java:35`）：`options`、`readBatchSize`、`writeBatchSize`、`writeBatchMemory`、`zstdLevel`、`blockSize`。这些值由 `CoreOptions` 在 `FileFormat.fromIdentifier` 里组装（`FileFormat.java:76`），避免逐方法传一堆参数。

### 3.2 格式工厂的 SPI 发现

**收益**：写读两端各取所需，对标 RocksDB 的分层策略。**代价**：同一张表里出现混合格式，运维要清楚每层是什么；读取时靠文件名后缀识别格式（§10），所以混合是安全的。

## 4. ORC 格式适配

**① 要解决什么问题**

ORC（Optimized Row Columnar）是 Hive 生态主推的列存格式。Paimon 用它支撑分析型读取（列裁剪、谓词下推、高压缩比），并与 Hive 表互通。难点不在"会用 ORC SDK"，而在三件适配工作：把 ORC SDK 的类型系统对齐 Paimon 的类型系统、把 ORC 的列向量零成本接入 Paimon 的读路径、让 ORC 的行组过滤与 Paimon 的 Deletion Vector 共存而不冲突。

**② 设计原理与取舍：三个工程优化**

| 优化点 | 做法 | 取舍 |
|--------|------|------|
| 类型精化 `refineDataType` | 在边界把 ORC 不区分的类型归一（BINARY/VARBINARY→bytes、MULTISET→MAP） | 内部保持 Paimon 类型系统纯净，代价是读写两端各做一次转换 |
| 零拷贝向量映射 | ORC `ColumnVector` 与 Paimon `ColumnVector` 共享底层数组 | 省内存拷贝/GC，代价是向量引用不能逃逸出当前批次 |
| Pool 复用批次 | `Pool<OrcReaderBatch>` 预分配批次对象，同步 size=1、异步 size=2 | 避免反复 new 大数组，代价是异步预取内存翻倍 |

一句话：**ORC 适配的核心是"贴着 SDK 做零拷贝 + 池化"，把列式读的内存与 GC 成本压到最低；类型差异则在进出边界一次性抹平。**

下面 4.1–4.4 分别讲入口与类型精化、读、写、谓词下推。

### 4.1 OrcFileFormat 入口与类型精化

**源码路径**: `paimon-format/src/main/java/org/apache/paimon/format/orc/OrcFileFormat.java`

类型精化 `refineDataType`（`OrcFileFormat.java:177`）把 ORC SDK 与 Paimon 不对齐的类型在边界归一：

- `BINARY`/`VARBINARY` → bytes（ORC 不区分定长/变长二进制，`OrcFileFormat.java:179`）
- `MULTISET<T>` → `MAP<T, INT(false)>`（ORC 无 multiset 概念，用计数 map 表达，`OrcFileFormat.java:193`）
- 嵌套类型（ARRAY/MAP/ROW）递归精化

**为什么在边界做转换而非读取时**：让 Paimon 内部的类型系统保持一致，避免核心代码到处处理"ORC 的特殊类型"。读出来再转回 Paimon 类型，写进去前从 Paimon 类型转成 ORC 能表达的形式。`deletionVectorsEnabled` 在构造时从 `DELETION_VECTORS_ENABLED` 取出（`OrcFileFormat.java:86`），传给 ReaderFactory 决定是否禁用行组过滤。

### 4.2 OrcReaderFactory：Pool、零拷贝、Stripe 裁剪

**源码路径**: `paimon-format/src/main/java/org/apache/paimon/format/orc/OrcReaderFactory.java`

读取关键步骤（行号已核对）：

1. **Stripe 范围裁剪**：`getOffsetAndLengthForSplit`（`OrcReaderFactory.java:333`）只读落在当前 Split `(offset,length)` 内的 Stripe，配合 `useZeroCopy`（`:289`）。
2. **批次池**：`Pool<OrcReaderBatch>`（`:104`、`createPoolOfBatches :152`）预分配并循环复用批次对象。批次内含多个列向量数组，反复 new 会造成 GC 压力，池化避开。
3. **谓词下推 + DV 互斥**：仅当 `!conjunctPredicates.isEmpty() && !deletionVectorsEnabled && selection == null` 时才启用行组级筛选（`:293`），随后设 `useSelected`/`allowSARGToFilter`（`:296-297`）。即——开了 DV 或带 bitmap selection 时，行组过滤被关掉。
4. **零拷贝向量映射**：`nextBatch` 从 ORC SDK 拿到 `VectorizedRowBatch`（`:328`），再包装成共享数组的 Paimon `ColumnVector`，无数据拷贝。

**为什么 DV 要禁行组过滤**：行组过滤会"跳过不匹配的行"，使返回行的相对位置发生变化；而 DV 用绝对行号标记删除，一旦跳行行号就对不上。所以必须二选一，Paimon 让位给更灵活的行级删除（Stripe 级裁剪不受影响，仍生效）。详见 §1.3 陷阱 4，DV 机制本身详见 04 文档。

### 4.3 OrcWriterFactory：双阈值 flush

**源码路径**: `paimon-format/src/main/java/org/apache/paimon/format/orc/OrcWriterFactory.java`、`OrcBulkWriter`

写入的核心是 `OrcBulkWriter.addElement`：把行 vectorize 进 `rowBatch`，**当达到行数上限 `writeBatchSize`（默认 1024）或累计内存达到 `writeBatchMemory`（默认 128MB）任一条件，就 flush（`writer.addRowBatch`）**。

**为什么要双阈值**：只看行数会在宽 STRING 列上 OOM——1024 行若每行携带几 KB 字符串，内存可能已远超预期；只看内存又会在窄行上批次过大。两个阈值取"先到为准"，既防 OOM 又保证批足够大、压缩有料。文件大小估算 `reachTargetSize` 用 `fileLength + estimateMemory * 0.2`，0.2 系数体现 ORC 内存缓冲数据落盘后会被压缩。

写入流被包成不可被 SDK 提前关闭的形式（`PhysicalFsWriter` 屏蔽 `close()`，关闭时机由上层 `SingleFileWriter` 掌控），ORC `WriterImpl` 的 path 只是个 UUID（仅供 ORC memory manager 做 key，不是真实落盘路径）。

### 4.4 ORC 谓词下推与 DV 互斥

**源码路径**: `paimon-format/.../orc/filter/OrcPredicateFunctionVisitor.java`

Paimon 的 `Predicate` 经 `OrcPredicateFunctionVisitor` 转成 ORC 的 `SearchArgument`，支持两级过滤：
1. **Stripe 级**：用 Stripe 列统计（min/max/count）跳过整个 Stripe——这级始终生效。
2. **Row Group 级**：用行组索引进一步精筛——**DV/bitmap 模式下被禁用**（见 4.2）。

谓词下推全链路（Predicate 如何从 SQL 一路传到这里）详见 12 文档，本篇只讲它在 ORC 层的落点。

## 5. Parquet 格式适配

**① 要解决什么问题**

Parquet 是 Spark 生态的事实标准，也是 Paimon 的**默认 `file.format`**（`CoreOptions.java:295`）。它的强项是嵌套类型（ROW/ARRAY/MAP，靠 Dremel 编码）能做递归列裁剪，以及对 Variant 半结构化类型的 Shredding 优化。适配难点：把文件 schema 裁剪映射成"只读需要列"的请求 schema 这件事不便宜，而且外部迁移来的 Parquet 文件 schema 可能各不相同。

**② 设计原理与取舍：两个关键优化**

| 优化点 | 做法 | 取舍 |
|--------|------|------|
| `RequestedSchema` 缓存 | 按 `fileSchema` 缓存裁剪结果（`requestedSchemaCache`） | 同 schema 文件免重复算 `clipParquetSchema`；按 fileSchema 做 key 而非"只算一次"，兼容外部异构文件 |
| Variant 推断装饰器 | `VariantInferenceWriterFactory` 包装写工厂，写入中推断 Variant schema | 把高频字段提成独立列加速查询；代价是第一个文件还没推断出来、仍是完整 Variant 列 |

设计哲学：**Parquet 适配把"schema 映射"做成可缓存的纯函数，把"Variant 优化"做成可装饰的旁路，两者都不侵入核心读写主干。**

下面 5.1–5.3 讲入口/Variant、读（缓存与递归裁剪）、写与列裁剪/谓词下推。

### 5.1 ParquetFileFormat 与 Variant 推断

**源码路径**: `paimon-format/.../parquet/ParquetFileFormat.java`

`createWriterFactory` 把基础 `ParquetWriterFactory` 包进 `VariantInferenceWriterFactory`：Variant Shredding 把 Variant 里高频出现的字段提取成独立 Parquet 列，查询时能像普通列一样裁剪/下推。装饰器在写入过程中推断 Variant 的 schema，**应用于下一个文件**——所以第一个文件仍是完整 Variant 列，这是"延迟生效"的边界。用装饰器模式而非改核心，是为了把这个 Parquet 特有优化关在格式层内。

### 5.2 ParquetReaderFactory：RequestedSchema 缓存与递归裁剪

**源码路径**: `paimon-format/.../parquet/ParquetReaderFactory.java`（行号已核对）

读取主线：建 `ParquetFileReader`（传入 filter、range，以及 bitmap 索引用的 `RoaringBitmap32` selection）→ 取文件 schema（`:126`）→ `getOrCreateRequestedSchema`（`:150`）→ 建 `VectorizedParquetRecordReader`（`:140`）向量化读、每次返回一批行。

**RequestedSchema 缓存**（`:96`）：

```java
private final Map<MessageType, RequestedSchema> requestedSchemaCache = new ConcurrentHashMap<>();
// createRequestedSchema：clipParquetSchema(fileSchema) + buildFieldsList，二者是 (readFields, fileSchema) 的纯函数（:160-163）
```

**为什么按 fileSchema 缓存而非"只算一次"**：`clipParquetSchema`/`buildFieldsList` 涉及 schema 解析与字段映射，对同一 Factory 而言 `readFields` 固定、大多数 Paimon 自产文件 `fileSchema` 也相同，缓存即可免算；但外部迁移来的 Parquet 文件 schema 可能不同，用 fileSchema 做 key 才正确——这是防御性设计，拿少量内存换正确性。

**递归列裁剪** `clipParquetSchema`（`:168`）：逐个 readField，文件缺该字段就填 null 列，存在就 `clipParquetType` 递归裁嵌套类型（ROW/MAP/ARRAY），最终 `buildMessage`。这是 Parquet 相对 ORC 的优势——能裁到嵌套字段粒度，只读 `ROW<a,b>` 里的 `a`。

### 5.3 ParquetWriterFactory 与列裁剪/谓词下推

**源码路径**: `paimon-format/.../parquet/ParquetWriterFactory.java`

写入：`StreamOutputFile` 包装 Paimon 的 `PositionOutputStream` 为 Parquet `OutputFile`（屏蔽 close，由上层控关闭时机）；压缩经 `CompressionCodecName.fromConf(compression)`，`"none"`→null。Row Group 默认 128MB（见 `file.block-size` 描述）、Page 默认 1MB。Parquet 支持的压缩：`SNAPPY/GZIP/LZO/BROTLI/LZ4/ZSTD/LZ4_RAW`。

**谓词下推**：`ParquetFileFormat.createReaderFactory` 用 `ParquetFilters.convert(filters)` 把 Paimon `Predicate` 转成 Parquet 的 `FilterCompat.Filter`，运行时用 Row Group 列统计（min/max/null_count）跳过不匹配的 Row Group。与 ORC 不同，Parquet 不需要因 DV 禁用过滤（Paimon 的 DV 行号映射在更上层处理）。谓词下推全链路详见 12 文档。

## 6. Avro 与文本格式（CSV/JSON/Text）

**① 要解决什么问题**

LSM 里有一类文件天生不需要列式格式的好处：L0 数据文件（高频溢写、马上被 compaction 吃掉）、Manifest（行少字段多、每次提交都写）、Changelog。它们要的是"写得快、文件头开销小、能自描述支持 schema 演进"。Avro 这种行式格式正好——直接序列化、无列编码/索引成本。文本格式（CSV/JSON/Text）则是另一类需求：和外部系统导入导出对接。

**② 设计原理与取舍**

Avro 是 Paimon 早期唯一格式，如今退居"写多读少、不分析"的场景：`manifest.format` 默认 avro（`CoreOptions.java:442`）、L0 推荐 `0:avro`、Changelog。它的根本特性是**行式 → 无谓词下推、无文件级列裁剪**：`AvroFileFormat.createReaderFactory`（`AvroFileFormat.java:76`）直接 `new AvroBulkFormat(projectedRowType)`，丢弃 `filters`，投影只在反序列化后裁。统计也只能逐行收集（无内建列统计，详见 §11）。

设计哲学：**Avro 用"放弃读优化"换"极致写简单"，恰好匹配那些写完即焚或频繁小写的文件。**

**Avro 的 ZSTD 特殊处理**（`AvroFileFormat.java:102`）：Avro SDK 的 zstd 必须显式给等级，所以 Paimon 在 `createCodecFactory` 里特判——配了 `avro.codec` 用它，否则压缩名是 "zstd" 时走 `CodecFactory.zstandardCodec(zstdLevel)`，其余走 `CodecFactory.fromString(compression)`。`avro.codec` 默认 snappy，但会被全局 `file.compression`（默认 zstd）覆盖。

**③ 文本格式（CSV/JSON/Text）一览**

三者都是文本行式、都不支持谓词下推、压缩走 Hadoop codec（GZIP/BZIP2/DEFLATE/SNAPPY/LZ4/ZSTD，在流层面应用），区别在支持的类型：

| 格式 | 入口类 | 支持类型 | 典型用途 |
|------|--------|----------|----------|
| CSV | `CsvFileFormat`（csv 包） | 基本类型（不支持复杂类型） | 数据导入导出 |
| JSON | `JsonFileFormat`（json 包） | 全类型（含 ARRAY/MAP/ROW/VECTOR，自描述） | 半结构化数据 |
| Text | `TextFileFormat`（text 包） | 仅单个 STRING 列（`validateDataFields` 强制） | 日志导出、外部对接 |

`TextFileFormat.validateDataFields` 在写入前校验"必须单 STRING 列"，否则抛异常——这是 §3.1 `validateDataFields` 在格式层兜底类型的实例。

**④ 风险/陷阱/边界**

- 分析型大表设 `file.format=avro` → 全扫退化（见 §1.3 陷阱 6）。
- CSV 建表时放 ROW/ARRAY 等复杂列 → `validateDataFields` 拒绝；JSON 则可以。
- Manifest 改列式 → 增加提交开销（见 §1.3 陷阱 7）。

**⑤ 收益与代价**

收益：写路径极轻、文件头开销小、自描述利于 schema 演进与跨系统交换。代价：读取只能全扫、压缩比中等、不适合大数据量分析。所以 Avro/文本格式是"配角"，分析主力始终是 ORC/Parquet。

## 7. 压缩机制：两套体系与代价权衡

**① 要解决什么问题**

存储成本、网络传输、I/O 都是数据湖的硬约束，压缩能在每一项上省钱省时间。但压缩不是越狠越好——高压缩比要花更多 CPU、拖慢写入。LSM 不同层级、不同用途的数据对"压缩比 vs 速度"的偏好完全不同：热数据/L0 要快，冷数据要省空间；文件格式内部的数据块要配合索引结构压，而 Paimon 自己 spill 到磁盘的临时数据要的是纯速度。

**② 设计原理与取舍：为什么是两套压缩体系**

Paimon 有**两套互不干扰**的压缩：

| 体系 | 压缩对象 | 由谁实现 | 追求 |
|------|----------|----------|------|
| 文件格式层压缩 | ORC Stripe Stream / Parquet Page / Avro Data Block | 各格式 SDK | 配合列存索引结构，压缩比与读性能 |
| 内部块压缩 | spill 排序缓冲、Lookup 缓存等内部临时数据 | Paimon 自实现（`compression` 包） | 极致压缩/解压速度，无需索引 |

设计哲学：**文件格式层压缩"贴着格式"做、要兼顾索引；内部块压缩"贴着内存/磁盘 IO"做、只要快。两者职责正交，所以分开实现而非复用。** 算法默认都是 **zstd**（`file.compression` 默认 zstd `CoreOptions.java:321`；`CompressOptions.defaultOptions()` = zstd/level1），因为 zstd level 1 速度接近 lz4 但压缩比更高、解压快，是大多数场景的最佳平衡点。

### 7.1 文件格式层压缩

Paimon 的压缩分为两个层次：

**层次一: 文件格式内部压缩 (ORC/Parquet/Avro/CSV/JSON/TEXT)**

每种文件格式有自己的压缩实现：

| 格式 | 支持的算法 | 默认 | 压缩单元 | 备注 |
|:---|:---|:---|:---|:---|
| ORC | NONE, ZLIB, SNAPPY, LZO, LZ4, ZSTD | ZSTD | Stripe 内的 Stream | 由 ORC SDK 实现 |
| Parquet | UNCOMPRESSED, SNAPPY, GZIP, LZO, BROTLI, LZ4, ZSTD, LZ4_RAW | ZSTD* | Page | 由 Parquet SDK 实现 |
| Avro | null, deflate, snappy, bzip2, xz, zstandard | ZSTD* | Data Block | 支持 ZSTD（需指定 level） |
| CSV | NONE, GZIP, BZIP2, DEFLATE, SNAPPY, LZ4, ZSTD | ZSTD* | 整个文件 | 使用 Hadoop 压缩 |
| JSON | NONE, GZIP, BZIP2, DEFLATE, SNAPPY, LZ4, ZSTD | ZSTD* | 整个文件 | 使用 Hadoop 压缩 |
| TEXT | NONE, GZIP, BZIP2, DEFLATE, SNAPPY, LZ4, ZSTD | ZSTD* | 整个文件 | 使用 Hadoop 压缩 |

> *注: Paimon 的 `file.compression` 默认值为 `zstd`，会传递给各格式。Avro 的 `avro.codec` 配置可以覆盖全局设置。CSV/JSON/TEXT 等文本格式通过 `HadoopCompressionUtils` 在流层面应用压缩。

**压缩配置传递链**:
```
CoreOptions.FILE_COMPRESSION ("file.compression", 默认 "zstd")
  → KeyValueFileWriterFactory.FileWriterContextFactory.key2Compress
    → FileWriterContext.compression
      → SingleFileWriter 构造函数
        → FormatWriterFactory.create(out, compression)
          → OrcWriterFactory: CompressionKind.valueOf(compression)
          → ParquetWriterFactory: CompressionCodecName.fromConf(compression)
          → AvroFileFormat: CodecFactory.fromString(compression)
```

### 7.2 内部块压缩

**源码路径**: `paimon-common/src/main/java/org/apache/paimon/compression/`

**层次二: Paimon 内部数据结构压缩**

用于 Spill 到磁盘的排序缓冲区、Lookup 缓存等内部场景：

```
BlockCompressionFactory (接口)
```

**为什么有两套压缩体系**:
- **文件格式压缩**: 由 ORC/Parquet/Avro SDK 实现，压缩的是文件内部的数据块（Stripe Stream/Page/Data Block），配合文件格式的索引结构工作。
- **内部块压缩**: 由 Paimon 自实现，用于内存中的 Block 压缩（如排序溢写到磁盘的临时文件），追求极致的压缩/解压速度，不需要文件格式的索引开销。

```java
}
```

### 7.3 按层级配置压缩算法

**源码路径**: `CoreOptions.java` 第 299-305 行

```java
// file.compression.per.level = '0:lz4,1:zstd'
public static final ConfigOption<Map<String, String>> FILE_COMPRESSION_PER_LEVEL =
    key("file.compression.per.level").mapType().defaultValue(new HashMap<>());
```

在 `KeyValueFileWriterFactory.FileWriterContextFactory` 中实现层级压缩路由：

```java
String defaultCompress = options.fileCompression();          // 全局默认
Map<Integer, String> fileCompressionPerLevel = options.fileCompressionPerLevel();
this.key2Compress = key -> {
    if (key.isChangelog && changelogCompression != null) {
        return changelogCompression;                         // Changelog 单独配置
    }
    return fileCompressionPerLevel.getOrDefault(key.level, defaultCompress);
};
```

**为什么支持按层级配置压缩**:
- **Level 0 (热数据)**: 写入频繁，使用 LZ4 获得最快的压缩/解压速度，牺牲一些压缩比。
- **高层级 (冷数据)**: 读取为主，Compaction 后数据量大，使用 ZSTD 获得更好的压缩比，节省存储空间。

**典型配置**:
```
file.compression.per.level = '0:lz4,1:zstd,2:zstd'
file.compression = zstd                    # 全局默认
file.compression.zstd-level = 1            # ZSTD 压缩等级 (1-9, 越高越慢但压缩比越好)
changelog-file.compression = zstd          # Changelog 单独配置
```

### 7.4 压缩算法选择建议

| 算法 | 压缩比 | 压缩速度 | 解压速度 | 适用场景 |
|:---|:---|:---|:---|:---|
| ZSTD (level 1) | 高 | 较快 | 快 | **默认推荐**, 大多数场景 |
| ZSTD (level 9) | 非常高 | 慢 | 快 | 冷数据归档, 存储空间敏感 |
| LZ4 | 中 | 非常快 | 非常快 | Level 0 热数据, 写入敏感 |
| Snappy | 中 | 快 | 快 | Parquet 默认, 兼容性好 |
| GZIP/ZLIB | 高 | 慢 | 慢 | 特殊兼容性需求 |
| LZO | 中低 | 快 | 快 | 旧系统兼容 |
| BROTLI | 非常高 | 非常慢 | 较快 | 极端空间优化 (仅 Parquet) |

> 上表数值为相对量级，实际取决于数据分布（低基数列压缩比远高于高基数列）。

**④ 风险/陷阱/边界**
- L0 配高 zstd-level → 高频溢写时压缩拖慢写入（见 §1.3 陷阱 3）。`file.compression.per.level='0:lz4'` 让 L0 走 lz4。
- 算法与格式不兼容：BROTLI 仅 Parquet 支持，ORC 不支持。
- per-level 语法必须是 `level:algorithm`（如 `'0:lz4,1:zstd'`），路由逻辑见 §7.3。

**⑤ 收益与代价**
收益：省存储、省 I/O、按层级精调省/快两全。代价：压缩吃 CPU，配错（如全表 zstd level 9）会让写入 CPU 成为瓶颈。原则：写敏感/热数据偏 lz4，读敏感/冷数据偏 zstd（必要时调高 level）。

## 8. KeyValue 写入工厂：格式/压缩/Thin Mode 路由

**① 要解决什么问题**

`KeyValueFileWriterFactory`（paimon-core 的 `io` 包）是格式层与 LSM 写路径之间的"调度中枢"。上层 MergeTreeWriter 写一个 level 的数据/changelog，不应关心"这层用什么格式、什么压缩、统计怎么收、文件名怎么生成、要不要 Thin Mode"——这些都被这个工厂按 `(level, isChangelog)` 一键路由。

**② 设计原理与取舍**

核心是内部类 `FileWriterContextFactory` 用三个函数把 `WriteFormatKey(level,isChangelog)` 映射到格式/压缩/统计模式，并缓存 `FormatWriterFactory`、`DataFilePathFactory`：

- **key2Format / key2Compress / key2Stats**（`KeyValueFileWriterFactory.java:304/315/326`）：都遵循"changelog 优先用 changelog 专属配置，否则按 level 查 per-level，再落全局默认"的同一套规则。这就是 §3.3、§7.3 里 per-level 配置的落地点。
- **缓存** `format2WriterFactory` / `format2PathFactory`：同一 `(level,isChangelog)` 共享工厂，避免反复创建。
- **Thin Mode**（`:296`）：`dataFileThinMode() && supportsThinMode(keyType, valueType)` 才启用，写入时省略 key 列（key 是 value 子集时冗余），读取时从 value 重建 key。

设计哲学：**把"按层级的格式/压缩/统计/Thin"全部收敛到一个 `WriteFormatKey → 配置` 的查表 + 缓存里，让写路径只管写、不管选。**

## 9. KeyValue 读取工厂：裁剪/下推/DV 落到格式层

**① 要解决什么问题**

读路径要把"上层想要的列与谓词"翻译成"格式层能执行的列裁剪与下推"，还要处理三件麻烦事：schema 演进（文件是旧 schema，当前表是新 schema）、Deletion Vector（行级删除）、大文件读取阻塞。`KeyValueFileReaderFactory`（paimon-core 的 `io` 包）就是这层翻译器。

**② 设计原理与取舍**

- **列裁剪 + 谓词下推的入口**：工厂把 `projectedRowType` 和 `filters` 传给 `FormatReaderFactory.createReaderFactory`（§3.1），ORC/Parquet 据此只读需要的列、跳过不匹配的块；Avro/文本格式则忽略 filters。
- **`FormatReaderMapping` 缓存**（`KeyValueFileReaderFactory.java:71/139`）：按 `FormatKey(schemaId, formatIdentifier)` 缓存"文件 schema → 当前 schema"的映射（列重排 `indexMapping` + 类型转换 `castMapping` + 分区列补充），避免每个文件重算。
- **异步预取仅 ORC**（`:120`）：`file.fileSize() >= asyncThreshold && file.fileName().endsWith(".orc")` 才用 `AsyncRecordReader`（Pool size=2 预取下一批）。Parquet 不走异步（见 §1.3 陷阱 5）。
- **DV 装饰器**：`ApplyDeletionVectorReader` 包装底层 reader，在文件读取层按行号过滤被删行，对上层透明。

设计哲学：**读工厂把"裁剪/下推交给格式层、schema 演进交给可缓存的 mapping、删除交给装饰器、大文件交给异步"四件事解耦，每件独立可关。**

下面 9.1–9.3 看工厂结构、转换链、DV 应用。

### 9.1 KeyValueFileReaderFactory 结构

**源码路径**: `paimon-core/src/main/java/org/apache/paimon/io/KeyValueFileReaderFactory.java`

```
├── fileIO, schemaManager, schema
```

### 9.2 列裁剪与谓词下推到格式层

**源码路径**: `paimon-core/src/main/java/org/apache/paimon/utils/FormatReaderMapping.java`

读取时的数据转换链：

```
文件中的原始数据 (dataSchema)
  ↓ [FormatReaderFactory 列裁剪 + 谓词下推]
读取到的行 (projectedRowType)
  ↓ [indexMapping: 列重排]
  ↓ [castMapping: 类型转换 (Schema Evolution)]
  ↓ [PartitionInfo: 补充分区列值]
```

- **indexMapping**: 当文件的 schema（旧版本）与当前 schema 的列顺序不同时，通过 index 映射重排列。

- **PartitionInfo**: 分区列的值不存储在数据文件中，而是从分区路径推导后补充到每一行中。

## 10. 文件路径管理与扩展名规则

**① 要解决什么问题**

分布式写入下，不同节点的 Writer 同时往同一个 bucket 写文件，文件名必须全局唯一且无需中心协调；分区表要有清晰的目录层级；冷热分离要支持把数据文件甩到外部存储；读取时还要能仅凭文件名反推出格式（因为同表可能混合格式，见 §3.3）。

**② 设计原理与取舍**

- **UUID + AtomicInteger 双保险**：`newFileName` = `prefix + uuid + "-" + pathCount.getAndIncrement() + extension`（`DataFilePathFactory.java:139`）。UUID 区分不同 Writer 实例（不同 Task），AtomicInteger 区分同一 Writer 的多个文件——组合即全局唯一，无需协调。
- **扩展名规则**（`makeExtension`，`DataFilePathFactory.java:116`）：默认只带格式后缀 `.orc`/`.parquet`（压缩信息在文件内部）；文本格式（csv/json，`isTextFormat :211`）压缩后缀放最后 `.json.zst`（需外部 codec 解压）；开 `file.suffix.include.compression`（默认 false，`CoreOptions.java:371`，**已修正：旧稿写成 `file-suffix-include-compression`**）则变 `.zst.orc`。
- **格式反推**：`formatIdentifier(fileName)`（`:188`）从文件名取格式，读取时据此选 `FileFormat`，这正是同表混合格式安全的前提。
- **External Path**：配了 `ExternalPathProvider` 时，数据文件走 `getNextExternalDataPath`（`:134`，轮询/权重策略），元数据仍留表路径。

设计哲学：**文件名自带"身份（UUID）+ 序号 + 格式后缀"三件信息，写时保证不撞、读时无需配置即可识别格式。**

下面 10.1–10.3 看根工厂目录布局、数据文件工厂、路径计算规则。

- 对文本格式（CSV/JSON），压缩扩展名始终包含，因为这些格式需要外部工具（如 Hadoop Codec）解压。

## 11. 文件元数据收集：行级收集 vs 文件级提取

**① 要解决什么问题**

数据跳过（data skipping）依赖每个文件的列统计（min/max/nullCount）——它们会被写进 Manifest，scan 时据此裁掉整个文件（见 12 文档的多级裁剪）。问题是怎么"低成本"拿到这些统计：行式格式（Avro）没有内建统计，列式格式（ORC/Parquet）的 Footer 里现成就有。而且对象存储上文件刚写完可能还读不到，不能"写完再 reopen 算统计"。

**② 设计原理与取舍：两条收集路径 + 一个对象存储优化**

| 路径 | 适用格式 | 做法 | 代价 |
|------|----------|------|------|
| `SimpleStatsCollector`（行级收集） | Avro/文本 | 写入时逐行 `collect(row)` 更新各列 min/max/null | 每行都算，大文件开销大 |
| `SimpleStatsExtractor`（文件级提取） | ORC/Parquet | 写完从 Footer/ColumnStatistics 读现成统计 | 几乎零额外成本 |

二者由 `SimpleStatsProducer` 统一接口对接写入器（见 §8.3）。**选哪条由 `FileFormat.createStatsExtractor` 是否返回提取器决定**（§3.1）：ORC/Parquet 返回提取器走文件级，Avro 不返回走逐行收集。

**对象存储优化 `writerMetadata`**（`SimpleStatsExtractor.java`）：提供 `extract(fileIO, path, length, writerMetadata)` 重载，直接从写入器内存元数据取统计，绕开"reopen 文件"——这正是 S3 写后可见延迟的对策（见 §1.3 陷阱 8）。

**isDisabled 优化**：若所有列都用 `NoneSimpleColStatsCollector`（不收集），标 `isDisabled=true` 跳过逐行循环；统计模式 `none/counts/truncate(默认)/full` 控制收集精度与开销，truncate 截断长字符串避免 min/max 把 Manifest 撑大。

下面 11.1–11.3 看行级收集器、文件级提取接口、ORC/Parquet 的提取差异。

### 11.1 SimpleStatsCollector 行内收集

**源码路径**: `paimon-common/src/main/java/org/apache/paimon/format/SimpleStatsCollector.java`

SimpleStatsCollector 逐行收集每列的统计信息：

```java
public class SimpleStatsCollector {
    private final RowDataToObjectArrayConverter converter;
    private final SimpleColStatsCollector[] statsCollectors; // 每列一个
    private final Serializer<Object>[] fieldSerializers;
    
    public void collect(InternalRow row) {
        Object[] objects = converter.convert(row);
        for (int i = 0; i < row.getFieldCount(); i++) {
            statsCollectors[i].collect(objects[i], fieldSerializers[i]);
        }
    }
    
    public SimpleColStats[] extract() {
        SimpleColStats[] stats = new SimpleColStats[statsCollectors.length];
        for (int i = 0; i < stats.length; i++) {
            stats[i] = statsCollectors[i].result();
        }
        return stats;
    }
}
```

**SimpleColStats** 包含: `minValue`, `maxValue`, `nullCount`

**为什么有 isDisabled 检查**: 某些列使用 `NoneSimpleColStatsCollector`（不收集统计），如果全部列都是 None，标记 `isDisabled=true`，写入时跳过逐行收集以提升性能。

### 11.2 SimpleStatsExtractor 文件级提取

**源码路径**: `paimon-common/src/main/java/org/apache/paimon/format/SimpleStatsExtractor.java`

```java
public interface SimpleStatsExtractor {
    // 从文件元数据提取统计
    SimpleColStats[] extract(FileIO fileIO, Path path, long length) throws IOException;
    
    // 带 writerMetadata: 避免重新读取文件 (对象存储可能文件还不可见)
    default SimpleColStats[] extract(FileIO fileIO, Path path, long length, 
            @Nullable Object writerMetadata) throws IOException {
        return extract(fileIO, path, length);
    }
    
    // 提取统计 + 文件信息 (rowCount)
    Pair<SimpleColStats[], FileInfo> extractWithFileInfo(FileIO fileIO, Path path, long length);
}
```

**为什么有 writerMetadata 参数**: 在对象存储（S3）上，文件关闭后可能存在短暂的"不可见"窗口。`writerMetadata` 允许从写入器的内存元数据直接提取统计，避免重新读取文件。

### 11.3 ORC 与 Parquet 的统计提取差异

**ORC SimpleStatsExtractor**:
```
读取 Reader.getStatistics()
  → ColumnStatistics[] (每列)
    → IntegerColumnStatistics: min, max
    → StringColumnStatistics: min, max
    → DecimalColumnStatistics: min, max
    → BooleanColumnStatistics: trueCount, falseCount
    → TimestampColumnStatistics: min, max
    → DateColumnStatistics: min, max
```

**Parquet SimpleStatsExtractor**:
```
读取 ParquetMetadata (Footer)
  → BlockMetaData (每个 Row Group)
    → ColumnChunkMetaData → Statistics
      → IntStatistics / LongStatistics / FloatStatistics / ...
      → 合并多个 Row Group 的统计
```

**关键差异**:

| 特性 | ORC | Parquet |
|:---|:---|:---|
| 统计粒度 | 文件级 (所有 Stripe 合并) | Row Group 级 (需合并) |
| Null Count | 从 `getNumberOfValues()` 推导 | 从 `getNumNulls()` 直接获取 |
| 嵌套类型统计 | 不支持 | 不支持 |
| 对象存储友好性 | 需要重新打开文件 | 需要重新打开文件 |

## 12. FileIO 抽象与原子提交

**① 要解决什么问题**

格式层负责"文件内部长什么样"，FileIO 层负责"文件落到哪种存储、怎么原子地出现"。对象存储（S3/OSS）与文件系统（HDFS/Local）的物理差异是本质的——不能原地改、rename 是 copy+delete（非原子）、写后有可见延迟——而 Paimon 的快照/schema 提交都需要"一个新文件原子地、要么完整出现要么不出现"。FileIO 把这套差异关进实现里，让上层无感；同时它必须可序列化，以便随作业分发到每个 Task。

**② 设计原理与取舍**

`FileIO`（`FileIO.java:66`，paimon-common）标注 `@Public @ThreadSafe`、`extends Serializable, Closeable`，关键方法：`isObjectStore()`（`:70`，让上层据存储类型选提交策略）、`newInputStream/newOutputStream`、`newTwoPhaseOutputStream`（`:111`）、`tryToWriteAtomic`（`:330`）、`readOverwrittenFileUtf8`（`:402`，带重试读元数据）。

分层取舍：`Path` 放在 `paimon-api`（纯路径、零依赖），`FileIO` 放在 `paimon-common`（要依赖 Hadoop 等），保持 api 层纯净。设计哲学：**用统一接口 + 各实现内部兜底，把"原子地让文件出现"在每种存储上各自做对——HDFS 靠原子 rename、对象存储靠 Multi-Part Upload 的 complete。**

### 12.1 FileIO 接口与实现谱系

| 实现 | 存储 | 原子性手段 | 要点 |
|------|------|-----------|------|
| `LocalFileIO` | 本地 | `Files.move(ATOMIC_MOVE)` + 全局 `RENAME_LOCK` | 单例；rename 前的"检查目标存在 + move"靠全局锁保证 check-then-move 原子；`SeekableInputStream` 实现 `VectoredReadable` 支持 pread |
| `HadoopFileIO` | HDFS 及兼容 | 三参数 `rename(src,dst,OVERWRITE)`（反射嗅探） | FileSystem 按 `(scheme,authority)` 缓存；小于 1MB 前向 seek 用 skip 代替；仅 `DistributedFileSystem`/`ViewFileSystem` 支持原子覆盖，对象存储优雅降级 |
| `PluginFileIO` | S3/OSS/OBS… 插件 | 委托底层 | 每操作 `wrap()` 切换到插件 ClassLoader；实际实现 `transient` 不序列化，反序列化后在目标节点重建 |
| `ResolvingFileIO` | 多协议 | 按 scheme 路由 | 数据在 S3、元数据在 HDFS 等场景；`CatalogOptions.RESOLVING_FILE_IO_ENABLED` 默认关 |

### 12.2 原子写入与两阶段提交流

**`tryToWriteAtomic` 默认实现**（`FileIO.java:330`，已核对）：先写临时文件 `createTempPath()`，再 `rename(tmp, path)`，失败则 `deleteQuietly`。这套在 HDFS/Local 上靠原子 rename 成立，在对象存储上 rename 是 copy+delete，需改走两阶段流。

**TwoPhaseOutputStream**（`:111` 默认返回 `RenamingTwoPhaseOutputStream`）：数据先写临时位置，`closeForCommit()` 返回 `Committer`，`commit(FileIO)` 才让数据可见、`discard` 丢弃。两种实现：

- `RenamingTwoPhaseOutputStream`：HDFS/本地，写 `_temporary` 临时文件 + commit 时 rename。
- `MultiPartUploadTwoPhaseOutputStream`：对象存储，原生 Multi-Part Upload 分片（`partSizeThreshold()`，`MultiPartUploadTwoPhaseOutputStream.java:69`）上传，complete 操作让文件原子可见——比 rename 可靠。

**为什么对象存储不用 rename**：copy+delete 中间态不一致，且无原子保证；Multi-Part Upload 的 complete 是服务端原子操作，是对象存储上的正解。

### 12.3 文件系统插件与 SPI 路由

存储插件用**双层机制**：SPI 层（`FileIOLoader`，轻量，主 classpath）负责发现；PluginLoader 层（从 `paimon-plugin-<fs>` 目录加载重量级实现到独立 ClassLoader）负责依赖隔离——不同插件可依赖不同版本的库而不冲突。`FileIO.get(path, config)` 路由优先级：① `preferIO`（用户指定）→ ② `ServiceLoader` 发现的 `FileIOLoader`（按 scheme 匹配 + 校验 `requiredOptions`）→ ③ `fallbackIO`（计算引擎提供的 FileSystem）→ ④ `HadoopFileIOLoader` 兜底，全失败抛 `UnsupportedSchemeException`。

支持的存储：S3（`paimon-s3`/`S3Loader`）、OSS、OBS、Azure（abfs/abfss）、GCS、COS、Jindo/JindoDLS，HDFS 内置（`HadoopFileIOLoader`）。每个插件分 `paimon-<fs>`（Loader + PluginFileIO 子类，SPI 注册）与 `paimon-<fs>-impl`（实际实现，PluginLoader 隔离）两个子模块。

**陷阱**：插件目录必须是 `paimon-plugin-<fs>` 命名、放 plugin 目录而非 lib（否则不隔离）；S3 等 `requiredOptions` 缺 access/secret-key 会在加载阶段就报错。

## 13. 设计决策总结

| 决策点 | 选择 | 取舍/代价 | 收益 |
|--------|------|-----------|------|
| 格式抽象位置 | 抽象在 common、实现在 format、SPI 串联 | 多一层 SPI 查表 | 加新格式零侵入核心 |
| 默认格式 | Parquet（`file.format` 默认） | 与 ORC 比少了 Paimon 深度集成（异步/零拷贝） | Spark 互通最成熟、嵌套裁剪强 |
| 按层级选格式 | `file.format.per.level`（L0 行式、高层列式） | 同表混合格式、运维认知成本 | 写读两端各取最优 |
| ORC 读优化 | Pool 复用 + 零拷贝向量 + 大文件异步预取 | 向量引用不能逃逸批次、异步内存翻倍 | 列式读内存/GC 成本最低 |
| DV 与行组过滤 | 开 DV 时禁 ORC 行组过滤 | 失去行组级精筛 | 行级删除可行（Stripe 裁剪仍在） |
| 两套压缩体系 | 格式层压缩 + 内部块压缩分离 | 两套代码 | 各自贴合场景，职责正交 |
| 默认压缩算法 | zstd（level 1） | 比 lz4 略慢 | 压缩比/速度最佳平衡 |
| Thin Mode | key 是 value 子集时省 key 列 | 读取需从 value 重建 key | 省存储 |
| 统计收集 | Avro 逐行收集 / ORC·Parquet 文件级提取 | 行式收集开销大 | 各格式都能产出 data skipping 统计 |
| FileIO 抽象 | 统一接口 + 实现内兜底 + 可序列化 | 牺牲部分存储特性 | 上层对存储无感、可分发 |
| 原子提交 | HDFS rename / 对象存储 Multi-Part Upload | 对象存储不能用 rename | 各存储上都能原子地让文件出现 |
| 存储插件 | SPI 发现 + PluginLoader ClassLoader 隔离 | 部署要正确放 JAR | 多云存储、依赖不冲突 |

### 13.1 ORC vs Parquet 速查

| 维度 | ORC | Parquet（**Paimon 默认**） |
|:---|:---|:---|
| 列存组织 | Stripe（默认 64MB）→ Stream → RLE | Row Group（默认 128MB）→ Column Chunk → Page |
| 谓词下推 | SearchArgument（Stripe + Row Group） | FilterCompat（Row Group） |
| 向量化读取 | VectorizedRowBatch → Paimon ColumnVector（零拷贝） | 自实现 VectorizedParquetRecordReader |
| 写入内存 | 估算式（`estimateMemory * 0.2`） | Row Group 缓冲 |
| 异步读取 | 支持（Pool size=2） | 不支持（Pool size 固定 1） |
| Deletion Vector | 兼容（禁用行组过滤） | 兼容 |
| 嵌套类型裁剪 | 需类型转换 | 原生递归裁剪 |
| 生态 | Hive 优先 | Spark 优先 |

> 默认压缩两者在 Paimon 里都由 `file.compression`（默认 zstd）传入；ORC/Parquet 各自的 SDK 默认值会被覆盖。

**选格式速查**：默认/Spark 互通 → Parquet；Hive 互通/想要异步预取与零拷贝深度集成 → ORC；L0 高频溢写 → Avro；Manifest → Avro（默认）。

**推荐混合配置**（实时数仓）：

```sql
'file.format' = 'parquet',                       -- 默认 Parquet（或 orc 以深度集成）
'file.format.per.level' = '0:avro',              -- L0 行式快写
'file.compression' = 'zstd',                     -- 默认 zstd
'file.compression.per.level' = '0:lz4,1:zstd',   -- L0 lz4 快压
'file.compression.zstd-level' = '1',             -- 速度优先
'manifest.format' = 'avro'                        -- Manifest 默认 avro
```
