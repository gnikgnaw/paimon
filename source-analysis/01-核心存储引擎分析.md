## 1. 快速理解（核心问题 / 概念 / 陷阱）

### 1.1 核心问题：为什么是 LSM 而不是 B+Tree

**① 要解决什么问题**

实时湖仓要在对象存储（S3/OSS/HDFS）上同时满足三个互相冲突的需求：高吞吐流式 upsert、主键更新/删除语义、PB 级规模的 ACID 快照隔离。传统 Hive 只能追加，传统数据库扛不住规模且依赖本地随机写，对象存储又不支持原地修改。

**② 设计原理与取舍**

对象存储的物理特性决定了一切：**只能顺序追加，不能原地改写，随机写极慢**。LSM-Tree 天然契合——所有写入都是追加新文件，更新通过"写新版本 + 后台合并"完成，把随机写转成顺序写。代价是读取需要跨多层归并（读放大），但读放大可以靠后台异步 compaction 持续收敛，而写入吞吐是数据湖的硬瓶颈。

| 维度 | B+Tree | LSM-Tree（Paimon） |
|------|--------|-------------------|
| 写入模式 | 随机写、原地更新 | 顺序写、追加新文件 |
| 对象存储适配 | 差（不支持原地改） | 优（纯顺序写） |
| 写放大 | 高（读-改-写整页） | 低（批量写 + 异步合并） |
| 读放大 | 低（一次定位） | 中（需归并多层） |
| 适用场景 | 读多写少 | 写多、流式更新 |

核心哲学一句话：**用可控的、可异步收敛的读放大，换取极低的写放大。**

与同类湖格式横向对比（理解 Paimon 定位的关键）：

| 方案 | 写入模式 | 更新支持 | 对象存储适配 | 典型更新延迟 |
|------|----------|----------|--------------|--------------|
| Hive | 批量追加 | 不支持 | 优 | 离线批 |
| Delta Lake | Copy-on-Write | 支持（重写整文件） | 中（写放大高） | 小时级 |
| Iceberg | CoW / MOR | 支持 | 中 | 小时级 |
| Hudi | CoW / MOR | 支持 | 中 | 分钟级 |
| **Paimon** | **LSM Merge-Tree** | 支持（增量合并） | **优（纯顺序写）** | **秒级流式** |

Paimon 的差异点：更新一条记录时 Delta/Iceberg CoW 需重写整个数据文件（可能 GB 级），Paimon 只追加一条新版本记录，后台 compaction 再异步合并——这是写放大数倍差距的根源。

**一条记录的全生命周期（串起全文各章）**：以主键表 upsert 一行 `(pk=42, v=...)` 为例——

```
1. TableWrite.write(row)        路由到 (partition, bucket=hash(42)%N)（§10.2），转成 KeyValue（§2）
2. MergeTreeWriter.write(kv)    分配 sequenceNumber，put 进 WriteBuffer（§4.1）
3. WriteBuffer 满 → flush       排序 + MergeFunction 合并同 key（§3.4/§4.2），落成 Level 0 文件（§3.1）
                                INPUT 模式同时双写 changelog（§8.6）
4. compaction 异步触发          UniversalCompaction.pick 选 run（§8.1），逐级推向高层
                                小文件 rewrite、大文件 upgrade 只改 level（§8.3）；MOW 顺带生成 DV（§7.2）
5. prepareCommit → commit       六增量打包 CommitIncrement（§4.3），APPEND+COMPACT 两阶段提交（§13）
                                ConflictDetection 校验、原子重命名生成 Snapshot（§12.2）
6. 读取该 pk                    Scan 三级裁剪定位文件（§12.4）→ Lookup 点查（§6）或 SortMergeReader 归并（§5.3）
                                MOW 用 DV 按行过滤、MOR 全量 merge、COW 几乎免 merge（§7.1）
7. TTL / compaction             record-level expire 搭车清理（§8.4）；snapshot/partition expire 回收
```

这条链贯穿全文：写快（顺序追加 + 异步 compaction）、读对（多级不变式 + 归并）、省（双清单增量 + upgrade 免重写）、可控（反压 + 内存抢占）。

### 1.3 六个高频生产陷阱

**陷阱 1：bucket 数配置不当。** 每个 bucket 是一棵独立 LSM 树，桶数同时决定写入并行度和单桶数据量。过少 → 单桶巨大、compaction 压力集中；过多 → 小文件泛滥、元数据爆炸。经验：单桶 1–5GB，主键表 `bucket ≈ 单分区预估数据量 / 2GB`；无法预估用 `bucket=-2`（POSTPONE）让系统自适应。

**陷阱 2：`num-sorted-run.stop-trigger` 过大导致 Level 0 堆积。** Level 0 文件 key 范围重叠，读取要逐个归并，文件数从 10 涨到 100，归并路数同步翻 10 倍。默认值是 `num-sorted-run.compaction-trigger + 3`（默认 5+3=8，见 `CoreOptions.numSortedRunStopTrigger()` 在 `CoreOptions.java:3068`，**已修正：旧稿误称默认 10**）；调大可提升写吞吐但查询更吃内存/CPU；Lookup 场景应配 `lookup-compact=radical` 强制清空 L0。

**陷阱 3：`write-buffer-size` 过小导致频繁 flush。** WriteBuffer 是内存排序缓冲，过小则每次 flush 出的文件很小、Level 0 增长快、compaction 压力大。默认 256MB；内存紧张开 `write-buffer-spillable=true` 允许溢写磁盘；监控 `bufferPreemptCount`（抢占次数），频繁抢占说明内存不足。

**陷阱 4：跨分区更新未用 KEY_DYNAMIC。** 若主键不含全部分区字段（如按日期分区、主键仅 `user_id`），用 HASH_FIXED/HASH_DYNAMIC 会在新旧两个分区都写一份造成重复。判定逻辑见 `KeyValueFileStore.java:97`：`bucket=-1` 且 `crossPartitionUpdate` 时自动选 KEY_DYNAMIC。代价是启动需读全表主键建索引。

**陷阱 5：`changelog-producer=full-compaction` 未配触发间隔。** 该模式下 changelog 只在全量压缩时产生，若仅靠 size-amplification（默认 200%）触发，可能数小时才出一次 changelog。需配 `optimized-compaction-interval`（走 EarlyFullCompaction），或改用 `input`/`lookup` 模式获得更低延迟。

**陷阱 6：Lookup 场景未启用远程 lookup 文件。** LookupLevels 默认从 Parquet/ORC 现场构建本地索引，GB 级文件构建耗时数分钟，Flink failover 后每次重建。开 `lookup.remote-file.enabled=true`，compaction 时顺带生成 `.lookup` 文件并上传，Task 启动直接下载。

**陷阱 7：误用 `num-sorted-run.compaction-trigger` 同时影响 compaction 频率、层数与归并内存。** 这一个参数有三重作用：① 触发 compaction 的 SortedRun 阈值（默认 5）；② 默认层数 = `trigger + 1`（`numLevels()` 在 `CoreOptions.java:3076`）；③ compaction 时归并的 run 数（write-performance 文档明确指出归并内存随它增长）。调大它会让 compaction 更少触发（写更快），但每次归并更多 run（峰值内存更高）、查询时 run 更多（读更慢）。量化：从 5 调到 10，写吞吐可能 +20% 但单次 compaction 内存翻倍、点查延迟上升。

**陷阱 8：MOW（deletion-vectors）下误开异步 compaction 致数据延迟可见。** MOW 模式 Level 0 文件需等 compaction 生成 deletion vector 后才对读可见（官方 table-mode 明确："files with level 0 will only be visible after compaction"，源码上 DV 在 compaction 阶段由 `BucketedDvMaintainer` 生成）。默认 compaction 同步；若强行异步，新写数据会有可见延迟。排障：发现 MOW 表"刚写的数据查不到"，先查是否把 compaction 改成了纯异步。

## 2. 总体架构：FileStore 三层体系

**① 要解决什么问题**

如何让 Flink/Spark 等计算引擎只依赖一个稳定接口，而把"主键表 LSM"与"追加表纯追加"两套截然不同的存储逻辑、以及 snapshot/manifest/commit 等横切逻辑解耦复用。

**② 设计原理与取舍**

采用模板方法模式的三层结构：

```
FileStore<T>              接口：scan/read/write/commit + 快照/清单/桶模式全部契约
   ↑
AbstractFileStore<T>      抽象类：pathFactory / snapshotManager / manifest 工厂 / newCommit 等公共逻辑
   ↑
KeyValueFileStore (T=KeyValue, LSM Merge-Tree)   |   AppendOnlyFileStore (T=InternalRow, 仅追加)
```

`AbstractFileStore` 承担约 60% 的公共逻辑（路径、快照管理、清单工厂、提交器、各类 deletion/expire/tag 管理），子类只需实现 `newScan/newRead/newWrite/bucketMode` 等抽象方法。新增存储模式（如未来列式引擎）只继承抽象类即可，不影响上层。

**③ 关键源码**

`FileStore<T>` 接口的核心方法（`FileStore.java:61-95`）：

```java
public interface FileStore<T> {
    FileStoreScan newScan();                                   // :77
    SplitRead<T> newRead();                                    // :89
    FileStoreWrite<T> newWrite(String commitUser);             // :91
    FileStoreWrite<T> newWrite(String commitUser, @Nullable Integer writeId); // :93
    FileStoreCommit newCommit(String commitUser, FileStoreTable table);       // :95
    BucketMode bucketMode();
    // … snapshotManager / manifestFileFactory / manifestListFactory / 各类 deletion、expire、tag 管理
}
```

KeyValue 行模型把版本号与操作类型嵌入数据行（`KeyValue.java:42-45`）：

```java
public class KeyValue {
    public static final long UNKNOWN_SEQUENCE = -1;   // :44
    public static final int  UNKNOWN_LEVEL    = -1;   // :45
    private InternalRow key;
    private long sequenceNumber;   // write() 时分配，参与排序键
    private RowKind valueKind;     // INSERT / UPDATE_BEFORE / UPDATE_AFTER / DELETE
    private InternalRow value;
    private int level;             // 读取时由文件元数据填充
}
```

物理 schema：`[key fields…] + _SEQUENCE_NUMBER(BIGINT) + _VALUE_KIND(TINYINT) + [value fields…]`（`KeyValue.schema()` 在 `KeyValue.java:113`，字段拼装见 `createKeyValueFields`，系统字段 `SEQUENCE_NUMBER`/`VALUE_KIND` 引自 `SpecialFields`）。

**④ 风险 / 陷阱 / 边界**

- 不要强转具体实现类（`(KeyValueFileStore) store`），应面向 `FileStore<?>` 编程，否则无法适配未来引擎。
- 区分 `FileStore`（存储引擎）与 `FileStoreTable`（表抽象层）：表级能力（schema evolution、分区裁剪、ReadBuilder）在 `FileStoreTable`，直接用 `FileStore` 会丢失这些。

**⑤ 收益与代价**

收益：存储与计算彻底解耦，横切逻辑单点维护。代价：多一层抽象，理解成本略高；每行多出约 9 字节（8B sequenceNumber + 1B valueKind）的元数据开销，小行场景占比偏高，但换来 merge 可完全基于 `(key, sequenceNumber)` 流式完成、无需外部索引。

**AbstractFileStore 承担的横切逻辑（子类不重复实现）**：路径工厂 `pathFactory()`（拼 `FileStorePathFactory`，处理外部路径策略 `ExternalPathStrategy`：NONE/SPECIFIC_FS/ROUND_ROBIN）、`snapshotManager()`（注入 snapshot 缓存）、`manifestFileFactory()`/`manifestListFactory()`（配压缩与 manifest 缓存）、`newCommit()`（装配 `FileStoreCommitImpl` + ConflictDetection + 回调链）、各类 `newSnapshotDeletion()`/`newChangelogDeletion()`/`newTagManager()`/`newPartitionExpire()`。原子提交器在 `AbstractFileStore` 里选择：优先 `catalogEnvironment.snapshotCommit()`（REST/Hive catalog 提供的带锁提交），否则回落 `RenamingSnapshotCommit`（文件系统原子重命名）。子类（KeyValue/AppendOnly）只填四个抽象方法：`newScan / newRead / newWrite / bucketMode` 以及 `newKeyComparator`——这是模板方法模式的"可变步骤"。

**完整调用链（计算引擎 → 存储引擎）**：
```
FileStoreTable（表抽象：schema evolution、分区裁剪、ReadBuilder/WriteBuilder）
   ├─ newReadBuilder() → ReadBuilder → TableScan(产 Split[]) → TableRead(读 Split)  →  store.newScan()/newRead()
   └─ newBatchWriteBuilder()/newStreamWriteBuilder() → TableWrite → TableCommit       →  store.newWrite()/newCommit()
        ↓
FileStore<T>（存储引擎：纯 LSM/manifest/snapshot，无表语义）
```
计算引擎只碰 `FileStoreTable`/`ReadBuilder`，绝不直接碰 `FileStore`——后者无 schema evolution、无分区裁剪 builder。这层划分是"为什么不能直接强转 FileStore 用"的根本原因。

## 3. LSM 数据结构：Level / SortedRun / Levels

### 3.1 Level 全景：什么是 Level 0，有几层

**Level 是数据文件的"年龄层级"标签。** 新写入从 Level 0 开始，经 compaction 逐级晋升至 maxLevel。每个 `DataFileMeta` 自带 `level` 字段。

默认层数 = `num-sorted-run.compaction-trigger + 1`（默认 6，即 Level 0–5）。配置见 `CoreOptions.java:773`（`num-levels`，无默认值）；默认计算见 `CoreOptions.java:3076`（**已修正行号**，旧稿标 2995）：

```java
public int numLevels() {                                  // CoreOptions.java:3076
    Integer numLevels = options.get(NUM_LEVELS);
    if (numLevels == null) {
        numLevels = MathUtils.incrementSafely(numSortedRunCompactionTrigger());
    }
    return numLevels;
}
```

**Level 0 是唯一形态不同的层**——它直接接收内存 flush 的产物，文件间 key 范围可能重叠：

| 维度 | Level 0 | Level 1 ~ maxLevel |
|------|---------|--------------------|
| 来源 | WriteBuffer flush 落盘 | compaction 合并产物 |
| 文件间 key 范围 | **可能重叠** | **绝不重叠** |
| 每层 SortedRun 数 | 多个（每文件一个） | 1 个（整层一个） |
| 存储结构 | `TreeSet<DataFileMeta>`（按 maxSeq 降序） | `SortedRun`（按 minKey 升序） |
| 查找方式 | 遍历（新鲜度优先） | 二分查找 |

flush 产物固定落到 Level 0（`MergeTreeWriter.java:220-221`，`createRollingMergeTreeFileWriter(0, ...)`）；写入 Level 0 强制校验 `file.level()==0`（`Levels.java:109-112`）。

晋升流程一图记忆：

```
WriteBuffer（内存排序缓冲）
     │ flushWriteBuffer → createRollingMergeTreeFileWriter(0, APPEND)
     ▼
Level 0   [file_a] [file_b] [file_c] …   ← TreeSet 按 maxSeq 降序、key 范围可重叠、每文件一个 SortedRun
     │ UniversalCompaction.pick() 选中部分 run，createUnit() 决定输出层
     ▼
Level 1   [minK..maxK][minK..maxK][minK..maxK]   ← 整层一个 SortedRun，files[i].maxKey < files[i+1].minKey
     │ 逐级 compaction 推进（输出到 next_run.level - 1）
     ▼
Level 2 … maxLevel   ← 冷数据归宿，全量 compaction 的输出层
```

**晋升的输出层**由 `UniversalCompaction.createUnit()` 决定（`UniversalCompaction.java:196-226`）：选中全部 run → 输出 maxLevel；否则输出 `next_run.level - 1`；**绝不输出到 Level 0**，若算得 0 则向后扩展直至命中非 0 层。逐级推进（而非一步到 maxLevel）的设计意图：把合并结果"填进下一层之前的空隙"，避免一次性把数据推到最底层造成与底层全量数据的反复重写——这正是 Universal Compaction 在写放大上的关键节流阀。

**单文件 upgrade 优化**：无 key 重叠的大文件晋升时只改元数据 level 字段、不重写（`AbstractCompactRewriter.java:33-36`）：

```java
public CompactResult upgrade(int outputLevel, DataFileMeta file) throws Exception {
    return new CompactResult(file, file.upgrade(outputLevel));   // 仅改 metadata.level
}
```

分层的本质收益：写入快（L0 不要求有序，直接落盘）、读取归并路数可控（`L0 文件数 + 非空高层数`）、空间放大随逐级 compaction 收敛、写放大因只选择性合并而受控。

### 3.2 Levels：层级的内存视图

**① 要解决什么问题**：在内存中高效维护一棵 LSM 树的全部文件，支持频繁增删、最新优先查找、以及反压/compaction 所需的 SortedRun 计数。

**② 设计原理与取舍**：Level 0 用 `TreeSet` 而非 `List`——文件频繁增删（每 flush 增、每 compaction 减），TreeSet 提供 O(log n) 增删并自动按新鲜度排序；Level 1+ 每层维护单个 `SortedRun`，保证"某 key 在每个高层最多落在一个文件"，从而二分查找 O(log F) 定位、每层最多贡献一路归并。

**③ 关键源码**

Level 0 的多级比较器（`Levels.java:59-83`）——降序为主，多 Job 并发同写时靠 minSeq/creationTime/fileName 兜底避免 TreeSet 去重丢文件：

```java
this.level0 = new TreeSet<>((a, b) -> {
    if (a.maxSequenceNumber() != b.maxSequenceNumber()) {
        return Long.compare(b.maxSequenceNumber(), a.maxSequenceNumber());  // 降序：最新在前
    }
    int c = Long.compare(a.minSequenceNumber(), b.minSequenceNumber());     // 并发写兜底
    if (c != 0) return c;
    c = a.creationTime().compareTo(b.creationTime());
    if (c != 0) return c;
    return a.fileName().compareTo(b.fileName());                            // 终极兜底，保唯一
});
```

核心度量 `numberOfSortedRuns()`（`Levels.java:127-135`）——反压与 compaction 触发的核心指标，L0 每文件计 1，高层每非空层计 1：

```java
public int numberOfSortedRuns() {
    int n = level0.size();
    for (SortedRun run : levels) if (run.nonEmpty()) n++;
    return n;
}
```

更新入口 `update(before, after)`（`Levels.java:174-193`）按 level 分组逐层更新，并对"被删但未在 after 中出现"的文件（即真正 drop、排除 upgrade）触发 `DropFileCallback`——这是 LookupFile 缓存失效的钩子。逐层更新 `updateLevel`（`:195-209`）对 L0 直接 `level0.remove/addAll`，对高层则取出整层文件、`removeAll(before)`+`addAll(after)` 后用 `SortedRun.fromUnsorted` 重建（内部重排+validate，保证不变式）。

**restore 时的层数推断**（`Levels.java:53-58`）：构造时 `restoredNumLevels = max(numLevels, 输入文件最大 level + 1)`——即使配置层数偏小，也按实际文件占用的最高层撑开，避免丢层；并校验 ≥ 2（至少 L0 + 一个高层）。`nonEmptyHighestLevel()`（`:137-146`）从高往低找首个非空层，返回 -1（全空）/0（仅 L0 非空）/i+1——它是 §8.3 `dropDelete` 判定"输出层是否已是最底层"的依据。`totalFileSize()`（`:148-151`）= L0 文件大小和 + 各高层 `SortedRun.totalSize` 和，供 metrics 与 EarlyFullCompaction 用。

**④ 风险 / 陷阱 / 边界**：不要直接操作 `level0` TreeSet 或 `levels` List，必须走 `update()`/`addLevel0File()`（后者断言 `file.level()==0`），否则破坏排序不变式导致查找失败。构造时校验 `restoredNumLevels > 1`（至少 2 层）。`update` 末尾文件数一致性 `checkState`（构造时 `:95-98`）确保存入文件数 == 输入文件数，防止 TreeSet 因比较器返回 0 而"去重丢文件"（这正是 L0 比较器要 fileName 兜底的原因）。

**⑤ 收益与代价**：自动维护排序、最新优先命中、O(1) 拿到 SortedRun 数；代价是 TreeSet 比 List 内存开销略高，但 L0 文件数通常 < 100 可忽略。

### 3.3 SortedRun 与 LevelSortedRun

**SortedRun** 不变式：相邻文件 `files[i].maxKey < files[i+1].minKey`，并预计算 `totalSize`。`validate()`（`SortedRun.java:87-94`）构建时校验，`totalSize` 在构造函数中一次性求和（`SortedRun.java:42-49`）。

```java
public void validate(Comparator<InternalRow> comparator) {           // SortedRun.java:87
    for (int i = 1; i < files.size(); i++)
        Preconditions.checkState(
            comparator.compare(files.get(i).minKey(), files.get(i - 1).maxKey()) > 0,
            "SortedRun is not sorted and may contain overlapping key intervals. This is a bug.");
}
```

为什么预计算 `totalSize`：UniversalCompaction 的 size-amp / size-ratio 判定每轮要访问数十次 `run.totalSize()`，预计算把每次访问从 O(F) 降到 O(1)。构建方式：`fromSingle`（L0 单文件）/ `fromSorted`（已排序）/ `fromUnsorted`（内部排序后 validate，`SortedRun.java:63-69`）。

**LevelSortedRun**（`LevelSortedRun.java`）是 `(level, SortedRun)` 的简单包装，让 compaction 策略知道每个 run 来自哪一层以决定输出层。`Levels.levelSortedRuns()`（`Levels.java:162-172`）把整树拍平成 `List<LevelSortedRun>`：L0 每文件一个（level=0），高层每非空层一个（level=i+1）。

### 3.4 MergeFunction 与四种 Merge Engine

**① 要解决什么问题**：同一主键的多个版本（不同 sequenceNumber）在归并时如何合并成一行——这是主键表"upsert 语义"的落点，且必须支持流式增量合并（不缓存该 key 全部版本）。

**② 设计原理与取舍**：`MergeFunction<T>`（`MergeFunction.java:37-53`）是合并算子，契约是 `reset() → add(kv)* → getResult()`：按 `(key, seq)` 有序喂入同一 key 的所有版本（seq 升序，越后越新），`getResult()` 给出合并结果。四种 `MergeEngine`（`CoreOptions.java:3910-3917`，默认 DEDUPLICATE）即四种 add/getResult 实现：

| Merge Engine | 语义 | 实现 | 关键边界 |
|--------------|------|------|----------|
| **deduplicate**（默认） | 保留最新一行 | `DeduplicateMergeFunction`：`latestKv = kv`，结果取最后一个 | `ignore-delete` 时跳过 retract 行 |
| **partial-update** | 非空字段逐列覆盖 | `PartialUpdateMergeFunction`：按列取最新非 null，支持 sequence-group | DELETE 默认不删整行（除非配 `partial-update.remove-record-on-delete`） |
| **aggregation** | 同 key 聚合（sum/max/last 等） | `AggregateMergeFunction` + 各 `FieldAggregator` | 多数聚合 `requireCopy()=true`（需缓存中间态） |
| **first-row** | 保留最早一行 | `FirstRowMergeFunction`：只认第一条，其余丢弃 | 触发 lookup compaction（§8.2）；`needLookup()` 含 first-row（`CoreOptions.java:3644`） |

**③ 关键源码**

最简单的 deduplicate（`DeduplicateMergeFunction.java`）——只记最后一个 kv，`requireCopy()=false`（无需深拷贝，因只持最后引用）：

```java
public void add(KeyValue kv) {
    if (ignoreDelete && kv.valueKind().isRetract()) return;  // ignore-delete：跳过 -U/-D
    latestKv = kv;                                            // 喂入有序，最后一个即最新
}
public KeyValue getResult() { return latestKv; }
public boolean requireCopy() { return false; }
```

`requireCopy()` 与 §4.2 MergeIterator 联动：deduplicate/first-row 只持单引用故 false；aggregation/partial-update 跨多条累积中间态，必须 true 让 MergeIterator 深拷贝输入，否则对象复用会损坏已累积的字段。

**partial-update 的逐列覆盖与 sequence-group**（`PartialUpdateMergeFunction`）：同 key 多条按 seq 升序喂入，每列取"最新非 null"值。问题：若一列的更新乱序到达（同分区内 seq 已保证有序，但跨源/回撤场景需更细控制），用 **sequence-group** 把若干数据列绑定到一个时间/版本列，按该列的值而非全局 seq 决定每组列的新旧——避免"老数据覆盖新数据"。worked example：

```
表 (k PK, a, b, c)，seq 升序喂入：
  +I (k=1, a=A1, b=null, c=null, seq=10)
  +I (k=1, a=null, b=B2, c=null, seq=11)
  +I (k=1, a=null, b=null, c=C3, seq=12)
  → partial-update 合并结果：(k=1, a=A1, b=B2, c=C3)   // 各列取最新非 null
```

DELETE 处理：partial-update 默认**不**因一条 -D 就删整行（因为部分列更新语义下"删某列"无意义），需 `partial-update.remove-record-on-delete`（`CoreOptions.java:977`）或 `partial-update.remove-record-on-sequence-group`（`:985`）显式开启整行删除。aggregation 类似——`AggregateMergeFunction` 持有每列 `FieldAggregator`（sum/max/min/last_value/listagg/collect…），`add` 时逐列聚合、`getResult` 输出累积态，故 `requireCopy()=true`。

**④ 风险 / 陷阱 / 边界**：
- partial-update 默认不处理 DELETE（一行被 -D 不会消失），需显式配 `partial-update.remove-record-on-delete` 或 sequence-group 控制。
- partial-update / aggregation 默认不接受 retract（-U/-D）输入做"反向聚合"，除非配 sequence-group + 特定 aggregator；否则回撤语义会出错。
- first-row 与 MOW 一样触发 radical lookup compaction（§8.2），因为"判断是不是第一条"需要点查 LSM 看 key 是否已存在。
- `changelog.row-deduplicate`（`CoreOptions.java:922`）可在产 changelog 时去掉无实际变化的 -U/+U 对，减少 changelog 量。

**⑤ 收益与代价**：一个 merge-engine 配置即覆盖去重/部分列更新/聚合/取首行四类典型 upsert 语义，且全部流式可算。代价是 partial-update/aggregation 需缓存中间态（requireCopy=true，内存与拷贝开销）、first-row 引入 lookup compaction 成本、sequence-group 增加配置复杂度。

## 4. 写入路径：MergeTreeWriter

`MergeTreeWriter` 同时实现 `RecordWriter<KeyValue>` 与 `MemoryOwner`，是主键表写入的核心执行器。

### 4.1 write 与 sequenceNumber 分配

**① 要解决什么问题**：在毫秒级延迟下接收流式数据，且保证同 key 多版本的确定性新旧关系。

**② 设计原理与取舍**：`write()` 只写内存 WriteBuffer，**不立即落盘**；sequenceNumber 在 **write 时**（而非 commit 时）分配。在 write 时分配的关键好处：flush 时数据已按 `(key, sequenceNumber)` 排好序，只需顺序写文件，避免 commit 时 O(N log N) 全局重排；不同 Writer 的 seq 区间通过 restore 时的 maxSequenceNumber 隔离。

**③ 关键源码**（`MergeTreeWriter.java:163-174`）：

```java
public void write(KeyValue kv) throws Exception {
    long sequenceNumber = newSequenceNumber();
    boolean success = writeBuffer.put(sequenceNumber, kv.valueKind(), kv.key(), kv.value());
    if (!success) {                          // buffer 满
        flushWriteBuffer(false, false);      // 先 flush 腾空间
        success = writeBuffer.put(sequenceNumber, kv.valueKind(), kv.key(), kv.value());
        if (!success) throw new RuntimeException("Mem table is too small to hold a single element.");
    }
}
```

**④ 风险 / 陷阱 / 边界**：误以为 `write()` 同步落盘会导致误判写性能、忽视 flush 时机；`changelog-producer=input` 会让每次 flush 多写一个 changelog 文件，吞吐降约 50%、存储翻倍，仅在需要 < 1 分钟 changelog 延迟时使用。

写入路径全景（一条记录的命运）：

```
write(KeyValue)
   │ newSequenceNumber() 分配递增 seq
   ▼
WriteBuffer.put(seq, kind, key, value)
   ├─ 成功 → 返回（数据仅在内存）
   └─ 失败（buffer 满）
        ▼ flushWriteBuffer(false,false)
        ├─ [反压点1] shouldWaitForLatestCompaction()?  是→等当前 compaction
        ├─ writeBuffer.forEach(keyComparator, mergeFunction, rawConsumer, mergedConsumer)
        │     ├─ rawConsumer  → changelog 文件（仅 INPUT 模式，merge 前原始记录）
        │     └─ mergedConsumer → 数据文件（merge 后结果，落 Level 0）
        ├─ compactManager.addNewFile(每个新文件)
        ├─ trySyncLatestCompaction() 收集已完成结果
        └─ compactManager.triggerCompaction() 异步触发
        ▼ 重新 put（仍失败则抛错：单条记录都放不下）
```

**⑤ 收益与代价**：写路径仅 O(1) seq 分配 + 内存 put，延迟稳定毫秒级；代价是数据在 flush 前仅存于内存，依赖 checkpoint 持久化。

### 4.2 WriteBuffer / SortBufferWriteBuffer 排序缓冲

**WriteBuffer 接口**（`WriteBuffer.java`）的两个设计点：`put()` 返回 `boolean`（满则 false，由调用方决定 flush 重试，避免异常驱动控制流）；`forEach()` 同时接收 `rawConsumer` 与 `mergedConsumer`，实现"一次遍历双路输出"——merge 前的原始记录写 changelog（INPUT 模式），merge 后写数据文件。

**SortBufferWriteBuffer** 是唯一实现。排序键 = `[key 字段] + [用户定义序列字段(可选)] + [sequenceNumber]`（`SortBufferWriteBuffer.java:80-94`）：

```java
IntStream sortFields = IntStream.range(0, keyType.getFieldCount());          // key 字段
if (userDefinedSeqComparator != null) {                                      // 用户序列字段
    IntStream udsFields = IntStream.of(userDefinedSeqComparator.compareFields())
            .map(operand -> operand + keyType.getFieldCount() + 2);          // 跳过 seq+kind 两列
    sortFields = IntStream.concat(sortFields, udsFields);
}
sortFields = IntStream.concat(sortFields, IntStream.of(keyType.getFieldCount())); // sequenceNumber
```

> **已修正**：旧稿代码片段写 `newRecordComparator(fieldTypes, sortFieldArray, true)`，与现源码不符。现版本支持**用户定义序列字段降序**：通过构造 `boolean[] ascendingOrders`（`SortBufferWriteBuffer.java:96-106`，UDS 字段按 `isAscendingOrder()` 置位），再传入 `CodeGenUtils.newNormalizedKeyComputer(fieldTypes, sortFieldArray, ascendingOrders)` 与 `newRecordComparator(fieldTypes, sortFieldArray, ascendingOrders)`（`:114-117`）。

用户定义序列字段用于 CDC：上游 binlog 的时间戳/GTID 比 Paimon 内部 seq 更能反映真实时序，加入排序键后 merge 按业务语义选"最新"。

内存 vs 磁盘（`SortBufferWriteBuffer.java:119-139`）：`ioManager != null && spillable` 时用 `BinaryExternalSortBuffer`（可溢写），否则纯内存 `BinaryInMemorySortBuffer`；最低要求 3 页内存（`:119-122`，否则抛 `IllegalArgumentException`）。

合并迭代器 `MergeIterator`（`SortBufferWriteBuffer.java:189-293`）是 flush 时"边遍历边 merge"的核心，用 **双 serializer 指针 `previous`/`current` + `swapSerializers()` 零拷贝交换** 收集相同 key 的连续记录（`advanceIfNeeded`，`:241-268`）：

```java
private void advanceIfNeeded() throws IOException {
    advanced = true;
    do {
        swapSerializers();                                  // current → previous
        if (previousRow == null) return;
        mergeFunctionWrapper.reset();
        mergeFunctionWrapper.add(requireCopy ? previous.getCopiedKv() : previous.getReusedKv());
        while (readOnce()) {                                // 读下一条
            if (keyComparator.compare(previous.getReusedKv().key(),
                                      current.getReusedKv().key()) != 0) break;  // key 变了→收尾
            mergeFunctionWrapper.add(requireCopy ? current.getCopiedKv() : current.getReusedKv());
            swapSerializers();
        }
        result = mergeFunctionWrapper.getResult();
    } while (result == null);                               // merge 返回 null（如 first-row 过滤）则跳过该 key 继续
}
```

三个设计细节决定其正确性与性能：① `rawConsumer` 在 `readOnce()`（`:270-283`）里每读一条即触发——这就是 INPUT changelog 的"原始记录"来源，与 merge 后的 `mergedConsumer` 形成双路输出；② `requireCopy()` 由 merge function 决定是否需深拷贝 KeyValue（接口注释明确：复用对象时第 3 个对象可能覆盖第 1 个的引用，故聚合类 merge 需 copy）；③ `swapSerializers()`（`:285-292`）只交换两个 serializer/BinaryRow 引用、不拷贝数据，使得"前进一步"几乎零开销。

> 为什么 merge function 可返回 null：`first-row` merge-engine 只保留首条、`partial-update`/aggregation 在某些情况合并后无有效行，此时 `getResult()` 返回 null，`do-while` 直接跳到下一个 key，等价于"该 key 不产出任何行"。

### 4.3 flush 与增量收集

`flushWriteBuffer()`（`MergeTreeWriter.java:209-249`）是内存到磁盘的关键：

```java
private void flushWriteBuffer(boolean waitForLatestCompaction, boolean forcedFullCompaction) {
    if (writeBuffer.size() > 0) {
        if (compactManager.shouldWaitForLatestCompaction()) waitForLatestCompaction = true; // 反压点1
        RollingFileWriter changelogWriter = changelogProducer == ChangelogProducer.INPUT
                ? writerFactory.createRollingChangelogFileWriter(0) : null;
        RollingFileWriter dataWriter = writerFactory.createRollingMergeTreeFileWriter(0, FileSource.APPEND);
        try {
            writeBuffer.forEach(keyComparator, mergeFunction,
                    changelogWriter == null ? null : changelogWriter::write, dataWriter::write);
        } finally { writeBuffer.clear(); /* close writers */ }
        if (changelogWriter != null) newFilesChangelog.addAll(changelogWriter.result());
        for (DataFileMeta f : dataWriter.result()) { newFiles.add(f); compactManager.addNewFile(f); }
    }
    trySyncLatestCompaction(waitForLatestCompaction);   // 收集已完成 compaction 结果
    compactManager.triggerCompaction(forcedFullCompaction); // 触发新 compaction
}
```

`RollingFileWriter` 在单文件超过 `target-file-size` 时自动滚动新文件，保证单文件不过大、便于后续 compaction 与并行读。

**六个增量集合 → CommitIncrement**（`MergeTreeWriter` 成员）：`newFiles`/`deletedFiles`/`newFilesChangelog`（→ DataIncrement），`compactBefore`/`compactAfter`/`compactChangelog`（→ CompactIncrement）。`compactBefore` 用 `LinkedHashMap`（key=fileName）以便 `updateCompactResult()`（`:309-329`）通过文件名判定"中间文件"。

`updateCompactResult` 的中间文件处理（`MergeTreeWriter.java:309-329`）是写路径正确性的精细点：

```java
private void updateCompactResult(CompactResult result) {
    Set<String> afterFiles = result.after().stream().map(DataFileMeta::fileName).collect(toSet());
    for (DataFileMeta file : result.before()) {
        if (compactAfter.remove(file)) {                       // 该输入文件曾是上次 compaction 的输出
            // 中间文件：本周期内产生又消费，从未进 snapshot，可直接物理删
            if (!compactBefore.containsKey(file.fileName())    // 不是已记录的待删
                    && !afterFiles.contains(file.fileName()))  // 也不是本次输出（upgrade 文件前后都引用）
                writerFactory.deleteFile(file);                // 直接删，不进 CommitIncrement
        } else {
            compactBefore.put(file.fileName(), file);          // 真正的旧文件 → 记为待删
        }
    }
    compactAfter.addAll(result.after());
    compactChangelog.addAll(result.changelog());
}
```

为什么要双重排除：若一个文件是 upgrade 产生（只改 level），它会同时被前一个 snapshot 与后一个 snapshot 引用，绝不能当中间文件删掉，否则两个 snapshot 都指向不存在的文件。这是"upgrade 文件须保留"在写路径侧的保障。

`drainIncrement()`（`MergeTreeWriter.java:280-302`）把六集合打包成 `CommitIncrement`（含 `DataIncrement` + `CompactIncrement` + `CompactDeletionFile`）后清空所有集合，准备下个 checkpoint 周期。

**为什么 data 与 compact increment 分离**：二者在 `FileStoreCommitImpl` 被提交为两个独立 snapshot（APPEND + COMPACT）。好处——APPEND 冲突检测更简单更快（§13）；COMPACT 失败不影响数据持久化（只是"错过一次优化"）；流式读可只消费 APPEND snapshot 拿增量（§12.4 ScanMode.DELTA）。

**⑤ 收益与代价**：写路径把"持久化数据"与"优化文件"解耦成两个增量、两个 snapshot，容错与增量读都受益；中间文件直接删避免无谓的 manifest 膨胀。代价是六集合 + 中间文件判定的状态管理较繁，drain 后必须完整清空否则跨周期串数据。

## 5. 读取路径：MergeTreeReaders

### 5.1 三层归并结构与 IntervalPartition

**① 要解决什么问题**：从可能数千个文件中读出正确的最新版本，又不能打开全部文件做全量归并（内存爆炸、归并路数过高）。

**② 设计原理与取舍**：三层读取器 Section → SortedRun → 单文件。先用 `IntervalPartition` 把待读文件按 key 区间切成多个 **Section（section 间 key 不重叠）**，section 之间用 `ConcatRecordReader` 串联（零归并开销），仅 section 内部做归并。section reader 通过 lambda 懒加载，前一个读完才创建下一个，避免同时打开全部文件。

**③ 关键源码**

入口（`MergeTreeReaders.java:44-65`）懒加载串联各 section：

```java
public static <T> RecordReader<T> readerForMergeTree(
        List<List<SortedRun>> sections, FileReaderFactory<KeyValue> readerFactory,
        Comparator<InternalRow> userKeyComparator, @Nullable FieldsComparator udsComparator,
        MergeFunctionWrapper<T> wrapper, MergeSorter mergeSorter) throws IOException {
    List<ReaderSupplier<T>> readers = new ArrayList<>();
    for (List<SortedRun> section : sections)
        readers.add(() -> readerForSection(section, readerFactory, userKeyComparator, udsComparator, wrapper, mergeSorter));
    return ConcatRecordReader.create(readers);   // 懒加载串联
}
```

`IntervalPartition`（`IntervalPartition.java:33`）两层划分：外层 `partition()`（`:67`）按 minKey 排序的文件流，遇到 `minKey > 当前 section 右边界` 就切新 section；内层 `partition(metas)`（`:93`）用贪心优先队列（取 maxKey 最小的 run 尝试追加），等价于**区间图着色**，保证 SortedRun 数 = 最大重叠深度（最少）。

三层读取器结构：

```
ConcatRecordReader（sections 间串联，零归并）
   ├─ Section[0] ─ SortMergeReader（section 内多 SortedRun 归并）
   │                  ├─ Run[0] ─ ConcatRecordReader（run 内多文件串联）─ f1,f2,f3
   │                  └─ Run[1] …
   ├─ Section[1] …
   └─ Section[2] …
```

量化直觉：1000 个文件若不分 section 要 1000 路归并；经 IntervalPartition 若分成 100 个 section、每个平均 10 路，则**任一时刻**只需 10 路归并 + section 间顺序切换。N 路归并比较代价 O(N log N)，10 路相对 100 路约降 70%，且更少路数带来更好的 CPU 缓存局部性。

**④ 风险 / 陷阱 / 边界**：误以为读取需归并所有文件；实际多数 section 仅含 1–2 个 SortedRun。`sort-engine` 大路数应保持默认 `loser-tree`（min-heap 常数因子大）；`sort-spill-threshold` 不要设成极大值（否则大路数归并 OOM），默认值更安全。

**⑤ 收益与代价**：归并路数从"文件总数"降到"section 内重叠深度"，内存与 CPU 大幅下降，缓存局部性更好。代价是 IntervalPartition 的排序与分割本身有 O(F log F) 开销，但远小于全量归并。

### 5.2 MergeSorter 与磁盘溢写

`MergeSorter`（`MergeSorter.java:60`）持有 `sortEngine`（默认 LOSER_TREE）、`spillThreshold`（`:66`，来自 `sortSpillThreshold()`）、压缩选项与内存池。

**LoserTree vs MinHeap 的算法差异**（为什么默认 LoserTree）：k 路归并每次要选出 k 个 reader 头部中最小者并前进它。MinHeap 前进堆顶后需 `siftDown`，每层要比较两个孩子选较小者再与父比，约 `2·log k` 次比较。LoserTree（败者树）是一棵完全二叉树，内部节点记"较大者（败者）"、根上方记全局最小（胜者）；前进某叶子后只需沿其到根的路径，每个内部节点与"已记的败者"比一次，约 `log k` 次比较——比较次数减半。大路数（如 spill 后几十路）时这个常数差异显著，故默认 LOSER_TREE；小路数 MinHeap 实现简单也够用。两者都通过 `SortMergeReader.createSortMergeReader`（§5.3）按 `sort-engine` 选择。

溢写策略（`mergeSort` 在 `:112`，路数 > `spillThreshold` 时走 `spillMergeSort` 在 `:148`）：先按 SortedRun 大小升序排，**把最小的若干个溢写到临时文件**（`spillSize = inputReaders.size() - spillThreshold`，`:156`），大的留内存，再做最终归并。溢写用 64KB 压缩块 + `KeyValueWithLevelNoReusingSerializer` 序列化（保留 level 字段，因 merge 后仍需知道来源层）。

**为什么溢写最小的**：小 run 溢写 IO 量最小，大 run 留内存可享连续读 IO 优势——典型"以最小代价腾资源"。调优：`sort-spill-threshold` 默认 none（不溢写，纯内存），大路数易 OOM 时设个值（compaction.md 异步配方给 10）；行很大时 `read.batch-size` 减小可降归并峰值内存（write-performance.md）。

### 5.3 完整归并链路：从 Split 到 DropDeleteReader

**① 要解决什么问题**：把"读一个 Split"翻译成一条精确的 reader 嵌套链，并在链路末端正确处理 DELETE 记录、deletion vector、record-level expire 等横切语义——这是理解 MOR 读性能与正确性的关键。

**② 设计原理与取舍**：主键表 MOR 读取由 `MergeFileSplitRead`（`KeyValueFileStore.newRead()` 创建，见 §15）组装，整条链由内向外是：

```
单文件 RecordReader（KeyValueFileReaderFactory 按 schemaId 做列映射/类型转换）
   ↓ run 内多文件
ConcatRecordReader（按 minKey 顺序串联，run 内不重叠故无需归并）
   ↓ section 内多 run
SortMergeReader（LoserTree / MinHeap，按 (key, seq) 归并 + MergeFunction 合并同 key 多版本）
   ↓ section 间
ConcatRecordReader（section 间不重叠，顺序串联）
   ↓ 可选包装
DropDeleteReader（dropDelete=true 时丢弃非 ADD 行）/ RecordLevelExpire.wrap（行级过期过滤）
```

**③ 关键源码**

归并引擎工厂（`SortMergeReader.java:41-57`）按 `sort-engine` 二选一，注释强调"同一个 reader 内不能出现相同 key"（即 run 内不重叠是前提）：

```java
static <T> SortMergeReader<T> createSortMergeReader(
        List<RecordReader<KeyValue>> readers, Comparator<InternalRow> userKeyComparator,
        @Nullable FieldsComparator udsComparator, MergeFunctionWrapper<T> wrapper, SortEngine sortEngine) {
    switch (sortEngine) {
        case MIN_HEAP:   return new SortMergeReaderWithMinHeap<>(readers, userKeyComparator, udsComparator, wrapper);
        case LOSER_TREE: return new SortMergeReaderWithLoserTree<>(readers, userKeyComparator, udsComparator, wrapper);
        default: throw new UnsupportedOperationException("Unsupported sort engine: " + sortEngine);
    }
}
```

末端 `DropDeleteReader`（`DropDeleteReader.java:33-69`）在 batch 迭代里循环跳过所有 `!kv.isAdd()` 的行：

```java
public KeyValue next() throws IOException {
    while (true) {
        KeyValue kv = batch.next();
        if (kv == null) return null;
        if (kv.isAdd()) return kv;   // 仅保留 INSERT / UPDATE_AFTER，丢弃 DELETE / UPDATE_BEFORE
    }
}
```

**④ 风险 / 陷阱 / 边界**：
- MOR 下"单棵 LSM 树只能单线程读"（官方 table-mode 明确），读并行度受 bucket 数限制——这是 §10 桶数建议 200MB–1GB 的根因：桶太大读不动，桶太小小文件多。
- MOR 因为要 merge，**无法对非主键列做 filter 下推**（否则会把待合并的旧版本误过滤导致结果错误），官方 query-performance 文档明确此限制；只有 MOW/COW/Read-Optimized 表才能用非主键列 data skipping。
- `DropDeleteReader` 只在 compaction 输出到最底层（`dropDelete=true`，见 §8.3）或最终读取时使用；中间层 compaction 必须保留 DELETE 记录，否则更高层的旧版本无法被正确"墓碑"覆盖。

**⑤ 收益与代价**：链路完全由 reader 组合而成、可懒加载、可逐层裁剪，正确性由"run 内不重叠 + section 间不重叠 + 归并按 (key,seq)"三层不变式保证。代价是 MOR 读路径的 merge 开销与 SortedRun 数线性相关——这正是 COW/MOW 用写期代价换读期性能的动机（见 §7）。

### 5.4 列裁剪、谓词下推与 RawFileSplitRead

**① 要解决什么问题**：不是所有读都需要走完整 merge 链。批量扫描只取部分列、或读已 full-compact 的文件时，应跳过 merge、只读需要的列，把读放大压到最低。

**② 设计原理与取舍**：两类优化——**列裁剪（projection）**：只反序列化查询涉及的列（列存格式天然支持），减少 IO 与解码；**谓词下推（filter pushdown）**：把过滤条件下推到文件级（统计/file index 跳文件）甚至 row-group 级。但 MOR 主键表有约束（§5.3）：非主键列谓词不能跳文件（merge 前过滤会丢旧版本）。`KeyValueFileStore` 提供两条读路径：`newRead()` → `MergeFileSplitRead`（带 merge，§15）；`newBatchRawFileRead()` → `RawFileSplitRead`（**跳过 merge** 的原始读）。

**③ 关键源码**

`RawFileSplitRead`（`KeyValueFileStore.newBatchRawFileRead()`，`KeyValueFileStore.java:115-124`）直接读数据文件、不做归并——适用于：① COW 表（数据已全量合并，单层有序无需 merge）；② Read-Optimized 系统表（只读已 full-compact 的高层文件，§7.3）；③ MOW 表配合 `ApplyDeletionVectorReader`（§7.2，DV 过滤代替 merge）。这是"为什么 COW/MOW 读不受 bucket 并行限制、可非主键列 skipping"的源码落点：它们走的是 raw read + DV 过滤，而非 MOR 的 SortMergeReader。

读类型由 `FileStoreScan.withReadType(RowType)`（§12.4，`FileStoreScan.java:100`）携带，传递投影后的列集合；过滤经 `withCompleteFilter`/`enableValueFilter` 下推到 scan 的统计裁剪。

**④ 风险 / 陷阱 / 边界**：
- 对 MOR 表强行用 raw read 会读到未合并的多版本（脏读），仅在确知数据已 full-compact（COW/Read-Optimized）或有 DV（MOW）时才安全。
- 列裁剪对行存（avro）收益小（无列投影，§15.2），列存（parquet/orc）收益大——这是分析查询选列存的根因。
- 谓词若涉及主键，固定桶可叠加桶裁剪（§10.2）+ 文件级 minKey/maxKey 裁剪（§12.1），多级叠加效果最佳。

**⑤ 收益与代价**：raw read + 投影 + 下推使"读得少"，COW/MOW 借此达到接近列式仓库的查询性能。代价是 raw read 不做 merge，正确性依赖"数据已合并/有 DV"前提，误用会脏读。

## 6. LookupLevels 点查加速

**① 要解决什么问题**：在为批量扫描设计的列式格式（Parquet/ORC）上实现微秒级点查。三个核心使用场景都依赖"按主键点查 LSM"：① **lookup join**（维表关联，每条流记录按 key 查维表）；② **`changelog-producer=lookup`**（compaction 时点查旧值生成 `-U`，§8.6）；③ **MOW deletion vector 生成**（写期点查 key 是否在高层已存在以标记被删行，§7.2）；④ **first-row merge-engine**（点查判断是否已有该 key，§3.4）。后三者统称 lookup compaction，共用本章基础设施。

**② 设计原理与取舍**：把远程列式文件**首次访问时转成本地磁盘的 KV lookup 文件**（当前为 sort-based 本地存储，块内有序 + 块索引 + BloomFilter，详见本节末），后续点查走本地索引。直接在 Parquet 上点查要读 footer、定位 row group、解压解码整段，延迟数十毫秒；本地 KV 索引点查微秒级。查找顺序从小 level 到大 level，**最新优先**，命中即返回，避免读到旧版本再判断是否被覆盖——这与 §3.1 的层级新旧语义、§5.3 的归并方向一致。

**③ 关键源码**

跨层查找骨架（`LookupUtils.java:35-57`）：

```java
public static <T> T lookup(Levels levels, InternalRow key, int startLevel,
        BiFunctionWithIOE<InternalRow, SortedRun, T> lookup,
        BiFunctionWithIOE<InternalRow, TreeSet<DataFileMeta>, T> level0Lookup) throws IOException {
    T result = null;
    for (int i = startLevel; i < levels.numberOfLevels(); i++) {
        result = (i == 0) ? level0Lookup.apply(key, levels.level0())
                          : lookup.apply(key, levels.runOfLevel(i));
        if (result != null) break;   // 最新优先
    }
    return result;
}
```

Level 0 遍历 TreeSet（新鲜度序）逐文件判 `minKey ≤ target ≤ maxKey`（`LookupUtils.java:59-77`）；Level 1+ 因不重叠用二分查找定位唯一候选文件（`LookupUtils.java:79-119`）。

本地缓存与构建：首次 lookup 某文件时若缓存未命中则 `createLookupFile()`（用 `fileReaderFactory` 读远程文件，逐行序列化为 `(keyBytes, valueBytes)`，`LookupStoreWriter` 建索引 + BloomFilter），结果存入 Caffeine LRU（`lookupFileCache`），并记入 `ownCachedFiles`。`LookupLevels` 在构造时 `levels.addDropFileCallback(this)`，compaction 删文件时 `notifyDropFile()` 自动 `invalidate` 缓存——这正是 §3.2 中 `Levels.update()` 的回调钩子。

`createLookupFile()`（`LookupLevels.java:182-207`）的构建步骤：① 本地磁盘建空文件（`localFile.createNewFile()`，失败抛 IOException）→ ② `tryToDownloadRemoteSst()`（`:209-233`）尝试下载远程 lookup 文件（校验 processorId/serVersion/schema 兼容）→ ③ 未命中则现场 `createSstFileFromDataFile`（读远程数据文件逐行写本地 sort-based 索引 + BloomFilter）→ ④ 包装成含 `LookupStoreReader` 的 `LookupFile`（带 `() -> ownCachedFiles.remove(...)` 清理回调）→ ⑤ 注册到 `ownCachedFiles`（保证 close 时只清自己拥有的，不误删其他 LookupLevels 实例的缓存）。下载命中时用远程文件的 `schemaId`，否则用当前 schema。BloomFilter 的价值：partial-update / 去重场景大量 key 实际不存在，Bloom 让这些查询无需打开本地索引即可返回 null。

**LookupStore 的本地实现**：当前由 `LookupStoreFactory.create()`（`LookupStoreFactory.java:67-72`）统一返回 `SortLookupStoreFactory`——**sort-based 本地存储**（块内有序 + 块索引，类 LevelDB SSTable 的查找结构），而非通用 RocksDB 实例（`local-sort` 对 Parquet lookup 已足够，CoreOptions 注释明确）。BloomFilter 由 `bfGenerator`（`:52-65`）按 `lookup.cache.bloom.filter.enabled` + `fpp`（默认 0.05，`CoreOptions.java:1314-1321`）生成，`rowCount>0` 才建，用于"不存在 key"的快速否定。`lookup.hash-load-factor`（`CoreOptions.java:1271-1275`，默认 0.75）调本地 hash 索引的空间/查找权衡。

**Caffeine LRU 缓存 + DropFileCallback 双触发失效**。缓存本体由 `LookupFile.createCache()`（`LookupFile.java:123-132`）构建，**以"本地磁盘占用"为权重做容量上限**：

```java
public static Cache<String, LookupFile> createCache(Duration fileRetention, MemorySize maxDiskSize) {
    return Caffeine.newBuilder()
        .expireAfterAccess(fileRetention)              // lookup.cache-file-retention，默认 1h，闲置即逐出
        .maximumWeight(maxDiskSize.getKibiBytes())     // lookup.cache-max-disk-size，按 KiB 计权
        .weigher(LookupFile::fileWeigh)                // 权重 = 本地文件大小（KiB），非条目数
        .removalListener(LookupFile::removalCallback)  // 逐出/失效 → 触发 close → 删本地文件
        .executor(Runnable::run).build();              // 同步执行 removalListener
}
```

`weigher` 用文件大小而非条目数，是因为本地磁盘是真正的稀缺资源——缓存总磁盘占用被 `lookup.cache-max-disk-size` 硬约束。**两条失效路径都汇入同一个 `removalListener → LookupFile.close()`**（`LookupFile.java:107-118`，`reader.close()` + `callback.run()` 清 `ownCachedFiles` + `FileIOUtils.deleteFileOrDirectory(localFile)` 删本地文件）：

1. **被动逐出**：磁盘超 `cache-max-disk-size`（LRU 逐出）或 `expireAfterAccess` 闲置超 `cache-file-retention`。
2. **主动失效（DropFileCallback）**：compaction 删了某 data 文件时，`Levels.update()` 回调 `LookupLevels.notifyDropFile(file)`（`LookupLevels.java:126-127`）→ `lookupFileCache.invalidate(file)` → removalListener 删本地文件。这是正确性关键：**底层 data 文件被 compaction 合并掉后，其对应的本地 lookup 索引必须立刻作废**，否则点查会命中已不存在数据文件的陈旧索引。这正是 §3.2 `Levels.update()` 的 DropFileCallback 钩子用途。

缓存命中逻辑（`LookupLevels.java:146-163`）：`getIfPresent(file.fileName())` 未命中才 `createLookupFile`，且**只有真正用到（finally 块）才 `addLocalFile` 入缓存**——避免构建失败的半成品污染缓存。

**远程 lookup 文件（冷启动/failover 加速）**：`lookup.remote-file.enabled=true`（`CoreOptions.java:1328-1331`）时检查 `DataFileMeta.extraFiles()` 中的 `.lookup` 后缀文件，校验 processorId/serVersion/schema 兼容后直接下载，命名 `{原文件名}.{fileSize}.{processorId}.{serVersion}.lookup`，并受 `lookup.remote-file.level-threshold` 控制只为高 level 大文件生成（小文件本地建索引更快、不值得占元数据）。没有它，job failover 后所有本地 lookup 文件丢失，需对全部活跃文件重新逐行构建，冷启动可达分钟级。



**lookup-wait 与可见性**：`lookup-wait`（`CoreOptions.java:1948-1954`，默认 **true**，旧 key `changelog-producer.lookup-wait`）决定 commit 是否等待 lookup compaction 完成。默认 true 时写入会同步等 lookup compaction，保证 L0 数据及时进入可读层（这与 MOW 的"L0 compaction 后才可见"一致）；设 false 可提吞吐但 changelog/数据可见性有延迟（compaction.md 的"完全异步"配方即 `lookup-wait=false` + 大 stop-trigger + sort-spill-threshold）。

**hash vs sort lookup**：本地 LookupStore 有两类查找结构。**sort-based**（当前 `LookupStoreFactory.create` 默认）按 key 排序、块内有序 + 块索引，点查走块索引 + 块内二分，对 Parquet/ORC 这种本身按 key 有序的数据文件构建快、空间紧凑；**hash-based** 用 hash 表（`lookup.hash-load-factor` 控装载因子），单点查 O(1) 更快但占空间多、不支持范围。lookup compaction / point-lookup 场景多为单 key 精确查，二者都够用，Paimon 当前定为 sort 优先（构建成本低、对有序数据天然友好）。`lookup.hash-load-factor` 越小冲突越少查得越快但越费磁盘。

**④ 风险 / 陷阱 / 边界**：
- **冷启动**：未开 `lookup.remote-file.enabled` → failover 后所有本地 lookup 索引丢失，需对全部活跃文件逐行重建，冷启动数分钟（大表更久）。生产 lookup 表建议开远程文件。
- **`lookup.cache-max-disk-size` 设太小** → LRU 频繁逐出，刚建好的本地索引被删又得重建，hit 率低、CPU/IO 抖动。看 `LookupFile.close` 日志里的 `hitCount/requestCount` 评估命中率；磁盘允许时调大它。
- **`lookup.cache-file-retention`（默认 1h）太短** → 闲置索引被 `expireAfterAccess` 逐出，周期性维表查询（间隔 > retention）每次都重建。按访问间隔调大。
- **内存块缓存**：`lookup.cache-max-memory-size`（`CoreOptions.java:1301-1302`）+ `lookup.cache.high-priority-pool-ratio`（`:1307`）是本地索引**数据块**的内存缓存（在磁盘文件之上再加一层热块缓存），与上面磁盘级文件缓存是两层。内存紧张时它会与 write buffer 争内存，需统筹。
- Lookup 场景默认 `lookup-compact=radical`（`CoreOptions.java:1956-1960`）让 L0 及时清空（否则点查要遍历多个 L0 文件），资源紧张可降为 gentle + `lookup-compact.max-interval` 牺牲新鲜度换资源。
- 不要手动删本地 lookup 文件，Caffeine LRU + DropFileCallback 会自动管理；手删会让 reader 指向不存在文件。lookup 场景写吞吐受 lookup compaction 拖累，快照同步阶段可临时关 `changelog-producer=lookup`（write-performance.md）。

**⑤ 收益与代价**：点查从数十毫秒降到微秒级，冷启动从分钟级降到秒级（远程文件）。代价分三层并各有上限约束：① 本地磁盘占用（由 `cache-max-disk-size` 权重上限 + LRU 控制）；② 首次构建 IO（读远程文件逐行序列化建索引，由远程文件/BloomFilter 摊薄）；③ lookup compaction 把点查代价前移到写期（`lookup-wait=true` 时拖慢 commit），换取读期/changelog 生成的低延迟。本质是用"写期多做功 + 本地磁盘空间"换"读期微秒级点查"，靠 LRU + DropFileCallback + BloomFilter 把代价控制在有界范围。

## 7. Table Mode：MOR / COW / MOW 与 Deletion Vector

> 官方 table-mode 文档只讲了三模式"怎么配、读写性能好不好"。本章在其之上讲清：每种模式在读写路径源码里走哪条分支、deletion vector 如何参与读写、各自的读/写放大代价与适用场景，以及 Read-Optimized 折中。

### 7.1 三模式的读写路径分支

**① 要解决什么问题**：主键表读取本质要 merge（§5.3），merge 发生在"读时"还是"写时"决定了读写性能的此消彼长。三种 Table Mode 就是把 merge 时机放在三个不同位置。

**② 设计原理与取舍**

| 模式 | 触发配置 | merge 时机 | 写路径分支 | 读路径分支 | 写放大 | 读放大 |
|------|----------|-----------|-----------|-----------|--------|--------|
| **MOR**（默认） | 无（minor compaction） | **读时** 全量归并 | 仅 minor compaction，L0 直接落 | §5.3 完整 SortMergeReader 链 | 低 | 高（受 SortedRun 数线性影响） |
| **COW** | `full-compaction.delta-commits=1` | **写时** 全量合到 maxLevel | 每次 commit 触发 full compaction | 几乎免归并（单层有序） | **极高**（每写全量重写） | 极低 |
| **MOW** | `deletion-vectors.enabled=true` | **写时**生成 DV，读时按位过滤 | compaction 时查 LSM 生成 DV 文件 | `ApplyDeletionVectorReader` 按行位过滤，无需 merge | 中 | 低（接近 COW，但无全量重写） |

三模式本质是 merge 成本在"写期 vs 读期"之间的连续谱：MOR 全压到读期、COW 全压到写期、MOW 用 DV 把"判断哪些行被覆盖/删除"的工作前移到写期但只付出生成位图的代价。官方推荐一般主键表用 MOW（`merge-engine` 默认 `deduplicate` 时）。

**③ 关键源码**

- COW：`full-compaction.delta-commits=1` 让每次 commit 都强制 full compaction（`MergeTreeWriter.compact(fullCompaction=true)` → `flushWriteBuffer(true, true)`，见 §4.3），所有 run 合到 maxLevel，读时单层不重叠近乎免归并。
- MOW：`deletionVectorsEnabled` 在 `KeyValueFileStore.newWrite()`（`KeyValueFileStore.java:168-171`）创建 `BucketedDvMaintainer.Factory`：

```java
BucketedDvMaintainer.Factory dvMaintainerFactory = null;
if (options.deletionVectorsEnabled()) {
    dvMaintainerFactory = BucketedDvMaintainer.factory(newIndexFileHandler());
}
```

`dvMaintainer != null` 还直接影响 compaction 的 `dropDelete` 判定（`MergeTreeCompactManager.java:179-183`，见 §8.3）——有 DV 时删除靠位图追踪而非 DELETE 墓碑，故可放心 dropDelete。

**④ 风险 / 陷阱 / 边界**：
- COW 写放大极严重（官方原话 "write amplification is very severe"），仅适合写少读多、对读延迟极敏感的维表。
- MOW 的 L0 文件须 compaction 后才可见（同步 compaction 默认开启）；异步会有可见延迟（见 §1.3 陷阱 8）。
- MOW 属于"radical lookup compaction"触发条件之一（与 `lookup` changelog-producer、`first-row` merge-engine 并列，见 `compaction.md`），故 MOW 表也走 ForceUpLevel0 强制清 L0（§8.2）。

**⑤ 收益与代价**：一个配置项即可在读写性能间切换，无需改 schema。代价是模式语义与 compaction/changelog/merge-engine 交织，配错会引发可见延迟或写放大失控。

### 7.2 Deletion Vector 源码机制

**① 要解决什么问题**：MOW 要在不重写数据文件的前提下，标记某数据文件中"哪些行已被新版本覆盖或删除"，使读取能直接跳过这些行、免去 merge。

**② 设计原理与取舍**：用**按行号（position）的位图**记录被删行。`DeletionVector` 接口（`DeletionVector.java:44-71`）核心是 `delete(long position)` / `isDeleted(position)` / `merge(DeletionVector)`，两种实现按文件行数自动二选一：`BitmapDeletionVector`（基于 `RoaringBitmap32`，`BitmapDeletionVector.java:34`，行数 ≤ `RoaringBitmap32.MAX_VALUE` 即约 21 亿）与 `Bitmap64DeletionVector`（64 位，超大文件）。选 RoaringBitmap 而非普通 bitset 的原因：行删除往往稀疏或成簇，Roaring 的"分桶 + 容器（array/bitmap/run）自适应"压缩比普通位图省得多。每个数据文件对应一个 DV，多个 DV 打包进一个 index 文件（`DeletionVectorsIndexFile`），`DeletionVectorMeta` 记录每个数据文件名 → 在 index 文件中的 offset/length。

**③ 关键源码**

DV 索引文件格式（`DeletionVectorsIndexFile.java:46-47`）：索引类型常量 `DELETION_VECTORS`，版本 `VERSION_ID_V1=1`；每个 DV 序列化时带 `MAGIC_NUMBER`（`BitmapDeletionVector.java:36` = 1581511376）校验。读取时校验 version 与 magic，不符抛错——防止跨版本/损坏的 DV 被误用。

读期应用：`ApplyDeletionVectorReader`（`ApplyDeletionVectorReader.java:31-61`）包装文件 reader，对每个 batch 套 `ApplyDeletionFileRecordIterator`，按物理行位置 `returnedPosition()` 查位图过滤被删行：

```java
public FileRecordIterator<InternalRow> readBatch() throws IOException {
    FileRecordIterator<InternalRow> batch = reader.readBatch();
    if (batch == null) return null;
    return new ApplyDeletionFileRecordIterator(batch, deletionVector);  // 迭代时 isDeleted(pos) 则跳过
}
```

写期生成：compaction 时 `BucketedDvMaintainer` 查询 LSM（点查每个新行的 key 是否在更高层已存在，存在则在旧文件对应 position 上 `delete()`）生成 DV；`merge()` 用于把同文件的多次删除合并。DV 的提交在 ConflictDetection 里有专门的 base-ADD-dv 与 delta-DELETE-dv 配对校验（`ConflictDetection.buildBaseEntriesWithDV` / `buildDeltaEntriesWithDV`，`:545-655`），保证"一个文件最多一个有效 DV"，并处理"为无 DV 的文件首次建 DV 时补一条 `DELETE<f,null>`"的微妙情形（`:645-650`）。

**④ 风险 / 陷阱 / 边界**：
- DV 是按 position 而非 key 的，因此数据文件一旦被 compaction 重写，其 position 失效，旧 DV 必须一并失效重建——这是 DV 与 compaction 强耦合的根因（也是 §13 commit 要严格配对校验 base-dv/delta-dv 的原因）。
- DV 模式支持聚合下推（`SELECT COUNT(*)`）与非主键列 data skipping（query-performance 文档），因为读不再需要 merge。
- BUCKET_UNAWARE（追加表）的 DV 走 `AppendDeleteFileMaintainer`/`BaseAppendDeleteFileMaintainer` 路径，提交冲突检测对其做特殊的 dv 配对（`ConflictDetection.java:165-186`），保证 base 文件的 ADD-dv 与 delta 的 DELETE-dv 严格成对。
- 行数超 21 亿的大文件自动用 `Bitmap64DeletionVector`，否则 RoaringBitmap32 会越界。

**⑤ 收益与代价**：读性能接近 COW 但无全量重写的写放大，是通用主键表的推荐模式。代价是写期要做点查（lookup）生成 DV、DV 与 compaction 的生命周期管理复杂、RoaringBitmap 在删除极度密集时压缩收益下降。

### 7.3 MOR Read-Optimized 折中

若不想用 MOW 但希望 MOR 读得快（可容忍读到稍旧数据）：① 写时配 `compaction.optimization-interval`（走 EarlyFullCompaction 的时间触发，§8.2/§8.4）定期产生 read-optimized 的全量文件；② 查询 read-optimized 系统表，直接读已 full-compact 的文件、跳过 merge。这本质是"用数据新鲜度换读性能"，介于 MOR 与 COW 之间。源码上由 `compaction.optimization-interval` 驱动 `EarlyFullCompaction` 定期把数据压到 maxLevel，使 read-optimized 视图有足够新的全量数据。

## 8. Compaction 压缩策略体系

### 8.1 UniversalCompaction 三级策略

**① 要解决什么问题**：在不阻塞写入的前提下，自动控制空间放大、读放大、文件数量三者——单一策略无法同时兼顾。

**② 设计原理与取舍**：移植 RocksDB Universal Compaction，按优先级递减三级（`UniversalCompaction.java:66-107`）：先 EarlyFullCompaction（可选）→ ① Size Amplification（空间放大）→ ② Size Ratio（大小比例）→ ③ File Num（文件数量兜底）。三级分别保障"空间不浪费 > 读取性能 > 文件数可控"。

**③ 关键源码**

主流程（`UniversalCompaction.java:66-107`）：

```java
public Optional<CompactUnit> pick(int numLevels, List<LevelSortedRun> runs) {
    int maxLevel = numLevels - 1;
    if (earlyFullCompact != null) {                      // 0 提前全量
        Optional<CompactUnit> u = earlyFullCompact.tryFullCompact(numLevels, runs);
        if (u.isPresent()) return u;
    }
    CompactUnit unit = pickForSizeAmp(maxLevel, runs);   // 1 空间放大
    if (unit != null) return Optional.of(unit);
    unit = pickForSizeRatio(maxLevel, runs);             // 2 大小比例
    if (unit != null) return Optional.of(unit);
    if (runs.size() > numRunCompactionTrigger) {         // 3 文件数兜底
        int candidateCount = runs.size() - numRunCompactionTrigger + 1;
        return Optional.ofNullable(pickForSizeRatio(maxLevel, runs, candidateCount));
    }
    return Optional.empty();
}
```

**第一级 Size Amplification**（`:124-147`）：`candidateSize`（除最后一个 run 外全部）与 `earliestRunSize`（最后一个 run，近似有效数据）比较，`candidateSize*100 > maxSizeAmp*earliestRunSize`（默认 200%）即全量压缩到 maxLevel。`candidateSize/earliestRunSize` 近似"冗余数据占比"。注意命中时会同步 `earlyFullCompact.updateLastFullCompaction()`（`:140-142`，**旧稿遗漏，已补**）。

> 举例：max level run 有 100GB（有效数据），其余 run 合计 250GB（多为旧版本/被覆盖数据），冗余占比 250% > 200%，触发全量压缩，把所有 run 合并到 maxLevel 彻底清旧版本。不做此检查的话，更新密集表可能占用 3–5 倍实际数据量。为何必须全量：冗余如此之高，局部合并无法有效回收，只有合到最底层才能清除被覆盖版本。

**第二级 Size Ratio**（`pickForSizeRatio` 在 `:163-182`）：从最新 run 起，只要候选集 `candidateSize*(100+sizeRatio+offPeakRatio)/100 ≥ next.totalSize` 就继续纳入，否则停止。`sizeRatio` 是写放大 vs 读放大的旋钮：越小（默认 1，即 1%）越积极合并、run 数少（读放大低）、写放大高。

输出层确定 `createUnit()`（`:196-226`）：

```java
CompactUnit createUnit(List<LevelSortedRun> runs, int maxLevel, int runCount) {
    int outputLevel;
    if (runCount == runs.size()) outputLevel = maxLevel;                     // 全量 → maxLevel
    else outputLevel = Math.max(0, runs.get(runCount).level() - 1);          // next.level - 1
    if (outputLevel == 0) {                                                  // 不允许输出 L0
        for (int i = runCount; i < runs.size(); i++) {
            runCount++;
            if (runs.get(i).level() != 0) { outputLevel = runs.get(i).level(); break; }
        }
    }
    if (runCount == runs.size()) { /* updateLastFullCompaction */ outputLevel = maxLevel; }
    return CompactUnit.fromLevelRuns(outputLevel, runs.subList(0, runCount));
}
```

逐级推进（输出到 `next.level-1`）避免层级跳跃过大、保持 LSM 平衡；不允许输出 L0 是为保住"L1+ 不重叠"不变式。

> size-ratio 直觉与举例：从最新（通常最小）run 起，尝试把"大小相近"的相邻 run 卷入同一次合并。判据是已积累的候选集大小未超过下一个 run 的 `sizeRatio%`。例如候选集 1GB、`sizeRatio=1`（默认 1%），则只要下一 run ≤ 约 1.01GB 就并入，否则停止——这保证只合并体量同量级的 run，避免"小 run 一遍遍地被卷进大 run 重写"的写放大灾难。`sizeRatio` 越小越积极合并（run 少、读放大低、写放大高），越大越保守。OffPeakHours 正是在此处给 `sizeRatio` 临时加码（见 §7.2）。

**第三级 File Num**：前两级可能因 run 大小悬殊都不触发，但 run 数持续增长会拖垮读取。第三级强制至少合并 `runs.size()-trigger+1` 个 run，是最后的安全网。

> 举例：trigger=5，当前已有 7 个 run（前两级都没选中，因大小差异大）。第三级算 `candidateCount = 7-5+1 = 3`，强制从最新 3 个 run 起做 size-ratio 合并，把 run 数压回 ≤5 附近。这保证 run 数有上界（约 trigger），从而 §5/§5.3 的归并路数有界、读延迟可预测。

**三级的优先级语义**：顺序即优先级——先保空间（amp），再保读性能（ratio），最后兜底文件数（num）。`forcePickL0()`（§8.2，`UniversalCompaction.java:109-122`）是第四条独立路径：收集开头所有 level==0 的 run 强制合并，仅 lookup/MOW/first-row 场景由 `ForceUpLevel0Compaction` 调用。

**与 Table Mode 的联动**：COW（`full-compaction.delta-commits=1`）走的不是这三级，而是 `triggerCompaction(fullCompaction=true)` → `CompactStrategy.pickFullCompaction` 选全部 run 合到 maxLevel（§8.3）；三级策略是 MOR/MOW 的常态增量 compaction。

**一次完整的 run 演化 trace**（trigger=5, sizeRatio=1%, maxSizeAmp=200%, maxLevel=5）——直观看三级如何协作：

```
初始：L0 有 5 个 flush 文件 [run0..run4]，每个约 100MB，高层空
  pick(): runs=5（=trigger，amp 需 runs<trigger 才早退？否，amp 检查 runs>=trigger）
  → size-amp: candidateSize(前4)=400MB, earliest(run4)=100MB, 400*100 > 200*100 ✓ → 全量合到 L5
  结果：L5 = 1 个 ~500MB run（去重后可能更小）

继续写：L0 再积 5 个 100MB → runs = 5(L0) + 1(L5) = 6
  → size-amp: candidate(前5 = 5×L0 + 0)=500MB, earliest(L5)=500MB, 500*100 > 200*500? 否(50000 < 100000)
  → size-ratio: 从最新 run0 起，候选 100MB，下一 run 100MB，100*(101)/100=101 ≥ 100 ✓ 纳入…
    连续把 5 个 100MB 的 L0 卷成候选 500MB，遇到 L5 的 500MB：500*101/100=505 ≥ 500 ✓ 也纳入 → 全量
    （若 L5 已很大如 5GB：505 < 5000，停止，候选 5 个 L0 → 输出到 L5-1=L4）
```

这解释了为什么"小 run 不断被 size-ratio 卷向高层、直到撞上一个大到 sizeRatio 兜不住的高层 run 才停"——逐级填充、写放大可控。

**④ 风险 / 陷阱 / 边界**：误以为 compaction 同步执行（实为异步，flush 只提交任务，§8.3）；`maxSizeAmp` 默认 200%，更新密集表若调高会显著增加空间占用；`sizeRatio` 过大（如 50）会让小 run 难被卷入合并、run 数逼近 trigger 上限、读放大升高。

**⑤ 收益与代价**：三级分层使空间/读/文件数各有侧重，写放大可控。代价是策略本身有调参复杂度，配错（如 sizeRatio 过大）会让 run 数失控。

### 8.2 ForceUpLevel0 / EarlyFullCompaction / OffPeakHours

**ForceUpLevel0Compaction**（`ForceUpLevel0Compaction.java`，装饰 UniversalCompaction）：先试 `universal.pick()`，无果则 `forcePickL0()` 强推 L0。Lookup 场景 L0 越多点查越慢，常规策略可能因 L0 文件太小不合并，故强制清空。两模式（`pick` 在 `:49-77`）：

```java
public Optional<CompactUnit> pick(int numLevels, List<LevelSortedRun> runs) {
    Optional<CompactUnit> pick = universal.pick(numLevels, runs);
    if (pick.isPresent()) return pick;
    if (maxCompactInterval == null) return universal.forcePickL0(numLevels, runs);  // RADICAL：每次推
    compactTriggerCount.getAndIncrement();
    if (compactTriggerCount.compareAndSet(maxCompactInterval, 0))
        return universal.forcePickL0(numLevels, runs);                              // GENTLE：每 N 次推
    return Optional.empty();
}
```

策略选择逻辑（`MergeTreeCompactManagerFactory`）：`needLookup` → ForceUpLevel0（GENTLE 带 interval / RADICAL 无限制）；否则 `compactionForceUpLevel0` → ForceUpLevel0(null)；再否则裸 UniversalCompaction。

**EarlyFullCompaction**（`EarlyFullCompaction.java:86-123`，**旧稿行号 70-107 已修正**）：`tryFullCompact()` 仅当 `runs.size() > 1` 且满足任一条件时全量压缩到 maxLevel，三条件独立：

| 条件 | 配置项 | 触发 |
|------|--------|------|
| 时间间隔 | `optimized-compaction-interval` | `now - lastFullCompaction > interval` |
| 总大小阈值 | `compaction.total-size-threshold` | `sum(all runs) < threshold`（数据少、合并便宜） |
| 增量大小阈值 | `compaction.incremental-size-threshold` | `sum(level != maxLevel runs) > threshold`（增量堆积过多） |

主要为 `changelog-producer=full-compaction` 服务：让用户用明确的时间/大小阈值控制 changelog 及时性，而非被动等 size-amp。

**OffPeakHours**（`OffPeakHours.java:38-46`）：低峰时段给 sizeRatio 加上 `compactOffPeakRatio`，使 §8.1 阈值变 `(100+sizeRatio+offPeakRatio)/100`，纳入更多 run。支持跨天判断（如 22:00–06:00，`startHour > endHour` 分支）。"借低峰偿高峰"：低峰更积极压缩，换高峰更好的读性能。

### 8.3 CompactManager 异步管理与 CompactTask 执行

`MergeTreeCompactManager`（继承 `CompactFutureManager`）管理异步任务。反压双阈值（`MergeTreeCompactManager.java:108-117`）：

```java

public boolean shouldWaitForPreparingCheckpoint()  { return levels.numberOfSortedRuns() > (long) numSortedRunStopTrigger + 1; }
```

`triggerCompaction()`（`:132-201`）：`fullCompaction` 走 `CompactStrategy.pickFullCompaction()`（选全部）；常规则先判 `taskFuture != null`（已有任务在跑则跳过），再由 strategy.pick() 选并过滤掉空 unit / 单文件且层级未变的 unit。`dropDelete` 判定（`:179-183`）——仅当 `!forceKeepDelete && outputLevel != 0 && (outputLevel >= nonEmptyHighestLevel || dvMaintainer != null)` 才能安全丢弃 DELETE 记录（输出到最底层、或用 deletion vector 追踪删除）。

Lookup 模式的额外反压（`:278-284`）：即使无运行中的 compaction，只要 L0 非空就视为"未完成"，逼上层持续触发直至 L0 清空：

```java
public boolean compactNotCompleted() {

}
```

`MergeTreeCompactTask.doCompact()`（`MergeTreeCompactTask.java:81-113`）：先 IntervalPartition 分 section，`section.size > 1`（有重叠）入 candidate 待 rewrite；`size == 1` 则逐文件——小文件（`< minFileSize`）入 candidate，大文件先 rewrite 已积压 candidate 再 `upgrade`。

`doCompact` 的核心循环（保留有序性、不能跳过中间文件）：

```java
for (List<SortedRun> section : partitioned) {
    if (section.size() > 1) {
        candidate.add(section);                                   // 有重叠 → 必须 rewrite
    } else {
        for (DataFileMeta file : section.get(0).files()) {
            if (file.fileSize() < minFileSize) candidate.add(singletonList(SortedRun.fromSingle(file)));  // 小文件凑批
            else { rewrite(candidate, result); upgrade(file, result); }                                   // 大文件：先清积压再 upgrade
        }
    }
}
rewrite(candidate, result);
result.setDeletionFile(compactDfSupplier.get());                  // MOW：附带生成的 DV
```

`upgrade()`（`:123-138`）的"改元数据 vs 重写"决策树：

```java
private void upgrade(DataFileMeta file, CompactResult toUpdate) {
    if ((outputLevel == maxLevel && containsDeleteRecords(file))   // ① 到最底层且含 DELETE → 必须重写清墓碑
            || forceRewriteAllFiles                                // ② 强制全重写
            || containsExpiredRecords(file)) {                     // ③ 含过期记录（record-level expire）
        rewriteImpl(singleton(file), toUpdate); return;
    }
    if (file.level() != outputLevel) {                             // 否则仅改 level 字段，零数据 IO
        toUpdate.merge(rewriter.upgrade(outputLevel, file));       // AbstractCompactRewriter.upgrade（§3.1）
        upgradeFilesNum++;
    }
}
```

`containsDeleteRecords`（`:166-168`）对旧版本无 `_DELETE_ROW_COUNT` 的文件 `orElse(true)` 保守当作"可能有删除"。`containsExpiredRecords`（`:170-172`）委托 `RecordLevelExpire.isExpireFile`（§8.4）。

upgrade 的价值：100MB 无重叠文件晋升，重写要 100MB 读 + 100MB 写，upgrade 仅改 ManifestEntry 的 level 字段，近零开销——这是 §3.1 单文件 upgrade 优化在 compaction task 里的落点。为什么只有"到 maxLevel 且含 DELETE"才重写：DELETE 是墓碑，只有推到最底层（其下再无更旧版本）时才能安全物理删除被它覆盖的行；中间层若提前清掉 DELETE，更高层的旧版本会"复活"。

### 8.4 Record-Level Expire 记录级过期

**① 要解决什么问题**：让表内记录按某个时间字段自动过期（TTL），无需用户写 DELETE，且过期发生在已有的 compaction IO 里、不额外扫描。

**② 设计原理与取舍**：配 `record-level.expire-time` + `record-level.time-field`。过期"搭便车"在 compaction 里完成，因此官方明确"无强时效保证"——不会按秒精确过期，只在该文件被 compaction 时清理。源码上有**两个层次**的过期：① **文件级跳过判定** `isExpireFile()`，决定 compaction 是否需要重写整个文件；② **行级过滤** `wrap()`，在读 KeyValue 流时逐行丢弃过期行。

**③ 关键源码**

文件级判定（`RecordLevelExpire.java:118-152`）用文件的 valueStats **最小时间**判断整个文件是否全过期，且正确处理 schema evolution 与稀疏统计（`metadata.stats-dense-store`）：

```java
public boolean isExpireFile(DataFileMeta file) {
    InternalRow minValues = file.valueStats().minValues();
    if (file.schemaId() != schema.id() || file.valueStatsCols() != null) {
        // schema 演化 或 稀疏统计：不能按原始字段下标直接取 min，需经 SimpleStatsEvolution 转换
        minValues = fieldValueStatsConverters.getOrCreate(file.schemaId())
                .evolution(file.valueStats(), file.rowCount(), file.valueStatsCols()).minValues();
    }
    long currentTime = System.currentTimeMillis() / 1000L;
    return fieldGetter.apply(minValues).map(min -> currentTime - expireTime > min).orElse(false);
}
```

行级过滤（`RecordLevelExpire.java:198-206`）包装 reader，对每条记录取时间字段，过期则丢弃：

```java
private RecordReader<KeyValue> wrap(RecordReader<KeyValue> reader) {
    long currentTime = System.currentTimeMillis() / 1000L;
    return reader.filter(kv -> fieldGetter.apply(kv.value())
            .map(t -> currentTime <= t + expireTime).orElse(true));   // null 时间字段视为不过期
}
```

时间字段类型适配（`:159-196`）：INT 直接当秒；BIGINT 自动识别毫秒（`>= 1e12` 时 `/1000`）；TIMESTAMP/LocalZonedTimestamp 取毫秒 `/1000`；其它类型直接抛错。

**④ 风险 / 陷阱 / 边界**：
- 过期不及时是**设计如此**：若某文件长期不被 compaction，其过期行不会被清理。需要兜底时手动触发 full compaction。
- 文件级判定用 **min 时间**——只有整文件最小时间都已过期才会被识别为全过期文件；部分过期靠行级 filter 在读/重写时清理。
- `record-level.time-field` 字段缺失或类型不支持会在 `create()`（`:65-96`）阶段直接抛 `IllegalArgumentException`。

**⑤ 收益与代价**：过期免额外 IO（搭 compaction 便车）、支持多种时间字段类型。代价是无精确时效保证、依赖 compaction 节奏。

### 8.5 Compaction 配置项全景与调优

> 这是把分散在 compaction.md / write-performance.md 的配置项，按"作用机制 + 调优方向 + 排障信号"整合的源码视角速查。

| 配置项 | 默认 | 作用机制（源码） | 调大效果 | 调小效果 |
|--------|------|-----------------|----------|----------|
| `num-sorted-run.compaction-trigger` | 5 | 触发 compaction 的 run 阈值；也决定默认层数(`+1`)与归并 run 数 | compaction 少触发、写更快、读更慢、归并内存更高 | compaction 频繁、读更快、写更慢 |
| `num-sorted-run.stop-trigger` | trigger+3(=8) | 反压硬阈值(`shouldWaitForLatestCompaction`) | 写停顿少、写吞吐高、查询更吃内存 | 写易被反压、L0 更受控 |
| `sort-spill-threshold` | (none) | 归并 reader 数超此值则溢写(`MergeSorter.spillMergeSort`) | 溢写少、用内存换速度、OOM 风险高 | 早溢写、防 OOM、多磁盘 IO |
| `compaction.max.file-num` / `compaction.min.file-num` | — | 控制单次 compaction 文件数范围 | 单次合并更多、写放大集中 | 合并更碎 |
| `full-compaction.delta-commits` | (none) | =1 即 COW，每次 commit 全量合并 | （仅 1 有特殊语义） | — |
| `compaction.optimization-interval` | (none) | EarlyFullCompaction 时间触发，保 read-optimized 时效 | full compaction 更频、读优化更新 | 读优化数据更旧 |
| `lookup-compact` | radical | Lookup 场景 L0 清理激进度（`ForceUpLevel0Compaction`） | gentle 省资源、freshness 降 | radical L0 即时清、点查快 |
| `lookup-compact.max-interval` | — | gentle 模式强制推 L0 的间隔(`compactTriggerCount`) | 推 L0 更稀疏 | 更频繁 |

**该看哪些 metrics（对照 CompactionMetrics 与 §9）**：`numberOfSortedRuns()`（run 数，反压与读放大直接信号）、`level0FileCount`（`reportLevel0FileCount`，L0 堆积信号）、`totalFileSize`、`compactNotCompleted()`（是否有未完成 compaction）、`bufferPreemptCount`（内存抢占次数，§11）。

**典型排障**：① 读越来越慢 → 看 run 数 / L0 文件数是否持续上涨（compaction 跟不上，扩并行度或调小 trigger）；② 写频繁停顿 → 看是否触达 stop-trigger（调大或加 compaction 资源）；③ compaction 引发 checkpoint 超时 → 桶太少导致单次 full compaction 太重，rescale bucket 或调大 `execution.checkpointing.timeout`（write-performance 文档建议 60min）。

### 8.6 Changelog Producer：四种 changelog 生成机制

**① 要解决什么问题**：主键表的下游（流式消费者）往往需要完整的 `+I / -U / +U / -D` 变更流（changelog），但 LSM 的 merge 默认只产出"合并后最新值"。如何在 merge 过程中额外重建出完整 changelog，是 compaction 与 merge function 协作的关键。

**② 设计原理与取舍**：四种 `ChangelogProducer`（`CoreOptions.java:4051-4060`，默认 NONE），本质是"在哪个时机、用什么代价重建 before/after 对"：

| 模式 | 时机 | 机制（源码） | 延迟 | 写代价 |
|------|------|-------------|------|--------|
| **NONE**（默认） | 不产生 | 下游需自行处理或无 changelog 需求 | — | 0 |
| **INPUT** | flush 时双写 | `MergeTreeWriter.flushWriteBuffer` 用 `rawConsumer` 把 merge 前原始记录写 changelog 文件（§4.2/§4.3） | 最低（与 flush 同步） | 高（每 flush 双写，吞吐≈-50%） |
| **FULL_COMPACTION** | full compaction 时 | `FullChangelogMergeFunctionWrapper` 在全量合并时对比新旧值产出 changelog | 取决于 full compaction 频率 | 中（仅 full compaction 时） |
| **LOOKUP** | lookup compaction 时 | `LookupChangelogMergeFunctionWrapper` 借 LookupLevels 点查旧值，每次 compaction 都能产 changelog | 较低（每次 compaction） | 中高（需点查旧值） |

INPUT 信任上游已是完整 changelog（如 MySQL CDC 含 UPDATE_BEFORE），直接把输入透传为 changelog——最快但要求输入本身完整。FULL_COMPACTION/LOOKUP 则在引擎内重建 changelog：前者搭 full compaction 便车（需 §8.4 EarlyFullCompaction 控延迟），后者用 §6 的点查能力在每次 compaction 时查出被覆盖的旧值生成 `-U`。

**③ 关键源码**

`MergeFunctionWrapper`（`MergeFunctionWrapper.java:30-38`）是产 changelog 的扩展点——它包装基础 `MergeFunction`（如 `DeduplicateMergeFunction`），在 `getResult()` 时除返回合并值外还可副产 changelog：

```java
public interface MergeFunctionWrapper<T> {
    void reset();
    void add(KeyValue kv);
    @Nullable T getResult();   // 不同 wrapper 在此重建 +I/-U/+U/-D
}
```

三个 wrapper 各司其职：`ReducerMergeFunctionWrapper`（无 changelog，仅合并，§4.2 WriteBuffer 用）、`FullChangelogMergeFunctionWrapper`（full compaction 产 changelog）、`LookupChangelogMergeFunctionWrapper`（lookup compaction 产 changelog）。对应 §15 的三种 CompactRewriter。

**④ 风险 / 陷阱 / 边界**：
- INPUT 要求输入流本身完整（含 UPDATE_BEFORE）；若上游只有 upsert（无 before），INPUT 产的 changelog 不完整，下游回撤会出错。
- FULL_COMPACTION 不配 `optimized-compaction-interval`/`full-compaction.delta-commits` 时，changelog 延迟由 size-amp 被动触发，可能数小时（§1.3 陷阱 5）。
- LOOKUP 与 MOW/first-row 共享 lookup compaction 基础设施，三者都会触发 ForceUpLevel0（§8.2）。

**⑤ 收益与代价**：让 Paimon 主键表能当作"带完整 changelog 的流式源"。代价是 INPUT 写翻倍、FULL_COMPACTION 延迟受 compaction 节奏制约、LOOKUP 需点查旧值的额外 IO——三者是延迟/吞吐/资源的三角权衡。

## 9. 反压机制（Backpressure）

**① 要解决什么问题**：compaction 跟不上写入时，自动减速写入，防止 L0 无限堆积引发读取雪崩。

**② 设计原理与取舍**：双阈值"宽松度递减"——flush 阶段用 `stopTrigger`（适度放宽，避免每次 flush 都等），prepareCommit 阶段用 `stopTrigger+1`（更严，因即将生成 snapshot）。注意 `stopTrigger` 本身默认是 `compaction-trigger+3`（§1.3 陷阱 2），这里的 `+1` 是 prepareCommit 阶段在该默认值之上再收紧一格的运行时检查。

**③ 关键源码**

阈值见 §8.3。两个判定方法（`MergeTreeCompactManager.java:108-117`），都看 `numberOfSortedRuns()`（L0 文件数 + L1..Ln 各算 1 个 run）与 `numSortedRunStopTrigger`（即 `num-sorted-run.stop-trigger`，默认 `compaction-trigger+3`）：

```java
public boolean shouldWaitForLatestCompaction() {           // flush 阶段：宽松
    return levels.numberOfSortedRuns() > numSortedRunStopTrigger;
}
public boolean shouldWaitForPreparingCheckpoint() {        // prepareCommit 阶段：严
    // cast to long to avoid Numeric overflow
    return levels.numberOfSortedRuns() > (long) numSortedRunStopTrigger + 1;
}
```

`+1` 看似更宽，实则更严：它在"即将生成 snapshot"的关键点上额外强制一次同步等待。`(long)` 强转是防 `stopTrigger=Integer.MAX_VALUE`（完全异步配方）时 `+1` 整型溢出成负数导致条件恒真。三个触发时机：

- **flush 时**（`MergeTreeWriter.java:212-213`）：`shouldWaitForLatestCompaction()` 为真则置 `waitForLatestCompaction`，`trySyncLatestCompaction()` 阻塞等当前任务。
- **prepareCommit 时**（`MergeTreeWriter.java:251-267`）：先 flush，`commitForceCompact` 或 `shouldWaitForPreparingCheckpoint()` 为真则 `waitCompaction=true`。注释（`:257-261`）说明动机：反复写入失败时 L0 可能已提交但 compaction 重启失败，导致 L0 持续增长，此 +1 检查可避免。
- **flushMemory 时**（`MergeTreeWriter.java:201-207`，内存抢占触发）：优先 `writeBuffer.flushMemory()` 溢写（保留数据在 buffer），不支持溢写才完整 flush。

```java
public void flushMemory() throws Exception {
    boolean success = writeBuffer.flushMemory();   // 优先溢写磁盘
    if (!success) flushWriteBuffer(false, false);  // 不支持则完整 flush
}
```

**Lookup 模式的额外反压**（`compactNotCompleted`，`MergeTreeCompactManager.java:279-283`）：

```java
public boolean compactNotCompleted() {
    // super: 当前是否有未完成的 compaction Future
    return super.compactNotCompleted() || (needLookup && !levels().level0().isEmpty());
}
```

Lookup 模式（`changelog-producer=lookup` 或 `deletion-vectors.enabled=true`，§6/§8）下，**只要 L0 非空就视为"compaction 未完成"**——因为 Lookup 索引建在更高 level，L0 数据未下沉就无法保证点查/DV 生成完整。`MergeTreeWriter.compactNotCompleted()`（`:269-273`）会先 `triggerCompaction(false)` 再判断。它被 `lookup-wait=true`（默认）用于 **commit 阶段决定是否阻塞等 lookup compaction 完成**——这是 lookup 表写入比普通 PK 表更"重"的根因。设 `lookup-wait=false` 可让 commit 不等，吞吐上升但点查/DV 短暂滞后。

**与 Flink checkpoint 的交互**：Flink checkpoint barrier 到达 → 触发 sink 的 `prepareCommit`/`snapshotState`，此时走 `shouldWaitForPreparingCheckpoint()`（+1 严判）+ lookup 表的 `compactNotCompleted`。所以**反压最终体现为 checkpoint 变慢甚至超时**：写太快 → L0 堆积超 stop-trigger → checkpoint 阶段同步等 compaction → barrier 对齐延迟 → 上游 source 被反压减速。这正是反压"自动减速写入"的传导链：不是直接卡 record，而是借 checkpoint 阻塞让 Flink 的信用反压机制回传到源头。

**④ 风险 / 陷阱 / 边界**：
- `num-sorted-run.stop-trigger` 过大（如 100）→ 点查延迟可数量级上升（要归并的 run 多）；Lookup 模式即使无任务在跑、只要 L0 非空就一直反压，属预期行为而非 bug（见 `compactNotCompleted`）。
- **"L0 堆积"排障**：现象是 checkpoint 时间持续上涨 / source 反压高。先看 `numberOfSortedRuns()` 与 `maxLevel0FileCount`（§15.5）是否逼近/超过 stop-trigger。若是 → compaction 追不上：① 加 compaction 线程（`num-sorted-run.compaction-trigger` 调小让早压、`compaction` 相关并行）；② 扩 sink 并行度分摊 bucket；③ 若读延迟可容忍则切完全异步模式（下文）。若 L0 不高但仍慢 → 多半是 lookup-wait 在等（lookup 表正常代价）。
- checkpoint 超时（`execution.checkpointing.timeout`）常因反压同步等 compaction 太久，可适当放宽超时或降写入速率，治本仍是让 compaction 跟上。

监控指标：`numberOfSortedRuns()`、`compactNotCompleted()`、`bufferPreemptCount`、`maxLevel0FileCount`（§15.5）。

**`numberOfSortedRuns` 的精确语义**（`Levels.java:127-134`）：`= level0.size() + 每个非空高层各计 1`。即 **L0 每个文件单独算一个 run**（L0 文件 key 区间互相重叠，读时必须各开一个 reader 归并），而 L1..Ln 每层内部已无重叠、整层只算 1 个 run。这解释了反压为什么紧盯 L0：L0 文件越多，run 数线性增长，既逼近 stop-trigger 触发反压，又让每次读/点查要归并的路数线性上升——读写两头都恶化。所以"控 L0 文件数"是反压机制的核心目标，compaction 的首要任务就是把 L0 文件尽快下沉合并进 L1。

**三种表的反压强度对照**：① 普通 PK 表（无 lookup）——只在 `numberOfSortedRuns > stopTrigger(+1)` 时反压，L0 可短暂堆积；② lookup 表（`changelog-producer=lookup` / DV）——额外 `compactNotCompleted` 让"L0 非空即视为未完成"，反压最频繁、写最重，但读/changelog 最及时；③ append-only 表——无 LSM 无 sorted run，本机制基本不参与（其反压来自小文件 compaction 与 checkpoint）。同一份写入压力下，lookup 表最易触发 checkpoint 变慢，调参时要把这部分代价算进预期而非当成异常。

：若追求最大写吞吐、可接受读期临时变慢，compaction.md 给出配方——`num-sorted-run.stop-trigger=2147483647`（实质关掉反压硬阈值）+ `sort-spill-threshold=10`（归并超 10 路就溢写防 OOM）+ `lookup-wait=false`（commit 不等 lookup compaction）。原理：让 L0 在写高峰自由堆积、compaction 在写低谷慢慢追，把写停顿换成读期短暂的多路归并。代价是高峰期间读会慢、L0 文件多。这与默认"边写边压、用反压兜底"是两种极端，按业务对读/写延迟的敏感度取舍。

**反压与异步 compaction 的关系**：compaction 本就异步（`triggerCompaction` 提交 `Future` 不阻塞，§8.3），反压只在"异步追不上"时才介入同步等待。三个时机层层设防：内存抢占（flushMemory，最轻，优先溢写）→ flush（stopTrigger）→ prepareCommit（stopTrigger+1，最严，因要生成 snapshot）。`commitForceCompact`（`commit.force-compact`）则强制每次 commit 前完成 compaction（COW 类需求）。

**⑤ 收益与代价**：写入自适应减速、避免雪崩。代价是高写入压力下吞吐被主动限制，需通过扩并行度 / 调 compaction 参数 / 切异步模式缓解。

## 10. 数据分布：Partition + Bucket + BucketMode

> 官方 data-distribution / overview 文档讲了"配几个桶、动态桶怎么扩"。本章在其之上讲清：partition+bucket 如何映射到物理文件路径、`KeyAndBucketExtractor` 与 `DefaultBucketFunction` 怎么算 bucket、动态桶的索引如何在 `HashBucketAssigner` 中维护与扩容。

### 10.1 Bucket 到物理路径的映射

**① 要解决什么问题**：把逻辑上的 `(partition, bucket)` 二元组确定性地映射到对象存储上的目录路径，使写入定位、读取裁剪、桶并行都有据可依。

**② 设计原理与取舍**：Bucket 是分区目录下的物理单元，**每桶一棵独立 LSM（主键表）或独立文件集（追加表）**，是写入并行度、数据路由、读取裁剪、LSM 状态隔离四重边界。层级关系：`Table → Partition（逻辑、可整体 drop） → Bucket（物理并行、LSM 隔离） → Level（年龄） → DataFile`。一个表可无 Partition，但主键表必有至少 1 个 Bucket。

物理目录布局（对照 datafile.md 的真实示例）：

```
warehouse/my_db.db/my_table/
  ├─ dt=2026-01-01/
  │    ├─ bucket-0/  data-{uuid}-0.orc  data-{uuid}-1.orc …   ← 桶内含该桶 Level 0..maxLevel 全部数据文件
  │    ├─ bucket-1/  …
  │    └─ bucket-2/  …
  ├─ dt=2026-01-02/ …
  ├─ snapshot/   ← 全表共享：snapshot-N + EARLIEST + LATEST 指针
  ├─ manifest/   ← 全表共享：manifest-* 与 manifest-list-*
  └─ schema/     ← 全表共享：schema-N
```

数据文件名 `data-${uuid}-${id}.${format}`（datafile.md）。POSTPONE 桶目录名特殊为 `bucket-postpone`。

**③ 关键源码**

路径前缀常量与构建（`FileStorePathFactory.java:64`、`:235-248`）：

```java
public static final String BUCKET_PATH_PREFIX = "bucket-";                 // :64

    String bucketName = String.valueOf(bucket);
    if (bucket == BucketMode.POSTPONE_BUCKET) bucketName = "postpone";      // :238  -2 → bucket-postpone
    Path relativeBucketPath = new Path(BUCKET_PATH_PREFIX + bucketName);
    String partitionPath = getPartitionString(partition);                  // dt=2026-01-01
    if (!partitionPath.isEmpty()) relativeBucketPath = new Path(partitionPath, relativeBucketPath);
    // …（外部路径策略时再拼 dataFilePathDirectory）
    return relativeBucketPath;
}
```

**④ 风险 / 陷阱 / 边界**：snapshot/manifest/schema 全表共享（非按分区），故"删一个分区目录"不等于"逻辑删分区"——必须走 partition expire 走 commit 才能让 manifest 一致；直接 rm 分区目录会让 manifest 指向不存在的文件而读失败。

**⑤ 收益与代价**：路径即元数据，写定位 O(1)、读裁剪可按目录跳过分区/桶。代价是分区数 × 桶数过多会产生海量目录与小文件。

### 10.2 KeyAndBucketExtractor 与 bucket 计算

**① 要解决什么问题**：写入侧每条记录都要被路由到确定的 `(partition, bucket)`，且 Flink/Spark 上游需据此做 hash shuffle 保证同桶单 Task 写。

**② 设计原理与取舍**：抽象接口 `KeyAndBucketExtractor<T>`（`KeyAndBucketExtractor.java:31-42`）暴露 `partition() / bucket() / trimmedPrimaryKey()`，按模式有多个实现：`FixedBucketRowKeyExtractor`（固定桶）、`DynamicBucketRowKeyExtractor`（动态桶，bucket 由外部 assigner 决定）、`PostponeBucketRowKeyExtractor`、`AppendTableRowKeyExtractor`。固定桶的核心是 bucket-key 投影 + 哈希取模。

**③ 关键源码**

固定桶 bucket 计算（`FixedBucketRowKeyExtractor`）——bucket-key 默认等于 trimmed 主键，否则用 codegen 投影，再交给 `BucketFunction`：

```java
public int bucket(int numBuckets) {
    return bucketFunction.bucket(bucketKey(), numBuckets);   // bucketKey() 默认=trimmedPrimaryKey
}
```

默认哈希函数（`DefaultBucketFunction.java:29-34`），与官方文档的 `Math.abs(key_hashcode % numBuckets)` 完全一致：

```java
public int bucket(BinaryRow row, int numBuckets) {
    assert numBuckets > 0 && row.getRowKind() == RowKind.INSERT;
    int hash = row.hashCode();
    return Math.abs(hash % numBuckets);   // BinaryRow.hashCode() 基于序列化字节，跨引擎稳定
}
```

另有 `ModBucketFunction`（按单数值列直接取模，用于 `bucket-function.type`）与 `HiveBucketFunction`（兼容 Hive 分桶哈希，跨引擎 bucketed join）。

worked example（bucket=4，主键 `order_id`）：写入 `order_id=42` → `BinaryRow(42).hashCode()` 设为 `h` → `Math.abs(h % 4)` 落到某固定桶（如 2）→ 该行进 `dt=.../bucket-2/`，并由上游 hash shuffle 保证所有 `order_id=42` 的记录都进同一 Task 的同一桶（LSM 状态隔离前提，§10.1）。读 `WHERE order_id=42` 时 scan 据同一公式直接定位 bucket-2，跳过 bucket-0/1/3 的全部文件——这就是桶裁剪。整个链路上游 shuffle、写入路由、读取裁剪用的是同一个 `BucketFunction`，故必须确定性且跨引擎一致。

**④ 风险 / 陷阱 / 边界**：
- bucket-key 默认是主键；若显式配 `bucket-key` 与主键不同，则桶裁剪只对 bucket-key 上的过滤生效。
- `Math.abs(Integer.MIN_VALUE)` 仍为负——但 `% numBuckets` 在哈希为 `MIN_VALUE` 的极端情况由实现保证落在 `[0, numBuckets)`，正常 hash 分布无忧。
- 固定桶数一旦确定不能在线改，改需 offline rescale（INSERT OVERWRITE），否则 ConflictDetection 的 `checkBucketKeepSame`（§13）会因 totalBuckets 变化而拒绝提交。

**⑤ 收益与代价**：基于 `BinaryRow.hashCode()` 的哈希跨引擎确定，支持 bucketed join 免 shuffle（query-performance 文档）。代价是哈希分桶可能数据倾斜，需 `local-merge-buffer-size`（§15.1）缓解。

### 10.3 五种 BucketMode 与动态桶索引维护

**① 要解决什么问题**：覆盖"可预估量、跨分区更新、追加表、无法预估"全场景。

**② 设计原理与取舍**

| 模式 | 配置 | 跨分区更新 | 多 Job 并发 | 桶裁剪 | 启动 |
|------|------|-----------|------------|--------|------|
| HASH_FIXED | `bucket=N(>0)` | 否 | 支持 | 支持 | 快 |
| HASH_DYNAMIC | `bucket=-1` 且主键含全部分区字段 | 否 | 不支持 | 否 | 快 |
| KEY_DYNAMIC | `bucket=-1` 且主键不含全部分区字段 | **支持** | 不支持 | 否 | 慢 |
| BUCKET_UNAWARE | `bucket=-1` 且无主键（追加表） | N/A | 支持 | 否 | 快 |
| POSTPONE_MODE | `bucket=-2` | 否 | 不支持 | 否 | 快 |

**③ 关键源码**

主键表判定（`KeyValueFileStore.java:90-101`）：

```java
public BucketMode bucketMode() {
    switch (options.bucket()) {
        case -2: return BucketMode.POSTPONE_MODE;                                                  // :95
        case -1: return crossPartitionUpdate ? BucketMode.KEY_DYNAMIC : BucketMode.HASH_DYNAMIC;    // :97
        default: return BucketMode.HASH_FIXED;                                                      // :99
    }
}
```

追加表判定（`AppendOnlyFileStore.java:65-68`）：`options.bucket() == -1 ? BUCKET_UNAWARE : HASH_FIXED`。`bucket` 配置（`CoreOptions.java:103-115`，default `-1`）。

**HASH_DYNAMIC 的索引维护**——核心在 `HashBucketAssigner`（`HashBucketAssigner.java:37`）。它对每个分区维护一个 `PartitionIndex`（key hash → bucket 的映射），新 key 来时：先查既有桶；若所有桶都达到 `dynamic-bucket.target-row-num`（默认 2,000,000 行）则分配新桶号，受 `dynamic-bucket.max-buckets` 上限约束。多 assigner 实例按 `assignId / numAssigners / numChannels` 分片，保证同一 key 始终由同一 assigner 处理——这是"动态桶只支持单写 job"的根因（多 job 各自维护索引会把同 key 分到不同桶，data-distribution 文档明确警告，连开 write-only + 专用 compaction 也不行）。`KeyValueFileStore.newWrite()`（`:164-167`）据此创建 `DynamicBucketIndexMaintainer.Factory`。

**KEY_DYNAMIC（跨分区 upsert）**：维护 `pk → (partition, bucket)` 全局索引（RocksDB 本地盘），启动读全表 key 初始化。不同 merge-engine 行为不同（data-distribution 文档）：deduplicate 删旧分区+插新分区；partial-update/aggregation 插旧分区；first-row 有旧值则忽略新值。可配 `cross-partition-upsert.index-ttl` 限制索引规模与初始化时间。性能：1 亿 key/分区约多占 1GB 内存（活跃分区才占）。

**POSTPONE_MODE**：写入先进 `bucket-postpone` 临时桶（`PostponeBucketFileStoreWrite`，构造见 `KeyValueFileStore.java:146-163`），对读不可见；需跑 compaction procedure 把数据搬到正式桶，首次桶数由 `postpone.default-bucket-num`（默认 1）决定，之后可 rescale。

**④ 风险 / 陷阱 / 边界**：见 §1.3 陷阱 1/4；HASH_DYNAMIC/KEY_DYNAMIC 不支持多 Job 并发写同分区与桶裁剪；POSTPONE 数据在 compaction 前对读不可见。

**⑤ 收益与代价**：覆盖全场景，固定桶性能最可预测、动态桶免预估。代价是动态/跨分区桶要维护索引（内存/盘 + 初始化时间），且都丧失桶裁剪能力。

## 11. 内存管理：共享池 + 抢占回收

**① 要解决什么问题**：一个 Job 可能同时有数百上千个 `(partition, bucket)` Writer，独立分配内存会导致利用率极低（多数 Writer 空闲）且数据倾斜时单 Writer OOM。

> 场景：Flink Job 写 100 个分区 × 16 桶 = 1600 个 Writer。若每个独占 256MB 需 400GB 内存且大多空闲；共享池让活跃 Writer 借用空闲者的份额，利用率从约 30% 提到 80%+。

**② 设计原理与取舍**：池化 + 抢占。所有 Writer 共享一个物理内存池 `innerPool`，各自通过 `OwnerMemoryPool` 代理按需取页；池耗尽时抢占**占用内存最多**的 Writer 强制 flush。抢占最大者而非 LRU 的原因：一次释放最多、其 buffer 数据量最大 flush 出的文件更合理（少小文件）、实现仅 O(N)。

**③ 关键源码**

**谁是 owner、怎么注册、契约三方法**。`MemoryOwner`（`MemoryOwner.java:22-31`）是被池管理的单元——实现者主要是各 `(partition,bucket)` 的 `MergeTreeWriter` 持有的 `SortBufferWriteBuffer`。接口只有三个方法，构成共享池的"管理契约"：

```java
public interface MemoryOwner {
    void setMemoryPool(MemorySegmentPool memoryPool);  // 池工厂把"代理子池"注入给 owner
    long memoryOccupancy();                            // 报告自己当前占用 → 抢占据此选最大者
    void flushMemory() throws Exception;               // 被抢占时落盘并归还页
}
```

注册走 `MemoryPoolFactory.addOwners`（`:49-57`，用 `Iterators.concat` 累加多批 owner，不复制）+ `notifyNewOwner`（`:59-61`）：后者为新 owner 调 `createSubPool(owner)` 造一个 `OwnerMemoryPool` 代理并 `setMemoryPool` 注入。`totalPages = innerPool.freePages()` 在工厂构造时一次性确定（`:45`），即全池总页数——所有 owner 共享这一个数字做"全局可用页"判断。抢占时遍历的正是 `addOwners` 累加出来的全体 owner：

抢占（`MemoryPoolFactory.java:73-93`）：

```java
private void preemptMemory(MemoryOwner owner) {
    long maxMemory = 0; MemoryOwner max = null;
    for (MemoryOwner other : owners)
        if (other != owner && other.memoryOccupancy() > maxMemory) {  // 关键：不抢自己
            maxMemory = other.memoryOccupancy(); max = other;
        }
    if (max != null) { max.flushMemory(); ++bufferPreemptCount; }
}
```

不抢自己（`other != owner`）：写入与 flush 同线程，抢自己会清空正在写入的 buffer 造成状态不一致。

代理分配（`OwnerMemoryPool.nextSegment`，`MemoryPoolFactory.java:139-150`）：每个 owner 持一个 `OwnerMemoryPool` 代理，自己记 `allocatedPages`，但 `freePages()` 返回的是 **全局** `totalPages - allocatedPages`（`:135-137`）——即"理论上我还能拿多少页"按全局总量算，而非按本 owner 配额。取页时先问 `innerPool`，拿不到才 `preemptMemory` 再重试一次：

```java
public MemorySegment nextSegment() {
    MemorySegment segment = innerPool.nextSegment();
    if (segment == null) { preemptMemory(owner); segment = innerPool.nextSegment(); }  // 抢占后重试
    if (segment != null) allocatedPages++;
    return segment;
}
public int freePages() { return totalPages - allocatedPages; }  // 全局视角的可用页
```

这种"代理记账 + 全局可用页"设计正是共享语义的来源：单个 owner 可以一直要页直到把整个池吃满，只有池真空了才触发抢占——没有人为的"每 owner 上限"。

**底层物理池：`MemorySegmentPool` / 惰性分配**。`innerPool` 实现是 `HeapMemorySegmentPool extends AbstractMemorySegmentPool`（`HeapMemorySegmentPool.java:22-31`，`allocateHeapMemory` 堆上分配定长页）。物理池本身不预分配全部内存，而是**惰性增长 + 空闲页复用**：

```java
// AbstractMemorySegmentPool.java:34, 40-49, 64-65
this.maxPages = (int) (maxMemory / pageSize);       // write-buffer-size / page-size
public MemorySegment nextSegment() {
    if (this.segments.size() > 0) return this.segments.poll();   // 优先复用已归还的空闲页
    else if (numPage < maxPages) { numPage++; return allocateMemory(); } // 未达上限才真分配新页
    else return null;                                            // 池满，触发上层 preemptMemory
}
public int freePages() { return segments.size() + maxPages - numPage; } // 空闲页 + 尚未分配的额度
```

`maxPages = write-buffer-size / page-size`（page-size 接口默认 32KB，`MemorySegmentPool.DEFAULT_PAGE_SIZE`），即"页数上限"。返回 `null` 是池满信号，逐层上传到 `OwnerMemoryPool.nextSegment` 触发抢占。`returnAll` 把 flush 后的页放回 `segments` 供后续复用——这是为什么 spill 后内存能立刻被其他 owner 借走。

`SortBufferWriteBuffer` 在 `spillable` 时用 `BinaryExternalSortBuffer`（`SortBufferWriteBuffer.java:128-139`），`flushMemory()` 把已排序数据写临时文件、归还内存页（`returnAll` 减 `allocatedPages`），`forEach()` 时外部归并合并内存与磁盘数据。**硬边界**：构造时校验 `memoryPool.freePages() < 3` 直接抛 `IllegalArgumentException("Write buffer requires a minimum of 3 page memory...")`（`SortBufferWriteBuffer.java:119-122`）——排序缓冲至少要 3 页（记录区 + 索引区 + 周转），所以 `write-buffer-size` 不能小于 `3 × page-size`，否则 Writer 起不来。

（write-performance.md "Write Memory" + 源码）：① WriteBuffer 共享池（`write-buffer-size`，本节）；② compaction 归并多个 run 的内存（受 `num-sorted-run.compaction-trigger` 控归并路数，§1.3 陷阱 7）；③ 大行读批 `read.batch-size` 与列存写批 `orc.write.batch-size`。监控/可观测：`bufferPreemptCount()`（`:95`，抢占次数）、`usedBufferSize()`（`:99`，遍历各 owner 求和）、`totalBufferSize()`。

**④ 风险 / 陷阱 / 边界**：
- 不要以为 `write-buffer-size` 是每 Writer 配额（实为共享池总量）；倾斜场景务必开 `write-buffer-spillable=true` 防 OOM。
- `bufferPreemptCount` 持续上涨即池不够：要么加 `write-buffer-size`，要么开溢写。抢占本身是同步 flush，抢占频繁会拖慢写入吞吐（每次抢占 = 一次落盘 + 排序）。
- **`page-size` 调优**：默认 32KB。`maxPages = write-buffer-size / page-size`。页太大（如 64KB+）会让 `freePages() < 3` 边界更易触发、小 Writer 浪费内存；页太小则页数多、`segments` 链表管理开销与碎片上升。一般不动，仅在单行特别大（宽表/大字段）时酌情增大避免单行跨多页。
- **OOM 边界**：共享池只约束 WriteBuffer，挡不住 compaction 同时打开多个 run 的内存（受 `num-sorted-run.compaction-trigger`）、大读批（`read.batch-size`）、列存写批（`orc.write.batch-size`）——这三处不走共享池，OOM 排查别只盯 `write-buffer-size`。
- 若 Flink job 不依赖 state，可 `taskmanager.memory.managed.size=1m` 把内存让给 write buffer；或反过来 `sink.use-managed-memory-allocator=true` 用 Flink managed memory 托管 write buffer 防 OOM（write-performance.md）。
- `write-buffer-size` < `3 × page-size` 时 Writer 构造直接抛异常（上文 3 页下限），极端调小配置要避开。

**⑤ 收益与代价**：内存利用率显著提升、弹性抗倾斜。代价是溢写引入磁盘 IO，靠压缩（`sortCompression`）和 `sortMaxFan` 限制。

| 9 | `_SCHEMA_ID` | BIGINT | N | 写入时 schema 版本 |
| 10 | `_LEVEL` | INT | N | LSM 层级 |
| 11 | `_EXTRA_FILES` | ARRAY(STRING) | N | 附属文件（如 `.lookup`） |
| 12 | `_CREATION_TIME` | TIMESTAMP | N | 创建时间 |
| 13 | `_DELETE_ROW_COUNT` | BIGINT | Y | DELETE 行数 |
| 14 | `_EMBEDDED_FILE_INDEX` | BYTES | Y | 内嵌索引（Bloom 等） |
| 15 | `_FILE_SOURCE` | TINYINT | Y | APPEND(0)/COMPACT(1) |
| 16 | `_VALUE_STATS_COLS` | ARRAY(STRING) | Y | value 统计列名 |
| 17 | `_EXTERNAL_PATH` | STRING | Y | 外部存储路径 |
| 18 | `_FIRST_ROW_ID` | BIGINT | Y | 首行全局 ID（row tracking） |
| 19 | `_WRITE_COLS` | ARRAY(STRING) | Y | 写入时列列表 |

关键字段设计：
- **`_MIN_KEY/_MAX_KEY` 单独存**（不只靠 `_KEY_STATS`）：复合主键 `(a,b)` 的范围比较需字典序整行，单列 min/max 无法推断。用于 SortedRun.validate、二分查找、L0 区间判定、IntervalPartition。
- **`_DELETE_ROW_COUNT`**：`containsDeleteRecords`（`MergeTreeCompactTask.java:166-168`）判 upgrade 是否需重写；旧文件无此字段时 `orElse(true)` 保守。
- **`_FILE_SOURCE`**：流式读可只消费 APPEND 文件拿增量，跳过 COMPACT（仅文件重组织）。
- **`_SCHEMA_ID`**：Schema Evolution，读取时若与当前 schema 不同需经 SchemaManager 取写入时 schema 做列映射；LookupLevels 也据此选正确 PersistProcessor。

**主键表数据文件的物理列布局**（datafile.md）：除用户列外还加系统列。默认 6 列示例（表 `(a PK, b, c)`）：`_KEY_a`（主键列加 `_KEY_` 前缀避免与普通列冲突）、`_VALUE_KIND`、`_SEQUENCE_NUMBER`、`a`、`b`、`c`。开 `data-file.thin-mode` 后省去 `_KEY_` 列、降为 5 列（`_VALUE_KIND`、`_SEQUENCE_NUMBER`、`a`、`b`、`c`），主键从 value 列回取——thin-mode 省存储但读时需从 value 重建 key。追加表无系统列，列即用户列。Changelog 文件与 Data 文件结构完全相同（仅主键表有，类似数据库 binlog）。

**列统计 `_KEY_STATS` / `_VALUE_STATS`（SimpleStats）与 data skipping**：每文件存 key/value 各列的 `min/max/nullCount`（`SimpleStats.SCHEMA`），用于读时按谓词跳过文件。注意稀疏统计：`metadata.stats-dense-store=true` 时只存部分列统计，`_VALUE_STATS_COLS`（字段 16）记哪些列有统计——这正是 §8.4 `RecordLevelExpire.isExpireFile()` 要走 `SimpleStatsEvolution.evolution()` 转换、不能按原始下标直接取 min 的原因（列对不上）。schema 演化时同理需经 `SimpleStatsEvolutions` 把旧文件统计映射到当前列序。

**内嵌文件索引 `_EMBEDDED_FILE_INDEX`（字段 14）**：Bloom/Bitmap/Range-Bitmap 等 file index，小则直接序列化进 DataFileMeta（随 manifest 持久化、读 manifest 即得，省一次文件打开），大则落数据文件目录的独立索引文件（query-performance.md）。这是"读侧按非主键列裁剪文件"的载体（仅 full-compacted 或 MOW 表可用，因 MOR 的 merge 语义不允许非主键列 skip，§5.3）。`_FIRST_ROW_ID` + `nextRowId`（Snapshot）支撑 row tracking / data-evolution（MERGE INTO 的 row-id 冲突检测，§13）。

**④ 风险 / 陷阱 / 边界**：`_DELETE_ROW_COUNT` 对 0.7- 旧文件可能缺失（`orElse(true)` 保守当作"可能有删除"，§8.3）；`metadata.stats-mode=none`（行存 avro 推荐，§14.2）会关掉统计，丧失 data skipping 但提写吞吐——是写/读权衡。

**⑤ 收益与代价**：20 字段一次性提供文件定位、skipping、schema 演化、DV/row-id、外部路径全部元数据，读侧几乎无需打开数据文件即可裁剪。代价是每文件元数据偏大（20 字段 + 统计 + 内嵌索引），海量小文件时 manifest 膨胀——这是控制桶数/文件数的元数据侧动机。

### 12.2 Snapshot 快照与双清单

**① 要解决什么问题**：用一个 JSON 表示表在某时刻的完整状态，支撑时间旅行、增量读、快照隔离、幂等提交，且 commit IO 不随总文件数膨胀。

**② 设计原理与取舍**：base + delta 双清单。`baseManifestList` 记录从创建到上一个 snapshot 的全部存活文件（回放 ADD/DELETE 相消重建全量视图）；`deltaManifestList` 仅记本次变更。写入只写 delta，base 通过 manifest 合并策略定期重建——commit IO 从 O(总文件数) 降到 O(变更文件数)。

```
Snapshot (JSON)
  ├─ baseManifestList ──→ ManifestList ──→ ManifestFileMeta[0..n] ──→ ManifestFile ──→ ManifestEntry(ADD/DELETE + DataFileMeta)
  ├─ deltaManifestList ─→ ManifestList（仅本次变更）
  ├─ changelogManifestList → ManifestList（changelog 文件）
  └─ indexManifest ─────→ IndexManifestFile（DV / 动态桶索引等）
```

> 量化：表有 10 万文件、每次 commit 仅变更 100 文件。单清单设计每次要写 10 万条 ManifestEntry；双清单只写 100 条 delta，base 复用——**IO 减少约 99.9%**，commit 延迟与变更文件数成正比而非总文件数，大表从秒级降到毫秒级。

**③ 关键源码**（`Snapshot.java`，paimon-api）

`CURRENT_VERSION = 3`（`:49`）、`FIRST_SNAPSHOT_ID = 1`（`:47`）。Snapshot JSON 关键字段：`id`、`schemaId`、`baseManifestList(+Size)`、`deltaManifestList(+Size)`、`changelogManifestList`、`indexManifest`、`commitUser`、`commitIdentifier`、`commitKind`、`timeMillis`、`totalRecordCount`、`deltaRecordCount`、`watermark`、`statistics`、`nextRowId` 等。

`commitIdentifier` 语义（`Snapshot.java:128-134` 注释）：相同 commitIdentifier 的多个 snapshot 读出的表必须一致；A 的 commitIdentifier 小于 B 则 A 必先于 B 提交、含更旧记录。用于 Flink checkpoint 恢复去重（`filterCommitted()`）。

`CommitKind` 是 Snapshot 的嵌套枚举（`Snapshot.java:454-470`，**旧稿当作独立文件，已说明**），四值：`APPEND`（新增数据）、`COMPACT`（文件合并/升级，无新数据）、`OVERWRITE`（覆盖写）、`ANALYZE`（统计）。区分 APPEND/COMPACT 的意义：流式读只关心 APPEND；冲突检测逻辑不同；COMPACT 失败仅"错过优化"。

**④ 风险 / 陷阱 / 边界**：别以为 base 与 delta 内容相同（只读 delta 会丢数据）；别让流式消费误读 COMPACT snapshot 造成重复。

**⑤ 收益与代价**：commit 延迟与变更文件数成正比（大表从秒级降毫秒级）、存储省、增量读高效。代价是需定期 manifest 合并维护 base。

### 12.3 Manifest 三层索引

三层：`Snapshot → ManifestList（含多个 ManifestFileMeta）→ ManifestFile（含多个 ManifestEntry）`。

为何三层而非两层：两层时每个 snapshot 要把全部存活文件写一份（IO ∝ 总文件数）。三层让 **ManifestFile 可跨 snapshot 共享**，每次 commit 只写新增/删除的 ManifestEntry 到新 ManifestFile，ManifestList 引用旧+新 ManifestFile，并由 `ManifestFileMerger` 按 `manifest-target-size` 定期合并小文件。

`ManifestEntry.SCHEMA`（`ManifestEntry.java:44-52`）：`_KIND`(TINYINT, ADD=0/DELETE=1)、`_PARTITION`(BYTES)、`_BUCKET`(INT)、`_TOTAL_BUCKETS`(INT)、`_FILE`(DataFileMeta)。一次 flush 产生若干 ADD；一次 compaction 产生 DELETE（旧输入）+ ADD（新输出）。回放全部 entry、ADD/DELETE 相消即可重建任意时刻文件视图——本质是增量日志。

`ManifestFileMeta.SCHEMA`（`ManifestFileMeta.java:44+`）含 `_PARTITION_STATS`、`_MIN/_MAX_BUCKET`、`_MIN/_MAX_LEVEL`、`_MIN/_MAX_ROW_ID` 等统计，用于**清单级裁剪**：查询只涉及特定分区/桶/层时跳过不匹配的 ManifestFile，大表（数万 ManifestFile）显著减少 Scan IO。

**ManifestFileMerger：为什么以及何时合并 manifest**。增量写法的副作用：随着 commit 累积，delta manifest 越来越多（每次 commit 至少一个），ManifestList 引用的 ManifestFile 数量膨胀，scan 时要读越来越多小 manifest。`ManifestFileMerger` 在 commit 时按 `manifest.target-file-size`（目标大小）与 `manifest.merge-min-count`（最少合并数）把若干小 ManifestFile 合并成大的，并在合并时**做一次 ADD/DELETE 全局相消**——把"加了又删"的文件条目直接抵消掉，使合并后的 base manifest 只含当前存活文件。这就是 §12.2 所说"base 通过 manifest 合并策略定期重建"的落点：delta 不断追加，merger 周期性把 delta 折叠进 base。

> 类比：manifest 体系本身就是一棵"元数据的 LSM"——ManifestEntry 是 record（ADD/DELETE 如同 +I/-D），delta manifest 如同 L0、合并后的 base 如同高层，ManifestFileMerger 如同 compaction。理解了主数据的 LSM，元数据层同构。

**索引清单 `indexManifest`**：与 data manifest 平行的一条线，记录非数据文件的索引——MOW 的 deletion vector index（§7.2）、动态桶的 hash index（§10.3）。`IndexManifestEntry` 也走 ADD/DELETE 增量语义，§13 的 ConflictDetection 对 DV index 做专门的 base-ADD/delta-DELETE 配对校验正是基于此。

**④ 风险 / 陷阱 / 边界**：commit 频繁但 manifest 不合并 → scan 读海量小 manifest 变慢（看 `lastScannedManifests` metric，§15.5）；`manifest.target-file-size` 太小则合并频繁、太大则单 manifest 读放大。manifest 也有缓存（`manifestHitCache`/`MissedCache`），大表可调大缓存减 IO。

**⑤ 收益与代价**：三层 + merger 使元数据写入 O(delta)、读取可裁剪、存储可折叠回收。代价是 merger 本身有合并 IO、delta 过多未及时合并会拖慢 scan。

### 12.4 Scan 路径与多级裁剪

**① 要解决什么问题**：从 snapshot 出发，只读出本次查询真正需要的 `ManifestEntry`（进而 DataFileMeta），把"扫全表元数据"降到"扫相关元数据"，并支撑读取的 split 生成。

**② 设计原理与取舍**：`FileStoreScan`（`FileStoreScan.java:50`）是流式 builder，可叠加分区/桶/层/文件名/谓词等过滤，最终 `plan()`（`:112`）产出 `Plan`（含选中的 ManifestEntry → Split）。裁剪在**三个粒度逐级收窄**：

```
Snapshot → ManifestList → [清单级裁剪] ManifestFileMeta.partition/bucket/level stats 跳过整个 ManifestFile
                        → ManifestFile → [条目级裁剪] ManifestEntry.partition/bucket + DataFileMeta key/value stats 跳过单文件
                        → DataFileMeta → [文件内裁剪] embedded file index / 列存 row-group 统计
```

**③ 关键源码**

`FileStoreScan` 的过滤维度（`FileStoreScan.java:52-102`）覆盖：`withPartitionFilter`（分区，多重载支持 Predicate/分区列表/PartitionPredicate）、`withBucket`/`withBucketFilter`/`withTotalAwareBucketFilter`（桶，固定桶可据 `hash%N` 裁剪）、`withLevel`/`withLevelFilter`/`withLevelMinMaxFilter`（层，如 read-optimized 只读高层）、`withKind(ScanMode)`（ALL/DELTA/CHANGELOG，区分全量/增量/changelog 读）、`withManifestEntryFilter`、`enableValueFilter`（开 value 统计过滤）、`dropStats`/`keepStats`（是否保留统计以省内存）。`readSimpleEntries()`（`:118`）只读关键字段、不能做统计过滤，用于提交侧冲突检测（§13 用的就是 SimpleFileEntry）。

`ScanMode` 三值对应读取语义：ALL（全量，合并所有存活文件）、DELTA（仅本次 delta，流式增量读）、CHANGELOG（读 changelog manifest 拿 +I/-U/+U/-D）——这与 §8.6 的 changelog-producer、§12.2 的双清单直接呼应。

**④ 风险 / 陷阱 / 边界**：
- 桶裁剪仅固定桶有效（动态桶映射随数据变化，§10.3），这是固定桶相对动态桶的核心读优势。
- MOR 表对非主键列的 value 谓词不能用于跳过文件（会误过滤待合并旧版本，§5.3）；`enableValueFilter` 主要服务 MOW/COW/append。
- `dropStats` 可在不需要统计的场景省 scan 内存（大表 manifest 统计可观），但会丧失 data skipping。

**⑤ 收益与代价**：三级裁剪使 scan 成本与"命中分区/桶/文件数"成正比而非总量，大表查询元数据 IO 大幅下降。代价是裁剪依赖统计准确性与谓词可下推性；统计关闭（`stats-mode=none`）则退化为全扫。

## 13. 写入提交全流程与冲突检测

**① 要解决什么问题**：分布式多 Writer 并发提交下保证原子性、一致性、幂等性，并在冲突时给出可恢复的失败（而非默默写坏元数据）。

**② 设计原理与取舍**：分两阶段提交（APPEND 数据 + COMPACT 压缩），数据可用性优先于文件优化。原子性靠外部 `SnapshotCommit` 或文件系统原子重命名（乐观锁 + 指数退避重试）。冲突检测是一条**有序短路链**：任一环节发现冲突立即返回异常，让 commit 重读最新 snapshot 后重试。

端到端流程：

```
TableWrite.write(record) → 路由 (partition,bucket) → MergeTreeWriter.write() → WriteBuffer
TableWrite.prepareCommit(id) → 每 writer flush + [反压点2] + drainIncrement() → CommitIncrement(DataIncrement + CompactIncrement)
TableCommit.commit(messages)
   └─ FileStoreCommitImpl.commit()
        ├─ collectChanges()：拆分 append / compact / index
        ├─ tryCommit(APPEND)   ① ConflictDetection ② 写 ManifestFile+ManifestList ③ 创建 Snapshot JSON ④ 原子重命名（乐观锁）
        │      └─ 失败 → 重读最新 snapshot → 指数退避重试（RetryWaiter）
        └─ tryCommit(COMPACT)  同上流程（失败仅"错过优化"，不丢数据）
```

**③ 关键源码**（`FileStoreCommitImpl.java` / `ConflictDetection.java`）

`commit()`（`FileStoreCommitImpl.java:290`，**旧稿标 288-355，已修正**）先 `collectChanges()`（`:685`）分类变更，再分别 `tryCommit()`（`:692`）提交 APPEND 与 COMPACT：

```java
public int commit(ManifestCommittable committable, boolean checkAppendFiles) {
    ManifestEntryChanges changes = collectChanges(committable.fileCommittables());
    if (!changes.appendTableFiles.isEmpty() || ...) {                 // 阶段1：数据
        CommitKind kind = CommitKind.APPEND;                          // :311
        if (conflictDetection.shouldBeOverwriteCommit(...)) kind = CommitKind.OVERWRITE;  // 含 DELETE 文件/DV → 转 OVERWRITE
        tryCommit(changes.appendTableFiles, ..., kind, ...);          // :329
    }
    if (!changes.compactTableFiles.isEmpty() || ...) {                // 阶段2：压缩
        tryCommit(changes.compactTableFiles, ..., CommitKind.COMPACT, ...); // :348/:356
    }
}
```

**ConflictDetection 的有序检查链**（`ConflictDetection.checkConflicts`，`:158-232`）——这是提交正确性的核心，按顺序短路：

1. **DV 富化**（仅 BUCKET_UNAWARE + DV，`:165-186`）：把 base 文件的 ADD-dv 与 delta 的 DELETE-dv 配对，保证"一个文件最多一个有效 DV"（`buildBaseEntriesWithDV`/`buildDeltaEntriesWithDV`，`:545-655`）。
2. **桶数不变检查** `checkBucketKeepSame`（`:267-317`）：同分区 `totalBuckets` 不能在非 OVERWRITE 下变化，否则抛"bucket num changed"（这是固定桶不能在线 rescale 的强制点）。用 LRU 缓存 `sameBucketCheckedPartitions`（上限 1000）避免重复校验。
3. **delta 内部自洽** `mergeEntries(deltaEntries)`（`:204`）：同一文件不能既 ADD 又 DELETE（否则 scan 去重会丢文件）。
4. **删除目标存在性** `checkDeleteInEntries`（`:407-426`）：合并 base+delta 后若仍有 DELETE 项，说明要删的文件并不存在 → 冲突（典型：并发 compaction 删了同一批文件）。但若是 partition expire 导致的删除则放行（`checkConflictForPartitionExpire`，`:428-454`）。
5. **LSM key 范围不重叠** `checkKeyRange`（`:337-388`）：对每个 `(partition, bucket, level≥1)` 分组（`LevelIdentifier`），按 minKey 排序后校验相邻文件 `a.maxKey < b.minKey`，违反即"LSM conflicts detected"——这正是 §3.3 SortedRun 不变式在提交侧的兜底校验。
6. **row-id 范围冲突**（data-evolution 表，`checkRowIdRangeConflicts` `:456` / `checkForRowIdFromSnapshot` `:491`）：多个 MERGE INTO/COMPACT 不能更新同一 row-id 区间。

```java
private Optional<RuntimeException> checkKeyRange(...) {
    if (keyComparator == null || pkClusteringOverride) return Optional.empty();  // 无 key 或聚类排序时跳过
    Map<LevelIdentifier, List<SimpleFileEntry>> levels = new HashMap<>();
    for (SimpleFileEntry e : mergedEntries) if (e.level() >= 1)                  // level 0 允许重叠，不查
        levels.computeIfAbsent(new LevelIdentifier(e.partition(), e.bucket(), e.level()), k -> new ArrayList<>()).add(e);
    for (List<SimpleFileEntry> entries : levels.values()) {
        entries.sort((a, b) -> keyComparator.compare(a.minKey(), b.minKey()));
        for (int i = 0; i + 1 < entries.size(); i++)
            if (keyComparator.compare(entries.get(i).maxKey(), entries.get(i+1).minKey()) >= 0)
                return Optional.of(/* LSM conflicts detected! */);
    }
    return Optional.empty();
}
```

冲突异常很贴心（`createConflictException`，`:662-743`）：列出 base/current commit user、冲突文件、可能原因（多 job 写同分区 / 从旧 savepoint 恢复），并在 entry 超 50 条时生成简化版异常避免日志爆炸。

`filterCommitted()`（`:248`，**旧稿标 246-285**）：用最新 snapshot 的 `commitIdentifier` 比对，`committable.identifier() > latest.commitIdentifier()` 的才需提交，已提交的重放回调——Flink checkpoint 恢复幂等去重的核心。

**乐观锁重试循环 `tryCommit`（`FileStoreCommitImpl.java:692-731`）**：每个阶段（APPEND/COMPACT）都在 while 循环里重试——

```java
private int tryCommit(...) {
    while (true) {
        Snapshot latest = snapshotManager.latestSnapshot();          // 每轮重读最新 snapshot
        CommitChanges changes = changesProvider.provide(latest);     // 基于最新 base 重算变更
        CommitResult result = tryCommitOnce(retryResult, ..., latest, detectConflicts, ...);
        if (result.isSuccess()) break;                               // 原子重命名成功 → 完成
        retryResult = (RetryCommitResult) result;                    // 失败：携带上轮结果继续
        if (now - startMillis > commitTimeout() || retryCount >= commitMaxRetries())
            throw ...;                                               // 超时或超次数才放弃
        retryWaiter.waitBeforeRetry();                               // 指数退避
    }
}
```

`tryCommitOnce` 流程：读最新 snapshot → `ConflictDetection.checkConflicts`（§13 上文的有序链）→ 写新 ManifestFile/ManifestList → 写 Snapshot JSON → **原子重命名 `snapshot-N`**（乐观锁：N 已存在即败、说明被人抢先）。失败回到循环顶重读 latest 重算——这就是乐观并发控制（OCC）：不加锁、冲突时重试。`generatedSnapshot`（`:300`）计数对应 `lastGeneratedSnapshots` metric（§14.5），APPEND+COMPACT 都成功则为 2。超时 `commit.timeout` / 次数 `commit.max-retries` 控制放弃阈值。

> 两 writer 并发模型（basic-concepts.md）：不同分区的并发提交可真正并行（互不冲突）；同分区只保证 snapshot isolation——最终状态可能是两次提交的混合，但不丢更新（OCC + 冲突检测保证）。多 writer 写同分区的标准解法是分离出专用 compaction job（避免双方都触发 compaction 而冲突）。

**④ 风险 / 陷阱 / 边界**：
- 提交冲突是**正常机制而非 bug**（异常文案首句即"Don't panic!"）：乐观锁下并发提交必然偶发冲突，重试即可解决；持续失败多半是多 job 写同分区（应上专用 compaction job）或从旧 savepoint 恢复。
- `checkKeyRange` 只查 level≥1（level 0 允许重叠，§3.1），且 `pkClusteringOverride`（Z-order 聚类）时跳过——因为聚类排序不再保证主键范围不重叠。
- 固定桶在线改桶数会被 `checkBucketKeepSame` 拒绝，必须 batch INSERT OVERWRITE 重排。

**⑤ 收益与代价**：数据先于优化可用、COMPACT 失败容错、冲突可恢复且诊断信息丰富。代价是两次提交两个 snapshot（snapshot 数增长更快，靠 expire 清理）、冲突检测在大 entry 集时有 O(N log N) 排序开销（用 LRU 缓存与 50 条上限缓解）。

## 14. Snapshot / Partition 过期与文件清理

> compaction.md 把"snapshot/partition expiration"列为 compaction 解决的事项之一，但只一笔带过。本章讲清过期的触发、保留策略、以及"删文件时如何不误删仍被存活 snapshot 引用的文件"这一正确性核心。

**① 要解决什么问题**：snapshot/manifest/data 文件只增不减会撑爆存储；但任意删文件可能破坏时间旅行或正在进行的读。需要一套"保留近期 + 安全回收"的机制。

**② 设计原理与取舍**：每次 commit 后按 `snapshot.num-retained.min/max` 与 `snapshot.time-retained` 触发 `ExpireSnapshots.expire()`（`ExpireSnapshots.java:24-30`）。保留逻辑：① 至少留 `num-retained.min` 个；② 超 `time-retained` 的可过期，但保留数不超过 `num-retained.max`（配置见 `CoreOptions.java:495-511`）。过期的不是简单删 snapshot JSON——而是要算出"该 snapshot 独有、且不被任何存活 snapshot 引用"的 data/manifest/changelog 文件才能删。

**③ 关键源码**

**(0) 过期主流程：三道闸门 + 区间删除**（`ExpireSnapshotsImpl.java:128-159, 200-225`）。先算"可过期到哪个 snapshot"（`maxExclusive`），它被三道闸门层层收紧，取最小值：

```java
// 闸门1：保留最近 num-retained.min 个
long maxExclusive = latestSnapshotId - retainMin + 1;
// 闸门2：consumer 正在读的 snapshot 不能删（consumer-id 占位）
if (!isConsumerChangelogOnly())
    maxExclusive = Math.min(maxExclusive, consumerManager.minNextSnapshot().orElse(MAX_VALUE));
// 闸门3：单次最多过期 snapshot.expire.limit 个（避免一次删太多卡 commit）
maxExclusive = Math.min(maxExclusive, earliest + maxDeletes);

for (long id = min; id < maxExclusive; id++) {     // time-retained 早退
    Snapshot nextSnapshot = snapshotManager.tryGetSnapshot(id + 1);
    // 只有当"下一个 snapshot 已存活超过 time-retained"才让 id 过期 —— 更强的保护
    if (olderThanMills <= nextSnapshot.timeMillis()) return expireUntil(earliest, id);
}
```

`time-retained` 的判定很巧：不是看 snapshot 自身年龄，而是看它的**下一个** snapshot 是否已存活够久——保证任意时刻"最近 time-retained 窗口内的历史"完整可时间旅行。`expireUntil` 删 `(beginInclusiveId, endExclusiveId]` 区间（`:200-225`，beginInclusive 自身保留，因下一个 snapshot 还引用它的文件），每个 snapshot 先 `createDataFileSkipperForTags` 算出 tag 引用的跳过集，再 `cleanUnusedDataFiles(snapshot, skipper)`。manifest 清理用 `manifestSkippingSet(skippingSnapshots)`（存活 snapshot + tag 引用的 manifest 全进跳过集），最后 `snapshotManager.deleteSnapshot` 删 JSON + `writeEarliestHint` 更新 earliest 提示文件（加速后续 `earliestSnapshotId` 查询，避免全目录扫描）。

**(a) ADD/DELETE 回放算"该删哪些 data 文件"（正确性核心）**. `SnapshotDeletion extends FileDeletionBase<Snapshot>`（`SnapshotDeletion.java`）。删除不是看文件名，而是回放过期 snapshot 引入的 manifest entry：遇到 ADD 则把该文件从"待删集合"移除，遇到 DELETE 才加入待删——**且 DELETE 时不能立刻删，因为该文件可能被 compaction "升级"（upgrade）到更高 level 后仍被引用**：

```java
// FileDeletionBase.java:221-246
protected void getDataFileToDelete(
        Map<Path, Pair<ExpireFileEntry, List<Path>>> dataFileToDelete,
        List<ExpireFileEntry> dataFileEntries) {
    // we cannot delete a data file directly when we meet a DELETE entry,
    // because that file might be upgraded
    for (ExpireFileEntry entry : dataFileEntries) {
        Path dataFilePath = ...toPath(entry);
        switch (entry.kind()) {
            case ADD:    dataFileToDelete.remove(dataFilePath); break;  // 仍被引用，撤销删除
            case DELETE: dataFileToDelete.put(dataFilePath, ...); break; // 候选删除
        }
    }
}
```

之后还要用 `createDataFileSkipperForTags`（`FileDeletionBase.java:354`）把 **tag 引用的文件**剔出待删集——tag 是显式快照，过期 snapshot 删文件时绝不能动 tag 引用的文件。最终用 `deleteFiles` 多线程（`fileExecutor`，受 `snapshot.expire.thread-num`/`COMMIT_FILE_DELETION_NUM` 控制）并发删，`cleanEmptyDirectories` 顺带清空 bucket/分区空目录。`changelogDecoupled`（`produceChangelog`）决定 changelog 文件是交给 `ChangelogDeletion` 单独保留（供下游消费）还是随 snapshot 一起回收。

**(b) 孤儿文件（orphan）两阶段清理**。孤儿文件成因：写入端写了 data 文件但 commit 失败 / job 崩溃，文件落盘却没进任何 snapshot 的 manifest——它"无主"，expire 的引用回放永远碰不到它，只能靠 `OrphanFilesClean` 扫目录比对。算法是**两阶段安全删除**（`OrphanFilesClean.java:73-85`）：

```java
// 阶段1：列出所有存活 snapshot/tag 引用的 used files（manifest→data）
// 阶段2：遍历表目录下实际文件，凡不在 used 集合、且 mtime < olderThanMillis 的即孤儿
// 关键保护（见类注释）：
//  - 只删 olderThanMillis（默认 1 天前）的文件，避开"正在写、还没 commit"的新文件
//  - 列文件遇 FileNotFoundException（snapshot/tag 正被 expire/rollback）→ 跳过该 snapshot
//  - 读 used files 失败 → 直接停止删除（宁可不删，绝不误删）
```

`olderThanMillis` 校验强制 `olderThan < now`（`:463-473`），否则报错——防止把"当前正在写、还没被 snapshot 引用"的数据文件当孤儿误删。`LocalOrphanFilesClean` 单机执行，分布式版在 Flink/Spark 侧。

**(c) Partition 过期**。`PartitionExpire`（`PartitionExpire.java:140-145`）按 `partition.expiration-time`（`CoreOptions.java:1153`）删整个分区，但用 `partition.expiration-check-interval`（默认 1h）做**节流 + 随机错峰**（`:83-85` 给首次检查加 `rndSeconds` 随机偏移，避免多 writer 同时 expire 打爆元数据）：`now.isAfter(lastCheck.plus(checkInterval))` 才真正 `doExpire`。策略二选一：`PartitionValuesTimeExpireStrategy`（按分区值里的时间，如 `dt=2026-01-01`）或 `PartitionUpdateTimeExpireStrategy`（按分区最后更新时间，写到 `.partition-stats`）。注意 §13 `ConflictDetection.checkConflictForPartitionExpire`（`:428-454`）专门处理"写入已过期分区"的冲突——避免 expire 与写入打架导致 job 反复 failover（异常文案明确提示过滤这些数据）。

**④ 风险 / 陷阱 / 边界**：
- **不要手动 rm snapshot/data 文件**：会让存活 snapshot 指向不存在文件（§10.1）。删 snapshot 必须走 expire 算引用；删孤儿必须走 `OrphanFilesClean`（带 `olderThanMillis` 保护），手动 rm 极易误删正在写的文件。
- **DELETE entry 不等于可删**：升级（upgrade，§5 L0→Ln 不重写仅改 level）的文件会在某 snapshot 出现 DELETE、又在另一 snapshot 出现 ADD，必须全量回放抵消后才知道是否真无人引用。这是为什么不能"看到 DELETE 就删"。
- `num-retained.min` 设太小 + 下游消费慢 → 下游要读的 snapshot 已被过期，流式读报 "snapshot not found"；可用 `consumer-id` 让下游"占住"未消费完的 snapshot 不被过期（consumer 记录的 snapshot 进入 expire 的 skipping 集合）。
- 时间旅行 / tag 依赖未过期 snapshot；要回溯多久就把 `time-retained` / `num-retained.max` 设够大（与存储成本权衡）。tag 一旦打上则永久钉住其引用文件，即使 snapshot 过期也删不掉，长期累积要留意。
- `partition.expiration-check-interval` 设太小会让每次 commit 都扫分区元数据，大表有开销；设太大则过期分区滞留。随机错峰仅缓解并发，单 writer 仍受 interval 节流。

**⑤ 收益与代价**：三套机制分工——expire 回收"曾被引用、现已无人引用"的文件（引用回放保证不误删存活/tag 引用），orphan clean 兜底"从未被引用"的崩溃残留（两阶段 + 时间窗 + 读失败即停保证安全），partition expire 按业务时间维度回收整分区。代价是 expire/orphan 都要回放或全量列 manifest 算引用（大表开销大，建议定时低峰跑）、过期太激进会断时间旅行与流式消费。

## 15. 写入 / 查询性能调优（源码视角）

> 整合 write-performance.md / query-performance.md，但每条都落到"动了哪个源码机制、读/写放大如何变、出问题看什么信号"。

### 15.1 写入吞吐：内存、checkpoint、并行度

**Writer 内存的三处占用**（write-performance.md "Write Memory" + §11）：① WriteBuffer 共享池（`write-buffer-size`，所有 writer 抢占共享，非每 writer 配额）；② compaction 归并时打开多个 run 的内存（受 `num-sorted-run.compaction-trigger` 控制归并路数）；③ 大行读批（`read.batch-size`）与 ORC 写批（`orc.write.batch-size`）。源码上 ① 由 `MemoryPoolFactory` 池化抢占（§11），抢占最大占用者 flush。调优：增大 `write-buffer-size` 让单次 flush 出更大文件（减小文件数→减 compaction 压力），内存不够则开 `write-buffer-spillable=true` 走 `BinaryExternalSortBuffer` 溢写（§4.2）。

**checkpoint 与写吞吐强相关**：Paimon 在 checkpoint 时 prepareCommit→commit 生成 snapshot（§13）。增大 `execution.checkpointing.interval`、`max-concurrent-checkpoints=3` 减少 commit 频次、增大每次 flush 数据量。快照/全量同步阶段可临时关掉 `changelog-producer=lookup/full-compaction` 与 `full-compaction.delta-commits`（这些会显著拖慢写，见 §7.1 COW），增量阶段再开。

**并行度**：`sink.parallelism` 建议 ≤ 桶数且最好相等（write-performance.md）——因为同桶必须单 Task 写（§10.2 LSM 状态隔离），并行度超桶数则多余 Task 空跑。

**数据倾斜的 local merge**：`local-merge-buffer-size`（建议起步 64mb）在 shuffle 前按主键先做一次本地 merge，缓解热点主键频繁更新（write-performance.md）；注意 CDC ingestion 场景不生效。

### 15.2 写期/读期文件格式与压缩

- **行存 AVRO 提升写/compaction 吞吐**：`file.format=avro` + `metadata.stats-mode=none`（统计采集对行存偏贵）。代价是分析查询慢（无列裁剪投影）。折中：`file.format.per.level=0:avro,1:avro` 只把低层（频繁 compaction 的热层）设为 avro，高层保持列存。这对应 §3.1 的层级模型——低层写得频、用行存；高层读得多、用列存。
- **压缩**：默认 zstd level 1；`file.compression.zstd-level` 调到 9 提压缩率但显著降读写速度。
- **大列字典**：compaction 时大列字典编码吃内存，可 `parquet.enable.dictionary=false` / `orc.dictionary.key.threshold=0` 关闭。

### 15.3 查询加速：四条独立路径

主键表查询性能首看 Table Mode（§7）。在此之上有四条 data skipping 路径（query-performance.md）：

1. **主键 filter 裁剪**：固定桶表对主键的过滤可基于 `hash(pk) mod N` 直接定位桶（§10.2），跳过其它桶的全部文件——这是固定桶相对动态桶的核心读优势。
2. **文件索引（file index）**：full-compacted 文件或 MOW 表可用 `file-index.bloom-filter.columns` / `file-index.bitmap.columns` / `file-index.range-bitmap.columns`。索引小则内嵌 manifest（`DataFileMeta._EMBEDDED_FILE_INDEX`，§12.1 字段 14），大则独立文件。Bloom 适合点查、Bitmap 精度高但占空间。存量表可用 `rewrite_file_index` procedure 补建。
3. **聚合下推**：MOW 表 `SELECT COUNT(*)` / `MIN/MAX`（Spark + 默认 stats-mode）可在编译期返回，因为 DV 模式读不需 merge（§7.2）。
4. **Bucketed join**：两表同 bucket 策略 + 同桶数时，Spark `v2.bucketing.enabled` 可免 shuffle（§10.2 哈希跨引擎稳定是前提）。

**关键边界**：MOR 表因 merge 不能对非主键列做 filter 下推（会误过滤旧版本，§5.3）；只有 COW/MOW/Read-Optimized 表才能用非主键列 data skipping。这是"为查询性能选 MOW"的最强理由之一。

### 15.4 Commit 端与初始化优化

- **Commit 内存**：写入量极大时 committer 可能 OOM，用 Flink fine-grained resource 单独给 `sink.committer-memory`（如 300MB）加内存，不必整体抬高 TaskManager。
- **Write 初始化**：writer 启动要读该桶全部历史文件（重建 Levels，§3.2）。大量分区同时写时是瓶颈，开 `sink.writer-coordinator.enabled` 用 Flink coordinator 缓存 manifest 加速（缓存内存 `sink.writer-coordinator.cache-memory` 默认 1GB）。
- **稳定性**：桶太少时 full compaction 可能撑爆默认 10min checkpoint 超时，调 `execution.checkpointing.timeout=60min` 或 rescale bucket。

### 15.5 生产实战：监控指标与故障排查

**关键 metrics（按 group，来自 `CompactionMetrics` / `CommitMetrics` / `ScanMetrics`）**：

| Group | 指标 | 含义 | 异常信号 |
|-------|------|------|---------|
| compaction | `maxLevel0FileCount` | 各桶 L0 文件数峰值 | 持续上涨 = compaction 跟不上写（§9 反压前兆） |

**故障排查 playbook（症状 → 源码侧根因 → 处置）**：

1. **读越来越慢**：`maxLevel0FileCount` / `numberOfSortedRuns` 持续上涨 → compaction 跟不上写（§8/§9）→ 扩 sink 并行度（≤桶数）、调小 `num-sorted-run.compaction-trigger`、或上专用 compaction job。
2. **写频繁卡顿（锯齿状吞吐）**：触达 `num-sorted-run.stop-trigger` 反压（§9）→ 调大 stop-trigger（吃内存）或加 compaction 资源；倾斜则开 local merge（§14.1）。
3. **提交反复失败**：`lastCommitAttempts` 飙升 + 日志 "conflicts detected"（§13）→ 多 job 写同分区 / 旧 savepoint 恢复 → 单写或专用 compaction，或回滚到对应 snapshot。
4. **OOM**：写期看 `bufferPreemptCount`（§11，频繁抢占=池不够）→ 加 `write-buffer-size` / 开 spillable / `sink.use-managed-memory-allocator`；commit 期 → `sink.committer-memory`；归并期 → 调小 `num-sorted-run.compaction-trigger` / `read.batch-size`。
5. **MOW 刚写查不到**：异步 compaction 致 L0 未生成 DV 不可见（§1.3 陷阱 8 / §7.1）→ 用同步 compaction 或接受延迟。
6. **changelog 延迟大**：`full-compaction` 没配 `optimized-compaction-interval` / `full-compaction.delta-commits`（§8.6）→ 配间隔或改 `lookup`/`input`。
7. **checkpoint 超时**：full compaction 太重（桶太少）→ rescale bucket 或调大 checkpoint timeout（§14.4）。

## 16. 工厂创建体系

`KeyValueFileStore` 各组件工厂组织：

```
KeyValueFileStore
  ├─ newWrite() → (POSTPONE → PostponeBucketFileStoreWrite) 否则 KeyValueFileStoreWrite
  │    └─ 每 (partition, bucket) 创建 writer：
  │         ├─ KeyValueFileWriterFactory
  │         ├─ CompactManager：writeOnly → NoopCompactManager；否则 MergeTreeCompactManager
  │         │     ├─ 策略：needLookup → ForceUpLevel0(Universal)；forceUpLevel0 → ForceUpLevel0(Universal,null)；默认 Universal
  │         │     └─ Rewriter：FULL_COMPACTION → FullChangelogMergeTreeCompactRewriter
  │         │                  needLookup → LookupMergeTreeCompactRewriter（含 LookupLevels）
  │         │                  默认 → MergeTreeCompactRewriter
  │         └─ MergeTreeWriter
  ├─ newRead() → MergeFileSplitRead（归并读，§5.3）
  ├─ newBatchRawFileRead() → RawFileSplitRead（跳过 merge 的原始读，Read-Optimized/COW 用）
  ├─ newScan() → KeyValueFileStoreScan
  └─ newCommit() → FileStoreCommitImpl（含 ConflictDetection，§13）
```

### 16.1 要解决什么问题

`FileStore` 是一个**配置驱动的对象图装配器**。同一张表，写路径要不要 compaction、用哪种归并策略、产不产 changelog、点查走不走 lookup、桶模式是 fixed/dynamic/postpone/unaware——这些都由 `CoreOptions` 决定，但 `TableWrite`/`TableRead` 的调用方代码完全一致。工厂体系的职责就是：**把"配置 → 具体实现类"的所有分支收敛到工厂方法里**，让上层只面向 `AbstractFileStoreWrite`/`SplitRead`/`FileStoreScan`/`FileStoreCommit` 这几个抽象接口编程，避免分支逻辑散落到读写主流程。

### 16.2 KeyValueFileStore 的工厂方法

`newWrite()` 第一道分叉就是桶模式（`KeyValueFileStore.java:146`）：

```java
public AbstractFileStoreWrite<KeyValue> newWrite(String commitUser, @Nullable Integer writeId) {
    if (options.bucket() == BucketMode.POSTPONE_BUCKET) {
        return new PostponeBucketFileStoreWrite(...);   // 写时不分桶，留到 compaction 期定桶
    }
    DynamicBucketIndexMaintainer.Factory indexFactory = null;
    if (bucketMode() == BucketMode.HASH_DYNAMIC) {       // 动态桶：维护 key→bucket 索引
        indexFactory = new DynamicBucketIndexMaintainer.Factory(newIndexFileHandler());
    }
    BucketedDvMaintainer.Factory dvMaintainerFactory = null;
    if (options.deletionVectorsEnabled()) {              // MOW：维护删除位图
        dvMaintainerFactory = BucketedDvMaintainer.factory(newIndexFileHandler());
    }
    return new KeyValueFileStoreWrite(...);              // 普通 PK 表
}
```
> `KeyValueFileStore.java:144-193`

注意三个 `Factory` 互斥/可叠加的设计：POSTPONE 直接短路返回另一套 write 实现；HASH_DYNAMIC 与 DV 各自决定是否注入对应的 maintainer 工厂（动态桶 + MOW 可同时成立）。这些工厂以**延迟构造**形态传入——只有真正写到某个 `(partition, bucket)` 时才 new 出索引/DV 维护器，避免空分区白白占内存。

读侧两条独立路径：`newRead()→MergeFileSplitRead`（归并读，MOR 主路径，§5.3）与 `newBatchRawFileRead()→RawFileSplitRead`（`KeyValueFileStore.java:104/115`，跳过 merge 直读底层文件，Read-Optimized 系统表与 COW 批读用）。`newScan()→KeyValueFileStoreScan`（`:196`，内含 `BucketSelectConverter` 做桶裁剪），`newCommit()→FileStoreCommitImpl`（含 ConflictDetection，§13）。

### 16.3 CompactManager 的选择：writeOnly 短路 + 三种 Rewriter

CompactManager 不在 `KeyValueFileStoreWrite` 里 new，而是委托给 `compactManagerFactory.create(...)`。`MergeTreeCompactManagerFactory.create()` 第一行就是 writeOnly 短路（`MergeTreeCompactManagerFactory.java:151`）：

```java
if (options.writeOnly()) {
    return new NoopCompactManager();   // 专用 compaction job 场景：写端只追加不压缩
}
CompactStrategy compactStrategy = createCompactStrategy(options, restoreFiles);
Levels levels = new Levels(keyComparator, restoreFiles, options.numLevels());
MergeTreeCompactRewriter rewriter = createRewriter(...);
return new MergeTreeCompactManager(compactExecutor, levels, compactStrategy, ...,
        options.numSortedRunStopTrigger(), rewriter, ..., options.needLookup(), ...);
```
> `MergeTreeCompactManagerFactory.java:144-189`

**策略分支**（`createCompactStrategy`，`:191`）：`needLookup`（MOW/lookup/first-row）→ `ForceUpLevel0Compaction(UniversalCompaction, compactMaxInterval)`，其中 GENTLE 模式带 `lookupCompactMaxInterval`、RADICAL 模式 interval 为 null（每次都强推 L0，`:195-211`）；否则纯 `UniversalCompaction`，再看 `compactionForceUpLevel0()` 决定要不要外套一层 `ForceUpLevel0Compaction(universal, null)`（`:221-224`）。ForceUpLevel0 的意义是：lookup 类场景 L0 文件会拖慢点查（§6），强制把 L0 尽快上推合并。

**Rewriter 三选一**（`createRewriter`，`:244`）按 changelog/lookup 优先级决策：

```java
if (changelogProducer.equals(FULL_COMPACTION)) {
    return new FullChangelogMergeTreeCompactRewriter(...);     // full-compaction：重写时比新旧值产 changelog
} else if (lookupStrategy.needLookup) {
    // 按 lookupStrategy 选 PersistProcessor：firstRow→PersistEmptyProcessor；
    // DV+需位置→PersistValueAndPosProcessor / PersistPositionProcessor；否则 PersistValueProcessor
    LookupLevels<?> lookupLevels = createLookupLevels(...);    // 本地 KV 索引，§6
    return new LookupMergeTreeCompactRewriter(...);            // 用 lookup 点查产增量 changelog
} else {
    return new MergeTreeCompactRewriter(...);                  // 默认标准 merge-sort
}
```
> `MergeTreeCompactManagerFactory.java:265-346`

三种 CompactRewriter 的语义：`MergeTreeCompactRewriter`（默认，纯 merge-sort 重写）、`FullChangelogMergeTreeCompactRewriter`（`changelog-producer=full-compaction`，full compaction 时对比新旧值产完整 changelog）、`LookupMergeTreeCompactRewriter`（`needLookup`，用 LookupLevels 点查上层产增量 changelog/DV；MOW、`changelog-producer=lookup`、`first-row` 三场景共用，但通过 `PersistProcessor`/`wrapperFactory` 的不同组合区分 persist 内容——firstRow 只存 key 不存 value，DV 模式存位置以便回标删除）。

### 16.4 AppendOnlyFileStore 与扩展点

AppendOnly 侧（`AppendOnlyFileStore.java:96`）按桶模式分叉：`BUCKET_UNAWARE`→`AppendFileStoreWrite`（无桶，compaction 仅做小文件合并），否则 `BucketedAppendFileStoreWrite`（带桶，可叠加 DV 维护器）。其 CompactManager 同样 writeOnly 短路（`BucketedAppendFileStoreWrite.java:100-101` 返回 `NoopCompactManager`），否则 `BucketedAppendCompactManager`（`:118`，按文件大小/数量合并小文件，无 LSM 多层概念）。

工厂的可插拔点是 `KvCompactionManagerFactory` 接口：默认 `MergeTreeCompactManagerFactory`；配 `pkClusteringOverride`（Z-order/Hilbert 多维聚簇优化范围查询）时换 `ClusteringCompactManagerFactory`，此时数据按聚簇键重排而非主键序，§13 ConflictDetection 的 `checkKeyRange` 会相应跳过（key 范围不再单调）。`LookupLevels` 的 `lookupFileCache`（Caffeine，`:365-369`）由工厂持有并在所有 `(partition,bucket)` 间共享，`close()` 时 `invalidateAll()` 统一清理（`:390-395`）——这是把"每桶一个 rewriter，但缓存全表一份"的资源边界放在工厂层管理。

**设计取舍**：把分支收敛到工厂换来主流程极简，但代价是 `create()` 方法参数多、分支深，新增一种 mode（如 POSTPONE）要同时改 `newWrite` 与对应 factory。好处是测试可针对单一工厂方法验证装配正确性，且 `restoreFiles`（恢复时已有文件）能在 `create()` 时就喂给 `Levels`/`estimateLastFullCompactionTime`，让 failover 后的 compaction 策略带着历史状态启动而非从零判断。

## 17. 设计决策总结

| 设计决策 | 为什么 | 收益 / 代价 |
|----------|--------|-------------|
| **LSM 替代 B+Tree** | 数据湖以追加写为主，对象存储不支持原地改 | 写放大降数倍、顺序 IO 充分利用带宽；读放大上升但可异步收敛 |
| **Level 0 用 TreeSet 按 maxSeq 降序** | L0 频繁增删、lookup 需最新优先 | O(log n) 增删 + 自动排序；内存略高 |
| **sequenceNumber 嵌入排序键、write 时分配** | 同 key 多版本需确定性排序 | merge 可流式、O(1) 分配；每行多约 9B |
| **WriteBuffer 可磁盘溢写** | 共享内存 + 倾斜致单 Writer 内存不足 | 防 OOM、弹性大分区；额外溢写 IO |
| **Compaction 异步执行** | 写路径不应被阻塞 | 写延迟稳定毫秒级；需反压兜底 |
| **UniversalCompaction 三级** | 单策略难同时优化空间/读/文件数 | 分层保障各有侧重；调参复杂 |
| **三种 Table Mode（MOR/COW/MOW）** | merge 成本可放写期或读期 | 一配置切换读写性能；语义与 compaction/changelog 交织易错 |
| **Deletion Vector 按行位图** | MOW 要免重写标记被删行 | 读接近 COW 无全量重写；DV 与 compaction 强耦合 |
| **ForceUpLevel0（RADICAL/GENTLE）** | Lookup/MOW/first-row 场景 L0 拖慢点查 | 点查延迟可控；强制合并增写放大 |
| **SortedRun 数量反压 + 双阈值** | 无限 L0 堆积致读取雪崩 | flush 放宽 / commit 严控、自适应减速；限吞吐 |
| **APPEND + COMPACT 分两次提交** | 数据可用性优先于文件优化 | COMPACT 失败不丢数据；snapshot 增长更快 |
| **ConflictDetection 有序短路链** | 乐观锁并发提交需兜底正确性 | 桶数/删除/key 范围/row-id 逐级校验、可恢复；大集合排序开销 |
| **三层 Manifest 索引** | 直存文件列表 IO ∝ 总文件数 | 增量写 O(delta)、ManifestFile 跨 snapshot 共享 |
| **DataFileMeta 内嵌 minKey/maxKey** | 复合主键范围查找需整行比较 | O(log F) 二分定位文件 |
| **data-file.thin-mode** | 主键已在 value 列、key 列冗余 | 省存储；读时需从 value 重建 key |
| **五种 BucketMode + 动态桶索引** | 业务对分桶需求差异极大 | 全场景覆盖；动态/跨分区桶丧失桶裁剪、需维护索引 |
| **bucket=Math.abs(hashCode % N)** | 跨引擎确定性路由 | 支持 bucketed join 免 shuffle；可能数据倾斜 |
| **共享内存池 + 抢占最大占用者** | 多 Writer 隔离浪费严重 | 利用率提升、一次释放最多；抢占带来 flush |
| **LookupFile 本地缓存 + DropFileCallback** | Parquet/ORC 不适合点查 | 微秒级点查、Caffeine LRU 自动管理 + compaction 失效 |
| **OffPeakHours 低峰优化** | 全天均匀压缩浪费低峰资源 | 低峰积极压缩换高峰读性能 |
| **IntervalPartition Section 分割** | 全量归并代价高 | 归并路数降至重叠深度、section 间零开销串联 |
| **Record-Level Expire 搭车 compaction** | TTL 不应额外扫描 | 免额外 IO；无精确时效保证 |
| **commitIdentifier 幂等去重** | Flink checkpoint 恢复需幂等 | 精确去重防重复 |
| **upgrade 只改 level 不重写** | 大文件无重叠时重写浪费 | 近零 IO 晋升、减写放大 |
| **EarlyFullCompaction 三条件** | full-compaction changelog 延迟需可控 | 时间/总量/增量三触发器灵活控时 |
| **远程 Lookup 文件** | failover 重建 lookup 耗时长 | 下载已生成文件加速恢复；compaction 时额外生成上传 |
| **per-level 文件格式（行存低层）** | 低层写频、高层读多 | 低层 avro 提 compaction 吞吐、高层列存利分析 |
