# Apache Paimon 内存管理与溢写机制深度分析

> **版本**：1.5-SNAPSHOT　**源码模块**：`paimon-core`（配置项在 `paimon-api`，Flink 适配在 `paimon-flink`）　**核对日期**：2026-06

**一句话定位**：Paimon 把一个 TaskManager 内成百上千个 Writer 的内存收拢进**一个共享内存池**，靠**抢占式回收 + 磁盘溢写**在有限堆内存里扛住流式 upsert 的写入压力，既不 OOM 又不退化成小文件雪崩。

读完本文你应能回答：① 一个 subtask 写 1000 个桶时，内存是怎么在这些 Writer 之间动态分配的；② "抢占式回收"为什么选占用最大的 Owner、为什么绝不抢占自己；③ 写入路径有两层溢写（buffer spill 与 flushWriteBuffer 兜底），各自在什么条件触发、谁先谁后；④ 写排序缓冲 `BinaryExternalSortBuffer` 与读路径 `MergeSorter` 同样叫"溢写"，本质差别在哪；⑤ `write-buffer-size` / `write-buffer-spillable` / `sort-spill-threshold` / `lookup.merge-buffer-size` 配错会触发哪种生产故障；⑥ Heap 模式与 Flink Managed Memory 模式的取舍，反射访问 offHeapBuffer 的风险在哪；⑦ 三类 OOM（Writer 爆炸 / reader 过多 / lookup 热 key）如何用真实 metric 定位。

## 1. 快速理解（核心问题 / 概念 / 陷阱）

### 1.1 核心问题：千个 Writer 抢一池内存

**① 要解决什么问题**

主键表的每个 `(partition, bucket)` 都是一棵独立 LSM 树，对应一个 `MergeTreeWriter`（写入全流程详见 [01 §4](01-核心存储引擎分析.md)）。一个 Flink subtask 高峰期可能同时持有成百上千个 Writer（如 30 个回填分区 × 256 桶 ≈ 7680 个）。每个 Writer 都要一块内存排序缓冲，如果各自独占固定内存，要么内存被稀释到无法工作（每个分不到 3 页就直接报错），要么热点 Writer 饿死冷 Writer。约束是硬性的：**堆内存有限、不能 OOM、又不能退化成小文件雪崩**。

**② 设计原理与取舍**

Paimon 的答案是"**一池共享 + 抢占回收 + 磁盘兜底**"三件套：

1. **共享池**：所有 Writer 共享一个 `MemorySegmentPool`（默认 `write-buffer-size=256MB`），按页（默认 64KB）先到先得，不预分配配额——空闲 Writer 不占内存，热点 Writer 能多拿。
2. **抢占回收**：某个 Writer 申请不到页时，由 `MemoryPoolFactory` 选出**当前占用最大的另一个** Owner，强制它 `flushMemory()` 把数据落盘腾出内存。
3. **磁盘兜底**：被抢占的 Writer 先尝试把排序缓冲溢写到本地临时文件（`write-buffer-spillable=true`）；连磁盘配额都满了，才退化为直接写出 LSM 数据文件。

核心哲学一句话：**内存是流动的公共资源，不是私有配额；不够用时让占得多的先吐出来，吐到磁盘也不让任务死。**

**③ 一次内存压力的全生命周期**（串起全文各章）

```
1. writer.write(kv)            put 进 SortBufferWriteBuffer（§4），向 OwnerMemoryPool 申请页
2. innerPool.nextSegment()==null   共享池没页了 → 触发 preemptMemory（§3.2）
3. 遍历所有 Owner 选 max         占用最大且非自己的 Writer 被选中（§3.2）
4. max.flushMemory()           被抢者先 buffer spill 到本地磁盘（§5.1）；磁盘满则写出数据文件（§5.2）
5. 重试 nextSegment()           腾出页，当前 Writer 继续写
6. checkpoint / prepareCommit   各 Writer 把缓冲 flush 成 Level 0 文件，metric 记录抢占次数（§3、§10）
```

### 1.3 高频生产陷阱

**陷阱 1：分区 × 桶数爆炸，每个 Writer 分不到 3 页。** `SortBufferWriteBuffer` 构造时硬性要求 `memoryPool.freePages() >= 3`（`BinaryInMemorySortBuffer.java:48,64`），否则抛 `Write buffer requires a minimum of 3 page memory`。`365 分区 × 512 桶` ≈ 18.7 万个潜在 Writer，即便 `write-buffer-size=1GB` 也不够每个分 3 页（192KB）。规避：减少桶数、用 dynamic-bucket（`bucket=-1`），或分批回填。详见 [01 §10](01-核心存储引擎分析.md) 的 bucket 取舍。

**陷阱 2：误把 `memoryOccupancy()` 当实时监控。** 它只在抢占决策时被遍历调用，不是逐条 write 更新的指标。真正的运行时指标在 `writerBuffer` metric 组：`numWriters`、`bufferPreemptCount`、`usedWriteBufferSizeByte`、`totalWriteBufferSizeByte`（`WriterBufferMetric.java:32-36`，**已修正：旧稿杜撰了 `writerCount`/`spillChannelIDs.size()` 等不存在的 metric**）。

**陷阱 3：以为关掉 `write-buffer-spillable` 会 OOM。** 实际不会——`BinaryInMemorySortBuffer.flushMemory()` 恒返回 false，抢占拿不到内存时 `MergeTreeWriter.flushMemory()` 会兜底 `flushWriteBuffer()` 写出数据文件（`MergeTreeWriter.java:202-207`）。代价是频繁 flush → 大量小文件 → compaction 压力暴增（小文件治理见 [11](11-小文件治理与优化.md)、[23](23-Compaction全链路深度分析.md)）。

**陷阱 4：`sort-spill-threshold` 配太小，compaction 频繁溢写已有序数据。** 该阈值控制 compaction 同时打开多少个 reader，无默认值时取 `numSortedRunStopTrigger + 1`（`CoreOptions.java:2867`）。配成 5 会让略多一点的 SortedRun 就触发溢写，而溢写的是**本就有序**的 SST，得不偿失。规避：让它跟随默认、并保证 compaction 不积压。

**陷阱 5：lookup 合并热 key 在无 IOManager 时无上限。** partial-update/aggregation 用 `KeyValueBuffer`，超过 `lookup.merge-records-threshold`（默认 1024）升级为 BinaryBuffer。Flink 有 IOManager 时受 `lookup.merge-buffer-size`（默认 8MB）约束并可溢写；Spark batch 无 IOManager 时走 `UnlimitedSegmentPool`，热 key 版本数无上限，只能靠 JVM 堆扛（`KeyValueBuffer.java:210-214`）。

**陷阱 6：盲目开 `sink.use-managed-memory-allocator` 以为一定更好。** Managed Memory 是 off-heap、对 GC 友好、多表按算子权重自动分配，但要走 Flink `MemoryManager` 分配、且**靠反射读 `offHeapBuffer` 字段**（`MemorySegmentAllocator.java:47-50`，FLINK-32213 未解决前的权宜），Flink 版本升级有失效风险。单表、堆充足时 Heap 模式（默认）更简单。

## 2. 内存所有者：MemoryOwner 三方法契约

**① 要解决什么问题**

抢占式回收的前提是：调度方能问每个内存使用者"你占了多少"，并能命令它"让出来"。`MemoryOwner` 就是这份契约——只有三个方法，刻意做得极薄，这样任何写入器（甚至未来的 reader）都能轻松实现，被纳入统一调度。

**② 设计原理与取舍**

设计成**接口而非抽象类**，是因为 `MergeTreeWriter` 已经实现了 `RecordWriter`，Java 不支持多继承；接口也给了未来扩展的空间。三个方法各司其职：

| 方法 | 语义 | 为什么这样设计 |
|------|------|-----------|
| `setMemoryPool(pool)` | 注入内存池（依赖注入） | Owner 不关心内存来自堆还是 Flink Managed Memory，只通过池接口申请页，解耦内存来源 |
| `memoryOccupancy()` | 返回当前占用字节数 | 抢占的**决策依据**，必须快速返回（纯内存计算，不做 I/O）；只在抢占时被遍历，不是实时监控指标 |
| `flushMemory()` | 落盘释放内存页 | "让出内存"的执行动作，**允许做 I/O**（写磁盘），是抢占链路里唯一可能阻塞的环节 |

一句话设计哲学：**报告要快（纯内存），让出可慢（容许 I/O），调度方只认这两个动作。**

**③ 关键源码**

接口本体极简（`MemoryOwner.java:22`，已修正——旧稿标 paimon-core 路径但未给行号）：

```java
public interface MemoryOwner {
    void setMemoryPool(MemorySegmentPool memoryPool);  // 注入池
    long memoryOccupancy();                            // 报告占用（快）
    void flushMemory() throws Exception;               // 让出内存（容 I/O）
}
```

唯一的核心实现者是 `MergeTreeWriter`（`MergeTreeWriter.java:58`），它在三方法上的落地（行号均已核对）：

- `setMemoryPool()`（:149）：用注入的池 new 一个 `SortBufferWriteBuffer`（§4）。
- `memoryOccupancy()`（:197）：委托 `writeBuffer.memoryOccupancy()`，返回数据区 + 排序索引字节数。
- `flushMemory()`（:202）：先 `writeBuffer.flushMemory()`（buffer spill），**返回 false 才**兜底 `flushWriteBuffer()`（§5.2）。

**④ 风险 / 陷阱 / 边界**

- `memoryOccupancy()` 里**绝不能放 I/O**：它在抢占时被全量遍历（§3.2），一旦慢就放大整条写入链路的延迟。
- `flushMemory()` 是**同步阻塞**的：抢占发生在写入线程的 `nextSegment()` 调用栈里，被抢者落盘期间，发起方的 write 是等着的——这是"内存不足是低频事件"假设下的有意取舍。

**⑤ 收益与代价**

收益：极薄契约让"谁用内存谁就能被统一调度"成为可能，是抢占机制的地基。代价：抢占同步、可能抖动写入延迟；但内存吃紧本就是低频事件，异步抢占带来的并发一致性复杂度更不划算。

## 3. 共享内存池与抢占式回收：MemoryPoolFactory

### 3.1 共享池 vs 配额制

**① 要解决什么问题**

多个 Writer 共用有限内存，怎么分？既要避免某个 Writer 独占导致别人无法工作，又要在负载随时间漂移（凌晨 10 个活跃分区、白天 100 个回填分区）时自动适应。

**② 设计原理与取舍**

两条路线，Paimon 选了共享池：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **配额制**（未采用） | 每个 Owner 预分配固定额度 | 隔离性好、无抢占 | Owner1 空闲时 Owner2 用不了它的额度，利用率低；额度难预估 |
| **共享池 + 抢占**（Paimon） | 所有 Owner 共享一个底层池，先到先得，不够时抢占 | 利用率高、自动适应负载漂移 | 需要抢占机制兜底，可能出现"大 Writer 被反复抢"的不公平 |

一句话哲学：**不预分配、不画地为牢；谁能用就先用，紧张了再让占得最多的吐出来。**

`MemoryPoolFactory` 名为"工厂"实为"代理层"：它把一个底层 `innerPool` 包装成多个 `OwnerMemoryPool`，每个 Owner 拿到一个代理，但物理内存全部来自共享的 `innerPool`（类比"虚拟地址空间共享物理内存"）。

**③ 关键源码**

构造时记录总页数，作为后续 `OwnerMemoryPool.freePages()` 的基准（`MemoryPoolFactory.java:43-47`）：

```java
public MemoryPoolFactory(MemorySegmentPool innerPool) {
    this.innerPool = innerPool;
    this.totalPages = innerPool.freePages();
    this.totalBufferSize = (long) totalPages * innerPool.pageSize();
}
```

`addOwners()` 接收的是 `Iterable<MemoryOwner>`（lazy 迭代器），不是固定列表——这样 Writer 动态增删时，抢占总能遍历到**最新**的 Owner 集合（`MemoryPoolFactory.java:49-57`）。

### 3.2 抢占算法：抢最大、不抢自己

**① 要解决什么问题**

共享池被申请空时，必须有人让出内存。让谁？让多少次？

**② 设计原理与取舍**

算法极简：遍历所有 Owner，挑出**占用最大且不是自己**的那个，命它 `flushMemory()`。

抢"最大"而非最老/随机的依据：设要释放 M 字节，抢最大者（占 X）期望抢占次数 ≈ M/X；抢小 Owner（占 Y≪X）则 ≈ M/Y，远多。**抢占次数 = I/O 次数**，越少越好，还能避免"反复抢小 Owner"的内存颠簸。

| 抢占策略 | Paimon | 优点 | 缺点 |
|------|------------|------|------|
| 抢占最大 Owner | ✓ | 单次释放最多、抢占次数最少 | 对持续高负载的大 Writer 略不公平 |
| 抢占最老 / 随机 | ✗ | 公平 / 实现简单 | 单次释放可能很少，抢占次数多 |
| LRU | ✗ | 符合局部性 | 要维护访问时间戳，得不偿失 |

**③ 关键源码**（`MemoryPoolFactory.java:73-93`）

```java
private void preemptMemory(MemoryOwner owner) {
    long maxMemory = 0;
    MemoryOwner max = null;
    for (MemoryOwner other : owners) {
        // Don't preempt yourself! Write and flush at the same time,
        // which may lead to inconsistent state
        if (other != owner && other.memoryOccupancy() > maxMemory) {
            maxMemory = other.memoryOccupancy();
            max = other;
        }
    }
    if (max != null) { max.flushMemory(); ++bufferPreemptCount; }
}
```

触发点在 `OwnerMemoryPool.nextSegment()`：拿不到页就抢占一次再重试（`MemoryPoolFactory.java:139-150`，已修正——旧稿标"第140-150行"）：

```java
public MemorySegment nextSegment() {
    MemorySegment segment = innerPool.nextSegment();
    if (segment == null) {
        preemptMemory(owner);           // 抢占
        segment = innerPool.nextSegment(); // 重试一次
    }
    if (segment != null) { allocatedPages++; }
    return segment;
}
```

**④ 风险 / 陷阱 / 边界**

- **绝不抢自己**（`other != owner`）：一个 Writer 正在 `write()` 调用栈里申请页，若此时触发自己的 `flushMemory()`，排序缓冲会处于"半写入半 flush"的不一致状态。源码注释直接点明这点——这是自研抢占最容易踩的坑。
- **只重试一次**：抢占后若仍拿不到页，`nextSegment()` 返回 null，由 `SortBufferWriteBuffer.put()` 返回 false，再由 `MergeTreeWriter.write()` 触发 `flushWriteBuffer()` 兜底（§5.2）。
- **抢占风暴**：若所有 Owner 占用都相近（各 10MB），而单次申请需求大，会连续抢多个 Owner，表现为 `bufferPreemptCount` 飙升、写入延迟抖动。规避：增大 `write-buffer-size` 或降低并发 Writer 数。

**⑤ 收益与代价**

收益：内存利用率高、自动适应负载、抢占次数被"抢最大"策略压到最低。代价：抢占同步阻塞、可能不公平、风暴场景下抖动——但都在"内存紧张是低频"的前提下可接受。

### 3.3 底层池四实现与 OwnerMemoryPool 视图

**① 要解决什么问题**

共享池底层从哪要内存？堆、Flink 托管内存、一次性使用、还是无限——不同场景需要不同的底层实现，但上层抢占逻辑不应感知差异。

**② 设计原理与取舍**

统一在 `MemorySegmentPool` 接口下，四个实现各管一类场景：

| 实现类 | 模块 | 归还行为 | 适用场景 |
|--------|------|----------|----------|
| `HeapMemorySegmentPool` | paimon-common | **缓存复用**（归还进 LinkedList） | 默认堆内存写缓冲，反复 allocate/GC 开销大，缓存划算 |
| `FlinkMemorySegmentPool` | paimon-flink | 缓存复用（继承 Abstract） | Flink Managed Memory（off-heap），只在缺页时调 allocator（§8） |
| `CachelessSegmentPool` | paimon-common | **只减计数**，由 GC 回收 | MergeSorter（§6）等一次性读取场景，用完即弃，缓存无意义 |
| `UnlimitedSegmentPool` | paimon-common | —— | `freePages()` 恒返回极大值，lookup 无 IOManager 时的兜底（§7） |

`HeapMemorySegmentPool` 与 `FlinkMemorySegmentPool` 共用父类 `AbstractMemorySegmentPool` 的空闲列表逻辑（`AbstractMemorySegmentPool.java:39-49`）：优先从归还的 segment 取，否则在未超限时新分配，超限返回 null。两者只差一个 `allocateMemory()`——前者 `allocateHeapMemory`，后者走 Flink allocator。

**③ 关键源码：OwnerMemoryPool 的"视图"语义**

`OwnerMemoryPool.freePages()` 返回的是 `totalPages - allocatedPages`——**该 Owner 视角的剩余**，不是全局真实剩余（`MemoryPoolFactory.java:134-137`，已修正——旧稿误把它说成"全局剩余页数"）：

```java
@Override
public int freePages() {
    return totalPages - allocatedPages;  // 该 Owner 的视图，非全局
}
```

这个"视图"语义是关键：`BinaryInMemorySortBuffer` 初始化要求 `freePages() >= 3`（§4），检查的是"这个 Owner 名义上还能不能要到 3 页"，而非全局是否还剩 3 页。理解错会误判初始化失败的根因。

**④ 风险 / 陷阱 / 边界**

- 误读 `freePages()` 为全局剩余 → 排查 3 页报错时找错方向。真实瓶颈是"总页数被太多 Owner 摊薄"，要么加大 `write-buffer-size`，要么减 Writer 数。
- `CachelessSegmentPool` 不缓存，频繁申请会增加 GC 压力，因此只用在生命周期短、不反复申请的读路径。

**⑤ 收益与代价**

收益：一套接口屏蔽四种内存来源，抢占逻辑零感知；归还策略按场景分化（缓存 vs 弃用）兼顾了写路径吞吐与读路径低碎片。代价：`freePages()` 的"视图"语义反直觉，是排障高频误区。

## 4. 写排序缓冲：SortBufferWriteBuffer

### 4.1 延迟排序与排序键设计

**① 要解决什么问题**

主键表要把同 key 的多次更新合并。要合并就得让同 key 记录相邻，最直接是排序。但写入是高频热路径，每条都维护有序结构（如 TreeMap/SkipList）开销太大；用 Java 对象集合存又有沉重的对象头开销和 GC 压力。`SortBufferWriteBuffer` 要在"写得快"和"能高效合并"之间找平衡。

**② 设计原理与取舍**

三个关键决策：

1. **二进制页式存储**：把 KeyValue 序列化成紧凑二进制存进 `MemorySegment`，无 Java 对象头开销、CPU cache 友好。
2. **延迟排序**：写入时 O(1) 直接追加，flush 时才一次性 QuickSort——把排序成本从"每条 O(log n)"挪到"批量一次 O(n log n)"。
3. **normalized key 加速比较**：排序索引条目里存 `8 字节数据指针 + N 字节 normalized key 前缀`，比较时先比前缀（纯内存字节比较），只有前缀分不出胜负才回退到完整记录反序列化比较。

| 存储方案 | 内存占用 | 写入 | 排序 | Paimon |
|------|---------|---------|---------|----------------|
| `ArrayList<KeyValue>` | 高（对象头） | 快 | 慢（对象比较） | ✗ |
| `TreeMap` | 高（红黑树节点） | 慢（插入即排序） | 无需排序 | ✗ |
| 页式二进制 + 索引 | 低（紧凑） | 快（追加） | 快（索引上排序） | ✓ |

排序算法是 Introsort（快排为主，递归过深退化为堆排，小段用插入排序）：`QuickSort.java:115` 元素 `< 13` 走插入排序，`getMaxDepth = 2*ceil(log n)` 超限切 HeapSort——平均 O(n log n) 且避免最坏退化。一句话哲学：**写时只管堆，合并前一次排好；比较先看前缀，省掉绝大多数反序列化。**

**③ 关键源码：排序键的拼接顺序**

排序键不止主键，构造时按 `主键字段 → 用户自定义 sequence → 系统 sequence` 拼接（`SortBufferWriteBuffer.java:80-92`，已修正——旧稿标"第79-91行"）：

```java
// key fields
IntStream sortFields = IntStream.range(0, keyType.getFieldCount());
// user-defined sequence fields（如 sequence.field，需 +keyFieldCount+2 映射到序列化 row）
if (userDefinedSeqComparator != null) {
    IntStream udsFields = IntStream.of(userDefinedSeqComparator.compareFields())
            .map(operand -> operand + keyType.getFieldCount() + 2);
    sortFields = IntStream.concat(sortFields, udsFields);
}
// system sequence field
sortFields = IntStream.concat(sortFields, IntStream.of(keyType.getFieldCount()));
```

序列化后的 row 结构：`[key 字段...] [系统 seq:BigInt] [valueKind:TinyInt] [value 字段...]`。排序键先主键（同 key 聚拢）、再用户 seq（支持自定义去重序）、最后系统 seq（同序按写入顺序），正好让 flush 时的 MergeIterator 能流式合并，无需额外 group-by。

**③（续）三页下限**

构造时硬性检查 `freePages() >= 3`，不足直接抛异常（`SortBufferWriteBuffer.java:119-122`，已修正——旧稿标"第106-109行"）：

```java
if (memoryPool.freePages() < 3) {
    throw new IllegalArgumentException(
        "Write buffer requires a minimum of 3 page memory, ...");
}
```

3 页的由来（`BinaryInMemorySortBuffer.java:48` `MIN_REQUIRED_BUFFERS = 3`）：1 页排序索引区、1 页数据区、1 页跨页写入时的 `SimpleCollectingOutputView` 缓冲。

**④ 风险 / 陷阱 / 边界**

- **3 页下限是 Writer 爆炸的直接死因**：`write-buffer-size` 被太多桶摊薄到每 Writer < 3 页就启动失败（见 §1.3 陷阱 1、§10 场景 1）。
- **长字符串主键退化 normalized key**：UUID 等长前缀主键的 normalized key 前缀易碰撞，频繁回退到完整比较，排序变慢——优先用数值主键。
- **`spillable=true` 时实际用的是 `BinaryExternalSortBuffer`**：构造里 `ioManager != null && spillable ? new BinaryExternalSortBuffer(...) : inMemorySortBuffer`（`SortBufferWriteBuffer.java:128-139`），溢写能力在 §5 展开。

**⑤ 收益与代价**

收益：写入近 O(1)、内存紧凑、排序与合并高效，是高吞吐流式写入的基础。代价：不支持随机点查（必须排序后遍历），但 Paimon 是写多读批的 OLAP 场景，这正合适。

### 4.2 forEach：flush 时同 key 合并

**① 要解决什么问题**

flush 时不能简单把缓冲区原样写盘——同 key 的多个版本应当先合并，既减少落盘数据量，也降低后续 compaction 压力。

**② 设计原理与取舍**

`forEach()` 在排序后的迭代器上跑一个内部 `MergeIterator`：相邻同 key 记录喂给 `MergeFunction` 合并，遇到不同 key 就吐出结果开启新组（`SortBufferWriteBuffer.java:163-177`，已修正——旧稿标"第151-164行"）。`MergeFunction` 的四种 merge engine（deduplicate/partial-update/aggregation/first-row）详见 [08](08-Merge引擎与聚合函数.md)，本文不展开。

**③ 关键源码**（`SortBufferWriteBuffer.java:163-177`）

```java
public void forEach(Comparator<InternalRow> keyComparator, MergeFunction<KeyValue> mergeFunction,
        @Nullable KvConsumer rawConsumer, KvConsumer mergedConsumer) throws IOException {
    MergeIterator mergeIterator =
            new MergeIterator(rawConsumer, buffer.sortedIterator(), keyComparator, mergeFunction);
    while (mergeIterator.hasNext()) {
        mergedConsumer.accept(mergeIterator.next());
    }
}
```

注意有两个 consumer：`rawConsumer` 拿到合并前的每条原始记录（INPUT 模式靠它产 changelog，见 [24](24-Changelog机制全链路分析.md)），`mergedConsumer` 拿合并后结果写数据文件。

**④ 风险 / 陷阱 / 边界**

`MergeIterator` 复用 BinaryRow 对象，`requireCopy` 由 MergeFunction 决定是否拷贝（`SortBufferWriteBuffer.java:216`）——若 MergeFunction 需要跨记录持有引用却没声明 requireCopy，会读到被覆写的脏数据。这是实现自定义聚合函数时的隐藏边界。

**⑤ 收益与代价**

收益：落盘前就完成同 key 合并，减少写放大与 compaction 负担。代价：flush 时多一次归并遍历，但摊到批量上可忽略。

## 5. 磁盘溢写：两层防线

**① 要解决什么问题**

写入速度可能超过内存容量（历史回填、突发流量）。要么 OOM、要么强行把内存数据写成大量小文件——都不可接受。溢写机制让超量数据有处可去，同时尽量不污染最终数据文件。

**② 设计原理与取舍：为什么是两层**

Paimon 在写路径上有两层溢写防线，触发条件和目的截然不同：

| 防线 | 实现 | 触发条件 | 落点 | 目的 |
|------|------|----------|------|------|
| 第一层 buffer spill | `BinaryExternalSortBuffer.spill()` | 内存池缺页（抢占链路）且 `spillable=true` | 本地**临时**文件（排序后、压缩） | 避免 OOM，数据仍"处理中"，后续读回归并 |
| 第二层兜底 | `MergeTreeWriter.flushWriteBuffer()` | 第一层 `flushMemory()` 返回 false（磁盘配额满） | **最终** LSM 数据文件 | 连临时盘都满了，宁可写小文件也不让任务死 |

关键区别：**spill 写临时文件（可回收），flush 写正式数据文件（进快照）**。一句话哲学：**先想方设法把数据暂存到磁盘维持有序归并；实在没地方了，才退而求其次落成正式文件。**

### 5.1 第一层 buffer spill：BinaryExternalSortBuffer

**② 设计原理与取舍：为什么 spill 前先排序**

`spillable=true` 时，`SortBufferWriteBuffer` 内部用的是 `BinaryExternalSortBuffer` 而非纯内存的 `BinaryInMemorySortBuffer`。它的 spill 在落盘**前先 QuickSort 排序**，让每个溢写文件内部有序。这样最终归并只需 N-way merge（小顶堆），无需重新排序——这是经典外部排序模式。代价是 spill 时多一次排序，但换来归并阶段大幅省 CPU。

**③ 关键源码**

`spill()`（`BinaryExternalSortBuffer.java:226-256`，已修正——旧稿标"第248-278行"）：

```java
private void spill() throws IOException {
    if (inMemorySortBuffer.isEmpty()) return;
    FileIOChannel.ID channel = enumerator.next();
    channelManager.addChannel(channel);
    ChannelWriterOutputView output =
            FileChannelUtil.createOutputView(ioManager, channel, compressionCodecFactory, compressionBlockSize);
    new QuickSort().sort(inMemorySortBuffer);   // 落盘前排序
    inMemorySortBuffer.writeToOutput(output);   // 写出（压缩，默认 zstd，block 64KB）
    output.close();
    spillChannelIDs.add(new ChannelWithMeta(channel, output.getBlockCount(), output.getWriteBytes()));
    inMemorySortBuffer.clear();                 // 释放内存
}
```

`write()` 在内存满时自旋：尝试写内存 → 满则 spill → 文件数超阈值则中间合并 → 重试（`BinaryExternalSortBuffer.java:171-192`，已修正——旧稿标"第194-214行"）。

### 5.2 第二层兜底：flushWriteBuffer

**③ 关键源码**

`MergeTreeWriter.flushMemory()`（`MergeTreeWriter.java:202-207`）：

```java
public void flushMemory() throws Exception {
    boolean success = writeBuffer.flushMemory();  // 先试第一层 buffer spill
    if (!success) {                               // 磁盘配额满，第一层失败
        flushWriteBuffer(false, false);           // 兜底：排序合并后写成数据文件
    }
}
```

`writeBuffer.flushMemory()` 返回 false 的判定在 `BinaryExternalSortBuffer.flushMemory()`：已用磁盘 `getDiskUsage()` 超过 `maxDiskSize`（`write-buffer-spill.max-disk-size`，默认无限）就拒绝（`BinaryExternalSortBuffer.java:144-152`，已修正——旧稿标"第166-174行"）。注意 `BinaryInMemorySortBuffer.flushMemory()` 恒返回 false（`BinaryInMemorySortBuffer.java:147`），所以 `spillable=false` 时永远走兜底。

**④ 风险 / 陷阱 / 边界**

- **磁盘配额耗尽 → 小文件雪崩**：`write-buffer-spill.max-disk-size` 配了上限但磁盘不够，第一层失败后大量走第二层 → 每次只攒一点就写出小文件 → compaction 压力暴增。监控数据文件数突增、平均文件大小骤降（治理见 [11](11-小文件治理与优化.md)）。
- **关掉 spillable 不会 OOM 但会小文件化**：见 §1.3 陷阱 3。

**⑤ 收益与代价**

收益：两层防线让"超量写入永不 OOM"成为硬保证。代价：第一层增加临时 I/O，第二层牺牲文件质量换存活——按"先保命再保质量"排序。

### 5.3 溢写文件合并与 N-way 归并

**① 要解决什么问题**

长时间运行持续 spill，溢写文件会越来越多。同时打开太多文件句柄会触发 "Too many open files"，归并 N-way 也越慢。

**② 设计原理与取舍**

溢写文件数达到 `maxNumFileHandles`（`local-sort.max-num-file-handles`，默认 128）时触发中间合并：先做一轮不完整合并（partial round）再做整轮 maxFanIn-way 合并，类似 LSM 的 tiered merge，最小化中间 I/O。读取时用 `BinaryMergeIterator` + 小顶堆 `PartialOrderPriorityQueue` 做 N-way 归并，复杂度 O(总记录数 × log N)，远优于两两合并的 O(总记录数 × N)。

**③ 关键源码**（`BinaryExternalSortBuffer.java:185-189`）

```java
if (spillChannelIDs.size() >= maxNumFileHandles) {
    List<ChannelWithMeta> merged = merger.mergeChannelList(spillChannelIDs);
    spillChannelIDs.clear();
    spillChannelIDs.addAll(merged);
}
```

溢写文件生命周期由 `SpillChannelManager` 统一管理（区分"已创建未打开"与"已打开"channel，`reset()` 时清理），保证异常恢复时临时文件不泄漏。

**④ 风险 / 陷阱 / 边界**

- `local-sort.max-num-file-handles` 太小 → 频繁中间合并 → 合并阻塞写入、磁盘 I/O 打满；合并速度跟不上 spill 速度时文件数仍持续涨。规避：调大该值或加大 `write-buffer-size` 降低 spill 频率。
- 压缩算法权衡：`spill-compression` 默认 zstd（压缩率高、CPU 重），CPU 紧张时改 lz4（更快、压缩率略低）。

**⑤ 收益与代价**

收益：句柄数受控、归并高效、临时文件不泄漏，支撑超大批写入。代价：中间合并是额外 I/O，但相比"句柄耗尽崩溃"是必要保险。

## 6. 读路径溢写：MergeSorter

**① 要解决什么问题**

compaction 要合并多个 SortedRun（SST 文件）。Level 0 积压几百个文件时，同时打开几百个 reader——每个 reader 都带解压缓冲、列缓存，内存数 GB，还可能触发"Too many open files"。必须限制同时打开的 reader 数。

**② 设计原理与取舍：与写路径溢写的本质区别**

读路径也叫"溢写"，但和 §5 的 buffer spill 是两码事：

| 维度 | `BinaryExternalSortBuffer`（写路径） | `MergeSorter`（读/compaction 路径） |
|------|-------------------------|-------------|
| 输入数据 | **无序**的内存数据 | **已有序**的 SST 文件 |
| spill 前是否排序 | **是**（数据本无序） | **否**（SST 已有序，只是序列化转存） |
| 内存池 | `HeapMemorySegmentPool`（缓存复用） | `CachelessSegmentPool`（用完即弃，默认 64MB） |
| 溢写目的 | 避免 OOM | 减少**同时打开的文件句柄数** |

`MergeSorter` 的 spill 只是把 reader 的有序数据序列化到临时文件、释放原文件句柄，不重排序。用 `CachelessSegmentPool` 是因为 sort-merge 读取是流式的、内存段用完即弃，缓存复用无意义、反而增碎片。一句话哲学：**它溢的是"句柄占用"，不是"无序数据"；目标是别同时开太多 reader。**

**③ 关键源码**

触发与阈值（`MergeSorter.java:112-125`、`:79`）：

```java
public <T> RecordReader<T> mergeSort(List<SizedReaderSupplier<KeyValue>> lazyReaders, ...) {
    if (ioManager != null && lazyReaders.size() > spillThreshold) {
        return spillMergeSort(...);   // reader 超阈值才溢写
    }
    return mergeSortNoSpill(...);
}
// spillThreshold = options.sortSpillThreshold()  // 无默认值时 = numSortedRunStopTrigger + 1
```

溢写策略——**优先溢写小文件**（`MergeSorter.java:148-165`）：

```java
sortedReaders.sort(Comparator.comparingLong(SizedReaderSupplier::estimateSize)); // 按大小升序
int spillSize = inputReaders.size() - spillThreshold;
// 大文件保留在 reader，小文件（subList(0, spillSize)）溢写
for (ReaderSupplier<KeyValue> supplier : sortedReaders.subList(0, spillSize)) {
    readers.add(spill(supplier));
}
```

**为什么优先溢写小文件**（反直觉）：溢写目的不是省内存而是减句柄；小文件读写快、溢写开销低，大文件保留在 reader 里直读避免二次 I/O 倍增。`spill()` 内把 reader 数据用 `KeyValueWithLevelNoReusingSerializer` 序列化到临时 channel，读回时不需排序（`MergeSorter.java:167-198`）。内存池在构造时独立创建，**与写 buffer 的池完全隔离**（`MergeSorter.java:83-84`）。

**④ 风险 / 陷阱 / 边界**

- `sort-spill-threshold` 太小 → 略多 SortedRun 就溢写已有序数据，纯浪费 I/O；太大或无 IOManager → 所有 reader 同开，OOM 风险。它无默认值，跟随 `numSortedRunStopTrigger + 1`（`CoreOptions.java:2867-2874`）。
- 误以为 MergeSorter 会"排序"——它只转存有序数据，不排序。
- `sort-spill-buffer-size`（默认 64MB）是这个独立池的大小，溢写时增大可减少 I/O 次数。

**⑤ 收益与代价**

收益：compaction 同时打开的 reader 数可控，避免句柄耗尽与 OOM，支撑大规模合并。代价：溢写增加读 I/O、compaction 延迟略升——但 compaction 是后台任务，延迟可接受。compaction 全链路见 [23](23-Compaction全链路深度分析.md)。

## 7. Lookup 合并缓冲：KeyValueBuffer 三层混合

**① 要解决什么问题**

Lookup 类 merge（partial-update、aggregation，详见 [08](08-Merge引擎与聚合函数.md)）合并某个 key 时，要缓存它在不同层级的多个候选版本。绝大多数 key 只有 1–3 个版本，但热 key（测试账号、爬虫、爆款商品）可能有数千个版本。全用 ArrayList 则热 key OOM，全用二进制则普通 key 浪费 CPU。

**② 设计原理与取舍：三层自适应**

`KeyValueBuffer.HybridBuffer` 按数据量动态升级存储形态：

| 形态 | 存储 | 适用 | 触发 |
|------|------|------|------|
| `ListBuffer` | `ArrayList<KeyValue>` | 版本数 ≤ threshold（绝大多数 key） | 默认起点 |
| `BinaryBuffer`(InMemory) | `InMemoryBuffer` + 内存页 | 无 IOManager（Spark batch） | 超 threshold 升级 |
| `BinaryBuffer`(External) | `ExternalBuffer` + 磁盘溢写 | 有 IOManager（Flink streaming） | 超 threshold 升级 |

threshold = `lookup.merge-records-threshold`（默认 1024）。`LazyField` 保证只有真正触发升级时才创建 BinaryBuffer，普通 key 不付出重量级资源代价。一句话哲学：**为常见情况优化（小 key 用对象集合），为极端情况兜底（热 key 转二进制 + 溢写）。**

**③ 关键源码**

升级逻辑（`KeyValueBuffer.java:85-95`，已修正——旧稿标"第86-94行"）：

```java
public void put(KeyValue kv) {
    if (binaryBuffer != null) {
        binaryBuffer.put(kv);
    } else {
        listBuffer.put(kv);
        if (listBuffer.list.size() > threshold) {
            spillToBinary();   // 超阈值，整体迁到二进制 buffer
        }
    }
}
```

BinaryBuffer 的内存来源按引擎分化（`KeyValueBuffer.java:203-226`）：

```java
MemorySegmentPool pool = ioManager == null
        ? new UnlimitedSegmentPool(options.pageSize())              // Spark batch：无上限
        : new HeapMemorySegmentPool(options.lookupMergeBufferSize(), // Flink：8MB 上限
                                    options.pageSize());
RowBuffer buffer = ioManager == null
        ? new InMemoryBuffer(pool, serializer)                       // 纯内存
        : new ExternalBuffer(ioManager, pool, serializer,            // 可溢写
                options.writeBufferSpillDiskSize(), options.spillCompressOptions());
```

**ExternalBuffer 溢写不排序、零拷贝**——它存的是同 key 不同版本，无需全局有序，spill 时直接把内存段写盘（`ExternalBuffer.java:147-182`）：

```java
for (int i = 0; i < numRecordBuffers; i++) {
    MemorySegment segment = segments.get(i);
    int bufferSize = i == numRecordBuffers - 1
            ? inMemoryBuffer.getNumBytesInLastBuffer() : segment.size();
    channelWriterOutputView.write(segment, 0, bufferSize);  // 零拷贝
}
```

**④ 风险 / 陷阱 / 边界**

- **无 IOManager 时（Spark batch）无内存上限**：走 `UnlimitedSegmentPool`，热 key 版本数无界，只能靠 JVM 堆，堆不够直接 OOM（见 §10 场景 3）。规避：用 Flink streaming（有 IOManager）、或及时 compaction 控制版本数。
- **threshold 配太小适得其反**：大量 key 被升级为 BinaryBuffer，序列化开销反而拖垮 CPU、增加元数据开销。版本通常 < 100 的表配大点（如 10000）更合适。
- `spillToBinary()` 是同步的：一次性序列化最多 threshold 个对象，热 key 大字段场景会抖动写入延迟。

**⑤ 收益与代价**

收益：99% 普通 key 走轻量 ArrayList，1% 热 key 自动转二进制 + 溢写不 OOM，兼顾性能与内存。代价：实现复杂、有升级开销；Spark batch 下缺溢写兜底是已知边界。

## 8. Flink 内存集成：Heap vs Managed Memory

**① 要解决什么问题**

Paimon 作为 Flink Sink 时，内存可以来自 JVM 堆，也可以来自 Flink 托管的 off-heap Managed Memory。前者简单但占堆、影响 GC，且和 RocksDB State Backend 抢同一块堆；后者由 Flink 按算子权重统一分配、对 GC 友好，但要走 Flink 的 `MemoryManager`。需要让两种来源在同一套抢占框架下工作。

**② 设计原理与取舍**

| 维度 | Heap 模式（默认） | Managed Memory 模式 |
|------|----------|---------------------|
| 开关 | —— | `sink.use-managed-memory-allocator=true` |
| 内存来源 | JVM 堆（`write-buffer-size`） | Flink off-heap（按算子权重分配） |
| 页大小 | `page-size`（默认 64KB） | Flink `MemoryManager` 页大小 |
| GC 影响 | 占堆、影响 GC | off-heap、无 GC 影响 |
| 多表写入 | 需手动配每表内存 | 自动按权重分配 |
| 实现成本 | 直接分配 | 走 MemoryManager + **反射读 offHeapBuffer** |
| 适用 | 单表、堆充足、配置简单 | 多表、需精细管理、State 占堆大 |

默认 Heap 是为了向后兼容与简单；跨引擎（Spark）也没有 Managed Memory 概念。一句话哲学：**抢占框架不变，只换底层池实现；off-heap 的代价是一处反射兼容。**

**③ 关键源码**

内存来源在 `PrepareCommitOperator.setup()` 决策（`PrepareCommitOperator.java:74-90`，已修正——旧稿标"第74-89行"且方法名标错）：

```java
MemorySegmentPool memoryPool;
if (options.get(SINK_USE_MANAGED_MEMORY)) {
    MemoryManager memoryManager = containingTask.getEnvironment().getMemoryManager();
    memoryAllocator = new MemorySegmentAllocator(containingTask, memoryManager);
    memoryPool = new FlinkMemorySegmentPool(
            computeManagedMemory(this), memoryManager.getPageSize(), memoryAllocator);
} else {
    CoreOptions coreOptions = new CoreOptions(options);
    memoryPool = new HeapMemorySegmentPool(coreOptions.writeBufferSize(), coreOptions.pageSize());
}
memoryPoolFactory = new MemoryPoolFactory(memoryPool);  // 之后流程与 Heap 模式完全一致
```

`FlinkMemorySegmentPool` 只是 `AbstractMemorySegmentPool` 的子类，复用父类空闲列表，只在缺页时调 `allocator.allocate()`（`FlinkMemorySegmentPool.java`）。`SINK_USE_MANAGED_MEMORY` 配置项名为 `sink.use-managed-memory-allocator`，默认 false（`FlinkConnectorOptions.java:175`）。

**③（续）反射桥接的关键与风险**（`MemorySegmentAllocator.java:41-71`）

```java
// 构造时反射拿到 Flink MemorySegment 的私有 offHeapBuffer 字段

        org.apache.flink.core.memory.MemorySegment.class.getDeclaredField("offHeapBuffer");

// allocate 时：Flink 分配 → 校验 off-heap → 反射取 ByteBuffer → 包成 Paimon MemorySegment

```

两个要点：① **反射读私有字段**是 FLINK-32213（`getOffHeapBuffer` 尚不可用）解决前的权宜，Flink 版本升级有失效风险；② 必须保留 `allocatedSegments`（Flink 原始段引用），否则 `release()` 无法归还给 Flink。

**③（续）内存流转**

`MemoryPoolFactory` 创建后由 `MemoryFileStoreWrite.withMemoryPoolFactory()` 接管，`memoryOwners()` 返回所有活跃 Writer 的 lazy 迭代器（`MemoryFileStoreWrite.java:84-109`，已修正）。外部未传入时，`notifyNewWriter()` 兜底创建默认堆池工厂（`MemoryFileStoreWrite.java:120-127`）。

**④ 风险 / 陷阱 / 边界**

- **盲目开 Managed Memory 未必更快**：要走 MemoryManager（有锁）、靠反射、页大小由 Flink 定。单表 + 堆充足时 Heap 更简单。
- **Managed Memory 不足时静默降级到抢占**：`allocate()` 拿不到页 → 池返回 null → 触发抢占。`bufferPreemptCount` 异常高，可能是 `taskmanager.memory.managed.size` 不够。
- **与 RocksDB State Backend 抢内存**：两者都用 Managed Memory 时按权重分；建议要么都用 Managed、要么 State 用 Managed 而 Paimon 用 Heap，别一半一半导致配置割裂。

**⑤ 收益与代价**

收益：同一抢占框架兼容两种内存来源，多表写入下 off-heap + 权重分配显著降低 GC 压力。代价：Managed 模式实现复杂、依赖反射、有版本兼容风险。

## 9. 文件滚动写入：RollingFileWriter

**① 要解决什么问题**

flush/compaction 写出数据文件时，单文件不能任意大也不能任意小：太大查询要读大量无关数据、compaction 内存吃紧、文件级 min/max 统计过滤失效；太小则文件数爆炸、元数据膨胀、句柄耗尽。`RollingFileWriter` 在写入过程中按目标大小自动切文件。

**② 设计原理与取舍**

两个关键决策：

1. **目标大小按表类型分**：主键表默认 128MB、append 表默认 256MB（`target-file-size` 无默认值，按是否有主键取）。主键表要频繁 compaction，文件小一点降低单次合并 I/O 与内存；append 表不常 compaction，文件大一点减少文件数。
2. **采样检查而非每条检查**：每 `CHECK_ROLLING_RECORD_CNT = 1000` 条才查一次当前文件字节数。因为 `reachTargetSize()` 要查 Parquet/ORC 内部统计、可能触发 flush，每条都查太贵。

| 检查策略 | 频率 | 开销 | Paimon |
|---------|------|------|--------|
| 每条检查 | 100% | 高（每条查格式统计） | ✗ |
| 采样检查（1/1000） | 0.1% | 低 | ✓ |
| 不检查 | 0 | 无 | ✗（无法控大小） |

一句话哲学：**目标大小按 compaction 频率分档；用千分之一采样把"测量成本"降到可忽略，容忍微小超出。**

**③ 关键源码**

采样逻辑（`RollingFileWriterImpl.java:64-67`）：

```java
private boolean rollingFile(boolean forceCheck) throws IOException {
    return currentWriter.reachTargetSize(
            forceCheck || recordCount % CHECK_ROLLING_RECORD_CNT == 0, targetFileSize);
}
```

常量定义在接口（`RollingFileWriter.java:43` `int CHECK_ROLLING_RECORD_CNT = 1000;`）。普通 `write()` 传 `forceCheck=false` 走采样（`:80`），`writeBundle()` 传 `forceCheck=true` 强制检查（`:105`）——因为一次 bundle 可能含数万条，不强制检查会大幅超目标。目标大小取值（`CoreOptions.java:713`，`targetFileSize(boolean hasPrimaryKey)` 主键 128MB / append 256MB）。

**④ 风险 / 陷阱 / 边界**

- **采样导致实际文件略超目标**：超出量取决于最后 1000 条的大小。单条很大（含 BLOB）时可能超出明显——大记录场景应调小 `target-file-size`。
- `target-file-size` 太大 → 过滤效果差、compaction 内存高；太小 → 文件数爆炸、句柄耗尽。建议主键表不低于 64MB。

**⑤ 收益与代价**

收益：文件大小受控，兼顾查询过滤、compaction 开销与元数据规模。代价：采样有微小精度损失（通常 < 1%），可忽略。

## 10. OOM 排查与调优手册

**① 要解决什么问题**

内存类故障在生产里最常见也最易"治标不治本"——一 OOM 就加内存。本节给出"先看 metric 定位类型、再对症调参"的系统化路径，并明确哪些是真实指标、哪些是旧稿杜撰。

**② 先看真实 metric，再决定动作**

Paimon 暴露的写缓冲指标全在 `writerBuffer` metric 组（`WriterBufferMetric.java:32-36`），**只有这四个**（已修正——旧稿杜撰了 `writerCount`、`spillChannelIDs.size()` 等不存在的指标）：

| metric | 含义 | 异常信号 |
|--------|------|---------|
| `numWriters` | 当前活跃 Writer 数 | 过高 → 分区/桶爆炸 |
| `bufferPreemptCount` | 累计抢占次数 | 持续快速增长 → Writer 间内存竞争激烈 |
| `usedWriteBufferSizeByte` | 已用写缓冲字节 | 长期顶满 `total` → 内存不足 |
| `totalWriteBufferSizeByte` | 写缓冲总字节 | 对照 used 看水位 |

排查顺序：**先看 metric 缩小范围 → 再决定加内存 / 减 Writer / 调溢写**，而非一上来 dump heap。

**③ 三类典型 OOM 及对症**

| 场景 | 根因 | 定位信号 | 对症 |
|------|------|---------|------|
| **Writer 数量爆炸** | 高基数分区 × 大桶数，每 Writer 至少 3 页（192KB），256MB 摊不开（§1.3 陷阱 1） | `numWriters` 极高、3 页报错 | 减桶数 / dynamic-bucket / 分批回填 / 加 `write-buffer-size` |
| **compaction reader 过多** | Level 0 积压，`sort-spill-threshold` 过大或无 IOManager，reader 同开（§6） | compaction 阶段 OOM | 让 `sort-spill-threshold` 跟默认、保证 compaction 不积压、加 `sort-spill-buffer-size` |
| **lookup 热 key** | partial-update/aggregation 某 key 版本数极多，Spark batch 无 IOManager 走 `UnlimitedSegmentPool`（§7） | lookup merge 阶段 OOM | 加 `lookup.merge-buffer-size`、用 Flink streaming（有 IOManager）、及时 compaction 控版本数 |

注意一个**反直觉点**：关闭 `write-buffer-spillable` 通常**不会 OOM**——`MergeTreeWriter.flushWriteBuffer()` 兜底写出数据文件（§5.2），代价是小文件雪崩而非崩溃。

**④ 调优参数全景**（默认值已逐条核对）

| 参数 | 默认值 | 源码 | 调优建议 |
|------|--------|------|----------|
| `write-buffer-size` | 256 MB | `CoreOptions.java:657` | Writer 多时增大到 512MB–1GB |
| `write-buffer-spillable` | true | `:672` | 保持开启 |
| `write-buffer-spill.max-disk-size` | 无限 | `:664` | 磁盘有限时设上限（注意上限满会退化小文件） |
| `page-size` | 64 KB | `:701` | 一般无需调 |
| `local-sort.max-num-file-handles` | 128 | `:692` | 溢写文件多时增大 |
| `sort-spill-threshold` | `numSortedRunStopTrigger+1` | `:606`（无默认值） | 一般跟随默认；reader 多时谨慎调 |
| `sort-spill-buffer-size` | 64 MB | `:614` | 溢写时增大减少 I/O 次数 |
| `spill-compression` | zstd | `:620` | CPU 紧张改 lz4 |
| `target-file-size` | 128MB/256MB | `:713` | 按查询模式与表类型调 |
| `sink.use-managed-memory-allocator` | false | `FlinkConnectorOptions.java:175` | 多表写入 + State 占堆大时开 |
| `write-max-writers-to-spill` | 10 | `:685` | 批量 append 插入时 Writer 超此值开缓冲溢写 |
| `lookup.merge-buffer-size` | 8 MB | `:2255` | 热 key 场景增大 |
| `lookup.merge-records-threshold` | 1024 | `:2261` | 控制升级 BinaryBuffer 的阈值 |

**⑤ 排查流程清单（文字版）**

```
1. 看 metric：numWriters 是否异常高？bufferPreemptCount 是否快速增长？
   ├─ numWriters 极高           → Writer 爆炸：减桶数 / dynamic-bucket
   └─ bufferPreemptCount 飙升   → 内存竞争：加 write-buffer-size / 减并发
2. 看 OOM 发生阶段：写入 / compaction / lookup merge？
   ├─ 写入阶段   → write buffer 不足或关了 spillable
   ├─ compaction → sort-spill-threshold / sort-spill-buffer-size
   └─ lookup     → lookup.merge-buffer-size / 确保 IOManager
3. 加内存无效？→ 怀疑根因不是"不足"而是"被稀释"（Writer 太多）或 GC 碎片
   └─ GC 频繁但回收少 → 考虑 sink.use-managed-memory-allocator（off-heap 免 GC）
4. 仅在前述都排除后，再 dump heap 看具体对象分布
```

> 注：旧稿提到的某次 commit 修复的 `ClusteringFileRewriter` key bytes OOM 属版本特定问题，已随版本修复，此处不展开为通用场景。

## 11. 设计决策总结

### 11.1 与 Flink RocksDB State Backend 的对比

Paimon 不复用 RocksDB 的原因：RocksDB 为随机读写设计（SkipList 插入即排序、Block Cache + Write Buffer + Index 多池、依赖重），而 Paimon 写入是单线程顺序、批量、只需排序缓冲 + 溢写——自研更轻、更可控。核心差异：

| 维度 | Paimon 内存管理 | Flink RocksDB State Backend |
|------|----------------|----------------------------|
| 用途 | 数据持久化（顺序写为主） | 算子内部状态（随机读写） |
| 内存模型 | 页式 `MemorySegment`（默认 64KB 页） | Block Cache + Write Buffer + Index |
| 排序 | **延迟排序**（flush 时 QuickSort/Introsort） | **插入即排序**（SkipList） |
| 内存回收 | **主动抢占**最大 Owner 并 flush | **被动 LRU** 淘汰 Block Cache |
| 溢写 | buffer spill 到临时文件（排序后） | memtable flush 到 SST |
| GC 影响 | Heap 模式有、Managed 模式无 | off-heap，无 |

设计哲学差异：Paimon 是"**协作式**"——`MemoryOwner` 主动报告占用、配合让出，内存使用透明、抢占策略可定制（代价是抢占抖动）；RocksDB 是"**委托式**"——交给 Block Cache 自动淘汰，省心但不透明。两者都用 Managed Memory 时按算子权重分配，建议要么都用 Managed、要么 State 用 Managed 而 Paimon 用 Heap，避免配置割裂（§8 陷阱）。

### 11.2 设计决策总表

| 决策点 | 选择 | 取舍 / 代价 | 收益 |
|--------|------|------------|------|
| 内存分配模型 | 共享池 + 抢占，不预分配配额（§3.1） | 需抢占兜底，可能"大 Writer 被反复抢" | 利用率高、自动适应负载漂移 |
| 抢占目标 | 抢**占用最大**且**非自己**的 Owner（§3.2） | 对持续高负载大 Writer 略不公平 | 单次释放最多、抢占次数最少、不抢自己防状态错乱 |
| `MemoryOwner` 契约 | 极薄三方法（报告快 / 让出容 I/O）（§2） | 抢占同步、可能阻塞写入线程 | 任何组件都易接入统一调度 |
| 排序时机 | **延迟排序** + 二进制页式 + normalized key（§4.1） | 不支持随机点查 | 写近 O(1)、内存紧凑、比较省反序列化 |
| 溢写分层 | 两层：buffer spill → flushWriteBuffer 兜底（§5） | 兜底会产小文件 | 超量写入永不 OOM |
| spill 前是否排序 | 写路径**排序**后落盘，读路径（MergeSorter）**不排序**（§5.1/§6） | 写路径多一次排序 | 归并只需 N-way merge，省 CPU |
| 读路径内存池 | `CachelessSegmentPool`（用完即弃）（§6） | 频繁申请增 GC | 流式读零碎片、独立于写池 |
| lookup 缓冲 | List→Binary→External 三层自适应（§7） | 实现复杂、升级有开销；Spark batch 无溢写 | 普通 key 轻量、热 key 不 OOM |
| Flink 内存来源 | Heap 默认 + Managed 可选（§8） | Managed 靠反射、有版本风险 | 兼容单表简单与多表 off-heap |
| 文件大小控制 | 目标大小分档 + 千分之一采样检查（§9） | 实际略超目标 | 测量成本可忽略、文件大小受控 |

### 11.3 核心类关系总览

```
MemorySegmentPool (接口)
├── AbstractMemorySegmentPool ── HeapMemorySegmentPool / FlinkMemorySegmentPool（缓存复用）
├── CachelessSegmentPool（用完即弃，MergeSorter 用）
├── UnlimitedSegmentPool（无上限，lookup 无 IOManager 兜底）
└── OwnerMemoryPool（MemoryPoolFactory 内部代理，记 allocatedPages，物理内存取自 innerPool）

MemoryPoolFactory ──持有── innerPool + Iterable<MemoryOwner>
    └── preemptMemory()：抢最大非自己的 Owner → flushMemory()

MemoryOwner (接口：setMemoryPool / memoryOccupancy / flushMemory)
    └── MergeTreeWriter（唯一核心实现，同时是 RecordWriter）
            └── WriteBuffer (接口) ── SortBufferWriteBuffer
                    └── SortBuffer (接口)
                            ├── BinaryInMemorySortBuffer（纯内存，flushMemory 恒 false）
                            └── BinaryExternalSortBuffer（可溢写，spill 前 QuickSort）

KeyValueBuffer (lookup 合并)：HybridBuffer → ListBuffer / BinaryBuffer(InMemoryBuffer | ExternalBuffer)
```
