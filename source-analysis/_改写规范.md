# 源码分析文档改写规范 (v1)

> 这份文件是所有参与改写 `source-analysis/04~24` 的 agent 的**统一作业基准**。开工前必读，读完按 §6 SOP 执行。
> 本文件是过程文件，任务结束后会删除，**不要在交付文档里引用它**。

---

## 1. 总目标（最重要，反复对照）

把文档从"**源码堆砌**"改造成"**读完成为该模块源码专家**"的实用深度文档。

铁律优先级：**实用 > 讲透原理 > 覆盖广度 > 源码量**。

- 核心是讲清三件事：**① 为什么这么设计（取舍）② 风险/陷阱是什么 ③ 收益与代价是什么**。
- **不要通篇粘贴源码**。源码只作为"论据"出现，精选最能说明问题的核心片段，其余用文字 + `路径:行号` 指引读者自己去看。
- 用户的判据原话："不要堆砌过多的源码，从底层原理来讲解为什么这么做、风险是什么、收益是什么。可以放源码，但是不要通篇地放。"

## 2. 唯一标杆

`01-核心存储引擎分析.md` 是唯一风格标杆。**改写前必须先 `Read` 01 的第 1–160 行 + 目录区**，对齐它的开篇模板、五段式、速查表、对比表、生命周期串讲的写法。你的产出风格要和 01 一致。

## 3. 强制文档骨架（顺序固定，不可调换）

```

> **版本**：1.5-SNAPSHOT　**源码模块**：`<主模块>`　**核对日期**：2026-06

**一句话定位**：<一句话说清这个机制在 Paimon 里解决什么、价值是什么>

读完本文你应能回答：① … ② … ③ …（列 5–8 个真正有深度的问题，作为学习目标）

> 阅读约定：本文每个机制按"① 要解决什么问题 → ② 设计原理与取舍 → ③ 关键源码（精选片段 + 路径:行号）→ ④ 风险/陷阱/边界 → ⑤ 收益与代价"组织。源码行号以本次核对为准；与旧稿不符处用"（已修正）"标注。

## 1. 快速理解（核心问题 / 概念速查 / 高频陷阱）
### 1.1 核心问题：<这个模块到底在解决什么>
### 1.2 核心概念速查表    <表格：概念 | 一句话定义 | 关键源码(路径:行号)>
### 1.3 高频生产陷阱      <表格或列表：陷阱 | 后果 | 规避>

## 2 ~ N. <各机制章节，每节走五段式>

## 末章. 设计决策总结      <表格：决策点 | 选择 | 取舍/代价 | 收益>
```

要点：
- **目录必须在正文最前面**（用户明确要求"目录然后才是具体内容"）。当前旧文档常把概念解释堆在目录前面，必须纠正。
- 开篇模板（一句话定位 + 能回答的问题 + 阅读约定）照搬 01 的风格，不能省。
- 难度/版本等元信息放引用块（`>`）里，简洁。

## 4. 每个机制章节的五段式（核心）

每个二级/三级机制章节，尽量按这个逻辑流推进（不必生硬地写五个小标题，但内容要覆盖到）：

1. **① 要解决什么问题**：不先讲实现，先讲动机和约束（对象存储特性、流式语义、规模压力等）。
2. **② 设计原理与取舍**：为什么选这个方案而不是别的？用**对比表**讲清替代方案的优劣。给出"一句话设计哲学"。
3. **③ 关键源码**：精选片段（一般单段 ≤ 30 行，只留最关键的逻辑），必带 `文件名:行号`。能用一行文字概括的逻辑就不要贴代码。
4. **④ 风险 / 陷阱 / 边界**：配错会怎样、并发/失败/规模边界、与默认值的关系、生产事故场景。
5. **⑤ 收益与代价**：这个设计带来什么、代价是什么、什么场景适用/不适用。

## 5. 写作铁律

- **忠于源码**：所有类名、方法名、配置项、行号必须用 `Grep`/`Read` 打开**真实源码核对**，严禁凭旧稿或记忆编造。拿不准就去读，读不到就不写。
- **行号核对**：旧文档行号大概率已过时。凡保留的源码引用都要重新核对行号；与旧稿不一致的，在该处标注"（已修正）"。
- **精选而非堆砌**：删掉冗长的、可被一句话概括的源码段。一篇文档源码占比建议 < 35%。
- **多用结构化表达**：对比表、决策表、流程清单（文字版箭头流程）、speca 比大段代码更有信息密度。
- **可视化流程**：关键链路用"步骤 1 → 2 → 3"文字流程或 ASCII 图，参考 01 的"记录全生命周期"。
- **重叠处理**：见 §8，重叠内容用交叉引用，不重复展开。
- **保留有价值的存量**：旧文档里准确、有深度的内容要保留并融入新骨架，不是推倒重写；只删冗余、补缺失、改结构、纠错误。

## 6. Agent 标准作业流程 (SOP)

1. `Read` 本规范 + `01` 的开篇与目录（对齐风格）。
2. **分段** `Read` 你负责的那篇文档**全文**（每次 ≤ 1500 行，读完为止——用户强调"完整阅读，不要大致阅读"）。边读边记：哪些是冗余源码、哪些结构混乱、哪些事实可能过时、缺哪些"为什么/风险/收益"。
3. 用 `Grep` 在对应源码模块定位关键类，**核对行号与签名**（§7 给了模块地图）。
4. 规划新目录结构（先在脑中或草稿里定好骨架）。
5. **用 `Edit` 分段改写**（严禁 `Write` 覆盖整文件！见 §7）。通常的做法：
   - 先 `Edit` 替换文件头部（标题→开篇模板→目录），把目录提到最前。
   - 再逐章 `Edit`：把旧章节内容替换为五段式新内容。
   - 删冗余源码段用 `Edit` 把大段代码替换为"精炼代码 + 文字解释 + 路径:行号"。
6. 收尾走 §9 自检清单。
7. 最终回复给主控：**只报告"改了什么、核对了哪些源码、行号修正了几处、还存疑的点"**，**不要把文档全文贴回来**。

## 7. 防请求超限铁律（必须遵守——上次因此报错 "Request too large 32MB"）

- **禁止 `Write` 覆盖已存在的大文档**。一律用 `Edit` 做局部替换。`Write` 4000 行文件 = 单请求塞十几万字符，会触发 32MB 上限。
- 单次 `Read` 文档 **≤ 1500 行**，大文件分多次读。
- 单次 `Edit` 的 `new_string` 控制在**一个章节**的体量（建议 < 600 行 / < 数万字符），不要一次替换上千行。
- 不要在你给主控的最终回复里**粘贴大段文档原文或大段源码**——只给结论和要点。
- 读源码时按需读片段（带 offset/limit），不要把整个大源码文件读进来。
