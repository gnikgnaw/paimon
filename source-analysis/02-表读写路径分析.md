## 1. Table 接口完整层次

### 1.1 继承层次总览

### 1.2 各层职责精确定义

**Table (L52, Table.java)** -- 公共 API 层

源码路径: `paimon-core/src/main/java/org/apache/paimon/table/Table.java`

Table 是面向用户的稳定接口，标注 `@Public`（自 0.4.0 起）。它定义了三组能力:

| 能力组 | 方法 | 说明 |
|--------|------|------|
| 表元数据 | `name()`, `fullName()`, `uuid()`, `rowType()`, `partitionKeys()`, `primaryKeys()`, `options()`, `comment()`, `statistics()` | 访问表的结构信息与统计 |
| 表管理 | `rollbackTo()`, `createTag()`, `deleteTag()`, `createBranch()`, `deleteBranch()`, `fastForward()`, `newExpireSnapshots()`, `newExpireChangelog()` | 表生命周期管理操作 |
| 读写入口 | `newReadBuilder()`, `newBatchWriteBuilder()`, `newStreamWriteBuilder()`, `newVectorSearchBuilder()`, `newFullTextSearchBuilder()` | 创建读写 Builder |

**为什么这样设计**: 将面向用户的 API 与内部实现完全分离。用户只依赖 Table 接口，底层实现可自由演进而不破坏兼容性。Builder 模式让用户通过链式调用配置参数，隐藏了 Scan/Read/Write 的创建细节。

**InnerTable (L41, InnerTable.java)** -- 内部实现入口层

源码路径: `paimon-core/src/main/java/org/apache/paimon/table/InnerTable.java`

InnerTable 是 Table 之下的第一层扩展，不对外暴露，只供引擎集成使用。它新增了直接创建底层组件的方法:

```
newScan()        -> InnerTableScan       (计划层)
newStreamScan()  -> StreamDataTableScan  (流式计划层)
newRead()        -> InnerTableRead       (执行层)
newWrite(user)   -> InnerTableWrite      (写入层)
newCommit(user)  -> InnerTableCommit     (提交层)
newWriteSelector() -> Optional<WriteSelector>  (写分发层)
```

同时提供了所有 Builder 的默认实现:
- `newReadBuilder()` 默认返回 `ReadBuilderImpl(this)`
- `newBatchWriteBuilder()` 默认返回 `BatchWriteBuilderImpl(this)`
- `newStreamWriteBuilder()` 默认返回 `StreamWriteBuilderImpl(this)`
- `newVectorSearchBuilder()` 默认返回 `VectorSearchBuilderImpl(this)`
- `newFullTextSearchBuilder()` 默认返回 `FullTextSearchBuilderImpl(this)`

**为什么需要这一层**: 系统表 (System Table) 也实现 InnerTable，但不需要 DataTable 的管理能力（快照、标签、分支）。InnerTable 作为最小的内部实现接口，让系统表和数据表共享 Builder 创建逻辑。

**DataTable (L33, DataTable.java)** -- 数据表管理层

源码路径: `paimon-core/src/main/java/org/apache/paimon/table/DataTable.java`

DataTable 在 InnerTable 基础上增加了"数据表"独有的管理能力:

```
newSnapshotReader()  -> SnapshotReader      (快照感知的读取协调器)
coreOptions()        -> CoreOptions         (核心配置访问)
snapshotManager()    -> SnapshotManager     (快照生命周期)
changelogManager()   -> ChangelogManager    (Changelog 生命周期)
consumerManager()    -> ConsumerManager     (消费者进度管理)
schemaManager()      -> SchemaManager       (Schema 演进)
tagManager()         -> TagManager          (标签管理)
branchManager()      -> BranchManager       (分支管理)
switchToBranch()     -> DataTable           (分支切换)
location()           -> Path                (表的存储路径)
```

关键细化: `newScan()` 的返回类型从 `InnerTableScan` 收窄为 `DataTableScan`，后者增加了 `withShard()` 能力（支持并行分片扫描）。

**为什么需要这一层**: 将数据表的管理能力（快照、标签、分支等）与基础读写抽象分离。系统表 (如 `$snapshots`, `$files`) 只需实现 InnerTable 提供读取能力，不需要自己管理快照。同时 DataTable 定义了 `SnapshotReader` -- 这是读取路径最核心的协调器，从这一层开始才有完整的快照感知读取能力。

**FileStoreTable (L56, FileStoreTable.java)** -- FileStore 桥接层

源码路径: `paimon-core/src/main/java/org/apache/paimon/table/FileStoreTable.java`

FileStoreTable 是连接"表"抽象与"存储引擎"的桥梁:

```
store()                -> FileStore<?>        核心: 获取底层存储引擎
schema()               -> TableSchema         表 Schema (包含完整的字段/分区/主键/bucket信息)
bucketMode()           -> BucketMode          桶模式
bucketSpec()           -> BucketSpec          桶规格 (mode + keys + num)
catalogEnvironment()   -> CatalogEnvironment  Catalog 环境信息

setManifestCache()     缓存: Manifest 文件缓存
setSnapshotCache()     缓存: Snapshot 对象缓存
setStatsCache()        缓存: Statistics 缓存
setDVMetaCache()       缓存: DV 元数据缓存

newWrite()             -> TableWriteImpl<?>   精确的写入实现类型
newCommit()            -> TableCommitImpl     精确的提交实现类型
newLocalTableQuery()   -> LocalTableQuery     本地点查接口
createRowKeyExtractor()-> RowKeyExtractor     行路由提取器

purgeFiles()           清除表的所有文件(分支/标签/消费者/快照)
```

**为什么需要这一层**: FileStore 是 Paimon 存储引擎的核心抽象（有 KeyValueFileStore 和 AppendOnlyFileStore 两种实现）。FileStoreTable 作为桥梁层，将 FileStore 的底层能力封装为表级操作。缓存设置方法 (`setManifestCache` 等) 体现了 `CachingCatalog` 模式 -- 由 Catalog 统一管理缓存，再注入到表中。

### 1.3 AbstractFileStoreTable 模板方法模式

源码路径: `paimon-core/src/main/java/org/apache/paimon/table/AbstractFileStoreTable.java` (L90)

AbstractFileStoreTable 是 FileStoreTable 的骨架实现，使用了经典的**模板方法模式**:

**固定实现** (所有子类共享):

```
newSnapshotReader()   创建 SnapshotReaderImpl，注入 scan/splitGenerator/pathFactory 等
newScan()             创建 DataTableBatchScan，可包装为 DataEvolutionBatchScan
newStreamScan()       创建 DataTableStreamScan
newCommit()           创建 TableCommitImpl，注入 expire/partitionExpire/tagAutoManager
copy(TableSchema)     根据 primaryKeys 是否为空决定创建哪个子类 (L396-L402)
createRowKeyExtractor() 根据 bucketMode 选择 RowKeyExtractor 实现 (L257-L270)
newWriteSelector()    根据 bucketMode 选择 WriteSelector 实现 (L229-L239)
```

**模板方法** (由子类实现):

```
abstract SplitGenerator splitGenerator();
    // 主键表: MergeTreeSplitGenerator  -- 基于 key 范围的 IntervalPartition
    // 追加表: AppendOnlySplitGenerator -- 基于文件大小的简单分组

abstract BiConsumer<FileStoreScan, Predicate> nonPartitionFilterConsumer();
    // 主键表: 只提取 key 列过滤条件 (L123-L144, PrimaryKeyFileStoreTable)
    // 追加表: 将完整谓词传给 scan (L107-L108, AppendOnlyFileStoreTable)
```

**关键的 copy() 决策逻辑** (L396-L402):

```
FileStoreTable copy(TableSchema newTableSchema) {
    return newTableSchema.primaryKeys().isEmpty()
        ? new AppendOnlyFileStoreTable(fileIO, path, newTableSchema, catalogEnvironment)
        : new PrimaryKeyFileStoreTable(fileIO, path, newTableSchema, catalogEnvironment);
}
```

**为什么在 copy 时重新判断**: 这确保了通过 `table.copy(dynamicOptions)` 或 `table.copyWithLatestSchema()` 获取的新表实例，总是与 Schema 的主键定义保持一致。如果 Schema 演进移除了主键，表类型会从 PrimaryKey 自动切换为 AppendOnly。

### 1.4 主键表与追加表的继承差异

**PrimaryKeyFileStoreTable** (L53, PrimaryKeyFileStoreTable.java)

```
store() -> KeyValueFileStore
    - 创建时从 schema 中提取 keyType、valueType
    - 创建 MergeFunctionFactory (根据 merge-engine 配置)
    - 如果 options.needLookup()=true，包装为 LookupMergeFunction (L84-L85)

splitGenerator() -> MergeTreeSplitGenerator
    - 需要 keyComparator 进行 key 范围分析
    - deletionVectorsEnabled 影响 rawConvertible 判定
    - mergeEngine 影响是否可跳过 merge (FIRST_ROW 引擎可以)

nonPartitionFilterConsumer():
    - 只提取 key 列过滤条件传给 KeyValueFileStoreScan.withKeyFilter()
    - 同时传全量过滤给 withValueFilter() 用于 bucket 级过滤
    - 原因: 值字段过滤可能导致数据不一致 (见 L127-L134 注释)

newRead() -> KeyValueTableRead
    - 4 个 SplitReadProvider: Raw/Merge/IncrementalChangelog/IncrementalDiff

newWrite() -> TableWriteImpl<KeyValue>
    - 将 InternalRow 转换为 KeyValue 结构 (L161-L172)
    - kv.replace(primaryKey, UNKNOWN_SEQUENCE, rowKind, row)
```

**AppendOnlyFileStoreTable** (L53, AppendOnlyFileStoreTable.java)

```
store() -> AppendOnlyFileStore
    - 创建时使用 rowType.notNull() 作为文件格式类型

splitGenerator() -> AppendOnlySplitGenerator
    - 无需 key 比较器
    - BUCKET_UNAWARE 模式下流式也按 batch 方式切分

nonPartitionFilterConsumer():
    - 将完整谓词直接传给 AppendOnlyFileStoreScan.withFilter()
    - 追加表无 key 重叠问题，可安全过滤任意列

newRead() -> AppendTableRead
    - 2 个 SplitReadProvider: RawFile / DataEvolution

newWrite() -> TableWriteImpl<InternalRow>
    - 直接写入 InternalRow，校验只接受 RowKind.isAdd() (L143-L146)
    - 如果不是 INSERT/INSERT_ONLY，抛出异常
```

### 1.5 与 Iceberg 的对比

| 维度 | Paimon | Iceberg |
|------|--------|---------|
| 表抽象层数 | 5 层继承 (Table -> ... -> PrimaryKeyFileStoreTable) | 2 层 (Table 接口 + BaseTable 实现) |
| 表类型区分 | 编译期: 两个子类 + 模板方法 | 运行时: format-version 控制 |
| 存储引擎 | FileStore 作为独立组件，有 KV 和 AppendOnly 两种 | TableOperations 统一管理 |
| 谓词下推 | 主键表只下推 key 列 (LSM 语义约束) | 可下推任意列 (无 merge 语义) |
| 缓存注入 | 通过 set 方法由 CachingCatalog 注入 | 通过构造参数传入 |
| 设计理念 | 面向流式更新的分层设计 | 面向分析的统一接口 |

**为什么 Paimon 需要更多层**: Paimon 同时支持主键表 (LSM-tree 语义) 和追加表 (Append-only 语义)，两者在谓词下推、Split 生成、数据写入上差异显著。编译期通过子类区分可避免运行时条件判断，且各子类职责清晰。同时 Paimon 在表级别集成了快照/标签/分支管理能力，需要单独的 DataTable 层来承载。

### 2.2 ReadBuilder 配置阶段

源码路径: `paimon-core/src/main/java/org/apache/paimon/table/source/ReadBuilderImpl.java`

ReadBuilderImpl 是 ReadBuilder 接口的唯一实现，在 `InnerTable.newReadBuilder()` 中被创建 (InnerTable.java L56-L58)。

ReadBuilder 采用 Builder 模式收集所有读取参数:

```
ReadBuilderImpl {
    InnerTable table;              // 关联的表
    RowType partitionType;         // 分区字段类型 (从 table.rowType() 投影)
    String defaultPartitionName;   // 默认分区名 (core option)

    Predicate filter;              // 过滤谓词 (可多次叠加, AND 合并)
    Integer limit;                 // 行数限制
    TopN topN;                     // TopN 优化
    Integer shardIndexOfThisSubtask;    // 分片索引
    Integer shardNumberOfParallelSubtasks;  // 并行度
    PartitionPredicate partitionFilter;  // 分区过滤器
    Integer specifiedBucket;       // 指定桶号
    Filter<Integer> bucketFilter;  // 桶过滤器
    RowType readType;              // 读取类型 (列裁剪)
    RowRangeIndex rowRangeIndex;   // 行范围索引
    boolean dropStats;             // 是否丢弃统计信息
}
```

**newScan() 的配置传递** (L181-L229):

```
ReadBuilderImpl.newScan()
    |
    v
configureScan(table.newScan())
    |
    +-- scan.withFilter(filter)               // 谓词传递
    +-- scan.withReadType(readType)           // 列裁剪传递
    +-- scan.withPartitionFilter(partFilter)  // 分区过滤
    +-- scan.withRowRangeIndex(rowRangeIndex) // 行范围
    +-- scan.withLimit(limit)                 // limit pushdown
    +-- scan.withTopN(topN)                   // topN pushdown
    |
    +-- [互斥检查] bucketFilter 和 shard 不能同时使用
    +-- scan.withShard(index, parallelism)    // 或
    +-- scan.withBucketFilter(bucketFilter)   // 二选一
```

**newRead() 的配置传递** (L232-L244):

```
ReadBuilderImpl.newRead()
    |
    v
table.newRead()
    .withFilter(filter)     // 谓词传递给 Read (行级过滤)
    .withReadType(readType) // 列裁剪传递
    .withTopN(topN)         // TopN 传递
    .withLimit(limit)       // Limit 传递
```

**为什么 filter 同时传给 Scan 和 Read**: Scan 层用 filter 做文件级过滤 (跳过不匹配的文件)；Read 层用 filter 做行级过滤 (跳过不匹配的行)。两层过滤互补，Scan 减少读取文件数量，Read 减少返回行数。

### 2.3 TableScan 计划阶段

源码路径: `paimon-core/src/main/java/org/apache/paimon/table/source/DataTableBatchScan.java`

**调用链**:

```
DataTableBatchScan.plan()                          (L99)
  |
  v
AbstractDataTableScan.plan()                       (L103)
  |
  +-- authQuery()                                   // 权限检查
  |
  v
DataTableBatchScan.planWithoutAuth()               (L99-L118)
  |
  +-- createStartingScanner(false)                  // 创建 StartingScanner
  |

  |     +-- 累加 Split 的 mergedRowCount             // 直到 >= limit
  |     +-- 截断多余的 Split                          // 减少读取量
  |
  +-- [尝试] applyPushDownTopN()                    // TopN 下推优化
  |     |
  |     +-- 条件: 非主键表 + 非DV + 单排序列 + limit<=100 + minmax可用
  |     +-- TopNDataSplitEvaluator.evaluate()       // 基于 min/max 过滤
  |
  +-- [默认] startingScanner.scan(snapshotReader)   // 标准扫描
  |
  v
DataFilePlan.fromResult(result)                     // 包装返回
```

**Limit 下推优化** (L128-L165):

当设置了 `withLimit(n)` 时，Paimon 不需要扫描所有 Split。它累加每个 Split 的 `mergedRowCount()`，当总行数达到 limit 时截断剩余 Split。

**为什么这能工作**: `mergedRowCount()` 对于 rawConvertible 的 Split 可以精确计算（文件行数 - DV 删除数）。对于需要 merge 的 Split 则返回 `OptionalLong.empty()` 表示无法估算，此时不做截断。

**TopN 下推优化** (L167-L205):

当查询是 `ORDER BY col LIMIT n` 且 n <= 100 时，利用文件的 min/max 统计信息过滤掉不可能包含 Top N 结果的 Split。

**为什么限制 n <= 100**: TopN 优化需要加载所有 Split 的统计信息 (`keepStats()`)，代价与 Split 数量成正比。n 太大时优化收益递减。

### 2.9 rawConvertible 优化

**定义**: 当一个 DataSplit 标记为 `rawConvertible=true` 时，意味着其中的文件可以直接逐文件读取，跳过 Paimon 的 merge 逻辑。

**启用条件**:

| 表类型 | 条件 | rawConvertible |
|--------|------|----------------|
| 追加表 | 始终 | true |
| 主键表 (DV) | `deletion-vectors.enabled=true` | true |
| 主键表 (FIRST_ROW) | `merge-engine=first-row` 且 level > 0 | true |
| 主键表 (普通) | 所有文件 level > 0 + 无 deleteRow + 同一 level | true |
| 主键表 (普通) | 有 level=0 文件或多 level 文件 key 重叠 | false |
| 主键表 (普通) | 单文件且无 deleteRow (即使跨 level) | true |

**为什么能跳过 merge**:

- DV 模式: 每个数据文件有对应的 Deletion Vector (Roaring Bitmap) 标记哪些行被删除。读取时只需跳过标记位，不需要和其他文件做归并。
- FIRST_ROW 引擎: 只取每个 key 的第一条记录，level > 0 的文件已经 compacted，保证每个 key 只有一条。
- 单 level + 无重叠: LSM-tree 的同 level 文件 key 不重叠，可独立读取。

**性能提升有多大**:

rawConvertible=true 时:
1. 跳过 LoserTree/MinHeap 排序归并 -- 避免 O(N log K) 比较 (K=文件数)
2. 跳过 MergeFunction 调用 -- 避免 CPU 密集的合并计算
3. 可利用列式格式的向量化读取 -- Parquet/ORC 的 batch 读取直接返回
4. 可利用引擎原生优化 -- Spark/Flink 可直接使用自己的 Parquet Reader

在 DV 模式下，主键表的批查询性能可接近 Iceberg 等 MoR (Merge-on-Read) 引擎的水平。

```
| 使用: MergeFileSplitRead (LoserTree 归并 + MergeFunction)         |
| 特点: withReadKeyType(RowType.of()) 不读 key 列，只读 value       |
+------------------------------------------------------------------+
                        |  不匹配则继续
                        v
+------------------------------------------------------------------+
| 优先级 3: IncrementalChangelogReadProvider                        |
|                                                                    |
| 匹配条件:                                                         |
|   split instanceof IncrementalSplit                               |
|   && incrementalSplit.isStreaming()  // 流式增量                   |
|                                                                    |
| 使用: ConcatRecordReader(ReverseReader(before), after)            |
| 原理: 将 before 文件反转 (行标记为 DELETE) 再与 after 文件拼接    |
+------------------------------------------------------------------+
                        |  不匹配则继续
                        v
+------------------------------------------------------------------+
| 优先级 4: IncrementalDiffReadProvider                             |
|                                                                    |
| 匹配条件:                                                         |
|   split instanceof IncrementalSplit                               |
|   && !incrementalSplit.isStreaming()  // 批式差异                  |
|                                                                    |
| 使用: IncrementalDiffSplitRead (diff 读取)                        |
+------------------------------------------------------------------+
```

**为什么优先级 1 在最前面**: RawFile 读取性能最高（直接利用列式格式），应优先使用。只有不满足 rawConvertible 条件时才降级到 merge 读取。

**forceKeepDelete 的作用**: 当 `forceKeepDelete=true` 时（如审计日志场景），即使 rawConvertible=true 也必须走 merge 路径，因为 merge 会生成完整的 +I/-D/-U/+U 标记，而 raw 读取只返回 +I。

### 2.13 谓词下推的保守策略

```
+====================================================================+
|                    谓词下推三层过滤                                   |
+====================================================================+
|                                                                      |
|  Layer 1: 分区过滤 (Manifest 级别)                                   |
|     SnapshotReader.withFilter() -> splitPartitionPredicates()        |
|     -> scan.withPartitionFilter(partitionPredicate)                  |
|     -> 在 Manifest 扫描时跳过不匹配的分区 ManifestEntry              |
|     效果: 跳过整个分区目录                                           |
|                                                                      |
|  Layer 2: 文件级过滤 (Stats 级别)                                    |
|     根据文件的 min/max/null 统计信息过滤整个文件                      |
|                                                                      |
|     主键表 (PrimaryKeyFileStoreTable.java L123-L144):               |
|       +-- 只提取 KEY 列谓词 -> KeyValueFileStoreScan.withKeyFilter() |
|       +-- 完整谓词 -> withValueFilter() (仅 bucket 级过滤)          |
|       原因: 值字段过滤可能导致数据不一致                              |
|                                                                      |
|     追加表 (AppendOnlyFileStoreTable.java L107-L108):               |
|       +-- 完整谓词 -> AppendOnlyFileStoreScan.withFilter()           |
|       原因: 追加表无 key 重叠，安全过滤任意列                        |
|                                                                      |
|  Layer 3: 行级过滤                                                   |
|     在 RecordReader 内部应用过滤谓词                                 |
|     文件索引 (bloom/bitmap/bsi): 在读文件前做进一步过滤              |
|                                                                      |
+====================================================================+
```

**主键表只对 key 列下推的根本原因** (PrimaryKeyFileStoreTable.java L127-L134):

```
考虑这个场景:
  data file 1: insert key=a, value=1    (level=1)
  data file 2: update key=a, value=2    (level=0)
  filter: value = 1

如果对 value 做文件级 filter push down:
  -> file 1 的 stats 显示 value min=1, max=1 => 匹配 => 选中
  -> file 2 的 stats 显示 value min=2, max=2 => 不匹配 => 跳过

最终结果: key=a, value=1
正确结果: 空集 (key=a 的最终值是 2，不满足 value=1)

原因: LSM-tree 中同一个 key 的不同版本分布在不同文件中。
      跳过任何一个版本都可能导致 merge 结果不正确。
      只有 KEY 列过滤是安全的——如果 key 范围不匹配，所有版本都不匹配。
```

**withValueFilter() 的 bucket 级过滤** (L143):

虽然不能在文件级过滤 value 列，但可以在 bucket 级别过滤: 如果一个 bucket 内所有文件的 value 列统计信息都不满足谓词，可以安全跳过整个 bucket。因为一个 bucket 内的所有 key 版本都在这个 bucket 中。

### 3.3 TableWriteImpl 核心逻辑

**writeAndReturn() 的 6 个步骤** (L183-L193):

```java
public SinkRecord writeAndReturn(InternalRow row, int bucket) throws Exception {
    // Step 1: 非空字段检查
    checkNullability(row);
    // 遍历 notNullFieldIndex，如果 row.isNullAt(idx) 则抛出异常

    // Step 2: 填充默认值
    row = wrapDefaultValue(row);
    // 如果 schema 定义了 default value，用 DefaultValueRow 包装

    // Step 3: 确定行类型 (RowKind)
    RowKind rowKind = RowKindGenerator.getRowKind(rowKindGenerator, row);
    // 如果配置了 rowkind-field，从指定字段读取 (+I/-D/-U/+U)
    // 否则使用 row.getRowKind() (来自 CDC 消息)

    // Step 4: RowKind 过滤
    if (rowKindFilter != null && !rowKindFilter.test(rowKind)) {
        return null;  // 被过滤的行直接丢弃
    }

    // Step 5: 提取 partition/bucket/primaryKey
    SinkRecord record = bucket == -1 ? toSinkRecord(row) : toSinkRecord(row, bucket);
    // toSinkRecord:
    //   keyAndBucketExtractor.setRecord(row)
    //   new SinkRecord(partition(), bucket(), trimmedPrimaryKey(), row)

    // Step 6: 转换为存储格式并写入
    write.write(record.partition(), record.bucket(), recordExtractor.extract(record, rowKind));
    // 主键表: recordExtractor 将 SinkRecord 转为 KeyValue
    //         kv.replace(primaryKey, UNKNOWN_SEQUENCE, rowKind, row)
    // 追加表: recordExtractor 直接返回 row (校验 rowKind.isAdd())

    return record;
}
```

**RowKindFilter 的作用**:

```
配置 'read.changelog-mode' 控制:
  - all: 不过滤
  - upsert: 过滤掉 UPDATE_BEFORE
  - 自定义: 保留指定的 RowKind 组合

```

```
    |
```

```
    |     if tagAutoManager != null:
    |       tagAutoCreation.run()
    |       // 根据时间/commit数自动创建 Tag (如 daily/hourly)

    |
    +-- Task 5: Snapshot 过期 (放在最后，耗时最长)
          if doExpire && expireSnapshots != null:
            expireSnapshots.run()
            // 根据 snapshot.num-retained-max/min 和 snapshot.time-retained 清理快照
            // 如果 changelog-lifecycle-decoupled=true，还独立清理 changelog
```

**执行模式** (L360-L378):

```
maintainExecutor 的选择:
  - SYNC 模式:  使用 DirectExecutorService (在当前线程同步执行)
  - ASYNC 模式: 使用单线程 Executor (后台异步执行)
```

**为什么 Task 4 (Consumer 过期) 在 Task 5 (Snapshot 过期) 之前**: Consumer 记录了某个消费者消费到的快照位置。如果先过期快照再过期 Consumer，可能出现 Consumer 引用了已删除的快照。先清理过期的 Consumer 记录，才能让 Snapshot 过期安全地删除不再被引用的快照。

### 3.8 乐观并发控制机制

**FileStoreCommitImpl.commit() 核心步骤**:

```
tryCommit(committable):
  |
  +-- Step 1: 读取当前最新快照 latestSnapshot
  |
  +-- Step 2: 从 CommitMessage 生成 ManifestEntry 列表
  |     newFiles -> FileKind.ADD 的 ManifestEntry
  |     compactBefore -> FileKind.DELETE 的 ManifestEntry
  |     compactAfter -> FileKind.ADD 的 ManifestEntry
  |
  +-- Step 3: 合并到 ManifestFile
  |     将新的 ManifestEntry 追加到现有 Manifest 中
  |     如果 Manifest 过大，触发 Manifest Compaction
  |
  +-- Step 4: 创建 Snapshot 对象
  |     snapshot-{latestId+1}
  |     引用: baseManifestList, deltaManifestList, changelogManifestList
  |
  +-- Step 5: 原子写入 Snapshot 文件
  |     try: fileIO.rename(tmpFile, snapshotFile)
  |     catch FileAlreadyExists:
  |       -> 读取冲突快照
  |       -> 冲突检测 (conflictCheck)
  |       -> if 无冲突: rebase 到新快照，重试
  |       -> if 有冲突: 抛出 ConflictException
  |
  +-- Step 6: 触发 CommitCallback
```

**冲突检测逻辑**:

```
conflictCheck(committable, conflict):
    |
    +-- APPEND 类型:
    |     检查新写入文件与冲突快照的文件是否有 key 范围重叠
    |     追加表: 如果 appendCommitCheckConflict=false，跳过检查
    |     主键表: 按 (partition, bucket) 检查 key 范围
    |
    +-- COMPACT 类型:
    |     检查 compactBefore 中要删除的文件是否仍存在于最新快照
    |     如果文件已被其他 Compaction 删除: 冲突
    |
    +-- OVERWRITE 类型:
          不做冲突检测 (overwrite 总是成功)
```

**为什么使用乐观并发控制而非分布式锁**:
1. 大多数场景下不同 Writer 写不同的 (partition, bucket)，冲突概率极低
2. 避免分布式锁的额外复杂度和延迟
3. 文件系统的 rename 操作提供了原子性保证
4. 重试代价低 -- 只需重新计算 Manifest，不需要重写数据

### 4.3 DV 缓存机制

源码路径: `SnapshotReaderImpl.java` (L624-L677)

```
scanDvIndex(snapshot, buckets):
    |
    +-- Step 1: 从 dvMetaCache 读取
    |     for each (partition, bucket):
    |       fromCache = dvMetaCache.read(indexManifestPath, partition, bucket)
    |       if fromCache != null:
    |         result.put(entry, fromCache)
    |         iterator.remove()          // 已缓存的不再读取
    |         dvMetaCacheMetrics.increaseHitObject()
    |       else:
    |         dvMetaCacheMetrics.increaseMissedObject()
    |
    +-- Step 2: 从文件系统读取未命中的
    |     indexFileHandler.scan(snapshot, DELETION_VECTORS_INDEX, partitions)
    |     -> 读取 IndexManifest -> IndexFileMeta 列表
    |
    +-- Step 3: 解析 DV 元数据
    |     for each IndexFileMeta:
    |       dvRanges -> Map<dataFileName, DeletionVectorMeta>
    |       -> 转换为 Map<fileName, DeletionFile(path, offset, length, cardinality)>
    |
    +-- Step 4: 写回缓存
          dvMetaCache.put(indexManifestPath, partition, bucket, deletionFiles)
```

**为什么需要 DV 缓存**: DV 元数据解析需要读取索引文件，涉及文件 IO。在连续查询场景下 (如 Flink 流式读取的多次 checkpoint)，同一个 snapshot 的 DV 信息可能被反复读取。缓存以 `indexManifestPath + partition + bucket` 为 key，避免重复 IO。

## 6. 主键表 vs 追加表全面对比

| 维度 | 主键表 (PrimaryKeyFileStoreTable) | 追加表 (AppendOnlyFileStoreTable) |
|------|----------------------------------|-----------------------------------|
| **存储引擎** | KeyValueFileStore (LSM-tree) | AppendOnlyFileStore |
| **数据模型** | KeyValue (key + seq + kind + value) | InternalRow (纯行数据) |
| **主键** | 必须定义 | 无 |
| **写入类型** | INSERT, UPDATE, DELETE | 仅 INSERT |
| **Merge** | 需要 MergeFunction 合并同 key | 无 merge |
| **Split 生成** | MergeTreeSplitGenerator (IntervalPartition) | AppendOnlySplitGenerator (简单装箱) |
| **rawConvertible** | 有条件 (DV/FIRST_ROW/单level) | 始终 true |
| **Read Provider 数** | 4 个 (Raw/Merge/IncrChangelog/IncrDiff) | 1-2 个 (Raw/DataEvolution) |
| **谓词下推** | 仅 key 列 (保守策略) | 全部列 (无限制) |
| **非分区过滤** | pickTransformFieldMapping -> key 列过滤 | withFilter -> 全字段过滤 |
| **Compaction** | 必须 (LSM-tree 语义) | 可选 |
| **Deletion Vectors** | 支持 (显著提升读性能) | 不适用 |
| **Bucket 模式** | HASH_FIXED / HASH_DYNAMIC / KEY_DYNAMIC / POSTPONE | HASH_FIXED / BUCKET_UNAWARE |
| **RowKeyExtractor** | Fixed/Dynamic/Postpone | Fixed/AppendTable |
| **MergeEngine** | deduplicate/partial-update/aggregation/first-row | 无 |
| **Changelog** | INPUT/LOOKUP/FULL_COMPACTION | 无 (追加表只有 INSERT) |
| **LocalTableQuery** | 支持 (点查加速) | 不支持 (抛异常) |
| **streamingReadOverwrite** | `streaming-read-overwrite` 控制 | `streaming-read-append-overwrite` 控制 |
| **batch-scan-skip-level0** | 支持 (跳过 L0，只读已 compacted) | 不适用 |
| **写入时格式转换** | InternalRow -> KeyValue (L161-L172) | InternalRow 直接写入 (L143-L146) |
| **文件层级** | 多层 (Level 0-N) | 通常单层 |
| **适用场景** | 实时更新、CDC、维表关联 | 日志、事件、ETL 输出 |

## 7. FollowUpScanner 流式增量扫描

在流式读取中，StartingScanner 确定起始位置后，由 FollowUpScanner 持续扫描新快照。

**工作模型**:

```
  StartingScanner                FollowUpScanner
  (确定起始位置)                (持续消费增量)
       |                              |
       v                              v
  scan() -> NextSnapshot(N)     for snapshot = N, N+1, N+2, ...:
       |                          followUp.scan(snapshot)
       |                          -> 读取 delta/changelog manifest
       v                          -> 生成增量 Split
  [一次性]                       [持续循环]
```

| FollowUpScanner | 读取内容 | 适用场景 |
|-----------------|---------|---------|
| DeltaFollowUpScanner | 读取 deltaManifestList (只看 APPEND commit) | 默认的流式增量消费 |
| AllDeltaFollowUpScanner | 读取所有 delta (包括 COMPACT) | 需要感知 Compaction 的场景 |
| ChangelogFollowUpScanner | 读取 changelogManifestList | changelog-producer 非 NONE 时 |

**与 Iceberg 增量读取对比**:
- Paimon: 通过 Snapshot 的 deltaManifestList 直接获取增量，O(1) 定位
- Iceberg: IncrementalAppendScan 需扫描指定范围内所有快照的 manifest
- Paimon 优势: delta manifest 将增量信息独立存储，不需要和 base 做 diff

**为什么 Paimon 能做到 O(1) 定位**: Paimon 的 Snapshot 同时记录了 baseManifestList (全量) 和 deltaManifestList (增量)。流式消费只需读 delta 部分，不需要解析全量 manifest 做差集运算。

## 8. 关键设计决策总结

### 8.1 为什么 Table 需要 5 层继承?

**决策**: Table -> InnerTable -> DataTable -> FileStoreTable -> AbstractFileStoreTable

**原因**: Paimon 的表体系需要同时支持:
1. 用户稳定 API (Table)
2. 系统表和数据表的共享逻辑 (InnerTable)
3. 数据表的管理能力 (DataTable, 快照/标签/分支)
4. 存储引擎桥接 (FileStoreTable)
5. 主键表与追加表的差异化 (子类 + 模板方法)

每层有明确的职责边界，新增表类型只需在对应层扩展。

### 8.2 为什么读取路径分离 Scan 和 Read?

**决策**: ReadBuilder 分别创建 TableScan (计划) 和 TableRead (执行)

**原因**:
1. **分布式计算**: Scan 在 Driver/Coordinator 执行，生成 Split 列表；Read 在各 Worker/TaskManager 上并行执行
2. **序列化需求**: Split 需要跨节点传输，必须 Serializable
3. **灵活调度**: 引擎可以根据 Split 的 rowCount/fileSize 做负载均衡

### 8.3 为什么主键表只对 key 列做谓词下推?

**决策**: PrimaryKeyFileStoreTable 的 nonPartitionFilterConsumer 只提取 key 列谓词

**原因**: LSM-tree 中同一个 key 的不同版本分布在不同层的不同文件中。对 value 列做文件级过滤可能导致某些版本被跳过，merge 结果不正确。key 列过滤是安全的——如果 key 不在范围内，该 key 的所有版本都不在范围内。

**好处**: 保证查询结果的正确性，这是数据一致性的底线。

### 8.4 为什么 Compaction/Expire 异步执行?

**决策**: TableCommitImpl 的 maintain() 可配置为 ASYNC 模式

**原因**: Snapshot 过期和 Compaction 涉及大量文件操作，如果同步执行会阻塞写入。异步执行让写入延迟不受过期操作影响。

**好处**: 写入延迟稳定，过期操作不影响吞吐。

**代价**: 异步错误需要在下次 commit 时检测 (`maintainError.get()`)。

### 8.5 为什么 Consumer 过期在 Snapshot 过期之前?

**决策**: maintain() 中 Task 4 (Consumer) 先于 Task 5 (Snapshot)

**原因**: Consumer 记录了消费到的快照位置。如果先过期快照，Consumer 可能引用已删除的快照。先清理过期 Consumer，才能让 Snapshot 安全回收。

### 8.6 为什么 rawConvertible 是性能关键?

**决策**: SplitGenerator 为每个 Split 标记 rawConvertible

**原因**: rawConvertible=true 时可跳过整个 merge 层，直接使用列式格式的向量化读取。这避免了:
1. LoserTree/MinHeap 的 O(N log K) 比较开销
2. MergeFunction 的逐行合并计算
3. KeyValue 到 InternalRow 的格式转换

**好处**: 在 DV 模式下，主键表的批查询性能接近纯追加表水平，与 Iceberg 相当。

### 8.7 为什么 StartingScanner 有 16+ 种策略?

**决策**: 每种时间旅行/增量读取模式对应一个 StartingScanner

**原因**: 不同的业务场景需要不同的起始位置:
- 全量分析: FullStartingScanner
- 流式 CDC: ContinuousFromSnapshotStartingScanner
- 增量 ETL: IncrementalDeltaStartingScanner
- 一致性读: CompactedStartingScanner
- 时间旅行: StaticFromTimestampStartingScanner

策略模式让每种场景的逻辑独立封装，避免巨大的 switch-case。

### 8.8 为什么需要 IncrementalSplit 和 DataSplit 两种 Split?

**决策**: 全量/批式读取使用 DataSplit，增量/差异读取使用 IncrementalSplit

**原因**:
- DataSplit: 包含一组同 (partition, bucket) 的文件，可能需要 merge 或直接读取
- IncrementalSplit: 包含 before 和 after 两组文件，用于计算变更 (changelog)

两种 Split 对应不同的 SplitReadProvider，读取逻辑完全不同。统一为一种 Split 会增加条件判断的复杂度。
