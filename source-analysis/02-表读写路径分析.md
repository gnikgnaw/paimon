## 1. Table 接口完整层次

### 解决什么问题

**核心业务问题**: 如何在保持用户 API 稳定的前提下,支持主键表和追加表两种完全不同的存储语义?

**没有这个设计的后果**:
- 用户代码在升级 Paimon 版本时频繁出现 API 不兼容
- 主键表和追加表的差异暴露给用户,增加学习成本
- 系统表和数据表无法共享读写逻辑,代码重复严重

**实际场景**:
```java
// 场景1: 用户代码只依赖 Table 接口,不关心底层是主键表还是追加表
Table table = catalog.getTable(identifier);
TableScan scan = table.newReadBuilder()
    .withFilter(predicate)
    .newScan();

// 场景2: 引擎集成需要访问底层组件
InnerTable innerTable = (InnerTable) table;
InnerTableScan scan = innerTable.newScan();  // 直接创建 Scan,跳过 Builder

// 场景3: 系统表复用数据表的 Builder 逻辑
// SnapshotsTable 实现 InnerTable,自动获得 ReadBuilderImpl
```

### 有什么坑

**误区1: 直接强转为具体实现类**
```java
// 错误: 依赖具体实现类
PrimaryKeyFileStoreTable pkTable = (PrimaryKeyFileStoreTable) table;

// 正确: 依赖接口
FileStoreTable fileStoreTable = (FileStoreTable) table;
KeyValueFileStore store = (KeyValueFileStore) fileStoreTable.store();
```
**后果**: 代码在遇到系统表或未来新表类型时崩溃。

**误区2: 在 copy() 后假设表类型不变**
```java
FileStoreTable original = ...;  // PrimaryKeyFileStoreTable
TableSchema newSchema = schema.copy(Collections.emptyList());  // 移除主键
FileStoreTable copied = original.copy(newSchema);
// 坑: copied 现在是 AppendOnlyFileStoreTable,不是 PrimaryKeyFileStoreTable
```
**原因**: `copy()` 会根据新 Schema 的主键定义重新判断表类型 (L396-L402)。

**误区3: 混淆 DataTable 和 InnerTable**
```java
// 错误: 系统表不是 DataTable
DataTable dataTable = (DataTable) snapshotsTable;  // ClassCastException

// 正确: 系统表只实现 InnerTable
InnerTable innerTable = (InnerTable) snapshotsTable;
```

**生产环境注意事项**:
- 使用 `CachingCatalog` 时,必须通过 `setManifestCache()` 等方法注入缓存,否则每次读取都会重新加载 Manifest
- 在 Schema 演进场景下,`copyWithLatestSchema()` 可能改变表类型,需要重新获取 store

### 核心概念解释

**Table vs InnerTable vs DataTable**:
- **Table**: 面向用户的稳定 API,标注 `@Public`,只提供 Builder 入口
- **InnerTable**: 面向引擎集成的内部 API,提供直接创建 Scan/Read/Write 的方法
- **DataTable**: 面向数据表的管理 API,增加快照/标签/分支管理能力

**为什么需要三层**: 系统表 (如 `$snapshots`) 需要读取能力但不需要快照管理,所以只实现 InnerTable。数据表需要完整的管理能力,所以实现 DataTable。

**FileStoreTable vs AbstractFileStoreTable**:
- **FileStoreTable**: 接口层,定义了 `store()` 方法桥接存储引擎
- **AbstractFileStoreTable**: 骨架实现,使用模板方法模式让子类定制 `splitGenerator()` 和 `nonPartitionFilterConsumer()`

**模板方法模式的应用**:
```java
// AbstractFileStoreTable 固定实现
public SnapshotReader newSnapshotReader() {
    return new SnapshotReaderImpl(
        scan, schema, options, snapshotManager, changelogManager,
        splitGenerator(),  // 由子类实现
        nonPartitionFilterConsumer(),  // 由子类实现
        pathFactory, tableName, indexFileHandler, dvMetaCache
    );
}

// PrimaryKeyFileStoreTable 实现
protected SplitGenerator splitGenerator() {
    return new MergeTreeSplitGenerator(...);  // 基于 key 范围的 IntervalPartition
}

// AppendOnlyFileStoreTable 实现
protected SplitGenerator splitGenerator() {
    return new AppendOnlySplitGenerator(...);  // 基于文件大小的简单装箱
}
```

**与 Iceberg 的对比**:
- Iceberg: 2 层 (Table 接口 + BaseTable 实现),通过 format-version 运行时区分
- Paimon: 5 层继承,编译期区分主键表和追加表,每层职责清晰

### 设计理念

**为什么选择 5 层继承而非运行时判断**:

**权衡1: 编译期类型安全 vs 运行时灵活性**
- Paimon 选择编译期: 主键表和追加表的差异太大 (谓词下推、Split 生成、写入逻辑完全不同)
- 编译期区分可避免大量 `if (isPrimaryKey)` 判断,代码更清晰
- 代价: 继承层次较深,需要理解模板方法模式

**权衡2: 接口隔离 vs 接口统一**
- Paimon 选择接口隔离: InnerTable 和 DataTable 分离,让系统表不需要实现不必要的方法
- 遵循接口隔离原则 (ISP): 客户端不应依赖它不使用的接口

**架构演进考虑**:
1. **向后兼容**: Table 接口标注 `@Public`,保证用户代码升级不破坏
2. **扩展性**: 新增表类型 (如未来的 Vector Table) 只需继承 AbstractFileStoreTable
3. **可测试性**: 每层接口都可以 Mock,单元测试不需要完整的表实例

**业界对比**:
- **Iceberg**: 统一接口,通过 TableOperations 抽象存储,适合分析场景
- **Paimon**: 分层接口,通过 FileStore 抽象存储,适合流式更新场景
- **Delta Lake**: Scala trait 组合,通过 mixin 实现功能扩展

**为什么 Paimon 需要更复杂的设计**: Paimon 同时支持批和流、主键表和追加表、LSM-tree 和 Append-only,需求复杂度远高于纯分析引擎。分层设计让每层职责单一,降低整体复杂度。

### 1.2 各层职责精确定义

**Table (L52, Table.java)** -- 公共 API 层

源码路径: `paimon-core/src/main/java/org/apache/paimon/table/Table.java`

Table 是面向用户的稳定接口，标注 `@Public`（自 0.4.0 起）。它定义了三组能力:

| 能力组 | 方法 | 说明 |
|--------|------|------|
| 表元数据 | `name()`, `fullName()`, `uuid()`, `rowType()`, `partitionKeys()`, `primaryKeys()`, `options()`, `comment()`, `statistics()` | 访问表的结构信息与统计 |
| 表管理 | `rollbackTo()`, `createTag()`, `deleteTag()`, `createBranch()`, `deleteBranch()`, `fastForward()`, `newExpireSnapshots()`, `newExpireChangelog()` | 表生命周期管理操作 |
| 读写入口 | `newReadBuilder()`, `newBatchWriteBuilder()`, `newStreamWriteBuilder()`, `newVectorSearchBuilder()`, `newFullTextSearchBuilder()` | 创建读写 Builder |

**为什么这样设计**: 将面向用户的 API 与内部实现完全分离。用户只依赖 Table 接口，底层实现可自由演进而不破坏兼容性。Builder 模式让用户通过链式调用配置参数，隐藏了 Scan/Read/Write 的创建细节。

**InnerTable (L41, InnerTable.java)** -- 内部实现入口层

源码路径: `paimon-core/src/main/java/org/apache/paimon/table/InnerTable.java`

InnerTable 是 Table 之下的第一层扩展，不对外暴露，只供引擎集成使用。它新增了直接创建底层组件的方法:

```
newScan()        -> InnerTableScan       (计划层)
newStreamScan()  -> StreamDataTableScan  (流式计划层)
newRead()        -> InnerTableRead       (执行层)
newWrite(user)   -> InnerTableWrite      (写入层)
newCommit(user)  -> InnerTableCommit     (提交层)
newWriteSelector() -> Optional<WriteSelector>  (写分发层)
```

同时提供了所有 Builder 的默认实现:
- `newReadBuilder()` 默认返回 `ReadBuilderImpl(this)` (L56-L58)
- `newBatchWriteBuilder()` 默认返回 `BatchWriteBuilderImpl(this)` (L71-L73)
- `newStreamWriteBuilder()` 默认返回 `StreamWriteBuilderImpl(this)` (L76-L78)
- `newVectorSearchBuilder()` 默认返回 `VectorSearchBuilderImpl(this)` (L61-L63)
- `newFullTextSearchBuilder()` 默认返回 `FullTextSearchBuilderImpl(this)` (L66-L68)

**为什么需要这一层**: 系统表 (System Table) 也实现 InnerTable，但不需要 DataTable 的管理能力（快照、标签、分支）。InnerTable 作为最小的内部实现接口，让系统表和数据表共享 Builder 创建逻辑。

**DataTable (L33, DataTable.java)** -- 数据表管理层

源码路径: `paimon-core/src/main/java/org/apache/paimon/table/DataTable.java`

DataTable 在 InnerTable 基础上增加了"数据表"独有的管理能力:

```
newSnapshotReader()  -> SnapshotReader      (快照感知的读取协调器)
coreOptions()        -> CoreOptions         (核心配置访问)
snapshotManager()    -> SnapshotManager     (快照生命周期)
changelogManager()   -> ChangelogManager    (Changelog 生命周期)
consumerManager()    -> ConsumerManager     (消费者进度管理)
schemaManager()      -> SchemaManager       (Schema 演进)
tagManager()         -> TagManager          (标签管理)
branchManager()      -> BranchManager       (分支管理)
switchToBranch()     -> DataTable           (分支切换)
location()           -> Path                (表的存储路径)
```

关键细化: `newScan()` 的返回类型从 `InnerTableScan` 收窄为 `DataTableScan` (L36)，后者增加了 `withShard()` 能力（支持并行分片扫描）。

**为什么需要这一层**: 将数据表的管理能力（快照、标签、分支等）与基础读写抽象分离。系统表 (如 `$snapshots`, `$files`) 只需实现 InnerTable 提供读取能力，不需要自己管理快照。同时 DataTable 定义了 `SnapshotReader` -- 这是读取路径最核心的协调器，从这一层开始才有完整的快照感知读取能力。

**FileStoreTable (L56, FileStoreTable.java)** -- FileStore 桥接层

源码路径: `paimon-core/src/main/java/org/apache/paimon/table/FileStoreTable.java`

FileStoreTable 是连接"表"抽象与"存储引擎"的桥梁:

```
store()                -> FileStore<?>        核心: 获取底层存储引擎
schema()               -> TableSchema         表 Schema (包含完整的字段/分区/主键/bucket信息)
bucketMode()           -> BucketMode          桶模式
bucketSpec()           -> BucketSpec          桶规格 (mode + keys + num)
catalogEnvironment()   -> CatalogEnvironment  Catalog 环境信息

setManifestCache()     缓存: Manifest 文件缓存
setSnapshotCache()     缓存: Snapshot 对象缓存
setStatsCache()        缓存: Statistics 缓存
setDVMetaCache()       缓存: DV 元数据缓存

newWrite()             -> TableWriteImpl<?>   精确的写入实现类型
newCommit()            -> TableCommitImpl     精确的提交实现类型
newLocalTableQuery()   -> LocalTableQuery     本地点查接口
createRowKeyExtractor()-> RowKeyExtractor     行路由提取器

purgeFiles()           清除表的所有文件(分支/标签/消费者/快照)
```

**为什么需要这一层**: FileStore 是 Paimon 存储引擎的核心抽象（有 KeyValueFileStore 和 AppendOnlyFileStore 两种实现）。FileStoreTable 作为桥梁层，将 FileStore 的底层能力封装为表级操作。缓存设置方法 (`setManifestCache` 等) 体现了 `CachingCatalog` 模式 -- 由 Catalog 统一管理缓存，再注入到表中。

### 1.3 AbstractFileStoreTable 模板方法模式

源码路径: `paimon-core/src/main/java/org/apache/paimon/table/AbstractFileStoreTable.java` (L90)

AbstractFileStoreTable 是 FileStoreTable 的骨架实现，使用了经典的**模板方法模式**:

**固定实现** (所有子类共享):

```
newSnapshotReader()   创建 SnapshotReaderImpl，注入 scan/splitGenerator/pathFactory 等
newScan()             创建 DataTableBatchScan，可包装为 DataEvolutionBatchScan
newStreamScan()       创建 DataTableStreamScan
newCommit()           创建 TableCommitImpl，注入 expire/partitionExpire/tagAutoManager
copy(TableSchema)     根据 primaryKeys 是否为空决定创建哪个子类 (L396-L402)
createRowKeyExtractor() 根据 bucketMode 选择 RowKeyExtractor 实现 (L257-L270)
newWriteSelector()    根据 bucketMode 选择 WriteSelector 实现 (L229-L239)
```

**模板方法** (由子类实现):

```
abstract SplitGenerator splitGenerator();
    // 主键表: MergeTreeSplitGenerator  -- 基于 key 范围的 IntervalPartition
    // 追加表: AppendOnlySplitGenerator -- 基于文件大小的简单分组

abstract BiConsumer<FileStoreScan, Predicate> nonPartitionFilterConsumer();
    // 主键表: 只提取 key 列过滤条件 (L123-L144, PrimaryKeyFileStoreTable)
    // 追加表: 将完整谓词传给 scan (L107-L108, AppendOnlyFileStoreTable)
```

**关键的 copy() 决策逻辑** (L396-L402):

```
FileStoreTable copy(TableSchema newTableSchema) {
    return newTableSchema.primaryKeys().isEmpty()
        ? new AppendOnlyFileStoreTable(fileIO, path, newTableSchema, catalogEnvironment)
        : new PrimaryKeyFileStoreTable(fileIO, path, newTableSchema, catalogEnvironment);
}
```

**为什么在 copy 时重新判断**: 这确保了通过 `table.copy(dynamicOptions)` 或 `table.copyWithLatestSchema()` 获取的新表实例，总是与 Schema 的主键定义保持一致。如果 Schema 演进移除了主键，表类型会从 PrimaryKey 自动切换为 AppendOnly。

### 1.4 主键表与追加表的继承差异

**PrimaryKeyFileStoreTable** (L53, PrimaryKeyFileStoreTable.java)

```
store() -> KeyValueFileStore
    - 创建时从 schema 中提取 keyType、valueType
    - 创建 MergeFunctionFactory (根据 merge-engine 配置)
    - 如果 options.needLookup()=true，包装为 LookupMergeFunction (L84-L85)

splitGenerator() -> MergeTreeSplitGenerator
    - 需要 keyComparator 进行 key 范围分析
    - deletionVectorsEnabled 影响 rawConvertible 判定
    - mergeEngine 影响是否可跳过 merge (FIRST_ROW 引擎可以)

nonPartitionFilterConsumer():
    - 只提取 key 列过滤条件传给 KeyValueFileStoreScan.withKeyFilter()
    - 同时传全量过滤给 withValueFilter() 用于 bucket 级过滤
    - 原因: 值字段过滤可能导致数据不一致 (见 L127-L134 注释)

newRead() -> KeyValueTableRead
    - 4 个 SplitReadProvider: Raw/Merge/IncrementalChangelog/IncrementalDiff

newWrite() -> TableWriteImpl<KeyValue>
    - 将 InternalRow 转换为 KeyValue 结构 (L161-L172)
    - kv.replace(primaryKey, UNKNOWN_SEQUENCE, rowKind, row)
```

**AppendOnlyFileStoreTable** (L53, AppendOnlyFileStoreTable.java)

```
store() -> AppendOnlyFileStore
    - 创建时使用 rowType.notNull() 作为文件格式类型

splitGenerator() -> AppendOnlySplitGenerator
    - 无需 key 比较器
    - BUCKET_UNAWARE 模式下流式也按 batch 方式切分

nonPartitionFilterConsumer():
    - 将完整谓词直接传给 AppendOnlyFileStoreScan.withFilter()
    - 追加表无 key 重叠问题，可安全过滤任意列

newRead() -> AppendTableRead
    - 1 个 SplitReadProvider (根据配置二选一): RawFile 或 DataEvolution

newWrite() -> TableWriteImpl<InternalRow>
    - 直接写入 InternalRow，校验只接受 RowKind.isAdd() (L143-L146)
    - 如果不是 INSERT/INSERT_ONLY，抛出异常
```

### 1.5 与 Iceberg 的对比

| 维度 | Paimon | Iceberg |
|------|--------|---------|
| 表抽象层数 | 5 层继承 (Table -> ... -> PrimaryKeyFileStoreTable) | 2 层 (Table 接口 + BaseTable 实现) |
| 表类型区分 | 编译期: 两个子类 + 模板方法 | 运行时: format-version 控制 |
| 存储引擎 | FileStore 作为独立组件，有 KV 和 AppendOnly 两种 | TableOperations 统一管理 |
| 谓词下推 | 主键表只下推 key 列 (LSM 语义约束) | 可下推任意列 (无 merge 语义) |
| 缓存注入 | 通过 set 方法由 CachingCatalog 注入 | 通过构造参数传入 |
| 设计理念 | 面向流式更新的分层设计 | 面向分析的统一接口 |

**为什么 Paimon 需要更多层**: Paimon 同时支持主键表 (LSM-tree 语义) 和追加表 (Append-only 语义)，两者在谓词下推、Split 生成、数据写入上差异显著。编译期通过子类区分可避免运行时条件判断，且各子类职责清晰。同时 Paimon 在表级别集成了快照/标签/分支管理能力，需要单独的 DataTable 层来承载。

## 2. 读取路径完整流程

### 解决什么问题

**核心业务问题**: 如何在保证数据一致性的前提下,支持批式全量读取、流式增量消费、时间旅行、增量差异等多种读取模式?

**没有这个设计的后果**:
- 每种读取模式都需要独立实现,代码重复严重
- 批和流的读取逻辑混在一起,难以维护
- 无法利用文件统计信息做优化,性能低下
- 分布式计算引擎无法有效并行化读取

**实际场景**:
```java
// 场景1: 批式全量读取 (OLAP 分析)
TableScan scan = table.newReadBuilder()
    .withFilter(Predicate.equal("status", "active"))
    .withProjection(new int[]{0, 2, 5})  // 只读3列
    .newScan();
List<Split> splits = scan.plan().splits();  // 生成 Split 列表
TableRead read = table.newReadBuilder().newRead();
for (Split split : splits) {
    RecordReader<InternalRow> reader = read.createReader(split);
    // 并行读取
}

// 场景2: 流式增量消费 (CDC)
StreamDataTableScan streamScan = table.newReadBuilder()
    .withFilter(predicate)
    .newStreamScan();
while (true) {
    Plan plan = streamScan.plan();  // 持续扫描新快照
    for (Split split : plan.splits()) {
        // 消费增量数据
    }
}

// 场景3: 时间旅行 (审计查询)
TableScan scan = table.newReadBuilder()
    .withFilter(predicate)
    .newScan()
    .withSnapshot(snapshotId);  // 读取历史快照

// 场景4: 增量差异 (ETL)
TableScan scan = table.newReadBuilder()
    .withIncrementalBetween("tag1", "tag2")  // 两个时间点的差异
    .newScan();
```

### 有什么坑

**误区1: 混淆 Scan 和 Read 的职责**
```java
// 错误: 在 Scan 阶段就开始读取数据
TableScan scan = table.newReadBuilder().newScan();
Plan plan = scan.plan();
for (Split split : plan.splits()) {
    // 错误: 直接用 scan 读取
    scan.read(split);  // 编译错误,scan 没有 read 方法
}

// 正确: Scan 只负责计划,Read 负责执行
TableScan scan = table.newReadBuilder().newScan();
TableRead read = table.newReadBuilder().newRead();  // 单独创建 Read
Plan plan = scan.plan();
for (Split split : plan.splits()) {
    RecordReader<InternalRow> reader = read.createReader(split);
}
```
**后果**: Scan 在 Driver 执行生成 Split,Read 在 Worker 执行读取数据。混淆两者会导致无法分布式执行。

**误区2: 假设 Split 的 mergedRowCount 总是可用**
```java
// 错误: 直接使用 mergedRowCount
long totalRows = 0;
for (DataSplit split : splits) {
    totalRows += split.mergedRowCount().getAsLong();  // 可能抛 NoSuchElementException
}

// 正确: 检查是否可用
long totalRows = 0;
for (DataSplit split : splits) {
    OptionalLong count = split.mergedRowCount();
    if (count.isPresent()) {
        totalRows += count.getAsLong();
    } else {
        // 无法估算,需要实际读取
    }
}
```
**原因**: 只有 rawConvertible=true 且 DV cardinality 已知时,mergedRowCount 才能精确计算。需要 merge 的 Split 返回 empty。

**误区3: 在主键表上对 value 列做文件级过滤**
```java
// 错误: 自己实现文件过滤逻辑
List<DataFileMeta> filtered = new ArrayList<>();
for (DataFileMeta file : allFiles) {
    if (file.valueStats().minValues().get(valueFieldIndex) >= threshold) {
        filtered.add(file);  // 跳过不满足条件的文件
    }
}
// 坑: 可能导致 merge 结果不正确
```
**后果**: LSM-tree 中同一个 key 的不同版本在不同文件中,跳过任何版本都可能导致错误结果。

**生产环境注意事项**:
- 使用 `withLimit()` 时,确保 Split 的 rawConvertible=true,否则 Limit 下推无效
- 流式消费必须配置 `consumer-id`,否则重启后从头消费
- 在 BUCKET_UNAWARE 模式下,流式读取也会切分 Split,需要注意并行度设置
- TopN 下推只对 n <= 100 生效,超过此值不会优化

### 核心概念解释

**Scan vs Read**:
- **Scan (计划层)**: 在 Driver/Coordinator 执行,负责扫描 Manifest、应用分区/桶过滤、生成 Split 列表
- **Read (执行层)**: 在 Worker/TaskManager 执行,负责读取 Split 中的文件、应用行级过滤、返回数据

**为什么分离**: 分布式计算引擎需要先在中心节点生成执行计划 (Split 列表),再分发到各节点并行执行。Scan 和 Read 的分离天然适配这种模式。

**Split 的本质**:
- Split 是读取任务的最小调度单元,包含一组需要一起读取的文件
- 每个 Split 对应一个 (partition, bucket) 对
- Split 必须 Serializable,可以跨节点传输
- Split 的大小影响并行度: Split 太大导致负载不均,太小导致调度开销大

**StartingScanner vs FollowUpScanner**:
- **StartingScanner**: 确定"从哪里开始读",只执行一次,返回起始快照 ID 或初始 Split 列表
- **FollowUpScanner**: 持续扫描新快照,只在流式模式下使用,循环调用

**rawConvertible 的含义**:
- rawConvertible=true: 文件可以直接读取,跳过 Paimon 的 merge 层
- rawConvertible=false: 文件之间有 key 重叠,必须通过 LoserTree 归并读取
- 影响: rawConvertible=true 时性能提升 2-10 倍 (取决于文件数量和 merge 复杂度)

**Manifest 的三种类型**:
- **baseManifestList**: 全量数据的 Manifest 列表,包含所有 FileKind.ADD 的文件
- **deltaManifestList**: 增量数据的 Manifest 列表,只包含本次 commit 新增的文件
- **changelogManifestList**: 变更日志的 Manifest 列表,包含 before/after 记录

**与 Iceberg 的对比**:
- Iceberg: TableScan 直接返回 FileScanTask,包含文件路径和过滤条件
- Paimon: TableScan 返回 DataSplit/IncrementalSplit,包含文件元数据和 DV 信息
- Paimon 优势: Split 包含更多元信息 (rawConvertible、mergedRowCount、DV),可做更多优化

### 设计理念

**为什么选择三阶段读取 (Builder -> Scan -> Read)**:

**权衡1: 灵活性 vs 简单性**
- Paimon 选择灵活性: Builder 收集参数,Scan 生成计划,Read 执行读取
- 三阶段让每个阶段职责单一,易于扩展新的读取模式
- 代价: 用户需要理解三个概念,学习曲线稍陡

**权衡2: 计划时优化 vs 执行时优化**
- Paimon 选择计划时优化: Limit/TopN 下推在 Scan 阶段完成,减少 Split 数量
- 好处: Worker 节点只需执行,不需要做优化决策,降低执行复杂度
- 代价: Scan 阶段需要加载文件统计信息,增加计划时间

**架构演进考虑**:
1. **向后兼容**: ReadBuilder 接口稳定,内部实现可自由演进
2. **扩展性**: 新增 StartingScanner 策略只需实现接口,不影响现有代码
3. **可测试性**: Scan 和 Read 分离,可以独立 Mock 测试

**业界对比**:
- **Iceberg**: TableScan 返回 FileScanTask,包含文件路径和 residual filter
- **Delta Lake**: Snapshot.scan() 返回 DataFrame,隐藏了 Split 概念
- **Paimon**: 显式的 Split 抽象,适配 Flink/Spark 的分布式执行模型

**为什么 Paimon 需要更复杂的 Split**: Paimon 的主键表需要 merge,Split 必须包含 rawConvertible 标记来决定读取策略。追加表虽然简单,但为了统一接口也使用 Split 抽象。

**为什么有 16+ 种 StartingScanner**: Paimon 同时支持批和流、全量和增量、时间旅行和差异读取,每种模式的起始位置逻辑完全不同。策略模式让每种场景独立封装,避免巨大的 if-else。

### 2.2 ReadBuilder 配置阶段

### 解决什么问题

**核心业务问题**: 如何让用户通过链式调用灵活配置读取参数,同时确保参数正确传递到 Scan 和 Read 两个阶段?

**没有这个设计的后果**:
- 用户需要直接操作 Scan 和 Read 对象,暴露内部实现细节
- 参数传递容易遗漏,导致过滤条件不生效
- 无法在编译期检查参数冲突 (如 bucketFilter 和 shard 互斥)

**实际场景**:
```java
// 场景1: 复杂查询配置
TableScan scan = table.newReadBuilder()
    .withFilter(Predicate.and(
        Predicate.equal("status", "active"),
        Predicate.greaterThan("amount", 1000)
    ))
    .withProjection(new int[]{0, 2, 5})  // 只读3列
    .withPartitionFilter(PartitionPredicate.of("dt", "2024-01-01"))
    .withLimit(1000)  // Limit 下推
    .newScan();

// 场景2: 分片读取 (Flink 并行度=4)
TableScan scan = table.newReadBuilder()
    .withShard(0, 4)  // 读取第0个分片,共4个分片
    .newScan();

// 场景3: 桶过滤 (只读特定桶)
TableScan scan = table.newReadBuilder()
    .withBucketFilter(Filter.in(0, 2, 5))  // 只读桶 0,2,5
    .newScan();
```

### 有什么坑

**误区1: 多次调用 withFilter 会覆盖而非叠加**
```java
// 错误: 以为会 AND 合并
ReadBuilder builder = table.newReadBuilder()
    .withFilter(Predicate.equal("status", "active"))
    .withFilter(Predicate.greaterThan("amount", 1000));  // 覆盖了前一个
// 实际: 只有第二个 filter 生效

// 正确: 手动 AND 合并
ReadBuilder builder = table.newReadBuilder()
    .withFilter(Predicate.and(
        Predicate.equal("status", "active"),
        Predicate.greaterThan("amount", 1000)
    ));
```
**原因**: ReadBuilderImpl.withFilter() 直接赋值 `this.filter = filter`,不做合并 (L93-L96)。

**误区2: 同时使用 bucketFilter 和 shard**
```java
// 错误: 两者互斥
ReadBuilder builder = table.newReadBuilder()
    .withBucketFilter(Filter.in(0, 2))
    .withShard(0, 4);  // 抛出 IllegalArgumentException
```
**后果**: configureScan() 会检测到冲突并抛异常 (L206-L209)。

**误区3: 假设 withProjection 会减少 Scan 阶段的开销**
```java
// 误解: 以为 projection 会减少 Manifest 扫描的开销
ReadBuilder builder = table.newReadBuilder()
    .withProjection(new int[]{0});  // 只读第0列
// 实际: Manifest 扫描仍然读取所有列的统计信息
```
**原因**: Projection 只影响 Read 阶段的列裁剪,Scan 阶段仍需读取完整的文件元数据。

**生产环境注意事项**:
- 在 BUCKET_UNAWARE 模式下,withShard 可能导致数据倾斜 (所有数据在 bucket=0)
- withLimit 只对 rawConvertible=true 的 Split 有效,主键表可能无法优化
- withRowRangeIndex 只在特定场景下生效 (如 Flink 的 FLIP-27 Source)

### 核心概念解释

**Builder 模式的应用**:
- ReadBuilder 收集所有读取参数,延迟到 newScan()/newRead() 时才创建对象
- 好处: 参数可以任意顺序设置,最后统一校验和传递
- 实现: ReadBuilderImpl 持有所有参数字段,newScan() 时调用 configureScan() 传递

**filter 的两次传递**:
```
withFilter(predicate)
    |
    +-- newScan() -> scan.withFilter(filter)      // 文件级过滤
    +-- newRead() -> read.withFilter(filter)      // 行级过滤
```
**为什么传两次**: Scan 用 filter 跳过不匹配的文件 (基于 min/max 统计),Read 用 filter 跳过不匹配的行。

**Projection 的表示**:
- `int[] projection`: 字段索引数组,如 `[0, 2, 5]` 表示只读第 0、2、5 列
- 转换为 RowType: `RowType.project(projection)` 生成裁剪后的类型
- 传递: `scan.withReadType(readType)` 和 `read.withReadType(readType)`

**Shard vs BucketFilter**:
- **Shard**: 按 bucket 编号取模分片,如 `shard(0, 4)` 读取 bucket % 4 == 0 的桶
- **BucketFilter**: 显式指定桶号列表,如 `Filter.in(0, 2, 5)` 只读这3个桶
- 互斥原因: 两者都控制桶的选择,同时使用会产生歧义

**PartitionFilter 的类型**:
- `PartitionPredicate`: 基于分区字段的谓词,如 `dt='2024-01-01' AND region='us'`
- 优化: 在 Manifest 扫描时直接跳过不匹配的分区,避免读取文件元数据

### 设计理念

**为什么选择 Builder 模式而非构造参数**:

**权衡1: 可读性 vs 简洁性**
- Paimon 选择可读性: 链式调用清晰表达每个参数的含义
- 如果用构造参数: `new TableScan(filter, projection, limit, topN, shard, ...)`  参数过多且顺序难记
- 代价: 需要额外的 Builder 类

**权衡2: 参数校验时机**
- Paimon 选择延迟校验: 在 newScan() 时才检查 bucketFilter 和 shard 冲突
- 好处: 用户可以先设置参数,最后统一校验
- 代价: 错误发现较晚,不是在 withXxx() 时立即报错

**架构演进考虑**:
1. **向后兼容**: 新增参数只需在 Builder 中添加方法,不影响现有代码
2. **扩展性**: 未来可以添加更多 withXxx() 方法,如 withHint()、withTimeout()
3. **可测试性**: Builder 可以独立测试参数传递逻辑

**业界对比**:
- **Iceberg**: TableScan 直接提供 filter()/select()/limit() 方法,没有 Builder
- **Delta Lake**: DataFrame API 隐藏了 Builder,用户直接操作 DataFrame
- **Paimon**: 显式的 Builder,适配 Table API 的设计风格

**为什么 Paimon 需要 Builder**: Paimon 的 Table 接口标注 @Public,必须保持稳定。Builder 作为中间层,可以灵活调整参数传递逻辑而不破坏 Table 接口。

源码路径: `paimon-core/src/main/java/org/apache/paimon/table/source/ReadBuilderImpl.java`

ReadBuilderImpl 是 ReadBuilder 接口的唯一实现，在 `InnerTable.newReadBuilder()` 中被创建 (InnerTable.java L56-L58)。

ReadBuilder 采用 Builder 模式收集所有读取参数:

```
ReadBuilderImpl {
    InnerTable table;              // 关联的表
    RowType partitionType;         // 分区字段类型 (从 table.rowType() 投影)
    String defaultPartitionName;   // 默认分区名 (core option)

    Predicate filter;              // 过滤谓词 (可多次叠加, AND 合并)
    Integer limit;                 // 行数限制
    TopN topN;                     // TopN 优化
    Integer shardIndexOfThisSubtask;    // 分片索引
    Integer shardNumberOfParallelSubtasks;  // 并行度
    PartitionPredicate partitionFilter;  // 分区过滤器
    Integer specifiedBucket;       // 指定桶号
    Filter<Integer> bucketFilter;  // 桶过滤器
    RowType readType;              // 读取类型 (列裁剪)
    RowRangeIndex rowRangeIndex;   // 行范围索引
    boolean dropStats;             // 是否丢弃统计信息
}
```

**newScan() 的配置传递** (L181-L229):

```
ReadBuilderImpl.newScan()
    |
    v
configureScan(table.newScan())
    |
    +-- scan.withFilter(filter)               // 谓词传递
    +-- scan.withReadType(readType)           // 列裁剪传递
    +-- scan.withPartitionFilter(partFilter)  // 分区过滤
    +-- scan.withRowRangeIndex(rowRangeIndex) // 行范围
    +-- scan.withLimit(limit)                 // limit pushdown (L184)
    +-- scan.withTopN(topN)                   // topN pushdown (L187)
    |
    +-- [互斥检查] bucketFilter 和 shard 不能同时使用 (L206-L209)
    +-- scan.withShard(index, parallelism)    // 或 (L212-L213)
    +-- scan.withBucketFilter(bucketFilter)   // 二选一 (L223)
```

**newRead() 的配置传递** (L232-L244):

```
ReadBuilderImpl.newRead()
    |
    v
table.newRead()
    .withFilter(filter)     // 谓词传递给 Read (行级过滤)
    .withReadType(readType) // 列裁剪传递
    .withTopN(topN)         // TopN 传递
    .withLimit(limit)       // Limit 传递
```

**为什么 filter 同时传给 Scan 和 Read**: Scan 层用 filter 做文件级过滤 (跳过不匹配的文件)；Read 层用 filter 做行级过滤 (跳过不匹配的行)。两层过滤互补，Scan 减少读取文件数量，Read 减少返回行数。

### 2.3 TableScan 计划阶段

### 解决什么问题

**核心业务问题**: 如何在不读取实际数据的情况下,生成最优的 Split 执行计划,并尽可能减少需要读取的数据量?

**没有这个设计的后果**:
- 无法利用文件统计信息做 Limit/TopN 下推,读取大量不必要的数据
- 无法在计划阶段过滤文件,所有文件都要传递到 Worker 节点
- 权限检查无法在计划阶段完成,可能在执行时才发现无权限

**实际场景**:
```java
// 场景1: Limit 下推优化
TableScan scan = table.newReadBuilder()
    .withLimit(100)
    .newScan();
Plan plan = scan.plan();
// 优化: 只生成能满足 100 行的 Split,不扫描所有文件

// 场景2: TopN 下推优化
TableScan scan = table.newReadBuilder()
    .withTopN(TopN.of("amount", 10, true))  // ORDER BY amount DESC LIMIT 10
    .newScan();
Plan plan = scan.plan();
// 优化: 利用 min/max 统计,跳过不可能包含 Top 10 的文件

// 场景3: 分区裁剪
TableScan scan = table.newReadBuilder()
    .withPartitionFilter(PartitionPredicate.of("dt", "2024-01-01"))
    .newScan();
Plan plan = scan.plan();
// 优化: 在 Manifest 扫描时直接跳过其他分区

// 场景4: 权限检查
TableScan scan = table.newReadBuilder().newScan();
Plan plan = scan.plan();  // 如果无权限,这里抛出 SecurityException
```

### 有什么坑

**误区1: 假设 Limit 下推总是生效**
```java
// 错误: 以为设置 limit 就能减少 Split 数量
TableScan scan = table.newReadBuilder()
    .withLimit(100)
    .newScan();
Plan plan = scan.plan();
// 坑: 如果 Split 的 rawConvertible=false,无法估算行数,Limit 下推失效
```
**原因**: applyPushDownLimit() 依赖 split.mergedRowCount(),只有 rawConvertible=true 时才能精确计算 (L128-L165)。

**误区2: 假设 TopN 下推对所有查询都有效**
```java
// 错误: 以为任何 TopN 都能优化
TableScan scan = table.newReadBuilder()
    .withTopN(TopN.of("amount", 10000, true))  // LIMIT 10000
    .newScan();
// 坑: TopN 下推只对 n <= 100 生效 (L167-L205)
```
**原因**: TopN 优化需要加载所有 Split 的统计信息,n 太大时代价超过收益。

**误区3: 在主键表上期望 value 列的文件过滤**
```java
// 错误: 以为 value 列的过滤会跳过文件
TableScan scan = table.newReadBuilder()
    .withFilter(Predicate.greaterThan("amount", 1000))  // amount 是 value 列
    .newScan();
Plan plan = scan.plan();
// 坑: 主键表只对 key 列做文件级过滤,value 列过滤只在行级生效
```
**后果**: 所有文件都会被包含在 Split 中,无法减少 IO。

**生产环境注意事项**:
- 在大表上使用 TopN 下推时,确保 n <= 100,否则反而增加计划时间
- 主键表的 Limit 下推效果取决于 DV 是否启用和 compaction 程度
- 权限检查在 plan() 时同步执行,可能阻塞较长时间

### 核心概念解释

**Plan 的结构**:
```java
interface Plan {
    Long watermark();           // 水位线 (流式场景)
    Long snapshotId();          // 快照 ID
    List<Split> splits();       // Split 列表
}
```

**Limit 下推的原理**:
1. 遍历 Split 列表,累加每个 Split 的 mergedRowCount()
2. 当累加值 >= limit 时,截断剩余 Split
3. 对最后一个 Split,设置 recordsToSkip 和 limit 参数

**TopN 下推的原理**:
1. 提取排序列的 min/max 统计信息
2. 根据排序方向 (ASC/DESC) 和 limit,计算阈值
3. 过滤掉不可能包含 Top N 结果的 Split
4. 例如: `ORDER BY amount DESC LIMIT 10`,如果某 Split 的 max(amount) < 当前第10大的值,则跳过

**StartingScanner 的选择**:
- createStartingScanner() 根据配置选择 16+ 种策略之一
- 批式: FullStartingScanner、CompactedStartingScanner、IncrementalDeltaStartingScanner 等
- 流式: ContinuousLatestStartingScanner、ContinuousFromSnapshotStartingScanner 等

**SnapshotReader 的职责**:
- 协调 FileStoreScan (Manifest 扫描) 和 SplitGenerator (Split 划分)
- 应用分区过滤、桶过滤、文件过滤
- 加载 DV 元数据并关联到 Split

**与 Iceberg 的对比**:
- Iceberg: TableScan.planFiles() 返回 CloseableIterable<FileScanTask>
- Paimon: TableScan.plan() 返回 Plan (包含 watermark、snapshotId、splits)
- Paimon 优势: Plan 包含更多元信息,支持流式场景的 watermark

### 设计理念

**为什么选择在 Scan 阶段做优化而非 Read 阶段**:

**权衡1: 计划时优化 vs 执行时优化**
- Paimon 选择计划时优化: Limit/TopN 在 Scan 阶段完成,减少 Split 数量
- 好处: Worker 节点只需执行,不需要做优化决策,降低执行复杂度
- 代价: Scan 阶段需要加载文件统计信息,增加计划时间 (通常 < 1s)

**权衡2: 保守优化 vs 激进优化**
- Paimon 选择保守优化: 只在能保证正确性的情况下才做优化
- 例如: 主键表只对 key 列做文件过滤,避免 merge 结果不正确
- 代价: 某些场景下无法充分利用统计信息

**架构演进考虑**:
1. **向后兼容**: Plan 接口稳定,内部优化逻辑可自由演进
2. **扩展性**: 新增优化策略只需在 planWithoutAuth() 中添加,不影响现有代码
3. **可测试性**: Scan 和 Read 分离,可以独立测试计划逻辑

**业界对比**:
- **Iceberg**: 支持 Limit 下推,但不支持 TopN 下推
- **Delta Lake**: 通过 Spark 的 CBO 做优化,不在 Scan 层
- **Paimon**: 在 Scan 层做优化,适配 Flink 的执行模型

**为什么 Paimon 需要在 Scan 层做优化**: Flink 的 Source 接口要求在 SplitEnumerator 中生成 Split,这对应 Paimon 的 Scan 阶段。如果不在 Scan 层优化,就只能在 Read 层优化,但 Read 层是分布式执行的,无法全局协调。

**为什么 TopN 限制 n <= 100**: TopN 优化需要加载所有 Split 的统计信息到内存,并排序找出阈值。当 n 很大时:
1. 阈值不够严格,过滤效果差
2. 加载统计信息的开销超过优化收益
3. 内存占用增加

实验表明 n=100 是一个较好的平衡点。

**Limit 下推优化** (L128-L165):

当设置了 `withLimit(n)` 时，Paimon 不需要扫描所有 Split。它累加每个 Split 的 `mergedRowCount()`，当总行数达到 limit 时截断剩余 Split。

**为什么这能工作**: `mergedRowCount()` 对于 rawConvertible 的 Split 可以精确计算（文件行数 - DV 删除数）。对于需要 merge 的 Split 则返回 `OptionalLong.empty()` 表示无法估算，此时不做截断。

**TopN 下推优化** (L167-L205):

**为什么限制 n <= 100**: TopN 优化需要加载所有 Split 的统计信息 (`keepStats()`)，代价与 Split 数量成正比。n 太大时优化收益递减。

**Consumer 优先机制** (L229-L239):

在流式模式下，如果配置了 `consumer-id` 且未设置 `consumer.ignore-progress=true`，系统优先从 Consumer 记录的进度恢复:

```
if (isStreaming && consumerId != null && !options.consumerIgnoreProgress()) {
    ConsumerManager consumerManager = snapshotReader.consumerManager();
    Optional<Consumer> consumer = consumerManager.consumer(consumerId);
    if (consumer.isPresent()) {
        return new ContinuousFromSnapshotStartingScanner(
            snapshotManager, changelogManager, consumer.get().nextSnapshot(), ...);

}
```

**为什么 Consumer 优先**: Consumer 记录了上次消费到的位置，这实现了"恰好一次"语义的消费进度持久化。即使作业重启，也能从断点恢复而不是重头消费。

### 2.9 rawConvertible 优化

**定义**: 当一个 DataSplit 标记为 `rawConvertible=true` 时，意味着其中的文件可以直接逐文件读取，跳过 Paimon 的 merge 逻辑。

**启用条件**:

| 表类型 | 条件 | rawConvertible |
|--------|------|----------------|
| 追加表 | 始终 | true |
| 主键表 (DV) | `deletion-vectors.enabled=true` | true |
| 主键表 (FIRST_ROW) | `merge-engine=first-row` 且 level > 0 | true |
| 主键表 (普通) | 所有文件 level > 0 + 无 deleteRow + 同一 level | true |
| 主键表 (普通) | 有 level=0 文件或多 level 文件 key 重叠 | false |
| 主键表 (普通) | 单文件且无 deleteRow (即使跨 level) | true |

**为什么能跳过 merge**:

- DV 模式: 每个数据文件有对应的 Deletion Vector (Roaring Bitmap) 标记哪些行被删除。读取时只需跳过标记位，不需要和其他文件做归并。
- FIRST_ROW 引擎: 只取每个 key 的第一条记录，level > 0 的文件已经 compacted，保证每个 key 只有一条。
- 单 level + 无重叠: LSM-tree 的同 level 文件 key 不重叠，可独立读取。

**性能提升有多大**:

rawConvertible=true 时:
1. 跳过 LoserTree/MinHeap 排序归并 -- 避免 O(N log K) 比较 (K=文件数)
2. 跳过 MergeFunction 调用 -- 避免 CPU 密集的合并计算
3. 可利用列式格式的向量化读取 -- Parquet/ORC 的 batch 读取直接返回
4. 可利用引擎原生优化 -- Spark/Flink 可直接使用自己的 Parquet Reader

在 DV 模式下，主键表的批查询性能可接近 Iceberg 等 MoR (Merge-on-Read) 引擎的水平。

### 2.13 谓词下推的保守策略

```
+====================================================================+
|                    谓词下推三层过滤                                   |
+====================================================================+
|                                                                      |
|  Layer 1: 分区过滤 (Manifest 级别)                                   |
|     SnapshotReader.withFilter() -> splitPartitionPredicates()        |
|     -> scan.withPartitionFilter(partitionPredicate)                  |
|     -> 在 Manifest 扫描时跳过不匹配的分区 ManifestEntry              |
|     效果: 跳过整个分区目录                                           |
|                                                                      |
|  Layer 2: 文件级过滤 (Stats 级别)                                    |
|     根据文件的 min/max/null 统计信息过滤整个文件                      |
|                                                                      |
|     主键表 (PrimaryKeyFileStoreTable.java L123-L144):               |
|       +-- 只提取 KEY 列谓词 -> KeyValueFileStoreScan.withKeyFilter() |
|       +-- 完整谓词 -> withValueFilter() (仅 bucket 级过滤)          |
|       原因: 值字段过滤可能导致数据不一致                              |
|                                                                      |
|     追加表 (AppendOnlyFileStoreTable.java L107-L108):               |
|       +-- 完整谓词 -> AppendOnlyFileStoreScan.withFilter()           |
|       原因: 追加表无 key 重叠，安全过滤任意列                        |
|                                                                      |
|  Layer 3: 行级过滤                                                   |
|     在 RecordReader 内部应用过滤谓词                                 |
|     文件索引 (bloom/bitmap/bsi): 在读文件前做进一步过滤              |
|                                                                      |
+====================================================================+
```

**主键表只对 key 列下推的根本原因** (PrimaryKeyFileStoreTable.java L127-L134):

```
考虑这个场景:
  data file 1: insert key=a, value=1    (level=1)
  data file 2: update key=a, value=2    (level=0)
  filter: value = 1

如果对 value 做文件级 filter push down:
  -> file 1 的 stats 显示 value min=1, max=1 => 匹配 => 选中
  -> file 2 的 stats 显示 value min=2, max=2 => 不匹配 => 跳过

最终结果: key=a, value=1
正确结果: 空集 (key=a 的最终值是 2，不满足 value=1)

原因: LSM-tree 中同一个 key 的不同版本分布在不同文件中。
      跳过任何一个版本都可能导致 merge 结果不正确。
      只有 KEY 列过滤是安全的——如果 key 范围不匹配，所有版本都不匹配。
```

**withValueFilter() 的 bucket 级过滤** (L143):

虽然不能在文件级过滤 value 列，但可以在 bucket 级别过滤: 如果一个 bucket 内所有文件的 value 列统计信息都不满足谓词，可以安全跳过整个 bucket。因为一个 bucket 内的所有 key 版本都在这个 bucket 中。

## 3. 写入路径完整流程

### 解决什么问题

**核心业务问题**: 如何在分布式环境下保证数据写入的原子性、一致性和高性能,同时支持批式和流式两种写入模式?

**没有这个设计的后果**:
- 多个 Writer 并发写入时可能产生数据冲突和丢失
- 无法支持 CDC 场景的 UPDATE/DELETE 操作
- 写入性能低下,无法利用内存缓冲和异步 Compaction
- 快照和文件无法自动过期,存储空间持续增长

**实际场景**:
```java
// 场景1: 批式写入 (ETL)
TableWrite write = table.newBatchWriteBuilder().newWrite();
TableCommit commit = table.newBatchWriteBuilder().newCommit();
for (InternalRow row : rows) {
    write.write(row);
}
List<CommitMessage> messages = write.prepareCommit(true, 0);
commit.commit(messages);  // 原子提交
write.close();
commit.close();

// 场景2: 流式写入 (Flink CDC)
TableWrite write = table.newStreamWriteBuilder()
    .withCommitUser("flink-job-1")
    .newWrite();
TableCommit commit = table.newStreamWriteBuilder()
    .withCommitUser("flink-job-1")
    .newCommit();
while (true) {
    for (InternalRow row : nextBatch()) {
        write.write(row);  // 支持 INSERT/UPDATE/DELETE
    }
    List<CommitMessage> messages = write.prepareCommit(false, checkpointId);
    commit.commit(messages);  // 每个 checkpoint 提交一次
}

// 场景3: 覆写分区 (Hive 兼容)
TableWrite write = table.newStreamWriteBuilder()
    .withOverwrite(partitionSpec)  // 覆写指定分区
    .newWrite();
// ... 写入数据 ...
commit.commit(messages);  // 原子替换分区数据
```

### 有什么坑

**误区1: 忘记调用 prepareCommit**
```java
// 错误: 直接 commit
TableWrite write = table.newBatchWriteBuilder().newWrite();
for (InternalRow row : rows) {
    write.write(row);
}
commit.commit(Collections.emptyList());  // 空提交,数据丢失

// 正确: 先 prepareCommit
List<CommitMessage> messages = write.prepareCommit(true, 0);
commit.commit(messages);
```
**后果**: prepareCommit() 会 flush 内存中的数据并生成 CommitMessage,不调用会导致数据丢失。

**误区2: 在追加表上写入 UPDATE/DELETE**
```java
// 错误: 追加表不支持 UPDATE
InternalRow row = GenericRow.of(...);
row.setRowKind(RowKind.UPDATE_AFTER);
write.write(row);  // 抛出 IllegalArgumentException
```
**原因**: AppendOnlyFileStoreTable 的 recordExtractor 会校验 rowKind.isAdd() (L143-L146)。

**误区3: 多个 Writer 使用相同的 commitUser**
```java
// 错误: 两个 Flink 作业使用相同的 commitUser
// Job 1:
TableWrite write1 = table.newStreamWriteBuilder()
    .withCommitUser("my-job")  // 相同
    .newWrite();

// Job 2:
TableWrite write2 = table.newStreamWriteBuilder()
    .withCommitUser("my-job")  // 相同
    .newWrite();
// 坑: 冲突检测可能失效,导致数据不一致
```
**后果**: commitUser 用于标识 Writer,相同的 commitUser 会被认为是同一个 Writer 的重试,可能跳过冲突检测。

**误区4: 在 overwrite 模式下多次 commit**
```java
// 错误: overwrite 模式不支持多次提交
TableWrite write = table.newStreamWriteBuilder()
    .withOverwrite(partition)
    .newWrite();
// ... 写入数据 ...
commit.commit(messages1);  // 第一次 OK
commit.commit(messages2);  // 抛出 IllegalArgumentException
```
**原因**: commitMultiple() 会检查 overwrite 模式下 committables.size() > 1 (L251-L254)。

**生产环境注意事项**:
- 流式写入必须配置唯一的 commitUser,建议使用 `jobId + "-" + taskId`
- prepareCommit(waitCompaction=true) 会阻塞等待 Compaction 完成,可能导致 checkpoint 超时
- 在高并发写入场景下,冲突重试可能导致延迟增加,需要调整 `commit.retry.max-attempts`
- 快照过期是异步执行的,如果过期失败会在下次 commit 时抛出异常

### 核心概念解释

**Write vs Commit 的分离**:
- **Write**: 负责数据写入,生成数据文件和 CommitMessage
- **Commit**: 负责原子提交,更新 Manifest 和 Snapshot
- 分离原因: 分布式环境下,多个 Writer 并行写入,但只有一个 Coordinator 负责提交

**CommitMessage 的结构**:
```java
CommitMessageImpl {
    partition, bucket, totalBuckets,
    DataIncrement {
        newFiles,        // 新写入的数据文件
        deletedFiles,    // 被删除的数据文件 (Compaction)
        changelogFiles,  // 变更日志文件
        newIndexFiles,   // 新索引文件 (DV/Bloom)
        deletedIndexFiles // 被删除的索引文件
    },
    CompactIncrement {
        compactBefore,   // Compaction 前的文件
        compactAfter,    // Compaction 后的文件
        ...
    }
}
```

**RowKind 的处理**:
- INSERT (+I): 新增记录
- UPDATE_BEFORE (-U): 更新前的记录
- UPDATE_AFTER (+U): 更新后的记录
- DELETE (-D): 删除记录
- 主键表: 所有 RowKind 都支持,转换为 KeyValue 结构
- 追加表: 只支持 INSERT,其他 RowKind 抛异常

**乐观并发控制**:
- 每个 Snapshot 有唯一的 ID (递增)
- Commit 时通过文件系统的原子 rename 操作创建 Snapshot 文件
- 如果 rename 失败 (文件已存在),说明有其他 Writer 先提交了
- 读取冲突的 Snapshot,检查是否有文件冲突
- 如果无冲突,rebase 到新 Snapshot 并重试
- 如果有冲突,抛出 ConflictException

**Compaction 的触发**:
- 写入时: 当 Level-0 文件数量达到阈值,触发异步 Compaction
- 提交时: prepareCommit(waitCompaction=true) 会等待 Compaction 完成
- 独立任务: 可以启动专门的 Compaction 作业 (CompactAction)

**Maintain 的 5 个任务**:
1. Tag 自动创建: 根据时间/commit数自动创建 Tag
2. Tag 过期: 删除超过保留时间的 Tag
3. 分区过期: 删除超过保留策略的分区
4. Consumer 过期: 删除超过保留时间的 Consumer 记录
5. Snapshot 过期: 根据保留策略清理快照

**与 Iceberg 的对比**:
- Iceberg: AppendFiles/OverwriteFiles/ReplaceFiles 等操作类
- Paimon: 统一的 TableWrite + TableCommit 接口
- Paimon 优势: 接口更简单,批和流使用相同的 API

### 设计理念

**为什么选择 Write/Commit 分离而非统一接口**:

**权衡1: 分布式友好 vs 简单性**
- Paimon 选择分布式友好: Write 在 Worker 执行,Commit 在 Coordinator 执行
- 好处: 天然适配 Flink/Spark 的分布式执行模型
- 代价: 用户需要理解两个概念,学习曲线稍陡

**权衡2: 乐观并发 vs 悲观锁**
- Paimon 选择乐观并发: 通过文件系统的原子操作保证一致性
- 好处: 无需分布式锁,性能高,实现简单
- 代价: 冲突时需要重试,可能增加延迟

**架构演进考虑**:
1. **向后兼容**: TableWrite/TableCommit 接口稳定,内部实现可自由演进
2. **扩展性**: 新增写入模式 (如 MERGE INTO) 只需扩展 WriteBuilder
3. **可测试性**: Write 和 Commit 分离,可以独立 Mock 测试

**业界对比**:
- **Iceberg**: 每种操作有独立的类 (AppendFiles/OverwriteFiles),需要调用 commit()
- **Delta Lake**: 通过 OptimisticTransaction 封装,自动处理冲突
- **Paimon**: 显式的 Write/Commit 分离,适配流式场景

**为什么 Paimon 需要显式的 Commit**: 流式场景下,Write 是持续的,但 Commit 是周期性的 (每个 checkpoint)。显式的 Commit 让用户可以控制提交时机,实现 exactly-once 语义。

**为什么 Maintain 任务在 Commit 时执行**: Maintain 任务 (快照过期、分区过期等) 需要在有新数据提交后才有意义。在 Commit 时执行可以确保:
1. 有新快照产生后才触发过期检查
2. 避免额外的后台线程,降低复杂度
3. 可以配置为异步执行,不阻塞写入

**为什么 Consumer 过期在 Snapshot 过期之前**: Consumer 记录了消费到的快照位置。如果先过期快照,Consumer 可能引用已删除的快照,导致消费失败。先清理过期 Consumer,才能让 Snapshot 安全回收。

### 3.3 TableWriteImpl 核心逻辑

### 解决什么问题

**核心业务问题**: 如何将用户的 InternalRow 正确路由到对应的 (partition, bucket),并转换为存储引擎需要的格式?

**没有这个设计的后果**:
- 非空字段校验缺失,导致写入 NULL 值到 NOT NULL 列
- 默认值不生效,导致数据不完整
- RowKind 处理错误,导致 CDC 数据丢失或重复
- 行路由错误,导致数据写入错误的分区或桶

**实际场景**:
```java
// 场景1: CDC 写入 (包含 UPDATE/DELETE)
TableWrite write = table.newStreamWriteBuilder().newWrite();
InternalRow updateBefore = GenericRow.of(1, "old");
updateBefore.setRowKind(RowKind.UPDATE_BEFORE);
write.write(updateBefore);  // -U

InternalRow updateAfter = GenericRow.of(1, "new");
updateAfter.setRowKind(RowKind.UPDATE_AFTER);
write.write(updateAfter);  // +U

// 场景2: 默认值填充
// Schema: id INT NOT NULL, name STRING DEFAULT 'unknown'
InternalRow row = GenericRow.of(1, null);  // name 为 null
write.write(row);  // 自动填充为 'unknown'

// 场景3: 非空校验
// Schema: id INT NOT NULL
InternalRow row = GenericRow.of(null);  // id 为 null
write.write(row);  // 抛出 RuntimeException

// 场景4: RowKind 过滤
// 配置: read.changelog-mode=upsert (只保留 +I/+U/-D)
InternalRow row = GenericRow.of(1, "test");
row.setRowKind(RowKind.UPDATE_BEFORE);  // -U
write.write(row);  // 被过滤,不写入
```

### 有什么坑

**误区1: 假设 NULL 值会自动填充默认值**
```java
// 错误: 以为 NULL 会自动变成默认值
// Schema: name STRING NOT NULL DEFAULT 'unknown'
InternalRow row = GenericRow.of(null);
write.write(row);  // 抛出 RuntimeException (非空校验先于默认值填充)
```
**原因**: checkNullability() 在 wrapDefaultValue() 之前执行 (L183-L184)。

**误区2: 在追加表上写入非 INSERT 的 RowKind**
```java
// 错误: 追加表只支持 INSERT
InternalRow row = GenericRow.of(1, "test");
row.setRowKind(RowKind.DELETE);
write.write(row);  // 抛出 IllegalArgumentException
```
**原因**: AppendOnlyFileStoreTable 的 recordExtractor 会校验 rowKind.isAdd() (L143-L146)。

**误区3: 假设 RowKindFilter 会抛异常**
```java
// 错误: 以为被过滤的行会报错
// 配置: read.changelog-mode=upsert
InternalRow row = GenericRow.of(1, "test");
row.setRowKind(RowKind.UPDATE_BEFORE);  // -U
SinkRecord record = write.writeAndReturn(row, -1);
// 坑: record == null,没有异常
```
**原因**: rowKindFilter 返回 false 时,writeAndReturn() 直接返回 null (L187-L189)。

**误区4: 手动指定 bucket 时传入错误的值**
```java
// 错误: 在 HASH_FIXED 模式下手动指定 bucket
// 表配置: bucket=10, bucket-key=id
InternalRow row = GenericRow.of(1, "test");  // id=1
write.write(row, 5);  // 手动指定 bucket=5
// 坑: 正确的 bucket 应该是 hash(1) % 10,可能不是 5
// 后果: 数据路由错误,查询时找不到
```
**原因**: write(row, bucket) 会跳过 RowKeyExtractor 的 bucket() 计算,直接使用传入的 bucket。

**生产环境注意事项**:
- 在 Schema 演进场景下,确保新增的 NOT NULL 列有默认值,否则旧数据写入会失败
- 使用 rowkind-field 配置时,确保字段值是合法的 RowKind 字符串 (+I/-D/-U/+U)
- 在动态桶模式下,必须使用 write(row, bucket) 并由外部分配 bucket

### 核心概念解释

**writeAndReturn() 的 6 个步骤**:
1. **checkNullability**: 检查 NOT NULL 列是否为 NULL
2. **wrapDefaultValue**: 为 NULL 列填充默认值
3. **getRowKind**: 从 rowkind-field 或 row.getRowKind() 获取 RowKind
4. **rowKindFilter**: 根据 changelog-mode 过滤不需要的 RowKind
5. **toSinkRecord**: 提取 partition/bucket/primaryKey
6. **recordExtractor**: 转换为存储格式 (KeyValue 或 InternalRow)

**SinkRecord 的结构**:
```java
SinkRecord {
    BinaryRow partition;      // 分区值
    int bucket;               // 桶号
    BinaryRow primaryKey;     // 主键值 (trimmed,不包含分区列)
    InternalRow row;          // 完整行数据
}
```

**RowKindGenerator 的作用**:
- 如果配置了 `rowkind-field`,从指定字段读取 RowKind
- 字段值必须是 "+I"/"-D"/"-U"/"+U" 之一
- 如果未配置,使用 row.getRowKind() (来自 CDC 消息)

**RowKindFilter 的配置**:
- `read.changelog-mode=all`: 不过滤
- `read.changelog-mode=upsert`: 过滤掉 UPDATE_BEFORE
- 自定义: 保留指定的 RowKind 组合

**RecordExtractor 的两种实现**:
- **KeyValueRecordExtractor** (主键表): 将 SinkRecord 转为 KeyValue
  ```java
  kv.replace(primaryKey, UNKNOWN_SEQUENCE, rowKind, row)
  ```
- **RowRecordExtractor** (追加表): 直接返回 row,校验 rowKind.isAdd()

**与 Iceberg 的对比**:
- Iceberg: 直接写入 GenericRecord,不支持 RowKind
- Paimon: 支持 RowKind,可以处理 CDC 数据
- Paimon 优势: 原生支持流式更新场景

### 设计理念

**为什么选择 6 步流水线而非一次性处理**:

**权衡1: 职责分离 vs 性能**
- Paimon 选择职责分离: 每步只做一件事,易于理解和维护
- 代价: 多次方法调用,可能有轻微性能损失
- 优化: 关键路径 (toSinkRecord、recordExtractor) 使用 CodeGen 生成高效代码

**权衡2: 校验时机**
- Paimon 选择早期校验: checkNullability 在第一步执行
- 好处: 尽早发现错误,避免无效计算
- 代价: 每行都要检查,增加 CPU 开销

**架构演进考虑**:
1. **向后兼容**: 6 步流水线是内部实现,可以调整顺序或增加步骤
2. **扩展性**: 新增处理逻辑 (如数据脱敏) 只需插入新步骤
3. **可测试性**: 每步可以独立测试

**业界对比**:
- **Flink Table API**: 在 Sink 算子中处理 RowKind
- **Spark DataSource V2**: 不支持 RowKind,只支持 Append/Overwrite
- **Paimon**: 在 TableWrite 层统一处理,适配多种引擎

**为什么 checkNullability 在 wrapDefaultValue 之前**: 如果先填充默认值,非空校验就失去意义了 (所有列都有值)。先校验可以确保用户显式提供了必需的字段。

**为什么需要 RowKindFilter**: CDC 场景下,某些下游系统只需要最终状态 (upsert),不需要中间的 UPDATE_BEFORE。RowKindFilter 可以在写入时就过滤掉不需要的记录,减少存储和计算开销。

**为什么 recordExtractor 是最后一步**: 转换为存储格式 (KeyValue) 是最重的操作,涉及内存分配和序列化。放在最后可以确保前面的过滤和校验都通过了,避免无效转换。

源码路径: `paimon-core/src/main/java/org/apache/paimon/table/sink/TableWriteImpl.java`

**writeAndReturn() 的 6 个步骤** (L183-L193):

```java
public SinkRecord writeAndReturn(InternalRow row, int bucket) throws Exception {
    // Step 1: 非空字段检查
    checkNullability(row);
    // 遍历 notNullFieldIndex，如果 row.isNullAt(idx) 则抛出异常 (L195-L203)

    // Step 2: 填充默认值
    row = wrapDefaultValue(row);
    // 如果 schema 定义了 default value，用 DefaultValueRow 包装 (L205-L207)

    // Step 3: 确定行类型 (RowKind)
    RowKind rowKind = RowKindGenerator.getRowKind(rowKindGenerator, row);
    // 如果配置了 rowkind-field，从指定字段读取 (+I/-D/-U/+U)
    // 否则使用 row.getRowKind() (来自 CDC 消息)

    // Step 4: RowKind 过滤
    if (rowKindFilter != null && !rowKindFilter.test(rowKind)) {
        return null;  // 被过滤的行直接丢弃 (L187-L189)
    }

    // Step 5: 提取 partition/bucket/primaryKey
    SinkRecord record = bucket == -1 ? toSinkRecord(row) : toSinkRecord(row, bucket);
    // toSinkRecord:
    //   keyAndBucketExtractor.setRecord(row)
    //   new SinkRecord(partition(), bucket(), trimmedPrimaryKey(), row)

    // Step 6: 转换为存储格式并写入
    write.write(record.partition(), record.bucket(), recordExtractor.extract(record, rowKind));
    // 主键表: recordExtractor 将 SinkRecord 转为 KeyValue
    //         kv.replace(primaryKey, UNKNOWN_SEQUENCE, rowKind, row)
    // 追加表: recordExtractor 直接返回 row (校验 rowKind.isAdd())

    return record;
}
```

**RowKindFilter 的作用**:

```
配置 'read.changelog-mode' 控制:
  - all: 不过滤
  - upsert: 过滤掉 UPDATE_BEFORE
  - 自定义: 保留指定的 RowKind 组合
```

### 3.4 RowKeyExtractor 行路由

源码路径: `paimon-core/src/main/java/org/apache/paimon/table/sink/RowKeyExtractor.java`

RowKeyExtractor 抽象类提供 `partition()` 和 `trimmedPrimaryKey()` 的通用实现，`bucket()` 由子类实现:

**5 种 RowKeyExtractor 实现**:

| BucketMode | 实现类 | bucket() 逻辑 | 适用场景 |

| HASH_DYNAMIC | DynamicBucketRowKeyExtractor | **抛出异常** -- bucket 由外部分配 | 动态桶模式，由 Assigner 分配 |
| KEY_DYNAMIC | DynamicBucketRowKeyExtractor | **抛出异常** -- bucket 由外部分配 | 跨分区更新模式 |

        // 优化: 如果 bucketKeys == trimmedPrimaryKeys，复用 trimmedPrimaryKey()

    return BucketFunction.bucket(bucketKey, numBuckets)

**为什么 DynamicBucket 的 bucket() 抛异常**: 动态桶模式下，bucket 的分配需要全局索引 (查看该 key 之前在哪个 bucket)。这个逻辑在 Flink 的 BucketAssigner 中实现，不在 RowKeyExtractor 中。调用者必须使用 `write(row, bucket)` 方法显式传入 bucket。

### 3.7 TableCommitImpl 提交与维护

### 解决什么问题

**核心业务问题**: 如何在分布式环境下保证提交的原子性,同时自动维护表的生命周期 (快照过期、分区过期、Tag 管理等)?

**没有这个设计的后果**:
- 多个 Writer 并发提交时可能产生数据冲突和丢失
- 快照和文件无法自动过期,存储空间持续增长
- Tag 和分区无法自动管理,需要手动维护
- Consumer 进度无法自动清理,阻止快照回收

**实际场景**:
```java
// 场景1: 批式提交 (一次性提交所有数据)
TableCommit commit = table.newBatchWriteBuilder().newCommit();
List<CommitMessage> messages = write.prepareCommit(true, 0);
commit.commit(messages);  // 原子提交 + 同步维护
commit.close();

// 场景2: 流式提交 (每个 checkpoint 提交一次)
TableCommit commit = table.newStreamWriteBuilder()
    .withCommitUser("flink-job-1")
    .newCommit();
for (long checkpointId = 1; ; checkpointId++) {
    // ... 写入数据 ...
    List<CommitMessage> messages = write.prepareCommit(false, checkpointId);
    commit.commit(messages);  // 原子提交 + 异步维护
}

// 场景3: 覆写分区
TableCommit commit = table.newStreamWriteBuilder()
    .withOverwrite(partitionSpec)
    .newCommit();
commit.commit(messages);  // 原子替换分区数据

// 场景4: 手动触发维护
commit.commit(Collections.emptyList());  // 空提交,只触发维护任务
```

### 有什么坑

**误区1: 假设 commit 总是成功**
```java
// 错误: 不处理冲突异常
try {
    commit.commit(messages);
} catch (Exception e) {
    // 忽略异常
}
// 坑: 如果是 ConflictException,数据没有提交成功
```
**后果**: 数据丢失,且无法感知。

**误区2: 在 overwrite 模式下多次 commit**
```java
// 错误: overwrite 模式不支持多次提交
TableCommit commit = table.newStreamWriteBuilder()
    .withOverwrite(partition)
    .newCommit();
commit.commit(messages1);  // 第一次 OK
commit.commit(messages2);  // 抛出 IllegalArgumentException
```
**原因**: commitMultiple() 会检查 overwrite 模式下 committables.size() > 1 (L251-L254)。

**误区3: 假设维护任务总是同步执行**
```java
// 错误: 以为 commit 返回后快照已过期
commit.commit(messages);
// 立即查询快照列表
List<Snapshot> snapshots = snapshotManager.snapshots();
// 坑: 如果是异步维护,快照可能还没过期
```
**原因**: 流式模式下维护任务默认异步执行 (L121-L123)。

**误区4: 忽略维护任务的异常**
```java
// 错误: 不检查 maintainError
commit.commit(messages1);  // 维护任务异步执行,出错了
commit.commit(messages2);  // 这里才抛出上次的维护异常
```
**原因**: 异步维护的异常会在下次 commit 时检查并抛出 (L361-L362)。

**生产环境注意事项**:
- 在高并发写入场景下,冲突重试可能导致延迟增加,需要调整 `commit.retry.max-attempts`
- 快照过期是耗时操作 (需要删除大量文件),建议配置为异步执行
- Consumer 过期必须在 Snapshot 过期之前,否则可能阻止快照回收
- 在 S3 等对象存储上,文件删除操作可能很慢,需要增加维护任务的超时时间

### 核心概念解释

**commitMultiple() 的两种模式**:
- **非 overwrite 模式**: 调用 FileStoreCommit.commit(),支持多次提交
- **overwrite 模式**: 调用 FileStoreCommit.overwritePartition(),只能提交一次

**maintain() 的 5 个任务**:
1. **Tag 自动创建**: 根据 `tag.automatic-creation` 配置,按时间/commit数自动创建 Tag
2. **Tag 过期**: 根据 `tag.time-retained` 删除超过保留时间的 Tag
3. **分区过期**: 根据 `partition.expiration-time` 删除超过保留策略的分区
4. **Consumer 过期**: 根据 `consumer.expiration-time` 删除超过保留时间的 Consumer 记录
5. **Snapshot 过期**: 根据 `snapshot.num-retained-max/min` 和 `snapshot.time-retained` 清理快照

**执行模式**:
- **SYNC 模式**: 在当前线程同步执行,commit 返回后维护任务已完成
- **ASYNC 模式**: 在后台线程异步执行,commit 立即返回,维护任务在后台进行
- 批式: 默认 SYNC (L119-L120)
- 流式: 默认 ASYNC (L121-L123)

**为什么 Consumer 过期在 Snapshot 过期之前**:
- Consumer 记录了消费到的快照位置
- 如果先过期快照,Consumer 可能引用已删除的快照
- 先清理过期 Consumer,才能让 Snapshot 安全回收

**CommitCallback 的作用**:
- 在 commit 成功后触发回调
- 用于通知外部系统 (如 Metastore) 更新元数据
- 例如: Hive Metastore 需要在 commit 后更新分区信息

**与 Iceberg 的对比**:
- Iceberg: 通过 TableOperations.commit() 提交,维护任务需要手动触发
- Paimon: commit 时自动触发维护任务,无需手动管理
- Paimon 优势: 自动化程度高,减少运维负担

### 设计理念

**为什么选择在 commit 时触发维护任务**:

**权衡1: 自动化 vs 可控性**
- Paimon 选择自动化: commit 时自动触发维护,无需额外的后台任务
- 好处: 减少运维负担,避免忘记清理导致存储爆炸
- 代价: commit 延迟可能增加 (如果是同步维护)

**权衡2: 同步 vs 异步**
- Paimon 选择根据场景区分: 批式同步,流式异步
- 批式同步: 只提交一次,同步执行可以确保维护完成
- 流式异步: 频繁提交,异步执行避免阻塞写入
- 代价: 异步模式下错误发现较晚

**架构演进考虑**:
1. **向后兼容**: maintain() 的任务列表可以扩展,不影响现有代码
2. **扩展性**: 新增维护任务只需在 maintain() 中添加
3. **可测试性**: 维护任务可以独立测试

**业界对比**:
- **Iceberg**: 通过 ExpireSnapshots/RemoveOrphanFiles 等 Action 手动触发
- **Delta Lake**: 通过 VACUUM 命令手动触发
- **Paimon**: 自动触发,也支持手动触发 (ExpireSnapshotsAction)

**为什么 Paimon 选择自动触发**: 流式场景下,表是持续写入的,如果不自动清理,快照和文件会无限增长。自动触发可以确保表的存储空间可控,无需人工干预。

**为什么维护任务的顺序很重要**:
1. **Tag 自动创建在最前**: 新创建的 Tag 会引用最新快照,必须在快照过期之前创建
2. **Consumer 过期在 Snapshot 过期之前**: Consumer 引用快照,必须先清理 Consumer
3. **Snapshot 过期在最后**: 最耗时的操作放在最后,避免阻塞其他任务

**为什么批式和流式的执行模式不同**:
- 批式: 只提交一次,同步执行可以确保作业结束时维护完成,存储空间立即释放
- 流式: 频繁提交 (每个 checkpoint),同步执行会显著增加 checkpoint 延迟,影响吞吐

**为什么需要 expireForEmptyCommit**: 即使没有新数据提交,维护任务也可能需要执行 (如定时过期快照)。expireForEmptyCommit 允许空提交触发维护。

### 3.8 乐观并发控制机制

### 解决什么问题

**核心业务问题**: 如何在分布式环境下,多个 Writer 并发写入时保证数据一致性,同时避免分布式锁的复杂度和性能开销?

**没有这个设计的后果**:
- 多个 Writer 同时提交可能产生数据覆盖和丢失
- 需要引入分布式锁 (如 ZooKeeper),增加系统复杂度和延迟
- 无法利用文件系统的原子操作,性能低下
- 冲突检测不准确,可能导致数据不一致

**实际场景**:
```java
// 场景1: 两个 Flink 作业并发写入不同分区 (无冲突)
// Job 1: 写入 partition=2024-01-01
// Job 2: 写入 partition=2024-01-02
// 结果: 两者都成功提交,各自创建快照

// 场景2: 两个作业并发写入相同分区 (有冲突)
// Job 1: 写入 partition=2024-01-01, bucket=0
// Job 2: 写入 partition=2024-01-01, bucket=0
// 结果: 一个成功,另一个检测到冲突并重试

// 场景3: Compaction 与写入并发 (可能冲突)
// Writer: 写入新数据到 bucket=0
// Compactor: 合并 bucket=0 的旧文件
// 结果: 如果 Compactor 先提交,Writer 检测到文件已删除,rebase 后重试

// 场景4: 追加表的并发写入 (默认无冲突)
// 配置: write.append-commit-check-conflict=false
// 多个 Writer 并发写入追加表
// 结果: 都成功,不做冲突检测 (性能最优)
```

### 有什么坑

**误区1: 假设冲突总是导致失败**
```java
// 错误: 以为冲突会抛异常
try {
    commit.commit(messages);
} catch (ConflictException e) {
    // 处理冲突
}
// 坑: 大多数冲突会自动 rebase 并重试,不会抛异常
```
**原因**: tryCommit() 会循环重试,只有达到最大重试次数或无法 rebase 时才抛异常。

**误区2: 在追加表上期望严格的冲突检测**
```java
// 错误: 以为追加表会检测所有冲突
// 配置: write.append-commit-check-conflict=false (默认)
// 两个 Writer 并发写入相同 bucket
// 坑: 不会检测冲突,都成功提交
```
**原因**: 追加表默认不检测冲突,因为追加操作通常是安全的 (无 key 重叠)。

**误区3: 假设 rebase 总是成功**
```java
// 错误: 不处理 rebase 失败
commit.commit(messages);
// 坑: 如果 rebase 失败 (如文件已被删除),会抛出 ConflictException
```
**原因**: rebase 需要检查 compactBefore 中的文件是否仍存在,如果不存在说明有冲突。

**误区4: 多个 Writer 使用相同的 commitUser**
```java
// 错误: 两个作业使用相同的 commitUser
// Job 1: commitUser="my-job"
// Job 2: commitUser="my-job"
// 坑: 冲突检测可能失效,因为被认为是同一个 Writer 的重试
```
**后果**: 可能导致数据覆盖和丢失。

**生产环境注意事项**:
- 在高并发写入场景下,增加 `commit.retry.max-attempts` (默认 3)
- 在追加表上,如果需要严格的冲突检测,设置 `write.append-commit-check-conflict=true`
- 在 S3 等对象存储上,rename 操作可能不是原子的,需要使用 S3 的条件写入
- 冲突重试会增加延迟,需要监控 commit 延迟指标

### 核心概念解释

**乐观并发控制的原理**:
1. **读取**: 读取当前最新快照 ID
2. **执行**: 在本地生成新的 Manifest 和 Snapshot 文件
3. **提交**: 通过文件系统的原子 rename 操作创建 Snapshot 文件
4. **冲突检测**: 如果 rename 失败 (文件已存在),说明有其他 Writer 先提交了
5. **Rebase**: 读取冲突的 Snapshot,检查是否有文件冲突
6. **重试**: 如果无冲突,rebase 到新 Snapshot 并重试;如果有冲突,抛出异常

**原子性保证**:
- 文件系统的 rename 操作是原子的 (POSIX 语义)
- Snapshot 文件名包含递增的 ID: `snapshot-{id}`
- 只有一个 Writer 能成功创建 `snapshot-{id}` 文件
- 其他 Writer 会收到 FileAlreadyExistsException

**冲突检测的三种类型**:
1. **APPEND 冲突**: 新写入文件与冲突快照的文件有 key 范围重叠
2. **COMPACT 冲突**: compactBefore 中要删除的文件已被其他 Compaction 删除
3. **OVERWRITE 冲突**: 不做冲突检测,overwrite 总是成功

**Rebase 的逻辑**:
```
当前 Writer 的 Manifest:
  - newFiles: [file1, file2]
  - compactBefore: [file3, file4]
  - compactAfter: [file5]

冲突 Snapshot 的 Manifest:
  - newFiles: [file6, file7]
  - compactBefore: [file8]
  - compactAfter: [file9]

Rebase 检查:
  1. file1/file2 与 file6/file7 是否有 key 范围重叠? (APPEND 冲突)
  2. file3/file4 是否在冲突 Snapshot 中被删除? (COMPACT 冲突)
  3. 如果都无冲突,将当前 Writer 的 Manifest 合并到冲突 Snapshot 之后
```

**与 Iceberg 的对比**:
- Iceberg: 使用 TableOperations.commit() 的 CAS 操作,基于 metadata.json 的版本号
- Paimon: 使用文件系统的原子 rename,基于 Snapshot 文件名的递增 ID
- Paimon 优势: 不依赖外部存储 (如 Hive Metastore),纯文件系统实现

### 设计理念

**为什么选择乐观并发而非悲观锁**:

**权衡1: 性能 vs 一致性**
- Paimon 选择乐观并发: 大多数场景下不同 Writer 写不同的 (partition, bucket),冲突概率极低
- 好处: 无需分布式锁,性能高,实现简单
- 代价: 冲突时需要重试,可能增加延迟

**权衡2: 原子性保证**
- Paimon 选择文件系统的原子操作: rename 是 POSIX 标准的原子操作
- 好处: 不依赖外部系统 (如 ZooKeeper),降低复杂度
- 代价: 在某些对象存储 (如 S3) 上,rename 可能不是原子的,需要特殊处理

**架构演进考虑**:
1. **向后兼容**: 冲突检测逻辑可以调整,不影响 Snapshot 格式
2. **扩展性**: 新增冲突类型只需在 conflictCheck() 中添加
3. **可测试性**: 冲突检测可以独立测试

**业界对比**:
- **Iceberg**: 乐观并发,基于 metadata.json 的版本号
- **Delta Lake**: 乐观并发,基于 _delta_log 的事务日志
- **Hudi**: 乐观并发,基于 Timeline 的 instant 时间戳

**为什么 Paimon 的设计更简单**: Paimon 直接使用 Snapshot 文件名的递增 ID,不需要额外的版本号或事务日志。文件系统的 rename 操作天然提供了原子性保证。

**为什么追加表默认不检测冲突**: 追加表没有主键,不同 Writer 写入的数据不会有 key 重叠。即使并发写入相同 bucket,也只是文件数量增加,不会导致数据不一致。跳过冲突检测可以显著提升性能。

**为什么 COMPACT 冲突需要特殊处理**: Compaction 会删除旧文件并生成新文件。如果两个 Compaction 并发执行,可能都尝试删除相同的文件。通过检查 compactBefore 中的文件是否仍存在,可以检测到这种冲突。

**为什么 rebase 可以自动解决大多数冲突**: 在分布式写入场景下,不同 Writer 通常写入不同的 (partition, bucket)。即使快照 ID 冲突,文件级别通常无冲突。rebase 可以将当前 Writer 的 Manifest 合并到冲突 Snapshot 之后,无需重新写入数据。

**为什么需要 commitUser**: commitUser 标识了 Writer 的身份。在重试场景下,如果 commitUser 相同,可以认为是同一个 Writer 的重试,跳过某些检查 (如幂等性检查)。如果 commitUser 不同,说明是不同的 Writer,需要严格的冲突检测。

## 6. 主键表 vs 追加表全面对比

| 维度 | 主键表 (PrimaryKeyFileStoreTable) | 追加表 (AppendOnlyFileStoreTable) |
|------|----------------------------------|-----------------------------------|
| **存储引擎** | KeyValueFileStore (LSM-tree) | AppendOnlyFileStore |
| **数据模型** | KeyValue (key + seq + kind + value) | InternalRow (纯行数据) |
| **主键** | 必须定义 | 无 |
| **写入类型** | INSERT, UPDATE, DELETE | 仅 INSERT |
| **Merge** | 需要 MergeFunction 合并同 key | 无 merge |
| **Split 生成** | MergeTreeSplitGenerator (IntervalPartition) | AppendOnlySplitGenerator (简单装箱) |
| **rawConvertible** | 有条件 (DV/FIRST_ROW/单level) | 始终 true |
| **Read Provider 数** | 4 个 (Raw/Merge/IncrChangelog/IncrDiff) | 1 个 (根据配置: Raw 或 DataEvolution) |
| **谓词下推** | 仅 key 列 (保守策略) | 全部列 (无限制) |
| **非分区过滤** | pickTransformFieldMapping -> key 列过滤 | withFilter -> 全字段过滤 |
| **Compaction** | 必须 (LSM-tree 语义) | 可选 |
| **Deletion Vectors** | 支持 (显著提升读性能) | 不适用 |
| **Bucket 模式** | HASH_FIXED / HASH_DYNAMIC / KEY_DYNAMIC / POSTPONE | HASH_FIXED / BUCKET_UNAWARE |
| **RowKeyExtractor** | Fixed/Dynamic/Postpone | Fixed/AppendTable |
| **MergeEngine** | deduplicate/partial-update/aggregation/first-row | 无 |
| **Changelog** | INPUT/LOOKUP/FULL_COMPACTION | 无 (追加表只有 INSERT) |
| **LocalTableQuery** | 支持 (点查加速) | 不支持 (抛异常) |
| **streamingReadOverwrite** | `streaming-read-overwrite` 控制 | `streaming-read-append-overwrite` 控制 |
| **batch-scan-skip-level0** | 支持 (跳过 L0，只读已 compacted) | 不适用 |
| **写入时格式转换** | InternalRow -> KeyValue (L161-L172) | InternalRow 直接写入 (L143-L146) |
| **文件层级** | 多层 (Level 0-N) | 通常单层 |
| **适用场景** | 实时更新、CDC、维表关联 | 日志、事件、ETL 输出 |

## 7. FollowUpScanner 流式增量扫描

在流式读取中，StartingScanner 确定起始位置后，由 FollowUpScanner 持续扫描新快照。

**工作模型**:

```
  StartingScanner                FollowUpScanner
  (确定起始位置)                (持续消费增量)
       |                              |
       v                              v
  scan() -> NextSnapshot(N)     for snapshot = N, N+1, N+2, ...:
       |                          followUp.scan(snapshot)
       |                          -> 读取 delta/changelog manifest
       v                          -> 生成增量 Split
  [一次性]                       [持续循环]
```

| FollowUpScanner | 读取内容 | 适用场景 |
|-----------------|---------|---------|
| DeltaFollowUpScanner | 读取 deltaManifestList (只看 APPEND commit) | 默认的流式增量消费 |
| AllDeltaFollowUpScanner | 读取所有 delta (包括 COMPACT) | 需要感知 Compaction 的场景 |
| ChangelogFollowUpScanner | 读取 changelogManifestList | changelog-producer 非 NONE 时 |

**与 Iceberg 增量读取对比**:
- Paimon: 通过 Snapshot 的 deltaManifestList 直接获取增量，O(1) 定位
- Iceberg: IncrementalAppendScan 需扫描指定范围内所有快照的 manifest
- Paimon 优势: delta manifest 将增量信息独立存储，不需要和 base 做 diff

**为什么 Paimon 能做到 O(1) 定位**: Paimon 的 Snapshot 同时记录了 baseManifestList (全量) 和 deltaManifestList (增量)。流式消费只需读 delta 部分，不需要解析全量 manifest 做差集运算。

## 8. 关键设计决策总结

### 8.1 为什么 Table 需要 5 层继承?

**决策**: Table -> InnerTable -> DataTable -> FileStoreTable -> AbstractFileStoreTable

**原因**: Paimon 的表体系需要同时支持:
1. 用户稳定 API (Table)
2. 系统表和数据表的共享逻辑 (InnerTable)
3. 数据表的管理能力 (DataTable, 快照/标签/分支)
4. 存储引擎桥接 (FileStoreTable)
5. 主键表与追加表的差异化 (子类 + 模板方法)

每层有明确的职责边界，新增表类型只需在对应层扩展。

### 8.2 为什么读取路径分离 Scan 和 Read?

**决策**: ReadBuilder 分别创建 TableScan (计划) 和 TableRead (执行)

**原因**:
1. **分布式计算**: Scan 在 Driver/Coordinator 执行，生成 Split 列表；Read 在各 Worker/TaskManager 上并行执行
2. **序列化需求**: Split 需要跨节点传输，必须 Serializable
3. **灵活调度**: 引擎可以根据 Split 的 rowCount/fileSize 做负载均衡

### 8.3 为什么主键表只对 key 列做谓词下推?

**决策**: PrimaryKeyFileStoreTable 的 nonPartitionFilterConsumer 只提取 key 列谓词

**原因**: LSM-tree 中同一个 key 的不同版本分布在不同层的不同文件中。对 value 列做文件级过滤可能导致某些版本被跳过，merge 结果不正确。key 列过滤是安全的——如果 key 不在范围内，该 key 的所有版本都不在范围内。

**好处**: 保证查询结果的正确性，这是数据一致性的底线。

### 8.4 为什么 Compaction/Expire 异步执行?

**决策**: TableCommitImpl 的 maintain() 可配置为 ASYNC 模式

**原因**: Snapshot 过期和 Compaction 涉及大量文件操作，如果同步执行会阻塞写入。异步执行让写入延迟不受过期操作影响。

**好处**: 写入延迟稳定，过期操作不影响吞吐。

**代价**: 异步错误需要在下次 commit 时检测 (`maintainError.get()`)。

### 8.5 为什么 Consumer 过期在 Snapshot 过期之前?

**决策**: maintain() 中 Task 4 (Consumer) 先于 Task 5 (Snapshot)

**原因**: Consumer 记录了消费到的快照位置。如果先过期快照，Consumer 可能引用已删除的快照。先清理过期 Consumer，才能让 Snapshot 安全回收。

### 8.6 为什么 rawConvertible 是性能关键?

**决策**: SplitGenerator 为每个 Split 标记 rawConvertible

**原因**: rawConvertible=true 时可跳过整个 merge 层，直接使用列式格式的向量化读取。这避免了:
1. LoserTree/MinHeap 的 O(N log K) 比较开销
2. MergeFunction 的逐行合并计算
3. KeyValue 到 InternalRow 的格式转换

**好处**: 在 DV 模式下，主键表的批查询性能接近纯追加表水平，与 Iceberg 相当。

### 8.7 为什么 StartingScanner 有 16+ 种策略?

**决策**: 每种时间旅行/增量读取模式对应一个 StartingScanner

**原因**: 不同的业务场景需要不同的起始位置:
- 全量分析: FullStartingScanner
- 流式 CDC: ContinuousFromSnapshotStartingScanner
- 增量 ETL: IncrementalDeltaStartingScanner
- 一致性读: CompactedStartingScanner
- 时间旅行: StaticFromTimestampStartingScanner

策略模式让每种场景的逻辑独立封装，避免巨大的 switch-case。

### 8.8 为什么需要 IncrementalSplit 和 DataSplit 两种 Split?

**决策**: 全量/批式读取使用 DataSplit，增量/差异读取使用 IncrementalSplit

**原因**:
- DataSplit: 包含一组同 (partition, bucket) 的文件，可能需要 merge 或直接读取
- IncrementalSplit: 包含 before 和 after 两组文件，用于计算变更 (changelog)

两种 Split 对应不同的 SplitReadProvider，读取逻辑完全不同。统一为一种 Split 会增加条件判断的复杂度。
