# Apache Paimon 表读写路径深度分析

> 版本: Paimon 1.5-SNAPSHOT　|　commit: e76fc41b7
> 源码模块: `paimon-core`（table / table.source / table.sink / table.system / operation / mergetree）、`paimon-api`（Table/Snapshot 接口）
> 分析日期: 2026/06/03

## 一句话定位

表读写路径是 Paimon 把"用户一行 SQL / 一条 `InternalRow`"翻译成"扫描哪些 manifest、归并哪些数据文件、写到哪个 bucket、提交成哪个 snapshot"的中枢。它由两条主干构成：**读路径** `ReadBuilder → TableScan(计划) → TableRead(执行)`，在 Driver 端裁剪 manifest/分区/文件、生成可序列化的 `Split`，在 Worker 端按 `rawConvertible` 决定走"直读"还是"LSM 归并"；**写路径** `WriteBuilder → TableWrite(写文件) → TableCommit(原子提交+维护)`，在 Worker 端缓冲排序、异步 compaction，在 Coordinator 端用文件系统原子 rename 做乐观并发控制。理解这两条主干，等于掌握了 Paimon 性能（读写放大）与一致性（冲突检测）的全部命门。本文逐机制给出源码、算法、边界与排障路径——读完应能改源码、能定位线上问题。

## 1. Table 接口层次与表类型分派

### ① 要解决的问题
在 `@Public` 用户 API 稳定不变的前提下，同时承载两套语义完全不同的存储引擎（主键表 LSM、追加表 Append-only），并让系统表（`$snapshots` 等）复用数据表的读取脚手架，又不被迫实现快照/标签/分支管理。若把差异暴露给用户，则升级即破坏兼容、学习成本陡增、系统表与数据表代码大量重复。

### ② 设计原理与取舍
Paimon 采用 **5 层接口继承 + 末端两个子类 + 模板方法**，而非 Iceberg 式"单接口 + 运行时 format-version 判断"：

```
Table(@Public, table/Table.java:52)        公共API：元数据 + 管理操作 + 读写Builder入口
  └─ InnerTable(InnerTable.java:41)         内部入口：newScan/newRead/newWrite/newCommit + 各 Builder 默认实现
       └─ DataTable(DataTable.java:33)      数据表管理：newSnapshotReader/snapshotManager/tagManager/branchManager
            └─ FileStoreTable(FileStoreTable.java:56)  桥接存储：store()→FileStore<?>，缓存注入，精确写入/提交类型
                 └─ AbstractFileStoreTable(:91)        骨架：copy分派、RowKeyExtractor选择、模板方法占位
                      ├─ AppendOnlyFileStoreTable(:54) store→AppendOnlyFileStore
                      └─ PrimaryKeyFileStoreTable(:53) store→KeyValueFileStore
```

分层取舍的核心是**接口隔离原则（ISP）**：系统表只实现到 `InnerTable`（要读、不要快照管理），数据表才实现到 `DataTable`。`FileStoreTable` 这一层独立存在的理由是暴露 `store()`——把"表"与"存储引擎 FileStore"解耦，从而让缓存（`setManifestCache/setSnapshotCache/setStatsCache/setDVMetaCache`）由 `CachingCatalog` 统一注入，而不是写死进构造函数。这是 Paimon 缓存能在 Catalog 层集中管理的前提。

末端两个子类把全部差异收敛到两个模板方法：

```java
// AbstractFileStoreTable.java:318-320
protected abstract SplitGenerator splitGenerator();               // PK→MergeTreeSplitGenerator / Append→AppendOnlySplitGenerator
protected abstract BiConsumer<FileStoreScan, Predicate> nonPartitionFilterConsumer();  // 谓词下推策略差异
```

`InnerTable` 还为所有 Builder 提供默认实现（`InnerTable.java:56-78`）：`newReadBuilder→ReadBuilderImpl(this)`、`newBatchWriteBuilder→BatchWriteBuilderImpl(this)`、`newStreamWriteBuilder→StreamWriteBuilderImpl(this)`。系统表正是靠这些默认实现"白嫖"到完整读取链路。

### ③ 关键源码
表类型在 `copy(TableSchema)` 时**重新判断**，这是"Schema 演进可改变表类型"的根因：

```java
// AbstractFileStoreTable.java:397-403
public FileStoreTable copy(TableSchema newTableSchema) {
    return newTableSchema.primaryKeys().isEmpty()
        ? new AppendOnlyFileStoreTable(fileIO, path, newTableSchema, catalogEnvironment)
        : new PrimaryKeyFileStoreTable(fileIO, path, newTableSchema, catalogEnvironment);
}
```

`newSnapshotReader()`（`AbstractFileStoreTable.java:275-288`）是所有读取的工厂，把 `store().newScan()`、`splitGenerator()`、`nonPartitionFilterConsumer()`、`store().newIndexFileHandler()`、`dvmetaCache` 一次性注入 `SnapshotReaderImpl`；`newScan()`（`:291-303`）在开启 data-evolution 时用 `DataEvolutionBatchScan` 装饰。

### ④ 风险/陷阱/边界
- **强转具体类是 bug 温床**：`(PrimaryKeyFileStoreTable) table` 遇到系统表、`FallbackReadFileStoreTable`、`DelegatedFileStoreTable` 直接 `ClassCastException`。应转到接口层 `FileStoreTable` 再 `store()`。
- **`copy()` 后表类型可能变**：`copyWithLatestSchema()` 若最新 Schema 去掉主键，返回 `AppendOnlyFileStoreTable`，缓存的旧 `store()` 引用失效；动态 options 改桶数也会重建 store。
- **系统表不是 `DataTable` 全集**：多数系统表只实现 `ReadonlyTable + InnerTable`，`newWrite/newCommit` 抛 `UnsupportedOperationException`。

### ⑤ 收益与代价
收益：编译期消除海量 `if(isPrimaryKey)` 分支，新增表类型（Object/Lance/Iceberg 兼容表）只需挂到对应层；缓存可被 Catalog 统一管理。代价：继承链深，定位"某行为在哪一层实现"须先理解模板方法，初学成本高。

## 1.5 底层数据结构基线：读写路径操作的物理对象

读写路径上的 Scan/Split/Commit 全部围绕同一套**磁盘元数据结构**运转，先建立这层基线，后续机制才不悬空（对照 docs/concepts/spec 的 snapshot/manifest/datafile，但深入到读写如何消费它们）。

**目录布局**（datafile spec）：
```
table/
├── dt=20240514/bucket-0/data-xxx.orc     数据文件（按 partition→bucket 目录组织）
├── manifest/
│   ├── manifest-list-xxx-0                ManifestList（base，引用全量 manifest）
│   ├── manifest-list-xxx-1                ManifestList（delta，引用本次增量 manifest）
│   └── manifest-xxx-0                     Manifest（记录若干 DataFile 的 ADD/DELETE entry）
├── index/                                 DV / 文件索引
├── schema/schema-0                        Schema 演进历史
└── snapshot/{EARLIEST,LATEST,snapshot-1}  Snapshot 文件 + 指针
```

**Snapshot**（snapshot spec）：读路径的入口对象，关键字段被各机制消费：
- `baseManifestList` / `deltaManifestList`：§6 `read()` 读 base+delta，§5 流式 FollowUpScanner 只读 delta（O(1) 增量）。
- `changelogManifestList`：§10 ChangelogFollowUpScanner 消费，无 changelog 时为 null。
- `indexManifest`：§18 `scanDvIndex` 读它定位 DV/文件索引。
- `commitKind`（append/compact/overwrite/analyze）：§5 CompactedStartingScanner 据此找 COMPACT 快照、§19 `$snapshots` 暴露。
- `commitIdentifier`：流式事务 id，§16 幂等提交与 partitionExpire 据此判断。
- `totalRecordCount` / `deltaRecordCount`：行数估算与监控。

**ManifestList**（manifest spec）：avro 文件，每条记录一个 manifest 的 `_FILE_NAME/_FILE_SIZE/_NUM_ADDED_FILES/_NUM_DELETED_FILES/_PARTITION_STATS/_SCHEMA_ID`。`_PARTITION_STATS`（SimpleStats）是 §6.5 **第一层裁剪**的依据——按 manifest 级 partition min/max 跳过整个 manifest 文件。

**Manifest**（avro）：每条 entry = `_KIND(ADD/DELETE) + _PARTITION + _BUCKET + _TOTAL_BUCKETS + _FILE(DataFileMeta)`。关键设计：**同一文件可被多次 ADD/DELETE，读时取最后版本**（LSM compaction 删旧文件靠追加 DELETE entry 而非物理改写，使 commit 轻量）。`_TOTAL_BUCKETS` 用于桶数变更后的校验，对应 §7 `DataSplit.totalBuckets` 可空。

**DataFileMeta**：读写路径最频繁触碰的对象，字段直接驱动各优化：
| 字段 | 被谁消费 |
|---|---|
| `rowCount` / `deleteRowCount` | §7 mergedRowCount、§8 withoutDeleteRow、§9 raw 判定 |
| `minKey` / `maxKey` / `keyStats` | §8 IntervalPartition、§11 key 列文件级过滤 |
| `valueStats` | §11 value 谓词 bucket 级裁剪、§12 MIN/MAX 下推 |
| `minSequenceNumber` / `maxSequenceNumber` | §8 追加表按序号排序、merge 时版本定序 |
| `level` | §9 raw 判定（level>0）、`$read_optimized` 取最高 level |
| `firstRowId` | §7 dataEvolution 行数估算、row-tracking |

**为什么读路径不能绕过这层**：Scan 本质是"解析 Snapshot→ManifestList→Manifest→DataFileMeta 四级，在每级用统计裁剪"，Split 则是 DataFileMeta 的分组打包。理解这四级的字段与裁剪点，等于掌握了读路径所有优化的物理基础。

## 2. 读路径总览：三阶段为何分离

### ① 要解决的问题
分布式引擎（Flink/Spark）要求"中心节点生成执行计划、各节点并行执行"。计划与执行不分离则无法序列化分发"读哪些文件"的决策，也无法做计划期的 Limit/TopN 下推与全局分片。

### ② 设计原理与取舍
```
  ReadBuilder(配置, Driver)
      ├── newScan() ──► TableScan.plan() ──► List<Split>   [计划层, Driver/Coordinator]
      └── newRead() ──► TableRead.createReader(Split)       [执行层, Worker/TaskManager]
```
- **Scan（计划层）**：扫 manifest、应用分区/桶过滤、生成 `Split`。Driver 单点执行，可做全局优化（Limit 截断、TopN 裁剪、shard 分片）。
- **Read（执行层）**：拿一个 `Split` 建 `RecordReader`，应用 DV、列裁剪、行级谓词、文件索引。各 Worker 并行执行。
- **Split 是边界契约**：必须 `Serializable`、自包含（`partition/bucket/dataFiles/deletionFiles/rawConvertible`），跨节点传输后 Worker 无需再访问 Catalog 或 manifest。

与 Iceberg 对比：Iceberg `FileScanTask` 只带文件路径 + residual filter；Paimon `DataSplit` 还带 `rawConvertible`（决定读取策略）、`mergedRowCount`（决定 Limit 下推）、`deletionFiles`（MoR），信息密度更高，专为 LSM merge 语义服务。

### ③ 关键源码
```java
// AbstractFileStoreTable.java:291-315  批/流入口分派
public DataTableScan newScan() {
    DataTableBatchScan scan = new DataTableBatchScan(tableSchema, schemaManager(), coreOptions(),
            newSnapshotReader(), catalogEnvironment.tableQueryAuth(coreOptions()));
    return coreOptions().dataEvolutionEnabled() ? new DataEvolutionBatchScan(this, scan) : scan;
}
public StreamDataTableScan newStreamScan() {
    return new DataTableStreamScan(tableSchema, coreOptions(), newSnapshotReader(),
            snapshotManager(), changelogManager(), supportStreamingReadOverwrite(),
            catalogEnvironment.tableQueryAuth(coreOptions()), !tableSchema.primaryKeys().isEmpty());
}
```

### ④ 风险/陷阱/边界
- Scan 在 Driver 同步执行，`plan()` 内先 `authQuery()` 权限检查再做 manifest IO；大表上是秒级阻塞，不是"立即返回"。
- 批式 `plan()` 一次性（再调抛 `EndOfScanException`，见 `DataTableBatchScan.java:128`）；流式 `StreamDataTableScan.plan()` 有状态、可反复调用，每次返回新快照增量。
- Split 必须可序列化：自定义 `DataFileMeta` 扩展字段若不可序列化会在分发时失败。

### ⑤ 收益与代价
收益：天然适配 FLIP-27 Source / Spark DSv2 的 enumerator-reader 模型；计划期优化集中、Worker 端纯执行。代价：用户须理解 Scan/Read 两对象不可混用（Scan 无 `read()`，Read 无 `plan()`）。

### ⑥ 深入：Scan/Read 到引擎接口的映射

Paimon 的 Scan/Read 分离不是凭空设计，而是为对齐两大引擎的 source 模型：

| Paimon | Flink FLIP-27 | Spark DSv2 |
|---|---|---|
| `TableScan.plan()` 产 Split | `SplitEnumerator`（JobManager 单点枚举 split） | `Scan.planInputPartitions()`（Driver 生成 InputPartition） |
| `Split`（可序列化） | `SourceSplit` | `InputPartition` |
| `TableRead.createReader(split)` | `SourceReader`（TaskManager 读 split） | `PartitionReaderFactory.createReader` |
| 流式 `StreamScan.plan()` 循环 | `SplitEnumerator` 持续 `addSplits` | （micro-batch）每批 `latestOffset`+`planInputPartitions` |

因此 Paimon 的 `DataSplit` 必须 `Serializable` 且自包含——它要在 JM→TM / Driver→Executor 间传输。流式的 `nextSnapshotId` 状态（§5.6）对应 enumerator 的 checkpoint 状态，作业失败恢复时从 checkpoint 恢复 `nextSnapshotId` 续读。这也解释了为何计划期优化（Limit/TopN/分区裁剪）必须在 Scan 完成：enumerator 是单点，能做全局决策；reader 是分布式的，只能本地执行。

## 3. ReadBuilder 配置阶段

### ① 要解决的问题
用户参数（filter/projection/limit/topN/shard/bucketFilter/partitionFilter）需被收集，并**分别正确地**传到 Scan（文件级）和 Read（行级）两个阶段——漏传任一处都会导致过滤失效或结果错误。

### ② 设计原理与取舍
`ReadBuilderImpl`（`table/source/ReadBuilderImpl.java`）是唯一实现，由 `InnerTable.newReadBuilder()` 默认创建。它是纯参数容器，延迟到 `newScan()/newRead()` 才通过 `configureScan()` 统一分发。两处分发是关键：

```
filter ──┬─► scan.withFilter()  →  文件级裁剪（基于 min/max/null 统计跳整文件）
         └─► read.withFilter()  →  行级过滤（RecordReader 内逐行判断）
projection ─► withReadType(RowType.project(proj))  同时传给 Scan 与 Read
```

之所以同一个 filter 传两次：Scan 用它跳文件减少读取量，Read 用它跳行减少返回量，两层互补。Projection 同理但只在 Read 端真正裁列。

### ③ 关键源码
```java
// ReadBuilderImpl.newScan() → configureScan() 要点
scan.withFilter(filter).withReadType(readType)
    .withPartitionFilter(partitionFilter).withLimit(limit).withTopN(topN);
if (shardIndexOfThisSubtask != null) {            // shard 与 bucketFilter 二选一
    checkArgument(bucketFilter == null, "Bucket filter and shard cannot be set together.");
    scan.withShard(shardIndexOfThisSubtask, shardNumberOfParallelSubtasks);
} else if (bucketFilter != null) {
    scan.withBucketFilter(bucketFilter);
}
// newRead() → table.newRead().withFilter(filter).withReadType(readType).withTopN(topN).withLimit(limit)
```

### ④ 风险/陷阱/边界
- **`withFilter` 多次调用是覆盖不是叠加**：`this.filter = filter` 直接赋值。多条件须自行 `PredicateBuilder.and(...)`。
- **`withProjection` 不减小 Scan 开销**：manifest 仍读全列统计；projection 只在 Read 端列裁剪。
- **`bucketFilter` 与 `shard` 互斥**：同时设置在 `configureScan()` 抛 `IllegalArgumentException`，因两者都控 bucket 选择会产生歧义。
- **`shard` 在 BUCKET_UNAWARE 下退化**：所有数据在 bucket=0，shard 取模会把全部数据分到一个 subtask，造成倾斜。

### ⑤ 收益与代价
收益：参数任意顺序设置、晚绑定、Table 接口稳定。代价：互斥/冲突在 `newScan()` 才暴露，错误发现偏晚。

### ⑥ 深入：shard / projection / rowRangeIndex 的传递语义

- **`withProjection(int[])`**：转 `RowType readType = RowType.project(proj)`，同时 `scan.withReadType` 与 `read.withReadType`。Scan 端 readType 不影响 manifest 扫描（仍读全列统计），但影响 split 携带的读类型；Read 端据此只向格式 reader 请求所需列（Parquet/ORC 列裁剪）。嵌套列投影（`a.b`）由 readType 的嵌套 RowType 表达。
- **`withShard(i, n)`**：转 `snapshotReader.withShard`，按 bucket 取模分片——`bucket % n == i` 的桶归本 subtask。要求表有多桶才均衡；BUCKET_UNAWARE 单桶下全部落 `i=hash(0)%n` 一个 subtask（§3 倾斜陷阱）。与 bucketFilter 互斥。
- **`withPartitionFilter(PartitionPredicate)`**：直达 §6.5 第一层/第二层 partition 裁剪，是省 IO 最多的过滤，引擎应尽量把分区谓词独立出来传此通道而非混在 filter 里（虽然 `withFilter` 内部也会 `splitPartitionPredicatesAndDataPredicates` 自动拆分）。
- **`withRowRangeIndex`**：用于按物理 row 范围读（如 row-tracking、特定 FLIP-27 场景），普通查询不用。

**filter 在 Scan 与 Read 的不对称**：Scan 端 filter 可能因表类型被"削弱"（主键表只下推 key 列，§11），但 Read 端始终持完整 filter 做行级兜底——所以即便文件级没跳掉，结果仍正确，只是多读了行。这是"下推保守不影响正确性"的整体保证。

## 4. TableScan 计划阶段与下推优化

### ① 要解决的问题
在不读实际数据的前提下尽量减少最终读取量：Limit 下推（少生成 Split）、TopN 下推（用统计裁掉不可能命中的 Split）。

### ② 设计原理与取舍
`DataTableBatchScan.planWithoutAuth()`（`:106-130`）优先级是 **Limit > TopN > 普通扫描**，三者互斥取第一个成功者：

```java
Optional<Result> pushed = applyPushDownLimit();
if (pushed.isPresent()) result = pushed.get();
else { pushed = applyPushDownTopN(); result = pushed.orElseGet(() -> startingScanner.scan(snapshotReader)); }
```

**Limit 下推**（`:140-177`）：前置 `pushDownLimit != null && !snapshotReader.hasNonPartitionFilter()`——一旦有非分区谓词，行会被行级过滤掉，按文件 rowCount 截断会少读，故放弃。满足时先正常 `scan()`，再按顺序累加每 split 的 `mergedRowCount()`，累计 `>= limit` 即截断剩余 split 重组 `PlanImpl`。

**TopN 下推**（`:179-217`）：条件极严，任一不满足即退化为全量扫描且不报错。

### ③ 关键源码
```java
// DataTableBatchScan.java:159-176  Limit 下推累加截断
for (Split split : splits) {
    OptionalLong mergedRowCount = split.mergedRowCount();
    if (mergedRowCount.isPresent()) {                // 需 merge 的 split 返回 empty → 跳过累加，可能不截断
        limitedSplits.add(split);
        scannedRowCount += mergedRowCount.getAsLong();
        if (scannedRowCount >= pushDownLimit)
            return Optional.of(new ScannedResult(new PlanImpl(watermark, snapshotId, limitedSplits)));
    }
}
// DataTableBatchScan.java:180-194  TopN 下推硬条件
if (topN == null || pushDownLimit != null
        || !schema.primaryKeys().isEmpty()       // 主键表禁用（merge 后顺序不可预测）
        || options().deletionVectorsEnabled())    // DV 表禁用
    return Optional.empty();
if (orders.size() != 1) return Optional.empty();   // 仅单列排序
if (topN.limit() > 100) return Optional.empty();    // n>100 收益不抵代价
// 通过后：startingScanner.scan(snapshotReader.keepStats()) → TopNDataSplitEvaluator.evaluate(order, limit, splits)
```

### ④ 风险/陷阱/边界
- **Limit 下推对需 merge 的 split 无效**：`mergedRowCount()` 对 `rawConvertible=false` 返回 `empty`（§7），循环跳过累加，可能完全不截断。
- **有非分区谓词时 Limit 下推关闭**：用户以为"加了 limit 就快"，实际带 `WHERE` 子句时退化。
- **主键表禁用 TopN 下推**：merge 折叠同 key 多版本后单文件 max 不代表 merge 后 max，裁剪会出错——正确性约束而非性能取舍。
- TopN 仅单列、n≤100、非 PK、非 DV，超出静默退化。

### ⑤ 收益与代价
收益：计划期裁剪把"读多少"前移到 Driver，Worker 端零优化负担。代价：TopN 下推需 `keepStats()` 加载全部 split 统计，代价随 split 数线性增长，故 n≤100 封顶。

## 5. StartingScanner / FollowUpScanner 策略族

### ① 要解决的问题
批/流、全量/增量、时间旅行/标签/差异——十余种"从哪里开始读"的语义，用一个巨型 switch 会不可维护且难测试。

### ② 设计原理与取舍
`StartingScanner` 决定**起点**（执行一次），`FollowUpScanner` 决定**后续增量**（流式循环）。`AbstractDataTableScan.createStartingScanner()`（`:214-491`）按 `CoreOptions.StartupMode` 分派。继承结构：

```
StartingScanner
  └─ AbstractStartingScanner (持 snapshotManager + startingSnapshotId)
       ├─ ReadPlanStartingScanner (configure() 模板)
       │    ├─ Full / Compacted / FullCompacted StartingScanner
       │    ├─ StaticFromSnapshot/Tag/Timestamp/Watermark StartingScanner
       │    ├─ FileCreationTimeStartingScanner / ContinuousFromSnapshotFullStartingScanner
       ├─ ContinuousLatest/FromSnapshot/FromTimestamp/Compactor StartingScanner
       └─ IncrementalDelta/IncrementalDiff/EmptyResult StartingScanner
```

返回两类：`ScannedResult`（直接给初始 split，批式/全量起点）与 `NextSnapshot`（只给下一快照号，流式纯增量起点）。

### ③ 关键源码
```java

                                     : new FullStartingScanner(..);
case COMPACTED_FULL: return deltaCommits!=null ? new FullCompactedStartingScanner(.., deltaCommits) // identifier%delta==0 的 COMPACT
                                               : new CompactedStartingScanner(..);    // 最新 COMPACT 快照
case FROM_TIMESTAMP: return isStreaming ? new ContinuousFromTimestampStartingScanner(..)
                                        : new StaticFromTimestampStartingScanner(.., startupMillis);
case FROM_SNAPSHOT: ... // tag-name→StaticFromTag；watermark→StaticFromWatermark；snapshot-id→Static/Continuous
case FROM_SNAPSHOT_FULL: return isStreaming ? new ContinuousFromSnapshotFullStartingScanner(..) // 先全量再增量
                                            : new StaticFromSnapshotStartingScanner(..);
case FROM_FILE_CREATION_TIME / FROM_CREATION_TIMESTAMP: ... // 按创建时间过滤文件/快照
case INCREMENTAL: ... // incremental-between → IncrementalDelta/Diff；start==end 或无快照 → EmptyResult
```

**Consumer 优先**：流式且配 `consumer-id` 且未设 `consumer.ignore-progress` 时，优先 `ContinuousFromSnapshotStartingScanner(consumer.nextSnapshot())`，从持久化进度恢复——这是 exactly-once 消费进度的落地点。

**Incremental 两模式**：`IncrementalDelta` 读 `(start,end]` 每个快照 deltaManifest 的 `FileKind.ADD`（原始增量记录，跳过 COMPACT/OVERWRITE）；`IncrementalDiff` 对 start/end 两快照全量做 diff（最终状态差异，见 §6 `readIncrementalDiff`）。`incremental-between=tag1,tag2` 默认走 Diff；`incremental-between-tag-to-snapshot=true` 才转 snapshot 区间走 Delta。

**FollowUpScanner 三实现**：`DeltaFollowUpScanner`（仅 APPEND delta，默认）、`AllDeltaFollowUpScanner`（含 COMPACT，需感知合并）、`ChangelogFollowUpScanner`（读 changelogManifestList，`changelog-producer != none` 时）。Paimon 的 Snapshot 同时记录 base+delta manifest，流式消费直接读 delta，**O(1) 定位**，无需像 Iceberg `IncrementalAppendScan` 那样扫范围内所有 manifest 做差集。

### ④ 风险/陷阱/边界
- 流式未配 `consumer-id`：作业重启从 `latest+1` 起，重启窗口外的历史增量丢失（"从头消费"是误区，`LATEST` 实际从重启时刻起）。
- `EmptyResultStartingScanner`：`start==end` 或目标快照不存在返回空，不报错——增量为空与"读失败"须用 `$snapshots` 区分。
- `COMPACTED_FULL` 依赖存在 COMPACT 快照；从未触发 full-compaction 的表会找不到合适起点。

### ⑤ 收益与代价
收益：每种语义独立类、可单测、易扩展。代价：策略多达 16+，定位"我这条 SQL 走了哪个 scanner"需结合 `scan.mode`/`scan.snapshot-id`/`consumer-id` 多配置回溯。

### ⑥ 深入：DataTableStreamScan 流式状态机

流式 `plan()` 是有状态循环（`DataTableStreamScan.java:56+`），核心字段 `nextSnapshotId`（下一个待消费快照）+ `currentWatermark`：

```

  if commitKind == OVERWRITE:                                   // :217-220
```

**OVERWRITE 快照的流式处理**：默认流式消费**跳过** overwrite 快照（避免把覆写当增量重放）；`streaming-read-overwrite=true`（追加表为 `streaming-read-append-overwrite`）时才产出 overwrite 的增量变更。这是"流读看不到 INSERT OVERWRITE 结果"问题的源码根因。

**FILE_MONITOR 模式**（`:156`）：compactor 专用，产出文件级监控信息而非数据。

**边界**：`nextSnapshotId` 指向的快照已过期（被 §16 snapshot 过期清理）会触发异常或数据丢失——流式消费滞后超过 `snapshot.num-retained` 是生产事故高发点，须配 `consumer-id` 让过期感知消费进度（`consumer.expiration-time` + Consumer 阻止快照回收，见 §16 Task4/Task5 顺序）。

## 7. Split 结构与 mergedRowCount 三档算法

### ① 要解决的问题
Split 需自包含可序列化，并能在**不读数据**的前提下估算"merge 后行数"，供 Limit 下推与引擎并行度规划。

### ② 设计原理与取舍
两种 Split：
- **`DataSplit`**（`table/source/DataSplit.java:62`）：`snapshotId/partition/bucket/bucketPath/totalBuckets/dataFiles/dataDeletionFiles/isStreaming/rawConvertible`。全量与批式读取单元。
- **`IncrementalSplit`**：含 `beforeFiles/afterFiles` 两组及各自 DV，`isStreaming` 区分流式 changelog（true）与批式 diff（false）。需 before 是因生成 `-D/-U/+I/+U` 完整 changelog 必须对比前后。

`mergedRowCount()` 三档（精确→估算→放弃），直接决定 Limit 下推/并行度规划是否生效。

### ③ 关键源码
```java
// DataSplit.java:132-186
public OptionalLong mergedRowCount() {
    if (rawMergedRowCountAvailable()) return OptionalLong.of(rawMergedRowCount());  // 精确档
    if (dataEvolutionRowCountAvailable()) return OptionalLong.of(dataEvolutionMergedRowCount()); // 估算档
    return OptionalLong.empty();                                                     // 放弃档
}
// 精确档：rawConvertible 且(无 DV 或所有 DV cardinality 已知) → Σ(file.rowCount - dv.cardinality)
private boolean rawMergedRowCountAvailable() {
    return rawConvertible && (dataDeletionFiles == null
            || dataDeletionFiles.stream().allMatch(f -> f == null || f.cardinality() != null));
}
// 估算档：所有文件有 firstRowId → 按 rowId 区间合并重叠，每组取 max(rowCount)
private long dataEvolutionMergedRowCount() {
    List<List<DataFileMeta>> ranges =
        new RangeHelper<>(DataFileMeta::nonNullRowIdRange).mergeOverlappingRanges(dataFiles);
    long sum = 0; for (List<DataFileMeta> g : ranges) { long m=0; for (DataFileMeta f:g) m=Math.max(m,f.rowCount()); sum+=m; }
    return sum;
}
```

精确档背后是 §12 聚合下推（`COUNT(*)` 可直接由此求和）；估算档服务 data-evolution 场景（同一 rowId 范围多版本取最大行数近似）。

### ④ 风险/陷阱/边界
- 非 DV 的普通 MoR split 多为 `rawConvertible=false` → 返回 empty → Limit 下推/精确 `COUNT(*)` 不可用。
- DV cardinality 缺失（旧索引）会让精确档退化为放弃档。
- `rowCount()`（裸行数，`:124-130`）≠ `mergedRowCount()`（merge 后行数），引擎做并行度规划时勿混用。

### ⑤ 收益与代价
收益：三档分级让"能精确就精确、能估算就估算、否则诚实放弃"，避免错误截断。代价：精确档对 DV 元数据完整性有依赖，老表升级后需重写才能享受。

### ⑥ 深入：估算档的 data-evolution 行数语义

估算档（`dataEvolutionMergedRowCount`，`DataSplit.java:174-186`）服务于 data-evolution（schema 演进 + row-tracking）场景：同一逻辑行的不同列版本可能分布在多个文件，靠 `firstRowId`+`rowCount` 形成的 `[firstRowId, firstRowId+rowCount)` 区间标识。算法用 `RangeHelper.mergeOverlappingRanges` 合并 rowId 重叠的文件分组，每组取 `max(rowCount)`——因为同一 rowId 区间的多文件代表"同一批行的不同列版本"，行数应取最大而非求和（求和会重复计数）。

```
文件 rowId 区间：[0,100) [0,100) [100,150)
合并重叠：group1={[0,100),[0,100)} → max=100；group2={[100,150)} → 50
mergedRowCount = 100 + 50 = 150（而非 100+100+50=250）
```

这与精确档（DV 减法）互补：DV 表用精确档、data-evolution 表用估算档、普通 nonRaw 主键表两者皆不可用返回 empty。三档的判定顺序（精确优先）保证：能精确就不估算。

### ⑦ 风险/陷阱/边界（mergedRowCount 进阶）
- 估算档是上界近似（取 max），Limit 下推用它截断是安全的（宁可多读一个 split 不可少读）。
- `IncrementalSplit` 不实现精确 mergedRowCount，增量读不参与 Limit 下推。

## 8. SplitGenerator：IntervalPartition + BinPacking

### ① 要解决的问题
把一个 (partition,bucket) 下的一堆文件切成"既能并行、又不破坏 merge 正确性"的 Split——主键表必须保证 key 重叠的文件进同一 Split。

### ② 设计原理与取舍
主键表 `MergeTreeSplitGenerator` 两段式：能直读则顺序装箱，否则先按 key 范围 IntervalPartition 切不重叠 section、再 BinPacking 装箱。追加表 `AppendOnlySplitGenerator` 无 key 概念，按序号排序后直接装箱。

### ③ 关键源码

**IntervalPartition 算法**（源码注释 `:92-103`）：文件按 key 范围 `[1,2][3,4][5,180][5,190][200,600][210,700]`、targetSplitSize=128M：


### ④ 风险/陷阱/边界
- `withoutDeleteRow` 把缺失 `deleteRowCount` 当 true（兼容旧文件），但下游 `PrimaryKeyTableRawFileSplitReadProvider` 又要求 `deleteRowCount` 必须存在才走 raw（§10），两处需配套理解：生成端乐观、消费端保守。
- `BUCKET_UNAWARE` 追加表流式不切分会单并行度倾斜，故强制 batch 切分。
- 主键表流式不切分：单 (partition,bucket) 一个 Split，热桶可能成为流式瓶颈。

### ⑤ 收益与代价
收益：IntervalPartition 保正确性、BinPacking 让 split 大小趋近 `targetSplitSize`（默认随 `source.split.target-size`）、并行均衡。代价：IntervalPartition 需 `keyComparator` 全量比较，文件极多时计划期 CPU 升高。

### ⑥ 深入：BinPacking 的 openFileCost 与权重函数

`BinPacking.packForOrdered(items, weightFunc, targetSize)` 是顺序贪心装箱：按输入顺序累加权重，超过 `targetSize` 就开新箱。两处细节决定 split 质量：

```java
// MergeTreeSplitGenerator.java:76-77 / AppendOnlySplitGenerator.java:55
Function<DataFileMeta, Long> weightFunc = file -> Math.max(file.fileSize(), openFileCost);
```

**`openFileCost`（`source.split.open-file-cost`，默认约 4MB）的作用**：权重取 `max(fileSize, openFileCost)`，给每个文件一个"打开成本下限"。否则大量小文件会被装进一个 split（按 fileSize 累加远不到 targetSize），导致单 split 要打开成百上千个文件——打开文件的固定开销（seek、元数据读）会主导耗时。openFileCost 把"打开 N 个小文件"等价为"读 N×4MB"，使 split 内文件数受控。

**顺序装箱而非最优装箱**：`packForOrdered` 保持输入顺序（主键表已按 IntervalPartition section 顺序、追加表按 sequenceNumber 顺序），不做全局最优（NP-hard）。保序的意义：追加表保证 split 内数据有序（利于下游有序消费）、主键表保证同 section 相邻文件尽量同箱。

**调优**：split 太多（调度开销大）→ 调大 `source.split.target-size`；单 split 文件过多（打开开销大）→ 调大 `source.split.open-file-cost`；两者共同决定 batch 读并行度（split 数）。

## 9. rawConvertible 与读放大

### ① 要解决的问题
主键表默认 MoR：读时把同 key 多版本归并，CPU 与格式转换放大严重。如何在保证正确前提下让一部分文件"直读"，逼近追加表/Iceberg 读性能？

### ② 设计原理与取舍
`rawConvertible=true` 意味着该 Split 内文件**互不需要 merge**，可逐文件向量化直读，跳过 LoserTree 归并与 MergeFunction。判定矩阵：

| 表/引擎 | rawConvertible 条件 | 原理 |
|---|---|---|
| 追加表 | 始终 true | 无 merge 语义 |
| 主键表 + DV | `deletion-vectors.enabled=true` | 每文件配独立 DV，删除靠 bitmap 跳行 |
| 主键表 + FIRST_ROW | level>0（已 compact） | 只取首条，compact 后每 key 唯一 |
| 主键表普通 | 全部 level>0 + 无 deleteRow + 同一 level | 同 level key 不重叠可独立读 |
| 主键表普通 | 单文件且无 deleteRow（即使跨 level） | 单文件无需归并 |
| 主键表普通 | 有 level=0 或多 level key 重叠 | false，必须 merge |

### ③ 关键源码
判定见 §8 `MergeTreeSplitGenerator.splitForBatch:70-75`；消费见 §10 `PrimaryKeyTableRawFileSplitReadProvider.match:36-58`。

读放大量化（同一份 PK 数据批查）：

```
nonRaw（merge）路径：  打开 K 个文件 → LoserTree O(N·logK) 比较 → 逐行 MergeFunction → KeyValue→InternalRow 转换
raw 路径：             逐文件引擎原生 Parquet/ORC 向量化 batch 读 →（DV 时按 bitmap 跳行）→ 直出 InternalRow
```

### ④ 风险/陷阱/边界
- **`forceKeepDelete` 一票否决**：审计/changelog 场景需保留 `-D` 时，即便 rawConvertible=true 也强制 merge（raw 只产 `+I`）。
- **新写入 L0 破坏 raw**：高频写下 L0 堆积让大量 split 退化为 nonRaw，读性能骤降——看 `$files` 的 level 分布判断 compaction 是否跟上。
- **DV 模式 raw 的代价**：删除靠 bitmap 跳行，被删比例高时仍读全文件再丢弃，省 CPU 不省 IO。

### ⑤ 收益与代价
收益：DV 模式下主键表批查性能接近 COW/Iceberg，且聚合可下推（§12）。代价：DV 维护增加写放大（§18）；非 DV 表需依赖充分 compaction 才能享受 raw。

### ⑥ 深入：用 level 与 key 范围判定 rawConvertible 的三个实例

LSM 分层语义：L0 是 flush 直出（同 key 可与任意层重叠）；L1+ 是 compaction 产物（同 level 内 key 不重叠、有序）。`rawConvertible` 本质是问"这组文件里有没有可能存在同 key 多版本"：

```
实例A（可 raw）：[L1: key 1-100] [L1: key 101-200]   同 level 不重叠 → oneLevel 分支 → raw
实例B（不可 raw）：[L0: key 1-200] [L1: key 50-150]   有 L0、且跨层重叠 → 必须 merge
实例C（可 raw）：[L2: key 1-300]（单文件，无 deleteRow） 单文件无需归并 → raw（即使非 L0）
```

源码两个布尔门（§8 `:70-75`）：`rawConvertible = 全部 level!=0 && 无 deleteRow`（排除 L0 与含删除标记），再 `&& (DV || FIRST_ROW || oneLevel)`（确保无跨层重叠的可能）。DV/FIRST_ROW 是"语义豁免"（各文件独立/只取首条），oneLevel 是"物理保证"（同层不重叠）。

**为什么 L0 一票否决**：L0 文件是写入直出，其 key 可能与下层任意文件重叠，无法独立读——这把"compaction 是否跟上"与"读性能"直接绑定：L0 越多，越多 split 退化为 nonRaw。线上看 `$files` 的 `level=0` 文件数即可预判读放大。

## 10. TableRead 执行阶段与 SplitReadProvider

### ① 要解决的问题
同一张表不同 Split 形态（raw / 需 merge / 流式增量 / 批式 diff）需不同 `RecordReader`。如何按形态动态选实现，又避免初始化用不到的 reader？

### ② 设计原理与取舍
`KeyValueTableRead`（主键表）持 **4 个 `SplitReadProvider`，按优先级 `match()`**（`KeyValueTableRead.java:68-74`），命中第一个即用其 `LazyField<SplitRead>`；`LazyField` 保证 4 选 1 时其余 3 个 reader 永不初始化：

```
① PrimaryKeyTableRawFileSplitReadProvider  → RawFileSplitRead        (直读，最优)
② MergeFileSplitReadProvider               → MergeFileSplitRead       (LoserTree 归并)
③ IncrementalChangelogReadProvider         → ConcatReader(ReverseReader(before), after)  (流式增量)
④ IncrementalDiffReadProvider              → IncrementalDiffSplitRead  (批式 diff)
```

`AppendTableRead`（追加表）只有 1 个：`dataEvolutionEnabled` 时 `DataEvolutionSplitReadProvider`，否则 `AppendTableRawFileSplitReadProvider`。

### ③ 关键源码
```java
// PrimaryKeyTableRawFileSplitReadProvider.java:36-58
public boolean match(Split split, Context context) {
    if (!(split instanceof DataSplit)) return false;
    DataSplit ds = (DataSplit) split;
    boolean matched = !context.forceKeepDelete() && !ds.isStreaming() && ds.rawConvertible();
    if (matched)
        for (DataFileMeta file : ds.dataFiles())
            if (!file.deleteRowCount().isPresent()) return false;  // 旧版本缺 deleteRowCount → 退 merge 求正确
    return matched;
}
```

优先级 3 的 changelog 还原原理：before 文件经 `ReverseReader` 翻转（每行标 `-D`），再 `ConcatRecordReader` 拼 after（`+I/+U`），由"前后两份全量"还原完整 changelog。优先级 2 的 `MergeFileSplitRead` 用 `withReadKeyType(RowType.of())` 不读 key 列只读 value，依赖文件内排序 + LoserTree 重建顺序。

### ④ 风险/陷阱/边界
- 优先级 1 对"缺 `deleteRowCount` 的旧文件"主动降级 merge，是正确性优先于性能的典型；升级后老文件未重写前读偏慢属预期。
- match 顺序是隐式契约，新增 Split 形态须谨慎插位（如把 diff 放在 changelog 前会错配流式 split）。

### ⑤ 收益与代价
收益：策略 + 惰性初始化，零浪费且形态可扩展。代价：4 个 provider 的 match 顺序耦合，调试需逐个看 `match` 返回。

### ⑥ 深入：MergeFileSplitRead 的归并内核

nonRaw split 走 `MergeFileSplitRead`，核心是**多路归并 + MergeFunction 折叠同 key**。归并器二选一（`mergetree/compact/`）：
- `SortMergeReaderWithLoserTree`（`LoserTree.java`）：败者树，K 路归并每次出最小元素只需 O(logK) 比较，K 大时优于堆（缓存友好、比较次数少）。
- `SortMergeReaderWithMinHeap`：最小堆，作为备选。

由 `sort-engine`（`loser-tree`/`min-heap`，默认 loser-tree）选择。输入是同 section 内多个文件的 `RecordReader`（按 key+sequenceNumber 有序），归并后同 key 的多版本连续到达，交 `MergeFunction` 折叠为一条。

**MergeFunction 家族**（`mergetree/compact/*MergeFunction.java`）对应 `merge-engine`：
| MergeFunction | merge-engine | 折叠语义 |
|---|---|---|
| DeduplicateMergeFunction | deduplicate(默认) | 取最高 sequenceNumber 的版本（最新覆盖旧） |
| PartialUpdateMergeFunction | partial-update | 逐列非空覆盖，支持 sequence-group / 聚合列 |
| AggregateMergeFunction | aggregation | 按列配置的 agg 函数（sum/max/last_value...）聚合 |
| FirstRowMergeFunction | first-row | 只保留首条（§9 raw 短路即源于此） |
| LookupMergeFunction | （包装层） | `needLookup` 时包装，配合 DV/changelog-producer=lookup 在写时 lookup 旧值 |

`LookupMergeFunction.wrap`（`PrimaryKeyFileStoreTable.java:84-85`）：当 `options.needLookup()` 为真（DV 或 lookup changelog），把上述 MergeFunction 再包一层，使其能在 compaction/写入时通过 `LookupLevels` 查到旧记录，从而产出 changelog 或维护 DV。

**为什么 raw 路径能跳过这一切**：raw split 内文件互不含同 key 多版本（DV 模式各文件独立、FIRST_ROW 已 compact、同 level 不重叠），无需归并与折叠，直接逐文件向量化读——这是 §9 读放大差异的根源（nonRaw 多付 O(N·logK) 比较 + 逐行 MergeFunction + KeyValue→InternalRow 转换）。

### ⑦ 风险/陷阱/边界（归并内核）
- `MergeFileSplitRead` 用 `withReadKeyType(RowType.of())` 不读 key 列只读 value，依赖文件内排序重建顺序——key projection 误传会破坏归并。
- partial-update / aggregation 引擎对 `-D/-U` 处理有特殊规则（部分配置下拒绝或忽略 retract），需结合 `partial-update.remove-record-on-delete` 等理解。

## 11. 谓词下推三层模型与保守策略

### ① 要解决的问题
谓词要尽早过滤以减 IO，但主键表 LSM 语义下"对 value 列做文件级过滤"会破坏 merge 正确性。如何分层下推且不牺牲一致性？

### ② 设计原理与取舍
三层过滤逐层收窄：

```
Layer1 分区过滤(manifest级)：withFilter 拆分区谓词 → scan.withPartitionFilter → 跳整分区 ManifestEntry（省 IO 最多）
Layer2 文件级过滤(stats级)：按文件 min/max/null 跳整文件
   ├ 追加表：完整谓词 → AppendOnlyFileStoreScan.withFilter（任意列安全）
   └ 主键表：仅 key 列谓词 → KeyValueFileStoreScan.withKeyFilter；完整谓词 → withValueFilter（仅 bucket 级）
Layer3 行级过滤：RecordReader 内逐行 + 文件索引(bloom/bitmap/bsi)读前裁剪
```

### ③ 关键源码
主键表只下推 key 列的根因（`PrimaryKeyFileStoreTable.java:123-144`，含源码注释）：

```java
protected BiConsumer<FileStoreScan, Predicate> nonPartitionFilterConsumer() {
    return (scan, predicate) -> {
        // data file1: insert key=a,value=1 ; data file2: update key=a,value=2 ; filter: value=1
        // 若对 value 文件级下推 → file1 选中、file2 跳过 → 结果(a,1)，但正确结果为空集(a 最终 value=2)
        List<Predicate> keyFilters = pickTransformFieldMapping(
                splitAnd(predicate), tableSchema.fieldNames(), tableSchema.trimmedPrimaryKeys());
        if (!keyFilters.isEmpty()) ((KeyValueFileStoreScan) scan).withKeyFilter(and(keyFilters));
        ((KeyValueFileStoreScan) scan).withValueFilter(predicate);  // value 谓词仅用于 bucket 级裁剪
    };
}
```

追加表（`AppendOnlyFileStoreTable.java:108-109`）：`(scan,p) -> ((AppendOnlyFileStoreScan)scan).withFilter(p)`，无 key 重叠，任意列文件级过滤安全。

### ④ 风险/陷阱/边界
- **主键表 value 列谓词不缩文件**：`WHERE amount>1000`（amount 非 key）只在 bucket 级（整桶 stats 都不满足才跳）与行级生效，文件级不跳。
- `withValueFilter` bucket 级裁剪安全的前提：同 key 所有版本必在同桶，故整桶 stats 不满足即可整桶跳。
- DV/COW/read-optimized 表无 merge 顾虑，可对非主键列做文件级过滤（官方 query-performance：DV 表"can also utilize filtering conditions for non-primary-key columns"）。

### ⑤ 收益与代价
收益：分区裁剪省最多 IO、文件级次之、行级兜底；保守策略守住主键表正确性底线。代价：主键表非 DV 时 value 过滤弱于追加表，是 LSM 固有代价。

### ⑥ 深入：Layer3 行级过滤与文件索引的协作

`scan.withCompleteFilter(predicate)`（§6 `withFilter`）保留完整谓词，最终在 Read 端两处使用：
1. **文件索引读前裁剪**（Layer3 上半）：若配 `file-index.{type}.columns`，读数据文件前先读其索引文件，按谓词判断"是否可能命中"。例：`WHERE id=12345` 且 id 配了 bloom-filter → bloom 查 12345 不存在则整文件跳过（bloom 无假阴性，跳过安全）；`WHERE age BETWEEN 20 AND 30` 且 age 配 range-bitmap → 直接定位命中行组。
2. **逐行谓词过滤**（Layer3 下半）：RecordReader 解出每行后用完整谓词 `test(row)`，跳不匹配行。这是正确性兜底——无论前面文件级/索引级是否生效，行级总能保证结果正确。

**三层的过滤强度递增、代价递增**：分区级（跳目录，几乎零成本）< 文件级 stats（读 manifest 即可，无需开文件）< 文件索引（需读索引文件）< 行级（需解数据行）。优化器/用户应尽量让谓词在更高层生效：把过滤列设为分区键 > bucket 键 > 主键 > file-index 列 > 普通列。

**主键表 value 谓词的完整命运**：`amount>5`（value 列）→ 分区级不适用 → 文件级不下推（§11 保守）→ bucket 级若整桶 stats 不满足则跳桶 → file-index 若配则读前裁剪 → 行级兜底。即"文件级失效但其余层仍可发力"。

## 12. 聚合/索引下推与查询加速

### ① 要解决的问题
`COUNT(*)`、`MIN/MAX`、点查、稀疏过滤若仍走"读文件 + 计算"则浪费。Paimon 如何用元数据在编译期/读前直接返回或大幅跳文件？（官方 query-performance.md 功能点）

### ② 设计原理与取舍
- **聚合下推（DV 表）**：`SELECT COUNT(*) ... WHERE dt=...` 可在编译期由 manifest/DV 元数据算出——每文件 `rowCount - dv.cardinality` 求和即精确行数（对应 `DataSplit.mergedRowCount` 精确档，§7），无需读数据文件。Spark 下 `MIN/MAX` 在 `metadata.stats-mode` 默认下也可由文件 min/max 直接得出。
- **PK Filter 数据跳过**：合理 bucket（如 bucket=5）下主键等值/范围条件经 `withKeyFilter` + bucket 选择，跳大量文件。
- **File Index 数据跳过**：full-compacted 文件或 DV 表可配 `file-index.{bloom-filter,bitmap,range-bitmap}.columns`，索引随数据文件生成（小则内联 manifest，大则旁路文件），Layer3 读前裁剪行组/文件。bloom 适合点查、bitmap 精度高但占空间、range-bitmap 适合范围。
- **Read Optimized 表**：`$read_optimized` 只读最高 level（已 compact）数据，跳过未 compact 的 L0，牺牲实时性换无 merge 直读（与 DV/COW 一样无并发读限制）。

### ③ 关键源码
聚合行数依赖 §7 `mergedRowCount` 精确档（`DataSplit.java:143-163`）；file index 落在 `paimon-common/fileindex/` 与读端格式 reader 的行组裁剪；read-optimized 走 `SnapshotReader.withLevel(maxLevel)` 路径。

### ④ 风险/陷阱/边界
- 聚合下推仅 DV 表精确；非 DV MoR 表 `COUNT(*)` 因需 merge 折叠重复 key，不能用文件 rowCount 直接求和。
- file index 需 full-compacted 或 DV 才有意义；对已存在表加索引用 `rewrite_file_index` procedure，且须先 `ALTER` 配 `file-index.*.columns`。
- read-optimized 读到的是"上次 compaction 时的快照视图"，可能滞后于最新写入。

### ⑤ 收益与代价
收益：聚合/点查可亚秒返回，索引在高基数过滤下跳文件比例高。代价：DV/索引/full-compaction 都增写侧成本与存储；read-optimized 牺牲数据新鲜度。

### ⑥ 深入：聚合下推为何只对 DV 表精确

`COUNT(*)` 下推的可行性完全取决于"能否不读数据就知道 merge 后行数"。三种表的差异：
- **追加表**：无 merge，`Σ file.rowCount` 即精确——任何时候可下推。
- **DV 主键表**：`Σ(file.rowCount - dv.cardinality)`（§7 精确档），删除已被 DV 记录，无重复 key 残留——精确可下推。
- **非 DV 主键表（MoR）**：同一 key 的多版本散落多文件，`Σ file.rowCount` 是"物理行数"含重复，merge 后真实行数未知——**不能**下推，必须实际 merge 才知道。

这就是官方 query-performance 强调"DV 表才能聚合下推"的源码本质。`MIN/MAX` 下推同理依赖文件 `valueStats`（min/max），DV 表与 full-compacted 文件的 stats 才完整可信；Spark 侧由 `metadata.stats-mode` 控制 stats 收集粒度（`full`/`truncate`/`none`），`none` 时无 stats 不能下推。

**file-index 的物理形态**：索引文件小则内联进 manifest（省一次 IO）、大则旁路存于数据文件目录，每数据文件一个独立索引文件，可含多列多类型索引。读端在打开数据文件前先读索引判断"是否可能命中"，命中才读对应行组——这是 Layer3 行级过滤之前的"读前裁剪"。

## 13. 写路径总览与 TableWriteImpl 流水线

### ① 要解决的问题
把用户 `InternalRow`（可能带 CDC 的 `-U/+U/-D`）正确路由到 (partition,bucket)、校验、填默认值、过滤 RowKind、转成存储格式（KeyValue 或 InternalRow），且批/流共用 API。

### ② 设计原理与取舍
`Write` 与 `Commit` 分离：Write 在 Worker 写文件产 `CommitMessage`，Commit 在 Coordinator 原子提交。`TableWriteImpl.writeAndReturn`（`table/sink/TableWriteImpl.java:183-192`）是 6 步流水线，顺序即设计：

```java
public SinkRecord writeAndReturn(InternalRow row, int bucket) throws Exception {
    checkNullability(row);                                    // 1 非空校验（先于默认值，否则校验失效）
    row = wrapDefaultValue(row);                              // 2 默认值填充（DefaultValueRow 包装）
    RowKind rowKind = RowKindGenerator.getRowKind(rowKindGenerator, row);  // 3 取 RowKind（rowkind-field 或 row 自带）
    if (rowKindFilter != null && !rowKindFilter.test(rowKind)) return null; // 4 过滤（被滤行返回 null，不报错）
    SinkRecord record = bucket == -1 ? toSinkRecord(row) : toSinkRecord(row, bucket); // 5 提取 partition/bucket/pk
    write.write(record.partition(), record.bucket(), recordExtractor.extract(record, rowKind)); // 6 转格式写入
    return record;
}
```

第 6 步 `recordExtractor` 是表类型差异点：主键表 `kv.replace(primaryKey, UNKNOWN_SEQUENCE, rowKind, row)`（`PrimaryKeyFileStoreTable.java:165-170`）；追加表直接返回 row 并校验 `rowKind.isAdd()`（`AppendOnlyFileStoreTable.java:147-150`）。`SinkRecord{partition, bucket, trimmedPrimaryKey, row}`。`RowKindFilter` 由 `read.changelog-mode` 驱动（`all` 不滤 / `upsert` 滤掉 `-U`）。

### ③ 关键源码
```java
// PrimaryKeyFileStoreTable.java:158-170  主键表写入：InternalRow → KeyValue
public TableWriteImpl<KeyValue> newWrite(String commitUser, Integer writeId) {
    KeyValue kv = new KeyValue();
    return new TableWriteImpl<>(rowType(), store().newWrite(commitUser, writeId), createRowKeyExtractor(),
            (record, rowKind) -> kv.replace(record.primaryKey(), KeyValue.UNKNOWN_SEQUENCE, rowKind, record.row()),
            rowKindGenerator(), ...);
}
```

`UNKNOWN_SEQUENCE` 是占位 seq，真实 sequenceNumber 在 `MergeTreeWriter` flush 时按写入顺序分配——这是 LSM 同 key 多版本能正确折叠的依据。

### ④ 风险/陷阱/边界
- **校验先于默认值**：`name NOT NULL DEFAULT 'x'` 传入 null 仍抛异常（`checkNullability` 在 `wrapDefaultValue` 前），默认值仅补"未提供"而非"显式 null"。
- **追加表拒非 INSERT**：`-D/-U/+U` 触发 `IllegalArgumentException`。
- **被 RowKindFilter 滤掉的行静默丢弃**（返回 null），不是异常。
- **`write(row, bucket)` 跳过 bucket 计算**：HASH_FIXED 表手动传错 bucket 会数据路由错乱、查不到；动态桶模式则**必须**用此重载由外部 assigner 传 bucket。

### ⑤ 收益与代价
收益：6 步职责单一、可独立测试；关键路径（toSinkRecord/recordExtractor）用 CodeGen 提速。代价：每行多次方法调用，CodeGen 抵消大部分但非零。

### ⑥ 深入：RowKind、sequence 与 changelog 的写时协作

**RowKind 来源三选一**：`rowkind-field` 配置时从该字段读字符串（`+I/-D/-U/+U`，由 `RowKindGenerator` 解析）；否则用 `row.getRowKind()`（CDC 引擎设置）；批写默认全 `+I`。追加表只接受 `isAdd()`（+I/+U 视为 add）。

**sequence 决定 merge 胜负**：写时占位 `UNKNOWN_SEQUENCE`，flush 时按写入顺序分配真实递增 seq；若配 `sequence.field`（如业务时间戳），则用该字段值作 seq，使"业务时间更新者胜"而非"物理写入晚者胜"——解决乱序 CDC 的关键。DeduplicateMergeFunction 取 max seq，故 seq 语义直接决定最终值。

**changelog 的三种产出方式**（`changelog-producer`）：
- `none`：不产 changelog，流读只能拿 upsert（无 -U）。
- `input`：直接把输入的 RowKind 当 changelog 写出（最轻，但要求输入已是完整 changelog）。
- `lookup`：写/compaction 时用 `LookupLevels` 查旧值，现算出 `-U/+U`（`LookupMergeFunction` 包装，§10.6），CPU 重但输入可以是 upsert。
- `full-compaction`：在 full compaction 时对比新旧产 changelog，延迟高但开销摊薄。

这解释了 §10 优先级 3 的 `IncrementalChangelogReadProvider`：当无独立 changelog 文件时，流式增量靠"before 反转 + after 拼接"现算 changelog；有 changelog 文件时则直接读（ChangelogFollowUpScanner，§5）。

## 14. RowKeyExtractor 行路由

### ① 要解决的问题
按 bucket 模式把行映射到正确 bucket（固定哈希/动态分配/无桶/延迟分配），且 partition 与 trimmedPrimaryKey 提取逻辑复用。

### ② 设计原理与取舍
`RowKeyExtractor`（`table/sink/RowKeyExtractor.java`）抽象 `bucket()`，`partition()/trimmedPrimaryKey()` 复用 `RowPartitionKeyExtractor`。`AbstractFileStoreTable.createRowKeyExtractor()`（`:258`）按 `BucketMode` 选实现：

| BucketMode | 实现 | bucket() |
|---|---|---|
| HASH_FIXED | FixedBucketRowKeyExtractor | `BucketFunction.bucket(bucketKey, numBuckets)` |
| HASH_DYNAMIC / KEY_DYNAMIC | DynamicBucketRowKeyExtractor | **抛异常**（须 `write(row,bucket)` 外部分配） |
| BUCKET_UNAWARE | AppendTableRowKeyExtractor | 恒返回 0 |
| POSTPONE_MODE | PostponeBucketRowKeyExtractor | 返回延迟标记 |

`FixedBucketRowKeyExtractor`：若 `bucketKeys == trimmedPrimaryKeys` 复用 pk 投影，否则用 CodeGen 投影提取 bucketKey，再 `hash(bucketKey) % numBuckets`。

### ④ 风险/陷阱/边界
动态桶 `bucket()` 抛异常是设计而非缺陷——分配需全局索引（该 key 历史在哪个 bucket），逻辑在 Flink `BucketAssigner`，不在 extractor。固定桶改桶数需 reshuffle/rescale，不能直接改配置。

### ⑤ 收益与代价
收益：路由策略与写入主流程解耦。代价：动态/延迟桶把分配责任外移给引擎层，单读 core 难看全貌。

### ⑥ 深入：动态桶分配为何不在 core

固定桶 `hash(bucketKey) % numBuckets` 是无状态纯函数，读写两侧用同一公式即可定位（BucketSelector 也据此反推，§6.5）。**动态桶（HASH_DYNAMIC/KEY_DYNAMIC）则有状态**：同一 key 第二次写必须落到它"上次所在的 bucket"，否则同 key 散落多桶破坏 merge。这要求一个"key→bucket"的全局索引：
- `HASH_DYNAMIC`：bucket=-1 默认模式，索引文件（hash-index）记录 hash(pk)→bucket，写时查索引；满桶（`dynamic-bucket.target-row-num`）则分配新桶。
- `KEY_DYNAMIC`：跨分区更新场景，索引粒度更细。

索引的查询与新桶分配逻辑在 Flink 算子（`HashBucketAssignerOperator`/`BucketAssigner`）中，因为它需要 keyed state 与跨记录的状态维护——这是 core 的 `RowKeyExtractor.bucket()` 直接抛异常、强制 `write(row, bucket)` 由引擎传入的根本原因。`POSTPONE_MODE` 更进一步：写时不定桶，提交后由专门的 compaction/重排过程延迟分配，适合写入时无法确定桶数的场景。

**边界**：固定桶改桶数需 `rescale` procedure 重排数据（不能直接改配置后读旧数据）；动态桶索引膨胀时点查变慢，需关注 hash-index 大小。

## 15. FileStoreWrite 与 prepareCommit

### ① 要解决的问题
内存缓冲排序、L0 flush、异步 compaction、DV 维护，最终汇成可原子提交的 `CommitMessage`。

### ② 设计原理与取舍
主键表 `KeyValueFileStoreWrite` → `MergeTreeWriter`：写 changelog（若配置）→ 写内存 `SortBufferMemTable`（按 key+seq 排序）→ 超阈值 flush 成 L0 SST → 异步 `CompactManager.triggerCompaction()`（L0 数达阈值时按 Universal/Level 策略合并）。追加表 `BaseAppendFileStoreWrite` → `AppendOnlyFileWriter`：写文件缓冲 → 达 `targetFileSize` rollover → 可选 compaction。

`prepareCommit(waitCompaction, commitIdentifier)`：遍历每个 `WriterContainer`（per partition+bucket），flush 残留、（waitCompaction 时）等异步 compaction 完成，收集 `newFiles/compactBefore/compactAfter/changelogFiles` 生成 `CommitMessageImpl`：

```
CommitMessageImpl{ partition, bucket, totalBuckets,
  DataIncrement{ newFiles, deletedFiles, changelogFiles, newIndexFiles, deletedIndexFiles },
  CompactIncrement{ compactBefore, compactAfter, changelogFiles, newIndexFiles, deletedIndexFiles } }
```

`DataIncrement` 描述本次写入新增/删除，`CompactIncrement` 描述 compaction 的文件替换——两者在 commit 时分别转成 `FileKind.ADD/DELETE` 的 ManifestEntry（§17）。

### ④ 风险/陷阱/边界
- `prepareCommit(true,...)` 阻塞等 compaction，流式下易致 checkpoint 超时——流式通常传 `waitCompaction=false`。
- 漏调 `prepareCommit` 直接 `commit(emptyList)` → 内存未 flush → 数据丢失。
- `totalBuckets` 可空（Schema 演进改桶数），下游并行度规划需容错。
- DV 模式下 `prepareCommit` 还会 `dvMaintainer.writeDeletionVector()` 产 DV 索引文件入 `newIndexFiles`（§18）。

### ⑤ 收益与代价
收益：写时仅排序 + 顺序写，compaction 异步化，写延迟稳定。代价：异步 compaction 落后 → L0 堆积 → 读放大（§9）；waitCompaction 同步则写延迟抖动。

### ⑥ 深入：MergeTreeWriter 内核与 compaction 策略

`MergeTreeWriter.write(KeyValue)` 的内存路径：
1. 写入 `WriteBuffer`（`SortBufferWriteBuffer`，按 key+seq 排序的内存缓冲）。
2. 缓冲满（`write-buffer-size`）或显式 flush → `mergeAndWriteBuffer`：用与读路径同源的 SortMergeReader 把缓冲内同 key 折叠后，写出一个 **level-0** SST（Parquet/ORC），产 `DataFileMeta`。
3. flush 后检查是否触发 compaction，提交给 `CompactManager` **异步**执行。

**Compaction 策略**（`mergetree/compact/`）：主键表用 `UniversalCompaction`（默认，`CompactStrategy` 接口），类似 RocksDB universal：按 `num-sorted-run.compaction-trigger` 触发，挑选 sorted run 集合合并，控制写放大与 run 数量；`num-sorted-run.stop-trigger` 达到时**反压写入**（拒绝新写直到 compaction 跟上），这是 L0 堆积的安全阀。

**sequenceNumber 的全局定序**：§13 写入时用 `UNKNOWN_SEQUENCE` 占位，真实 seq 在 flush 写 SST 时按写入顺序递增分配，保证"同 key 后写的 seq 更大"，从而 DeduplicateMergeFunction 取 max seq 即取最新——这是 LSM 正确折叠的底层契约。

**写放大量化**：一条记录可能被写多次（L0→Ln 每层 compaction 重写一次），写放大 ≈ compaction 层数；universal 策略以较低写放大换取读时 run 数受控。DV 模式额外加 lookup + bitmap 重写（§18）。

### ⑦ 风险/陷阱/边界（写内核）
- `num-sorted-run.stop-trigger` 触发反压：写入吞吐突降常因 compaction 跟不上，查 `$files` 的 level-0 run 数确认。
- `write-buffer-size` 过小 → 频繁 flush 出大量小 L0 文件 → 读放大；过大 → 内存压力 + flush 长尾。
- 流式 `waitCompaction=false` 下 compaction 异步，作业反复重启可能让 L0 持续堆积。

## 16. TableCommitImpl：提交与 maintain

### ① 要解决的问题
原子提交 + 自动维护（Tag 创建/过期、分区过期、Consumer 过期、Snapshot 过期），且批/流采用不同同步/异步策略。

### ② 设计原理与取舍
`commitMultiple`（`table/sink/TableCommitImpl.java:237-271`）：非 overwrite 逐个 `FileStoreCommit.commit` 累加新快照数，再 `maintain`；overwrite 模式 `committables.size()>1` 抛异常（`:251-254`），走 `overwritePartition`（`:264-265`）。

执行模式由 `maintain(identifier, executor, doExpire)`（`:360-378`）决定：批式 `batchCommitted=true` 当前线程同步执行；流式提交到单线程 `maintainExecutor` 异步，**异步异常存入 `maintainError`，下次 commit 入口抛出**。

### ③ 关键源码
```java
// TableCommitImpl.java:360-378  执行模式 + 异步错误回传
private void maintain(long identifier, ExecutorService executor, boolean doExpire) {
    if (maintainError.get() != null) throw new RuntimeException(maintainError.get()); // 上次异步异常滞后抛出
    if (batchCommitted) maintain(identifier, doExpire);                                // 批：同步
    else executor.execute(() -> { try { maintain(identifier, doExpire); }
                                  catch (Throwable t) { maintainError.compareAndSet(null, t); } }); // 流：异步
}
// TableCommitImpl.java:380-405  5 任务严格顺序
private void maintain(long identifier, boolean doExpire) {
    if (tagAutoManager != null) { tagAutoCreation.run();                       // 1 Tag 自动创建（须先于过期，引用最新快照）
                                  if (doExpire) tagTimeExpire.expire(); }       // 2 Tag 过期
    if (doExpire && partitionExpire != null) partitionExpire.expire(identifier); // 3 分区过期
    // expire consumer first to avoid preventing snapshot expiration
    if (doExpire && consumerExpireTime != null)                                  // 4 Consumer 过期（须先于快照）
        consumerManager.expire(LocalDateTime.now().minus(consumerExpireTime));
    // snapshot expiration usually take the most time, so put it at the end
    if (doExpire && expireSnapshots != null) expireSnapshots.run();             // 5 Snapshot 过期（最耗时，置末）
}
```

顺序的三条理由（均有源码注释佐证）：Tag 创建须先于过期（否则可能引用已删快照）；Consumer 过期须先于 Snapshot 过期（否则 Consumer 引用已删快照、消费失败）；Snapshot 过期最耗时置末，避免阻塞前序。

### ④ 风险/陷阱/边界
- overwrite 模式只能单次提交。
- 异步 maintain 异常滞后一次 commit 才暴露——监控需关注 commit 抛错而非仅看本次。
- 空提交 `commit(emptyList)` 仍可触发 maintain（受 `expireForEmptyCommit` 控制），可用于纯过期。
- `expireSnapshots` 若 `changelog-lifecycle-decoupled=true` 还独立清理 changelog。

### ⑤ 收益与代价
收益：免后台守护进程，流式不阻塞写。代价：异步错误发现晚；S3 等对象存储删文件慢，Snapshot 过期可能长尾，需调维护超时。

### ⑥ 深入：五个 maintain 任务各自的机制

- **Task1 Tag 自动创建**（`TagAutoCreation`）：按 `tag.creation-mode`（process-time / watermark）与周期（daily/hourly）在跨越时间边界时打 Tag，Tag 指向当时的快照。`tag.automatic-creation` 开启。Tag 是"被命名的快照"，过期不会回收被 Tag 引用的快照（Tag 是快照的强引用）。
- **Task2 Tag 过期**（`TagTimeExpire`）：按 `tag.num-retained-max` / `tag.default-time-retained` 删超期 Tag，解除其对快照的强引用。
- **Task3 分区过期**（`PartitionExpire`）：按 `partition.expiration-time` + `partition.timestamp-formatter`（从分区值解析时间）删超期分区，经 Commit 产生 OVERWRITE/DELETE。
- **Task4 Consumer 过期**：删超 `consumer.expiration-time` 未活动的 Consumer 记录。Consumer 记录 `nextSnapshot`，是快照的强引用（防止还在消费的快照被回收）——故必须先于 Task5。
- **Task5 Snapshot 过期**（`ExpireSnapshotsImpl`）：按 `snapshot.num-retained-min/max` + `snapshot.time-retained` 保留窗口，删窗口外快照，并物理删除"不再被任何保留快照/Tag/Consumer 引用"的数据/manifest/index 文件。`changelog-lifecycle-decoupled=true` 时 changelog 独立按 `changelog.num-retained` 过期，使流读可保留更长 changelog 而数据更早回收。

**引用计数式安全回收**：快照过期删文件前要确认该文件不被任何"存活快照"引用（manifest 中仍有未被 DELETE 的 ADD entry）。这就是为何 Tag/Consumer（强引用）必须先处理——它们决定哪些快照"存活"。

### ⑦ 风险/陷阱/边界（maintain 进阶）
- `snapshot.num-retained-min` 是硬下限，即使超时也保留，防止过度回收导致流读断点丢失。
- 分区过期依赖能从分区值解析出时间，非时间分区表不可用。
- S3 上 Task5 删大量小文件是长尾，可调 `snapshot.expire.limit`（单次过期快照数上限）分摊。

## 17. 乐观并发控制与冲突检测

### ① 要解决的问题
多 Writer 并发提交时保证一致性，又不引入 ZooKeeper 等分布式锁的复杂度与延迟。

### ② 设计原理与取舍
基于**文件系统原子 rename**：`snapshot-{id}` 文件名递增，只有一个 Writer 能成功创建某 id，其余收到 `FileAlreadyExistsException` 触发重试（rebase）。`tryCommit` 循环：

```
tryCommit(committable):
  1 读 latest snapshot
  2 CommitMessage → ManifestEntry：newFiles→ADD、compactBefore→DELETE、compactAfter→ADD
  3 合并到 ManifestFile（过大触发 manifest compaction）
  4 建 Snapshot 对象（snapshot-{latest+1}，引用 base/delta/changelog manifestList）
  5 原子写：rename(tmp, snapshot-{id})
      └ FileAlreadyExists → 读冲突快照 → conflictCheck → 无冲突 rebase 重试 / 有冲突抛 ConflictException
  6 触发 CommitCallback（如 Hive Metastore 更新分区）
```

冲突三类：
- **APPEND**：新文件与冲突快照文件 key 范围重叠；追加表 `write.append-commit-check-conflict=false` 时跳过。
- **COMPACT**：compactBefore 要删的文件已被他人删（两个 compaction 抢同一文件）。
- **OVERWRITE**：不检测，总成功。

### ④ 风险/陷阱/边界
- 多数冲突自动 rebase 重试，**不抛异常**；只在达 `commit.retry.max-attempts`（默认 3）或不可 rebase 时抛 `ConflictException`，需捕获处理。
- 不同作业误用相同 `commitUser` → 被当作同一 Writer 重试，冲突检测可能失效、数据覆盖。生产建议 `jobId + taskId`。
- S3 等 rename 非严格原子的存储需依赖条件写/外部一致性保障。
- 追加表默认不检测冲突（无 key 重叠通常安全），需严格则开 `write.append-commit-check-conflict=true`。

### ⑤ 收益与代价
收益：无锁、纯文件系统、重试只重算 Manifest 不重写数据，不依赖外部存储（对比 Iceberg 依赖 metadata.json 版本号/Hive Metastore）。代价：高冲突场景重试增延迟；对象存储原子性需额外保障。

### ⑥ 深入：tryCommitOnce 与 manifest 合并

`FileStoreCommitImpl.tryCommit`（`:692+`）循环调 `tryCommitOnce`（`:799+`），单次提交的关键步骤：
1. **CommitMessage → ManifestEntry**：`DataIncrement.newFiles`→ADD、`CompactIncrement.compactBefore`→DELETE、`compactAfter`→ADD、index 文件入 indexManifest。
2. **生成 delta manifest**：把本次 entry 写成新 manifest 文件，连同上一快照的 base 一起组成新快照的 `baseManifestList`（`:927`）+ `deltaManifestList`（`:928`）。
3. **manifest 合并**（`:958-959`）：当小 manifest 数超 `manifest.merge-min-count` 或单个超 `manifest.target-file-size` 时触发 manifest compaction（合并小 manifest，避免 §6.5 第一层裁剪扫太多文件）——这是 `$manifests` 膨胀的自愈机制。
4. **原子写快照**：`rename(tmp, snapshot-{id})`，失败则 `RetryCommitResult` 回到循环、读冲突快照做检测。

**overwrite 路径**（`overwritePartition`，`:418+`）：先生成"删除目标分区现有全部文件"的 DELETE entry，再叠加新文件 ADD entry，原子提交。`commitKind=OVERWRITE`，不做 APPEND/COMPACT 冲突检测（§17 三类），但只能单次（§16 `committables.size()>1` 抛错）。流式消费侧据 `commitKind` 决定是否产出（§5.6）。

**冲突检测落点**：rebase 时 `noConflictsOrFail`/`assertNoConflicts` 比对当前 committable 与冲突快照的 ManifestEntry：APPEND 看 key 范围重叠（主键表按 (partition,bucket)）、COMPACT 看 compactBefore 文件是否还在最新快照（被他人删则冲突）。无冲突则把本次 entry 接到冲突快照之后重写 delta，**不重写数据文件**——这是重试代价低的根本原因。

### ⑦ 风险/陷阱/边界（提交内核）
- manifest compaction 触发时 commit 变重（要重写 manifest），表现为偶发 commit 延迟尖刺。
- `commit.max-retries` 耗尽抛 `RuntimeException`（含 `ConflictException`），上游须有重启/告警。
- overwrite 与并发写同分区：overwrite 不检测冲突，可能覆盖并发写入的数据，生产应避免对热写分区做 overwrite。

## 18. Deletion Vectors 在读写路径的集成

### ① 要解决的问题
让主键表 MoR 在支持删除/更新的同时实现读时直读（rawConvertible），用 bitmap 标删除行替代读时 merge。

### ② 设计原理与取舍
**写路径**：`deletion-vectors.enabled=true` 时 `MergeTreeWriter` 经 `LookupCompactedFileWriter` 用 `LookupLevels.lookup(key)` 定位旧记录 (fileName,position)，`dvMaintainer.notifyNewDeletion` 在该文件 Roaring Bitmap 标删；prepareCommit 时 `writeDeletionVector` 序列化 bitmap，产 `IndexFileMeta`（含 `dvRanges: fileName→DeletionVectorMeta`）入 `DataIncrement.newIndexFiles`。

**读路径**：`SnapshotReaderImpl.generateSplits` 调 `scanDvIndex` 读 `snapshot.indexManifest`，解析出 `Map<(partition,bucket), Map<fileName, DeletionFile>>`，与每个 DataSplit 的 dataFiles 一一对应挂上；`RawFileSplitRead` 为有 DV 的文件包 `ApplyDeletionVectorReader`，逐行 `dv.isDeleted(position)` 跳行。

### ③ 关键源码
`scanDvIndex`（`SnapshotReaderImpl.java:631+`）带缓存：

```
scanDvIndex(snapshot, buckets):
  Step1 查缓存：dvMetaCache.read(indexManifestPath, partition, bucket)
        命中 → 计 hit、从待读集移除；未命中 → 计 miss
  Step2 读文件：indexFileHandler.scan(snapshot, DELETION_VECTORS_INDEX, partitions) → IndexFileMeta 列表
  Step3 解析：dvRanges → Map<fileName, DeletionFile(path, offset, length, cardinality)>
  Step4 写回缓存：key = indexManifestPath + partition + bucket
```

缓存避免流式多 checkpoint 反复读同一快照 DV，命中率是关键性能指标。

### ④ 风险/陷阱/边界
- DV 写放大：每次更新需 lookup 旧位置（RocksDB/内存索引）并重写 bitmap，写成本高于纯 LSM。
- 被删比例高时 raw 直读仍读全文件再丢弃行，省 CPU 不省 IO（§9）。
- DV 元数据未缓存时每 (partition,bucket) 一次文件 IO，宽分区场景计划期 IO 高。

### ⑤ 收益与代价
收益：主键表批查直读、聚合可下推（§12），读性能接近 COW。代价：写放大 + DV 索引存储 + lookup 索引内存。

### ⑥ 深入：DV 的位图结构、生命周期与索引模型

**位图结构**：DV 用 Roaring Bitmap（`Bitmap64DeletionVector`/32 位变体），按数据文件内 row position 标删除。Roaring 对稀疏/稠密区间自适应编码，序列化后体积远小于 boolean 数组；`cardinality`（删除行数）随位图一起持久化，供 §7 `mergedRowCount` 精确档与 §12 `COUNT(*)` 下推直接使用，无需解位图。

**DV 与数据文件的绑定**：一个数据文件对应一个 DV（多个 DV 物理上打包进同一 index 文件，靠 `DeletionVectorMeta{offset,length,cardinality}` 在 `dvRanges` 中按 fileName 寻址）。这就是为何 `scanDvIndex` 产出 `Map<fileName, DeletionFile>`，再与 split 的 dataFiles 一一对应（§6 `generateSplits`）。

**生命周期**：compaction 重写数据文件时，旧文件的 DV 随旧文件一起被 DELETE，新文件按合并后的删除状态生成新 DV——即"删除信息在 compaction 时物化进数据、DV 重置"。这避免 DV 无限膨胀，但也意味着 DV 表更依赖 compaction（删除比例高时及时 compact 才能回收空间）。

**与文件索引的关系**：DV（删除标记）与 file-index（bloom/bitmap/bsi，过滤索引）都存于 index/ 目录、都经 `indexManifest` 引用，但用途正交：DV 用于读时跳已删行、file-index 用于读前跳不匹配行（§12）。两者可同时存在。

### ⑦ 风险/陷阱/边界（DV 进阶）
- DV 表的 `first-row`/`partial-update` 等引擎与 DV 的交互需逐配置验证；并非所有 merge-engine 都支持 DV。
- `compaction.max.file-num` 等若让 compaction 长期不触发，DV 累积导致"读全文件丢大部分行"的极端读放大。
- 旧版本数据文件无 `deleteRowCount` 时，§10 raw provider 会拒绝直读退回 merge，DV 加速失效——升级后需 rewrite。

## 19. System Tables 与线上排障

### ① 要解决的问题
把 manifest/snapshot/file/index 等内部元数据以 SQL 表暴露，供调优与排障，无需读二进制元文件。

### ② 设计原理与取舍
`SystemTableLoader.SYSTEM_TABLE_LOADERS`（`table/system/SystemTableLoader.java:61-80`）注册 **18 张表级系统表**（`table$xxx` 访问），多数实现 `ReadonlyTable + InnerTable` 复用读取脚手架。`GLOBAL_SYSTEM_TABLES`（`:85-86`）4 张全局表（`sys.xxx`）。

### ③ 关键源码（系统表全集，已据 `:61-86` 核对）
表级（18 张）：`$manifests $snapshots $options $schemas $partitions $buckets $audit_log $files $file_key_ranges $tags $branches $consumers $read_optimized $aggregation_fields $statistics $binlog $table_indexes $row_tracking`。

全局（4 张）：`all_tables all_partitions all_table_options catalog_options`。

**两张高频排障表的列模式**（直接来自源码 `DataField` 定义，列名/可空性可据此写精确 SQL）：

`$snapshots`（`SnapshotsTable.java:87-111`）：`snapshot_id, schema_id, commit_user, commit_identifier, commit_kind(append/compact/overwrite/analyze), commit_time, base_manifest_list, delta_manifest_list, changelog_manifest_list, total_record_count, delta_record_count, changelog_record_count, watermark, next_row_id`。回滚/时间旅行靠 `snapshot_id`；判断"上次提交是写还是 compaction"靠 `commit_kind`；估算数据量靠 `*_record_count`。

`$files`（`FilesTable.java:101-126`）：`partition, bucket, file_path, file_format, schema_id, level, record_count, file_size_in_bytes, min_key, max_key, null_value_counts, min_value_stats, max_value_stats, min/max_sequence_number, creation_time, deleteRowCount, file_source, first_row_id, write_cols`。读放大排查的黄金表：`level` 看 LSM 分层（L0 过多=compaction 落后）、`deleteRowCount` 看 DV 删除占比、`record_count/file_size` 看小文件、`min_key/max_key` 看 key 范围重叠（决定 rawConvertible，§9）。

排障对照：
| 现象 | 查 | 看什么 |
|---|---|---|
| 读慢/读放大 | `$files` | level 分布、L0 文件数、单文件 rowCount/大小 |
| 计划慢/manifest 膨胀 | `$manifests` | manifest 文件数与大小 |
| 时间旅行/回滚 | `$snapshots` | id、commitKind(APPEND/COMPACT/OVERWRITE)、schemaId、记录数 |
| 流消费卡住 | `$consumers` | nextSnapshot 是否前进 |
| DV/索引未生效 | `$table_indexes` | DV/Bloom/Bitmap 是否存在 |
| Schema 变更审计 | `$schemas` | 字段/主键/分区演进历史 |
| 最优读不滞后 | `$read_optimized` | 仅最高 level 视图 |
| CDC 审计 | `$audit_log` / `$binlog` | rowkind 列 / binlog 数组 |
| key 分布/倾斜 | `$file_key_ranges` / `$buckets` | 每文件 key 范围 / 桶数据量 |

### ④ 风险/陷阱/边界
- **`$compact_buckets`（CompactBucketsTable）未注册在 `SYSTEM_TABLE_LOADERS`**：它是 compactor 内部用（构造需 `isContinuous` 参数），**不能**经 `table$compact_buckets` SQL 访问——旧文档将其列为第 19 张用户系统表，本文据 `:61-80` 修正剔除。
- 系统表只读，`newWrite/newCommit` 抛 `UnsupportedOperationException`。
- `$files` 等在超大表上自身扫描成本不低，排障也要带分区谓词。

### ⑤ 收益与代价
收益：纯 SQL 排障，免解析元文件。代价：部分系统表扫描成本高，需带过滤。

## 19.5 SQL 功能面到源码机制映射

把官方 flink/spark 的 query/write SQL 功能点逐一落到本文已剖析的源码机制——读完应知"每条 SQL 走哪条路径、受哪个配置控制、有何边界"。

### 读侧（sql-query）

| SQL 功能 | 源码机制 | 关键配置/边界 |
|---|---|---|
| Batch Query（全量） | `FullStartingScanner` + `read()` 产 DataSplit（§5/§6） | 默认 `scan.mode=latest-full` |
| Batch Time Travel | `StaticFromSnapshot/Tag/Timestamp StartingScanner`（§5） | `VERSION AS OF` / `scan.snapshot-id` / `scan.tag-name` / `scan.timestamp-millis` |
| Batch Incremental | `IncrementalDeltaStartingScanner`（原始增量）（§5） | `incremental-between='start,end'`，跳过 COMPACT/OVERWRITE |
| Incremental between Tags | `IncrementalDiffStartingScanner`（默认 diff）（§5） | `incremental-between-tag-to-snapshot` 切换 delta/diff |
| Streaming Query | `ContinuousLatestStartingScanner` + FollowUpScanner 循环（§5.6） | 默认从 latest+1，配 `consumer-id` 持久化进度 |
| Streaming Time Travel | `ContinuousFromSnapshot/Timestamp/SnapshotFull StartingScanner`（§5） | `scan.snapshot-id` + streaming；FROM_SNAPSHOT_FULL 先全量再增量 |
| Read Overwrite（流读覆写） | §5.6 OVERWRITE 快照处理 | `streaming-read-overwrite`（PK）/`streaming-read-append-overwrite`（Append），默认跳过 |
| Read Parallelism | Split 数（batch）/ bucket 数（stream）推导（§7/§8） | `scan.infer-parallelism`(默认 true)、`scan.infer-parallelism.max`(1024)、`scan.parallelism` |
| Query Optimization（data skipping） | 三层谓词下推 + BucketSelector（§11/§6.5） | 加速谓词 `= < <= > >=`；分区 + PK + bucketKey 过滤最有效 |
| Dedicated Split Generation | `MergeTree/AppendOnly SplitGenerator`（§8） | `source.split.target-size`、`source.split.open-file-cost` |
| COUNT(*) / MIN / MAX 下推 | `mergedRowCount` 精确档 + valueStats（§7/§12） | 仅 DV 表精确；Spark 默认 stats-mode 支持 MIN/MAX |

### 写侧（sql-write）

| SQL 功能 | 源码机制 | 关键配置/边界 |
|---|---|---|
| INSERT INTO | `TableWriteImpl` 6 步流水线 + FileStoreWrite（§13/§15） | 主键表转 KeyValue；追加表仅 INSERT |
| Clustering（追加表聚簇） | 写入按聚簇列排序后装箱 | `sink.clustering.by-columns`，提升 data skipping |
| INSERT OVERWRITE（整表/分区） | `FileStoreCommit.overwritePartition`（§17.6） | `commitKind=OVERWRITE`，仅单次提交 |
| Dynamic Overwrite | overwrite 仅删命中分区 | Flink `dynamic-partition-overwrite`（默认动态）；静态需显式 PARTITION |
| Truncate / Purge Partitions | overwrite 空数据 / 分区删除 | 经 Commit 原子替换 |
| UPDATE / DELETE FROM | 主键表转 `-U/+U` / `-D` 经 MergeFunction（§13/§10.6） | 追加表不支持（`rowKind.isAdd` 校验） |
| Partition Mark Done | Commit 后 CommitCallback 触发 | `partition.mark-done-action` |
| 流式 exactly-once | `commit.commit(messages)` per checkpoint + commitUser 幂等（§16/§17） | 唯一 `commitUser`，`commitIdentifier`=checkpointId |

### 关键澄清（doc 未明说、源码可证）
- **time travel 的实现差异**：批式 time travel 走 `Static*StartingScanner` 一次性读指定快照全量；流式 time travel 走 `Continuous*StartingScanner` 从指定点起增量。两者返回类型不同（ScannedResult vs NextSnapshot，§5）。
- **INSERT OVERWRITE 对并发写不安全**：overwrite 不做冲突检测（§17.6），与并发写同分区会互相覆盖。
- **流读看不到 OVERWRITE**：默认 `streaming-read-overwrite=false`，§5.6 状态机直接 `nextSnapshotId++` 跳过——这是常见困惑点。
- **Read Parallelism 推导**：batch=split 数（受 SplitGenerator 的 target-size 影响）、stream=bucket 数（受桶数与 `infer-parallelism.max` 上限影响），改并行度本质是改桶数或 split target-size。

## 21. 端到端追踪与读写放大调优速查

本节把前 20 节串成两条可在线上对照的完整调用链，并给出读写放大的调优旋钮，作为"拿到一个慢查询/写入抖动该看哪里"的速查。

### ① 读路径端到端追踪：`SELECT c FROM pk_t WHERE dt='d1' AND pk=100 AND amount>5`

```
Driver(计划):
 1 table.newReadBuilder().withFilter(p).withProjection([c])      ReadBuilderImpl 收参

 4 SnapshotReaderImpl.withFilter(p)                                §6
   ├ 拆分：dt='d1'→partitionFilter；pk=100,amount>5→nonPartition

Worker(执行, 每 split 一个):
 8 read.createReader(split) → KeyValueTableRead                    §10
   ├ ① RawFileSplitReadProvider.match：!forceKeepDelete&&!streaming&&rawConvertible&&有deleteRowCount
   │    命中 → RawFileSplitRead 直读 Parquet/ORC（向量化）
```

### ③ 读写放大调优速查

| 目标 | 旋钮 | 机制 | 代价 |
|---|---|---|---|
| 降读放大（PK 表） | `deletion-vectors.enabled=true` | raw 直读（§9） | 写放大↑、DV 存储 |
| 降读放大 | 提高 compaction 频率（`num-sorted-run.compaction-trigger`↓） | 减少 L0 run、更多 raw split | 写放大↑ |
| 降读放大（不要实时） | `$read_optimized` 表 | 只读最高 level（§12） | 数据滞后 |
| 控并行/split 大小 | `source.split.target-size` | SplitGenerator 装箱（§8） | 过小调度开销、过大倾斜 |
| 控流读并行 | bucket 数 / `scan.infer-parallelism.max` | stream 并行=bucket 数（§19.5） | 桶数定后难改 |
| 加速过滤 | 分区 + bucketKey + PK 上加等值谓词；file-index | 三层裁剪 + BucketSelector（§11/§6.5） | bucketKey 选错则无效 |
| 降写放大 | 增大 `write-buffer-size` | 减少 flush 次数与小文件 | 内存↑、flush 长尾 |
| 降 commit 延迟 | 流式 `waitCompaction=false` | compaction 异步（§15） | L0 可能堆积 |
| 降冲突重试 | 唯一 commitUser、写不同 (partition,bucket) | 乐观并发（§17） | — |
| 控 manifest 膨胀 | `manifest.merge-min-count`/`manifest.target-file-size` | manifest 合并（§17.6） | 合并时 commit 变重 |
| 防流读丢数据 | `consumer-id` + `consumer.expiration-time` | Consumer 阻止快照回收（§16/§5.6） | 快照延迟回收 |

## 22. 设计决策总结

| # | 决策 | 核心权衡 | 收益 | 代价 |
|---|---|---|---|---|
| 1 | Table 5 层继承 + 末端两子类 + 模板方法 | 编译期类型安全 vs 运行时灵活 | 消除海量 isPrimaryKey 分支、接口隔离、易扩展表类型 | 继承链深、行为定位需懂模板方法 |
| 2 | Scan/Read 分离 | 分布式友好 vs 概念简单 | 计划期全局优化 + Worker 纯执行、Split 可序列化 | 用户须理解两对象不可混用 |
| 3 | 计划期 Limit/TopN 下推 | 计划期 vs 执行期优化 | 少生成/少分发 Split | Scan 阶段加载统计、TopN n≤100 封顶 |
| 4 | StartingScanner 16+ 策略 | 策略模式 vs 巨型 switch | 每语义独立可测可扩展、delta manifest O(1) 增量定位 | 定位"走哪个 scanner"需回溯多配置 |
| 5 | IntervalPartition + BinPacking 切 Split | 正确性(key 不跨 split) vs 并行均衡 | LSM 并行读正确、split 趋近目标大小 | 文件多时计划期 CPU 高 |
| 6 | mergedRowCount 三档算法 | 精确 vs 诚实放弃 | 能精确则精确、能估算则估算、否则不误截断 | 精确档依赖 DV 元数据完整 |
