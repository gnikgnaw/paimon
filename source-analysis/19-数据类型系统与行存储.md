# Apache Paimon 数据类型系统与行存储深度分析

> **版本**：1.5-SNAPSHOT　**源码模块**：`paimon-api`（类型定义）、`paimon-common`（行存储/序列化/casting/codegen）、`paimon-core`（KeyValue）　**核对日期**：2026-06

**一句话定位**：Paimon 用一套**与计算引擎解耦的类型系统**（`paimon-api/types`）描述列的语义，用一族**面向内存的行表示**（`paimon-common/data` 的 `InternalRow` 实现）在热路径上零拷贝地搬运数据——前者保证 Flink/Spark/Hive 写出来的数据语义一致、可演化，后者保证每行的读写、排序、序列化都不为通用性付出运行时代价。

读完本文你应能回答：① 为什么类型定义要单独放在 `paimon-api` 而不是和实现放一起；② `DataTypeRoot` 与 `DataTypeFamily` 两层枚举各自在解决什么；③ `BinaryRow` 的固定 8 字节槽 + 变长区两层布局，为什么能同时支持 O(1) 投影、字节序比较和零拷贝序列化；④ 短变长数据"7 字节内联"省掉了什么开销、边界在哪；⑤ `Decimal`/`Timestamp` 的"紧凑 / 非紧凑"双模存储分界（18 / 3）从何而来、配错会怎样；⑥ `KeyValue` 为什么把 key/value 拆开、`_SEQUENCE_NUMBER`/`_VALUE_KIND` 为什么夹在中间；⑦ `BinaryRowSerializer` 和 `RowCompactedSerializer` 各自牺牲了什么换什么；⑧ Flink/Spark 适配层哪些类型真零拷贝、哪些必须拷贝、为什么不能直接引用对方的内存段。

> 阅读约定：本文每个机制按"① 要解决什么问题 → ② 设计原理与取舍（含对比表）→ ③ 关键源码（精选片段 + `路径:行号`）→ ④ 风险/陷阱/边界 → ⑤ 收益与代价"组织。源码行号以本次核对（commit `e76fc41b7`）为准；与旧稿不符处用 `（已修正）` 标注。文件格式的列式落盘编码（ORC/Parquet 物理类型、向量化读）主讲 **20**，本篇只讲到"内存类型 → 文件类型"的映射边界并引用之；Schema 演化中的类型提升规则主讲 **22**，本篇只讲 casting 机制本身。

## 1. 快速理解（核心问题 / 概念速查 / 高频陷阱）

### 1.1 核心问题：类型系统与行存储各自在解决什么

本篇其实是两个相邻但目标不同的子系统：

**① 类型系统（`paimon-api/types`）要解决什么**

湖仓表会被 Flink、Spark、Hive 等多个引擎同时读写。若每个引擎各用自己的类型系统，跨引擎传数据时语义就会漂移：Flink 的 `TIMESTAMP(9)` 写出去、Spark 只支持到微秒读回来；`DECIMAL(10,2)` 被某一侧当成 `DECIMAL(10,3)`，金额直接错位。Paimon 的对策是把"列的类型语义"沉淀成一份**与引擎无关的契约**，放在最底层、依赖最少的 `paimon-api` 里，引擎只需各自实现"我的类型 ↔ Paimon 类型"的双向映射即可。类型系统还要服务于 Schema 演化（哪些类型变更是安全的）与谓词下推（"这是不是一个数值类型"这类批量判断）。

**② 行存储（`paimon-common/data`）要解决什么**

存储引擎在 LSM compaction、排序归并、文件读写这些热路径上，每秒要搬运数千万行。若用 `Object[]` 表示一行，序列化要逐字段拆箱、比较要逐字段装箱拆箱、GC 压力随行数线性上涨。行存储的对策是：用 `InternalRow` 抽象"一行"的读写契约，再针对不同场景给出不同实现——文件 I/O 用紧凑二进制的 `BinaryRow`（序列化即内存拷贝、比较即字节比较），内存构造用 `GenericRow`，key/value 拼接用零拷贝的 `JoinedRow`。**一句话设计哲学：类型系统为"跨引擎正确性"服务，行存储为"单机热路径性能"服务，两者通过 `InternalRow.createFieldGetter(DataType, pos)` 这类按类型生成的访问器衔接。**

|------|-----------|---------|

### 1.3 高频生产陷阱

**陷阱 1：用 `==` 比较 `DataType` 或忽略 nullability。** `DataType` 的 `equals` 同时比较 `isNullable` 与 `typeRoot`（`DataType.java:121-131`），`DecimalType(10,2)`（默认 nullable）与 `DecimalType(false,10,2)` 不相等。要忽略可空性比较时用 `equalsIgnoreNullable()`（`DataType.java:138`）。直接 `==` 永远 false。

**陷阱 2：Decimal/Timestamp 精度跨过"紧凑分界"导致性能与存储突变。** `DECIMAL(18,s)` 是紧凑模式（一个 `long`，可内联进 BinaryRow 固定槽），`DECIMAL(19,s)` 退化为 `BigDecimal` 且数据落到变长区（`Decimal.isCompact`:135，分界 `MAX_COMPACT_PRECISION=18`，`DecimalType.java:35`）。同理 `TIMESTAMP(3)` 紧凑、`TIMESTAMP(4)` 非紧凑（`Timestamp.isCompact`:241）。建模时若精度可控，压到分界内可显著降 GC 与存储。

**陷阱 3：BinaryString/BinaryRow 脱离其 `MemorySegment` 生命周期使用。** 这些对象只是指向内存段的"视图"（`BinarySection` 持 `segments/offset/sizeInBytes`，`BinarySection.java:64-66`）。读出的 `BinaryString` 若在底层段被复用/回收后再 `toString()`，会读到脏数据。热路径上复用 reader 缓冲行时尤其要注意：要么立即 `toString()` 物化，要么 `copy()`。

**陷阱 4：误以为 BinaryRow 变长字段能原地更新。** 固定长度字段（int/long/double/紧凑 Decimal）可 `setInt` 原地改（`BinaryRow.java:148-160`），但变长字段（String、外置 Decimal）一旦写定，长度变化就会破坏后续布局，必须重建整行。

**陷阱 5：多线程共享有状态 Serializer。** `InternalRowSerializer` 内部持有 `RowHelper`（含可复用的 `BinaryRowWriter`），非线程安全（`InternalRowSerializer.java:43`）。每线程应 `duplicate()` 一份。

**陷阱 6：`TIMESTAMP` 与 `TIMESTAMP_LTZ` 混用导致跨时区错乱。** 两者 `DataTypeRoot` 不同（`TIMESTAMP_WITHOUT_TIME_ZONE` vs `TIMESTAMP_WITH_LOCAL_TIME_ZONE`，`DataTypeRoot.java:94-101`），内部都是 `Timestamp`，但语义一为"无时区墙上时间"、一为"带本地时区瞬时点"。Spark 适配层会对后者做 UTC 调整（见 §11.2），引擎侧时区配置不一致就会偏移。

**陷阱 7：高精度 Decimal 的外置字节上限是 16 字节。** `setDecimal` 在非紧凑分支断言 `bytes.length <= 16`（`BinaryRow.java:191`），即 unscaled 值的字节表示不超过 16 字节——这正好覆盖 `DECIMAL(38)`，但说明 Paimon 的 Decimal 上限就是 38（`DecimalType.MAX_PRECISION=38`，`DecimalType.java:39`），不可能再高。

## 2. 类型系统：DataType / Root / Family / Visitor

### 2.1 DataType 继承体系与双层模块设计

**① 要解决什么问题。** 类型定义是引擎与 Paimon 之间的"公共契约"。如果把它和存储实现（`paimon-core`）放在一起，外部 Connector 想引用 `DataType` 就得拖入整个核心引擎及其传递依赖。

**② 设计原理与取舍。** Paimon 把类型定义下沉到依赖最少的 `paimon-api`，类型检查/大小估算等"操作"放到上一层 `paimon-common`。这是典型的"稳定接口与易变实现分离"：

| 放置方案 | 外部依赖重量 | 演化耦合 | Paimon 选择 |
|----------|--------------|----------|-------------|
| 类型与存储实现同层 | 重（拖入 core） | 高 | 否 |
| 类型在独立 api 层、操作在 common 层 | 轻（只依赖 api） | 低 | **是** |

`DataType` 是抽象基类，只持有两个不可变字段 `isNullable` 与 `typeRoot`，并声明 4 个抽象方法：`defaultSize()`（大小估算）、`copy(boolean)`（改可空性深拷贝）、`asSQLString()`（SQL 文本）、`accept(DataTypeVisitor)`（Visitor 入口）。继承树覆盖布尔/整数/浮点/Decimal/字符/二进制/日期时间/数组/向量/多重集/Map/Row/Variant/Blob 全部类型，类文件都在 `paimon-api/types/` 下（如 `IntType.java`、`DecimalType.java`、`RowType.java`、`VariantType.java`、`BlobType.java`、`VectorType.java`）。

**③ 关键源码。** 基类字段与相等性（`DataType.java:43-49`、`121-131`）：

```java
// paimon-api/.../types/DataType.java:43
private final boolean isNullable;
private final DataTypeRoot typeRoot;
// equals 同时比较 isNullable 与 typeRoot（:130）
return isNullable == that.isNullable && typeRoot == that.typeRoot;
```

`is(DataTypeRoot)`/`isAnyOf(...)`/`is(DataTypeFamily)`（`DataType.java:69-99`）是上层做类型判断的入口，其中 family 判断委托给 `typeRoot.getFamilies()`。

**④ 风险 / 陷阱 / 边界。** 所有类型默认 `isNullable=true`；`equals` 把可空性算进去（见 §1.3 陷阱 1）。`equalsIgnoreFieldId` 默认等价于 `equals`，仅 `RowType` 等含字段 ID 的类型才覆写（用于 schema 演化比较，详见 22）。

**⑤ 收益与代价。** 收益：Connector 仅依赖轻量 api，类型契约稳定；新增类型只动 api 层。代价：多一个模块边界，类型定义与"对类型的操作"被强制分离，写跨层逻辑时要在两个模块间跳转。

### 2.2 DataTypeRoot 与 DataTypeFamily 两层枚举

**① 要解决什么问题。** 对类型有两种截然不同的判断需求：精确分发（"它到底是 INT 还是 BIGINT"，决定走哪条序列化分支）与批量归类（"它是不是数值类型"，决定能否套用某条 cast 规则或下推某个谓词）。只用一层枚举满足不了。

**② 设计原理与取舍。** `DataTypeRoot` 是精确标识（switch/case 友好），`DataTypeFamily` 是分组标签；**每个 Root 在定义处就静态声明自己属于哪些 Family**，一个 Root 可属多个 Family。这样"是否数值类型"退化为一次集合包含判断，而不必枚举所有数值 Root。

**③ 关键源码。** Family 在枚举常量声明处绑定（`DataTypeRoot.java:60-84`）：

```java
// paimon-api/.../types/DataTypeRoot.java:60
DECIMAL(PREDEFINED, NUMERIC, EXACT_NUMERIC),
TINYINT(PREDEFINED, NUMERIC, INTEGER_NUMERIC, EXACT_NUMERIC),
// ...
INTEGER(PREDEFINED, NUMERIC, INTEGER_NUMERIC, EXACT_NUMERIC),
```

构造器把首个 + 其余 Family 收进不可变 `EnumSet`（`DataTypeRoot.java:119-125`），`DataType.is(DataTypeFamily)` 即查该集合。

**④ 风险 / 陷阱 / 边界。** Family 归属是写死在 Root 上的，新增类型时必须同时维护 Root 与其 Family 列表，漏配会让 cast/谓词逻辑"看不见"新类型。注意 `MAP` 的 Family 是 `CONSTRUCTED, EXTENSION`、`ROW` 仅 `CONSTRUCTED`、`TIMESTAMP_WITH_LOCAL_TIME_ZONE` 含 `EXTENSION`（`DataTypeRoot.java:97-115`，旧稿注释把 MAP 单标 EXTENSION、ROW 标 CONSTRUCTED，已按源码修正）。

**⑤ 收益与代价。** 收益：一条 cast 规则可凭 Family 覆盖一整类类型（如 `NumericPrimitiveCastRule` 覆盖所有整数互转，见 §10），谓词下推可批量判类。代价：双重枚举的维护成本，且 Family 集合是编译期固定的，无法运行时扩展。

### 2.3 完整类型清单与内部表示

下表是"SQL 类型 → DataTypeRoot → 内存内部表示"的对照（内部表示均为 `paimon-common/data` 下的类）：

| SQL 类型 | DataTypeRoot | Java 内部表示 | 参数 | 紧凑/内联 |
|---------|-------------|--------------|------|-----------|
| BOOLEAN | BOOLEAN | boolean | — | 内联（8 字节槽） |
| TINYINT/SMALLINT/INT/BIGINT | 对应 Root | byte/short/int/long | — | 内联 |
| FLOAT/DOUBLE | FLOAT/DOUBLE | float/double | — | 内联 |
| DECIMAL(p,s) | DECIMAL | `Decimal` | precision,scale | p≤18 内联，否则外置 |
| CHAR(n)/VARCHAR(n) | CHAR/VARCHAR | `BinaryString` | length | ≤7 字节内联，否则外置 |
| BINARY/VARBINARY(n) | BINARY/VARBINARY | byte[] | length | 同上 |
| DATE | DATE | int（epoch day） | — | 内联 |
| TIME | TIME_WITHOUT_TIME_ZONE | int（毫秒/天） | — | 内联 |
| TIMESTAMP(p) | TIMESTAMP_WITHOUT_TIME_ZONE | `Timestamp` | precision | p≤3 内联，否则外置 |
| TIMESTAMP_LTZ(p) | TIMESTAMP_WITH_LOCAL_TIME_ZONE | `Timestamp` | precision | 同上 |
| ARRAY&lt;T&gt; | ARRAY | `InternalArray` | elementType | 外置（变长区） |

各类型的精度/长度边界由对应 `*Type` 类常量约束：`DecimalType.MAX_PRECISION=38`、`MAX_COMPACT_PRECISION=18`（`DecimalType.java:35-39`）；`TimestampType.MAX_PRECISION=9`、`MIN_PRECISION=0`（`TimestampType.java:40-42`）。

### 2.4 Visitor 模式与类型检查

**① 要解决什么问题。** 类型集合相对封闭（新增类型罕见），但"对所有类型做同一件事"的操作很多：序列化、大小估算、转 ORC/Parquet schema、生成字段访问器。若把这些操作都塞进各类型类，类型类会越来越臃肿，且加一个操作要改 20 多个类。

**② 设计原理与取舍。** 用 Visitor 把"操作"从"类型"里抽出来：每个类型实现 `accept(DataTypeVisitor<R>)` 把自己分派给 visitor 的对应方法，新增操作只写一个新 visitor。`DataTypeDefaultVisitor` 提供默认 fallback，省去为每个类型都写一遍样板。

**③ 关键源码 / 路径。**
- `DataTypeVisitor.java`（接口，paimon-api/types）— 为每个具体类型声明一个 `visit` 方法
- `DataTypeDefaultVisitor.java`（默认基类，paimon-api/types）— 未覆写的类型走 `defaultMethod`
- `DataTypeChecks.java`（静态检查工具，paimon-api/types）— 如 `getFieldCount`/取精度等
- `InternalRowToSizeVisitor.java`（paimon-common/types）— 行大小估算 visitor 的实例

**④ 风险 / 陷阱 / 边界。** `DataTypeDefaultVisitor` 的默认 fallback 是双刃剑：新增类型若没在某个 visitor 里显式处理，会静默走 default，可能产生错误结果（源码注释专门警告"default fallback is dangerous"，`DataTypeRoot.java:42-43`）。

**⑤ 收益与代价。** 收益：操作与类型解耦，加操作零侵入类型类。代价：调用链多一跳虚分派；新增类型时要审视所有 visitor 是否需要相应分支。

## 3. InternalRow 体系：一接口五实现

### 3.1 接口设计：为什么不固定一种行格式

**① 要解决什么问题。** "一行数据"在 Paimon 内部要走过多个性质迥异的场景：从列式文件读出来（紧凑二进制）、在 merge engine 里拼接 key+value、在引擎侧用 Java 对象现造、在投影里只取个别字段。任何单一格式都会在某些场景上吃亏——纯 `Object[]` 序列化慢、纯 `byte[]` 构造难。

**② 设计原理与取舍。** 定义只读接口 `InternalRow extends DataGetters`，把"怎么存"留给实现。上层（merge、compaction、序列化）只面向接口编程，运行时可以无差别处理来自文件的 `BinaryRow` 和引擎现造的 `GenericRow`。写能力单独抽成 `DataSetters`，只有可变实现（`BinaryRow`/`GenericRow`）才同时实现它。热路径上避免每个字段 switch 类型，靠 `InternalRow.createFieldGetter(DataType, pos)` 预生成按类型特化的 `FieldGetter`。

**③ 关键源码。** 接口骨架（`InternalRow.java:91-112`）：

```java
// paimon-common/.../data/InternalRow.java:91
public interface InternalRow extends DataGetters {
    int getFieldCount();
    RowKind getRowKind();         // 行的变更语义（+I/-U/+U/-D）
    void setRowKind(RowKind kind);
    // getter 全部继承自 DataGetters：getInt/getString/getDecimal/getRow...
}
```

`createFieldGetter` 在 `InternalRow.java:168` 按 `DataType` 生成一次性的字段访问器（含 null 判断），供排序、序列化等热路径复用，避免每行每字段都做类型分派。

**④ 风险 / 陷阱 / 边界。** 接口是只读契约，拿到一个 `InternalRow` 不能假设它可写；要改值得先确认是 `DataSetters` 实现或转成 `GenericRow`。`getRowKind` 对 `BinaryRow` 来自 header 字节，对 `GenericRow` 是独立字段——两者语义一致但来源不同。

**⑤ 收益与代价。** 收益：上层逻辑与底层布局彻底解耦，新增一种行实现不影响调用方。代价：getter 是虚方法，JIT 需靠去虚化优化；多实现并存增加理解成本。

### 3.2 五种实现各自的取舍

下表是五种 `InternalRow` 实现的横向对比（均在 `paimon-common/data/`）：

| 实现 | 底层存储 | 可变性 | 序列化成本 | GC 友好 | 典型场景 |
|------|---------|--------|-----------|---------|---------|
| **BinaryRow** | `MemorySegment[]`（继承 `BinarySection`） | 固定字段可原地改 | 极低（字节拷贝） | 高 | 文件 I/O、排序、网络 |
| **GenericRow** | `Object[] fields` | 完全可变 | 高（须先转 BinaryRow） | 低 | 引擎现造、测试、中间计算 |
| **JoinedRow** | 两个 `InternalRow` 引用 | 可 `replace` 替换引用 | 不直接序列化 | 中 | key+value 拼接、字段追加 |
| **NestedRow** | `MemorySegment[]`（跨段读） | 固定字段可原地改 | 极低 | 高 | 读 ROW 类型嵌套字段 |
| **LazyGenericRow** | `Supplier<Object>[]` + 缓存 | 字段级延迟物化 | 高 | 低 | 投影、字段计算昂贵且未必访问 |

几个关键点（均经源码核对）：

- **JoinedRow 的零拷贝拼接**（`JoinedRow.java:40-41`、`60`）：持 `row1`/`row2` 两个引用，按 `pos` 是否 `< row1.getFieldCount()` 分发到哪个子行。`isNullAt`/全部 getter 都是这个模式（`JoinedRow.java:121-126` 起，每个 getter 一段相同的分发）。`replace(row1,row2)` 支持对象复用，避免热路径上反复 new。代价：它只是引用视图，子行被覆盖/回收后 JoinedRow 读到的就是脏数据——把它放进集合再复用同一对象是经典 bug。

```java
// paimon-common/.../data/JoinedRow.java:121
public boolean isNullAt(int pos) {
    if (pos < row1.getFieldCount()) {
        return row1.isNullAt(pos);
    } else {
        return row2.isNullAt(pos - row1.getFieldCount());
    }
}
```

- **NestedRow vs BinaryRow**：二者内存格式完全一致（字节级可互相 `equals`），区别只在读取方式——`BinaryRow` 用 `segments[0].getInt(offset)` 直接读第一个段（要求固定区不跨段），`NestedRow` 用 `MemorySegmentUtils` 处理跨段。原因是嵌套 ROW 存在父行变长区，可能横跨多个 `MemorySegment`。
- **GenericRow**（`GenericRow.java:49-54`）：`getField` 即 `(Type) fields[pos]` 强转，`setField` 不做类型检查（`GenericRow.java:98`），类型放错要等到 getter 才 `ClassCastException`。它要落盘必须先经 `InternalRowSerializer.toBinaryRow()` 转成 `BinaryRow`。
- **LazyGenericRow**：用 `Supplier[]` 延迟字段计算，首次访问才物化——投影里大量字段不被读时收益明显，但每个被访问字段多一层 supplier 调用。

## 4. BinaryRow 内存布局

### 4.1 总体布局与 Header / Null Bits

**① 要解决什么问题。** 同一份行数据要同时满足三个互相拉扯的需求：序列化快（落盘/传输）、随机访问快（投影只读个别列）、比较快（排序/去重）。`Object[]` 序列化慢、Protobuf 式紧凑编码随机访问要从头扫、未对齐的定长编码又踩 CPU 对齐惩罚。

**② 设计原理与取舍。** `BinaryRow` 把一行切成两段：**固定区**（Header + Null Bits + 每字段 8 字节槽）保证落在单个 `MemorySegment` 内、支持 O(1) 定位；**变长区**存放放不下 8 字节的数据，可跨多个段。整体布局：

```
固定区（必在 segments[0] 内）            变长区（可跨段）
┌────────┬──────────────┬───────────────────────┐  ┌──────────────────┐
│ Header │ Null Bit Set │ field0 | field1 | ...  │  │ var_data0 | ...  │
│ 1 byte │ 8 字节对齐    │ 每字段 8 bytes          │  │                  │
└────────┴──────────────┴───────────────────────┘  └──────────────────┘
```

- **Header**：固定 1 字节，存 `RowKind` 的字节值（+I=0/-U=1/+U=2/-D=3）。`getRowKind`/`setRowKind` 直接读写 `segments[0]` 的 offset 处（`BinaryRow.java:112-121`）。注意 RowKind 占整整一个字节，不是 1 bit（旧稿 §5.2 称"只占 1 bit"，已修正）。
- **Null Bits**：位图标记哪些字段为 null，**向上对齐到 8 字节边界**，宽度 = `((arity + 63 + 8) / 64) * 8`（`HEADER_SIZE_IN_BITS=8`，`BinaryRow.java:77-79`）。第 i 个字段的 null 位在 `i + HEADER_SIZE_IN_BITS` 处（前 8 bit 给 header 让路，`BinaryRow.java:128-130`）。

**③ 关键源码。** 尺寸计算（`BinaryRow.java:77-83`）：

```java
// paimon-common/.../data/BinaryRow.java:77
public static int calculateBitSetWidthInBytes(int arity) {
    return ((arity + 63 + HEADER_SIZE_IN_BITS) / 64) * 8;   // 对齐到 8 字节
}
public static int calculateFixPartSizeInBytes(int arity) {
    return calculateBitSetWidthInBytes(arity) + 8 * arity;  // null 区 + 定长槽
}
```

对齐到 8 字节的收益在 `anyNull()`：它逐 `long` 扫描 null 区，一次比较 64 个字段（`BinaryRow.java:377`），比逐位检查快得多。

**④ 风险 / 陷阱 / 边界。** Null Bits 宽度不是简单的 `arity/8`——它含 header 占位且按 64 bit 对齐，手算字段偏移极易错位，永远用 `getInt(pos)` 而非手算。字段数过多时固定区可能超过单个 `MemorySegment` 容量，源码注释建议调大 `MemorySegment` 的 pageSize（`BinaryRow.java:50-53`）。

**⑤ 收益与代价。** 收益：批量 null 检查 O(nullBits/8)、布局规整。代价：字段少时 null 区因 8 字节对齐有浪费（哪怕 1 个字段也占 8 字节 null 区 + header）。

### 4.2 固定 8 字节槽：用空间换 O(1) 与对齐

**① 要解决什么问题。** 投影查询常常只读第 5、第 10 个字段，若字段宽度不一就得从头累加偏移（O(n)）。

**② 设计原理与取舍。** 每个字段无论实际类型一律占 **8 字节槽**，于是字段偏移退化为纯算术：`offset + nullBitsSize + pos*8`（`BinaryRow.java:94-96`），任意字段 O(1) 直达。8 字节槽还顺带满足 CPU 字对齐（`putLong`/`getLong` 一次读写），并能把变长数据的"偏移+长度"打包进一个 long。

| 方案 | 随机访问 | 空间 | 对齐 | Paimon 选择 |
|------|----------|------|------|-------------|
| 按实际宽度紧凑排列 | O(n) 累加偏移 | 省 | 可能未对齐 | 否（用于 RowCompacted，见 §7.4） |
| 统一 8 字节槽 | O(1) | 费（boolean 也占 8 字节） | 天然对齐 | **是** |

**③ 关键源码。** 固定字段直写槽位，先清 null 位再写值（`BinaryRow.java:148-160`）：

```java
// paimon-common/.../data/BinaryRow.java:148
public void setInt(int pos, int value) {
    assertIndexIsValid(pos);
    setNotNullAt(pos);
    segments[0].putInt(getFieldOffset(pos), value);   // getFieldOffset = offset + nullBits + pos*8
}
```

各类型在 8 字节槽内的存法：boolean 占最低位；byte/short/int 小端存低位；long/double 占满 8 字节；紧凑 `Decimal(p≤18)` 存 unscaled long；紧凑 `Timestamp(p≤3)` 存毫秒值。

**④ 风险 / 陷阱 / 边界。** 固定字段可 `setInt` 原地改并自动清 null 位（`setNotNullAt`），但变长字段不能原地改长度（见 §4.3）。`setNullAt` 会把该槽清零（`BinaryRow.java:139-146`），以保证字节级 equals/hash 的确定性。

**⑤ 收益与代价。** 收益：投影、排序极快。代价：宽表 + 小类型时空间浪费明显（这正是存储层在 KV 序列化处改用 `RowCompactedSerializer` 的动机，见 §7.4）。

### 4.3 变长区两层编码：7 字节内联 vs 外置

**① 要解决什么问题。** 字符串、byte[]、高精度 Decimal/Timestamp 长度不定，塞不进 8 字节槽。但大量字符串其实很短（状态码 "OK"、国家码 "CN"），若一律外置则每次读都要追一次指针（pointer chasing），cache miss 频发。

**② 设计原理与取舍。** 按长度分两层编码（阈值 `MAX_FIX_PART_DATA_SIZE = 7`，`BinarySection.java:45`，已修正：旧稿标 "BinarySection.java 第 37-63 行"，常量实际在 45/53/62 行）：

- **≤7 字节内联**：`[1-bit 标记=1][7-bit 长度][7-byte 数据]`，整段就在 8 字节槽里，读取无需访问变长区。
- **>7 字节外置**：`[1-bit 标记=0][31-bit 偏移][4-byte 长度]`，数据落在变长区。

判别用最高位标记（`HIGHEST_FIRST_BIT = 0x80L << 56`，`BinarySection.java:53`；长度掩码 `HIGHEST_SECOND_TO_EIGHTH_BIT = 0x7FL << 56`，`:62`）。

**③ 关键源码。** 高精度 Decimal 的外置分支（`BinaryRow.java:170-198`）——unscaled 值转字节存变长区，槽里存"偏移<<32 | 长度"：

```java
// paimon-common/.../data/BinaryRow.java:188
byte[] bytes = value.toUnscaledBytes();
assert bytes.length <= 16;   // 上限 16 字节，恰好覆盖 DECIMAL(38)
MemorySegmentUtils.copyFromBytes(segments, offset + cursor, bytes, 0, bytes.length);
setLong(pos, ((long) cursor << 32) | ((long) bytes.length));
```

高精度 Timestamp（`BinaryRow.java:200-224`）：毫秒存变长区，纳秒偏移存槽位低 32 位，即 `[offset<<32 | nanoOfMillisecond]`。（已修正：旧稿写"用 `unscaledValue().toByteArray()`"，实际方法是 `value.toUnscaledBytes()`。）

**④ 风险 / 陷阱 / 边界。** 内联阈值是 **7 而非 8**（很多人误记 8）。变长字段一旦写定，改成不同长度会破坏后续布局，必须重建整行。高精度 Decimal 外置字节上限 16 字节，这也是 Paimon Decimal 精度上限 38 的物理根由（见 §1.3 陷阱 7）。

**⑤ 收益与代价。** 收益：短字符串零指针追踪、读取就在槽内完成，cache 友好。代价：编码逻辑更复杂（写时要判长度），跨过 7 字节阈值会有一次"内联→外置"的形态突变。

### 4.4 BinaryRowWriter 写入约定

**① 要解决什么问题。** BinaryRow 在热路径上被反复复用（reader 缓冲、排序缓冲），需要一套高效、可复用且不出错的写入流程。

**② 设计原理与取舍。** `BinaryRowWriter`（`paimon-common/data/BinaryRowWriter.java`）按"reset → 逐字段写 → complete"三步走。关键约定：**null 区默认全 0（即默认非 null）**，只有显式 `setNullAt` 才置 1。这样绝大多数非空字段无需任何 null 操作。

**③ 关键流程（文字版）。**
1. `reset()`：清零 null 区，cursor 复位到固定区末尾（变长区起点）。
2. 写字段：定长字段直写槽；变长字段先在槽写指针/内联数据，外置部分追加到 cursor 并前移。
3. `complete()`：把累计的总字节数写回 `BinaryRow.setTotalSize`。

**④ 风险 / 陷阱 / 边界。** **复用 writer 写下一行前必须 `reset()`**，否则上一行的 null 位残留会污染本行。这是复用场景最常见的 bug。

**⑤ 收益与代价。** 收益：默认非空 + 对象复用，写入几乎零额外分配。代价：调用方要自己保证 reset 纪律，漏 reset 是隐蔽错误。

## 5. 列值类型：BinaryString / Decimal / Timestamp / Array / Map

**本节通用动机。** §4 讲了"一行怎么排布"，本节讲"放进 8 字节槽或变长区的那些复杂列值各自怎么编码"。共同诉求：避免 Java 标准类型（`String` UTF-16、`BigDecimal`、`java.sql.Timestamp`）带来的对象分配与编码开销，在常见取值上尽量做到零拷贝或紧凑内联。

### 5.1 BinaryString：UTF-8 + 零拷贝

**① 要解决什么问题。** Java `String` 用 UTF-16，每字符至少 2 字节；且每次从行里取字符串都要新建 `String` 对象，热路径上 GC 压力大。

**② 设计原理与取舍。** `BinaryString`（`paimon-common/data/BinaryString.java`，继承 `BinarySection`）直接用 UTF-8 字节、并指向底层 `MemorySegment`——取字符串时返回一个"视图"而非拷贝。比较走字节序，与文件中排序一致。

**③ 关键取舍对比。**

| 维度 | Java String（UTF-16） | BinaryString（UTF-8 视图） |
|------|----------------------|----------------------------|
| ASCII 文本空间 | 2 字节/字符 | 1 字节/字符 |
| 中文文本空间 | 2 字节/字符 | 3 字节/字符 |
| 取值开销 | 新建 String 对象 | 零拷贝视图 |
| 比较 | 逐 char | 字节序，与落盘一致 |

SQL 负载以 ASCII 为主（列名、关键字、状态码），UTF-8 整体更省。

**④ 风险 / 陷阱 / 边界。** ① `BinaryString` 只是视图，底层段被复用/回收后再 `toString()` 读脏数据（见 §1.3 陷阱 3）；要长期持有需 `copy()`。② 中文按 3 字节算，`VARCHAR(10)` 只能装约 3 个汉字。③ 热路径里反复 `toString()` 会反复建 `String`，应一次物化后复用。

**⑤ 收益与代价。** 收益：内存省、取值零拷贝、比较与落盘一致。代价：中文比 UTF-16 多占 50%，且视图生命周期需谨慎管理。

### 5.2 Decimal：18 位为界的双模存储

**① 要解决什么问题。** 全用 `BigDecimal` 表示小数，则每个值都是堆对象、加减乘除都走大数运算，金额/百分比这类高频场景代价过高。

**② 设计原理与取舍。** `Decimal`（`paimon-common/data/Decimal.java:44`）按精度分两模：

- **紧凑（precision ≤ 18 = `MAX_COMPACT_PRECISION`，`Decimal.java:136`）**：用一个 `long` 存 unscaled 值，可内联进 BinaryRow 8 字节槽，比较即 `Long.compare`。
- **非紧凑（precision > 18）**：退化为 `BigDecimal`，数据外置到变长区。

分界 18 的由来：Java `long` 约 9.2×10¹⁸，恰能精确表示 18 位十进制（`POW10` 预算了 10 的幂，`Decimal.java:51`）。

**③ 关键源码。** 构造时即按精度分流（`Decimal.java:182-210`）：

```java
// paimon-common/.../data/Decimal.java:189
if (precision <= MAX_COMPACT_PRECISION) {
    return new Decimal(precision, scale, bd.unscaledValue().longValueExact(), null);
}
return new Decimal(precision, scale, -1, bd);   // 非紧凑：保留 BigDecimal
```

**④ 风险 / 陷阱 / 边界。** ① 跨过 18 位精度会从"一个 long"突变为"BigDecimal + 外置"，性能/存储不连续（见 §1.3 陷阱 2）。② `fromUnscaledLong` 仅适用于紧凑精度，超 long 范围的值必须走 `fromBigDecimal`。③ 上限 `DECIMAL(38)`（外置字节 ≤16，见 §4.3）。精度截断/溢出的语义由 casting 层处理，规则详见 22。

**⑤ 收益与代价。** 收益：常见精度下 Decimal ≈ long，零对象、运算极快。代价：双套逻辑（紧凑/非紧凑）维护成本，分界处行为突变。

### 5.3 Timestamp：millis + nanoOfMillis 分离

**① 要解决什么问题。** 若统一用纳秒 long 表示时间戳，64 位只能覆盖约 292 年范围；而多数场景只需毫秒/秒精度。

**② 设计原理与取舍。** `Timestamp`（`paimon-common/data/Timestamp.java`）拆成两段：`millisecond:long`（`:57`）+ `nanoOfMillisecond:int`（`:60`，0–999999）。

- **紧凑（precision ≤ 3，`Timestamp.isCompact`:241）**：只用 millisecond，单 long 内联。
- **非紧凑（precision > 3）**：毫秒落变长区、纳秒偏移存槽位低 32 位（见 §4.3 的 setTimestamp）。

**③ 关键源码。** 微秒入口拆分毫秒/纳秒（`Timestamp.java:232-235`）：

```java
// paimon-common/.../data/Timestamp.java:232
public static Timestamp fromMicros(long micros) {
    long mills = Math.floorDiv(micros, MICROS_PER_MILLIS);
    long nanos = Math.floorMod(micros, MICROS_PER_MILLIS) * NANOS_PER_MICROS;
    return Timestamp.fromEpochMillis(mills, (int) nanos);
}
```

**④ 风险 / 陷阱 / 边界。** ① `TIMESTAMP(3)` 紧凑、`TIMESTAMP(4)` 非紧凑，精度选择影响内联与性能。② 上限 `TIMESTAMP(9)`（`TimestampType.MAX_PRECISION=9`）。③ `TIMESTAMP` vs `TIMESTAMP_LTZ` 仅类型 root 不同、内部同为 `Timestamp`，时区语义差异在引擎适配层处理（见 §11.2、§1.3 陷阱 6）。

**⑤ 收益与代价。** 收益：低精度单 long 内联、范围与精度兼顾。代价：高精度需 millis+nanos 两处协同，setter 逻辑较绕。

### 5.4 BinaryArray / BinaryMap

**① 要解决什么问题。** ARRAY/MAP 是复合类型，要能内嵌进 BinaryRow 变长区并支持零拷贝读取，不能每次都反序列化成 Java 集合。

**② 设计原理与取舍。** `BinaryArray`（`paimon-common/data/BinaryArray.java:46`，继承 `BinarySection`）的布局：`[size(4字节)] + [null bits(4字节对齐)] + [元素值或 offset&length] + [变长区]`（`BinaryArray.java:40`）。头部宽度 `4 + ((numFields + 31) / 32) * 4`（`:60-61`）——null 区按 **4 字节（32 位）对齐**，而非 BinaryRow 的 8 字节。

`BinaryMap`（`paimon-common/data/BinaryMap.java`）= 并列的两个 `BinaryArray`（key 数组 + value 数组），布局 `[keyArraySize(4字节)] + [Key BinaryArray] + [Value BinaryArray]`，复用 BinaryArray 的全部编解码逻辑。

**③ 设计要点对比。**

| 维度 | BinaryRow | BinaryArray |
|------|-----------|-------------|
| null 区对齐 | 8 字节 | 4 字节 |
| 头部 | 1 字节 header + null 区 | 4 字节 size + null 区 |
| 动机差异 | 行常被批量 null 检查，8 字节利于 long 扫描 | 数组元素多，4 字节对齐更省空间 |

**④ 风险 / 陷阱 / 边界。** Array/Map 一律外置在变长区，深层嵌套（`ARRAY<ARRAY<...>>`）会层层展开变长区，过深嵌套增加序列化深度与单行体积。MULTISET 复用 `InternalMap`（value 为计数），见 §7.5 工厂映射。

**⑤ 收益与代价。** 收益：复合类型零拷贝内嵌、Map 复用 Array 逻辑。代价：相对 Java 集合可读性差、随机按 key 取值需先定位（非 O(1) 哈希）。

## 6. KeyValue：主键表的物理行单元

### 6.1 核心字段与 key/value 分离

**① 要解决什么问题。** 主键表跑在 LSM 上，排序归并、版本去重、CDC 语义这三件事要在同一份行结构上高效完成。若用一整行表示，排序得比较整行（含无关 value 列），且无处安放版本号与变更类型。

**② 设计原理与取舍。** `KeyValue`（`paimon-core/KeyValue.java:42`）把一条记录拆成 5 部分：`key`（主键字段）、`sequenceNumber`（写入后分配的递增版本号）、`valueKind`（RowKind 变更类型）、`value`（非主键字段）、`level`（读取后回填的 LSM 层级）。

- **key/value 分离**：排序/查找只比 key 列，省 CPU；投影时可只取 key 或只取 value；key 与 value 可用不同压缩/索引策略。
- **sequenceNumber**：LSM 归并靠它判定"谁更新"，保留同 key 中 seq 最大者。
- **level**：是文件属性而非数据属性，读出来才回填，compaction 据此选层。

**③ 关键源码。** 字段定义（`KeyValue.java:42-53`）：

```java
// paimon-core/.../KeyValue.java:42
public class KeyValue {
    private InternalRow key;           // 主键字段
    private long sequenceNumber;       // 写入后分配
    private RowKind valueKind;         // 变更类型
    private InternalRow value;         // 值字段
    private int level;                 // 读取后回填的 LSM 层级
}
```

落盘时 key 与 value 用 `JoinedRow` 思路拼成整行（见 §3.2），读取时按布局再拆开（见 §6.3）。

**④ 风险 / 陷阱 / 边界。** `sequenceNumber`/`level` 由引擎分配，构造 KeyValue 时不该手动设。`valueKind`（KeyValue 的变更类型）与 `value.getRowKind()`（值行自身的 RowKind）是两个独立概念，可能不同，别混用。同 key 多版本必须严格递增 seq，否则归并取错版本。

**⑤ 收益与代价。** 收益：排序只比 key、版本语义清晰、支持完整 CDC。代价：结构比"一整行"复杂，需要 schema 在 key/value 间插入元数据列（见 §6.3）。

### 6.2 RowKind 与 changelog 语义

**① 要解决什么问题。** 要支持 CDC（增删改的流式回放），必须能在行级别区分"插入 / 更新前 / 更新后 / 删除"，仅 Put/Delete 两态不够。

**② 设计原理与取舍。** `RowKind`（`paimon-api/types/RowKind.java:35`，已修正：旧稿标 paimon-common）是 4 值枚举，每值带短串与字节值：

| RowKind | 短串 | 字节值 | 语义 | 源码 |
|---------|------|--------|------|------|
| INSERT | +I | 0 | 新增 | `RowKind.java:35` |
| UPDATE_BEFORE | -U | 1 | 更新前旧值（回撤用） | `:44` |
| UPDATE_AFTER | +U | 2 | 更新后新值 | `:53` |
| DELETE | -D | 3 | 删除 | `:56` |

字节值固定（注释专门指出枚举的 hashCode 跨 JVM 不稳定，必须用 `toByteValue()`，`RowKind.java:31`）。`isAdd()`=INSERT|UPDATE_AFTER（`:106`），`isRetract()`=UPDATE_BEFORE|DELETE（`:101`），供 merge engine 快速判语义。

**③ 关键源码。** 语义判定（`RowKind.java:101-107`）：

```java
// paimon-api/.../types/RowKind.java:101
public boolean isRetract() { return this == UPDATE_BEFORE || this == DELETE; }
public boolean isAdd()     { return this == INSERT || this == UPDATE_AFTER; }
```

**④ 风险 / 陷阱 / 边界。** RowKind 在 `BinaryRow` 里占 header 整整 1 字节（不是 1 bit，旧稿误述已修正，见 §4.1）。changelog 完整性要求 UPDATE_BEFORE/UPDATE_AFTER 成对——这由 changelog producer 保证，机制详见 24。

**⑤ 收益与代价。** 收益：单字节编码 + 语义方法，CDC 判别零开销。代价：四态语义要求上下游（写入、compaction、changelog）一致维护，否则 changelog 不可回放。

### 6.3 KeyValue 的 schema 布局

**① 要解决什么问题。** KeyValue 落到列式文件时需要一个扁平 schema，且要能在读取时把 key 列、value 列、元数据列三者干净地切开做投影。

**② 设计原理与取舍。** 物理 schema 是一个扁平 `RowType`，字段顺序为：

```
[key_0 ... key_n] [_SEQUENCE_NUMBER] [_VALUE_KIND] [value_0 ... value_m]
```

两个元数据列 `_SEQUENCE_NUMBER`/`_VALUE_KIND`（取自 `SpecialFields`，`KeyValue.java:35-36`）夹在 key 与 value 之间，形成清晰分界：key 投影取前 n 列、value 投影取后 m 列、中间 2 列恒读。

**③ 关键源码。** schema 构造与字段拼装（`KeyValue.java:113-135`）：

```java
// paimon-core/.../KeyValue.java:113
public static RowType schema(RowType keyType, RowType valueType) {
    return new RowType(false, createKeyValueFields(keyType.getFields(), valueType.getFields()));
}
// createKeyValueFields(:130) 中插入 SEQUENCE_NUMBER(:134) 与 VALUE_KIND(:135)
```

`KeyValue.project()`（`KeyValue.java:140`）基于此布局构造投影映射（旧稿标"第 140-165 行"，起点 140 已核对）。

**④ 风险 / 陷阱 / 边界。** 元数据列名以 `_` 前缀且属保留字段，用户表不应使用同名列。整行 schema 的 nullability 顶层为 false（`new RowType(false, ...)`）。

**⑤ 收益与代价。** 收益：投影边界清晰、key/value 可独立裁剪。代价：每条记录在文件里多 2 个元数据列的存储开销（seq=long、kind=1 字节）。

## 7. 序列化框架：三套行序列化方案

### 7.0 概览

**①要解决什么问题。** 内存里的 `InternalRow` 要在三类场景间往返字节流：文件 I/O（求快）、分页内存缓冲（求跨页正确）、LSM KV 存储与 Manifest（求省空间）。单一方案无法同时最优，于是 Paimon 备了三套行序列化器，各占一个权衡点。

**②设计总览（对比表）。**

| 序列化器 | 字节布局 | 随机访问 | 空间 | 典型场景 | 源码 |
|----------|----------|----------|------|----------|------|
| **BinaryRowSerializer** | `[4字节长度]+[BinaryRow 字节]` | O(1) | 大（8 字节对齐） | 文件 I/O、排序、分页 | `BinaryRowSerializer.java` |
| **InternalRowSerializer** | 先转 BinaryRow 再写 | O(1) | 同上 | 任意 InternalRow 落盘的桥 | `InternalRowSerializer.java` |
| **RowCompactedSerializer** | 按实际宽度 + VarInt | 否（顺序扫） | 小 30–50% | LSM KV、Manifest | `RowCompactedSerializer.java` |

### 7.1 Serializer 接口层次

**① 要解决什么问题。** 上层（写文件、排序缓冲、KV 存储）只想要"把对象写成字节/读回来"，不关心实现；且部分序列化器有可复用的内部状态，多线程要安全隔离。

**② 设计原理与取舍。** `Serializer<T>`（`Serializer.java:32`）定义 `duplicate()`/`copy()`/`serialize()`/`deserialize()`，并提供 `serializeToBytes`/`deserializeFromBytes` 默认便捷方法（`:79`/`:92`）。分页能力由子接口 `PagedTypeSerializer<T>` 扩展，行序列化的共同基类是 `AbstractRowDataSerializer<T>`（含 `toBinaryRow`）。

**③ 关键源码。** 接口核心（`Serializer.java:41-71`）：

```java
// paimon-common/.../data/serializer/Serializer.java:41
Serializer<T> duplicate();            // 有状态实现据此每线程复制
T copy(T from);
void serialize(T record, DataOutputView target) throws IOException;
T deserialize(DataInputView source) throws IOException;
```

**④ 风险 / 陷阱 / 边界。** 有状态序列化器（如 `InternalRowSerializer` 持 `RowHelper`）非线程安全，跨线程必须 `duplicate()`（见 §1.3 陷阱 5）。无状态原始类型序列化器返回单例（见 §7.5）。

**⑤ 收益与代价。** 收益：统一接口 + 分页/紧凑能力按需扩展。代价：层次较多（Serializer → PagedTypeSerializer → AbstractRowDataSerializer），理解曲线略陡。

### 7.2 BinaryRowSerializer：分页 + 零拷贝映射

**① 要解决什么问题。** BinaryRow 已是紧凑二进制，序列化不该再逐字段遍历；分页缓冲里又必须保证固定区不跨页（否则破坏 §4.2 的 O(1) 直读前提）。

**② 设计原理与取舍。** 格式极简：`[4字节长度] + [BinaryRow 原始字节]`，序列化≈内存拷贝。分页写入时先探测当前页剩余空间，不够装下固定区就跳到下一页；读取支持 `mapFromPages` 把 reuse 行直接 `pointTo` 分页内存，零拷贝。

**③ 关键源码。** 固定区不跨页的探测（`BinaryRowSerializer.java:258-263`，旧稿标 258-279，起点核对一致）：

```java
// paimon-common/.../serializer/BinaryRowSerializer.java:258
private int checkSkipWriteForFixLengthPart(AbstractPagedOutputView out) throws IOException {
    int available = out.getSegmentSize() - out.getCurrentPositionInSegment();
    if (available < getSerializedRowFixedPartLength()) {
        out.advance();                 // 跳到下一页，避免固定区跨页
        return available;
    }
    return 0;
}
```

`serializeToPages`:116、`mapFromPages`:166（零拷贝读）。

**④ 风险 / 陷阱 / 边界。** 固定区跨页会迫使读取走跨段逻辑（像 NestedRow），显著拖慢热路径——这正是 `checkSkip` 存在的理由。变长区允许跨页。

**⑤ 收益与代价。** 收益：序列化/反序列化纯内存拷贝、可零拷贝映射。代价：格式不灵活（不支持字段重排/演化），空间随 8 字节对齐偏大。

### 7.3 InternalRowSerializer：统一到 BinaryRow 的桥

**① 要解决什么问题。** 上层经常拿到的是 `GenericRow`/`JoinedRow` 等非二进制实现，但落盘、排序、分页都要求 `BinaryRow`。

**② 设计原理与取舍。** `InternalRowSerializer`（`:37`）持一个 `RowHelper`（含可复用 BinaryRowWriter + 各列 FieldGetter，`:43`），`toBinaryRow` 做"已是 BinaryRow 则零成本返回、否则逐字段拷进复用行"。

**③ 关键源码。**（`InternalRowSerializer.java:119-124`，旧稿标 119-125，已核对）：

```java
// paimon-common/.../serializer/InternalRowSerializer.java:119
public BinaryRow toBinaryRow(InternalRow row) {
    if (row instanceof BinaryRow) {
        return (BinaryRow) row;        // 零成本
    }
    rowHelper.copyInto(row);           // 逐字段拷进复用 BinaryRow
    return rowHelper.reuseRow();
}
```

**④ 风险 / 陷阱 / 边界。** `reuseRow()` 返回的是复用对象——同一序列化器再调一次会覆盖上次结果，不能跨调用持有。非线程安全（见 §7.1）。

**⑤ 收益与代价。** 收益：上层可透明处理任意 InternalRow 实现。代价：非 BinaryRow 输入要逐字段拷贝（GenericRow 落盘的固有成本）。

### 7.4 RowCompactedSerializer：存储场景的紧凑编码

**① 要解决什么问题。** LSM KV 的 key/value、Manifest 这类存储场景数据量大、以顺序读为主，BinaryRow 的 8 字节对齐在这里纯属空间浪费。

**② 设计原理与取舍。** `RowCompactedSerializer`（`:59`）放弃 O(1) 随机访问，换真正紧凑的编码：

| 维度 | BinaryRowSerializer | RowCompactedSerializer |
|------|---------------------|------------------------|
| null 区对齐 | 8 字节 | 1 字节 |
| 定长字段 | 统一 8 字节 | 按实际宽度（bool=1/int=4/long=8） |
| 变长长度 | 4 字节 int | VarInt（`VarLengthIntUtils`，`:40`） |
| 随机访问 | O(1) | 顺序扫描 |

它还提供 `createSliceComparator()`（`:166`），可直接在序列化字节上比较行、免反序列化。

**③ 关键源码。**（`RowCompactedSerializer.java:106-107`）写入时以 VarInt 编码长度前缀：

```java
// paimon-common/.../serializer/RowCompactedSerializer.java:106
byte[] bytes = serializeToBytes(record);
VarLengthIntUtils.encodeInt(target, bytes.length);   // VarInt 长度前缀
```

**④ 风险 / 陷阱 / 边界。** 不支持 O(1) 随机访问，要取第 k 列必须从头扫——只适合顺序读场景，误用于随机访问会退化。

**⑤ 收益与代价。** 收益：比 BinaryRow 格式省 30–50% 空间，I/O 密集场景（compaction）收益明显。代价：随机访问能力丧失，编解码比纯字节拷贝慢。

### 7.5 InternalSerializers 工厂

**① 要解决什么问题。** 给定一个 `DataType`，要能拿到对应的列序列化器，且无状态的原始类型不该反复 new。

**② 设计原理与取舍。** `InternalSerializers.create(DataType)`（`:38`，另有 `create(RowType)`:43）按 `typeRoot` 分发：原始类型返回单例 `INSTANCE`，参数化/复合类型按参数构造。

**③ 映射要点（节选，`InternalSerializers.java:50-92`）。**

| DataType | Serializer | 说明 |
|----------|-----------|------|
| CHAR/VARCHAR | `BinaryStringSerializer.INSTANCE`（:52） | 单例 |
| INTEGER/DATE/TIME | `IntSerializer.INSTANCE`（:67） | 单例 |
| DECIMAL | `DecimalSerializer(p,s)`（:58） | 按精度选紧凑/非紧凑 |
| TIMESTAMP / _LTZ | `TimestampSerializer(p)`（:74-75） | 按精度 |
| ARRAY/MAP/ROW | `InternalArray/Map/RowSerializer`（:77/:86/:89） | 复合递归 |
| MULTISET | 复用 `InternalMapSerializer(elem, Int)` | value 为计数 |
| VARIANT | `VariantSerializer.INSTANCE`（:92） | 单例 |

**④ 风险 / 陷阱 / 边界。** 单例仅用于无成员变量的原始类型；带精度/元素类型的序列化器是有状态的，不可共享。新增类型必须在此 switch 补分支，否则 default 抛不支持。

**⑤ 收益与代价。** 收益：原始类型零分配复用、类型→序列化器一处集中。代价：switch 与类型枚举强耦合，演化要同步维护。

## 8. 文件格式类型映射（边界，详见 20）

### 8.0 概览

> 本节只讲"内存 `DataType` → ORC/Parquet 物理类型"的映射规则与其取舍；列式落盘的编码细节、向量化读取（ColumnVector 适配）主讲 **20**，本节点到即引用。

**① 要解决什么问题。** Paimon 的内存类型要落进 ORC/Parquet 的物理类型体系，必须保证：语义不漂移（`DECIMAL(10,2)` 读回仍是 `(10,2)`）、精度不丢、且字段改名后仍能对齐（Schema 演化）。

**② 设计原理与取舍。** 两条主线：① **字段 ID 而非字段名**——名可被重命名，ID 在演化中不变，故映射时把 Paimon 字段 ID 写进文件元数据；② **精度分档映射**——列存按列同型存储，用更小的物理类型省压缩后体积。

### 8.1 ORC 类型映射与字段 ID

**① 要解决什么问题。** ORC 默认靠字段名匹配且不区分大小写，列改名后历史文件就读不对了。

**② 设计原理与取舍。** `OrcTypeUtil`（`paimon-format/format/orc/OrcTypeUtil.java`）把每个 Paimon 类型转成 ORC `TypeDescription`，并通过 `setAttribute(PAIMON_ORC_FIELD_ID_KEY, id)` 把字段 ID 钉进去（key 值为 `"paimon.id"`）。

**③ 关键源码。**（已修正：旧稿写方法名 `toOrcType`，实际为 `convertToOrcSchema`/`convertToOrcType`，`OrcTypeUtil.java:40/50`）


### 8.2 Parquet 类型映射的精度分档

**① 要解决什么问题。** Parquet 列存按列同型，Decimal/Timestamp 用过宽的物理类型会显著放大压缩后体积；高精度时间戳还要兼容 Hive/Impala 的 INT96 习惯。

**② 设计原理与取舍。** `ParquetSchemaConverter`（`paimon-format/format/parquet/ParquetSchemaConverter.java`，方法 `convertToParquetType`，已修正：旧稿泛称 `toParquetType`）按精度分档：

| Paimon 类型 | Parquet 物理类型 | 源码 |
|------------|------------------|------|
| DECIMAL(p≤9) | INT32 + Decimal | `is32BitDecimal`:278 |
| DECIMAL(9&lt;p≤18) | INT64 + Decimal | — |
| DECIMAL(p&gt;18) | FIXED_LEN_BYTE_ARRAY + Decimal | — |
| TIMESTAMP(p≤3) | INT64 + MILLIS | `createTimestampWithLogicalType`:252 |
| TIMESTAMP(3&lt;p≤6) | INT64 + MICROS | `:261` |
| TIMESTAMP(p&gt;6) | INT96（兼容 Hive/Impala） | `:259` |
| TINYINT/SMALLINT | INT32 + intType(8/16) | `:119/:124` |

**③ 关键源码。** 时间戳分档（`ParquetSchemaConverter.java:250-266`）：

```java
// paimon-format/.../parquet/ParquetSchemaConverter.java:252
if (precision <= 3)      return INT64 + MILLIS;
else if (precision > 6)  return INT96;          // 遗留格式，兼容 Hive/Impala
else                     return INT64 + MICROS;
```

**④ 风险 / 陷阱 / 边界。** INT96 是遗留格式、不推荐主动使用；纳秒级时间戳走 INT96。注意 Parquet Decimal 分三档而 BinaryRow 内存只分两档（18 为界）——因为内存里每字段都 8 字节，INT32/INT64 无差别，列存才有省空间动机。

**⑤ 收益与代价。** 收益：小精度 Decimal 用 INT32 省一半列空间、时间戳兼容老生态。代价：映射分档逻辑较碎，纳秒精度被迫用遗留 INT96。

## 9. 代码生成（Codegen）

### 9.0 概览

**① 要解决什么问题。** 投影、记录比较、等值判断这些操作在 compaction、sort-merge 等热路径上被调用数十亿次。若按 `RowType` 在运行时用 switch/case 逐字段分派类型，哪怕每次只多几纳秒，累积也是可观开销，且虚分派阻碍 JIT 内联。

**② 设计原理与取舍。** Schema 是运行时才确定的，无法预编译，于是 Paimon 在运行时按具体 `RowType` 生成 Java 源码、用 **Janino** 编译成类并缓存，把动态分派固化成静态调用（`row.getInt(3)` 这种直调）。只对热路径操作做生成（编译有 10–100ms 首次成本，仅高频调用才划算），其余仍解释执行。

| 方案 | 首次成本 | 稳态性能 | Paimon 选择 |
|------|----------|----------|-------------|
| 运行时 switch/case 解释 | 0 | 有分派/装箱开销 | 仅冷路径 |
| 运行时 codegen + 缓存 | 10–100ms 编译 | 静态调用、JIT 易内联 | **热路径** |

**③ 关键源码。** `CodeGenerator` 接口（`paimon-common/codegen/CodeGenerator.java:27`，实现 `paimon-codegen/codegen/CodeGeneratorImpl.java`）声明四类生成入口：

```java
// paimon-common/.../codegen/CodeGenerator.java:27
GeneratedClass<Projection> generateProjection(RowType inputType, int[] inputMapping);     // :29
GeneratedClass<NormalizedKeyComputer> generateNormalizedKeyComputer(...);                 // :38/:48
GeneratedClass<RecordComparator> generateRecordComparator(...);                           // :59/:69
GeneratedClass<RecordEqualiser> generateRecordEqualiser(...);                             // :73
```

四类生成物的用途：**Projection** 做 InternalRow→BinaryRow 列投影（直生 `result.setInt(0, row.getInt(3))`）；**NormalizedKeyComputer** 把多字段比较编译为连续字节写入（支持前缀排序）；**RecordComparator/RecordEqualiser** 生成类型特化的比较/等值逻辑，消除字段类型分派。编译走 `CompileUtils`（Janino），结果按生成代码缓存、相同代码只编一次。

**④ 风险 / 陷阱 / 边界。** ① 缓存 key 必须包含 `RowType`——只用 `inputMapping` 当 key、schema 变了会命中错误缓存。② Janino 不支持 Java 8+ 的 lambda/stream，生成代码须用传统语法。③ 生成类占 Metaspace，海量不同 schema 会累积。

**⑤ 收益与代价。** 收益：热路径消除虚分派与装箱、JIT 更易内联。代价：首次编译延迟、Metaspace 占用、生成代码难调试（与 Flink 共用 Janino 思路，跨引擎一致）。

## 10. Casting：Schema 演化的类型转换（机制，规则详见 22）

### 10.0 概览

> 本节讲 casting 的**机制**（Rule/Executor/CastedRow 懒转换）；"哪些类型变更算安全升级"这类演化规则主讲 **22**，本节只列规则全景作索引。

**① 要解决什么问题。** Schema 演化里旧文件的列类型可能与新 schema 不同（`INT`→`BIGINT`、`DECIMAL(10,2)`→`(12,4)`）。读取时要按需把旧值转成新类型，且不能"为改了 2 列就把整行 100 列都转一遍"。

**② 设计原理与取舍。** 两层机制：

- **Rule + Predicate**（`paimon-common/casting/CastRule.java`、`CastRulePredicate.java`）：一条 `CastRule` 通过 `CastRulePredicate` 声明它能匹配哪些输入/输出类型——常借 `DataTypeFamily` 批量匹配（如 `NumericPrimitiveCastRule` 一条覆盖所有整数互转），避免为每对具体类型写规则。`CastRule.create(in,out)` 产出无副作用的 `CastExecutor`（`CastExecutor.java`，单方法 `cast(IN)→OUT`）。
- **CastedRow 懒转换**（`paimon-common/casting/CastedRow.java`）：一个 `InternalRow` 包装器，持 `CastFieldGetter[]`，每次读字段才从底层行取原值并转换。100 列只访问 5 列，就只转这 5 列。

**③ 关键结构。**

```
CastRule<IN,OUT>           getPredicateDefinition() → CastRulePredicate（声明可匹配类型）
                           create(in,out)           → CastExecutor<IN,OUT>
CastedRow (InternalRow)    持 CastFieldGetter[]，按 pos 懒转换
```

`CastExecutors` 是规则注册/查找入口；`CastedArray`/`CastedMap`/`CastedVector` 是复合类型的对应懒包装。

**④ 风险 / 陷阱 / 边界。** ① 并非所有转换安全（`BIGINT`→`INT` 可能溢出、`DECIMAL(10,4)`→`(10,2)` 截断）——安全性边界由演化层把关，详见 22。② `STRING`→数值要解析，比直接读数值慢一两个数量级。③ `CastedRow` 是包装视图，底层行被替换后语义随之变。

**⑤ 收益与代价。** 收益：一条 Rule 覆盖一族类型、懒转换让演化读几乎零额外成本（未访问列不转）。代价：规则集需维护（下表 30+ 条），转换失败的"返 null vs 抛错"语义需明确。

### 10.1 转换规则全景（索引，逐条语义见 22）

均位于 `paimon-common/casting/`（已逐一核对文件存在）：

- **数值/Decimal**：`NumericPrimitiveCastRule`、`DecimalToDecimalCastRule`、`NumericPrimitiveToDecimalCastRule`、`DecimalToNumericPrimitiveCastRule`、`NumericPrimitiveToTimestamp`
- **布尔↔数值**：`BooleanToNumericCastRule`、`NumericToBooleanCastRule`
- **To String**：`NumericToStringCastRule`、`BooleanToStringCastRule`、`TimestampToStringCastRule`、`DateToStringCastRule`、`TimeToStringCastRule`、`StringToStringCastRule`、`ArrayToStringCastRule`、`MapToStringCastRule`、`RowToStringCastRule`
- **From String**：`StringToBooleanCastRule`、`StringToDecimalCastRule`、`StringToNumericPrimitiveCastRule`、`StringToDateCastRule`、`StringToTimeCastRule`、`StringToTimestampCastRule`、`StringToBinaryCastRule`、`StringToArrayCastRule`、`StringToMapCastRule`、`StringToRowCastRule`
- **日期/时间互转**：`TimestampToTimestampCastRule`、`TimestampToDateCastRule`、`TimestampToTimeCastRule`、`DateToTimestampCastRule`、`TimeToTimestampCastRule`、`TimestampToNumericPrimitiveCastRule`
- **二进制**：`BinaryToBinaryCastRule`、`BinaryToStringCastRule`

## 11. 与 Flink / Spark 的类型映射

### 11.0 概览

**① 要解决什么问题。** Paimon 与 Flink/Spark 之间每条记录都要跨内存模型传递。若每次都深拷贝，吞吐损失巨大；若直接共享对方的内存段，又会因双方内存管理（生命周期、回收策略）不同而 use-after-free。

**② 设计原理与取舍。** 一律走 **Wrapper（包装视图）而非 Converter（深拷贝）**：包装对方的行对象、按 pos 委托取值。基本数值类型内部表示一致（Java 原生），可真零拷贝；`String`/非紧凑 `Decimal` 因分属两套内存系统，必须做字节拷贝换取内存安全；复合类型递归包装、延迟到字段访问时再转。

### 11.1 Flink 适配层

**① 要解决什么问题。** Paimon `InternalRow` 与 Flink `RowData` 接口形似但分属两套框架，读写双向都要桥接。

**② 设计原理与取舍。** 两个 wrapper：`FlinkRowData`（Paimon→Flink，读路径，`FlinkRowData.java:42`）与 `FlinkRowWrapper`（Flink→Paimon，写路径）。基本类型直接委托（零拷贝）；String/Decimal/Timestamp 走轻量转换。

**③ 关键源码。** 读路径的类型转换（`FlinkRowData.java:111-122`、`:314/:318`）：

```java
// paimon-flink/.../flink/FlinkRowData.java:111
public StringData getString(int pos) { return toFlinkString(row.getString(pos)); }  // :314 StringData.fromBytes(str.toBytes()) —— 字节拷贝
public TimestampData getTimestamp(int pos, int p) { return toFlinkTimestamp(row.getTimestamp(pos, p)); } // :318 fromEpochMillis(millis, nanoOfMillis) —— 零拷贝
```

基本类型（boolean/byte/short/int/long/float/double/byte[]）直接 `row.getXxx(pos)` 委托，无转换。

**④ 风险 / 陷阱 / 边界。** `FlinkRowData` 只是引用包装，底层 Paimon 行必须在其存活期内有效。`StringData.fromBytes(str.toBytes())` 会拷一次字节——这是两套内存系统隔离的必要代价，不是零拷贝。

**⑤ 收益与代价。** 收益：数值类型零拷贝、与 Flink RowData 无缝。代价：String 等有一次字节拷贝（通常 <5%）。

### 11.2 Spark 适配层

**① 要解决什么问题。** Spark 有两套行表示（V1 的外部 `Row`、V2 的 catalyst `InternalRow`），且其 data schema 与 table schema 字段顺序可能不同，时区语义也与 Paimon 有别。

**② 设计原理与取舍。** 三个 wrapper：`SparkRow`（Spark 外部 `Row`→Paimon，V1 写）、`SparkInternalRowWrapper`（catalyst `InternalRow`→Paimon，V2 写，`SparkInternalRowWrapper.java:57`）、`AbstractSparkInternalRow`（Paimon→Spark，读）。V2 路径靠 `fieldIndexMap`（`:62`）做字段重映射，并对带时区类型做 UTC 调整。

**③ 关键源码。** 字段索引映射 + 缺失字段处理（`SparkInternalRowWrapper.java:107-113`，已修正：旧稿标"第 92-107 行"）：

```java
// paimon-spark/.../spark/SparkInternalRowWrapper.java:107
if (fieldIndexMap == null) { return pos; }
if (pos < 0 || pos >= fieldIndexMap.length) { ... }
return fieldIndexMap[pos];      // data schema → table schema 重映射，-1 表示缺失列→null
```

时区处理区分 `TimestampNTZType` 与带时区 `TimestampType`（`:213` getTimestamp、`:300` 类型判定，已修正：旧稿标"第 304-316 行"）。

**④ 风险 / 陷阱 / 边界。** ① 忽略 `fieldIndexMap` 会按错位置取列（取到 name 当 age）。② Spark `TIMESTAMP` 是 UTC、Paimon `TIMESTAMP` 无时区，引擎时区配置不一致会偏移（见 §1.3 陷阱 6）。③ `SparkRow`（V1，包装外部 `Row`）每个 getter 都要 Object 级类型转换，比 `SparkInternalRowWrapper`（V2，读 Tungsten 二进制）重。

**⑤ 收益与代价。** 收益：兼容 V1/V2 两套 API、支持字段重排与列裁剪。代价：字段映射多一次数组查找、时区调整增加复杂度。

### 11.3 零拷贝边界总表

| 字段类型 | Flink/Spark ↔ Paimon | 说明 |
|----------|----------------------|------|
| boolean/整数/浮点/byte[] | **零拷贝** | 内部表示同为 Java 原生 |
| 紧凑 Decimal、Timestamp | 轻量转换 | 传 unscaled long / millis+nanos |
| String | 字节拷贝 | UTF-8 字节跨内存系统复制 |
| 非紧凑 Decimal | 转换 | BigDecimal 重建 |
| Array/Map/Row | 递归包装 | 延迟到字段访问时转 |

**为什么不直接引用对方的 `MemorySegment`/`UnsafeRow`？** Flink 与 Spark 各有独立的内存管理与回收策略，跨框架直接持有底层内存会导致 use-after-free 或泄漏。字节拷贝是用少量开销换内存安全的刻意取舍。

## 12. 设计决策总结

| 设计决策 | 为什么这么选 | 取舍/代价 | 收益 |
|---------|------------|-----------|------|
| 类型定义下沉 paimon-api | 类型是引擎间公共契约，应独立于实现 | 多一个模块边界、定义与操作分层 | Connector 仅依赖轻量 API、契约稳定 |
| DataTypeRoot + DataTypeFamily 双层枚举 | 既要精确分发又要批量归类 | 新增类型须同步维护 Family、编译期固定 | 一条 cast 规则覆盖一族类型、谓词批量判类 |
| Visitor 模式处理类型操作 | 类型封闭、操作频繁且多 | 多一跳虚分派、default fallback 有风险 | 加操作零侵入类型类 |
| InternalRow 定义为接口、五实现并存 | 文件 I/O / 内存计算 / 拼接需求迥异 | 多实现维护、getter 虚分派 | 上层逻辑与底层布局解耦 |
| BinaryRow 每字段固定 8 字节槽 | O(1) 投影 + CPU 字对齐 + 打包变长指针 | 小类型空间浪费 | 投影/排序极快 |
| Null 区 8 字节对齐（数组 4 字节） | 行常批量 null 检查、数组重空间 | 少字段时浪费 | `anyNull` 一次 long 查 64 列 |
| 变长数据 ≤7 字节内联 | 大量短字符串无需访问变长区 | 编码更复杂、阈值处形态突变 | 短串零指针追踪、cache 友好 |
| Decimal 双模（long ≤18 / BigDecimal） | 绝大多数精度 ≤18 | 双套逻辑、分界处性能突变 | 紧凑模式 Decimal≈long |
| Timestamp millis+nanoOfMillis 分离 | 统一纳秒 long 只覆盖约 292 年 | 高精度需两处协同 | 低精度单 long 内联、兼顾范围精度 |
| KeyValue key/value 分离 + 中置元数据列 | 排序只比 key、需安放 seq/kind | 结构更复杂、多 2 元数据列 | 排序快、CDC 语义清晰、投影边界明确 |
| 三套行序列化器并存 | 文件 I/O 求快、KV 存储求省 | 维护三套逻辑 | BinaryRow 极速 / RowCompacted 省 30–50% |
| BinaryRowSerializer 固定区不跨页 | 保住 O(1) 直读前提 | 页尾可能浪费 | 反序列化可零拷贝映射 |
| 运行时 Codegen（Janino） | Schema 运行时定、热路径量大 | 首次编译延迟、Metaspace 占用 | 消除虚分派、JIT 易内联 |
| Casting：Rule+Predicate + CastedRow 懒转换 | 一规则覆盖一族、演化常只改少数列 | 规则集需维护、失败语义需定义 | 演化读几乎零额外成本 |
| Flink/Spark 适配用 Wrapper 而非 Converter | 两套内存体系不能直接共享 | String 等须字节拷贝 | 数值零拷贝、内存安全 |
