## 1. 类型系统总览

### 解决什么问题

**核心业务问题：** 如何在一个湖仓系统中统一表达所有数据类型，并让 Flink、Spark、Hive 等不同计算引擎都能正确理解和处理？

**没有这个设计的后果：**
- 每个计算引擎使用自己的类型系统，导致数据在引擎间传递时需要大量转换代码
- 类型语义不一致（如 Flink 的 TIMESTAMP 与 Spark 的 TIMESTAMP 含义不同），导致查询结果错误
- Schema Evolution 时无法保证类型兼容性，可能导致历史数据无法读取

**实际场景：**
```java
// 场景1：Flink 写入数据，Spark 读取
// 如果没有统一类型系统，Decimal(10,2) 可能在 Spark 中被误读为 Decimal(10,3)

// 场景2：Schema 演化
// 原表：age INT
// 新表：age BIGINT
// 需要类型系统明确定义 INT -> BIGINT 的转换规则

// 场景3：文件格式适配
// Paimon TIMESTAMP(6) -> Parquet INT64(MICROS) -> ORC TIMESTAMP
// 需要类型系统保证精度不丢失
```

### 有什么坑

**误区陷阱：**
1. **VARCHAR 长度陷阱**：`VARCHAR(10)` 和 `VARCHAR(MAX)` 在 Paimon 中都是 `VarCharType`，但 `length` 不同。直接用 `typeRoot == VARCHAR` 判断会忽略长度约束
2. **Nullable 默认值**：所有 `DataType` 默认 `isNullable=true`，如果业务逻辑假设字段非空，必须显式检查
3. **Decimal 精度丢失**：`Decimal(38,10)` 转 `Decimal(10,2)` 会截断，但类型系统不会自动报错，需要在 Casting 层处理

**错误配置：**
```java
// 错误：直接比较 DataType 对象
if (type1 == type2) { ... }  // 永远为 false，因为是不同对象

// 正确：使用 equals
if (type1.equals(type2)) { ... }

// 错误：忽略 nullable 差异
DecimalType dt1 = new DecimalType(10, 2);  // nullable=true
DecimalType dt2 = new DecimalType(false, 10, 2);  // nullable=false
// dt1.equals(dt2) 返回 false
```

**生产环境注意事项：**
1. **类型升级路径**：只支持安全的类型升级（INT -> BIGINT），不支持降级（BIGINT -> INT）
2. **时区敏感类型**：`TIMESTAMP` 与 `TIMESTAMP_LTZ` 在跨时区查询时行为不同，必须明确选择
3. **复合类型嵌套深度**：`ARRAY<ARRAY<ARRAY<...>>>` 嵌套过深会导致序列化栈溢出，建议不超过 5 层

**性能陷阱：**
```java
// 陷阱：频繁调用 asSQLString()
for (DataType type : types) {
    String sql = type.asSQLString();  // 每次都重新构造字符串
}

// 优化：缓存结果
Map<DataType, String> sqlCache = new HashMap<>();
```

### 核心概念解释

**术语定义：**
- **DataTypeRoot**：类型的精确标识符，如 `INTEGER`、`VARCHAR`、`DECIMAL`。每个具体类型有唯一的 Root
- **DataTypeFamily**：类型的分组分类，如 `NUMERIC`（包含所有数值类型）、`CHARACTER_STRING`（包含 CHAR/VARCHAR）
- **Nullable**：字段是否允许 NULL 值。Paimon 中所有类型都支持 nullable，这是与 Java 原生类型的关键区别
- **Parameterized Type**：带参数的类型，如 `DECIMAL(precision, scale)`、`VARCHAR(length)`

**概念关系：**
```
DataType (抽象基类)
  ├── typeRoot: DataTypeRoot (精确标识)
  ├── isNullable: boolean (是否可空)
  └── 子类实现具体类型
      ├── 简单类型：IntType, DoubleType, BooleanType
      ├── 参数化类型：DecimalType(p,s), VarCharType(len)
      └── 复合类型：ArrayType, MapType, RowType

DataTypeFamily (枚举集合)
  ├── PREDEFINED (所有预定义类型)
  ├── NUMERIC (所有数值类型)
  ├── CHARACTER_STRING (CHAR, VARCHAR)
  └── COLLECTION (ARRAY, VECTOR, MULTISET)

一个 DataTypeRoot 可以属于多个 Family
例如：INTEGER 属于 PREDEFINED, NUMERIC, INTEGER_NUMERIC, EXACT_NUMERIC
```

**与其他系统对比：**

| 特性 | Paimon | Flink | Spark | Iceberg |
|------|--------|-------|-------|---------|
| 类型定义位置 | paimon-api (独立模块) | flink-table-common | catalyst (内部) | iceberg-api |
| Nullable 支持 | 所有类型都支持 | 所有类型都支持 | 所有类型都支持 | 通过 Schema 字段控制 |
| Decimal 最大精度 | 38 | 38 | 38 | 38 |
| Timestamp 精度 | 0-9 | 0-9 | 0-6 | 0-6 (微秒) |
| 复合类型嵌套 | 无限制 | 无限制 | 无限制 | 无限制 |
| 类型演化 | 支持安全升级 | 支持 | 支持 | 支持 |

### 设计理念

**为什么这样设计：**

1. **双层模块设计（paimon-api + paimon-common）**
   - **原因**：类型定义是公共契约，应该独立于实现细节
   - **权衡**：增加了模块数量，但降低了外部依赖的复杂度
   - **好处**：Flink/Spark Connector 只需依赖轻量的 paimon-api，不需要引入整个 paimon-common

2. **DataTypeRoot + DataTypeFamily 双层枚举**
   - **原因**：既需要精确匹配（switch/case），又需要批量判断（"是否数值类型"）
   - **权衡**：增加了枚举维护成本（新增类型需要同时更新 Root 和 Family）
   - **好处**：类型转换规则可以用 Family 批量匹配，避免为每对类型写一条规则

3. **Visitor 模式处理类型操作**
   - **原因**：类型集合是封闭的（新增类型罕见），但操作频繁（序列化、比较、转换）
   - **权衡**：Visitor 代码相对复杂，但操作逻辑集中
   - **好处**：新增操作只需实现一个 Visitor，不需要修改任何类型类

**架构演进：**
- **早期版本**：类型定义与序列化逻辑耦合在一起，导致类型类膨胀
- **当前版本**：类型定义（paimon-api）、类型检查（paimon-common）、序列化（paimon-common/serializer）三层分离
- **未来方向**：可能引入更多半结构化类型（如 JSON、XML），需要保持类型系统的扩展性

**业界对比：**
- **Iceberg**：类型系统更简单，只有 Schema 层面的类型定义，没有 Family 概念
- **Delta Lake**：直接使用 Spark 的类型系统，与 Spark 深度绑定
- **Hudi**：类型系统基于 Avro Schema，与 Avro 强耦合
- **Paimon 优势**：独立的类型系统，不绑定任何计算引擎，同时支持 Flink/Spark/Hive

### 1.1 DataType 继承体系

Paimon 的类型系统采用**双层模块设计**：类型定义位于 `paimon-api`（公共 API），类型检查工具位于 `paimon-common`。

**为什么这样设计？** 将类型定义放在 API 层，使得外部系统（Flink、Spark、Hive）可以仅依赖轻量的 API 模块进行类型交互，不需要引入整个 paimon-common。

**好处：** 降低模块耦合度，外部 Connector 只需关注类型契约，不需要关注内部实现。

```
DataType (abstract)                           [paimon-api/types/DataType.java]
├── isNullable: boolean                       -- 所有类型均支持 nullable
├── typeRoot: DataTypeRoot                    -- 类型根枚举
├── abstract defaultSize(): int               -- 默认大小估算
├── abstract copy(boolean): DataType          -- 深拷贝
├── abstract asSQLString(): String            -- SQL 表示
└── abstract accept(DataTypeVisitor<R>): R    -- Visitor 模式
    │
    ├── BooleanType
    ├── TinyIntType / SmallIntType / IntType / BigIntType
    ├── FloatType / DoubleType
    ├── DecimalType(precision, scale)
    ├── CharType(length) / VarCharType(length)
    ├── BinaryType(length) / VarBinaryType(length)
    ├── DateType / TimeType
    ├── TimestampType(precision) / LocalZonedTimestampType(precision)
    ├── ArrayType(elementType)
    ├── MapType(keyType, valueType)
    ├── MultisetType(elementType)
    ├── RowType(List<DataField>)
    ├── VectorType(elementType, length)
    ├── VariantType
    └── BlobType
```

**源码路径：**
- `paimon-api/src/main/java/org/apache/paimon/types/DataType.java`
- 所有子类型位于同一包：`paimon-api/src/main/java/org/apache/paimon/types/`

### 1.2 DataTypeRoot 与 DataTypeFamily

```java
// paimon-api/src/main/java/org/apache/paimon/types/DataTypeRoot.java
public enum DataTypeRoot {
    CHAR, VARCHAR,                                          // CHARACTER_STRING
    BOOLEAN,                                                // PREDEFINED
    BINARY, VARBINARY,                                      // BINARY_STRING
    DECIMAL, TINYINT, SMALLINT, INTEGER, BIGINT,           // NUMERIC
    FLOAT, DOUBLE,                                          // NUMERIC
    DATE, TIME_WITHOUT_TIME_ZONE,                           // DATETIME
    TIMESTAMP_WITHOUT_TIME_ZONE, TIMESTAMP_WITH_LOCAL_TIME_ZONE, // TIMESTAMP
    VARIANT, BLOB,                                          // PREDEFINED
    ARRAY, VECTOR, MULTISET,                                // COLLECTION
    MAP,                                                    // EXTENSION
    ROW                                                     // CONSTRUCTED
}
```

**为什么用两层枚举（Root + Family）？**

- `DataTypeRoot` 是每个具体类型的**精确标识**，用于 switch/case 分发
- `DataTypeFamily` 是类型的**分组分类**，用于批量判断（如"所有数值类型"）
- 一个 Root 可以属于多个 Family（如 `INTEGER` 同时属于 `PREDEFINED`、`NUMERIC`、`INTEGER_NUMERIC`、`EXACT_NUMERIC`）

**好处：** 在类型转换、过滤谓词下推等场景中，可以用 Family 快速匹配一类类型，而不用逐一枚举。

### 1.4 Visitor 模式与类型检查

`DataType` 提供 `accept(DataTypeVisitor<R>)` 方法，配合 `DataTypeDefaultVisitor<R>` 基类实现 Visitor 模式。

**为什么用 Visitor 模式？** 类型系统是一个封闭的枚举集合，新增类型相对罕见，但对类型的操作（序列化、比较、转换、大小估算）非常频繁。Visitor 模式将操作逻辑从类型类中解耦。

**好处：** 新增操作只需实现一个 Visitor，不需要修改任何类型类。

```
源码路径：
- paimon-api/types/DataTypeVisitor.java        -- Visitor 接口
- paimon-api/types/DataTypeDefaultVisitor.java  -- 默认实现基类
- paimon-api/types/DataTypeChecks.java          -- 静态检查工具
- paimon-common/types/InternalRowToSizeVisitor.java -- 大小估算 Visitor
```

## 2. InternalRow 体系

### 解决什么问题

**核心业务问题：** 如何在内存中高效表示一行数据，同时满足文件 I/O、内存计算、网络传输等不同场景的性能需求？

**没有这个设计的后果：**
- 如果只用 Java 对象（如 `Object[]`），文件读写需要逐字段序列化，性能极差
- 如果只用二进制格式（如 `byte[]`），内存计算需要频繁编解码，代码复杂
- 如果为每种场景实现独立的行格式，上层逻辑需要大量 if/else 分支处理不同格式

**实际场景：**
```java
// 场景1：文件读取 -> 内存计算 -> 文件写入
BinaryRow row = fileReader.read();           // 从文件读取二进制行
int age = row.getInt(2);                     // 直接从二进制读取，无需反序列化
if (age > 30) {
    GenericRow newRow = GenericRow.of(...);  // 构造新行
    fileWriter.write(newRow);                // 写入时自动转为 BinaryRow
}

// 场景2：Key-Value 拼接
InternalRow key = ...;
InternalRow value = ...;
JoinedRow fullRow = new JoinedRow(key, value);  // 零拷贝拼接
fileWriter.write(fullRow);                      // 写入时才真正合并

// 场景3：嵌套行读取
BinaryRow parentRow = ...;
InternalRow childRow = parentRow.getRow(5, 3);  // 读取嵌套行
// childRow 是 NestedRow，直接指向 parentRow 的变长区域，无需拷贝
```

### 有什么坑

**误区陷阱：**
1. **BinaryRow 可变性误解**：BinaryRow 的固定长度字段可以原地更新（`setInt`），但变长字段不能更新（会破坏内存布局）
2. **JoinedRow 的生命周期**：JoinedRow 只是引用，如果 row1/row2 被修改或回收，JoinedRow 会读到错误数据
3. **GenericRow 的类型安全**：`GenericRow.setField(pos, value)` 不检查类型，传错类型会在 getter 时抛 ClassCastException

**错误配置：**
```java
// 错误：复用 BinaryRow 时忘记 reset
BinaryRowWriter writer = new BinaryRowWriter(row);
writer.writeInt(0, 100);
writer.complete();
// 第二次写入前必须 reset，否则 null bits 不会清零
writer.reset();  // 必须调用
writer.writeInt(0, 200);
writer.complete();

// 错误：JoinedRow 引用了临时对象
JoinedRow joined = new JoinedRow();
for (InternalRow row : rows) {
    joined.replace(key, row);  // row 是循环变量，会被覆盖
    list.add(joined);          // 错误：所有元素都指向最后一个 row
}
// 正确：每次创建新的 JoinedRow 或深拷贝 row
```

**生产环境注意事项：**
1. **BinaryRow 的 MemorySegment 生命周期**：BinaryRow 指向的 MemorySegment 可能被回收，必须在使用前检查有效性
2. **GenericRow 的 GC 压力**：大量 GenericRow 会产生大量 Java 对象，在高吞吐场景下可能触发频繁 GC
3. **NestedRow 的跨段读取开销**：如果嵌套行的字段跨越多个 MemorySegment，读取性能会下降

**性能陷阱：**
```java
// 陷阱：频繁创建 JoinedRow
for (int i = 0; i < 1000000; i++) {
    JoinedRow joined = new JoinedRow(key, value);  // 每次都创建新对象
}

// 优化：复用 JoinedRow
JoinedRow joined = new JoinedRow();
for (int i = 0; i < 1000000; i++) {
    joined.replace(key, value);  // 复用对象
}
```

### 核心概念解释

**术语定义：**
- **InternalRow**：Paimon 内部行数据的统一接口，定义了所有 getter 方法
- **BinaryRow**：二进制行格式，数据存储在 MemorySegment 上，适合文件 I/O 和网络传输
- **GenericRow**：通用对象行格式，数据存储在 `Object[]` 中，适合内存计算和数据构造
- **JoinedRow**：拼接行，逻辑上将两个 InternalRow 拼接为一个，不做数据拷贝
- **NestedRow**：嵌套行，用于读取 BinaryRow 变长区域中的嵌套 ROW 类型字段
- **MemorySegment**：内存段，Paimon 的底层内存抽象，可以是堆内 `byte[]` 或堆外 `DirectByteBuffer`

**概念关系：**
```
InternalRow (interface)
  ├── 实现1：BinaryRow (二进制格式)
  │   └── 继承自 BinarySection (持有 MemorySegment[])
  ├── 实现2：GenericRow (对象格式)
  │   └── 持有 Object[] fields
  ├── 实现3：JoinedRow (拼接格式)
  │   └── 持有 InternalRow row1, InternalRow row2
  ├── 实现4：NestedRow (嵌套格式)
  │   └── 继承自 BinarySection，处理跨段读取
  └── 实现5：LazyGenericRow (延迟格式)
      └── 持有 Supplier<Object>[] suppliers

DataGetters (interface)
  └── 定义所有 getter 方法 (getInt, getString, ...)

DataSetters (interface)
  └── 定义所有 setter 方法 (setInt, setString, ...)

BinaryRow 同时实现 InternalRow 和 DataSetters
GenericRow 同时实现 InternalRow 和 DataSetters
JoinedRow 只实现 InternalRow (只读)
```

**与其他系统对比：**

| 特性 | Paimon | Flink | Spark | Arrow |
|------|--------|-------|-------|-------|
| 行接口 | InternalRow | RowData | InternalRow | - (列式) |
| 二进制格式 | BinaryRow | BinaryRowData | UnsafeRow | - |
| 对象格式 | GenericRow | GenericRowData | GenericInternalRow | - |
| 拼接格式 | JoinedRow | JoinedRowData | JoinedRow | - |
| 内存管理 | MemorySegment | MemorySegment | UnsafeRow (堆外) | Buffer |
| 字段对齐 | 8 字节 | 8 字节 | 8 字节 | 64 字节 (SIMD) |

### 设计理念

**为什么这样设计：**

1. **接口而非具体类**
   - **原因**：不同场景对行数据的性能需求完全不同
   - **权衡**：增加了实现复杂度（需要维护多个实现），但提升了灵活性
   - **好处**：上层逻辑（Merge Engine、Compaction）不需要关心底层行的具体实现

2. **BinaryRow 的 8 字节对齐**
   - **原因**：O(1) 随机访问 + CPU 字对齐 + 简化变长指针
   - **权衡**：牺牲了空间（小字段也占 8 字节），但换来了极致的读取性能
   - **好处**：投影（Projection）场景下，任何字段的读取都是 O(1) 的

3. **JoinedRow 的零拷贝设计**
   - **原因**：KeyValue 场景中，key 和 value 是分开的，但文件格式需要整行
   - **权衡**：JoinedRow 只是引用，生命周期管理复杂
   - **好处**：避免了热路径上的内存拷贝和对象分配

4. **NestedRow 的跨段读取**
   - **原因**：嵌套行存储在父行的变长部分，可能跨越多个 MemorySegment
   - **权衡**：跨段读取比单段读取慢，但避免了额外的反序列化
   - **好处**：支持 ROW 类型嵌套时的正确读取

**架构演进：**
- **早期版本**：只有 BinaryRow 和 GenericRow，JoinedRow 是后来为 KeyValue 场景引入的
- **当前版本**：五种实现各司其职，覆盖了所有常见场景
- **未来方向**：可能引入 ColumnarBatch（列式批处理），进一步提升向量化计算性能

**业界对比：**
- **Flink**：RowData 体系与 Paimon 几乎一致（Paimon 最初从 Flink 借鉴）
- **Spark**：UnsafeRow 是唯一的二进制格式，没有 JoinedRow（Join 时会拷贝数据）
- **Arrow**：纯列式存储，没有行的概念，适合 OLAP 但不适合 OLTP
- **Paimon 优势**：多种行实现灵活组合，既支持高效的文件 I/O，又支持灵活的内存计算

### 2.1 接口设计哲学

`InternalRow` 是 Paimon 内部行数据的统一接口，位于 `paimon-common/data/InternalRow.java`。

```
InternalRow (interface) extends DataGetters
├── getFieldCount(): int                      -- 字段数量
├── getRowKind(): RowKind                     -- Changelog 语义（INSERT/UPDATE_BEFORE/UPDATE_AFTER/DELETE）
├── setRowKind(RowKind): void                 -- 设置变更类型
└── [继承自 DataGetters 的 getter 方法]
    ├── isNullAt(int pos)
    ├── getBoolean/getByte/getShort/getInt/getLong/getFloat/getDouble
    ├── getString -> BinaryString
    ├── getDecimal(pos, precision, scale) -> Decimal
    ├── getTimestamp(pos, precision) -> Timestamp
    ├── getBinary -> byte[]
    ├── getVariant -> Variant
    ├── getBlob -> Blob
    ├── getArray -> InternalArray
    ├── getVector -> InternalVector
    ├── getMap -> InternalMap
    └── getRow(pos, numFields) -> InternalRow
```

**为什么抽象为接口而非具体类？**

不同场景对行数据有完全不同的性能需求：
- 文件 I/O 需要紧凑二进制格式（`BinaryRow`）减少序列化开销
- 内存计算需要灵活的 Java 对象格式（`GenericRow`）方便字段操作
- 跨表 Join 需要零拷贝拼接（`JoinedRow`）避免数据复制

**好处：** 统一接口使上层逻辑（Merge Engine、Compaction）不需要关心底层行的具体实现，同一段代码可以处理从文件读取的 BinaryRow 和从引擎构造的 GenericRow。

**关键辅助接口：**
- `DataGetters`：纯读接口，定义所有 getter（`paimon-common/data/DataGetters.java`）
- `DataSetters`：纯写接口，定义 setNullAt/setInt/setLong 等（`paimon-common/data/DataSetters.java`）
- `FieldGetter`：按类型生成的字段访问器（`InternalRow.createFieldGetter()`），避免在热路径上 switch/case

### 2.2 BinaryRow -- 紧凑二进制行

```
源码路径：paimon-common/src/main/java/org/apache/paimon/data/BinaryRow.java
```

**为什么需要 BinaryRow？** 它是文件读写和网络传输的核心格式。数据存储在 `MemorySegment` 上而非 Java 堆对象中，直接操作字节级数据。

**好处：**
1. **减少 GC 压力** -- 数据存储在连续的 byte[] 中，不产生大量 Java 对象
2. **高效序列化** -- 序列化即内存拷贝，无需字段级遍历
3. **高效比较** -- equals/hashCode 基于字节比较，无需逐字段拆箱
4. **零拷贝可能性** -- 多个 BinaryRow 可指向同一块 MemorySegment 的不同 offset

**核心特性：**
- 继承自 `BinarySection`，持有 `MemorySegment[]`、`offset`、`sizeInBytes`
- 同时实现 `InternalRow`（读）和 `DataSetters`（写）
- 固定长度部分保证在单个 MemorySegment 内

### 2.3 GenericRow -- 通用对象行

```
源码路径：paimon-common/src/main/java/org/apache/paimon/data/GenericRow.java
```

**为什么需要 GenericRow？** 用于中间计算、构造测试数据、类型转换等场景，其底层是 `Object[]` 数组。

**好处：**
1. **构造简便** -- `GenericRow.of(val1, val2, ...)` 一行代码创建
2. **字段可变** -- 通过 `setField(pos, value)` 可随时修改
3. **支持任意类型** -- 不受二进制编码限制

**与 BinaryRow 的关键区别：** GenericRow 的 getter 方法是直接 `(Type) fields[pos]` 强制类型转换，而 BinaryRow 是从 MemorySegment 读取字节后解码。GenericRow 的序列化需要通过 `InternalRowSerializer.toBinaryRow()` 先转为 BinaryRow。

### 2.4 JoinedRow -- 拼接行

```
源码路径：paimon-common/src/main/java/org/apache/paimon/data/JoinedRow.java
```

**为什么需要 JoinedRow？** 在 KeyValue 场景中，key 和 value 是分开的两个 InternalRow，但文件格式需要写入一个整行。JoinedRow 将两个行逻辑拼接为一个行，**不做任何数据拷贝**。

**好处：**
1. **零拷贝拼接** -- 不创建新的内存区域，只是根据 pos 分发到 row1 或 row2
2. **可复用** -- `replace(row1, row2)` 方法允许对象复用，避免热路径上的对象分配
3. **支持任意组合** -- row1/row2 可以是 BinaryRow、GenericRow 或其他任何 InternalRow 实现

**分发逻辑（第 120-126 行）：**
```java
if (pos < row1.getFieldCount()) {
    return row1.isNullAt(pos);
} else {
    return row2.isNullAt(pos - row1.getFieldCount());
}
```

### 2.5 NestedRow -- 嵌套行

```
源码路径：paimon-common/src/main/java/org/apache/paimon/data/NestedRow.java
```

**为什么需要 NestedRow？** BinaryRow 的固定长度部分必须在第一个 MemorySegment 内，但嵌套行存储在父行的变长部分中，其字段可能跨越多个 MemorySegment 边界。

**与 BinaryRow 的区别：**
- 内存格式完全相同（二进制兼容，equals 可互比）
- BinaryRow 通过 `segments[0].getInt(offset)` 直接读第一个 segment
- NestedRow 通过 `MemorySegmentUtils.getInt(segments, offset)` 处理跨 segment 读取

**好处：** 支持 ROW 类型嵌套时的正确读取，而不需要额外的反序列化步骤。

### 2.6 LazyGenericRow -- 延迟初始化行

```
源码路径：paimon-common/src/main/java/org/apache/paimon/data/LazyGenericRow.java
```

**为什么需要 LazyGenericRow？** 某些场景（如 Projection）中，并非所有字段都会被访问。LazyGenericRow 使用 `Supplier<Object>[]` 数组，只在首次访问时才计算字段值。

**好处：** 避免不必要的计算开销，特别是当字段计算涉及反序列化或类型转换等昂贵操作时。

### 2.7 实现对比与适用场景

| 实现 | 存储方式 | 适用场景 | 可变性 | 序列化开销 | GC 友好度 |
|------|---------|---------|--------|-----------|----------|
| BinaryRow | MemorySegment | 文件读写、网络传输、排序 | 固定长度字段可原地更新 | 极低（直接拷贝字节） | 高 |
| GenericRow | Object[] | 中间计算、数据构造、测试 | 完全可变 | 高（需转 BinaryRow） | 低 |
| JoinedRow | 两个 InternalRow 引用 | Key+Value 拼接、字段追加 | row1/row2 可替换 | 不直接序列化 | 中 |
| NestedRow | MemorySegment（跨段） | ROW 类型嵌套字段读取 | 固定长度字段可原地更新 | 极低 | 高 |
| LazyGenericRow | Supplier[] + Object[] | 延迟计算、Projection | 字段级延迟初始化 | 高 | 低 |

## 3. BinaryRow 内存布局深度解析

### 解决什么问题

**核心业务问题：** 如何设计一种内存布局，使得行数据既能高效序列化（文件 I/O），又能快速随机访问（投影查询），还能支持高效比较（排序）？

**没有这个设计的后果：**
- 如果用 Java 对象数组，序列化需要逐字段遍历，性能极差
- 如果用紧凑编码（如 Protobuf），随机访问需要从头扫描，O(n) 复杂度
- 如果用固定长度编码但不对齐，CPU 访问未对齐内存会有性能损失

**实际场景：**
```java
// 场景1：投影查询（只读第 5 和第 10 个字段）
BinaryRow row = ...;
int field5 = row.getInt(5);    // O(1) 直接计算偏移：offset + nullBits + 5*8
int field10 = row.getInt(10);  // O(1) 直接计算偏移：offset + nullBits + 10*8
// 不需要读取前面的字段

// 场景2：排序比较
BinaryRow row1 = ...;
BinaryRow row2 = ...;
int cmp = row1.equals(row2);  // 直接按字节比较，无需逐字段拆箱

// 场景3：序列化
BinaryRow row = ...;
outputStream.write(row.getSegments()[0].getArray(), row.getOffset(), row.getSizeInBytes());
// 直接写入字节，无需字段级遍历
```

### 有什么坑

**误区陷阱：**
1. **变长字段的内联阈值**：只有 <= 7 字节的变长数据才内联，8 字节及以上会外置。很多人以为是 8 字节
2. **Null Bits 的对齐计算**：`(arity + 63 + 8) / 64 * 8`，不是简单的 `arity / 8`，容易算错
3. **Fixed Part 必须在单个 Segment**：如果 BinaryRow 跨越多个 MemorySegment，Fixed Part 必须完整在第一个 Segment 内

**错误配置：**
```java
// 错误：手动计算字段偏移
int offset = row.getOffset() + 1 + arity / 8 + pos * 8;  // 错误的 null bits 计算
int value = segments[0].getInt(offset);

// 正确：使用 BinaryRow 的 getter
int value = row.getInt(pos);

// 错误：修改变长字段
BinaryRow row = ...;
row.setString(2, BinaryString.fromString("new value"));  // 可能破坏内存布局
// 正确：变长字段不能原地更新，需要重新构造整行
```

**生产环境注意事项：**
1. **MemorySegment 的边界检查**：在分页场景下，Fixed Part 可能跨页，需要调用 `checkSkipWriteForFixLengthPart` 跳到下一页
2. **变长数据的累积**：大量变长字段会导致 Variable Part 膨胀，单行可能超过 1MB，需要限制字段长度
3. **Null Bits 的批量检查**：`anyNull()` 方法通过逐 long 检查实现，但如果字段数很少（< 8），单次检查更快

**性能陷阱：**
```java
// 陷阱：频繁调用 getString
for (int i = 0; i < 1000000; i++) {
    BinaryString str = row.getString(5);  // 每次都解码变长指针
    if (str.toString().equals("target")) { ... }
}

// 优化：缓存 BinaryString
BinaryString str = row.getString(5);
String javaStr = str.toString();  // 只转换一次
for (int i = 0; i < 1000000; i++) {
    if (javaStr.equals("target")) { ... }
}
```

### 核心概念解释

**术语定义：**
- **Header**：1 字节，存储 RowKind（INSERT/UPDATE_BEFORE/UPDATE_AFTER/DELETE）
- **Null Bit Set**：位图，标记哪些字段为 NULL，对齐到 8 字节边界
- **Fixed-Length Part**：固定长度部分，每个字段占 8 字节，存储固定长度值或变长指针
- **Variable-Length Part**：变长部分，存储字符串、数组、高精度 Decimal 等变长数据
- **内联（Inline）**：<= 7 字节的变长数据直接存储在 Fixed Part 中，不占用 Variable Part
- **外置（Outline）**：> 7 字节的变长数据存储在 Variable Part，Fixed Part 存储指针

**内存布局示意：**
```
+------------------------------------------------------------------+
| Header | Null Bits      | Field 0 | Field 1 | ... | Field N     |
| 1 byte | 对齐到 8 字节   | 8 bytes | 8 bytes |     | 8 bytes     |
+------------------------------------------------------------------+
| Variable Part: var_data_0 | var_data_1 | ...                    |
+------------------------------------------------------------------+

Fixed Part 中的 8 字节编码：
- 固定长度字段（int/long/double）：直接存储值
- 短变长字段（<= 7 字节）：[1-bit=1][7-bit 长度][7-byte 数据]
- 长变长字段（> 7 字节）：[1-bit=0][31-bit 偏移][32-bit 长度]
```

**与其他系统对比：**

| 特性 | Paimon BinaryRow | Flink BinaryRowData | Spark UnsafeRow | Arrow |
|------|-----------------|-------------------|----------------|-------|
| Header 大小 | 1 字节 | 1 字节 | 无 | 无 |
| Null Bits 对齐 | 8 字节 | 8 字节 | 8 字节 | 64 字节 |
| 字段大小 | 8 字节 | 8 字节 | 8 字节 | 变长 |
| 变长内联阈值 | 7 字节 | 7 字节 | 7 字节 | 无（纯列式） |
| 随机访问 | O(1) | O(1) | O(1) | O(1) 列内 |
| 序列化开销 | 极低（直接拷贝） | 极低 | 极低 | 低（列压缩） |

### 设计理念

**为什么这样设计：**

1. **8 字节对齐**
   - **原因**：CPU 字对齐 + O(1) 随机访问 + 简化变长指针
   - **权衡**：牺牲空间（boolean 也占 8 字节），换取极致性能
   - **好处**：投影查询时，`getFieldOffset(pos) = offset + nullBits + pos * 8`，无需扫描前面字段

2. **Null Bits 对齐到 8 字节**
   - **原因**：批量 null 检查可以用 long 操作，一次检查 64 个字段
   - **权衡**：字段数少时（< 8）会浪费空间
   - **好处**：`anyNull()` 方法通过逐 long 比较实现 O(nullBitsSize/8) 的批量检测

3. **变长数据的两层编码（内联 vs 外置）**
   - **原因**：大量字符串字段实际值很短（如状态码 "OK"、国家代码 "CN"）
   - **权衡**：增加了编码复杂度（需要判断长度）
   - **好处**：短字符串读取跳过了指针追踪（pointer chasing），减少 cache miss

4. **Fixed Part 必须在单个 Segment**
   - **原因**：BinaryRow 的 getter 通过 `segments[0].getInt(offset)` 直接读第一个 segment
   - **权衡**：分页写入时需要检查剩余空间，可能浪费页尾空间
   - **好处**：避免了跨段读取的开销（像 NestedRow 那样）

**架构演进：**
- **早期版本**：所有变长数据都外置，没有内联优化
- **当前版本**：引入 7 字节内联阈值，大幅提升短字符串性能
- **未来方向**：可能引入字典编码（Dictionary Encoding），进一步压缩重复字符串

**业界对比：**
- **Flink BinaryRowData**：与 Paimon BinaryRow 几乎一致（Paimon 从 Flink 借鉴）
- **Spark UnsafeRow**：类似设计，但没有 Header（RowKind 存储在外部）
- **Arrow**：纯列式存储，每列独立编码，不适合行级操作
- **Parquet/ORC**：列式文件格式，行内编码更紧凑（按类型实际大小），但不支持随机访问
- **Paimon 优势**：在行式操作（排序、Join）和列式操作（投影、聚合）之间取得平衡

### 3.1 总体布局

```
+------------------------------------------------------------------+
|                      Fixed-Length Part                             |
| +--------+------------------+-----------------------------------+ |
| | Header | Null Bit Set     | Field Values (8 bytes each)       | |
| | 1 byte | 对齐到 8 字节边界 | field_0 | field_1 | ... | field_n | |
| +--------+------------------+-----------------------------------+ |
+------------------------------------------------------------------+
|                    Variable-Length Part                            |
| +-----------------------------------------------------------+    |
| | var_data_0 | var_data_1 | ...                              |    |
| +-----------------------------------------------------------+    |
+------------------------------------------------------------------+
```

### 3.2 Header 与 Null Bits

**Header（1 byte）：** 存储 `RowKind` 的字节值（0=INSERT, 1=UPDATE_BEFORE, 2=UPDATE_AFTER, 3=DELETE）。

**Null Bit Set（计算方法）：**
```java
// BinaryRow.java 第 77-79 行
public static int calculateBitSetWidthInBytes(int arity) {
    return ((arity + 63 + HEADER_SIZE_IN_BITS) / 64) * 8;
}
```
- `HEADER_SIZE_IN_BITS = 8`（1 字节 header 占用 8 bit）
- 公式含义：header(8 bits) + arity 个 null 位，向上对齐到 64 bit（8 字节）边界

**为什么对齐到 8 字节边界？** 这样每次可以用一个 `long` 读取完整的 null bits 段，利用 CPU 的 64 位字操作进行高效的 null 检查。`anyNull()` 方法（第 377-388 行）通过逐 long 检查实现 O(nullBitsSize/8) 的批量 null 检测。

**好处：** 对齐后的 null bits 在内存中天然适配 SIMD/CPU 字操作，bulk null check 只需一次 long 比较。

### 3.3 Fixed-Length Part

**每个字段固定占用 8 字节**，无论实际类型大小。

```java
// BinaryRow.java 第 80-82 行
public static int calculateFixPartSizeInBytes(int arity) {
    return calculateBitSetWidthInBytes(arity) + 8 * arity;
}
```

**为什么每个字段都是 8 字节？**

1. **统一偏移计算**：`getFieldOffset(pos) = offset + nullBitsSizeInBytes + pos * 8`，O(1) 随机访问
2. **天然对齐**：8 字节对齐可以用 `putLong`/`getLong` 一次操作读写，避免未对齐内存访问
3. **简化变长指针**：变长数据的（offset, length）可以打包到一个 long 中

**好处：** 任何字段的读取都是 O(1) 的，不需要扫描前面的字段计算偏移。这在投影（Projection）场景下特别重要。

**固定长度字段的存储方式：**
- `boolean`：存在 8 字节的最低位
- `byte/short/int`：小端序存在 8 字节的低位
- `long/double`：直接占满 8 字节
- `Decimal(p<=18)`：紧凑格式，unscaled long 直接存入
- `Timestamp(p<=3)`：毫秒值直接存入

### 3.4 Variable-Length Part 编码策略

变长数据（String、byte[]、高精度 Decimal、高精度 Timestamp 等）使用**两层编码策略**：

**策略一：短数据内联（<= 7 字节）**

```
Fixed Part 中的 8 字节：
[1-bit 标记=1] [7-bit 长度] [7-byte 数据]
```

**策略二：长数据外置（> 7 字节）**

```
Fixed Part 中的 8 字节：
[1-bit 标记=0] [31-bit 偏移量] [4-byte 长度]

Variable Part 中：
[offset 处的实际字节数据]
```

**源码定义（BinarySection.java 第 37-63 行）：**
```java
public static final int MAX_FIX_PART_DATA_SIZE = 7;
public static final long HIGHEST_FIRST_BIT = 0x80L << 56;        // 最高位标记
public static final long HIGHEST_SECOND_TO_EIGHTH_BIT = 0x7FL << 56; // 长度掩码
```

**为什么需要两层策略？**

大量字符串字段实际值很短（如状态码 "OK"、国家代码 "CN"），内联后完全不需要访问变长区域。

**好处：** 短字符串读取跳过了一次指针追踪（pointer chasing），减少 cache miss。

**高精度 Decimal 的特殊处理（BinaryRow.java 第 170-198 行）：**
```java
// precision > 18 时，用 BigDecimal 的 unscaledValue().toByteArray() 存到变长区域
// fixed part 存 [32-bit offset | 32-bit length]
setLong(pos, ((long) cursor << 32) | ((long) bytes.length));
```

**高精度 Timestamp 的特殊处理（BinaryRow.java 第 200-224 行）：**
```java
// precision > 3 时，millisecond 存到变长区域，nanoOfMillisecond 存到 fixed part 低 32 位
// fixed part 存 [32-bit offset | 32-bit nanoOfMillisecond]
setLong(pos, ((long) cursor << 32) | (long) value.getNanoOfMillisecond());
```

### 3.5 BinaryRowWriter 写入流程

```
源码路径：paimon-common/src/main/java/org/apache/paimon/data/BinaryRowWriter.java
```

写入分三步：
1. **reset()** -- 清零 null bits 区域，cursor 重置到 fixedSize
2. **写入字段** -- 固定长度字段直接写 fixed part；变长字段先写 fixed part 指针，再追加到 cursor 位置
3. **complete()** -- 设置 BinaryRow 的 totalSize

**为什么要求先 reset 再写入？** null bits 默认全 0（即所有字段默认非 null），只在 `setNullAt()` 时才设为 1。这避免了对每个字段都进行 null 设置的开销。

## 4. 列值类型实现

### 解决什么问题

**核心业务问题：** 如何在二进制格式（BinaryRow）中高效存储和访问复杂类型（字符串、Decimal、Timestamp、数组、Map）？

**没有这个设计的后果：**
- 如果用 Java String，每个字符串都是独立对象，GC 压力大，且 UTF-16 编码浪费空间
- 如果用 BigDecimal，所有精度的 Decimal 都需要对象分配，性能差
- 如果用 java.sql.Timestamp，时区处理复杂，且不支持纳秒精度

**实际场景：**
```java
// 场景1：字符串处理
BinaryRow row = ...;
BinaryString str = row.getString(2);  // 零拷贝，直接指向 BinaryRow 的变长区域
int len = str.getSizeInBytes();       // UTF-8 字节长度
String javaStr = str.toString();      // 需要时才转为 Java String

// 场景2：高精度 Decimal
Decimal d1 = Decimal.fromLong(12345, 10, 2);  // 紧凑模式，123.45
Decimal d2 = Decimal.fromBigDecimal(new BigDecimal("123456789012345678901234567890.12"), 38, 2);  // 非紧凑模式

// 场景3：Timestamp 精度
Timestamp ts1 = Timestamp.fromEpochMillis(1000);  // 毫秒精度，紧凑模式
Timestamp ts2 = Timestamp.fromMicros(1000000);    // 微秒精度，非紧凑模式
```

### 有什么坑

**误区陷阱：**
1. **BinaryString 的生命周期**：BinaryString 直接指向 BinaryRow 的内存，如果 BinaryRow 被回收，BinaryString 会读到错误数据
2. **Decimal 的精度边界**：`Decimal(18, 0)` 是紧凑模式，`Decimal(19, 0)` 是非紧凑模式，性能差异巨大
3. **Timestamp 的时区陷阱**：`TIMESTAMP` 是无时区的，`TIMESTAMP_LTZ` 是带本地时区的，混用会导致时间错误

**错误配置：**
```java
// 错误：BinaryString 脱离 BinaryRow 使用
BinaryString str = row.getString(2);
row = null;  // BinaryRow 被回收
String javaStr = str.toString();  // 可能读到错误数据

// 正确：立即转为 Java String
String javaStr = row.getString(2).toString();

// 错误：Decimal 精度不匹配
Decimal d = Decimal.fromLong(123456789012345678L, 38, 0);  // 超过 long 范围
// 正确：使用 BigDecimal
Decimal d = Decimal.fromBigDecimal(new BigDecimal("123456789012345678"), 38, 0);
```

**生产环境注意事项：**
1. **BinaryString 的 UTF-8 编码**：中文字符占 3 字节，`VARCHAR(10)` 只能存 3 个中文字符
2. **Decimal 的溢出检查**：`Decimal(10, 2)` 最大值是 99999999.99，超过会抛异常
3. **Timestamp 的精度丢失**：`Timestamp(3)` 只保留毫秒，微秒和纳秒会被截断

**性能陷阱：**
```java
// 陷阱：频繁调用 BinaryString.toString()
for (int i = 0; i < 1000000; i++) {
    String str = row.getString(2).toString();  // 每次都创建新 String
}

// 优化：缓存 String
String str = row.getString(2).toString();
for (int i = 0; i < 1000000; i++) {
    // 使用 str
}
```

### 核心概念解释

**术语定义：**
- **BinaryString**：UTF-8 编码的字符串，存储在 MemorySegment 上，零拷贝
- **Decimal**：精确小数，支持紧凑模式（long）和非紧凑模式（BigDecimal）
- **Timestamp**：时间戳，由 millisecond（long）和 nanoOfMillisecond（int）组成
- **InternalArray**：数组接口，实现类为 BinaryArray 和 GenericArray
- **InternalMap**：Map 接口，实现类为 BinaryMap 和 GenericMap
- **紧凑模式**：Decimal(p<=18) 和 Timestamp(p<=3) 使用 long 存储，可内联到 BinaryRow
- **非紧凑模式**：Decimal(p>18) 和 Timestamp(p>3) 使用复合结构，存储在变长区域

**类型对应关系：**
```
SQL 类型              Java 内部表示           BinaryRow 存储
-----------------------------------------------------------------
CHAR/VARCHAR         BinaryString           变长区域（UTF-8 字节）
DECIMAL(p<=18, s)    Decimal (long)         Fixed Part（8 字节）
DECIMAL(p>18, s)     Decimal (BigDecimal)   Variable Part
TIMESTAMP(p<=3)      Timestamp (long)       Fixed Part（8 字节）
TIMESTAMP(p>3)       Timestamp (long+int)   Variable Part（millis）+ Fixed Part（nanos）
ARRAY<T>             InternalArray          Variable Part（BinaryArray 格式）
MAP<K,V>             InternalMap            Variable Part（BinaryMap 格式）
ROW<...>             InternalRow            Variable Part（NestedRow 格式）
```

**与其他系统对比：**

| 特性 | Paimon | Flink | Spark | Java 标准库 |
|------|--------|-------|-------|------------|
| 字符串编码 | UTF-8 | UTF-8 | UTF-8 | UTF-16 |
| Decimal 最大精度 | 38 | 38 | 38 | 无限 |
| Decimal 紧凑阈值 | 18 | 18 | 无（统一用 Decimal128） | 无 |
| Timestamp 精度 | 0-9 | 0-9 | 0-6 | 纳秒 |
| Timestamp 紧凑阈值 | 3 | 3 | 无 | 无 |

### 设计理念

**为什么这样设计：**

1. **BinaryString 使用 UTF-8 编码**
   - **原因**：UTF-8 对 ASCII 字符只占 1 字节，比 UTF-16 节省 50% 空间
   - **权衡**：中文字符占 3 字节，比 UTF-16 多 50%
   - **好处**：SQL 字符串以 ASCII 为主（表名、列名、关键字），UTF-8 更紧凑

2. **Decimal 的双模存储**
   - **原因**：绝大多数业务场景的精度在 18 以内（如金额、百分比）
   - **权衡**：需要维护两套逻辑（紧凑 vs 非紧凑）
   - **好处**：紧凑模式下 Decimal 就是一个 long，比较、加减、乘除都是 long 操作

3. **Timestamp 的分离存储**
   - **原因**：统一纳秒需要 64 位 long，但范围只能表示约 292 年
   - **权衡**：非紧凑模式需要两个字段（millis + nanos）
   - **好处**：紧凑模式（毫秒）覆盖了绝大多数场景，且只需一个 long

4. **BinaryArray/BinaryMap 的二进制格式**
   - **原因**：数组和 Map 是复合类型，需要递归序列化
   - **权衡**：二进制格式不如 JSON 可读，但性能高
   - **好处**：零拷贝，直接指向 BinaryRow 的变长区域

**架构演进：**
- **早期版本**：所有 Decimal 都用 BigDecimal，性能差
- **当前版本**：引入紧凑模式，大幅提升常见精度的性能
- **未来方向**：可能引入 Decimal256（256 位精度），支持更高精度的科学计算

**业界对比：**
- **Flink**：与 Paimon 几乎一致（Paimon 从 Flink 借鉴）
- **Spark**：Decimal 统一用 Decimal128（128 位），没有紧凑模式
- **Arrow**：纯列式存储，Decimal 用 128 位或 256 位定长编码
- **Paimon 优势**：紧凑模式在常见精度下性能极高，且与 Flink 无缝集成

### 4.1 BinaryString

```
源码路径：paimon-common/src/main/java/org/apache/paimon/data/BinaryString.java
```

继承 `BinarySection`，存储 UTF-8 编码的字符串字节。

**为什么不用 Java String？** Java String 使用 UTF-16 编码，每个字符 2 字节。对于以 ASCII 为主的数据（如 SQL 字符串），UTF-8 编码体积更小。更重要的是，BinaryString 直接指向 MemorySegment，避免了 String 对象的创建和 UTF-16 转换。

**好处：**
1. 内存效率 -- UTF-8 编码更紧凑
2. 零拷贝 -- 可以直接引用 BinaryRow 的变长区域，不需要额外分配
3. 高效比较 -- 直接按字节比较，与文件格式中的排序一致

### 4.2 Decimal

```
源码路径：paimon-common/src/main/java/org/apache/paimon/data/Decimal.java
```

**双模存储策略：**
- **紧凑模式**（precision <= 18 即 `MAX_COMPACT_PRECISION`）：用 `long` 存储 unscaled value
- **非紧凑模式**（precision > 18）：用 `BigDecimal` 存储

```java
// Decimal.java 第 60-70 行
// 如果 precision <= MAX_COMPACT_PRECISION:
//   (longVal, scale) 表示值，decimalVal 可能被缓存
// 否则:
//   decimalVal 表示值，longVal 未定义
```

**为什么 18 是分界线？** Java `long` 最大值约 9.2 * 10^18，可以精确表示 18 位十进制数字。`Decimal.POW10` 数组预计算了 10 的幂次用于快速转换。

**好处：**
1. 大多数业务场景的精度在 18 以内，紧凑模式下 Decimal 就是一个 long，在 BinaryRow 中内联存储
2. 比较操作在紧凑模式下是 `Long.compare`，极快
3. 不需要 BigDecimal 对象，减少 GC 压力

### 4.3 Timestamp

```
源码路径：paimon-common/src/main/java/org/apache/paimon/data/Timestamp.java
```

由两部分组成：
- `millisecond: long` -- 自 epoch 以来的毫秒数
- `nanoOfMillisecond: int` -- 毫秒内的纳秒偏移（0-999,999）

**紧凑模式（precision <= 3）：** 只需 `millisecond`，存入一个 long。
**非紧凑模式（precision > 3）：** 需要额外的 `nanoOfMillisecond`。

**为什么用 millisecond + nanoOfMillisecond 而非统一用纳秒？** 统一纳秒需要 64 位 long，但 long 的范围只能表示约 292 年（至 epoch 前后）。分离存储可以覆盖更大的时间范围。

**好处：** 低精度（秒/毫秒级）时间戳只需一个 long，在 BinaryRow 中内联存储，读写极快。

### 4.4 BinaryArray

```
源码路径：paimon-common/src/main/java/org/apache/paimon/data/BinaryArray.java
```

**内存布局：**
```
[4-byte size] + [null bits (4-byte 边界对齐)] + [values 或 offset&length] + [变长部分]
```

**为什么 null bits 使用 4 字节对齐（而非 BinaryRow 的 8 字节对齐）？** 数组通常元素数量大，4 字节对齐可以节省空间。在数组场景中，批量 null 检查的性能收益不如行场景明显。

### 4.5 BinaryMap

```
源码路径：paimon-common/src/main/java/org/apache/paimon/data/BinaryMap.java
```

**内存布局：**
```
[4-byte keyArraySize] + [Key BinaryArray] + [Value BinaryArray]
```

**为什么用两个并列的 BinaryArray？** 这种设计使得 key 和 value 可以独立遍历，支持按 key 排序后的高效查找。同时复用了 BinaryArray 的所有序列化/反序列化逻辑。

## 5. KeyValue 结构

### 解决什么问题

**核心业务问题：** 在 LSM-tree 存储引擎中，如何组织主键表的数据结构，使得排序、合并、查找都能高效进行？

**没有这个设计的后果：**
- 如果 key 和 value 不分离，排序时需要比较整行，浪费 CPU
- 如果没有 sequenceNumber，无法区分同一 key 的多个版本，Merge 时会出错
- 如果没有 RowKind，无法支持 CDC（Change Data Capture），只能做全量更新

**实际场景：**
```java
// 场景1：LSM Compaction
List<KeyValue> kvs = readFromMultipleFiles();
kvs.sort((kv1, kv2) -> {
    int cmp = compareKey(kv1.key(), kv2.key());  // 只比较 key
    if (cmp == 0) {
        return Long.compare(kv2.sequenceNumber(), kv1.sequenceNumber());  // 序列号降序
    }
    return cmp;
});

// 场景2：Merge Engine
KeyValue latest = null;
for (KeyValue kv : kvs) {
    if (kv.valueKind() == RowKind.DELETE) {
        latest = null;  // 删除记录
    } else if (kv.valueKind().isAdd()) {
        latest = kv;    // 新增或更新
    }
}

// 场景3：CDC 写入
KeyValue insertKv = new KeyValue()
    .replace(key, RowKind.INSERT, value)
    .setSequenceNumber(100);
KeyValue deleteKv = new KeyValue()
    .replace(key, RowKind.DELETE, value)
    .setSequenceNumber(101);
```

### 有什么坑

**误区陷阱：**
1. **sequenceNumber 的分配时机**：sequenceNumber 是写入后分配的，构造 KeyValue 时不需要设置
2. **level 的含义**：level 是文件读取后分配的 LSM 层级，不是用户设置的
3. **valueKind 与 RowKind 的区别**：valueKind 是 KeyValue 的变更类型，RowKind 是 InternalRow 的变更类型，两者可能不同

**错误配置：**
```java
// 错误：手动设置 sequenceNumber
KeyValue kv = new KeyValue()
    .replace(key, RowKind.INSERT, value)
    .setSequenceNumber(100);  // 错误：sequenceNumber 应该由引擎分配

// 正确：不设置 sequenceNumber
KeyValue kv = new KeyValue().replace(key, RowKind.INSERT, value);

// 错误：混淆 valueKind 和 value.getRowKind()
KeyValue kv = ...;
if (kv.valueKind() == RowKind.INSERT) {  // valueKind 是 KeyValue 的变更类型
    if (kv.value().getRowKind() == RowKind.INSERT) {  // value 的 RowKind 可能不同
        // ...
    }
}
```

**生产环境注意事项：**
1. **sequenceNumber 的单调性**：同一 key 的多个版本，sequenceNumber 必须严格递增，否则 Merge 会出错
2. **RowKind 的语义**：UPDATE_BEFORE 和 UPDATE_AFTER 必须成对出现，否则 Changelog 不完整
3. **level 的范围**：level 从 0 开始，最大值取决于 LSM 配置，通常不超过 10

**性能陷阱：**
```java
// 陷阱：频繁创建 KeyValue
for (int i = 0; i < 1000000; i++) {
    KeyValue kv = new KeyValue().replace(key, RowKind.INSERT, value);  // 每次都创建新对象
}

// 优化：复用 KeyValue
KeyValue kv = new KeyValue();
for (int i = 0; i < 1000000; i++) {
    kv.replace(key, RowKind.INSERT, value);  // 复用对象
}
```

### 核心概念解释

**术语定义：**
- **KeyValue**：主键表的数据单元，由 key（主键字段）、value（非主键字段）、sequenceNumber（版本号）、valueKind（变更类型）、level（LSM 层级）组成
- **sequenceNumber**：写入后分配的全局递增序列号，用于区分同一 key 的多个版本
- **valueKind**：变更类型，包括 INSERT、UPDATE_BEFORE、UPDATE_AFTER、DELETE
- **level**：文件读取后分配的 LSM 层级，level 0 是最新数据，level 越大数据越旧
- **KeyValue Schema**：KeyValue 在文件中的物理格式，是一个扁平的 RowType

**KeyValue 结构：**
```
KeyValue
├── key: InternalRow              -- 主键字段
├── sequenceNumber: long          -- 版本号（写入后分配）
├── valueKind: RowKind            -- 变更类型
├── value: InternalRow            -- 非主键字段
└── level: int                    -- LSM 层级（读取后分配）

文件中的物理格式：
[key_field_0, ..., key_field_n, _SEQUENCE_NUMBER, _VALUE_KIND, value_field_0, ..., value_field_m]
```

**与其他系统对比：**

| 特性 | Paimon | RocksDB | LevelDB | HBase |
|------|--------|---------|---------|-------|
| Key-Value 分离 | 是 | 是 | 是 | 是 |
| 版本号 | sequenceNumber (long) | SequenceNumber (uint64) | SequenceNumber (uint64) | Timestamp (long) |
| 变更类型 | RowKind (4 种) | Type (Put/Delete) | Type (Put/Delete) | Type (Put/Delete) |
| LSM 层级 | level (int) | level (int) | level (int) | 无（Region 级别） |
| 物理格式 | 扁平 RowType | 二进制 key+value | 二进制 key+value | 二进制 key+value |

### 设计理念

**为什么这样设计：**

1. **Key-Value 分离**
   - **原因**：排序只涉及 key，Merge 操作需要分别处理 key 和 value
   - **权衡**：增加了结构复杂度（需要 JoinedRow 拼接）
   - **好处**：排序性能提升（只比较 key），投影优化（只读 key 或只读 value）

2. **sequenceNumber 的全局递增**
   - **原因**：LSM Compaction 需要根据版本号决定保留哪个版本
   - **权衡**：需要全局计数器，可能成为并发瓶颈
   - **好处**：Merge 逻辑简单，只需保留 sequenceNumber 最大的版本

3. **valueKind 的 4 种类型**
   - **原因**：支持 CDC（Change Data Capture），需要区分 INSERT/UPDATE/DELETE
   - **权衡**：增加了 Merge 逻辑的复杂度
   - **好处**：支持 Changelog 语义，可以回溯历史变更

4. **level 的后置分配**
   - **原因**：level 是文件的属性，不是数据的属性
   - **权衡**：KeyValue 对象需要额外的 level 字段
   - **好处**：Compaction 时可以根据 level 优化合并策略

**架构演进：**
- **早期版本**：没有 valueKind，只支持全量更新
- **当前版本**：引入 valueKind，支持 CDC 和 Changelog
- **未来方向**：可能引入 TTL（Time To Live），支持自动过期删除

**业界对比：**
- **RocksDB/LevelDB**：只有 Put/Delete 两种类型，不支持 UPDATE_BEFORE/UPDATE_AFTER
- **HBase**：使用 Timestamp 作为版本号，支持多版本读取
- **Iceberg**：没有 KeyValue 概念，使用 Parquet 的 Row Group 作为数据单元
- **Paimon 优势**：支持完整的 CDC 语义，与 Flink CDC 无缝集成

### 5.1 核心字段

```
源码路径：paimon-core/src/main/java/org/apache/paimon/KeyValue.java
```

```java
public class KeyValue {
    private InternalRow key;           // 主键字段
    private long sequenceNumber;       // 写入后分配的序列号
    private RowKind valueKind;         // 变更类型 (INSERT/UPDATE_BEFORE/UPDATE_AFTER/DELETE)
    private InternalRow value;         // 值字段
    private int level;                 // 文件读取后分配的 LSM 层级
}
```

**为什么 key 和 value 分开而不是一个整体行？**

1. **排序只涉及 key** -- LSM-tree 的排序和查找只需要 key 部分
2. **Merge 操作需要分别处理** -- key 作为合并键，value 作为合并内容
3. **投影优化** -- 查询时可以只读 key 或只读 value

**好处：** 存储层可以对 key 和 value 使用不同的压缩和索引策略。

### 5.2 RowKind 与 Changelog 语义

```
源码路径：paimon-api/src/main/java/org/apache/paimon/types/RowKind.java
```

| RowKind | 短字符串 | 字节值 | 语义 |
|---------|---------|--------|------|
| INSERT | +I | 0 | 新增记录 |
| UPDATE_BEFORE | -U | 1 | 更新前的旧值（用于回撤） |
| UPDATE_AFTER | +U | 2 | 更新后的新值 |
| DELETE | -D | 3 | 删除记录 |

**为什么存储字节值而非字符串？** 字节值只占 1 bit（存储在 BinaryRow header 的第一个字节），而字符串需要 2 字节。KeyValue 是热路径上最频繁创建的对象，每个字节都重要。

**好处：** RowKind 的 `isAdd()` 和 `isRetract()` 方法提供了简洁的语义判断，Merge Engine 可以快速决定记录是新增还是撤回。

### 5.3 KeyValue Schema 组织

KeyValue 在文件中的物理格式是一个扁平的 RowType：

```java
// KeyValue.java 第 113-115 行
public static RowType schema(RowType keyType, RowType valueType) {
    return new RowType(false, createKeyValueFields(keyType.getFields(), valueType.getFields()));
}
```

**文件中的字段顺序：**
```
[key_field_0, key_field_1, ..., key_field_n, _SEQUENCE_NUMBER, _VALUE_KIND, value_field_0, ..., value_field_m]
```

**为什么把 SEQUENCE_NUMBER 和 VALUE_KIND 放在 key 和 value 之间？**

1. 它们在逻辑上是 key-value 对的元数据，放在中间形成清晰的分界
2. 文件读取时，key projection 是前 n 个字段，value projection 是后 m 个字段，中间 2 个固定字段总是需要读取
3. `KeyValue.project()` 方法（第 140-165 行）基于此布局构建投影映射

## 6. 序列化框架

### 解决什么问题

**核心业务问题：** 如何在内存对象（InternalRow）和字节流（文件/网络）之间高效转换，同时支持多种场景（文件 I/O、分页缓冲、LSM KV 存储）？

**没有这个设计的后果：**
- 如果用 Java 序列化，性能极差且字节流不可控
- 如果用 Protobuf/Avro，需要额外的 Schema 定义，且不支持零拷贝
- 如果只有一种序列化方案，无法在空间和性能之间权衡

**实际场景：**
```java
// 场景1：文件写入（使用 BinaryRowSerializer）
BinaryRowSerializer serializer = new BinaryRowSerializer(rowType);
DataOutputView output = ...;
for (InternalRow row : rows) {
    BinaryRow binaryRow = serializer.toBinaryRow(row);  // 统一转为 BinaryRow
    serializer.serialize(binaryRow, output);            // 直接写入字节
}

// 场景2：LSM KV 存储（使用 RowCompactedSerializer）
RowCompactedSerializer keySerializer = new RowCompactedSerializer(keyType);
byte[] keyBytes = keySerializer.serializeToBytes(key);  // 紧凑编码，节省空间

// 场景3：分页缓冲（使用 BinaryRowSerializer 的分页优化）
AbstractPagedOutputView pagedOutput = ...;
serializer.serializeToPages(row, pagedOutput);  // 自动处理跨页
```

### 有什么坑

**误区陷阱：**
1. **Serializer 的线程安全性**：有状态的 Serializer（如 InternalRowSerializer）不是线程安全的，必须每个线程一个实例
2. **BinaryRow 的分页约束**：Fixed Part 必须在单个页内，Variable Part 可以跨页
3. **RowCompactedSerializer 的随机访问**：紧凑编码不支持 O(1) 随机访问，只能顺序读取

**错误配置：**
```java
// 错误：多线程共享 Serializer
InternalRowSerializer serializer = new InternalRowSerializer(rowType);
ExecutorService executor = Executors.newFixedThreadPool(10);
for (int i = 0; i < 10; i++) {
    executor.submit(() -> {
        serializer.serialize(row, output);  // 线程不安全
    });
}

// 正确：每个线程一个 Serializer
InternalRowSerializer serializer = new InternalRowSerializer(rowType);
ExecutorService executor = Executors.newFixedThreadPool(10);
for (int i = 0; i < 10; i++) {
    executor.submit(() -> {
        InternalRowSerializer threadSerializer = (InternalRowSerializer) serializer.duplicate();
        threadSerializer.serialize(row, output);
    });
}

// 错误：用 RowCompactedSerializer 做随机访问
RowCompactedSerializer serializer = new RowCompactedSerializer(rowType);
byte[] bytes = serializer.serializeToBytes(row);
// 无法直接读取第 5 个字段，必须从头反序列化
```

**生产环境注意事项：**
1. **序列化格式的兼容性**：BinaryRowSerializer 的格式与 BinaryRow 内存布局一致，升级时需要保证兼容
2. **RowCompactedSerializer 的空间节省**：通常比 BinaryRowSerializer 小 30-50%，但反序列化慢 10-20%
3. **分页写入的页大小**：页太小会导致频繁跨页，页太大会浪费内存，建议 32KB-64KB

**性能陷阱：**
```java
// 陷阱：频繁调用 serializeToBytes
for (int i = 0; i < 1000000; i++) {
    byte[] bytes = serializer.serializeToBytes(row);  // 每次都创建新数组
}

// 优化：复用 DataOutputView
ByteArrayOutputStreamWithPos output = new ByteArrayOutputStreamWithPos();
for (int i = 0; i < 1000000; i++) {
    output.reset();
    serializer.serialize(row, new DataOutputViewStreamWrapper(output));
    byte[] bytes = output.toByteArray();
}
```

### 核心概念解释

**术语定义：**
- **Serializer**：序列化器接口，定义 serialize/deserialize 方法
- **BinaryRowSerializer**：BinaryRow 的原生序列化器，格式为 `[4-byte length] + [BinaryRow 字节]`
- **InternalRowSerializer**：通用 InternalRow 序列化器，先转为 BinaryRow 再序列化
- **RowCompactedSerializer**：紧凑编码序列化器，按类型实际大小编码，节省空间
- **PagedTypeSerializer**：支持分页读写的序列化器，处理跨页场景
- **DataOutputView/DataInputView**：字节流抽象，类似 Java 的 DataOutputStream/DataInputStream

**序列化器层次：**
```
Serializer<T>
├── duplicate(): Serializer<T>      -- 深拷贝（线程安全）
├── copy(T): T                      -- 深拷贝对象
├── serialize(T, DataOutputView)    -- 序列化
└── deserialize(DataInputView): T   -- 反序列化

PagedTypeSerializer<T> extends Serializer<T>
├── serializeToPages(T, AbstractPagedOutputView)  -- 分页写入
└── mapFromPages(AbstractPagedInputView): T       -- 分页读取（零拷贝）

AbstractRowDataSerializer<T> extends PagedTypeSerializer<T>
├── toBinaryRow(T): BinaryRow       -- 统一转为 BinaryRow
├── BinaryRowSerializer             -- BinaryRow 原生序列化
└── InternalRowSerializer           -- 通用 InternalRow 序列化

RowCompactedSerializer              -- 紧凑编码（独立实现）
```

**与其他系统对比：**

| 特性 | Paimon | Flink | Spark | Kryo | Protobuf |
|------|--------|-------|-------|------|----------|
| 序列化格式 | 自定义二进制 | 自定义二进制 | UnsafeRow | 自定义二进制 | Protobuf |
| 零拷贝支持 | 是（BinaryRow） | 是（BinaryRowData） | 是（UnsafeRow） | 否 | 否 |
| 分页支持 | 是 | 是 | 否 | 否 | 否 |
| 紧凑编码 | 是（RowCompacted） | 否 | 否 | 是 | 是 |
| Schema 演化 | 支持 | 支持 | 支持 | 有限支持 | 支持 |

### 设计理念

**为什么这样设计：**

1. **多种序列化器并存**
   - **原因**：不同场景对空间和性能的需求不同
   - **权衡**：增加了维护成本（需要维护多套序列化逻辑）
   - **好处**：文件 I/O 用 BinaryRowSerializer（快），LSM KV 用 RowCompactedSerializer（省空间）

2. **BinaryRowSerializer 的简单格式**
   - **原因**：BinaryRow 本身就是紧凑的二进制格式
   - **权衡**：格式不如 Protobuf 灵活（不支持字段重排序）
   - **好处**：序列化/反序列化是纯内存拷贝，性能极高

3. **RowCompactedSerializer 的紧凑编码**
   - **原因**：LSM KV 存储场景，数据量大且主要是顺序读取
   - **权衡**：不支持 O(1) 随机访问，反序列化需要从头扫描
   - **好处**：比 BinaryRowSerializer 小 30-50%，减少 I/O 开销

4. **分页支持（PagedTypeSerializer）**
   - **原因**：大行数据可能超过单页大小，需要跨页存储
   - **权衡**：增加了序列化复杂度（需要检查页边界）
   - **好处**：支持任意大小的行，且 Fixed Part 不跨页保证了性能

**架构演进：**
- **早期版本**：只有 BinaryRowSerializer，所有场景都用同一种格式
- **当前版本**：引入 RowCompactedSerializer，针对存储场景优化
- **未来方向**：可能引入列式序列化（Columnar Serializer），进一步提升压缩率

**业界对比：**
- **Flink**：与 Paimon 类似，但没有 RowCompactedSerializer
- **Spark**：只有 UnsafeRow 一种格式，不支持紧凑编码
- **Arrow**：纯列式序列化，不适合行级操作
- **Protobuf/Avro**：通用序列化框架，但不支持零拷贝，性能不如自定义格式
- **Paimon 优势**：多种序列化器灵活组合，既有极致性能（BinaryRow），又有极致空间（RowCompacted）

### 6.1 Serializer 接口体系

```
源码路径：paimon-common/src/main/java/org/apache/paimon/data/serializer/Serializer.java
```

```
Serializer<T> (interface)
├── duplicate(): Serializer<T>      -- 深拷贝（无状态可返回 this）

├── serialize(T, DataOutputView)    -- 序列化到输出视图
```

**为什么 Serializer 需要 `duplicate()` 方法？** 有状态的 Serializer（如 InternalRowSerializer 内部持有 BinaryRowWriter）不是线程安全的。`duplicate()` 允许每个线程持有独立的实例。

```

    ├── toBinaryRow(T): BinaryRow               -- 关键：任意行 -> BinaryRow

    └── InternalRowSerializer                   -- 通用 InternalRow 序列化器
```

### 6.2 BinaryRowSerializer

```
源码路径：paimon-common/src/main/java/org/apache/paimon/data/serializer/BinaryRowSerializer.java
```

**序列化格式：** `[4-byte length] + [BinaryRow 字节内容]`

**为什么格式如此简单？** BinaryRow 本身就是紧凑的二进制格式，序列化只需要写长度前缀然后拷贝字节。

**好处：**
1. 序列化/反序列化是纯内存拷贝，性能极高
2. 支持直接映射（`mapFromPages`），反序列化时不拷贝数据，直接 pointTo 分页内存

**分页操作的核心优化（第 258-279 行）：**
```java
// 写入时检查当前页剩余空间是否能容纳固定长度部分
// 不够则跳到下一页，避免固定长度部分跨页
private int checkSkipWriteForFixLengthPart(AbstractPagedOutputView out) throws IOException {
    int available = out.getSegmentSize() - out.getCurrentPositionInSegment();
    if (available < getSerializedRowFixedPartLength()) {
        out.advance();
        return available;
    }
    return 0;
}
```

**为什么固定长度部分不能跨页？** BinaryRow 的 getter 通过 `segments[0].getInt(offset)` 直接读第一个 segment。如果 fixed part 跨页，就需要处理跨段读取（像 NestedRow 那样），显著增加热路径开销。

### 6.3 InternalRowSerializer

```
源码路径：paimon-common/src/main/java/org/apache/paimon/data/serializer/InternalRowSerializer.java
```

**核心职责：** 将任意 `InternalRow` 实现统一为 `BinaryRow` 格式。

```java
// InternalRowSerializer.java 第 119-125 行
public BinaryRow toBinaryRow(InternalRow row) {
    if (row instanceof BinaryRow) {
        return (BinaryRow) row;   // 已经是 BinaryRow，零成本
    }
    rowHelper.copyInto(row);      // 使用 RowHelper 逐字段拷贝到 BinaryRow
    return rowHelper.reuseRow();
}
```

**为什么需要 toBinaryRow 转换？** 文件写入、排序、分页存储都要求 BinaryRow 格式。InternalRowSerializer 作为桥接层，使上层代码可以透明地处理 GenericRow、JoinedRow 等非二进制实现。

### 6.4 RowCompactedSerializer

```
源码路径：paimon-common/src/main/java/org/apache/paimon/data/serializer/RowCompactedSerializer.java
```

**这是 BinaryRowSerializer 之外的第二种行序列化方案。** 与 BinaryRow 的 8 字节对齐不同，它使用真正紧凑的编码：

| 特性 | BinaryRowSerializer | RowCompactedSerializer |
|------|-------------------|----------------------|
| null bits 对齐 | 8 字节 | 1 字节 |
| 固定字段大小 | 统一 8 字节 | 按类型实际大小（boolean=1, int=4, long=8） |
| 变长数据长度 | 4 字节 int | VarInt 编码 |
| 随机访问 | O(1) | 需要顺序扫描 |
| 适用场景 | 内存排序、分页缓冲 | LSM KV 存储、Manifest 序列化 |

**为什么需要两种序列化方案？**

BinaryRow 为 O(1) 随机访问牺牲了空间（每字段 8 字节对齐）。在存储场景（如 LSM KV 的 key 和 value 序列化），数据量大且主要是顺序读取，紧凑编码可以显著减少存储和 I/O 开销。

**好处：** RowCompactedSerializer 生成的字节通常比 BinaryRowSerializer 小 30-50%（取决于字段类型分布），在 LSM compaction 等 I/O 密集场景中带来可观的性能提升。

**内建 SliceComparator：** RowCompactedSerializer 提供了 `createSliceComparator()` 方法，可以直接在序列化字节上比较行，无需先反序列化。

## 7. 文件格式适配

### 解决什么问题

**核心业务问题：** 如何将 Paimon 的类型系统映射到 ORC、Parquet 等列式文件格式，同时保证类型语义不丢失、精度不损失？

**没有这个设计的后果：**
- 如果类型映射不正确，Decimal(10,2) 可能被读成 Decimal(10,3)，导致金额错误
- 如果精度映射不当，Timestamp(6) 可能丢失微秒精度，导致时间错误
- 如果字段 ID 不保存，Schema Evolution 时无法正确匹配字段，导致数据错乱

**实际场景：**
```java
// 场景1：Paimon 写入 ORC
RowType paimonType = RowType.of(
    new DecimalType(10, 2),
    new TimestampType(6),
    new VarCharType(100)
);
TypeDescription orcType = OrcTypeUtil.toOrcType(paimonType);
// ORC: struct<decimal(10,2), timestamp, varchar(100)>

// 场景2：Schema Evolution
// 原 Schema: id INT, name STRING
// 新 Schema: id BIGINT, name STRING, age INT
// ORC 通过字段 ID 匹配：id(fieldId=0), name(fieldId=1), age(fieldId=2)

// 场景3：Parquet 精度映射
TimestampType ts1 = new TimestampType(3);  // 毫秒
TimestampType ts2 = new TimestampType(6);  // 微秒
TimestampType ts3 = new TimestampType(9);  // 纳秒
// Parquet: INT64(MILLIS), INT64(MICROS), INT96
```

### 有什么坑

**误区陷阱：**
1. **Parquet 的 INT96 陷阱**：INT96 是 Parquet 的遗留格式，只为兼容 Hive/Impala，不推荐使用
2. **ORC 的字段名大小写**：ORC 默认不区分大小写，`name` 和 `NAME` 会被视为同一字段
3. **Decimal 精度的上下界**：Parquet 的 Decimal(p<=9) 用 INT32，Decimal(9<p<=18) 用 INT64，边界容易搞错

**错误配置：**
```java
// 错误：手动构造 ORC TypeDescription 时忘记设置字段 ID
TypeDescription orcType = TypeDescription.createStruct()
    .addField("id", TypeDescription.createInt())
    .addField("name", TypeDescription.createString());
// 缺少字段 ID，Schema Evolution 时会出错

// 正确：使用 OrcTypeUtil.toOrcType
TypeDescription orcType = OrcTypeUtil.toOrcType(paimonType);

// 错误：Parquet Timestamp 精度不匹配
TimestampType paimonType = new TimestampType(9);  // 纳秒
// Parquet 只支持到微秒（INT64 MICROS），纳秒会被截断
```

**生产环境注意事项：**
1. **ORC 的时区处理**：ORC TIMESTAMP 是无时区的，TIMESTAMP_INSTANT 是带时区的，必须明确选择
2. **Parquet 的嵌套类型**：Parquet 的 List/Map 有多种编码方式（2-level, 3-level），需要保证兼容性
3. **字段 ID 的持久化**：字段 ID 必须持久化到文件元数据中，否则 Schema Evolution 会失败

**性能陷阱：**
```java
// 陷阱：频繁调用类型转换
for (int i = 0; i < 1000000; i++) {
    TypeDescription orcType = OrcTypeUtil.toOrcType(paimonType);  // 每次都重新转换
}

// 优化：缓存转换结果
TypeDescription orcType = OrcTypeUtil.toOrcType(paimonType);
for (int i = 0; i < 1000000; i++) {
    // 使用 orcType
}
```

### 核心概念解释

**术语定义：**
- **ORC TypeDescription**：ORC 的类型描述，包括类型名称、精度、字段列表等
- **Parquet Type**：Parquet 的物理类型（INT32/INT64/BINARY/...）和逻辑类型（DecimalLogicalType/TimestampLogicalType/...）
- **字段 ID（Field ID）**：Paimon 为每个字段分配的唯一标识，用于 Schema Evolution 时的字段匹配
- **列式向量化（Columnar Vectorization）**：批量读取列数据，减少虚方法调用和对象分配
- **ColumnVector**：列向量接口，ORC/Parquet 的列数据表示

**类型映射关系：**
```

-----------------------------------------------------------------
CHAR(n)              CHAR(n)                 BINARY + STRING
VARCHAR(n)           VARCHAR(n) / STRING     BINARY + STRING
DECIMAL(p<=9)        DECIMAL(p,s)            INT32 + DECIMAL
DECIMAL(9<p<=18)     DECIMAL(p,s)            INT64 + DECIMAL
DECIMAL(p>18)        DECIMAL(p,s)            FIXED_LEN_BYTE_ARRAY + DECIMAL
TIMESTAMP(p<=3)      TIMESTAMP               INT64 + TIMESTAMP(MILLIS)
TIMESTAMP(3<p<=6)    TIMESTAMP               INT64 + TIMESTAMP(MICROS)

```

### 设计理念

**为什么这样设计：**

1. **字段 ID 的引入**
   - **原因**：字段名可能被重命名，需要不变的唯一标识
   - **权衡**：需要在文件元数据中额外存储字段 ID
   - **好处**：Schema Evolution 时可以正确匹配字段，即使字段名改变

2. **Parquet Decimal 的三档映射**
   - **原因**：Parquet 是列式存储，用更小的物理类型可以显著减少列压缩后的体积
   - **权衡**：增加了映射复杂度（需要判断精度范围）
   - **好处**：p<=9 的 Decimal 用 INT32，比 INT64 节省 50% 空间

3. **Timestamp 精度的分档映射**
   - **原因**：Parquet 的 INT64 只能表示毫秒或微秒，纳秒需要 INT96
   - **权衡**：INT96 是遗留格式，不推荐使用，但为了兼容 Hive/Impala 必须支持
   - **好处**：p<=6 的 Timestamp 用 INT64，性能更好且兼容性更好

4. **列式向量化读取**
   - **原因**：ORC/Parquet 是列式存储，批量读取列数据可以减少虚方法调用
   - **权衡**：需要实现 ColumnVector 适配器，增加了代码复杂度
   - **好处**：向量化读取比逐行读取快 5-10 倍

**架构演进：**
- **早期版本**：只支持 Parquet，类型映射简单
- **当前版本**：支持 ORC/Parquet/Avro，引入字段 ID 支持 Schema Evolution
- **未来方向**：可能支持 Arrow IPC 格式，进一步提升向量化性能

**业界对比：**
- **Iceberg**：与 Paimon 类似，使用字段 ID 支持 Schema Evolution
- **Delta Lake**：不使用字段 ID，依赖字段名匹配，Schema Evolution 受限
- **Hudi**：不使用字段 ID，依赖 Avro Schema 的字段顺序
- **Paimon 优势**：字段 ID + 多种文件格式，Schema Evolution 更灵活


- `OrcLongColumnVector` -- 处理整数类型
- `OrcDoubleColumnVector` -- 处理浮点类型
- `OrcBytesColumnVector` -- 处理字符串和二进制
- `OrcDecimalColumnVector` -- 处理 Decimal
- `OrcTimestampColumnVector` -- 处理 Timestamp
- `OrcArrayColumnVector` / `OrcMapColumnVector` / `OrcRowColumnVector` -- 处理复合类型

**为什么用适配器模式而非直接读取？** ORC 的 ColumnVector 使用自己的数据表示（如 `HiveDecimalWritable`），Paimon 需要统一转换为 `Decimal`、`Timestamp` 等内部类型。适配器将转换逻辑封装在一处。

## 8. 代码生成 (Codegen)

### 解决什么问题

**核心业务问题：** 如何消除热路径上的运行时类型分发（switch/case）开销，将动态操作转化为静态调用，提升性能？

**没有这个设计的后果：**
- 如果用反射或 switch/case，每次字段访问都需要类型判断，性能损失 10-50%
- 如果用虚方法调用，JIT 编译器难以内联，无法充分优化
- 如果用手写代码，每种类型组合都需要一个类，代码膨胀且难以维护

**实际场景：**
```java
// 场景1：投影（Projection）
// 原始代码（运行时分发）
InternalRow project(InternalRow row) {
    BinaryRow result = new BinaryRow(2);
    switch (inputMapping[0]) {
        case 0: result.setInt(0, row.getInt(0)); break;
        case 1: result.setString(0, row.getString(1)); break;
        // ...
    }
    return result;
}

// 生成代码（静态调用）
InternalRow project(InternalRow row) {
    BinaryRow result = new BinaryRow(2);
    result.setInt(0, row.getInt(3));      // 直接调用，无 switch
    result.setString(1, row.getString(5));
    return result;
}

// 场景2：记录比较（RecordComparator）
// 原始代码（运行时分发）
int compare(InternalRow row1, InternalRow row2) {
    for (int i = 0; i < fieldCount; i++) {
        int cmp = compareField(row1, row2, i, fieldTypes[i]);  // 虚方法调用
        if (cmp != 0) return cmp;
    }
    return 0;
}

// 生成代码（静态调用）
int compare(InternalRow row1, InternalRow row2) {
    int cmp = Integer.compare(row1.getInt(0), row2.getInt(0));  // 直接调用
    if (cmp != 0) return cmp;
    cmp = row1.getString(1).compareTo(row2.getString(1));
    if (cmp != 0) return cmp;
    return 0;
}
```

### 有什么坑

**误区陷阱：**
1. **生成代码的缓存失效**：如果 RowType 改变，必须重新生成代码，否则会读到错误数据
2. **Janino 的编译限制**：Janino 不支持 Java 8+ 的 lambda 和 stream，生成代码必须用传统语法
3. **生成代码的调试困难**：生成的代码没有源文件，调试时只能看到字节码

**错误配置：**
```java
// 错误：缓存生成代码时忘记考虑 RowType
Map<int[], GeneratedClass<Projection>> cache = new HashMap<>();
GeneratedClass<Projection> gen = cache.computeIfAbsent(inputMapping, k -> {
    return codeGenerator.generateProjection(inputType, inputMapping);
});
// 如果 inputType 改变但 inputMapping 相同，会用到错误的缓存

// 正确：缓存 key 包含 RowType
Map<Pair<RowType, int[]>, GeneratedClass<Projection>> cache = new HashMap<>();

// 错误：生成代码中使用 lambda
String code = "return row -> row.getInt(0);";  // Janino 不支持
// 正确：使用匿名类
String code = "return new Function() { public Object apply(Object row) { return ((InternalRow)row).getInt(0); } };";
```

**生产环境注意事项：**
1. **生成代码的编译时间**：首次编译需要 10-100ms，在高并发场景下可能成为瓶颈
2. **生成代码的内存占用**：每个生成的类占用 1-10KB，大量生成会增加 Metaspace 压力
3. **生成代码的 GC 影响**：生成的类在 Metaspace 中，Full GC 时会扫描，影响 GC 时间

**性能陷阱：**
```java
// 陷阱：频繁生成代码
for (int i = 0; i < 1000; i++) {
    GeneratedClass<Projection> gen = codeGenerator.generateProjection(inputType, inputMapping);
    Projection projection = gen.newInstance(classLoader);  // 每次都编译
}

// 优化：缓存生成代码
GeneratedClass<Projection> gen = codeGenerator.generateProjection(inputType, inputMapping);
gen.compile(classLoader);  // 只编译一次
for (int i = 0; i < 1000; i++) {
    Projection projection = gen.newInstance(classLoader);
}
```

### 核心概念解释

**术语定义：**
- **CodeGenerator**：代码生成器接口，定义生成各种操作的方法
- **GeneratedClass**：生成的类，包含 Java 源码和编译后的 Class 对象
- **Janino**：轻量级 Java 编译器，用于运行时编译生成的代码
- **Projection**：列投影，将输入行的部分字段映射到输出行
- **NormalizedKeyComputer**：归一化排序键计算器，将多字段比较转化为字节序比较
- **RecordComparator**：记录比较器，比较两行记录的大小
- **RecordEqualiser**：记录等值判断器，判断两行记录是否相等

**代码生成流程：**
```
1. 分析输入类型（RowType）和操作（Projection/Comparator/...）
2. 生成 Java 源码（字符串拼接）
3. 使用 Janino 编译源码为 Class 对象
4. 缓存 Class 对象（避免重复编译）
5. 实例化 Class 对象，返回操作接口
```

**与其他系统对比：**

| 特性 | Paimon | Flink | Spark | Calcite |
|------|--------|-------|-------|---------|
| 代码生成框架 | Janino | Janino | Janino | Janino |
| 生成场景 | Projection, Comparator, Equaliser | 所有算子 | 所有算子 | SQL 表达式 |
| 缓存策略 | 基于 RowType + 操作 | 基于算子配置 | 基于 SQL 计划 | 基于表达式 |
| 编译时间 | 10-100ms | 10-100ms | 10-100ms | 10-100ms |
| 性能提升 | 10-50% | 10-50% | 10-50% | 10-50% |

### 设计理念

**为什么这样设计：**

1. **运行时代码生成而非预编译**
   - **原因**：Schema 是运行时确定的，无法预编译
   - **权衡**：首次编译有开销（10-100ms），但后续调用极快
   - **好处**：生成的代码针对具体 Schema 优化，性能最优

2. **使用 Janino 而非 javac**
   - **原因**：Janino 是纯 Java 实现，无需依赖 JDK，且编译速度快
   - **权衡**：Janino 不支持 Java 8+ 的新特性（lambda、stream）
   - **好处**：编译速度比 javac 快 5-10 倍，且无需 JDK 依赖

3. **只生成热路径操作**
   - **原因**：代码生成有编译开销，只对频繁调用的操作有收益
   - **权衡**：需要维护两套逻辑（生成版本 + 解释版本）
   - **好处**：Projection、Comparator 等热路径操作性能提升 10-50%

4. **缓存生成的 Class 对象**
   - **原因**：相同 Schema 的操作可以复用生成的代码
   - **权衡**：缓存占用 Metaspace 空间
   - **好处**：避免重复编译，首次编译后性能稳定

**架构演进：**
- **早期版本**：所有操作都是解释执行，性能较差
- **当前版本**：热路径操作使用代码生成，性能提升显著
- **未来方向**：可能引入 LLVM 或 GraalVM，生成机器码而非字节码

**业界对比：**
- **Flink**：与 Paimon 类似，使用 Janino 生成所有算子代码
- **Spark**：Tungsten 引擎使用代码生成，性能提升显著
- **Calcite**：只生成 SQL 表达式，不生成算子
- **Paimon 优势**：代码生成与 Flink 一致，无缝集成

### 8.1 CodeGenerator 接口

```
源码路径：
- paimon-common/src/main/java/org/apache/paimon/codegen/CodeGenerator.java（接口）
- paimon-codegen/src/main/java/org/apache/paimon/codegen/CodeGeneratorImpl.java（实现）
```

```java
public interface CodeGenerator {
    GeneratedClass<Projection> generateProjection(RowType inputType, int[] inputMapping);
    GeneratedClass<NormalizedKeyComputer> generateNormalizedKeyComputer(...);
    GeneratedClass<RecordComparator> generateRecordComparator(...);
    GeneratedClass<RecordEqualiser> generateRecordEqualiser(...);
}
```

### 8.2 生成场景

| 生成类 | 功能 | 为什么代码生成 |
|--------|------|-------------|
| **Projection** | InternalRow -> BinaryRow 的列投影 | 避免运行时 switch/case 分发，直接生成 `row.getInt(3)` 这样的调用 |
| **NormalizedKeyComputer** | 计算行的归一化排序 key | 将多字段比较编译为连续的字节写入，支持前缀排序优化 |
| **RecordComparator** | 两行记录的字段级比较 | 消除字段类型 dispatch 开销，直接生成类型特化的比较逻辑 |
| **RecordEqualiser** | 两行记录的等值判断 | 同上，消除类型 dispatch |

**为什么需要代码生成？** 这些操作在 LSM Compaction、Sort Merge Join 等热路径上被调用数十亿次。即使每次 switch/case 只增加几纳秒，累积起来也是可观的开销。代码生成将运行时的动态分发转化为编译时的静态调用。

**好处：**
1. JIT 编译器可以更好地内联生成的代码
2. 消除 autoboxing/unboxing
3. 消除虚方法调用

### 8.3 编译与缓存机制

```
源码路径：paimon-common/src/main/java/org/apache/paimon/codegen/CompileUtils.java
```

`GeneratedClass` 包含生成的 Java 源码，通过 Janino 编译器在运行时编译。编译结果被缓存，相同的生成代码只编译一次。

## 9. Casting 机制

### 解决什么问题

**核心业务问题：** 在 Schema Evolution 场景下，如何安全、高效地将旧 Schema 的数据转换为新 Schema 的数据？

**没有这个设计的后果：**
- 如果不支持类型转换，Schema Evolution 只能添加字段，不能修改字段类型
- 如果转换规则不完整，某些类型组合无法转换，导致历史数据无法读取
- 如果转换是全量的，即使只访问少数字段，也需要转换整行，性能差

**实际场景：**
```java
// 场景1：字段类型升级
// 原 Schema: age INT
// 新 Schema: age BIGINT
// 读取时自动转换：INT -> BIGINT

// 场景2：精度调整
// 原 Schema: price DECIMAL(10, 2)
// 新 Schema: price DECIMAL(12, 4)
// 读取时自动转换：DECIMAL(10,2) -> DECIMAL(12,4)

// 场景3：字符串转数值
// 原 Schema: count STRING
// 新 Schema: count INT
// 读取时自动转换：STRING -> INT（如果字符串是数字）

// 场景4：懒转换
CastedRow castedRow = new CastedRow(oldRow, castFieldGetters);
int age = castedRow.getInt(0);  // 只转换第 0 个字段
// 其他字段不访问，不转换
```

### 有什么坑

**误区陷阱：**
1. **转换的安全性**：并非所有类型转换都是安全的，如 BIGINT -> INT 可能溢出
2. **转换的精度丢失**：DECIMAL(10,4) -> DECIMAL(10,2) 会截断小数位
3. **转换的性能开销**：STRING -> INT 需要解析字符串，比直接读取 INT 慢 10-100 倍

**错误配置：**
```java
// 错误：不安全的类型转换
// 原 Schema: id BIGINT (值为 10000000000)
// 新 Schema: id INT
// 转换时会溢出，导致数据错误

// 错误：忽略转换失败
CastExecutor<BinaryString, Integer> executor = ...;
Integer value = executor.cast(BinaryString.fromString("abc"));  // 抛异常
// 正确：捕获异常或使用 trycast
try {
    Integer value = executor.cast(BinaryString.fromString("abc"));
} catch (Exception e) {
    // 处理转换失败
}

// 错误：频繁创建 CastedRow
for (int i = 0; i < 1000000; i++) {
    CastedRow castedRow = new CastedRow(row, castFieldGetters);  // 每次都创建新对象
}
// 正确：复用 CastedRow
CastedRow castedRow = new CastedRow(row, castFieldGetters);
for (int i = 0; i < 1000000; i++) {
    castedRow.replace(row);  // 复用对象
}
```

**生产环境注意事项：**
1. **转换规则的完整性**：确保所有需要的类型转换都有对应的 CastRule
2. **转换失败的处理**：转换失败时应该返回 NULL 还是抛异常，需要明确定义
3. **转换的性能影响**：大量字段转换会显著降低查询性能，建议重写数据

**性能陷阱：**
```java
// 陷阱：全量转换
InternalRow newRow = convertRow(oldRow, oldSchema, newSchema);  // 转换所有字段
int age = newRow.getInt(0);  // 只访问一个字段

// 优化：懒转换
CastedRow castedRow = new CastedRow(oldRow, castFieldGetters);
int age = castedRow.getInt(0);  // 只转换访问的字段
```

### 核心概念解释

**术语定义：**
- **CastRule**：类型转换规则，定义支持的输入/输出类型和转换逻辑
- **CastExecutor**：类型转换执行器，执行具体的转换操作
- **CastRulePredicate**：转换规则谓词，声明支持的输入/输出类型
- **CastedRow**：懒转换包装器，按需转换字段
- **CastFieldGetter**：字段转换访问器，从底层行读取原始值并执行转换
- **安全转换**：不会丢失精度或溢出的转换，如 INT -> BIGINT
- **不安全转换**：可能丢失精度或溢出的转换，如 BIGINT -> INT

**转换规则分类：**
```
数值类转换
├── NumericPrimitiveCastRule (int/long/float/double 互转)
├── DecimalToDecimalCastRule (Decimal 精度调整)
├── NumericPrimitiveToDecimalCastRule (int/long -> Decimal)
└── DecimalToNumericPrimitiveCastRule (Decimal -> int/long)

布尔与数值互转
├── BooleanToNumericCastRule (boolean -> 0/1)
└── NumericToBooleanCastRule (0 -> false, 非 0 -> true)

To String 转换
├── NumericToStringCastRule
├── BooleanToStringCastRule
├── TimestampToStringCastRule
└── ...

From String 转换
├── StringToNumericPrimitiveCastRule
├── StringToBooleanCastRule
├── StringToTimestampCastRule
└── ...

日期/时间互转
├── TimestampToTimestampCastRule (精度调整)
├── TimestampToDateCastRule
├── DateToTimestampCastRule
└── ...
```

**与其他系统对比：**

| 特性 | Paimon | Flink | Spark | Hive |
|------|--------|-------|-------|------|
| 类型转换支持 | 完整 | 完整 | 完整 | 部分 |
| 懒转换 | 是（CastedRow） | 否 | 否 | 否 |
| 转换规则数量 | 30+ | 50+ | 100+ | 20+ |
| 安全性检查 | 运行时 | 运行时 | 编译时 | 运行时 |
| 性能优化 | 懒转换 | 无 | 无 | 无 |

### 设计理念

**为什么这样设计：**

1. **Rule + Predicate 模式**
   - **原因**：一条 Rule 可以匹配一类类型（通过 DataTypeFamily）
   - **权衡**：增加了规则匹配的复杂度
   - **好处**：`NumericPrimitiveCastRule` 一条规则覆盖了所有整数类型之间的转换

2. **懒转换（CastedRow）**
   - **原因**：Schema Evolution 场景下，新旧 Schema 之间可能只有少数字段类型不同
   - **权衡**：CastedRow 的实现复杂，需要维护 CastFieldGetter 数组
   - **好处**：如果一行有 100 个字段但只访问了 5 个，只有这 5 个字段会执行转换

3. **CastExecutor 接口**
   - **原因**：转换逻辑与规则匹配分离，便于测试和复用
   - **权衡**：增加了接口层次
   - **好处**：同一个 CastExecutor 可以被多个 CastFieldGetter 复用

4. **完整的转换规则集**
   - **原因**：Schema Evolution 需要支持尽可能多的类型转换
   - **权衡**：需要维护大量转换规则（30+ 条）
   - **好处**：支持几乎所有常见的类型转换场景

**架构演进：**
- **早期版本**：只支持安全转换（INT -> BIGINT），不支持 String 转换
- **当前版本**：支持完整的转换规则集，包括 String 转换和复合类型转换
- **未来方向**：可能引入用户自定义转换规则（UDF）

**业界对比：**
- **Flink**：转换规则更多（50+），但没有懒转换
- **Spark**：转换规则最多（100+），且在编译时检查安全性
- **Hive**：转换规则较少（20+），且不支持复合类型转换
- **Paimon 优势**：懒转换在 Schema Evolution 场景下性能最优

### 9.1 CastRule 与 CastExecutor

```
源码路径：paimon-common/src/main/java/org/apache/paimon/casting/
```

类型转换采用**规则匹配**模式：

```
CastRule<IN, OUT> (interface)
├── getPredicateDefinition(): CastRulePredicate  -- 声明支持的输入/输出类型
└── create(inputType, outputType): CastExecutor   -- 创建具体的转换执行器

CastExecutor<IN, OUT> (interface)
└── cast(IN value): OUT                           -- 执行转换
```

**为什么用 Rule + Predicate 而非直接注册 (inputType, outputType) -> executor？** 一条 Rule 可以匹配一类类型（通过 DataTypeFamily），避免为每对具体类型都写一条规则。例如 `NumericPrimitiveCastRule` 一条规则覆盖了所有整数类型之间的转换。

### 9.2 CastedRow 懒转换包装

```
源码路径：paimon-common/src/main/java/org/apache/paimon/casting/CastedRow.java
```

`CastedRow` 是一个 `InternalRow` 包装器，持有 `CastFieldGetter[]` 数组。每次读取字段时，通过对应的 CastFieldGetter 从底层行读取原始值并执行类型转换。

**为什么是懒转换？** 如果一行有 100 个字段但只访问了 5 个，只有这 5 个字段会执行转换。

**好处：** Schema Evolution 场景下，新旧 schema 之间可能只有少数字段类型不同。CastedRow 使得这些字段可以按需转换，不需要预先转换整行。

### 9.3 完整转换规则列表

**数值类转换：**
- `DecimalToDecimalCastRule` -- Decimal 精度/scale 调整
- `NumericPrimitiveToDecimalCastRule` -- int/long -> Decimal
- `DecimalToNumericPrimitiveCastRule` -- Decimal -> int/long
- `NumericPrimitiveCastRule` -- int/long/float/double 之间互转
- `NumericPrimitiveToTimestamp` -- 数值 -> 时间戳

**布尔与数值互转：**
- `BooleanToNumericCastRule` -- boolean -> 0/1
- `NumericToBooleanCastRule` -- 0 -> false, 非 0 -> true

**To String 转换：**
- `NumericToStringCastRule` / `BooleanToStringCastRule` / `TimestampToStringCastRule` / `DateToStringCastRule` / `TimeToStringCastRule` / `StringToStringCastRule` / `ArrayToStringCastRule` / `MapToStringCastRule` / `RowToStringCastRule`

**From String 转换：**
- `StringToBooleanCastRule` / `StringToDecimalCastRule` / `StringToNumericPrimitiveCastRule` / `StringToDateCastRule` / `StringToTimeCastRule` / `StringToTimestampCastRule` / `StringToBinaryCastRule` / `StringToArrayCastRule` / `StringToMapCastRule` / `StringToRowCastRule`

**日期/时间互转：**
- `TimestampToTimestampCastRule` / `TimestampToDateCastRule` / `TimestampToTimeCastRule` / `DateToTimestampCastRule` / `TimeToTimestampCastRule` / `TimestampToNumericPrimitiveCastRule`

**二进制转换：**
- `BinaryToBinaryCastRule` / `BinaryToStringCastRule`

## 10. 与 Flink/Spark 的类型映射

### 解决什么问题

**核心业务问题：** 如何在 Paimon 和 Flink/Spark 之间高效传递数据，同时保证类型语义一致、减少数据拷贝？

**没有这个设计的后果：**
- 如果每次传递都深拷贝，性能损失 50-90%
- 如果类型映射不正确，Decimal/Timestamp 等复杂类型会出现精度错误
- 如果时区处理不当，TIMESTAMP 和 TIMESTAMP_LTZ 会混淆，导致时间错误

**实际场景：**
```java
// 场景1：Flink 读取 Paimon（零拷贝）
InternalRow paimonRow = paimonReader.read();
RowData flinkRow = new FlinkRowData(paimonRow);  // 零拷贝包装
int age = flinkRow.getInt(0);  // 直接委托到 paimonRow.getInt(0)

// 场景2：Spark 写入 Paimon（字段映射）
catalyst.InternalRow sparkRow = ...;
InternalRow paimonRow = new SparkInternalRowWrapper(sparkRow, fieldIndexMap);
// fieldIndexMap 处理字段顺序不同的情况

// 场景3：Timestamp 时区处理
// Spark TIMESTAMP 是 UTC 时间
// Paimon TIMESTAMP 是无时区时间
// 需要在适配层处理时区转换
```

### 有什么坑

**误区陷阱：**
1. **BinaryString 的字节拷贝**：Flink/Spark 的 StringData 和 Paimon 的 BinaryString 都是 UTF-8，但需要字节拷贝
2. **Timestamp 的时区陷阱**：Spark 的 TIMESTAMP 是 UTC，Paimon 的 TIMESTAMP 是无时区，混用会导致时间错误
3. **字段顺序的映射**：Spark 的 data schema 和 table schema 字段顺序可能不同，需要 fieldIndexMap

**错误配置：**
```java
// 错误：假设 FlinkRowData 是深拷贝
InternalRow paimonRow = ...;
RowData flinkRow = new FlinkRowData(paimonRow);
paimonRow = null;  // paimonRow 被回收
int age = flinkRow.getInt(0);  // 可能读到错误数据

// 正确：FlinkRowData 只是包装，paimonRow 必须保持有效
InternalRow paimonRow = ...;
RowData flinkRow = new FlinkRowData(paimonRow);
int age = flinkRow.getInt(0);  // paimonRow 仍然有效

// 错误：忽略字段顺序映射
// Spark data schema: [name, age, city]
// Paimon table schema: [age, name, city]
InternalRow paimonRow = new SparkInternalRowWrapper(sparkRow, null);  // 错误：没有 fieldIndexMap
int age = paimonRow.getInt(0);  // 读到的是 name，不是 age

// 正确：使用 fieldIndexMap
int[] fieldIndexMap = {1, 0, 2};  // data[0] -> table[1], data[1] -> table[0], data[2] -> table[2]
InternalRow paimonRow = new SparkInternalRowWrapper(sparkRow, fieldIndexMap);
```

**生产环境注意事项：**
1. **内存生命周期管理**：Wrapper 对象只是引用，底层数据必须保持有效
2. **时区配置一致性**：Flink/Spark 的时区配置必须与 Paimon 一致，否则 Timestamp 会错误
3. **字段映射的性能开销**：fieldIndexMap 会增加字段访问开销，建议保持字段顺序一致

**性能陷阱：**
```java
// 陷阱：频繁调用 toString()
for (int i = 0; i < 1000000; i++) {
    String str = flinkRow.getString(0).toString();  // 每次都创建新 String
}

// 优化：缓存 String
String str = flinkRow.getString(0).toString();
for (int i = 0; i < 1000000; i++) {
    // 使用 str
}
```

### 核心概念解释

**术语定义：**
- **FlinkRowData**：Paimon InternalRow -> Flink RowData 的包装器，用于读路径
- **FlinkRowWrapper**：Flink RowData -> Paimon InternalRow 的包装器，用于写路径
- **SparkRow**：Spark Row -> Paimon InternalRow 的包装器，用于 V1 API 写路径
- **SparkInternalRowWrapper**：Spark InternalRow -> Paimon InternalRow 的包装器，用于 V2 API 写路径
- **AbstractSparkInternalRow**：Paimon -> Spark InternalRow 的基类，用于读路径
- **零拷贝（Zero-Copy）**：直接传递引用，不拷贝数据
- **字段映射（Field Mapping）**：处理字段顺序不同的情况

**适配器层次：**
```
Flink 适配层
├── FlinkRowData (Paimon -> Flink 读路径)
│   └── 基本类型零拷贝，String/Decimal/Timestamp 轻量转换
└── FlinkRowWrapper (Flink -> Paimon 写路径)
    └── 基本类型零拷贝，String/Decimal/Timestamp 轻量转换

Spark 适配层
├── SparkRow (Spark Row -> Paimon 写路径，V1 API)
│   └── 所有字段都需要 Object 级别的类型转换
├── SparkInternalRowWrapper (Spark InternalRow -> Paimon 写路径，V2 API)
│   ├── 基本类型零拷贝
│   ├── 字段索引映射（fieldIndexMap）
│   └── Timestamp 时区处理
└── AbstractSparkInternalRow (Paimon -> Spark 读路径)
    └── 基本类型零拷贝，String/Decimal/Timestamp 轻量转换
```

**与其他系统对比：**

| 特性 | Paimon-Flink | Paimon-Spark | Iceberg-Flink | Iceberg-Spark |
|------|-------------|-------------|--------------|--------------|
| 基本类型零拷贝 | 是 | 是 | 是 | 是 |
| String 零拷贝 | 否（字节拷贝） | 否（字节拷贝） | 否 | 否 |
| Decimal 零拷贝 | 紧凑模式是 | 紧凑模式是 | 否 | 否 |
| Timestamp 零拷贝 | 是 | 是（需时区转换） | 是 | 是 |
| 字段映射 | 否 | 是 | 否 | 是 |
| 适配器类型 | Wrapper | Wrapper | Converter | Converter |

### 设计理念

**为什么这样设计：**

1. **Wrapper 模式而非 Converter**
   - **原因**：Wrapper 是零拷贝的，Converter 需要深拷贝
   - **权衡**：Wrapper 的生命周期管理复杂，底层数据必须保持有效
   - **好处**：基本类型（int/long/double）完全零拷贝，性能极高

2. **String 的字节拷贝**
   - **原因**：Flink/Spark 的 StringData 和 Paimon 的 BinaryString 指向各自的内存管理系统
   - **权衡**：String 拷贝有少量开销（通常 < 5%）
   - **好处**：内存安全，避免 use-after-free 或内存泄漏

3. **Spark 的字段映射**
   - **原因**：Spark 的 data schema 和 table schema 字段顺序可能不同
   - **权衡**：字段访问需要额外的数组查找（fieldIndexMap[pos]）
   - **好处**：支持 Spark 的列裁剪和字段重排序优化

4. **Timestamp 的时区处理**
   - **原因**：Spark 的 TIMESTAMP 是 UTC，Paimon 的 TIMESTAMP 是无时区
   - **权衡**：需要在适配层处理时区转换，增加复杂度
   - **好处**：保证时间语义正确，避免跨时区查询错误

**架构演进：**
- **早期版本**：使用 Converter 模式，所有字段都深拷贝，性能差
- **当前版本**：使用 Wrapper 模式，基本类型零拷贝，性能提升 50-90%
- **未来方向**：可能引入 Arrow 格式，进一步减少拷贝

**业界对比：**
- **Iceberg**：使用 Converter 模式，所有字段都深拷贝，性能不如 Paimon
- **Delta Lake**：直接使用 Spark 的类型系统，与 Spark 深度绑定，不支持 Flink
- **Hudi**：使用 Avro 作为中间格式，需要两次转换（Flink/Spark -> Avro -> Hudi）
- **Paimon 优势**：Wrapper 模式 + 零拷贝，性能最优，且同时支持 Flink/Spark

### 10.1 Flink 适配层

```
源码路径：paimon-flink/paimon-flink-common/src/main/java/org/apache/paimon/flink/
```

**两个方向的适配器：**

| 类 | 方向 | 作用 |
|----|------|------|
| `FlinkRowData` | Paimon -> Flink | 将 Paimon `InternalRow` 包装为 Flink `RowData` |
| `FlinkRowWrapper` | Flink -> Paimon | 将 Flink `RowData` 包装为 Paimon `InternalRow` |

**FlinkRowData（Paimon -> Flink 读路径）：**
- 基本类型（boolean/byte/short/int/long/float/double/byte[]）: **零拷贝** -- 直接委托调用
- `BinaryString -> StringData`: 需要字节拷贝 -- `StringData.fromBytes(str.toBytes())`
- `Decimal -> DecimalData`: 紧凑模式零拷贝 -- `DecimalData.fromUnscaledLong(...)`
- `Timestamp -> TimestampData`: 零拷贝 -- `TimestampData.fromEpochMillis(millis, nanoOfMillisecond)`
- `InternalArray/InternalMap/InternalRow`: 递归包装 -- 创建 `FlinkArrayData/FlinkMapData/FlinkRowData`

**FlinkRowWrapper（Flink -> Paimon 写路径）：**
- 基本类型: **零拷贝** -- 直接委托
- `StringData -> BinaryString`: 字节拷贝 -- `BinaryString.fromBytes(str.toBytes())`
- `DecimalData -> Decimal`: 紧凑模式零拷贝
- `TimestampData -> Timestamp`: 零拷贝
- RowKind 映射: Flink `RowKind` <-> Paimon `RowKind` 枚举转换

**为什么 Flink 适配层不能完全零拷贝？** Flink 的 `StringData` 和 Paimon 的 `BinaryString` 都基于 UTF-8 字节数组，但分别指向各自框架的内存管理系统。`toBytes()` 会产生一次拷贝。然而，在基本数值类型上确实是零拷贝的，因为它们的内部表示完全一致（Java 原生类型）。

### 10.2 Spark 适配层

```
源码路径：paimon-spark/paimon-spark-common/src/main/java/org/apache/paimon/spark/
```

**三个适配器类：**

| 类 | 方向 | 作用 |
|----|------|------|
| `SparkRow` | Spark Row -> Paimon InternalRow | V1 API 写路径，包装 Spark 外部 `Row` |
| `SparkInternalRowWrapper` | Spark InternalRow -> Paimon InternalRow | V2 API 写路径，包装 Spark 内部行 |
| `AbstractSparkInternalRow` | Paimon -> Spark InternalRow | 读路径基类 |

**SparkInternalRowWrapper 的设计亮点：**

1. **字段索引映射**（第 92-107 行）：当 data schema 与 table schema 字段顺序不同时，通过 `fieldIndexMap` 做重映射
2. **缺失字段处理**：映射为 -1 的字段自动返回 null
3. **Timestamp 的时区处理**（第 304-316 行）：区分 `TimestampType`（NTZ）和 `TimestampNTZType`，处理 Spark 的 UTC 调整逻辑

**SparkRow 与 SparkInternalRowWrapper 的区别：**
- `SparkRow` 包装 Spark 的外部 `Row` 对象（Java 对象），所有 getter 需要 Object 级别的类型转换
- `SparkInternalRowWrapper` 包装 Spark 的内部 `catalyst.InternalRow`（Tungsten 格式），getter 直接读取二进制数据

**为什么 Spark 需要两套适配器？** Spark 的 V1 API（DataFrame API）使用外部 `Row` 格式，V2 API（DataSource V2）使用内部 `catalyst.InternalRow` 格式。两者的数据表示完全不同。

### 10.3 零拷贝与数据拷贝的权衡

**完全零拷贝的字段类型（Flink/Spark -> Paimon）：**
- boolean、byte、short、int、long、float、double
- byte[]（二进制数据直接传递引用）

**需要轻量转换的字段类型：**
- Decimal 紧凑模式（unscaled long 传递）
- Timestamp（millisecond + nanoOfMillisecond 传递）

**需要字节拷贝的字段类型：**
- String（UTF-8 字节数组拷贝）
- Decimal 非紧凑模式（BigDecimal 转换）

**需要递归包装的字段类型：**
- Array、Map、Row（创建 wrapper 对象，延迟到字段访问时转换）

**为什么不在 Flink/Spark 适配层中直接引用 Paimon 的 MemorySegment？** Flink 和 Spark 各有自己的内存管理体系（Flink 的 MemorySegment、Spark 的 Tungsten UnsafeRow），它们的生命周期和内存回收策略不同。直接引用可能导致 use-after-free 或内存泄漏。字节拷贝虽然有少量开销，但保证了内存安全。

## 总结：核心设计决策一览

| 设计决策 | 为什么 | 好处 |
|---------|--------|------|
| 类型系统放在 paimon-api 模块 | 外部 Connector 只需依赖轻量 API | 降低模块耦合 |
| DataTypeRoot + DataTypeFamily 双层枚举 | 精确匹配 + 批量匹配 | 简化类型转换和谓词下推逻辑 |
| InternalRow 定义为接口 | 不同场景需要不同的内存布局 | 上层逻辑与底层表示解耦 |
| BinaryRow 每字段 8 字节对齐 | O(1) 随机访问 + CPU 字对齐 | 投影和排序场景极快 |
| 变长数据短内联策略（<=7 字节） | 大量短字符串无需访问变长区域 | 减少 cache miss |
| Decimal 双模存储（long / BigDecimal） | 绝大多数精度 <= 18 | 紧凑模式下操作与 long 等价 |
| Timestamp 分离存储（millis + nanoOfMillisecond） | 低精度只需一个 long | 兼顾范围和精度 |
| RowCompactedSerializer 紧凑编码 | 存储场景 I/O 敏感 | 比 BinaryRow 格式小 30-50% |
| 代码生成（Janino） | 热路径消除 switch/case | JIT 更好内联，消除虚调用 |
| Casting 使用 Rule + Predicate | 一条规则覆盖一类类型 | Schema Evolution 灵活高效 |
| Flink/Spark 适配使用 Wrapper 模式 | 两套内存管理体系不能直接引用 | 内存安全，按需延迟转换 |
